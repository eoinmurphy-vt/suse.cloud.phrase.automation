= コンポーネントにカスタムチャートを追加する
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: SUSE Observability

== 概要

{stackstate-product-name} は、Kubernetesリソースを表すコンポーネントのほとんどのタイプで、すでにデフォルトで多くのメトリックチャートを提供している。必要なときはいつでも、どのコンポーネント・セットにも、追加のメートル法チャートを追加することができる。コンポーネントにメトリクスを追加する場合、2つのオプションがあります：

. メトリクスは{stackstate-product-name} によって既に収集されていますが、デフォルトではコンポーネント上で視覚化されていません。
. このメトリクスはまだ{stackstate-product-name} 、まったく収集されていないため、まだ利用できない。

オプション1については、以下の手順で、{stackstate-product-name} 、特定のコンポーネントのセットに特定のメトリックを追加するように設定するメトリックバインディングの作成方法を説明します。

オプション 2 では、xref:/use/metrics/k8s-prometheus-remote-write.adoc[Prometheus のリモート書き込みプロトコルを]使用して{stackstate-product-name} にメトリッ クを送信し、{stackstate-product-name} で利用可能であることを確認します。メトリクスが利用可能であることを確認した後、メトリクスのチャートをコンポーネントに追加します。

== メトリック・バインディングの作成

. <<_create_an_outline_of_the_metric_binding,メトリックバインディングのアウトラインを作成する>>
. <<_select_the_components_to_bind_to,バインドするコンポーネントを選択する>>
. <<_write_the_promql_query,希望するメトリックのPromQLクエリを書く>>
. <<_bind_the_correct_time_series_to_each_component,各コンポーネントに正しい時系列をバインドする>>

例えば、Kubernetesデプロイの`+Replica counts+` のメトリックバインディングを追加します。このメトリック・バインディングは、デフォルトで{stackstate-product-name} にすでに存在している。

=== メトリックバインディングのアウトラインを作成する

`+metricbindings.yaml+` YAMLファイルを好きなコードエディターで開き、このガイドを通して変更してください。 CLIを使ってxref:/setup/custom-integrations/develop.adoc#_test_your_stackpack[StackPackをテスト]することができます。

----
- _type: MetricBinding
  chartType: line
  enabled: true
  identifier: urn:stackpack:my-stackpack:shared:metric-binding:node-memory-bytes-available-scheduling
  layout:
    metricPerspective:
      section: Resources
      tab: Kubernetes Node
  name: Memory available for scheduling (Custom)
  priority: medium
  queries:
  - alias: ${cluster_name} - ${node}
    expression: max_over_time(kubernetes_state_node_memory_allocatable{cluster_name="${tags.cluster-name}", node="${name}"}[${__interval}])
  scope: (label = "stackpack:kubernetes" and type = "node")
  unit: bytes(IEC)
----

クエリーとスコープのセクションは次のステップで記入する。使用される単位は`+short+` で、単に数値を表示するだけであることに注意。PromQLクエリを作成する際に、正しい単位を決めることができます。

=== バインドするコンポーネントを選択する

xref:/use/views/k8s-topology-perspective.adoc[Topology」パースペクティブの]ビューを保存し、フィルタ（Filters-> Topology-> Switch to STQL）を使用して、新しいメトリックを表示する必要があるコンポーネントをクエリします。メトリックバインディングのためにトポロジーを選択する最も一般的なフィールドは、コンポーネントタイプのための`+type+` 、すべてのラベルを選択するための`+label+` 。例えばデプロイメント：

----
type = "deployment" and label = "stackpack:kubernetes"
----

タイプフィルターはすべてのデプロイメントを選択し、ラベルフィルターはKubernetes stackpackによって作成されたコンポーネントのみを選択します（ラベル名は`+stackpack+` 、ラベル値は`+kubernetes+` ）。後者は省略しても同じ結果になる。 すべてのxref:/develop/reference/k8sTs-stql_reference.adoc#_component_filters[STQLクエリ・コンポーネント・フィルターは]フィルタリングに使用できます。

アドバンスモードに切り替えて、結果のトポロジークエリーをコピーし、それをメトリックバインディングの`+scope+` フィールドに入れる。

[NOTE]
====
メトリックバインディングはクエリフィルタのみをサポートします。`+withNeighborsOf+` のようなクエリー関数はサポートされておらず、使用できない。
====


=== PromQLクエリを書く

{stackstate-product-name} インスタンスのxref:/use/metrics/k8sTs-explore-metrics.adoc[メトリックエクスプローラー]（http://your-instance/#/metrics, ）にアクセスし、目的のメトリックのクエリーに使用する。エクスプローラーには、メトリクス、ラベル、ラベル値だけでなく、PromQL関数や演算子のオートコンプリートが用意されています。最良の結果を得るために、例えば1時間といった短い時間範囲から始める。

レプリカの総数については、`+kubernetes_state_deployment_replicas+` メトリックを使用する。時系列データのメトリクス・チャートを表示するには、`+${__interval}+` パラメータを使用して集約を行うようにクエリを拡張します：

----
max_over_time(kubernetes_state_deployment_replicas[${__interval}])
----

この具体的なケースでは、`+max_over_time+` を使用して、チャートが常に任意の時点で最も多いレプリカ数を表示するようにする。より長い時間範囲では、レプリカの短いディップは表示されない。最小のレプリカ数を強調したい場合は、代わりに`+min_over_time+` 。

このクエリーを、メトリックバインディングの`+queries+` フィールドの最初のエントリーの`+expression+` プロパティにコピーする。チャート凡例に表示するには、`+Total replicas+` をエイリアスとして使用する。

[NOTE]
====
{stackstate-product-name} では、メトリックス・チャートのサイズは、チャートに表示されるメトリックスの粒度を自動的に決定する。PromQLのクエリはこの動作を最適に利用するように調整することができ、指標の代表的なチャートを得ることができます。xref:/setup/custom-integrations/metric-bindings/writing-promql.adoc[チャート用のPromQLの書き方で]詳しく説明しています。
====


=== 各コンポーネントに正しい時系列をバインドする

すべてのフィールドが入力されたメトリックバインディング：

----
_type: MetricBinding
chartType: line
enabled: true
tags: {}
unit: short
name: Replica counts
priority: MEDIUM
identifier: urn:stackpack:my-stackpack:metric-binding:my-deployment-replica-counts
queries:
  - expression: max_over_time(kubernetes_state_deployment_replicas[${__interval}])
    alias: Total replicas
scope: type = "deployment" and label = "stackpack:kubernetes"
----

{stackstate-product-name} で作成し、デプロイメント・コンポーネントの「Replica count」チャートを表示すると、予期しない結果が得られました。このグラフは、すべてのデプロイメントのレプリカ数を示している。論理的には、この特定のデプロイメントのレプリカカウントという1つの時系列しか期待できない。

image::k8s/k8s-replica-counts-without-binding.png[単一配備の誤ったチャート, it shows the replica count for all deployments]

この問題を解決するには、コンポーネントからの情報を使用して、コンポーネントに特化したPromQLクエリを作成します。十分なメトリック・ラベルでフィルタリングし、コンポーネントの特定の時系列のみを選択する。これは、正しい時系列をコンポーネントに「バインド」することである。Grafanaダッシュボードを作った経験のある人にとっては、これはダッシュボード上のクエリで使用されるパラメータを持つダッシュボードに似ている。メトリック・バインディングのクエリをこう変えてみよう：

----
max_over_time(kubernetes_state_deployment_replicas{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", deployment="${name}"}[${__interval}])
----

image::k8s/k8s-replica-counts-with-binding.png[パラメタライズド・フィルターを追加すると、チャートは期待通りになる。, only one time series for this component]

PromQLクエリは、`+cluster_name+` 、`+namespace+` 、`+deployment+` の3つのラベルでフィルタリングするようになりました。これらのラベルに実際の値を指定する代わりに、コンポーネントのフィールドへの変数参照が使われる。この場合、`+cluster-name+` と`+namespace+` のラベルが使用され、`+${tags.cluster-name}+` と`+${tags.namespace}+` が参照される。さらに、コンポーネント名は`+${name}+` で参照される。

サポートされている変数参照は以下の通り：

* コンポーネントのラベルは `+${tags.<label-name>}+`
* コンポーネント名は `+${name}+`

image::k8s/k8s-carts-highlights.png[ラベルとコンポーネント名を表示するコンポーネントハイライトページ（どちらも赤でハイライトされている）]

[NOTE]
====
Kubernetesから特定のコンポーネントのメトリクスを選択するには、通常、クラスタ名、名前空間、コンポーネントタイプと名前の組み合わせで十分です。これらのラベル、または同様のラベルは、通常、ほとんどの測定基準や部品に貼られている。
====


== 上級

=== チャート内に複数の時系列を表示

[NOTE]
====
メートル装丁の単位は1つしかない（チャートのY軸にプロットされる）。その結果、1つのメトリックバインディングで同じ単位の時系列を生成するクエリのみを組み合わせる必要があります。単位を変換できる場合もある。例えば、CPU使用率はミリコアまたはコアで報告されるかもしれない。ミリコアは、次のように1000倍することでコアに変換できる`+(<original-query>) * 1000+` 。
====


1つのメトリック・バインディングで複数の時系列を取得し、したがって1つのチャートで取得するには、2つの方法がある：

. 1つのコンポーネントについて複数の時系列を返すPromQLクエリーを書く
. メトリック・バインディングにPromQLクエリを追加

最初のオプションについては、xref:/setup/custom-integrations/metric-bindings/index.adoc#_using_metric_labels_in_aliases[次のセクションで]例を挙げる。2番目のオプションは、関連する測定基準を比較するのに便利です。典型的な使用例をいくつか挙げる：

* 総レプリカ数と希望レプリカ数と利用可能レプリカ数の比較
* リソースの使用状況：制限、リクエスト、使用量を1つのグラフで表示

メトリックバインディングにさらにクエリーを追加するには、xref:/setup/custom-integrations/metric-bindings/index.adoc#_steps[手順]3.と4.を繰り返し、クエリーリストに追加項目としてクエリーを追加するだけです。デプロイメントレプリカ数については、同じチャートに含めることができるいくつかの関連するメトリクスがあります：

----
- _type: MetricBinding
  chartType: line
  enabled: true
  tags: {}
  unit: short
  name: Replica counts
  priority: MEDIUM
  identifier: urn:stackpack:my-stackpack:metric-binding:my-deployment-replica-counts
  queries:
    - expression: max_over_time(kubernetes_state_deployment_replicas{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", deployment="${name}"}[${__interval}])
      alias: Total replicas
    - expression: max_over_time(kubernetes_state_deployment_replicas_available{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}",  deployment="${name}"}[${__interval}])
      alias: Available - ${cluster_name} - ${namespace} - ${deployment}
    - expression: max_over_time(kubernetes_state_deployment_replicas_unavailable{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}",  deployment="${name}"}[${__interval}])
      alias: Unavailable - ${cluster_name} - ${namespace} - ${deployment}
    - expression: min_over_time(kubernetes_state_deployment_replicas_desired{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}",  deployment="${name}"}[${__interval}])
      alias: Desired - ${cluster_name} - ${namespace} - ${deployment}
  scope: type = "deployment" and label = "stackpack:kubernetes"
----

image::k8s/k8s-replica-counts-multiple-timeseries.png[複数のメトリクスによるメトリック・バインディング]

=== エイリアスにメトリック・ラベルを使用する

1つのクエリがコンポーネントごとに複数の時系列を返す場合、これはチャートに複数の線として表示される。しかし、凡例ではすべて同じエイリアスを使用する。異なる時系列間の違いを見ることができるように、エイリアスは、`+${label}+` 構文を使用してメトリック・ラベルへの参照を含めることができる。ポッドは複数のコンテナを持つことができることに注意してください：

----
type: MetricBinding
chartType: line
enabled: true
id: -1
identifier: urn:stackpack:my-stackpack:metric-binding:my-pod-restart-count
name: Container restarts
priority: MEDIUM
queries:
- alias: Restarts - ${container}
  expression: max by (cluster_name, namespace, pod_name, container) (kubernetes_state_container_restarts{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", pod_name="${name}"})
scope: (label = "stackpack:kubernetes" and type = "pod")
unit: short
----

`+alias+` 、メトリックの`+container+` ラベルを参照していることに注意。ラベルがない場合、`+${container}+` はトラブルシューティングに役立つリテラルテキストとしてレンダリングされます。

=== レイアウト

各コンポーネントは、k8s、ネットワーキング、実行環境（JVMなど）、プロトコル（HTTP、AMQP）など、さまざまな技術やプロトコルに関連付けることができる。
その結果、各コンポーネントについて多数の異なるメトリクスを表示することができる。読みやすくするために、{stackstate-product-name} 、これらのチャートをタブやセクションに整理することができる。
チャート(`+MetricBinding+`)を特定のタブやセクション内に表示するには、レイアウト・プロパティを設定する必要があります。
レイアウトが指定されていないMetricsBindingは、`+Other+` という名前のタブとセクションに表示されます。
