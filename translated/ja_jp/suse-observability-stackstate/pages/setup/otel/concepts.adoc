= オープン・テレメトリーのコンセプト
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: SUSE Observability

これは、オープン・テレメトリーにおける最も重要なコンセプトの要約であり、始めるには十分であろう。より詳細な紹介は、https://opentelemetry.io/docs/concepts/[Open Telemetry のドキュメントを]ご覧ください。

== 信号

オープン・テレメトリーは3つのテレメトリー信号を認識します：

* 痕跡
* 指標
* 過去ログ

現時点では、SUSE Observabilityはトレースとメトリクスをサポートしており、ログは将来のバージョンでサポートされる予定です。Kubernetesのログについては、代わりにxref:/k8s-quick-start-guide.adoc[SUSE Observabilityエージェントを]使用することが可能です。

=== 痕跡

トレースによって、アプリケーションを通過するリクエストのパスを可視化することができます。トレースは、1つまたは複数のスパンから構成され、それらが組み合わさってツリーを形成する。1つのトレースは、完全に1つのサービス内に存在することもあれば、多くのサービスにまたがることもある。それぞれのスパンは、リクエストの処理におけるオペレーションを表し、以下のような特徴がある：

* 名前
* 開始時刻と終了時刻から、継続時間を計算することができる。
* ステータス
* 属性
* リソース属性（<<_resources,リソース>>参照）
* イベント

スパンの属性は、スパンのメタデータを提供するために使用されます。例えば、注文を行う操作のスパンは、属性として`+orderId+` 、HTTP操作のスパンは、属性としてHTTPメソッドとURLを持つことができます。

スパン・イベントは、スパンの運用の中で何か重要なことが起こった時点を表すために使用することができる。例えば、スパンが失敗した場合、エラーメッセージ、スタックトレース、エラーが発生した正確な時点をキャプチャする`+exception+` または`+error+` イベントがあります。

=== 指標

メトリクスは、実行時に取得される測定値であり、メトリクス・イベントをもたらす。メトリクスは、アプリケーションのパフォーマンスと可用性の重要な指標であり、停止やパフォーマンスの問題を警告するためによく使用されます。メトリックスがある：

* 名前
* タイムスタンプ
* 一種
* 属性
* リソース属性（<<_resources,リソース>>参照）

属性はメトリックのメタデータを提供する。

== リソース

リソースは遠隔測定データを生成するエンティティである。リソース属性は、リソースのメタデータを提供します。たとえば、コンテナ、ポッド、Kubernetesクラスタのネームスペースで実行されるプロセスは、これらすべてのエンティティに対するリソース属性を持つことができる。

リソース属性はSDKによって自動的に割り当てられることが多い。ただし、`+service.name+` と`+service.namespace+` 属性は常に明示的に設定することを推奨します。最初のものはサービスの論理名です。設定されていない場合、SDKは`+unknown_service+` 値を設定し、後でSUSE Observabilityでデータを使用するのが非常に困難になります。名前空間は、サービスを整理する便利な方法で、特に同じサービスを複数の場所で実行する場合に便利です。

== 意味上の規則

Open Telemetry は、オペレーションとデータに共通の名前を定義しており、これをセマンティック規約と呼んでいる。意味規約は、言語、ライブラリ、コードベース間でデータの処理を標準化することを可能にする命名スキームに従う。すべてのシグナルとリソース属性にはセマンティックな規約がある。これらは、https://opentelemetry.io/docs/specs/semconv/attributes-registry/[Open Telemetryのウェブサイト]上で、多くの異なるプラットフォームやオペレーション用に定義されている。SDKはこれらの属性を割り当てるためにセマンティック規約を利用し、SUSE Observabilityも規約を尊重し、例えばKubernetesリソースを認識するために規約に依存します。
