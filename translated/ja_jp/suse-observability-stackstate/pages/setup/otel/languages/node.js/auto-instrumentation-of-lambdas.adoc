= NodeJSラムダを自動計測する
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: SUSE Observability

== はじめに

このドキュメントでは、OpenTelemetry を使用した NodeJS Lambda 関数の自動計測について説明します。自動計測は、パフォーマンスメトリクスとトレース情報を自動的に取得することで、Lambda関数に観測可能性を追加するプロセスを簡素化します。

== 前提条件

作業を始める前に、以下のものが揃っていることを確認してください：

* *AWS Lambda関数：*計測したい機能。
* *OpenTelemetry SDK:*ラムダ関数にインストールします。
* *OpenTelemetryコレクター：*配備され、設定された。
* *SUSE Observability：*遠隔測定データを送信するSUSE Observabilityのアカウント。
* *メモリ：*インスツルメンテーションを含むラムダを実行するのに十分なメモリ。

== 環境から供給される値

OpenTelemetry が正しく機能するためには、様々な設定値に依存します。これらの値は、データの収集、エクスポート、バックエンドシステムとの通信などの側面を制御する。OpenTelemetryのデプロイメントをフレキシブルにし、異なる環境に適応できるようにするために、環境変数を通してこれらの設定を提供することができます。このアプローチにはいくつかの利点がある：

* *ダイナミック・コンフィギュレーション：*コードを変更することなく、簡単に設定を調整できます。
* *環境固有の設定：*OpenTelemetryを開発、テスト、本番用に別々に設定する。
* *秘密の管理*APIキーなどの機密情報を安全に保管。

このドキュメントで説明する OpenTelemetry のセットアップでは、以下の環境変数を定義する必要があります：

* *`+VERBOSITY+`:*OpenTelemetry ログの詳細レベルを制御します。
* *`+OTLP_API_KEY+`:*SUSE Observabilityにデータを送信するLambda関数を認証します。
* *`+OTLP_ENDPOINT+`:*SUSE Observabilityインスタンスのアドレスを指定します。
* *`+OPENTELEMETRY_COLLECTOR_CONFIG_FILE+`:*OpenTelemetry Collector の設定ファイルを指す。
* *`+AWS_LAMBDA_EXEC_WRAPPER+`:*OpenTelemetry ハンドラを使用するように Lambda 実行環境を設定します。
* *`+OTLP_INSTR_LAYER_ARN+`:*自動計測に必要なコンポーネントを追加する OpenTelemetry 計測レイヤーの ARN (Amazon Resource Name) を提供します。
* *`+OTLP_COLLECTOR_LAYER_ARN+`:*テレメトリ・データの受信、処理、エクスポートを行う OpenTelemetry コレクター・レイヤーの ARN を提供する。

*重要な考慮事項*

* *GRPCエンドポイント：*`+OTLP_ENDPOINT+` には、SUSE Observability インスタンスの gRPC エンドポイントを指定します。`+http+` または`+https+` のプレフィックスは使用しないでください。安全な通信には443番ポートを使用してください。
* *地域固有のレイヤー：*ラムダ層はリージョンに縛られる。`+OTLP_INSTR_LAYER_ARN+` と`+OTLP_COLLECTOR_LAYER_ARN+` に使用する ARN が、Lambda 関数がデプロイされている AWS リージョンと一致していることを確認してください。
* *建築のマッチング：*OpenTelemetry Collector レイヤーはアーキテクチャ固有である。Lambda関数のアーキテクチャに適したARNを選択する（例：`+amd64+` または`+arm64+` ）。

*完全な例：自分の値を入力する必要があることに注意。*

[,yaml]
----
VERBOSITY: "normal"
OTLP_API_KEY: "<your api key for sending data to SUSE Observability here>"
OTLP_ENDPOINT: "<your-dns-name-for-suse-observability-here>:443"
OPENTELEMETRY_COLLECTOR_CONFIG_FILE: "/var/task/collector.yaml"
AWS_LAMBDA_EXEC_WRAPPER: "/opt/otel-handler"
OTLP_INSTR_LAYER_ARN: "arn:aws:lambda:<aws-region>:184161586896:layer:opentelemetry-nodejs-0_11_0:1"
OTLP_COLLECTOR_LAYER_ARN: "arn:aws:lambda:<aws-region>:184161586896:layer:opentelemetry-collector-<amd64|arm64>-0_12_0:1"
----

== collector.yamlファイル

OTEL収集構成は、収集したデータをどのように分配するかを設定する。 これは、ラムダ・ファイルがあるsrcディレクトリにあるcollector.yamlファイルで行う。 以下はcollector.yamlファイルの例である。

[,yaml]
----
# collector.yaml in the root directory
# Set an environemnt variable 'OPENTELEMETRY_COLLECTOR_CONFIG_FILE' to
# '/var/task/collector.yaml'

receivers:
  otlp:
    protocols:
      grpc:
      http:

exporters:
  debug:
    verbosity: "${env:VERBOSITY}"
  otlp/stackstate:
    headers:
      Authorization: "SUSEObservability ${env:OTLP_API_KEY}"
    endpoint: "${env:OTLP_ENDPOINT}"

service:
  pipelines:
    traces:
      receivers: [otlp]
      exporters: [debug, otlp/stackstate]
      processors: []
    metrics:
      receivers: [otlp]
      exporters: [debug, otlp/stackstate]
      processors: []
----

このコレクタは、SUSE Observabilityにデータを送信する前に、テールサンプリング、メトリック集約などに使用される次のコレクタにデータを送信するために使用されることに注意してください。この 2 番目のコレクターも、顧客の環境で実行する必要がある。

希望する機能に応じて、またはこの方法でインスツルメンテーションされたラムダによって生成されるデータ量などの要因に基づいて、SUSE Observabilityへの影響を軽減するために、バッチ処理、テールサンプリング、その他の前処理技術用にコレクタを設定できます。

SUSE Observabilityのセキュリティプロキシとして機能するバッチコレクタの設定xref:/setup/otel/proxy-collector.adoc[方法については]、このページを参照してください。
テールサンプリングも行なうコレクタのセットアップxref:/setup/otel/collector.adoc[方法については]、 このページを参照してください。
opentelemetry コレクタのプロセッサ構成については、https://github.com/open-telemetry/opentelemetry-collector/blob/main/processor/README.md[公式ドキュメントを]参照してください。

== パッケージ.json

`+package.json+` に`+"@opentelemetry/auto-instrumentations-node": "^0.55.2",++` を追加し、`+npm install+` を実行して、自動計測クライアント・ライブラリを NodeJS Lambda に追加してください。

== トラブルシューティング

=== タイムアウト

OTELラムダレイヤーを追加した結果、ラムダがタイムアウトした場合（ログをチェックすると、コレクターがビジー状態のままシャットダウンするように要求されたことがわかるかもしれません：）

[,json]
----
{
    "level": "info",
    "ts": 1736867469.2312617,
    "caller": "internal/retry_sender.go:126",
    "msg": "Exporting failed. Will retry the request after interval.",
    "kind": "exporter",
    "data_type": "traces",
    "name": "otlp/stackstate",
    "error": "rpc error: code = Canceled desc = context canceled",
    "interval": "5.125929689s"
}
----

シャットダウンの指示を受けた直後だった：

[,json]
----
{
    "level": "info",
    "ts": 1736867468.4311068,
    "logger": "lifecycle.manager",
    "msg": "Received SHUTDOWN event"
}
----

上記は、ラムダの割り当てられたリソースが、ラムダの実行とOTELインストルメンテーションによって追加された負荷を可能にするには十分でないことを示している。 この問題を解決するために、メモリ割り当てとラムダのタイムアウト設定を必要に応じて調整し、ラムダが作業を完了できるようにすると同時に、テレメトリ収集も成功するようにします。

失敗しているラムダのMemorySizeとTimeOutプロパティを変更してみてください：

[,yaml]
----
MemorySize: 256
Timeout: 25
----

デフォルトのメモリ割り当ては128MBであることに注意

メモリの増分は128MBであることに注意

注 タイムアウトは秒を表す整数値である。

=== 認証とソースIPフィルタリング

コレクタのデータをクラスタ、または前処理やプロキシコレクタに送信するときに`+error 403 Unauthorized+` が発生する場合は、VPC NAT ゲートウェイのソース IP アドレスがコレクタのイングレスでホワイトリストに登録されているものと一致していることを再確認してください。また、選択した認証メカニズムがソースと宛先に一致していること、および資格情報 (シークレットなど) が正しく設定されていることも再確認してください。

opentelemetry コレクターの認証の設定については、https://github.com/open-telemetry/opentelemetry-collector/blob/main/config/configauth/README.md[公式ドキュメントを]参照。
