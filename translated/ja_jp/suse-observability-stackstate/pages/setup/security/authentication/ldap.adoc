== 概要

SUSE Observabilityでは、LDAPサーバ(ADを含む)を使用して認証したり、ロール/グループを取得したりできます。ただし、SUSE Observabilityにアクセス可能なLDAPサーバーが必要です。

LDAPメイン・ディレクトリとすべてのサブ・ディレクトリにユーザー・ファイルがあるかどうかがチェックされる。SUSE Observability設定のバインド資格情報は、LDAPサーバでSUSE Observabilityを認証するために使用されます。認証後、SUSE ObservabilityはSUSE ObservabilityにログインするユーザのトップLDAPディレクトリ名を渡します。

== LDAP用のSUSE Observabilityの設定

=== Kubernetes

KubernetesでLDAP認証サーバーを使用して認証するようにSUSE Observabilityを設定するには、LDAPの詳細とユーザーロールマッピングをファイル`+authentication.yaml+` に追加する必要があります。例えば、こうだ：

[tabs]
====
authentication.yaml::
+
--

[,yaml]
----
stackstate:
  authentication:
    ldap:
      host: sts-ldap
      port: 10389 # For most LDAP servers 389 for plain, 636 for ssl connections
      #ssl:
      #  sslType: ssl
      #  trustStore: <see below>
      #  trustCertificates <see below>
      bind:
        dn: "cn=admin,ou=employees,dc=acme,dc=com"
        password: "password"
      userQuery:
        parameters:
          - ou: employees
          - dc: acme
          - dc: com
        usernameKey: cn
        emailKey: mail
      groupQuery:
        parameters:
          - ou: groups
          - dc: acme
          - dc: com
        rolesKey: cn
        groupMemberKey: member
        # to return all nested groups, use:
        # groupMemberKey: "member:1.2.840.113556.1.4.1941:"

    # map the groups from LDAP to the
    # standard subjects in SUSE Observability (guest, powerUser and admin)
    roles:
      guest: ["ldap-guest-role-for-stackstate"]
      powerUser: ["ldap-power-user-role-for-stackstate"]
      admin: ["ldap-admin-role-for-stackstate"]
----

--
====

以下の手順に従って、LDAPを使用して認証するようにSUSE Observabilityを設定します：

. `+authentication.yaml+` - LDAPの詳細を追加する（上記の例を参照）：
 ** *host*- LDAPサーバーのホスト名。
 ** *port*- LDAPサーバーがリッスンしているポート。
 ** *sslType* - Optional.LDAP セキュア接続のタイプ`+ssl+` または`+startTls+` 。プレーンなLDAP接続を使用する場合は省略する。
 ** *trustCertificates*- オプション、SSL 用証明書ファイル。形式はPEM、DER、PKCS7に対応しています。
 ** *trustStore*- オプション。SSL用のJavaトラスト・ストア・ファイル。`+trustCertificates+` 、`+trustStore+` の両方が指定されている場合は、`+trustCertificatesPath+` が優先される。
 ** *bind*- オプション。LDAPサーバが匿名LDAP検索をサポートしていない場合に、SUSE ObservabilityをLDAPサーバに認証するために使用します。
 ** *userQueryパラメータとgroupQueryパラメータ*- LDAPのベースDNに対応するパラメータのセット。1つ目はSUSE Observabilityでユーザを認証するために使用され、2つ目はユーザが管理者、パワーユーザ、ゲストのいずれであるかを判断するために、そのユーザのグループを取得するために使用されます。
 ** *usernameKey*- ユーザー名を格納する属性名。値はログイン画面で指定されたユーザー名と照合されます。
 ** *emailKey*- SUSE Observabilityでメールアドレスとして使用される属性名。
 ** *rolesKey*- グループ名を格納する属性名。
 ** *groupMemberKey*- ユーザがグループのメンバーであるかどうかを示す属性名。構築されたLDAPフィルターは次のパターンに従っている：`+<groupMemberKey>=<user.dn>,ou=groups,dc=acme,dc=com+`.すべてのネストしたグループを返すには、`+groupMemberKey: "member:1.2.840.113556.1.4.1941:"+` を使う。
. `+authentication.yaml+` - LDAPから正しいSUSE Observabilityサブジェクトにユーザロールをマッピングします（上記の例を参照してください）：
 ** *ロール*- 詳細は、xref:/setup/security/rbac/rbac_permissions.adoc#_predefined_roles[デフォルトのSUSE Observabilityロールを]参照してください。その他のSUSE Observabilityロールも作成できます。xref:/setup/security/rbac/README.adoc[RBACドキュメントを]参照してください。
. ファイル`+authentication.yaml+` は、SUSE Observabilityのインストール手順にある`+values.yaml+` と一緒に保存してください。
. Helmアップグレードを実行して変更を適用します。カスタム証明*書*付きSSLを使用している場合、LDAPに接続する際に使用するバイナリ証明書ファイルは、コマンドラインから設定する必要があります：

[,text]
----
helm upgrade \
  --install \
  --namespace suse-observability \
  --values values.yaml \
  --values authentication.yaml \
suse-observability \
suse-observability/suse-observability
----

*trustCertificates*

[,bash]
----
helm upgrade \
  --install \
  --namespace suse-observability \
  --values values.yaml \
  --values authentication.yaml \
  --set-file stackstate.authentication.ldap.ssl.trustCertificates=./ldap-certificate.pem \
suse-observability \
suse-observability/suse-observability
----

*trustStore*

[,bash]
----
helm upgrade \
  --install \
  --namespace suse-observability \
  --values values.yaml \
  --values authentication.yaml \
  --set-file stackstate.authentication.ldap.ssl.trustStore=./ldap-cacerts \
suse-observability \
suse-observability/suse-observability
----

[NOTE]
====
*注：*

* helm upgrade コマンドを最初に実行するとポッドが再起動し、可用性が短時間中断する可能性があります。
* `+helm upgrade+` 実行するたびに`+authentication.yaml+` を含める。
* 認証設定はKubernetesシークレットとして保存される。
====


=== 外部秘密の使用

ldapパスワードが外部シークレットに由来する場合は、xref:/setup/security/external-secrets.adoc#_getting_authentication_data_from_an_external_secret[以下の手順に]従うが、以下のデータを記入すること：

[,yaml]
----
kind: Secret
metadata:
   name: "<custom-secret-name>"
type: Opaque
data:
  ldap_password: <base64 of ldap password>
----
