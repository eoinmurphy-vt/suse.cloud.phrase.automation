= データ保持
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: SUSE Observability セルフホスト

== 概要

SUSE Observabilityは、ストレージスペースを節約し、パフォーマンスを向上させるために、データ保持の制限を課しています。データ保存期間を設定して、保存データ量とSUSE Observabilityのパフォーマンスおよびデータの可用性のバランスをとることができます。

== トポロジー・グラフ・データの保持

デフォルトでは、トポロジーグラフデータは30日間保持される。これは、トポロジー・グラフの最新の状態が常に保持され、30日以上前の履歴だけが削除されるという仕組みである。
場合によっては、履歴データを30日以上保存したり、ディスク容量を節約するために30日未満にしたりすることが有効なこともある。トポロジー保持はヘルムチャートで設定できる：

[,yaml]
----
stackstate:
  topology:
    # Retention set to 1 week
    retentionHours: 144
----

データ保持期間を長くすることで、保存されるデータ量も増え、より多くのストレージ容量が必要になることに注意。これはビューのパフォーマンスにも影響するかもしれない。

保存期間を短くすると、ディスク領域が解放されるまで時間がかかることがあります（少なくとも15分）。

=== トポロジーのディスク容量の問題のトラブルシューティング
ディスク容量の問題が発生した場合、namenodeにログ行 -`+Not enough replicas was chosen. Reason: {NOT_ENOUGH_STORAGE_SPACE=1+` が表示されます。このシナリオに対処するには、以下の手順に従ってください：

* リテンションを下げ、インスタンスがすぐにディスクスペースを回復できるように準備し、ヘルムアップグレードをトリガーする：
[,yaml]
----
stackstate:
  topology:
    # Retention set to 1 week in case you are running with the default 1 month
    retentionHours: 144
hbase:
  console:
    enabled: true
    replicaCount: 1
  hdfs:
    datanode:
      extraEnv:
        open:
          HDFS_CONF_dfs_datanode_du_reserved_pct: "0"
----

[NOTE]
====
次のステップに進む前に、すべてのhbaseとhdfsポッドが安定するまで待つ。
====

* 履歴データの圧縮をトリガーする：
[,bash]
----
kubectl exec -t --namespace suse-observability  $(kubectl get pods --namespace suse-observability --no-headers | grep "console" | awk '{print $1}' | head -n 1) -- /bin/bash -c "stackgraph-console run println\(retention.removeExpiredDataImmediately\(\)\)"
----

* を使って経過を追う：
----
kubectl exec -t --namespace suse-observability  $(kubectl get pods --namespace suse-observability --no-headers | grep "console" | awk '{print $1}' | head -n 1) -- /bin/bash -c "stackgraph-console run println\(retention.removeExpiredDataImmediatelyStatus\(\)\)"
----

* ディスク容量が足りない場合は、<support-portal-link> までご連絡ください。

* 設定を元に戻す。ステータスが進行中でなくなったら -`+Status(inProgress = false, lastFailure = null)+` 、ヘルムアップグレードをトリガーし、新しい保持をあなたの価値の一部として保持する。
[,yaml]
----
stackstate:
  topology:
    # Retention set to 1 week in case you are running with the default 1 month
    retentionHours: 144
----

== イベントとログの保持

=== SUSE Observabilityデータストア

SUSE Observabilityで提供されるイベント/ログストアを使用している場合、データはデフォルトで30日間保持されます。ほとんどの場合、デフォルトの設定で、この時間、すべてのインデックスを保存するのに十分である。

==== Elasticsearchのディスク容量を設定する

状況によっては、Elasticsearchで利用可能なディスク容量や、ログやイベントへの割り当て方法を調整する必要があるかもしれません。例えば、特定のデータタイプについて多くのデータが到着することが予想される場合などです。

以下は、デフォルト値を含む、Elasticsearchのディスク容量とリテンション設定のスニペットです。

[,yaml]
----
elasticsearch:
  volumeClaimTemplate:
    resources:
      requests:
        storage: 250Gi
stackstate:
  components:
    receiver:
      esDiskSpaceShare: 70
      # Number of days to keep the logs data on Es
      retention: 7
    e2es:
      esDiskSpaceShare: 30
      # Number of days to keep the events data on Es
      retention: 30
----

Elasticsearch で利用可能なディスク容量は、`+elasticsearch.volumeClaimTemplate.resources.requests.storage+` キーで設定します。最初のインストール後にこの値を変更するには、いくつかのxref:/setup/data-management/data_retention.adoc#_resizing_storage[特別な手順が必要です]。

NOTE: これは、ElasticSearchの各インスタンスのディスク容量です。HAでない場合、これは利用可能なディスク容量の合計であるが、HAの場合は3つのインスタンスがあり、レプリケーション係数は1である。最終的に、利用可能な Elasticsearch ストレージの合計は`+(250Gi * 3) / 2 = 375Gi+` となります。
`+esDiskSpaceShare+` と`+retention+` に基づいて、Elasticsearch のディスクスペースの一部が各データタイプ用に予約されています。

== メトリクスの保持

SUSE ObservabilityはVictoriaMetricsを使用してメトリクスを保存します。デフォルトの保存期間は30日間です。ヘルムチャートは、このように1つまたは2つのビクトリア・メトリクス・インスタンスにディスク領域を割り当て、保持期間を設定する：

----
victoria-metrics-0:
  server:
    persistentVolume:
      size: 250Gi
    retentionPeriod: 1 # month
# For HA setups:
victoria-metrics-1:
  server:
    persistentVolume:
      size: 250Gi
    retentionPeriod: 1 # month
----

初回インストール後にボリュームサイズを変更するには、いくつかのxref:/setup/data-management/data_retention.adoc#_resizing_storage[追加手順が必要です]。

保持期間を変更するには、カスタムvalues.yamlで両方の`+retentionPeriod+` キーを同じ値で上書きし、xref:/setup/data-management/data_retention.adoc#_update_stackstate[SUSE Observabilityを更新]します：

* h（時間）、d（日）、w（週）、y（年）。サフィックスが設定されていない場合、期間は月単位となる。
* 最低保存期間は24時間または1日です。

== SUSE Observabilityのアップデート

values.yamlに変更を加えた後、その変更をランタイムに適用するためにSUSE Observabilityを更新する必要があります。このため、サービスが再起動するまでの短いダウンタイムが発生する可能性があります。SUSE Observabilityをアップデートするには、SUSE Observabilityのインストール時に使用したものと同じコマンドを使用し、変更内容を含む同じ設定ファイルを含めるようにしてください：

== ストレージのサイズ変更

ほとんどのクラスタでは、永続ボリュームが作成された後でも、アプリケーションの動作をまったく中断することなく、永続ボリュームのサイズを変更することが可能です。しかし、これはSUSE Observability Helmチャートのvalues.yamlで設定されたストレージサイズを変更するだけではできません。その代わり、いくつかのステップが必要だ：

. 使用するストレージ・クラスのサイズを変更できることを確認する。
. ボリュームのサイズを変更する
. values.yamlを更新し、変更を適用する。

以下の例では、VictoriaMetrics ストレージを例にしている。SUSE Observabilityは`+suse-observability+` 名前空間にインストールされます。ボリュームは500Giにリサイズされる。

=== ストレージクラスがリサイズをサポートしていることを確認する。

以下の`+kubectl+` コマンドを使用して、使用されているストレージクラスを取得し、`+allowVolumeExpansion+` が true に設定されていることを確認する。

[,bash]
----
# Get the PVC's for SUSE Observability
kubectl get pvc --namespace suse-observability

# There is a storage class column in the output, copy it and use it to describe the storage class
kubectl describe storageclass <storage-class-name>
----

出力にこの行が含まれていることを確認する：

----
AllowVolumeExpansion:  True
----

この行がない場合、または`+False+` に設定されている場合は、リサイズがサポートされ、有効にできるかどうかをKubernetes管理者に相談してください。

=== ボリュームのサイズを変更する

SUSE Observability Helmチャートは、永続ボリュームクレーム(PVC)を作成するテンプレートを持つステートフルセットを作成します。このテンプレートはPVCの作成に一度だけ使用され、それ以降は適用されず、変更もできない。だから、PVCのサイズを大きくするには、PVC自体を編集する必要がある。

PVCサイズを変更するには、以下のコマンドを使用する。

[,bash]
----
# Get the PVC's for SUSE Observability, allows us to check the current size and copy the name of the PVC to modify it with the next command
kubectl get pvc --namespace suse-observability

# Patch the PVC's specified size, change it to 500Gi
kubectl patch pvc server-volume-stackstate-victoria-metrics-0-0 -p '{"spec":{"resources": { "requests": { "storage": "500Gi" }}}}'

# Get the PVC's again to verify if it was resized, depending on the provider this can take a while
kubectl get pvc --namespace suse-observability
----

=== values.yamlを更新し、変更を適用する

パーシステント・ボリューム・クレーム（PVC）に加えられた変更は、PVCの有効期間中は残りますが、クリーン・インストールを行うたびに失われます。しかし、より重要なのは、PVCのサイズを変更した後、クラスタの状態とvalues.yamlの望ましい状態の定義との間に矛盾が生じたことだ。したがって、values.yamlも更新することをお勧めする。この変更が許可されていないことを回避するには、まずステートフル・セットを削除して（ただしポッドは実行したまま）、新しい設定で再作成する。

[NOTE]
====
このステップではPVC自体のサイズは変更されないので、このステップを実行するだけで、実行環境はまったく変更されない。
====


まずvalues.yamlを編集して、リサイズしたPVCのボリュームサイズを更新します。xref:/setup/data-management/data_retention.adoc#_retention_of_metrics[メトリクス]またはxref:/setup/data-management/data_retention.adoc#_retention_of_events_traces_and_logs[イベントとログの]セクションを参照してください。

ストレージが変更されたアプリケーションのステートフルセットを削除する：

[,bash]
----
# List all stateful sets, check that all are ready, if not please troubleshoot that first
kubectl get statefulset --namespace suse-observability

# Delete the
kubectl delete statefulset --namespace suse-observability stackstate-victoria-metrics-0 --cascade=orphan
----

最後に、新しい設定でxref:/setup/data-management/data_retention.adoc#_update_[SUSE Observabilityをアップデート]します。
