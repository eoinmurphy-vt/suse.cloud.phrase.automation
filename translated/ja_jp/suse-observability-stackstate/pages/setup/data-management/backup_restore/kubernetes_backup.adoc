== 概要

{stackstate-product-name} にはバックアップとリストアのメカニズムが組み込まれており、バックアップをローカルクラスタ、AWS S3、またはAzure Blob Storageに保存するように設定できる。

=== バックアップ範囲

以下のデータを自動的にバックアップすることができます：

* StackGraphに保存された*コンフィギュレーションとトポロジーデータ*
* SUSE ObservabilityのVictoria Metricsインスタンスに格納されている*メトリクス*
* SUSE ObservabilityのElasticsearchインスタンスに保存された*遠隔測定データ*
* SUSE ObservabilityのClickHouseインスタンスに保存された*OpenTelemetryデータ*

以下のデータはバックアップさ*れません*：

* Kafkaに保存されたトランジットのトポロジーとテレメトリーの更新 - これらは一時的な価値しかなく、バックアップが復元されたときには役に立たない。
* ZooKeeperに保存されたマスターノードの交渉状態 - この実行時の状態は、復元されたときに不正確となるため、実行時に自動的に決定されます。
* Kubernetesの設定状態と生の永続ボリュームの状態 - この状態は、SUSE Observabilityを再インストールしてバックアップを復元することで再構築できます。
* Kubernetesのログ - これらはエフェメラルである。

=== ストレージ・オプション

バックアップは、https://min.io/[MinIO（min.io]）のインスタンスに送信される。自動バックアップが有効になっている場合、`+suse-observability+` Helmチャートによって自動的に開始される。MinIOは、AWS S3と同じAPIを持つオブジェクトストレージシステムである。データをローカルに保存したり、https://docs.min.io/docs/minio-gateway-for-s3.html[AWS S3 (min.io)] https://docs.min.io/docs/minio-gateway-for-azure.html[、]、その他のシステムへのゲートウェイとして機能する。

内蔵のMinIOインスタンスは、バックアップを3つの場所に保存するように設定できる：

* xref:/setup/data-management/backup_restore/kubernetes_backup.adoc[AWS S3]
* xref:/setup/data-management/backup_restore/kubernetes_backup.adoc[Azure Blob Storage]
* xref:/setup/data-management/backup_restore/kubernetes_backup.adoc[Kubernetesストレージ]

== バックアップを有効にする

=== AWS S3

[CAUTION]
====

*暗号化*

バックアップを保存するS3バケットを暗号化する場合は、Amazon S3管理キー（SSE-S3）を使用する必要があります。

⚠️ AWS Key Management Service (SSE-KMS)に保存されたAWS KMSキーによる暗号化はサポートされていません。その結果、Elasticsearchのログにこのようなエラーが表示されます：

`+Caused by: org.elasticsearch.common.io.stream.NotSerializableExceptionWrapper: sdk_client_exception: Unable to verify integrity of data upload. Client calculated content hash (contentMD5: ZX4D/ZDUzZWRhNDUyZTI1MTc= in base 64) didn't match hash (etag: c75faa31280154027542f6530c9e543e in hex) calculated by Amazon S3. You may need to delete the data stored in Amazon S3. (metadata.contentMD5: null, md5DigestStream: com.amazonaws.services.s3.internal.MD5DigestCalculatingInputStream@5481a656, bucketName: suse-observability-elasticsearch-backup, key: tests-UG34QIV9s32tTzQWdPsZL/master.dat)",+`
====


AWS S3バケットへのスケジュールバックアップを有効にするには、SUSE Observabilityのインストールに使用するHelm`+values.yaml+` ファイルに次のYAMLフラグメントを追加します：

[,yaml]
----
global:
  backup:
    enabled: true
backup:
  stackGraph:
    bucketName: AWS_STACKGRAPH_BUCKET
  elasticsearch:
    bucketName: AWS_ELASTICSEARCH_BUCKET
  configuration:
    bucketName: AWS_CONFIGURATION_BUCKET
victoria-metrics-0:
  backup:
    bucketName: AWS_VICTORIA_METRICS_BUCKET
victoria-metrics-1:
  backup:
    bucketName: AWS_VICTORIA_METRICS_BUCKET
clickhouse:
  backup:
    bucketName: AWS_CLICKHOUSE_BUCKET
minio:
  accessKey: YOUR_ACCESS_KEY
  secretKey: YOUR_SECRET_KEY
  s3gateway:
    enabled: true
    accessKey: AWS_ACCESS_KEY
    secretKey: AWS_SECRET_KEY
----

以下の値を置き換える：

* `+YOUR_ACCESS_KEY+` および`+YOUR_SECRET_KEY+` は、MinIOシステムを保護するために使用される認証情報である。これらの認証情報は、MinIOシステムで設定され、自動バックアップジョブとリストアジョブで使用されます。また、MinIOシステムに手動でアクセスする場合にも必要だ。
 ** YOUR_ACCESS_KEYには5文字から20文字の英数字を使用します。
 ** YOUR_SECRET_KEYには8～40文字の英数字を使用する。
* `+AWS_ACCESS_KEY+` と`+AWS_SECRET_KEY+` は、バックアップが保存されるS3バケットへのアクセス権を持つIAMユーザーのAWS認証情報です。そのユーザーに添付する必要がある許可ポリシーについては、以下を参照してください。
* `+AWS_STACKGRAPH_BUCKET+``+AWS_CONFIGURATION_BUCKET+`,`+AWS_ELASTICSEARCH_BUCKET+`,`+AWS_VICTORIA_METRICS_BUCKET+` および`+AWS_CLICKHOUSE_BUCKET+` は、バックアップを保存するS3バケットの名前である。注：AWSのS3バケットの名前はAWS全体でグローバルなので、デフォルトの名前のS3バケット(`+sts-elasticsearch-backup+`,`+sts-configuration-backup+`,`+sts-stackgraph-backup+`,`+sts-victoria-metrics-backup+`,`+sts-clickhouse-backup+` )はおそらく利用できません。

`+AWS_ACCESS_KEY+` と`+AWS_SECRET_KEY+` で識別される IAM ユーザーは、S3 バケットにアクセスするために、以下のパーミッションポリシーを設定する必要がある：

[,javascript]
----
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "AllowListMinioBackupBuckets",
            "Effect": "Allow",
            "Action": [
                "s3:ListBucket",
                "s3:GetBucketLocation"
            ],
            "Resource": [
                "arn:aws:s3:::AWS_STACKGRAPH_BUCKET",
                "arn:aws:s3:::AWS_ELASTICSEARCH_BUCKET",
                "arn:aws:s3:::AWS_VICTORIA_METRICS_BUCKET",
                "arn:aws:s3:::AWS_CLICKHOUSE_BUCKET",
                "arn:aws:s3:::AWS_CONFIGURATION_BUCKET"
            ]
        },
        {
            "Sid": "AllowWriteMinioBackupBuckets",
            "Effect": "Allow",
            "Action": [
                "s3:PutObject",
                "s3:GetObject",
                "s3:DeleteObject"
            ],
            "Resource": [
                "arn:aws:s3:::AWS_STACKGRAPH_BUCKET/*",
                "arn:aws:s3:::AWS_ELASTICSEARCH_BUCKET/*",
                "arn:aws:s3:::AWS_VICTORIA_METRICS_BUCKET/*",
                "arn:aws:s3:::AWS_CLICKHOUSE_BUCKET/*",
                "arn:aws:s3:::AWS_CONFIGURATION_BUCKET"
            ]
        }
    ]
}
----

=== Azure Blob Storage

Azure Blob Storageアカウントへのバックアップを有効にするには、SUSE Observabilityのインストールに使用するHelm`+values.yaml+` ファイルに次のYAMLフラグメントを追加します：

[,yaml]
----
global:
  backup:
    enabled: true
minio:
  accessKey: AZURE_STORAGE_ACCOUNT_NAME
  secretKey: AZURE_STORAGE_ACCOUNT_KEY
  azuregateway:
    enabled: true
----

以下の値を置き換える：

* `+AZURE_STORAGE_ACCOUNT_NAME+` -https://learn.microsoft.com/en-us/azure/storage/common/storage-account-create?tabs=azure-portal[Azure ストレージアカウント名 (learn.microsoft.com)]
* `+AZURE_STORAGE_ACCOUNT_KEY+` - バックアップを保存するhttps://learn.microsoft.com/en-us/azure/storage/common/storage-account-keys-manage?tabs=azure-portal[Azure ストレージ・アカウント・キー（learn.microsoft.com）]。

StackGraph、Elasticsearch、Victoria Metricsのバックアップは、それぞれ`+sts-stackgraph-backup+`,`+sts-configuration-backup+`,`+sts-elasticsearch-backup+`,`+sts-victoria-metrics-backup+`,`+sts-clickhouse-backup+` というBLOBコンテナに格納されている。これらの名前は、Helmの値`+backup.stackGraph.bucketName+` 、`+backup.elasticsearch.bucketName+` 、`+victoria-metrics-0.backup.bucketName+` 、`+victoria-metrics-1.backup.bucketName+` 、`+clickhouse.backup.bucketName+` をそれぞれ設定することで変更できる。

=== Kubernetesストレージ

[CAUTION]
====
MinIOがKubernetesストレージにデータを格納するように構成されている場合、Kubernetesクラスタにストレージを要求するためにPersistentVolumeClaim（PVC）が使用されます。割り当てられるストレージの種類はクラスタの構成によって異なる。

Amazon AWS上で稼働しているクラスタにはAWS S3を、Azure上で稼働しているクラスタにはAzure Blob Storageを使用することをお勧めします：

. クラウドプロバイダーで稼働しているKubernetesクラスタは通常、AWSのElastic Block StorageやAzure Block StorageなどのブロックストレージにPVCをマッピングする。ブロック・ストレージは、特に大容量のデータには高価だ。
. 永続ボリュームは、それを作成したクラスタが破棄されると破棄されます。つまり、クラスタを（誤って）削除すると、Persistent Volumesに保存されているすべてのバックアップも破壊されます。
. 永続ボリュームは別のクラスタからアクセスできません。つまり、別のクラスタで取得したバックアップからSUSE Observabilityをリストアすることはできません。
====


クラスタローカルストレージへのバックアップを有効にするには、SUSE Observabilityのインストールに使用するHelm`+values.yaml+` ファイルに次のYAMLフラグメントを追加して、MinIOを有効にします：

[,yaml]
----
global:
  backup:
    enabled: true
minio:
  accessKey: YOUR_ACCESS_KEY
  secretKey: YOUR_SECRET_KEY
  persistence:
    enabled: true
----

以下の値を置き換える：

* `+YOUR_ACCESS_KEY+` および`+YOUR_SECRET_KEY+` - MinIOシステムのセキュリティに使用される認証情報。自動バックアップジョブとリストアジョブはこれらを使用する。また、MinIOのストレージに手動でアクセスする際にも必要となる。`+YOUR_ACCESS_KEY+` には5～20文字の英数字、`+YOUR_SECRET_KEY+` には8～40文字の英数字を入れる。

== コンフィギュレーションとトポロジーデータ（StackGraph）

コンフィギュレーションとトポロジーデータ（StackGraph）のバックアップはフルバックアップで、拡張子`+.graph+` の単一ファイルに保存されます。各ファイルにはフルバックアップが含まれており、必要に応じて移動、コピー、削除が可能です。

=== バックアップ・スケジュール

デフォルトでは、StackGraphのバックアップは毎日サーバー時間の午前3時に作成されます。

バックアップスケジュールは、https://kubernetes.io/docs/concepts/workloads/controllers/cron-jobs/#_cron_schedule_syntax[] Kubernetesで指定されているHelm値`+backup.stackGraph.scheduled.schedule+` を使用して設定することができます。

=== バックアップ保持

デフォルトでは、StackGraphのバックアップは30日間保存されます。StackGraphのバックアップはフルバックアップであるため、多くのストレージを必要とする可能性がある。

バックアップ保持のデルタは、GNU日付`+--date+` 引数のフォーマットで指定されたHelm値`+backup.stackGraph.scheduled.backupRetentionTimeDelta+` を使用して設定できます。例えば、デフォルトは`+30 days ago+` 。その他の例については、https://www.gnu.org/software/coreutils/manual/html_node/Relative-items-in-date-strings.html[日付文字列の相対項目を]参照してください。

=== スケジュールバックアップを無効にする

スケジュールされたStackGraphバックアップを無効にするには、Helmの値`+backup.stackGraph.scheduled.schedule+` を使用して、バックアップスケジュールを過去の日付に設定します：

[,yaml]
----
backup:
  stackGraph:
    scheduled:
      schedule: '0 0 1 1 1970'  # January 1, 1970 (epoch start)
----

== メトリックス（ヴィクトリア・メトリックス）

[WARNING]
====
Victoria Metricsはバケットのバージョン管理を行わずに増分バックアップを行うため、新しいバックアップは*以前のバックアップを完全に置き換える*ことになります。

万が一、以下のような事態に遭遇した場合：

* 空のボリュームをVictoria Metricsインスタンスの`+/storage+` ディレクトリにマウントする。
* Victoria Metricsインスタンスから`+/storage+` ディレクトリまたはその中のファイルを削除する。

次に作成される（空の）バックアップには新しいバージョンのラベルが付けられ、ボリュームが空になる前のものは保存されます。
両方のバックアップはその瞬間から<<_list_victoria_metrics_backups,リストア可能なものとしてリストさ>>れます。
====


メトリックス（ビクトリア・メトリックス）は、増分バックアップのデータを保存するためにインスタント・スナップショットを使用する。Victoria Metricsの多くのインスタンスは同じバケットにバックアップを保存することができ、それぞれのバックアップは別々のディレクトリに保存されます。ディレクトリにあるファイルはすべて1つの全体として扱われ、全体としてのみ移動、コピー、削除が可能です。

[NOTE]
====
高可用性のデプロイメントでは、Victoria Metrics のインスタンスを 2 つデプロイする必要がある。バックアップはそれぞれ独立して有効化／設定される。

以下のコードスニペット/コマンドは、ビクトリア・メトリックの最初のインスタンス`+victoria-metrics-0+` 。2つ目のインスタンスをバックアップ／設定するには、次のようにします。 `+victoria-metrics-1+`
====


=== バックアップ・スケジュール

デフォルトでは、Victoria Metricsのバックアップは1時間ごとに作成される：

* `+victoria-metrics-0+` - 時25分
* `+victoria-metrics-1+` - 時35分

バックアップスケジュールは、Helm の値`+victoria-metrics-0.backup.scheduled.schedule+` https://github.com/aptible/supercronic/tree/master/cronexpr[ cronexpr フォーマットに従って]設定できます。

=== スケジュールバックアップを無効にする

スケジュールされた Victoria Metrics のバックアップを無効にするには、両方のインスタンスのバックアップスケジュールを過去の日付に設定します：

[,yaml]
----
victoria-metrics-0:
  backup:
    scheduled:
      schedule: '0 0 1 1 1970'  # January 1, 1970 (epoch start)
victoria-metrics-1:
  backup:
    scheduled:
      schedule: '0 0 1 1 1970'  # January 1, 1970 (epoch start)
----

== オープンテレメトリー（クリックハウス）

ClickHouseは増分バックアップと完全バックアップの両方を使用します。デフォルトでは、フルバックアップは毎日午前0時45分に実行され、増分バックアップは1時間ごとに実行される。バックアップごとに新しいディレクトリが作成され、古いバックアップ（ディレクトリ）は自動的に削除されます。バックアップ・ディレクトリにあるファイルはすべて1つのグループとして扱われ、グループとしてのみ移動、コピー、削除が可能です。バックアップの管理には、`+suse-observability-clickhouse-shard0-0+` Podで利用可能な`+clickhouse-backup+` ツールを使用することをお勧めします。

=== バックアップ・スケジュール

デフォルトでは、ClickHouseのバックアップが作成されます：

* フルバックアップ - 毎日午前0時45分
* 増分バックアップ - 毎時45分（午前3時から午前12時まで）

[CAUTION]
====
バックアップは並列実行で苦労する。1つ目のバックアップが完了する前に2つ目のバックアップが開始されると、1つ目のバックアップが中断されます。したがって、並列実行を避けることは極めて重要である。例えば、最初の増分バックアップは完全バックアップの3時間後に実行する。
====


バックアップスケジュールは、Helmの値`+clickhouse.backup.scheduled.full_schedule+` 、`+clickhouse.backup.scheduled.incremental_schedule+`  https://github.com/aptible/supercronic/tree/master/cronexpr[（]）に従って設定できます。

=== バックアップ保持

デフォルトでは、このツールは直近の308個のバックアップ（フルおよびインクリメンタル）を保持しており、これは～14日間に相当する。

バックアップの保持は、Helm の値`+clickhouse.backup.config.keep_remote+` を使用して設定できます。

=== スケジュールバックアップを無効にする

スケジュールされたClickHouseバックアップを無効にするには、フルバックアップと増分バックアップの両方のスケジュールを過去の日付に設定します：

[,yaml]
----
clickhouse:
  backup:
    scheduled:
      full_schedule: '0 0 1 1 1970'         # January 1, 1970 (epoch start)
      incremental_schedule: '0 0 1 1 1970'  # January 1, 1970 (epoch start)
----

== テレメトリーデータ（Elasticsearch）

遠隔測定データ（Elasticsearch）のスナップショットは増分であり、`+.dat+` という拡張子のファイルに保存されます。Elasticsearchのバックアップ保存場所にあるファイルは1つの全体として扱われ、全体としてのみ移動、コピー、削除が可能です。

xref:/setup/data-management/backup_restore/kubernetes_backup.adoc#_enable_backups[バックアップを有効にする]セクションで提供されている設定スニペットは、毎日のElasticsearchスナップショットを有効にします。

=== スケジュールスナップショットを無効にする

スケジュールされた Elasticsearch スナップショットを無効にするには、Helm の値`+backup.elasticsearch.scheduled.schedule+` を使用してスナップショットスケジュールを過去の日付に設定します：

[,yaml]
----
backup:
  elasticsearch:
    scheduled:
      schedule: '0 0 1 1 1970'  # January 1, 1970 (epoch start)
----

=== スナップショット・スケジュール

デフォルトでは、Elasticsearchスナップショットは毎日サーバ時間の午前3時に作成されます。

バックアップスケジュールは、https://www.elastic.co/guide/en/elasticsearch/reference/7.6/cron-expressions.html[Elasticsearch cron schedule syntax (elastic.co])で指定されているHelm値`+backup.elasticsearch.scheduled.schedule+` を使って設定することができます。

=== スナップショットの保持

デフォルトでは、Elasticsearchのスナップショットは30日間保存され、最小スナップショット数は5、最大スナップショット数は30です。

保持時間とスナップショットの保持数は、以下のHelmの値を使用して設定できます：

* `+backup.elasticsearch.scheduled.snapshotRetentionExpireAfter+`https://www.elastic.co/guide/en/elasticsearch/reference/7.6/common-options.html#_time_units[Elasticsearchの時間単位（elastic.co]）で指定します。
* `+backup.elasticsearch.scheduled.snapshotRetentionMinCount+`
* `+backup.elasticsearch.scheduled.snapshotRetentionMaxCount+`

[NOTE]
====
デフォルトでは、リテンションタスク自体https://www.elastic.co/guide/en/elasticsearch/reference/7.6/slm-settings.html#_slm_retention_schedule[、毎日午前1時30分（UTC）に実行されます（elastic.co]）。テスト目的などでスナップショットの有効期限を1日以内に設定する場合は、保持タスクのスケジュールを変更する必要があります。
====


=== スナップショット・インデックス

デフォルトでは、スナップショットは`+sts+` で始まる名前の Elasticsearch インデックスに対して作成されます。

スナップショットが作成されるインデックスは、https://www.w3schools.com/js/js_json_arrays.asp[] JSON）で指定されたHelm値`+backup.elasticsearch.scheduled.indices+` を使用して設定できます。

== バックアップとスナップショットの復元

[NOTE]
====
*スクリプトを使用する前に、以下を確認してください：*

* `+kubectl+` のバイナリがインストールされ、接続できるように設定されている：
 .. SUSE ObservabilityがインストールされているKubernetesクラスタ。
 .. SUSE Observabilityがインストールされているクラスタ内の名前空間。
* Helm の値`+global.backup.enabled+` は`+true+` に設定される。
====

=== Elasticsearchスナップショットのリストア

[WARNING]
====
*スナップショットがリストアされると、既存のインデックスは上書きされません。*

`+--delete-all-indices+` フラグを使用することで、リストアスクリプトによってすべてのインデックスを自動的に削除することもできますし、xref:/setup/data-management/backup_restore/kubernetes_backup.adoc#_delete_elasticsearch_indices[delete Elasticsearch indices] の説明に従って手動で削除することもできます。

====

[WARNING]
====
*Elasticsearch`+PersistentVolumes+` が再作成された場合*（例えば、誤って削除され、その後ポッドが再起動された場合）、以下の2つの方法のいずれかを使用して Elasticsearch のバックアップ設定を再作成する必要があります：

* 初期インストール(OR)で使用したのと同じ設定でSUSE Observability Helmチャートを再インストールします。
* バックアップ初期化CronJobを手動でトリガーする：
+
[,bash]
----
kubectl create job --from=cronjob.batch/suse-observability-backup-init "suse-observability-backup-init-$(date +%s)"
----
====
Elasticsearchスナップショットをリストアするには、スナップショット名を選択し、最初のパラメータとして渡します。オプションで指定できる：

* リストアするインデックスのカンマ区切りリスト。指定しない場合、Helmの値`+backup.elasticsearch.scheduled.indices+` に一致するすべてのインデックスがリストアされる。デフォルト値は`+"sts*"+`)。
* バージョン`+2.6.1+` で導入された "--delete-all-indices" フラグを使用すると、リストア前に既存のすべてのインデックスを自動的に削除することができる。

==== 基本リストア

[,bash]
----
./restore/restore-elasticsearch-snapshot.sh \
  sts-backup-20210223-0300-ppss8nx40ykppss8nx40yk
----

==== 特定のインデックスを復元する

[,bash]
----
./restore/restore-elasticsearch-snapshot.sh \
  sts-backup-20210223-0300-ppss8nx40ykppss8nx40yk \
  "<INDEX_TO_RESTORE>,<INDEX_TO_RESTORE>"
----

==== インデックスの自動削除によるリストア

[,bash]
----
./restore/restore-elasticsearch-snapshot.sh \
  sts-backup-20210223-0300-ppss8nx40ykppss8nx40yk \
