== 概要

このページでは、健全性同期Repeat States一貫性モデルで送信できる正確なJSONメッセージについて説明します。

== JSONプロパティ: "health"

ヘルスは、xref:/configure/health/send-health-data/send-health-data.adoc#_common_json_object[共通 JSON オブジェクトの] `+"health"+` プロパティを使用して SUSE Observability Receiver API に送信できます。

[tabs]
====
健康状態の例`+repeat_states+` JSON::
+
--

[,javascript]
----
   "apiKey":"<STACKSTATE_RECEIVER_API_KEY>",
   "collection_timestamp":1585818978,
   "internalHostname":"lnx-343242.srv.stackstate.com",
   "events":{},
   "metrics":[],
   "service_checks":[],
   "health":[
      {
        "consistency_model": "REPEAT_STATES",
        "expiry": {
          "repeat_interval_s": 50,
          "expiry_interval_s": 100
        },
        "stream": {
          "urn": "urn:health:sourceId:streamId"
          //"sub_stream_id": "subStreamId" Optional
        },
        "check_states": [
          {
            "checkStateId": "checkStateId1",
            "message": "Server Running out of disk space",
            "health": "Deviating",
            "topologyElementIdentifier": "server-1",
            "name": "Disk Usage"
          },
          {
            "checkStateId": "checkStateId2",
            "message": "Provisioning failed. [Learn more](https://www.any-link.com)",
            "health": "critical",
            "topologyElementIdentifier": "server-2",
            "name": "Health monitor"
          }
        ]
      }
   ],
   "topologies":[]
----

--
====

すべてのヘルス・リピート・ステーツ・データ・ペイロードには、以下の詳細がある：

* *expiry*- オプション。`+check_states+` を処理する前に、期限切れアップデートを処理する必要がある。これにより、SUSE Observabilityは、外部チェックが再度送信されない場合、システムに存在すべき期間を追跡できます。期限切れメタデータとして以下のフィールドを持つ：
 ** *repeat_interval_s*- 秒単位の時間．外部ソースがSUSE Observabilityにヘルスデータを送信する頻度。最大許容値は1800（30分）。
 ** *expiry_interval_s*- 秒単位の時間。外部チェックが再び観測されない場合、SUSE Observabilityによって外部チェックが削除されるまでの最終更新後の待機時間。
* *stream*- どのスナップショットと`+check_states+` が一緒になっているかという識別を提供するオブジェクト。以下のフィールドがある：
 ** *urn*- 以下の規約に一致するSUSE Observabilityxref:/configure/topology/identifiers.adoc[URNとして]エンコードされたデータソースおよびストリームID:`+urn:health:<sourceId>:<streamId>+` `+<sourceId>+` は外部データソースの名前、`+<streamId>+` はヘルスデータストリームの一意の識別子です。
 ** *sub_stream_id* - Optional.ストリームの健全性データのサブセットの識別子。ストリームデータが分散され、複数のエージェントによって報告される場合、これにより、各エージェントのスナップショットライフサイクルが可能になる。 `+sub_stream_id+`
* *check_states*- チェック状態のリスト。各チェック状態は以下のフィールドを持つことができる：
 ** *checkStateId*- 外部システムにおけるチェック状態の識別子。
 ** *message*- オプション。SUSE Observability UIに表示するメッセージ。データはマークダウンとして解釈され、外部チェック状態を生成した外部システムチェックへのリンクを持つことができる。
 ** *health*- 以下のSUSE Observability Health状態値のいずれか：`+Clear+`,`+Deviating+`,`+Critical+`.
 ** *topologyElementIdentifier*- チェック状態をSUSE Observabilityトポロジ要素にバインドするために使用されます。
 ** *name*- 外部チェック状態の名前。

== ヘルスをSUSE Observabilityに送信する

健康状態は、HTTP POSTを介して1つのJSONメッセージで送信することができます。以下の例では、2つのチェック状態を含むスナップショットが、1つの外部監視システムからSUSE Observabilityに送信されます。

[,bash]
----
curl -X POST \
 '<STACKSTATE_RECEIVER_API_ADDRESS>' \
 -H 'Content-Type: application/json' \
 -d '{
  "collection_timestamp": 1548857167,
  "internalHostname": "local.test",
  "health": [
    {
      "consistency_model": "REPEAT_STATES",
      "expiry": {
        "repeat_interval_s": 300,
        "expiry_interval_s": 600
      },
      "stream": {
        "urn": "urn:health:sourceId:streamId"
      },
      "check_states": [
        {
          "checkStateId": "checkStateId1",
          "message": "Server Running out of disk space",
          "health": "Deviating",
          "topologyElementIdentifier": "server-1",
          "name": "Disk Usage"
        },
        {
          "checkStateId": "checkStateId2",
          "message": "Provisioning failed. [Learn more](https://www.any-link.com)",
          "health": "critical",
          "topologyElementIdentifier": "server-2",
          "name": "Health monitor"
        }
      ]
    }
  ]
}'
----
