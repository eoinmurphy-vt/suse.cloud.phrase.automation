= フィルター
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: SUSE Observability

== 概要

ビュー UI の右上隅にある*フィルター*メニューを使用すると、ビューに表示されるコンポーネント（トポロジー）とイベントをフィルタリングできます。フィルタを適用すると、ビュー内のすべてのパースペクティブのコンテンツに影響します。

image::k8s/k8s-filters.png[フィルターを見る]

== フィルター・トポロジー

トポロジーフィルターを使用して、利用可能なパースペクティブのいずれかに表示するトポロジーコンポーネントのサブセットを選択することができます。イベントフィルターはすべてのビュータイプで同じですが、トポロジーフィルターはビューのタイプによって異なります。続きを読む

* xref:/use/views/k8s-filters.adoc#_topology_filters_on_kubernetes_views[Kubernetesビューのトポロジーフィルター]
* xref:/use/views/k8s-filters.adoc#_topology_filters_on_other_view_types[他のビュータイプのトポロジーフィルター]

=== Kubernetesビューのトポロジーフィルター

xref:/use/views/k8s-views.adoc[kubernetesビューでは]、トポロジーフィルタは、すべてのKubernetesビューにまたがって持続する基本的なフィルタの小さなセットに制限されています：`+clusters+` および/または`+namespaces+` 。Kubernetesビューの永続的なトポロジーフィルターは、UIの通常の`+filters+` メニューの外側に配置され、相互に依存しません（例えば、クラスタを選択しても、`+namespaces+` フィルターが自動的に更新され、そのクラスタのネームスペースのみが反映される*わけではありません*）。

image::k8s/k8s-filters-kube-topology.png[Kubernetesトポロジーフィルター]

=== 他のビュータイプのトポロジーフィルター

その他のビュー・タイプでは、基本または高度なトポロジー・フィルターを使用してトポロジーをブラウズできる。続きを読む

* xref:/use/views/k8s-filters.adoc#_basic_topology_filters[基本的なトポロジー・フィルター]
* xref:/use/views/k8s-filters.adoc#_advanced_topology_filters[高度なトポロジー・フィルター]
* xref:/use/views/k8s-filters.adoc#_topology_filtering_limits[トポロジー・フィルタリングの制限]

==== 基本的なトポロジー・フィルター

トポロジーをフィルタリングする主な方法は、利用可能な基本フィルタを使用することである。フィルタを設定すると、開いたパースペクティブが更新され、フィルタに一致するトポロジのサブセットの視覚化またはデータのみが表示されます。複数のフィルターを設定すれば、さらに絞り込むことができる。各フィルターに複数の値を設定して、検索範囲を広げることができます。

|===
| フィルター | 説明

| レイヤー、ドメイン、環境、コンポーネントタイプ
| コンポーネントがインポートまたは作成されたときに含まれるコンポーネントの詳細によってフィルタリングします。

| コンポーネント・ヘルス
| 関連する健全性チェックによって報告された、指定された健全性状態を持つコンポーネントのみを含める。

| コンポーネント・ラベル
| 特定のラベルの付いたコンポーネントのみを含める。

| コンポーネントを含む
| ここで指定されたコンポーネントは、他のフィルターから返された*コンポーネントに加えて*トポロジーに含まれる。
|===

基本的なフィルターを使用してトポロジーをフィルターするには、UIの右上隅にある*Filters*メニューをクリックし、*Topology*verticalタブの下にある*Switch to basicを*選択する。

image::k8s/k8s-filters-basic-topology.png[基本的なトポロジー・フィルター]

==== 高度なトポロジー・フィルター

内蔵のxref:/develop/reference/k8sTs-stql_reference.adoc[SUSE Observability Query Language（STQL]）を使用して、トポロジの特定の領域にズームインする高度なトポロジフィルタを構築できます。

STQLクエリーを使用してトポロジーをフィルターするには、UIの右上隅にある*［フィルター］*メニューをクリックし、［*トポロジー］*垂直タブで*［STQLに切り替える*］を選択します。

以下のSTQLクエリー例は、以下の条件に一致するコンポーネントを返します：

* *ドメイン*内 `+security check+`
* かつ*健康*状態が`+Clear+` または `+Deviating+`
* ORは*コンポーネント*名 `+ai_engine+`

[,yaml]
----
(domain IN ("security check") AND healthstate IN ("CLEAR", "DEVIATING")) OR name IN ("ai_engine")
----

image::k8s/k8s-filters-advanced-topology.png[高度なトポロジー・フィルター]

==== ベーシック・フィルターとアドバンス・フィルターの互換性

*Filters*メニューの*Topology*verticalタブで*Switch to basic*または*Switch to STQLを*選択すると、基本フィルタリングと高度フィルタリングを切り替えることができます。

基本的なフィルタリングから高度なフィルタリングへの切り替えはいつでも可能だ。選択された基本フィルターは、直接STQLクエリーに変換されます。単純なクエリの場合、高度なフィルタリングから基本的なフィルタリングに切り替えることも可能ですが、一部の高度なクエリは基本的なフィルタリングと互換性がありません。

➡️xref:/develop/reference/k8sTs-stql_reference.adoc#_compatibility_basic_and_advanced_filters[基本トポロジー・フィルターとアドバンスド・トポロジー・フィルターの互換性についての詳細はこちら]

==== その他のフィルター

以下の高度なフィルターは、基本的なフィルターと互換性がありますが、基本的なフィルターとして設定または調整することはできません。

* *withNeighborsOf*- 高度なフィルターにxref:/develop/reference/k8sTs-stql_reference.adoc#_withneighborsof[withNeighborsOf]関数が含まれている場合、隣接するコンポーネントの数が*他のフィルター*ボックスに表示されます。基本的なフィルタリングと互換性を持たせるために、`+withNeighborsOf+` 関数は、`+OR+` 演算子を使用して他のフィルタと結合する必要があります。
* *identifier*- 高度なフィルターがxref:/develop/reference/k8sTs-stql_reference.adoc#_filters[識別子で]コンポーネントを選択した場合、照会されたコンポーネントの識別子の数が*Other filters*ボックスに報告されます。基本的なフィルタリングと互換性を持たせるためには、`+identifier+` フィルタを指定し、演算子`+OR identifier IN (...)+` を使って他のフィルタと結合する必要があります。

基本トポロジーフィルターの*「その他のフィルター*」ボックスには、これらすべての高度なフィルターと、影響を受けるコンポーネントの数が表示されます。

[NOTE]
====
*その他のフィルター*」ボックスには、設定され、基本的なフィルタリングと互換性のある高度なフィルターの詳細のみが表示されます。
====


==== トポロジー・フィルタリングの制限

パフォーマンスを最適化するために、トポロジーを視覚化するためにロードできる要素の量に制限が設けられている。フィルタリング制限のデフォルト値は10000要素。xref:/use/views/k8s-filters.adoc#_basic_topology_filters[基本フィルター]またはxref:/use/views/k8s-filters.adoc#_advanced_topology_filters[詳細フィルターのクエリーが]フィルター制限を超えると、画面にメッセージが表示され、トポロジーの可視化は表示されません。

フィルタリングの制限は、最終的に表示される要素の量ではなく、*読み込む*必要のある要素の総量に適用されることに注意してください。

以下の例では、まず、トポロジー内のすべてのコンポーネントのすべての近傍をLOADし、次に、`+applications+` レイヤーに属するものだけをDISPLAYする。これは、トポロジー内のすべてのコンポーネントをロードする必要があるため、フィルタリングの制限エラーで失敗する可能性が高い。

[,text]
----
withNeighborsOf(direction = "both", components = (name = "*"), levels = "15")
   AND layer = "applications"
----

このトポロジーの視覚化を成功させるには、ロードされるコンポーネントの数を設定されたフィルタリングの上限以下に保つようにクエリを書き直すか、フィルタリングの上限を増やす必要がある。`+applications+` レイヤーにあるコンポーネントだけをフィッティングすることで、すべてのコンポーネントを最初に LOAD することなく、上のクエリーと同じコンポーネントを表示することができます。そのため、このクエリでは、フィルタリングの制限エラーが発生する可能性は低くなる。

[,yaml]
----
layer = "applications"
----

== フィルターイベント

SUSE Observability UIの左側にある[*View Filters]*パネルを使用して、xref:/use/views/k8s-events-perspective.adoc[イベントパースペクティブに]表示されるイベントをフィルタリングできます。また、右パネルの*「概要」*タブと「詳細」タブ（*「コンポーネントの詳細*」と「*直接関係の詳細」*）の*「イベント」*リストにも含まれる。

以下のイベントフィルターが利用可能です：

|===
| フィルター | 説明

| *カテゴリー*
| 1つまたは複数のxref:/use/views/k8s-events-perspective.adoc#_event_category[カテゴリの]イベントのみを表示します。

| *タイプ*
| *タイプ・*フィルター・ボックスをクリックすると、現在フィルタリングされているコンポーネントについて、現在のタイム・ウィンドウで生成されたすべてのイベント・タイプのリストが表示されます。1つまたは複数のイベントタイプを選択して、表示されるイベントを絞り込むことができます。

| *ソース*
| イベントは、SUSE Observabilityで生成することも、統合によってKubernetesやServiceNowなどの外部ソースシステムから取得することもできます。*ソース*フィルタ ボックスをクリックすると、現在のタイム ウィンドウで現在フィルタリングされているコンポー ネントに対して生成されたイベントのすべてのソース システムのリストが表示されます。1つまたは複数のソース・システムを選択すると、それらのイベントのみが表示されます。

| *タグ*
| イベントが外部システムから取得されると、関連するイベント・プロパティがタグとして追加される。例えば`+status:open+` や`+status:production+` 。これは、特定の問題や環境に関連する事象を特定するのに役立つ。
|===
