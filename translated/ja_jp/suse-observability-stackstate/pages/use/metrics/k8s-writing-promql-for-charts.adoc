== ガイドライン

SUSE Observabilityがデータをチャートに表示する場合、ほとんどの場合、チャートの空きスペースに収まるように保存データの解像度を変更する必要があります。可能な限り代表的なチャートを得るためには、以下のガイドラインに従ってください：

* 未加工の指標を問い合わせるのではなく、常に（`+*_over_time+` または`+rate+` 関数を使って）経時的に集計すること。
* `+${__interval}+` パラメータを経時的な集計の範囲として使用する。チャートの解像度に応じて自動的に調整される。
* `+rate+` 集計の範囲として、`+${__rate_interval}+` パラメータを使用する。これは、チャートの解像度に連動して自動的に調整されるが、`+rate+` の特定の動作を考慮に入れている。

集計を適用するということは、多くの場合、メトリクスの特定のパターンを他のものよりも強調するというトレードオフが行われることを意味する。例えば、大きな時間窓の場合、`+max_over_time+` はすべてのピークを表示するが、すべての谷は表示しない。一方、`+min_over_time+` は正反対で、`+avg_over_time+` は山と谷の両方を滑らかにする。この動作を示すために、ポッドのCPU使用率を使用したメトリックバインディングの例を示します。自分で試すには、これをYAMLファイルにコピーし、xref:/use/metrics/k8s-add-charts.adoc#_create_or_update_the_metric_binding_in_stackstate[CLIを使用して]SUSE Observabilityにxref:/use/metrics/k8s-add-charts.adoc#_create_or_update_the_metric_binding_in_stackstate[適用]します(後で削除できます)。

----
nodes:
- _type: MetricBinding
  chartType: line
  enabled: true
  tags: {}
  unit: short
  name: CPU Usage (different aggregations and intervals)
  priority: HIGH
  identifier: urn:custom:metric-binding:pod-cpu-usage-a
  queries:
    - expression: sum(max_over_time(container_cpu_usage{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", pod_name="${name}"}[${__interval}])) by (cluster_name, namespace, pod_name) /1000000000
      alias: max_over_time dynamic interval
    - expression: sum(min_over_time(container_cpu_usage{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", pod_name="${name}"}[${__interval}])) by (cluster_name, namespace, pod_name) /1000000000
      alias: min_over_time dynamic interval
    - expression: sum(avg_over_time(container_cpu_usage{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", pod_name="${name}"}[${__interval}])) by (cluster_name, namespace, pod_name) /1000000000
      alias: avg_over_time dynamic interval
    - expression: sum(last_over_time(container_cpu_usage{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", pod_name="${name}"}[${__interval}])) by (cluster_name, namespace, pod_name) /1000000000
      alias: last_over_time dynamic interval
    - expression: sum(max_over_time(container_cpu_usage{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", pod_name="${name}"}[1m])) by (cluster_name, namespace, pod_name) /1000000000
      alias: max_over_time 1m interval
    - expression: sum(min_over_time(container_cpu_usage{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", pod_name="${name}"}[1m])) by (cluster_name, namespace, pod_name) /1000000000
      alias: min_over_time 1m interval
    - expression: sum(avg_over_time(container_cpu_usage{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", pod_name="${name}"}[1m])) by (cluster_name, namespace, pod_name) /1000000000
      alias: avg_over_time 1m interval
    - expression: sum(last_over_time(container_cpu_usage{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", pod_name="${name}"}[1m])) by (cluster_name, namespace, pod_name) /1000000000
      alias: last_over_time 1m interval
  scope: (label = "stackpack:kubernetes" and type = "pod")
----

適用後、SUSE Observabilityでポッドのメトリクス・パースペクティブを開きます(CPU使用率にスパイクと谷があるポッドが望ましい)。右上のアイコンを使ってチャートを拡大すると、より見やすくなります。また、時間窓を変えて、異なる集計による効果を見ることもできる（たとえば30分と24時間）。

[CAUTION]
====
メトリックバインディングで集計が指定されていない場合、SUSE Observabilityは自動的に`+last_over_time+` 集計を使用して、グラフのデータポイント数を減らします。xref:/use/metrics/k8s-writing-promql-for-charts.adoc#_why[これについては、「なぜこれが必要なのか？]
====

== なぜこれが必要なのか？

まず、なぜアグリゲーションを使う必要があるのか？チャートに収まる以上のデータポイントをメトリックスストアから取得するのは意味がない。したがって、SUSE Observabilityは、良い結果を得るために必要な2つのデータポイント間のステップを自動的に決定します。短い時間ウィンドウ（例えば1時間分のデータしか表示しないチャート）では、この結果、ステップが小さくなる（約10秒）。メトリクスは30秒ごとにしか収集されないことが多いので、10秒のステップでは、次の値に変わる前に同じ値が3ステップ繰り返される。1週間のタイム・ウィンドウにズーム・アウトするには、もっと大きなステップが必要になる（画面上のチャートの正確なサイズにもよるが、約1時間）。

ステップが収集されたデータポイントの分解能よりも大きくなった場合、1時間の時間範囲のデータポイントをどのように1つの値にまとめるかを決定する必要がある。クエリで時間に対する集計がすでに指定されている場合は、それを使用する。しかし、集計が指定されていない場合、または集計間隔がステップより小さい場合、`+step+` サイズを間隔として、`+last_over_time+` 集計が使用される。その結果、各時間の最後のデータポイントだけが、その時間のすべてのデータポイントを「要約」するために使われる。

要約すると、PromQLクエリを1週間、1時間のステップで実行した場合、このクエリは次のようになります：

----
container_cpu_usage /1000000000
----

に自動的に変換される：

----
last_over_time(container_cpu_usage[1h]) /1000000000
----

image::k8s/k8s-metric-queries-for-chart-last-over-time.png[Last over time]image::k8s/k8s-metric-queries-for-chart-max-over-time-fixed-range.png[Max over time with fixed range]image::k8s/k8s-metric-queries-for-chart-max-over-time-interval.png[Max over time with automatic range]

多くの場合、この動作は意図されたものではないので、どのような集約が必要かは自分で判断した方がいい。異なる集計関数を使用することで、（他の振る舞いを隠す代償として）特定の振る舞いを強調することができる。ピークや谷、滑らかなチャートなどを見ることがより重要なのでしょうか？そして、`+${__interval}+` パラメータを範囲に使用すると、クエリに使用される`+step+` サイズに自動的に置き換えられます。その結果、そのステップのすべてのデータポイントが使用される。

image::k8s/k8s-metric-queries-small-range.png[固定レンジ, shorter than the data resolution]image::k8s/k8s-metric-queries-interval-for-range.png[自動レンジ, based on step but with a lower limit]

`+${__interval}+` 。`+step+` 、したがって`+${__interval}+` 、保存されているメートルデータの解像度よりも小さいサイズに縮小されると、チャートにギャップが生じる。

したがって、`+${__interval}+` 、SUSE Observabilityエージェントのデフォルトのスクレイプ間隔（デフォルトのスクレイプ間隔は30秒）の2*より小さくなることはありません。

最後に、`+rate()+` 関数は、率を計算するために、少なくとも2つのデータ点が区間内にあることを必要とする。データポイントが2つ以下では、レートは値を持たない。従って、`+${__rate_interval}+` は常に少なくとも4 *スクレイプ間隔であることが保証される。これにより、データが欠落していない限り、レート・チャートに予期せぬギャップやその他の奇妙な動作が発生しないことが保証される。
