= コンポーネントにカスタムチャートを追加する
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: SUSE Observability

== 概要

SUSE Observabilityは、Kubernetesリソースを表すコンポーネントのほとんどのタイプで、デフォルトですでに多くのメトリクスチャートを提供しています。必要なときはいつでも、どのコンポーネント・セットにも、追加のメートル法チャートを追加することができる。コンポーネントにメトリクスを追加する場合、2つのオプションがあります：

. メトリクスはSUSE Observabilityによってすでに収集されていますが、デフォルトではコンポーネント上に可視化されていません。
. このメトリクスはまだSUSE Observabilityで収集されていないため、まだ利用できません。

オプション1の場合、以下の手順では、特定のコンポーネントのセットに特定のメトリックを追加するようにSUSE Observabilityを設定するメトリックバインディングの作成方法を説明します。

オプション2の場合、まずxref:/use/metrics/k8s-prometheus-remote-write.adoc[Prometheusリモート書き込みプロトコルを]使用してSUSE Observabilityにメトリクスを送信し、SUSE Observabilityでメトリクスが利用可能であることを確認します。その後、メトリクスのチャートをコンポーネントに追加していきます。

== ステップ

メトリックバインディングを作成する手順

. <<_write_the_outline_of_the_metric_binding,メートル装丁の概要を書く>>
. <<_write_the_topology_query,コンポーネントを選択するためのトポロジークエリー（STQL）を書く。>>
. <<_write_the_promql_query,希望するメトリックのPromQLクエリを書く>>
. <<_bind_the_correct_time_series_to_each_component,各コンポーネントに正しい時系列をバインドする>>
. <<_create_or_update_the_metric_binding_in_suse_observability,SUSE Observabilityでメトリックバインディングを作成または更新する>>

例として、Kubernetesデプロイの`+Replica counts+` のメトリックバインディングを追加します。これは単なる例であり、このメトリック・バインディングはSUSE Observabilityにデフォルトで既に存在しています。

== メートル装丁の概要を書く

`+metric-bindings.yaml+` という名前の新しい YAML ファイルを作成し、独自のメトリックバインディングを作成するためにこの YAML テンプレートを追加します。お好きなコード・エディターで開いて、このガイド全体を通して変更してください。最後に、SUSE Observability CLIを使用して、SUSE Observabilityのメトリックバインディングを作成および更新します。

----
nodes:
- _type: MetricBinding
  chartType: line
  enabled: true
  tags: {}
  unit:
  name:
  description:
  priority:
  identifier: urn:custom:metric-binding:...
  queries:
    - expression:
      alias:
  scope:
  layout:
    metricPerspective:
      tab:
      section:
      weight:
    componentSummary:
      weight:
----

このテンプレートのフィールドは以下の通り：

* `+_type+`:SUSE Observabilityは、これがメトリックバインディングであることを認識する必要があるため、値は常に次のようにする必要があります。 `+MetricBinding+`
* `+chartType+`:SUSE Observabilityは、さまざまなチャート・タイプ（`+line+` 、`+bar+` など）をサポートする予定です。現在サポートされているのは`+line+` のみです。
* `+enabled+`:`+false+` に設定すると、メトリックのバインディングは維持されるが、ユーザーには表示されない。
* `+tags+`:ユーザー・インターフェイスでメトリクスを整理するために使用されます。 `+{}+`
* `+unit+`:チャートのY軸を表示するために使用される、クエリまたはクエリによって返される時系列の値の単位。すべてのユニットについては、xref:/develop/reference/k8sTs-chart-units.adoc[サポートされているユニットの]リファレンスを参照してください。
* `+name+`:メトリックバインディングの名前
* `+description+`:名前の上に表示される説明文（オプション
* `+priority+`:[非推奨]`+HIGH+`,`+MEDIUM+`,`+LOW+` のいずれか。コンポーネントのメトリクスの主なソート順（ここに記載されている順）。副次的なソート順は`+name+` 。
* `+identifier+`:メトリックバインディングの一意な識別子として使われるURN（ユニバーサルリソース識別子）。これは`+urn:custom:metric-binding:+` で始まらなければならない。残りの部分は、すべてのメートル法バインディングの中でユニークである限り、フリーフォーマットである。
* `+queries+`:メトリック・バインディングのチャートに表示するクエリのリスト（以下のセクションも参照）。
* `+scope+`:メ ト リ ッ ク バ イ ンデ ィ ン グの ト ポ ロ ジース コ ープ。 こ の メ ト リ ッ ク バ イ ンデ ィ ン グが表示 さ れ る コ ン ポーネ ン ト を選択す る ト ポ ロ ジ ク エ リ。
* `+layout+`:異なるパースペクティブ・ビュー（例：xref:/use/views/k8s-metrics-perspective.adoc[メトリクス・パースペクティブ]）でチャートをグループ化する方法

まず既知の部分をすべて埋める（デプロイメント・レプリカのカウントを例に）。

----
_type: MetricBinding
chartType: line
enabled: true
tags: {}
unit: short
name: Replica counts
priority: MEDIUM
identifier: urn:custom:metric-binding:my-deployment-replica-counts
queries:
  - expression:
    alias:
scope:
----

クエリーとスコープのセクションは次のステップで記入する。使用される単位は`+short+` で、単に数値を表示するだけであることに注意。PromQLクエリを作成する際に、正しい単位を決めることができます。

== トポロジークエリーを書く

xref:/use/views/k8s-topology-perspective.adoc[Topology］パースペクティブの]［Explore］ビューを使用して、http://your-instance/#/views/explore, 、新しいメトリックを表示する必要があるコンポーネントを選択します。選択には、ベーシック・ビューとアドバンスド・ビューの両方が使用できる。メトリックバインディングのためにトポロジーを選択する最も一般的なフィールドは、コンポーネントタイプのための`+type+` 、すべてのラベルを選択するための`+label+` 。例えばデプロイメント：

----
type = "deployment" and label = "stackpack:kubernetes"
----

タイプフィルターはすべてのデプロイメントを選択し、ラベルフィルターはKubernetes stackpackによって作成されたコンポーネントのみを選択します（ラベル名は`+stackpack+` 、ラベル値は`+kubernetes+` ）。後者は省略しても同じ結果になる。

アドバンスモードに切り替えて、結果のトポロジークエリーをコピーし、それをメトリックバインディングの`+scope+` フィールドに入れる。

[NOTE]
====
メトリックバインディングはクエリーフィルターのみをサポートし、`+withNeighborsOf+` のようなクエリー関数はサポートされておらず、使用できません。
====


== PromQLクエリを書く

SUSE Observabilityインスタンスのxref:/use/metrics/k8sTs-explore-metrics.adoc[メトリックエクスプローラ]、http://your-instance/#/metrics, 、それを使用して目的のメトリックをクエリします。エクスプローラーには、メトリクス、ラベル、ラベル値だけでなく、PromQL関数や演算子のオートコンプリートが用意されています。最良の結果を得るために、例えば1時間といった短い時間範囲から始める。

レプリカの総数については、`+kubernetes_state_deployment_replicas+` メトリックを使用する。このメトリクスのチャートを時系列データを代表するものにするために、`+${__interval}+` パラメータを使用して集約を行うようにクエリを拡張する：

----
max_over_time(kubernetes_state_deployment_replicas[${__interval}])
----

この具体的なケースでは、`+max_over_time+` を使用して、チャートが常に任意の時点で最も多いレプリカ数を表示するようにする。より長い時間範囲では、レプリカの短いディップが表示されないことを意味する。レプリカの最低数を強調するには、代わりに`+min_over_time+` 。

このクエリーを、メトリックバインディングの`+queries+` フィールドの最初のエントリーの`+expression+` プロパティにコピーする。エイリアスとして`+Total replicas+` 。これはチャートの凡例に表示される名前である。

[NOTE]
====
SUSE Observabilityでは、メトリックチャートのサイズが、チャートに表示されるメトリックの粒度を自動的に決定します。PromQLのクエリはこの動作を最適に利用するように調整することができ、指標の代表的なチャートを得ることができます。xref:/use/metrics/k8s-writing-promql-for-charts.adoc[チャート用PromQLの書き方で]詳しく説明しています。
====


== 各コンポーネントに正しい時系列をバインドする

すべてのフィールドが入力されたメトリックバインディング：

----
_type: MetricBinding
chartType: line
enabled: true
tags: {}
unit: short
name: Replica counts
priority: MEDIUM
identifier: urn:custom:metric-binding:my-deployment-replica-counts
queries:
  - expression: max_over_time(kubernetes_state_deployment_replicas[${__interval}])
    alias: Total replicas
scope: type = "deployment" and label = "stackpack:kubernetes"
----

SUSE Observabilityで作成し、デプロイメントコンポーネントの「Replica count」チャートを表示すると、予期しない結果が得られます。このグラフは、すべてのデプロイメントのレプリカ数を示している。論理的には、この特定のデプロイメントのレプリカカウントという1つの時系列しか期待できない。

image::k8s/k8s-replica-counts-without-binding.png[単一配備の誤ったチャート, it shows the replica count for all deployments]

この問題を解決するには、コンポーネントからの情報を使用して、コンポーネントに特化したPromQLクエリを作成します。十分なメトリック・ラベルでフィルタリングし、コンポーネントの特定の時系列のみを選択する。これは、正しい時系列をコンポーネントに「バインド」することである。Grafanaダッシュボードを作った経験のある人にとっては、これはダッシュボード上のクエリで使用されるパラメータを持つダッシュボードに似ている。メトリック・バインディングのクエリをこう変えてみよう：

----
max_over_time(kubernetes_state_deployment_replicas{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", deployment="${name}"}[${__interval}])
----

image::k8s/k8s-replica-counts-with-binding.png[パラメタライズド・フィルターを追加すると、チャートは期待通りになる。, only 1 time series for this component]

PromQLクエリは、`+cluster_name+` 、`+namespace+` 、`+deployment+` の3つのラベルでフィルタリングするようになりました。これらのラベルに実際の値を指定する代わりに、コンポーネントのフィールドへの変数参照が使われる。この場合、`+cluster-name+` と`+namespace+` のラベルが使用され、`+${tags.cluster-name}+` と`+${tags.namespace}+` が参照される。さらに、コンポーネント名は`+${name}+` で参照される。

サポートされている変数参照は以下の通り：

* コンポーネントのラベルは `+${tags.<label-name>}+`
* コンポーネント名は `+${name}+`

image::k8s/k8s-carts-highlights.png[ラベルとコンポーネント名を表示するコンポーネントハイライトページ（どちらも赤でハイライトされている）]

[NOTE]
====
Kubernetesから特定のコンポーネントのメトリクスを選択するには、通常、クラスタ名、名前空間、コンポーネントタイプと名前の組み合わせで十分です。これらのラベル、または同様のラベルは、通常、ほとんどの測定基準や部品に貼られている。
====


== SUSE Observabilityでメトリックバインディングを作成または更新する

SUSE Observability CLIを使用して、SUSE Observabilityでメトリックバインディングを作成します。`+metric-bindings.yaml+` が保存され、以下のようになっていることを確認する：

----
nodes:
- _type: MetricBinding
  chartType: line
  enabled: true
  tags: {}
  unit: short
  name: Replica counts
  priority: MEDIUM
  identifier: urn:custom:metric-binding:my-deployment-replica-counts
  queries:
    - expression: max_over_time(kubernetes_state_deployment_replicas{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", deployment="${name}"}[${__interval}])
      alias: Total replicas
  scope: type = "deployment" and label = "stackpack:kubernetes"
----

xref:/setup/cli/cli-sts.adoc[SUSE Observability CLIを]使用して、メトリックバインディングを作成します：

[,bash]
----
sts settings apply -f metric-bindings.yaml
----

デプロイのメトリクスパースペクティブを開いて、SUSE Observabilityで結果を確認します。結果に満足できない場合は、YAMLファイルのメトリックバインディングを変更し、コマンドを再度実行して更新するだけです。ノードのリストは、多くのメトリック・バインディングの追加をサポートしている。先ほどと同じ手順で、YAML配列に別のメトリックバインディングのエントリを追加するだけです。

[CAUTION]
====
識別子は、メトリックバインディングの一意のキーとして使用される。識別子を変更すると、既存のバインディングが更新されるのではなく、新しいメトリックバインディングが作成される。
====


`+sts settings+` コマンドにはさらに多くのオプションがあり、たとえば、すべてのメトリックバインディングを一覧表示することができる：

[,bash]
----
sts settings list --type MetricBinding
----

最後に、メトリックバインディングを削除するには

[,bash]
----
sts settings delete --ids <id>
----

このコマンドの`+<id>+` は識別子ではなく、`+sts settings list+` 出力の`+Id+` 列の番号である。

[NOTE]
====
推奨される作業方法は、メトリックバインディング(およびSUSE Observabilityで作成されたその他のカスタムリソース)をYAMLファイルとしてGitリポジトリに保存することです。そこから変更を手動で適用することも、GitHubアクションやGitLabパイプラインなどのCI/CDシステムでSUSE Observability CLIを使用して完全に自動化することもできます。
====


== その他のオプション

=== 1つ以上の時系列をチャートに表示

[NOTE]
====
メートル装丁には1単位しかない（チャートのY軸にプロットされる）。その結果、1つのメトリックバインディングで同じ単位の時系列を生成するクエリのみを組み合わせる必要があります。単位を変換できる場合もある。例えば、CPU使用率はミリコアまたはコアで報告されるかもしれない。ミリコアは、次のように1000倍することでコアに変換できる`+(<original-query>) * 1000+` 。
====


1つのメトリック・バインディングで複数の時系列を取得し、したがって1つのチャートで取得するには、2つの方法がある：

. 1つのコンポーネントについて複数の時系列を返すPromQLクエリーを書く
. メトリック・バインディングにPromQLクエリを追加

最初のオプションについては、xref:/use/metrics/k8s-add-charts.adoc#_using_metric_labels_in_aliases[次のセクションで]例を挙げる。2番目のオプションは、関連する測定基準を比較するのに便利です。典型的な使用例をいくつか挙げる：

* 総レプリカ数と希望レプリカ数と利用可能レプリカ数の比較
* リソースの使用状況：制限、リクエスト、使用量を1つのグラフで表示

メトリックバインディングにさらにクエリーを追加するには、xref:/use/metrics/k8s-add-charts.adoc#_steps[手順]3.と4.を繰り返し、クエリーリストに追加項目としてクエリーを追加するだけです。デプロイメントレプリカ数については、同じチャートに含めることができるいくつかの関連するメトリクスがあります：

----
nodes:
- _type: MetricBinding
  chartType: line
  enabled: true
  tags: {}
  unit: short
  name: Replica counts
  priority: MEDIUM
  identifier: urn:custom:metric-binding:my-deployment-replica-counts
  queries:
    - expression: max_over_time(kubernetes_state_deployment_replicas{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", deployment="${name}"}[${__interval}])
      alias: Total replicas
    - expression: max_over_time(kubernetes_state_deployment_replicas_available{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}",  deployment="${name}"}[${__interval}])
      alias: Available - ${cluster_name} - ${namespace} - ${deployment}
    - expression: max_over_time(kubernetes_state_deployment_replicas_unavailable{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}",  deployment="${name}"}[${__interval}])
      alias: Unavailable - ${cluster_name} - ${namespace} - ${deployment}
    - expression: min_over_time(kubernetes_state_deployment_replicas_desired{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}",  deployment="${name}"}[${__interval}])
      alias: Desired - ${cluster_name} - ${namespace} - ${deployment}
  scope: type = "deployment" and label = "stackpack:kubernetes"
----

image::k8s/k8s-replica-counts-multiple-timeseries.png[複数のメトリクスによるメトリック・バインディング]

=== エイリアスにメトリック・ラベルを使用する

1つのクエリがコンポーネントごとに複数の時系列を返す場合、これはチャートに複数の線として表示される。しかし、凡例ではすべて同じエイリアスを使用する。異なる時系列間の違いを見ることができるように、エイリアスは、`+${label}+` 構文を使用してメトリック・ラベルへの参照を含めることができる。ポッドは複数のコンテナを持つことができることに注意してください：

----
type: MetricBinding
chartType: line
enabled: true
id: -1
identifier: urn:custom:metric-binding:my-pod-restart-count
name: Container restarts
priority: MEDIUM
queries:
- alias: Restarts - ${container}
  expression: max by (cluster_name, namespace, pod_name, container) (kubernetes_state_container_restarts{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", pod_name="${name}"})
scope: (label = "stackpack:kubernetes" and type = "pod")
unit: short
----

`+alias+` 、メトリックの`+container+` ラベルを参照していることに注意。ラベルがない場合、`+${container}+` はトラブルシューティングに役立つリテラルテキストとしてレンダリングされます。

=== レイアウト

各コンポーネントは、k8s、ネットワーキング、実行環境（JVMなど）、プロトコル（HTTP、AMQP）など、さまざまな技術やプロトコルに関連付けることができる。
その結果、各コンポーネントについて多数の異なるメトリクスを表示することができる。読みやすくするために、SUSE Observabilityではこれらのグラフをタブやセクションに整理することができます。
チャート（`+MetricBinding+` ）を特定のタブやセクション内に表示するには、レイアウト・プロパティを設定する必要があります。
レイアウトが指定されていないMetricsBindingは、`+Other+` という名前のタブとセクションに表示されます。

以下に設定例を示す：

----
layout:
  metricPerspective:
    tab: Kubernetes Pod
    section: Performance
    weight: 2
  componentSummary:
    weight: 2
----

フィールド

* `+layout.metricPerspective+` -`+Metrics Perspective+` に表示するメトリクスを定義する。指標はタブとセクションに分類される。
* `+layout.metricPerspective.tab+` - タブ名。タブはアルファベット順に並んでいる。
* `+layout.metricPerspective.section+` - セクション名セクションはアルファベット順に並んでいる。
* `+layout.metricPerspective.weight+` - セクション内の指標は、主に重み（昇順）、次に名前（アルファベット順）でソートされる。
* `+layout.componentSummary+` - コンポーネント選択時に`+Components details+` サイドバーに表示するメトリクスを指定します。チャートは、このプロパティが定義されている場合にのみ表示される。
* `+layout.componentSummary.weight+` - これはチャートの重さを表している。チャートは重量の昇順にソートされ、最初の3つのチャートが表示される。
