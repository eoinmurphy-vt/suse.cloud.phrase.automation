ポリシーの動作は厳密ではないので、実行時にポリシーに設定の詳細を与えることで設定できる。ポリシー作成者には、ポリシー設定の構造を定義する自由がある。

{project-name} はポリシー設定をJSONにシリアライズし、呼び出されるたびにポリシーに提供します。

== 設定の検証

ポリシーは、ユーザーが提供した設定を検証し、正しさをチェックする必要があります。

各ポリシーは、ポリシーの設定を検証する`+validate_settings+` というwaPC関数を登録する。

`+validate_settings+` 関数は、ユーザーによって提供された設定のJSON表現を入力として受け取る。この関数はそれらを検証し、レスポンスとして`+SettingsValidationResponse+` オブジェクトを返す。

`+SettingsValidationResponse+` ：

[subs="+attributes",yaml]
----
{
  # mandatory
  "valid": <boolean>,

  # optional, ignored if accepted - recommended for rejections
  "message": <string>,
}
----

ユーザーが提供した設定が`+valid+` の場合、`+validate_settings+` は`+message+` の内容を無視する。そうでない場合は、`+validate_settings+` 、`+message+`の内容が表示される。

[NOTE]
====

{short-project-name}https://github.com/chimera-kube/policy-server[policy-serverは]、開始時にユーザーから提供されたすべてのポリシー設定を検証する。ポリシーサーバは、そのポリシーの少なくとも1つが間違った設定パラメータを受け取った場合、直ちにエラーで終了します。

====


== 例

例として、https://github.com/kubewarden/psp-capabilities[]psp-capabilitiesポリシーを考えてみましょう。このポリシーは次のような構成形式を持っています：

[subs="+attributes",yaml]
----
allowed_capabilities:
- CHOWN

required_drop_capabilities:
- NET_ADMIN

default_add_capabilities:
- KILL
----

`+validate_settings+` 関数は、以下のJSONドキュメントを入力として受け取る：

[subs="+attributes",json]
----
{
  "allowed_capabilities": [
    "CHOWN"
  ],
  "required_drop_capabilities": [
    "NET_ADMIN"
  ],
  "default_add_capabilities": [
    "KILL"
  ]
}
----

== 総括

各ポリシーはwaPC関数`+validate_settings+` を登録しなければならない。
