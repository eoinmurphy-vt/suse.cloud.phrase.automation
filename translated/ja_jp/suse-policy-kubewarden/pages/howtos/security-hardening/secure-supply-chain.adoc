安全なサプライチェーン・インフラストラクチャは、その部品（リンク）の妥当性を検証することができる。これにより、ユーザーや開発者は、ソフトウェア・コンポーネント（成果物）の管理チェーンを示すことができる。セキュリティ問題を軽減するための積極的なアプローチだ。

https://sigstore.dev/[]Sigstoreプロジェクトは、そのためのツールとインフラを提供している。工芸品のサプライチェーンの完全性を検証するためだ。

{short-project-name} 用途https://github.com/sigstore/cosign[`+cosign+`]シグストア・プロジェクトが提供するhttps://github.com/SigStore/fulcio[フルチオと]https://github.com/sigstore/rekor[レコールの]インフラとともに。

クラスタ・オペレータは、{project-name} 、信頼できるエンティティによって署名されたポリシーのみを実行するように設定できる。ポリシー開発者はポリシーに署名し、レジストリに公開することができる。

== 前提条件

以下のセクションでは、いくつかのツールをインストールする必要があります。これらは、ユーザーがOCIアーティファクトの署名と検証を行えるようにするためのものである。例ではhttps://docs.sigstore.dev/quickstart/quickstart-cosign/[`+cosign+`]およびhttps://github.com/kubewarden/kwctl[`+kwctl+`]ユーティリティを使用しています。

GitHubを使ってポリシーに署名するには、https://docs.sigstore.dev/cosign/system_config/installation/#github-actions[GitHub actions]をインストールする必要があります。

キーレス署名は、Sigstoreプロジェクトが提供するデフォルトのhttps://github.com/SigStore/fulcio[fulcioと]https://github.com/sigstore/rekor[rekor]インスタンスを使用する。必要に応じて、独自のインフラを使用する方法の詳細については、Sigstoreのドキュメントを確認してください。

== 署名ポリシー

{short-project-name} は、Sigstoreのhttps://github.com/sigstore/cosign[コサイン・]ユーティリティを使用してポリシーに署名することを推奨している。このセクションでは、鍵ベースのポリシー署名方法を示す。このためには、秘密鍵と公開鍵のペアを生成する必要がある。生成された鍵は、署名された成果物が期待されるユーザーによるものかどうかを検証するのに役立つ。このキー・ペアを生成するには、`+cosign generate-key-pair+` ：

[subs="+attributes",bash]
----
cosign generate-key-pair
----

パスワードを入力し、確認するプロンプトが表示される：

[subs="+attributes",console]
----
Enter password for private key: ●●●●●●●●
Enter password for private key again: ●●●●●●●●
Private key written to cosign.key
Public key written to cosign.pub
----

これで、このキーを使ってポリシーに署名できる。

秘密鍵ファイルは共有しないでください、`+cosign.key+` 。これは、鍵の所有者だけがポリシーに署名するために使用する秘密ファイルである。

ポリシーに署名するには、`+cosign sign+` を使用し、`+--key+` コマンドライン引数に秘密鍵ファイルを渡します：

[subs="+attributes",bash]
----
cosign sign --key cosign.key ghcr.io/kubewarden/policies/user-group-psp:latest
----

指定された秘密鍵のパスワードの入力を求めるプロンプトが表示される：

[subs="+attributes",console]
----
an error occurred: no provider found for that key reference, will try to load key from disk...
Enter password for private key: ●●●●●●●●
Pushing signature to: ghcr.io/kubewarden/policies/user-group-psp
----

このコマンドは、新しい署名オブジェクトを作成してポリシーに署名します。署名オブジェクトは、ポリシーとともにレジストリにアップロードされる。これで、署名検証を使用した{short-project-name} インストールでポリシーを使用する準備ができた。

同じポリシーは、同じユーザーまたは異なるユーザーによって、複数回署名することができる。これらの署名は、オリジナルの署名とともに署名オブジェクトに追加される。

署名プロセスの仕組みについての詳細は、https://docs.sigstore.dev/[] Sigstoreご覧ください。

=== キーレス署名

多くの場合、ポリシーはCI/CDパイプラインを使って自動的に構築される。これは鍵の生成プロセスを複雑にする。このシグストア・キーレス・ワークフローは、このような状況のためにある。キーレス・ワークフローでは、長寿命の歌唱キーを使用する代わりに、認証局（CA）と証明書チェーンを使用する。

生成された短命の証明書キーを信頼の連鎖に結びつける。これは、署名者の身元を確認するためのIDチャレンジによって行われる。証明書鍵の有効期間は、署名が行われるのに十分な長さである。ID チャレンジは、OpenID Connect（OIDC）プロバイダに対する認証によって行われる。シグストアのフルシオ公共インフラは信頼の連鎖を提供する。

署名にはSigstoreのコサイン・ユーティリティを使用する。

[subs="+attributes",bash]
----
$ cosign sign ghcr.io/kubewarden/policies/user-group-psp:latest
----

=== GitHub ワークフローで成果物に署名する方法

キーレス署名を使う場合、GitHubのアクションでは、`+cosign+` 、ユーザーがOIDCプロバイダーにログインする必要はありません。GitHubトークンは、GitHubワークフローの実行中に利用できます。これは、ユーザーの認証とエフェメラル・キーの生成に使われる。署名プロセスは、キーレス・モードで使用されるものと同じである。これは、{short-project-name} ・プロジェクトがどのようにポリシーに署名しているかの一例である：

.{short-project-name} ポリシー署名を記述する YAML
[%collapsible]
======

[,yaml]
----
# ... beginning of the workflow file ...
jobs:
  build:
    name: Build container image
    runs-on: ubuntu-latest
    steps:
      # ... other steps building the container image ...
      -
      name: Login to GitHub Container Registry
      uses: docker/login-action@v1
      with:
        registry: ghcr.io
        username: ${{ github.repository_owner }}
        password: ${{ inputs.GITHUB_TOKEN }}
      -
      name: Publish Wasm policy artifact to OCI registry with the 'latest' tag
      shell: bash
      if: ${{ startsWith(github.ref, 'refs/heads/') }}
      env:
        COSIGN_EXPERIMENTAL: 1
      run: |
        set -ex
        echo Pushing policy to OCI container registry
        IMMUTABLE_REF=$(kwctl push -o json ${{ PATH_TO_BUILT_WASM_FILE }} ghcr.io/myorg/policies/my-great-policy:latest | jq -r .immutable_ref)
        echo Keyless signing of policy using cosign
        cosign sign ${IMMUTABLE_REF}
      # ... other build steps ...

# ... remainder of the workflow file ...
----

== ポリシーの検証

[subs="+attributes",console]
----
$ kwctl verify -k cosign.pub ghcr.io/kubewarden/policies/user-group-psp:latest
2022-03-29T14:49:31.878180Z  INFO kwctl::verify: Policy successfully verified
----

== ポリシー署名をチェックするためにポリシーサーバーを設定する

{short-project-name} 、信頼できるポリシーのみを実行するように`+ConfigMap+` 。`+ConfigMap+` の構造については、『xref:reference/verification-config.adoc#_signature_configuration_reference[Signature Configuration Reference]』を参照のこと。これは、`+kwctl+` を使ってポリシーを検証するために使われる。`+ConfigMap+` は、`+verification-config+` フィールドの下に許容されるコンフィギュレーションを定義すべきである。

例えば、{short-project-name} GitHub組織によって署名されたポリシーを実行したいとします。このシナリオの`+ConfigMap+` ：

署名要件を格納する`+ConfigMap+` を作成した後、ポリシー サーバを設定できます。`+verificationConfig+` （➀マーク）フィールドに`+ConfigMap+` 名を設定して、ポリシー署名の検証を開始する。

[subs="+attributes",yaml]
----
apiVersion: policies.kubewarden.io/v1alpha2
kind: PolicyServer
metadata:
  name: default
  finalizers:
    - kubewarden
spec:
  image: ghcr.io/kubewarden/policy-server:v0.2.7
  serviceAccountName: policy-server
  replicas: 1
//highlight-next-l
  # name of the confimap with the signatures requirements
  verificationConfig: your_configmap <1>
  env:
    - name: KUBEWARDEN_ENABLE_METRICS
      value: "1"
    - name: KUBEWARDEN_LOG_FMT
      value: otlp
    - name: "KUBEWARDEN_LOG_LEVEL"
      value: "info"
----
<1> `+verificationConfig+`

`+kubewarden-defaults+` Helm chart を使用してデフォルトの Policy Server を配置する場合は、`+policyServer.verificationConfig+` の値に`+ConfigMap+` 名を設定して、このフィールドを構成します。

現在、PolicyServerは信頼されていないAdmissionPoliciesとClusterAdmissionPoliciesの起動を拒否します。実行中の PolicyServer に対して、信頼されていないポリシーを削除するか、署名要件を変更する必要があります。

== シグネチャ設定リファレンス

ファイルに含まれる署名要件を検証することができます。例を挙げよう：

.署名要件ファイル
[%collapsible]
======
[subs="+attributes",yaml]
----
apiVersion: v1

allOf: <1>
  - kind: githubAction
    owner: kubewarden   # mandatory
    annotations:
      env: prod

anyOf: # at least `+anyOf.minimumMatches+` are required to match <2>
  minimumMatches: 2 # default is 1
  signatures:
  - kind: pubKey
    owner: flavio # optional
    key: .... # mandatory
    annotations:  # optional
      env: prod
      foo: bar
  - kind: pubKey
    owner: victor # optional
    key: .... # mandatory
  - kind: genericIssuer
    issuer: https://github.com/login/oauth
    subject:
      equal: alice@example.com
  - kind: genericIssuer
    issuer: https://token.actions.githubusercontent.com
    subject:
      equal: https://github.com/flavio/policy-secure-pod-images/.github/workflows/release.yml@refs/heads/main
  - kind: genericIssuer
    issuer: https://token.actions.githubusercontent.com
    subject:
      urlPrefix: https://github.com/flavio/
  - kind: genericIssuer
    issuer: https://token.actions.githubusercontent.com
    subject:
      urlPrefix: https://github.com/kubewarden # <- it will be post-fixed with `+/+` for security reasons
  - kind: githubAction
    owner: flavio   # mandatory
    repo: policy1 # optional
  - kind: pubKey
    owner: alice # optional
    key: .... # mandatory
----
<1> `+allOf+`
<2> `+anyOf+`
======

=== 署名検証

上記のコンフィギュレーションには、ハイライトされた2つのセクション、`+allOf+` と`+anyOf+` が含まれている：

* `+allOf+`ポリシーは、ここにあるすべての署名要件が有効である場合にのみ信頼される。
* `+anyOf+``+minimumMatches+` の基準を満たせば、ポリシーは信頼される。

上記、`+minimumMatches+` フィールドは2。つまり、署名の条件のうち少なくとも2つは満たさなければならない。`+minimumMatches+` フィールドのデフォルト値は`+1+` である。

`+allOf+` *、*`+anyOf+` の最低署名数を満たす必要があります。

=== 公開鍵の検証

ポリシーが正しい公開鍵で署名されていることを確認するには、鍵データと鍵の所有者を指定する。この例では、`+kind+` を`+pubKey+` に設定し、`+key+` が公開鍵を持っている。オーナー・フィールドは任意だが、誰がキーの所有者かを明確にするのに便利である。

[subs="+attributes",yaml]
----
  - kind: pubKey
    owner: bob # optional
    key: |
      -----BEGIN PUBLIC KEY-----
      MBFKHFDGHKIJH0CAQYIKoZIzj0DAQcDQgAEX0HFTtCfTtPmkx5p1RbDE6HJSGAVD
      BVDF6SKFSF87AASUspkQsN3FO4iyWodCy5j3o0CdIJD/KJHDJFHDFIu6sA==
      -----END PUBLIC KEY-----
----

=== キーレス署名検証

キーレス・モードで署名されたポリシーは、検証する公開鍵を持たない。署名プロセスで使用されたOIDCデータでポリシーを検証することはできる。そのためには、`+genericIssuer+` として署名検証を定義する必要がある。

署名から情報を確認することは可能だ：

* `+issuer+`(必須）：これは、Fulcioが生成した証明書の`+Issuer+` 属性と一致する。これは、ポリシーの署名に使用されたOIDCを示している。
* `+subject+`:Fulcioの証明書の`+Subject+` 属性と一致させるために使用されるフィールド。`+Subject+` (Fulcio)フィールドには、OIDCプロバイダーに対する認証に使用されるユーザーが含まれる。検証フィールド、`+subject+` は、2つのサブフィールドのうちの1つを持つことができる：
 ** `+equal+`:証明書の`+Subject+` （Fulcio）は、署名検証の値と等しくなければならない；
 ** `+urlPrefix+`:証明書の`+Subject+` （Fulcio）フィールドの値の先頭に、署名検証で定義された値を付加する必要がある。

[NOTE]
====

`+cosign verify+` 、`+kwctl inspect+` 、キーレス署名に関する情報を表示することができる。

====


例えば、この設定は、ポリシーがGitHub OIDCを使ったアリスからのキーレス署名を持っていなければならないことを意味する：

[subs="+attributes",yaml]
----
- kind: genericIssuer
  issuer: https://github.com/login/oauth
  subject:
    equal: alice@example.com
----

この設定には、GitHub ユーザー`+flavio+` が所有するリポジトリから GitHub アクションで署名されたポリシーが必要です：

[subs="+attributes",yaml]
----
- kind: genericIssuer
  issuer: https://token.actions.githubusercontent.com
  subject:
    urlPrefix: https://github.com/flavio
----

=== GitHub アクション署名検証

kind",`+githubAction+` は、GitHub Actions で署名されたポリシーを検証するためのものです。
`+genericIssuer+` 。署名要求プロセスを簡素化するために、`+githubAction+` に2つの追加フィールドを使用する：

* `+owner+` (必須）：信頼するユーザーまたは組織のGitHub ID
* `+repo+`信頼するリポジトリ名

例えば、最後のスニペットは、`+genericIssuer+` を使って、次のように書き換えることができる：

[subs="+attributes",yaml]
----
- kind: githubAction
  owner: flavio
----

=== 署名注釈の検証

すべての署名タイプは、他のオプションの検証フィールド、`+annotations+` を持つことができる。
これらのフィールドは、署名プロセス中に追加されるキー／バリュー・データである。

{short-project-name} を使えば、ポリシー署名が信頼されたユーザーによるものであること*、そして*そのユーザーが特定の注釈を持っていることをチェックできる。

次の検証では、ポリシーの2つの条件をチェックする：

* 特定の鍵で署名されていること
* 本番環境アノテーション

[subs="+attributes",yaml]
----
- kind: pubKey
  key: |
    -----BEGIN PUBLIC KEY-----
    MBFKHFDGHKIJH0CAQYIKoZIzj0DAQcDQgAEX0HFTtCfTtPmkx5p1RbDE6HJSGAVD
    BVDF6SKFSF87AASUspkQsN3FO4iyWodCy5j3o0CdIJD/KJHDJFHDFIu6sA==
    -----END PUBLIC KEY-----
  annotations:
    environment: production
----
