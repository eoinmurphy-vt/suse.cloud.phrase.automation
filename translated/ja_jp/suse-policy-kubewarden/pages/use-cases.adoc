== ケースA：Kubernetesのオペレーターとして、私は自分のクラスタが安全でコンプライアンスに準拠していることを保証したい。

`+kubewarden+` Namespaceに、{project-name} とそのデフォルトのコンフィギュレーション、`+kubewarden-defaults+` Helmチャートをデプロイする。これにより、デフォルトのPolicyServerと推奨されるClusterAdmissionPoliciesが`+kubewarden+` 名前空間にデプロイされ、Kubernetesオペレータの制御下にのみ置かれる。

オペレータとして、管理されたネームスペース（`+kubewarden+` など）の下にさらにPolicyServersを追加して、負荷とフォールトトレランスを分散することができます。

オペレータは、より多くのClusterAdmissionPoliciesとClusterAdmissionPolicyGroupsを配置できます。これらは、あらゆるタイプの操作（GET、CREATE、UPDATE、PATCH、DELETE、PROXY）について、すべてのKubernetesリソースをチェックします。これにより、クラスターへのオペレーションが安全かつコンプライアンスに適合したものとなる。

これらには以下が含まれる：

* セキュリティ
* コンプライアンス
* リソースの最適化
* Kubernetes環境のガバナンス（ラベルと命名規則による）
* ベストプラクティス
* イメージ検証

セキュリティへの期待は時代とともに変化する。以前は正しく配置されていたクラスタが、そうでなくなる可能性がある。しかし、{short-project-name} 、すでにクラスターでのオペレーションを受け入れている。このような状況では、オペレータは定期的に実行され、クラスタ内の既存のリソースを評価するCronJobであるAudit Scanner機能を展開することができます。これにより、クラスターが長期間にわたって安全でコンプライアンスに適合していることが確認される。

オペレータは、`+protect+`モードではなく、`+monitor+` モードでポリシーを設定し、操作をブロックすることなくクラスタの状態から学習することができる。

オペレータは、メトリクスとトレースのためにログと OpenTelemetry 情報を消費することで、ポリシーと{short-project-name} スタックから情報を受け取ることができる。

== ケースB：Kubernetesのオペレーターとして、私はKubernetesユーザーにフレームワークを提供し、彼らがNamespacesでセルフサービスできるようにしたい。

オペレーターとして、ケースAと同じように{short-project-name} 。これにより、他のユーザーが回避できない安全なベースラインをクラスタ内で提供することができる。

ケースAと同様に、私はネームスペースごとに、チーム、チーム管理者、テストデプロイメントなど、さまざまなペルソナを持っています。私は、ネームスペースの管理者が、ネームスペースのAdmissionPoliciesとAdmissionPolicyGroupsとともに、自分のネームスペースにPolicyServersをデプロイすることによって、各ネームスペースの管理者がセルフサービスできるようにしている。
このアーキテクチャは、PolicyServerとポリシーをコントロールできることを意味する。ポリシーはそのNamespaceにのみ適用され、リソースの使用はそのNamespaceに限定される。

また、オペレータはノイズの多いテナントを分離し、例えば、高いスループットと低いレイテンシを必要とするテナントやタスクのためにパフォーマンスの高いPolicyServersを予約することができる。

== ケースC：ポリシー作成者として、私は新しいポリシーを書くために自分の知っているツールと言語を使いたい。

{short-project-name} これは、ポリシーのターゲット言語としてWebAssemblyにコンパイル可能なあらゆる言語をサポートすることによって達成される。これは、ポリシー作成者がワークフロー（`+git+` 、CI、エディター、ピアレビューなど）やツール（言語、言語ライブラリ、テストハーネス、フレームワークなど）を再利用できることを意味する。

ドメイン固有言語（Rego、CEL、KyvernoのYAML+マクロなど）や汎用言語（Go、Rust、C#、Javascriptなど、Wasmにコンパイルできるもの）を再利用することができる。{short-project-name} 、いくつかの言語のxref:/tutorials/writing-policies/index.adoc[SDKを]ファーストクラスのサポートとして提供している。

{short-project-name} ポリシーは、単純なものもあれば、複雑なxref:/explanations/context-aware-policies.adoc[コンテキストを考慮した]ポリシーもある。
コンテキストを意識したポリシーは、別のワークロードとのインターフェースにも使用される（たとえば、イメージスキャナの長時間稼動サービスから情報を取得する場合など）。

== ケースD：システム・インテグレーターとして、セキュリティとコンプライアンス・ソリューションの一部として{short-project-name} 。

システムインテグレーターとして、{short-project-name} を再利用したい。また、他のソリューションを{short-project-name} に含めることで、再利用できる可能性もある。

システムインテグレーターとして、私は{short-project-name} の一部を再利用することができる。例えば、`+policy-server+` 、xref:/howtos/raw-policies.adoc["raw policies "]機能を介してKubernetesクラスタの内部または外部のリソースを取り締まる。

システムインテグレーターは、`+kubewarden-controller+` 、またはCRDを自社で管理することを選択できる。監査スキャナーの導入や拡張は、必要に応じて選択できる。

システム・インテグレーターは新しいコンポーネントを作ることができる。例えば、イメージスキャナであれば、Kubernetesコントローラにモノリシックな実装をすることなく、コンテキストを意識したポリシーでインターフェイスすることができる。

== 非ゴール

{short-project-name} はそのつもりはない：

- Kubernetesに組み込まれたセキュリティ機能を置き換えるのではなく、補完する：
  - {short-project-name} はPSPからの移行を提供する。
  - ValidatingAdmissionPolicies と CEL ポリシーは、{short-project-name} の`+cel-policy+` で再利用できます。
  - {short-project-name} 一方、ポッドセキュリティアドミッションはそうではない。
  - {short-project-name} ポリシーは、{short-project-name} スタック機能（監査スキャナー、テレメトリー、CRD管理）の恩恵を受ける。
- 侵入検知やランタイム・コンテナの分離のようなランタイム・セキュリティを提供する。
- クラスタのホストシステム保護を提供する。
- 政策実行の柔軟性を無限に提供する。DoS攻撃を防ぐため、ポリシーの処理時間は制限されている。
