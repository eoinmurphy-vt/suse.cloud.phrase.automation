= 分散型ハイブリッドまたはマルチクラウドクラスター
:page-languages: [en、ja、ko、zh]。
:revdate: 2025-05-26
:page-revdate: {revdate}

K3sクラスタは、共通のプライベートネットワークを共有せず、直接接続されていないノード（異なるパブリッククラウドのノードなど）にもデプロイできます。これを実現するには、組み込み型のk3sマルチクラウド・ソリューションと、`+tailscale+` VPNプロバイダーとの統合という2つのオプションがある。

[CAUTION]
====
外部接続に必要なホップ数が増えるため、ノード間の待ち時間は長くなる。これはネットワーク・パフォーマンスを低下させ、レイテンシーが高すぎる場合はクラスタの健全性にも影響を与える可能性がある。
====


[CAUTION]
====
このタイプのデプロイメントでは、埋め込みetcdはサポートされていません。組み込みetcdを使用する場合、すべてのサーバーノードはプライベートIPを介して互いに到達可能でなければならない。エージェントは複数のネットワークに分散していてもよいが、すべてのサーバーは同じ場所にあるべきである。
====


== 組み込み型k3sマルチクラウドソリューション

K3sはクラスタトラフィックのVPNメッシュを確立するためにwireguardを使用しています。ノードはそれぞれ、到達可能な一意のIP（通常はパブリックIP）を持っていなければならない。K3sのスーパーバイザートラフィックはwebsocketトンネルを使用し、クラスタ（CNI）トラフィックはwireguardトンネルを使用します。

このタイプの配置を有効にするには、サーバーに次のパラメータを追加する必要があります：

[,bash]
----
--node-external-ip=<SERVER_EXTERNAL_IP> --flannel-backend=wireguard-native --flannel-external-ip
----

そしてエージェントについて：

[,bash]
----
--node-external-ip=<AGENT_EXTERNAL_IP>
----

ここで、`+SERVER_EXTERNAL_IP+` はサーバーノードに到達できるIPであり、`+AGENT_EXTERNAL_IP+` はエージェントノードに到達できるIPである。エージェントの`+K3S_URL+` コンフィグパラメータは、`+SERVER_EXTERNAL_IP+` を使用しないと接続できないことに注意してください。xref:installation/requirements.adoc#_networking[Networking Requirements（ネットワーク要件]）をチェックし、内部および外部アドレスのリストされたポートへのアクセスを許可することを忘れないでください。

`+SERVER_EXTERNAL_IP+` 、`+AGENT_EXTERNAL_IP+` の両方が接続可能でなければならず、通常はパブリックIPである。

[IMPORTANT]
.ダイナミックIP
====
ノードに動的IPが割り当てられ、IPが変更された場合（AWSなど）、新しいIPを反映させるために`+--node-external-ip+` パラメータを変更する必要があります。K3sをサービスとして実行する場合は、`+/etc/systemd/system/k3s.service+` を修正してから実行してください：

[,bash]
----
systemctl daemon-reload
systemctl restart k3s
----
====

[#_integration_with_the_tailscale_vpn_provider_experimental]
== Tailscale VPNプロバイダーとの統合（実験的）

v1.27.3、v1.26.6、v1.25.11以降で使用可能。

K3はhttps://tailscale.com/[]Tailscaleと統合することができ、ノードはTailscale VPNサービスを使用してノード間のメッシュを構築する。

K3をデプロイする前に、Tailscaleで行うべき4つのステップがある：

. Tailscaleアカウントにログイン
. `+Settings > Keys+` で、認証キー ($AUTH-KEY) を生成します。これはクラスタ内のすべてのノードで再利用できます。
. クラスタが使用するPodCIDRを決定します（デフォルトでは`+10.42.0.0/16+` ）。アクセス制御のCIDR（デュアルスタックの場合はCIDR）をスタンザに追加する：
+
[,yaml]
----
"autoApprovers": {
     "routes": {
         "10.42.0.0/16":        ["your_account@xyz.com"],
         "2001:cafe:42::/56": ["your_account@xyz.com"],
     },
 },
----

. ノードにTailscaleをインストールする：
+
[,bash]
----
curl -fsSL https://tailscale.com/install.sh | sh
----

Tailscale統合を有効にしてK3をデプロイするには、各ノードに以下のパラメータを追加する必要があります：

[,bash]
----
--vpn-auth="name=tailscale,joinKey=$AUTH-KEY"
----

または、その情報をファイルに記述し、そのパラメータを使用する：

[,bash]
----
--vpn-auth-file=$PATH_TO_FILE
----

オプションとして、独自のTailscaleサーバー（headscaleなど）を持っている場合は、vpn-authパラメーターに`+,controlServerURL=$URL+` を追加することで接続できる。

次に、以下のコマンドを使ってサーバーを作成する：

[,bash]
----
k3s server --token <token> --vpn-auth="name=tailscale,joinKey=<joinKey>" --node-external-ip=<TailscaleIPOfServerNode>
----

このコマンドを実行したら、Tailscaleの管理コンソールにアクセスして、Tailscaleノードとサブネットを承認します（autoApproversでまだ承認されていない場合）。

サーバーがセットアップされたら、エージェントと接続する：

[,bash]
----
k3s agent --token <token> --vpn-auth="name=tailscale,joinKey=<joinKey>" --server https://<TailscaleIPOfServerNode>:6443 --node-external-ip=<TailscaleIPOfAgentNode>
----

この場合も、サーバーの場合と同様に、Tailscaleノードとサブネットを承認する。

TailscaleでACLを有効にしている場合は、ポッド同士の通信を許可する「accept」ルールを追加する必要があります。作成した認証キーが自動的にTailscaleノードにタグを付けると仮定すると、`+testing-k3s+` 、ルールは次のようになります：

[,yaml]
----
"acls": [
    {
        "action": "accept",
        "src":    ["tag:testing-k3s", "10.42.0.0/16"],
        "dst":    ["tag:testing-k3s:*", "10.42.0.0/16:*"],
    },
],
----

[CAUTION]
====
同じtailscaleネットワークを使用して複数のK3sクラスタを実行する予定がある場合、IPの競合を避けるために適切なhttps://tailscale.com/kb/1018/acls/[ACLを]作成するか、クラスタごとに異なるpodCIDRサブネットを使用してください。
====
