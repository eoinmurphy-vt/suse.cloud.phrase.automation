== 概要

このドキュメントは、xref:security/hardening-guide.adoc[K3sセキュリティハードニングガイドに]付随するものです。ハードニングガイドは、K3sの本番インストールをハードニングするための規定ガイダンスを提供しており、このベンチマークガイドは、CIS Kubernetes Benchmarkの各コントロールに対してハードニングされたクラスタのセキュリティレベルを評価するためのものです。K3sのオペレーター、セキュリティチーム、監査役、意思決定者が使用する。

このガイドは、K3sの*v1.26*リリースラインとCIS Kubernetes Benchmarkの*v1.8*リリースに特化したものです。

各コントロールの詳細については、詳細な説明やテストに失敗した場合の対処法を含め、CIS Kubernetes Benchmark v1.8の該当セクションを参照してください。https://www.cisecurity.org/benchmark/kubernetes/[Center for Internet Security (CIS])で無料アカウントを作成後、ベンチマークをダウンロードできます。

=== テストコントロールの方法論

CIS Kubernetes Benchmarkの各コントロールは、付属のハードニングガイドに従って構成されたK3sクラスタに対して評価された。

コントロール監査がオリジナルのCISベンチマークと異なる場合、K3に固有の監査コマンドをテスト用に提供する。

これらは各コントロールで考えられる結果である：

* *合格*- テスト対象のK3sクラスタは、ベンチマークで概説されている監査に合格した。
* *非適用*- このコントロールは、操作の設計上、K3には適用されない。その理由については、救済のセクションで説明する。
* *警告*- CISベンチマークでは制御は手動であり、クラスタのユースケースまたはクラスタオペレータが決定しなければならない他の要因に依存する。これらのコントロールは、K3sがその実装を妨げないことを確認するために評価されているが、テスト対象のクラスタのさらなる設定や監査は行われていない。

このガイドでは、K3sがSystemdユニットとして動作していることを前提としています。あなたのインストールは異なるかもしれないので、あなたのシナリオに合うように「監査」コマンドを調整する必要があります。

=== 1.1.12 etcdデータディレクトリの所有者がetcd:etcdに設定されていることを確認する。

*結果*該当なし

*理由*

K3sの場合、etcdはk3sのプロセスに組み込まれている。独立したetcdプロセスは存在しない。
したがって、etcdデータ・ディレクトリの所有者はk3sプロセスによって管理され、root:rootでなければならない。
