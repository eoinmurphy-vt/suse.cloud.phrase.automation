= リセットのトラブルシューティング
:revdate: 2025-07-03
:page-revdate: {revdate}

`+elemental.cattle.io/resettable: "true"+` アノテーションを持つ各`+MachineInventory+` は、削除時にリセットプランの実行をトリガーする。 +
`+machineinventory.elemental.cattle.io+` ファイナライザは、マシン上で実行されている`+elemental-system-agent+` によってプランが正常に実行された後にのみ削除される。

`+MachineInventory+` 、まだ削除されていない理由を調べることができる：

[,yaml]
----
apiVersion: elemental.cattle.io/v1beta1
kind: MachineInventory
metadata:
  # deletionTimestamp has been set. This object has been marked for deletion.
  deletionTimestamp: "2023-08-04T08:41:25Z"
  annotations:
    # `+resettable+` annotation is enabled.
    # This means the machine has to go through reset, before deletion of this object.
    elemental.cattle.io/resettable: "true"
  # `+machineinventory.elemental.cattle.io+` finalizer is set.
  #  The `+elemental-operator+` is going to create a reset plan for the machine to execute.
  #  After successful execution of the reset plan, the finalizer is removed and the object will be deleted.
  finalizers:
  - machineinventory.elemental.cattle.io
status:
  conditions:
  # Most recent condition shows that the MachineInventory is waiting for a plan to be applied.
  - lastTransitionTime: "2023-08-04T08:41:25Z"
    message: waiting for plan to be applied
    reason: WaitingForPlan
    status: "False"
    type: Ready
  # The plan to be executed is referenced.
  # Normally it has the same name of the MachineInventory and lives within the same namespace.
  plan:
    checksum: 5aba8b6b3161bc52d8953b2428e54ecda3b59e8e0043b49d761d1e79174eded6
    secretRef:
      name: m-bf1008a1-61d6-4355-b5f5-f7d1c527affe
      namespace: fleet-default
----

`+elemental-system-agent+` がプランの実行に成功した場合、`+MachineInventory+` 上の`+machineinventory.elemental.cattle.io+` ファイナライザは削除され、`+MachineInventory+` は削除される。 +
これは、マシンがまだ完全にリセットされていないことを示すものではないことに注意してください。 +
これは現在の実装の限界であり、いずれ改善され、リセット状態を完全に追跡できるようになるだろう。

しかし、この段階では、ホストが再起動し、リカバリーモードで再起動することが予想されます。 +
リカバリーモードに入ったら、`+cos-setup-network+` 、`+/oem/reset-cloud-config.yaml+` に書き込まれたcloud-init configを実行する。 +
`+journalctl -u cos-setup-network -f+` でステータスを監視できます。

Cloud-initの手順は以下のようになるはずだ：

[,yaml]
----
name: {elemental-product-name} Reset
stages:
    network.after:
        - if: '[ -f /run/cos/recovery_mode ]'
          name: Runs elemental reset
          commands:
            - systemctl start elemental-register-reset
----

`+elemental-register+` cliは新しいマシンとして`+elemental-operator+` 。これにより、新しい`+MachineInventory+` オブジェクトが作成される。 +
リモートの`+MachineRegistration+` コンフィギュレーションも、例えば`+reset-persistent+` や`+reset-oem+` のようなリセットオプション、あるいは`+reboot+` や`+power-off+` のような電源設定を適用するためにフェッチされる。 +
リセット後、設定に応じてマシンはシャットダウンするか再起動し、新しいクラスタ内で採用できるようになる。

== リセット中のMachineInventoryの強制削除

マシンがリセット命令を実行できず、関連する`+MachineInventory+` が削除されない場合、手動で問題を解決する方法が2つあります。

* `+MachineInventory+` から`+elemental.cattle.io/resettable: "true"+` のアノテーションを削除する。
* `+machineinventory.elemental.cattle.io+` ファイナライザーを`+MachineInventory+` から取り外す。

マシンを完全に再ビジョニングするか、リカバリーモードに再起動して`+elemental reset+` 。
