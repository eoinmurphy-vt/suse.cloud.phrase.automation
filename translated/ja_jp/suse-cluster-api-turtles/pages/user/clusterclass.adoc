= クラスタクラス
:revdate: 2026-02-18
:page-revdate: {revdate}

このセクションでは、{product_name} でhttps://cluster-api.sigs.k8s.io/tasks/experimental-features/cluster-class/[ClusterClass を]使用する方法について説明する。

== 前提条件

* {product_name} がインストールされた Rancher Manager クラスタ
* すべてのClusterClassを使用するための前提条件として、https://github.com/rancher/cluster-api-addon-provider-fleet[CAAPF]プロバイダをインストールする必要があります。
** CAAPF プロバイダのインストール例：
+
.クリックして拡大
[%collapsible]
======
[,yaml]
----
apiVersion: v1
kind: Namespace
metadata:
  name: fleet-addon-system
---
apiVersion: turtles-capi.cattle.io/v1alpha1
kind: CAPIProvider
metadata:
  name: fleet
  namespace: fleet-addon-system
spec:
  name: rancher-fleet
  type: addon
----
======

[WARNING]
====
CAAPFは`+WatchList+` Kubernetes feature gateに依存している。この機能は、Kubernetes`+1.33+` バージョンで明示的に有効にする必要がある。詳細については、https://kubernetes.io/docs/reference/command-line-tools-reference/feature-gates/[]Kubernetes参照してください。
====

* お使いのエンジンによっては、CAPIProviderのセットアップの前に、https://github.com/rancher/cluster-api-provider-rke2[]CAPRKE2またはhttps://github.com/kubernetes-sigs/cluster-api[]Kubeadm Bootstrap/Control Planeのインストールが必要です：
+

== セットアップ

ifeval::["{build-type}" == "community"]
[NOTE]
.バージョンのピン留め
====
CAPIProvider`+spec.version+` は、プロバイダを特定のバージョンに固定するために、いつでも設定できる。不変性を持続させるために、`+spec.enableAutomaticUpdate: false+` 、それぞれのCAPIProviderの自動バージョン更新を無効にすることもできます。
====
endif::[]

[tabs]
======
Azure::
+
--
管理クラスタを準備するために、https://capz.sigs.k8s.io/[Cluster API Provider Azureを]インストールし、https://capz.sigs.k8s.io/topics/identities#service-principal[ServicePrincipal]IDを作成して、Azure上に新しいクラスタをプロビジョニングします。

開始する前に、少なくともAzureサブスクリプションへのContributorアクセス権を持つServicePrincipalを作成する必要があります。
詳しくは、https://capz.sigs.k8s.io/topics/identities[]CAPZの参照してください。

* プロバイダー設置
+
[source,yaml]
----
apiVersion: v1
kind: Namespace
metadata:
  name: capz-system
---
apiVersion: turtles-capi.cattle.io/v1alpha1
kind: CAPIProvider
metadata:
  name: azure
  namespace: capz-system
spec:
  type: infrastructure
  name: azure
----

* アイデンティティ・セットアップ
+
[WARNING]
.AKSクラスタ・プロビジョニングのための認証
====
**AKSで**マネージドクラスタをデプロイしている場合は、**Identity Setup**ステップをスキップしてxref:#_create_a_cluster_from_a_clusterclass[ClusterClassからCreate a Clusterに]ジャンプできます。
`+AzureClusterIdentity+` を使用する代わりに、クラスタ マニフェストを適用するときに使用する Azure Service Operator 用のサービス プリンシパル資格情報で`+Secret+` を作成します。
====
+
この例では、`+AzureClusterIdentity+` を使用して Azure クラスターをプロビジョニングする。
サービスプリンシパルの認証情報を含むシークレットを最初に作成し、`+AzureClusterIdentity+` リソースから参照する必要があります。
`+AzureClusterIdentity+` は名前空間リソースであり、クラスタと同じ名前空間に作成する必要があることに注意してください。
Azure ID を使用する際のベストプラクティスの詳細については、https://capz.sigs.k8s.io/topics/identities-use-cases[の]公式参照してください。
+
いくつかの変数は、ユーザーが代入できるようになっている。
+
[source,yaml]
----
apiVersion: v1
kind: Secret
metadata:
  name: <AZURE_CLUSTER_IDENTITY_SECRET_NAME>
  namespace: <AZURE_CLUSTER_IDENTITY_SECRET_NAMESPACE>
type: Opaque
stringData:
  clientSecret: <AZURE_CLIENT_SECRET>
---
apiVersion: infrastructure.cluster.x-k8s.io/v1beta1
kind: AzureClusterIdentity
metadata:
  labels:
    clusterctl.cluster.x-k8s.io/move-hierarchy: "true"
  name: cluster-identity
spec:
  allowedNamespaces: {}
  clientID: <AZURE_APP_ID>
  clientSecret:
    name: <AZURE_CLUSTER_IDENTITY_SECRET_NAME>
    namespace: <AZURE_CLUSTER_IDENTITY_SECRET_NAMESPACE>
  tenantID: <AZURE_TENANT_ID>
  type: ServicePrincipal
----
--

AWS::
+
--
管理クラスタを準備するために、https://cluster-api-aws.sigs.k8s.io/[Cluster API Provider AWSを]インストールし、AWS上で新しいクラスタをプロビジョニングするために必要な認証情報を持つシークレットを作成します。
グローバル認証情報は空白に設定されている。代わりに`+AWSClusterStaticIdentity+` 。

* プロバイダー設置
+
[source,yaml]
----
apiVersion: turtles-capi.cattle.io/v1alpha1
kind: CAPIProvider
metadata:
  name: aws
  namespace: capa-system
spec:
  type: infrastructure
  variables:
    AWS_B64ENCODED_CREDENTIALS: ""
----

* アイデンティティ・セットアップ
+
この例では、`+AWSClusterStaticIdentity+` を使用して AWS クラスターをプロビジョニングする。 +
AWS プロバイダーがインストールされているネームスペースに、認証情報を含むシークレットを作成する必要があります。 +
認証情報のセットアップ方法の詳細については、link:https://cluster-api-aws.sigs.k8s.io/clusterawsadm/clusterawsadm[clusterawsadmドキュメントを]参照してください。 +
`+AWSClusterStaticIdentity+` はこのシークレットを参照してクラスタのプロビジョニングを許可できます。この例では、簡単に再利用できるように、すべてのネームスペースでの ID の使用を許可しています。 +
ID管理の詳細については、link:https://cluster-api-aws.sigs.k8s.io/topics/multitenancy[の公式ドキュメントを]参照してください。
+
いくつかの変数は、ユーザーが代入できるようになっている。 +
+
[source,yaml]
----
apiVersion: v1
kind: Secret
metadata:
  name: <AWS_IDENTITY_SECRET_NAME>
  namespace: capa-system
type: Opaque
stringData:
  AccessKeyID: <AWS_ACCESS_KEY_ID>
  SecretAccessKey: <AWS_SECRET_ACCESS_KEY>
---
apiVersion: infrastructure.cluster.x-k8s.io/v1beta2
kind: AWSClusterStaticIdentity
metadata:
  name: cluster-identity
spec:
  secretRef: <AWS_IDENTITY_SECRET_NAME>
  allowedNamespaces: {}
----
--

GCP::
+
--
管理Clusterを準備するために、https://cluster-api-gcp.sigs.k8s.io/[Cluster API Provider GCPを]インストールし、GCP上で新しいClusterをプロビジョニングするために必要な認証情報を含むシークレットを作成します。
GCPでクラスタを作成および管理するにはサービス・アカウントが必要で、これには`+Editor+` の権限が必要です。https://cluster-api-gcp.sigs.k8s.io/quick-start#create-a-service-account[] CAPG公式ガイドに従ってください。
base64エンコードされたサービス・アカウント・キーは、プロバイダーの`+GCP_B64ENCODED_CREDENTIALS+` 変数に設定する必要がある。

* プロバイダー設置
+
[source,yaml]
----
apiVersion: turtles-capi.cattle.io/v1alpha1
kind: CAPIProvider
metadata:
  name: gcp
  namespace: capg-system
spec:
  type: infrastructure
  variables:
    GCP_B64ENCODED_CREDENTIALS: xxx
----

* ネットワーク設定
+
セルフマネージドGCPクラスタをプロビジョニングするには、Kubernetesノードがコントロールプレーンと通信し、コンテナレジストリからイメージをプルできるようにGCPネットワークを設定する必要がある。
デフォルトのプロバイダーの動作は、パブリックIPがアタッチされていない仮想マシンを作成することなので、ノードがロードバランサーや外部との接続を確立できるようにするには、https://cloud.google.com/nat/docs/overview[クラウドNATが]必要です。
自己管理型GCPクラスタをプロビジョニングするためにGCPネットワークを準備する方法については、https://cluster-api-gcp.sigs.k8s.io/prerequisites#configure-network-and-cloud-nat[]CAPG公式ガイドを参照してください。
+
[NOTE]
====
クラスタ・プロビジョニングのためにGCPネットワークを準備するには、以下の手順が必要です：

- ルーターを作る。
- ルーターに関連付けられたNATを作成する。
====

vSphere::
+
--
管理クラスタを準備するために、https://github.com/kubernetes-sigs/cluster-api-provider-vsphere/blob/main/docs/getting_started.md[Cluster API Provider vSphereを]インストールする。
グローバル認証情報は空白に設定されている。代わりに`+VSphereClusterIdentity+` 。

* プロバイダー設置
+
[source,yaml]
----
apiVersion: v1
kind: Namespace
metadata:
  name: capv-system
---
apiVersion: turtles-capi.cattle.io/v1alpha1
kind: CAPIProvider
metadata:
  name: vsphere
  namespace: capv-system
spec:
  type: infrastructure
  variables:
    VSPHERE_USERNAME: "" 
    VSPHERE_PASSWORD: ""
----

* アイデンティティ・セットアップ
+
この例では、`+VSphereClusterIdentity+` を使用して vSphere クラスターをプロビジョニングします。 +
資格情報を含むシークレットは、vSphere プロバイダがインストールされているネームスペースに作成する必要があります。
`+VSphereClusterIdentity+` はこのシークレットを参照してクラスタのプロビジョニングを許可できます。この例では、簡単に再利用できるように、すべてのネームスペースでの ID の使用を許可しています。
ID管理の詳細については、https://github.com/kubernetes-sigs/cluster-api-provider-vsphere/blob/main/docs/identity_management.md[の公式ドキュメントを]参照してください。
+
[source,yaml]
----
apiVersion: v1
kind: Secret
metadata:
  name: cluster-identity
  namespace: capv-system
type: Opaque
stringData:
  username: xxx
  password: xxx
---
apiVersion: infrastructure.cluster.x-k8s.io/v1beta1
kind: VSphereClusterIdentity
metadata:
  name: cluster-identity
spec:
  secretName: cluster-identity
  allowedNamespaces:
    selector:
      matchLabels: {}
----

== ClusterClassからクラスタを作成する

[tabs]
======

Azure RKE2::
+
--
* Azure ClusterClassは、https://github.com/rancher/turtles/tree/main/examples/clusterclasses[] Turtlesの中にあります。
+
Azure Cloud ProviderやCalico CNIなどのアプリケーションはダウンストリームクラスターにインストールされる。これは、`+cloud-provider: azure+` や`+cni: calico+` のような特定のラベルを持つクラスタを対象とすることで、クラスタ作成時に自動的に行われる。
+
[tabs]
=======
CLI::
+
Azure RKE2 ClusterClassと関連アプリケーションは、サンプルツールを使用して適用できます：
+
[source,bash, subs="attributes"]
----
go run github.com/rancher/turtles/examples@{turtles_version} -r azure-rke2 | kubectl apply -f -
----
kubectl::
+
* または、kubectl を使用して Azure RKE2 ClusterClass を直接適用することもできます：
+
[source,bash, subs="attributes"]
----
kubectl apply -f https://raw.githubusercontent.com/rancher/turtles/refs/tags/{turtles_version}/examples/clusterclasses/azure/rke2/clusterclass-rke2-example.yaml
----

* さらに、ノードを正しく初期化するために、https://capz.sigs.k8s.io/self-managed/cloud-provider-config[Azure Cloud Providerを]各ダウンストリームクラスターにインストールする必要があります。
この例では、https://docs.tigera.io/calico/latest/about/[Calicoを]デフォルトのCNIとしてインストールする。
+
これはhttps://rancher.github.io/cluster-api-addon-provider-fleet/[Cluster API Add-on Provider Fleet を使って]クラスタ作成時に自動的に行うことができます。
ラベル・セレクタを使って新しいクラスタに適用するために、まず2つの`+HelmOps+` 。
+
[source,bash, subs="attributes"]
----
kubectl apply -f https://raw.githubusercontent.com/rancher/turtles/refs/tags/{turtles_version}/examples/applications/ccm/azure/helm-chart.yaml
kubectl apply -f https://raw.githubusercontent.com/rancher/turtles/refs/tags/{turtles_version}/examples/applications/cni/calico/helm-chart.yaml
----
=======

* サンプルのClusterClassからAzure Clusterを作成する
+
いくつかの変数は、ユーザーが代入できるようになっている。
+
[source,yaml]
----
apiVersion: cluster.x-k8s.io/v1beta2
kind: Cluster
metadata:
  labels:
    cluster-api.cattle.io/rancher-auto-import: "true"
    cloud-provider: azure
    cni: calico
  name: azure-quickstart
spec:
  clusterNetwork:
    pods:
      cidrBlocks:
      - 192.168.0.0/16
  topology:
    classRef: 
      name: azure-rke2-example
    controlPlane:
      replicas: 1
    variables:
    - name: subscriptionID
      value: <AZURE_SUBSCRIPTION_ID>
    - name: location
      value: <AZURE_LOCATION>
    - name: resourceGroup
      value: <AZURE_RESOURCE_GROUP>
    - name: azureClusterIdentityName
      value: cluster-identity
    version: v1.34.3+rke2r1
    workers:
      machineDeployments:
      - class: rke2-default-worker
        name: md-0
        replicas: 1
----

* サンプルのClusterClassからAzure AKS Clusterを作成します。
+
`+Cluster+` が作成される Azure AKS`+ClusterClass+` は、Azure Service Operator が管理する`+AzureASOManagedCluster+` 、`+AzureASOManagedControlPlane+` および`+AzureASOManagedMachinePool+` リソースを使用します。
このため、セルフマネージドAzureクラスタ（RKE2付きAzureなど）のように`+AzureClusterIdentity+` 。その代わりに、`+Cluster+` と同じ名前空間にサービスプリンシパルの認証情報を持つ`+Secret+` が作成される。
`+Secret+` 定義は`+Cluster+` マニフェストに埋め込まれています。
+
この認証方法の詳細については、https://azure.github.io/azure-service-operator/guide/authentication/[Azure Service Operator のドキュメントを]参照してください。
+
いくつかの変数は、ユーザーが代入できるようになっている。
+
[source,yaml]
----
apiVersion: v1
kind: Secret
metadata:
  name: aks-credentials
stringData:
  AZURE_SUBSCRIPTION_ID: <AZURE_SUBSCRIPTION_ID>
  AZURE_TENANT_ID: <AZURE_TENANT_ID>
  AZURE_CLIENT_ID: <AZURE_CLIENT_ID>
  AZURE_CLIENT_SECRET: <AZURE_CLIENT_SECRET>
type: Opaque
---
apiVersion: cluster.x-k8s.io/v1beta2
kind: Cluster
metadata:
  labels:
    cluster-api.cattle.io/rancher-auto-import: "true"
  name: azure-aks-quickstart
spec:
  clusterNetwork:
    pods:
      cidrBlocks:
      - 192.168.0.0/16
  topology:
    class: azure-aks-example
    variables:
    - name: location
      value: <AZURE_LOCATION>
    - name: resourceGroup
      value: <AZURE_RESOURCE_GROUP>
    - name: asoCredentialSecretName
      value: aks-credentials
    version: v1.34.2
    workers:
      machinePools:
      - class: default-system
        name: system-1
        replicas: 1
      - class: default-worker
        name: worker-1
        replicas: 1
----
--

* クラスタ構成
+
vSphere Cloud Provider と vSphere CSI コントローラは、ダウンストリーム クラスタに適用される追加の設定が必要です。
上記の手順と同様に、ダウンストリームクラスターに適用するフリートバンドルも2つ追加で作成します。
これらのバンドルは、`+vsphere-kubeadm-quickstart+` という名前で下流のクラスタをターゲットに設定されていることに注意してください。
クラスタに別の名前を使用する場合は、それに応じてBundleターゲットを変更してください。
+
[source,yaml]
----
kind: Bundle
apiVersion: fleet.cattle.io/v1alpha1
metadata:
  name: vsphere-csi-config
spec:
  resources:
  - content: |-
      apiVersion: v1
      kind: Secret
      type: Opaque
      metadata:
        name: vsphere-config-secret
        namespace: vmware-system-csi
      stringData:
        csi-vsphere.conf: |+
          [Global]
          thumbprint = "<VSPHERE_THUMBPRINT>"
          [VirtualCenter "<VSPHERE_SERVER>"]
          user = "<VSPHERE_USER>"
          password = "<VSPHERE_PASSWORD>"
          datacenters = "<VSPHERE_DATACENTED>"
          [Network]
          public-network = "<VSPHERE_NETWORK>"
          [Labels]
          zone = ""
          region = ""
  targets:
  - clusterSelector:
      matchLabels:
        csi: vsphere
        cluster.x-k8s.io/cluster-name: 'vsphere-kubeadm-quickstart'
---
kind: Bundle
apiVersion: fleet.cattle.io/v1alpha1
metadata:
  name: vsphere-cloud-credentials
spec:
  resources:
  - content: |-
      apiVersion: v1
      kind: Secret
      type: Opaque
      metadata:
        name: vsphere-cloud-secret
        namespace: kube-system
      stringData:
        <VSPHERE_SERVER>.password: "<VSPHERE_PASSWORD>"
        <VSPHERE_SERVER>.username: "<VSPHERE_USER>"
  targets:
  - clusterSelector:
      matchLabels:
        cloud-provider: vsphere
        cluster.x-k8s.io/cluster-name: 'vsphere-kubeadm-quickstart'
----

* サンプルのClusterClassからvSphereクラスタを作成する
+
この例では、https://kube-vip.io/[]kube-vipをControl Planeロードバランサーとして使用していることに注意してください。
`+KUBE_VIP_INTERFACE+` 、ARPモードで`+CONTROL_PLANE_IP+` 。オペレーティング・システムやネットワーク・デバイスの構成に応じて、この値を適宜設定する必要があります。例えば、`+eth0+` 。
`+kube-vip+` 静的マニフェストは ClusterClass 定義に埋め込まれます。独自のClusterClasses用に静的なkube-vipマニフェストを生成する方法の詳細については、https://kube-vip.io/docs/installation/static/[の]公式参照してください。
+
[source,yaml]
----
apiVersion: cluster.x-k8s.io/v1beta2
kind: Cluster
metadata:
  labels:
    cni: calico
    cloud-provider: vsphere
    csi: vsphere
    cluster-api.cattle.io/rancher-auto-import: "true"
  name: 'vsphere-kubeadm-quickstart'
spec:
  clusterNetwork:
    pods:
      cidrBlocks:
      - 192.168.0.0/16
  topology:
    classRef: 
      name: vsphere-kubeadm-example
    version: v1.35.0
    controlPlane:
      replicas: 3
    workers:
      machineDeployments:
      - class: vsphere-kubeadm-example-worker
        name: md-0
        replicas: 1
    variables:
    - name: vSphereClusterIdentityName
      value: cluster-identity
    - name: vSphereTLSThumbprint
      value: <VSPHERE_THUMBPRINT>
    - name: vSphereDataCenter
      value: <VSPHERE_DATACENTER>
    - name: vSphereDataStore
      value: <VSPHERE_DATASTORE>
    - name: vSphereFolder
      value: <VSPHERE_FOLDER>
    - name: vSphereNetwork
      value: <VSPHERE_NETWORK>
    - name: vSphereResourcePool
      value: <VSPHERE_RESOURCE_POOL>
    - name: vSphereServer
      value: <VSPHERE_SERVER>
    - name: vSphereTemplate
      value: <VSPHERE_TEMPLATE>
    - name: controlPlaneIpAddr
      value: <CONTROL_PLANE_IP>
    - name: controlPlanePort
      value: 6443
    - name: sshKey
      value: <SSH_KEY>
    - name: kubeVIPInterface
      value: <KUBE_VIP_INTERFACE>
----
