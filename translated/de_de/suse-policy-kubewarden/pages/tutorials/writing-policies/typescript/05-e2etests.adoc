In diesem Abschnitt wird gezeigt, wie Sie End-to-End-Tests schreiben können, die gegen die tatsächliche WebAssembly-Binärdatei laufen, die von der Javy-Kompilierungs-Toolchain erzeugt wird.

== Voraussetzungen

Denken Sie daran, dass Sie diese Werkzeuge auf Ihrem Entwicklungsrechner benötigen:

* `+bats+`: Wird verwendet, um die Tests zu schreiben und ihre Ausführung zu automatisieren.
* `+kwctl+` >= v1.30: CLI-Tool, das von {project-name} bereitgestellt wird, um unter anderem seine Richtlinien außerhalb von Kubernetes auszuführen. Dies wird in xref:/tutorials/testing-policies/index.adoc[den Testrichtlinien] in der Dokumentation behandelt.

== Tests schreiben

Sie werden `+bats+` verwenden, um Ihre Tests zu schreiben und zu automatisieren. Jeder Test besteht aus den folgenden Schritten:

. Führen Sie die Richtlinie unter `+kwctl run+` direkt mit der JSON-Ressourcendatei aus.
. Führen Sie Assertions gegen die von `+kwctl+` erzeugte Ausgabe durch.

Alle End-to-End-Tests werden in einer Datei namens `+e2e.bats+` gespeichert. Das Projekt "Gerüst" enthält ein Beispiel, `+e2e.bats+`. Sie müssen den Inhalt erweitern, um eine umfassende Testabdeckung für das Verhalten Ihrer Police zu gewährleisten.

== Testdaten-Dateien

Die End-to-End-Testfälle testen die Erstellung von spezifischen Kubernetes-Ressourcen.

Zum Beispiel, wenn Sie die Erstellung einer Pod-Ressource testen:

[source,json]
----
{
  "apiVersion": "v1",
  "kind": "Pod",
  "metadata": {
    "name": "test-pod"
  },
  "spec": {
    "containers": [
      {
        "name": "nginx",
        "image": "nginx:latest"
      }
    ]
  }
}
----

Der End-to-End-Testfall verwendet eine JSON-Datei, die eine Kubernetes AdmissionReview enthält. Diese JSON-Objekte AdmissionReviews JSONs sind diejenigen, die an die Kubernetes-API gesendet werden, damit sie xref:/reference/spec/03-validating-policies.adoc#an-example[validiert] werden.

Sie können die JSON-Objekte von AdmissionReviews mit Hilfe von `+kwctl scaffold admission-request+` erhalten.
Unter `+test_data/pod.json+` finden Sie einen AdmissionRequest JSON mit einem Pod.

== Grundlegende Testfälle

Die Vorlage enthält bereits einige grundlegende Tests. So sollte die vollständige Datei `+e2e.bats+` aussehen:

[source,bash]
----
#!/usr/bin/env bats
----

Hier sind die bestehenden Tests mit Erklärungen:

=== Test 1: Verweigerte Hostnamen ablehnen

[source,bash]
----
@test "reject because hostname is on deny list" {
  run kwctl run annotated-policy.wasm -r test_data/pod_with_hostname.json --settings-json '{"denied_hostnames": ["forbidden-host", "test-hostname"]}'

  # this prints the output when one of the checks below fails
  echo "output = ${output}"

  # request rejected
  [ "$status" -eq 0 ]
  [ $(expr "$output" : '.*allowed.*false') -ne 0 ]
  [ $(expr "$output" : ".*Pod hostname 'test-hostname' is not allowed.*") -ne 0 ]
}
----

Dieser Test stellt sicher, dass die Richtlinie Pods ablehnt, wenn ihr Hostname in der Ablehnungsliste steht.

=== Test 2: Erlaubte Hostnamen akzeptieren

[source,bash]
----
@test "accept because hostname is not on the deny list" {
  run kwctl run annotated-policy.wasm -r test_data/pod_with_hostname.json --settings-json '{"denied_hostnames": ["forbidden-host"]}'

  # this prints the output when one of the checks below fails
  echo "output = ${output}"

  # request accepted
  [ "$status" -eq 0 ]
  [ $(expr "$output" : '.*allowed.*true') -ne 0 ]
}
----

Mit diesem Test wird überprüft, ob die Richtlinie Pods akzeptiert, deren Hostname nicht in der Ablehnungsliste enthalten ist.

=== Test 3: Akzeptieren, wenn keine Einstellungen vorhanden sind

[source,bash]
----
@test "accept because the deny list is empty" {
  run kwctl run annotated-policy.wasm -r test_data/pod_with_hostname.json

  # this prints the output when one of the checks below fails
  echo "output = ${output}"

  # request accepted
  [ "$status" -eq 0 ]
  [ $(expr "$output" : '.*allowed.*true') -ne 0 ]
}
----

Dieser Test stellt sicher, dass die Richtlinie Anfragen annimmt, wenn Sie keine Einstellungen vornehmen.

=== Test 4: Pods ohne Hostnamen akzeptieren

[source,bash]
----
@test "accept because pod has no hostname set" {
  run kwctl run annotated-policy.wasm -r test_data/pod.json --settings-json '{"denied_hostnames": ["forbidden-host"]}'

  # this prints the output when one of the checks below fails
  echo "output = ${output}"

  # request accepted (no hostname to check)
  [ "$status" -eq 0 ]
  [ $(expr "$output" : '.*allowed.*true') -ne 0 ]
}
----

Mit diesem Test wird überprüft, ob die Richtlinie unabhängig von der Ablehnungsliste auch Pods ohne Hostnamen akzeptiert.

=== Test 5: Nicht-Pod-Ressourcen akzeptieren

[source,bash]
----
@test "accept non-pod resources" {
  run kwctl run annotated-policy.wasm -r test_data/service.json --settings-json '{"denied_hostnames": ["forbidden-host"]}'

  # this prints the output when one of the checks below fails
  echo "output = ${output}"

  # request accepted (not a pod)
  [ "$status" -eq 0 ]
  [ $(expr "$output" : '.*allowed.*true') -ne 0 ]
}
----

Dieser Test stellt sicher, dass die Richtlinie auch Ressourcen akzeptiert, die keine Pods sind, da die Richtlinie nur die Hostnamen von Pods überprüft.

== Erweiterte Testabdeckung

Sie können die Testabdeckung erweitern, indem Sie weitere Testszenarien hinzufügen:

=== Prüfung 6: Validierung der Einstellungen

Sie können auch Tests hinzufügen, um zu überprüfen, ob die Validierung der Einstellungen korrekt funktioniert:

[source,bash]
----
@test "accept valid settings" {
  run kwctl run annotated-policy.wasm -r test_data/pod.json --settings-json '{"denied_hostnames": ["host1", "host2"]}'

  # this prints the output when one of the checks below fails
  echo "output = ${output}"

  # settings are valid, request processed normally
  [ "$status" -eq 0 ]
  [ $(expr "$output" : '.*allowed.*true') -ne 0 ]
}
----

=== Test 7: Grenzfälle

[source,bash]
----
@test "reject with multiple denied hostnames" {
  run kwctl run annotated-policy.wasm -r test_data/pod_with_hostname.json --settings-json '{"denied_hostnames": ["bad-host", "test-hostname", "forbidden-host"]}'

  # this prints the output when one of the checks below fails
  echo "output = ${output}"

  # request rejected
  [ "$status" -eq 0 ]
  [ $(expr "$output" : '.*allowed.*false') -ne 0 ]
  [ $(expr "$output" : ".*Pod hostname 'test-hostname' is not allowed.*") -ne 0 ]
}
----

[source,bash]
----
@test "accept with empty denied hostnames array" {
  run kwctl run annotated-policy.wasm -r test_data/pod_with_hostname.json --settings-json '{"denied_hostnames": []}'

  # this prints the output when one of the checks below fails
  echo "output = ${output}"

  # request accepted
  [ "$status" -eq 0 ]
  [ $(expr "$output" : '.*allowed.*true') -ne 0 ]
}
----

== Durchführung der Tests

Sie können alle End-to-End-Tests mit diesem Befehl ausführen:

[source,console]
----
make e2e
----

Dies ergibt eine Ausgabe wie:

[source,console]
----
bats e2e.bats
e2e.bats
 ✓ reject because hostname is on the deny list
 ✓ accept because hostname is not on the deny list
 ✓ accept because the deny list is empty
 ✓ accept because pod has no hostname set
 ✓ accept non-pod resources
 ✓ accept valid settings
 ✓ reject with multiple denied hostnames
 ✓ accept with empty denied hostnames array

8 tests, 0 failures
----

== Verstehen der Testausgabe

Jeder Test verwendet `+kwctl+`, um die Richtlinien und Prüfungen durchzuführen:

* Status der Beendigung: Der Befehl `+kwctl+` sollte bei erfolgreicher Ausführung der Richtlinie mit Status 0 beendet werden.
* Erlaubtes Feld: Die JSON-Ausgabe enthält ein Feld `+allowed+`, das angibt, ob die Anfrage akzeptiert wurde.
* Inhalt der Nachricht: Bei abgelehnten Anfragen enthält die Ausgabe eine beschreibende Fehlermeldung.

Die `+echo "output = $\{output}"+` Anweisungen in jedem Test helfen bei der Fehlersuche, indem sie die tatsächliche Richtlinienausgabe anzeigen, wenn ein Test fehlschlägt.

== Schlussfolgerung

Die End-to-End-Tests bieten eine umfassende Abdeckung des Richtlinienverhaltens, indem sie mit der tatsächlichen WebAssembly-Binärdatei getestet werden. Dadurch wird sichergestellt, dass die Richtlinie korrekt funktioniert, wenn sie in {short-project-name} bereitgestellt wird, nicht nur in der TypeScript-Entwicklungsumgebung.
