Sie haben Ihre Rego-Police geschrieben, erstellt und ausgeführt.
Jetzt ist es an der Zeit, die Police zu verteilen.

Die Richtlinien müssen mit Anmerkungen versehen werden, damit sie in der `+policy-server+` ausgeführt werden können.
Die `+policy-server+` ist der Teil, der die Richtlinien ausführt, wenn sie in einem Kubernetes-Cluster ausgeführt werden.

== Kommentierung der Politik

Die Kommentierung einer Richtlinie ist ein Prozess, der die Metadaten der Richtlinie mit relevanten Informationen anreichert.
Informationen wie Urheberschaft, Lizenz, Ort des Quellcodes, Regeln, die beschreiben, welche Art von Ressourcen diese Richtlinie versteht und auswertet.

Um Ihre Police mit Anmerkungen zu versehen, müssen Sie eine Datei `+metadata.yaml+` schreiben:

[subs="+attributes",yaml]
----
rules:
- apiGroups: [""]
  apiVersions: ["*"]
  resources: ["*"]
  operations: ["CREATE"]
mutating: false
contextAware: false
executionMode: opa
annotations:
  io.kubewarden.policy.title: no-default-namespace
  io.kubewarden.policy.version: 0.1.0 # should match the OCI tag
  io.kubewarden.policy.description: This policy will reject any resource created inside the default namespace
  io.kubewarden.policy.author: The {project-name} Authors
  io.kubewarden.policy.url: https://github.com/kubewarden/some-policy
  io.kubewarden.policy.source: https://github.com/kubewarden/some-policy
  io.kubewarden.policy.license: Apache-2.0
  io.kubewarden.policy.usage: |
      This policy is just an example.

      You can write interesting descriptions about the policy here.
----

Sie können mehrere Details sehen:

* Regeln:
Auf welche Ressourcen diese Politik abzielt.
* Mutieren:
Ob sich diese Politik ändert.
In diesem Fall handelt es sich um eine reine Validierung.
* Kontextabhängig:
Ob diese Richtlinie einen Kontext aus dem Cluster benötigt, um die Anfrage zu bewerten.
* Ausführungsmodus:
Da es sich um eine Rego-Police handelt, muss angegeben werden, welcher Ausführungsmodus erwartet wird:`+opa+` oder `+gatekeeper+`.
Diese Richtlinie ist im Stil von `+opa+` geschrieben und gibt ein ganzes `+AdmissionReview+` Objekt zurück.
* Anmerkungen: Metadaten, die in der Richtlinie selbst gespeichert sind.

Vermerken Sie Ihre Police:

[subs="+attributes",console]
----
$ kwctl annotate policy.wasm --metadata-path metadata.yaml --output-path annotated-policy.wasm
----

Jetzt können Sie die Richtlinie überprüfen, indem Sie `+kwctl inspect annotated-policy.wasm+` ausführen.

== Die Politik vorantreiben

Nun, da die Richtlinie mit Anmerkungen versehen ist, können Sie sie in ein OCI-Register übertragen.

[subs="+attributes",console]
----
$ kwctl push annotated-policy.wasm registry.my-company.com/kubewarden/no-default-namespace:v0.0.1
Policy successfully pushed
----

Ihre Rego-Richtlinie, die auf das OPA-Framework abzielt, hat alles, was sie braucht, um in der Produktion eingesetzt zu werden, indem sie eine `+ClusterAdmissionPolicy+` erstellt.
Auch das können Sie vorbereiten.
Zunächst müssen Sie die Police in der lokalen Filiale von `+kwctl+` abschließen:

[subs="+attributes",console]
----
$ kwctl pull registry://registry.my-company.com/kubewarden/no-default-namespace:v0.0.1
pulling policy...
----

Erstellen Sie daraus eine `+ClusterAdmissionPolicy+`.
Dieser Vorgang berücksichtigt die Metadaten, die ihm über die Richtlinie vorliegen:

[subs="+attributes",console]
----
$ kwctl manifest registry://registry.my-company.com/kubewarden/no-default-namespace:v0.0.1 --type ClusterAdmissionPolicy
---
apiVersion: policies.kubewarden.io/v1alpha2
kind: ClusterAdmissionPolicy
metadata:
  name: generated-policy
spec:
  module: "registry://registry.my-company.com/kubewarden/no-default-namespace:v0.0.1"
  settings: {}
  rules:
    - apiGroups:
        - ""
      apiVersions:
        - "*"
      resources:
        - "*"
      operations:
        - CREATE
  mutating: false
----

Sie können nun diese `+ClusterAdmissionPolicy+` als Basis verwenden, um die gewünschten Ressourcen anzuvisieren, oder sie unverändert in Kubernetes bereitstellen.
