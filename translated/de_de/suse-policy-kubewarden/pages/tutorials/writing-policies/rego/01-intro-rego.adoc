Die Rego-Sprache ist eine domänenspezifische Sprache, die Richtlinien als Code ermöglicht.https://www.openpolicyagent.org/docs/latest/policy-language/[Rego]ist eine Sprache, die von https://en.wikipedia.org/wiki/Datalog[Datalog] inspiriert ist.

Es gibt zwei Möglichkeiten, Rego-Richtlinien zu schreiben, um Richtlinien als Code in Kubernetes zu implementieren: Open Policy Agent und Gatekeeper.

In den nächsten Abschnitten wird gezeigt, wie sich die beiden Rahmenwerke unterscheiden, obwohl sie dieselbe Sprache verwenden.

== Eine Sprache, zwei Rahmenwerke

=== Open Policy Agent (OPA)

Open Policy Agent ist ein Projekt, das es Ihnen ermöglicht, Richtlinien als Code in jedes Projekt zu implementieren.
Sie können OPA für jede beliebige Police verwenden, um Ihre Anwendungsanforderungen zu prüfen.

In diesem Zusammenhang ist das Schreiben von Richtlinien für Kubernetes einfach die Ausübung von OPA.
Durch die Verwendung von Kubernetes-Zulassungs-Webhooks ist es möglich, Anfragen mit OPA zu bewerten, das die in Rego geschriebenen Richtlinien ausführt.

OPA verfügt über eine optionale Integration mit Kubernetes durch sein `+kube-mgmt+` Sidecar.
Nach dem Einsatz in Kubernetes und der Auswertung der Rego-Richtlinien durch den OPA-Server kann dieser die konfigurierten Kubernetes-Ressourcen in Rego replizieren.
Diese Kubernetes-Ressourcen sind also für alle Richtlinien sichtbar.
Mit OPA können Sie Richtlinien innerhalb der ConfigMap-Objekte von Kubernetes definieren.
Sie können mehr darüber auf derhttps://github.com/open-policy-agent/kube-mgmt[Projektseite] lesen.

=== Torwächter

Gatekeeper konzentriert sich ausschließlich auf die Verwendung in Kubernetes und nutzt dies so weit wie möglich aus, wodurch Kubernetes-Workflows in bestimmten Fällen einfacher sind als OPA.

== Betrachtet man die Unterschiede

Sowohl OPA- als auch Gatekeeper-Richtlinien verwenden Rego, um Richtlinien als Code zu beschreiben.
Jede Lösung unterscheidet sich in Bezug auf die Erstellung von Richtlinien in Rego, wie in den nächsten Abschnitten gezeigt wird.

== Einstiegspunkt

Der Einstiegspunkt ist der Name einer Regel innerhalb eines Pakets und ist die Regel, die von der Laufzeit aufgerufen wird, wenn die Richtlinie ausgeführt wird.

== Aktuelle Einschränkungen

=== Kontextabhängige Richtlinien

Kontextabhängige Richtlinien sind Richtlinien, die die Eingabeanforderung nicht isoliert bewerten.
Sie berücksichtigen andere Faktoren, um eine Entscheidung zu treffen.
Zum Beispiel eine Richtlinie, die Ressourcen mit Namespacing auswertet und eine Anmerkung zum übergeordneten Namespace verwendet, um etwas in der Richtlinie zu konfigurieren.
Ein anderes Beispiel wäre eine Politik, die die Ressourcen von `+Ingress+` bewertet, aber um eine Entscheidung zu treffen, auch die Liste der vorhandenen Ressourcen von `+Ingress+` hat.

Die Idee der kontextbezogenen Richtlinien kann auch auf benutzerdefinierte Ressourcen ausgedehnt werden, so dass Ihre Richtlinie eine Anfrage auch auf der Grundlage der aktuell verwendeten benutzerdefinierten Ressourcen bewerten kann.

Sowohl OPA als auch Gatekeeper unterstützen kontextabhängige Richtlinien, beginnend mit der Version {short-project-name} 1.9.

Weitere Einzelheiten zu kontextbezogenen Maßnahmen finden Siexref:reference/spec/05-context-aware-policies.adoc[hier].

=== Mutierende Politik

Gatekeeper bietet Unterstützung für mutierende Richtlinien, aber {short-project-name} hat noch keine mutierenden Richtlinien mit Gatekeeper-Kompatibilität implementiert.
Sie können Richtlinien verwenden, die das {short-project-name} SDK verwenden, um mutierende Richtlinien zu schreiben, aber derzeit können Sie keine mutierenden Gatekeeper-Richtlinien in {short-project-name} ausführen.
