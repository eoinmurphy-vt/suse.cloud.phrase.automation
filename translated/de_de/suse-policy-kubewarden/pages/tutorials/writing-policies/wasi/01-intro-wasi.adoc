Das https://wasi.dev/[WebAssembly System Interface (WASI)] ist ein WebAssembly-Standard, der eine Reihe von Schnittstellen bereitstellt, die die Ausführung von WebAssembly außerhalb des Browsers ermöglichen.

[CAUTION]
====

Autoren, die reguläre Richtlinien schreiben, sollten niemals einfache WASI-Systemschnittstellen zum Schreiben von Richtlinien verwenden.

Diese Seite ist für {project-name} Maintainer oder Low-Level-Policy-Autoren, die mit neuen WASM-Plattformen experimentieren wollen.
====


Mit WASI können Sie ein WebAssembly-Modul erstellen, das mit Systemprimitiven wie STDOUT, STDERR, STDIN, Umgebungsvariablen und anderen interagiert.

Viele der Compiler, die zum Kompilieren von {short-project-name} -Richtlinien verwendet werden, erzeugen WebAssembly-Module, die auf WASI-Schnittstellen abzielen.
{short-project-name} Policen verwenden jedoch das https://github.com/wapc[waPC-Projekt], um die bidirektionale Kommunikation zwischen der Policy und der Policy-Laufzeit (`+kwctl+` oder `+policy-server+`) zu implementieren. Die Verwendung des Kommunikationsprotokolls{short-project-name} wird xref:reference/spec/01-intro-spec.adoc[hier] beschrieben.

Es gibt spezielle Fälle, in denen das waPC-Projekt noch nicht verwendet werden kann.
Unter diesen Umständen können Sie eine Richtlinie unter Verwendung der von WASI bereitgestellten Schnittstellen erstellen.

[IMPORTANT]
====

{short-project-name} unterstützt WASI-Richtlinien ab der Version {short-project-name} 1.7.0.
====


== Beschränkungen

Sie sollten unter normalen Umständen keine WASI-Richtlinien verwenden, da sie bei der Auswertung schlechter abschneiden als waPC/Rego-Richtlinien.

[NOTE]
====

Eine bidirektionale Kommunikation zwischen der Richtlinie und dem Host kann erreicht werden, erfordert jedoch Änderungen innerhalb des Sprach-SDK. Dies ist erforderlich, um xref:/reference/spec/host-capabilities/01-intro-host-capabilities.adoc[Host-Fähigkeiten] zu nutzen undxref:/explanations/context-aware-policies.adoc[kontextabhängige] Richtlinien zu schreiben.

Derzeit werden sie nur von den {short-project-name} Go und JavaScript/TypeScript SDKs für WASI-Richtlinien freigegeben.

Wenn dies für Sie von Interesse ist, nehmen Sie bitte Kontakt auf. Wir können dann Prioritäten für die Bemühungen setzen.

====


== Anwendungsfälle

Der einzige Grund, eine "reine WASI"-Richtlinie zu schreiben, ist, wenn Sie den waPC-Kommunikationsmechanismus nicht verwenden können.

Derzeit (Stand Juni 2023) ist der einzige gute Grund, dies zu tun, die Verwendung des offiziellen Go-Compilers zur Erstellung eines WebAssembly-Moduls.

Seit der Version 1.21 ist der offizielle Go-Compiler in der Lage, WebAssembly-Module zu erzeugen, die auf die WASI-Schnittstelle abzielen.
Diese Module können jedoch noch keine Funktionen an die WebAssembly-Laufzeitumgebung exportieren.
Diese Einschränkung, die unter https://github.com/golang/go/issues/42372[] nachverfolgt werden kann, verhindert die Einführung des waPC-Protokolls.

Das {short-project-name} Projektteam empfiehlt, dass Sie {short-project-name} Go-Policies mit dem TinyGo-Compiler schreiben, wie xref:tutorials/writing-policies/go/01-intro-go.adoc[hier] beschrieben.

Bestimmte komplexe Go-Codebasen können jedoch nicht mit dem TinyGo-Compiler kompiliert werden.
Dazu gehören z. B. Codebasen wie https://github.com/google/cel-go[Kyverno] CEL-go oder https://github.com/kyverno/kyverno/[].
Unter diesen Umständen kann die Verwendung des offiziellen Go-Compilers helfen.

== Kommunikationsprotokoll

In diesem Abschnitt wird beschrieben, wie man eine einfache WASI-Richtlinie schreibt.

Sie müssen den Code wie ein normales CLI-Programm schreiben.
Das Programm muss die folgenden Unterbefehle enthalten:

* `+validate+`Dieser Befehl wird von der Policy Engine aufgerufen, um eine Zulassungsanfrage zu bewerten.
* `+validate-settings+`Dieser Befehl wird von der Policy Engine aufgerufen, um die Policy-Einstellungen zu validieren.

In beiden Fällen werden die zu prüfenden Daten über STDIN bereitgestellt.
Die Richtlinie muss die Antwort über STDOUT bereitstellen.
Sie können STDERR für Debug- oder Fehlermeldungen verwenden.

=== Validierung

Die Validierung einer Anfrage erfolgt beim Aufrufen des CLI-Programms für Richtlinien mit dem Unterbefehl `+validate+`.

STDIN muss ein JSON-Dokument enthalten, das ein `+ValidationRequest+` Objekt beschreibt.
Die Richtlinie muss ein JSON-Dokument, das ein `+ValidationResponse+` Objekt enthält, an STDOUT schreiben.

Die beiden Objekte `+ValidationRequest+` und `+ValidationResponse+` werden xref:reference/spec/03-validating-policies.adoc[hier] beschrieben.

=== Mutation

Mutating Policies funktionieren auf die gleiche Weise wie Validating Policies.
Das CLI-Programm für Richtlinien wird mit dem Unterbefehl `+validate+` aufgerufen.

STDIN muss ein JSON-Dokument enthalten, das ein `+ValidationRequest+` Objekt beschreibt.
Die Richtlinie muss ein JSON-Dokument, das ein `+ValidationResponse+` -Objekt enthält, an STDOUT schreiben.

Die beiden Objekte `+ValidationRequest+` und `+ValidationResponse+` werden xref:reference/spec/03-validating-policies.adoc[hier] beschrieben.

Wenn eine Mutation erforderlich ist, muss das Objekt `+ValidationResponse+` einen Schlüssel `+mutated_object+` haben, der das zu erstellende Objekt enthält.
Dieser Prozess wird xref:reference/spec/04-mutating-policies.adoc[hier] beschrieben.

== Context-aware

Zurzeit nur über das Go SDK unterstützt. Das Go SDK stellt die kontextabhängigen Fähigkeiten wie üblich zur Verfügung, weitere Informationen finden Sie xref:explanations/context-aware-policies.adoc[hier].

Ein Beispiel für eine kontextabhängige WASI Go-Richtlinie finden Sie unterhttps://github.com/kubewarden/go-wasi-context-aware-test-policy[go-wasi-context-aware-test-policy].

=== Validierung der Einstellungen

Die Richtlinie muss einen Unterbefehl namens `+validate-settings+` enthalten.
Dieser Befehl wird verwendet, um die vom Benutzer angegebenen Einstellungen zu überprüfen.

Das Programm muss auf STDIN ein JSON-Objekt empfangen, das die vom Benutzer angegebenen Einstellungen enthält.
Anschließend werden sie validiert und ein `+SettingsValidationResponse+` -Objekt auf STDOUT geschrieben.

Das Format der Website `+SettingsValidationResponse+` und der Prozess der Validierung der Einstellungen werden xref:reference/spec/02-settings.adoc[hier] beschrieben.

== Metadaten zur Politik

Jede {short-project-name} Richtlinie muss mit dem Befehl `+kwctl annotate+` kommentiert werden.
Die Richtlinien-Metadaten einer einfachen WASI-Richtlinie müssen diesen Wert haben:

[subs="+attributes",yaml]
----
executionMode: wasi
----
