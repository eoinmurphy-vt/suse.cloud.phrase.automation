Entwickler können Richtlinien erstellen, die zur Laufzeit Informationen aus einem Kubernetes-Cluster abrufen. Dies sind kontextbewusste Maßnahmen. Kontextabhängige Richtlinien können anhand von Informationen aus den im Cluster bereitgestellten Ressourcen feststellen, ob eine `+AdmissionRequest+` akzeptabel ist.

[NOTE]
====
Kontextabhängige Richtlinien sind nur in {project-name} Versionen ≥ v1.6.0 verfügbar.
====

Die Kontrolle darüber, auf welche Ressourcen eine Richtlinie im Cluster zugreifen kann, erfolgt über dashttps://kubernetes.io/docs/concepts/security/service-accounts/[Service Account] des Richtlinienservers. Ein Cluster-Administrator kontrolliert, worauf eine Richtlinie über Kubernetes RBAC-Regeln zugreifen kann. Kontextabhängige Richtlinien haben nur *Lesezugriff* auf die angeforderten Ressourcen.

Aus Sicherheitsgründen können nur `+ClusterAdmissionPolicy+` Richtlinien Informationen aus dem Kubernetes-Cluster abrufen. Dies liegt daran, dass nicht privilegierte Benutzer `+AdmissionPolicy+` Ressourcen bereitstellen können.

Wenn ein unprivilegierter Benutzer eine kontextbezogene Richtlinie als`+AdmissionPolicy+` einsetzt, wird das System:

- Blockiert alle Versuche, auf Kubernetes-Ressourcen zuzugreifen.
- Meldet sie an den Cluster-Administrator.

Standardmäßig blockiert {short-project-name} alle Cluster-Ressourcen. Ein {project-name} -Administrator legt fest, welche Kubernetes-Ressourcen jede kontextbewusste Richtlinie lesen kann.`+contextAwareResources+` dazu.

Im folgenden Beispiel wird eine Richtlinie umgesetzt, die den Zugriff auf die Ressourcen `+Deployment+`und `+Pod+` erfordert:

[subs="+attributes",yaml]
----
apiVersion: policies.kubewarden.io/v1
kind: ClusterAdmissionPolicy
metadata:
  name: context-aware-policy
  namespace: default
spec:
  policyServer: default
  module: "registry://ghcr.io/kubewarden/policies/context-aware-policy:v1.0.0"
  settings: {}
  contextAwareResources:
    - apiVersion: "apps/v1"
      kind: "deployment"
    - apiVersion: "v1"
      kind: "pod"
  rules:
    - apiGroups: ["apps"]
      apiVersions: ["v1"]
      resources: ["deployment"]
      operations:
        - CREATE
        - UPDATE
  mutating: false
----

Nach der Bereitstellung kann diese Richtlinie die Daten der Ressourcen `+deployment+` und `+pod+` lesen.

Richtlinienautoren stellen Listen von Kubernetes-Ressourcen für ihre kontextbewusste Richtlinie bereit. Richtlinienautoren tun dies, indem sie die Richtlinie mit Anmerkungen versehen. {project-name}Administratoren können die Metadaten der Richtlinie mit dem Befehl `+kwctl inspect+` einsehen. Sie können eine Liste der Ressourcen erhalten, auf die die Richtlinie Zugriff haben muss. Ein Administrator verwendet diese Liste, um die Definition von `+ClusterAdmissionPolicy+` zu füllen.

[WARNING]
====

Um Systemmissbrauch zu verhindern, müssen die Administratoren von {project-name} die Ressourcen überprüfen, auf die die Richtlinie zugreift.

Beispielsweise hätte eine Richtlinie, die Ingress-Objekte auswertet, gute Gründe, die im Cluster definierten `+Ingress+` Ressourcen zu lesen. Dieselbe Politik kann den Zugang zu den Ressourcen von `+Secret+` nicht rechtfertigen.

Die Richtlinien sollten den für ein korrektes Funktionieren erforderlichen Mindestzugang haben.

====


[NOTE]
====

Die Identifizierung von Kubernetes-Ressourcen erfolgt über `+apiVersion+` und `+kind+`.

Normalerweise ist `+apiVersion+` eine Zeichenkette im Format `+<api>/<version>+`. Ressourcen aus der `+core+` API-Gruppe (Pod, Service und andere) sollten den Gruppennamen nicht definieren, `+<api>+`. Sie sollten nur die `+<version>+` definieren (z. B. `+v1+`).

Bei einer Kernressource wird die erste nicht funktionieren, die zweite schon.

[subs="+attributes",console]
----
- apiVersion: "core/v1"
  kind: "pod"
----

[subs="+attributes",console]
----
- apiVersion: "v1"
  kind: "pod"
----

Alle anderen Kubernetes-Ressourcen benötigen die vollständige Definition: `+<api>/<version>+`.
====
