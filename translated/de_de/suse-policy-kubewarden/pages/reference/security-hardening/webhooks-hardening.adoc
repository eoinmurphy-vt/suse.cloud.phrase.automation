= Härtung der {short-project-name} Webhooks
:revdate: 2026-03-18
:page-revdate: {revdate}
:sidebar_label: Webhaken
:description: Konfigurieren Sie Netzwerkrichtlinien und Authentifizierung, um den Zugriff auf Webhooks in Ihrem Kubernetes-Cluster einzuschränken, insbesondere um Verbindungen von ungültigen Anrufern zu beschränken.
:keywords: kubewarden, kubernetes, sicherheit
:doc-persona: kubewarden-Bediener, kubewarden-Integrator
:doc-type: Referenz
:doc-topic: Referenz, Sicherheit

Der {project-name} Stack verwendet Webhooks zur Durchsetzung von Richtlinien in einem Kubernetes-Cluster.
Jede `+PolicyServer+` -Instanz stellt einen Webhook zur Verfügung, den der Kubernetes-API-Server aufruft, um Ressourcen zu validieren und zu ändern. Darüber hinaus stellt `+kubewarden-controller+`Webhooks zur Verfügung, um die vom Projekt{short-project-name} bereitgestellten benutzerdefinierten Ressourcen zu validieren und zu ändern.

Um die Angriffsfläche zu verringern, sollten Sie den Zugriff auf diese Webhooks auf die einzigen gültigen Anrufer beschränken, die sie haben:

* den Kubernetes-API-Server
* die Komponente xref:/explanations/audit-scanner/audit-scanner.adoc[Audit-Scanner].

Sie können dies mithilfe von Netzwerkrichtlinien und Authentifizierung unabhängig voneinander oder zusammen tun, um die Webhooks gegen Angriffe zu schützen.

== Externen Datenverkehr mithilfe von Netzwerkrichtlinien blockieren

Es wird erwartet, dass Webhooks nur Anfragen vom Kubernetes-API-Server und der Audit-Scanner-Komponente annehmen. Standardmäßig können Webhooks jedoch Datenverkehr aus jeder beliebigen Quelle annehmen. Wenn Sie eine Container-Netzwerkschnittstelle (CNI) verwenden, die Netzwerkrichtlinien unterstützt, können Sie eine Richtlinie erstellen, die Datenverkehr blockiert, der nicht vom API-Server stammt.

Die in Kubernetes integrierte NetworkPolicy-Ressource kann den Verkehr von den Cluster-Hosts weder blockieren noch zulassen. Außerdem läuft der Prozess `+kube-apiserver+` immer auf dem Host-Netzwerk. Daher müssen Sie die erweiterten Netzwerkrichtlinienressourcen des verwendeten CNI verwenden. Es folgen Beispiele für Calico und Cilium. Weitere Einzelheiten finden Sie in der Dokumentation zu Ihrem CNI.

=== Kattun

Verwenden Sie die Ressource NetworkPolicy in der Gruppe `+crd.projectcalico.org/v1+` API, um eine Netzwerkrichtlinie wie die folgende zu definieren:

[source,yaml]
----
apiVersion: crd.projectcalico.org/v1
kind: NetworkPolicy
metadata:
  name: allow-k8s-and-audit-scanner
  namespace: kubewarden
spec:
  selector: 'app.kubernetes.io/component in {"kubewarden-controller", "policy-server"}'
  types:
    - Ingress
  ingress:
    - action: Allow
      protocol: TCP
      source:
        nets:
        - 192.168.42.0/24
      destination:
        selector: 'app.kubernetes.io/component in {"kubewarden-controller", "policy-server"}'
    - action: Allow
      protocol: TCP
      source:
        namespaceSelector: 'kubernetes.io/metadata.name == "kubewarden"'
      destination:
        selector: 'app.kubernetes.io/component in {"kubewarden-controller", "policy-server"}'
----

[NOTE]
====

Diese Netzwerkrichtlinie verwendet Label-Selektoren, die in {short-project-name} 1.23.0 eingeführt wurden. Wenn Sie eine ältere Version verwenden, aktualisieren Sie die Bezeichnungen in der Richtlinie, damit sie mit Ihrer Bereitstellung übereinstimmen.

Genauer gesagt, schreiben Sie die

[source,yaml]
----
selector: 'app.kubernetes.io/component in {"kubewarden-controller", "policy-server"}'
----

Selektoren als:

[source,yaml]
----
selector: 'app.kubernetes.io/name == "kubewarden-controller" || has(kubewarden/policy-server)'
----
====

=== Cilium

Verwenden Sie die Ressource CiliumNetworkPolicy in der Gruppe `+cilium.io/v2+` API, um eine Netzwerkrichtlinie wie die folgende zu definieren:

[source,yaml]
----
apiVersion: "cilium.io/v2"
kind: CiliumNetworkPolicy
metadata:
  name: allow-k8s-and-audit-scanner
  namespace: kubewarden
spec:
  endpointSelector:
    matchExpressions:
      - key: app.kubernetes.io/component
        operator: In
        values:
          - policy-server
          - controller
  ingress:
    - fromEntities:
      - host
      - remote-node
    - fromEndpoints:
        - matchLabels:
            k8s:io.kubernetes.pod.namespace: kubewarden
----

[NOTE]
====

Diese Netzwerkrichtlinie verwendet Label-Selektoren, die in {short-project-name} 1.23.0 eingeführt wurden. Wenn Sie eine ältere Version verwenden, aktualisieren Sie die Bezeichnungen in der Richtlinie, damit sie mit Ihrer Bereitstellung übereinstimmen.

Genauer gesagt, schreiben Sie die

[source,yaml]
----
matchExpressions:
  - key: app.kubernetes.io/component
    operator: In
    values:
      - policy-server
      - controller
----

Ausdruck als:

[source,yaml]
----
endpointSelector:
matchExpressions:
  - key: app.kubernetes.io/name
    operator: In
    values:
      - kubewarden-controller
  - key: kubewarden/policy-server
    operator: Exists
----
====

== Erfordert den Kubernetes-API-Server zur Authentifizierung beim Webhook

[TIP]
====

Eine Schritt-für-Schritt-Anleitung zur Konfiguration des Kubernetes-API-Servers von K3s für die Authentifizierung gegenüber dem Webhook finden Sie in der xref:/howtos/security-hardening/webhook-mtls.adoc[Anleitung zu webook mTLS].

====

Die vom {short-project-name} Stack bereitgestellten Webhooks sollten nur Anfragen vom Kubernetes-API-Server oder von der Audit-Scanner-Komponente akzeptieren. Bei diesen Webhooks ist standardmäßig keine Authentifizierung der Kunden erforderlich. Sie akzeptieren jede Anfrage.

Sie können die Webhooks so konfigurieren, dass Anmeldeinformationen erforderlich sind, damit nur der API-Server und die Audit-Scanner-Prozesse auf sie zugreifen können. Weitere Informationen finden Sie in derhttps://kubernetes.io/docs/reference/access-authn-authz/extensible-admission-controllers/#authenticate-apiservers[Kubernetes-Dokumentation].

. Konfigurieren Sie den API-Server so, dass er dem Webhook ein Client-Zertifikat vorlegt, das auf eine `+AdmissionConfiguration+` -Datei zur Konfiguration der Plug-ins`+ValidatingAdmissionWebhook+` und `+MutatingAdmissionWebhook+` verweist:
+
Erstellen Sie eine Datei namens `+admission.yaml+` mit folgendem Inhalt:
+
[source,yaml]
----
apiVersion: apiserver.config.k8s.io/v1
kind: AdmissionConfiguration
plugins:
- name: ValidatingAdmissionWebhook
 configuration:
   apiVersion: apiserver.config.k8s.io/v1
   kind: WebhookAdmissionConfiguration
   kubeConfigFile: "/etc/k8s/admission/kubeconfig"
- name: MutatingAdmissionWebhook
 configuration:
   apiVersion: apiserver.config.k8s.io/v1
   kind: WebhookAdmissionConfiguration
   kubeConfigFile: "/etc/k8s/admission/kubeconfig"
----
+
Dies ist dieselbe Konfigurationsdatei, die auch für die Konfiguration anderer Plug-ins, wie z. B.`+PodSecurity+`, verwendet wird. Wenn Ihre Distribution oder Einrichtung zusätzliche Zulassungs-Plug-ins verwendet, sollten Sie auch diese konfigurieren.

. Erstellen Sie die Datei `+kubeconfig+`, auf die sich die Zulassungs-Plug-ins beziehen. {short-project-name} unterstützt nur die Authentifizierung mit Client-Zertifikaten, erzeugen Sie also ein TLS-Schlüsselpaar und stellen Sie kubeconfig so ein, dass entweder client-certificate und client-key oder client-certificate-data und client-key-data verwendet werden.
+
Zum Beispiel:
+
[source,yaml]
----
apiVersion: v1
kind: Config
users:
- name: '*.kubewarden.svc'
  user:
    client-certificate: /path/to/client/cert
    client-key: /path/to/client/key
----

. Starten Sie die Binärdatei `+kube-apiserver+` mit dem Flag`+--admission-control-config-file+`, das auf Ihre Datei `+AdmissionConfiguration+`verweist. Die Art und Weise, dies zu tun, variiert von Distribution zu Distribution und wird nicht überall unterstützt, z. B. bei gehosteten Kubernetes-Anbietern. Konsultieren Sie die Dokumentation zu Ihrer Kubernetes-Distribution.

. Machen Sie das Zertifikat der Stammzertifizierungsstelle, die das API-Server-Client-Zertifikat ausgestellt hat, für den {short-project-name} Stack verfügbar.
+
Legen Sie den Inhalt in ein `+ConfigMap+` unter dem Namensraum `+kubewarden+`mit einem Schlüssel namens `+client-ca.crt+`.
+
Unter der Annahme, dass die Root-CA unter `+/etc/k8s/admission/certs/rootCA.crt+` verfügbar ist, erstellen Sie die `+ConfigMap+` mit dem folgenden Befehl:
+
[source,console]
----
kubectl create configmap -n kubewarden api-server-mtls \
   --from-file=client-ca.crt=/etc/k8s/admission/certs/rootCA.crt
----

. Stellen Sie bei der Installation der `+kubewarden-controller+` Helm-Karte sicher, dass Sie die folgenden Werte aktivieren:
+
- Stellen Sie `+mTLS.enable+` auf `+true+` ein.
- Setzen Sie `+mTLS.configMapName+` auf den Namen der `+ConfigMap+`, die zuvor erstellt wurde.
+
Der Name von `+ConfigMap+` ist `+api-server-mtls+`, also lautet der Helm-Befehl zur Installation von `+kubewarden-controller+`:
+
[source,console]
----
helm install --wait -n kubewarden kubewarden-controller kubewarden/kubewarden-controller \
   --set mTLS.enable=true \
   --set mTLS.configMapName=api-server-mtls
----
+
Der {short-project-name} Controller erstellt ein Client-Zertifikat zur Verwendung durch die Audit-Scanner-Komponente. Das Zertifikat wird bei Bedarf automatisch vom Controller gedreht.
