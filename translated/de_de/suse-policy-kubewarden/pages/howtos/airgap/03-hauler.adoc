= Luftspaltinstallation mit Hauler
:revdate: 2026-03-18
:page-revdate: {revdate}
:description: Verwenden Sie Hauler zur Installation von {project-name} in luftdurchlässigen Umgebungen.
:keywords: ["{project-name}", kubewarden, kubernetes, luftspaltinstallation, hauler]
:doc-persona: kubewarden-Bediener, kubewarden-Integrator
:doc-type: howto
:doc-topic: Bediener-Handbuch, Luftspalt, Installation, Schlepper

Diese Anleitung zeigt Ihnen, wie Sie {project-name} in luftdurchlässigen Umgebungen mitlink:https://docs.hauler.dev/docs/intro[Hauler] installieren. Hauler ist ein Tool, das Benutzern hilft, Workloads in Air-Gapped-Umgebungen auszuführen. Es verlagert die erforderlichen Ressourcen für Anwendungen in diese luftdichten Umgebungen.

{short-project-name} bietet eine Manifestdatei mit allen erforderlichen Ressourcen, um {short-project-name} in einer privaten Umgebung auszuführen. In dieser Dokumentation wird beschrieben, wie Sie es verwenden können. In der Hauler-Dokumentation finden Sie weitere Informationen dazu.

Der grundlegende Arbeitsablauf unter Verwendung des Hauler-Manifests ist folgender:

. Holen Sie es sich von einer {short-project-name} Release-Seite
. Laden Sie alle für die Ausführung von {short-project-name} erforderlichen Ressourcen in einen lokalen Speicher.
. Exportieren Sie sie in eine Datei
. Verschieben Sie die Datei in die private Umgebung
. Laden Sie die Ressourcen in den Hauler, der in der privaten Umgebung läuft.
. Kopieren Sie alles in eine Registrierung, die bei der Installation in Ihrer abgeschirmten Umgebung verwendet werden soll

== Download des Hauler-Manifests von der Helm Chart Release Seite

Laden Sie `+hauler_manifest.yml+` von der {short-project-name}link:https://github.com/kubewarden/helm-charts/releases/[ Release-Seite] herunter.

Synchronisieren Sie die im Manifest definierten Ressourcen mit Ihrem Hauler-Speicher:

[source,shell]
----
hauler store sync --filename hauler_manifest.yaml
----

Der Hauler lädt alle Ressourcen aus dem Manifest in den lokalen Speicher herunter. Dieser Vorgang dauert ein paar Minuten. Nach der Fertigstellung können Sie die synchronisierten Ressourcen mit dem Befehl `+hauler store info+` anzeigen.

== Erzeugen Sie die Tarball-Datei mit allen {short-project-name} Ressourcen

Führen Sie den Befehl Hauler aus, um alle zuvor in den Speicher geladenen Ressourcen in eine Datei zu exportieren:

[source,shell]
----
hauler store save --filename kubewarden-resources.tar.zst
----

[NOTE]
====

{short-project-name} Container-Images unterstützen x86_64- und Arm-Architekturen. Wenn Sie die Ressource in der Datei speichern, können Sie daher Warnmeldungen wie die folgende sehen:

[source,shell]
----
2025-08-12 18:55:54 WRN specify an export platform to prevent potential platform mismatch on import of index [ghcr.io/kyverno/policy-reporter:3.3.3]
2025-08-12 18:55:54 WRN specify an export platform to prevent potential platform mismatch on import of index [ghcr.io/kyverno/policy-reporter-ui:2.4.1]
2025-08-12 18:55:54 WRN specify an export platform to prevent potential platform mismatch on import of index [ghcr.io/kubewarden/policy-server:v1.27.0]
2025-08-12 18:55:54 WRN specify an export platform to prevent potential platform mismatch on import of index [ghcr.io/kubewarden/audit-scanner:v1.27.0]
2025-08-12 18:55:54 WRN specify an export platform to prevent potential platform mismatch on import of index [ghcr.io/rancher/kuberlr-kubectl:v5.0.0]
2025-08-12 18:55:54 WRN specify an export platform to prevent potential platform mismatch on import of index [ghcr.io/kubewarden/kubewarden-controller:v1.27.0]
----

Um diese Warnmeldung zu vermeiden, können Sie das CLI-Flag `+--platform+` setzen, um festzulegen, welche Plattformarchitektur Sie in der Datei speichern möchten.

====

== Übertragen Sie den Tarball in Ihre isolierte Umgebung

Jetzt, wo Sie alle Ressourcen von {short-project-name} in`+kubewarden-resources.tar.zst+` haben, kopieren Sie sie in Ihre Umgebung und laden sie dort in den Hauler-Speicher:

[source,shell]
----
hauler store load --filename kubewarden-resources.tar.zst
# Check if the resources are loaded
hauler store info
----

Jetzt befinden sich alle für die Installation von {short-project-name} erforderlichen Ressourcen im Hauler-Speicher in Ihrer isolierten Umgebung.

== Privates Register befüllen

Um die Ressourcen aus Ihrem Hauler-Lager zu nutzen, müssen Sie sie in einer internen Registrierung verfügbar machen. Sie können Hauler-Befehle verwenden, um sie in Ihre private Registry zu kopieren.

[source,shell]
----
hauler store copy registry://localhost:5000
----

Sie können auch Hauler ausführen, um eine Registry mit allen Ressourcen aus dem Store zu starten. Diese Registrierung ist unsicher, und Sie müssen die Clusterkonfiguration anpassen:

[source,shell]
----
# Find IP address of your host
# hostname -I

# Update registries.yaml for k3s based cluster to allow insecure access
# mirrors:
#   "<HOST_IP>:5000":
#     endpoint:
#       - "http://<HOST_IP>:5000"
# configs:
#   "<HOST_IP>:5000":
#     tls:
#       insecure_skip_verify: true

# Configure policy-server to allow pulling policies from insecure sources
# helm install .. kubewarden-defaults .. --set policyServer.insecureSources[0]=<HOST_IP:5000>

hauler store serve registry
----

Dadurch wird eine Registrierung unter der Adresse `+localhost:5000+` gestartet. Von diesem Punkt aus können Sie andere Befehle wie Skopeo verwenden, um alle von {short-project-name} verwendeten Container-Images, Richtlinienmodule und Helm-Diagramme in Ihre private Registry zu kopieren.

== Installieren Sie {short-project-name}

Nun, da Ihre private Registrierung alles Notwendige enthält, können Sie{short-project-name} installieren. Der Unterschied zu einer Standardinstallation von {short-project-name}besteht darin, dass Sie die Registrierung in den Container-Images und Richtlinien so ändern müssen, dass sie die private Registrierung ist. Außerdem müssen Sie die Helm-Diagramme aus OCI-Artefakten installieren.

Installieren Sie den {short-project-name} Stack:

[source,shell]
----
helm install --wait -n kubewarden kubewarden-crds --create-namespace \
    oci://<REGISTRY.YOURDOMAIN.COM:PORT>/hauler/kubewarden-crds
helm install --wait -n kubewarden kubewarden-controller \
    --set "global.cattle.systemDefaultRegistry=<REGISTRY.YOURDOMAIN.COM:PORT>" \
    oci://<REGISTRY.YOURDOMAIN.COM:PORT>/hauler/kubewarden-controller
----

[CAUTION]
====

Um die PolicyReporter-Subkarte, die in der Karte `+kubewarden-controller+`verfügbar ist, zu verwenden, müssen Sie andere Werte definieren, die für die Subkarte in einer Air-Gapped-Umgebung spezifisch sind. Siehe ein Beispiel unten:

[source,shell]
----
helm install --wait -n kubewarden kubewarden-controller oci://<REGISTRY.YOURDOMAIN.COM:PORT>/hauler/kubewarden-controller \
    --set global.cattle.systemDefaultRegistry=<REGISTRY.YOURDOMAIN.COM:PORT> \
    --set auditScanner.policyReporter=true \
    --set policy-reporter.image.registry=<REGISTRY.YOURDOMAIN.COM:PORT> \
    --set policy-reporter.image.repository=kyverno/policy-reporter \
    --set policy-reporter.ui.image.registry=<REGISTRY.YOURDOMAIN.COM:PORT> \
    --set policy-reporter.ui.image.repository=kyverno/policy-reporter-ui
----

Es ist notwendig, `+auditScanner.policyReporter+` und vier weitere Werte zu definieren, um das Unterdiagramm zu aktivieren und die Registrierung und das Repository mit dem Speicherort des Policy Reporter-Images zu konfigurieren. Weitere Informationen zu den Werten der Unterdiagramme des Richtlinienberichts finden Sie unter \link:https://kyverno.github.io/policy-reporter-docs/getting-started/helm.html[Policy Reporter documentation].

====

[source,shell]
----
helm install --wait -n kubewarden \
  kubewarden-defaults oci://<REGISTRY.YOURDOMAIN.COM:PORT>/hauler/kubewarden-defaults \
  --set global.cattle.systemDefaultRegistry=<REGISTRY.YOURDOMAIN.COM:PORT>
----

Konfigurieren Sie schließlich jeden Richtlinienserver so, dass er Richtlinien aus Ihrer privaten Registrierung abruft. Siehe den Abschnitt xref:/howtos/policy-servers/02-private-registry.adoc[über die Verwendung der privaten Registry] in der Dokumentation.

Erstellen Sie nun {short-project-name} Richtlinien in Ihrem Cluster. Die Policen müssen in Ihrer privaten Registratur verfügbar sein.

[source,yaml]
----
kubectl apply -f - <<EOF
apiVersion: policies.kubewarden.io/v1
kind: ClusterAdmissionPolicy
metadata:
  name: privileged-pods
spec:
  module: registry://<REGISTRY.YOURDOMAIN.COM:PORT>/kubewarden/policies/pod-privileged:v0.2.2
  rules:
  - apiGroups: [""]
    apiVersions: ["v1"]
    resources: ["pods"]
    operations:
    - CREATE
  mutating: false
EOF
----

[CAUTION]
====

`+PolicyServer+` Ressourcen müssen das in Ihrer privaten Registratur verfügbare Bild verwenden.
Zum Beispiel:

[source,yaml]
----
apiVersion: policies.kubewarden.io/v1
kind: PolicyServer
metadata:
  name: reserved-instance-for-tenant-a
spec:
  image: <REGISTRY.YOURDOMAIN.COM:PORT>/kubewarden/policy-server:v1.3.0
  replicas:
----
