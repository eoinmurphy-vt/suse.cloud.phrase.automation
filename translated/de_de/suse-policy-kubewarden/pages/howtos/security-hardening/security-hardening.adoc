{project-name} ist bestrebt, mit wenig Konfiguration sicher zu sein. In diesem Abschnitt und seinen Unterseiten finden Sie Härtungstipps (mit den entsprechenden Kompromissen), um {short-project-name} selbst zu sichern.

Weitere Informationen entnehmen Sie bitte dem xref:/reference/threat-model.adoc[Bedrohungsmodell].

== `+kubewarden-defaults+` Steuerkarte

Die Betreiber können eine sichere Bereitstellung erhalten, indem sie alle {short-project-name} Helm-Karten installieren. Es wird empfohlen, die `+kubewarden-defaults+` Helm-Karte zu installieren und die empfohlenen Richtlinien mit zu aktivieren:

[source,console]
----
helm install --wait -n kubewarden kubewarden-defaults kubewarden/kubewarden-defaults \
  --set recommendedPolicies.enabled=True \
  --set recommendedPolicies.defaultPolicyMode=protect
----

Dies stellt einen Standard-PolicyServer und Standardrichtlinien im Schutzmodus bereit, um sicherzustellen, dass der {short-project-name} Stack vor anderen Arbeitslasten geschützt ist.

== RBAC

{short-project-name} beschreibt RBAC-Konfigurationen in verschiedenen _Erklärungsabschnitten_. Die Benutzer können die erforderlichen Berechtigungen für diexref:/explanations/audit-scanner/audit-scanner.adoc#_permissions_and_serviceaccounts[Audit-Scanner-Funktion] sowie xref:/explanations/context-aware-policies.adoc[für die] kontextabhängige Funktion xref:/explanations/context-aware-policies.adoc[pro Policy Server] Service-Konto feinabstimmen.

Um alle Rollen anzuzeigen:

[source,console]
----
kubectl get clusterroles,roles -A | grep kubewarden
----

== Berechtigungen pro Richtlinie

Für kontextabhängige Richtlinien legen die Betreiber unter `+spec.contextAwareResources+` fein abgestufte Berechtigungen pro Richtlinie fest, die in Verbindung mit dem für den Richtlinienserver, auf dem die Richtlinie ausgeführt wird, konfigurierten Dienstkonto funktionieren.

== Abdeckung der Arbeitslast

Standardmäßig schließt {short-project-name} bestimmte Namespaces von der Abdeckung durch{short-project-name} aus. Dies dient der Vereinfachung der erstmaligen Nutzung und der Interoperabilität mit anderen Workloads.

Sicherheitsbewusste Betreiber können diese Namespace-Liste über den Wert`+.global.skipNamespaces+` sowohl für die `+kubewarden-controller+` als auch für die`+kubewarden-defaults+` Helm-Diagramme anpassen.

== Pod Security Zulassung

Ab Version 1.23 kann der Stack von {short-project-name} in einem Namespace ausgeführt werden, in dem diehttps://kubernetes.io/docs/concepts/security/pod-security-standards/#restricted[`+restricted+`` Pod Security Standards] mit den aktuellen Best Practices zur Pod-Härtung vorhanden sind.

Dazu müssen Sie das Label `pod-security.kubernetes.io/enforce: restricted` zum Namespace der Bereitstellung {short-project-name} hinzufügen.

```console
kubectl label namespace kubewarden pod-security.kubernetes.io/enforce=restricted --overwrite
```

Weitere Einzelheiten finden Sie in derhttps://kubernetes.io/docs/concepts/security/pod-security-admission/[offizielle Dokumentation] von Kubernetes' Pod Security Admission.

=== SecurityContexts

Das `+kubewarden-controller+` Helm-Diagramm konfiguriert die SecurityContexts und stellt sie in seinem `+values.yaml+` dar.

Das `+kubewarden-defaults+` Helm-Diagramm ermöglicht die Konfiguration des Standard-Policy-Servers `+.spec.securityContexts+` unter `+.Values.policyServer.securityContexts+`.

Die von Betreibern verwalteten Policy Server können Sie über ihrexref:/reference/CRDs.adoc#_policyserversecurity[`+spec.securityContexts+`].
