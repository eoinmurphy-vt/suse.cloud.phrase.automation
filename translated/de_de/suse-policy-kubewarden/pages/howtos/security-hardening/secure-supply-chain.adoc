Eine sichere Infrastruktur für die Lieferkette kann die Gültigkeit ihrer Teile bzw. Verbindungen überprüfen. Es ermöglicht Benutzern und Entwicklern, die Überwachungskette ihrer Softwarekomponenten oder Artefakte zu zeigen. Es ist ein aktiver Ansatz zur Abschwächung von Sicherheitsproblemen.

Das Projekt https://sigstore.dev/[Sigstore] bietet hierfür Werkzeuge und Infrastruktur. Sie dient der Überprüfung der Integrität der Lieferkette von Artefakten.

{short-project-name} verwendet https://github.com/sigstore/cosign[`+cosign+`] zusammen mit der vom Sigstore-Projekt angebotenen Infrastruktur https://github.com/SigStore/fulcio[fulcio] undhttps://github.com/sigstore/rekor[rekor].

Cluster-Betreiber können {project-name} so konfigurieren, dass nur Richtlinien ausgeführt werden, die von vertrauenswürdigen Stellen signiert wurden. Richtlinienentwickler können ihre Richtlinien signieren und sie in einem Register veröffentlichen.

== Voraussetzungen

In den folgenden Abschnitten müssen Sie ein paar Tools installieren. Damit können Benutzer die Signaturen von OCI-Artefakten unterschreiben und überprüfen. Die Beispiele zeigen die Verwendung vonhttps://docs.sigstore.dev/quickstart/quickstart-cosign/[`+cosign+`] undhttps://github.com/kubewarden/kwctl[`+kwctl+`] zum Signieren und Prüfen von Richtlinien.

Um GitHub zum Signieren von Richtlinien zu verwenden, müssen Siehttps://docs.sigstore.dev/cosign/system_config/installation/#github-actions[GitHub-Aktionen] installieren.

Das schlüssellose Signieren verwendet die Standardinstanzen https://github.com/SigStore/fulcio[fulcio] undhttps://github.com/sigstore/rekor[rekor], die vom Sigstore-Projekt bereitgestellt werden. Lesen Sie in der Sigstore-Dokumentation nach, wie Sie bei Bedarf Ihre eigene Infrastruktur dafür nutzen können.

== Unterzeichnungsrichtlinien

{short-project-name} empfiehlt die Verwendung des Dienstprogrammshttps://github.com/sigstore/cosign[Mitunterzeichnung] von Sigstore zur von Policen. Dieser Abschnitt zeigt eine schlüsselbasierte Methode zum Signieren von Richtlinien. Dazu müssen Sie ein privat-öffentliches Schlüsselpaar erzeugen. Anhand der generierten Schlüssel lässt sich überprüfen, ob die signierten Artefakte vom erwarteten Benutzer stammen. Um dieses Schlüsselpaar zu erzeugen, verwenden Sie den Befehl `+cosign generate-key-pair+`:

[subs="+attributes",bash]
----
cosign generate-key-pair
----

Daraufhin werden Sie aufgefordert, ein Passwort einzugeben und zu bestätigen:

[subs="+attributes",console]
----
Enter password for private key: ●●●●●●●●
Enter password for private key again: ●●●●●●●●
Private key written to cosign.key
Public key written to cosign.pub
----

Jetzt können Sie diesen Schlüssel zum Unterzeichnen von Policen verwenden.

Geben Sie die Datei mit dem privaten Schlüssel nicht weiter, `+cosign.key+`. Dies ist eine geheime Datei, die nur vom Eigentümer des Schlüssels zum Signieren von Richtlinien verwendet werden darf.

Um eine Richtlinie zu signieren, können Sie `+cosign sign+` verwenden und das Befehlszeilenargument `+--key+` mit Ihrer privaten Schlüsseldatei übergeben:

[subs="+attributes",bash]
----
cosign sign --key cosign.key ghcr.io/kubewarden/policies/user-group-psp:latest
----

Das Ergebnis ist eine Aufforderung zur Eingabe des Passworts für den angegebenen privaten Schlüssel:

[subs="+attributes",console]
----
an error occurred: no provider found for that key reference, will try to load key from disk...
Enter password for private key: ●●●●●●●●
Pushing signature to: ghcr.io/kubewarden/policies/user-group-psp
----

Dieser Befehl signiert die Richtlinie, indem er ein neues Signaturobjekt erstellt. Das Signaturobjekt wird dann zusammen mit der Richtlinie in die Registrierung hochgeladen. Jetzt ist die Richtlinie bereit für die Verwendung in einer {short-project-name} -Installation mit Signaturprüfung.

Dieselbe Richtlinie kann mehrfach signiert werden, von demselben oder von verschiedenen Benutzern. Diese Signaturen werden dem Signaturobjekt zusammen mit der ursprünglichen Signatur hinzugefügt.

Weitere Informationen zur Funktionsweise des Signiervorgangs finden Sie in derhttps://docs.sigstore.dev/[Projektdokumentation].

=== Schlüssellose Unterzeichnung

Häufig werden Richtlinien automatisch über CI/CD-Pipelines erstellt. Dies erschwert den Prozess der Schlüsselerzeugung. Dieser schlüssellose Arbeitsablauf von Sigstore ist für diese Situationen gedacht. Anstelle von langlebigen singenden Schlüsseln verwendet der schlüssellose Arbeitsablauf Zertifizierungsstellen (CAs) und Zertifikatsketten.

Sie binden den generierten, kurzlebigen Zertifikatsschlüssel in eine Vertrauenskette ein. Dies geschieht durch eine Identitätsprüfung, um die Identität des Unterzeichners zu bestätigen. Die Lebensdauer des Zertifikatsschlüssels ist lang genug, um die Signierung zu ermöglichen. Die Identitätsprüfung erfolgt durch Authentifizierung gegenüber einem OpenID Connect (OIDC)-Anbieter. Die öffentliche Fulcio-Infrastruktur von Sigstore bildet die Vertrauenskette.

Zum Signieren wird das Sigstore-Dienstprogramm für Mitsignieren verwendet.

[subs="+attributes",bash]
----
$ cosign sign ghcr.io/kubewarden/policies/user-group-psp:latest
----

=== Wie man Artefakte in GitHub-Workflows signiert

Bei der Verwendung des schlüssellosen Signierens in einer GitHub-Aktion muss sich der Benutzer auf `+cosign+` nicht bei einem OIDC-Anbieter anmelden. Ein GitHub-Token ist während der Ausführung des GitHub-Workflows verfügbar. Er wird zur Authentifizierung des Benutzers und zur Erzeugung der ephemeren Schlüssel verwendet. Der Signiervorgang ist der gleiche wie im schlüssellosen Modus. Dies ist ein Beispiel dafür, wie das Projekt {short-project-name} seine Richtlinien unterzeichnet:

.YAML zur Beschreibung von {short-project-name} policy signing
[%collapsible]
======

[,yaml]
----
# ... beginning of the workflow file ...
jobs:
  build:
    name: Build container image
    runs-on: ubuntu-latest
    steps:
      # ... other steps building the container image ...
      -
      name: Login to GitHub Container Registry
      uses: docker/login-action@v1
      with:
        registry: ghcr.io
        username: ${{ github.repository_owner }}
        password: ${{ inputs.GITHUB_TOKEN }}
      -
      name: Publish Wasm policy artifact to OCI registry with the 'latest' tag
      shell: bash
      if: ${{ startsWith(github.ref, 'refs/heads/') }}
      env:
        COSIGN_EXPERIMENTAL: 1
      run: |
        set -ex
        echo Pushing policy to OCI container registry
        IMMUTABLE_REF=$(kwctl push -o json ${{ PATH_TO_BUILT_WASM_FILE }} ghcr.io/myorg/policies/my-great-policy:latest | jq -r .immutable_ref)
        echo Keyless signing of policy using cosign
        cosign sign ${IMMUTABLE_REF}
      # ... other build steps ...

# ... remainder of the workflow file ...
----

== Überprüfung von Richtlinien

Ob eine Police korrekt signiert ist, können Sie mit `+cosign+` oder `+kwctl+` überprüfen. Sie verfügen über ähnliche Befehlszeilenoptionen zur Überprüfung von Richtliniensignaturen. Um die Binärsignatur mit einem Schlüssel zu prüfen, verwenden Sie `+kwctl+` wie folgt:

[subs="+attributes",console]
----
$ kwctl verify -k cosign.pub ghcr.io/kubewarden/policies/user-group-psp:latest
2022-03-29T14:49:31.878180Z  INFO kwctl::verify: Policy successfully verified
----

Oder `+cosign+`:

[subs="+attributes",console]
----
$ cosign verify --key cosign.pub ghcr.io/kubewarden/policies/user-group-psp:latest

Verification for ghcr.io/kubewarden/policies/user-group-psp:latest --
The following checks were performed on each of these signatures:
  - The cosign claims were validated
  - The signatures were verified against the specified public key
  - Any certificates were verified against the Fulcio roots.

== Konfigurieren des Richtlinienservers zur Prüfung von Richtliniensignaturen

Sie können {short-project-name} mit einer `+ConfigMap+` so konfigurieren, dass nur vertrauenswürdige Richtlinien ausgeführt werden. Die Struktur von `+ConfigMap+` ist in derxref:reference/verification-config.adoc#_signature_configuration_reference[Signaturkonfigurationsreferenz] beschrieben. Es wird verwendet, um eine Richtlinie mit `+kwctl+` zu überprüfen. Die`+ConfigMap+` sollte unter dem Feld`+verification-config+` die zulässigen Konfigurationen festlegen.

Sie möchten beispielsweise Richtlinien ausführen, die von der Organisation {short-project-name} GitHub signiert wurden. Ein Beispiel `+ConfigMap+` für dieses Szenario wäre dann:

.$ cat kubewarden_signatures.yaml
[%collapsible]
======

[subs="+attributes",console]
----
$ cat kubewarden_signatures.yaml
apiVersion: v1
allOf:
  - kind: githubAction
    owner: kubewarden

# note that the data is stored under verification-config field
$ kubectl  create configmap my-signatures-configuration --from-file=verification-config=kubewarden_signatures.yaml

$ kubectl get configmap -o yaml my-signatures-configuration
apiVersion: v1
data:
  verification-config: |
    apiVersion: v1
    allOf:
      - kind: githubAction

== Referenz der Signaturkonfiguration

Sie können die in einer Datei enthaltenen Signaturanforderungen validieren. Hier ist ein Beispiel:

.Eine Datei mit Unterschriftsanforderungen
[%collapsible]
======
[subs="+attributes",yaml]
----
apiVersion: v1

allOf: <1>
  - kind: githubAction
    owner: kubewarden   # mandatory
    annotations:
      env: prod

anyOf: # at least `+anyOf.minimumMatches+` are required to match <2>
  minimumMatches: 2 # default is 1
  signatures:
  - kind: pubKey
    owner: flavio # optional
    key: .... # mandatory
    annotations:  # optional
      env: prod
      foo: bar
  - kind: pubKey
    owner: victor # optional
    key: .... # mandatory
  - kind: genericIssuer
    issuer: https://github.com/login/oauth
    subject:
      equal: alice@example.com
  - kind: genericIssuer
    issuer: https://token.actions.githubusercontent.com
    subject:
      equal: https://github.com/flavio/policy-secure-pod-images/.github/workflows/release.yml@refs/heads/main
  - kind: genericIssuer
    issuer: https://token.actions.githubusercontent.com
    subject:
      urlPrefix: https://github.com/flavio/
  - kind: genericIssuer
    issuer: https://token.actions.githubusercontent.com
    subject:
      urlPrefix: https://github.com/kubewarden # <- it will be post-fixed with `+/+` for security reasons
  - kind: githubAction
    owner: flavio   # mandatory
    repo: policy1 # optional
  - kind: pubKey
    owner: alice # optional
    key: .... # mandatory
----
<1> `+allOf+`
<2> `+anyOf+`
======

=== Validierung der Unterschrift

Die obige Konfiguration enthält die beiden hervorgehobenen Abschnitte, `+allOf+` und`+anyOf+`:

* `+allOf+`: Der Richtlinie wird nur dann vertraut, wenn alle hier aufgeführten Signaturanforderungen erfüllt sind.
* `+anyOf+`: Die Richtlinie ist vertrauenswürdig, wenn das Kriterium `+minimumMatches+` erfüllt ist.

Oben ist das Feld `+minimumMatches+` 2. Es müssen also mindestens zwei der Unterschriftenanforderungen erfüllt sein. Der Standardwert für das Feld `+minimumMatches+` ist `+1+`.

Alle erforderlichen Unterschriften von `+allOf+` *und* die Mindestanzahl von`+anyOf+` müssen erfüllt sein.

=== Validierung des öffentlichen Schlüssels

Um zu prüfen, ob eine Richtlinie mit dem richtigen öffentlichen Schlüssel signiert ist, geben Sie die Schlüsseldaten und den Eigentümer des Schlüssels an. In diesem Beispiel setzen Sie `+kind+`, auf `+pubKey+` und `+key+` hat den öffentlichen Schlüssel. Das Eigentümerfeld ist optional, kann aber nützlich sein, um zu klären, wer der Eigentümer des Schlüssels ist.

[subs="+attributes",yaml]
----
  - kind: pubKey
    owner: bob # optional
    key: |
      -----BEGIN PUBLIC KEY-----
      MBFKHFDGHKIJH0CAQYIKoZIzj0DAQcDQgAEX0HFTtCfTtPmkx5p1RbDE6HJSGAVD
      BVDF6SKFSF87AASUspkQsN3FO4iyWodCy5j3o0CdIJD/KJHDJFHDFIu6sA==
      -----END PUBLIC KEY-----
----

=== Schlüssellose Signaturvalidierung

Eine im Keyless-Modus signierte Richtlinie hat keinen öffentlichen Schlüssel, der verifiziert werden muss. Sie können die Richtlinie weiterhin mit den OIDC-Daten überprüfen, die während des Signiervorgangs verwendet wurden. Dazu ist es notwendig, die Signaturvalidierung als `+genericIssuer+` zu definieren.

Es ist möglich, Informationen anhand der Unterschrift zu überprüfen:

* `+issuer+`(obligatorisch): Dies entspricht dem Attribut `+Issuer+` in dem von Fulcio erstellten Zertifikat. Hier wird der OIDC angezeigt, der zur Unterzeichnung der Police verwendet wird.
* `+subject+`: Feld, das verwendet wird, um das Attribut `+Subject+` im Zertifikat von Fulcio abzugleichen. Das Feld `+Subject+` (Fulcio) enthält den Benutzer, der für die Authentifizierung gegenüber dem OIDC-Anbieter verwendet wird. Das Verifizierungsfeld, `+subject+`, kann eines von zwei Unterfeldern haben:
 ** `+equal+`: Die `+Subject+` (Fulcio) aus dem Zertifikat muss mit dem Wert in der Signaturvalidierung übereinstimmen;
 ** `+urlPrefix+`: Dem Wert des Feldes `+Subject+` (Fulcio) des Zertifikats muss der in der Signaturvalidierung definierte Wert vorangestellt werden.

[NOTE]
====

Sowohl auf `+cosign verify+` als auch auf `+kwctl inspect+` können Informationen über schlüssellose Signaturen angezeigt werden.

====


Diese Konfiguration bedeutet zum Beispiel, dass die Richtlinie eine schlüssellose Signatur von Alice unter Verwendung des GitHub OIDC haben muss:

[subs="+attributes",yaml]
----
- kind: genericIssuer
  issuer: https://github.com/login/oauth
  subject:
    equal: alice@example.com
----

Bei dieser Konfiguration muss die Richtlinie in GitHub-Aktionen signiert werden, und zwar von einem Repository, das dem GitHub-Benutzer `+flavio+` gehört:

[subs="+attributes",yaml]
----
- kind: genericIssuer
  issuer: https://token.actions.githubusercontent.com
  subject:
    urlPrefix: https://github.com/flavio
----

=== GitHub-Aktionen Signaturprüfung

Die "Art" `+githubAction+` dient dazu, in GitHub-Aktionen signierte Richtlinien zu validieren.
Sie können dies auch mit der Art `+genericIssuer+` tun. Um den Prozess der Unterschriftsanforderung zu vereinfachen, verwenden Sie zwei zusätzliche Felder für `+githubAction+`:

* `+owner+` (obligatorisch): GitHub-ID des Benutzers oder der Organisation, dem/der vertraut werden soll
* `+repo+`: der Name des zu vertrauenden Repository

Zum Beispiel könnte der letzte Ausschnitt unter Verwendung von `+genericIssuer+` wie folgt umgeschrieben werden:

[subs="+attributes",yaml]
----
- kind: githubAction
  owner: flavio
----

=== Validierung von Unterschriftenvermerken

Alle Signaturtypen können weitere optionale Validierungsfelder haben, `+annotations+`.
Bei diesen Feldern handelt es sich um Schlüssel/Wert-Daten, die während des Signiervorgangs hinzugefügt werden.

Mit {short-project-name} können Sie überprüfen, ob die Richtliniensignatur von vertrauenswürdigen Benutzern stammt *und* ob diese bestimmte Anmerkungen haben.

Bei der nächsten Validierung werden 2 Bedingungen für die Police geprüft:

* dass sie mit einem bestimmten Schlüssel signiert ist
* er hat eine Anmerkung zur Produktionsumgebung

[subs="+attributes",yaml]
----
- kind: pubKey
  key: |
    -----BEGIN PUBLIC KEY-----
    MBFKHFDGHKIJH0CAQYIKoZIzj0DAQcDQgAEX0HFTtCfTtPmkx5p1RbDE6HJSGAVD
    BVDF6SKFSF87AASUspkQsN3FO4iyWodCy5j3o0CdIJD/KJHDJFHDFIu6sA==
    -----END PUBLIC KEY-----
  annotations:
    environment: production
----

=== Verwendung einer Signaturprüfungs-Konfigurationsdatei zur Prüfung eines OCI-Artefakts für Richtlinien

Sie können testen, ob eine Richtlinie die Überprüfung besteht, indem Sie die Konfigurationsdatei zur Überprüfung verwenden.
Verwenden Sie das Flag `+--verification-config-path+` des Befehls `+kwctl verify+` 

[subs="+attributes",console]
----
$ cat signatures_requirements.yaml
apiVersion: v1
allOf:
  - kind: pubKey
    key: |
      -----BEGIN PUBLIC KEY-----
      MFkwEwYHKoZIzj0CAQYIKoZIzj0DAQcDQgAE5Q+cN1Jj2S7N05J4AXnqwP2DyzSg
      Mc+raYce2Wthrd30MSgFtoh5ADAkCd/nML2Nx8UD9KBuASRb0gG5jXqgMQ==
      -----END PUBLIC KEY-----

$ kwctl verify --verification-config-path signatures_requirements.yaml ghcr.io/kubewarden/policies/user-group-psp:latest
2022-03-29T17:34:37.847169Z  INFO kwctl::verify: Policy successfully verified
----

  - kind: githubAction
    owner: kubewarden

$ kwctl verify --verification-config-path kubewarden_signatures.yaml ghcr.io/kubewarden/policies/user-group-psp:latest
