Vielen Dank, dass Sie sich die Zeit nehmen, zu den Rancher CAPI-Erweiterung https://github.com/rancher?q=turtles&type=all&language=&sort=[Projekten der] beizutragen.

Verbesserungen in allen Bereichen des Projekts, vom Code bis zur Dokumentation, von Fehlerberichten bis zum Design von Funktionen und der Verbesserung der Benutzeroberfläche, sind dankbar willkommen.
Dieser Leitfaden sollte alle Aspekte der Interaktion mit dem Projekt und der möglichst reibungslosen Einbindung in die Entwicklung abdecken.

Das Lesen von Dokumenten ist oft mühsam, daher sollten wir die wichtigsten Regeln, die dazu beitragen, gleich an den Anfang stellen: *Sei immer freundlich!*

Ich freue mich darauf, Ihre Beiträge im Repo zu sehen!

== Wie kann man sich beteiligen?

Wir würden gerne Ihre Patches in so ziemlich allen Bereichen der Projektentwicklung annehmen!

Wenn Sie neu bei dem Projekt sind und helfen möchten, aber nicht wissen, wo Sie anfangen sollen, finden Sie hier eine nicht erschöpfende Liste von Möglichkeiten, wie Sie mithelfen können:

. Einen <<_submitting_prs,Pull Request>> einreichen
+
Neben der Behebung von Fehlern und dem Einreichen neuer Funktionen gibt es noch andere Dinge, die Sie einreichen können, die zwar weniger auffällig sind, aber von allen, die mit der Codebasis arbeiten, sehr geschätzt werden:

 ** Ausweitung der Testabdeckung!
 ** Refactoring!
ifeval::["{build-type}" == "product"]
 ** Überprüfen und Aktualisieren der xref:index.adoc[Dokumentation]!
endif::[]
ifeval::["{build-type}" == "community"]
 ** Überprüfen und Aktualisieren der xref:index.adoc[Dokumentation]!
endif::[]
 ** Hinzufügen einer neuen UI-Funktionalität!

+
(Siehe auch <<_choosing_something_to_work_on,Auswahl einer Aufgabe>> weiter unten.)

. Ein <<_opening_issues,Thema>> eröffnen
+
Wir haben 2 Arten von Problemen: Fehlerberichte und Funktionsanfragen. Wenn Sie nicht sicher sind, welche Kategorie Sie benötigen, schätzen Sie einfach und geben Sie so viele Informationen wie möglich an.

. Interessiert daran, zur Verbesserung beizutragen:

* Rancher CAPI-Erweiterung Backend? Informieren Sie sich über https://github.com/rancher/turtles/issues?q=is%3Aopen+ist%3Ausgabe+Label%3Art%2Fbug+[`+bugs+`] oder https://github.com/rancher/turtles/labels/help-wanted[`+help wanted+` Ausgaben].
  Wenn Sie eine größere Herausforderung oder einen erfahrenen Mitarbeiter suchen, besuchen Sie https://github.com/rancher/turtles/issues?q=is%3Aopen+is%3Aissue+label%3Akind%2Ffeature[`+feature requests+`].
* Erweiterung UI? Werfen Sie einen Blick auf https://github.com/rancher/capi-ui-extension[`+open+`] oder https://github.com/rancher/capi-ui-extension/labels/help-wanted[`+help wanted+` aus].
* vielleicht, Benutzerdokumente? Starten Sie dann direkt https://github.com/rancher/turtles-product-docs/issues[`+open+` im Docs Repository.] 

== Eröffnungsthemen

Diese Leitfäden sollen Ihnen dabei helfen, Ihre Anfragen so zu verfassen, dass sie so schnell wie möglich bearbeitet werden.

_Siehe unten, um zu erfahren <<_prioritizing_issues,, wie die Probleme priorisiert werden>>_.

*Allgemeine Regeln*:

. Bevor Sie etwas öffnen, sollten Sie sich die vorhandenen Ausgaben genau ansehen.
. Mehr ist mehr: Geben Sie so viele Informationen, wie es möglich ist.
  Bei sehr detaillierten Problemen ist es wahrscheinlicher, dass sie aufgegriffen werden, da sie schneller nach Prioritäten geordnet und für die Bearbeitung eingeplant werden können. Außerdem sind sie für die Gemeinschaft leichter zugänglich, d. h. Sie müssen nicht darauf warten, dass sich das Kernteam damit befasst.
. Bitte eröffnen Sie kein Problem mit einer Beschreibung, die *nur* ein Link zu einem anderen Problem, ein Link zu einer Slack-Konversation, ein Zitat aus einem dieser Themen oder etwas anderes, das ebenso undurchsichtig ist, ist. Dies erhöht die Zugangsvoraussetzungen und erschwert es der Gemeinschaft, sich zu beteiligen. Nehmen Sie sich die Zeit, eine angemessene Beschreibung zu verfassen und die wichtigsten Punkte zusammenzufassen.
. Achten Sie auf die Formatierung. Stellen Sie sicher, dass die https://docs.github.com/en/free-pro-team@latest/github/writing-on-github/getting-started-with-writing-and-formatting-on-github[Markdown ordentlich ist], verwenden Sie https://docs.github.com/en/free-pro-team@latest/github/writing-on-github/creating-and-highlighting-code-blocks[Codeblöcke] usw. usw.
  Je schneller etwas gelesen werden kann, desto schneller kann es bearbeitet werden.
. Seien Sie höflich. Ja, es ist ärgerlich, wenn etwas nicht funktioniert, aber es macht viel mehr Spaß, jemandem zu helfen, der nicht... das Schlimmste ist. Denken Sie daran, dass eine Unterhaltung per Text die Negativität aller Beteiligten noch verstärkt, also fügen Sie im Zweifelsfall ein paar Emoji hinzu.

== Einreichen von PRs

=== Etwas auswählen, an dem man arbeiten möchte

Wenn Sie hier um Hilfe bitten oder ein neues Verhalten einfordern möchten, ist dies der richtige Bereich für Sie. Wir haben eine Reihe von Themen für alle zusammengestellt, die einfach ihre Open-Source-Kompetenz ausbauen wollen.

* Themen mit der Bezeichnung https://github.com/search?q=org%3Agithub%2Francher+repo%3Arancher%2Fturtles+repo%3Arancher%2Fcapi-ui-extension+repo%3Arancher%2Fturtles-docs+is%3Aopen+label%3A%22good+first+issue%22+&type=issues&ref=advsearch[`+good first issues+`] sollten für Leute zugänglich sein, die neu in den Repos sind, sowie für Open Source im Allgemeinen.
+
Diese Themen sollten eine niedrige/nicht vorhandene Einstiegshürde darstellen, mit einer gründlichen Beschreibung, einer leicht nachvollziehbaren Reproduktion (falls zutreffend) und genügend Kontext, um von jedem verstanden zu werden.
Das Ziel sollte klar sein, möglicherweise mit einem Lösungsvorschlag oder einem Pseudocode.
Wenn etwas Ähnliches bereits gemacht wurde, sollte diese Arbeit verlinkt werden.
+
Wenn Sie auf ein Thema gestoßen sind, das mit `+good first issue+` gekennzeichnet ist und von dem Sie glauben, dass Sie es beanspruchen möchten, das aber nicht 100%ig klar ist, fragen Sie bitte nach weiteren Informationen! Beim Schreiben von Themen wird _viel_ Wissen vorausgesetzt, das oft als selbstverständlich angesehen wird. Das ist etwas, was wir alle besser machen können, also scheuen Sie sich nicht, nach dem zu fragen, was Sie brauchen, um gute Arbeit zu leisten.
+
Weitere Informationen über das <<_asking_for_help,Bitten um Hilfe>> finden Sie weiter unten!

* https://github.com/search?q=org%3Agithub%2Francher+repo%3Arancher%2Fturtles+repo%3Arancher%2Fcapi-ui-extension+repo%3Arancher%2Fturtles-docs+is%3Aopen+label%3A%22help+wanted%22+&type=issues&ref=advsearch[`+help wanted+` issues] sind für diejenigen, die mit der Codebasis etwas vertrauter sind, sollten aber auch für Neulinge noch zugänglich genug sein.
* Alle anderen Themen, die mit `+kind/<x>+` oder `+area/<x>+` gekennzeichnet sind, stehen ebenfalls zur Debatte, erfordern aber wahrscheinlich ein gewisses Maß an Kontext.

=== Entwicklung von Rancher-Schildkröten

Lesen Sie die speziellen xref:../developer/development.adoc[Hinweise] zum Einstieg in die Entwicklung.

=== Um Hilfe bitten

Wenn Sie in irgendeiner Phase Ihrer Arbeit Hilfe benötigen, zögern Sie nicht, uns zu fragen!

* Um mehr Details über das von Ihnen gewählte Thema zu erfahren, sollten Sie zunächst den Ersteller des Themas bitten, weitere Informationen zu liefern.
* Wenn Sie bei der Arbeit an Ihrer PR mit etwas nicht zurechtkommen oder sich nicht ganz sicher sind, wie Sie vorgehen wollen, können Sie einen https://github.blog/2019-02-14-introducing-draft-pull-requests/[Entwurf auf]öffnen (stellen Sie dem Titel `+WIP:+` voran) und erklären, was Sie denken.

=== PR-Einreichungsrichtlinien

. Forken Sie das gewünschte Repo, entwickeln und testen Sie Ihre Codeänderungen.
. Verschieben Sie Ihre Änderungen in den Zweig Ihres Forks und reichen Sie eine Pull-Anfrage an das ursprüngliche Repository gegen den Zweig `+main+` ein.

[source,bash]
----
git push <remote-name> <feature-name>
----

. Reichen Sie einen Pull-Request ein.
 .. Jeder Code PR muss mit einer der folgenden Angaben gekennzeichnet sein
  *** ⚠️ (`+:warning:+`, größere oder grundlegende Änderungen)
  *** ✨ (`+:sparkles:+`, Ergänzungen zu Funktionen)
  *** 🐛 (`+:bug:+`, Patch und Fehlerbehebungen)
  *** 📖 (`+:book:+`, Dokumentation oder Vorschläge)
  *** 🌱 (`+:seedling:+`, minderjährig oder andere)

Wenn möglich, zerquetschen Sie bitte Ihre Commits, um einen ordentlichen und anschaulichen Verlauf zu gewährleisten.

Wenn Ihr PR noch in Arbeit ist, eröffnen Sie bitte einen https://github.blog/2019-02-14-introducing-draft-pull-requests/[PR-Entwurf]und stellen Sie Ihrem Titel das Wort `+WIP+` voran. Wenn Ihre PR zur Überprüfung bereit ist, können Sie den Titel ändern und die Einstellung "Entwurf" entfernen.

Wir empfehlen, dass Sie regelmäßig von `+main+` des ursprünglichen Repos zurückbasen, um Ihren Zweig auf dem neuesten Stand zu halten.

Im Allgemeinen werden wir einen PR zusammenführen, sobald ein Betreuer ihn überprüft und genehmigt hat.
Triviale Änderungen (z. B. Korrekturen der Rechtschreibung) können durchgewunken werden.
Bei umfangreichen Änderungen können mehr Personen beteiligt werden, und Sie werden möglicherweise aufgefordert, den PR erneut einzureichen oder die Änderungen in mehrere PR aufzuteilen.

==== Formatierung der Nachricht bestätigen

_Weitere Informationen darüber, wie Sie gute Commit-Nachrichten schreiben und warum Sie das tun sollten, finden Sie unterhttps://chris.beams.io/posts/git-commit/[in diesem ausgezeichneten Blogbeitrag]._

Wir folgen einer groben Konvention für Commit-Meldungen, die drei Fragen beantworten soll: Was wurde geändert, warum wurde die Änderung vorgenommen und wie wurde sie vorgenommen.

Die Betreffzeile sollte das _"Was"_ enthalten und der Text der Mitteilung sollte das _"Warum"_ und _"Wie"_ beschreiben.
Wenn Sie unterwegs auf seltsame Dinge gestoßen sind, können Sie hier notieren, was Sie entdeckt und wie Sie das Problem gelöst haben.

Das Format lässt sich formal wie folgt beschreiben:

[source,text]
----
<short title for what changed>
<BLANK LINE>
<why this change was made and what changed>
<BLANK LINE>
<any interesting details>
<footer>
----

Die erste Zeile ist der Betreff und sollte nicht länger als 70 Zeichen sein, die zweite Zeile ist immer leer, und die anderen Zeilen sollten nicht länger als 80 Zeichen sein.
Dadurch wird die Nachricht sowohl auf GitHub als auch in verschiedenen Git-Tools besser lesbar.

Es gibt eine empfohlene Vorlage für die Verwendung https://gist.github.com/yitsushi/656e68c7db141743e81b7dcd23362f1a[hier].

== Wie die Maintainer Beiträge bearbeiten

=== Priorisierung von Themen

Das Kernteam bearbeitet regelmäßig eingehende Probleme. Während der Ferienzeiten kann es zu Verzögerungen kommen.

Jede Ausgabe wird mit einem `+priority/<x>+` Label versehen. Die Prioritätsebenen sind:

* https://github.com/rancher/turtles/labels/priority%2Fcritical-urgent[`+critical-urgent+`]: Dies sind zeitkritische Fragen, die sofort aufgegriffen werden sollten.
Wenn ein Thema, das mit `+critical+` gekennzeichnet ist, nicht zugewiesen ist oder aktiv bearbeitet wird, wird erwartet, dass jemand seine Arbeit sofort unterbricht, um es zu bearbeiten.
Normalerweise ist damit das Kernteam gemeint, aber Mitglieder der Gemeinschaft sind willkommen, Probleme auf jeder Prioritätsstufe zu reklamieren, wenn sie diese zuerst erreichen. _Sollte jedoch ein Mitarbeiter, der nicht zum Kernteam gehört, darum bitten, einem Problem auf `+critical+` zugewiesen zu werden, wird ihm ein Mitglied des Kernteams zur Seite gestellt, um die Nachverfolgung, Kommunikation und Freigabe von Korrekturen zu verwalten und die Verantwortung für alle Fortschritte zu übernehmen._
* https://github.com/rancher/turtles/labels/priority%2Fimportant-soon[`+important-soon+`]: Muss zugewiesen werden, sobald Kapazitäten frei werden.
Im Idealfall sollte etwas rechtzeitig für die nächste Version geliefert werden.
* https://github.com/rancher/turtles/labels/priority%2Fimportant-longterm[`+important-longterm+`]: Langfristig wichtig, aber möglicherweise nicht ausreichend besetzt und/oder mehrere Freisetzungen erforderlich, um sie abzuschließen.
* https://github.com/rancher/turtles/labels/priority%2Fbacklog[`+backlog+`]: Es scheint allgemeines Einvernehmen darüber zu herrschen, dass dies gut wäre, aber wir haben vielleicht niemanden, der jetzt oder in naher Zukunft daran arbeiten kann.
PRs sind nach wie vor sehr willkommen, auch wenn es eine Weile dauern kann, bis sie geprüft werden, wenn die Prüfer mit Themen von höherer Priorität ausgelastet sind, zum Beispiel unmittelbar vor einer Veröffentlichung.

Diese Prioritätskategorien wurden von https://github.com/kubernetes/community/blob/master/contributors/guide/issue-triage.md[, dem Leitfaden für Kubernetes-Mitarbeiter] inspiriert.

Andere Labels sind:

* https://github.com/rancher/turtles/labels/adr-required[`+adr-required+`]:
Zeigt an, dass die Ausgabe oder der PR eine Entscheidung enthält, die in einem <<_adrs_architectural_decision_records,ADR>> dokumentiert werden muss _, bevor_sie bearbeitet werden kann.
* https://github.com/rancher/turtles/labels/needs-investigation[`+needs-investigation+`]:  Derzeit gibt es nicht genügend Informationen, um eine richtige Kategorisierung vorzunehmen oder eine Lösung zu verstehen und umzusetzen. Dies könnte daran liegen, dass der Themeneröffner nicht genügend relevante Informationen geliefert hat oder dass eine eingehendere Recherche erforderlich ist, bevor mit der Arbeit begonnen werden kann.

=== Überprüfung von PRs

Das Kernteam ist bestrebt, die PR-Warteschlange so schnell wie möglich abzubauen. Auch die Mitglieder der Gemeinschaft sollten sich frei fühlen, die Dinge im Auge zu behalten und ihre eigenen Gedanken und ihr Fachwissen einzubringen.

Beiträge mit hohem Wert und/oder hoher Priorität werden so schnell wie möglich bearbeitet, während Dinge mit geringerer Priorität oder "Nice-to-have"-Beiträge möglicherweise etwas länger brauchen, um genehmigt zu werden.

Um eine reibungslosere und schnellere Überprüfung zu ermöglichen, sollten Sie die <<_pr_submission_guidelines,oben genannten>> Leitlinien befolgen.
Einreichungen, die den Standards nicht entsprechen, werden bei der Prüfung nicht berücksichtigt.

== ADRs (Architectural Decision Records)

[NOTE]
====
Sie können <<_adrs_architectural_decision_records,diesen>> und den folgenden <<_process,Abschnitt>> gerne überspringen, da sie nur für das https://github.com/rancher/turtles[rancher-turtles] Repository relevant sind.
====


Alle Entscheidungen, die Auswirkungen auf die Architektur, das Design, die Entwicklung und das Verhalten von Rancher-Schildkröten haben, müssen in Form eines https://engineering.atspotify.com/2020/04/14/when-should-i-write-an-architecture-decision-record/[ADR] festgehalten werden.

Eine Vorlage ist zu finden unter https://github.com/rancher/turtles/blob/main/docs/adr/0000-template.md[`+docs/adr/0000-template.md+`] des Repos zu finden, mit zahlreichen Beispielen für fertige Datensätze im selben Verzeichnis.

Die Mitwirkenden können auch ADRs ergänzen, wenn sie fehlen.

=== Prozess

. Starten Sie eine neue https://github.com/rancher/turtles/discussions/new?category=adr[Diskussion] in der Kategorie `+ADR+`.
. Wählen Sie einen angemessenen, klaren und prägnanten Titel (z. B. `+ADR: Implement X in Go+`).
. Geben Sie den Kontext der zu treffenden Entscheidung an. Beschreiben Sie die verschiedenen Optionen, wenn es mehr als eine gibt, und erläutern Sie die Vor- und Nachteile. Heben Sie die Bereiche hervor, auf die Sie die Gutachter aufmerksam machen möchten oder zu denen Sie speziell eine Stellungnahme wünschen.
. Beziehen Sie die https://github.com/rancher/turtles/blob/main/CODEOWNERS[-Maintainer] als "Entscheider" ein und laden Sie sie ein, sich an der Entscheidung und ihren Folgen zu beteiligen und diese zu bewerten.
. Sobald eine Entscheidung getroffen wurde, eröffnen Sie eine PR und fügen ein neues ADR in das https://github.com/rancher/turtles/blob/main/docs/adr[Verzeichnis] ein.
  Kopieren Sie die https://github.com/rancher/turtles/blob/main/docs/adr/0000-template.md[Vorlage] und füllen Sie sie aus;
 ** Erhöht die Dateinummer um eins
 ** Setzen Sie den Status auf "Angenommen".
 ** Legen Sie die Entscheider als diejenigen fest, die das Ergebnis der Diskussion genehmigt haben.
 ** Fassen Sie die Entscheidung und die Konsequenzen aus dem Diskussionsfaden zusammen
 ** Link zurück zur Diskussion aus dem ADR-Dokument
