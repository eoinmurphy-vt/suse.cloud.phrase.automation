== Übersicht

https://slsa.dev/spec/v1.0/about[SLSA] ist eine Reihe von schrittweise annehmbaren Leitlinien für die Sicherheit der Lieferkette, die im Konsens mit der Industrie erstellt wurden. Die von SLSA festgelegten Spezifikationen sind sowohl für Softwarehersteller als auch für Verbraucher nützlich: Hersteller können die SLSA-Leitlinien befolgen, um ihre Software-Lieferkette sicherer zu machen, und Verbraucher können SLSA nutzen, um zu entscheiden, ob sie einem Softwarepaket vertrauen.

== Plattform aufbauen

* Das Projekt {product_name} verwendet Git für die Quellcodeverwaltung.
* Alle Betreuer von {product_name} müssen die Zwei-Faktor-Authentifizierung aktivieren, um alle ihre Beiträge zu unterzeichnen und abzuzeichnen.
* Das Projekt {product_name} verwendet GitHub Actions und GitHub Runners für die Erstellung aller Release-Artefakte.
* Der Build- und Release-Prozess läuft isoliert in einer ephemeren Umgebung ab, die von auf GitHub gehosteten Runnern bereitgestellt wird.

== Prozess erstellen

* Der Build- und Release-Prozess ist im Code definiert und wird unter Versionskontrolle gehalten.
* Die GitHub-Workflows nutzen GitHub-Aktionen, die an bestimmte Versionen angeheftet sind, und werden mit GitHub Dependabot auf dem neuesten Stand gehalten.
* Alle Änderungen am Build- und Release-Prozess werden über Pull Requests vorgenommen, die von mindestens einem {product_name} Maintainer genehmigt werden müssen.
* Der Freigabeprozess kann nur von einem {product_name} -Maintainer angestoßen werden, indem ein Git-Tag im Semver-Format veröffentlicht wird.

== Provenienz

* Das Projekt {product_name} nutzt das offizielle https://github.com/slsa-framework/slsa-github-generator[SLSA GitHub Generator] Projekt für die Generierung und Verteilung der Provenienz.
* Die Provenienz für die in der GitHub Container Registry und in der Rancher Prime Registry veröffentlichten Release-Artefakte wird mit dem generator_container_slsa3 GitHub Workflow generiert, der vom https://github.com/slsa-framework/slsa-github-generator[SLSA GitHub Generator] Projekt bereitgestellt wird.
* Die Provenienz identifiziert die {product_name} Container-Images anhand ihres Digest im SHA-256-Format.
* Die Provenienz wird von Sigstore Cosign unter Verwendung der GitHub OIDC-Identität signiert, und der öffentliche Schlüssel zur Verifizierung der Provenienz wird im öffentlichen https://docs.sigstore.dev/logging/overview/[Rekor-Transparenzprotokoll] gespeichert.
* Der Freigabeprozess und die Provenance-Generierung laufen isoliert in einer ephemeren Umgebung, die von auf GitHub gehosteten Runnern bereitgestellt wird.
* Die Herkunft der {product_name} Container-Images kann mit dem offiziellen https://github.com/slsa-framework/slsa-verifier[SLSA Verifier Tool] überprüft werden.
* Die Arbeitsabläufe zur Provenance-Generierung laufen auf ephemeren und isolierten virtuellen Maschinen, die vollständig von GitHub verwaltet werden.
* Die Provenance-Signiergeheimnisse sind flüchtig und werden durch das Signierverfahren von Sigstore https://github.com/sigstore/cosign/blob/main/KEYLESS.md[schlüssellose] erzeugt.
* Der https://github.com/slsa-framework/slsa-github-generator[SLSA GitHub Generator] läuft auf anderen virtuellen Maschinen als der Build- und Release-Prozess, so dass die {product_name} Build-Skripte keinen Zugriff auf die Signiergeheimnisse haben.

== Isolierung

* Der Freigabeprozess und die Provenance-Generierung laufen isoliert in einer ephemeren Umgebung, die von auf GitHub gehosteten Runnern bereitgestellt wird.
* Die Provenance-Generierung ist vom Build-Prozess entkoppelt; der https://github.com/slsa-framework/slsa-github-generator[SLSA GitHub Generator] läuft auf separaten virtuellen Maschinen, die vollständig von GitHub verwaltet werden.
* Der Freigabeprozess kann nicht auf den Provenance-Signierschlüssel zugreifen, da der Provenance-Generator isoliert auf separaten, von GitHub gehosteten Runnern läuft.
