= GitOps Best Practices für Turtles und CAPI
:revdate: 2025-10-02
:page-revdate: {revdate}

ifeval::["{build-type}" == "product"]
Dieser Abschnitt enthält Best Practices für die Implementierung von GitOps mit Turtles und Cluster API (CAPI). Der Schwerpunkt liegt dabei auf der Verwendung von https://documentation.suse.com/cloudnative/continuous-delivery/v0.12/en/index.html[Continuous Delivery] in Kombination mit der *Cluster API Addon Provider Fleet*(`+CAAPF+`) zur Automatisierung der Agentenbereitstellung und der Integration in das `+CAPI+` Ökosystem.
endif::[]
ifeval::["{build-type}" == "community"]
Dieser Abschnitt enthält Best Practices für die Implementierung von GitOps mit Turtles und Cluster API (CAPI), wobei der Schwerpunkt auf der Verwendung von https://fleet.rancher.io/[Fleet] in Kombination mit dem *Cluster API Addon Provider Fleet*(`+CAAPF+`) zur Automatisierung der Agentenbereitstellung und der Integration in das `+CAPI+` Ökosystem liegt.
endif::[]

Diese Tipps sind übertragbar und können auch bei Verwendung einer anderen GitOps-Lösung angewendet werden.

== Allgemeine Tipps

ifeval::["{build-type}" == "product"]
- **Getrennte CAPI-Cluster und ClusterClass-Konfigurationen:**
  Halten Sie CAPI-Cluster-Konfigurationen (`+clusters/+`) getrennt von CAPI `+ClusterClass+` Definitionen (`+templates/+`). Diese Trennung ermöglicht die Bereitstellung mehrerer Cluster aus gemeinsamen Vorlagen und verringert gleichzeitig das Risiko unbeabsichtigter Infrastrukturänderungen.
- **Addon-Konfigurationen isolieren:**
  Speichern Sie Anwendungsmanifeste in `+applications/+`, um sicherzustellen, dass sie unabhängig von `+Clusters+` und `+ClusterClasses+` sind. Dies ermöglicht eine selektive oder gruppierte Bereitstellung auf der Grundlage der Nutzernachfrage oder übereinstimmender Selektoren, wodurch Störungen der Cluster-Infrastruktur vermieden werden. Bitte beachten Sie, dass die Helm-Diagrammabhängigkeiten vor der Bereitstellung von Addons auf https://documentation.suse.com/cloudnative/continuous-delivery/v0.12/en/gitrepo-content.html#_fleet_yaml[aktualisiert] werden müssen.
- **Nutzen Sie verschachtelte GitRepo-Ressourcen:**
  Nutzen Sie https://documentation.suse.com/cloudnative/continuous-delivery/v0.12/en/gitrepo-content.html#_nested_gitrepo_crs[] verschachtelte `+GitRepo+` Ressourcen in Fleet, um Repository-Inhalte effizient zu strukturieren. Dieser Ansatz ermöglicht eine modulare Bereitstellung, bei der `+ClusterClass+` Definitionen, erforderliche Addons/Anwendungen und Clusterkonfigurationen kombiniert werden, wobei die Trennung der Bereiche beibehalten wird.
- **Verwenden Sie die GitOps-Integrationswerkzeuge, wenn sie verfügbar sind:**
  Berücksichtigen Sie bei der Konfiguration eines GitOps-Setups die verfügbaren Werkzeuge und Funktionen, die sie bieten. Ein Beispiel hierfür ist https://rancher.github.io/cluster-api-addon-provider-fleet/00_intro.html[`+CAAPF+`]das die Installation von Fleet GitOps auf `+CAPI+` Clustern automatisiert und eine bequeme Automatisierung ermöglicht.
- **Wenden Sie Kustomize Overlays für umgebungsspezifische Anpassungen an:**
  Verwenden Sie `+overlays/+`, um Kustomize Overlays für verschiedene Umgebungen zu speichern. Dadurch wird sichergestellt, dass Konfigurationsänderungen, wie z. B. Skalierungsanpassungen, Netzwerkmodifikationen oder Komponenten-Upgrades, dynamisch vorgenommen werden können, ohne die Basis-Manifeste zu verändern.
- **Verwenden Sie Compare Patches:**
  Fleet ermöglicht das selektive Ignorieren von Ressourcendifferenzen, wenn das Bündel geeignete Regeln für die https://documentation.suse.com/cloudnative/continuous-delivery/v0.12/en/troubleshooting.html#_suse_rancher_prime_continuous_delivery_deployment_stuck_in_modified_state[Erkennung von] Dies würde verhindern, dass Bündel in einem geänderten Zustand verharren.
endif::[]

== Strategie zur Trennung von Ordnern

Ein gut strukturiertes Repository vereinfacht die Verwaltung und verbessert die Übersichtlichkeit. Die folgende Strategie zur Trennung von Ordnern wird empfohlen:

----
repo-root/
│── fleet/
│   ├── providers/         # CAPIProvider files to use with templates
│   │   ├── core/          # Core provider configuration
│   │   ├── infra-docker/  # Docker provider configuration
│   │   ├── infra-aws/     # AWS provider configuration
│   ├── clusters/          # Cluster-specific configurations
│   │   ├── prod/          # Production clusters
│   │   ├── staging/       # Staging clusters
│   │   ├── dev/           # Development clusters
│   ├── addons/            # Application-specific manifests focused on cluster bootstrapping, like CNI, CCM, CPI configurations.
│   │   ├── cni/           # Group applications based on purpose. Provide fleet.yaml per each sub-directory to maintain bundle separation.
│   │   ├── ccm/
│   ├── templates/         # CAPI ClusterClass templates for cluster provisioning
│   ├── fleet.yaml         # Fleet bundle configuration
│── overlays/              # Kustomize overlays for different environments
│   ├── prod/.             # Kustomize overlays for production environments
│   ├── staging/.          # Kustomize overlays for staging environments
│   ├── dev/.              # Kustomize overlays for development environments
----

== Anbieter Konfiguration

CAPI-Anbieter definieren die für die Bereitstellung von Clustern erforderlichen Infrastrukturkomponenten. Ein einfacher Aufbau könnte wie folgt aussehen:

----
repo-root/
│── fleet/
│   ├── providers/
│   │   ├── core/
│   │   |   ├── core.yaml  # Core `+CAPIProvider+` definition
│   │   |   ├── fleet.yaml # Fleet.yaml definition for the provider setup
----

in dem der Kernanbieter enthalten ist:

[source,yaml]
----
apiVersion: turtles-capi.cattle.io/v1alpha1
kind: CAPIProvider
metadata:
  name: cluster-api
spec:
  type: core
----

== Schablonen Konfiguration

Geben Sie `+dependsOn+` in `+fleet.yaml+` für bestimmte `+CAPIProvider+` -Bündel oder externe `+GitRepo+` -Komponenten an. Dadurch wird sichergestellt, dass die Vorlagen in der richtigen Reihenfolge ausgerollt werden und Probleme vermieden werden, bei denen Komponenten-Controller fehlen oder nicht bereit sind, `+ClusterClass+` Definitionen zu akzeptieren.

[source,yaml]
----
namespace: capa-clusters
dependsOn:
  - name: core      # Ensure core bundle is installed first
  - name: infra-aws # Wait for AWS provider installation
----

== Konfiguration von Clustern

Verwenden Sie Labels, um Cluster dynamisch mit den erforderlichen Addons abzugleichen, so dass nur die erforderlichen Workloads bereitgestellt werden können.

Beispiel für die Verwendung des Etiketts:

[source,yaml]
----
apiVersion: cluster.x-k8s.io/v1beta1
kind: Cluster
metadata:
  name: aws-cluster
  labels:
    cni: calico
    ccm: aws
    env: dev
----

[WARNING]
====
Es wird empfohlen, die Cluster-Definitionen von den Vorlagen zu trennen, da eine solche Cluster-Bündel-Entfernung es dem CAPI-Controller ermöglichen würde, die Löschung in der richtigen Reihenfolge zu verarbeiten. Die gleichzeitige Entfernung von `+Clusters+` und `+ClusterClasses+` kann zu unerwarteten Ergebnissen führen.
====

== Konfiguration des Zusatzes

Die Addon-Konfiguration ist für die Rationalisierung der Bereitstellung von Clustern von entscheidender Bedeutung und stellt sicher, dass sie den Zustand `+Ready+` ohne manuelle Eingriffe oder Tools von Drittanbietern erreichen.

Cluster-Addons sollten präzise Auswahlregeln für den Abgleich mit entsprechend gekennzeichneten Clustern festlegen.

Das folgende Beispiel zeigt einen Cluster, der die Installation von `+Calico+` CNI und `+AWS+` Cloud Controller Manager (CCM) erfordert:

[source,yaml]
----
apiVersion: cluster.x-k8s.io/v1beta1
kind: Cluster
metadata:
  name: aws-cluster
  labels:
    cni: calico
    ccm: aws
----

Dieses Setup erfordert zwei `+GitRepo+` Ressourcen, die jeweils für die Bereitstellung von Addons auf der Grundlage bestimmter Labels zuständig sind - eine für `+cni: calico+` und eine für `+ccm: aws+`.

Hier ist ein Beispiel für eine `+GitRepo+` Ressource für den Einsatz der `+Calico+` CNI:

[source,yaml]
----
apiVersion: fleet.cattle.io/v1alpha1
kind: GitRepo
metadata:
  name: calico
spec:
  branch: main
  paths:
  - /fleet/applications/calico
  repo: https://github.com/rancher-sandbox/cluster-api-addon-provider-fleet.git
  targets:
  - clusterSelector:
      matchLabels:
        cni: calico
        env: dev
----

Dadurch wird sichergestellt, dass die CNI `+Calico+` nur auf Clustern mit den Kennzeichnungen `+cni: calico+` und `+env: dev+` eingesetzt wird, was eine selektive Bereitstellung von Umgebungen ermöglicht.

=== Helm Chart Werte Templating

Die Addon-Konfiguration muss die Arbeitslast von `+Calico+` auf der Grundlage der passenden Cluster-Definition und des spezifischen Cluster-Status dynamisch anpassen. Dies kann durch eine `+fleet.yaml+` Einrichtung erreicht werden, die der folgenden Struktur ähnelt:

----
repo-root/
│── fleet/
│   ├── applications/
│   │   ├── calico/
│   │   │   ├── fleet.yaml # Fleet configuration for the Calico setup
----

Die Datei `+fleet.yaml+` definiert Schablonenregeln und `+comparePatches+`, um ein reibungsloses Rollout unabhängig von der Clusterkonfiguration zu gewährleisten. Um mehr über `+CAAPF+` Templating zu erfahren, das hier verwendet wird, lesen Sie die offizielle https://rancher.github.io/cluster-api-addon-provider-fleet/04_reference/02_templating-strategy.html[Dokumentation].

[source,yaml]
----
helm:
  releaseName: projectcalico
  repo: https://docs.tigera.io/calico/charts
  chart: tigera-operator
  templateValues:
    installation: |-
      cni:
        type: Calico
        ipam:
          type: HostLocal
      calicoNetwork:
        bgp: Disabled
        mtu: 1350
        ipPools:
          ${- range $cidr := .ClusterValues.Cluster.spec.clusterNetwork.pods.cidrBlocks }
          - cidr: "${ $cidr }"
            encapsulation: None
            natOutgoing: Enabled
            nodeSelector: all()${- end}

diff:
  comparePatches:
  - apiVersion: operator.tigera.io/v1
    kind: Installation
    name: default
    operations:
    - {"op":"remove", "path":"/spec/kubernetesProvider"}
----
