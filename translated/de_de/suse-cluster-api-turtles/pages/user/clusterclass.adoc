= ClusterClass
:revdate: 2026-02-18
:page-revdate: {revdate}

In diesem Abschnitt geht es um die Verwendung von https://cluster-api.sigs.k8s.io/tasks/experimental-features/cluster-class/[ClusterClass] mit der {product_name}.

== Voraussetzungen

* Rancher Manager-Cluster mit {product_name} installiert
* Der https://github.com/rancher/cluster-api-addon-provider-fleet[CAAPF-Provider] muss als Voraussetzung für die Nutzung aller ClusterClasses installiert sein.
** Beispiel für die Installation eines CAAPF-Anbieters:
+
.Klicken zum Erweitern
[%collapsible]
======
[,yaml]
----
apiVersion: v1
kind: Namespace
metadata:
  name: fleet-addon-system
---
apiVersion: turtles-capi.cattle.io/v1alpha1
kind: CAPIProvider
metadata:
  name: fleet
  namespace: fleet-addon-system
spec:
  name: rancher-fleet
  type: addon
----
======

[WARNING]
====
CAAPF hängt vom `+WatchList+` Kubernetes Feature Gate ab. Diese Funktion muss bei den Kubernetes-Versionen `+1.33+` explizit aktiviert werden. Weitere Informationen finden Sie in der https://kubernetes.io/docs/reference/command-line-tools-reference/feature-gates/[Upstream-Dokumentation] Kubernetes.
====

* Je nach Motor ist entweder die Installation von https://github.com/rancher/cluster-api-provider-rke2[CAPRKE2] oder https://github.com/kubernetes-sigs/cluster-api[] Kubeadm Bootstrap/Control Plane erforderlich, bevor Sie CAPIProvider einrichten können:

== Einrichtung

ifeval::["{build-type}" == "community"]
[NOTE]
.Version Pinning
====
Der CAPIProvider `+spec.version+` kann jederzeit eingestellt werden, um den Provider an eine bestimmte Version zu binden. Um die Unveränderlichkeit zu erhalten, können Sie auch `+spec.enableAutomaticUpdate: false+` so einstellen, dass automatische Versions-Updates für Ihren jeweiligen CAPIProvider deaktiviert werden.
====
endif::[]

[tabs]
======
Azure::
+
--
Um den Verwaltungscluster vorzubereiten, installieren wir den https://capz.sigs.k8s.io/[Cluster API Provider Azure] und erstellen eine https://capz.sigs.k8s.io/topics/identities#service-principal[ServicePrincipal-Identität] zur Bereitstellung eines neuen Clusters auf Azure.

Bevor wir beginnen, muss ein ServicePrincipal erstellt werden, das zumindest über einen Contributor-Zugriff auf ein Azure-Abonnement verfügt.
Weitere Einzelheiten finden Sie in der https://capz.sigs.k8s.io/topics/identities[Dokumentation].

* Provider-Installation
+
[source,yaml]
----
apiVersion: v1
kind: Namespace
metadata:
  name: capz-system
---
apiVersion: turtles-capi.cattle.io/v1alpha1
kind: CAPIProvider
metadata:
  name: azure
  namespace: capz-system
spec:
  type: infrastructure
  name: azure
----

* Identität einrichten
+
[WARNING]
.Authentifizierung für die Bereitstellung von AKS-Clustern
====
Wenn Sie einen verwalteten Cluster auf **AKS** bereitstellen, können Sie den Schritt **Identitätseinrichtung** überspringen und zu xref:#_create_a_cluster_from_a_clusterclass[Cluster aus einer Clusterklasse erstellen] wechseln.
Anstatt `+AzureClusterIdentity+` zu verwenden, erstellen Sie eine `+Secret+` mit den Anmeldeinformationen für den Service Principal, die der Azure Service Operator bei der Anwendung des Clustermanifests verwenden kann.
====
+
In diesem Beispiel werden wir eine `+AzureClusterIdentity+` verwenden, um Azure-Cluster bereitzustellen.
Zunächst muss ein Secret erstellt werden, das die Anmeldeinformationen des Service Principal enthält, damit die Ressource `+AzureClusterIdentity+` darauf verweisen kann.
Beachten Sie, dass `+AzureClusterIdentity+` eine Ressource mit Namensraum ist und im gleichen Namensraum wie der Cluster erstellt werden muss.
Weitere Informationen zu bewährten Verfahren bei der Verwendung von Azure-Identitäten finden Sie in der offiziellen https://capz.sigs.k8s.io/topics/identities-use-cases[Dokumentation].
+
Beachten Sie, dass einige Variablen vom Benutzer selbst ersetzt werden können.
+
[source,yaml]
----
apiVersion: v1
kind: Secret
metadata:
  name: <AZURE_CLUSTER_IDENTITY_SECRET_NAME>
  namespace: <AZURE_CLUSTER_IDENTITY_SECRET_NAMESPACE>
type: Opaque
stringData:
  clientSecret: <AZURE_CLIENT_SECRET>
---
apiVersion: infrastructure.cluster.x-k8s.io/v1beta1
kind: AzureClusterIdentity
metadata:
  labels:
    clusterctl.cluster.x-k8s.io/move-hierarchy: "true"
  name: cluster-identity
spec:
  allowedNamespaces: {}
  clientID: <AZURE_APP_ID>
  clientSecret:
    name: <AZURE_CLUSTER_IDENTITY_SECRET_NAME>
    namespace: <AZURE_CLUSTER_IDENTITY_SECRET_NAMESPACE>
  tenantID: <AZURE_TENANT_ID>
  type: ServicePrincipal
----
--

AWS::
+
--
Um den Verwaltungscluster vorzubereiten, installieren wir den https://cluster-api-aws.sigs.k8s.io/[Cluster API Provider AWS] und erstellen ein Geheimnis mit den erforderlichen Anmeldeinformationen für die Bereitstellung eines neuen Clusters auf AWS.
Die globalen Anmeldeinformationen sind auf Leerzeichen gesetzt, da wir stattdessen `+AWSClusterStaticIdentity+` verwenden werden.

* Provider-Installation
+
[source,yaml]
----
apiVersion: turtles-capi.cattle.io/v1alpha1
kind: CAPIProvider
metadata:
  name: aws
  namespace: capa-system
spec:
  type: infrastructure
  variables:
    AWS_B64ENCODED_CREDENTIALS: ""
----

* Identität einrichten
+
In diesem Beispiel verwenden wir eine `+AWSClusterStaticIdentity+` für die Bereitstellung von AWS-Clustern. +
Ein Secret, das die Anmeldeinformationen enthält, muss in dem Namespace erstellt werden, in dem der AWS-Anbieter installiert ist. +
Weitere Informationen zur Einrichtung der Anmeldeinformationen finden Sie in der link:https://cluster-api-aws.sigs.k8s.io/clusterawsadm/clusterawsadm[Dokumentation]. +
Die `+AWSClusterStaticIdentity+` kann auf dieses Secret verweisen, um die Bereitstellung von Clustern zu ermöglichen. In diesem Beispiel erlauben wir die Verwendung der Identität in allen Namespaces, so dass sie leicht wiederverwendet werden kann. +
Weitere Informationen zum Identitätsmanagement finden Sie in der link:https://cluster-api-aws.sigs.k8s.io/topics/multitenancy[offiziellen Dokumentation].
+
Beachten Sie, dass einige Variablen vom Benutzer selbst ersetzt werden können. +
+
[source,yaml]
----
apiVersion: v1
kind: Secret
metadata:
  name: <AWS_IDENTITY_SECRET_NAME>
  namespace: capa-system
type: Opaque
stringData:
  AccessKeyID: <AWS_ACCESS_KEY_ID>
  SecretAccessKey: <AWS_SECRET_ACCESS_KEY>
---
apiVersion: infrastructure.cluster.x-k8s.io/v1beta2
kind: AWSClusterStaticIdentity
metadata:
  name: cluster-identity
spec:
  secretRef: <AWS_IDENTITY_SECRET_NAME>
  allowedNamespaces: {}
----
--

GCP::
+
--
Um den Verwaltungscluster vorzubereiten, installieren wir den https://cluster-api-gcp.sigs.k8s.io/[Cluster API Provider GCP] und erstellen ein Geheimnis mit den Anmeldeinformationen, die für die Bereitstellung eines neuen Clusters auf GCP erforderlich sind.
Zum Erstellen und Verwalten von Clustern in GCP ist ein Dienstkonto erforderlich, für das die Berechtigungen `+Editor+` erforderlich sind. Sie können den offiziellen Leitfaden aus dem https://cluster-api-gcp.sigs.k8s.io/quick-start#create-a-service-account[CAPG Book] befolgen.
Der base64-kodierte Schlüssel des Dienstkontos muss in der Variablen `+GCP_B64ENCODED_CREDENTIALS+` des Anbieters angegeben werden.

* Provider-Installation
+
[source,yaml]
----
apiVersion: turtles-capi.cattle.io/v1alpha1
kind: CAPIProvider
metadata:
  name: gcp
  namespace: capg-system
spec:
  type: infrastructure
  variables:
    GCP_B64ENCODED_CREDENTIALS: xxx
----

* Netzwerk-Einrichtung
+
Für die Bereitstellung eines selbstverwalteten GCP-Clusters muss ein GCP-Netzwerk konfiguriert werden, damit Kubernetes-Knoten mit der Steuerebene kommunizieren und Images aus der Container-Registry abrufen können, wofür die Maschinen über NAT-Zugang oder eine öffentliche IP verfügen müssen.
Das Standardverhalten des Anbieters besteht darin, virtuelle Maschinen ohne öffentliche IP zu erstellen. Daher ist ein https://cloud.google.com/nat/docs/overview[Cloud NAT] erforderlich, damit die Knoten eine Verbindung mit dem Load Balancer und der Außenwelt herstellen können.
Wie Sie Ihr GCP-Netzwerk für die Bereitstellung eines selbstverwalteten GCP-Clusters vorbereiten, entnehmen Sie bitte der offiziellen Anleitung https://cluster-api-gcp.sigs.k8s.io/prerequisites#configure-network-and-cloud-nat[CAPG Book].
+
[NOTE]
====
Die folgenden Schritte sind erforderlich, um das GCP-Netzwerk für die Cluster-Bereitstellung vorzubereiten:

- Erstellen Sie einen Router.
- Erstellen Sie ein mit dem Router verbundenes NAT.
====

vSphere::
+
--
Um den Verwaltungscluster vorzubereiten, installieren wir den https://github.com/kubernetes-sigs/cluster-api-provider-vsphere/blob/main/docs/getting_started.md[Cluster API Provider vSphere].
Die globalen Anmeldeinformationen sind auf Leerzeichen gesetzt, da wir stattdessen `+VSphereClusterIdentity+` verwenden werden.

* Provider-Installation
+
[source,yaml]
----
apiVersion: v1
kind: Namespace
metadata:
  name: capv-system
---
apiVersion: turtles-capi.cattle.io/v1alpha1
kind: CAPIProvider
metadata:
  name: vsphere
  namespace: capv-system
spec:
  type: infrastructure
  variables:
    VSPHERE_USERNAME: "" 
    VSPHERE_PASSWORD: ""
----

* Identität einrichten
+
In diesem Beispiel wird eine `+VSphereClusterIdentity+` zur Bereitstellung von vSphere-Clustern verwendet. +
In dem Namespace, in dem der vSphere-Anbieter installiert ist, muss ein Secret erstellt werden, das die Anmeldeinformationen enthält.
Die `+VSphereClusterIdentity+` kann auf dieses Secret verweisen, um die Bereitstellung von Clustern zu ermöglichen. In diesem Beispiel erlauben wir die Verwendung der Identität in allen Namespaces, so dass sie leicht wiederverwendet werden kann.
Weitere Informationen zum Identitätsmanagement finden Sie in der https://github.com/kubernetes-sigs/cluster-api-provider-vsphere/blob/main/docs/identity_management.md[offiziellen Dokumentation].
+
[source,yaml]
----
apiVersion: v1
kind: Secret
metadata:
  name: cluster-identity
  namespace: capv-system
type: Opaque
stringData:
  username: xxx
  password: xxx
---
apiVersion: infrastructure.cluster.x-k8s.io/v1beta1
kind: VSphereClusterIdentity
metadata:
  name: cluster-identity
spec:
  secretName: cluster-identity
  allowedNamespaces:
    selector:
      matchLabels: {}
----

== Erstellen eines Clusters aus einer ClusterClass

[tabs]
======

Azure RKE2::
+
--
* Eine Azure ClusterClass ist unter den https://github.com/rancher/turtles/tree/main/examples/clusterclasses[Beispielen] Turtles zu finden.
+
Anwendungen wie der Azure Cloud Provider und Calico CNI werden auf nachgelagerten Clustern installiert. Dies geschieht automatisch bei der Erstellung von Clustern, indem die Cluster mit bestimmten Bezeichnungen versehen werden, z. B. `+cloud-provider: azure+` und `+cni: calico+`.
+
[tabs]
=======
CLI::
+
Eine Azure RKE2 ClusterClass und zugehörige Anwendungen können mit dem Beispieltool angewendet werden:
+
[source,bash, subs="attributes"]
----
go run github.com/rancher/turtles/examples@{turtles_version} -r azure-rke2 | kubectl apply -f -
----
kubectl::
+
* Alternativ können Sie die Azure RKE2 ClusterClass auch direkt mit kubectl anwenden:
+
[source,bash, subs="attributes"]
----
kubectl apply -f https://raw.githubusercontent.com/rancher/turtles/refs/tags/{turtles_version}/examples/clusterclasses/azure/rke2/clusterclass-rke2-example.yaml
----

* Zusätzlich muss der https://capz.sigs.k8s.io/self-managed/cloud-provider-config[Azure Cloud Provider] auf jedem nachgelagerten Cluster installiert werden, damit die Knoten korrekt initialisiert werden können.
Für dieses Beispiel werden wir auch https://docs.tigera.io/calico/latest/about/[Calico] als Standard-CNI installieren.
+
Wir können dies automatisch bei der Cluster-Erstellung mit dem https://rancher.github.io/cluster-api-addon-provider-fleet/[Cluster API Add-on Provider Fleet] tun.
Zunächst müssen zwei `+HelmOps+` erstellt werden, die über Label-Selektoren auf den neuen Cluster angewendet werden.
+
[source,bash, subs="attributes"]
----
kubectl apply -f https://raw.githubusercontent.com/rancher/turtles/refs/tags/{turtles_version}/examples/applications/ccm/azure/helm-chart.yaml
kubectl apply -f https://raw.githubusercontent.com/rancher/turtles/refs/tags/{turtles_version}/examples/applications/cni/calico/helm-chart.yaml
----

* Erstellen Sie den Azure Cluster aus dem Beispiel ClusterClass
+
Beachten Sie, dass einige Variablen vom Benutzer selbst ersetzt werden können.
+
[source,yaml]
----
apiVersion: cluster.x-k8s.io/v1beta2
kind: Cluster
metadata:
  labels:
    cluster-api.cattle.io/rancher-auto-import: "true"
    cloud-provider: azure
    cni: calico
  name: azure-quickstart
spec:
  clusterNetwork:
    pods:
      cidrBlocks:
      - 192.168.0.0/16
  topology:
    classRef: 
      name: azure-rke2-example
    controlPlane:
      replicas: 1
    variables:
    - name: subscriptionID
      value: <AZURE_SUBSCRIPTION_ID>
    - name: location
      value: <AZURE_LOCATION>
    - name: resourceGroup
      value: <AZURE_RESOURCE_GROUP>
    - name: azureClusterIdentityName
      value: cluster-identity
    version: v1.34.3+rke2r1
    workers:
      machineDeployments:
      - class: rke2-default-worker
        name: md-0
        replicas: 1
----

* Erstellen Sie den Azure AKS Cluster aus der Beispiel ClusterClass.
+
Die Azure AKS `+ClusterClass+`, aus der `+Cluster+` erstellt wird, verwendet die Ressourcen `+AzureASOManagedCluster+`, `+AzureASOManagedControlPlane+` und `+AzureASOManagedMachinePool+`, die von Azure Service Operator verwaltet werden.
Aus diesem Grund werden Sie nicht wie bei einem selbstverwalteten Azure-Cluster (z.B. Azure mit RKE2) ein `+AzureClusterIdentity+` verwenden. Stattdessen wird eine `+Secret+` mit den Anmeldeinformationen des Service Principal im gleichen Namensraum wie die `+Cluster+` erstellt.
Beachten Sie, dass die Definition von `+Secret+` der Einfachheit halber in das Manifest `+Cluster+` eingebettet ist, sie kann aber auch vorher erstellt werden.
+
Weitere Informationen zu dieser Authentifizierungsmethode finden Sie in der https://azure.github.io/azure-service-operator/guide/authentication/[Dokumentation].
+
Beachten Sie, dass einige Variablen vom Benutzer selbst ersetzt werden können.
+
[source,yaml]
----
apiVersion: v1
kind: Secret
metadata:
  name: aks-credentials
stringData:
  AZURE_SUBSCRIPTION_ID: <AZURE_SUBSCRIPTION_ID>
  AZURE_TENANT_ID: <AZURE_TENANT_ID>
  AZURE_CLIENT_ID: <AZURE_CLIENT_ID>
  AZURE_CLIENT_SECRET: <AZURE_CLIENT_SECRET>
type: Opaque
---
apiVersion: cluster.x-k8s.io/v1beta2
kind: Cluster
metadata:
  labels:
    cluster-api.cattle.io/rancher-auto-import: "true"
  name: azure-aks-quickstart
spec:
  clusterNetwork:
    pods:
      cidrBlocks:
      - 192.168.0.0/16
  topology:
    class: azure-aks-example
    variables:
    - name: location
      value: <AZURE_LOCATION>
    - name: resourceGroup
      value: <AZURE_RESOURCE_GROUP>
    - name: asoCredentialSecretName
      value: aks-credentials
    version: v1.34.2
    workers:
      machinePools:
      - class: default-system
        name: system-1
        replicas: 1
      - class: default-worker
        name: worker-1
        replicas: 1
----
--

  name: gcp-quickstart

  clusterNetwork:
    pods:
      cidrBlocks: 
      - 192.168.0.0/16

      replicas: 1
    workers:
      machineDeployments:
        - class: "default-worker"
          name: "md-0"
          replicas: 1
    variables:
      - name: gcpProject
        value: <GCP_PROJECT>
      - name: region
        value: <GCP_REGION>
      - name: gcpNetworkName
        value: <GCP_NETWORK_NAME>
      - name: clusterFailureDomains
        value:
          - "<GCP_REGION>-a"
      - name: imageId
        value: <GCP_IMAGE_ID>
      - name: machineType
        value: <GCP_MACHINE_TYPE>
    version: v1.34.1

* Cluster-Konfiguration
+
Der vSphere Cloud Provider und der vSphere CSI-Controller benötigen eine zusätzliche Konfiguration, die auf den Downstream-Cluster angewendet werden muss.
Ähnlich wie bei den obigen Schritten können wir zwei zusätzliche Flottenbündel erstellen, die auf den nachgeschalteten Cluster angewendet werden.
Bitte beachten Sie, dass diese Bundles so konfiguriert sind, dass sie auf den nachgelagerten Cluster mit dem Namen `+vsphere-kubeadm-quickstart+` abzielen.
Wenn Sie einen anderen Namen für Ihren Cluster verwenden, ändern Sie die Bündelziele entsprechend.
+
[source,yaml]
----
kind: Bundle
apiVersion: fleet.cattle.io/v1alpha1
metadata:
  name: vsphere-csi-config
spec:
  resources:
  - content: |-
      apiVersion: v1
      kind: Secret
      type: Opaque
      metadata:
        name: vsphere-config-secret
        namespace: vmware-system-csi
      stringData:
        csi-vsphere.conf: |+
          [Global]
          thumbprint = "<VSPHERE_THUMBPRINT>"
          [VirtualCenter "<VSPHERE_SERVER>"]
          user = "<VSPHERE_USER>"
          password = "<VSPHERE_PASSWORD>"
          datacenters = "<VSPHERE_DATACENTED>"
          [Network]
          public-network = "<VSPHERE_NETWORK>"
          [Labels]
          zone = ""
          region = ""
  targets:
  - clusterSelector:
      matchLabels:
        csi: vsphere
        cluster.x-k8s.io/cluster-name: 'vsphere-kubeadm-quickstart'
---
kind: Bundle
apiVersion: fleet.cattle.io/v1alpha1
metadata:
  name: vsphere-cloud-credentials
spec:
  resources:
  - content: |-
      apiVersion: v1
      kind: Secret
      type: Opaque
      metadata:
        name: vsphere-cloud-secret
        namespace: kube-system
      stringData:
        <VSPHERE_SERVER>.password: "<VSPHERE_PASSWORD>"
        <VSPHERE_SERVER>.username: "<VSPHERE_USER>"
  targets:
  - clusterSelector:
      matchLabels:
        cloud-provider: vsphere
        cluster.x-k8s.io/cluster-name: 'vsphere-kubeadm-quickstart'
----

* Erstellen Sie den vSphere-Cluster aus der Beispiel-ClusterClass
+
Beachten Sie, dass wir in diesem Beispiel https://kube-vip.io/[] kube-vip als Control Plane Load Balancer verwenden.
Die `+KUBE_VIP_INTERFACE+` wird verwendet, um die `+CONTROL_PLANE_IP+` im ARP-Modus zu binden. Je nach Betriebssystem und Netzwerkgerätekonfiguration müssen Sie diesen Wert entsprechend konfigurieren - zum Beispiel auf `+eth0+`.
Das statische Manifest `+kube-vip+` ist in die ClusterClass-Definition eingebettet. Weitere Informationen über die Erstellung eines statischen kube-vip-Manifests für Ihre eigenen ClusterClasses finden Sie in der offiziellen https://kube-vip.io/docs/installation/static/[Dokumentation].
+
[source,yaml]
----
apiVersion: cluster.x-k8s.io/v1beta2
kind: Cluster
metadata:
  labels:
    cni: calico
    cloud-provider: vsphere
    csi: vsphere
    cluster-api.cattle.io/rancher-auto-import: "true"
  name: 'vsphere-kubeadm-quickstart'
spec:
  clusterNetwork:
    pods:
      cidrBlocks:
      - 192.168.0.0/16
  topology:
    classRef: 
      name: vsphere-kubeadm-example
    version: v1.35.0
    controlPlane:
      replicas: 3
    workers:
      machineDeployments:
      - class: vsphere-kubeadm-example-worker
        name: md-0
        replicas: 1
    variables:
    - name: vSphereClusterIdentityName
      value: cluster-identity
    - name: vSphereTLSThumbprint
      value: <VSPHERE_THUMBPRINT>
    - name: vSphereDataCenter
      value: <VSPHERE_DATACENTER>
    - name: vSphereDataStore
      value: <VSPHERE_DATASTORE>
    - name: vSphereFolder
      value: <VSPHERE_FOLDER>
    - name: vSphereNetwork
      value: <VSPHERE_NETWORK>
    - name: vSphereResourcePool
      value: <VSPHERE_RESOURCE_POOL>
    - name: vSphereServer
      value: <VSPHERE_SERVER>
    - name: vSphereTemplate
      value: <VSPHERE_TEMPLATE>
    - name: controlPlaneIpAddr
      value: <CONTROL_PLANE_IP>
    - name: controlPlanePort
      value: 6443
    - name: sshKey
      value: <SSH_KEY>
    - name: kubeVIPInterface
      value: <KUBE_VIP_INTERFACE>
----

vSphere RKE2::
+
--
* Eine vSphere ClusterClass finden Sie unter den https://github.com/rancher/turtles/tree/main/examples/clusterclasses[Beispielen] Turtles.
+
Anwendungen wie der vSphere Cloud Provider, der vSphere CSI-Treiber und Calico CNI werden auf Downstream-Clustern installiert. Dies geschieht automatisch bei der Erstellung von Clustern durch gezielte Cluster mit bestimmten Bezeichnungen, wie `+cloud-provider: vsphere+`, `+csi: vsphere+` und `+cni: calico+`.
+
[tabs]
=======

CLI::
+
Eine vSphere RKE2 ClusterClass und zugehörige Anwendungen können mit dem Beispieltool angewendet werden:
+
[source,bash, subs="attributes"]
----
go run github.com/rancher/turtles/examples@{turtles_version} -r vsphere-rke2 | kubectl apply -f -
----

kubectl::
+
* Alternativ können Sie die vSphere RKE2 ClusterClass auch direkt mit kubectl anwenden:
+
[source,bash, subs="attributes"]
----
kubectl apply -f https://raw.githubusercontent.com/rancher/turtles/refs/tags/{turtles_version}/examples/clusterclasses/vsphere/rke2/clusterclass-rke2-example.yaml
----

* Zusätzlich muss der https://github.com/kubernetes/cloud-provider-vsphere[Cloud Provider] vSphere auf jedem Downstream-Cluster installiert werden, damit die Knoten korrekt initialisiert werden können.
Als Speicherlösung wird der https://github.com/kubernetes-sigs/vsphere-csi-driver[Container Storage Interface (CSI) Treiber für vSphere] verwendet.
Für dieses Beispiel werden wir https://docs.tigera.io/calico/latest/about/[Calico] als Standard-CNI installieren.
+
Wir können alle Anwendungen automatisch bei der Cluster-Erstellung installieren, indem wir die https://rancher.github.io/cluster-api-addon-provider-fleet/[Cluster API Add-on Provider Fleet] verwenden.
Zunächst müssen zwei `+HelmOps+` erstellt werden, die über Label-Selektoren auf den neuen Cluster angewendet werden.
+
[source,bash, subs="attributes"]
----
kubectl apply -f https://raw.githubusercontent.com/rancher/turtles/refs/tags/{turtles_version}/examples/applications/ccm/vsphere/helm-chart.yaml
kubectl apply -f https://raw.githubusercontent.com/rancher/turtles/refs/tags/{turtles_version}/examples/applications/cni/calico/helm-chart.yaml
----
+
Da der vSphere CSI-Treiber nicht in Helm verpackt ist, werden wir sein gesamtes Manifest in ein Fleet-Bundle aufnehmen, das auf den Downstream-Cluster angewendet wird.
+
[source,bash, subs="attributes"]
----
kubectl apply -f https://raw.githubusercontent.com/rancher/turtles/refs/tags/{turtles_version}/examples/applications/csi/vsphere/bundle.yaml
----
