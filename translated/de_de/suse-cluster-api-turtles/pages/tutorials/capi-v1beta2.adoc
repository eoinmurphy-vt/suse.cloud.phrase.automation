= Aufrüstung auf CAPI v1beta2

Die Version `+v0.26.0+` von Rancher Turtles verwendet den CAPI `+v1beta2+` Vertrag, der mit CAPI core v1.12.2 geliefert wird. Wenn Sie die CAPI-basierte Bereitstellung verwenden, sollten Sie vor dem Upgrade einige Dinge überprüfen.

Die Änderungen fallen in drei Bereiche:

* xref:#_clusterclass[Wie `+ClusterClass+` Ressourcen strukturiert sind].
* xref:#_cluster[Wie `+Cluster+` Topologie auf seine Klasse verweist].
* xref:#_rke2_provider_caprke2[Was sich beim RKE2-Anbieter geändert hat].

Lesen Sie die Abschnitte durch, die auf Ihre Einrichtung zutreffen, bevor Sie Änderungen vornehmen.

== Ein Hinweis zur Abwärtskompatibilität

CAPI v1.12 stellt weiterhin neben `+v1beta2+` auch `+v1beta1+` zur Verfügung. Vorhandene Ressourcen in Ihrem Cluster funktionieren auch ohne sofortige Migration weiter, da der API-Server transparent zwischen den beiden Versionen konvertiert. Das heißt, alle neuen Manifeste, die Sie schreiben, sollten auf `+v1beta2+` ausgerichtet sein, und Sie sollten planen, Ihre bestehenden Manifeste im Laufe der Zeit zu migrieren. Die API `+v1beta1+` soll in einer künftigen CAPI-Version (August 2026) entfernt werden, sobald das Ökosystem Zeit hatte, sich auf `+v1beta2+` zu einigen.

*Empfehlung*: Beurteilen Sie zunächst, ob Sie bereit sind, `+v1beta2+` zu übernehmen, und handeln Sie entsprechend. Wenn Sie sich dafür entscheiden, nicht sofort zu migrieren, erstellen Sie einen Plan für die Umstellung auf die neue API, um Unterbrechungen in der Zukunft zu vermeiden.

Außerdem stellt die Upstream-Community einen offiziellen Migrationsleitfaden zur Verfügung:

* https://cluster-api.sigs.k8s.io/developer/providers/migrations/v1.10-to-v1.11[CAPI v1.10 bis v1.11]: Dies umfasst alle Änderungen in der CAPI-Kernversion von `+v1beta1+` bis `+v1beta2+`.
* https://cluster-api.sigs.k8s.io/developer/providers/migrations/v1.11-to-v1.12[CAPI v1.11 bis v1.12]: Die meisten Änderungen in CAPI v1.12 können als geringfügig angesehen werden.

== ClusterClass

=== Referenzen der Vorlage

In `+v1beta1+` wurde das Feld, das auf die `+ClusterClass+` Vorlageobjekte verweist, `+ref+` genannt. Auf `+v1beta2+` heißt sie `+templateRef+`. Dies gilt für `+spec.infrastructure+`, `+spec.controlPlane+` und `+spec.controlPlane.machineInfrastructure+`.

Vorher:

[,yaml]
----
spec:
  infrastructure:
    ref:
      apiVersion: infrastructure.cluster.x-k8s.io/v1beta1
      kind: DockerClusterTemplate
      name: my-cluster-template
----

Danach:

[,yaml]
----
spec:
  infrastructure:
    templateRef:
      apiVersion: infrastructure.cluster.x-k8s.io/v1beta2
      kind: DockerClusterTemplate
      name: my-cluster-template
----

=== Bereich Arbeitnehmer

In `+v1beta1+` wurden Bootstrap- und Infrastrukturreferenzen für Maschinenbereitstellungen unter einem `+template+` Feld verschachtelt. Dieser Root-Schlüssel ist in `+v1beta2+` verschwunden. `+bootstrap+` und `+infrastructure+` sind nun direkte Kinder des Eintrags für die Maschinenbereitstellung.

Vorher:

[,yaml]
----
workers:
  machineDeployments:
    - class: default-worker
      template:
        bootstrap:
          ref:
            apiVersion: bootstrap.cluster.x-k8s.io/v1beta1
            kind: KubeadmConfigTemplate
            name: my-bootstrap-template
        infrastructure:
          ref:
            apiVersion: infrastructure.cluster.x-k8s.io/v1beta1
            kind: DockerMachineTemplate
            name: my-worker-template
----

Danach:

[,yaml]
----
workers:
  machineDeployments:
    - class: default-worker
      bootstrap:
        templateRef:
          apiVersion: bootstrap.cluster.x-k8s.io/v1beta2
          kind: KubeadmConfigTemplate
          name: my-bootstrap-template
      infrastructure:
        templateRef:
          apiVersion: infrastructure.cluster.x-k8s.io/v1beta2
          kind: DockerMachineTemplate
          name: my-worker-template
----

== Cluster

Die Ressource `+Cluster+` verweist nun auf `+ClusterClass+` über ein strukturiertes Objekt statt über einen Inline-String. In `+v1beta1+` wurden der Klassenname und der Namespace als `+spec.topology.class+` und `+spec.topology.classNamespace+` angegeben. In `+v1beta2+` werden beide Felder durch ein einziges `+spec.topology.classRef+` Objekt ersetzt.

Vorher:

[,yaml]
----
apiVersion: cluster.x-k8s.io/v1beta1
kind: Cluster
spec:
  topology:
    class: my-clusterclass
    classNamespace: capi-system
----

Danach:

[,yaml]
----
apiVersion: cluster.x-k8s.io/v1beta2
kind: Cluster
spec:
  topology:
    classRef:
      name: my-clusterclass
      namespace: capi-system
----

Der Rest der Spezifikation `+Cluster+` (Netzwerkeinstellungen, `+controlPlane.replicas+`, `+workers.machineDeployments+`, `+variables+`, `+version+`, usw.) bleibt unverändert.

== RKE2-Anbieter (CAPRKE2)

Mit dieser Version wird CAPRKE2 v0.23.1 ausgeliefert, eine Verbesserung gegenüber v0.22.1. Die `+v1beta2+` API für die RKE2-Bootstrap- und Control-Plane-Ressourcen wurde in v0.22.0 zusammen mit der Unterstützung für CAPI v1.11 eingeführt, und v0.23.0 vervollständigte diese Arbeit durch die Umstellung auf CAPI v1.12.2. Sowohl v0.22.1 als auch v0.23.1 sind Patch-Versionen, die zusätzlich zu den jeweiligen Minor-Versionen veröffentlicht wurden.

*Hinweis*: Lesen Sie die Provider-Dokumentation zu den API-Änderungen aus dem link:https://caprke2.docs.rancher.com/05_reference/01_api_versions.html#v1beta1-to-v1beta2[Buch]

=== Neue v1beta2 API

CAPRKE2 bietet nun eine eigene `+v1beta2+` API für `+RKE2ControlPlane+`, `+RKE2ControlPlaneTemplate+` und `+RKE2Config+`/`+RKE2ConfigTemplate+` an. Die API `+v1beta1+` ist weiterhin verfügbar, aber mehrere Felder, die in `+v1beta1+` als veraltet gekennzeichnet waren, sind nun in `+v1beta2+` entfernt.

=== Entfernte Felder in RKE2ControlPlane

Die folgenden Felder wurden von `+RKE2ControlPlane.spec+` in `+v1beta2+` entfernt:

* `+infrastructureRef+` - verwenden Sie stattdessen `+spec.machineTemplate.spec.infrastructureRef+`.
* `+nodeDrainTimeout+` - verwenden Sie stattdessen `+spec.machineTemplate.spec.deletion.nodeDrainTimeout+`.

=== Umbenannte Timeout-Felder

Die Timeout-Felder in `+RKE2ControlPlaneMachineTemplate+` wurden in ein Unterobjekt `+deletion+` verschoben und umbenannt. Sie haben auch den Typ von `+metav1.Duration+` auf `+int32+` geändert und erwarten nun Werte in Sekunden anstelle einer Dauer.

|===
| v1beta1 | v1beta2

| `+nodeDrainTimeout+`
| `+deletion.nodeDrainTimeoutSeconds+`

| `+nodeVolumeDetachTimeout+`
| `+deletion.nodeVolumeDetachTimeoutSeconds+`

| `+nodeDeletionTimeout+`
| `+deletion.nodeDeletionTimeoutSeconds+`
|===

Das Objekt `+RKE2ControlPlaneMachineTemplate+` benötigt jetzt auch ein Feld `+spec+`.

=== v1alpha1 Verwerfung

Die `+v1alpha1+` API wurde in CAPRKE2 v0.22.0 veraltet. Wenn Sie noch die Ressourcen von `+v1alpha1+` verwenden, ist jetzt ein guter Zeitpunkt, um zumindest auf `+v1beta1+` zu migrieren.

== Empfohlene Schritte vor der Aufrüstung

. Aktualisieren Sie alle `+ClusterClass+` Manifeste, die Sie pflegen: Benennen Sie `+ref+` in `+templateRef+` um, entfernen Sie den `+template+` Wrapper im Worker-Bereich und aktualisieren Sie die Core CAPI `+apiVersion+` Strings in `+v1beta2+`.
. Aktualisieren Sie alle `+Cluster+` Manifeste: Ersetzen Sie `+spec.topology.class+` und `+spec.topology.classNamespace+` durch `+spec.topology.classRef.name+` und `+spec.topology.classRef.namespace+`.
. Wenn Sie CAPRKE2 `+v1beta2+` -Ressourcen verwenden, entfernen Sie alle direkten `+infrastructureRef+` - oder `+nodeDrainTimeout+` -Felder aus `+RKE2ControlPlane.spec+` und verschieben Sie sie nach `+spec.machineTemplate.spec+`.
. In den aktualisierten Beispielen unter `+examples/clusterclasses/+` finden Sie vollständige, funktionierende Manifeste für jeden zertifizierten CAPI-Anbieter.
