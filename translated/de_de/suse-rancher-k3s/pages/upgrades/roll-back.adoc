= Rolling Back K3s
:page-languages: [en, ja, ko, zh]
:revdate: 2026-03-03
:page-revdate: {revdate}

Sie können die K3s-Kubernetes-Version nach einem Upgrade zurücksetzen, indem Sie eine Kombination aus K3s-Binär-Downgrade und Wiederherstellung des Datenspeichers verwenden. Rollback kann auf Clustern aller Art durchgeführt werden, einschließlich eines Single-Node-SQLite, eines externen Datenspeichers oder eines eingebetteten etcd. Wenn Sie zu einer früheren Kubernetes-Unterversion zurückkehren möchten, müssen Sie einen Snapshot des Datenspeichers mit der Kubernetes-Unterversion erstellen, zu der Sie zurückkehren möchten.

[WARNING]
====
Wenn Sie die Datenbank nicht wiederherstellen können, können Sie auch nicht auf eine frühere Unterversion zurückgehen.
====

== Wichtige Überlegungen

* *Backups:* Stellen Sie vor dem Upgrade sicher, dass Sie eine gültige Datenbank oder einen etcd-Snapshot von Ihrem Cluster haben, auf dem die ältere Version von K3s läuft. Ohne ein Backup ist ein Rollback unmöglich.
* *Potenzieller Datenverlust:* Das Skript `+k3s-killall.sh+` beendet zwangsweise K3s-Prozesse und kann zu Datenverlusten führen, wenn die Anwendungen nicht ordnungsgemäß beendet werden.
* *Besonderheiten der Version:* Überprüfen Sie immer die K3s und Komponentenversionen vor und nach dem Rollback.

== Zurücksetzen eines K3s-Clusters

[tabs]
======

SQLite::
+
--
Um ein Rollback eines K3s-Clusters bei Verwendung einer SQLite-Datenbank durchzuführen, ersetzen Sie die Datei `+.db+` durch die Kopie der Datei `+.db+`, die Sie beim Backup Ihrer Datenbank erstellt haben.
--

Eingebettetes etcd::
+
--
Führen Sie die folgenden Schritte aus, um einen K3s-Cluster bei Verwendung eines eingebetteten etcd zurückzusetzen: 

. Wenn der Cluster läuft und die Kubernetes-API verfügbar ist, beenden Sie die Arbeitslasten, indem Sie alle Knoten entleeren: 
+
[,bash]
----
kubectl drain --ignore-daemonsets --delete-emptydir-data <NODE-ONE-NAME> <NODE-TWO-NAME> <NODE-THREE-NAME> ...
----

. Halten Sie auf jedem Knoten den K3s-Dienst und alle laufenden Pod-Prozesse an:
+
[,bash]
----
k3s-killall.sh
----

. Setzen Sie auf jedem Knoten die K3s-Binärdatei auf die vorherige Version zurück, aber starten Sie K3s *nicht*.

* Clusters mit Internetzugang:

** Server-Knoten: 
+
[,bash]
----
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s INSTALL_K3S_VERSION=vX.Y.Zk3s1 INSTALL_K3S_EXEC="server" INSTALL_K3S_SKIP_START="true" sh -
----

** Agenten-Knoten:
+
[,bash]
----
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s INSTALL_K3S_VERSION=vX.Y.Zk3s1 INSTALL_K3S_EXEC="agent" INSTALL_K3S_SKIP_START="true" sh -
----

* Air-gapped Clusters:

** Laden Sie die Artefakte herunter und führen Sie das xref:installation/airgap.adoc#_install_k3s[Installationsskript] lokal aus. Fügen Sie die Umgebungsvariable `+INSTALL_K3S_SKIP_START="true"+` hinzu, wenn Sie das Installationsskript ausführen, um den Start von K3s zu verhindern.

. Starten Sie die Cluster-Wiederherstellung auf dem ersten Serverknoten oder dem Knoten ohne einen `+server:+` -Eintrag in seiner xref:installation/configuration.adoc[K3s-Konfigurationsdatei]. Weitere Informationen finden Sie in den xref:cli/etcd-snapshot.adoc#_snapshot_restore_steps[Schritten zur Wiederherstellung von Snapshots]: 
+
[,bash]
----
k3s server --cluster-reset --cluster-reset-restore-path=<PATH-TO-SNAPSHOT>
----
+
[WARNING]
====
Dadurch werden alle Daten im etcd-Datenspeicher überschrieben. Überprüfen Sie vor der Wiederherstellung die Integrität des Snapshots. Beachten Sie, dass die Wiederherstellung großer Snapshots sehr lange dauern kann.
====

. Starten Sie den K3s-Dienst auf dem ersten Serverknoten: 
+
[,bash]
----
systemctl start k3s
----
 
. Entfernen Sie auf den anderen Serverknoten das K3s-Datenbankverzeichnis:
+
[,bash]
----
rm -rf /var/lib/rancher/k3s/server/db
----
 
. Starten Sie den K3s-Dienst auf den anderen Serverknoten: 
+
[,bash]
----
systemctl start k3s
----

. Starten Sie den K3s-Dienst auf allen Agentenknoten:
+
[,bash]
----
systemctl start k3s
----

. Überprüfen Sie den Dienststatus des K3s mit `+systemctl status k3s+`. 
--

Externe Datenbank:: 
+
--
Gehen Sie wie folgt vor, um ein Rollback eines K3s-Clusters durchzuführen, wenn Sie eine externe Datenbank (z. B. PostgreSQL, MySQL) verwenden: 

. Wenn der Cluster läuft und die Kubernetes-API verfügbar ist, beenden Sie die Arbeitslasten, indem Sie alle Knoten entleeren: 
+
[,bash]
----
kubectl drain --ignore-daemonsets --delete-emptydir-data <NODE-ONE-NAME> <NODE-TWO-NAME> <NODE-THREE-NAME> ...
----
+
[NOTE]
====
Dieser Vorgang kann laufende Anwendungen stören.
====

. Halten Sie auf jedem Knoten den K3s-Dienst und alle laufenden Pod-Prozesse an:
+
[,bash]
----
k3s-killall.sh
----

. Stellen Sie einen Datenbank-Snapshot wieder her, der vor dem Upgrade von K3s erstellt wurde, und überprüfen Sie die Integrität der Datenbank. Wenn Sie beispielsweise PostgreSQL verwenden, führen Sie den folgenden Befehl aus:
+
[,bash]
----
pg_restore -U <DB-USER> -d <DB-NAME> <BACKUP-FILE>
----

. Führen Sie auf jedem Knoten ein Rollback der K3s-Binärdatei auf die vorherige Version durch. 

* Clusters mit Internetzugang:

** Server-Knoten:
+
[,bash]
----
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s INSTALL_K3S_VERSION=vX.Y.Zk3s1 INSTALL_K3S_EXEC="server" sh -
----

** Agenten-Knoten:
+
[,bash]
----
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s INSTALL_K3S_VERSION=vX.Y.Zk3s1 INSTALL_K3S_EXEC="agent" sh -
----

* Air-gapped Clusters:

** Laden Sie die Artefakte herunter und führen Sie das xref:installation/airgap.adoc#_install_k3s[Installationsskript] lokal aus. Überprüfen Sie die K3s-Version nach der Installation mit `+k3s --version+` und wenden Sie alle benutzerdefinierten Konfigurationen, die vor dem Upgrade verwendet wurden, erneut an. 

. Starten Sie den K3s-Dienst auf jedem Knoten:
+
[,bash]
----
systemctl start k3s
----

. Überprüfen Sie den Dienststatus des K3s mit `+systemctl status k3s+`.
--
======

== Überprüfung 

Überprüfen Sie nach dem Rollback die folgenden Punkte: 

* K3s-Version: `+k3s --version+` 
* Zustand des Kubernetes-Clusters: `+kubectl get nodes+` 
* Anwendungsfunktionalität. 
* Überprüfen Sie die K3s-Protokolle auf Fehler.
