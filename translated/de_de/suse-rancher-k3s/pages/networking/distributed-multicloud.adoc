= Verteilter Hybrid- oder Multicloud-Cluster
:page-languages: [en, ja, ko, zh]
:revdate: 2025-05-26
:page-revdate: {revdate}

Ein K3s-Cluster kann auch auf Knoten eingesetzt werden, die nicht über ein gemeinsames privates Netzwerk verfügen und nicht direkt miteinander verbunden sind (z. B. Knoten in verschiedenen öffentlichen Clouds). Hierfür gibt es zwei Möglichkeiten: die eingebettete k3s-Multicloud-Lösung und die Integration mit dem VPN-Anbieter `+tailscale+`.

[CAUTION]
====
Die Latenzzeit zwischen den Knoten wird zunehmen, da die externe Anbindung mehr Sprünge erfordert. Dies verringert die Netzwerkleistung und kann sich bei zu hohen Latenzzeiten auch auf den Zustand des Clusters auswirken.
====


[CAUTION]
====
Eingebettetes etcd wird bei dieser Art der Bereitstellung nicht unterstützt. Bei Verwendung von eingebettetem etcd müssen alle Serverknoten untereinander über ihre privaten IPs erreichbar sein. Die Agenten können über mehrere Netze verteilt sein, aber alle Server sollten sich am selben Ort befinden.
====


== Eingebettete k3s-Multicloud-Lösung

K3s verwendet wireguard, um ein VPN-Mesh für den Cluster-Verkehr aufzubauen. Jeder Knoten muss eine eindeutige IP haben, über die er erreicht werden kann (in der Regel eine öffentliche IP). Der K3s-Supervisor-Verkehr wird über einen Websocket-Tunnel abgewickelt, der Cluster-(CNI-)Verkehr über einen Wireguard-Tunnel.

Um diese Art der Bereitstellung zu ermöglichen, müssen Sie die folgenden Parameter auf den Servern hinzufügen:

[,bash]
----
--node-external-ip=<SERVER_EXTERNAL_IP> --flannel-backend=wireguard-native --flannel-external-ip
----

und auf Agenten:

[,bash]
----
--node-external-ip=<AGENT_EXTERNAL_IP>
----

wobei `+SERVER_EXTERNAL_IP+` die IP ist, über die wir den Serverknoten erreichen können, und `+AGENT_EXTERNAL_IP+` die IP ist, über die wir den Agentenknoten erreichen können. Beachten Sie, dass der Parameter `+K3S_URL+` config im Agenten die `+SERVER_EXTERNAL_IP+` verwenden sollte, um eine Verbindung herstellen zu können. Denken Sie daran, die xref:installation/requirements.adoc#_networking[Netzwerkanforderungen] zu überprüfen und den Zugriff auf die aufgelisteten Ports sowohl für interne als auch für externe Adressen zuzulassen.

Sowohl `+SERVER_EXTERNAL_IP+` als auch `+AGENT_EXTERNAL_IP+` müssen miteinander verbunden sein und sind normalerweise öffentliche IPs.

[IMPORTANT]
.Dynamische IPs
====
Wenn Knoten dynamische IPs zugewiesen werden und sich die IP ändert (z. B. in AWS), müssen Sie den Parameter `+--node-external-ip+` ändern, um die neue IP zu berücksichtigen. Wenn Sie K3s als Dienst ausführen, müssen Sie `+/etc/systemd/system/k3s.service+` ändern und dann ausführen:

[,bash]
----
systemctl daemon-reload
systemctl restart k3s
----
====

[#_integration_with_the_tailscale_vpn_provider_experimental]
== Integration mit dem Tailscale VPN-Anbieter (experimentell)

Verfügbar in v1.27.3, v1.26.6, v1.25.11 und neuer.

K3s kann mit https://tailscale.com/[Tailscale] integriert werden, so dass die Knoten den Tailscale VPN-Service nutzen, um ein Netz zwischen den Knoten aufzubauen.

Vor dem Einsatz von K3s müssen vier Schritte mit Tailscale durchgeführt werden:

. Melden Sie sich bei Ihrem Tailscale-Konto an
. Erzeugen Sie unter `+Settings > Keys+` einen Authentifizierungsschlüssel ($AUTH-KEY), der für alle Knoten in Ihrem Cluster wiederverwendet werden kann.
. Legen Sie den podCIDR fest, den der Cluster verwenden soll (standardmäßig `+10.42.0.0/16+`). Fügen Sie die CIDR (oder CIDRs für Dual-Stack) in Access controls mit der Stanza an:
+
[,yaml]
----
"autoApprovers": {
     "routes": {
         "10.42.0.0/16":        ["your_account@xyz.com"],
         "2001:cafe:42::/56": ["your_account@xyz.com"],
     },
 },
----

. Installieren Sie Tailscale auf Ihren Knotenpunkten:
+
[,bash]
----
curl -fsSL https://tailscale.com/install.sh | sh
----

Um K3s mit aktivierter Tailscale-Integration einzusetzen, müssen Sie den folgenden Parameter auf jedem Ihrer Knoten hinzufügen:

[,bash]
----
--vpn-auth="name=tailscale,joinKey=$AUTH-KEY"
----

oder geben Sie diese Informationen in einer Datei an und verwenden Sie den Parameter:

[,bash]
----
--vpn-auth-file=$PATH_TO_FILE
----

Wenn Sie einen eigenen Tailscale-Server haben (z. B. headscale), können Sie sich optional mit diesem verbinden, indem Sie `+,controlServerURL=$URL+` an die vpn-auth-Parameter anhängen

Anschließend können Sie den Server mit dem folgenden Befehl erstellen:

[,bash]
----
k3s server --token <token> --vpn-auth="name=tailscale,joinKey=<joinKey>" --node-external-ip=<TailscaleIPOfServerNode>
----

Nach der Ausführung dieses Befehls greifen Sie auf die Tailscale-Administrationskonsole zu, um den Tailscale-Knoten und das Subnetz zu genehmigen (falls nicht bereits durch autoApprovers genehmigt). 

Sobald der Server eingerichtet ist, verbinden Sie die Agenten über:

[,bash]
----
k3s agent --token <token> --vpn-auth="name=tailscale,joinKey=<joinKey>" --server https://<TailscaleIPOfServerNode>:6443 --node-external-ip=<TailscaleIPOfAgentNode>
----

Bestätigen Sie auch hier den Tailscale-Knoten und das Subnetz, wie Sie es für den Server getan haben.

Wenn Sie ACLs in Tailscale aktiviert haben, müssen Sie eine "Accept"-Regel hinzufügen, damit die Pods miteinander kommunizieren können. Unter der Annahme, dass der von Ihnen erstellte Authentifizierungsschlüssel die Tailscale-Knoten automatisch mit dem Tag `+testing-k3s+` versieht, sollte die Regel wie folgt aussehen:

[,yaml]
----
"acls": [
    {
        "action": "accept",
        "src":    ["tag:testing-k3s", "10.42.0.0/16"],
        "dst":    ["tag:testing-k3s:*", "10.42.0.0/16:*"],
    },
],
----

[CAUTION]
====
Wenn Sie vorhaben, mehrere K3s-Cluster über dasselbe Tailscale-Netzwerk zu betreiben, erstellen Sie bitte entsprechende https://tailscale.com/kb/1018/acls/[ACLs], um IP-Konflikte zu vermeiden, oder verwenden Sie für jeden Cluster unterschiedliche podCIDR-Subnetze.
====
