= k3s-Server
:page-languages: [en, ja, ko, zh]
:revdate: 2025-05-26
:page-revdate: {revdate}

In diesem Abschnitt erfahren Sie, wie Sie den K3s-Server konfigurieren.

Beachten Sie, dass auch auf Servern ein Agent läuft, so dass alle in der xref:cli/agent.adoc[Dokumentation`+k3s agent+` ] aufgeführten Konfigurationsoptionen auch auf Servern unterstützt werden.

Optionen sind auf dieser Seite als CLI-Flags dokumentiert, können aber auch als Konfigurationsdateioptionen übergeben werden. Weitere Informationen zur Verwendung von YAML-Konfigurationsdateien finden Sie in der Dokumentation zur xref:installation/configuration.adoc#_configuration_file[Konfigurationsdatei].

== Kritische Konfigurationswerte

Die folgenden Optionen müssen auf allen Servern des Clusters auf denselben Wert gesetzt werden. Andernfalls werden neue Server bei der Verwendung von eingebettetem etcd nicht in den Cluster aufgenommen, und bei der Verwendung eines externen Datenspeichers funktioniert der Cluster nicht richtig.

* `+--agent-token+`
* `+--cluster-cidr+`
* `+--cluster-dns+`
* `+--cluster-domain+`
* `+--disable-cloud-controller+`
* `+--disable-helm-controller+`
* `+--disable-network-policy+`
* `+--disable=servicelb+` _Hinweis: Andere Komponenten des Pakets können für jeden Server einzeln deaktiviert werden._
* `+--egress-selector-mode+`
* `+--embedded-registry+`
* `+--flannel-backend+`
* `+--flannel-external-ip+`
* `+--flannel-ipv6-masq+`
* `+--secrets-encryption+`
* `+--secrets-encryption-provider+`
* `+--service-cidr+`

|===
| Flagge | Standard | Beschreibung

| Cluster-Bereich

| `+--flannel-backend+` Wert
| "vxlan"
| Eine der Optionen "keine", "vxlan", "ipsec" (veraltet), "host-gw", "wireguard-native" oder "wireguard" (veraltet)

| `+--flannel-ipv6-masq+`
| "N/A"
| IPv6-Masquerading für Pods aktivieren

| `+--flannel-external-ip+`
| "N/A"
| Externe IP-Adressen des Knotens für Flannel-Verkehr verwenden

| `+--servicelb-namespace+` Wert
| "kube-system"
| Namespace der Pods für die Komponente servicelb

| `+--egress-selector-mode+` Wert
| "Agent"
a| Muss einer der folgenden sein: 

* deaktiviert: Der Apiserver verwendet keine Agententunnel zur Kommunikation mit Knoten. Erfordert, dass auf den Servern Agenten laufen und eine direkte Verbindung zum Kubelet auf den Agenten besteht, da der Apiserver sonst nicht in der Lage ist, auf Service-Endpunkte zuzugreifen oder kubectl exec und kubectl logs auszuführen.
* Agent: Der Apiserver verwendet Agententunnel, um mit den Knoten zu kommunizieren. Knoten erlauben die Tunnelverbindung von Loopback-Adressen. Erfordert, dass auf den Servern auch Agenten laufen, sonst kann der Apiserver nicht auf Dienstendpunkte zugreifen. Der historische Standard für k3s.
* pod: Der Apiserver verwendet Agententunnel für die Kommunikation mit Knoten und Dienstendpunkten und leitet Endpunktverbindungen an den richtigen Agenten weiter, indem er Knoten überwacht. Knoten erlauben die Tunnelverbindung von Loopback-Adressen oder einem ihrem Knoten zugewiesenen CIDR.
* cluster: Der Apiserver verwendet Agententunnel zur Kommunikation mit Knoten und Dienstendpunkten und leitet Endpunktverbindungen an den richtigen Agenten weiter, indem er Endpunkte beobachtet. Die Knoten erlauben die Tunnelverbindung von Loopback-Adressen oder dem konfigurierten Cluster-CIDR-Bereich.
