= Kanäle
:revdate: 2025-09-30
:page-revdate: {revdate}

{elemental_operator_name} ermöglicht die Anmeldung bei einem oder mehreren xref:references/managedosversionchannel-reference.adoc[ManagedOSVersionChannels], um automatisch eine Liste von xref:references/managedosversion-reference.adoc[ManagedOSVersionen] zu erstellen, die zur Erstellung neuer ISOs unter Verwendung eines xref:references/seedimage-reference.adoc[SeedImage] oder zur Aktualisierung bestehender {elemental-product-name} -Knoten auf neue Betriebssystemversionen unter Verwendung des xref:references/managedosimage-reference.adoc[ManagedOSImage] genutzt werden können.

Ein Kanal wird normalerweise als OCI-Container-Image verteilt, aber es ist auch möglich, direkt auf die URI einer JSON-Datei zu verweisen, die eine Liste von `+ManagedOSVersion+` enthält. Beachten Sie, dass es am besten ist, die Kanäle mit Bildern zu verteilen, damit die Verteilung mit allen anderen Bildern, die von {elemental_operator_name} benötigt werden, konsistent ist. Dies kann z. B. beim Einsatz in einer Airgapped-Umgebung von Vorteil sein.

Ein benutzerdefinierter Syncer bietet mehr Flexibilität bei der Erfassung von `+ManagedOSVersion+`, da er benutzerdefinierte Befehle mit benutzerdefinierten Bildern erlaubt. Diese Art von Syncer erlaubt es, einen bestimmten Befehl mit Argumenten und Umgebungsvariablen in einem benutzerdefinierten Image auszuführen und eine json-Datei an `+/data/output+` auszugeben. == Lebenszyklus von Kanälen und bewährte Verfahren

Sobald eine neue `+ManagedOSVersionChannel+` erstellt ist, wird {elemental_operator_name} die vom Kanal bereitgestellte JSON-Liste regelmäßig synchronisieren und in eine neue `+ManagedOSVersions+` konvertieren. +
Alle synchronisierten `+ManagedOSVersions+` sind Eigentum der `+ManagedOSVersionChannel+`. Die Löschung von `+ManagedOSVersionChannel+` führt zur Löschung aller `+ManagedOSVersions+` auf der Kaskade.

Beachten Sie, dass `+ManagedOSVersionChannel+` die automatische Bereinigung von nicht mehr synchronen `+ManagedOSVersions+` unterstützt, wenn die Option `+ManagedOSVersionChannel.spec.deleteNoLongerInSyncVersions+` aktiviert ist.

Wenn eine `+ManagedOSVersion+` zur Löschung vorgesehen ist, stellt ein Finalizer sicher, dass keine aktive Referenz auf einer `+ManagedOSImage+` vorhanden ist.

Wenn eine `+ManagedOSVersion+` nicht gelöscht werden kann, können Sie herausfinden, von welchen Ressourcen sie referenziert wird:

[,shell]
----
kubectl -n fleet-default get managedosimages -l elemental.cattle.io/managed-os-version-name=my-deleted-os-version
----

Bei der Verwendung mehrerer Kanäle ist es wichtig, eine angemessene Benennungsstrategie zu verfolgen, um immer eine schnelle, für Menschen lesbare Referenz auf dem eigenen `+ManagedOSVersions+` zu haben. +
Es wird empfohlen, jeden Kanal als zu benennen: `+{BaseOS}-{BaseOSVersion}-{Flavor}+`.

Dies sollte es dem Benutzer ermöglichen, den `+ManagedOSVersion+` Namen als die spezifische {elemental-product-name} Build-Version des Images zu verwenden, während ein Verweis auf das Basisbetriebssystem und die Basisbetriebssystemversion aus dem übergeordneten Channel beibehalten wird. +

== Eigene Channels erstellen

Die einzige Voraussetzung für einen eigenen Syncer ist, dass er eine JSON-Datei an `+/data/output+` ausgibt und die korrekte JSON-Struktur beibehält.

Die Datei ist ein JSON-Array mit ISO- und Container-Einträgen.
Jeder Eintrag im Array wird 1:1 einem xref:references/managedosversion-reference.adoc[ManagedOSVersion-Objekt] zugeordnet.

`+"type": "iso"+` Einträge müssen eine bootfähige {elemental-product-name} ISO enthalten und werden von xref:references/seedimage-reference.adoc[SeedImages] verwendet, während `+"type": "container"+` Einträge von xref:references/managedosimage-reference.adoc[ManagedOSImage] für {elemental-product-name} Upgrades verwendet werden.

Im Zweifelsfall kann das Projekt https://github.com/rancher-sandbox/elemental-channels[elemental-channels] als Referenzimplementierung für die Erstellung und Pflege eigener Kanäle verwendet werden.

Achten Sie bei der Erstellung neuer Einträge auf die von Ihnen gewählte Namensstrategie, um Kollisionen mit anderen Kanälen zu vermeiden, da diese unter Umständen verschiedene `+ManagedOSVersion+` mit demselben Namen synchronisieren.
Eine bewährte Praxis ist die Verwendung der Konvention: `+\{Flavor}-\{Version}-\{Type}+`
