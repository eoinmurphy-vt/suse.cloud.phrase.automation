= Überschreiben von Schwellenwertargumenten für die Überwachung über Kubernetes-Anmerkungen
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: SUSE Observabilität

== Übersicht

SUSE Observability stellt xref:/use/alerting/k8s-monitors.adoc[standardmäßig Monitore] zur Verfügung, mit denen gängige Probleme, die in einem Kubernetes-Cluster auftreten können, überwacht werden können. Diese Monitore arbeiten mit bestimmten Standardargumenten, die für die meisten Anwendungsfälle geeignet sind, aber manchmal ist es notwendig, das Verhalten anzupassen, indem einige dieser Standardargumente wie `+threshold+` oder `+failureState+` überschrieben werden.
Der Mechanismus zur Deklaration der Überschreibungen erfolgt über Kubernetes-Ressourcen-Annotationen, die angeben, für welchen Monitor und welche Komponente sie gelten sollen. Wir könnten zum Beispiel den `+failureState+` für den `+Available service endpoints+` Monitor für einen bestimmten Dienst überschreiben, wenn wir einen `+CRITICAL+` Zustand signalisieren wollen, wenn er ausfällt, anstatt den Standard `+DEVIATING+`.

Als Beispiel werden die Schritte die Argumente für den `+Available service endpoints+` Monitor von Kubernetes HTTP-Diensten überschreiben.

== Wie erstelle ich meinen Kommentar?

Die Schlüssel der Überschreibungsanmerkungen für SUSE Observability-Monitore folgen der folgenden Konvention:

----
monitor.${owner}.stackstate.io/${monitorShorName}
----

Die Eigenschaft `+owner+` gibt an, wer einen solchen Monitor erstellt hat, für die Standardmonitore lautet sie `+kubernetes-v2+`, und die Eigenschaft `+monitorShorName+` gibt die ID des Monitors an und kann aus der Eigenschaft `+identifier+` eines Monitors extrahiert werden, die beim Auflisten oder Überprüfen von Monitoren aus der Kli gelesen werden kann.

----
sts monitor list

ID              | STATUS  | IDENTIFIER                                                                          | NAME                                        | FUNCTION ID     | TAGS
8051105457030   | ENABLED | urn:stackpack:kubernetes-v2:shared:monitor:kubernetes-v2:service-available-endpoint | Available service endpoints                 | 233276809885571 | [services]
----

In unserem Beispiel ist der Bezeichner `+urn:stackpack:kubernetes-v2:shared:monitor:kubernetes-v2:service-available-endpoint+` und `+monitorShorName+` entspricht dem allerletzten Segment wie in `+service-available-endpoint+`, daher lautet der Anmerkungsschlüssel:

[,bash]
----
monitor.kubernetes-v2.stackstate.io/service-available-endpoint
----

Die Nutzlast der Anmerkung ist ein JSON-Objekt, in dem die folgenden optionalen Argumente definiert werden können:

* `+threshold+`Optional: Ein numerischer Schwellenwert, mit dem verglichen wird.
* `+failureState+`: fakultativ. Entweder "KRITISCH" oder "ABWEICHEND". "CRITICAL" wird in SUSE Observability als gelesen und "DEVIATING" als orange angezeigt, um den unterschiedlichen Schweregrad zu kennzeichnen.
* `+enabled+`: fakultativ. Boolescher Wert, der bestimmt, ob der Monitor einen Gesundheitszustand für diese Komponente erzeugen würde.

Die vollständige Anmerkung würde dann wie folgt aussehen

[,bash]
----
    monitor.kubernetes-v2.stackstate.io/service-available-endpoint: |-
      {
        "threshold": 0.0,
        "failureState": "CRITICAL",
        "enabled": true
      }
----

== Erstellen einer Überschreibung für einen benutzerdefinierten Monitor

Jeder benutzerdefinierte Schwellenwertmonitor, der mit Hilfe der Anleitung unter xref:/use/alerting/k8s-add-monitors-cli.adoc[Hinzufügen eines Schwellenwertmonitors zu Komponenten unter Verwendung der CLI] erstellt wurde, eignet sich für die Überschreibung von Argumenten, da, wie xref:/use/alerting/k8s-add-monitors-cli.adoc#_write_the_outline_of_the_monitor[das Beispiel zeigt,] ein Bezeichner für einen benutzerdefinierten Monitor als ``+urn:custom:monitor:{monitorShortName}``strukturiert ist +und der Überschreibungsschlüssel für einen solchen Bezeichner erwartet wird

[,bash]
----
monitor.custom.stackstate.io/${monitorShortName}
----

Im Beispiel wird der Bezeichner `+urn:custom:monitor:deployment-has-replicas+` verwendet, so dass der Anmerkungsschlüssel wie folgt lautet

[,bash]
----
monitor.custom.stackstate.io/deployment-has-replicas
----

Und die vollständige Anmerkung würde wie folgt aussehen

[,bash]
----
    monitor.custom.stackstate.io/deployment-has-replicas: |-
      {
        "threshold": 0.0,
        "failureState": "CRITICAL"
        "enabled": true
      }
----
