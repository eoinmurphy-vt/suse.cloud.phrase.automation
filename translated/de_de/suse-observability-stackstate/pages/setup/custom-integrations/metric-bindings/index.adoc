= Hinzufügen von benutzerdefinierten Diagrammen zu Komponenten
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: SUSE Observabilität

== Übersicht

{stackstate-product-name} bietet bereits standardmäßig viele metrische Diagramme für die meisten Arten von Komponenten, die Kubernetes-Ressourcen darstellen. Zusätzliche metrische Diagramme können bei Bedarf zu jedem Komponentensatz hinzugefügt werden. Beim Hinzufügen von Metriken zu Komponenten gibt es zwei Möglichkeiten:

. Die Metriken werden bereits von {stackstate-product-name} gesammelt, werden aber standardmäßig nicht auf einer Komponente visualisiert
. Die Metriken werden von {stackstate-product-name} noch nicht erfasst und sind daher noch nicht verfügbar.

Für Option 1 werden Sie in den folgenden Schritten angewiesen, eine Metrikbindung zu erstellen, die {stackstate-product-name} so konfiguriert, dass eine bestimmte Metrik zu einer bestimmten Gruppe von Komponenten hinzugefügt wird.

Bei Option 2 stellen Sie sicher, dass die Metriken in {stackstate-product-name} verfügbar sind, indem Sie sie mit dem xref:/use/metrics/k8s-prometheus-remote-write.adoc[Prometheus-Protokoll für Fernschreiben] an {stackstate-product-name} senden. Fahren Sie mit dem Hinzufügen von Diagrammen für die Metriken zu den Komponenten fort, NUR nachdem Sie sichergestellt haben, dass die Metriken verfügbar sind.

== Erstellen einer metrischen Bindung

. <<_create_an_outline_of_the_metric_binding,Erstellen Sie eine Gliederung der metrischen Bindung>>
. <<_select_the_components_to_bind_to,Wählen Sie die Komponenten aus, an die gebunden werden soll>>

Die Schritte fügen zum Beispiel eine Metrikbindung für die `+Replica counts+` von Kubernetes-Bereitstellungen hinzu. Diese metrische Bindung existiert bereits standardmäßig in {stackstate-product-name}.

=== Erstellen Sie eine Gliederung der metrischen Bindung

Öffnen Sie die YAML-Datei `+metricbindings.yaml+` in Ihrem bevorzugten Code-Editor, um sie im Verlauf dieser Anleitung zu ändern.  Sie können die CLI verwenden, um xref:/setup/custom-integrations/develop.adoc#_test_your_stackpack[Ihr StackPack zu testen].

----
- _type: MetricBinding
  chartType: line
  enabled: true
  identifier: urn:stackpack:my-stackpack:shared:metric-binding:node-memory-bytes-available-scheduling
  layout:
    metricPerspective:
      section: Resources
      tab: Kubernetes Node
  name: Memory available for scheduling (Custom)
  priority: medium
  queries:
  - alias: ${cluster_name} - ${node}
    expression: max_over_time(kubernetes_state_node_memory_allocatable{cluster_name="${tags.cluster-name}", node="${name}"}[${__interval}])
  scope: (label = "stackpack:kubernetes" and type = "node")
  unit: bytes(IEC)
----

Der Abschnitt "Abfragen und Umfang" wird in den nächsten Schritten ausgefüllt. Beachten Sie, dass die verwendete Einheit `+short+` ist, die einfach einen numerischen Wert wiedergibt. Falls Sie sich über die Einheit der Metrik noch nicht sicher sind, können Sie sie offen lassen und sich beim Schreiben der PromQL-Abfrage für die richtige Einheit entscheiden.

=== Wählen Sie die Komponenten aus, an die gebunden werden soll

Speichern Sie eine Ansicht der xref:/use/views/k8s-topology-perspective.adoc[Topologieperspektive] und verwenden Sie die Filter (Filters -> Topology -> Switch to STQL), um die Komponenten abzufragen, die die neue Metrik anzeigen sollen. Die gebräuchlichsten Felder für die Auswahl der Topologie für metrische Bindungen sind `+type+` für den Komponententyp und `+label+` für die Auswahl aller Labels. Zum Beispiel für die Einsätze:

----
type = "deployment" and label = "stackpack:kubernetes"
----

Der Typ-Filter wählt alle Bereitstellungen aus, während der Label-Filter nur Komponenten auswählt, die vom Kubernetes-Stackpack erstellt wurden (Label-Name ist `+stackpack+` und Label-Wert ist `+kubernetes+`). Letzteres kann auch weggelassen werden, um das gleiche Ergebnis zu erzielen.  Alle xref:/develop/reference/k8sTs-stql_reference.adoc#_component_filters[STQL-Abfrage-Komponentenfilter] können zum Filtern verwendet werden.

Wechseln Sie in den erweiterten Modus, um die resultierende Topologieabfrage zu kopieren und in das Feld `+scope+` der Metrikbindung einzufügen.

[NOTE]
====
Metrische Bindungen unterstützen nur die Abfragefilter. Abfragefunktionen wie `+withNeighborsOf+` werden nicht unterstützt und können nicht verwendet werden.
====


=== Schreiben Sie die PromQL-Abfrage

Rufen Sie den xref:/use/metrics/k8sTs-explore-metrics.adoc[Metrik-Explorer] Ihrer {stackstate-product-name} -Instanz auf, http://your-instance/#/metrics,, und verwenden Sie ihn zur Abfrage der gewünschten Metrik. Der Explorer verfügt über eine Autovervollständigung für Metriken, Labels, Labelwerte, aber auch PromQL-Funktionen und Operatoren, um Ihnen zu helfen. Beginnen Sie mit einer kurzen Zeitspanne von z. B. einer Stunde, um die besten Ergebnisse zu erzielen.

Für die Gesamtzahl der Replikate verwenden Sie die Metrik `+kubernetes_state_deployment_replicas+`. Um die Metrikdiagramme der Zeitreihendaten anzuzeigen, erweitern Sie die Abfrage, um eine Aggregation mit dem Parameter `+${__interval}+` durchzuführen:

----
max_over_time(kubernetes_state_deployment_replicas[${__interval}])
----

In diesem speziellen Fall sollten Sie `+max_over_time+` verwenden, um sicherzustellen, dass das Diagramm immer die höchste Anzahl von Replikaten zu einem bestimmten Zeitpunkt anzeigt. Bei längeren Zeiträumen wird ein kurzer Einbruch der Wiederholungen nicht angezeigt. Um die geringste Anzahl von Replikaten hervorzuheben, verwenden Sie stattdessen `+min_over_time+`.

Kopieren Sie die Abfrage in die Eigenschaft `+expression+` des ersten Eintrags im Feld `+queries+` der metrischen Bindung. Verwenden Sie `+Total replicas+` als Alias. damit es in der Diagrammlegende angezeigt wird.

[NOTE]
====
In {stackstate-product-name} bestimmt die Größe des Metrikdiagramms automatisch die Granularität der im Diagramm angezeigten Metrik. PromQL-Abfragen können so angepasst werden, dass sie dieses Verhalten optimal ausnutzen, um ein repräsentatives Diagramm für die Metrik zu erhalten. xref:/setup/custom-integrations/metric-bindings/writing-promql.adoc[Das Schreiben von PromQL für Diagramme] erklärt dies im Detail.
====


=== Binden Sie die richtigen Zeitreihen an jede Komponente

Der metrische Datenfluss mit allen ausgefüllten Feldern:

----
_type: MetricBinding
chartType: line
enabled: true
tags: {}
unit: short
name: Replica counts
priority: MEDIUM
identifier: urn:stackpack:my-stackpack:metric-binding:my-deployment-replica-counts
queries:
  - expression: max_over_time(kubernetes_state_deployment_replicas[${__interval}])
    alias: Total replicas
scope: type = "deployment" and label = "stackpack:kubernetes"
----

Die Erstellung in {stackstate-product-name} und die Anzeige des Diagramms "Anzahl der Replikate" in einer Verteilungskomponente führt zu einem unerwarteten Ergebnis. Das Diagramm zeigt die Anzahl der Replikate für alle Bereitstellungen. Logischerweise würde man nur eine Zeitreihe erwarten: die Anzahl der Replikate für diesen speziellen Einsatz.

image::k8s/k8s-replica-counts-without-binding.png[Das falsche Diagramm für einen einzelnen Einsatz, it shows the replica count for all deployments]

Um dies zu beheben, machen Sie die PromQL-Abfrage spezifisch für eine Komponente unter Verwendung von Informationen aus der Komponente. Filtern Sie nach genügend metrischen Bezeichnungen, um nur die spezifischen Zeitreihen für die Komponente auszuwählen. Dies ist die "Bindung" der richtigen Zeitreihe an die Komponente. Für alle, die Erfahrung mit der Erstellung von Grafana-Dashboards haben, ist dies ähnlich wie ein Dashboard mit Parametern, die in Abfragen auf dem Dashboard verwendet werden. Ändern wir die Abfrage in der metrischen Bindung wie folgt:

----
max_over_time(kubernetes_state_deployment_replicas{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", deployment="${name}"}[${__interval}])
----

image::k8s/k8s-replica-counts-with-binding.png[Nach dem Hinzufügen der parametrisierten Filter sieht das resultierende Diagramm wie erwartet aus, only one time series for this component]

Die PromQL-Abfrage filtert jetzt nach drei Bezeichnungen, `+cluster_name+`, `+namespace+` und `+deployment+`. Anstatt einen tatsächlichen Wert für diese Bezeichnungen anzugeben, wird eine variable Referenz auf Felder der Komponente verwendet. In diesem Fall werden die Etiketten `+cluster-name+` und `+namespace+` verwendet, auf die mit `+${tags.cluster-name}+` und `+${tags.namespace}+` verwiesen wird. Weiterhin wird der Komponentenname mit `+${name}+` referenziert.

Unterstützte Variablenreferenzen sind:

* Jedes Komponentenetikett, das mit `+${tags.<label-name>}+`
* Der Komponentenname, unter Verwendung von `+${name}+`

image::k8s/k8s-carts-highlights.png[Seite "Komponentenhervorhebungen", auf der die Beschriftungen und der Komponentenname angezeigt werden (beide sind rot hervorgehoben)]

[NOTE]
====
Der Clustername, der Namespace und eine Kombination aus Komponententyp und -name reichen in der Regel aus, um die Metriken für eine bestimmte Komponente aus Kubernetes auszuwählen. Diese oder ähnliche Kennzeichnungen sind in der Regel auf den meisten Messinstrumenten und Komponenten vorhanden.
====


== Fortgeschrittene

=== Mehr als eine Zeitreihe in einem Diagramm

[NOTE]
====
Es gibt nur eine Einheit für eine metrische Bindung (sie wird auf der y-Achse des Diagramms aufgetragen). Daher sollten Sie nur Abfragen, die Zeitreihen mit der gleichen Einheit ergeben, in einer metrischen Bindung kombinieren. Manchmal ist es möglich, die Einheit umzurechnen. Beispielsweise kann die CPU-Nutzung in Millicores oder Cores angegeben werden. Millicores können in Cores umgewandelt werden, indem sie wie folgt mit 1000 multipliziert werden `+(<original-query>) * 1000+`.
====


Es gibt zwei Möglichkeiten, mehr als eine Zeitreihe in einer einzigen metrischen Bindung und damit in einem einzigen Diagramm zu erhalten:

. Schreiben Sie eine PromQL-Abfrage, die mehrere Zeitreihen für eine einzelne Komponente zurückgibt
. Weitere PromQL-Abfragen zur Metrikbindung hinzufügen

Für die erste Option wird im xref:/setup/custom-integrations/metric-bindings/index.adoc#_using_metric_labels_in_aliases[nächsten Abschnitt] ein Beispiel gegeben. Die zweite Option kann für den Vergleich verwandter Metriken nützlich sein. Einige typische Anwendungsfälle:

* Vergleich der gesamten Replikate mit den gewünschten und verfügbaren
* Ressourcennutzung: Grenzen, Anforderungen und Nutzung in einem einzigen Diagramm

Um weitere Abfragen zu einer metrischen Bindung hinzuzufügen, wiederholen Sie einfach die xref:/setup/custom-integrations/metric-bindings/index.adoc#_steps[Schritte] 3. und 4. und fügen die Abfrage als zusätzlichen Eintrag in die Liste der Abfragen ein. Für die Anzahl der Bereitstellungsreplikate gibt es mehrere verwandte Metriken, die in das gleiche Diagramm aufgenommen werden können:

----
- _type: MetricBinding
  chartType: line
  enabled: true
  tags: {}
  unit: short
  name: Replica counts
  priority: MEDIUM
  identifier: urn:stackpack:my-stackpack:metric-binding:my-deployment-replica-counts
  queries:
    - expression: max_over_time(kubernetes_state_deployment_replicas{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", deployment="${name}"}[${__interval}])
      alias: Total replicas
    - expression: max_over_time(kubernetes_state_deployment_replicas_available{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}",  deployment="${name}"}[${__interval}])
      alias: Available - ${cluster_name} - ${namespace} - ${deployment}
    - expression: max_over_time(kubernetes_state_deployment_replicas_unavailable{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}",  deployment="${name}"}[${__interval}])
      alias: Unavailable - ${cluster_name} - ${namespace} - ${deployment}
    - expression: min_over_time(kubernetes_state_deployment_replicas_desired{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}",  deployment="${name}"}[${__interval}])
      alias: Desired - ${cluster_name} - ${namespace} - ${deployment}
  scope: type = "deployment" and label = "stackpack:kubernetes"
----

image::k8s/k8s-replica-counts-multiple-timeseries.png[Metrikbindung mit mehreren Metriken]

=== Metrische Bezeichnungen in Aliasen verwenden

Wenn eine einzelne Abfrage mehrere Zeitreihen pro Komponente liefert, wird dies als mehrere Linien im Diagramm angezeigt. In der Legende werden sie jedoch alle denselben Alias verwenden. Um den Unterschied zwischen den verschiedenen Zeitreihen zu erkennen, kann der Alias Verweise auf die metrischen Bezeichnungen in der Syntax `+${label}+` enthalten. Hier ist zum Beispiel eine Metrikbindung für die Metrik "Container-Neustarts" auf einem Pod, wobei zu beachten ist, dass ein Pod mehrere Container haben kann:

----
type: MetricBinding
chartType: line
enabled: true
id: -1
identifier: urn:stackpack:my-stackpack:metric-binding:my-pod-restart-count
name: Container restarts
priority: MEDIUM
queries:
- alias: Restarts - ${container}
  expression: max by (cluster_name, namespace, pod_name, container) (kubernetes_state_container_restarts{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", pod_name="${name}"})
scope: (label = "stackpack:kubernetes" and type = "pod")
unit: short
----

Beachten Sie, dass die `+alias+` auf die `+container+` Bezeichnung der Metrik verweist. Vergewissern Sie sich, dass die Beschriftung im Abfrageergebnis vorhanden ist. Wenn die Beschriftung fehlt, wird `+${container}+` als wörtlicher Text wiedergegeben, um die Fehlersuche zu erleichtern.

=== Layouts

Jede Komponente kann mit verschiedenen Technologien oder Protokollen wie k8s, Netzwerken, Laufzeitumgebungen (z. B. JVM), Protokollen (HTTP, AMQP) usw. verbunden sein.
Folglich kann für jede Komponente eine Vielzahl verschiedener Metriken angezeigt werden. Zur besseren Lesbarkeit kann {stackstate-product-name} diese Diagramme in Registerkarten und Abschnitte einteilen.
Um ein Diagramm (`+MetricBinding+`) innerhalb einer bestimmten Registerkarte oder eines Abschnitts anzuzeigen, müssen Sie die Layout-Eigenschaft konfigurieren.
Jede MetricsBinding ohne angegebenes Layout wird in einer Registerkarte und einem Abschnitt namens `+Other+` angezeigt.
