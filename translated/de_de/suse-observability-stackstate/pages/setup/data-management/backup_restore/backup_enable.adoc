= Aktivieren von Backups
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: SUSE Observability Selbstgehostet

[NOTE]
====
Diese Seite gilt für {stackstate-product-name} v2.7.0 oder neuer.
====

== Übersicht

{stackstate-product-name} verfügt über einen integrierten Backup-Mechanismus, der so konfiguriert werden kann, dass Backups in AWS S3, Azure Blob Storage oder einem Kubernetes Persistent Volume gespeichert werden.

Die meisten Backups werden mit einem einzigen Helm-Wert `+global.backup.enabled+` aktiviert. Einstellungssicherungen sind immer aktiviert, verhalten sich aber je nach den folgenden Werten unterschiedlich:

=== Sicherungsbereich

Die folgenden Daten können automatisch gesichert werden:

* *Einstellungen* (StackPacks, Monitore, Ansichten, Token) - immer aktiviert:
** Wenn `+global.backup.enabled+` `+true+` ist, werden die Backups über MinIO in Ihrem konfigurierten Speicher-Backend gespeichert.
** Wenn `+global.backup.enabled+` `+false+` ist: Backups werden in einem dedizierten Kubernetes Persistent Volume gespeichert, wodurch MinIO umgangen wird.
* *Topologie-Daten und Einstellungen* in StackGraph gespeichert - aktiviert, wenn `+global.backup.enabled+` ist `+true+`
* *Metriken*, die in {stackstate-product-name}'s Victoria Metrics Instanz(en) gespeichert sind - aktiviert, wenn `+global.backup.enabled+` ist `+true+`
* *Telemetriedaten* werden in der Elasticsearch-Instanz von {stackstate-product-name} gespeichert - aktiviert, wenn `+global.backup.enabled+` ist `+true+`
* *OpenTelemetry-Daten*, die in der ClickHouse-Instanz von {stackstate-product-name} gespeichert sind - aktiviert, wenn `+global.backup.enabled+` ist `+true+`

=== Speicheroptionen

Backups verwenden https://min.io/[MinIO (min.io)] als S3-kompatibles Gateway zu Ihrem Speicher-Backend. MinIO wird automatisch durch das Helm-Diagramm bereitgestellt, wenn Backups aktiviert sind.

Die integrierte MinIO-Instanz kann so konfiguriert werden, dass die Backups an drei Orten gespeichert werden:

* xref:/setup/data-management/backup_restore/backup_enable.adoc[AWS S3]
* xref:/setup/data-management/backup_restore/backup_enable.adoc[Azure Blob-Speicher]
* xref:/setup/data-management/backup_restore/backup_enable.adoc[Kubernetes-Speicher]

== Aktivieren von Backups

==== Verwendung separater S3-Buckets

Um geplante Sicherungen in separaten AWS S3-Buckets (eines pro Datenspeicher) zu aktivieren, fügen Sie das folgende YAML-Fragment in die Helm `+values.yaml+` -Datei ein, die zur Installation von {stackstate-product-name} verwendet wird:

[,yaml]
----
global:
  backup:
    enabled: true
backup:
  stackGraph:
    bucketName: AWS_STACKGRAPH_BUCKET
  elasticsearch:
    bucketName: AWS_ELASTICSEARCH_BUCKET
  configuration:
    bucketName: AWS_CONFIGURATION_BUCKET
victoria-metrics-0:
  backup:
    bucketName: AWS_VICTORIA_METRICS_BUCKET
victoria-metrics-1:
  backup:
    bucketName: AWS_VICTORIA_METRICS_BUCKET
clickhouse:
  backup:
    bucketName: AWS_CLICKHOUSE_BUCKET
minio:
  accessKey: YOUR_ACCESS_KEY
  secretKey: YOUR_SECRET_KEY
  s3gateway:
    enabled: true
    accessKey: AWS_ACCESS_KEY
    secretKey: AWS_SECRET_KEY
----

Ersetzen Sie die folgenden Werte:

* `+YOUR_ACCESS_KEY+` und `+YOUR_SECRET_KEY+` sind die Anmeldedaten, die zur Sicherung des MinIO-Systems verwendet werden. Diese Anmeldedaten werden auf dem MinIO-System festgelegt und von den automatischen Sicherungs- und Wiederherstellungsaufträgen verwendet. Sie sind auch erforderlich, wenn Sie manuell auf das MinIO-System zugreifen möchten.
 ** DEIN_ZUGRIFFSSCHLÜSSEL sollte 5 bis 20 alphanumerische Zeichen enthalten.
 ** YOUR_SECRET_KEY sollte 8 bis 40 alphanumerische Zeichen enthalten.
* `+AWS_ACCESS_KEY+` und `+AWS_SECRET_KEY+` sind die AWS-Anmeldeinformationen für den IAM-Benutzer, der Zugriff auf die S3-Buckets hat, in denen die Backups gespeichert werden. Siehe unten für die Berechtigungsrichtlinie, die diesem Benutzer zugewiesen werden muss.
* `+AWS_*_BUCKET+` sind die Namen der S3-Buckets, in denen die Backups gespeichert werden sollen.
+
[Note]
====
Die Namen von AWS S3-Buckets sind für ganz AWS global. Daher werden die S3-Buckets mit dem Standardnamen (`+sts-elasticsearch-backup+`, `+sts-configuration-backup+`, `+sts-stackgraph-backup+`, `+sts-victoria-metrics-backup+` und `+sts-clickhouse-backup+` ) wahrscheinlich nicht verfügbar sein.
====

==== Verwendung eines einzigen S3-Buckets mit Präfixen

Anstatt separate Buckets für jeden Datenspeicher zu verwenden, können Sie einen einzigen S3-Bucket mit unterschiedlichen Präfixen verwenden:

[,yaml]
----
global:
  backup:
    enabled: true
backup:
  elasticsearch:
    bucketName: BUCKET
    s3Prefix: elasticsearch
  stackGraph:
    bucketName: BUCKET
    s3Prefix: stackgraph
  configuration:
    bucketName: BUCKET
    s3Prefix: configuration
victoria-metrics-0:
  backup:
    bucketName: BUCKET
    s3Prefix: victoria-metrics-0
victoria-metrics-1:
  backup:
    bucketName: BUCKET
    s3Prefix: victoria-metrics-1
clickhouse:
  backup:
    bucketName: BUCKET
    s3Prefix: clickhouse
minio:
  accessKey: YOUR_ACCESS_KEY
  secretKey: YOUR_SECRET_KEY
  s3gateway:
    enabled: true
    accessKey: AWS_ACCESS_KEY
    secretKey: AWS_SECRET_KEY
----

Ersetzen Sie `+BUCKET+` durch den Namen Ihres S3-Buckets. Die Backups für die verschiedenen Datenspeicher werden anhand der konfigurierten `+s3Prefix+` Werte organisiert. Es gelten die gleichen Werte `+YOUR_ACCESS_KEY+`, `+YOUR_SECRET_KEY+`, `+AWS_ACCESS_KEY+` und `+AWS_SECRET_KEY+` wie im vorherigen Abschnitt.

=== AWS S3-Berechtigungen

Der durch `+AWS_ACCESS_KEY+` und `+AWS_SECRET_KEY+` identifizierte IAM-Benutzer muss mit der folgenden Berechtigungsrichtlinie konfiguriert werden, um auf die S3-Buckets zugreifen zu können:

----

            ],
            "Resource": [
                "arn:aws:s3:::AWS_STACKGRAPH_BUCKET",

        },
        {
            "Sid": "AllowWriteMinioBackupBuckets",
            "Effect": "Allow",
            "Action": [
                "s3:PutObject",
                "s3:GetObject",
                "s3:DeleteObject"
            ],
            "Resource": [
                "arn:aws:s3:::AWS_STACKGRAPH_BUCKET/*",
                "arn:aws:s3:::AWS_ELASTICSEARCH_BUCKET/*",
                "arn:aws:s3:::AWS_VICTORIA_METRICS_BUCKET/*",
                "arn:aws:s3:::AWS_CLICKHOUSE_BUCKET/*",
                "arn:aws:s3:::AWS_CONFIGURATION_BUCKET"
            ]
        }
    ]
}
----

=== Azure Blob-Speicher

==== Verwendung separater Behälter

Um Backups in separaten Azure Blob Storage-Containern (einer pro Datenspeicher) zu ermöglichen, fügen Sie das folgende YAML-Fragment in die Helm `+values.yaml+` -Datei ein, die zur Installation von {stackstate-product-name} verwendet wird:

[,yaml]
----
global:
  backup:
    enabled: true
minio:
  accessKey: AZURE_STORAGE_ACCOUNT_NAME
  secretKey: AZURE_STORAGE_ACCOUNT_KEY
  azuregateway:
    enabled: true
----

Ersetzen Sie die folgenden Werte:

* `+AZURE_STORAGE_ACCOUNT_NAME+` - den https://learn.microsoft.com/en-us/azure/storage/common/storage-account-create?tabs=azure-portal[Azure-Speicher-Kontonamen (learn.microsoft.com)]
* `+AZURE_STORAGE_ACCOUNT_KEY+` - den https://learn.microsoft.com/en-us/azure/storage/common/storage-account-keys-manage?tabs=azure-portal[Azure-Speicherkontoschlüssel (learn.microsoft.com)], unter dem die Backups gespeichert werden sollen.

Die Backups von StackGraph, Elasticsearch, Victoria Metrics und ClickHouse werden in BLOB-Containern mit den Bezeichnungen `+sts-stackgraph-backup+`, `+sts-configuration-backup+`, `+sts-elasticsearch-backup+`, `+sts-victoria-metrics-backup+`, `+sts-clickhouse-backup+` gespeichert. Diese Namen können durch Setzen der Helm-Werte `+backup.stackGraph.bucketName+`, `+backup.elasticsearch.bucketName+`, `+victoria-metrics-0.backup.bucketName+`, `+victoria-metrics-1.backup.bucketName+` und `+clickhouse.backup.bucketName+` geändert werden.

==== Verwendung eines einzigen Containers mit Präfixen

Anstatt separate Container für jeden Datenspeicher zu verwenden, können Sie einen einzigen Azure Blob Storage-Container mit verschiedenen Präfixen verwenden:

[,yaml]
----
global:
  backup:
    enabled: true
backup:
  elasticsearch:
    bucketName: CONTAINER
    s3Prefix: elasticsearch
  stackGraph:
    bucketName: CONTAINER
    s3Prefix: stackgraph
  configuration:
    bucketName: CONTAINER
    s3Prefix: configuration
victoria-metrics-0:
  backup:
    bucketName: CONTAINER
    s3Prefix: victoria-metrics-0
victoria-metrics-1:
  backup:
    bucketName: CONTAINER
    s3Prefix: victoria-metrics-1
clickhouse:
  backup:
    bucketName: CONTAINER
    s3Prefix: clickhouse
minio:
  accessKey: AZURE_STORAGE_ACCOUNT_NAME
  secretKey: AZURE_STORAGE_ACCOUNT_KEY
  azuregateway:
    enabled: true
----

Ersetzen Sie `+CONTAINER+` durch den Namen Ihres Azure Blob Storage-Containers. Die Backups für die verschiedenen Datenspeicher werden anhand der konfigurierten `+s3Prefix+` Werte organisiert. Es gelten die gleichen Werte `+AZURE_STORAGE_ACCOUNT_NAME+` und `+AZURE_STORAGE_ACCOUNT_KEY+` wie im vorherigen Abschnitt.

=== Persistentes Kubernetes-Volumen

[CAUTION]
====
Die Verwendung von Kubernetes Persistent Volumes für Backups hat erhebliche Einschränkungen:

* Teuer - Cloud-Anbieter verwenden in der Regel Blockspeicher (EBS/Azure Block), die für umfangreiche Backups kostspielig sind.
* Keine Notfallwiederherstellung - PVs werden zerstört, wenn der Cluster gelöscht wird
* Nicht portabel - Backups können nicht in einem anderen Cluster wiederhergestellt werden

**Empfehlung:** Verwenden Sie für Produktionsumgebungen stattdessen AWS S3 oder Azure Blob Storage.
====

==== Grundlegende Konfiguration

Um Backups auf clusterlokalen Speicher zu ermöglichen, aktivieren Sie MinIO, indem Sie das folgende YAML-Fragment in die Helm `+values.yaml+` -Datei einfügen, die zur Installation von {stackstate-product-name} verwendet wird:

[,yaml]
----
global:
  backup:
    enabled: true
minio:
  accessKey: YOUR_ACCESS_KEY
  secretKey: YOUR_SECRET_KEY
  persistence:
    enabled: true
----

Ersetzen Sie die folgenden Werte:

* `+YOUR_ACCESS_KEY+` und `+YOUR_SECRET_KEY+` - die Anmeldedaten, die zur Sicherung des MinIO-Systems verwendet werden. Die automatischen Sicherungsaufträge und die Wiederherstellungsaufträge verwenden sie. Sie werden auch für den manuellen Zugriff auf den MinIO-Speicher benötigt. `+YOUR_ACCESS_KEY+` sollte 5 bis 20 alphanumerische Zeichen und `+YOUR_SECRET_KEY+` sollte 8 bis 40 alphanumerische Zeichen enthalten.

== Konfigurations- und Topologiedaten (StackGraph)

Sicherungen von Konfigurations- und Topologiedaten (StackGraph) sind Vollsicherungen, die in einer einzigen Datei mit der Erweiterung `+.graph+` gespeichert werden. Jede Datei enthält eine vollständige Sicherung und kann nach Bedarf verschoben, kopiert oder gelöscht werden.

=== Sicherungszeitplan

Standardmäßig werden die StackGraph-Backups täglich um 03:00 Uhr Serverzeit erstellt.

Der Sicherungszeitplan kann mit dem Helm-Wert `+backup.stackGraph.scheduled.schedule+` konfiguriert werden, der in der https://kubernetes.io/docs/concepts/workloads/controllers/cron-jobs/#_cron_schedule_syntax[Syntax] Kubernetes angegeben wird.

=== Aufbewahrung von Sicherungskopien

Standardmäßig werden die Backups von StackGraph 30 Tage lang aufbewahrt. Da es sich bei den StackGraph-Sicherungen um Vollsicherungen handelt, kann dies viel Speicherplatz erfordern.

Das Delta der Sicherungsaufbewahrung kann mit dem Helm-Wert `+backup.stackGraph.scheduled.backupRetentionTimeDelta+` konfiguriert werden, der im Format des GNU-Datums `+--date+` angegeben wird. Die Standardeinstellung ist `+30 days ago+`. Weitere Beispiele finden Sie unter https://www.gnu.org/software/coreutils/manual/html_node/Relative-items-in-date-strings.html[Relative Elemente in Datumsstrings].

=== Geplante Backups deaktivieren

Um geplante StackGraph-Backups zu deaktivieren, setzen Sie den Backup-Zeitplan mit dem Helm-Wert `+backup.stackGraph.scheduled.schedule+` auf ein weit zurückliegendes Datum:

[,yaml]
----
backup:
  stackGraph:
    scheduled:
      schedule: '0 0 1 1 1970'  # January 1, 1970 (epoch start)
----

== Einstellungen

Einstellungen (zuvor Konfiguration) enthält installierte Instanzen von StackPacks mit ihrer Konfiguration und anderen vom Benutzer erstellten Anpassungen, wie z. B. Monitore, benutzerdefinierte Ansichten und Dienst-Tokens.

Einstellungssicherungen sind leicht (in der Regel nur einige Megabyte) und lassen sich schnell und mit minimaler Ausfallzeit wiederherstellen. Nach der Wiederherstellung eines Einstellungs-Backups werden die neuen Daten wie zuvor verarbeitet und die Topologie, die Zustände und die Warnmeldungen neu erstellt. Der Topologieverlauf (einschließlich des Zustands) bleibt jedoch bei der Sicherung der Einstellungen nicht erhalten - verwenden Sie zu diesem Zweck die oben beschriebene StackGraph-Sicherung.

[NOTE]
====
*Einstellungssicherungen sind immer aktiviert*, unabhängig vom Wert `+global.backup.enabled+`:

* Unter `+global.backup.enabled+` finden Sie `+true+`: Backups der Einstellungen werden über MinIO in Ihrem konfigurierten Speicher-Backend (AWS S3, Azure Blob Storage oder Kubernetes Persistent Volume) gespeichert.
* Unter `+global.backup.enabled+` finden Sie `+false+`: Backups der Einstellungen werden in einem dedizierten Kubernetes Persistent Volume gespeichert, wodurch MinIO umgangen wird.
====

=== Sicherungszeitplan

Standardmäßig werden Sicherungen der Einstellungen täglich um 04:00 Uhr Serverzeit erstellt.

Der Sicherungszeitplan kann mit dem Helm-Wert `+backup.configuration.scheduled.schedule+` konfiguriert werden, der in der https://kubernetes.io/docs/concepts/workloads/controllers/cron-jobs/#_cron_schedule_syntax[Syntax] Kubernetes angegeben wird.

=== Aufbewahrung von Sicherungskopien

Die Aufbewahrung von Sicherungskopien hängt von der Einstellung `+global.backup.enabled+` ab:

**Wenn `+global.backup.enabled+` `+true+` ist** (Backups werden über MinIO gespeichert):

* Standardmäßig werden Sicherungen der Einstellungen 365 Tage lang aufbewahrt.
* Konfigurieren Sie die Aufbewahrung mit `+backup.configuration.scheduled.backupRetentionTimeDelta+` - angegeben im Format des GNU-Datums `+--date+` argument. Der Standardwert ist `+365 days ago+`

**Wenn `+global.backup.enabled+` `+false+` ist** (Backups werden auf einem speziellen PV gespeichert):

* Standardmäßig werden die letzten 10 Sicherungsdateien auf dem Persistent Volume gespeichert.
* Konfigurieren Sie die maximale Anzahl von Dateien mit `+backup.configuration.maxLocalFiles+` (Standard: `+10+`)
* Konfigurieren Sie die PV-Größe mit `+backup.configuration.scheduled.pvc.size+` (Standard: `+1Gi+`)

Beispielkonfiguration für einen dedizierten PV-Speicher:

[,yaml]
----
backup:
  configuration:
    maxLocalFiles: 10
    scheduled:
      pvc:
        size: '1Gi'
----

=== Geplante Backups deaktivieren

Um geplante Backups der Einstellungen zu deaktivieren, setzen Sie den Backup-Zeitplan mit dem Helm-Wert `+backup.configuration.scheduled.schedule+` auf ein weit zurückliegendes Datum:

[,yaml]
----
backup:
  configuration:
    scheduled:
      schedule: '0 0 1 1 1970'  # January 1, 1970 (epoch start)
----

== Metriken (Victoria Metriken)

Victoria Metrics erstellt Sicherungen, indem es einen Schnappschuss der aktuellen Metrikdaten erstellt und nur die Änderungen seit der letzten Sicherung speichert (inkrementeller Ansatz).

[WARNING]
====
*Beschränkung der Versionierung von Sicherungskopien*

Victoria Metrics-Sicherungen ersetzen die vorherige Sicherung bei jeder neuen Sicherung. Das bedeutet, dass Sie nur auf *die jüngste Sicherung* zugreifen können und nicht auf eine Historie mehrerer Sicherungsversionen.

**Das ist wichtig**: Wenn eine Sicherung fehlschlägt oder die Daten beschädigt werden, können Ihre vorherigen Sicherungsdaten verloren gehen. Nur die letzte erfolgreiche Sicherung ist für die Wiederherstellung verfügbar.

**Eine Ausnahme**: Wenn das Victoria Metrics-Speichervolumen geleert oder zurückgesetzt wird (z. B. wenn das Verzeichnis `+/storage+` leer gemountet oder gelöscht wird), erkennt das System dies und erstellt automatisch eine neue Backup-Serie. In diesem Fall bleiben sowohl die alte Sicherung (vor dem Zurücksetzen) als auch die neue Sicherung erhalten und stehen zur Wiederherstellung zur Verfügung.
====

[NOTE]
====
*Hochverfügbarkeitseinsätze*

Hochverfügbare Bereitstellungen verwenden zwei Instanzen von Victoria Metrics (`+victoria-metrics-0+` und `+victoria-metrics-1+`). Backups werden unabhängig für jede Instanz konfiguriert.
====

=== Sicherungszeitplan

Die Victoria Metrics-Backups werden alle 1 Stunde erstellt:

* `+victoria-metrics-0+` - 25 Minuten nach der vollen Stunde
* `+victoria-metrics-1+` - 35 Minuten nach der vollen Stunde

=== Geplante Backups deaktivieren

Um geplante Victoria Metrics-Sicherungen zu deaktivieren, setzen Sie den Sicherungszeitplan für beide Instanzen auf ein Datum, das weit in der Vergangenheit liegt:

[,yaml]
----
victoria-metrics-0:
  backup:
    scheduled:
      schedule: '0 0 1 1 1970'  # January 1, 1970 (epoch start)
victoria-metrics-1:
  backup:
    scheduled:
      schedule: '0 0 1 1 1970'  # January 1, 1970 (epoch start)
----

== OpenTelemetry (KlickHaus)

ClickHouse verwendet sowohl inkrementelle als auch vollständige Backups. Die alten Backups werden entsprechend der konfigurierten Aufbewahrungsrichtlinie gelöscht.

=== Sicherungszeitplan

Die ClickHouse-Backups werden erstellt:

* Vollständige Datensicherung - jeden Tag um 00:45 Uhr
* Inkrementelle Sicherung - 45 Minuten nach der vollen Stunde (von 3 Uhr morgens bis 12 Uhr nachts)

=== Aufbewahrung von Sicherungskopien

Standardmäßig behält das Tool die letzten 308 Backups (vollständig und inkrementell), was etwa 14 Tagen entspricht.

Die Backup-Aufbewahrung kann mit dem Helm-Wert `+clickhouse.backup.config.keep_remote+` konfiguriert werden.

=== Geplante Backups deaktivieren

Um geplante ClickHouse-Sicherungen zu deaktivieren, setzen Sie sowohl die Zeitpläne für vollständige als auch für inkrementelle Sicherungen auf ein Datum, das weit in der Vergangenheit liegt:

[,yaml]
----
clickhouse:
  backup:
    scheduled:
      full_schedule: '0 0 1 1 1970'         # January 1, 1970 (epoch start)
      incremental_schedule: '0 0 1 1 1970'  # January 1, 1970 (epoch start)
----

== Telemetriedaten (Elasticsearch)

Elasticsearch-Snapshots sind inkrementell.

=== Schnappschuss-Zeitplan

Die Elasticsearch-Snapshots werden täglich um 03:00 Uhr Serverzeit erstellt.

=== Speicherung von Schnappschüssen

Standardmäßig werden Elasticsearch-Snapshots 30 Tage lang aufbewahrt, mit einem Minimum von 5 Snapshots und einem Maximum von 30 Snapshots.

Die Aufbewahrungszeit und die Anzahl der aufbewahrten Snapshots können mit den folgenden Helm-Werten konfiguriert werden:

* `+backup.elasticsearch.scheduled.snapshotRetentionExpireAfter+`, angegeben in https://www.elastic.co/guide/en/elasticsearch/reference/7.6/common-options.html#_time_units[Elasticsearch-Zeiteinheiten (elastic.co)].
* `+backup.elasticsearch.scheduled.snapshotRetentionMinCount+`
* `+backup.elasticsearch.scheduled.snapshotRetentionMaxCount+`

=== Geplante Schnappschüsse deaktivieren

Um geplante Elasticsearch-Snapshots zu deaktivieren, setzen Sie den Snapshot-Zeitplan mit dem Helm-Wert `+backup.elasticsearch.scheduled.schedule+` auf ein weit zurückliegendes Datum:

[,yaml]
----
backup:
  elasticsearch:
    scheduled:
      schedule: '0 0 1 1 1970'  # January 1, 1970 (epoch start)
----
