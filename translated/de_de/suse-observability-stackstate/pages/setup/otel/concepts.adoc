= Offene Telemetriekonzepte
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: SUSE Observabilität

Dies ist eine Zusammenfassung der wichtigsten Konzepte der offenen Telemetrie und sollte für den Einstieg ausreichend sein. Eine ausführlichere Einführung finden Sie in der https://opentelemetry.io/docs/concepts/[Dokumentation]

== Signale

Open Telemetry erkennt 3 Telemetriesignale:

* Spurensuche
* Metriken
* Protokolle

Momentan unterstützt SUSE Observability Traces und Metriken, Protokolle werden in einer zukünftigen Version unterstützt. Für Kubernetes-Protokolle kann stattdessen der xref:/k8s-quick-start-guide.adoc[SUSE Observability-Agent] verwendet werden.

=== Spurensuche

Traces ermöglichen es uns, den Weg einer Anfrage durch Ihre Anwendung zu visualisieren. Ein Trace besteht aus einem oder mehreren Spans, die zusammen einen Baum bilden. Ein einzelner Trace kann vollständig innerhalb eines einzigen Dienstes liegen, er kann sich aber auch über mehrere Dienste erstrecken. Jeder Bereich stellt einen Vorgang bei der Bearbeitung des Antrags dar und hat:

* ein Name
* Start- und Endzeit, daraus kann eine Dauer berechnet werden
* status
* Eigenschaften
* Ressourcenattribute (siehe <<_resources,Ressourcen>>)
* Veranstaltungen

Bereichsattribute werden verwendet, um Metadaten für den Bereich bereitzustellen, z. B. kann ein Bereich für einen Vorgang, der eine Bestellung aufgibt, die `+orderId+` als Attribut haben, oder ein Bereich für einen HTTP-Vorgang kann die HTTP-Methode und URL als Attribute haben.

Spannenereignisse können verwendet werden, um einen Zeitpunkt darzustellen, an dem etwas Wichtiges im Rahmen des Betriebs der Spannweite passiert ist. Wenn z. B. der Span fehlgeschlagen ist, kann es ein `+exception+` oder ein `+error+` Ereignis geben, das die Fehlermeldung, einen Stacktrace und den genauen Zeitpunkt des Auftretens des Fehlers aufzeichnet.

=== Metriken

Metriken sind Messungen, die zur Laufzeit erfasst werden und zu einem metrischen Ereignis führen. Metriken sind wichtige Indikatoren für die Anwendungsleistung und -verfügbarkeit und werden häufig verwendet, um auf Ausfälle oder Leistungsprobleme hinzuweisen. Metriken haben:

* ein Name
* einen Zeitstempel
* eine Art (Zähler, Messgerät, Histogramm, usw.)
* Eigenschaften
* Ressourcenattribute (siehe <<_resources,Ressourcen>>)

Attribute liefern die Metadaten für eine Metrik.

== Ressourcen

Eine Ressource ist die Einheit, die die Telemetriedaten erzeugt. Die Ressourcenattribute liefern die Metadaten für die Ressource. Beispielsweise kann ein Prozess, der in einem Container, in einem Pod oder in einem Namespace in einem Kubernetes-Cluster läuft, Ressourcenattribute für all diese Entitäten haben.

Ressourcenattribute werden von den SDKs oft automatisch zugewiesen. Es wird jedoch empfohlen, die Attribute `+service.name+` und `+service.namespace+` immer explizit zu setzen. Der erste ist der logische Name für den Service. Wenn er nicht gesetzt ist, setzt das SDK den Wert `+unknown_service+`, was die spätere Verwendung der Daten in SUSE Observability sehr erschwert. Der Namespace ist eine praktische Möglichkeit, Ihre Dienste zu organisieren, insbesondere dann, wenn Sie dieselben Dienste an mehreren Orten laufen lassen.

== Semantische Konventionen

Open Telemetry definiert gemeinsame Namen für Operationen und Daten, sie nennen dies die semantischen Konventionen. Semantische Konventionen folgen einem Benennungsschema, das die Standardisierung der Datenverarbeitung in verschiedenen Sprachen, Bibliotheken und Codebasen ermöglicht. Es gibt semantische Konventionen für alle Signale und für Ressourcenattribute. Sie sind für viele verschiedene Plattformen und Operationen auf der https://opentelemetry.io/docs/specs/semconv/attributes-registry/[Website] definiert. SDKs nutzen die semantischen Konventionen, um diese Attribute zuzuweisen, und auch SUSE Observability respektiert die Konventionen und stützt sich auf sie, um zum Beispiel Kubernetes-Ressourcen zu erkennen.
