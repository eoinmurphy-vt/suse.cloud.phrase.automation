= Unterstützung des SUSE Observability Open Telemetry Protocol
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: SUSE Observabilität

SUSE Observability unterstützt 2 Versionen des OTLP-Protokolls, die Version `+grpc+` (auch als OTLP bezeichnet) und `+http/protobuf+` (auch als OTLP over HTTP bezeichnet). In der Kollektorkonfiguration können Sie wählen, welchen Exporter Sie verwenden möchten. Stellen Sie jedoch sicher, dass Sie die korrekte URL für SUSE Observability konfigurieren. Die `+grpc+` Version des Protokolls wird bevorzugt, da sie größere Nutzlasten und einen höheren Durchsatz ermöglicht. Falls Ihre Infrastruktur jedoch `+grpc+` nicht unterstützt, können Sie auf die HTTP-Version umsteigen. Siehe auch xref:/setup/otel/troubleshooting.adoc#_some_proxies_and_firewalls_dont_work_well_with_grpc[Fehlersuche]

== SUSE Cloud Beobachtbarkeit

Die Endpunkte für SUSE Cloud Observability sind:

* OTLP: `+otlp-<your-suse-observability>.app.stackstate.io:443+` (ohne das URL-Schema)
* OTLP über HTTP: `+https://otlp-http-<your-suse-observability>.app.stackstate.io+` (mit dem URL-Schema: `+https+`)

== Selbstgehostetes SUSE Observability

Bei einer selbst gehosteten Installation müssen Sie einen der Endpunkte oder beide aktivieren, indem Sie den Ingress für SUSE Observability wie xref:/setup/install-stackstate/kubernetes_openshift/ingress.adoc#_configure_ingress_rule_for_open_telemetry[hier beschrieben] konfigurieren.

Wenn SUSE Observability im selben Cluster wie der Collector läuft, können Sie ihn auch ohne Ingress nutzen, indem Sie die Service-Endpunkte verwenden. Dies ist jedoch nur über das HTTP-Protokoll möglich:

* OTLP über HTTP: `+http://suse-observability-otel-collector.<namespace>.svc.cluster.local:4318+`

Stellen Sie sicher, dass Sie `+insecure: true+` in der Kollektorkonfiguration einstellen (siehe nächster Abschnitt), um die Verwendung von einfachen http-Endpunkten anstelle von https zu ermöglichen. 

Das GRPC-Protokoll unterstützt das Senden von Anmeldeinformationen über eine unsichere Verbindung nicht, daher können Sie es nur in Kombination mit einer sicheren Eingangseinrichtung verwenden.

== Konfiguration des Kollektors

=== OTLP-Protokoll

Die Beispiele in der Kollektorkonfiguration verwenden das OTLP-Protokoll wie folgt:

----
extensions:
  bearertokenauth:
    scheme: SUSEObservability
    token: "${env:API_KEY}"
exporters:
  otlp/suse-observability:
    auth:
      authenticator: bearertokenauth
    # Put in your own otlp endpoint, for example otlp-suse-observability.my.company.com:443
    endpoint: <otlp-suse-observability-endpoint>
    # Optional TLS configurations:
    #tls:
    # To disable certificate verification (but still use TLS):
    #  insecure_skip_verify: true
----

=== OTLP HTTP-Protokoll

Wenn Sie stattdessen das OTLP-over-HTTP-Protokoll verwenden möchten, benutzen Sie stattdessen den `+otlphttp+` exporter. Aktualisieren Sie auch alle Exporter-Referenzen in Ihren Pipelines von `+otlp/suse-observability+` auf `+otlphttp/suse-observability+`! Verwenden Sie eine Suchen/Ersetzen-Funktion, um sicherzustellen, dass Sie alle Vorkommen ändern.

----
extensions:
  bearertokenauth:
    scheme: SUSEObservability
    token: "${env:API_KEY}"
exporters:
  otlphttp/stackstate:
    auth:
      authenticator: bearertokenauth
    # Put in your own otlp-http endpoint, for example https://otlp-http-suse-observability.my.company.com:443
    endpoint: <otlp-http-suse-observability-endpoint>
    # Optional TLS configurations:
    #tls:
    # To disable TLS entirely:
    #  insecure: true
    # To disalbe certificate verification (but still use TLS):
    #  insecure_skip_verify: true
----

Es gibt mehr Konfigurationsmöglichkeiten, um die genauen Anforderungen und das Verhalten des Exporteurs zu steuern. Es ist zum Beispiel auch möglich, ein benutzerdefiniertes CA-Root-Zertifikat zu verwenden oder Client-Zertifikate zu aktivieren. Einzelheiten dazu finden Sie in der https://github.com/open-telemetry/opentelemetry-collector/blob/main/exporter/otlpexporter/README.md[Dokumentation].
