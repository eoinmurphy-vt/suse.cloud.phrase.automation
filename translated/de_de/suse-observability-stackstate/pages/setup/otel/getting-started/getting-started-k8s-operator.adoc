= Erste Schritte mit Open Telemetry Operator auf Kubernetes
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: SUSE Observabilität

Hier ist das Setup, das wir für eine zu überwachende Anwendung erstellen werden:

* Die überwachte Anwendung/Arbeitslast, die in Cluster A läuft und vom Bediener automatisch gemessen wird
* Der Open Telemetry-Operator in Cluster A
* Ein vom Betreiber erstellter Kollektor
* SUSE Observability, das in Cluster B ausgeführt wird, oder SUSE Cloud Observability

image::otel/open-telemetry-kubernetes-operator.png[Container-Instrumentierung mit Open Telemetry Bediener-Autoinstrumentierung]

== Installieren Sie den Bediener

Der Open Telemetry-Operator bietet einige zusätzliche Funktionen gegenüber der normalen Kubernetes-Einrichtung:

* Es kann Ihre Anwendungs-Pods automatisch für unterstützte Sprachen (Java, .NET, Python, Golang, Node.js) instrumentieren, ohne dass Sie die Anwendungen oder Docker-Images überhaupt ändern müssen
* Er kann als Ersatz für den Prometheus-Operator eingesetzt werden und beginnt mit dem Scraping von Prometheus-Exporter-Endpunkten auf der Grundlage von Dienst- und Pod-Monitoren

=== Erstellen eines Service-Tokens

Es gibt zwei Möglichkeiten, ein Service-Token zu erstellen:

* **SUSE Observability UI** - öffnen Sie das Hauptmenü, indem Sie oben links auf den Bildschirm klicken und zu `+StackPacks+` > `+Open Telemetry+` gehen.  Falls Sie dies noch nicht getan haben, klicken Sie auf die Schaltfläche `+INSTALL+`.  Klicken Sie auf die Schaltfläche `+CREATE NEW SERVICE TOKEN+` und kopieren Sie den Wert in Ihre Zwischenablage.
* **SUSE Observability CLI** - siehe xref:/use/security/k8s-service-tokens.adoc#_manage_service_tokens[Verwalten von Service-Tokens]

Der Wert des Service-Tokens muss verwendet werden, wenn in den nachstehenden Anweisungen `+<SERVICE_TOKEN>+` erwähnt wird.

=== Erstellen Sie den Namensraum und ein Geheimnis für das Service-Token

Wir installieren im Namespace `+open-telemetry+` und verwenden das Service-Token:

[,bash]
----
kubectl create namespace open-telemetry
kubectl create secret generic open-telemetry-collector \
    --namespace open-telemetry \
    --from-literal=API_KEY='<SERVICE_TOKEN>'
----

=== Konfigurieren und Installieren des Operators

Der Operator wird mit einer Helm-Karte installiert, also konfigurieren Sie zunächst das Kartendepot.

[,bash]
----
helm repo add open-telemetry https://open-telemetry.github.io/opentelemetry-helm-charts
----

Erstellen wir eine `+otel-operator.yaml+` Datei, um den Operator zu konfigurieren:

.otel-operator.yaml
[,yaml]
----
# Add image pull secret for private registries
imagePullSecrets: []
manager:
  image:
    # Uses chart.appVersion for the tag
    repository: ghcr.io/open-telemetry/opentelemetry-operator/opentelemetry-operator
  collectorImage:
    # find the latest collector releases at https://github.com/open-telemetry/opentelemetry-collector-releases/releases
    repository: otel/opentelemetry-collector-k8s
    tag: 0.123.0
  targetAllocatorImage:
    repository: ""
    tag: ""
  # Only needed when overriding the image repository, make sure to always specify both the image and tag:
  autoInstrumentationImage:
    java:
      repository: ""
      tag: ""
    nodejs:
      repository: ""
      tag: ""
    python:
      repository: ""
      tag: ""
    dotnet:
      repository: ""
      tag: ""
    # The Go instrumentation support in the operator is disabled by default.
    # To enable it, use the operator.autoinstrumentation.go feature gate.
    go:
      repository: ""
      tag: ""

admissionWebhooks:
  # A production setup should use certManager to generate the certificate, without certmanager the certificate will be generated during the Helm install
  certManager:
    enabled: false
  # The operator has validation and mutation hooks that need a certificate, with this we generate that automatically
  autoGenerateCert:
    enabled: true
----


Installieren Sie nun den Kollektor mit Hilfe der Konfigurationsdatei:

[,bash]
----
helm upgrade --install opentelemetry-operator open-telemetry/opentelemetry-operator \
  --namespace open-telemetry \
  --values otel-operator.yaml
----

Damit wird nur der Operator installiert. Fahren Sie mit der Installation des Kollektors fort und aktivieren Sie die Auto-Instrumentierung.

== Der offene Telemetriesammler

Der Betreiber verwaltet eine oder mehrere Collector-Bereitstellungen über eine benutzerdefinierte Kubernetes-Ressource der Art `+OpenTelemetryCollector+`. Wir erstellen eine solche mit der gleichen Konfiguration wie in der xref:/setup/otel/getting-started/getting-started-k8s.adoc[Kubernetes-Einführungsanleitung].

Dabei wird das zuvor im Leitfaden erstellte Geheimnis verwendet. Ersetzen Sie `+<otlp-suse-observability-endpoint:port>+` durch Ihren OTLP-Endpunkt (siehe xref:/setup/otel/otlp-apis.adoc[OTLP API] für Ihren Endpunkt) und fügen Sie den Namen Ihres Kubernetes-Clusters anstelle von `+<your-cluster-name>+` ein:

.collector.yaml
[,yaml]
----
apiVersion: opentelemetry.io/v1beta1
kind: OpenTelemetryCollector
metadata:
  name: otel-collector
spec:
  mode: deployment
  envFrom:
  - secretRef:
      name: open-telemetry-collector
  # optional service-account for pulling the collector image from a private registries
  # serviceAccount: otel-collector
  config:
    receivers:
      otlp:
        protocols:
          grpc:
            endpoint: 0.0.0.0:4317
          http:
            endpoint: 0.0.0.0:4318
      # Scrape the collectors own metrics
      prometheus:
        config:
          scrape_configs:
          - job_name: opentelemetry-collector
            scrape_interval: 10s
            static_configs:
            - targets:
              - ${env:MY_POD_IP}:8888
    extensions:
      health_check:
        endpoint: ${env:MY_POD_IP}:13133
      # Use the API key from the env for authentication
      bearertokenauth:
        scheme: SUSEObservability
        token: "${env:API_KEY}"
    exporters:
      debug: {}
      nop: {}
      otlp/suse-observability:
        auth:
          authenticator: bearertokenauth
        # Put in your own otlp endpoint, for example otlp-suse-observability.my.company.com:443
        endpoint: <otlp-suse-observability-endpoint:port>
        compression: snappy
    processors:
      memory_limiter:
        check_interval: 5s
        limit_percentage: 80
        spike_limit_percentage: 25
      batch: {}
      resource:
        attributes:
        - key: k8s.cluster.name
          action: upsert
          # Insert your own cluster name
          value: <your-cluster-name>
        - key: service.instance.id
          from_attribute: k8s.pod.uid
          action: insert
          # Use the k8s namespace also as the open telemetry namespace
        - key: service.namespace
          from_attribute: k8s.namespace.name
          action: insert
    connectors:
      # Generate metrics for spans
      spanmetrics:
        metrics_expiration: 5m
        namespace: otel_span
    service:
      extensions: [ health_check,  bearertokenauth ]
      pipelines:
        traces:
          receivers: [otlp]
          processors: [memory_limiter, resource, batch]
          exporters: [debug, spanmetrics, otlp/suse-observability]
        metrics:
          receivers: [otlp, spanmetrics, prometheus]
          processors: [memory_limiter, resource, batch]
          exporters: [debug, otlp/suse-observability]
        logs:
          receivers: [otlp]
          processors: []
          exporters: [nop]
      telemetry:
        metrics:
          address: ${env:MY_POD_IP}:8888
----


[CAUTION]
====
*Verwenden Sie denselben Clusternamen wie bei der Installation des SUSE Observability-Agenten*, wenn Sie den SUSE Observability-Agenten auch mit dem Kubernetes-Stackpack verwenden. Die Verwendung eines anderen Clusternamens führt zu einer leeren Trace-Perspektive für Kubernetes-Komponenten und erschwert die Korrelation von Informationen für SUSE Observability und Ihre Benutzer insgesamt erheblich.
====


Wenden Sie nun diese `+collector.yaml+` im `+open-telemetry+` Namespace an, um einen Collector einzusetzen:

[,bash]
----
kubectl apply --namespace open-telemetry -f collector.yaml
----

Der Collector bietet eine Menge weiterer Konfigurationsempfänger, -verarbeiter und -exporteure, weitere Details finden Sie auf unserer xref:/setup/otel/collector.adoc[Collector-Seite]. Bei der Verwendung in der Produktion werden oft große Mengen an Spans erzeugt, und Sie werden mit der Einrichtung von xref:/setup/otel/sampling.adoc[Stichproben] beginnen wollen.

== Auto-Instrumentierung

=== Auto-Instrumentierung konfigurieren

Nun müssen wir dem Bediener mitteilen, wie er die automatische Instrumentierung für die verschiedenen Sprachen konfigurieren kann, indem wir eine weitere benutzerdefinierte Ressource der Art `+Instrumentation+` verwenden. Es wird hauptsächlich dazu verwendet, den Kollektor zu konfigurieren, der gerade als Telemetrie-Endpunkt für die instrumentierten Anwendungen bereitgestellt wurde.

Sie kann an einer einzigen Stelle definiert und von allen Pods im Cluster verwendet werden, aber es ist auch möglich, in jedem Namespace eine andere `+Instrumentation+` zu haben. Wir werden hier das Erstere tun. Beachten Sie, dass der Endpunkt in dieser Datei entsprechend aktualisiert werden muss, wenn Sie einen anderen Namensraum oder einen anderen Namen für den otel-Kollektor verwenden.

Erstellen Sie eine `+instrumentation.yaml+`:

.instrumentation.yaml
[,yaml]
----
apiVersion: opentelemetry.io/v1alpha1
kind: Instrumentation
metadata:
  name: otel-instrumentation
spec:
  exporter:
    # default endpoint for the instrumentation
    endpoint: http://otel-collector-collector.open-telemetry.svc.cluster.local:4317
  propagators:
    - tracecontext
    - baggage
  defaults:
    # To use the standard app.kubernetes.io/ labels for the service name, version and namespace:
    useLabelsForResourceAttributes: true
  python:
    env:
      # Python autoinstrumentation uses http/proto by default, so data must be sent to 4318 instead of 4317.
      - name: OTEL_EXPORTER_OTLP_ENDPOINT
        value: http://otel-collector-collector.open-telemetry.svc.cluster.local:4318
  dotnet:
    env:
      # Dotnet autoinstrumentation uses http/proto by default, so data must be sent to 4318 instead of 4317.
      - name: OTEL_EXPORTER_OTLP_ENDPOINT
        value: http://otel-collector-collector.open-telemetry.svc.cluster.local:4318
  go:
    env:
      # Go autoinstrumentation uses http/proto by default, so data must be sent to 4318 instead of 4317.
      - name: OTEL_EXPORTER_OTLP_ENDPOINT
        value: http://otel-collector-collector.open-telemetry.svc.cluster.local:4318
----


Wenden Sie nun die `+instrumentation.yaml+` auch im Namensraum `+open-telemetry+` an:

[,bash]
----
kubectl apply --namespace open-telemetry -f instrumentation.yaml
----

=== Auto-Instrumentierung für einen Pod aktivieren

Um den Operator anzuweisen, Ihre Applikations-Pods automatisch zu instrumentieren, müssen wir dem Pod eine Anmerkung hinzufügen:

* Java: `+instrumentation.opentelemetry.io/inject-java: open-telemetry/otel-instrumentation+`
* NodeJS: `+instrumentation.opentelemetry.io/inject-nodejs: open-telemetry/otel-instrumentation+`
* Python: `+instrumentation.opentelemetry.io/inject-python: open-telemetry/otel-instrumentation+`
* Gehen Sie: `+instrumentation.opentelemetry.io/inject-go: open-telemetry/otel-instrumentation+`

Beachten Sie, dass sich der Wert der Anmerkung auf den Namespace und den Namen der von uns erstellten Ressource `+Instrumentation+` bezieht. Andere Optionen sind:

* "true" - Injizieren und `+Instrumentation+` benutzerdefinierte Ressource aus dem Namespace.
* "my-instrumentation" - Name der benutzerdefinierten Ressource `+Instrumentation+` im aktuellen Namensraum.
* "my-other-namespace/my-instrumentation" - Namespace und Name der benutzerdefinierten Ressource `+Instrumentation+` in einem anderen Namespace.
* "false" - nicht injizieren

Wenn ein Pod mit einer der Annotationen erstellt wird, ändert der Operator den Pod über einen Mutationshaken:

* Es fügt einen init-Container hinzu, der die Auto-Instrumentierungsbibliothek bereitstellt
* Es modifiziert den ersten Container des Pods, um die Instrumente während des Starts zu laden und fügt Umgebungsvariablen hinzu, um die Instrumente zu konfigurieren

Wenn Sie anpassen müssen, welche Container instrumentiert werden sollen, verwenden Sie die https://github.com/open-telemetry/opentelemetry-operator?tab=readme-ov-file#_multi_container_pods_with_multiple_instrumentations[Dokumentation].

[CAUTION]
====
Die automatische Instrumentierung von Go erfordert erhöhte Berechtigungen. Diese Berechtigungen werden automatisch vom Betreiber festgelegt:

[,yaml]
----
securityContext:
  privileged: true
  runAsUser: 0
----

====


== Die Ergebnisse anzeigen

Gehen Sie zu SUSE Observability und stellen Sie sicher, dass das Open Telemetry Stackpack installiert ist (über das Hauptmenü \-> Stackpacks).

Nach einer kurzen Zeit und wenn Ihre Pods etwas Traffic erhalten, sollten Sie sie unter ihrem Dienstnamen in den Übersichten Open Telemetry \-> Services und Service Instances finden können. Traces werden im xref:/use/traces/k8sTs-explore-traces.adoc[Trace-Explorer] und in der xref:/use/views/k8s-traces-perspective.adoc[Trace-Perspektive] für die Service- und Service-Instanz-Komponenten angezeigt. Bereichsmetriken und sprachspezifische Metriken (falls vorhanden) werden in der xref:/use/views/k8s-metrics-perspective.adoc[Metrikperspektive] für die Komponenten verfügbar.

Wenn Sie auch das Kubernetes-Stackpack installiert haben, werden die instrumentierten Pods auch in der xref:/use/views/k8s-traces-perspective.adoc[Trace-Perspektive] verfügbar sein.

== Rancher RBAC

Damit xref:/setup/security/rbac/rbac_rancher.adoc[Rancher RBAC] funktioniert, müssen die Telemetriedaten die folgenden Ressourcenattribute aufweisen:

* `+k8s.cluster.name+` - der *Cluster-Name*, wie er vom Kubernetes-Stackpack verwendet wird
* `+k8s.namespace.name+` - ein *Namensraum*, der von einem *Rancher-Projekt* verwaltet wird

Der Kubernetes-Operator fügt diese Attribute standardmäßig in alle gesendeten Telemetriedaten ein.
