= Telemetrie-Kollektor öffnen
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: SUSE Observabilität

Der OpenTelemetry Collector bietet eine herstellerunabhängige Implementierung zum Empfang, zur Verarbeitung und zum Export von Telemetriedaten. Anwendungen, die mit Open Telemetry SDKs instrumentiert sind, können den Collector verwenden, um Telemetriedaten an SUSE Observability zu senden (Traces und Metriken).

Wenn Ihre Anwendungen mit den OpenTelemetry-SDKs eingerichtet sind, können Sie den Collector verwenden, um Telemetriedaten wie Traces und Metriken an SUSE Observability oder einen anderen Collector (zur weiteren Verarbeitung) zu senden. Der Kollektor ist standardmäßig so eingestellt, dass er diese Daten über OTLP, das native offene Telemetrieprotokoll, empfängt. Es kann auch Daten in anderen Formaten empfangen, die von anderen Instrumentierungs-SDKs wie Jaeger und Zipkin für Traces und Influx und Prometheus für Metriken bereitgestellt werden.

Der Kollektor läuft in der Nähe Ihrer Anwendung, im selben Kubernetes-Cluster, auf derselben virtuellen Maschine usw. Dadurch können SDKs Daten schnell an den Collector weiterleiten, der dann Umwandlungen, Stapelverarbeitung und Filterung vornehmen kann. Sie kann von mehreren Anwendungen genutzt werden und ermöglicht einfache Änderungen an Ihrer Datenverarbeitungspipeline.

Installationsanleitungen finden Sie in den verschiedenen xref:/setup/otel/getting-started/README.adoc[Einführungshandbüchern]. Die Anleitungen für die ersten Schritte bieten eine grundlegende Collector-Konfiguration für den Anfang, aber im Laufe der Zeit werden Sie sie an Ihre Bedürfnisse anpassen und zusätzliche Empfänger, Prozessoren und Exporteure hinzufügen wollen, um Ihre Ingestion-Pipeline an Ihre Bedürfnisse anzupassen.

== Konfiguration

Die Kollektorkonfiguration definiert Pipelines für die Verarbeitung der verschiedenen Telemetriesignale. Die Komponenten in der Verarbeitungspipeline können in mehrere Kategorien unterteilt werden, und jede Komponente hat ihre eigene Konfiguration. Hier geben wir einen Überblick über die verschiedenen Konfigurationsabschnitte und ihre Verwendung.

=== Empfänger

Die Empfänger empfangen Telemetriedaten von instrumentierten Anwendungen, hier über OTLP:

[,yaml]
----
receivers:
  otlp:
    protocols:
      grpc:
        endpoint: 0.0.0.0:4317
      http:
        endpoint: 0.0.0.0:4318
----

Es gibt noch viele weitere Empfänger, die Daten über andere Protokolle entgegennehmen, z. B. Zipkin-Traces, oder die aktiv Daten aus verschiedenen Quellen sammeln, z. B:

* Host-Metriken
* Kubernetes-Metriken
* Prometheus-Metriken (OpenMetrics)
* Datenbanken

Einige Empfänger unterstützen alle 3 Signale (Traces, Metriken, Protokolle), andere nur 1 oder 2, z. B. kann der Prometheus-Empfänger nur Metriken sammeln. Das opentelemetry-collector-contrib Repository hat unter https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/receiver[alle Empfänger] mit Dokumentation zu deren Konfiguration.

=== Prozessoren

Die Daten von den Empfängern können von Prozessoren umgewandelt oder gefiltert werden.

[,yaml]
----
processors:
  batch: {}
----

Der Batch-Prozessor bündelt alle 3 Signale, wodurch die Komprimierung verbessert und die Anzahl der ausgehenden Verbindungen reduziert wird. Das opentelemetry-collector-contrib Repository hat unter https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/processor[alle Prozessoren] mit Dokumentation zu deren Konfiguration.

=== Exporteure

Um Daten an das SUSE Observability-Backend zu senden, verfügt der Collector über Exporteure. Es gibt Exporteure für verschiedene Protokolle, push- oder pull-basiert, und verschiedene Backends. Mit den OTLP-Protokollen ist es auch möglich, einen anderen Kollektor als Ziel für die weitere Verarbeitung zu verwenden.

[,yaml]
----
exporters:
  # The gRPC otlp exporter
  otlp/suse-observability:
    auth:
      authenticator: bearertokenauth
    # Put in your own otlp endpoint
    endpoint: <otlp-suse-observability-endpoint>
    # Use snappy compression, if no compression specified the data will be uncompressed
    compression: snappy
----

Der SUSE Observability-Exporter erfordert eine Authentifizierung mit einem API-Schlüssel, um die Verwendung einer <<_extensions,Authentifizierungserweiterung>> zu konfigurieren. Das opentelemetry-collector-contrib Repository hat unter https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/exporter[alle Exporteure] mit Dokumentation zu deren Konfiguration.

Wenn der gRPC-Exporter bei Ihnen nicht funktioniert (siehe auch xref:/setup/otel/troubleshooting.adoc#_some_proxies_and_firewalls_dont_work_well_with_grpc[Fehlerbehebung]), können Sie auf das etwas weniger effiziente OTLP-over-HTTP-Protokoll umsteigen, indem Sie stattdessen den `+otlphttp+` -Exporter verwenden. Ersetzen Sie alle Verweise auf `+otlp/suse-observability+` in den Abschnitten `+pipelines+` und `+exporter+` durch `+otlphttp/suse-observability+` und stellen Sie sicher, dass Sie die Konfiguration des Exporters aktualisieren:

[,yaml]
----
exporters:
  # The gRPC otlp exporter
  otlphttp/suse-observability:
    auth:
      authenticator: bearertokenauth
    # Put in your own otlp HTTP endpoint
    endpoint: <otlp-http-suse-observability-endpoint>
    # Use snappy compression, if no compression specified the data will be uncompressed
    compression: snappy
----

[CAUTION]
====
Der OTLP-HTTP-Endpunkt für SUSE Observability unterscheidet sich vom OTLP-Endpunkt. Verwenden Sie die xref:/setup/otel/otlp-apis.adoc[OTLP-APIs], um die richtige URL zu finden.
====


=== Service-Pipeline

Für jedes Telemetriesignal wird eine eigene Pipeline konfiguriert. Die Pipelines werden im Abschnitt `+service.pipeline+` konfiguriert und legen fest, welche Empfänger, Prozessoren und Exporteure in welcher Reihenfolge verwendet werden sollen. Bevor eine Komponente in der Pipeline verwendet werden kann, muss sie zunächst in ihrem Konfigurationsabschnitt definiert werden. Der Prozessor `+batch+` zum Beispiel hat keine Konfiguration, muss aber trotzdem im Abschnitt `+processors+` deklariert werden. Komponenten, die zwar konfiguriert, aber nicht in einer Pipeline enthalten sind, werden überhaupt nicht aktiv.

[,yaml]
----
service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [memory_limiter, resource, batch]
      exporters: [debug, spanmetrics, otlp/suse-observability]
    metrics:
      receivers: [otlp, spanmetrics, prometheus]
      processors: [memory_limiter, resource, batch]
      exporters: [debug, otlp/suse-observability]
----

=== Erweiterungen

Erweiterungen werden nicht direkt in Pipelines zur Verarbeitung von Daten verwendet, sondern erweitern die Möglichkeiten des Kollektors auf andere Weise. Für SUSE Observability wird sie zur Konfiguration der Authentifizierung mit einem API-Schlüssel verwendet. Erweiterungen müssen in einem Konfigurationsabschnitt definiert werden, bevor sie verwendet werden können. Ähnlich wie bei den Pipeline-Komponenten ist eine Erweiterung nur aktiv, wenn sie im Abschnitt `+service.extensions+` aktiviert ist.

[,yaml]
----
extensions:
  bearertokenauth:
    scheme: SUSEObservability
    token: "${env:API_KEY}"
service:
  extensions: [ bearertokenauth ]
----

Das opentelemetry-collector-contrib Repository hat https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/extension[alle Erweiterungen] mit Dokumentation zu deren Konfiguration.

== Umwandlung der Telemetrie

Es gibt viele Prozessoren im https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/processor[opentelemetry-collector-contrib Repository]. Hier versuchen wir, einen Überblick über die gebräuchlichen Prozessoren und ihre Fähigkeiten zu geben. Für weitere Details und viele weitere Prozessoren verwenden Sie das https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/processor[opentelemetry-collector-contrib Repository].

=== Filtern

Einige Instrumente oder Anwendungen können eine Menge Telemetriedaten erzeugen, die einfach nur verrauscht sind und für Ihren Anwendungsfall nicht benötigt werden. Mit dem https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/processor/filterprocessor[Filterprozessor] können Sie Daten, die Sie nicht benötigen, aus dem Collector entfernen, um zu vermeiden, dass die Daten an SUSE Observability gesendet werden. Zum Beispiel, um alle Daten von 1 bestimmten Dienst zu löschen:

[,yaml]
----
processors:
  filter/ignore-service1:
    error_mode: ignore
    traces:
      span:
        - resource.attributes["service.name"] == "service1"
----

Der Filterprozessor verwendet die https://github.com/open-telemetry/opentelemetry-collector-contrib/blob/main/pkg/ottl/README.md[Open Telemetry Transformation Lanuage (OTTL)], um die Filter zu definieren.

=== Hinzufügen, Ändern oder Löschen von Attributen

Der https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/processor/attributesprocessor[Attribute-Prozessor] kann Attribute von Spans, Protokollen oder Metriken ändern.

[,yaml]
----
processors:
  attributes/accountid:
    actions:
      - key: account_id
        value: 2245
        action: insert
----

Der https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/processor/resourceprocessor[Ressourcenprozessor] kann die Attribute einer xref:/setup/otel/concepts.adoc#_resources[Ressource] ändern. Zum Beispiel, um einen Kubernetes-Cluster-Namen zu jeder Ressource hinzuzufügen:

[,yaml]
----
  processors:
    resource/add-k8s-cluster:
      attributes:
      - key: k8s.cluster.name
        action: upsert
        value: my-k8s-cluster
----

Zum Ändern von Metriknamen und anderen metrikspezifischen Informationen gibt es auch den https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/processor/metricstransformprocessor[metrics transformer].

=== Verwandlungen

Der https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/processor/transformprocessor[Transformationsprozessor] kann z. B. zum Setzen eines Spannenstatus verwendet werden:

[,yaml]
----
processors:
  transform:
    error_mode: ignore
    trace_statements:
      - set(span.status.code, STATUS_CODE_OK) where span.attributes["http.request.status_code"] == 400
----

Es unterstützt viele weitere Transformationen, wie z. B. die Änderung des Bereichsnamens, die Konvertierung von Metrik-Typen oder die Änderung von Log-Events. Alle Möglichkeiten finden Sie in der https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/processor/transformprocessor[Readme-Datei]. Es verwendet die https://github.com/open-telemetry/opentelemetry-collector-contrib/blob/main/pkg/ottl/README.md[Open Telemetry Transformation Lanuage (OTTL)], um die Filter zu definieren.

== Scrub-empfindliche Daten

Der Collector ist der ideale Ort, um sensible Daten zu entfernen oder zu verschleiern, da er sich direkt zwischen Ihren Anwendungen und SUSE Observability befindet und über Prozessoren zum <<_transforming_telemetry,Filtern und Umwandeln Ihrer Daten>> verfügt. Neben den bereits besprochenen Filter- und Transformationsfunktionen steht auch ein https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/processor/redactionprocessor[Redaktionsprozessor] zur Verfügung, der Attributwerte maskieren kann, die einer Blockliste entsprechen. Es können auch Attribute entfernt werden, die nicht mit einer bestimmten Liste zulässiger Attribute übereinstimmen. Dies kann jedoch schnell dazu führen, dass die meisten Attribute wegfallen, was zu einer sehr eingeschränkten Beobachtbarkeit führt. Beachten Sie, dass es keine Ressourcenattribute verarbeitet.

Ein Beispiel, das nur bestimmte Attribute und/oder Werte ausblendet:

[,yaml]
----
processors:
  redaction:
    allow_all_keys: true
    # attributes matching the regexes on the list are masked.
    blocked_key_patterns:
      - ".*token.*"
      - ".*api_key.*"
    blocked_values: # Regular expressions for blocking values of allowed span attributes
      - '4[0-9]{12}(?:[0-9]{3})?' # Visa credit card number
      - '(5[1-5][0-9]{14})' # MasterCard number
    summary: debug
----

== Ausprobieren des Kollektors

Die Anleitungen für die ersten Schritte zeigen, wie der Collector in Kubernetes oder mit Linux-Paketen für eine produktionsreife Einrichtung bereitgestellt wird. Es ist auch möglich, es direkt als Docker-Container laufen zu lassen, um es auszuprobieren, zum Beispiel für Tests:

[,bash]
----
docker run \
  -p 127.0.0.1:4317:4317 \
  -p 127.0.0.1:4318:4318 \
  -v $(pwd)/config.yaml:/etc/otelcol-contrib/config.yaml \
  ghcr.io/open-telemetry/opentelemetry-collector-releases/opentelemetry-collector-contrib:latest
----

Dabei wird das Collector-Contrib-Image verwendet, das alle beigetragenen Komponenten (Empfänger, Prozessoren usw.) enthält. Eine kleinere, eingeschränkte Version des Bildes ist ebenfalls verfügbar, die jedoch nur eine sehr begrenzte Anzahl von Komponenten enthält:

[,bash]
----
docker run \
  -p 127.0.0.1:4317:4317 \
  -p 127.0.0.1:4318:4318 \
  -v $(pwd)/config.yaml:/etc/otelcol/config.yaml \
  ghcr.io/open-telemetry/opentelemetry-collector-releases/opentelemetry-collector:latest
----

Beachten Sie, dass die Kubernetes-Installation standardmäßig die Kubernetes-Distribution des Collector-Images `+ghcr.io/open-telemetry/opentelemetry-collector-releases/opentelemetry-collector-k8s+` verwendet, die mehr Komponenten als das Basic-Image, aber weniger als das Contrib-Image enthält. Wenn Sie mit diesem Image auf fehlende Komponenten stoßen, können Sie einfach das Contrib-Image , `+ghcr.io/open-telemetry/opentelemetry-collector-releases/opentelemetry-collector-contrib+`, stattdessen verwenden.

== Fehlersuche

=== HTTP-Anfragen vom Exporteur sind zu groß

In einigen Fällen können HTTP-Anforderungen für Telemetriedaten sehr groß werden und von SUSE Observability abgelehnt werden. SUSE Observability hat ein Limit von 4 MB für das gRPC-Protokoll. Wenn Sie bei HTTP-Anfragen an Grenzen stoßen, können Sie die Anfragegröße verringern, indem Sie den Komprimierungsalgorithmus ändern und die maximale Stapelgröße begrenzen.

==== Komprimierung von HTTP-Anfragen

In den Anleitungen für die ersten Schritte wird die Komprimierung `+snappy+` für den Collector aktiviert. Dies ist nicht die beste Komprimierung, verbraucht aber weniger CPU-Ressourcen als `+gzip+`. Wenn Sie die Komprimierung entfernt haben, können Sie sie wieder aktivieren, oder Sie können zu einem Komprimierungsalgorithmus wechseln, der ein besseres https://github.com/open-telemetry/opentelemetry-collector/blob/main/config/configgrpc/README.md#_compression_comparison[Komprimierungsverhältnis] bietet. Für gRPC- und HTTP-Protokolle sind dieselben Komprimierungsarten verfügbar.

==== Maximale Losgröße

Die Größe der HTTP-Anfrage kann durch Hinzufügen einer Konfiguration zum `+batch+` Prozessor reduziert werden, die die Stapelgröße begrenzt:

[,yaml]
----
processor:
  batch: {}
    send_batch_size: 8192 # This is the default value
    send_batch_max_size: 10000 # The default is 0, meaning no max size at all
----

Die Stapelgröße wird in der Anzahl der Spans, metrischen Datenpunkte oder Protokolldatensätze (nicht in Bytes) definiert, so dass Sie möglicherweise etwas experimentieren müssen, um die richtige Einstellung für Ihre Situation zu finden. Weitere Einzelheiten entnehmen Sie bitte der https://github.com/open-telemetry/opentelemetry-collector/blob/main/processor/batchprocessor/README.md[Dokumentation].
