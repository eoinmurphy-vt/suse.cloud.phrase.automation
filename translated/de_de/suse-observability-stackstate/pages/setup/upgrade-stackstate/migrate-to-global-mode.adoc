= Wechseln Sie in den globalen Konfigurationsmodus
:description: {stackstate-product-name} Selbst gehostet
:page-toclevels: 3

== Übersicht

In diesem Leitfaden wird erläutert, wie Sie von der herkömmlichen zweistufigen Installation unter Verwendung des `+suse-observability-values+` Diagramms auf die vereinfachte Ein-Diagramm-Installation unter Verwendung der `+global.suseObservability+` Konfiguration migrieren.

[NOTE]
====
Die Konfigurationsmethode `+global.suseObservability+` ist ab der Version `+2.8.0+` verfügbar und ist jetzt die empfohlene Installationsmethode. Die alte `+suse-observability-values+` Diagramm-Methode ist veraltet, wird aber für bestehende Installationen weiterhin funktionieren.
====

=== Ältere Installation (veraltet)

Die alte Installationsmethode erforderte zwei Schritte:

. Erzeugen von Wertedateien mit Hilfe des `+suse-observability-values+` Diagramms
. Installieren Sie das `+suse-observability+` Diagramm mit den generierten Werten

[,text]
----
# Step 1: Generate values
helm template suse-observability-values suse-observability/suse-observability-values \
  --set license="YOUR-LICENSE-KEY" \
  --set baseUrl="https://observability.example.com" \
  --set sizing.profile="150-ha" \
  --set adminPassword="your-password" \
  --set pullSecret.username="registry-user" \
  --set pullSecret.password="registry-pass" \
  > generated-values.yaml

# Step 2: Install with generated values
helm install suse-observability suse-observability/suse-observability \
  -f generated-values.yaml \
  -n suse-observability
----

=== Neuinstallation (empfohlen)

Die neue Methode verwendet eine einzige Wertedatei mit der Konfiguration `+global.suseObservability+`:

[,text]
----
helm install suse-observability suse-observability/suse-observability \
  -f values.yaml \
  -n suse-observability
----

== Vor der Installation umwandeln

Wenn Sie die Parameter für das alte `+suse-observability-values+` Diagramm vorbereitet, aber noch nicht installiert haben, können Sie direkt auf die neue Methode umstellen, ohne Dateien mit Zwischenwerten zu erstellen.

=== Direkte Parameterumwandlung

Konvertieren Sie Ihren `+helm template+` Befehl in einen `+helm upgrade --install+` Befehl, indem Sie die Parameter zuordnen:

[cols="1,1"]
|===
| Alte Parameter | Neue Parameter

| `+--set license="..."+`
| `+--set global.suseObservability.license="..."+`

| `+--set baseUrl="..."+`
| `+--set global.suseObservability.baseUrl="..."+`

| `+--set sizing.profile="..."+`
| `+--set global.suseObservability.sizing.profile="..."+`

| `+--set receiverApiKey="..."+`
| `+--set global.suseObservability.receiverApiKey="..."+`

| `+--set adminPassword="..."+`
| `+--set global.suseObservability.adminPassword="..."+`

| `+--set imageRegistry="..."+`
| `+--set global.imageRegistry="..."+`

| `+--set pullSecret.username="..."+`
| `+--set global.suseObservability.pullSecret.username="..."+`

| `+--set pullSecret.password="..."+`
| `+--set global.suseObservability.pullSecret.password="..."+`
|===

=== Beispiel: Vorher und nachher

[NOTE]
====
Vergewissern Sie sich vor dem Ausführen von Steuerkommandos, dass Sie die neueste Kartenversion haben:

[,bash]
----
helm repo update
----
====

.Legacy-Befehl (nicht verwenden)
[,bash]
----
export VALUES_DIR=.
helm template \
  --set license="YOUR-LICENSE-KEY" \
  --set baseUrl="https://observability.example.com" \
  --set sizing.profile="10-nonha" \
  --set receiverApiKey="YOUR-API-KEY" \
  suse-observability-values \
  suse-observability/suse-observability-values --output-dir $VALUES_DIR

# Then: helm install with generated files...
----

.Neuer Befehl (empfohlen)
[,bash]
----
helm upgrade --install \
  --namespace suse-observability \
  --create-namespace \
  --set global.suseObservability.license="YOUR-LICENSE-KEY" \
  --set global.suseObservability.baseUrl="https://observability.example.com" \
  --set global.suseObservability.sizing.profile="10-nonha" \
  --set global.suseObservability.receiverApiKey="YOUR-API-KEY" \
  --set global.suseObservability.adminPassword='your-password' \
  suse-observability suse-observability/suse-observability
----

[NOTE]
====
* Die `+adminPassword+` muss ein bcrypt-Hash in der neuen Methode sein. Unter <<bcrypt-hash>> erfahren Sie, wie Sie eine solche erstellen können.
* Für Umgebungen mit Luftabdeckung oder benutzerdefinierte Registrierungen siehe xref:/k8s-suse-rancher-prime-air-gapped.adoc[ {stackstate-product-name} im Luftabdeckungsmodus installieren] für zusätzliche Konfiguration einschließlich `+imageRegistry+` und `+pullSecret+`.
====

=== Stattdessen eine Wertedatei verwenden

Wenn Sie die Verwendung einer Wertedatei bevorzugen, erstellen Sie `+values.yaml+` mit der in <<_step_3_create_new_values_file>> gezeigten Konfiguration, stellen Sie sicher, dass Sie die neueste Tabelle mit `+helm repo update+` haben, und führen Sie sie aus:

[,bash]
----
helm upgrade --install \
  --namespace suse-observability \
  --create-namespace \
  --values values.yaml \
  suse-observability suse-observability/suse-observability
----

== Bevor Sie migrieren

Wenn Sie eine Migration von der alten Installationsmethode durchführen, sollten Sie Folgendes beachten:

[CAUTION]
====
Dieser Migrationsleitfaden richtet sich an Benutzer, die derzeit die `+suse-observability-values+` Diagramm-Methode verwenden. Wenn Sie ein Upgrade von StackState 6.0 durchführen, folgen Sie stattdessen der xref:/setup/upgrade-stackstate/migrate-from-6.adoc[Anleitung zur Migration von StackState].
====

[NOTE]
====
* Die Migration wird mit einem Standard-Helm-Upgrade-Befehl durchgeführt
* Ihre bestehenden Daten und Konfigurationen werden beibehalten.
* Die Migration kann während eines Wartungsfensters durchgeführt werden
* Ein Rollback ist mit den Standard-Rollback-Verfahren von Helm möglich
====

== Schritte der Migration

=== Schritt 1: Sicherung der aktuellen Konfiguration

Bevor Sie mit der Migration beginnen, speichern Sie Ihre bestehenden Helm-Werte und alle Anpassungen:

[,text]
----
# Save current Helm values
helm get values suse-observability -n suse-observability > backup-values.yaml

# Save your original suse-observability-values input (if available)
# This is typically your values file used with helm template

# Backup important resources
kubectl get secrets -n suse-observability -o yaml > backup-secrets.yaml
----

=== Schritt 2: Identifizieren Sie Ihre aktuelle Konfiguration

Überprüfen Sie Ihre bestehende Konfiguration, um die folgenden Einstellungen zu ermitteln:

* *Größenprofil*: Überprüfen Sie `+sizing.profile+` in Ihrer Eingabe für die Suse-Beobachtungswerte
* *Lizenzschlüssel*: Ihre {stackstate-product-name} Lizenz
* *Basis-URL*: Die URL, unter der {stackstate-product-name} zugänglich ist
* *Administrator-Passwort*: Das Admin-Passwort (Sie können auch den bcrypt-Hash des Passworts verwenden, der Wert dafür lautet `+adminPasswordBcrypt+`)
* *Geheime Zugangsdaten abrufen*: Registry-Benutzername und -Passwort (falls verwendet)
* *Benutzerdefinierte Affinitätseinstellungen*: Beliebige Anpassungen der Knotenaffinität oder Pod-Anti-Affinität

Sie können einige von ihnen aus Ihrer aktuellen Installation extrahieren:

[,text]
----
# Check current values
helm get values suse-observability -n suse-observability

# The sizing profile is visible in resource configurations
# Look for patterns like replica counts, resource limits, etc.
----

=== Automatisch generierte und benutzerdefinierte Werte verstehen

Wenn Sie `+helm get values+` ausführen, sehen Sie möglicherweise Hunderte von Konfigurationszeilen. Die meisten davon werden jedoch *automatisch* vom Größenprofil *generiert* und müssen *nicht* manuell in Ihrer Migrationswertedatei gespeichert werden.

==== Was das Größenprofil automatisch liefert

Wenn Sie `+global.suseObservability.sizing.profile+` einstellen, wird das Diagramm automatisch konfiguriert:

* Ressourcenanforderungen und -grenzen für alle Komponenten (Elasticsearch, Kafka, ClickHouse, Server, Empfänger usw.)
* Anzahl der Replikate für zustandsabhängige Dienste
* Speichergrößen für persistente Volumes
* HBase-Bereitstellungsmodus (Mono für Nicht-HA, Verteilt für HA)
* Split-Konfiguration von Server und Empfänger
* Victoria Metrics HA-Einstellungen
* Umgebungsvariablen (`+extraEnv+`) einschließlich `+CONFIG_FORCE_*+` Einstellungen für die Leistungsoptimierung

==== Was Sie bewahren müssen

Behalten Sie nur Werte bei, die *wirklich benutzerdefiniert* sind, d. h. Konfigurationen, die Sie hinzugefügt oder geändert haben und die über die im Größenprofil angegebenen Werte hinausgehen:

[cols="1,1"]
|===
| Bewahren (Benutzerdefiniert) | Nicht aufbewahren (automatisch generiert)

| Grundeinstellungen (Lizenz, baseUrl, adminPassword, receiverApiKey)
| Ressourcenbeschränkungen und -anforderungen

| Ingress-Konfiguration (Hosts, TLS, Anmerkungen)
| Replik zählt

| Benutzerdefinierte Umgebungsvariablen, die Sie hinzugefügt haben
| Speichergrößen

| Authentifizierungseinstellungen (LDAP, OIDC)
| `+CONFIG_FORCE_*+` Umgebungsvariablen

| Benutzerdefiniert `+allowedOrigins+`
| Kafka/Zookeeper-Replikationsfaktoren

| OpenTelemetry-Sammler benutzerdefinierter Eingang
| HBase-Einsatzmodus
|===

==== Beispiel: Datei mit minimalen Migrationswerten

Ihre Datei mit Migrationswerten sollte *viel kleiner* sein als die Ausgabe von `+helm get values+`:

[,yaml]
----
global:
  suseObservability:
    license: "YOUR-LICENSE"
    baseUrl: "https://your-url.example.com"
    sizing:
      profile: "10-nonha"  # This handles all resource configuration!
    adminPassword: "your-password"
    receiverApiKey: "your-api-key"

# Only include truly custom configurations below
ingress:
  enabled: true
  ingressClassName: your-ingress-class
  annotations:
    # your custom annotations
  hosts:
  - host: your-url.example.com
  tls:
  - hosts:
    - your-url.example.com
    secretName: your-tls-secret
----

[TIP]
====
Wenn Sie sich nicht sicher sind, ob ein Wert benutzerdefiniert ist oder automatisch generiert wurde, vergleichen Sie ihn mit den Standardwerten des Größenprofils. Werte wie die Umgebungsvariablen `+elasticsearch.resources+`, `+kafka.resources+`, `+stackstate.components.server.resources+` und `+CONFIG_FORCE_*+` werden alle vom Profil erzeugt und können getrost weggelassen werden.
====

=== Schritt 3: Neue Wertedatei erstellen

Erstellen Sie eine neue `+values.yaml+` mit der Struktur `+global.suseObservability+`:

[,yaml]
----
global:
  suseObservability:
    # Required: Your license key
    license: "YOUR-LICENSE-KEY"

    # Required: Base URL for {stackstate-product-name}
    baseUrl: "https://observability.example.com"

    # Required: Sizing profile
    # Available: trial, 10-nonha, 20-nonha, 50-nonha, 100-nonha,
    #            150-ha, 250-ha, 500-ha, 4000-ha
    sizing:
      profile: "150-ha"

    adminPassword: "your-password"
    # Instead of adminPassword you can provide a bcrypt hashed password
    # Generate with: htpasswd -bnBC 10 "" "your-password" | tr -d ':\n'
    # adminPasswordBcrypt: "$2a$10$..."

    # Optional: Receiver API key (auto-generated if not provided)
    receiverApiKey: "your-receiver-api-key"

    # Optional: Affinity configuration
    # These settings apply to all infrastructure components (Elasticsearch,
    # Kafka, ClickHouse, etc.) automatically - no need to configure per-component
    affinity:
      # Node affinity for all components
      nodeAffinity: null

      # Pod anti-affinity for HA profiles (infrastructure components)
      podAntiAffinity:
        requiredDuringSchedulingIgnoredDuringExecution: true
        topologyKey: "kubernetes.io/hostname"

  # For air-gapped or custom registry configurations, see:
  # xref:/k8s-suse-rancher-prime-air-gapped.adoc[Air-Gapped Installation Guide]
----

=== Schritt 4: Referenz der Konfigurationsabbildung

Verwenden Sie diese Tabelle, um Ihre alte Konfiguration auf das neue Format abzubilden:

[cols="1,1"]
|===
| Alte Konfiguration (suse-observability-values) | Neue Konfiguration (global.suseObservability)

| `+license+`
| `+global.suseObservability.license+`

| `+baseUrl+`
| `+global.suseObservability.baseUrl+`

| `+sizing.profile+`
| `+global.suseObservability.sizing.profile+`

| `+adminPassword+`
| `+global.suseObservability.adminPassword+`

| `+receiverApiKey+`
| `+global.suseObservability.receiverApiKey+`

| `+affinity.nodeAffinity+`
| `+global.suseObservability.affinity.nodeAffinity+`

| `+affinity.podAntiAffinity.requiredDuringSchedulingIgnoredDuringExecution+`
| `+global.suseObservability.affinity.podAntiAffinity.requiredDuringSchedulingIgnoredDuringExecution+`

| `+affinity.podAntiAffinity.topologyKey+`
| `+global.suseObservability.affinity.podAntiAffinity.topologyKey+`

| `+rancherUrl+`
| `+stackstate.allowedOrigins+` (manuelle Konfiguration)

| `+imageRegistry+` (nur luftgekühlt)
| `+global.imageRegistry+`

| `+pullSecret.username+` (nur luftgekühlt)
| `+global.suseObservability.pullSecret.username+`

| `+pullSecret.password+` (nur luftgekühlt)
| `+global.suseObservability.pullSecret.password+`
|===

[NOTE]
====
Für Umgebungen mit Luftabdeckung oder benutzerdefinierte Registrierungen, die eine Konfiguration von `+imageRegistry+` und `+pullSecret+` erfordern, siehe xref:/k8s-suse-rancher-prime-air-gapped.adoc[Installation von {stackstate-product-name} im Luftabdeckungsmodus].
====

=== Schritt 5: Umgang mit benutzerdefinierten Überschreibungen

Wenn Sie benutzerdefinierte Überschreibungen in Ihrer generierten Wertedatei hatten, können Sie diese weiterhin verwenden. Der neue globale Modus bietet sinnvolle Standardeinstellungen, die für jede einzelne Komponente überschrieben werden können:

[,yaml]
----
global:
  suseObservability:
    sizing:
      profile: "150-ha"
    license: "YOUR-LICENSE-KEY"
    baseUrl: "https://observability.example.com"
    adminPassword: "your-password"
    # Global affinity - applied to all components by default
    affinity:
      nodeAffinity:
        requiredDuringSchedulingIgnoredDuringExecution:
          nodeSelectorTerms:
          - matchExpressions:
            - key: topology.kubernetes.io/zone
              operator: In
              values:
              - us-west-2a

# Custom per-component overrides still work - they take precedence over global defaults
stackstate:
  components:
    api:
      resources:
        requests:
          memory: 16Gi  # Override default from sizing profile
        limits:
          memory: 20Gi

elasticsearch:
  volumeClaimTemplate:
    resources:
      requests:
        storage: 500Gi  # Override default storage size
  # Override affinity for Elasticsearch specifically
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: node-type
            operator: In
            values:
            - storage-optimized
----

=== Schritt 6: Führen Sie das Upgrade durch

Vergewissern Sie sich zunächst, dass Sie die neueste Version der Karte haben:

[,text]
----
helm repo update
----

Führen Sie dann das Upgrade durch:

[,text]
----
# Dry-run first to see what will change
helm upgrade suse-observability suse-observability/suse-observability \
  -n suse-observability \
  -f values.yaml \
  --dry-run

# If everything looks good, perform the actual upgrade
helm upgrade suse-observability suse-observability/suse-observability \
  -n suse-observability \
  -f values.yaml
----

=== Schritt 7: Überprüfen Sie das Upgrade

[,text]
----
# Check all pods are running
kubectl get pods -n suse-observability

# Check for any errors in events
kubectl get events -n suse-observability --sort-by='.lastTimestamp' | tail -20

# Verify the Helm release
helm status suse-observability -n suse-observability

# Check the applied values
helm get values suse-observability -n suse-observability
----

[[bcrypt-hash]]
== Administrator-Passwort

Bei der alten Methode wurde die `+adminPassword+` als Klartext an die Tabelle suse-observability-values übergeben, die sie in einen bcrypt-Hash umwandelte. Bei der neuen Methode können Sie den Klartext `+adminPassword+` an die Suse-Beobachtungstabelle weitergeben oder den bcrypt-Hash direkt an die Suse-Beobachtungstabelle mit `+adminPasswordBcrypt+` weitergeben.

Wenn Sie einen neuen bcrypt-Hash erzeugen müssen:

[,bash]
----
htpasswd -bnBC 10 "" "your-password" | tr -d ':\n'
----

== Fehlersuche

=== Pods stecken im Status "Ausstehend" fest

Dies deutet in der Regel auf ein Planungsproblem mit Anti-Affinitätsregeln hin. Bei Clustern mit einer begrenzten Anzahl von Knoten sollten Sie Soft Anti-Affinity verwenden:

[,yaml]
----
global:
  suseObservability:
    affinity:
      podAntiAffinity:
        requiredDuringSchedulingIgnoredDuringExecution: false  # Use soft anti-affinity
----

=== Ressourcen unterscheiden sich von der vorherigen Installation

Sizing-Profile können aktualisierte Ressourcenempfehlungen enthalten. Um Ihre bisherigen Einstellungen beizubehalten, fügen Sie explizite Überschreibungen hinzu:

[,yaml]
----
global:
  suseObservability:
    sizing:
      profile: "150-ha"

# Override with your previous resource values
stackstate:
  components:
    api:
      resources:
        requests:
          memory: 8Gi  # Your previous value
----

=== Konflikte zwischen Konfigurationsstilen

Mischen Sie nicht alte und neue Konfigurationsstile. Verwenden Sie entweder:

* `+global.suseObservability.*+` (neuer Stil), ODER
* `+stackstate.license.key+`, `+stackstate.baseUrl+`, usw. (alter Stil)

[,text]
----
# Check for conflicting values
helm get values suse-observability -n suse-observability | grep -E "license|baseUrl"
----

=== Rollback, falls erforderlich

[,text]
----
# List release history
helm history suse-observability -n suse-observability

# Rollback to previous revision
helm rollback suse-observability <revision-number> -n suse-observability
----

== Wesentliche Unterschiede

[cols="1,1,1"]
|===
| Aspekt | Vermächtnis (suse-observability-values) | Neu (global.suseObservability)

| Installationsschritte
| 2 (Vorlage + Installation)
| 1 (installieren)

| Verwaltung von Wertedateien
| Erzeugt, muss gespeichert werden
| Eine einzige Quelle der Wahrheit

| Profil-Updates
| Manuelle Regeneration erforderlich
| Automatisch mit Karten-Upgrades

| Geheimnis ziehen
| Separates Unterdiagramm
| Eingebaut, automatisch konfiguriert

| Victoria Metrics HA
| Handbuch `+enabled: true/false+`
| Automatisch pro Profil konfiguriert

| Konfiguration der Affinität
| Erzeugt pro Komponente (separate Einstellungen für jeden Dienst)
| Zentralisierte globale Einstellungen (können bei Bedarf immer noch pro Komponente überschrieben werden)
|===

[NOTE]
====
Die globale Affinitätskonfiguration bietet Standardeinstellungen für alle Infrastrukturkomponenten. Sie können die Affinität für bestimmte Komponenten (wie Elasticsearch oder Kafka) bei Bedarf immer noch außer Kraft setzen, indem Sie Affinitätswerte auf Komponentenebene festlegen. Einstellungen auf Komponentenebene haben Vorrang vor globalen Standardwerten.
====

== Aufräumen

Nach erfolgreicher Migration können Sie:

. Entfernen Sie die generierten Wertedateien (sie werden nicht mehr benötigt)
. Aktualisieren Sie Ihre Verteilungsskripte, um die neue einstufige Installation zu verwenden
. Entfernen Sie alle Verweise auf `+suse-observability-values+` chart in Ihren CI/CD-Pipelines
