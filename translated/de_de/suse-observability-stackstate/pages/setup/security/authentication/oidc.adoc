= Offene ID-Verbindung (OIDC)
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: SUSE Observability Selbstgehostet

== Übersicht

SUSE Observability kann sich über einen OIDC-Authentifizierungsanbieter authentifizieren. Um dies zu ermöglichen, müssen Sie sowohl SUSE Observability als auch den OIDC-Anbieter so konfigurieren, dass sie miteinander kommunizieren können. In den folgenden Abschnitten werden die jeweiligen Einstellungen beschrieben.

== Konfigurieren Sie den OIDC-Anbieter

Bevor Sie SUSE Observability für die Authentifizierung über OIDC konfigurieren können, müssen Sie einen Client für SUSE Observability auf Ihrem OIDC-Anbieter erstellen.

=== Rancher

[NOTE]
Dies funktioniert nur mit Rancher 2.12 oder höher.  Sie müssen https://documentation.suse.com/cloudnative/rancher-manager/latest/en/rancher-admin/users/authn-and-authz/configure-oidc-provider.html[Rancher als OIDC-Anbieter konfigurieren].

Erstellen Sie eine OIDCClient-Ressource im lokalen Rancher-Cluster:
[,yaml]
----
apiVersion: management.cattle.io/v3
kind: OIDCClient
metadata:
  name: oidc-observability
spec:
  tokenExpirationSeconds: 600
  refreshTokenExpirationSeconds: 3600
  redirectURIs:
    - "https://<observability-base-url>/loginCallback?client_name=StsOidcClient"
----

Rancher macht die Client-ID und das Geheimnis in seinem Statusfeld verfügbar:
[,yaml]
----
apiVersion: management.cattle.io/v3
kind: OIDCClient
metadata:
  name: oidc-observability
spec:
  tokenExpirationSeconds: 600
  refreshTokenExpirationSeconds: 3600
  redirectURIs:
    - "https://<observability-base-url>/loginCallback?client_name=StsOidcClient"
status:
  clientID: <oidc-client-id>
  clientSecrets:
    client-secret-1:
      createdAt: "xxx"
      lastFiveCharacters: xxx
----

Und Sie können den geheimen Wert erhalten mit
[,bash]
----
kubectl get secret <oidc-client-id> -n cattle-oidc-client-secrets -o jsonpath="{.data.client-secret-1}" | base64 -d
----


=== Andere OIDC-Anbieter

Verwenden Sie die folgenden Einstellungen für den Client (falls vom OIDC-Anbieter benötigt):

* Verwenden Sie den OIDC-Autorisierungsfluss, der oft auch als Autorisierungscodefluss bezeichnet wird. SUSE Observability unterstützt keine impliziten Grant- und hybriden Flüsse, sodass diese nicht aktiviert werden müssen.
* Legen Sie die *Redirect-URI* auf die Basis-URL von SUSE Observability mit der Endung `+/loginCallback?client_name=StsOidcClient+` fest. Zum Beispiel `+https://stackstate.acme.com/loginCallback?client_name=StsOidcClient+`. Bei einigen OIDC-Anbietern, wie z. B. Google und Azure Entra ID, muss der Redirect-URI genau übereinstimmen, einschließlich aller Abfrageparameter. In diesem Fall sollten Sie den URI wie folgt konfigurieren: `+https://stackstate.acme.com/loginCallback?client_name=StsOidcClient+`.
* Ermöglichen Sie SUSE Observability mindestens den Zugriff auf die Bereiche `+openid+` und `+email+` oder das Äquivalent dieser Bereiche für Ihren OIDC-Anbieter. Je nach Anbieter können weitere Bereiche erforderlich sein; falls eine separate `+profile+` vorhanden ist, ist diese ebenfalls einzubeziehen.
* SUSE Observability benötigt OIDC-Offline-Zugriff. Bei einigen Identitätsanbietern ist dafür ein zusätzlicher Bereich erforderlich, der in der Regel `+offline_access+` heißt.

Das Ergebnis dieser Konfiguration sollte eine *clientId* und ein *Geheimnis* ergeben. Kopieren Sie diese und bewahren Sie sie für die Konfiguration von SUSE Observability auf. Notieren Sie auch die *discoveryUri* des Anbieters. Normalerweise ist dies entweder auf demselben Bildschirm oder in der Dokumentation zu finden.

== Konfigurieren von SUSE Observability für OIDC

=== Rancher

[NOTE]
Dies funktioniert nur mit Rancher 2.12 oder höher.  Sie müssen https://documentation.suse.com/cloudnative/rancher-manager/latest/en/rancher-admin/users/authn-and-authz/configure-oidc-provider.html[Rancher als OIDC-Anbieter konfigurieren].

Um Rancher als OIDC-Anbieter für SUSE Observability zu konfigurieren, müssen Sie die OIDC-Details zu den Authentifizierungswerten hinzufügen:
[,yaml]
----
stackstate:
  authentication:
    rancher:
      clientId: "<oidc-client-id>"
      secret: "<oidc-secret>"
      baseUrl: "<rancher-url>"
----
Sie können die OIDC-Konfiguration für Rancher mit den folgenden Feldern außer Kraft setzen und erweitern:

* **discoveryUri** - URI, die Sie verwenden können, um den OIDC-Anbieter zu finden. Normalerweise auch dokumentiert oder zurückgegeben beim Anlegen des Mandanten im OIDC-Provider.
* **redirectUri** - Optional (nicht in diesem Beispiel): Der URI, über den der Login-Callback-Endpunkt von SUSE Observability erreichbar ist. Wird standardmäßig mit `+stackstate.baseUrl+` gefüllt, kann aber überschrieben werden. Dies muss eine vollständig qualifizierte URL sein, die auf den Pfad `+/loginCallback+` verweist.
* **customParameters** - Optionale Zuordnung von Schlüssel/Wertpaaren, die Sie als benutzerdefinierte Anfrageparameter an den OIDC-Anbieter senden. Bestimmte OIDC-Anbieter benötigen zusätzliche Anfrageparameter, die nicht standardmäßig gesendet werden.

Wenn Sie die TLS-Überprüfung deaktivieren müssen, weil eine Einrichtung keine überprüfbaren SSL-Zertifikate verwendet, können Sie die SSL-Überprüfung in der Anwendungskonfiguration deaktivieren (nicht in der Produktion verwenden):

Für **Nicht-HA-Setup**:
[,yaml]
----
stackstate:
  components:
    server:
      extraEnv:
        open:
          CONFIG_FORCE_stackstate_misc_sslCertificateChecking: false
----
Für **HA (High Availability)** Setup:
[,yaml]
----
stackstate:
  components:
    api:
      extraEnv:
        open:
          CONFIG_FORCE_stackstate_misc_sslCertificateChecking: false
----

Bei HA-Bereitstellungen geht die API-Komponente anders mit der Konfiguration um. Verwenden Sie das Folgende:=== Kubernetes

Um SUSE Observability für die Verwendung eines OIDC-Authentifizierungsanbieters in Kubernetes zu konfigurieren, müssen Sie die OIDC-Details und die Benutzerrollenzuordnung in der Datei `+authentication.yaml+` hinzufügen. Zum Beispiel:

[,yaml]
----
stackstate:
  authentication:
    oidc:
      clientId: "<client-id-from-oidc-provider>"
      secret: "<secret-from-oidc-provider>"
      discoveryUri: "https://oidc.acme.com/.well-known/openid-configuration"
      jwsAlgorithm: RS256
      scope: ["openid", "email"]
      jwtClaims:
        usernameField: email
        displayNameField: name
        groupsField: groups
      customParameters:
        access_type: offline

    # map the groups from OIDC provider
    # to the 4 standard roles in SUSE Observability (guest, powerUser, k8sTroubleshooter and admin)
    roles:
      guest: ["guest-group-in-oidc-provider"]
      powerUser: ["powerUser-group-in-oidc-provider"]
      admin: ["admin-group-in-oidc-provider"]
      k8sTroubleshooter: ["troubleshooter-group-in-oidc-provider"]
----

Führen Sie die folgenden Schritte aus, um SUSE Observability für die Authentifizierung mit OIDC zu konfigurieren:

. In `+authentication.yaml+` - fügen Sie die Details des OIDC-Authentifizierungsanbieters hinzu (siehe das obige Beispiel):
 ** *clientId* - Die ID des xref:/setup/security/authentication/oidc.adoc#_configure_the_oidc_provider[OIDC-Clients, den Sie für SUSE Observability erstellt haben].
 ** *secret* - Das Geheimnis für den xref:/setup/security/authentication/oidc.adoc#_configure_the_oidc_provider[OIDC-Client, den Sie für SUSE Observability erstellt haben]
 ** *discoveryUri* - URI, die zum Auffinden des OIDC-Anbieters verwendet werden kann. Normalerweise auch dokumentiert oder zurückgegeben beim Anlegen des Mandanten im OIDC-Provider.
 ** *jwsAlgorithmus* - Der Standard für OIDC ist `+RS256+`. Wenn Ihr OIDC-Anbieter einen anderen verwendet, kann dieser hier eingestellt werden.
 ** *scope* - Sollte mit dem in der OIDC-Anbieterkonfiguration angegebenen Bereich übereinstimmen oder eine Teilmenge davon sein. SUSE Observability verwendet dies, um den Zugriff auf diese Teile eines Benutzerprofils im OIDC-Anbieter anzufordern.
 ** *redirectUri* - Optional (nicht in diesem Beispiel): Der URI, über den der Login-Callback-Endpunkt von SUSE Observability erreichbar ist. Wird standardmäßig mit `+stackstate.baseUrl+` gefüllt, kann aber überschrieben werden. Dies muss eine vollständig qualifizierte URL sein, die auf den Pfad `+/loginCallback+` verweist.
 ** *customParameters* - Optionale Zuordnung von Schlüssel/Wertpaaren, die als benutzerdefinierte Anfrageparameter an den OIDC-Anbieter gesendet werden. Einige OIDC-Anbieter benötigen zusätzliche Anfrageparameter, die nicht standardmäßig gesendet werden.
 ** *jwtClaims* -
  *** *usernameField* - Das Feld im OIDC-Benutzerprofil, das als Benutzername verwendet werden soll. Standardmäßig ist dies die Adresse `+preferred_username+`. Viele Anbieter lassen dieses Feld jedoch weg. Eine gute Alternative ist `+email+`.
  *** *displayNameField* - Das Feld im OIDC-Benutzerprofil, das als DisplayName verwendet werden soll. Standardmäßig wird dies die `+name+` sein.
  *** *groupsField* - Das Feld, aus dem SUSE Observability die Rolle/Gruppe für einen Benutzer liest.
. In `+authentication.yaml+` - ordnen Sie die Benutzerrollen von OIDC den richtigen SUSE Observability-Subjekten zu, indem Sie die Einstellungen `+roles.guest+`, `+roles.powerUser+`, `+roles.admin+` oder `+roles.platformAdmin+` verwenden (siehe das Beispiel oben). Weitere Informationen finden Sie unter den xref:/setup/security/rbac/rbac_permissions.adoc#_predefined_roles[Standardrollen von SUSE Observability]. Es können auch weitere SUSE Observability-Rollen erstellt werden, siehe die xref:/setup/security/rbac/README.adoc[RBAC-Dokumentation].
. Speichern Sie die Datei `+authentication.yaml+` zusammen mit der Datei `+values.yaml+` aus der Installationsanleitung von SUSE Observability.
. Führen Sie ein Helm-Upgrade durch, um die Änderungen zu übernehmen:
+
[,text]
----
 helm upgrade \
   --install \
   --namespace suse-observability \
   --values values.yaml \
   --values authentication.yaml \
 suse-observability \
 suse-observability/suse-observability
----

[NOTE]
====
*Anmerkung:*

* Die erste Ausführung des Befehls helm upgrade führt zu einem Neustart der Pods, was zu einer kurzen Unterbrechung der Verfügbarkeit führen kann.
* Fügen Sie `+authentication.yaml+` in jeden `+helm upgrade+` Lauf ein.
* Die Authentifizierungskonfiguration wird als Kubernetes-Geheimnis gespeichert.
====

== Verwendung eines externen Geheimnisses

Wenn die oidc-Geheimnisse von einem externen Geheimnis stammen sollen, befolgen Sie xref:/setup/security/external-secrets.adoc#_getting_authentication_data_from_an_external_secret[diese Schritte], geben aber die folgenden Daten ein:

[,yaml]
----
kind: Secret
metadata:
   name: "<custom-secret-name>"
type: Opaque
data:
  oidc_client_id: <base64 of client id>
  oidc_secret: <base64 of secret>
----
