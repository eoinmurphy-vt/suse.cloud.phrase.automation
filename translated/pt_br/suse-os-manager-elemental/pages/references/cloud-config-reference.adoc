= Referência do Cloud-config
:revdate: 2025-10-21
:page-revdate: {revdate}

Espera-se que as imagens do Node OS criadas com o https://github.com/elemental-toolkit[kit de ferramentas] https://github.com/rancher/yip[{elemental-product-name} sejam inicializadas e configuradas com o uso do] yip. O Yip é um pequeno utilitário para aplicar um conjunto de ações e configurações ao sistema descrito com arquivos yaml. O Yip é integrado e consumido como uma biblioteca no binário do cliente elementar (consulte `+elemental run-stage --help+`). Yip agrupa as configurações e ações a serem aplicadas em _estágios_ arbitrários; por exemplo, a chamada de comando `+elemental run-stage network+` aplicaria somente as ações e configurações definidas para o estágio denominado `+network+`. Observe que, na perspectiva de Yip, os estágios podem ser executados a qualquer momento, pois eles simplesmente agrupam um conjunto de ações com um nome arbitrário.

{elemental-product-name} O Toolkit integra cinco estágios predefinidos ao processo de inicialização do sistema operacional.

. *`+pre-rootfs+`*: Esse estágio é executado na inicialização inicial dentro do disco ram de inicialização, logo antes de montar o dispositivo raiz (normalmente em `+/sysroot+`). Esse estágio pode ser usado para definir as etapas da primeira inicialização que são necessárias antes de montar o dispositivo raiz. Um bom exemplo é a expansão da partição do dispositivo raiz. Executado como parte do `+initrd-root-device.target+`.
. *`+rootfs+`*: Esse estágio é executado na inicialização inicial dentro do disco ram de inicialização, logo após a montagem do dispositivo raiz (normalmente em `+/sysroot+`). Esse estágio pode ser usado para definir as etapas da primeira inicialização, como a criação de novas partições. Os caminhos efêmeros e persistentes são normalmente definidos nesse estágio. Executado como parte do `+initrd-root-fs.target+`.
. *`+initramfs+`*: Esse estágio também é executado dentro do disco ram de inicialização, mas em um estágio posterior, logo antes de mudar para a raiz. Esse estágio é executado em um ambiente com chroot para o dispositivo raiz real. Essa etapa é útil para definir alguns parâmetros do sistema que podem ser relevantes para o systemd após a mudança de raiz. Por exemplo, arquivos de unidade systemd adicionais podem ser adicionados aqui antes que o systemd da raiz real seja executado. Executado como parte do `+initrd.target+`.
. *`+fs+`*: Esse estágio é o primeiro a ser executado após a mudança de raiz e é executado como parte do site `+sysinit.target+`, que é executado quando todos os sistemas de arquivos locais e pontos de montagem definidos no fstab estão montados e prontos.
. *`+network+`*: Esse estágio é executado como parte do `+multi-user.target+` e depois que o `+network-online.target+` é alcançado. Esse estágio pode ser usado para executar ações e configurações que exigem conectividade de rede. Por exemplo, esse estágio é usado para executar o primeiro registro de nó e a instalação a partir de um ISO ativo.
. *`+boot+`*: Esse estágio é executado como parte do multi-user.target e antes do getty.target. Esse é o estágio padrão para executar dados de configuração de nuvem fornecidos usando a sintaxe de inicialização de nuvem compatível. Consulte a seção xref:references/cloud-config-reference.adoc#_compatibility_with_cloud_init_format[de compatibilidade de inicialização na nuvem].

Por padrão, o site `+elemental+` lê os arquivos de configuração yaml dos seguintes caminhos, em ordem: `+/system/oem+`, `+/oem+` e `+/usr/local/cloud-config+`.

No {elemental-product-name} Operator, todos os recursos do kubernetes, incluindo um campo `+cloud-config+`, podem ser expressos em uma sintaxe <<_compatibility_with_cloud_init_format,compatível com>> <<_configuration_syntax,yip>> ou <<_compatibility_with_cloud_init_format,cloud-init>>. Isso inclui recursos como `+MachineRegistration+`, `+SeedImage+` e `+ManagedOSImage+`.

[NOTE]
====
Ao contrário de projetos semelhantes, como o _Cloud Init_, o Yip não mantém registros ou caches de estágios e etapas executados; todos os estágios e suas configurações associadas são executados a cada inicialização.
====

== Ganchos de configuração de nuvem do cliente elementar

Além dos estágios de configuração de nuvem definidos na inicialização descritos na seção anterior, o cliente Elemental também honra alguns estágios específicos, referenciados como hooks, para personalizar o comportamento desses subcomandos: `+install+`, `+upgrade+`, `+reset+` e `+build-disk+`. Cada um desses subcomandos tem seu próprio conjunto de quatro estágios diferentes de configuração de nuvem executados em fases análogas da execução do subcomando específico.

Os ganchos são basicamente uma forma de fornecer alterações permanentes ao sistema que não podem ser facilmente expressas como parte de um contêiner OCI ou que não são facilmente alcançáveis com as opções de configuração do cliente `+elemental+`. Um bom exemplo poderia ser o manuseio do firmware na partição EFI para dispositivos Raspberry Pi.

[WARNING]
====
Observe que a maioria dos hooks é executada no ambiente do host com privilégios, portanto, são operações potencialmente destrutivas. Na maioria dos casos, as operações regulares de configuração da nuvem no momento da inicialização são suficientes para configurar o sistema. Além disso, para incluir software adicional em uma imagem, a opção preferida é criar uma xref:operator-operational-tasks/custom-images.adoc[imagem derivada] e não abusar de ganchos para instalar software adicional.
====

=== Estágios do gancho

Estágios anteriores: before-install (antes da instalação), before-upgrade (antes da atualização), before-reset (antes da reinicialização), before-disk (antes do disco) Esses estágios são executados depois que os diretórios de trabalho e o ambiente são preparados, mas antes de iniciar a ação real. Nas etapas de instalação, upgrade e redefinição, isso acontece depois que todas as partições associadas são criadas e montadas, mas antes de declarar a implementação de qualquer imagem.

Estágios pós-croot: after-install-chroot, after-upgrade-chroot, after-reset-chroot, after-disk-chroot Esses estágios são executados após a implantação do sistema de destino na área de trabalho em um ambiente chroot enraizado na imagem implantada real. Como isso acontece em um chroot, o cliente elementar analisa os ganchos presentes na imagem implantada, não no host. Apenas /oem é compartilhado com o host, se disponível.

Estágios posteriores: after-install, after-upgrade, after-reset, after-disk Esses estágios são executados após a implantação do sistema de destino na área de trabalho a partir do ambiente do host. Nesse estágio, todas as partições ainda estão montadas e disponíveis no modo RW. Esse conjunto específico de estágios analisa os hooks presentes no host e no conjunto equivalente de caminhos com raiz na imagem implantada. No entanto, o hook não é executado em um ambiente chroot. Isso é útil para fornecer `+after-*+` hooks dentro da imagem implantada.

Estágios posteriores: post-install, post-upgrade, post-reset, post-disk Esses estágios são executados no final, antes de encerrar o comando e executar um processo de limpeza. Nesse estágio, a imagem já está implantada e bloqueada em um subvolume ou sistema de arquivos somente leitura. As partições ainda estão montadas nesse estágio.

[NOTE]
====
Observe que os ganchos de instalação não são aplicados como parte do xref:references/machineregistration-reference.adoc#_config_cloud_config[`+MachineRegistration.config.cloud-config+`]. Para fornecer ganchos de instalação, eles podem ser incluídos como parte do arquivo xref:[`+SeedImage.cloud-config+`]pois eles precisam estar presentes na mídia de instalação.
====

== Sintaxe de configuração

O Yip tem sua própria sintaxe e, basicamente, requer a definição de _etapas_ e uma lista de etapas para cada etapa. As etapas são executadas em ordem e cada etapa pode ser uma combinação de diferentes tipos de ação (por exemplo, executar comandos, criar arquivos, definir nome de host etc.).

Considere o exemplo a seguir:

[,yaml]
----
stages:
  initramfs:
  - name: "Setup users"
    ensure_entities:
    - path: /etc/shadow
      entity: |
          kind: "shadow"
          username: "root"
          password: "root"
  boot:
  - files:
    - path: /tmp/script.sh
      content: |
        #!/bin/sh
        echo "test"
      permissions: 0777
      owner: 1000
      group: 100
  - commands:
    - /tmp/script.sh
----

No exemplo acima, há dois estágios: `+initramfs+` e `+boot+`. O estágio `+initramfs+` inicializa um usuário de amostra. O estágio `+boot+` inclui duas etapas: uma para criar um arquivo de script executável e uma segunda que realmente executa o script.

O Yip também oferece suporte aos modificadores de sufixo `+*.before+` e `+*.after+` para qualquer estágio. Por exemplo, a execução do estágio `+network+` resulta na execução dos primeiros estágios `+network.before+` encontrados nos arquivos de configuração e, em seguida, `+network+` e, finalmente, `+network.after+`.

Veja a referência completa das chaves aplicáveis nas etapas documentadas no própriohttps://github.com/rancher/yip?tab=readme-ov-file#configuration-reference[projeto] yip.

== Sincronização

Durante a execução de um estágio, todos os arquivos são carregados e um gráfico é computado e, em seguida, executado. Isso significa que se pode esperar que as etapas no mesmo estágio sejam executadas em qualquer ordem e independentemente do nome do arquivo.

Para sincronizar a execução, o usuário pode usar os estágios post/pre ou consultar a palavra-chave `+after+` na https://github.com/rancher/yip/blob/ee4051d9ec2782989344f813bc6a0975bbd8f3fe/pkg/executor/default_test.go#L222[sintaxe] yip.

== Compatibilidade com o formato Cloud Init

Um subconjunto da oficial http://cloudinit.readthedocs.org/en/latest/topics/format.html#cloud-config-data[especificação] é implementado pelo yip. Mais especificamente, as chaves de inicialização de nuvem compatíveis são: `+users+`, `+ssh_authorized_keys+`, `+runcmd+`, `+hostname+` e `+write_files+` são implementadas.

Se um arquivo yaml começar com `+#cloud-config+`, ele será analisado como um cloud-init padrão, associado ao estágio yip `+boot+`.

Por exemplo:

[,yaml]
----
#cloud-config
# Note groups are delivered as list, not as comma separated values
users:
- name: "bar"
  passwd: "foo"
  groups:
  - "users"
  homedir: "/home/foo"
  shell: "/bin/bash"
  ssh_authorized_keys:
  - faaapploo
# Assigns these keys to the first user in users or root if there
# is none
ssh_authorized_keys:
- asddadfafefa
# Run these commands once the system has fully booted
# Each command is expressed as a sinlge string, no nested lists
runcmd:
- echo hello world
# Hostname
hostname: myserver
# Write arbitrary files
write_files:
- encoding: b64
  content: CiMgVGhpcyBmaWxlIGNvbnRyb2xzIHRoZSBzdGF0ZSBvZiBTRUxpbnV4
  path: /foo/bar
  permissions: "0644"
  owner: "bar"
----
