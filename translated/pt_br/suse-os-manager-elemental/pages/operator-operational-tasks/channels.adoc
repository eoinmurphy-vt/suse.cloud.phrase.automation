= Canais
:revdate: 2025-09-30
:page-revdate: {revdate}

O site {elemental_operator_name} permite a assinatura de um ou mais xref:references/managedosversionchannel-reference.adoc[ManagedOSVersionChannels] para preencher automaticamente uma lista de xref:references/managedosversion-reference.adoc[ManagedOSVersions] prontas para serem consumidas para criar novos ISOs usando uma xref:references/seedimage-reference.adoc[SeedImage] ou para atualizar os nós existentes do site {elemental-product-name} para novas versões do sistema operacional usando a xref:references/managedosimage-reference.adoc[ManagedOSImage].

Normalmente, um canal é distribuído como uma imagem de contêiner OCI, mas também é possível fazer referência ao URI de um arquivo JSON que contém diretamente uma lista de `+ManagedOSVersion+`. Observe que a prática recomendada é distribuir canais usando imagens, de modo que a distribuição seja consistente com todas as outras imagens necessárias para o {elemental_operator_name}. Isso pode ser benéfico, por exemplo, ao implantar em um ambiente Airgapped.

Um sincronizador personalizado permite mais flexibilidade sobre como coletar `+ManagedOSVersion+`, permitindo comandos personalizados com imagens personalizadas. Esse tipo de sincronizador permite executar um determinado comando com argumentos e variáveis de ambiente em uma imagem personalizada e gerar um arquivo json para `+/data/output+`. == Ciclo de vida dos canais e práticas recomendadas

Quando um novo `+ManagedOSVersionChannel+` for criado, o {elemental_operator_name} sincronizará periodicamente a lista JSON fornecida pelo canal e a converterá em um novo `+ManagedOSVersions+`. +
Todos os `+ManagedOSVersions+` sincronizados serão de propriedade do `+ManagedOSVersionChannel+`. A exclusão do site `+ManagedOSVersionChannel+` resultará na exclusão de todos os sites `+ManagedOSVersions+` em cascata.

Observe que o `+ManagedOSVersionChannel+` é compatível com a limpeza automática de itens que não estão mais sincronizados `+ManagedOSVersions+`, quando a opção `+ManagedOSVersionChannel.spec.deleteNoLongerInSyncVersions+` está ativada.

Quando um `+ManagedOSVersion+` estiver programado para ser excluído, um finalizador se certificará de que não haja nenhuma referência ativa em qualquer `+ManagedOSImage+`.

Se um `+ManagedOSVersion+` não puder ser excluído, você poderá descobrir por quais recursos ele é referenciado:

[,shell]
----
kubectl -n fleet-default get managedosimages -l elemental.cattle.io/managed-os-version-name=my-deleted-os-version
----

Ao usar vários canais, é importante manter uma estratégia de nomenclatura adequada para ter sempre uma referência rápida e legível no site `+ManagedOSVersions+`. +
Recomenda-se nomear qualquer canal como: `+{BaseOS}-{BaseOSVersion}-{Flavor}+`.

Isso deve permitir que o usuário use o nome `+ManagedOSVersion+` como a versão de compilação {elemental-product-name} específica da imagem e, ao mesmo tempo, mantenha uma referência no SO básico e na versão do SO básico do canal principal. +

== Criando seus próprios canais

O único requisito para criar seu próprio sincronizador personalizado é fazer com que ele envie um arquivo JSON para `+/data/output+` e mantenha a estrutura JSON correta.

O arquivo é uma matriz JSON que contém entradas ISO e de contêineres.
Cada entrada na matriz é mapeada 1:1 com um objeto xref:references/managedosversion-reference.adoc[ManagedOSVersion].

`+"type": "iso"+` devem conter um ISO {elemental-product-name} inicializável e são usadas por xref:references/seedimage-reference.adoc[SeedImages], enquanto as entradas `+"type": "container"+` são usadas por xref:references/managedosimage-reference.adoc[ManagedOSImage] para atualizações de {elemental-product-name}.

Em caso de dúvida, o projeto https://github.com/rancher-sandbox/elemental-channels[elemental-channels] pode ser usado como uma implementação de referência sobre como criar e manter seus próprios canais.

Ao criar novas entradas, esteja atento à estratégia de nomenclatura escolhida para evitar colisões com outros canais, pois eles podem acabar sincronizando diferentes `+ManagedOSVersion+` com o mesmo nome.
Uma prática recomendada é usar a convenção: `+\{Flavor}-\{Version}-\{Type}+`
