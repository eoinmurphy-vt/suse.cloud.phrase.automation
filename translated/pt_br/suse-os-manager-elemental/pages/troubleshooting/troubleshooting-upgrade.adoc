= Solução de problemas de upgrade
:revdate: 2025-09-08
:page-revdate: {revdate}

Para obter uma visão geral de alto nível do ciclo de vida e dos componentes do upgrade, consulte o xref:rancher-os-management/upgrade-lifecycle.adoc[documento relacionado].

== Lado rancheiro

Neste exemplo, fizemos o upgrade dos nós do cluster com a seguinte definição de ManagedOSImage:

[,yaml]
----
apiVersion: elemental.cattle.io/v1beta1
kind: ManagedOSImage
metadata:
  name: my-upgrade
  namespace: fleet-default
spec:
  # Set to the new {elemental-product-name} version you would like to upgrade to or track the latest tag
  osImage: "registry.suse.com/rancher/sle-micro/5.5:latest"
  clusterTargets:
    - clusterName: my-cluster
----

Depois que o `+ManagedOSImage+` for aplicado, o `+elemental-operator+` o verificará e gerará um `+Bundle+` relacionado.
O nome `+Bundle+` será prefixado com `+mos+` e, em seguida, com o nome `+ManagedOSImage+`. Nesse caso, `+mos-my-upgrade+`.

Na definição de `+Bundle+`, você encontrará os detalhes sobre o plano de upgrade e o destino desejado.
Por exemplo:

[,shell]
----

kubectl -n fleet-default get bundle mos-my-upgrade -o yaml
----

== {elemental-product-name} Lado do cluster

Qualquer nó do {elemental-product-name} registrado corretamente e que faça parte do cluster de destino buscará o pacote e começará a aplicá-lo. +
Essa operação é realizada pelo site `+system-upgrade-controller+` do Rancher em execução no cluster {elemental-product-name}. +
Para monitorar a operação correta desse controlador, você pode ler seus registros:

[,shell]
----
kubectl -n cattle-system logs deployment/system-upgrade-controller
----

Se tudo estiver correto, o `+system-upgrade-controller+` criará um plano de upgrade no cluster:

[,shell]
----
kubectl -n cattle-system get plans
----

Para cada plano, o controlador orquestrará os trabalhos que o aplicarão em cada nó de destino. +
Os nomes dos trabalhos usarão o nome do Plano (`+os-upgrader-my-upgrade+`) e o nome do host da máquina de destino (`+my-host+`) para facilitar a descoberta. +
Por exemplo: `+apply-os-upgrader-my-upgrade-on-my-host-7a25e+` +
Você pode monitorar esses trabalhos com:

[,shell]
----
kubectl -n cattle-system get jobs
----

Cada trabalho usará um contêiner `+privileged: true+` com a imagem do SLE Micro especificada na definição `+ManagedOSImage+`. Esse contêiner tentará atualizar o sistema e realizar uma reinicialização.

Se o trabalho falhar, você poderá verificar seu status examinando os logs:

[,shell]
----
kubectl -n cattle-system logs job.batch/apply-os-upgrader-my-upgrade-on-my-host-7a25e
----

[IMPORTANT]
.Processo de trabalho em dois estágios
====

Observe que o processo de upgrade é realizado em duas etapas. +
Você perceberá que o mesmo trabalho é executado duas vezes e o primeiro termina com o status `+Unknown+` e não é concluído. +
*Isso é esperado*, pois o {elemental-product-name} depende da execução do trabalho novamente após a reinicialização do computador, para que ele possa verificar se a nova versão foi instalada corretamente. +
Você notará uma segunda execução do trabalho, dessa vez concluída corretamente.

[,shell]
----
kubectl -n cattle-system get jobs
NAMESPACE       NAME                                            COMPLETIONS   DURATION   AGE
cattle-system   apply-os-upgrader-my-upgrade-on-my-host-0b392   1/1           2m34s      6m23s
cattle-system   apply-os-upgrader-my-upgrade-on-my-host-7a25e   0/1           6m23s      6m23s

----

[,shell]
----
kubectl -n cattle-system get pods
NAME                                            READY   STATUS      RESTARTS      AGE
apply-os-upgrader-my-upgrade-on-my-host-zbkrh   0/1     Completed   0             9m40s
apply-os-upgrader-my-upgrade-on-my-host-zvrff   0/1     Unknown     0             12m
----

== Recuperação de falhas

É possível que o processo de atualização do ManagedOSImage falhe, deixando um ou mais nós em um estado defeituoso. Por exemplo, se a imagem a ser atualizada não for encontrada no registro ou estiver quebrada, o trabalho de atualização em execução nos clusters downstream não será bem-sucedido.

Quando esse for o caso, os nós que estiverem executando o trabalho de upgrade com falha permanecerão isolados. Você pode atualizar a ManagedOSImage com um `+osImage+` funcional ou, alternativamente, excluí-la para interromper qualquer tentativa de atualização futura. De qualquer forma, para restaurá-los e poder programar também os upgrades seguintes, os nós afetados precisam ser descadastrados manualmente.
