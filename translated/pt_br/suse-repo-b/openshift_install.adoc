Crie um arquivo `+values.yaml+` com a estrutura de configuração `+global.suseObservability+`:

global:

  # imageRegistry: "your-private-registry.example.com"

    license: "YOUR-LICENSE-KEY"

    # Required: Base URL for {stackstate-product-name}
    # Use http://localhost:8080 if ingress is not configured yet
    baseUrl: "https://observability.example.com"

    # Required: Sizing profile
    # Available: trial, 10-nonha, 20-nonha, 50-nonha, 100-nonha,
    #            150-ha, 250-ha, 500-ha, 4000-ha
    sizing:
      profile: "150-ha"

    # Required: Plain text Admin password
    adminPassword: "your-password"
    # Instead of adminPassword you can provide a bcrypt hashed password with adminPasswordBcrypt
    # Generate with: htpasswd -bnBC 10 "" "your-password" | tr -d ':\n'
    # adminPasswordBcrypt: "$2a$10$..."

    # Optional: Receiver API key (auto-generated if not provided)
    # receiverApiKey: "your-receiver-api-key"

    # Optional: Pull secret for private registry
    # pullSecret:
    #   username: "registry-user"
    #   password: "registry-pass"
----

TIP: Para gerar um hash de senha bcrypt, use: `+htpasswd -bnBC 10 "" "your-password" | tr -d ':\n'+`

Você pode gerar os arquivos {stackstate-product-name} `+baseConfig_values.yaml+`, `+sizing_values.yaml+` e `+affinity_values.yaml+` executando um Helm Chart separado. O gráfico `+suse-observability/suse-observability-values+`. Um exemplo de linha de comando é:

[,text]
----
export VALUES_DIR=.
helm template \
  --set license='<your license>' \
  --set baseUrl='<suse-observability-base-url>' \
  --set sizing.profile='<sizing.profile>' \
  suse-observability-values \
  suse-observability/suse-observability-values --output-dir $VALUES_DIR
----

Esse comando gera os arquivos `+$VALUES_DIR/suse-observability-values/templates/baseConfig_values.yaml+`, `+$VALUES_DIR/suse-observability-values/templates/sizing_values.yaml+` e `+$VALUES_DIR/suse-observability-values/templates/affinity_values.yaml+` que contêm a configuração necessária para a instalação do Helm Chart {stackstate-product-name}.

[NOTE]
=====
As senhas de administrador de {stackstate-product-name} serão geradas automaticamente pelo comando acima e serão exibidas como comentários no arquivo `+basicConfig.yaml+` gerado. Os valores reais contêm os hashes `+bcrypt+` dessas senhas para que elas sejam armazenadas com segurança na versão do Helm no cluster.
=====


Os valores que podem ser passados para esse gráfico são:

[cols="1,1,2"]
|===
| Configuração | Valor | Descrição

| Geração de configuração básica
| `+basicConfig.generate+`
| Ativar ou desativar a geração do arquivo `+basicConfig_values.yaml+`.

| Chave de API do receptor
| `+receiverApiKey+`
| A chave de API usada por {stackstate-product-name} para receber dados de agentes. Essa é uma chave secreta que deve ser mantida em sigilo. Se você omitir isso, uma chave aleatória será gerada para você.

| URL de base
| `+baseUrl+`
| O `+<STACKSTATE_BASE_URL>+`. O URL externo para {stackstate-product-name} que os usuários e agentes usarão para se conectar. Por exemplo, `+https://suse-observability.internal+`. Se você ainda não tiver decidido sobre uma configuração de Ingress, use `+http://localhost:8080+`. Isso pode ser atualizado posteriormente no arquivo gerado.

| Nome de usuário e senha**
| `+-u+` `+-p+`
| O nome de usuário e a senha usados pelo {stackstate-product-name} para extrair imagens. Para ambientes com air-gap, eles precisam ser o nome de usuário e a senha do registro local do docker.

| Chave de licença
| `+license+`
| A chave de licença {stackstate-product-name}.

| Senha padrão
| `+adminPassword+`
| A senha do usuário padrão (`+admin+`) para acessar a interface do usuário de {stackstate-product-name}. Como alternativa, você pode fornecer uma senha com hash bcrypt usando `+adminPasswordBcrypt+`. Somente uma das duas opções pode ser usada ao mesmo tempo.

| Registro de imagens
| `+imageRegistry+`
| O registro onde as imagens do {stackstate-product-name} estão hospedadas. Se não for fornecido, o valor padrão será 'quay.io'

| Puxar nome de usuário secreto
| `+pullSecret.username+`
| O nome de usuário usado para extrair imagens do registro do Docker onde as imagens do {stackstate-product-name} estão hospedadas.

| Puxar senha secreta
| `+pullSecret.password+`
| A senha usada para extrair imagens do registro do Docker onde as imagens do {stackstate-product-name} estão hospedadas.

| Geração de dimensionamento
| `+sizing.generate+`
| Ativar ou desativar a geração do arquivo `+sizing_values.yaml+`.

| Perfil de dimensionamento
| `+sizing.profile+`
| Ensaio OneOf, 10-nonha, 20-nonha, 50-nonha, 100-nonha, 150-ha, 250-ha, 500-ha. Com base nesses perfis, você pode gerar o arquivo `+sizing_values.yaml+` que contém os tamanhos padrão para os recursos e a configuração do {stackstate-product-name} a serem implementados no modo HA ou Non-HA. Por exemplo, 10-nonha produz um `+sizing_values.yaml+` para implantar uma instância Non-HA {stackstate-product-name} para observar um cluster de 10 nós em um modo de não alta disponibilidade. Atualmente, não é possível passar de um ambiente não HA para um ambiente HA. Se você espera que seu ambiente exija a observação de cerca de 150 nós, é melhor usar o HA imediatamente.

| Pod AntiAffinity
| `+affinity.podAntiAffinity.requiredDuringSchedulingIgnoredDuringExecution+`
| Impor que as réplicas dos mesmos serviços de dados sejam agendadas para diferentes nós do Kubernetes. Aplicável somente aos perfis `+HA+`. `+true+` ou `+false+`. O padrão é `+true+` - o agendamento para nós diferentes é aplicado.
|===

[NOTE]
=====
Armazene os arquivos `+basicConfig.yaml+` e `+sizing_values.yaml+` gerados com segurança. Você pode reutilizar esses arquivos para atualizações, o que economiza tempo e garante que o {stackstate-product-name} continue a usar a mesma chave de API. Isso é desejável, pois significa que os agentes e outros provedores de dados para {stackstate-product-name} não precisarão ser atualizados.
Você pode gerar novamente os arquivos de forma independente usando as opções `+basicConfig.generate=false+` e `+sizing.generate=false+` para desativar qualquer um deles e ainda manter a versão gerada anteriormente do arquivo em `+output-dir+`.

=== Criar `+openshift-values.yaml+`

Como o OpenShift tem um modelo de segurança mais rigoroso do que o Kubernetes simples, todos os contextos de segurança padrão na implantação precisam ser desativados.

Crie um arquivo de valores do Helm `+openshift-values.yaml+` com o seguinte conteúdo e armazene-o ao lado do arquivo `+values.yaml+` gerado. Contém os valores necessários para uma implantação do OpenShift.

[,yaml]
----
elasticsearch:
  prometheus-elasticsearch-exporter:
    podSecurityContext: ""
  sysctlInitContainer:
    enabled: false
scc:
  enabled: true
----

=== Acesse a interface do usuário {stackstate-product-name} 

Depois que o {stackstate-product-name} tiver sido implantado, você poderá verificar se todos os pods estão em funcionamento:

[,bash]
----
kubectl get pods --namespace suse-observability
----

Quando todos os pods estiverem ativos, você poderá ativar um encaminhamento de porta:

[,bash]
----
kubectl port-forward service/suse-observability-router 8080:8080 --namespace suse-observability
----

{stackstate-product-name} agora estará disponível em seu navegador em `+https://localhost:8080+`. Faça login com o nome de usuário `+admin+` e a senha padrão fornecida no arquivo `+values.yaml+`.

== Criar manualmente objetos `+SecurityContextConfiguration+` 

Se não for possível usar uma conta de administrador para instalar o {stackstate-product-name} no OpenShift, peça ao administrador para aplicar os objetos `+SecurityContextConfiguration+` abaixo.

[,yaml]
----
apiVersion: security.openshift.io/v1
kind: SecurityContextConstraints
metadata:
  name: {{ template "common.fullname.short" . }}-{{ .Release.Namespace }}
  labels:
    {{- include "common.labels.standard" . | nindent 4 }}
  annotations:
    helm.sh/hook: pre-install
    suse-observability.io/note: "Ignored by helm uninstall, has to be deleted manually"
fsGroup:
  type: RunAsAny
groups:
- system:serviceaccounts:{{ .Release.Namespace }}
runAsUser:
  type: RunAsAny
seLinuxContext:
  type: MustRunAs
supplementalGroups:
  type: RunAsAny
volumes:
- configMap
- downwardAPI
- emptyDir
- ephemeral
- persistentVolumeClaim
- projected
- secret
allowHostDirVolumePlugin: false
allowHostIPC: false
allowHostNetwork: false
allowHostPID: false
allowHostPorts: false
allowPrivilegeEscalation: true
allowPrivilegedContainer: false
readOnlyRootFilesystem: false
seccompProfiles:
  - runtime/default
----
