=== Criar o arquivo de configuração `+values.yaml+` 

O arquivo `+values.yaml+` contém todas as configurações necessárias para implementar o {stackstate-product-name}, incluindo a chave de licença, o URL da base, a senha do administrador e o perfil de dimensionamento.

Método recomendado::
+
--
[NOTE]
=====
O método de configuração `+global.suseObservability+` está disponível a partir da versão `+2.8.0+`. Para versões anteriores, use o método legado.
=====

Crie um arquivo `+values.yaml+` com a estrutura de configuração `+global.suseObservability+`:

[source,yaml]
----
global:
  # Optional: Override image registry (defaults to registry.rancher.com)
  # Only needed for air-gapped environments or custom registries
  # imageRegistry: "your-private-registry.example.com"

  # Optional: Override storage class (uses cluster default if not set)
  # storageClass: "your-storage-class"

  suseObservability:
    # Required: Your {stackstate-product-name} license key
    license: "YOUR-LICENSE-KEY"

    # Required: Base URL for {stackstate-product-name}
    # Use http://localhost:8080 if ingress is not configured yet
    baseUrl: "https://observability.example.com"

    # Required: Sizing profile
    # Available: trial, 10-nonha, 20-nonha, 50-nonha, 100-nonha,
    #            150-ha, 250-ha, 500-ha, 4000-ha
    sizing:
      profile: "150-ha"

    # Required: Plain text admin password
    adminPassword: "your-password"
    # Instead of adminPassword you can provide a bcrypt hashed password with adminPasswordBcrypt
    # Generate with: htpasswd -bnBC 10 "" "your-password" | tr -d ':\n'
    # adminPasswordBcrypt: "$2a$10$..."

    # Optional: Receiver API key (auto-generated if not provided)
    # receiverApiKey: "your-receiver-api-key"

    # Optional: Pull secret for private registry
    # pullSecret:
    #   username: "registry-user"
    #   password: "registry-pass"
----

Para obter mais opções de configuração, incluindo configurações de afinidade, consulte xref:/setup/install-stackstate/kubernetes_openshift/affinity.adoc[Configurar afinidades do Kubernetes].

Você pode gerar os arquivos de configuração do {stackstate-product-name} executando um Helm Chart separado. Este é o gráfico `+suse-observability/suse-observability-values+`:

[,text]
----
export VALUES_DIR=.
helm template \
  --set license='<your license>' \
  --set baseUrl='<suse-observability-base-url>' \
  --set sizing.profile='<sizing.profile>' \
  suse-observability-values \
  suse-observability/suse-observability-values --output-dir $VALUES_DIR
----

Esse comando gera os arquivos `+$VALUES_DIR/suse-observability-values/templates/baseConfig_values.yaml+`, `+$VALUES_DIR/suse-observability-values/templates/sizing_values.yaml+` e `+$VALUES_DIR/suse-observability-values/templates/affinity_values.yaml+`.

[NOTE]
=====
As senhas de administrador de {stackstate-product-name} serão geradas automaticamente pelo comando acima e serão exibidas como comentários no arquivo `+basicConfig.yaml+` gerado. Os valores reais contêm os hashes `+bcrypt+` dessas senhas para que elas sejam armazenadas com segurança na versão do Helm no cluster.
=====

Os valores que podem ser passados para esse gráfico são:

[cols="1,1,2"]
|===
| Configuração | Valor | Descrição

| Geração de configuração básica
| `+basicConfig.generate+`
| Ativar ou desativar a geração do arquivo `+basicConfig_values.yaml+`.

| Chave de API do receptor
| `+receiverApiKey+`
| A chave de API usada por {stackstate-product-name} para receber dados de agentes. Essa é uma chave secreta que deve ser mantida em sigilo. Se você omitir isso, uma chave aleatória será gerada para você.

| URL de base
| `+baseUrl+`
| O `+<STACKSTATE_BASE_URL>+`. O URL externo para {stackstate-product-name} que os usuários e agentes usarão para se conectar. Por exemplo, `+https://suse-observability.internal+`. Se você ainda não tiver decidido sobre uma configuração de Ingress, use `+http://localhost:8080+`. Isso pode ser atualizado posteriormente no arquivo gerado.

| Nome de usuário e senha**
| `+-u+` `+-p+`
| O nome de usuário e a senha usados pelo {stackstate-product-name} para extrair imagens. Para ambientes com air-gap, eles precisam ser o nome de usuário e a senha do registro do contador local.

| Chave de licença
| `+license+`
| A chave de licença {stackstate-product-name}.

| Senha padrão
| `+adminPassword+`
| A senha do usuário padrão (`+admin+`) para acessar a interface do usuário de {stackstate-product-name}. Como alternativa, você pode fornecer uma senha com hash bcrypt usando `+adminPasswordBcrypt+`. Somente uma das duas opções pode ser usada ao mesmo tempo.

| Registro de imagens
| `+imageRegistry+`
| O registro onde as imagens do {stackstate-product-name} estão hospedadas. Se não for fornecido, o valor padrão será 'registry.rancher.com'. Necessário apenas para ambientes com air-gap ou registros personalizados.

| Puxar nome de usuário secreto
| `+pullSecret.username+`
| O nome de usuário usado para extrair imagens do registro do Docker onde as imagens do {stackstate-product-name} estão hospedadas. Necessário apenas para registros personalizados.

| Puxar senha secreta
| `+pullSecret.password+`
| A senha usada para extrair imagens do registro do Docker onde as imagens do {stackstate-product-name} estão hospedadas. Necessário apenas para registros personalizados.

| Geração de dimensionamento
| `+sizing.generate+`
| Ativar ou desativar a geração do arquivo `+sizing_values.yaml+`.

| Perfil de dimensionamento
| `+sizing.profile+`
| Ensaio OneOf, 10-nonha, 20-nonha, 50-nonha, 100-nonha, 150-ha, 250-ha, 500-ha. Com base nesses perfis, você pode gerar o arquivo `+sizing_values.yaml+` que contém os tamanhos padrão para os recursos e a configuração do {stackstate-product-name} para implantação no modo HA ou não HA. Por exemplo, 10-nonha produz um `+sizing_values.yaml+` para implantar uma instância NonHa {stackstate-product-name} para observar um cluster de 10 nós em um modo de não alta disponibilidade. Atualmente, não é possível passar de um ambiente não HA para um ambiente HA. Se você espera que seu ambiente exija a observação de cerca de 150 nós, é melhor usar o HA imediatamente.

| Pod AntiAffinity
| `+affinity.podAntiAffinity.requiredDuringSchedulingIgnoredDuringExecution+`
| Impor que as réplicas dos mesmos serviços de dados sejam agendadas para diferentes nós do Kubernetes. Aplicável somente aos perfis `+HA+`. `+true+` ou `+false+`. O padrão é `+true+` - o agendamento para nós diferentes é aplicado.
|===

[NOTE]
=====
Armazene os arquivos `+basicConfig.yaml+` e `+sizing_values.yaml+` gerados com segurança. Você pode reutilizar esses arquivos para atualizações, economizando tempo e garantindo que o {stackstate-product-name} continue a usar a mesma chave de API. Isso é desejável, pois significa que os agentes e outros provedores de dados para {stackstate-product-name} não precisarão ser atualizados.
Você pode gerar novamente os arquivos de forma independente usando as opções `+basicConfig.generate=false+` e `+sizing.generate=false+` para desativar qualquer um deles e ainda manter a versão gerada anteriormente do arquivo em `+output-dir+`.
=====

=====
O gráfico {stackstate-product-name} Values gera configurações de afinidade que você pode usar com o gráfico {stackstate-product-name} principal para controlar o comportamento de agendamento do pod. Consulte a documentação xref:/setup/install-stackstate/kubernetes_openshift/affinity.adoc[Configure Kubernetes Affinities] para obter mais informações.

=== Implante {stackstate-product-name} com o Helm

A implementação recomendada do {stackstate-product-name} é uma configuração pronta para produção e de alta disponibilidade, com muitos serviços executados de forma redundante. Se necessário, também é possível executar o {stackstate-product-name} em uma configuração não redundante, em que cada serviço tem apenas uma única réplica. Essa configuração é recomendada apenas para um ambiente de teste.

Para ambientes com vedação de ar, siga as instruções para as instalações com vedação de ar.

=== Acesse a interface do usuário {stackstate-product-name} 

Depois que o {stackstate-product-name} tiver sido implantado, você poderá verificar se todos os pods estão em funcionamento:

[,bash]
----
kubectl get pods --namespace suse-observability
----

Quando todos os pods estiverem ativos, você poderá ativar um encaminhamento de porta:

[,bash]
----
kubectl port-forward service/<helm-release-name>-suse-observability-router 8080:8080 --namespace suse-observability
----

Para a configuração de entrada, crie um arquivo `+ingress_values.yaml+` e adicione o código abaixo.

[NOTE]
====
Lembre-se de definir um segredo para a configuração do TLS. Para obter detalhes sobre como fazer isso, consulte a documentação do kubectl.
====

[,yaml]
----
ingress:
  enabled: true
  class: ""
  annotations:
    nginx.ingress.kubernetes.io/proxy-body-size: "50m"
    nginx.ingress.kubernetes.io/ssl-redirect: "true"
    #if external dns (cloudflare and similars) configure:
    #external-dns.alpha.kubernetes.io/target: <INGRESS_FQDN>
    #external-dns.alpha.kubernetes.io/ttl: "60"
    #end if
  hosts:
  - host: <SUSE_OBSERVABILITY_HOST>
  tls:
  - hosts:
    - <SUSE_OBSERVABILITY_HOST>
    secretName: suse-private-ai-tls
  existingSecret: suse-private-ai-tls
  # Set up your secrets in whichever way feels fit. Check k8s docs for details: https://kubernetes.io/docs/reference/kubectl/generated/kubectl_create/kubectl_create_secret_tls/
----
