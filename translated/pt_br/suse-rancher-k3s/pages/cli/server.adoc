= servidor k3s
:page-languages: [en, ja, ko, zh]
:revdate: 2025-05-26
:page-revdate: {revdate}

Nesta seção, você aprenderá a configurar o servidor K3s.

Observe que os servidores também executam um agente, portanto, todas as opções de configuração listadas na xref:cli/agent.adoc[documentação do site`+k3s agent+` ] também são compatíveis com os servidores.

As opções são documentadas nesta página como sinalizadores da CLI, mas também podem ser passadas como opções do arquivo de configuração. Consulte a documentação xref:installation/configuration.adoc#_configuration_file[do arquivo de configuração] para obter mais informações sobre o uso de arquivos de configuração YAML.

== Valores críticos de configuração

As opções a seguir devem ser definidas com o mesmo valor em todos os servidores do cluster. Se isso não for feito, os novos servidores não conseguirão ingressar no cluster ao usar o etcd incorporado ou a operação incorreta do cluster ao usar um armazenamento de dados externo.

* `+--agent-token+`
* `+--cluster-cidr+`
* `+--cluster-dns+`
* `+--cluster-domain+`
* `+--disable-cloud-controller+`
* `+--disable-helm-controller+`
* `+--disable-network-policy+`
* `+--disable=servicelb+` _Observação: outros componentes do pacote podem ser desativados de acordo com o servidor_
* `+--egress-selector-mode+`
* `+--embedded-registry+`
* `+--flannel-backend+`
* `+--flannel-external-ip+`
* `+--flannel-ipv6-masq+`
* `+--secrets-encryption+`
* `+--secrets-encryption-provider+`
* `+--service-cidr+`

=== Opções do administrador do Kubeconfig

|===
| Flag | Variável de ambiente | Descrição

| `+--write-kubeconfig value, -o+` valor
| `+K3S_KUBECONFIG_OUTPUT+`
| Escreva o kubeconfig para o cliente administrador nesse arquivo

| `+--write-kubeconfig-mode+` valor
| `+K3S_KUBECONFIG_MODE+`
| Escreva o kubeconfig com esse https://en.wikipedia.org/wiki/Chmod[modo] O arquivo kubeconfig pertence ao root e é gravado com um modo padrão de 600. Alterar o modo para 644 permitirá que ele seja lido por outros usuários sem privilégios no host.

| `+--write-kubeconfig-group+` valor
| `+K3S_KUBECONFIG_GROUP+`
| Gravar o grupo kubeconfig. Em combinação com `+--write-kubeconfig-mode+`, ele permitirá que os administradores do k3s acessem o arquivo kubeconfig, mas mantendo o arquivo como propriedade do root.

=== Trabalho em rede

|===
| Flag | Padrão | Descrição

| `+--cluster-cidr+` valor
| "10.42.0.0/16"
| CIDRs de rede IPv4/IPv6 a serem usados para IPs de pod

| `+--service-cidr+` valor
| "10.43.0.0/16"
| CIDRs de rede IPv4/IPv6 a serem usados para IPs de serviço

| `+--service-node-port-range+` valor
| "30000-32767"
| Intervalo de portas a ser reservado para serviços com visibilidade NodePort

| `+--cluster-dns+` valor
| "10.43.0.10"
| IPv4 Cluster IP para o serviço coredns. Deve estar em sua faixa de service-cidr

| `+--cluster-domain+` valor
| "cluster.local"
| Domínio do cluster

| `+--flannel-backend+` valor
| "vxlan"
| Uma das opções "none", "vxlan", "ipsec" (obsoleto), "host-gw", "wireguard-native" ou "wireguard" (obsoleto)

| `+--flannel-ipv6-masq+`
| "N/A"
| Ativar o mascaramento de IPv6 para o pod

| `+--flannel-external-ip+`
| "N/A"
| Usar endereços IP externos do nó para tráfego Flannel

| `+--servicelb-namespace+` valor
| "kube-system"
| Namespace dos pods para o componente servicelb

| `+--egress-selector-mode+` valor
| "agente"
a| Deve ser uma das seguintes opções: 

* desativado: O apiserver não usa túneis de agente para se comunicar com os nós. Requer que os servidores executem agentes e tenham conectividade direta com o kubelet nos agentes, caso contrário, o apiserver não conseguirá acessar os pontos de extremidade do serviço ou executar o kubectl exec e os registros do kubectl.
* agente: O apiserver usa túneis de agente para se comunicar com os nós. Os nós permitem a conexão de túnel a partir de endereços de loopback. Requer que os servidores também executem agentes, ou o apiserver não poderá acessar os pontos de extremidade do serviço. O padrão histórico para k3s.
* pod: O apiserver usa túneis de agente para se comunicar com nós e pontos de extremidade de serviço, encaminhando as conexões de ponto de extremidade para o agente correto observando os nós. Os nós permitem a conexão de túnel a partir de endereços de loopback ou de um CIDR atribuído a seu nó.
* cluster: O apiserver usa túneis de agente para se comunicar com nós e pontos de extremidade de serviço, encaminhando as conexões de ponto de extremidade para o agente correto observando os pontos de extremidade. Os nós permitem a conexão do túnel a partir de endereços de loopback ou do intervalo CIDR do cluster configurado.

== Ajuda da CLI do servidor K3s
