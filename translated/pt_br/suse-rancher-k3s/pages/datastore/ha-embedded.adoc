= etcd incorporado de alta disponibilidade
:page-languages: [en, ja, ko, zh]
:revdate: 2026-03-03
:page-revdate: {revdate}

[CAUTION]
====
O etcd (HA) incorporado pode ter problemas de desempenho em discos mais lentos, como Raspberry Pis executados com cartões SD.
====

.Por que um número ímpar de nós de servidor?
[%collapsible]
======
O cluster do etcd incorporado ao HA deve ser composto por um número ímpar de nós de servidor para que o etcd mantenha o quorum. Para um cluster com n servidores, o quorum é (n/2)+1. Em qualquer cluster de tamanho ímpar, a adição de um nó sempre aumentará o número de nós necessários para o quorum. Embora adicionar um nó a um cluster de tamanho ímpar pareça melhor, pois há mais máquinas, a tolerância a falhas é pior, pois exatamente o mesmo número de nós pode falhar sem perder o quórum, mas há mais nós que podem falhar.
======

Um cluster HA K3s com etcd incorporado é composto por:

* Três ou mais *nós de servidor* que atenderão à API do Kubernetes e executarão outros serviços de plano de controle, além de hospedar o datastore etcd incorporado.
* Opcional: Zero ou mais *nós de agente* designados para executar seus aplicativos e serviços
* Opcional: Um *endereço de registro fixo* para que os nós de agente se registrem no cluster

[NOTE]
====
Para implementar rapidamente grandes clusters de HA, consulte xref:related-projects.adoc[Projetos relacionados]
====


Para começar, primeiro inicie um nó de servidor com o sinalizador `+cluster-init+` para ativar o clustering e um token que será usado como um segredo compartilhado para unir servidores adicionais ao cluster.

[,bash]
----
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s K3S_TOKEN=SECRET sh -s - server \
    --cluster-init \
    --tls-san=<FIXED_IP> # Optional, needed if using a fixed registration address
----

Depois de iniciar o primeiro servidor, junte o segundo e o terceiro servidores ao cluster usando o segredo compartilhado:

[,bash]
----
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s K3S_TOKEN=SECRET sh -s - server \
    --server https://<ip or hostname of server1>:6443 \
    --tls-san=<FIXED_IP> # Optional, needed if using a fixed registration address
----

Onde `+INSTALL_K3S_ARTIFACT_URL+` é o https://scc.suse.com/rancher-docs/rancherprime/latest/en/reference-guide.html#prime-artifacts-url[URL dos artefatos principais]

Verifique se o segundo e o terceiro servidores agora fazem parte do cluster:

[,bash]
----
$ kubectl get nodes
NAME        STATUS   ROLES                       AGE   VERSION
server1     Ready    control-plane,etcd,master   28m   vX.Y.Z
server2     Ready    control-plane,etcd,master   13m   vX.Y.Z
server3     Ready    control-plane,etcd,master   10m   vX.Y.Z
----

Agora você tem um plano de controle altamente disponível. Todos os servidores agrupados com êxito podem ser usados no argumento `+--server+` para unir nós de servidor e agente adicionais. A associação de nós de agente adicionais ao cluster segue o mesmo procedimento dos servidores:

[,bash]
----
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s K3S_TOKEN=SECRET sh -s - agent --server https://<ip or hostname of server>:6443
----

Existem alguns sinalizadores de configuração que devem ser os mesmos em todos os nós do servidor:

* Sinalizadores relacionados à rede: `+--cluster-dns+`, `+--cluster-domain+`, `+--cluster-cidr+`, `+--service-cidr+`
* Sinalizadores que controlam a implantação de determinados componentes: `+--disable-helm-controller+`, `+--disable-kube-proxy+`, `+--disable-network-policy+` e qualquer componente passado para `+--disable+`
* Sinalizadores relacionados a recursos: `+--secrets-encryption+`

== Clusters de nó único existentes

Se você tiver um cluster existente usando o banco de dados SQLite incorporado padrão, poderá convertê-lo em etcd simplesmente reiniciando o servidor K3s com o sinalizador `+--cluster-init+`. Depois de fazer isso, você poderá adicionar outras instâncias, conforme descrito acima.

Se um datastore do etcd for encontrado no disco porque o nó já foi inicializado ou entrou em um cluster, os argumentos do datastore (`+--cluster-init+`, `+--server+`, `+--datastore-endpoint+`, etc) serão ignorados.
