= Primeiros passos com a telemetria aberta no Kubernetes
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: {stackstate-product-name}

Este guia fornece instruções sobre o monitoramento de um aplicativo.

* O aplicativo/carga de trabalho monitorado em execução no cluster A.
* O coletor do Open Telemetry é executado próximo ao(s) aplicativo(s) observado(s), portanto, no cluster A, e envia os dados para {stackstate-product-name}.
* {stackstate-product-name} em execução no cluster B, ou SUSE Cloud Observability.

image::otel/open-telemetry-collector-kubernetes.png[Instrumentação de contêineres com Open Telemetry por meio de um coletor executado como implantação do Kubernetes]

== O coletor de telemetria aberto

[NOTE]
====
Para uma configuração de produção, é altamente recomendável instalar o coletor, pois ele permite que seu serviço descarregue dados rapidamente e o coletor pode cuidar do tratamento adicional, como novas tentativas, lotes, criptografia ou até mesmo filtragem de dados confidenciais.
====

Instale o coletor OTel (Open Telemetry) no cluster A e configure-o para:

* Receber dados de, potencialmente, muitos aplicativos instrumentados.
* Enriqueça os dados coletados com atributos do Kubernetes.
* Gerar métricas para rastros.
* Encaminhe os dados para {stackstate-product-name}, incluindo a autenticação usando a chave da API.

NOTE: {stackstate-product-name} também tenta novamente enviar dados quando há problemas de conexão.

=== Criar um token de serviço

Há duas maneiras de criar um token de serviço:

* **{stackstate-product-name} UI** - abra o menu principal clicando no canto superior esquerdo da tela e vá para `+StackPacks+` > `+Open Telemetry+`.  Se você não tiver feito isso antes, clique no botão `+INSTALL+`. Clique no botão `+CREATE NEW SERVICE TOKEN+` e copie o valor para sua área de transferência.
* **{stackstate-product-name} CLI** - consulte xref:/use/security/k8s-service-tokens.adoc#_manage_service_tokens[Gerenciar tokens de serviço]

O valor do token de serviço deve ser usado quando as instruções abaixo mencionarem `+<SERVICE_TOKEN>+`.

=== Criar o namespace e um segredo para o token de serviço

Instale no namespace `+open-telemetry+` e use o token de serviço:

[,bash]
----
kubectl create namespace open-telemetry
kubectl create secret generic open-telemetry-collector \
    --namespace open-telemetry \
    --from-literal=API_KEY='<SERVICE_TOKEN>'
----

=== Configurar e instalar o coletor

Instalamos o coletor com um gráfico Helm fornecido pelo projeto Open Telemetry. Certifique-se de que você tenha configurado o repositório de gráficos do Open Telemetry helm:

[,bash]
----
helm repo add open-telemetry https://open-telemetry.github.io/opentelemetry-helm-charts
----

Crie um arquivo de valores `+otel-collector.yaml+` para o gráfico do Helm. Este é um bom ponto de partida para uso com {stackstate-product-name}, substitua `+<otlp-suse-observability-endpoint:port>+` pelo seu endpoint OTLP (consulte a xref:/setup/otel/otlp-apis.adoc[API OTLP] para seu endpoint) e insira o nome do seu cluster Kubernetes em vez de `+<your-cluster-name>+`:

.otel-collector.yaml
[,yaml]
----
# Set the API key from the secret as an env var:
extraEnvsFrom:
  - secretRef:
      name: open-telemetry-collector
mode: deployment
image:
  # Use the collector container image that has all components important for k8s. In case of missing components the ghcr.io/open-telemetry/opentelemetry-collector-releases/opentelemetry-collector-contrib image can be used which
  # has all components in the contrib repository: https://github.com/open-telemetry/opentelemetry-collector-contrib
  repository: "ghcr.io/open-telemetry/opentelemetry-collector-releases/opentelemetry-collector-k8s"
ports:
  metrics:
    enabled: true
presets:
  kubernetesAttributes:
    enabled: true
    extractAllPodLabels: true
# This is the config file for the collector:
config:
  receivers:
    otlp:
      protocols:
        grpc:
          endpoint: 0.0.0.0:4317
        http:
          endpoint: 0.0.0.0:4318
    # Scrape the collectors own metrics
    prometheus:
      config:
        scrape_configs:
        - job_name: opentelemetry-collector
          scrape_interval: 10s
          static_configs:
          - targets:
            - ${env:MY_POD_IP}:8888
  extensions:
    # Use the API key from the env for authentication
    bearertokenauth:
      scheme: SUSEObservability
      token: "${env:API_KEY}"
  exporters:
    nop: {}
    otlp/suse-observability:
      auth:
        authenticator: bearertokenauth
      # Put in your own otlp endpoint, for example otlp-suse-observability.my.company.com:443
      endpoint: <otlp-suse-observability-endpoint:port>
      compression: snappy
  processors:
    memory_limiter:
      check_interval: 5s
      limit_percentage: 80
      spike_limit_percentage: 25
    batch: {}
    resource:
      attributes:
      - key: k8s.cluster.name
        action: upsert
        # Insert your own cluster name
        value: <your-cluster-name>
      - key: service.instance.id
        from_attribute: k8s.pod.uid
        action: insert
        # Use the k8s namespace also as the open telemetry namespace
      - key: service.namespace
        from_attribute: k8s.namespace.name
        action: insert
  connectors:
    # Generate metrics for spans
    spanmetrics:
      metrics_expiration: 5m
      namespace: otel_span
  service:
    extensions: [ health_check,  bearertokenauth ]
    pipelines:
      traces:
        receivers: [otlp]
        processors: [memory_limiter, resource, batch]
        exporters: [debug, spanmetrics, otlp/suse-observability]
      metrics:
        receivers: [otlp, spanmetrics, prometheus]
        processors: [memory_limiter, resource, batch]
        exporters: [debug, otlp/suse-observability]
      logs:
        receivers: [otlp]
        processors: []
        exporters: [nop]
    telemetry:
      metrics:
        address: ${env:MY_POD_IP}:8888
----


[CAUTION]
====
*Use o mesmo nome de cluster usado para instalar o Agente {stackstate-product-name} * se você também usar o agente {stackstate-product-name} com o stackpack do Kubernetes. O uso de um nome de cluster diferente resultará em uma perspectiva de rastreamento vazia para os componentes do Kubernetes e, em geral, tornará a correlação de informações muito mais difícil para o {stackstate-product-name} e seus usuários.
====

Instale o coletor usando o arquivo de configuração:

[,bash]
----
helm upgrade --install opentelemetry-collector open-telemetry/opentelemetry-collector \
  --values otel-collector.yaml \
  --namespace open-telemetry
----

O coletor oferece muito mais receptores, processadores e exportadores de configuração. Para obter mais detalhes, consulte nossa xref:/setup/otel/collector.adoc[página do coletor]. Para uso na produção, muitas vezes são geradas grandes quantidades de vãos. Portanto, configure xref:/setup/otel/sampling.adoc[a amostragem].

== Colete dados de telemetria do seu aplicativo

A maneira comum de coletar dados de telemetria é instrumentar seu aplicativo usando o Open Telemetry SDK.

* xref:/setup/otel/instrumentation/java.adoc[Java]
* xref:/setup/otel/instrumentation/dot-net.adoc[.NET]
* xref:/setup/otel/instrumentation/node.js.adoc[Node.js]

Para outros idiomas, siga a documentação em https://opentelemetry.io/docs/languages/[opentelemetry.io] e certifique-se de configurar o exportador do SDK para enviar dados para o coletor que você acabou de instalar, seguindo xref:/setup/otel/instrumentation/sdk-exporter-config.adoc[estas instruções].

== Veja os resultados

Acesse {stackstate-product-name} e verifique se o Open Telemetry Stackpack está instalado (por meio do menu principal \-> Stackpacks).

Se os pods estiverem recebendo tráfego, você poderá encontrá-los sob o nome do serviço nas visões gerais de Open Telemetry \-> serviços e instâncias de serviço. Os rastreamentos aparecem no xref:/use/traces/k8sTs-explore-traces.adoc[explorador] de rastreamento e na xref:/use/views/k8s-traces-perspective.adoc[perspectiva de rastreamento] dos componentes de serviço e instância de serviço. As métricas de abrangência e as métricas específicas do idioma (se disponíveis) ficam disponíveis na xref:/use/views/k8s-metrics-perspective.adoc[perspectiva de métricas] dos componentes.

Se você também tiver o stackpack do Kubernetes instalado, os pods instrumentados também terão os rastreamentos disponíveis na xref:/use/views/k8s-traces-perspective.adoc[perspectiva de rastreamento].

== Rancher RBAC

Para que xref:/setup/security/rbac/rbac_rancher.adoc[o RBAC do Rancher] funcione, os dados de telemetria precisam ter os seguintes atributos de recursos presentes:

* `+k8s.cluster.name+` - o nome *do cluster* conforme usado pelo empilhamento do Kubernetes
* `+k8s.namespace.name+` - um *Namespace* gerenciado por um *projeto* Rancher

Isso pode ser obtido por meio de uma configuração como a mencionada acima.  Lá, a predefinição `+kubernetesAttributes+` do gráfico `+opentelemetry-collector+` Helm injeta um processador `+k8sattributes+` em cada pipeline.
