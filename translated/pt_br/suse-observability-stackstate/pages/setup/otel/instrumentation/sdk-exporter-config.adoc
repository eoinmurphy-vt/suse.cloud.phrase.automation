= Configuração de exportadores de SDK
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: Observabilidade do SUSE

Para enviar dados ao SUSE Observability, os SDKs usados para instrumentar seu aplicativo usam um exportador integrado. Uma configuração pronta para produção usa <<_with_a_collector_production_setup,um coletor>> próximo aos aplicativos instrumentados para enviar os dados para o SUSE Observability, mas também é possível fazer com que o aplicativo instrumentado <<_without_a_collector,envie diretamente>> os dados de telemetria para o SUSE Observability.

== Com um coletor (configuração de produção)

=== Configuração do exportador de SDK para Kubernetes

Todos os SDKs, independentemente do idioma, usam a mesma configuração básica para definir o Open Telemetry https://opentelemetry.io/docs/concepts/glossary/#_service[nome do serviço] e o endpoint do exportador (ou seja, para onde a telemetria é enviada).

Eles podem ser configurados definindo variáveis de ambiente para seu aplicativo instrumentado.

No Kubernetes, defina essas variáveis de ambiente no manifesto de sua carga de trabalho (substitua `+<the-service-name>+` por um nome para seu serviço de aplicativo):

[,yaml]
----
...
spec:
  containers:
  - env:
    - name: OTEL_EXPORTER_OTLP_ENDPOINT
      value: http://opentelemetry-collector.open-telemetry.svc.cluster.local:4317
    - name: OTEL_SERVICE_NAME
      value: <the-service-name>
    - name: OTEL_EXPORTER_OTLP_PROTOCOL
      value: grpc
...
----

O ponto de extremidade especificado no exemplo pressupõe que o coletor foi instalado usando os padrões do xref:/setup/otel/collector.adoc[guia de instalação]. Ele usa a porta `+4317+`, que usa a versão `+gRPC+` do protocolo OTLP. Algumas instrumentações são compatíveis apenas com HTTP; nesse caso, use a porta `+4318+`.

O nome do serviço também pode ser derivado dos rótulos do Kubernetes que já podem estar presentes. Por exemplo, assim:

[,yaml]
----
spec:
  containers:
  - env:
    - name: OTEL_SERVICE_NAME
      valueFrom:
        fieldRef:
          apiVersion: v1
          fieldPath: metadata.labels['app.kubernetes.io/component']
----

=== Configuração do SDK Exporter para outras instalações

Para configurar o SDK, defina essas variáveis de ambiente para seu aplicativo:

[,bash]
----
export OTEL_EXPORTER_OTLP_ENDPOINT="http://<the-host-for-the-collector>:4317"
export OTEL_EXPORTER_OTLP_PROTOCOL="grpc"
export OTEL_SERVICE_NAME="<the-service-name>"
export OTEL_RESOURCE_ATTRIBUTES='service.namespace=<the-namespace>'
----

O exemplo usa a porta `+4317+`, que usa a versão `+gRPC+` do protocolo OTLP. Algumas instrumentações suportam apenas HTTP, que usa a porta `+4318+` com o protocolo definido como `+http+`. Use a documentação do SDK de seu idioma para verificar qual protocolo é compatível com o SDK. Os campos `+OTEL_EXPORTER_OLTP_ENDPOINT+` e `+OTEL_EXPORTER_OTLP_PROTOCOL+` podem ser omitidos; eles têm valores padrão que enviam dados para o endpoint preferencial no host local.

O `+OTEL_RESOURCE_ATTRIBUTES+` é opcional e, além de definir um namespace de serviço, pode ser usado para definir mais atributos de recursos em uma lista separada por vírgulas.

=== gRPC vs HTTP

OTLP, o Open Telemetry Protocol, oferece suporte a gRPC e protobuf por HTTP. Na seção anterior, o protocolo do exportador foi definido como `+gRPC+`, o que geralmente proporciona o melhor desempenho. Além do fato de o SDK não oferecer suporte ao gRPC, pode haver outros motivos para preferir o HTTP:

* Alguns firewalls não estão configurados para lidar com gRPC
* Os proxies (reversos) e os balanceadores de carga podem não suportar o gRPC sem configuração adicional
* As conexões de longa duração do gRPC podem causar problemas no balanceamento de carga.

Para mudar para HTTP em vez de gRPC, altere o protocolo para `+http+` _e_ use a porta `+4318+`.

Para resumir, você pode tentar usar o HTTP caso o gRPC não esteja funcionando para você:

* `+grpc+` O protocolo usa a porta `+4317+` no coletor
* `+http+` O protocolo usa a porta `+4318+` no coletor

== Sem um coletor

Em pequenas configurações de teste, pode ser conveniente enviar diretamente os dados do seu aplicativo instrumentado para o SUSE Observability. A única diferença em relação à configuração do coletor documentada acima é usar um valor diferente para `+OTEL_EXPORTER_OTLP_ENDPOINT+`:

* Para gRPC, use o ponto de extremidade OTLP para o SUSE Observability, consulte xref:/setup/otel/otlp-apis.adoc[a página APIs OTLP].
* Para HTTP, use o ponto de extremidade OTLP sobre HTTP para o SUSE Observability, consulte xref:/setup/otel/otlp-apis.adoc[a página APIs OTLP].

[NOTE]
====
Substitua o URL do coletor *e* a porta pelos pontos de extremidade do SUSE Observability. Dependendo da sua instalação do SUSE Observability, as portas serão diferentes.
====
