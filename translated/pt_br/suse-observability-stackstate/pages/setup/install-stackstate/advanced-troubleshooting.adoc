Quando você for um cliente principal, entre em contato com o suporte do SUSE Observability em https://scc.suse.com/ para obter ajuda na configuração do SUSE Observability em seu cluster local. Use xref:/setup/install-stackstate/support-package-logs.adoc[o Support Package (Logs)] para coletar informações sobre a sua instância para a equipe de suporte.

Esta página fornece informações detalhadas sobre os subsistemas da plataforma SUSE Observability para solucionar problemas operacionais e de implantação. Esta página só deve ser consultada quando as etapas da xref:/setup/install-stackstate/troubleshooting.adoc[solução de problemas] não produzirem uma solução.

== Abordagem geral de solução de problemas

A abordagem geral para solucionar problemas operacionais da plataforma SUSE Observability é a seguinte:

* Obter uma visão geral de como os pods estão se comportando por meio de `+kubectl get pods+`
* Use as informações detalhadas do subsistema neste documento, juntamente com os sintomas do problema, para determinar quais pods/subsistemas podem ser a causa principal
* Inspecione os registros/metadados dos pods suspeitos:
 ** `+kubectl logs <pod-name> --all-containers=true+`
 ** `+kubectl describe pod <pod-name>+`
 ** Uma maneira rápida de obter todos os registros/descrições relacionados ao SUSE Observability é por meio do xref:/setup/install-stackstate/support-package-logs.adoc[Support Package (Logs)].
* Pode ser que os registros apontem para o mau funcionamento de alguma dependência; nesse caso, investigue a dependência.

== Visão geral dos subsistemas

=== Bancos de dados

O SUSE Observability é alimentado por vários bancos de dados; sempre que um banco de dados apresentar problemas, ele deve ser investigado primeiro, pois todos os outros serviços dependem dele

* `+Zookeeper+`: O Zookeeper é usado para descoberta de serviços, orquestração e failover. O Zookeeper é implantado usando 1 ou mais pods com o nome:
 ** `+suse-observability-zookeeper-<n>+`
* `+Kafka+`: O Kafka é usado para a passagem de mensagens entre quase todos os serviços: O Kafka é implantado pelos seguintes pods:
 ** `+suse-observability-kafka-<n>+`: Implantação principal do kafka
 ** `+<release-name>-kafkaup-operator-kafkaup-*+`: Operador auxiliar que realiza atualizações do kafka
* `+StackGraph+`: O StackGraph armazenava as configurações (do usuário) e a topologia. O StackGraph foi criado a partir de vários componentes e tem dois modos de implantação. HA e não HA.
 ** `+Tephra+`: Gerencia inícios, confirmações e conflitos de transações de banco de dados. Servido por cápsula `+<release-name>-hbase-tephra-<n>+`
  *** `+<release-name>-hbase-tephra-<n>+`: Pod do servidor de transações Tephra. Mantém o controle de transações e conflitos.
 ** `+HBase-HA+`: Armazena os dados do StackGraph, distribuídos em vários pods com diferentes responsabilidades:
  *** `+<release-name>-hbase-hdfs-nn-0+`: Nó de nome para HDFS, mantém o controle do índice do arquivo
  *** `+<release-name>-hbase-hdfs-snn-0+`: Nó de nome secundário, faz o trabalho de limpeza após o nó de nome
  *** `+<release-name>-hbase-hdfs-dn-<n>+`: HDFS Datanode, armazena os dados reais
  *** `+<release-name>-hbase-hbase-master-<n>+`: HBase Master, coordena tabelas e regiões
  *** `+<release-name>-hbase-hbase-rs-<n>+`: HBase Region Server, serve tabelas e regiões, armazena seus dados no HDFS
 ** `+HBase-non-HA+`:
  *** `+<release-name>-hbase-stackgraph-0+`: Todos os componentes do StackGraph implantados como um único pod na configuração `+non-HA+`. Isso também inclui sua própria instância do zookeeper.
* `+VictoriaMetrics+`: Armazena dados métricos. É implantado pelos pods:
 ** `+suse-observability-victoria-metrics-<n>-0+`: Principal nó de consulta/armazenamento de dados da VictoriaMetrics
 ** `+suse-observability-vmagent-0+`: Agente de ingestão para VictoriaMetrics. Os dados são enviados para o vmagent antes de serem encaminhados e armazenados.
* `+ClickHouse+`: Armazena dados de rastreamento. Implantado pelo(s) seguinte(s) pod(s):
 ** `+suse-observability-clickhouse-shard0-<n>+`: Loja principal da clickhouse
* `+ElasticSearch+`: Armazena eventos e registros. Implantado pelos seguintes pods:
 ** `+suse-observability-elasticsearch-master-<n>+`: Armazenamento principal do Elasticsearch
 ** `+<release-name>-prometheus-elasticsearch-exporter-*+`: Exporta métricas de desempenho das instâncias do Elasticsearch

=== Serviços de ingestão

A plataforma SUSE Observability recebe dados enviados pelo agente e pelo agente OpenTelemetry (OTEL). Os serviços de ingestão realizam o processamento inicial e levam os dados para o armazenamento.

* `+Receiver+`: O receptor implementa a API do lado da coleção para o agente do SUSE Observability. Ele aceita e autoriza dados de telemetria (registros, eventos, métricas ou topologia) e os encaminha para o armazenamento de dados correspondente ou para o Kafka. Ele pode ser implementado em modo único ou dividido:
 ** `+Receiver-Split+`:
  *** `+<release-name>-suse-observability-receiver-logs-*+`: Recebe os registros e os coloca no Elasticsearch
  *** `+<release-name>-suse-observability-receiver-process-agent-*+`: Recebe informações sobre processos e conectividade de rede e as encaminha para tópicos do Kafka
  *** `+<release-name>-suse-observability-receiver-base+`-*: Todos os outros dados do SUSE Observability Agent são transmitidos aqui.
 ** `+Receiver-NonSplit+`:
  *** `+<release-name>-suse-observability-receiver-*+`: Todos os dados do SUSE Observability Agent são transmitidos aqui.
* `+OpenTelemetry Collector+`: Fornece um ponto de extremidade para o qual os agentes do OpenTelemetry podem enviar dados do OpenTelemetry e produz rastreamentos, métricas e topologia com base nos dados enviados.
 ** `+suse-observability-otel-collector-0+`: Pod único que implementa o coletor OTEL

=== Processamento e serviço

A plataforma SUSE Observability executa a correlação e o monitoramento dos dados de telemetria recebidos. Os resultados são fornecidos ao cliente sob demanda por meio da API. A plataforma principal pode ser executada no modo distribuído e não distribuído. A distribuição permite uma maior taxa de transferência.

* `+Correlator+`: Correlaciona informações de conexão tcp para transformá-las em topologia. Implementado por pod:
 ** `+<release-name>-suse-observability-correlate-*+`
* `+Events2Elasticsearch+`: Processa eventos e os armazena no Elasticsearch: Implementado por pod:
 ** `+<release-name>-suse-observability-e2es-*+`
* `+Anomaly Detection+`: A plataforma SUSE Observability faz a detecção de anomalias (desativada por padrão) nas métricas, produzindo violações de integridade:
 ** `+<release-name>-anomaly-detection-spotlight-manager-*+`: Trabalho de detecção de anomalias distribuídas
 ** `+<release-name>-anomaly-detection-spotlight-worker-*+`: Realiza a detecção de anomalias em fluxos de métricas
* `+Platform-Distributed+`: A plataforma contém os principais componentes de processamento e a API de atendimento. No modo distribuído, as unidades funcionais são divididas. Os pods que pertencem à plataforma:
 ** `+<release-name>-suse-observability-api-*+`: Fornece todos os dados ao usuário e gerencia a instalação/desinstalação do StackPack.
 ** `+<release-name>-suse-observability-checks-*+`: Executa os monitores
 ** `+<release-name>-suse-observability-health-sync-*+`: Processa informações de integridade (violação) de monitores e do SUSE Observability Agent e as anexa à topologia.
 ** `+<release-name>-suse-observability-initializer-*+`: Coordena a inicialização dos repositórios de dados e das migrações
 ** `+<release-name>-suse-observability-notification-*+`: Encaminha notificações com base em violações de integridade e configurações do usuário para sistemas downstream, como Slack/Opsgenie.
 ** `+<release-name>-suse-observability-slicing-*+`: Otimiza continuamente o histórico da topologia para uma recuperação rápida
 ** `+<release-name>-suse-observability-state-*+`: Processamento de violações de saúde e agregação das mesmas em componentes de saúde
 ** `+<release-name>-suse-observability-sync-*+`: Processa os dados de topologia combinados com as configurações do usuário e os transforma no gráfico de topologia.
* `+Platform-Mono+`:
 ** `+<release-name>-suse-observability-server-*+`: Contém todas as funcionalidades da configuração `+Platform-Distributed+`, mas em um único pod.

=== Diversos

* `+Routing+`: Aceitar conexões e rotear para o serviço de backend correto:
 ** `+<release-name>-suse-observability-router-+`: Roteador baseado no Envoy
* `+UI+`: UI baseada em React
 ** `+<release-name>-suse-observability-ui+`: Serve apenas o código e os ativos estáticos da interface do usuário; todo o comportamento dinâmico é feito pelo `+api+`
* `+Backup/Restore+`: Executar periodicamente tarefas para fazer backup dos vários armazenamentos de dados. Possui um pod em funcionamento contínuo:
 ** `+suse-observability-minio-*+`: Fornece uma interface abstrata para interagir com o armazenamento de backup.
