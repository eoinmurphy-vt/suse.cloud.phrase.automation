= Open ID Connect (OIDC)
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: SUSE Observability Self-hosted

== Visão geral

O SUSE Observability pode se autenticar usando um provedor de autenticação OIDC. Para habilitar isso, você precisará configurar o SUSE Observability e o provedor OIDC para que possam se comunicar entre si. As seções a seguir descrevem as respectivas configurações.

== Configurar o provedor OIDC

Antes de configurar o SUSE Observability para autenticar usando o OIDC, você precisa criar um cliente para o SUSE Observability em seu provedor OIDC.

=== Fazendeiro

[NOTE]
Isso só funciona com o Rancher 2.12 ou posterior.  Você precisa https://documentation.suse.com/cloudnative/rancher-manager/latest/en/rancher-admin/users/authn-and-authz/configure-oidc-provider.html[configurar o Rancher como um provedor OIDC].

Crie um recurso OIDCClient no cluster local do Rancher:
[,yaml]
----
apiVersion: management.cattle.io/v3
kind: OIDCClient
metadata:
  name: oidc-observability
spec:
  tokenExpirationSeconds: 600
  refreshTokenExpirationSeconds: 3600
  redirectURIs:
    - "https://<observability-base-url>/loginCallback?client_name=StsOidcClient"
----

O Rancher disponibilizará o ID e o segredo do cliente em seu campo de status:
[,yaml]
----
apiVersion: management.cattle.io/v3
kind: OIDCClient
metadata:
  name: oidc-observability
spec:
  tokenExpirationSeconds: 600
  refreshTokenExpirationSeconds: 3600
  redirectURIs:
    - "https://<observability-base-url>/loginCallback?client_name=StsOidcClient"
status:
  clientID: <oidc-client-id>
  clientSecrets:
    client-secret-1:
      createdAt: "xxx"
      lastFiveCharacters: xxx
----

E você pode obter o valor secreto com
[,bash]
----
kubectl get secret <oidc-client-id> -n cattle-oidc-client-secrets -o jsonpath="{.data.client-secret-1}" | base64 -d
----


=== Outros provedores de OIDC

Use as seguintes configurações para o cliente (se necessário pelo provedor do OIDC):

* Use o fluxo de autorização do OIDC, que também costuma ser chamado de fluxo de código de autorização. O SUSE Observability não oferece suporte à concessão implícita e aos fluxos híbridos, portanto, não há necessidade de ativar o suporte a eles.
* Defina o *URI de redirecionamento* para o URL base do SUSE Observability com o sufixo `+/loginCallback?client_name=StsOidcClient+`. Por exemplo, `+https://stackstate.acme.com/loginCallback?client_name=StsOidcClient+`. Para alguns provedores de OIDC, como o Google e o Azure Entra ID, o URI de redirecionamento deve corresponder exatamente, incluindo todos os parâmetros de consulta. Nesse caso, você deve configurar o URI da seguinte forma `+https://stackstate.acme.com/loginCallback?client_name=StsOidcClient+`.
* Conceda ao SUSE Observability acesso a pelo menos os escopos `+openid+` e `+email+` ou o equivalente a eles para seu provedor OIDC. Dependendo do provedor, podem ser necessários mais escopos; se houver um `+profile+` separado, inclua-o também.
* O SUSE Observability precisa de acesso off-line ao OIDC. Para alguns provedores de identidade, isso requer um escopo extra, geralmente chamado `+offline_access+`.

O resultado dessa configuração deve produzir um *clientId* e um *segredo*. Copie-os e mantenha-os por perto para configurar o SUSE Observability. Anote também o *discoveryUri* do provedor. Normalmente, isso está na mesma tela ou pode ser encontrado na documentação.

== Configurar o SUSE Observability para OIDC

=== Fazendeiro

[NOTE]
Isso só funciona com o Rancher 2.12 ou posterior.  Você precisa https://documentation.suse.com/cloudnative/rancher-manager/latest/en/rancher-admin/users/authn-and-authz/configure-oidc-provider.html[configurar o Rancher como um provedor OIDC].

Para configurar o Rancher como o provedor de OIDC para o SUSE Observability, você precisa adicionar os detalhes do OIDC aos valores de autenticação:
[,yaml]
----
stackstate:
  authentication:
    rancher:
      clientId: "<oidc-client-id>"
      secret: "<oidc-secret>"
      baseUrl: "<rancher-url>"
----
Você pode substituir e estender a configuração do OIDC para o Rancher com os seguintes campos:

* **discoveryUri** - URI que você pode usar para descobrir o provedor OIDC. Normalmente, também documentado ou retornado ao criar o cliente no provedor OIDC.
* **redirectUri** - Opcional (não está no exemplo): O URI em que o ponto de extremidade de retorno de chamada de login do SUSE Observability pode ser acessado. Preenchido por padrão usando o endereço `+stackstate.baseUrl+`, mas pode ser substituído. Esse deve ser um URL totalmente qualificado que aponte para o caminho `+/loginCallback+`.
* **customParameters** - Mapa opcional de pares de chave/valor que você envia ao provedor OIDC como parâmetros de solicitação personalizados. Alguns provedores de OIDC exigem parâmetros de solicitação adicionais que não são enviados por padrão.

Se você precisar desativar a verificação de TLS devido a uma configuração que não usa certificados SSL verificáveis, poderá desativar as verificações de SSL com a configuração do aplicativo (não use em produção):

Para configuração **não HA**:
[,yaml]
----
stackstate:
  components:
    server:
      extraEnv:
        open:
          CONFIG_FORCE_stackstate_misc_sslCertificateChecking: false
----
Para a configuração de **HA (alta disponibilidade)**:
[,yaml]
----
stackstate:
  components:
    api:
      extraEnv:
        open:
          CONFIG_FORCE_stackstate_misc_sslCertificateChecking: false
----

Em implementações de HA, o componente api lida com a configuração de forma diferente. Use o seguinte:=== Kubernetes

Para configurar o SUSE Observability para usar um provedor de autenticação OIDC no Kubernetes, é necessário adicionar os detalhes do OIDC e o mapeamento da função do usuário ao arquivo `+authentication.yaml+`. Por exemplo:

[,yaml]
----
stackstate:
  authentication:
    oidc:
      clientId: "<client-id-from-oidc-provider>"
      secret: "<secret-from-oidc-provider>"
      discoveryUri: "https://oidc.acme.com/.well-known/openid-configuration"
      jwsAlgorithm: RS256
      scope: ["openid", "email"]
      jwtClaims:
        usernameField: email
        displayNameField: name
        groupsField: groups
      customParameters:
        access_type: offline

    # map the groups from OIDC provider
    # to the 4 standard roles in SUSE Observability (guest, powerUser, k8sTroubleshooter and admin)
    roles:
      guest: ["guest-group-in-oidc-provider"]
      powerUser: ["powerUser-group-in-oidc-provider"]
      admin: ["admin-group-in-oidc-provider"]
      k8sTroubleshooter: ["troubleshooter-group-in-oidc-provider"]
----

Siga as etapas abaixo para configurar o SUSE Observability para autenticar usando o OIDC:

. Em `+authentication.yaml+` - adicione detalhes do provedor de autenticação OIDC (veja o exemplo acima):
 ** *clientId* - O ID do xref:/setup/security/authentication/oidc.adoc#_configure_the_oidc_provider[cliente OIDC que você criou para o SUSE Observability].
 ** *secret* - O segredo do xref:/setup/security/authentication/oidc.adoc#_configure_the_oidc_provider[cliente OIDC que você criou para o SUSE Observability]
 ** *discoveryUri* - URI que pode ser usado para descobrir o provedor OIDC. Normalmente também documentado ou retornado ao criar o cliente no provedor OIDC.
 ** *jwsAlgorithm* - O padrão para o OIDC é `+RS256+`. Se o seu provedor de OIDC usar um diferente, ele poderá ser definido aqui.
 ** *escopo* - Deve corresponder ou ser um subconjunto do escopo fornecido na configuração do provedor do OIDC. O SUSE Observability usa isso para solicitar acesso a essas partes de um perfil de usuário no provedor OIDC.
 ** *redirectUri* - Opcional (não está no exemplo): O URI em que o ponto de extremidade de retorno de chamada de login do SUSE Observability pode ser acessado. Preenchido por padrão usando o endereço `+stackstate.baseUrl+`, mas pode ser substituído. Esse deve ser um URL totalmente qualificado que aponte para o caminho `+/loginCallback+`.
 ** *customParameters* - Mapa opcional de pares de chave/valor que são enviados ao provedor OIDC como parâmetros de solicitação personalizados. Alguns provedores de OIDC exigem parâmetros de solicitação adicionais que não são enviados por padrão.
 ** *jwtClaims* -
  *** *usernameField* - O campo no perfil de usuário do OIDC que deve ser usado como nome de usuário. Por padrão, esse será o `+preferred_username+`, no entanto, muitos provedores omitem esse campo. Uma boa alternativa é o site `+email+`.
  *** *displayNameField* - O campo no perfil de usuário do OIDC que deve ser usado como o displayName. Por padrão, esse será o `+name+`.
  *** *groupsField* - O campo do qual o SUSE Observability lerá a função/grupo de um usuário.
. Em `+authentication.yaml+` - mapeie as funções de usuário do OIDC para os assuntos corretos do SUSE Observability usando as configurações `+roles.guest+`, `+roles.powerUser+`, `+roles.admin+` ou `+roles.platformAdmin+` (veja o exemplo acima). Para obter detalhes, consulte as xref:/setup/security/rbac/rbac_permissions.adoc#_predefined_roles[funções padrão do SUSE Observability]. Mais funções do SUSE Observability também podem ser criadas, consulte a xref:/setup/security/rbac/README.adoc[documentação do RBAC].
. Armazene o arquivo `+authentication.yaml+` junto com o arquivo `+values.yaml+` das instruções de instalação do SUSE Observability.
. Execute um upgrade do Helm para aplicar as alterações:
+
[,text]
----
 helm upgrade \
   --install \
   --namespace suse-observability \
   --values values.yaml \
   --values authentication.yaml \
 suse-observability \
 suse-observability/suse-observability
----

[NOTE]
====
*Observação:*

* A primeira execução do comando helm upgrade resultará na reinicialização dos pods, o que pode causar uma breve interrupção da disponibilidade.
* Inclua `+authentication.yaml+` em cada execução de `+helm upgrade+`.
* A configuração de autenticação é armazenada como um segredo do Kubernetes.
====

== Usando um segredo externo

Quando os segredos do oidc forem provenientes de um segredo externo, siga xref:/setup/security/external-secrets.adoc#_getting_authentication_data_from_an_external_secret[estas etapas], mas preencha os dados a seguir:

[,yaml]
----
kind: Secret
metadata:
   name: "<custom-secret-name>"
type: Opaque
data:
  oidc_client_id: <base64 of client id>
  oidc_secret: <base64 of secret>
----
