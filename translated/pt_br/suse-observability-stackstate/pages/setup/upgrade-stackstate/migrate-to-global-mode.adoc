= Migrar para o modo de configuração global
:description: {stackstate-product-name} Auto-hospedado
:page-toclevels: 3

== Visão geral

Este guia explica como migrar da instalação antiga de duas etapas usando o gráfico `+suse-observability-values+` para a instalação simplificada de um único gráfico usando a configuração `+global.suseObservability+`.

[NOTE]
====
O método de configuração `+global.suseObservability+` está disponível a partir da versão `+2.8.0+` e agora é a abordagem de instalação recomendada. O método legado do gráfico `+suse-observability-values+` está obsoleto, mas continuará funcionando nas instalações existentes.
====

=== Instalação herdada (obsoleta)

O método de instalação antigo exigia duas etapas:

. Gerar arquivos de valores usando o gráfico `+suse-observability-values+` 
. Instale o gráfico `+suse-observability+` com os valores gerados

[,text]
----
# Step 1: Generate values
helm template suse-observability-values suse-observability/suse-observability-values \
  --set license="YOUR-LICENSE-KEY" \
  --set baseUrl="https://observability.example.com" \
  --set sizing.profile="150-ha" \
  --set adminPassword="your-password" \
  --set pullSecret.username="registry-user" \
  --set pullSecret.password="registry-pass" \
  > generated-values.yaml

# Step 2: Install with generated values
helm install suse-observability suse-observability/suse-observability \
  -f generated-values.yaml \
  -n suse-observability
----

=== Nova instalação (recomendada)

O novo método usa um único arquivo de valores com a configuração `+global.suseObservability+`:

[,text]
----
helm install suse-observability suse-observability/suse-observability \
  -f values.yaml \
  -n suse-observability
----

== Converter antes da instalação

Se você tiver preparado parâmetros para o gráfico `+suse-observability-values+` legado, mas ainda não o tiver instalado, poderá converter diretamente para o novo método sem criar arquivos de valores intermediários.

=== Conversão direta de parâmetros

Converta seu comando `+helm template+` em um comando `+helm upgrade --install+` mapeando os parâmetros:

[cols="1,1"]
|===
| Parâmetro antigo | Novo parâmetro

| `+--set license="..."+`
| `+--set global.suseObservability.license="..."+`

| `+--set baseUrl="..."+`
| `+--set global.suseObservability.baseUrl="..."+`

| `+--set sizing.profile="..."+`
| `+--set global.suseObservability.sizing.profile="..."+`

| `+--set receiverApiKey="..."+`
| `+--set global.suseObservability.receiverApiKey="..."+`

| `+--set adminPassword="..."+`
| `+--set global.suseObservability.adminPassword="..."+`

| `+--set imageRegistry="..."+`
| `+--set global.imageRegistry="..."+`

| `+--set pullSecret.username="..."+`
| `+--set global.suseObservability.pullSecret.username="..."+`

| `+--set pullSecret.password="..."+`
| `+--set global.suseObservability.pullSecret.password="..."+`
|===

=== Exemplo: Antes e depois

[NOTE]
====
Antes de executar os comandos do helm, verifique se você tem a versão mais recente do gráfico:

[,bash]
----
helm repo update
----
====

.Comando legado (não usar)
[,bash]
----
export VALUES_DIR=.
helm template \
  --set license="YOUR-LICENSE-KEY" \
  --set baseUrl="https://observability.example.com" \
  --set sizing.profile="10-nonha" \
  --set receiverApiKey="YOUR-API-KEY" \
  suse-observability-values \
  suse-observability/suse-observability-values --output-dir $VALUES_DIR

# Then: helm install with generated files...
----

.Novo comando (recomendado)
[,bash]
----
helm upgrade --install \
  --namespace suse-observability \
  --create-namespace \
  --set global.suseObservability.license="YOUR-LICENSE-KEY" \
  --set global.suseObservability.baseUrl="https://observability.example.com" \
  --set global.suseObservability.sizing.profile="10-nonha" \
  --set global.suseObservability.receiverApiKey="YOUR-API-KEY" \
  --set global.suseObservability.adminPassword='your-password' \
  suse-observability suse-observability/suse-observability
----

[NOTE]
====
* O `+adminPassword+` deve ser um hash bcrypt no novo método. Consulte <<bcrypt-hash>> para saber como gerar um.
* Para ambientes com air-gap ou registros personalizados, consulte xref:/k8s-suse-rancher-prime-air-gapped.adoc[Instalação do {stackstate-product-name} no modo com air-gap] para obter configurações adicionais, incluindo as configurações `+imageRegistry+` e `+pullSecret+`.
====

=== Usando um arquivo de valores em vez disso

Se preferir usar um arquivo de valores, crie `+values.yaml+` com a configuração mostrada em <<_step_3_create_new_values_file>>, certifique-se de que você tenha o gráfico mais recente com `+helm repo update+` e, em seguida, execute-o:

[,bash]
----
helm upgrade --install \
  --namespace suse-observability \
  --create-namespace \
  --values values.yaml \
  suse-observability suse-observability/suse-observability
----

== Antes de migrar

Ao executar uma migração do método de instalação herdado, esteja ciente do seguinte:

[CAUTION]
====
Este guia de migração destina-se aos usuários que usam atualmente o método de gráfico `+suse-observability-values+`. Se estiver fazendo upgrade do StackState 6.0, siga o xref:/setup/upgrade-stackstate/migrate-from-6.adoc[guia de migração do StackState].
====

[NOTE]
====
* A migração é realizada usando um comando padrão de upgrade do Helm
* Seus dados e configurações existentes serão preservados
* A migração pode ser realizada durante uma janela de manutenção
* A reversão é possível usando os procedimentos padrão de reversão do Helm
====

== Etapas de migração

=== Etapa 1: Backup da configuração atual

Antes de iniciar a migração, salve os valores existentes do Helm e todas as personalizações:

[,text]
----
# Save current Helm values
helm get values suse-observability -n suse-observability > backup-values.yaml

# Save your original suse-observability-values input (if available)
# This is typically your values file used with helm template

# Backup important resources
kubectl get secrets -n suse-observability -o yaml > backup-secrets.yaml
----

=== Etapa 2: Identifique sua configuração atual

Analise sua configuração existente para identificar as seguintes configurações:

* *Perfil de dimensionamento*: Verifique `+sizing.profile+` em sua entrada suse-observability-values
* *Chave de licença*: Sua licença {stackstate-product-name} 
* *URL de base*: O URL em que {stackstate-product-name} pode ser acessado
* *Senha de administrador*: A senha do administrador (você também pode usar o hash bcrypt da senha; o valor para isso é `+adminPasswordBcrypt+`)
* *Obter credenciais secretas*: Nome de usuário e senha do registro (se usados)
* *Configurações de afinidade personalizadas*: Quaisquer personalizações de afinidade de nó ou antiafinidade de pod

Você pode extrair alguns deles de sua instalação atual:

[,text]
----
# Check current values
helm get values suse-observability -n suse-observability

# The sizing profile is visible in resource configurations
# Look for patterns like replica counts, resource limits, etc.
----

=== Compreensão dos valores gerados automaticamente e dos valores personalizados

Ao executar `+helm get values+`, você poderá ver centenas de linhas de configuração. No entanto, a maioria deles é *gerada automaticamente* pelo perfil de dimensionamento e *não* precisa ser preservada manualmente em seu arquivo de valores de migração.

==== O que o perfil de dimensionamento fornece automaticamente

Quando você define `+global.suseObservability.sizing.profile+`, o gráfico é configurado automaticamente:

* Solicitações e limites de recursos para todos os componentes (Elasticsearch, Kafka, ClickHouse, servidor, receptor, etc.)
* Contagens de réplicas para serviços com estado
* Tamanhos de armazenamento para volumes persistentes
* Modo de implantação do HBase (Mono para não HA, Distribuído para HA)
* Configuração de divisão de servidor e receptor
* Configurações de HA do Victoria Metrics
* Variáveis de ambiente (`+extraEnv+`), incluindo configurações de `+CONFIG_FORCE_*+` para ajuste de desempenho

==== O que você precisa preservar

Preserve apenas os valores que são *realmente personalizados* - configurações que você adicionou ou modificou além do que o perfil de dimensionamento fornece:

[cols="1,1"]
|===
| Preservar (personalizado) | Não preservar (gerado automaticamente)

| Configurações principais (licença, baseUrl, adminPassword, receiverApiKey)
| Limites e solicitações de recursos

| Configuração de entrada (hosts, TLS, anotações)
| Contagens de réplicas

| Variáveis de ambiente personalizadas que você adicionou
| Tamanhos de armazenamento

| Configurações de autenticação (LDAP, OIDC)
| `+CONFIG_FORCE_*+` variáveis de ambiente

| Personalizado `+allowedOrigins+`
| Fatores de replicação do Kafka/Zookeeper

| Entrada personalizada do coletor OpenTelemetry
| Modo de implantação do HBase
|===

==== Exemplo: Arquivo de valores mínimos de migração

Seu arquivo de valores de migração deve ser *muito menor* do que a saída do `+helm get values+`:

[,yaml]
----
global:
  suseObservability:
    license: "YOUR-LICENSE"
    baseUrl: "https://your-url.example.com"
    sizing:
      profile: "10-nonha"  # This handles all resource configuration!
    adminPassword: "your-password"
    receiverApiKey: "your-api-key"

# Only include truly custom configurations below
ingress:
  enabled: true
  ingressClassName: your-ingress-class
  annotations:
    # your custom annotations
  hosts:
  - host: your-url.example.com
  tls:
  - hosts:
    - your-url.example.com
    secretName: your-tls-secret
----

[TIP]
====
Se você não tiver certeza se um valor é personalizado ou gerado automaticamente, compare-o com os padrões do perfil de dimensionamento. Valores como as variáveis de ambiente `+elasticsearch.resources+`, `+kafka.resources+`, `+stackstate.components.server.resources+` e `+CONFIG_FORCE_*+` são todos gerados pelo perfil e podem ser omitidos com segurança.
====

=== Etapa 3: Criar novo arquivo de valores

Crie um novo `+values.yaml+` com a estrutura `+global.suseObservability+`:

[,yaml]
----
global:
  suseObservability:
    # Required: Your license key
    license: "YOUR-LICENSE-KEY"

    # Required: Base URL for {stackstate-product-name}
    baseUrl: "https://observability.example.com"

    # Required: Sizing profile
    # Available: trial, 10-nonha, 20-nonha, 50-nonha, 100-nonha,
    #            150-ha, 250-ha, 500-ha, 4000-ha
    sizing:
      profile: "150-ha"

    adminPassword: "your-password"
    # Instead of adminPassword you can provide a bcrypt hashed password
    # Generate with: htpasswd -bnBC 10 "" "your-password" | tr -d ':\n'
    # adminPasswordBcrypt: "$2a$10$..."

    # Optional: Receiver API key (auto-generated if not provided)
    receiverApiKey: "your-receiver-api-key"

    # Optional: Affinity configuration
    # These settings apply to all infrastructure components (Elasticsearch,
    # Kafka, ClickHouse, etc.) automatically - no need to configure per-component
    affinity:
      # Node affinity for all components
      nodeAffinity: null

      # Pod anti-affinity for HA profiles (infrastructure components)
      podAntiAffinity:
        requiredDuringSchedulingIgnoredDuringExecution: true
        topologyKey: "kubernetes.io/hostname"

  # For air-gapped or custom registry configurations, see:
  # xref:/k8s-suse-rancher-prime-air-gapped.adoc[Air-Gapped Installation Guide]
----

=== Etapa 5: Lidar com substituições personalizadas

Se você tiver substituições personalizadas em seu arquivo de valores gerados, ainda poderá usá-las. O novo modo global fornece padrões sensatos que podem ser substituídos por componente:

[,yaml]
----
global:
  suseObservability:
    sizing:
      profile: "150-ha"
    license: "YOUR-LICENSE-KEY"
    baseUrl: "https://observability.example.com"
    adminPassword: "your-password"
    # Global affinity - applied to all components by default
    affinity:
      nodeAffinity:
        requiredDuringSchedulingIgnoredDuringExecution:
          nodeSelectorTerms:
          - matchExpressions:
            - key: topology.kubernetes.io/zone
              operator: In
              values:
              - us-west-2a

# Custom per-component overrides still work - they take precedence over global defaults
stackstate:
  components:
    api:
      resources:
        requests:
          memory: 16Gi  # Override default from sizing profile
        limits:
          memory: 20Gi

elasticsearch:
  volumeClaimTemplate:
    resources:
      requests:
        storage: 500Gi  # Override default storage size
  # Override affinity for Elasticsearch specifically
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: node-type
            operator: In
            values:
            - storage-optimized
----

=== Etapa 6: Executar o upgrade

Primeiro, verifique se você tem a versão mais recente do gráfico:

[,text]
----
helm repo update
----

Em seguida, execute a atualização:

[,text]
----
# Dry-run first to see what will change
helm upgrade suse-observability suse-observability/suse-observability \
  -n suse-observability \
  -f values.yaml \
  --dry-run

# If everything looks good, perform the actual upgrade
helm upgrade suse-observability suse-observability/suse-observability \
  -n suse-observability \
  -f values.yaml
----

[[bcrypt-hash]]
== Senha do administrador

Para o método legado, o `+adminPassword+` foi fornecido como texto simples para o gráfico suse-observability-values, que o transformou em um hash bcrypt. No novo método, você pode fornecer o texto simples `+adminPassword+` para o gráfico de suse-observability ou fornecer o hash bcrypt diretamente para o gráfico de suse-observability usando `+adminPasswordBcrypt+`.

Se você precisar gerar um novo hash bcrypt:

[,bash]
----
htpasswd -bnBC 10 "" "your-password" | tr -d ':\n'
----

== Solução de problemas

=== Pods presos no estado Pendente

Isso geralmente indica um problema de agendamento com regras de antiafinidade. Para clusters com nós limitados, use a antiafinidade suave:

[,yaml]
----
global:
  suseObservability:
    affinity:
      podAntiAffinity:
        requiredDuringSchedulingIgnoredDuringExecution: false  # Use soft anti-affinity
----

=== Os recursos são diferentes da instalação anterior

Os perfis de dimensionamento podem ter recomendações de recursos atualizadas. Para preservar suas configurações anteriores, adicione substituições explícitas:

[,yaml]
----
global:
  suseObservability:
    sizing:
      profile: "150-ha"

# Override with your previous resource values
stackstate:
  components:
    api:
      resources:
        requests:
          memory: 8Gi  # Your previous value
----

=== Conflitos de estilo de configuração

Não misture estilos de configuração antigos e novos. Use um dos dois:

* `+global.suseObservability.*+` (novo estilo), OU
* `+stackstate.license.key+`, `+stackstate.baseUrl+`, etc. (estilo antigo)

[,text]
----
# Check for conflicting values
helm get values suse-observability -n suse-observability | grep -E "license|baseUrl"
----

=== Reversão, se necessário

[,text]
----
# List release history
helm history suse-observability -n suse-observability

# Rollback to previous revision
helm rollback suse-observability <revision-number> -n suse-observability
----

== Principais diferenças

[cols="1,1,1"]
|===
| Aspecto | Legado (suse-observability-values) | Novo (global.suseObservability)

| Etapas de instalação
| 2 (modelo + instalação)
| 1 (instalar)

| Gerenciamento de arquivos de valores
| Gerado, deve ser armazenado
| Fonte única de verdade

| Atualizações de perfil
| Necessidade de regeneração manual
| Automático com atualizações de gráficos

| Puxar segredo
| Subgráfico separado
| Integrado, configurado automaticamente

| Victoria Metrics HA
| Manual `+enabled: true/false+`
| Configuração automática por perfil

| Configuração de afinidade
| Gerado por componente (configurações separadas para cada serviço)
| Configurações globais centralizadas (ainda podem ser substituídas por componente, se necessário)
|===

[NOTE]
====
A configuração global de afinidade fornece configurações padrão para todos os componentes da infraestrutura. Você ainda pode substituir a afinidade de componentes específicos (como Elasticsearch ou Kafka) quando necessário, definindo valores de afinidade no nível do componente. As configurações em nível de componente têm precedência sobre os padrões globais.
====

== Limpeza

Após a migração bem-sucedida, você pode:

. Remova os arquivos de valores gerados (eles não são mais necessários)
. Atualize seus scripts de implantação para usar a nova instalação em uma única etapa
. Remova todas as referências ao gráfico `+suse-observability-values+` em seus pipelines de CI/CD
