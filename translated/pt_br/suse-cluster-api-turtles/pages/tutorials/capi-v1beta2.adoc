= Atualizando para o CAPI v1beta2

A versão `+v0.26.0+` do Rancher Turtles adota o contrato CAPI `+v1beta2+`, que vem com o núcleo CAPI v1.12.2. Se você estiver usando o provisionamento baseado em CAPI, há alguns aspectos que devem ser revisados antes da atualização.

As mudanças se dividem em três áreas:

* xref:#_clusterclass[Como os recursos do `+ClusterClass+` são estruturados].
* xref:#_cluster[Como a topologia do `+Cluster+` faz referência à sua classe].
* xref:#_rke2_provider_caprke2[O que mudou no provedor RKE2].

Leia as seções que se aplicam à sua configuração antes de fazer qualquer alteração.

== Uma observação sobre compatibilidade com versões anteriores

O CAPI v1.12 continua a atender aos recursos do `+v1beta1+` juntamente com o `+v1beta2+`. Os recursos existentes em seu cluster continuarão funcionando sem uma migração imediata, pois o servidor de API faz a conversão entre as duas versões de forma transparente. Dito isso, todos os novos manifestos que você escrever devem ter como alvo o site `+v1beta2+`, e você deve planejar a migração dos existentes ao longo do tempo. A API `+v1beta1+` está programada para ser removida em uma versão futura do CAPI (agosto de 2026) quando o ecossistema tiver tido tempo de se estabelecer em `+v1beta2+`.

*Recomendação*: primeiro avalie se você está pronto para adotar o site `+v1beta2+` e aja de acordo. Se você optar por não migrar imediatamente, crie um plano para mudar para o uso da nova API a fim de evitar interrupções no futuro.

Além disso, a comunidade upstream fornece um guia oficial de migração:

* https://cluster-api.sigs.k8s.io/developer/providers/migrations/v1.10-to-v1.11[CAPI v1.10 a v1.11]: abrange todas as alterações `+v1beta1+` a `+v1beta2+` no núcleo do CAPI.
* https://cluster-api.sigs.k8s.io/developer/providers/migrations/v1.11-to-v1.12[CAPI v1.11 para v1.12]: a maioria das alterações no CAPI v1.12 pode ser considerada pequena.

== ClusterClass

=== Referências de modelos

Em `+v1beta1+`, o campo que aponta para os objetos do modelo `+ClusterClass+` foi denominado `+ref+`. Em `+v1beta2+` ele é chamado de `+templateRef+`. Isso se aplica a `+spec.infrastructure+`, `+spec.controlPlane+` e `+spec.controlPlane.machineInfrastructure+`.

Antes:

[,yaml]
----
spec:
  infrastructure:
    ref:
      apiVersion: infrastructure.cluster.x-k8s.io/v1beta1
      kind: DockerClusterTemplate
      name: my-cluster-template
----

Depois:

[,yaml]
----
spec:
  infrastructure:
    templateRef:
      apiVersion: infrastructure.cluster.x-k8s.io/v1beta2
      kind: DockerClusterTemplate
      name: my-cluster-template
----

=== Seção de trabalhadores

Em `+v1beta1+`, as referências de bootstrap e infraestrutura para implantações de máquinas foram aninhadas em um campo `+template+`. Essa chave raiz desapareceu em `+v1beta2+`. `+bootstrap+` e `+infrastructure+` agora são filhos diretos da entrada de implantação da máquina.

Antes:

[,yaml]
----
workers:
  machineDeployments:
    - class: default-worker
      template:
        bootstrap:
          ref:
            apiVersion: bootstrap.cluster.x-k8s.io/v1beta1
            kind: KubeadmConfigTemplate
            name: my-bootstrap-template
        infrastructure:
          ref:
            apiVersion: infrastructure.cluster.x-k8s.io/v1beta1
            kind: DockerMachineTemplate
            name: my-worker-template
----

Depois:

[,yaml]
----
workers:
  machineDeployments:
    - class: default-worker
      bootstrap:
        templateRef:
          apiVersion: bootstrap.cluster.x-k8s.io/v1beta2
          kind: KubeadmConfigTemplate
          name: my-bootstrap-template
      infrastructure:
        templateRef:
          apiVersion: infrastructure.cluster.x-k8s.io/v1beta2
          kind: DockerMachineTemplate
          name: my-worker-template
----

== Grupo

O recurso `+Cluster+` agora faz referência ao seu `+ClusterClass+` por meio de um objeto estruturado em vez de uma cadeia de caracteres em linha. Em `+v1beta1+`, o nome da classe e o namespace foram especificados como `+spec.topology.class+` e `+spec.topology.classNamespace+`. Em `+v1beta2+`, ambos os campos são substituídos por um único objeto `+spec.topology.classRef+`.

Antes:

[,yaml]
----
apiVersion: cluster.x-k8s.io/v1beta1
kind: Cluster
spec:
  topology:
    class: my-clusterclass
    classNamespace: capi-system
----

Depois:

[,yaml]
----
apiVersion: cluster.x-k8s.io/v1beta2
kind: Cluster
spec:
  topology:
    classRef:
      name: my-clusterclass
      namespace: capi-system
----

O restante da especificação `+Cluster+` (configurações de rede, `+controlPlane.replicas+`, `+workers.machineDeployments+`, `+variables+`, `+version+`, etc.) permanece inalterado.

== Provedor de RKE2 (CAPRKE2)

Esta versão traz o CAPRKE2 v0.23.1, acima da v0.22.1. A API `+v1beta2+` para os recursos de bootstrap e plano de controle do RKE2 foi introduzida na v0.22.0 juntamente com o suporte para CAPI v1.11, e a v0.23.0 concluiu esse trabalho ao passar para CAPI v1.12.2. Tanto a v0.22.1 quanto a v0.23.1 são lançamentos de patches sobre essas respectivas versões secundárias.

*Observação*: leia a documentação do provedor sobre as alterações da API no link:https://caprke2.docs.rancher.com/05_reference/01_api_versions.html#v1beta1-to-v1beta2[livro]

=== Nova API v1beta2

O CAPRKE2 agora oferece sua própria API `+v1beta2+` para `+RKE2ControlPlane+`, `+RKE2ControlPlaneTemplate+` e `+RKE2Config+`/`+RKE2ConfigTemplate+`. A API `+v1beta1+` continua disponível, mas vários campos que foram marcados como obsoletos em `+v1beta1+` agora foram removidos em `+v1beta2+`.

=== Campos removidos no RKE2ControlPlane

Os seguintes campos foram removidos de `+RKE2ControlPlane.spec+` em `+v1beta2+`:

* `+infrastructureRef+` - em vez disso, use `+spec.machineTemplate.spec.infrastructureRef+`.
* `+nodeDrainTimeout+` - em vez disso, use `+spec.machineTemplate.spec.deletion.nodeDrainTimeout+`.

=== Campos de tempo limite renomeados

Os campos de tempo limite em `+RKE2ControlPlaneMachineTemplate+` foram movidos para um subobjeto `+deletion+` e renomeados. Eles também mudaram o tipo de `+metav1.Duration+` para `+int32+` e agora esperam valores expressos em segundos em vez de uma string de duração.

|===
| v1beta1 | v1beta2

| `+nodeDrainTimeout+`
| `+deletion.nodeDrainTimeoutSeconds+`

| `+nodeVolumeDetachTimeout+`
| `+deletion.nodeVolumeDetachTimeoutSeconds+`

| `+nodeDeletionTimeout+`
| `+deletion.nodeDeletionTimeoutSeconds+`
|===

O objeto `+RKE2ControlPlaneMachineTemplate+` agora também exige um campo `+spec+`.

=== Prevalência da v1alpha1

A API `+v1alpha1+` foi descontinuada no CAPRKE2 v0.22.0. Se você ainda estiver usando os recursos do `+v1alpha1+`, agora é um bom momento para migrar para, pelo menos, o `+v1beta1+`.

== Etapas recomendadas antes de fazer o upgrade

. Atualize os manifestos `+ClusterClass+` que você mantém: renomeie `+ref+` para `+templateRef+`, remova o wrapper `+template+` na seção workers e atualize as cadeias de caracteres CAPI `+apiVersion+` para `+v1beta2+`.
. Atualize os manifestos `+Cluster+`: substitua `+spec.topology.class+` e `+spec.topology.classNamespace+` por `+spec.topology.classRef.name+` e `+spec.topology.classRef.namespace+`.
. Se você usar os recursos do CAPRKE2 `+v1beta2+`, remova todos os campos diretos `+infrastructureRef+` ou `+nodeDrainTimeout+` de `+RKE2ControlPlane.spec+` e mova-os para `+spec.machineTemplate.spec+`.
. Consulte os exemplos atualizados em `+examples/clusterclasses/+` para obter manifestos completos e funcionais para cada provedor de CAPI certificado.
