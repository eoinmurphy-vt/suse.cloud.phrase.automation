== {project-name} controle de versão da pilha

O projeto {short-project-name} usa o https://semver.org/[controle de versão semântico] para definir a versão "stack" de todos os seus componentes: a versão segue o padrão`+MAJOR.MINOR.PATCH+`. A versão compatível é a mais recente.

Os componentes do site {short-project-name} que seguem as regras da versão "stack" são:

* `+kubewarden-crds+` em seu campo appVersion.

* `+kubewarden-controller+` em seu campo appVersion.

* `+kubewarden-defaults+` em seu campo appVersion.

* `+policy-server+` para as imagens implantadas manualmente. O recurso para o padrão já é gerenciado pelo gráfico `+kubewarden-defaults+`.

* `+kwctl+` binário.

== Compatibilidade de versão de pilha entre componentes

O gráfico `+kubewarden-crds+`, o gráfico `+kubewarden-controller+`, o gráfico`+kubewarden-defaults+`, qualquer imagem `+policy-server+` implantada manualmente e o`+kwctl+` devem executar a mesma versão `+MAJOR+`.`+MINOR+`.`+PATCH+`.

Portanto, se a versão do `+kubewarden-controller+` em execução for `+1.1.2+`, a versão`` do \+policy-server++s`` e + do `kwctl`+ em uso também deverá ser`+1.1.2`.

== Versões do gráfico do leme

Os gráficos do leme definem o campo `+version+` e o campo `+appVersion+`. O campo`+appVersion+` informa a versão da "pilha" {short-project-name}, conforme mencionado anteriormente. O campo `+version+` também segue https://semver.org/[semantic versioning] e descreve alterações compatíveis com versões anteriores nos modelos de gráficos e `+values.yaml+`.

== Caminhos de atualização

Ao fazer upgrade de componentes, é possível fazer upgrade de várias versões da pilha `+PATCH+` em uma única operação. No entanto, *não* há suporte para a atualização de várias versões da pilha `+MAJOR+` ou `+MINOR+` em uma única atualização.

Por exemplo, você pode fazer upgrade de componentes da versão `+1.1.10+` para `+1.1.nn+` em um único upgrade. Mas a atualização de `+1.1.10+` para `+1.5.0+` não é suportada. Nesses casos, você deve fazer upgrade individualmente para cada versão do `+MAJOR+`/`+MINOR+` entre as duas versões. Portanto, é necessário atualizar `+1.1.10+` para`+1.2.0+`, depois para `+1.3.0+`, depois para `+1.4.0+` e, finalmente, para `+1.5.0+`. Para atualizar uma versão`+MAJOR+` para outra, você precisa aplicar todas as atualizações `+MINOR+` entre as duas versões `+MAJOR+`.

== Pedido de upgrade

{short-project-name} Os usuários devem atualizar a pilha começando com o gráfico `+kubewarden-crds+`e, em seguida, com o gráfico `+kubewarden-controller+`. Depois disso, atualize os sites`+policy-server+` (por meio do gráfico `+kubewarden-defaults+` ou das imagens dos personalizados) e `+kwctl+`.

== Rebaixamentos

*Não* há suporte para downgrades, e o site {short-project-name} não os testa. No entanto, há uma expectativa razoável de que eles funcionem.

== SDKs, políticas

Os SDKs de política para os diferentes idiomas e políticas mantidos pela equipe do{short-project-name} seguem seu próprio controle de versão semântico. O {short-project-name} é compatível com a versão mais recente. Não há necessidade de um caminho de atualização para eles, apenas um aumento para a versão mais recente.

As alterações na pilha do {short-project-name} podem significar que as políticas e os SDKs recebem atualizações para usar os recursos mais recentes do {short-project-name}. O {short-project-name} tem o cuidado de realizar essas atualizações de forma compatível com as versões anteriores.

Por exemplo, uma versão secundária do site {short-project-name} que adiciona suporte ao Audit Scanner (`+v1.7.0+`) significa que as políticas ganharam um novo campo `+spec.backgroundAudit+`. Isso é opcional, compatível com versões anteriores e definido como `+true+` por padrão.
