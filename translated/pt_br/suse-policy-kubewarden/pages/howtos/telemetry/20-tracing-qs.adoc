Esta seção mostra como ativar o suporte de rastreamento para o Policy Server.

[NOTE]
====

Antes de continuar, certifique-se de ter concluído a seção anterior doxref:howtos/telemetry/10-opentelemetry-qs.adoc#_install_opentelemetry[OpenTelemetry]desta documentação. Ela é necessária para que esta seção funcione corretamente.

====


O rastreamento permite coletar detalhes minuciosos sobre as avaliações de políticas. Pode ser uma ferramenta útil para depurar problemas em sua implementação e políticas do {project-name}.

Você usa o https://www.jaegertracing.io/[site] -- usado para receber, armazenar e visualizar eventos de rastreamento.

== Instalar o Jaeger

Você usa o https://github.com/jaegertracing/jaeger-operator[Jaeger Operator]para gerenciar todos os diferentes componentes do Jaeger. Você pode instalar o Jaeger Operator usando os gráficos do Helm.

[NOTE]
====

No momento da redação deste artigo (2022-06-21), somente versões específicas do Jaeger são compatíveis com o Cert Manager,https://github.com/jaegertracing/helm-charts/blob/main/charts/jaeger-operator/COMPATIBILITY.md[consulte a tabela de compatibilidade].

====

Para instalar o Helm Chart:

[subs="+attributes",console]
----
helm repo add jaegertracing https://jaegertracing.github.io/helm-charts

helm upgrade -i --wait \
  --namespace jaeger \
  --create-namespace \
  --version 2.49.0 \
  jaeger-operator jaegertracing/jaeger-operator \
  --set rbac.clusterRole=true
----

[CAUTION]
====

Isso *não é adequado para a implantação de produção*. Você deve consultar o site https://www.jaegertracing.io/docs/latest/kubernetes/[documentação oficial] do Jaeger.

====

Para criar um recurso Jaeger:

[subs="+attributes",console]
----
kubectl apply -f - <<EOF
apiVersion: jaegertracing.io/v1
kind: Jaeger
metadata:
  name: my-open-telemetry
  namespace: jaeger
spec: {}
EOF
----

Após a criação dos recursos do Jaeger Operator, você terá um serviço em`+my-open-telemetry-collector.jaeger.svc.cluster.local+`. A interface do usuário do Jaeger Query é exposta pelo serviço `+my-open-telemetry-query+` no espaço de nomes `+jaeger+`.

== Instalar {short-project-name}

Agora você pode prosseguir com a implementação do {short-project-name} da maneira usual.

[NOTE]
====

O `+cert-manager+` é um requisito do OpenTelemetry, mas você já o instalou em uma seção anterior desta documentação.

====

Como primeira etapa, você adiciona o repositório do Helm que contém{short-project-name}:

[subs="+attributes",console]
----
helm repo add kubewarden https://charts.kubewarden.io
----

Em seguida, você instala as Custom Resource Definitions (CRDs) definidas por{short-project-name}:

[subs="+attributes",console]
----
helm install --wait \
  --namespace kubewarden \
  --create-namespace \
  kubewarden-crds kubewarden/kubewarden-crds
----

Agora você pode implementar o restante da pilha {short-project-name}. O helm chart oficial`+kubewarden-defaults+` cria um PolicyServer chamado `+default+`. Você deseja que essa instância do PolicyServer tenha o rastreamento ativado.

Para fazer isso, você precisa especificar alguns valores extras para o gráfico`+kubewarden-controller+`. Você deve criar um arquivo `+values.yaml+` com o seguinte conteúdo:

[subs="+attributes",yaml]
----
telemetry:
  mode: sidecar
  tracing: True
  sidecar:
    tracing:
      jaeger:
        endpoint: "my-open-telemetry-collector.jaeger.svc.cluster.local:4317"
        tls:
          insecure: true
----

[CAUTION]
====

Para simplificar, não há criptografia da comunicação entre o coletor OpenTelemetry e o ponto de extremidade Jaeger.

Novamente, isso não é *adequado para uma implementação de produção*. Consulte do Jaegerhttps://www.jaegertracing.io/docs/latest/kubernetes/[a documentação oficial].

====

Em seguida, você pode prosseguir com a instalação dos gráficos do leme:

[subs="+attributes",console]
----
helm install --wait --namespace kubewarden --create-namespace \
  --values values.yaml \
  kubewarden-controller kubewarden/kubewarden-controller

helm install --wait --namespace kubewarden --create-namespace \
  kubewarden-defaults kubewarden/kubewarden-defaults
----

Isso cria a instância `+default+` de `+PolicyServer+`:

[subs="+attributes",console]
----
kubectl get policyservers.policies.kubewarden.io
NAME      AGE
default   3m7s
----

Observando mais de perto o pod que executa a instância do PolicyServer, você pode ver que ele tem dois contêineres. O `+policy-server+` e o sidecar do OpenTelemetry Collector `+otc-container+`.

== Aplicação de uma política

Você começa implantando a política dehttps://github.com/kubewarden/safe-labels-policy[rótulos seguros].

Você deseja que a política seja aplicada somente em Namespaces que tenham um rótulo `+environment+`com um valor de `+production+`.

Para um Namespace que tenha esse rótulo:

[subs="+attributes",console]
----
kubectl apply -f - <<EOF
apiVersion: v1
kind: Namespace
metadata:
  name: team-alpha-prod
  labels:
    environment: production
EOF
----

Em seguida, você pode definir uma ClusterAdmissionPolicy:

[subs="+attributes",yaml]
----
kubectl apply -f - <<EOF
apiVersion: policies.kubewarden.io/v1
kind: ClusterAdmissionPolicy
metadata:
  name: safe-labels
spec:
  module: registry://ghcr.io/kubewarden/policies/safe-labels:v0.1.6
  settings:
    mandatory_labels:
    - owner
  rules:
    - apiGroups:
        - apps
      apiVersions:
        - v1
      resources:
        - deployments
      operations:
        - CREATE
        - UPDATE
  namespaceSelector:
    matchExpressions:
    - key: environment
      operator: In
      values: ["production"]
  mutating: false
EOF
----

Você precisa esperar que a política se torne ativa:

[subs="+attributes",console]
----
kubectl wait --for=condition=PolicyActive clusteradmissionpolicy/safe-labels
----

Quando a política estiver ativa, você poderá experimentá-la:

[subs="+attributes",console]
----
kubectl apply -f - <<EOF
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
  namespace: team-alpha-prod
  labels:
    owner: octocat
spec:
  selector:
    matchLabels:
      app: nginx
  replicas: 0
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:latest
        ports:
        - containerPort: 80
EOF
----

A política permite a criação desse objeto de implementação, pois ele não viola a política.

A política bloqueia esse objeto de implementação:

[subs="+attributes",console]
----
kubectl apply -f - <<EOF
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment-without-labels
  namespace: team-alpha-prod
spec:
  selector:
    matchLabels:
      app: nginx
  replicas: 0
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:latest
        ports:
        - containerPort: 80
EOF
----

A política não é imposta em outro Namespace.

Esse comando cria um novo Namespace chamado `+team-alpha-staging+`:

[subs="+attributes",console]
----
kubectl apply -f - <<EOF
apiVersion: v1
kind: Namespace
metadata:
  name: team-alpha-staging
  labels:
    environment: staging
EOF
----

A política permite a criação de um recurso de implantação, sem rótulos, no espaço de nomes `+team-alpha-staging+`:

[subs="+attributes",console]
----
kubectl apply -f - <<EOF
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment-without-labels
  namespace: team-alpha-staging
spec:
  selector:
    matchLabels:
      app: nginx
  replicas: 0
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:latest
        ports:
        - containerPort: 80
EOF
----

Como esperado, esse recurso foi criado com sucesso.

== Explorando a interface do usuário do Jaeger

Você pode ver os eventos de rastreamento enviados pela instância do PolicyServer para o Jaeger, pois há um novo serviço `+kubewarden-policy-server+` listado na interface do usuário:

image::jaeger-ui-home.png[Painel de controle do Jaeger]

O coletor Jaeger está recebendo os rastreamentos gerados pelo seu PolicyServer.

Se você instalou o Traefik no guia anterior, exponha a interface do usuário do Jaeger Query com este `+Ingress+`:

[,yaml]
----
kubectl apply -f - <<EOF
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-open-telemetry-query
  namespace: jaeger
spec:
  ingressClassName: traefik
  rules:
    - http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: my-open-telemetry-query
                port:
                  number: 16686
EOF
----

Para acessar a interface do usuário do Jaeger com o Traefik, execute:

[,console]
----
kubectl -n traefik port-forward service/traefik 8080:80
----

Em seguida, acesse `+\http://localhost:8080+`.

Se preferir ignorar o Traefik, acesse o Jaeger diretamente:

[,console]
----
kubectl -n jaeger port-forward service/my-open-telemetry-query 16686
----

Em seguida, acesse `+\http://localhost:16686+`.
