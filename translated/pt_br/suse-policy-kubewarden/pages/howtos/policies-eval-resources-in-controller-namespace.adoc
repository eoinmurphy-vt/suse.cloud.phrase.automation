= Permitir avaliações no namespace do Controlador de Admissão
:description: Como configurar o Controlador de Admissão para permitir a avaliação da política em seu próprio namespace
:keywords: ["policies", "ClusterAdmissionPolicies", "ClusterAdmissionPoliciesGroup", "configuration", "namespaces"]
:doc-persona: ["kubewarden-operator"]
:doc-type: ["howto"]
:doc-topic: ["operator-manual", "policies"]

Por padrão, o Controlador de Admissão não permite que as políticas avaliem recursos em seu próprio namespace. Isso evita configurações incorretas que poderiam interromper a instalação do {short-project-name}.

Essa medida de segurança é implementada em duas camadas. A primeira camada é controlada pelo sinalizador `+--always-accept-admission-reviews-on-deployments-namespace+` CLI.
Esse sinalizador, que é ativado por padrão nos gráficos do Helm, faz com que o controlador configure a variável de ambiente `+KUBEWARDEN_ALWAYS_ACCEPT_ADMISSION_REVIEWS_ON_NAMESPACE+`nas implantações do PolicyServer. Essa variável informa ao servidor de políticas para sempre aceitar revisões de admissão do namespace definido na variável. A segunda camada é o próprio controlador de admissão, que, por padrão, adiciona um seletor de namespace às políticas de todo o cluster para ignorar o namespace em que o controlador é executado.

No entanto, em alguns cenários, os administradores de cluster podem querer que as políticas avaliem os recursos nesse namespace. Esta página explica como fazer isso desativando as duas camadas de proteção.

== Remova a variável de ambiente de proteção PolicyServer

A primeira etapa é configurar o controlador para não definir a variável de ambiente`+KUBEWARDEN_ALWAYS_ACCEPT_ADMISSION_REVIEWS_ON_NAMESPACE+` nas implantações do PolicyServer.

Para fazer isso, atualize sua instalação do {short-project-name} e defina o valor`+alwaysAcceptAdmissionReviewsOnDeploymentsNamespace+` do gráfico do`+kubewarden-controller+` Helm para `+false+`:

[source,console]
----
helm upgrade -n kubewarden
  --set "alwaysAcceptAdmissionReviewsOnDeploymentsNamespace=false"
  kubewarden-controller kubewarden/kubewarden-controller
----

Essa alteração permite que os servidores de políticas avaliem recursos de qualquer namespace.
Uma vez feito isso, as políticas podem ser configuradas para avaliar as revisões de admissão do espaço de nomes {short-project-name}.

== Configurar políticas para avaliar recursos no namespace do Controlador de Admissão

Para permitir que uma política avalie os recursos criados no namespace do Controlador de Admissão, implante políticas em todo o cluster com o campo `+allowInsideAdmissionControllerNamespace+`na especificação definida como `+true+`:

[NOTE]
.E as políticas de namespaced?
====
Esse campo de especificação está disponível somente em políticas de todo o cluster.
Como as políticas namespaced avaliam os recursos no mesmo namespace em que são implantados, não faz sentido adicionar esse campo aos seus CRDs.
Além disso, o namespace do controlador de admissão deve ser gerenciado pelos operadores do cluster, que podem decidir se uma política deve ser implantada nesse local.
====

[source,yaml]
----
apiVersion: policies.kubewarden.io/v1
kind: ClusterAdmissionPolicy
metadata:
  annotations:
    io.kubewarden.policy.category: PSP
    io.kubewarden.policy.severity: medium
  name: pod-privileged-policy
spec:
  module: registry://ghcr.io/kubewarden/policies/pod-privileged:v1.0.10
  settings: {}
  rules:
    - apiGroups:
        - ""
      apiVersions:
        - v1
      resources:
        - pods
      operations:
        - CREATE
  mutating: false
  allowInsideAdmissionControllerNamespace: true
----

Essa opção de configuração, que é `+false+` por padrão, faz com que o controlador ignore a adição do seletor de namespace que exclui o namespace do Controlador de Admissão, permitindo que a política avalie os recursos de todos os namespaces, incluindo esse namespace.

[NOTE]
.E quanto aos meus seletores de namespace personalizados?
====
Os CRDs de políticas também permitem que os usuários definam seus próprios seletores de namespace. Esses seletores não são alterados. Isso significa que, se um usuário definir um seletor que ignora o namespace {short-project-name}, a política ainda ignorará esse namespace conforme esperado.
====
