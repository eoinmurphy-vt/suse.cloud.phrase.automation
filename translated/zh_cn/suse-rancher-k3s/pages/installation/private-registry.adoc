= 私人注册表配置
:page-languages: [EN, JA, KO, ZH]
:revdate: 2025-12-30
:page-revdate: {revdate}

Containerd 可配置为连接到私有注册表，并根据 kubelet 的需要使用注册表提取图像。

启动时，K3s 将检查`+/etc/rancher/k3s/registries.yaml+` 是否存在。如果是，则在生成 containerd 配置时使用该文件中包含的注册表配置。

* 如果要将私有注册表用作公共注册表（如 docker.io）的镜像，则需要在每个要使用镜像的节点上配置`+registries.yaml+` 。
* 如果您的私有注册表需要身份验证、使用自定义 TLS 证书或不使用 TLS，则需要在每个将从注册表中提取映像的节点上配置`+registries.yaml+` 。

请注意，服务器节点默认是可调度的。如果您尚未对服务器节点进行污点处理，并且将在这些节点上运行工作负载，请确保在每台服务器上也创建`+registries.yaml+` 文件。

[#_default_endpoint_fallback]
== 默认端点回退

Containerd 对所有注册表都有一个隐含的 "默认端点"。
即使`+registries.yaml+` 中列出了该注册表的其他端点，默认端点始终会作为最后手段进行尝试。
重写不会应用于针对默认端点的拉取。
例如，当拉取`+registry.example.com:5000/rancher/mirrored-pause:3.6+` 时，containerd 将使用默认端点`+https://registry.example.com:5000/v2+` 。

* `+docker.io+` 的默认端点是`+\https://index.docker.io/v2+` 。
* 所有其他注册表的默认端点是`+\https://<REGISTRY>/v2+` ，其中`+<REGISTRY>+` 是注册表主机名和可选端口。

为了被识别为注册表，图像名称的第一个组成部分必须至少包含一个句号或冒号。
由于历史原因，名称中未指明注册表的图像被默认为来自`+docker.io+` 。

[IMPORTANT]
.版本闸门
====
`+--disable-default-registry-endpoint+` 选项在 2024 年 1 月发布的以下版本中可用：v1.26.13+k3s1、v1.27.10+k3s1、v1.28.6+k3s1、v1.29.1+k3s1
====


节点可以使用`+--disable-default-registry-endpoint+` 选项启动。
设置此项后，containerd 将不会回退到默认注册表端点，而只会从配置的镜像端点以及分布式注册表（如果已启用）中提取数据。

如果您的集群处于真正的 air-gapped 环境中，上游注册表不可用，或者您只想让某些节点从上游注册表中提取数据，则可能需要这样做。

禁用默认注册表端点仅适用于通过`+registries.yaml+` 配置的注册表。
如果注册表未通过`+registries.yaml+` 中的镜像条目进行显式配置，则仍将使用默认的回退行为。

== 注册表配置文件

文件由两个顶层键值和每个注册表的子键值组成：

[,yaml]
----
mirrors:
  <REGISTRY>:
    endpoint:
      - https://<REGISTRY>/v2
configs:
  <REGISTRY>:
    auth:
      username: <BASIC AUTH USERNAME>
      password: <BASIC AUTH PASSWORD>
      token: <BEARER TOKEN>
    tls:
      ca_file: <PATH TO SERVER CA>
      cert_file: <PATH TO CLIENT CERT>
      key_file: <PATH TO CLIENT KEY>
      insecure_skip_verify: <SKIP TLS CERT VERIFICATION BOOLEAN>
----

=== 镜子

例如，镜像部分定义了注册表的名称和端点：

----
mirrors:
  registry.example.com:
    endpoint:
      - "https://registry.example.com:5000"
----

每个镜像必须有一个名称和一组端点。从注册表中提取映像时，containerd 会尝试这些端点 URL 和默认端点，并使用第一个能正常工作的端点。

==== 重定向

如果私有注册表被用作另一个注册表的镜像，例如在配置https://docs.docker.com/registry/recipes/mirror/[pull through cache] 时，图像拉取会透明地重定向到列出的端点。原始注册表名称通过`+ns+` 查询参数传递给镜像端点。

例如，如果您为`+docker.io+` 配置了一个镜像：

[,yaml]
----
mirrors:
  docker.io:
    endpoint:
      - "https://registry.example.com:5000"
----

然后拉取`+docker.io/rancher/mirrored-pause:3.6+` ，就会以透明的方式拉取图像，即`+registry.example.com:5000/rancher/mirrored-pause:3.6+` 。

==== 改写

每个镜像都可以有一组重写，当从镜像中提取图像时，这些重写使用正则表达式来匹配和转换图像名称。
如果私有注册表中的组织/项目结构与镜像的注册表不同，这将非常有用。
重写只匹配和转换图像名称，而不是标签。

例如，以下配置将以透明方式提取图像`+docker.io/rancher/mirrored-pause:3.6+` ，即`+registry.example.com:5000/mirrorproject/rancher-images/mirrored-pause:3.6+` ：

[,yaml]
----
mirrors:
  docker.io:
    endpoint:
      - "https://registry.example.com:5000"
    rewrite:
      "^rancher/(.*)": "mirrorproject/rancher-images/$1"
----

[NOTE]
.版本闸门
====
从 2024 年 1 月发布的以下版本起，重写不再应用于xref:#_default_endpoint_fallback[默认端点]：v1.26.13+k3s1、v1.27.10+k3s1、v1.28.6+k3s1、v1.29.1+k3s1。
在这些版本发布之前，重写也应用于默认端点，这将阻止 K3 从上游注册表中提取镜像，如果镜像无法从镜像端点中提取，并且镜像在上游中的修改名称下不可用。
====

如果你想在直接从注册表中提取数据时应用重写（当注册表没有被用作不同上游注册表的镜像时），你必须提供一个与默认端点不匹配的镜像端点。
`+registries.yaml+` 中与默认端点相匹配的镜像端点会被忽略；如果没有禁用回退，默认端点总是最后尝试，不会重写。

例如，如果注册表位于`+https://registry.example.com/+` ，并希望在明确提取`+registry.example.com/rancher/mirrored-pause:3.6+` 时应用重写，则可以添加一个列出端口的镜像端点。
因为镜像端点与默认端点不匹配，所以 **`+"https://registry.example.com:443/v2" != "https://registry.example.com/v2"+`**- 端点被接受为镜像并应用重写，尽管它实际上与默认端点相同。

[,yaml]
----
mirrors:
 registry.example.com
   endpoint:
     - "https://registry.example.com:443"
   rewrite:
     "^rancher/(.*)": "mirrorproject/rancher-images/$1"
----

请注意，在使用镜像和重写时，图像仍将以原始名称存储。
例如，`+crictl image ls+` 会显示`+docker.io/rancher/mirrored-pause:3.6+` 在节点上可用，即使镜像是从名称不同的镜像中提取的。

=== 配置

`+configs+` 部分定义了每个镜像的 TLS 和凭据配置。您可以为每个镜像定义`+auth+` 和/或`+tls+` 。

`+tls+` 部分包括

|===
| 指令 | 说明

| `+cert_file+`
| 用于与注册表进行身份验证的客户证书路径

| `+key_file+`
| 用于与注册表进行身份验证的客户密钥路径

| `+ca_file+`
| 定义用于验证注册表服务器证书文件的 CA 证书路径

| `+insecure_skip_verify+`
| 布尔值，用于定义是否跳过注册表的 TLS 验证
|===

`+auth+` 部分包括用户名/密码或认证令牌：

|===
| 指令 | 说明

| `+username+`
| 私人注册表基本授权的用户名

| `+password+`
| 私人注册表基本授权的用户密码

| `+auth+`
| 私人注册表基本认证的认证令牌
|===

以下是在不同模式下使用私人注册表的基本示例：

=== 通配符支持

[IMPORTANT]
.版本闸门
====
通配符支持从 2024 年 3 月发布的以下版本开始提供：v1.26.15+k3s1、v1.27.12+k3s1、v1.28.8+k3s1、v1.29.3+k3s1
====


`+"*"+` 通配符条目可用于`+mirrors+` 和`+configs+` 部分，为所有注册表提供默认配置。
只有在注册表没有特定条目的情况下，才会使用默认配置。注意星号必须加引号。

在下面的示例中，所有注册表都将使用本地注册表镜像。除`+docker.io+` 外，所有注册表都将禁用 TLS 验证。

[,yaml]
----
mirrors:
  "*":
    endpoint:
      - "https://registry.example.com:5000"
configs:
  "docker.io":
  "*":
    tls:
      insecure_skip_verify: true
----

=== 使用 TLS

下面的示例显示了使用 TLS 时如何在每个节点上配置`+/etc/rancher/k3s/registries.yaml+` 。

[tabs]
======
通过身份验证::
+
[,yaml]
----
mirrors:
  docker.io:
    endpoint:
      - "https://registry.example.com:5000"
configs:
  "registry.example.com:5000":
    auth:
      username: xxxxxx # this is the registry username
      password: xxxxxx # this is the registry password
    tls:
      cert_file: # path to the cert file used in the registry
      key_file:  # path to the key file used in the registry
      ca_file:   # path to the ca file used in the registry
----

无认证::
+
[,yaml]
----
mirrors:
  docker.io:
    endpoint:
      - "https://registry.example.com:5000"
configs:
  "registry.example.com:5000":
    tls:
      cert_file: # path to the cert file used in the registry
      key_file:  # path to the key file used in the registry
      ca_file:   # path to the ca file used in the registry
----
======

=== 无 TLS

下面的示例显示了_不_使用 TLS 时如何在每个节点上配置`+/etc/rancher/k3s/registries.yaml+` 。

[tabs]
======
通过身份验证::
+
[,yaml]
----
mirrors:
  docker.io:
    endpoint:
      - "http://registry.example.com:5000"
configs:
  "registry.example.com:5000":
    auth:
      username: xxxxxx # this is the registry username
      password: xxxxxx # this is the registry password
----

无认证::
+
[,yaml]
----
mirrors:
  docker.io:
    endpoint:
      - "http://registry.example.com:5000"
----
======

____
如果没有 TLS 通信，则需要为端点指定`+http://+` ，否则会默认为 https。
____

为了使注册表更改生效，需要在每个节点上重新启动 K3。

== 图像拉动故障排除

当 Kubernetes 在拉取映像时遇到问题时，kubelet 显示的错误可能只反映了针对默认端点进行的拉取尝试所返回的终端错误，从而让人觉得配置的端点未被使用。

查看`+/var/lib/rancher/k3s/agent/containerd/containerd.log+` 节点上的 containerd 日志，了解故障根本原因的详细信息。请注意，您必须查看 pod 计划所在节点上的日志。您可以通过发布`+kubectl get pod -o wide -n NAMESPACE POD+` 并查看 NODE 列来检查 pod 计划在哪个节点上运行。

== 将图像添加到私人注册表

要将映像镜像到私有注册表，需要主机配备 Docker 或其他第三方工具，以便拉取和推送映像。 +
下面的步骤假定你有一台装有 dockerd 和 docker CLI 工具的主机，并且可以访问 docker.io 和你的私有注册表。

. 从 GitHub 获取所使用版本的`+k3s-images.txt+` 文件。
. 从 docker.io 获取 k3s-images.txt 文件中列出的每个 K3s 映像。 +
例如 `+docker pull docker.io/rancher/mirrored-pause:3.6+`
. 将图像重新标记到私人注册表。 +
例如 `+docker tag docker.io/rancher/mirrored-pause:3.6 registry.example.com:5000/rancher/mirrored-pause:3.6+`
. 将图像推送到私人注册表。 +
例如 `+docker push registry.example.com:5000/rancher/mirrored-pause:3.6+`
