= 开放式遥测的故障排除
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: SUSE 可观察性

有很多配置选项，但更重要的是，每个（Kubernetes）环境都略有不同。要找出问题所在，最快捷的方法是选择一个可提供遥测数据的吊舱：

. 检查 pod 的日志开头，当仪器启动失败时，SDK 会记录警告或错误日志
. 还可检查日志，查看与向采集器发送数据有关的任何错误
. 检查收集器 pod 的日志，查看是否存在配置或初始化错误，这些错误会在 pod 启动后立即记录下来
. 还可检查收集器日志，查看是否存在与向 SUSE Observability 发送数据有关的错误

日志中的错误通常能很好地说明问题。我们列出了导致部分或全部仪器应用程序无法使用开放式遥测数据的最常见原因。如果这里没有列出问题，您也可以查看https://opentelemetry.io/docs/languages/[语言特定的 SDK 文档]或 Open Telemetry 的https://opentelemetry.io/docs/collector/troubleshooting/[收集器文档]。

== 使用相同的 Kubernetes 集群名称

确保在同一集群中使用相同的 Kubernetes 集群名称：

* 安装开放式遥测采集器
* 安装 SUSE Observability 代理
* 安装 Kubernetes StackPack

当同一集群使用不同名称时，SUSE Observability 将无法将 Open Telemetry 的数据与 SUSE Observability 代理的数据相匹配，跟踪视角将保持为空。

== 收集器无法向 SUSE Observability 发送数据

=== SUSE Observability 的 OTLP 端点和 API 密钥配置错误

如果出现连接错误，可能是 OTLP 端点不正确。如果出现身份验证/授权错误（状态代码 401 和 403），很可能是 API 密钥不再有效。检查配置的 OTLP 端点是否为 SUSE Observability 的 URL，前缀为`+otlp-+` ，后缀为`+:443+` 。例如，`+play.stackstate.com+` 的 OTLP 端点是`+otlp-play.stackstate.com:443+` 。

为确保 api 密钥配置正确，请检查以下内容：

. 该秘密包含有效的 API 密钥（在 SUSE Observability 中进行验证）
. 秘密会被用作 pod 上的环境变量
. `+bearertokenauth+` 扩展使用了正确的方案和`+API_KEY+` 环境变量的值
. `+bearertokenauth+` 扩展名由`+otlp/suse-observability+` 输出程序使用

=== 某些代理和防火墙不能很好地与 gRPC 配合使用

如果收集器需要通过代理或防火墙向 SUSE Observability 发送数据，它们可能会完全阻断流量，或丢弃 gRPC 消息的某些部分，或意外地完全丢弃长期存在的 gRPC 连接。最简单的解决方法是从 gRPC 转为使用 HTTP，将`+otlp/suse-observability+` 输出程序配置**和管道部分中对它的引用**替换为`+otlphttp/suse-observability+` 输出程序。

`+<otlp-http-suse-observability-endpoint>+` 与`+<otlp-suse-observability-endpoint>+` 类似，但前缀不是`+otlp-+` ，而是`+otlp-http-+` ，例如`+otlp-http-play.stackstate.com+` 。更多详情，请参阅xref:/setup/otel/collector.adoc#_exporters[采集器配置]。

== 仪器应用程序无法向采集器发送数据

=== URL 不正确或流量受阻

如果 SDK 记录了无法解析采集器 DNS 名称的错误，则可能是配置的采集器 URL 不正确。在 Kubernetes 中，应用程序通常部署在与收集器不同的命名空间中。这意味着 SDK 需要配置收集器服务的完全合格域名：`+http://<service-name>.<namespace>.svc.cluster.local:4317+` 。在xref:/setup/otel/collector.adoc[采集器安装步骤]中，这是`+http://opentelemetry-collector.open-telemetry.svc.cluster.local:4317+` ，但如果您使用的是不同的命名空间或采集器发布名称，这可能与您的情况不同。

如果 SDK 记录了网络连接超时，可能是采集器配置错误或使用了<<_the_language_sdk_uses_the_wrong_port,错误的端口>>。但也有可能是 Kubernetes 网络策略阻止了从应用程序到收集器的网络流量。最好与 Kubernetes 管理员核实。网络策略至少应允许所有应用程序通过配置的端口（4317 和/或 4318）向收集器传输 TCP 流量。

=== 语言 SDK 不支持 gRPC

并非所有语言 SDK 都支持 gRPC。如果不支持通过 gRPC 的 OTLP，最好改用通过 HTTP 的 OTLP。xref:/setup/otel/instrumentation/sdk-exporter-config.adoc#_grpc_vs_http[SDK 输出程序配置]描述了如何进行此切换。

=== 语言 SDK 使用了错误的端口

使用错误的端口通常表现为连接错误，但也可能表现为网络连接意外关闭。确保 SDK 输出程序在发送数据时使用正确的端口。请参阅xref:/setup/otel/instrumentation/sdk-exporter-config.adoc#_grpc_vs_http[SDK 输出程序配置]。

== 启用 hostNetwork 的 Kubernetes pod

Open Telemetry 收集器用 Kubernetes 元数据丰富了遥测数据。根据配置方式，所有无法丰富的遥测数据都会被丢弃。但是，收集器无法对运行时自动设置了`+hostNetwork: true+` 的 pod 进行浓缩。这是不可能的，因为 pod 识别使用的是 pod 的 IP 地址，而使用主机网络的 pod 使用的是主机的 IP 地址。

为了帮助收集器识别 pod，我们可以通过指示 SDK 直接在元数据中添加`+k8s.pod.uid+` 属性。为此，请修改 pod 规范，并将以下环境变量添加到仪器应用容器中：

[,yaml]
----
env:
  - name: POD_UID
    valueFrom:
      fieldRef:
        apiVersion: v1
        fieldPath: metadata.uid
  - name: OTEL_RESOURCE_ATTRIBUTES
    value: k8s.pod.uid=$(POD_UID)
----

如果`+OTEL_RESOURCE_ATTRIBUTES+` env var 已经设置，只需添加`+k8s.pod.uid+` ，用逗号作为分隔符。值是以逗号分隔的列表。

== 谷歌 Kubernetes 引擎上的 Node.js 应用程序

仅在 GKE 上，Node.js SDK 希望通过`+NAMESPACE+` 环境变量设置 Kubernetes 命名空间。如果未设置，则仍会添加`+k8s.namespace.name+` 属性，但值为空。 这会阻止 Kubernetes 属性处理器插入正确的命名空间名称。在解决这个问题之前，一种解决方法是更新 pod 规格，并将此环境变量添加到仪器容器中：

[,yaml]
----
env:
  - name: NAMESPACE
    valueFrom:
      fieldRef:
        apiVersion: v1
        fieldPath: metadata.namespace
----

== Node.js 应用程序没有可用度量指标

通过环境变量配置的 Node.js 自动仪器仅支持跟踪。至少在https://github.com/open-telemetry/opentelemetry-js/issues/4551[Open Telemetry 问题]解决之前是这样。要启用自动仪器的度量指标，需要修改代码。请按照https://opentelemetry.io/docs/languages/js/exporters/#_usage_with_nodejs[Open Telemetry 文档]中的说明进行更改。

== 无法添加 Kubernetes 属性

安装收集器时，会在 Kubernetes 中创建一个群集角色和群集角色绑定，允许收集器从 Kubernetes 资源读取元数据。如果这样做失败或它们被移除，收集器将无法再查询 Kubernetes API。这会在收集器日志中显示为错误，错误包括无法检索元数据的资源类型。

要解决这个问题，请使用 Helm 图表重新安装收集器，并确保您拥有创建群集角色和群集角色绑定所需的权限。或者，请群集管理员以所需权限进行收集器安装。
