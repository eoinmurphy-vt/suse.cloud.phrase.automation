= 开始使用 Kubernetes 上的 Open Telemetry 操作员
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: SUSE 可观察性

下面是我们要为需要监控的应用程序创建的设置：

* 在群集 A 中运行的受监控应用程序/工作负载，由操作员自动监测
* A 组的开放式遥测操作员
* 操作员创建的收集器
* 在集群 B 中运行的 SUSE Observability 或 SUSE Cloud Observability

image::otel/open-telemetry-kubernetes-operator.png[带有开放式遥测操作员自动仪表的集装箱仪表]

== 安装操作器

与普通 Kubernetes 设置相比，Open Telemetry 操作员提供了一些额外功能：

* 它可以为支持的语言（Java、.NET、Python、Golang、Node.js）自动生成应用程序 pod，而无需修改应用程序或 docker 镜像。
* 它可以作为 Prometheus 操作员的替代品，开始根据服务和 pod 监视器搜索 Prometheus 输出端点

=== 创建服务令牌

创建服务令牌有两种方法：

* **SUSE Observability UI**- 点击屏幕左上角打开主菜单，进入`+StackPacks+` >`+Open Telemetry+` 。 如果之前没有这样做，请单击`+INSTALL+` 按钮。 单击`+CREATE NEW SERVICE TOKEN+` 按钮，将数值复制到剪贴板上。
* **SUSE Observability CLI**- 请参阅xref:/use/security/k8s-service-tokens.adoc#_manage_service_tokens[管理服务令牌]

在以下说明中提到`+<SERVICE_TOKEN>+` 时，必须使用服务令牌值。

=== 为服务令牌创建命名空间和密文

我们将在`+open-telemetry+` 命名空间中安装，并使用服务令牌：

[,bash]
----
kubectl create namespace open-telemetry
kubectl create secret generic open-telemetry-collector \
    --namespace open-telemetry \
    --from-literal=API_KEY='<SERVICE_TOKEN>'
----

=== 配置和安装操作员

操作员安装了 Helm 海图，因此首先要配置海图存储库。

[,bash]
----
helm repo add open-telemetry https://open-telemetry.github.io/opentelemetry-helm-charts
----

让我们创建一个`+otel-operator.yaml+` 文件来配置操作员：

.otel-operator.yaml
[,yaml]
----
# Add image pull secret for private registries
imagePullSecrets: []
manager:
  image:
    # Uses chart.appVersion for the tag
    repository: ghcr.io/open-telemetry/opentelemetry-operator/opentelemetry-operator
  collectorImage:
    # find the latest collector releases at https://github.com/open-telemetry/opentelemetry-collector-releases/releases
    repository: otel/opentelemetry-collector-k8s
    tag: 0.123.0
  targetAllocatorImage:
    repository: ""
    tag: ""
  # Only needed when overriding the image repository, make sure to always specify both the image and tag:
  autoInstrumentationImage:
    java:
      repository: ""
      tag: ""
    nodejs:
      repository: ""
      tag: ""
    python:
      repository: ""
      tag: ""
    dotnet:
      repository: ""
      tag: ""
    # The Go instrumentation support in the operator is disabled by default.
    # To enable it, use the operator.autoinstrumentation.go feature gate.
    go:
      repository: ""
      tag: ""

admissionWebhooks:
  # A production setup should use certManager to generate the certificate, without certmanager the certificate will be generated during the Helm install
  certManager:
    enabled: false
  # The operator has validation and mutation hooks that need a certificate, with this we generate that automatically
  autoGenerateCert:
    enabled: true
----


现在使用配置文件安装采集器：

[,bash]
----
helm upgrade --install opentelemetry-operator open-telemetry/opentelemetry-operator \
  --namespace open-telemetry \
  --values otel-operator.yaml
----

这只是安装操作器。继续安装采集器并启用自动仪器。

== 开放式遥测采集器

操作员通过一种名为`+OpenTelemetryCollector+` 的 Kubernetes 自定义资源来管理一个或多个收集器部署。我们将使用与xref:/setup/otel/getting-started/getting-started-k8s.adoc[Kubernetes 入门指南]中相同的配置创建一个。

它使用的是指南前面创建的秘密。确保将`+<otlp-suse-observability-endpoint:port>+` 替换为您的 OTLP 端点（请参阅xref:/setup/otel/otlp-apis.adoc[OTLP API]了解您的端点），并插入 Kubernetes 集群的名称，而不是`+<your-cluster-name>+` ：

.collector.yaml
[,yaml]
----
apiVersion: opentelemetry.io/v1beta1
kind: OpenTelemetryCollector
metadata:
  name: otel-collector
spec:
  mode: deployment
  envFrom:
  - secretRef:
      name: open-telemetry-collector
  # optional service-account for pulling the collector image from a private registries
  # serviceAccount: otel-collector
  config:
    receivers:
      otlp:
        protocols:
          grpc:
            endpoint: 0.0.0.0:4317
          http:
            endpoint: 0.0.0.0:4318
      # Scrape the collectors own metrics
      prometheus:
        config:
          scrape_configs:
          - job_name: opentelemetry-collector
            scrape_interval: 10s
            static_configs:
            - targets:
              - ${env:MY_POD_IP}:8888
    extensions:
      health_check:
        endpoint: ${env:MY_POD_IP}:13133
      # Use the API key from the env for authentication
      bearertokenauth:
        scheme: SUSEObservability
        token: "${env:API_KEY}"
    exporters:
      debug: {}
      nop: {}
      otlp/suse-observability:
        auth:
          authenticator: bearertokenauth
        # Put in your own otlp endpoint, for example otlp-suse-observability.my.company.com:443
        endpoint: <otlp-suse-observability-endpoint:port>
        compression: snappy
    processors:
      memory_limiter:
        check_interval: 5s
        limit_percentage: 80
        spike_limit_percentage: 25
      batch: {}
      resource:
        attributes:
        - key: k8s.cluster.name
          action: upsert
          # Insert your own cluster name
          value: <your-cluster-name>
        - key: service.instance.id
          from_attribute: k8s.pod.uid
          action: insert
          # Use the k8s namespace also as the open telemetry namespace
        - key: service.namespace
          from_attribute: k8s.namespace.name
          action: insert
    connectors:
      # Generate metrics for spans
      spanmetrics:
        metrics_expiration: 5m
        namespace: otel_span
    service:
      extensions: [ health_check,  bearertokenauth ]
      pipelines:
        traces:
          receivers: [otlp]
          processors: [memory_limiter, resource, batch]
          exporters: [debug, spanmetrics, otlp/suse-observability]
        metrics:
          receivers: [otlp, spanmetrics, prometheus]
          processors: [memory_limiter, resource, batch]
          exporters: [debug, otlp/suse-observability]
        logs:
          receivers: [otlp]
          processors: []
          exporters: [nop]
      telemetry:
        metrics:
          address: ${env:MY_POD_IP}:8888
----


[CAUTION]
====
如果在 Kubernetes 堆栈包中也使用*SUSE Observability 代理，请使用与安装*SUSE Observability 代理时*相同的群集名称*。使用不同的群集名称将导致 Kubernetes 组件的跟踪视角为空，并将在整体上大大增加 SUSE Observability 和用户关联信息的难度。
====


现在在`+open-telemetry+` 命名空间中应用`+collector.yaml+` 来部署收集器：

[,bash]
----
kubectl apply --namespace open-telemetry -f collector.yaml
----

采集器提供更多配置接收器、处理器和出口器，详情请查看我们的xref:/setup/otel/collector.adoc[采集器页面]。在生产使用中，经常会产生大量的跨度，因此需要开始设置xref:/setup/otel/sampling.adoc[采样]。

== 自动仪表

=== 配置自动仪器

现在，我们需要告诉操作员如何使用另一种自定义资源为不同语言配置自动仪表，资源类型为`+Instrumentation+` 。它主要用于配置刚刚部署的收集器，将其作为仪器应用程序的遥测端点。

它可以定义在一个地方，供群集中的所有 pod 使用，但也可以在每个命名空间中使用不同的`+Instrumentation+` 。我们将在这里做前者。请注意，如果您使用了不同的命名空间或不同的otel 收集器名称，则需要相应更新该文件中的端点。

创建`+instrumentation.yaml+` ：

.instrumentation.yaml
[,yaml]
----
apiVersion: opentelemetry.io/v1alpha1
kind: Instrumentation
metadata:
  name: otel-instrumentation
spec:
  exporter:
    # default endpoint for the instrumentation
    endpoint: http://otel-collector-collector.open-telemetry.svc.cluster.local:4317
  propagators:
    - tracecontext
    - baggage
  defaults:
    # To use the standard app.kubernetes.io/ labels for the service name, version and namespace:
    useLabelsForResourceAttributes: true
  python:
    env:
      # Python autoinstrumentation uses http/proto by default, so data must be sent to 4318 instead of 4317.
      - name: OTEL_EXPORTER_OTLP_ENDPOINT
        value: http://otel-collector-collector.open-telemetry.svc.cluster.local:4318
  dotnet:
    env:
      # Dotnet autoinstrumentation uses http/proto by default, so data must be sent to 4318 instead of 4317.
      - name: OTEL_EXPORTER_OTLP_ENDPOINT
        value: http://otel-collector-collector.open-telemetry.svc.cluster.local:4318
  go:
    env:
      # Go autoinstrumentation uses http/proto by default, so data must be sent to 4318 instead of 4317.
      - name: OTEL_EXPORTER_OTLP_ENDPOINT
        value: http://otel-collector-collector.open-telemetry.svc.cluster.local:4318
----


现在也在`+open-telemetry+` 命名空间中应用`+instrumentation.yaml+` ：

[,bash]
----
kubectl apply --namespace open-telemetry -f instrumentation.yaml
----

=== 启用 pod 的自动仪表功能

要指示操作员自动测量应用行动 pod，我们需要为 pod 添加注释：

* Java： `+instrumentation.opentelemetry.io/inject-java: open-telemetry/otel-instrumentation+`
* NodeJS `+instrumentation.opentelemetry.io/inject-nodejs: open-telemetry/otel-instrumentation+`
* Python： `+instrumentation.opentelemetry.io/inject-python: open-telemetry/otel-instrumentation+`
* 去吧 `+instrumentation.opentelemetry.io/inject-go: open-telemetry/otel-instrumentation+`

请注意，注解的值指的是我们创建的`+Instrumentation+` 资源的命名空间和名称。其他选项包括

* "true" - 从命名空间注入自定义资源并`+Instrumentation+` 。
* "my-instrumentation" - 当前命名空间中`+Instrumentation+` 自定义资源的名称。
* "my-other-namespace/my-instrumentation" - 另一个命名空间中`+Instrumentation+` 自定义资源的命名空间和名称。
* "false"\--不注入

当创建一个带有注释的 pod 时，操作员会通过突变钩子修改 pod：

* 它添加了一个 init 容器，提供自动仪器库
* 它修改 pod 的第一个容器，以便在启动时加载仪器，并添加环境变量来配置仪器

如果您需要自定义哪些容器应使用仪器，请使用https://github.com/open-telemetry/opentelemetry-operator?tab=readme-ov-file#_multi_container_pods_with_multiple_instrumentations[操作员文档]。

[CAUTION]
====
转到自动仪器需要提升权限。这些权限由操作员自动设置：

[,yaml]
----
securityContext:
  privileged: true
  runAsUser: 0
----

====


== 查看结果

进入 SUSE Observability，确保已安装 Open Telemetry Stackpack（通过主菜单 \-> Stackpacks）。

过了一会儿，如果您的 pod 有了一些流量，您就可以在 Open Telemetry \-> services（打开遥测）和 service instances（服务实例）概览中的服务名称下找到它们。跟踪将出现在服务和服务实例组件的xref:/use/views/k8s-traces-perspective.adoc[跟踪] xref:/use/traces/k8sTs-explore-traces.adoc[资源管理器]和xref:/use/views/k8s-traces-perspective.adoc[跟踪视角]中。跨度指标和特定语言指标（如有）将出现在组件的xref:/use/views/k8s-metrics-perspective.adoc[指标视角]中。

如果您还安装了 Kubernetes stackpack，则仪器化 pod 也会在xref:/use/views/k8s-traces-perspective.adoc[跟踪视角]中提供跟踪信息。

== Rancher RBAC

要使xref:/setup/security/rbac/rbac_rancher.adoc[Rancher RBAC]正常工作，遥测数据需要具备以下资源属性：

* `+k8s.cluster.name+` - Kubernetes 堆栈包使用的*集群*名称
* `+k8s.namespace.name+` - 由 Rancher*项目管理*的*命名空间*

Kubernetes 操作员会将这些属性默认注入发送的任何遥测数据中。
