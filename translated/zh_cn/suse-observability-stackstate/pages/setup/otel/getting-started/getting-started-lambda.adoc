为一个或多个 AWS Lambda 函数设置监控：

* 受监控的 AWS Lambda 函数（使用 Open Telemetry 进行检测
* 开放式遥测采集器
* SUSE Observability 或 SUSE Cloud Observability

image::otel/open-telemetry-collector-lambda.png[使用在 Kubernetes 中运行的 Open Telemetry 收集器和 Open Telemetry 的 AWS Lambda Instrumentation]

== 开放式遥测采集器

[NOTE]
====
对于生产设置，强烈建议安装收集器，因为它可以让您的服务快速卸载数据，而且收集器还可以处理重试、批处理、加密甚至敏感数据过滤等额外事务。
====


安装 OTel（Open Telemetry）采集器，在本示例中，我们使用 Kubernetes 集群在靠近 Lambda 功能的地方运行它。也可以使用安装在虚拟机上的收集器进行类似设置。这里使用的配置仅作为一个安全代理，从 Lambda 功能中快速卸载数据，并在受信任的网络基础设施中运行。

=== 创建服务令牌

创建服务令牌有两种方法：

* **SUSE Observability UI**- 点击屏幕左上角打开主菜单，进入`+StackPacks+` >`+Open Telemetry+` 。 如果之前没有这样做，请单击`+INSTALL+` 按钮。单击`+CREATE NEW SERVICE TOKEN+` 按钮，将数值复制到剪贴板上。
* **SUSE Observability CLI**- 请参阅xref:/use/security/k8s-service-tokens.adoc#_manage_service_tokens[管理服务令牌]

在以下说明中提到`+<SERVICE_TOKEN>+` 时，必须使用服务令牌值。

=== 为服务令牌创建命名空间和密文

安装在`+open-telemetry+` 命名空间并使用服务令牌：

[,bash]
----
kubectl create namespace open-telemetry
kubectl create secret generic open-telemetry-collector \
    --namespace open-telemetry \
    --from-literal=API_KEY='<SERVICE_TOKEN>'
----

=== 配置和安装采集器

我们使用开放遥测项目提供的 Helm 图表安装采集器。确保已配置 Open Telemetry helm 图表存储库：

[,bash]
----
helm repo add open-telemetry https://open-telemetry.github.io/opentelemetry-helm-charts
----

为 Helm 图表创建`+otel-collector.yaml+` 值文件。以下是使用 SUSE Observability 的良好起点，请将`+<otlp-suse-observability-endpoint:port>+` 替换为您的 OTLP 端点（请参阅xref:/setup/otel/otlp-apis.adoc[OTLP API]了解您的端点），并插入 Kubernetes 集群的名称，而不是`+<your-cluster-name>+` 。使用入口配置时，也请确保在标记位置插入自己的域名和相应的 TLS 证书密钥。

.otel-collector.yaml
[,yaml]
----
mode: deployment
presets:
  kubernetesAttributes:
    enabled: true
    # You can also configure the preset to add all the associated pod's labels and annotations to you telemetry.
    # The label/annotation name will become the resource attribute's key.
    extractAllPodLabels: true
extraEnvsFrom:
  - secretRef:
      name: open-telemetry-collector
image:
  repository: "ghcr.io/open-telemetry/opentelemetry-collector-releases/opentelemetry-collector-k8s"

config:
  receivers:
    otlp:
      protocols:
        grpc:
          endpoint: 0.0.0.0:4317
        http:
          endpoint: 0.0.0.0:4318
  extensions:
    # Use the API key from the env for authentication
    bearertokenauth:
      scheme: SUSEObservability
      token: "${env:API_KEY}"
  exporters:
    otlp:
      auth:
        authenticator: bearertokenauth
      # Put in your own otlp endpoint, for example otlp-suse-observability.my.company.com:443
      endpoint: <otlp-suse-observability-endpoint:port>

  service:
    extensions: [health_check, bearertokenauth]
    pipelines:
      traces:
        receivers: [otlp]
        processors: [batch]
        exporters: [otlp]
      metrics:
        receivers: [otlp]
        processors: [batch]
        exporters: [otlp]
      logs:
        receivers: [otlp]
        processors: [batch]
        exporters: [otlp]

ingress:
  enabled: true
  annotations:
    kubernetes.io/ingress.class: ingress-traefik-external
    traefik.ingress.kubernetes.io/ingress.class: ingress-traefik-external
    traefik.ingress.kubernetes.io/backend-protocol: GRPC
    # "12.34.56.78/32" IP address of NatGateway in the VPC where the otel data is originating from
    #  traefik.ingress.kubernetes.io/whitelist-source-range: "12.34.56.78/32"
  hosts:
    - host: "otlp-collector-proxy.<your-domain>"
      paths:
        - path: /
          pathType: ImplementationSpecific
          port: 4317
  tls:
    - secretName: <secret-for-tls-certificate>
      hosts:
        - "otlp-collector-proxy.<your-domain>"

# Instead of ingress:

# Alternative 1, load balancer service
#service:
#  type: LoadBalancer
#  loadBalancerSourceRanges: 12.34.56.78/32 # The IP address of NatGateway in the VPC for the lambda functions

# Alternative 2, node port service
#service:
#  type: NodePort
#ports:
#  otlp:
#    nodePort: 30317
----


现在使用配置文件安装采集器：

[,bash]
----
helm upgrade --install opentelemetry-collector open-telemetry/opentelemetry-collector \
  --values otel-collector.yaml \
  --namespace open-telemetry
----

确保 Lambda 功能可以访问代理收集器，方法是公开访问入口或将收集器 IP 设置在与 Lambda 功能相同的 VPC 中。 建议使用来源范围白名单来过滤来自不信任和/或未知来源的数据（参见 yaml 中的注释）。除了入口设置，还可以通过以下方式将收集器暴露给 Lambda 函数：

* 通过限制源范围来限制访问的 LoadBalancer 服务，请参见 "替代方案 1"。
* 收集器的 NodePort 服务，请参见 "备选方案 2"。

采集器提供更多配置接收器、处理器和出口器，详情请查看我们的xref:/setup/otel/collector.adoc[采集器页面]。在生产使用中，经常会产生大量的跨度，因此需要开始设置xref:/setup/otel/sampling.adoc[采样]。

== 仪器 Lambda 函数

Open Telemetry 支持使用 Lambda 层对多种语言的 Lambda 函数进行仪表化。这些 Lambda 层的配置应使用上一步中收集器的地址来传输数据。要在 Node.js lambda 中安装仪器，请按xref:/setup/otel/instrumentation/node.js/auto-instrumentation-of-lambdas.adoc[此处的详细说明]操作。对于其他语言的仪器，应用与 Node.js 相同的配置，但使用其他https://opentelemetry.io/docs/platforms/faas/lambda-auto-instrument/[Open Telemetry Lambda 层]之一。

== 查看结果

进入 SUSE Observability，确保已安装 Open Telemetry Stackpack（通过主菜单 \-> Stackpacks）。

过了一会儿，如果您的 Lambda 函数获得了一些流量，您就可以在 Open Telemetry \-> services（打开遥测）和 service instances（服务和服务实例）概览中找到其服务名称下的函数。跟踪将出现在服务和服务实例组件的xref:/use/views/k8s-traces-perspective.adoc[跟踪] xref:/use/traces/k8sTs-explore-traces.adoc[资源管理器]和xref:/use/views/k8s-traces-perspective.adoc[跟踪视角]中。跨度指标和特定语言指标（如有）将出现在组件的xref:/use/views/k8s-metrics-perspective.adoc[指标视角]中。
