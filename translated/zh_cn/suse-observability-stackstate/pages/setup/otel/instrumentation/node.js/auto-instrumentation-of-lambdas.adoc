= 自动监测 NodeJS Lambda
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: SUSE 可观察性

== 导言

本文档将指导您使用 OpenTelemetry 对 NodeJS Lambda 函数进行自动监测。自动仪表通过自动捕获性能指标和跟踪信息，简化了为 Lambda 函数添加可观察性的过程。

== 先决条件

在开始之前，请确保您具备以下条件：

* *AWS Lambda 函数：*您要测量的功能。
* *OpenTelemetry SDK：*安装在您的 Lambda 函数中。
* *OpenTelemetry 采集器：*已部署和配置。
* *SUSE Observability：*用于发送遥测数据的 SUSE Observability 账户。
* *内存*足够的内存来运行 Lambda，包括仪器。

== 环境提供的数值

OpenTelemetry 依靠各种配置值正常运行。这些值控制着数据收集、导出以及与后台系统通信等方面。为了使您的 OpenTelemetry 部署灵活适应不同的环境，您可以通过环境变量提供这些设置。这种方法有几个好处：

* *动态配置：*无需修改代码即可轻松调整设置。
* *特定环境设置*为开发、测试和生产配置不同的 OpenTelemetry。
* *秘密管理：*安全存储 API 密钥等敏感信息。

对于本文档中描述的 OpenTelemetry 设置，您需要定义以下环境变量：

* *`+VERBOSITY+`:*控制 OpenTelemetry 日志的详细程度。
* *`+OTLP_API_KEY+`:*验证您的 Lambda 函数，以便向 SUSE Observability 发送数据。
* *`+OTLP_ENDPOINT+`:*指定 SUSE Observability 实例的地址。
* *`+OPENTELEMETRY_COLLECTOR_CONFIG_FILE+`:*指向 OpenTelemetry 采集器的配置文件。
* *`+AWS_LAMBDA_EXEC_WRAPPER+`:*配置 Lambda 执行环境以使用 OpenTelemetry 处理程序。
* *`+OTLP_INSTR_LAYER_ARN+`:*提供 OpenTelemetry 仪器层的 ARN（亚马逊资源名称），它为自动仪器添加了必要的组件。
* *`+OTLP_COLLECTOR_LAYER_ARN+`:*提供 OpenTelemetry 采集器层的 ARN，该层负责接收、处理和导出遥测数据。

*重要考虑因素：*

* *GRPC 端点：*`+OTLP_ENDPOINT+` 应指定 SUSE Observability 实例的 gRPC 端点，不含任何`+http+` 或`+https+` 前缀。使用 443 端口进行安全通信。
* *特定区域图层：*Lambda 图层受区域限制。确保您用于`+OTLP_INSTR_LAYER_ARN+` 和`+OTLP_COLLECTOR_LAYER_ARN+` 的 ARN 与部署 Lambda 功能的 AWS 区域相匹配。
* *建筑匹配：*OpenTelemetry 采集器层是针对特定架构的。为 Lambda 函数的架构选择正确的 ARN（如`+amd64+` 或`+arm64+` ）。

*完整示例：请注意，您需要输入自己的值。*

[,yaml]
----
VERBOSITY: "normal"
OTLP_API_KEY: "<your api key for sending data to SUSE Observability here>"
OTLP_ENDPOINT: "<your-dns-name-for-suse-observability-here>:443"
OPENTELEMETRY_COLLECTOR_CONFIG_FILE: "/var/task/collector.yaml"
AWS_LAMBDA_EXEC_WRAPPER: "/opt/otel-handler"
OTLP_INSTR_LAYER_ARN: "arn:aws:lambda:<aws-region>:184161586896:layer:opentelemetry-nodejs-0_11_0:1"
OTLP_COLLECTOR_LAYER_ARN: "arn:aws:lambda:<aws-region>:184161586896:layer:opentelemetry-collector-<amd64|arm64>-0_12_0:1"
----

== collector.yaml 文件

OTEL 收集配置设置了如何分发收集到的数据。 收集器.yaml 文件放置在可以找到 lambda 文件的 src 目录中。 下面是一个 collector.yaml 文件示例。

[,yaml]
----
# collector.yaml in the root directory
# Set an environemnt variable 'OPENTELEMETRY_COLLECTOR_CONFIG_FILE' to
# '/var/task/collector.yaml'

receivers:
  otlp:
    protocols:
      grpc:
      http:

exporters:
  debug:
    verbosity: "${env:VERBOSITY}"
  otlp/stackstate:
    headers:
      Authorization: "SUSEObservability ${env:OTLP_API_KEY}"
    endpoint: "${env:OTLP_ENDPOINT}"

service:
  pipelines:
    traces:
      receivers: [otlp]
      exporters: [debug, otlp/stackstate]
      processors: []
    metrics:
      receivers: [otlp]
      exporters: [debug, otlp/stackstate]
      processors: []
----

请注意，该收集器用于将数据发送到下一个收集器，下一个收集器用于尾部采样、指标聚合等，然后再将数据发送到 SUSE Observability。第二个收集器也需要在客户的环境中运行。

根据所需的功能，或基于以这种方式使用仪器的 lambdas 生成的数据量等因素，可以为批处理、尾部采样和其他预处理技术设置收集器，以减少对 SUSE Observability 的影响。

按照xref:/setup/otel/getting-started/getting-started-lambda.adoc[入门指南]设置收集器，将数据发送到 SUSE Observability。定制采集器配置以设置采样、过滤等功能，请参阅我们的xref:/setup/otel/collector.adoc[采集器文档]。

== Package.json

确保将`+"@opentelemetry/auto-instrumentations-node": "^0.55.2",++` 添加到`+package.json+` 并执行`+npm install+` ，以便将自动仪表客户端库添加到 NodeJS Lambda 中。

== 故障排除

=== 超时

如果添加 OTEL Lambda 层导致 lambdas 超时（检查日志可能会发现收集器在繁忙时被要求关闭，例如看到以下日志条目）：

[,json]
----
{
    "level": "info",
    "ts": 1736867469.2312617,
    "caller": "internal/retry_sender.go:126",
    "msg": "Exporting failed. Will retry the request after interval.",
    "kind": "exporter",
    "data_type": "traces",
    "name": "otlp/stackstate",
    "error": "rpc error: code = Canceled desc = context canceled",
    "interval": "5.125929689s"
}
----

在收到停机指令后不久：

[,json]
----
{
    "level": "info",
    "ts": 1736867468.4311068,
    "logger": "lifecycle.manager",
    "msg": "Received SHUTDOWN event"
}
----

上述情况表明，分配给 lambda 的资源不足以执行 lambda 和 OTEL 仪器增加的额外压力。 为了解决这个问题，可以根据需要调整内存分配和 lambda 超时设置，以便让 lambda 完成工作，同时也让遥测收集成功。

尝试修改出现故障的 lambdas 的 MemorySize 和 TimeOut 属性：

[,yaml]
----
MemorySize: 256
Timeout: 25
----

请注意，默认内存分配为 128MB

注意内存增量为 128MB

注意 超时是一个整数值，表示秒数。

=== 身份验证和源 IP 过滤

如果在向群集或任何预处理或代理收集器提交收集器数据时遇到`+error 403 Unauthorized+` ，请仔细检查 VPC NAT 网关的源 IP 地址是否与收集器入口的白名单相匹配，还要仔细检查所选的验证机制是否与源和目的地相匹配，以及凭证（机密等）是否设置正确。

有关为 opentelemetry 采集器配置身份验证的更多信息，请参阅https://github.com/open-telemetry/opentelemetry-collector/blob/main/config/configauth/README.md[官方文档]。
