安全的供应链基础设施可以验证其各个部分或环节的有效性。它可以让用户和开发人员展示其软件组件或人工制品的监管链。这是一种缓解安全问题的积极方法。

https://sigstore.dev/[]Sigstore 项目为此提供了工具和基础设施。它用于验证工艺品供应链的完整性。

{short-project-name} 使用https://github.com/sigstore/cosign[`+cosign+`]以及 Sigstore 项目提供的https://github.com/SigStore/fulcio[fulcio]和https://github.com/sigstore/rekor[rekor]基础设施。

集群操作员可以配置{project-name} ，使其只运行由可信实体签署的策略。政策开发人员可以签署他们的政策，并将其发布到注册表中。

== 先决条件

在下面的章节中，您需要安装一些工具。这样用户就可以签署和验证 OCI 人工制品的签名。示例显示了https://docs.sigstore.dev/quickstart/quickstart-cosign/[`+cosign+`]和https://github.com/kubewarden/kwctl[`+kwctl+`]工具来签署和检查策略。

要使用 GitHub 签署策略，需要安装https://docs.sigstore.dev/cosign/system_config/installation/#github-actions[GitHub actions]。

无密钥签名使用 Sigstore 项目提供的默认https://github.com/SigStore/fulcio[fulcio]和https://github.com/sigstore/rekor[rekor]实例。如有需要，请查看 Sigstore 文档，了解如何使用自己的基础架构来实现这一点。

== 签署政策

{short-project-name} 建议使用 Sigstore 的https://github.com/sigstore/cosign[] 联署工具签署保单。本节介绍一种基于密钥的策略签名方法。为此，您需要生成私钥对-公钥对。生成的密钥有助于验证已签名的人工制品是否来自预期用户。要生成这个密钥对，请使用`+cosign generate-key-pair+` 命令：

[subs="+attributes",bash]
----
cosign generate-key-pair
----

提示输入并验证密码：

[subs="+attributes",console]
----
Enter password for private key: ●●●●●●●●
Enter password for private key again: ●●●●●●●●
Private key written to cosign.key
Public key written to cosign.pub
----

现在，您可以使用此密钥来签署政策。

不要共享私钥文件，`+cosign.key+` 。这是一个秘密文件，仅供密钥所有者在签署策略时使用。

要签署策略，可以使用`+cosign sign+` ，将`+--key+` 命令行参数与私钥文件一起传入：

[subs="+attributes",bash]
----
cosign sign --key cosign.key ghcr.io/kubewarden/policies/user-group-psp:latest
----

提示输入指定私人密钥的密码：

[subs="+attributes",console]
----
an error occurred: no provider found for that key reference, will try to load key from disk...
Enter password for private key: ●●●●●●●●
Pushing signature to: ghcr.io/kubewarden/policies/user-group-psp
----

该命令通过创建新的签名对象对策略进行签名。然后，签名对象会与策略一起上传到注册表中。现在，该策略已准备就绪，可在{short-project-name} 安装中使用签名验证。

同一策略可由同一用户或不同用户签署多次。这些签名与原始签名一起添加到签名对象中。

如需了解更多有关签名过程的信息，请查阅https://docs.sigstore.dev/[项目文档] Sigstore。

=== 无钥匙签约

通常情况下，策略是通过 CI/CD 管道自动构建的。这使得密钥生成过程变得更加复杂。Sigstore 无钥匙工作流程就是针对这些情况设计的。无密钥工作流程使用证书颁发机构（CA）和证书链，而不是使用长期存在的演唱密钥。

您要将生成的短期证书密钥与信任链联系起来。这是通过身份挑战来确认签名者的身份。证书密钥的有效期足够进行签名。身份挑战是通过对 OpenID Connect (OIDC) 提供商进行身份验证来完成的。Sigstore 的 Fulcio 公共基础设施提供了信任链。

签署使用 Sigstore 的共同签署工具。

[subs="+attributes",bash]
----
$ cosign sign ghcr.io/kubewarden/policies/user-group-psp:latest
----

=== 如何在 GitHub 工作流中签署工件

在 GitHub 操作中使用无密钥签名时，`+cosign+` 不需要用户登录到 OIDC 提供商。GitHub 令牌可在执行 GitHub 工作流程期间使用。它用于验证用户身份和生成短暂密钥。签名过程与无钥匙模式相同。这是{short-project-name} 项目如何签署政策的一个例子：

.描述{short-project-name} 策略签名的 YAML
[%collapsible]
======

[,yaml]
----
# ... beginning of the workflow file ...
jobs:
  build:
    name: Build container image
    runs-on: ubuntu-latest
    steps:
      # ... other steps building the container image ...
      -
      name: Login to GitHub Container Registry
      uses: docker/login-action@v1
      with:
        registry: ghcr.io
        username: ${{ github.repository_owner }}
        password: ${{ inputs.GITHUB_TOKEN }}
      -
      name: Publish Wasm policy artifact to OCI registry with the 'latest' tag
      shell: bash
      if: ${{ startsWith(github.ref, 'refs/heads/') }}
      env:
        COSIGN_EXPERIMENTAL: 1
      run: |
        set -ex
        echo Pushing policy to OCI container registry
        IMMUTABLE_REF=$(kwctl push -o json ${{ PATH_TO_BUILT_WASM_FILE }} ghcr.io/myorg/policies/my-great-policy:latest | jq -r .immutable_ref)
        echo Keyless signing of policy using cosign
        cosign sign ${IMMUTABLE_REF}
      # ... other build steps ...

# ... remainder of the workflow file ...
----

== 验证政策

[subs="+attributes",console]
----
$ kwctl verify -k cosign.pub ghcr.io/kubewarden/policies/user-group-psp:latest
2022-03-29T14:49:31.878180Z  INFO kwctl::verify: Policy successfully verified
----

或`+cosign+` ：

== 配置策略服务器以检查策略签名

您可以通过`+ConfigMap+` 配置{short-project-name} ，使其只运行受信任的策略。`+ConfigMap+` 结构参见《xref:reference/verification-config.adoc#_signature_configuration_reference[签名配置参考》]。它用于使用`+kwctl+` 验证策略。`+ConfigMap+` 应在`+verification-config+` 字段下定义允许的配置。

例如，您想运行由{short-project-name} GitHub 组织签署的策略。那么，这种情况下的`+ConfigMap+` 示例为

.$ cat kubewarden_signatures.yaml
[%collapsible]
======

[subs="+attributes",console]
----
$ cat kubewarden_signatures.yaml
apiVersion: v1
allOf:
  - kind: githubAction
    owner: kubewarden

# note that the data is stored under verification-config field
$ kubectl  create configmap my-signatures-configuration --from-file=verification-config=kubewarden_signatures.yaml

$ kubectl get configmap -o yaml my-signatures-configuration
apiVersion: v1
data:
  verification-config: |
    apiVersion: v1
    allOf:
      - kind: githubAction
        owner: kubewarden
kind: ConfigMap
metadata:
  creationTimestamp: "2022-03-29T18:27:20Z"
  name: my-signatures-configuration
  namespace: default
  resourceVersion: "10279"
  uid: d53e1c56-1fee-45de-92f5-9bd73b8cead4
----

======

您可以使用`+kwctl scaffold verification-config+` 为`+ConfigMap+` 生成默认验证配置文件：

[subs="+attributes",console]
----

== 签名配置参考

您可以验证文件中包含的签名要求。下面就是一个例子：

.签名要求文件
[%collapsible]
======
[subs="+attributes",yaml]
----
apiVersion: v1

allOf: <1>
  - kind: githubAction
    owner: kubewarden   # mandatory
    annotations:
      env: prod

anyOf: # at least `+anyOf.minimumMatches+` are required to match <2>
  minimumMatches: 2 # default is 1
  signatures:
  - kind: pubKey
    owner: flavio # optional
    key: .... # mandatory
    annotations:  # optional
      env: prod
      foo: bar
  - kind: pubKey
    owner: victor # optional
    key: .... # mandatory
  - kind: genericIssuer
    issuer: https://github.com/login/oauth
    subject:
      equal: alice@example.com
  - kind: genericIssuer
    issuer: https://token.actions.githubusercontent.com
    subject:
      equal: https://github.com/flavio/policy-secure-pod-images/.github/workflows/release.yml@refs/heads/main
  - kind: genericIssuer
    issuer: https://token.actions.githubusercontent.com
    subject:
      urlPrefix: https://github.com/flavio/
  - kind: genericIssuer
    issuer: https://token.actions.githubusercontent.com
    subject:
      urlPrefix: https://github.com/kubewarden # <- it will be post-fixed with `+/+` for security reasons
  - kind: githubAction
    owner: flavio   # mandatory
    repo: policy1 # optional
  - kind: pubKey
    owner: alice # optional
    key: .... # mandatory
----
<1> `+allOf+`
<2> `+anyOf+`
======

=== 签名验证

上述配置包含两个突出显示的部分`+allOf+` 和`+anyOf+` ：

* `+allOf+`签名：只有当此处的所有签名要求都有效时，政策才会被信任。
* `+anyOf+`：如果满足`+minimumMatches+` 标准，则该策略是可信的。

以上，`+minimumMatches+` 字段为 2。因此，必须满足至少两项签名要求。`+minimumMatches+` 字段的默认值是`+1+` 。

必须满足`+allOf+` 中的所有签名要求*和*`+anyOf+` 中的最低人数要求。

=== 公钥验证

要检查策略是否使用正确的公开密钥签署，需要指定密钥数据和密钥所有者。在此示例中，您将`+kind+`, 设置为`+pubKey+` ，而`+key+` 拥有公钥。所有者字段是可选的，但对于明确密钥的所有者很有用。

[subs="+attributes",yaml]
----
  - kind: pubKey
    owner: bob # optional
    key: |
      -----BEGIN PUBLIC KEY-----
      MBFKHFDGHKIJH0CAQYIKoZIzj0DAQcDQgAEX0HFTtCfTtPmkx5p1RbDE6HJSGAVD
      BVDF6SKFSF87AASUspkQsN3FO4iyWodCy5j3o0CdIJD/KJHDJFHDFIu6sA==
      -----END PUBLIC KEY-----
----

=== 无钥匙签名验证

以无密钥模式签署的策略没有公钥可验证。您仍然可以使用签署过程中使用的 OIDC 数据来验证策略。为此，有必要将签名验证定义为`+genericIssuer+` 。

可以通过签名验证信息：

* `+issuer+`(强制性）：这与 Fulcio 生成的证书中的`+Issuer+` 属性相匹配。这显示了用于签署政策的 OIDC。
* `+subject+`:用于匹配 Fulcio 证书中`+Subject+` 属性的字段。`+Subject+` (Fulcio) 字段包含用于验证 OIDC 提供商的用户。验证字段`+subject+` 可以有两个子字段：
 ** `+equal+`:证书中的`+Subject+` (Fulcio) 必须等于签名验证中的值；
 ** `+urlPrefix+`:证书的`+Subject+` (Fulcio) 字段值必须以签名验证中定义的值为前缀。

[NOTE]
====

`+cosign verify+` 和`+kwctl inspect+` 都可以显示有关无钥匙签名的信息。

====


例如，该配置意味着策略必须由 Alice 使用 GitHub OIDC 进行无密钥签名：

[subs="+attributes",yaml]
----
- kind: genericIssuer
  issuer: https://github.com/login/oauth
  subject:
    equal: alice@example.com
----

该配置需要在 GitHub 操作中对策略进行签名，签名来自 GitHub 用户`+flavio+` 所拥有的版本库：

[subs="+attributes",yaml]
----
- kind: genericIssuer
  issuer: https://token.actions.githubusercontent.com
  subject:
    urlPrefix: https://github.com/flavio
----

=== GitHub 行动签名验证

种"，`+githubAction+` 用于验证在 GitHub Actions 中签署的策略。
`+genericIssuer+` 也可以这样做。为简化签名要求程序，`+githubAction+` ：

* `+owner+` (强制性）：要信任的用户或组织的 GitHub ID
* `+repo+`：要信任的存储库名称

例如，使用`+genericIssuer+` 的最后一个片段可改写为

[subs="+attributes",yaml]
----
- kind: githubAction
  owner: flavio
----

=== 签名注释验证

所有签名类型都可以有其他可选的验证字段，`+annotations+` 。
这些字段是在签名过程中添加的键/值数据。

通过{short-project-name} ，可以检查策略签名是否由受信任的用户签署*，以及*他们是否有特定的注释。

下一步验证将检查政策的两个条件：

* 是用特定的密钥签署的
* 它有一个生产环境注释

[subs="+attributes",yaml]
----
- kind: pubKey
  key: |
    -----BEGIN PUBLIC KEY-----
    MBFKHFDGHKIJH0CAQYIKoZIzj0DAQcDQgAEX0HFTtCfTtPmkx5p1RbDE6HJSGAVD
    BVDF6SKFSF87AASUspkQsN3FO4iyWodCy5j3o0CdIJD/KJHDJFHDFIu6sA==
    -----END PUBLIC KEY-----
  annotations:
    environment: production
----
