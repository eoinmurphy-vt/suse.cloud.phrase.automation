= 允许在准入控制器命名空间中进行评估
:description: 如何配置准入控制器，使其允许在自己的命名空间中进行策略评估
:keywords: ["策略"、"群集许可策略"、"群集许可策略组"、"配置"、"命名空间"].
:doc-persona: ["kubewarden-operator"]
:doc-type: ["howto"]
:doc-topic: ["operator-manual", "policies"]

默认情况下，接纳控制器不允许策略评估自身命名空间中的资源。这样可以防止错误配置，以免破坏{short-project-name} 的安装。

这项安全措施分两层实施。第一层由`+--always-accept-admission-reviews-on-deployments-namespace+` CLI 标志控制。
该标记在 Helm 图表中默认启用，可使控制器在 PolicyServer 部署中配置`+KUBEWARDEN_ALWAYS_ACCEPT_ADMISSION_REVIEWS_ON_NAMESPACE+`环境变量。该变量告诉策略服务器始终接受来自变量中定义的命名空间的录入审查。第二层是准入控制器本身，默认情况下，它会在全群集策略中添加命名空间选择器，以跳过控制器运行所在的命名空间。

不过，在某些情况下，群集管理员可能希望策略能评估该命名空间中的资源。本页将介绍如何通过禁用这两层保护来实现这一目的。

== 删除 PolicyServer 保护环境变量

第一步是配置控制器，使其不在 PolicyServer 部署中设置`+KUBEWARDEN_ALWAYS_ACCEPT_ADMISSION_REVIEWS_ON_NAMESPACE+` 环境变量。

为此，请升级{short-project-name} 安装，并将`+kubewarden-controller+` 舵手图表中的`+alwaysAcceptAdmissionReviewsOnDeploymentsNamespace+` 值设置为`+false+` ：

[source,console]
----
helm upgrade -n kubewarden
  --set "alwaysAcceptAdmissionReviewsOnDeploymentsNamespace=false"
  kubewarden-controller kubewarden/kubewarden-controller
----

这一变更允许策略服务器评估来自任何命名空间的资源。
配置完成后，就可以对政策进行配置，以评估来自{short-project-name} 命名空间的入院审查。

== 配置策略以评估 Admission Controller 名称空间中的资源

要允许策略评估在 Admission Controller 命名空间中创建的资源，可部署全群集策略，并将规范中的`+allowInsideAdmissionControllerNamespace+`字段设置为`+true+` ：

[NOTE]
.命名策略呢？
====
此规范字段仅在全群集策略中可用。
由于命名空间策略是在部署策略的同一命名空间中对资源进行评估，因此在其 CRD 中添加此字段并不合理。
此外，接纳控制器命名空间应由群集操作员管理，他们可以决定是否在此部署策略。
====

[source,yaml]
----
apiVersion: policies.kubewarden.io/v1
kind: ClusterAdmissionPolicy
metadata:
  annotations:
    io.kubewarden.policy.category: PSP
    io.kubewarden.policy.severity: medium
  name: pod-privileged-policy
spec:
  module: registry://ghcr.io/kubewarden/policies/pod-privileged:v1.0.10
  settings: {}
  rules:
    - apiGroups:
        - ""
      apiVersions:
        - v1
      resources:
        - pods
      operations:
        - CREATE
  mutating: false
  allowInsideAdmissionControllerNamespace: true
----

该配置选项（默认为`+false+` ）可使控制器跳过添加排除 Admission Controller 命名空间的命名空间选择器，从而允许策略评估包括该命名空间在内的所有命名空间的资源。

[NOTE]
.我的自定义命名空间选择器怎么办？
====
策略 CRD 还允许用户定义自己的命名空间选择器。这些选择器不会更改。这意味着，如果用户定义的选择器跳过了{short-project-name} 命名空间，策略仍会按照预期忽略该命名空间。
====
