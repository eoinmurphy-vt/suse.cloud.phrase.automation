== 政策类型

OPA Gatekeeper 和 Kubernetes 都可以编写验证和突变策略。

这些策略可以针对任何类型的 Kubernetes 资源，包括自定义资源。

== 编写政策

您可以使用https://www.openpolicyagent.org/docs/latest/#rego[Rego] 编写 OPA Gatekeeper 策略。Rego 是开放政策代理项目创建的一种查询语言。

[IMPORTANT]
====

您只能使用 Rego 编写验证策略。变异策略不使用 Rego，而是使用 YAML 中定义的临时规则（参见https://open-policy-agent.github.io/gatekeeper/website/docs/mutation[] ）。

====


您可以使用不同的范式编写{short-project-name} 策略。政策作者既可以使用传统编程语言（如 Go、Rust 等），也可以使用https://en.wikipedia.org/wiki/Domain-specific_language[特定领域语言]，如 Rego 和 CEL。在{short-project-name} 中，验证策略和变异策略的编写方法是一样的。

[CAUTION]
====

https://github.com/open-policy-agent/kube-mgmt[]kube-mgmt 开源项目是开放政策代理项目的一部分，它使用 Rego。

尽管使用了相同的语言，但为 kube-mgmt 编写的策略与 OPA Gatekeeper 并不兼容，反之亦然。

{short-project-name} 可以使用为 Open Policy Agent 和 OPA Gatekeeper 编写的 Rego 策略。更多信息请xref:/tutorials/writing-policies/rego/01-intro-rego.adoc[点击此处]。

====


== 情境感知

有时，策略需要有关群组当前状态的数据来做出验证或突变决定。例如，验证 Ingress 资源的策略可能需要了解集群中已定义的其他 Ingress 资源，以确保不会发生冲突。{short-project-name} 称这些策略为 "上下文感知 "策略。

OPA Gatekeeper 和{short-project-name} 都支持这些类型的策略。

在部署 OPA Gatekeeper 时，Kubernetes 管理员会决定在评估时向策略提供哪种类型的群集数据。

重要的是要强调这些数据如何被所有已部署的政策访问。

例如，如果 OPA Gatekeeper 策略要求访问 Kubernetes Secrets，那么部署的所有其他策略也都能读取这些数据。

相反，{short-project-name} 提供了对集群资源的细粒度访问。
每个策略只能访问 Kubernetes 管理员指定的资源。试图读取未经授权数据的行为会立即被阻止，并报告给群集管理员。

== Kubernetes 集成

OPA Gatekeeper 有一个集群范围的自定义资源，允许定义策略，以及如何和在哪里执行这些策略。

{short-project-name} 有两种不同类型的自定义资源用于声明策略。一个在集群级别工作，另一个则以名称为分隔。命名自定义资源的名称是`+AdmissionPolicy+` 。

通过`+AdmissionPolicy+` 资源部署的策略只影响在其所属命名空间内创建的资源。因此，您可以允许非管理 Kubernetes 用户在他们可以访问的命名空间中拥有管理`+AdmissionPolicy+` 资源所需的 RBAC 权限。

这样，Kubernetes 管理员就可以委托与策略相关的工作。

== 政策分配

OPA Gatekeeper 和{short-project-name} 策略的策略源代码（Rego 用于 OPA Gatekeeper，CEL 用于{short-project-name} ）位于 Kubernetes 中定义策略的自定义资源中。

此外，{short-project-name} 策略可以像管理容器映像一样管理策略的源代码（Rego、Go、Rust、... ）。一旦构建完成，它们就会作为 OCI 工件被推送到容器注册中心。

您可以使用https://sigstore.dev[项目] Sigstore中的容器镜像工具（如`+cosign+` ）签署和验证{short-project-name} 策略。

您可以使用容器映像所采用的传统工具和流程，在地理分布广泛的容器注册中心之间分发{short-project-name} 策略。

== CI/CD 整合

OPA Gatekeeper 和{short-project-name} 均采用 GitOps 方法进行管理。

但是，对于 OPA Gatekeeper 而言，策略的 Rego 代码与用于在 Kubernetes 中部署策略的自定义资源之间存在耦合。
这就在 CI/CD 管道中引入了额外的步骤。

Rego 拥有https://www.openpolicyagent.org/docs/latest/policy-testing/[测试工具]，允许创建单元测试套件。编写测试并在 CI/CD 管道中执行测试对于确保策略按预期运行至关重要。

要使用这些测试工具，必须在专用文本文件中提供政策的源代码。我们无法从用于声明 OPA Gatekeeper 策略的 YAML 文件中读取源代码。CI/CD 管道必须将要测试的 Rego 源代码与 OPA Gatekeeper 自定义资源中定义的代码同步。您可以使用第三方工具来实现这一功能。

{short-project-name} 政策与传统微服务一样拥有 CI/CD 管道。
通常，他们的源代码保存在 Git 代码库中，然后使用传统的 CI/CD 系统对其进行单元测试。您可以使用编写策略所用语言的测试框架来编写单元测试。一旦所有测试通过，就将策略编译成 WebAssembly，并将其推送到容器注册表。这种管道通常由政策制定者负责维护。

Kubernetes 管理员通常会维护其他自动化管道，以便对策略的新版本（使用https://docs.github.com/en/code-security/dependabot/working-with-dependabot[Dependabot]、https://www.mend.io/renovate/[Renovate bot]、https://www.updatecli.io/[updatecli]等自动化工具）或策略配置的变更做出反应。

管道针对不同类型的请求测试策略。您可以使用https://github.com/kubewarden/kwctl[`+kwctl+`]CLI 工具进行测试，无需运行 Kubernetes 集群。`+kwctl+` CLI 工具使用与部署在 Kubernetes 集群中的{short-project-name} 堆栈相同的评估引擎。

== 政策执行模式

OPA Gatekeeper 和{short-project-name} 可使用两种不同的操作模式部署策略：

* `+deny+`：违反政策的申请将被拒绝
* `+warn+`：违反政策不会导致拒绝，但会记录在案

== 部署模式

由同一台服务器评估所有 OPA 网守策略。相反，{short-project-name} 允许定义多个评估服务器。您可以通过名为`+PolicyServer+` 的自定义资源来定义这些服务器。

在声明{short-project-name} 策略时，Kubernetes 管理员会决定由哪个`+PolicyServer+` 托管该策略。

[NOTE]
====

`+PolicyServer+` 对象是{short-project-name} 引入的高级抽象概念。
在幕后，会创建一个具有特定复制大小的`+Deployment+` 。

每个`+PolicyServer+` 都可以有不同于其他网站的复制大小。

====

这样就会出现以下有趣的情况：

* 将关键策略部署到专用策略服务器池。
* 将嘈杂租户的策略部署到专用策略服务器池中。

== 背景调查

随着策略的添加、删除和重新配置，群集中已有的资源可能会变得不合规。

OPA Gatekeeper 和{short-project-name} 都有一个在后台运行的扫描仪。该扫描程序会评估群集中已定义的资源，并标记不符合要求的资源。

OPA Gatekeeper 与{short-project-name} 的唯一区别在于如何保存扫描结果。

OPA Gatekeeper 会将违规详情添加到指定`+Constraint+` 自定义资源的`+status+` 字段（请参阅https://open-policy-agent.github.io/gatekeeper/website/docs/audit#constraint-status[] ）。

{short-project-name} 而不是将结果存储在https://openreports.io[openreports.io] 定义的一组报告自定义资源中。
