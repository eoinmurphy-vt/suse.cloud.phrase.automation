[[kwctl]]
=== `+kwctl+`

管理{project-name} 政策的工具

*使用方法* `+kwctl [OPTIONS] <COMMAND>+`

==== 子命令

* `+annotate+` - 为 WebAssembly 模块添加{short-project-name} 元数据
* `+bench+` - 基准 a{short-project-name} 政策
* `+completions+` - 生成外壳补全
* `+digest+` - 从保单的 OCI 清单中获取摘要
* `+docs+` - 生成 kwctl 命令的 markdown 文档
* `+info+` - 显示系统信息
* `+inspect+` - 检查{short-project-name} 政策
* `+load+` - 从 tar.gz 文件加载策略
* `+policies+` - 列出所有已下载的政策
* `+pull+` - 从给定的 URI 中提取{short-project-name} 策略
* `+push+` - 将{short-project-name} 政策推送至 OCI 注册表
* `+rm+` - 从商店中删除{short-project-name} 政策
* `+run+` - 从给定的 URI 运行{short-project-name} 策略
* `+save+` - 将策略保存到 tar.gz 文件中
* `+scaffold+` - 为 Kubernetes 资源或配置文件搭建脚手架
* `+verify+` - 使用 Sigstore 从给定 URI 验证{short-project-name} 策略

==== 选项

* `+-v+`,`+--verbose <VERBOSE>+` - 增加冗长度
* `+--no-color <NO-COLOR>+` - 禁用彩色输出

[[kwctl-bench]]
=== `+kwctl bench+`

以{short-project-name} 政策为基准。

可通过以下方式指定策略：- URI：如`+registry://ghcr.io/kubewarden/policies/psp-policy:latest+` 或`+\https://example.com/kubewarden/policies/main/psp-policy/psp-policy.wasm+`- SHA 前缀：如`+c3b80a10f9c3+` （要求已提取策略） - 本地 WASM 文件：如`+file://home/tux/new-policy/psp-policy.wasm+`- 本地 YAML 文件：如`+file://home/tux/cluster-admission-policy.yaml+` （包含{short-project-name} 自定义资源的声明，如`+ClusterAdmissionPolicy+`,`+AdmissionPolicy+` 等）

默认行为：
如果省略模式，则假定`+file://+` 根植于当前目录。

{short-project-name} 自定义资源的注意事项：- 标志`+--request-path+` 、`+--settings-path+` 和`+--settings-json+` 将被忽略；设置将从自定义资源定义中读取。-`+--execution-mode+` 标志适用于 YAML 文件中的所有策略。- 不能使用`+--raw+` 标志，因为{short-project-name} 的自定义资源不支持`+raw+` 策略。

只对自定义资源定义 (CRD) 的以下属性进行评估：- 策略模块 - 策略设置 - 策略可访问的上下文感知资源

其他字段，如`+rules+`,`+matchConditions+`,`+objectSelector+`, 和`+namespaceSelector+` 将被忽略。

一个 YAML 文件可能包含多个自定义资源声明。在这种情况下，`+kwctl+` 在每次评估过程中使用相同的请求来评估文件中的每个策略。

*使用方法* `+kwctl bench [OPTIONS] --request-path <PATH> <uri_or_sha_prefix_or_yaml_file>+`

==== 论据

* `+<URI_OR_SHA_PREFIX_OR_YAML_FILE>+` - 包含{short-project-name} 策略资源的策略 URI、SHA 前缀或 YAML 文件。支持的方案：registry://、https://, file://。如果省略 schema，则假定 file:// 根植于当前目录。

==== 选项

* `+--allow-context-aware <ALLOW-CONTEXT-AWARE>+` - 授权访问策略`+contextAwareResources+` 部分中定义的 Kubernetes 资源。警告：请仔细阅读资源清单，避免滥用。默认已禁用
* `+--cert-email <VALUE>+` - Fulcio 证书中的预期电子邮件
* `+--cert-oidc-issuer <VALUE>+` - Fulcio 证书中的预期 OIDC 签发人
* `+--disable-wasmtime-cache <DISABLE-WASMTIME-CACHE>+` - 关闭使用 wasmtime 缓存
* `+--docker-config-json-path <PATH>+` - 包含 Docker "config.json "文件的目录路径。可用于显示注册表身份验证详细信息
* `+--dump-results-to-disk <DUMP_RESULTS_TO_DISK>+` - 如果能找到目标，则将结果放入 target/tiny-bench/label/\... 中。用于比较以前的运行结果
* `+-e+`,`+--execution-mode <MODE>+` - 用于执行该策略的运行时 +
  可能的值：`+opa+`,`+gatekeeper+`,`+kubewarden+` 、 `+wasi+`
* `+--github-owner <VALUE>+` - 在 CD 管道中生成的证书中预期的 GitHub 所有者
* `+--github-repo <VALUE>+` - CD 管道中生成的证书中预期的 GitHub 仓库
* `+--measurement-time <SECONDS>+` - 如果要测试的代码速度慢于所允许的时间限制，那么测试将需要更长的时间来收集样本数。
* `+--num-resamples <NUM>+` - 应进行多少次重新采样
* `+--num-samples <NUM>+` - 应该做多少次重新采样。建议至少 50 个，超过 100 个似乎不会产生明显不同的结果
* `+--raw <RAW>+` - 验证原始请求 +
  默认值： `+false+`
* `+--record-host-capabilities-interactions <FILE>+` - 将所有策略和主机能力通信记录到给定文件中。稍后与 "--replay-host-capabilities-interactions" 标志结合使用，非常有用
* `+--replay-host-capabilities-interactions <FILE>+` - 在策略和主机能力交换过程中，主机会回放所提供文件中的答案。由于不与 OCI 注册机构、DNS、Kubernetes 进行外部交互，这有助于以可重现的方式测试策略。
* `+-r+`,`+--request-path <PATH>+` - 文件包含 JSON 格式的 Kubernetes 准入请求对象
* `+--settings-json <VALUE>+` - 包含该策略设置的 JSON 字符串
* `+-s+`,`+--settings-path <PATH>+` - 包含此策略设置的文件
* `+--sigstore-trust-config <PATH>+` - 符合 Sigstore protobuf 规范中 ClientTrustConfig 信息的 JSON 格式文件。该文件配置整个 Sigstore 实例状态，包括用于访问 CA 和人工制品透明度服务以及加密信任根本身的 URI。
* `+--sources-path <PATH>+` - 保存源信息的 YAML 文件（https、不安全主机注册表、自定义 CA...)
* `+-a+`,`+--verification-annotation <KEY=VALUE>+` - 键=值格式的注释。可重复多次
* `+--verification-config-path <PATH>+` - 保存验证配置信息（签名、公钥... ）的 YAML 文件
* `+-k+`,`+--verification-key <PATH>+` - 用于验证策略的密钥路径。可重复多次
* `+--warm-up-time <SECONDS>+` - 热身时间

[[kwctl-push]]
=== `+kwctl push+`

将{short-project-name} 政策推送至 OCI 注册表

*使用方法* `+kwctl push [OPTIONS] <policy> <uri>+`

政策元数据中的注释将成为 OCI 清单的一部分。
跳过多行注释是因为它们与 OCI 规范不兼容。
io.kubewarden.policy.source "注释被传播为 "org.opencontainers.image.source"，以便让 renovatebot 等工具检测策略更新。

==== 论据

* `+<POLICY>+` - 政策推动。可以是本地文件的路径、策略 URI 或存储中策略的 SHA 前缀。
* `+<URI>+` - 政策 URI。支持的方案：注册表://

==== 选项

* `+--docker-config-json-path <PATH>+` - 包含 Docker "config.json "文件的目录路径。可用于显示注册表身份验证详细信息
* `+-f+`,`+--force <FORCE>+` - 也推送未加注释的政策
* `+-o+`,`+--output <PATH>+` - 输出格式 +
  默认值：`+text+` +
  可能的值`+text+`, `+json+`
* `+--sources-path <PATH>+` - 保存源信息的 YAML 文件（https、不安全主机注册表、自定义 CA...)
