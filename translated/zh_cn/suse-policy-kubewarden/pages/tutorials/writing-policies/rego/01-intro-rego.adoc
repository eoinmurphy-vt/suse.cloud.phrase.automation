Rego 语言是一种针对特定领域的语言，可将政策作为代码使用。https://www.openpolicyagent.org/docs/latest/policy-language/[Rego]语言的灵感来自https://en.wikipedia.org/wiki/Datalog[Datalog]。

在 Kubernetes 中，有两种编写 Rego 策略以代码形式实施策略的方法：Open Policy Agent 和 Gatekeeper。

接下来的几节将展示这两个框架的不同之处，尽管它们使用的是同一种语言。
如果您对{short-project-name} 的实现更感兴趣，可以直接访问下面链接的特定框架文档。

* xref:tutorials/writing-policies/rego/open-policy-agent/01-intro.adoc[开放政策代理]
* xref:tutorials/writing-policies/rego/gatekeeper/01-intro.adoc[守门人]

== 一种语言，两个框架

=== 开放式策略代理（OPA）

Open Policy Agent 是一个允许你在任何项目中以代码形式实施策略的项目。
您可以根据自己的应用需求，在任何政策中使用 OPA。

在这种情况下，为 Kubernetes 编写策略就是在使用 OPA。
通过使用 Kubernetes 访问 webhooks，可以使用 OPA 评估请求，并执行用 Rego 编写的策略。

OPA 可通过其`+kube-mgmt+` sidecar 与 Kubernetes 进行可选集成。
在 Kubernetes 上部署后，通过 OPA 服务器评估 Rego 策略，就能将配置好的 Kubernetes 资源复制到 Rego 中。
因此，所有策略都能看到这些 Kubernetes 资源。
通过 OPA，您可以在 Kubernetes 的 ConfigMap 对象中定义策略。
如需了解更多信息，请访问https://github.com/open-policy-agent/kube-mgmt[项目页面]。

=== 守门人

Gatekeeper 只专注于在 Kubernetes 中使用，并尽可能利用这一优势，在某些情况下让 Kubernetes 工作流程比 OPA 更简单。

== 审视差异

OPA 和网守策略都使用 Rego 将策略描述为代码。
在使用 Rego 编写策略时，每种解决方案都有不同之处，下文将一一介绍。

== 入口点

入口点是软件包中规则的名称，是运行时在策略运行时调用的规则。

== 目前的局限性

=== 情境感知政策

情境感知策略是一种不单独评估输入请求的策略。
他们在做出决定时会考虑其他因素。
例如，在父命名空间上评估命名空间资源并使用注解的策略，可以在策略中配置某些内容。
另一个例子是对`+Ingress+` 资源进行评估的政策，但在做出决定时还需要现有`+Ingress+` 资源的清单。

情境感知策略的理念也可以扩展到自定义资源，因此您的策略可能也需要根据当前使用的自定义资源来评估请求。

从{short-project-name} 1.9 版本开始，OPA 和 Gatekeeper 都支持上下文感知策略。

有关情境感知政策的更多详情，请xref:reference/spec/05-context-aware-policies.adoc[点击此处]。

=== 变异政策

Gatekeeper 支持突变策略，但{short-project-name} 还没有实现与 Gatekeeper 兼容的突变策略。
您可以使用{short-project-name} SDK 编写变异策略，但目前无法在{short-project-name} 中运行 Gatekeeper 变异策略。
