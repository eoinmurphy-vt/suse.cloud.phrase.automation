https://wasi.dev/[WebAssembly 系统接口（WASI）]是一种 WebAssembly 标准，它提供了一套允许在浏览器之外执行 WebAssembly 的接口。

[CAUTION]
====

编写常规策略的作者绝不能使用普通的 WASI 系统接口来编写策略。

本页面向{project-name} 维护者或希望尝试使用最先进 WASM 平台的低级策略作者。
====


使用 WASI，你可以让 WebAssembly 模块与 STDOUT、STDERR、STDIN、环境变量等系统原语交互。

许多用于编译{short-project-name} 策略的编译器会生成针对 WASI 接口的 WebAssembly 模块。
不过，{short-project-name} 政策使用https://github.com/wapc[waPC]项目来实现政策与政策运行时（`+kwctl+` 或`+policy-server+` ）之间的双向通信。xref:reference/spec/01-intro-spec.adoc[这里]介绍{short-project-name} 对通信协议的使用。

在一些特殊情况下，还不能使用 waPC 项目。
在这种情况下，您可以使用 WASI 提供的接口编写策略。

[IMPORTANT]
====

{short-project-name} 从{short-project-name} 1.7.0 版开始支持 WASI 策略。
====


== 局限性

在常规情况下，您不应该使用 WASI 策略，因为与 waPC/Rego 策略相比，它们在评估时的性能较差。

[NOTE]
====

策略和主机之间可以实现双向通信，但需要在语言 SDK 内部进行更改。这是使用xref:/reference/spec/host-capabilities/01-intro-host-capabilities.adoc[主机功能]和编写xref:/explanations/context-aware-policies.adoc[上下文感知]策略所必需的。

目前，只有{short-project-name} Go 和 JavaScript/TypeScript SDK 将它们暴露于 WASI 策略。

如果您对此感兴趣，请联系我们。这样，我们就能确定工作的优先次序。

====


== 使用案例

编写 "纯 WASI "策略的唯一原因是无法使用 waPC 通信机制。

目前（截至 2023 年 6 月），只有在使用官方 Go 编译器生成 WebAssembly 模块时，才有理由这样做。

从 1.21 版开始，Go 官方编译器可以生成针对 WASI 接口的 WebAssembly 模块。
不过，这些模块还不能向 WebAssembly 运行时导出函数。
https://github.com/golang/go/issues/42372[] 跟踪这一限制，阻止了 waPC 协议的采用。

{short-project-name} 项目团队建议您使用 TinyGo 编译器编写{short-project-name} Go 策略，如xref:tutorials/writing-policies/go/01-intro-go.adoc[此处]所述。

不过，某些复杂的 Go 代码库无法使用 TinyGo 编译器编译。
例如，这包括https://github.com/google/cel-go[Kyverno] CEL-go 或https://github.com/kyverno/kyverno/[] 等代码库。
在这种情况下，使用官方 Go 编译器会有所帮助。

== 通信协议

本节介绍如何编写普通 WASI 策略。

您需要将代码编写成普通的 CLI 程序。
程序必须接受以下子命令：

* `+validate+`该命令由策略引擎调用，以评估准入请求
* `+validate-settings+`该命令由策略引擎调用，以验证策略设置

在这两种情况下，需要验证的数据都是通过 STDIN 提供的。
策略必须通过 STDOUT 提供答案。
您可以使用 STDERR 来获取调试或错误信息。

=== 验证

使用`+validate+` 子命令调用策略 CLI 程序时，会对请求进行验证。

STDIN 必须包含一个描述`+ValidationRequest+` 对象的 JSON 文档。
策略必须将包含`+ValidationResponse+` 对象的 JSON 文档写入 STDOUT。

xref:reference/spec/03-validating-policies.adoc[这里]对`+ValidationRequest+` 和`+ValidationResponse+` 两个对象进行了说明。

=== 突变

变异策略的工作方式与验证策略相同。
使用`+validate+` 子命令调用策略 CLI 程序。

STDIN 必须包含一个描述`+ValidationRequest+` 对象的 JSON 文档。
策略必须将包含`+ValidationResponse+` 对象的 JSON 文档写入 STDOUT。

xref:reference/spec/03-validating-policies.adoc[这里]对`+ValidationRequest+` 和`+ValidationResponse+` 两个对象进行了说明。

当需要突变时，`+ValidationResponse+` 对象必须有一个包含要创建对象的键`+mutated_object+` 。
xref:reference/spec/04-mutating-policies.adoc[在此]介绍这一过程。

== 情境感知

目前仅支持 Go SDK。Go SDK 照常提供上下文感知功能，更多信息请参阅xref:explanations/context-aware-policies.adoc[此处]。

有关 WASI Go 上下文感知策略的示例，请参阅https://github.com/kubewarden/go-wasi-context-aware-test-policy[go-wasi-context-aware-test-policy]。

=== 设置验证

策略必须提供一个名为`+validate-settings+` 的子命令。
该命令用于验证用户提供的设置。

程序必须在 STDIN 上接收一个 JSON 对象，其中包含用户提供的设置。
然后验证它们，并将`+SettingsValidationResponse+` 对象写入 STDOUT。

`+SettingsValidationResponse+` 的格式和设置验证过程xref:reference/spec/02-settings.adoc[在此]说明。

== 政策元数据

每个{short-project-name} 策略都必须通过`+kwctl annotate+` 命令进行注释。
纯 WASI 策略的策略元数据必须具有此值：

[subs="+attributes",yaml]
----
executionMode: wasi
----

https://github.com/kubewarden/go-wasi-policy-template[该 GitHub 仓库]包含一个使用 WASI 协议的基于 Go 的策略模板。
