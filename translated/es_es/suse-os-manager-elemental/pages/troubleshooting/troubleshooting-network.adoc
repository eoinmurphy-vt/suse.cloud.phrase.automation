= Resolución de problemas en redes declarativas
:revdate: 2025-09-08
:page-revdate: {revdate}

Dado el siguiente ejemplo de registro:

.ejemplo MachineRegistration utilizando Declarative Networking
[,yaml]
----
include::example$network/machineregistration.yaml[]
----

Podemos esperar que cada Máquina Elemental se configure utilizando la plantilla definida `+nm-configurator+` `+_all.yaml+`.

En el primer arranque, `+elemental-register+` intentará ponerse en contacto con la API de Rancher para registrar un nuevo `+MachineInventory+`.

En esta fase, la red de la máquina no está configurada y utilizará DHCP por defecto. Es un requisito que la máquina sea capaz de contactar con la API de Rancher en esta configuración, de lo contrario el registro no puede tener lugar.

Una vez registrado el `+MachineInventory+`, el `+elemental-operator+` va a reclamar un `+IPAddress+` para cada referencia `+IPPool+` definida en la configuración de la red.

En la página `+MachineInventory+`, tendrá el siguiente aspecto:

[,yaml]
----
apiVersion: elemental.cattle.io/v1beta1
kind: MachineInventory
metadata:
  finalizers:
  - machineinventory.elemental.cattle.io
  name: m-e5331e3b-1e1b-4ce7-b080-235ed9a6d07c
  namespace: fleet-default
spec:
  ipAddressClaims:
    inventory-ip:
      apiVersion: ipam.cluster.x-k8s.io/v1beta1
      kind: IPAddressClaim
      name: m-e5331e3b-1e1b-4ce7-b080-235ed9a6d07c-inventory-ip
      namespace: fleet-default
      uid: 78f2d07a-7b6d-4b58-b615-c4108b7964b9
  ipAddressPools:
    inventory-ip:
      apiGroup: ipam.cluster.x-k8s.io
      kind: InClusterIPPool
      name: elemental-inventory-pool
  network:
    config:
      dns-resolver:
        config:
          search: []
          server:
          - 192.168.122.1
      interfaces:
      - description: Main-NIC
        ipv4:
          address:
          - ip: "{inventory-ip}"
            prefix-length: 24
          dhcp: false
          enabled: true
        ipv6:
          enabled: false
        name: eth0
        state: up
        type: ethernet
      routes:
        config:
        - destination: 0.0.0.0/0
          metric: 150
          next-hop-address: 192.168.122.1
          next-hop-interface: eth0
          table-id: 254
    ipAddresses:
      inventory-ip: 192.168.122.150
status:
  conditions:
  - lastTransitionTime: "2024-07-30T11:50:47Z"
    message: NetworkConfig is ready
    reason: ReconcilingNetworkConfig
    status: "True"
    type: NetworkConfigReady
----

Observará que `+MachineInventory+` lleva el mismo `+network.config+` que `+MachineRegistration+`, sin embargo, en lugar de hacer referencia a IPAddressPools, ahora tenemos un mapa de IPAddresses reales:

[,yaml]
----
    ipAddresses:
      inventory-ip: 192.168.122.150
----

Este `+inventory-ip+` será sustituido en el config `+nm-configurator+` siempre que se haya definido `+{inventory-ip}+`.

Observe también que `+MachineInventory+` hace referencia y es propietario de cada `+IPAddressClaim+` asociado a él. Cada reclamación sigue la previsible convención de nomenclatura `+$MachineInventoryName-$IPPoolRefKey+`: `+m-e5331e3b-1e1b-4ce7-b080-235ed9a6d07c-inventory-ip+`.

Estas reclamaciones seguirán el ciclo de vida del objeto `+MachineInventory+` y se eliminarán en cascada, por ejemplo durante el xref:node-operational-tasks/reset.adoc[flujo de trabajo de restablecimiento].

Si el `+IPAddresses+` no puede ser reclamado, la condición `+NetworkConfigReady+` será `+False+`, impidiendo que la máquina complete la instalación. Este puede ser el caso si `+IPPool+` no dispone de más `+IPAddresses+`.

== Del lado de la máquina

Durante la fase de instalación, el proceso `+elemental-register+` que se ejecuta en la máquina recibirá la plantilla de configuración `+nm-configurator+` `+_all.yaml+` y la lista de IPAddresses reclamadas con sus claves. Esta información se digerirá en una configuración aplicable de `+nm-configurator+`:

[,yaml]
----
config:
  dns-resolver:
    config:
      search: []
      server:
      - 192.168.122.1
  interfaces:
  - description: Main-NIC
    ipv4:
      address:
      - ip: "192.168.122.150"
        prefix-length: 24
      dhcp: false
      enabled: true
    ipv6:
      enabled: false
    name: eth0
    state: up
    type: ethernet
  routes:
    config:
    - destination: 0.0.0.0/0
      metric: 150
      next-hop-address: 192.168.122.1
      next-hop-interface: eth0
      table-id: 254
----

A continuación, `+elemental-register+` invocará a `+nmc generate+` y `+nmc apply+` para aplicar esta configuración en el sistema en ejecución.

A partir de este momento y hasta su reinicio, la máquina utilizará siempre la configuración aplicada.

También tenga en cuenta que fuera de la instalación y el reinicio, `+nm-configurator+` ya no se utiliza, ya que el `+elemental-register+` persistirá los archivos `+/etc/NetworkManager/system-connection/*.nmconnection+` generados por `+nmc+` en lugar de la propia configuración `+nmc+`.

Por ejemplo, en cualquier sistema en ejecución, encontrará un archivo de configuración https://github.com/rancher/yip[] yip (`+/oem/elemental-network.yaml+`) para aplicar el `+nmconnections+` deseado, por ejemplo:

[,yaml]
----
name: Apply network config
stages:
    initramfs:
        - files:
            - path: /etc/NetworkManager/system-connections/Wired connection 1.nmconnection
              permissions: 384
              owner: 0
              group: 0
              content: |
                [connection]
                id=Wired connection 1
                uuid=d26b4ae4-d525-3cbf-a557-33feb60343c0
                type=ethernet
                autoconnect-priority=-999
                interface-name=eth0
                timestamp=1722340245

                [ethernet]

                [ipv4]
                address1=192.168.122.150/24
                dhcp-timeout=2147483647
                dns=192.168.122.1;
                dns-options=
                dns-priority=40
                method=manual
                route1=0.0.0.0/0,192.168.122.1,150
                route1_options=table=254

                [ipv6]
                addr-gen-mode=eui64
                dhcp-timeout=2147483647
                method=disabled

                [proxy]

                [user]
                nm-configurator.interface.description=Main-NIC
              encoding: ""
              ownerstring: ""
----

=== Durante el reinicio

Cada vez que se activa xref:node-operational-tasks/reset.adoc[el reinicio], el `+elemental-register+` que se ejecuta en la máquina borrará cualquier archivo `+/etc/NetworkManager/system-connection/*.nmconnection+` y reiniciará la pila de red. La máquina debería entonces volver a DHCP y después de eso confirmar a la `+elemental-operator+` en el lado de gestión, que el reinicio ha tenido éxito.

Tenga en cuenta que esto sólo se aplica cuando el valor de `+MachineInventory.spec.network.configurator+` es diferente de `+none+`. De lo contrario, no se realizará ninguna acción para restablecer la red durante la fase de restablecimiento de la máquina.

Tras el restablecimiento de la red, la máquina debería reiniciarse en modo de recuperación, realizar el restablecimiento real y recibir una nueva configuración de red para aplicarla. Potencialmente esta será la misma que antes (si el `+MachineRegistration+` no ha sido actualizado), o puede tener diferentes IPs ya que las anteriores pueden haber sido reclamadas por otras máquinas mientras tanto.

Si la reversión a DHCP falló o la máquina es de alguna manera incapaz de contactar a la API de Rancher para confirmar, notará que el `+MachineInventory+` no será borrado, a pesar de tener una marca de tiempo de borrado. Como la máquina tiene ahora un problema de red, no será posible arreglarla a distancia.

Tiene la opción de llegar físicamente a la máquina o por cualquier medio arreglar la configuración de red controlada por DHCP, o alternativamente puede quitar el finalizador `+machineinventory.elemental.cattle.io+` del `+MachineInventory+`, para permitir su eliminación, si tiene la intención de retirar la máquina del servicio.
