= Rancher Turtles Solución de problemas
:revdate: 2025-12-17
:page-revdate: {revdate}

Esta guía proporciona pasos para la resolución de problemas en clusters Rancher Turtles y Cluster API (CAPI). Abarca los problemas más comunes y sus soluciones para ayudarle a mantener un entorno estable y funcional.

Antes de leer esta guía, puede consultar la documentación oficial https://cluster-api.sigs.k8s.io/user/troubleshooting[`+troubleshooting+`] del proyecto CAPI.

Comprender los entresijos tanto de Rancher Turtles como de CAPI es crucial para una resolución de problemas eficaz. Esta guía pretende dotarle de los conocimientos necesarios para diagnosticar y resolver problemas básicos en Rancher Turtles y CAPI.

== Requisitos previos

Para solucionar eficazmente problemas de Rancher Turtles y clusters CAPI, necesita:

* Acceso a un terminal con kubectl configurado para acceder al clúster de gestión de Rancher.
* El gestor de plugins Krew para kubectl instalado.
* La herramienta Helm CLI instalada.

Puedes instalar Krew siguiendo las instrucciones de la https://krew.sigs.k8s.io/docs/user-guide/setup/install/[guía de instalación de] Krew.

== Cómo comprobar los registros

Puede ver los siguientes pods y comprobar sus registros en los siguientes espacios de nombres:


|===
| Espacio de nombres | Despliegue | Descripción

| cattle-turtles-system
| rancher-turtles-controller-manager
| Se encarga de la integración con Rancher y gestiona los recursos personalizados para las tortugas.

| fleet-addon-system
| caapf-controller-manager
| Controla el Cluster API Provider Framework para proveedores de infraestructura personalizados

| cattle-capi-system
| capi-controller-manager
| Controlador de la API del clúster principal que concilia los recursos del clúster principal

| rke2-bootstrap-system
| rke2-bootstrap-controller-manager
| Gestiona la configuración de arranque de los nodos RKE2

| rke2-control-plane-system
| rke2-control-plane-controller-manager
| Se encarga de la inicialización y gestión de los componentes del plano de control de RKE2

| *-sistema
| *-controller-manager
| Se encarga de la inicialización y gestión de los componentes del proveedor de infraestructura.

| $-sistema-bootstrap
| $-bootstrap-controller-manager
| Gestiona la configuración bootstrap para proveedores personalizados

| $-control-plane-system
| $-control-plane-controller-manager
| Se encarga de la inicialización y gestión de los componentes del plano de control de los proveedores personalizados.
|===

'*' - dependiendo del proveedor de infraestructura en uso '$' - dependiendo del proveedor de plano de control/bootstrap en uso, por ejemplo kubeadm o rke2

Una forma de supervisar los registros de los pods del plano de control de Cluster API es utilizando `+kubectl logs+` con el selector de etiquetas.

[source,bash]
----
kubectl logs -l control-plane=controller-manager -n cattle-turtles-system -f
kubectl logs -l control-plane=controller-manager -n cattle-capi-system -f
kubectl logs -l control-plane=controller-manager -n rke2-bootstrap-system -f
kubectl logs -l control-plane=controller-manager -n rke2-control-plane-system -f
----

[CAUTION]
====
El uso de `+kubectl krew+` para instalar el plugin de popa implica software de terceros que puede plantear riesgos de seguridad en entornos de producción. Aunque es una herramienta útil para ver los registros, debería utilizarse principalmente en entornos de desarrollo o pruebas. Revise siempre los plugins de terceros antes de instalarlos en clusters sensibles o de producción.
====

El mejor método para supervisar los registros de diferentes pods es utilizar el siguiente método.

[source,bash]
----
kubectl krew install stern
----

A continuación, puede cola registros de todos los pods en tiempo real.

[source,bash]
----
kubectl stern . -n cattle-turtles-system -n rke2-bootstrap-system -n rke2-control-plane-system -n cattle-capi-system
----

o

[source,bash]
----
kubectl stern -A -l control-plane=controller-manager
----

== Cómo gestionar las tortugas ganaderas y los recursos CAPI

*Lista de recursos API de Rancher y Cluster*

Los usuarios a menudo preguntan cómo pueden lidiar con largas listas de diferentes CRDs creadas por CAPI y Rancher, aquí hay algunas ideas de cómo puede lidiar con ello.

. Enumere todas las credenciales relacionadas con CAPI y Rancher.
+
[source,bash]
----
kubectl api-resources | grep 'cattle.io\|cluster.x-k8s.io'
----
+
. Más información sobre el recurso. Puedes obtener una explicación completa de lo que hace o de las opciones de configuración.
+
[source,bash]
----
kubectl api-resources | grep clusterclass
kubectl explain clusterclass.cluster.x-k8s.io
----
+
. A continuación, obtendrá una descripción de cada campo y sección de configuración.
+
[source,bash]
----
kubectl explain clusterclass.spec
kubectl explain clusterclass.spec.controlPlane
kubectl explain clusterclasses.spec.controlPlane.machineInfrastructure
----

*Lista de recursos existentes de objetos API de Rancher y Cluster*

El error más común es utilizar el espacio de nombres `+default+` en Kubernetes para su trabajo. Esto hace que su configuración sea realmente desordenada y difícil de gestionar y solucionar problemas. Recomendamos encarecidamente utilizar espacios de nombres independientes para desplegar y crear clústeres descendentes CAPI y evitar el uso de espacios de nombres del sistema Kubernetes, como `+default+` y `+kube-*+`.

Al aplicar plantillas con `+kubectl+`, especifique siempre `+-n <NAMESPACE>+` para aprovisionar recursos en una ubicación conocida. Puede cambiar entre espacios de nombres en *KUBECONFIG* con este comando:

[source,bash]
----
kubectl config view
kubectl config set-context --current --namespace <NAMESPACE>
----

Con espacios de nombres separados, puede gestionar fácilmente los recursos siguiendo este método.

[CAUTION]
====
El uso de `+kubectl krew+` para instalar el plugin de linaje implica software de terceros que puede plantear riesgos de seguridad en entornos de producción. Aunque es una herramienta útil para ver los registros, debería utilizarse principalmente en entornos de desarrollo o pruebas. Revise siempre los plugins de terceros antes de instalarlos en clusters sensibles o de producción.

== Cómo activar el modo de depuración para los proveedores {product_name} y CAPI

* {product_name} - Para habilitar el modo de depuración para {product_name}, es necesario parchear `+cattle-system/rancher-config+` configmap de la siguiente manera:
+
[source,bash]
----
kubectl patch configmap -n cattle-system rancher-config --type merge --patch '{"data":{"rancher-turtles": "managerArguments:\n  - --v=5\n"}}'
----
(5 equivale a DEBUG)
+
Esto hace que se reinicie el pod de Tortugas, para utilizar la nueva configuración.

* Cluster API Providers - Edite los recursos CAPIProvider para los proveedores en los que sea necesario aumentar el nivel de registro. Cambia al nivel que desees:
+
[source,bash]
----
CAPIProvider.Spec.Manager.Verbosity=5
----
(5 equivale a DEBUG)

== Cómo recoger información de las tortugas ganaderas y el CAPI

https://github.com/crust-gather/crust-gather[crust-gather] es un proyecto creado por desarrolladores de la API Cluster específicamente diseñado para recopilar logs y estados de recursos de entornos CAPI. Se trata de una herramienta segura y oficial que puede utilizarse en cualquier tipo de entorno (producción, pruebas o desarrollo) para recopilar información de diagnóstico exhaustiva para la resolución de problemas.

Puede instalarlo mediante las siguientes instrucciones:

[source,bash]
----
kubectl krew install crust-gather

kubectl crust-gather --help
----

También puede instalarse de forma independiente mediante el script *install.sh*:

[source,bash]
----
curl -sSfL https://github.com/crust-gather/crust-gather/raw/main/install.sh | sh

crust-gather --help
----

Puede especificar una lista de filtros para recopilar datos. Por defecto toma una instantánea completa del cluster. Por defecto, los datos se almacenan en el directorio crust-gather.
Crust-gather acepta una configuración de filtro predefinida para Rancher o el clúster hijo. Vea a continuación ejemplos de configuraciones.

Ejemplo de configuración de filtro YAML para el archivo `+child-crust-gather.yaml+`:

[source,yaml]
----
filters:
  - include_kind:
    - Node
    - Namespace
    - CustomResourceDefinition
  - include_group:
    - management.cattle.io/.*
    - provisioning.cattle.io/.*
  - include_namespace:
    - cattle.*
    - fleet.*
    - kube-system
settings:
  insecure_skip_tls_verify: true
  secrets_file: secrets.txt
----

Ejemplo de configuración de filtro YAML para el archivo `+rancher-crust-gather.yaml+`:

[source,yaml]
----
filters:
  - include_group:
    - .*cluster.x-k8s.io/.*
    - turtles-capi.cattle.io/.*
  - include_kind:
    - CustomResourceDefinition
    - Node
    - Namespace
  - include_group:
    - management.cattle.io/.*
    - provisioning.cattle.io/.*
  - include_namespace:
    - cattle-turtles-system
    - capi-.*
    - cattle.*
    - fleet.*
    - c-.*
    - rke2.*
    - cert-manager
    - kube-system
settings:
  insecure_skip_tls_verify: true
  secrets_file: secrets.txt
----

Asegúrese de que kubeconfig apunta al clúster correcto y ejecute el siguiente comando:

[source,bash]
----
kubectl crust-gather collect-from-config -c config.yaml
----

Utilización mediante banderas regulares:

[source,bash]
----
kubectl crust-gather collect --include-namespace cattle-turtles-system --include-namespace capi-* --include-namespace cattle* --include-namespace c-* --include-namespace=<any-capi-cluster-namespace> --kubeconfig=<KUBECONFIG>
----

Puede especificar un archivo con secretos o variables de entorno con cadenas de secretos a excluir. 

Por ejemplo:

[source,bash]
----
kubectl crust-gather collect -s ENV_WITH_SECRET --secrets-file=secrets.txt
----

O excluir todos los recursos secretos de la recogida: 

[source,bash]
----
kubectl crust-gather collect --exclude-kind Secret
----

== Cómo limpiar los recursos de Rancher Turtles y CAPI

A veces, la limpieza de su infraestructura puede fallar y puede provocar recursos pendientes. En esta situación, tiene que eliminar los recursos manualmente.

Tenga en cuenta que para eliminar los finalizadores manualmente **es necesario** que el proveedor de infraestructura limpie manualmente los recursos aprovisionados.

[source,bash]
----
export NAMESPACE=capi-clusters

for RESOURCE in `+kubectl get-all -n $NAMESPACE -o name | grep 'cattle.io\|cluster.x-k8s.io'+`; 
do 
        echo "Patching $RESOURCE in namespace $NAMESPACE";
        kubectl patch $RESOURCE -n $NAMESPACE -p '{"metadata":{"finalizers":[]}}' --type=merge;
        kubectl delete $RESOURCE -n $NAMESPACE;
done
----

== Cómo desinstalar Rancher Turtles y el proyecto CAPI

Para desinstalar Rancher Turtles y los componentes CAPI de su clúster de gestión, siga estos pasos en orden:

. En primer lugar, elimine todos los clusters posteriores creados con CAPI. Para cada agrupación:
+
[source,bash]
----
kubectl delete -n capi-clusters cluster.cluster.x-k8s.io cluster1
----
+
[NOTE]
====
Sustituya `+capi-clusters+` por el espacio de nombres donde están desplegados sus clusters y `+cluster1+` por el nombre de su cluster. Espere a que cada clúster se elimine por completo antes de proceder al siguiente paso.
====

. Desinstala el cuadro de mando de Rancher Turtles:
+
[source,bash]
----
helm uninstall -n cattle-turtles-system rancher-turtles
----

. Elimine cualquier configuración de webhook que pueda haber sido creada por proveedores CAPI:
+
[source,bash]
----
# List all webhook configurations
kubectl get validatingwebhookconfigurations.admissionregistration.k8s.io

# Delete provider-specific webhooks (examples)
kubectl delete validatingwebhookconfigurations.admissionregistration.k8s.io azureserviceoperator-validating-webhook-configuration
kubectl delete validatingwebhookconfigurations.admissionregistration.k8s.io capz-validating-webhook-configuration
kubectl delete validatingwebhookconfigurations.admissionregistration.k8s.io capi-validating-webhook-configuration
kubectl delete validatingwebhookconfigurations.admissionregistration.k8s.io rke2-webhook-configuration

# Also check and delete mutating webhooks if present
kubectl get mutatingwebhookconfigurations
kubectl delete mutatingwebhookconfigurations [webhook-name]
----

. Limpia los espacios de nombres y recursos sobrantes. Los siguientes espacios de nombres pueden permanecer después de la desinstalación:
   * cattle-turtles-system
   * rke2-bootstrap-system
   * rke2-control-plane-system
   * capi-system
   * capz-system (u otros espacios de nombres específicos de proveedores como capv-system, capa-system, etc.)
   * capi-clusters (u otros espacios de nombres en los que haya desplegado clusters)
+
Para eliminar estos espacios de nombres:
+
[source,bash]
----
# First remove any finalizers that might be blocking deletion
for NS in cattle-turtles-system rke2-bootstrap-system rke2-control-plane-system capi-system capz-system capi-clusters; do
  kubectl get namespace $NS -o json | jq '.spec.finalizers = []' | kubectl replace --raw "/api/v1/namespaces/$NS/finalize" -f -
done

# Then delete the namespaces
kubectl delete namespace cattle-turtles-system rke2-bootstrap-system rke2-control-plane-system capi-system capz-system capi-clusters
----

. Por último, elimine los CRD relacionados con Cluster API y Rancher Turtles:
+
[source,bash]
----
# Delete all Cluster API and Rancher Turtles CRDs
kubectl get crds | grep 'cluster.x-k8s.io\|turtles-capi.cattle.io' | awk '{print $1}' | xargs kubectl delete crd

# Or manually delete them one by one
kubectl delete crd clusters.cluster.x-k8s.io
kubectl delete crd clusterclasses.cluster.x-k8s.io
kubectl delete crd machines.cluster.x-k8s.io
kubectl delete crd machinepools.cluster.x-k8s.io
kubectl delete crd providers.turtles-capi.cattle.io
kubectl delete crd clusterconfigs.turtles-capi.cattle.io
# ... and other related CRDs
----
+
[CAUTION]
====
Asegúrate de que todos los clusters están completamente borrados antes de eliminar estos CRDs, o puedes dejar recursos en la nube huérfanos.
====
