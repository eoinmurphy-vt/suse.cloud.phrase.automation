== Descripciﾃｳn general

SUSE Observability ya proporciona muchos grﾃ｡ficos mﾃｩtricos por defecto en la mayorﾃｭa de los tipos de componentes que representan recursos de Kubernetes. . Las mﾃｩtricas ya son recopiladas por SUSE Observability pero no se visualizan en un componente por defecto.
. Las mﾃｩtricas aﾃｺn no son recopiladas por SUSE Observability en absoluto y, por lo tanto, no estﾃ｡n disponibles aﾃｺn.

Para la opciﾃｳn 1, los pasos a continuaciﾃｳn te indicarﾃ｡n cﾃｳmo crear un enlace de mﾃｩtrica que configurarﾃ｡ SUSE Observability para aﾃｱadir una mﾃｩtrica especﾃｭfica a un conjunto especﾃｭfico de componentes.

== Escribe el esquema del enlace de mﾃｩtrica

Crea un nuevo archivo YAML llamado `+metric-bindings.yaml+` y aﾃｱade esta plantilla YAML para crear tu propio enlace de mﾃｩtrica. ﾃ｜relo en tu editor de cﾃｳdigo favorito para modificarlo a lo largo de esta guﾃｭa. Al final, se utilizarﾃ｡ la CLI de SUSE Observability para crear y actualizar los enlaces de mﾃｩtrica en SUSE Observability.

----
nodes:
- _type: MetricBinding
  chartType: line
  enabled: true
  tags: {}
  unit:
  name:
  description:
  priority:
  identifier: urn:custom:metric-binding:...
  queries:
    - expression:
      alias:
  scope:
  layout:
    metricPerspective:
      tab:
      section:
      weight:
    componentSummary:
      weight:
----

Los campos en esta plantilla son:

* `+_type+`: SUSE Observability necesita saber que esto es un enlace mﾃｩtrico, por lo que el valor siempre debe ser `+MetricBinding+`.
* `+chartType+`: SUSE Observability soportarﾃ｡ diferentes tipos de grﾃ｡ficos (`+line+`, `+bar+`, etc.), actualmente solo se soporta `+line+`.
* `+enabled+`: Establecer en `+false+` para mantener el enlace mﾃｩtrico pero no mostrarlo a los usuarios.
* `+tags+`: Se utilizarﾃ｡ para organizar mﾃｩtricas en la interfaz de usuario, se puede dejar vacﾃｭo usando `+{}+`.
* `+unit+`: La unidad de los valores en la serie temporal devuelta por la consulta o consultas, utilizada para renderizar el eje Y del grﾃ｡fico. Consulta la referencia de xref:/develop/reference/k8sTs-chart-units.adoc[unidades soportadas] para todas las unidades.
* `+name+`: El nombre del enlace mﾃｩtrico.
* `+description+`: Descripciﾃｳn opcional, mostrada al pasar el ratﾃｳn sobre el nombre.
* `+priority+`: [Obsoleto] Uno de `+HIGH+`, `+MEDIUM+` o `+LOW+`. Orden principal de clasificaciﾃｳn para mﾃｩtricas en un componente (en el orden en que se mencionan aquﾃｭ), el orden secundario de clasificaciﾃｳn es el `+name+`.
* `+identifier+`: Un URN (identificador de recurso universal), utilizado como el identificador ﾃｺnico del enlace mﾃｩtrico. Debe comenzar con `+urn:custom:metric-binding:+`, el resto es de formato libre siempre que sea ﾃｺnico entre todos los enlaces mﾃｩtricos.
* `+queries+`: Una lista de consultas para mostrar en el grﾃ｡fico para el enlace mﾃｩtrico (ver tambiﾃｩn las secciones siguientes).
* `+scope+`: El ﾃ｡mbito de topologﾃｭa del enlace mﾃｩtrico, una consulta de topologﾃｭa que selecciona los componentes en los que se mostrarﾃ｡ este enlace mﾃｩtrico.
* `+layout+`: Cﾃｳmo agrupar grﾃ｡ficos en diferentes vistas de perspectiva, por ejemplo, en xref:/use/views/k8s-metrics-perspective.adoc[perspectiva de mﾃｩtricas].

Rellena primero todas las partes ya conocidas (con los recuentos de rﾃｩplicas de ampliaciﾃｳn como ejemplo).

----
_type: MetricBinding
chartType: line
enabled: true
tags: {}
unit: short
name: Replica counts
priority: MEDIUM
identifier: urn:custom:metric-binding:my-deployment-replica-counts
queries:
  - expression:
    alias:
scope:
----

La secciﾃｳn de consultas y ﾃ｡mbito se completarﾃ｡ en los siguientes pasos. Ten en cuenta que la unidad utilizada es `+short+`, que simplemente renderizarﾃ｡ un valor numﾃｩrico. En caso de que aﾃｺn no estﾃｩs seguro sobre la unidad de la mﾃｩtrica, puedes dejarlo abierto y decidir la unidad correcta al escribir la consulta PromQL.

== Escribe la consulta de topologﾃｭa

Utiliza la vista Explorar de la xref:/use/views/k8s-topology-perspective.adoc[perspectiva de Topologﾃｭa], http://your-instance/#/views/explore, y selecciona los componentes que necesitan mostrar la nueva mﾃｩtrica. Se pueden utilizar tanto las vistas bﾃ｡sica como avanzada para hacer la selecciﾃｳn. Los campos mﾃ｡s comunes para seleccionar la topologﾃｭa en los enlaces mﾃｩtricos son `+type+` para el tipo de componente y `+label+` para seleccionar todas las etiquetas. Por ejemplo, para las ampliaciones:

----
type = "deployment" and label = "stackpack:kubernetes"
----

El filtro de tipo selecciona todas las ampliaciones, mientras que el filtro de etiquetas selecciona solo los componentes creados por el stackpack de Kubernetes (el nombre de la etiqueta es `+stackpack+` y el valor de la etiqueta es `+kubernetes+`). Este ﾃｺltimo tambiﾃｩn se puede omitir para obtener el mismo resultado.

Cambia al modo avanzado para copiar la consulta de topologﾃｭa resultante y ponerla en el campo `+scope+` del enlace mﾃｩtrico.

[NOTE]
====
Los enlaces mﾃｩtricos solo admiten los filtros de consulta, las funciones de consulta como `+withNeighborsOf+` no son compatibles y no se pueden utilizar.
====


== Escribe la consulta PromQL

Ve al xref:/use/metrics/k8sTs-explore-metrics.adoc[explorador de mﾃｩtricas] de tu instancia de SUSE Observability, http://your-instance/#/metrics, y utilﾃｭzalo para consultar la mﾃｩtrica de interﾃｩs. El explorador tiene autocompletado para mﾃｩtricas, etiquetas, valores de etiquetas, pero tambiﾃｩn funciones PromQL y operadores para ayudarte. Comienza con un rango de tiempo corto de, por ejemplo, una hora para obtener los mejores resultados.

Para el nﾃｺmero total de rﾃｩplicas utiliza la mﾃｩtrica `+kubernetes_state_deployment_replicas+`. Para hacer que los grﾃ｡ficos mostrados para esta mﾃｩtrica sean representativos de los datos de series temporales, extiende la consulta para hacer una agregaciﾃｳn utilizando el parﾃ｡metro `+${__interval}+`:

----
max_over_time(kubernetes_state_deployment_replicas[${__interval}])
----

En este caso especﾃｭfico, utiliza `+max_over_time+` para asegurarte de que el grﾃ｡fico siempre muestre el mayor nﾃｺmero de rﾃｩplicas en cualquier momento dado. Para rangos de tiempo mﾃ｡s largos, esto significa que una pequeﾃｱa caﾃｭda en las rﾃｩplicas no se mostrarﾃ｡; para enfatizar el nﾃｺmero mﾃ｡s bajo de rﾃｩplicas, utiliza `+min_over_time+` en su lugar.

Copia la consulta en la propiedad `+expression+` de la primera entrada en el campo `+queries+` del enlace mﾃｩtrico. Utiliza `+Total replicas+` como un alias. Este es el nombre que aparecerﾃ｡ en la leyenda del grﾃ｡fico.

[NOTE]
====
En SUSE Observability, el tamaﾃｱo del grﾃ｡fico de mﾃｩtricas determina automﾃ｡ticamente la granularidad de la mﾃｩtrica mostrada en el grﾃ｡fico. Las consultas de PromQL se pueden ajustar para hacer un uso ﾃｳptimo de este comportamiento y obtener un grﾃ｡fico representativo de la mﾃｩtrica. xref:/use/metrics/k8s-writing-promql-for-charts.adoc[Escribir PromQL para grﾃ｡ficos] explica esto en detalle.
====


== Vincula la serie temporal correcta a cada componente.

El enlace de mﾃｩtrica con todos los campos completados:

----
_type: MetricBinding
chartType: line
enabled: true
tags: {}
unit: short
name: Replica counts
priority: MEDIUM
identifier: urn:custom:metric-binding:my-deployment-replica-counts
queries:
  - expression: max_over_time(kubernetes_state_deployment_replicas[${__interval}])
    alias: Total replicas
scope: type = "deployment" and label = "stackpack:kubernetes"
----

Crearlo en SUSE Observability y ver el grﾃ｡fico de "Conteo de rﾃｩplicas" en un componente de ampliaciﾃｳn da un resultado inesperado. El grﾃ｡fico muestra los conteos de rﾃｩplicas para todas las ampliaciones. Lﾃｳgicamente, uno esperarﾃｭa solo 1 serie temporal: el conteo de rﾃｩplicas para esta ampliaciﾃｳn especﾃｭfica.

image::k8s/k8s-replica-counts-without-binding.png[El grﾃ｡fico incorrecto para una sola ampliaciﾃｳn, it shows the replica count for all deployments]

Para solucionar esto, haz que la consulta de PromQL sea especﾃｭfica para un componente utilizando informaciﾃｳn del componente. Filtra en suficientes etiquetas de mﾃｩtricas para seleccionar solo la serie temporal especﾃｭfica para el componente. Esta es la "vinculaciﾃｳn" de la serie temporal correcta al componente. Para cualquiera con experiencia en la creaciﾃｳn de paneles de Grafana, esto es similar a un panel con parﾃ｡metros que se utilizan en las consultas del panel. Cambiemos la consulta en el enlace de mﾃｩtrica a esto:

----
max_over_time(kubernetes_state_deployment_replicas{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", deployment="${name}"}[${__interval}])
----

image::k8s/k8s-replica-counts-with-binding.png[Despuﾃｩs de aﾃｱadir los filtros parametrizados, el grﾃ｡fico resultante se ve como se esperaba, only 1 time series for this component]

La consulta de PromQL ahora filtra en 3 etiquetas, `+cluster_name+`, `+namespace+` y `+deployment+`. En lugar de especificar un valor real para estas etiquetas, se utiliza una referencia variable a los campos del componente. En este caso, se utilizan las etiquetas `+cluster-name+` y `+namespace+`, referenciadas usando `+${tags.cluster-name}+` y `+${tags.namespace}+`. Ademﾃ｡s, el nombre del componente se referencia con `+${name}+`.

Las referencias de variable soportadas son:

* Cualquier etiqueta de componente, usando `+${tags.<label-name>}+`
* El nombre del componente, usando `+${name}+`

image::k8s/k8s-carts-highlights.png[Pﾃ｡gina de Destacados del Componente que muestra las etiquetas y el nombre del componente (ambos destacados en rojo)]

[NOTE]
====
El nombre del clﾃｺster, el espacio de nombres y una combinaciﾃｳn del tipo y nombre del componente suelen ser suficientes para seleccionar las mﾃｩtricas de un componente especﾃｭfico de Kubernetes. Estas etiquetas, o etiquetas similares, suelen estar disponibles en la mayorﾃｭa de mﾃｩtricas y componentes.
====


== Crea o actualiza el enlace de mﾃｩtrica en SUSE Observability

Usa la SUSE Observability CLI para crear el enlace de mﾃｩtrica en SUSE Observability. Asegﾃｺrate de que el `+metric-bindings.yaml+` estﾃｩ guardado y se vea asﾃｭ:

----
nodes:
- _type: MetricBinding
  chartType: line
  enabled: true
  tags: {}
  unit: short
  name: Replica counts
  priority: MEDIUM
  identifier: urn:custom:metric-binding:my-deployment-replica-counts
  queries:
    - expression: max_over_time(kubernetes_state_deployment_replicas{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", deployment="${name}"}[${__interval}])
      alias: Total replicas
  scope: type = "deployment" and label = "stackpack:kubernetes"
----

Usa la xref:/setup/cli/cli-sts.adoc[SUSE Observability CLI] para crear el enlace de mﾃｩtricas:

[,bash]
----
sts settings apply -f metric-bindings.yaml
----

Verifica los resultados en SUSE Observability abriendo la perspectiva de mﾃｩtricas para una ampliaciﾃｳn. Si no estﾃ｡s satisfecho con el resultado, simplemente cambia el enlace de mﾃｩtricas en el archivo YAML y ejecuta el comando nuevamente para actualizarlo. La lista de nodos admite agregar muchos enlaces de mﾃｩtricas. Simplemente aﾃｱade otra entrada de enlace de mﾃｩtricas al array YAML usando los mismos pasos que antes.

[CAUTION]
====
El identificador se utiliza como la clave ﾃｺnica de un enlace de mﾃｩtricas. Cambiar el identificador crearﾃ｡ un nuevo enlace de mﾃｩtricas en lugar de actualizar el existente.
====


El comando `+sts settings+` tiene mﾃ｡s opciones; por ejemplo, puede listar todos los enlaces de mﾃｩtrica:

[,bash]
----
sts settings list --type MetricBinding
----

Finalmente, para eliminar un enlace de mﾃｩtricas, utiliza

[,bash]
----
sts settings delete --ids <id>
----

El `+<id>+` en este comando no es el identificador, sino el nﾃｺmero en la columna `+Id+` de la salida `+sts settings list+`.

[NOTE]
====
La forma recomendada de trabajar es almacenar los enlaces de mﾃｩtrica (y cualquier otro recurso personalizado creado en SUSE Observability) como archivos YAML en un repositorio de Git. Desde allﾃｭ, los cambios pueden aplicarse manualmente o puede ser totalmente automatizado utilizando la CLI de SUSE Observability en un sistema CI/CD como GitHub actions o GitLab pipelines.
====


== Otras opciones

=== Mﾃ｡s de 1 series temporales en un grﾃ｡fico

[NOTE]
====
Solo hay 1 unidad para un enlace de mﾃｩtricas (se representa en el eje y del grﾃ｡fico). Como resultado, solo deberﾃｭas combinar consultas que produzcan series temporales con la misma unidad en 1 enlace de mﾃｩtricas. A veces puede ser posible convertir la unidad. Por ejemplo, el uso de CPU puede ser reportado en milicores o nﾃｺcleos; los milicores se pueden convertir a nﾃｺcleos multiplicando por 1000 asﾃｭ: `+(<original-query>) * 1000+`.
====


Hay 2 formas de obtener mﾃ｡s de 1 series temporales en un ﾃｺnico enlace de mﾃｩtricas y, por lo tanto, en un ﾃｺnico grﾃ｡fico:

. Escribe una consulta PromQL que devuelva mﾃｺltiples series temporales para un ﾃｺnico componente
. Aﾃｱade mﾃ｡s consultas PromQL al enlace de mﾃｩtricas

Para la primera opciﾃｳn se da un ejemplo en la xref:/use/metrics/k8s-add-charts.adoc#_using_metric_labels_in_aliases[prﾃｳxima secciﾃｳn]. La segunda opciﾃｳn puede ser ﾃｺtil para comparar mﾃｩtricas relacionadas. Algunos casos de uso tﾃｭpicos:

* Comparando rﾃｩplicas totales frente a las deseadas y disponibles
* Uso de recursos: lﾃｭmites, solicitudes y uso en un solo grﾃ｡fico

Para aﾃｱadir mﾃ｡s consultas a un enlace de mﾃｩtricas simplemente repite xref:/use/metrics/k8s-add-charts.adoc#_steps[pasos] 3. y 4. y aﾃｱade la consulta como una entrada adicional en la lista de consultas. Para los recuentos de rﾃｩplicas de ampliaciﾃｳn, hay varias mﾃｩtricas relacionadas que se pueden incluir en el mismo grﾃ｡fico:

----
nodes:
- _type: MetricBinding
  chartType: line
  enabled: true
  tags: {}
  unit: short
  name: Replica counts
  priority: MEDIUM
  identifier: urn:custom:metric-binding:my-deployment-replica-counts
  queries:
    - expression: max_over_time(kubernetes_state_deployment_replicas{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", deployment="${name}"}[${__interval}])
      alias: Total replicas
    - expression: max_over_time(kubernetes_state_deployment_replicas_available{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}",  deployment="${name}"}[${__interval}])
      alias: Available - ${cluster_name} - ${namespace} - ${deployment}
    - expression: max_over_time(kubernetes_state_deployment_replicas_unavailable{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}",  deployment="${name}"}[${__interval}])
      alias: Unavailable - ${cluster_name} - ${namespace} - ${deployment}
    - expression: min_over_time(kubernetes_state_deployment_replicas_desired{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}",  deployment="${name}"}[${__interval}])
      alias: Desired - ${cluster_name} - ${namespace} - ${deployment}
  scope: type = "deployment" and label = "stackpack:kubernetes"
----

=== Usando etiquetas de mﾃｩtrica en alias

Cuando una ﾃｺnica consulta devuelve mﾃｺltiples series temporales por componente, esto se mostrarﾃ｡ como mﾃｺltiples lﾃｭneas en el grﾃ｡fico. Pero en la leyenda, todas usarﾃ｡n el mismo alias. Para poder ver la diferencia entre las diferentes series temporales, el alias puede incluir referencias a las etiquetas de mﾃｩtrica usando la sintaxis `+${label}+`. Por ejemplo, aquﾃｭ hay un enlace de mﾃｩtricas para la mﾃｩtrica "Reinicios de contenedor" en un pod, ten en cuenta que un pod puede tener mﾃｺltiples contenedores:

----
type: MetricBinding
chartType: line
enabled: true
id: -1
identifier: urn:custom:metric-binding:my-pod-restart-count
name: Container restarts
priority: MEDIUM
queries:
- alias: Restarts - ${container}
  expression: max by (cluster_name, namespace, pod_name, container) (kubernetes_state_container_restarts{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", pod_name="${name}"})
scope: (label = "stackpack:kubernetes" and type = "pod")
unit: short
----

Ten en cuenta que el `+alias+` hace referencia a la etiqueta `+container+` de la mﾃｩtrica. Asegﾃｺrate de que la etiqueta estﾃｩ presente en el resultado de la consulta; cuando falta la etiqueta, el `+${container}+` se renderizarﾃ｡ como texto literal para ayudar en la resoluciﾃｳn de problemas.

=== Distribuciones

Cada componente puede estar asociado con varias tecnologﾃｭas o protocolos como k8s, redes, entornos de tiempo de ejecuciﾃｳn (por ejemplo, JVM), protocolos (HTTP, AMQP), etc.
En consecuencia, se puede mostrar una multitud de mﾃｩtricas diferentes para cada componente. Para una lectura mﾃ｡s fﾃ｡cil, SUSE Observability puede organizar estos grﾃ｡ficos en pestaﾃｱas y secciones.
Para mostrar un grﾃ｡fico (`+MetricBinding+`) dentro de una pestaﾃｱa o secciﾃｳn especﾃｭfica, necesitas configurar la propiedad de diseﾃｱo.
Cualquier MetricsBinding sin un diseﾃｱo especificado se mostrarﾃ｡ en una pestaﾃｱa y secciﾃｳn llamada `+Other+`.

Aquﾃｭ hay un ejemplo de configuraciﾃｳn:

----
layout:
  metricPerspective:
    tab: Kubernetes Pod
    section: Performance
    weight: 2
  componentSummary:
    weight: 2
----

Campos:

* `+layout.metricPerspective+` - Define las mﾃｩtricas a mostrar en `+Metrics Perspective+`. Las mﾃｩtricas se agrupan en pestaﾃｱas y luego en secciones.
* `+layout.metricPerspective.tab+` - Nombre de la pestaﾃｱa. Las pestaﾃｱas se ordenan alfabﾃｩticamente.
* `+layout.metricPerspective.section+` - Nombre de la secciﾃｳn. Las secciones se ordenan alfabﾃｩticamente.
* `+layout.metricPerspective.weight+` - Las mﾃｩtricas dentro de una secciﾃｳn se ordenan principalmente por peso (ascendente) y secundariamente por nombre (alfabﾃｩtico).
* `+layout.componentSummary+` - Especifica las mﾃｩtricas a mostrar en la barra lateral `+Components details+` al seleccionar el componente. Los grﾃ｡ficos aparecen solo cuando esta propiedad estﾃ｡ definida.
* `+layout.componentSummary.weight+` - Esto representa el peso del grﾃ｡fico. Los grﾃ｡ficos se ordenan en orden ascendente por peso y luego se muestran los primeros 3ﾂgrﾃ｡ficos.
