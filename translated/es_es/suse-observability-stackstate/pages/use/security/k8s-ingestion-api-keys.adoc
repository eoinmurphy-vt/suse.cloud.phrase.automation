= Claves API (obsoletas)
:revdate: 2025-07-22
:page-revdate: {revdate}
:description: Observabilidad de SUSE

Para enviar los datos telemétricos a {stackstate-product-name} se utilizó una clave API para autenticarse en {stackstate-product-name}. En una instalación del agente Helm, esto se hacía estableciendo la variable helm `+stackstate.apiKey+`.

Las claves API y las claves API de ingestión están obsoletas y se eliminarán en el cuarto trimestre de 2026. Los tokens de servicio son ahora el mecanismo de autenticación preferido, también para el envío de datos telemétricos. 

El xref:/k8s-suse-rancher-prime.adoc#_installing_the_suse_observability_agent[flujo] para añadir nuevos clústeres a {stackstate-product-name} en el xref:/k8s-suse-rancher-prime.adoc#_installing_the_suse_observability_agent[flujo de instalación de StackPack] incluye la creación de un token específico de clúster compatible con el Agente RBAC para Rancher RBAC.

== Migración de la clave API al token de servicio

Existen 2 opciones para migrar del uso de la clave única de API al uso de tokens de servicio para los datos de telemetría. 

1. Migrar a un token de servicio por clúster. Esto es un poco más de trabajo para configurar, pero es compatible con Rancher RBAC.
2. Migra a un único Service Token que utilices desde todos los clústeres Kubernetes y recolectores OTel. Esto no es compatible con Rancher RBAC.

Después de migrar todos los agentes y colectores de Open Telemetry a tokens de servicio ahora puede eliminar los valores `+global.receiverApiKey+` o `+stackstate.apiKey+` de Helm para instalar {stackstate-product-name} y ejecutar un `+helm upgrade+` para aplicar los cambios.

=== Migrar a un token de servicio por clúster

1. Abra la mochila de Kubernetes en la interfaz de usuario
2. Amplíe el clúster que va a migrar
3. Haga clic en el botón "Crear nuevo token de servicio" para obtener el token de servicio específico del clúster y copie el token
4. Siga las instrucciones de instalación/actualización del agente del stackpack para volver a desplegar el agente en el clúster con el token de servicio como clave API.

En lugar de utilizar la interfaz de usuario también puede generar el token con la CLI como xref:setup/security/authentication/service_tokens#_telemetry_data_ingestion[se documenta aquí]. Asegúrese también de asignar el `+update-scoped-permissions+` scoped al cluster que está migrando. Los demás pasos siguen siendo los mismos que para la interfaz de usuario.

=== Migrar a un único token de servicio

Generar un token de servicio con la CLI como xref:setup/security/authentication/service_tokens#_telemetry_data_ingestion[se documenta aquí].:

[,sh]
----
> sts rbac create-subject --subject my-agent
✅ Created subject 'my-agent'
> sts rbac grant --subject my-agent --permission update-metrics
✅ Granted permission 'update-metrics' on 'system' to subject 'my-agent'
PERMISSION   | RESOURCE
update-metrics | system
----

No añada un permiso `+update-scoped-permissions+` ya que no funcionará para un token que se utilice desde varios clusters.

Vuelva a desplegar los agentes y los recopiladores xref:setup/otel/getting-started/getting-started-k8s.adoc[de telemetría] abierta siguiendo las instrucciones disponibles en la interfaz de usuario de Kubernetes StackPack y la xref:setup/otel/getting-started/getting-started-k8s.adoc[documentación del recopilador de telemetría] abierta. Utilice el token de servicio generado como clave de API.

== Migración de claves API de ingestión

Las claves API de ingestión fueron utilizadas por herramientas externas para ingestar datos (como métricas, eventos y trazas) en el clúster de SUSE Observability.
Estas herramientas incluyen el Agente STS y/o el Colector OTel. Las fichas de servicio tienen las mismas capacidades, y más. Siga los mismos pasos que para la clave API (en la sección anterior) para migrar a <<_migrate_to_a_service_token_per_cluster, un token de servicio por clúster>> o a <<_migrate_to_a_single_service_token, un token de servicio único>>.
