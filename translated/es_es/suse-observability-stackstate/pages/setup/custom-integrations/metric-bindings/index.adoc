= Añadir gráficos personalizados a los componentes
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: Observabilidad de SUSE

== Visión general

{stackstate-product-name} proporciona ya muchos gráficos de métricas por defecto en la mayoría de los tipos de componentes que representan recursos de Kubernetes. Se pueden añadir gráficos métricos adicionales a cualquier conjunto de componentes siempre que sea necesario. Al añadir métricas a los componentes hay dos opciones:

. Las métricas ya están recogidas por {stackstate-product-name} pero no se visualizan en un componente, por defecto
. {stackstate-product-name} aún no recoge ninguna métrica, por lo que todavía no están disponibles.

Para la opción 1, los pasos siguientes le indicarán cómo crear un enlace métrico que configurará {stackstate-product-name} para añadir una métrica específica a un conjunto específico de componentes.

Para la opción 2, asegúrese de que las métricas están disponibles en {stackstate-product-name} enviándolas a {stackstate-product-name} mediante el xref:/use/metrics/k8s-prometheus-remote-write.adoc[protocolo de escritura remota de Prometheus]. Continúe añadiendo gráficos para las métricas a los componentes SÓLO después de asegurarse de que las métricas están disponibles.

== Creación de un enlace métrico

. <<_create_an_outline_of_the_metric_binding,Crear un esquema del encuadernado métrico>>
. <<_select_the_components_to_bind_to,Seleccione los componentes que desea vincular>>
. <<_write_the_promql_query,Escriba la consulta PromQL para la métrica deseada>>
. <<_bind_the_correct_time_series_to_each_component,Vincular las series temporales correctas a cada componente>>

Por ejemplo, los pasos añadirán una métrica vinculante para `+Replica counts+` de despliegues de Kubernetes. Este enlace métrico ya existe en {stackstate-product-name}, por defecto.

=== Crear un esquema del encuadernado métrico

Abra el archivo YAML `+metricbindings.yaml+` en su editor de código favorito para modificarlo a lo largo de esta guía.  Puede utilizar la CLI para xref:/setup/custom-integrations/develop.adoc#_test_your_stackpack[probar su StackPack].

----
- _type: MetricBinding
  chartType: line
  enabled: true
  identifier: urn:stackpack:my-stackpack:shared:metric-binding:node-memory-bytes-available-scheduling
  layout:
    metricPerspective:
      section: Resources
      tab: Kubernetes Node
  name: Memory available for scheduling (Custom)
  priority: medium
  queries:
  - alias: ${cluster_name} - ${node}
    expression: max_over_time(kubernetes_state_node_memory_allocatable{cluster_name="${tags.cluster-name}", node="${name}"}[${__interval}])
  scope: (label = "stackpack:kubernetes" and type = "node")
  unit: bytes(IEC)
----

La sección de consultas y ámbito de aplicación se rellenará en los próximos pasos. Tenga en cuenta que la unidad utilizada es `+short+`, que simplemente mostrará un valor numérico. En caso de que aún no estés seguro de la unidad de la métrica, puedes dejarla abierta y decidir la unidad correcta al escribir la consulta PromQL.

=== Seleccione los componentes que desea vincular

Guarde una vista de la xref:/use/views/k8s-topology-perspective.adoc[perspectiva Topología] y utilice los filtros (Filtros -> Topología -> Cambiar a STQL) para consultar los componentes que deben mostrar la nueva métrica. Los campos más comunes para seleccionar la topología en los enlaces métricos son `+type+` para el tipo de componente y `+label+` para seleccionar todas las etiquetas. Por ejemplo, para las implantaciones:

----
type = "deployment" and label = "stackpack:kubernetes"
----

El filtro de tipo selecciona todos los despliegues, mientras que el filtro de etiqueta selecciona únicamente los componentes creados por el stackpack Kubernetes (el nombre de la etiqueta es `+stackpack+` y el valor de la etiqueta es `+kubernetes+`). Este último también puede omitirse para obtener el mismo resultado.  Todos los xref:/develop/reference/k8sTs-stql_reference.adoc#_component_filters[filtros de componentes de consulta STQL] se pueden utilizar para filtrar.

Cambie al modo avanzado para copiar la consulta de topología resultante y ponerla en el campo `+scope+` del enlace métrico.

[NOTE]
====
Los enlaces métricos sólo admiten los filtros de consulta. Las funciones de consulta como `+withNeighborsOf+` no son compatibles y no pueden utilizarse.
====


=== Escriba la consulta PromQL

Vaya al xref:/use/metrics/k8sTs-explore-metrics.adoc[explorador de métricas] de su instancia {stackstate-product-name}, http://your-instance/#/metrics, y utilícelo para consultar la métrica de interés. El explorador tiene autocompletado para métricas, etiquetas, valores de etiquetas, pero también funciones PromQL y operadores para ayudarte. Comience con un intervalo de tiempo corto de, por ejemplo, una hora para obtener los mejores resultados.

Para el número total de réplicas, utilice la métrica `+kubernetes_state_deployment_replicas+`. Para mostrar los gráficos de métricas de los datos de series temporales, amplíe la consulta para realizar una agregación utilizando el parámetro `+${__interval}+`:

----
max_over_time(kubernetes_state_deployment_replicas[${__interval}])
----

En este caso concreto, utilice `+max_over_time+` para asegurarse de que el gráfico muestra siempre el mayor número de réplicas en un momento dado. Para intervalos de tiempo más largos, no se muestra un breve descenso de las réplicas. Para destacar el menor número de réplicas, utilice `+min_over_time+` en su lugar.

Copie la consulta en la propiedad `+expression+` de la primera entrada del campo `+queries+` del enlace métrico. Utilice `+Total replicas+` como alias. para que aparezca en la leyenda del gráfico.

[NOTE]
====
En {stackstate-product-name}, el tamaño del gráfico métrico determina automáticamente la granularidad de la métrica mostrada en el gráfico. Las consultas PromQL pueden ajustarse para hacer un uso óptimo de este comportamiento y obtener un gráfico representativo de la métrica. xref:/setup/custom-integrations/metric-bindings/writing-promql.adoc[Escribir PromQL para gráficos] explica esto en detalle.
====


=== Vincular las series temporales correctas a cada componente

El enlace métrico con todos los campos rellenados:

----
_type: MetricBinding
chartType: line
enabled: true
tags: {}
unit: short
name: Replica counts
priority: MEDIUM
identifier: urn:stackpack:my-stackpack:metric-binding:my-deployment-replica-counts
queries:
  - expression: max_over_time(kubernetes_state_deployment_replicas[${__interval}])
    alias: Total replicas
scope: type = "deployment" and label = "stackpack:kubernetes"
----

Si se crea en {stackstate-product-name} y se visualiza el gráfico "Replica count" en un componente de despliegue, se obtiene un resultado inesperado. El gráfico muestra los recuentos de réplicas de todas las implantaciones. Lógicamente, sólo cabría esperar una serie temporal: el recuento de réplicas para este despliegue específico.

image::k8s/k8s-replica-counts-without-binding.png[El gráfico incorrecto para una única implantación, it shows the replica count for all deployments]

Para solucionar esto, haga la consulta PromQL específica para un componente utilizando la información del componente. Filtre en suficientes etiquetas métricas para seleccionar sólo las series temporales específicas para el componente. Se trata de la "vinculación" de la serie temporal correcta al componente. Para cualquiera que tenga experiencia en hacer cuadros de mando Grafana esto es similar a un cuadro de mando con parámetros que se utilizan en las consultas en el cuadro de mando. Cambiemos la consulta en el enlace métrico a esto:

----
max_over_time(kubernetes_state_deployment_replicas{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", deployment="${name}"}[${__interval}])
----

image::k8s/k8s-replica-counts-with-binding.png[Tras añadir los filtros parametrizados, el gráfico resultante tiene el aspecto esperado, only one time series for this component]

La consulta PromQL filtra ahora sobre tres etiquetas, `+cluster_name+`, `+namespace+` y `+deployment+`. En lugar de especificar un valor real para estas etiquetas, se utiliza una referencia variable a campos del componente. En este caso se utilizan las etiquetas `+cluster-name+` y `+namespace+`, referenciadas mediante `+${tags.cluster-name}+` y `+${tags.namespace}+`. Además, el nombre del componente se referencia con `+${name}+`.

Las referencias de variables admitidas son:

* Cualquier etiqueta de componente, utilizando `+${tags.<label-name>}+`
* El nombre del componente, utilizando `+${name}+`

image::k8s/k8s-carts-highlights.png[Página de componentes destacados que muestra las etiquetas y el nombre del componente (ambos resaltados en rojo)]

[NOTE]
====
El nombre del clúster, el espacio de nombres y una combinación del tipo y el nombre del componente suelen ser suficientes para seleccionar las métricas de un componente específico de Kubernetes. Estas etiquetas, o etiquetas similares, suelen estar disponibles en la mayoría de las métricas y componentes.
====


== Avanzado

=== Más de una serie temporal en un gráfico

[NOTE]
====
Sólo hay una unidad para un enlace métrico (se traza en el eje y del gráfico). En consecuencia, sólo debe combinar consultas que produzcan series temporales con la misma unidad en un enlace métrico. A veces es posible convertir la unidad. Por ejemplo, el uso de la CPU puede indicarse en mili-núcleos o en núcleos. Los mili-núcleos pueden convertirse en núcleos multiplicándolos por 1.000, como en `+(<original-query>) * 1000+`.
====


Hay dos formas de obtener más de una serie temporal en una única métrica vinculante y, por tanto, en un único gráfico:

. Escriba una consulta PromQL que devuelva múltiples series temporales para un único componente
. Añadir más consultas PromQL al enlace métrico

Para la primera opción se ofrece un ejemplo en la xref:/setup/custom-integrations/metric-bindings/index.adoc#_using_metric_labels_in_aliases[siguiente sección]. La segunda opción puede ser útil para comparar métricas relacionadas. Algunos casos típicos:

* Comparación de las réplicas totales con las deseadas y las disponibles
* Utilización de recursos: límites, solicitudes y uso en un solo gráfico

Para añadir más consultas a un enlace métrico, basta con repetir xref:/setup/custom-integrations/metric-bindings/index.adoc#_steps[los pasos] 3. y 4. y añadir la consulta como una entrada más en la lista de consultas. Para los recuentos de réplicas de despliegue hay varias métricas relacionadas que pueden incluirse en el mismo gráfico:

----
- _type: MetricBinding
  chartType: line
  enabled: true
  tags: {}
  unit: short
  name: Replica counts
  priority: MEDIUM
  identifier: urn:stackpack:my-stackpack:metric-binding:my-deployment-replica-counts
  queries:
    - expression: max_over_time(kubernetes_state_deployment_replicas{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", deployment="${name}"}[${__interval}])
      alias: Total replicas
    - expression: max_over_time(kubernetes_state_deployment_replicas_available{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}",  deployment="${name}"}[${__interval}])
      alias: Available - ${cluster_name} - ${namespace} - ${deployment}
    - expression: max_over_time(kubernetes_state_deployment_replicas_unavailable{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}",  deployment="${name}"}[${__interval}])
      alias: Unavailable - ${cluster_name} - ${namespace} - ${deployment}
    - expression: min_over_time(kubernetes_state_deployment_replicas_desired{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}",  deployment="${name}"}[${__interval}])
      alias: Desired - ${cluster_name} - ${namespace} - ${deployment}
  scope: type = "deployment" and label = "stackpack:kubernetes"
----

image::k8s/k8s-replica-counts-multiple-timeseries.png[Vinculación métrica con múltiples métricas]

=== Utilización de etiquetas métricas en los alias

Cuando una sola consulta devuelve múltiples series temporales por componente, esto se mostrará como múltiples líneas en el gráfico. Pero en la leyenda todos utilizarán el mismo alias. Para poder ver la diferencia entre las distintas series temporales, el alias puede incluir referencias a las etiquetas métricas utilizando la sintaxis `+${label}+`. Por ejemplo, aquí hay una métrica vinculante para la métrica "Reinicios de contenedor" en un pod, tenga en cuenta que un pod puede tener varios contenedores:

----
type: MetricBinding
chartType: line
enabled: true
id: -1
identifier: urn:stackpack:my-stackpack:metric-binding:my-pod-restart-count
name: Container restarts
priority: MEDIUM
queries:
- alias: Restarts - ${container}
  expression: max by (cluster_name, namespace, pod_name, container) (kubernetes_state_container_restarts{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", pod_name="${name}"})
scope: (label = "stackpack:kubernetes" and type = "pod")
unit: short
----

Observe que `+alias+` hace referencia a la etiqueta `+container+` de la métrica. Asegúrese de que la etiqueta está presente en el resultado de la consulta; si falta la etiqueta, `+${container}+` se mostrará como texto literal para facilitar la resolución de problemas.

=== Diseños

Cada componente puede estar asociado a diversas tecnologías o protocolos como k8s, redes, entornos de ejecución (por ejemplo, JVM), protocolos (HTTP, AMQP), etc.
En consecuencia, se pueden mostrar multitud de métricas diferentes para cada componente. Para facilitar la lectura, {stackstate-product-name} puede organizar estos gráficos en pestañas y secciones.
Para mostrar un gráfico (`+MetricBinding+`) dentro de una pestaña o sección específica, es necesario configurar la propiedad layout.
Cualquier MetricsBinding sin un diseño especificado se mostrará en una pestaña y sección llamada `+Other+`.
