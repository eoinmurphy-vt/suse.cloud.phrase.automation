== Visión general

En esta página se describe cómo restaurar las copias de seguridad de los almacenes de datos de {stackstate-product-name} mediante la CLI de copia de seguridad.

[WARNING]
====
*La restauración a partir de una copia de seguridad purgará o sobrescribirá los datos existentes. Esta operación no puede deshacerse.*

Verifique siempre el nombre de la copia de seguridad y el espacio de nombres antes de confirmar una operación de restauración.

*Flujo de trabajo de GitOps afectado*Esta herramienta de copia de seguridad modifica los recursos K8s directamente en su clúster:

- Escala StatefulSets/Deployments a 0 réplicas
- Añade anotaciones para el seguimiento de la restauración

Estos cambios entrarán en conflicto con su flujo de trabajo GitOps, ya que eluden la reconciliación basada en Git. Los controladores de GitOps pueden intentar revertir estos cambios durante las operaciones de copia de seguridad.
*Importante:* La reconciliación automática para la implantación del cuadro de SUSE Observability Helm debe desactivarse durante la restauración de la copia de seguridad para evitar conflictos.
====

[NOTE]
====
*Antes de utilizar la CLI, asegúrese de que:*

* Está utilizando la última versión de la CLI de copia de seguridad.
* La CLI espera un kubeconfig con un contexto actual al cluster donde SUSE Observability está instalado.
====

== Datos de configuración y topología (StackGraph)

=== Lista de copias de seguridad de StackGraph

Para listar las copias de seguridad de StackGraph, ejecute el siguiente comando:

[,bash]
----
sts-backup stackgraph list --namespace <NAMESPACE>
----

Sustituya `+<NAMESPACE>+` por el espacio de nombres donde está instalado {stackstate-product-name}.

El resultado debería ser el siguiente:

[,bash]
----
Setting up port-forward to suse-observability-minio:9000 in namespace <NAMESPACE>...
✓ Port-forward established successfully
Listing Stackgraph backups in bucket 'sts-stackgraph-backup'...
NAME                            LAST MODIFIED            SIZE
sts-backup-20251128-0300.graph  2025-11-28 03:08:42 UTC  2GiB
----

La fecha y hora en que se realiza la copia de seguridad forma parte del nombre de la copia.


=== Restaurar una copia de seguridad de StackGraph

Para restaurar una copia de seguridad de StackGraph, utilice uno de los siguientes métodos:

==== Restaurar una copia de seguridad específica

[,bash]
----
sts-backup stackgraph restore --namespace <NAMESPACE> --archive <BACKUP_NAME>
----

==== Restaurar la última copia de seguridad

[,bash]
----
sts-backup stackgraph restore --namespace <NAMESPACE> --latest
----

==== Banderas comunes

* `+--yes+` o `+-y+` - Omitir la pregunta de confirmación (útil para la automatización)
* `+--background+` - Ejecutar la restauración en segundo plano sin esperar a que se complete

==== Restauración del fondo

Cuando se utiliza `+--background+`, la restauración se ejecuta de forma asíncrona. Después de iniciar la restauración, utilice el siguiente comando para comprobar el estado y finalizar:

[,bash]
----
sts-backup stackgraph check-and-finalize --job <JOB_NAME> --wait --namespace <NAMESPACE>
----

El comando `+check-and-finalize+`:

* Comprueba el estado del trabajo de restauración
* Con la bandera `+--wait+`, espera la finalización del trabajo
* Amplía automáticamente las implantaciones que se redujeron durante la restauración.
* Limpia los recursos tras la finalización

[NOTE]
====
Si se interrumpe una restauración que se está ejecutando sin `+--background+` (por ejemplo, mediante Ctrl+C), deberá ejecutar `+check-and-finalize+` para ampliar las implantaciones y limpiar los recursos.
====

== Ajustes

Las copias de seguridad de la configuración incluyen los StackPacks instalados con su configuración y otras personalizaciones creadas por el usuario, como monitores, vistas personalizadas y tokens de servicio. Las copias de seguridad de la configuración son ligeras (normalmente sólo varios megabytes) y rápidas de restaurar, con un tiempo de inactividad mínimo.

=== Lista de copias de seguridad de la configuración

[,bash]

Sustituya `+<NAMESPACE>+` por el espacio de nombres donde está instalado SUSE Observability.

El resultado debería ser el siguiente:

[,bash]
----
Setting up port-forward to suse-observability-minio:9000 in namespace <NAMESPACE>...
✓ Port-forward established successfully
Listing Settings backups in bucket 'sts-configuration-backup'...
NAME                          LAST MODIFIED            SIZE
sts-backup-20251128-1328.sty  2025-11-28 13:29:12 UTC  2MiB
----

La fecha y hora en que se realizó la copia de seguridad forma parte del nombre de la copia.

== Métricas (Victoria Metrics)

Dependiendo del perfil, `+nonha+` o `+ha+`, Victoria Metrics se despliega en diferentes modos:

* *perfil nonha* - Modo de nodo único con una instancia de Victoria Metrics (`+victoria-metrics-0+`)
* *perfil ha* - modo HA (espejo) con dos instancias de Victoria Metrics (`+victoria-metrics-0+` y `+victoria-metrics-1+`)

=== Lista de copias de seguridad de Victoria Metrics

Para listar las copias de seguridad de Victoria Metrics, ejecute el siguiente comando:

[,bash]
----
sts-backup victoria-metrics list --namespace <NAMESPACE>
----

Sustituya `+<NAMESPACE>+` por el espacio de nombres donde está instalado SUSE Observability.

==== Salida en modo de nodo único (perfil nonha)

[,bash]
----
Setting up port-forward to suse-observability-minio:9000 in namespace <NAMESPACE>...
✓ Port-forward established successfully
Listing VictoriaMetrics backups in bucket ...
NAME ({bucket}/{instance}-{created})                           UPDATED
sts-victoria-metrics-backup/victoria-metrics-0-20251030152500  2025-11-28 09:25:05 UTC
----

==== Salida del modo HA (perfil ha)

[,bash]
----
Setting up port-forward to suse-observability-minio:9000 in namespace <NAMESPACE>...
✓ Port-forward established successfully
Listing VictoriaMetrics backups in bucket ...
NAME ({bucket}/{instance}-{created})                           UPDATED
sts-victoria-metrics-backup/victoria-metrics-1-20251030152500  2025-11-28 09:35:08 UTC
sts-victoria-metrics-backup/victoria-metrics-0-20251030152500  2025-11-28 09:25:04 UTC

En modo HA, las copias de seguridad se crean para ambas instancias con prefijos diferentes (`+victoria-metrics-0+` y `+victoria-metrics-1+`). Al restaurar, puede especificar cualquiera de las copias de seguridad: la operación de restauración restaurará la copia de seguridad seleccionada en ambas instancias.

== OpenTelemetry (ClickHouse)

=== Lista de copias de seguridad de ClickHouse

Para listar las copias de seguridad de ClickHouse, ejecute el siguiente comando:

[,bash]
----
sts-backup clickhouse list --namespace <NAMESPACE>
----

Sustituya `+<NAMESPACE>+` por el espacio de nombres donde está instalado SUSE Observability.

El resultado debería ser el siguiente:

[,bash]
----
Setting up port-forward to suse-observability-clickhouse-backup:7171 in namespace <NAMESPACE>...
✓ Port-forward established successfully
Listing Clickhouse backups...
NAME                             CREATED              SIZE
incremental_2025-11-28T09-45-00  2025-11-28 09:45:03  65MiB
incremental_2025-11-28T08-45-00  2025-11-28 08:45:03  223MiB

incremental_2025-11-27T23-45-00  2025-11-27 23:45:03  118MiB
----

Los nombres de copia de seguridad que empiezan por `+full_+` son copias de seguridad completas, mientras que los nombres que empiezan por `+incremental_+` son copias de seguridad incrementales.

=== Restaurar una copia de seguridad de ClickHouse

[NOTE]
====
El proceso de restauración reduce automáticamente las cargas de trabajo que producen datos (como los exportadores de OpenTelemetry) para evitar la pérdida de datos durante la restauración.
====

Para restaurar una copia de seguridad de ClickHouse, utilice uno de los siguientes métodos:

==== Restaurar una copia de seguridad específica

[,bash]
----
sts-backup clickhouse restore --namespace <NAMESPACE> --snapshot <BACKUP_NAME>
----

==== Restaurar la última copia de seguridad

[,bash]
----
sts-backup clickhouse restore --namespace <NAMESPACE> --latest
----

==== Banderas comunes

* `+--yes+` o `+-y+` - Omitir la pregunta de confirmación (útil para la automatización)
* `+--background+` - Ejecutar la restauración en segundo plano sin esperar a que se complete

==== Restauración del fondo

Cuando se utiliza `+--background+`, la restauración se ejecuta de forma asíncrona. Después de iniciar la restauración, utilice el siguiente comando para comprobar el estado y finalizar:

[,bash]
----
sts-backup clickhouse check-and-finalize --operation-id <OPERATION_ID> --wait --namespace <NAMESPACE>
----

El comando `+check-and-finalize+`:

* Comprueba el estado de la operación de restauración
* Con la bandera `+--wait+`, espera la finalización de la operación
* Ejecuta comandos SQL posteriores a la restauración
* Amplía automáticamente los StatefulSets que se redujeron durante la restauración.
* Limpia los recursos tras la finalización

[NOTE]
====
Si se interrumpe una restauración que se está ejecutando sin `+--background+` (por ejemplo, mediante Ctrl+C), debe ejecutar `+check-and-finalize+` con el ID de la operación para escalar StatefulSets y limpiar los recursos.
====
