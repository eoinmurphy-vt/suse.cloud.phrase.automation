= Muestreo
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: Observabilidad de SUSE

El muestreo se utiliza para reducir el volumen de datos que se exportan a SUSE Observability, comprometiendo lo menos posible la calidad de los datos telemétricos. La principal razón para aplicar el muestreo es reducir costes (de red, almacenamiento, etc.).

Si sus aplicaciones generan pocos datos no hay necesidad de muestreo e incluso puede dificultar la observabilidad debido a la falta de datos de telemetría. Sin embargo, si su aplicación tiene un tráfico importante, por ejemplo más de 1.000 tramos por segundo, ya puede tener sentido aplicar el muestreo.

Existen dos tipos principales de muestreo: el muestreo de cabeza y el muestreo de cola.

== Muestreo de cabezas

El muestreo de cabecera toma la decisión de muestreo (exportar o no los datos) lo antes posible. Por tanto, la decisión no puede basarse en todo el rastro, sino sólo en la información, muy limitada, de que se dispone. El recolector otel cuenta con el https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/processor/probabilisticsamplerprocessor[procesador de muestreo probabilístico] que implementa el Muestreo Probabilístico Consistente. El muestreador es configurable y toma una decisión de muestreo basada en el id de la traza (útil para trazas) o en el hash de un atributo (útil para logs). Esto garantiza que todos los intervalos de una traza se muestrean siempre o no y que tendrá trazas completas en SUSE Observability.

Las ventajas del muestreo de cabezas son:

* Fácil de entender
* Eficaz
* Fácil de configurar

Pero una desventaja es que es imposible tomar decisiones de muestreo sobre toda una traza, por ejemplo, muestrear todas las trazas fallidas y sólo una pequeña selección de las trazas exitosas.

Para activar el muestreo de cabezales, configure el procesador e inclúyalo en las canalizaciones. Este ejemplo muestrea 1 de cada 4 trazas basándose en el id de la traza:

[,yaml]
----
processors:
  probabilistic_sampler:
    sampling_percentage: 25
    mode: "proportional"
----

== Muestreo de cola

El muestreo de cola pospone la decisión de muestreo hasta que una traza está (casi) completa. Esto permite al muestreador de cola tomar decisiones de muestreo basadas en toda la traza, por ejemplo, muestrear siempre trazas fallidas y/o lentas. Existen muchas más posibilidades, como el muestreo basado en atributos específicos o la participación en servicios.

OpenTelemetry Collector proporciona un https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/processor/tailsamplingprocessor[procesador de muestreo de cola] que puede utilizarse para aplicar políticas de muestreo de cola.

La principal ventaja del muestreo de colas es la flexibilidad adicional que proporciona a la hora de tomar decisiones de muestreo basadas en datos de trazas completos, en lugar de en la información parcial disponible cuando se crean tramos individuales.


[WARNING]
====
Aunque ventajoso, el muestreo de colas presenta los siguientes inconvenientes:

* Más difícil de configurar y comprender.
* Debe ser de estado para almacenar los intervalos de las trazas hasta que se tome una decisión de muestreo.
* Por lo tanto también (mucho) más uso de recursos
* Es posible que el muestreador no pueda seguir el ritmo, por lo que necesita supervisión y escalado adicionales.
====

Para habilitar el muestreo de cola, configure el procesador e inclúyalo en los pipelines.

[,yaml]
----
processors:
  tail_sampling:
    decision_wait: 10s
    policies:
    - name: rate-limited-composite
      type: composite
      composite:
        max_total_spans_per_second: 500
        policy_order: [errors, slow-traces, rest]
        composite_sub_policy:
        - name: errors
          type: status_code
          status_code:
            status_codes: [ ERROR ]
        - name: slow-traces
          type: latency
          latency:
            threshold_ms: 1000
        - name: rest
          type: always_sample
        rate_allocation:
        - policy: errors
          percent: 33
        - policy: slow-traces
          percent: 33
        - policy: rest
          percent: 34
----

Las muestras de ejemplo:

* Un máximo de 500 tramos por segundo
* todos los tramos en las trazas que tienen errores de hasta el 33% del máximo
* todos los tramos en trazados más lentos que 1 segundo hasta el 33% del máximo
* otros tramos hasta la tasa máxima permitida

Para obtener más información sobre las opciones de configuración y las distintas políticas, consulte https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/processor/tailsamplingprocessor[el archivo "readme" de].

Sin embargo, no se trata de un sistema de "todo listo". Si el uso de recursos comienza a crecer, es necesario escalar el uso de múltiples recolectores para manejar el muestreo de cola, lo que también requerirá el https://github.com/open-telemetry/opentelemetry-collector-contrib/blob/main/connector/routingconnector/README.md[enrutamiento] para enrutar el tráfico basado en el ID de rastreo.

== Muestreo de trazas en combinación con métricas de tramo

En la sección de introducción, la configuración del colector no incluye el muestreo. Al añadir el muestreo, queremos tener cuidado de que las métricas que se calculan a partir de las trazas sean lo más precisas posible. Especialmente el muestreo de colas puede dar lugar a métricas muy sesgadas, porque normalmente la cantidad relativa de errores es mucho mayor. Para evitarlo, dividimos la tubería de trazas en varias partes y las conectamos con el conector de avance. Modifica la configuración para incluir el conector adicional y el procesador de muestreo. Y modificar las tuberías como se muestra aquí:

[,yaml]
----
connectors:
  # enable the forwarder
  forward:
processors:
  # Configure the probabilistic sampler to sample 25% of the traffic
  probabilistic_sampler:
    sampling_percentage: 25
    mode: "proportional"
service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [memory_limiter, resource]
      exporters: [forward]
    traces/spanmetrics:
      receivers: [forward]
      processors: []
      exporters: [spanmetrics]
    traces/sampling:
      receivers: [forward]
      processors: [probabilistic_sampler, batch]
      exporters: [debug, otlp/stackstate]
    metrics:
      receivers: [otlp, spanmetrics, prometheus]
      processors: [memory_limiter, resource, batch]
      exporters: [debug, otlp/stackstate]
----

El ejemplo utiliza el muestreador probabilístico configurado para muestrear el 25% del tráfico. Es probable que desee ajustar el porcentaje para su situación o cambiar al <<_tail_sampling,muestreador de>> cola en su lugar. La configuración de la tubería es la misma para el muestreador de cola, sólo tiene que sustituir la referencia a la `+probabilistic_sampler+` con `+tail_sampling+`.
