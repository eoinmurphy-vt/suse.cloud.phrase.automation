== Instrumentación automática

La instrumentación automática para .NET puede capturar automáticamente trazas y métricas para una gran variedad de https://github.com/open-telemetry/opentelemetry-dotnet-instrumentation/blob/main/docs/internal/instrumentation-libraries.md[bibliotecas y frameworks de].

La instrumentación automática no requiere ninguna modificación de la aplicación. Para configurarlo, sigue estos pasos:

. Descargue la versión https://github.com/open-telemetry/opentelemetry-dotnet-instrumentation/releases/latest/download/opentelemetry-dotnet-instrumentation-linux-glibc.zip[glibc] o https://github.com/open-telemetry/opentelemetry-dotnet-instrumentation/releases/latest/download/opentelemetry-dotnet-instrumentation-linux-musl.zip[musl] de las bibliotecas de instrumentación (musl para Alpine, glibc para la mayoría de las demás imágenes docker) de las https://github.com/open-telemetry/opentelemetry-java-instrumentation/releases[versiones] del repositorio opentelemetry-dotnet-instrumentation. Descomprime los archivos e inclúyelos en la imagen docker de tu aplicación en un directorio, aquí usamos `+/autoinstrumentation+`.
. Establecer los siguientes env vars, aquí lo hacemos a través de la `+env+` del contenedor en el Kubernetes pod spec:
+
[,yaml]
----
env:
- name: CORECLR_ENABLE_PROFILING
  value: "1"
- name: CORECLR_PROFILER
  value: "{918728DD-259F-4A6A-AC2B-B85E1B658318}"
- name: CORECLR_PROFILER_PATH
  # for glibc:
  value: "/autoinstrumentation/linux-x64/OpenTelemetry.AutoInstrumentation.Native.so"
  # For musl use instead:
  # value: "/autoinstrumentation/linux-musl-x64/OpenTelemetry.AutoInstrumentation.Native.so"
- name: DOTNET_ADDITIONAL_DEPS
  value: "/autoinstrumentation/AdditionalDeps"
- name: DOTNET_SHARED_STORE
  value: "/autoinstrumentation/store"
- name: DOTNET_STARTUP_HOOKS
  value: "/autoinstrumentation/net/OpenTelemetry.AutoInstrumentation.StartupHook.dll"
- name: OTEL_DOTNET_AUTO_HOME
  value: "/autoinstrumentation"
----

. Añada también las variables de entorno adicionales xref:/setup/otel/instrumentation/sdk-exporter-config.adoc[para configurar el nombre del servicio y el punto final del exportador] en el pod, los protocolos soportados son gRPC y protobuf sobre HTTP.
. Despliegue su aplicación con los cambios
. Verifique que SUSE Observability está recibiendo trazas y/o métricas buscando las métricas / trazas en el explorador de métricas / trazas para su nombre de servicio

Para más detalles, consulte la https://opentelemetry.io/docs/languages/java/automatic/[documentación de].

== Instrumentación manual

La instrumentación manual puede utilizarse cuando se necesitan métricas, trazas o registros de partes del código que no son compatibles con la instrumentación automática. Por ejemplo, bibliotecas no compatibles, código interno o métricas a nivel empresarial.

Para capturar esos datos es necesario modificar la aplicación.

. Incluya el SDK de telemetría abierta como dependencia
. Añada código a su aplicación para capturar métricas, intervalos o registros cuando sea necesario

Existe documentación detallada al respecto en https://opentelemetry.io/docs/languages/net/instrumentation/[las páginas de documentación de].

Asegúrese de utilizar el exportador OTLP (es el predeterminado) y https://opentelemetry.io/docs/languages/java/instrumentation/#_autoconfiguration[auto-configuration]. Al desplegar la aplicación, el nombre del servicio y el exportador se xref:/setup/otel/instrumentation/sdk-exporter-config.adoc[configuran mediante variables de entorno].

== Métricas en SUSE Observability

Para Kubernetes, los gráficos están disponibles en los pods. Es posible xref:/setup/custom-integrations/metric-bindings/index.adoc[añadir gráficos para más métricas], esto funciona para métricas de instrumentación automática pero también para métricas específicas de la aplicación de instrumentación manual.
