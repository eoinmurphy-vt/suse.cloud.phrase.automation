= Primeros pasos con Open Telemetry
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: Observabilidad de SUSE

Esta es la configuración que vamos a crear, para una aplicación que necesita ser monitorizada:

* La aplicación / carga de trabajo supervisada que se ejecuta en un host Linux
* El colector de telemetría abierta que se ejecuta en el mismo host Linux
* SUSE Observability o SUSE Cloud Observability

image::otel/open-telemetry-collector-linux.png[Instrumentación de aplicaciones en un host linux con Open Telemetry colector que se ejecuta en el host]

== Instalar el colector abierto de telemetría

[NOTE]
====
Para una configuración de producción, se recomienda encarecidamente instalar el recopilador, ya que permite que su servicio descargue datos rápidamente y el recopilador puede encargarse de tareas adicionales como reintentos, procesamiento por lotes, cifrado o incluso filtrado de datos confidenciales.
====


Primero instalaremos el colector. Lo configuramos para:

* Recibir datos de, potencialmente, muchas aplicaciones instrumentadas.
* Enriquecer los datos recogidos con atributos del anfitrión
* Generar métricas para las trazas
* Reenvíe los datos a SUSE Observability, incluida la autenticación mediante la clave API

Además, también reintentará enviar datos cuando haya problemas de conexión.

=== Crear un token de servicio

Hay dos formas de crear un token de servicio:

* **SUSE Observability UI** - abra el menú principal haciendo clic en la parte superior izquierda de la pantalla y vaya a `+StackPacks+` > `+Open Telemetry+`.  Si no lo ha hecho antes, pulse el botón `+INSTALL+`.  Haga clic en el botón `+CREATE NEW SERVICE TOKEN+` y copie el valor en el portapapeles.
* **SUSE Observability CLI**: consulte xref:/use/security/k8s-service-tokens.adoc#_manage_service_tokens[Gestión de tokens de servicio]

El valor del testigo de servicio debe utilizarse cuando en las instrucciones siguientes se mencione `+<SERVICE_TOKEN>+`.

=== Instalar y configurar el colector

El recopilador proporciona paquetes (apk, deb y rpm) para la mayoría de las versiones y arquitecturas de Linux y utiliza `+systemd+` para la configuración automática de servicios. Para instalarlo, busque la https://github.com/open-telemetry/opentelemetry-collector-releases/releases[última versión de] y actualice la URL del ejemplo para utilizar la última versión:

Para otras opciones de instalación, utilice las https://opentelemetry.io/docs/collector/installation/#_linux[instrucciones de].

Después de la instalación, modifique la configuración del colector editando `+/etc/otelcol-contrib/config.yaml+`. Cambie el archivo para que se vea como el ejemplo `+config.yaml+` aquí, reemplace `+<otlp-suse-observability-endpoint:port>+` con su punto final xref:/setup/otel/otlp-apis.adoc[OTLP] (ver xref:/setup/otel/otlp-apis.adoc[API OTLP] para su punto final) e inserte su token de servicio para `+<SERVICE_TOKEN>+`:

.config.yaml
[,yaml]
----
receivers:
  otlp:
    protocols:
      # Only bind to localhost to keep the collector secure, see https://github.com/open-telemetry/opentelemetry-collector/blob/main/docs/security-best-practices.md#safeguards-against-denial-of-service-attacks
      grpc:
        endpoint: 127.0.0.1:4317
      http:
        endpoint: 127.0.0.1:4318
  # Collect own metrics
  prometheus:
    config:
      scrape_configs:
      - job_name: 'otel-collector'
        scrape_interval: 10s
        static_configs:
        - targets: ['0.0.0.0:8888']
extensions:
  health_check: {}
  pprof:
    endpoint: 0.0.0.0:1777
  zpages:
    endpoint: 0.0.0.0:55679
  # Use the API key from the env for authentication
  bearertokenauth:
    scheme: SUSEObservability
    token: "<SERVICE_TOKEN>"
exporters:
  nop: {}
  debug: {}
  otlp/suse-observability:
    compression: snappy
    auth:
      authenticator: bearertokenauth
    # Put in your own otlp endpoint, for example otlp-suse-observability.my.company.com:443
    endpoint: <otlp-suse-observability-endpoint:port>
processors:
  memory_limiter:
    check_interval: 5s
    limit_percentage: 80
    spike_limit_percentage: 25
  batch: {}
  # Optionally include resource information from the system running the collector
  resourcedetection/system:
    detectors: [env, system] # Replace system with gcp, ec2, azure when running in cloud environments
    system:
      hostname_sources: ["os"]
connectors:
  # Generate metrics for spans
  spanmetrics:
    metrics_expiration: 5m
    namespace: otel_span
service:
  extensions: [ bearertokenauth, health_check, pprof, zpages ]
  pipelines:
    traces:
      receivers: [otlp]
      processors: [memory_limiter, resourcedetection/system, batch]
      exporters: [debug, spanmetrics, otlp/suse-observability]
    metrics:
      receivers: [otlp, spanmetrics, prometheus]
      processors: [memory_limiter, batch, resourcedetection/system]
      exporters: [debug, otlp/suse-observability]
    logs:
      receivers: [otlp]
      processors: []
      exporters: [nop]
----


Por último, reinicie el colector:

[,bash]
----
sudo systemctl restart otelcol-contrib
----

Para ver los registros del colector utilice:

[,bash]
----
sudo journalctl -u otelcol-contrib
----

== Recopile datos telemétricos de su aplicación

La forma habitual de recopilar datos de telemetría es instrumentar su aplicación utilizando los SDK de telemetría abiertos. Hemos documentado algunas guías de inicio rápido para algunos idiomas, pero hay muchos más:

* xref:/setup/otel/instrumentation/java.adoc[Java]
* xref:/setup/otel/instrumentation/dot-net.adoc[.NET]
* xref:/setup/otel/instrumentation/node.js.adoc[Node.js]

No es necesaria ninguna configuración adicional para los SDKs, exportan a localhost vía OTLP o OTLP sobre HTTP (dependiendo de los protocolos soportados) por defecto.

Para otros idiomas siga la documentación en https://opentelemetry.io/docs/languages/[opentelemetry.io].

== Ver los resultados

Vaya a SUSE Observability y asegúrese de que el Stackpack Open Telemetry está instalado (a través del menú principal \-> Stackpacks).

Después de un corto tiempo y si su aplicación está procesando algo de tráfico, usted debería ser capaz de encontrarlo bajo su nombre de servicio en la Telemetría abierta \-> servicios e instancias de servicio. Las trazas aparecerán en el xref:/use/traces/k8sTs-explore-traces.adoc[explorador de trazas] y en la xref:/use/views/k8s-traces-perspective.adoc[perspectiva de trazas] para los componentes de servicio e instancia de servicio. Las métricas de alcance y las métricas específicas del idioma (si están disponibles) estarán disponibles en la xref:/use/views/k8s-metrics-perspective.adoc[perspectiva de métricas] de los componentes.
