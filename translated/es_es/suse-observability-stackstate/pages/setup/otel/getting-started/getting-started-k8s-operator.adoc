= Primeros pasos con el operador Open Telemetry en Kubernetes
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: Observabilidad de SUSE

Esta es la configuración que vamos a crear, para una aplicación que necesita ser monitorizada:

* La aplicación / carga de trabajo supervisada que se ejecuta en el clúster A, autoinstrumentada por el operador.
* El operador de telemetría abierta en el clúster A
* Un colector creado por el operador
* SUSE Observability ejecutándose en el clúster B, o SUSE Cloud Observability

image::otel/open-telemetry-kubernetes-operator.png[Instrumentación de contenedores con autoinstrumentación del operador de telemetría abierta]

== Instalar el automatismo

El operador Open Telemetry ofrece algunas características adicionales sobre la configuración normal de Kubernetes:

* Puede auto-instrumentar sus pods de aplicaciones para los lenguajes soportados (Java, .NET, Python, Golang, Node.js), sin tener que modificar las aplicaciones o las imágenes docker en absoluto.
* Puede utilizarse como sustituto del operador de Prometheus y comenzar a rastrear los puntos finales del exportador de Prometheus basándose en los monitores de servicios y pods.

=== Crear un token de servicio

Hay dos formas de crear un token de servicio:

* **SUSE Observability UI** - abra el menú principal haciendo clic en la parte superior izquierda de la pantalla y vaya a `+StackPacks+` > `+Open Telemetry+`.  Si no lo ha hecho antes, pulse el botón `+INSTALL+`.  Haga clic en el botón `+CREATE NEW SERVICE TOKEN+` y copie el valor en el portapapeles.
* **SUSE Observability CLI**: consulte xref:/use/security/k8s-service-tokens.adoc#_manage_service_tokens[Gestión de tokens de servicio]

El valor del testigo de servicio debe utilizarse cuando en las instrucciones siguientes se mencione `+<SERVICE_TOKEN>+`.

=== Crear el espacio de nombres y un secreto para el token de servicio

Instalaremos en el espacio de nombres `+open-telemetry+` y utilizaremos el token de servicio:

[,bash]
----
kubectl create namespace open-telemetry
kubectl create secret generic open-telemetry-collector \
    --namespace open-telemetry \
    --from-literal=API_KEY='<SERVICE_TOKEN>'
----

=== Configurar e instalar el operador

El operador se instala con una carta Helm, así que primero configure el repositorio de cartas.

[,bash]
----
helm repo add open-telemetry https://open-telemetry.github.io/opentelemetry-helm-charts
----

Vamos a crear un archivo `+otel-operator.yaml+` para configurar el operador:

.otel-operator.yaml
[,yaml]
----
# Add image pull secret for private registries
imagePullSecrets: []
manager:
  image:
    # Uses chart.appVersion for the tag
    repository: ghcr.io/open-telemetry/opentelemetry-operator/opentelemetry-operator
  collectorImage:
    # find the latest collector releases at https://github.com/open-telemetry/opentelemetry-collector-releases/releases
    repository: otel/opentelemetry-collector-k8s
    tag: 0.123.0
  targetAllocatorImage:
    repository: ""
    tag: ""
  # Only needed when overriding the image repository, make sure to always specify both the image and tag:
  autoInstrumentationImage:
    java:
      repository: ""
      tag: ""
    nodejs:
      repository: ""
      tag: ""
    python:
      repository: ""
      tag: ""
    dotnet:
      repository: ""
      tag: ""
    # The Go instrumentation support in the operator is disabled by default.
    # To enable it, use the operator.autoinstrumentation.go feature gate.
    go:
      repository: ""
      tag: ""

admissionWebhooks:
  # A production setup should use certManager to generate the certificate, without certmanager the certificate will be generated during the Helm install
  certManager:
    enabled: false
  # The operator has validation and mutation hooks that need a certificate, with this we generate that automatically
  autoGenerateCert:
    enabled: true
----


Ahora instale el colector, utilizando el archivo de configuración:

[,bash]
----
helm upgrade --install opentelemetry-operator open-telemetry/opentelemetry-operator \
  --namespace open-telemetry \
  --values otel-operator.yaml
----

Esto sólo instala el operador. Continúe instalando el colector y active la autoinstrumentación.

== El colector de telemetría abierto

El operador gestiona uno o varios despliegues de recopiladores a través de un recurso personalizado de Kubernetes del tipo `+OpenTelemetryCollector+`. Vamos a crear uno utilizando la misma configuración que se utiliza en la xref:/setup/otel/getting-started/getting-started-k8s.adoc[guía de inicio de Kubernetes].

Utiliza el secreto creado anteriormente en la guía. Asegúrese de sustituir `+<otlp-suse-observability-endpoint:port>+` por su punto final xref:/setup/otel/otlp-apis.adoc[OTLP] (consulte la xref:/setup/otel/otlp-apis.adoc[API OTLP] para su punto final) e inserte el nombre de su clúster Kubernetes en lugar de `+<your-cluster-name>+`:

.collector.yaml
[,yaml]
----
apiVersion: opentelemetry.io/v1beta1
kind: OpenTelemetryCollector
metadata:
  name: otel-collector
spec:
  mode: deployment
  envFrom:
  - secretRef:
      name: open-telemetry-collector
  # optional service-account for pulling the collector image from a private registries
  # serviceAccount: otel-collector
  config:
    receivers:
      otlp:
        protocols:
          grpc:
            endpoint: 0.0.0.0:4317
          http:
            endpoint: 0.0.0.0:4318
      # Scrape the collectors own metrics
      prometheus:
        config:
          scrape_configs:
          - job_name: opentelemetry-collector
            scrape_interval: 10s
            static_configs:
            - targets:
              - ${env:MY_POD_IP}:8888
    extensions:
      health_check:
        endpoint: ${env:MY_POD_IP}:13133
      # Use the API key from the env for authentication
      bearertokenauth:
        scheme: SUSEObservability
        token: "${env:API_KEY}"
    exporters:
      debug: {}
      nop: {}
      otlp/suse-observability:
        auth:
          authenticator: bearertokenauth
        # Put in your own otlp endpoint, for example otlp-suse-observability.my.company.com:443
        endpoint: <otlp-suse-observability-endpoint:port>
        compression: snappy
    processors:
      memory_limiter:
        check_interval: 5s
        limit_percentage: 80
        spike_limit_percentage: 25
      batch: {}
      resource:
        attributes:
        - key: k8s.cluster.name
          action: upsert
          # Insert your own cluster name
          value: <your-cluster-name>
        - key: service.instance.id
          from_attribute: k8s.pod.uid
          action: insert
          # Use the k8s namespace also as the open telemetry namespace
        - key: service.namespace
          from_attribute: k8s.namespace.name
          action: insert
    connectors:
      # Generate metrics for spans
      spanmetrics:
        metrics_expiration: 5m
        namespace: otel_span
    service:
      extensions: [ health_check,  bearertokenauth ]
      pipelines:
        traces:
          receivers: [otlp]
          processors: [memory_limiter, resource, batch]
          exporters: [debug, spanmetrics, otlp/suse-observability]
        metrics:
          receivers: [otlp, spanmetrics, prometheus]
          processors: [memory_limiter, resource, batch]
          exporters: [debug, otlp/suse-observability]
        logs:
          receivers: [otlp]
          processors: []
          exporters: [nop]
      telemetry:
        metrics:
          address: ${env:MY_POD_IP}:8888
----


[CAUTION]
====
*Utilice el mismo nombre de clúster que el utilizado para instalar el* agente de SUSE Observability si también utiliza el agente de SUSE Observability con el stackpack de Kubernetes. El uso de un nombre de clúster diferente dará lugar a una perspectiva de trazas vacía para los componentes de Kubernetes y, en general, hará que la correlación de la información sea mucho más difícil para SUSE Observability y sus usuarios.
====


Ahora aplique este `+collector.yaml+` en el espacio de nombres `+open-telemetry+` para implementar un recopilador:

[,bash]
----
kubectl apply --namespace open-telemetry -f collector.yaml
----

El xref:/setup/otel/collector.adoc[recopilador] ofrece muchos más receptores de configuración, procesadores y exportadores, para más detalles consulte nuestra xref:/setup/otel/collector.adoc[página del recopilador]. Para el uso en producción, a menudo se generan grandes cantidades de tramos y querrá empezar a configurar el xref:/setup/otel/sampling.adoc[muestreo].

== Autoinstrumentación

=== Configurar la autoinstrumentación

Ahora tenemos que decirle al operador cómo configurar la instrumentación automática para los diferentes idiomas utilizando otro recurso personalizado, del tipo `+Instrumentation+`. Se utiliza principalmente para configurar el recolector que se acaba de implementar como punto final de telemetría para las aplicaciones instrumentadas.

Puede definirse en un único lugar y ser utilizado por todos los pods del cluster, pero también es posible tener un `+Instrumentation+` diferente en cada espacio de nombres. Aquí haremos lo primero. Ten en cuenta que si utilizaste un espacio de nombres diferente o un nombre diferente para el recopilador otel, el punto final de este archivo debe actualizarse en consecuencia.

Cree una página `+instrumentation.yaml+`:

.instrumentation.yaml
[,yaml]
----
apiVersion: opentelemetry.io/v1alpha1
kind: Instrumentation
metadata:
  name: otel-instrumentation
spec:
  exporter:
    # default endpoint for the instrumentation
    endpoint: http://otel-collector-collector.open-telemetry.svc.cluster.local:4317
  propagators:
    - tracecontext
    - baggage
  defaults:
    # To use the standard app.kubernetes.io/ labels for the service name, version and namespace:
    useLabelsForResourceAttributes: true
  python:
    env:
      # Python autoinstrumentation uses http/proto by default, so data must be sent to 4318 instead of 4317.
      - name: OTEL_EXPORTER_OTLP_ENDPOINT
        value: http://otel-collector-collector.open-telemetry.svc.cluster.local:4318
  dotnet:
    env:
      # Dotnet autoinstrumentation uses http/proto by default, so data must be sent to 4318 instead of 4317.
      - name: OTEL_EXPORTER_OTLP_ENDPOINT
        value: http://otel-collector-collector.open-telemetry.svc.cluster.local:4318
  go:
    env:
      # Go autoinstrumentation uses http/proto by default, so data must be sent to 4318 instead of 4317.
      - name: OTEL_EXPORTER_OTLP_ENDPOINT
        value: http://otel-collector-collector.open-telemetry.svc.cluster.local:4318
----


Ahora aplique el `+instrumentation.yaml+` también en el espacio de nombres `+open-telemetry+`:

[,bash]
----
kubectl apply --namespace open-telemetry -f instrumentation.yaml
----

=== Activar la autoinstrumentación de un pod

Para indicar al operador que auto-instrumente sus pods de aplicación, necesitamos añadir una anotación al pod:

* Java: `+instrumentation.opentelemetry.io/inject-java: open-telemetry/otel-instrumentation+`
* NodeJS: `+instrumentation.opentelemetry.io/inject-nodejs: open-telemetry/otel-instrumentation+`
* Python: `+instrumentation.opentelemetry.io/inject-python: open-telemetry/otel-instrumentation+`
* Vamos: `+instrumentation.opentelemetry.io/inject-go: open-telemetry/otel-instrumentation+`

Observe que el valor de la anotación hace referencia al espacio de nombres y al nombre del recurso `+Instrumentation+` que hemos creado. Otras opciones son:

* "true" - inyectar y `+Instrumentation+` recurso personalizado del espacio de nombres.
* "mi-instrumentación" - nombre del recurso personalizado `+Instrumentation+` en el espacio de nombres actual.
* "mi-otro-espacio-de-nombres/mi-instrumentación" - espacio de nombres y nombre del recurso personalizado `+Instrumentation+` en otro espacio de nombres.
* "false" - no inyectar

Cuando se crea un pod con una de las anotaciones, el operador modifica el pod mediante un gancho de mutación:

* Añade un contenedor init que proporciona la biblioteca de autoinstrumentación
* Modifica el primer contenedor del pod para cargar la instrumentación durante el arranque y añade variables de entorno para configurar la instrumentación

Si necesita personalizar qué contenedores deben instrumentarse, utilice la https://github.com/open-telemetry/opentelemetry-operator?tab=readme-ov-file#_multi_container_pods_with_multiple_instrumentations[documentación del operador].

[CAUTION]
====
La auto-instrumentación requiere permisos elevados. Estos permisos son establecidos automáticamente por el operador:

[,yaml]
----
securityContext:
  privileged: true
  runAsUser: 0
----

====


== Ver los resultados

Vaya a SUSE Observability y asegúrese de que el Stackpack Open Telemetry está instalado (a través del menú principal \-> Stackpacks).

Después de un corto tiempo y si sus pods están recibiendo algo de tráfico, debería ser capaz de encontrarlos bajo su nombre de servicio en el Open Telemetry \-> services and service instances overviews. Las trazas aparecerán en el xref:/use/traces/k8sTs-explore-traces.adoc[explorador de trazas] y en la xref:/use/views/k8s-traces-perspective.adoc[perspectiva de trazas] para los componentes de servicio e instancia de servicio. Las métricas de alcance y las métricas específicas del idioma (si están disponibles) estarán disponibles en la xref:/use/views/k8s-metrics-perspective.adoc[perspectiva de métricas] de los componentes.

Si también tiene instalado el stackpack de Kubernetes, los pods instrumentados también tendrán las trazas disponibles en la xref:/use/views/k8s-traces-perspective.adoc[perspectiva de trazas].

== Rancher RBAC

Para que xref:/setup/security/rbac/rbac_rancher.adoc[Rancher RBAC] funcione, los datos de telemetría necesitan tener presentes los siguientes atributos de recursos:

* `+k8s.cluster.name+` - el nombre *del clúster* utilizado por el stackpack de Kubernetes
* `+k8s.namespace.name+` - un *espacio de nombres* gestionado por un *proyecto* Rancher

El Operador Kubernetes inyectará estos atributos por defecto en cualquier dato de telemetría que se envíe.
