= Opciones de configuración

Esta página se centra en las opciones que se utilizan habitualmente cuando se configura el K3 por primera vez. Consulte la documentación sobre xref:advanced.adoc[Opciones avanzadas y configuración] y la documentación sobre comandos de xref:cli/server.adoc[servidor] y xref:cli/agent.adoc[agente] para obtener información más detallada.

[#_configuration_with_install_script]
== Configuración con script de instalación

Como se menciona en la xref:quick-start.adoc[Guía de inicio rápido], puede utilizar el script de instalación disponible en https://get.k3s.io para instalar K3s como un servicio en sistemas basados en systemd y openrc.

Puede utilizar una combinación de `+INSTALL_K3S_EXEC+`, `+K3S_+` variables de entorno y banderas de comando para pasar la configuración a la configuración del servicio.
Las variables de entorno prefijadas, el valor `+INSTALL_K3S_EXEC+` y los argumentos de shell finales se guardan en la configuración del servicio.
Tras la instalación, la configuración puede modificarse editando el archivo de entorno, editando la configuración del servicio o simplemente volviendo a ejecutar el instalador con nuevas opciones.
Para {k3s-product-name} debe utilizar `+INSTALL_K3S_ARTIFACT_URL+` que apunta a la https://scc.suse.com/rancher-docs/rancherprime/latest/en/reference-guide.html#prime-artifacts-url[URL de]

Para ilustrarlo, los siguientes comandos tienen todos el mismo comportamiento de registro de un servidor sin franela y con token:

[,bash]
----
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s INSTALL_K3S_EXEC="server" sh -s - --flannel-backend none --token 12345
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s INSTALL_K3S_EXEC="server --flannel-backend none" K3S_TOKEN=12345 sh -s -
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s K3S_TOKEN=12345 sh -s - server --flannel-backend none
# server is assumed below because there is no K3S_URL
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s INSTALL_K3S_EXEC="--flannel-backend none --token 12345" sh -s -
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s sh -s - --flannel-backend none --token 12345
----

Al registrar un agente, los siguientes comandos tienen todos el mismo comportamiento:

[,bash]
----
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s INSTALL_K3S_EXEC="agent --server https://k3s.example.com --token mypassword" sh -s -
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s INSTALL_K3S_EXEC="agent" K3S_TOKEN="mypassword" sh -s - --server https://k3s.example.com
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s K3S_URL=https://k3s.example.com sh -s - agent --token mypassword
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s K3S_URL=https://k3s.example.com K3S_TOKEN=mypassword sh -s - # agent is assumed because of K3S_URL
----

Para obtener más información sobre todas las variables de entorno, consulte xref:reference/env-variables.adoc[Variables de entorno.]

[IMPORTANT]
.Nota
====
Si establece la configuración al ejecutar el script de instalación, pero no la establece de nuevo al volver a ejecutar el script de instalación, se perderán los valores originales.

El contenido del <<_configuration_file,archivo de configuración>> no es gestionado por el script de instalación.
Si desea que su configuración sea independiente del script de instalación, debe utilizar un archivo de configuración en lugar de pasar variables de entorno o argumentos al script de instalación.
====


== Configuración con binario

El script de instalación se ocupa principalmente de configurar K3s para que se ejecute como un servicio del sistema. Si decide no utilizar el script de instalación, puede ejecutar K3s simplemente descargando el binario de nuestra GitHub https://github.com/k3s-io/k3s/releases/latest[página de publicación], colocándolo en su ruta y ejecutándolo. Esto no es particularmente útil para instalaciones permanentes, pero puede ser útil cuando se realizan pruebas rápidas que no merecen la gestión de K3s como un servicio del sistema.

[,bash]
----
curl -Lo /usr/local/bin/k3s <PRIME-ARTIFACTS-URL>/k3s/v1.35.1%2Bk3s1/k3s; chmod a+x /usr/local/bin/k3s
----

Puede pasar la configuración estableciendo variables de entorno `+K3S_+`:

[,bash]
----
K3S_KUBECONFIG_MODE="644" k3s server
----

O banderas de mando:

[,bash]
----
k3s server --write-kubeconfig-mode=644
----

El agente k3s también puede configurarse de esta manera:

[,bash]
----
k3s agent --server https://k3s.example.com --token mypassword
----

Para más detalles sobre la configuración del servidor K3s, consulte la xref:cli/server.adoc[documentación de`+k3s server+` ]. +
Para más detalles sobre la configuración del agente K3s, consulte la xref:cli/agent.adoc[documentación de`+k3s agent+` ]. +
También puede utilizar la opción `+--help+` para ver una lista de todas las opciones disponibles y sus correspondientes variables de entorno.

[IMPORTANT]
.Banderas a juego
====
Es importante hacer coincidir las banderas críticas en sus nodos de servidor. Por ejemplo, si utiliza la bandera`+--disable servicelb+` o `+--cluster-cidr=10.200.0.0/16+` en su nodo maestro, pero no la establece en otros nodos del servidor, los nodos fallarán al unirse. Imprimirán errores como:
`+failed to validate server configuration: critical configuration value mismatch.+`
Consulte la documentación Configuración del servidor (enlazada más arriba) para obtener más información sobre qué banderas deben configurarse de forma idéntica en los nodos del servidor.
====

== Configuración con imagen de contenedor

La imagen del contenedor K3s (https://hub.docker.com/r/rancher/k3s[\`+docker.io/rancher/k3s+`]) admite los mismos métodos de configuración que el binario disponible en la página de versiones de GitHub.

[#_configuration_file]
== Archivo de configuración

Además de configurar K3s con variables de entorno y argumentos CLI, K3s también puede utilizar un archivo de configuración. El archivo de configuración se carga independientemente de cómo se instale o ejecute K3s.

Por defecto, la configuración se carga desde `+/etc/rancher/k3s/config.yaml+`, y los archivos drop-in se cargan desde `+/etc/rancher/k3s/config.yaml.d/*.yaml+` en orden alfabético.
Esta ruta es configurable a través de `+--config+` CLI flag o `+K3S_CONFIG_FILE+` env var.
Cuando se anula el nombre del archivo de configuración por defecto, también se modifica la ruta del directorio drop-in.

A continuación se muestra un ejemplo de archivo de configuración básico de `+server+`:

[,yaml]
./etc/rancher/k3s/config.yaml
----
write-kubeconfig-mode: "0644"
tls-san:
  - "foo.local"
node-label:
  - "foo=bar"
  - "something=amazing"
cluster-init: true
----

Esto es equivalente a los siguientes argumentos CLI:

[,bash]
----
k3s server \
  --write-kubeconfig-mode "0644"    \
  --tls-san "foo.local"             \
  --node-label "foo=bar"            \
  --node-label "something=amazing"  \
  --cluster-init
----

En general, los argumentos CLI se asignan a sus respectivas claves YAML, y los argumentos CLI repetibles se representan como listas YAML. Los indicadores booleanos se representan como `+true+` o `+false+` en el archivo YAML.

También es posible utilizar tanto un archivo de configuración como argumentos CLI. En estas situaciones, los valores se cargarán desde ambas fuentes, pero los argumentos CLI tendrán prioridad. Para argumentos repetibles como `+--node-label+`, los argumentos CLI sobrescribirán todos los valores de la lista.

=== Comportamiento de la fusión de valores

Si está presente en varios archivos, se utilizará el último valor encontrado para una clave determinada. Se puede añadir un `+++` a la clave para añadir el valor a la cadena o porción existente, en lugar de sustituirla. Todas las apariciones de esta clave en archivos posteriores también requerirán un `+++` para evitar sobrescribir el valor acumulado.

A continuación se muestra un ejemplo de valores combinados de varios archivos de configuración:

[,yaml]
./etc/rancher/k3s/config.yaml
----
token: boop
node-label:
  - foo=bar
  - bar=baz
----

[,yaml]
./etc/rancher/k3s/config.yaml.d/test1.yaml
----
write-kubeconfig-mode: 600
node-taint:
  - alice=bob:NoExecute
----

[,yaml]
./etc/rancher/k3s/config.yaml.d/test2.yaml
----
write-kubeconfig-mode: 777
node-label:
  - other=what
  - foo=three
node-taint+:
  - charlie=delta:NoSchedule
----

El resultado es una configuración final de:

[,yaml]
----
write-kubeconfig-mode: 777
token: boop
node-label:
  - other=what
  - foo=three
node-taint:
  - alice=bob:NoExecute
  - charlie=delta:NoSchedule
----

== Ponerlo todo junto

Todas las opciones anteriores pueden combinarse en un solo ejemplo.

Se crea un archivo `+config.yaml+` en `+/etc/rancher/k3s/config.yaml+`:

[,yaml]
----
token: "secret"
debug: true
----

A continuación, se ejecuta el script de instalación con una combinación de variables de entorno y banderas:

[,bash]
----
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s K3S_KUBECONFIG_MODE="644" INSTALL_K3S_EXEC="server" sh -s - --flannel-backend none
----

O si ya ha instalado el K3s Binary:

[,bash]
----
K3S_KUBECONFIG_MODE="644" k3s server --flannel-backend none
----

Esto resulta en un servidor con:

* Un archivo kubeconfig con permisos `+644+`
* Flannel backend configurado en `+none+`
* El token fijado en `+secret+`
* Registro de depuración activado

== Archivos de configuración de Kubelet

Kubernetes admite la configuración del kubelet a través de banderas CLI y archivos de configuración. La configuración del kubelet a través de las banderas de la CLI está obsoleta desde hace tiempo, pero sigue siendo compatible y es la forma más sencilla de configurar las opciones básicas. Algunas configuraciones avanzadas de kubelet sólo pueden establecerse mediante un archivo de configuración. Para obtener más información, consulte la documentación de Kubernetes para el kubelet https://kubernetes.io/docs/reference/command-line-tools-reference/kubelet/[configuración de los parámetros del kubelet a través de un archivo de configuración] y https://kubernetes.io/docs/tasks/administer-cluster/kubelet-config-file/[].

La compatibilidad con los archivos de configuración drop-in de kubelet o el archivo config (opciones 1 y 2 a continuación) sólo está disponible en v1.32 y superiores. Para versiones anteriores, debe utilizar los args de kubelet directamente (opción número 3 más abajo).

K3s utiliza una configuración kubelet por defecto que se almacena en `+/var/lib/rancher/k3s/agent/etc/kubelet.conf.d/00-k3s-defaults.conf+`. Si desea cambiar los parámetros de configuración por defecto, hay tres maneras de hacerlo:

. Coloque un archivo de configuración drop-in en `+/var/lib/rancher/k3s/agent/etc/kubelet.conf.d/+` *(recomendado)*.
. Utilizando el indicador `+--kubelet-arg=config=$PATHTOFILE+`, donde `+$PATHTOFILE+` es la ruta a un archivo que incluye parámetros de configuración de kubelet (por ejemplo, `+/etc/rancher/k3s/kubelet.conf+`) o el indicador `+--kubelet-arg=config-dir=$PATHTODIR+`, donde `+$PATHTODIR+` es la ruta a un directorio que puede incluir archivos que contienen parámetros de configuración de kubelet (por ejemplo, `+/etc/rancher/k3s/kubelet.conf.d+`).
. Utilizando el indicador `+--kubelet-arg=$FLAG+`, donde `+$FLAG+` es un parámetro de configuración del kubelet (por ejemplo, `+image-gc-high-threshold=100+`). 

Al mezclar banderas CLI kubelet y drop-ins de archivos de configuración, prestar atención a la https://kubernetes.io/docs/tasks/administer-cluster/kubelet-config-file/#kubelet-configuration-merging-order[orden de precedencia].
