= Guía de endurecimiento del CIS
:page-languages: [en, ja, ko, zh]
:revdate: 2025-12-31
:page-revdate: {revdate}

Este documento proporciona una guía prescriptiva para el endurecimiento de una instalación de producción de K3s. Describe las configuraciones y controles necesarios para abordar los controles de referencia de Kubernetes del Centro de Seguridad de Internet (CIS).

K3s tiene una serie de mitigaciones de seguridad aplicadas y activadas por defecto y pasará una serie de controles de Kubernetes CIS sin modificación. Hay algunas excepciones notables que requieren una intervención manual para cumplir plenamente el criterio de referencia CIS:

. Los K3 no modificarán el sistema operativo anfitrión. Cualquier modificación a nivel de host deberá hacerse manualmente.
. Determinados controles de política de CIS para `+NetworkPolicies+` y `+PodSecurityStandards+` (`+PodSecurityPolicies+` en v1.24 y versiones anteriores) restringirán la funcionalidad del clúster. Debe optar por que K3s configure estos añadiendo las opciones apropiadas (habilitación de plugins de admisión) a sus banderas de línea de comandos o archivo de configuración, así como aplicando manualmente las políticas apropiadas. En las secciones siguientes se ofrecen más detalles.

La primera sección (1.1) del CIS Benchmark se ocupa principalmente de los permisos y la propiedad del manifiesto de vaina. K3s no los utiliza para los componentes principales, ya que todo está empaquetado en un único binario.

== Requisitos a nivel de host

Existen dos áreas de requisitos a nivel de host: los parámetros del kernel y la configuración del proceso/directorio etcd. En esta sección se describen estos aspectos.

=== Asegúrese de que `+protect-kernel-defaults+` está configurado

Esta es una bandera de kubelet que hará que el kubelet salga si los parámetros requeridos del kernel no están configurados o están configurados con valores diferentes a los predeterminados del kubelet.

NOTE: `+protect-kernel-defaults+` se expone como una bandera de nivel superior para los K3.

==== Establecer los parámetros del núcleo

Crea un archivo llamado `+/etc/sysctl.d/90-kubelet.conf+` y añade el siguiente fragmento. A continuación, ejecute `+sysctl -p /etc/sysctl.d/90-kubelet.conf+`.

[,bash]
----
vm.panic_on_oom=0
vm.overcommit_memory=1
kernel.panic=10
kernel.panic_on_oops=1
----

== Requisitos de tiempo de ejecución de Kubernetes

Los requisitos de tiempo de ejecución para cumplir con el CIS Benchmark se centran en la seguridad del pod (a través de PSP o PSA), las políticas de red y los registros de auditoría del servidor API. En esta sección se describen estos aspectos.

Por defecto, K3s no incluye ninguna política de seguridad de pods o de red. Sin embargo, el K3s incluye un controlador que aplicará las políticas de red, si se crea alguna. K3s no habilita la auditoría por defecto, por lo que la configuración del registro de auditoría y la política de auditoría deben crearse manualmente. Por defecto, el K3s se ejecuta con los controladores de admisión `+PodSecurity+` y `+NodeRestriction+` habilitados, entre otros.

=== Seguridad

[tabs,sync-group-id=pod-sec]
======
v1.25 y posteriores::
+
--
K3s v1.25 y posteriores admiten https://kubernetes.io/docs/concepts/security/pod-security-admission/[Pod Security Admissions (PSAs)] para controlar la seguridad de los pods. Los PSAs se habilitan pasando la siguiente bandera al servidor K3s:

----
--kube-apiserver-arg="admission-control-config-file=/var/lib/rancher/k3s/server/psa.yaml"
----

La política debe escribirse en un archivo llamado `+psa.yaml+` en el directorio `+/var/lib/rancher/k3s/server+`.

He aquí un ejemplo de anuncio de servicio público conforme:

[,yaml]
----
apiVersion: apiserver.config.k8s.io/v1
kind: AdmissionConfiguration
plugins:
- name: PodSecurity
  configuration:
    apiVersion: pod-security.admission.config.k8s.io/v1beta1
    kind: PodSecurityConfiguration
    defaults:
      enforce: "restricted"
      enforce-version: "latest"
      audit: "restricted"
      audit-version: "latest"
      warn: "restricted"
      warn-version: "latest"
    exemptions:
      usernames: []
      runtimeClasses: []
      namespaces: [kube-system, cis-operator-system]
----
--

v1.24 y anteriores::
+
--
K3s v1.24 y anteriores admiten https://kubernetes.io/docs/concepts/security/pod-security-policy/[Pod Security Policies (PSPs)] para controlar la seguridad de los pods. Los PSP se activan pasando la siguiente bandera al servidor K3s:

----
--kube-apiserver-arg="enable-admission-plugins=NodeRestriction,PodSecurityPolicy"
----

Esto tendrá el efecto de mantener el plugin `+NodeRestriction+` así como habilitar el `+PodSecurityPolicy+`.

Cuando las PSP están activadas, puede aplicarse una política para satisfacer los controles necesarios descritos en la sección 5.2 del Criterio de Referencia CIS.

He aquí un ejemplo de PSP conforme:

[,yaml]
----
apiVersion: policy/v1beta1
kind: PodSecurityPolicy
metadata:
  name: restricted-psp
spec:
  privileged: false                # CIS - 5.2.1
  allowPrivilegeEscalation: false  # CIS - 5.2.5
  requiredDropCapabilities:        # CIS - 5.2.7/8/9
    - ALL
  volumes:
    - 'configMap'
    - 'emptyDir'
    - 'projected'
    - 'secret'
    - 'downwardAPI'
    - 'csi'
    - 'persistentVolumeClaim'
    - 'ephemeral'
  hostNetwork: false               # CIS - 5.2.4
  hostIPC: false                   # CIS - 5.2.3
  hostPID: false                   # CIS - 5.2.2
  runAsUser:
    rule: 'MustRunAsNonRoot'       # CIS - 5.2.6
  seLinux:
    rule: 'RunAsAny'
  supplementalGroups:
    rule: 'MustRunAs'
    ranges:
      - min: 1
        max: 65535
  fsGroup:
    rule: 'MustRunAs'
    ranges:
      - min: 1
        max: 65535
  readOnlyRootFilesystem: false
----

Para que la PSP anterior sea efectiva, necesitamos crear un ClusterRole y un ClusterRoleBinding. También necesitamos incluir una "system unrestricted policy" que es necesaria para los pods a nivel de sistema que requieren privilegios adicionales, y una política adicional que permita las sysctls necesarias para que servicelb funcione correctamente.

Combinando la configuración anterior con la <<_networkpolicies,Política de Red>> descrita en la siguiente sección, se puede colocar un único archivo en el directorio `+/var/lib/rancher/k3s/server/manifests+`. He aquí un ejemplo de archivo `+policy.yaml+`:

NOTE: Las adiciones críticas de Kubernetes como CNI, DNS e Ingress se ejecutan como pods en el espacio de nombres `+kube-system+`. Por lo tanto, este espacio de nombres tendrá una política menos restrictiva para que estos componentes puedan ejecutarse correctamente.

=== NetworkPolicies

CIS requiere que todos los espacios de nombres tengan una política de red aplicada que limite razonablemente el tráfico en los espacios de nombres y pods.

Las políticas de red deben colocarse en el directorio `+/var/lib/rancher/k3s/server/manifests+`, donde se desplegarán automáticamente al iniciarse.

He aquí un ejemplo de política de red conforme.

[,yaml]
----
kind: NetworkPolicy
apiVersion: networking.k8s.io/v1
metadata:
  name: intra-namespace
  namespace: kube-system
spec:
  podSelector: {}
  ingress:
    - from:
      - namespaceSelector:
          matchLabels:
            kubernetes.io/metadata.name: kube-system
----

Con las restricciones aplicadas, el DNS se bloqueará a menos que se permita a propósito. A continuación se muestra una política de red que permitirá la existencia de tráfico para DNS.

[,yaml]
----
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-network-dns-policy
  namespace: <NAMESPACE>
spec:
  ingress:
  - ports:
    - port: 53
      protocol: TCP
    - port: 53
      protocol: UDP
  podSelector:
    matchLabels:
      k8s-app: kube-dns
  policyTypes:
  - Ingress
----

El servidor de métricas y el controlador de entrada Traefik se bloquearán por defecto si no se crean políticas de red que permitan el acceso. Asegúrese de utilizar el yaml de ejemplo que aparece a continuación:

[,yaml]
----
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-all-metrics-server
  namespace: kube-system
spec:
  podSelector:
    matchLabels:
      k8s-app: metrics-server
  ingress:
  - {}
  policyTypes:
  - Ingress
---
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-all-svclbtraefik-ingress
  namespace: kube-system
spec:
  podSelector:
    matchLabels:
      svccontroller.k3s.cattle.io/svcname: traefik
  ingress:
  - {}
  policyTypes:
  - Ingress
---
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-all-traefik-v121-ingress
  namespace: kube-system
spec:
  podSelector:
    matchLabels:
      app.kubernetes.io/name: traefik
  ingress:
  - {}
  policyTypes:
  - Ingress
---
----

[IMPORTANT]
====
Los operadores deben gestionar las políticas de red con normalidad para los espacios de nombres adicionales que se creen.
====


=== Configuración de la auditoría del servidor API

Los requisitos 1.2.22 a 1.2.25 del CIS están relacionados con la configuración de registros de auditoría para el Servidor API. K3s no crea por defecto el directorio de registro y la política de auditoría, ya que los requisitos de auditoría son específicos de las políticas y el entorno de cada usuario.

El directorio de registro, idealmente, debe crearse antes de iniciar K3s. Se recomienda un permiso de acceso restrictivo para evitar la filtración de información potencialmente sensible.

[,bash]
----
sudo mkdir -p -m 700 /var/lib/rancher/k3s/server/logs
----

A continuación se proporciona una política de auditoría inicial para registrar los metadatos de las solicitudes. La política debe escribirse en un archivo llamado `+audit.yaml+` en el directorio `+/var/lib/rancher/k3s/server+`. Encontrará información detallada sobre la configuración de políticas para el servidor API en la Kubernetes https://kubernetes.io/docs/tasks/debug-application-cluster/audit/[documentación de].

[,yaml]
----
apiVersion: audit.k8s.io/v1
kind: Policy
rules:
- level: Metadata
----

Ambas configuraciones deben pasarse como argumentos al Servidor API como:

[tabs]
======
config::
+
[,yaml]
----
kube-apiserver-arg:
  - 'admission-control-config-file=/var/lib/rancher/k3s/server/psa.yaml'
  - 'audit-log-path=/var/lib/rancher/k3s/server/logs/audit.log'
  - 'audit-policy-file=/var/lib/rancher/k3s/server/audit.yaml'
  - 'audit-log-maxage=30'
  - 'audit-log-maxbackup=10'
  - 'audit-log-maxsize=100'
----

cmdline::
+
[,bash]
----
--kube-apiserver-arg='audit-log-path=/var/lib/rancher/k3s/server/logs/audit.log'
--kube-apiserver-arg='audit-policy-file=/var/lib/rancher/k3s/server/audit.yaml'
----
======

Los K3 deben reiniciarse para cargar la nueva configuración.

[,bash]
----
sudo systemctl daemon-reload
sudo systemctl restart k3s.service
----

== Operaciones manuales
Los siguientes son controles por los que el K3s actualmente no pasa con la configuración anterior aplicada. Estos controles requieren una intervención manual para cumplir plenamente el criterio de referencia CIS.

=== Control 1.1.20

Asegúrese de que los permisos del archivo del certificado PKI de Kubernetes están configurados en 600 o más restrictivo (Manual).

.Remediación
[%collapsible]
======
Los archivos de certificados PKI de K3 se almacenan en `+/var/lib/rancher/k3s/server/tls/+` con permiso 644.
Para remediarlo, ejecute el siguiente comando:

[,bash]
----
chmod -R 600 /var/lib/rancher/k3s/server/tls/*.crt
----
======

=== Control 1.2.9

Asegúrese de que el plugin de control de admisión EventRateLimit está activado.

.Remediación
[%collapsible]
======
Siga la https://kubernetes.io/docs/reference/access-authn-authz/admission-controllers/#eventratelimit[documentación de] Kubernetes y establezca los límites deseados en un archivo de configuración.
Para esta y otras configuraciones psa, esta documentación utiliza /var/lib/rancher/k3s/server/psa.yaml.
A continuación, edite el archivo de configuración de K3s /etc/rancher/k3s/config.yaml y establezca los siguientes parámetros.

[,yaml]
----
kube-apiserver-arg:
  - "enable-admission-plugins=NodeRestriction,EventRateLimit"
  - "admission-control-config-file=/var/lib/rancher/k3s/server/psa.yaml"
----
======

=== Control 1.2.11

Asegúrese de que el plugin de control de admisión AlwaysPullImages está activado.

.Remediación
[%collapsible]
======
Permisivo, según las directrices del CIS: "Esta configuración podría afectar a los clústeres aislados o sin conexión, que tienen imágenes precargadas y no tienen acceso a un registro para extraer imágenes en uso". Esta configuración no es apropiada para los clusters que utilizan esta configuración."
Edite el archivo de configuración de K3s /etc/rancher/k3s/config.yaml y establezca el siguiente parámetro.

[,yaml]
----
kube-apiserver-arg:
  - "enable-admission-plugins=...,AlwaysPullImages,..."
----
======

=== Control 1.2.21

Asegúrese de que el argumento --request-timeout está configurado correctamente.

.Remediación
[%collapsible]
======
Permisivo, según las directrices del CIS, "se recomienda establecer este límite según convenga y cambiar el límite predeterminado de 60 segundos sólo si es necesario".
Edite el archivo de configuración de K3s /etc/rancher/k3s/config.yaml y establezca el siguiente parámetro si es necesario. Por ejemplo,

[,yaml]
----
kube-apiserver-arg:
  - "request-timeout=300s"
----
======

=== Control 4.2.13 

Asegúrese de que se establece un límite en los pod PID

.Remediación
[%collapsible]
======
Si utiliza un archivo de configuración K3s /etc/rancher/k3s/config.yaml, edite el archivo para establecer `+podPidsLimit+` a

[,yaml]
----
kubelet-arg:
  - "pod-max-pids=<value>"
----
======

=== Control 5.X

Todos los Controles 5.X están relacionados con la configuración de políticas de Kubernetes. Los K3 no aplican estos controles por defecto.

Consulte la xref:security/self-assessment-1.8.adoc#_5_1_rbac_and_service_accounts[sección 5 de CIS 1.8] para obtener más información sobre cómo crear y aplicar estas políticas.

==== Control 5.1.5

[NOTE]
====
La corrección para alcanzar la puntuación de aprobado sólo es necesaria para cis-1,9.
====

.Remediación
[%collapsible]
======
Cree cuentas de servicio explícitas siempre que una carga de trabajo de Kubernetes requiera acceso específico al servidor API de Kubernetes.
Como mínimo, parchea lo siguiente:

[,bash]
----
kubectl patch serviceaccount --namespace default default --patch '{"automountServiceAccountToken": false}'
kubectl patch serviceaccount --namespace kube-node-lease default --patch '{"automountServiceAccountToken": false}'
kubectl patch serviceaccount --namespace kube-public default --patch '{"automountServiceAccountToken": false}'
----
======
