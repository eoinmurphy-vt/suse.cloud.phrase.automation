= Almacén de datos en clúster
:page-languages: [en, ja, ko, zh]
:revdate: 2025-06-16
:page-revdate: {revdate}

La capacidad de ejecutar Kubernetes utilizando un almacén de datos distinto de etcd diferencia a K3s de otras distribuciones de Kubernetes. Esta función proporciona flexibilidad a los operadores de Kubernetes. Las opciones de almacén de datos disponibles le permiten seleccionar el que mejor se adapte a su caso de uso. Por ejemplo:

* Si su equipo no tiene experiencia en el funcionamiento de etcd, puede elegir una base de datos SQL de nivel empresarial como MySQL o PostgreSQL.
* Si necesita ejecutar un clúster sencillo y de corta duración en su entorno CI/CD, puede utilizar la base de datos SQLite incorporada
* Si desea implementar Kubernetes en el borde y necesita una solución de alta disponibilidad, pero no puede permitirse la sobrecarga operativa de gestionar una base de datos en el borde, puede utilizar el almacén de datos de HA integrado de K3s construido sobre etcd integrado.

K3s admite las siguientes opciones de almacén de datos:

* *Embedded https://www.sqlite.org/index.html[SQLite]* +
SQLite no puede utilizarse en clusters con múltiples servidores. +
SQLite es el almacén de datos por defecto, y se utilizará si no hay otra configuración de almacén de datos, y no hay archivos de base de datos etcd incrustados en el disco.
* *etcd integrado* +
Consulte la documentación de etcd incrustado de xref:datastore/ha-embedded.adoc[alta] disponibilidad para obtener más información sobre el uso de etcd incrustado con varios servidores.
Embedded etcd se seleccionará automáticamente si K3s está configurado para inicializar un nuevo clúster etcd, unirse a un clúster etcd existente, o si los archivos de base de datos etcd están presentes en el disco durante el arranque.
* *Base de datos externa* +
Consulte la documentación xref:datastore/ha.adoc[BD externa de alta] disponibilidad para obtener más información sobre el uso de almacenes de datos externos con varios servidores. +
Se admiten los siguientes almacenes de datos externos:
 ** https://etcd.io/[etcd] (certificado para la versión 3.5.21)
 ** https://www.mysql.com/[MySQL] (certificado para las versiones 5.7 y 8.0)
 ** https://mariadb.org/[MariaDB] (certificado para las versiones 10.11 y 11.4)
 ** https://www.postgresql.org/[PostgreSQL] (certificado para las versiones 15.12, 16.7 y 17.3)

[CAUTION]
.Declaración preparada Apoyo
====
K3s requiere el apoyo de los estados preparados de la BD. Esto significa que los agrupadores de conexiones como https://www.pgbouncer.org/faq.html#how-to-use-prepared-statements-with-transaction-pooling[] PgBouncer pueden requerir una configuración adicional para funcionar con los K3.
====

[WARNING]
.Configuraciones Multimaster
====
Las bases de datos multimaestro que establecen `+auto_increment_increment+` o `+auto_increment_offset+` mayores que 1 no son compatibles. Kine espera que la revisión comience en 0 en 0 y siempre avance exactamente 1 cuando se inserta con éxito una tecla. Esto afecta a productos como Galera para MySQL/MariaDB. 
====

[#_external_datastore_configuration_parameters]
== Parámetros de configuración del almacén de datos externo

Si desea utilizar un almacén de datos externo, como PostgreSQL, MySQL o etcd, debe establecer el parámetro `+datastore-endpoint+` para que K3s sepa cómo conectarse a él. También puede especificar parámetros para configurar la autenticación y el cifrado de la conexión. La siguiente tabla resume estos parámetros, que se pueden pasar como indicadores de la CLI o como variables de entorno.

|===
| CLI Flag | Variable de entorno | Descripción

| `+--datastore-endpoint+`
| `+K3S_DATASTORE_ENDPOINT+`
| Especifique una cadena de conexión PostgreSQL, MySQL o etcd. Se trata de una cadena utilizada para describir la conexión al almacén de datos. La estructura de esta cadena es específica de cada backend y se detalla a continuación.

| `+--datastore-cafile+`
| `+K3S_DATASTORE_CAFILE+`
| Archivo de autoridad de certificación (CA) TLS utilizado para ayudar a proteger la comunicación con el almacén de datos. Si su almacén de datos sirve solicitudes a través de TLS utilizando un certificado firmado por una autoridad de certificación personalizada, puede especificar esa CA utilizando este parámetro para que el cliente K3s pueda verificar correctamente el certificado.

| `+--datastore-certfile+`
| `+K3S_DATASTORE_CERTFILE+`
| Archivo de certificado TLS utilizado para la autenticación basada en certificados de cliente en su almacén de datos. Para utilizar esta función, el almacén de datos debe estar configurado para admitir la autenticación basada en certificados de cliente. Si especifica este parámetro, debe especificar también el parámetro `+datastore-keyfile+`.

| `+--datastore-keyfile+`
| `+K3S_DATASTORE_KEYFILE+`
| Archivo de claves TLS utilizado para la autenticación basada en certificados de cliente en su almacén de datos. Consulte el parámetro anterior `+datastore-certfile+` para más detalles.
|===

Como práctica recomendada, recomendamos establecer estos parámetros como variables de entorno en lugar de como argumentos de línea de comandos, de modo que las credenciales de la base de datos u otra información confidencial no queden expuestas como parte de la información del proceso.

[#_datastore_endpoint_format_and_functionality]
== Formato y funcionalidad del almacén de datos

Como ya se ha mencionado, el formato del valor pasado al parámetro `+datastore-endpoint+` depende del backend del datastore. A continuación se detalla este formato y funcionalidad para cada almacén de datos externo compatible.

[tabs,sync-group-id=ext-db]
======
PostgreSQL::
+
--
En su forma más común, el parámetro datastore-endpoint para PostgreSQL tiene el siguiente formato:

`+postgres://username:password@hostname:port/database-name+`

Existen parámetros de configuración más avanzados. Para más información, consulte https://godoc.org/github.com/lib/pq.

Si especifica un nombre de base de datos y no existe, el servidor intentará crearla.

Si sólo proporciona `+postgres://+` como punto final, K3s intentará hacer lo siguiente:

* Conéctese a localhost utilizando `+postgres+` como nombre de usuario y contraseña
* Crear una base de datos denominada `+kubernetes+`
--

MySQL / MariaDB::
+
--
En su forma más común, el parámetro `+datastore-endpoint+` para MySQL y MariaDB tiene el siguiente formato:

`+mysql://username:password@tcp(hostname:3306)/database-name+`

Existen parámetros de configuración más avanzados. Para más información, consulte https://github.com/go-sql-driver/mysql#dsn-data-source-name

Tenga en cuenta que, debido a un de https://github.com/k3s-io/k3s/issues/1093[problema conocido] en los K3, no puede configurar el parámetro `+tls+`. Se admite la comunicación TLS, pero no puede, por ejemplo, establecer este parámetro en "skip-verify" para hacer que K3s omita la verificación del certificado.

Si especifica un nombre de base de datos y no existe, el servidor intentará crearla.

Si sólo proporciona `+mysql://+` como punto final, K3s intentará hacer lo siguiente:

* Conéctese al socket MySQL en `+/var/run/mysqld/mysqld.sock+` utilizando el usuario `+root+` y sin contraseña
* Crear una base de datos con el nombre `+kubernetes+`
--

etcd::
+
--
En su forma más común, el parámetro `+datastore-endpoint+` para etcd tiene el siguiente formato:

`+\https://etcd-host-1:2379,https://etcd-host-2:2379,https://etcd-host-3:2379\+`

Lo anterior supone un clúster típico de tres nodos etcd. El parámetro puede aceptar una o más URLs etcd separadas por comas.
--
