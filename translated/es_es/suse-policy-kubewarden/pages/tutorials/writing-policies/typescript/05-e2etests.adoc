Esta sección muestra cómo puedes escribir pruebas de extremo a extremo que se ejecuten contra el binario WebAssembly real producido por la cadena de herramientas de compilación Javy.

== Requisitos previos

Recuerda que necesitas estas herramientas en tu máquina de desarrollo:

* `+bats+`: Se utiliza para escribir las pruebas y automatizar su ejecución.
* `+kwctl+` >= v1.30: Herramienta CLI proporcionada por {project-name} para ejecutar sus políticas fuera de Kubernetes, entre otras acciones. Se trata en xref:/tutorials/testing-policies/index.adoc[la sección de políticas de pruebas] de la documentación.

== Pruebas de redacción

Utilizará `+bats+` para escribir y automatizar sus pruebas. Cada prueba consta de los siguientes pasos:

. Ejecute la política utilizando `+kwctl run+` directamente con el archivo de recursos JSON.
. Realice aserciones contra la salida producida por `+kwctl+`.

Todas las pruebas de extremo a extremo van en un archivo llamado `+e2e.bats+`. El proyecto de andamiaje incluye un ejemplo, `+e2e.bats+`. Es necesario ampliar su contenido para proporcionar una cobertura de prueba completa del comportamiento de su póliza.

== Archivos de datos de prueba

Los casos de prueba de extremo a extremo prueban la creación de Recursos Kubernetes específicos.

Por ejemplo, al probar la creación de un recurso Pod:

[source,json]
----
{
  "apiVersion": "v1",
  "kind": "Pod",
  "metadata": {
    "name": "test-pod"
  },
  "spec": {
    "containers": [
      {
        "name": "nginx",
        "image": "nginx:latest"
      }
    ]
  }
}
----

El caso de prueba de extremo a extremo utiliza un archivo JSON que contiene un AdmissionReview de Kubernetes. Estos objetos JSON AdmissionReviews JSON son los que se envían contra la API de Kubernetes para que sean xref:/reference/spec/03-validating-policies.adoc#an-example[validados].

Puede obtener objetos JSON de AdmissionReviews utilizando `+kwctl scaffold admission-request+`.
Eche un vistazo a `+test_data/pod.json+` para ver un AdmissionRequest JSON con un Pod.

== Casos de prueba básicos

La plantilla ya proporciona varias pruebas básicas. Este es el aspecto que debería tener el archivo `+e2e.bats+` completo:

[source,bash]
----
#!/usr/bin/env bats
----

Aquí están las pruebas existentes con explicaciones:

=== Prueba 1: Rechazar nombres de host denegados

[source,bash]
----
@test "reject because hostname is on deny list" {
  run kwctl run annotated-policy.wasm -r test_data/pod_with_hostname.json --settings-json '{"denied_hostnames": ["forbidden-host", "test-hostname"]}'

  # this prints the output when one of the checks below fails
  echo "output = ${output}"

  # request rejected
  [ "$status" -eq 0 ]
  [ $(expr "$output" : '.*allowed.*false') -ne 0 ]
  [ $(expr "$output" : ".*Pod hostname 'test-hostname' is not allowed.*") -ne 0 ]
}
----

Esta prueba asegura que la política rechaza Pods cuando su nombre de host está en la lista de denegación.

=== Prueba 2: Aceptar nombres de host permitidos

[source,bash]
----
@test "accept because hostname is not on the deny list" {
  run kwctl run annotated-policy.wasm -r test_data/pod_with_hostname.json --settings-json '{"denied_hostnames": ["forbidden-host"]}'

  # this prints the output when one of the checks below fails
  echo "output = ${output}"

  # request accepted
  [ "$status" -eq 0 ]
  [ $(expr "$output" : '.*allowed.*true') -ne 0 ]
}
----

Esta prueba verifica que la política acepta Pods cuando su nombre de host no está en la lista de denegación.

=== Prueba 3: Aceptar cuando no se proporciona ninguna configuración

[source,bash]
----
@test "accept because the deny list is empty" {
  run kwctl run annotated-policy.wasm -r test_data/pod_with_hostname.json

  # this prints the output when one of the checks below fails
  echo "output = ${output}"

  # request accepted
  [ "$status" -eq 0 ]
  [ $(expr "$output" : '.*allowed.*true') -ne 0 ]
}
----

Esta prueba garantiza que la política acepta solicitudes cuando no se proporciona ninguna configuración.

=== Prueba 4: Aceptar pods sin nombre de host

[source,bash]
----
@test "accept because pod has no hostname set" {
  run kwctl run annotated-policy.wasm -r test_data/pod.json --settings-json '{"denied_hostnames": ["forbidden-host"]}'

  # this prints the output when one of the checks below fails
  echo "output = ${output}"

  # request accepted (no hostname to check)
  [ "$status" -eq 0 ]
  [ $(expr "$output" : '.*allowed.*true') -ne 0 ]
}
----

Esta prueba verifica que la política acepta Pods sin nombres de host independientemente de la lista de denegación.

=== Prueba 5: Aceptar recursos que no sean Pod

[source,bash]
----
@test "accept non-pod resources" {
  run kwctl run annotated-policy.wasm -r test_data/service.json --settings-json '{"denied_hostnames": ["forbidden-host"]}'

  # this prints the output when one of the checks below fails
  echo "output = ${output}"

  # request accepted (not a pod)
  [ "$status" -eq 0 ]
  [ $(expr "$output" : '.*allowed.*true') -ne 0 ]
}
----

Esta prueba asegura que la política acepta recursos que no son Pods, ya que la política sólo valida nombres de host Pod.

== Ejecución de las pruebas

Puede ejecutar todas las pruebas de extremo a extremo utilizando este comando:

[source,console]
----
make e2e
----

Esto produce resultados como:

[source,console]
----
bats e2e.bats
e2e.bats
 ✓ reject because hostname is on the deny list
 ✓ accept because hostname is not on the deny list
 ✓ accept because the deny list is empty
 ✓ accept because pod has no hostname set
 ✓ accept non-pod resources
 ✓ accept valid settings
 ✓ reject with multiple denied hostnames
 ✓ accept with empty denied hostnames array

8 tests, 0 failures
----

== Comprender los resultados de las pruebas

Cada prueba utiliza `+kwctl+` para ejecutar la política y las comprobaciones:

* Estado de salida: El comando `+kwctl+` debe salir con el estado 0 para una ejecución exitosa de la política.
* Campo permitido: La salida JSON contiene un campo `+allowed+` que indica si la solicitud ha sido aceptada.
* Contenido del mensaje: Para las solicitudes rechazadas, la salida contiene un mensaje de error descriptivo.

Las sentencias `+echo "output = $\{output}"+` de cada prueba ayudan a depurar mostrando la salida real de la política cuando falla una prueba.

== Conclusión

Las pruebas de extremo a extremo proporcionan una cobertura completa del comportamiento de la política mediante pruebas con el binario WebAssembly real. Esto asegura que la política funciona correctamente cuando se despliega en {short-project-name}, no sólo en el entorno de desarrollo de TypeScript.
