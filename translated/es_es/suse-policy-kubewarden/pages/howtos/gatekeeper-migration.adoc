= Migración de directivas Gatekeeper a {project-name}
:revdate: 2026-03-18
:page-revdate: {revdate}
:description: Configure una política {project-name} migrando una política Gatekeeper existente a través de dos pasos principales: compilación del código Rego en WebAssembly.
:keywords: ["{project-name}", kubewarden, gatekeeper]
:doc-persona:
  [kubewarden-operator, kubewarden-integrator, kubewarden-policy-developer]
:doc-type: [howto]
:doc-topic: [operator-manual]

Esta guía le muestra cómo convertir una política Gatekeeper existente en una política{project-name}. Este proceso consta de dos pasos principales:
. Compilar el programa Rego en un módulo WebAssembly (Wasm).
. Distribuya el módulo WebAssembly como una directiva {short-project-name}.

El xref:/tutorials/writing-policies/rego/01-intro-rego.adoc[tutorial de directivas Rego] cubre la mayor parte del proceso de compilación del código Rego en un módulo WebAssembly. Esta guía se centra en el proceso paso a paso de extracción de las definiciones de recursos personalizados (CRD) de Gatekeeper y su migración a una política funcional de {short-project-name}. Utiliza una política de demostración Gatekeeper básica.

== Requisitos previos

* link:https://github.com/open-policy-agent/opa/releases[opa]: utiliza esta herramienta para construir el código en wasm. Esta guía se ha redactado utilizando la versión `+v1.5.1+`.
* link:https://github.com/kubewarden/kwctl/releases[`+kwctl+`]herramienta que utiliza para preparar y ejecutar el módulo de montaje web {short-project-name} 
* link:https://github.com/bats-core/bats-core[`+bats+`]herramienta utilizada para realizar pruebas de extremo a extremo. Si decide escribir este tipo de pruebas
* link:https://github.com/mikefarah/yq[`+yq+`]: herramienta para extraer datos de archivos yaml

== Antes de migrar sus pólizas

Antes de iniciar el proceso de migración de políticas Gatekeeper, considere la posibilidad de utilizar políticas ya disponibles en el {short-project-name}link:https://artifacthub.io/packages/search?kind=13[catálogo]. Algunas de laslink:https://github.com/kubewarden/rego-policies-library[políticas de] están disponibles públicamente Las políticas de OPA y Gatekeeper se han migrado a {short-project-name}.

Eche también un vistazo a nuestra documentaciónxref:/explanations/comparisons/opa-comparison.adoc[comparativa]entre {short-project-name} y Gatekeeper.

== Primer paso: Inicialice su proyecto de política {short-project-name} 

En primer lugar, utilice la {short-project-name} Gatekeeperlink:https://github.com/kubewarden/gatekeeper-policy-template[plantilla] para crear una estructura de proyecto de política básica. Esto proporciona objetivos Makefile para construir y probar su política. Después de crear el código de la política a partir de la plantilla, ejecute los comandos Makefile para comprobar que la política se compila y las pruebas se ejecutan correctamente:

[source,console]
----
$ make policy.wasm test
opa build -t wasm -e policy/violation -o bundle.tar.gz policy.rego
tar xvf bundle.tar.gz /policy.wasm
tar: Removing leading `/' from member names
/policy.wasm
rm bundle.tar.gz
touch policy.wasm # opa creates the bundle with unix epoch timestamp, fix it
opa test *.rego
PASS: 2/2
----

== Segundo paso: Migrar el código de la política Gatekeeper

Ahora, comience a migrar la política del Gatekeeper. Se trata de convertir un`+ConstraintTemplate+` y sus recursos asociados `+Constraint+` en una política {short-project-name}. En un contexto {short-project-name}, considere `+ConstraintTemplate+` como el código de política central, mientras que las instancias `+Constraint+` se traducen en instancias de política que se ejecutan dentro de {short-project-name}.

En primer lugar, copie el código Rego de su `+ConstraintTemplate+` en el archivo `+policy.rego+`que ha generado la plantilla {short-project-name}. Para este ejemplo, debe utilizar la siguientebásicalink:https://github.com/open-policy-agent/gatekeeper/blob/896d6620f9c16d7a5d91a74a6a4260db8d735640/demo/basic/demo.sh#L1[política]del repositorio Gatekeeper.

[.details]
.Política de Gatekeeper yaml
[%collapsible]
====
[source,yaml]
----
apiVersion: templates.gatekeeper.sh/v1
kind: ConstraintTemplate
metadata:
  name: k8srequiredlabels
spec:
  crd:
    spec:
      names:
        kind: K8sRequiredLabels
      validation:
        # Schema for the `+parameters+` field
        openAPIV3Schema:
          type: object
          properties:
            labels:
              type: array
              items:
                type: string
  targets:
    - target: admission.k8s.gatekeeper.sh
      rego: |
        package k8srequiredlabels

        violation[{"msg": msg, "details": {"missing_labels": missing}}] {
          provided := {label | input.review.object.metadata.labels[label]}
          required := {label | label := input.parameters.labels[_]}
          missing := required - provided
          count(missing) > 0
          msg := sprintf("you must provide labels: %v", [missing])
        }
----
====

Copie el fragmento de código Rego del campo `+rego+` en su archivo `+policy.rego+`:

[source,console]
----
cat gatekeeper/demo/basic/templates/k8srequiredlabels_template.yaml | yq ".spec.targets[0].rego" > policy.rego
----

=== Adaptar el código Rego para {short-project-name}

Debe asegurarse de que el nombre `+package+` utilizado dentro del código Rego es `+policy+`.
Este es el valor esperado en muchos lugares por la plantilla {short-project-name} Gatekeeper.

Si no lo cambia, tendrá errores al crear la política y ejecutar sus pruebas de extremo a extremo.

Por ejemplo, la política de demostración que estamos convirtiendo está definida dentro del paquete `+k8srequiredlabels+`, este valor debe cambiarse para que sea `+policy+`.

Así debe ser el contenido del archivo `+policy.rego+`:

[source,rego]
----
package policy

violation[{"msg": msg, "details": {"missing_labels": missing}}] {
provided := {label | input.review.object.metadata.labels[label]}
required := {label | label := input.parameters.labels[_]}
missing := required - provided
count(missing) > 0
msg := sprintf("you must provide labels: %v", [missing])
}
----

Intentar compilar el código después de este cambio podría revelar nuevos errores de compilación:

[source,console]
----
opa build -t wasm -e policy/violation -o bundle.tar.gz policy.rego
error: load error: 2 errors occurred during loading:
policy.rego:3: rego_parse_error: `+if+` keyword is required before rule body
policy.rego:3: rego_parse_error: `+contains+` keyword is required for partial set rules
make: *** [Makefile:4: policy.wasm] Error 1
----

El autor de la política debe corregir estos errores para que la CLI `+opa+` pueda compilar el código correctamente. Los cambios específicos pueden variar en función de la versión de `+opa+` y del código de la política original. Como estamos migrando una política de rego anterior a OPA v1, necesitamos actualizar el código para que sea compatible con v1. El código final de `+policy.rego+` es el siguiente:

[source,rego]
----
package policy

violation contains {"msg": msg, "details": {"missing_labels": missing}} if {
  provided := {label | input.review.object.metadata.labels[label]}
  required := {label | label := input.parameters.labels[_]}
  missing := required - provided
  count(missing) > 0
  msg = sprintf("you must provide labels: %v", [missing])
}
----

Después de ajustar el código, construya la política:

[source,console]
----
$ make policy.wasm
opa build -t wasm -e policy/violation -o bundle.tar.gz policy.rego
tar xvf bundle.tar.gz /policy.wasm
tar: Removing leading `/' from member names
/policy.wasm
rm bundle.tar.gz
touch policy.wasm # opa creates the bundle with unix epoch timestamp, fix it
----

Hay más información sobre cómo construir políticas Gatekeeper en nuestroxref:/tutorials/writing-policies/rego/gatekeeper/03-build-and-run.adoc[tutorial].

=== Código de la Política de Rego y compatibilidad con OPA v1.0.0

Con el lanzamiento de OPA (Open Policy Agent)link:https://github.com/open-policy-agent/opa/releases/tag/v1.0.0[.0.0] v1 en diciembre de 2024, se introdujo un cambio de última hora en relación con la sintaxis de la política de Rego.

Antes, `+if+` para todas las definiciones de reglas y `+contains+` para las reglas multivalor eran opcionales; ahora, son obligatorias. Este cambio afecta a la mayoría de las pólizas más antiguas.

He aquí un resumen de lo que debe saber:

* *OPA v1.0.0 Sintaxis*: OPA v1.0.0 exige el uso de `+if+` para todas las definiciones de reglas y `+contains+` para las reglas multivalor. Las políticas que no se adhieran a esta sintaxis se romperán.
* *Compatibilidad con versiones anteriores*: Si necesita compilar políticas anteriores que no utilizan la nueva sintaxis v1.0.0, debe proporcionar la bandera `+--v0-compatible+` al comando `opa build`.
* *Integración de Gatekeeper*: Gatekeeper actualizó su dependencia de OPA a v1.0.0 en su link:https://github.com/open-policy-agent/gatekeeper/releases/tag/v3.19.0[versión] v3.
* *Versión Rego en plantillas Gatekeeper*: Gatekeeper asume que se utiliza la sintaxis `+v0+` a menos que la plantilla especifique explícitamente `+version: "v1"+` dentro del campo `+source+` bajo `+code.engine: Rego+`.

Consulte link:https://open-policy-agent.github.io/gatekeeper/website/docs/constrainttemplates/#enable-opa-rego-v1-syntax-in-constrainttemplates[esta sección]de la documentación de Gatekeeper para obtener más detalles sobre cómo se gestionan las versiones `+v0+` y `+v1+` de Rego.

Lo que esto significa para usted:

* Si el CR de Gatekeeper no especifica una versión de Rego, implica que se va a utilizar `+v0+`. Debe crear la política utilizando el comando `make OPA_V0_COMPATIBLE=true`.
* Si el CR Gatekeeper especifica explícitamente `+version: "v1"+`, debe construir la política sin ninguna variable de entorno establecida.

== Tercer paso: Actualizar y ejecutar pruebas

Aunque es muy recomendable, los autores de políticas pueden omitir la creación de pruebas para la versión inicial de una política. Si esto le ocurre, tendrá que desactivar los objetivos Makefile utilizados para ejecutar las pruebas. No puede eliminar estos objetivos por completo, ya que las tareas de CI predeterminadas esperan que estén definidos. En su lugar, debe sustituir los comandos que llaman a `+opa+` y `+bats+` por una operación "no-op". Por ejemplo, puede utilizar un comando `+echo+` para imprimir una explicación de por qué no se están ejecutando las pruebas.

La plantilla {short-project-name} Gatekeeper incluye tanto pruebas unitarias de Rego como pruebas de extremo a extremo (e2e) mediante Bats y `+kwctl+`. Si tiene previsto incluir pruebas, ambos conjuntos deben adaptarse a su política.

Si su política Gatekeeper ya tiene pruebas Rego, puede copiarlas en el archivo`+policy_test.rego+`. Se ejecutan automáticamente al ejecutar el comando`+make test+`.

[CAUTION]
====
Tenga en cuenta que cualquier prueba de Rego que escriba en `+policy_test.rego+` está sujeta a los mismos problemas de compatibilidad que se detallan en la sección <<_rego_policy_code_and_opa_v1_0_0_compatibility,Compatibilidad del código de Rego Policy y OPA v1.0.0>>.
====

La política que está migrando en esta guía no tiene pruebas; tenemos que añadirlas nosotros mismos. Por lo tanto, actualizaremos el archivo de pruebas `+policy_test.rego+` con algunas pruebas básicas:

[source,rego]
----
package policy

review_required_labels := {
    "parameters": {"labels": ["test"]},
    "review": {"object": {"metadata": {"labels": {"test": "value"}}}},
}

review_missing_labels := {
    "parameters": {"labels": ["test"]},
    "review": {"object": {"metadata": {"labels": {"other": "value"}}}},
}

test_accept if {
    r = review_required_labels
    res = violation with input as r
    count(res) = 0
}

test_reject if {
    r = review_missing_labels
    res = violation with input as r
    count(res) = 1
}
----

Ahora, ejecutando `+make test+` debería validar su política:

[source,console]
----
$ make policy.wasm test
opa build -t wasm -e policy/violation -o bundle.tar.gz policy.rego
tar xvf bundle.tar.gz /policy.wasm
tar: Removing leading `/' from member names
/policy.wasm
rm bundle.tar.gz
touch policy.wasm # opa creates the bundle with unix epoch timestamp, fix it
opa test *.rego
PASS: 2/2
----

A continuación, actualice el archivo de pruebas e2e (`+e2e.bats+`):

[source,bash]
----
#!/usr/bin/env bats

@test "accept because required label is present" {
  run kwctl run -e gatekeeper annotated-policy.wasm --settings-path test_data/settings.json --request-path test_data/accept_deploy_request.json

  # this prints the output when one the checks below fails
  echo "output = ${output}"

  # request accepted
  [ "$status" -eq 0 ]
  [ $(expr "$output" : '.*allowed.*true') -ne 0 ]
}

@test "reject because required label is missing" {
run kwctl run -e gatekeeper annotated-policy.wasm --settings-path test_data/settings.json --request-path test_data/reject_deploy_request.json

  # this prints the output when one the checks below fails
  echo "output = ${output}"

  # request rejected
  [ "$status" -eq 0 ]
  [ $(expr "$output" : '.*allowed.*false') -ne 0 ]
  [ $(expr "$output" : '.*message.*you must provide labels: \[test\]') -ne 0 ]
}
----

Deberá crear los archivos `+test_data/settings.json+`,`+test_data/accept_deploy_request.json+` y `+test_data/reject_deploy_request.json+`para poder realizar estas pruebas.

[.details]
.test_data/settings.json
[%collapsible]
====
[source,json]
----
{
  "labels": ["test"]
}
----
====

No incluiremos aquí el contenido completo de `+accept_deploy_request.json+` y`+reject_deploy_request.json+`, ya que el JSON de `+AdmissionRequest+` puede ser bastante largo y queremos que esta guía sea concisa. Sin embargo, puede utilizar el comandoxref:/reference/kwctl-cli.adoc#_kwctl_scaffold_admission_request[`+kwctl scaffold+`]para generar estos archivos. La clave para esta guía es que a una solicitud le debe faltar la etiqueta requerida, mientras que la otra debe tener la etiqueta definida.

Compruebe si las pruebas e2e están pasando:

[source,console]
----
$ make e2e-tests
bats e2e.bats
e2e.bats
  ✓ accept because required label is present
  ✓ reject because required label is missing
----

[IMPORTANT]
====
Los parámetros de la política (por ejemplo, las etiquetas en este ejemplo) tienen su origen en la configuración de la política. Esto le permite desplegar múltiples instancias de la misma política con diferentes parámetros/configuraciones, de forma similar a como funcionan las Restricciones en Gatekeeper.
====

== Paso 4: Prepare `+metadata.yml+` para su distribución

Ahora que ya dispone de una política funcional, prepare el archivo `+metadata.yml+` para su distribución. Este archivo define anotaciones con la descripción de la política, el autor, la licencia y otra información esencial. Fundamentalmente, define las`+rules+` que especifican qué recursos y verbos puede validar la política. Esta información hace que el comando `+kwctl scaffold+` genere el manifiesto para desplegar la política en su clúster.

Los CRD de `+Constraints+` de Gatekeeper, que son instancias de políticas definidas en`+ConstraintTemplates+`, especifican qué recursos evalúa una instancia de política.
Por lo tanto, si tiene `+Constraints+` existentes que aplican un `+ConstraintTemplate+`, ofrecen una buena referencia para los recursos que debe definir en su archivo`+metadata.yml+`. Por ejemplo, en el ejemplo de Gatekeeper utilizado anteriormente, el`+K8sRequiredLabels+` `+Constraint+` creado a partir del `+k8srequiredlabels+``+ConstraintTemplate+` se aplica a `+Namespaces+`:

[source,yaml]
----
apiVersion: constraints.gatekeeper.sh/v1beta
kind: K8sRequiredLabels
metadata:
  name: ns-must-have-gk
spec:
  match:
    kinds:
      - apiGroups: [""]
        kinds: ["Namespace"]
  parameters:
    labels: ["gatekeeper"]
----

Basándose en esto, actualice la sección `+rules+` de su `+metadata.yml+` para incluir un nuevo`+rule+` para validar `+namespaces+` durante las operaciones `+CREATE+` y `+UPDATE+`:

[source,yaml]
----
rules:
  - apiGroups: ["apps"]
    apiVersions: ["v1"]
    resources: ["deployments"]
    operations: ["CREATE", "UPDATE"]
  - apiGroups: [""]
    apiVersions: ["v1"]
    resources: ["namespaces"]
    operations: ["CREATE", "UPDATE"]
mutating: false
contextAware: false
executionMode: gatekeeper
backgroundAudit: true
annotations:
  io.artifacthub.displayName: Policy Name
  io.artifacthub.resources: Pod
  io.artifacthub.keywords: pod, cool policy, kubewarden
  io.kubewarden.policy.ociUrl: ghcr.io/yourorg/policies/policy-name
  io.kubewarden.policy.title: policy-name
  io.kubewarden.policy.version: 0.0.1-unreleased
  io.kubewarden.policy.description: Short description
  io.kubewarden.policy.author: "Author name <author-email@example.com>"
  io.kubewarden.policy.url: https://github.com/yourorg/policy-name
  io.kubewarden.policy.source: https://github.com/yourorg/policy-name
  io.kubewarden.policy.license: Apache-2.0
  io.kubewarden.policy.severity: medium
  io.kubewarden.policy.category: Resource validation
----

Ahora, su política está lista para su distribución y despliegue. Consulte la secciónxref:/tutorials/writing-policies/rego/gatekeeper/04-distribute.adoc#_pushing_the_policy[Publicación de la directiva]del tutorial para aprender a enviarla a un registro remoto.

Puede organizar el manifiesto de la política utilizando `+kwctl+`:

[source,console]
----
$ kwctl scaffold manifest --type ClusterAdmissionPolicy annotated-policy.wasm
apiVersion: policies.kubewarden.io/v1
kind: ClusterAdmissionPolicy
metadata:
  annotations:
    io.kubewarden.policy.category: Resource validation
    io.kubewarden.policy.severity: medium
  name: policy-name
spec:
  module: file:///home/jvanz/SUSE/mygatekeeperpolicy/annotated-policy.wasm
  settings: {}
  rules:
  - apiGroups:
    - apps
    apiVersions:
    - v1
    resources:
    - deployments
    operations:
    - CREATE
    - UPDATE
  - apiGroups:
    - ''
    apiVersions:
    - v1
    resources:
    - namespaces
    operations:
    - CREATE
    - UPDATE
  mutating: false
----

=== Definir la configuración de la política

Esta política tiene parámetros que Gatekeeper define en`+Constraint+`. Debe actualizar la sección `+settings+` en el manifiesto de la política {short-project-name}generada para incluir estos parámetros necesarios. En el siguiente ejemplo, además de definir la configuración, puede probar la política desde el registro OCI:

[source,yaml]
----
apiVersion: policies.kubewarden.io/v1
kind: ClusterAdmissionPolicy
metadata:
  annotations:
    io.kubewarden.policy.category: Resource validation
    io.kubewarden.policy.severity: medium
  name: policy-name
spec:
  module: registry://ghcr.io/jvanz/policies/mygatekeeperpolicy:latest
  settings:
    labels:
      - "gatekeeper"
  rules:
    - apiGroups:
        - apps
      apiVersions:
        - v1
      resources:
        - deployments
      operations:
        - CREATE
        - UPDATE
    - apiGroups:
        - ""
      apiVersions:
        - v1
      resources:
        - namespaces
      operations:
        - CREATE
        - UPDATE
  mutating: false
----
