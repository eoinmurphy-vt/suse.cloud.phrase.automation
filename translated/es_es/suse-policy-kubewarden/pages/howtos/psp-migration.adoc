Para Kubernetes ≥ v1.25.
https://kubernetes.io/docs/concepts/security/pod-security-policy/[PodSecurityPolicy]Se elimina (PSP). Ahora puede utilizar {project-name} para el control de admisión en sus clústeres Kubernetes.

{short-project-name} tiene políticas separadas para lograr el mismo objetivo que una configuración PSP monolítica. Cada definición de política de {short-project-name} funciona como una sección de configuración diferente en la especificación de un PSP. La asignación de los campos de configuración PSP a sus respectivas políticas {short-project-name} se encuentra en la siguiente<<_mapping_kubewarden_policies_to_psp_fields,tabla de asignación>>.

Con {short-project-name}, los operadores tienen un control granular de la configuración de políticas en sus clusters.

Con una instancia de {short-project-name}, puede desplegar políticas para reemplazar el objeto`+PodSecurityPolicy+`. Consideramos estas reglas en este ejemplo:

* una PSP que deshabilite la escalada de privilegios
* contenedores privilegiados
* bloqueo de pods ejecutados como root
* forzar a un grupo de usuarios determinado
* bloquear espacios de nombres de host
* permitir que un pod utilice sólo el puerto 443

La definición YAML de esta PSP es:

.Definición PSP YAML
[%collapsible]
======

[subs="+attributes",yaml]
----
apiVersion: policy/v1beta1
kind: PodSecurityPolicy
metadata:
  name: restricted
spec:
  allowPrivilegeEscalation: false
  runAsUser:
    rule: MustRunAsNonRoot
  supplementalGroups:
    rule: MustRunAs
    ranges:
      - min: 1000
        max: 65535
  privileged: false
  hostNetwork: false
  hostIPC: false
  hostPID: false
  hostPorts:
    - min: 443
      max: 443
----

======

== {short-project-name} sustitutos para PSP

Ahora crearemos políticas {short-project-name} para lograr el mismo objetivo.
Cada regla se aplica con una política independiente en {short-project-name}.
Así que, en este ejemplo, necesita una política separada para la aplicación de cada una de:

* escalada de privilegios
* configuración de usuarios y grupos
* espacios de nombres de host
* configuración privilegiada del contenedor.

=== Bloqueo de la escalada de privilegios de los contenedores

Puede desplegar una política como se muestra a continuación:

.Comando kubectl para el despliegue de políticas
[%collapsible]
======

[subs="+attributes",shell]
----
$ kubectl apply -f - <<EOF
apiVersion: policies.kubewarden.io/v1
kind: ClusterAdmissionPolicy
metadata:
  name: psp-allow-privilege-escalation
spec:
  module: ghcr.io/kubewarden/policies/allow-privilege-escalation-psp:v0.2.6
  rules:
    - apiGroups:
        - ""
      apiVersions:
        - v1
      resources:
        - pods
      operations:
        - CREATE
        - UPDATE
  mutating: false
  settings:
    default_allow_privilege_escalation: false
EOF
----

======

En ese comando, hemos especificado que `+default_allow_privilege_escalation+` sea`+false+`. Esta política restringe los pods que intentan ejecutarse con más privilegios que el contenedor padre.

.Salida de kubectl que intenta elevar el privilegio
[%collapsible]
======

[subs="+attributes",shell]
----
$ kubectl apply -f - <<EOF
apiVersion: v1
kind: Pod
metadata:
  name: nginx
spec:
  containers:
  - name: nginx
    image: nginx
    securityContext:
      allowPrivilegeEscalation: true
  - name: sidecar
    image: sidecar
EOF
Error from server: error when creating "STDIN": admission webhook "clusterwide-psp-allow-privilege-escalation.kubewarden.admission" denied the request: one of the containers has privilege escalation enabled
----

======

=== Configuración de usuarios y grupos

Ahora, para aplicar la configuración de usuarios y grupos, puede utilizar lalink:https://github.com/kubewarden/user-group-psp-policy[política].

.`+kubectl+` para utilizar user-group-psp-policy
[%collapsible]
======
[subs="+attributes",shell]
----
$ kubectl apply -f - <<EOF
apiVersion: policies.kubewarden.io/v1
kind: ClusterAdmissionPolicy
metadata:
  name: psp-user-group
spec:
  module: ghcr.io/kubewarden/policies/user-group-psp:v0.4.9
  rules:
    - apiGroups:
        - ""
      apiVersions:
        - v1
      resources:
        - pods
      operations:
        - CREATE
        - UPDATE
  mutating: true
  settings:
    run_as_user:
      rule: MustRunAsNonRoot
    supplemental_groups:
      rule: MustRunAs
      ranges:
        - min: 1000
          max: 65535
EOF
----
======

Debe configurar la política con `+mutation: true+`. Es necesario porque la política añadiráhttps://kubernetes.io/docs/concepts/security/pod-security-policy/#users-and-groups[supplementalGroups]cuando el usuario no los defina.

Por lo tanto, ahora los usuarios no pueden desplegar pods ejecutándose como root:

.Ejemplo de salida donde runAsNonRoot: false
[%collapsible]
======

[subs="+attributes",shell]
----
$ kubectl apply -f - <<EOF
apiVersion: v1
kind: Pod
metadata:
  name: nginx
spec:
  containers:
  - name: nginx
    image: nginx
    securityContext:
      runAsNonRoot: false
      runAsUser: 0
EOF
Error from server: error when creating "STDIN": admission webhook "clusterwide-psp-user-group-fb836.kubewarden.admission" denied the request: RunAsNonRoot should be set to true
----

======

.Ejemplo de salida donde runAsUser: 0
[%collapsible]
======

[subs="+attributes",shell]
----
kubectl apply -f - <<EOF
apiVersion: v1
kind: Pod
metadata:
  name: nginx
spec:
  containers:
  - name: nginx
    image: nginx
    securityContext:
      runAsNonRoot: true
      runAsUser: 0
EOF
Error from server: error when creating "STDIN": admission webhook "clusterwide-psp-user-group-fb836.kubewarden.admission" denied the request: Invalid user ID: cannot run container with root ID (0)
----

======

Este ejemplo muestra la adición de un https://kubernetes.io/docs/concepts/security/pod-security-policy/#users-and-groups[grupo suplementario], a pesar de no haber sido definido por nosotros.

=== Configuración de contenedores privilegiados

Es necesario sustituir la antigua configuración de PSP que bloquea los contenedores privilegiados. Es necesario desplegar la https://github.com/kubewarden/pod-privileged-policy[pod-privileged policy]. Esta política no necesita ninguna configuración. Una vez en marcha, bloqueará los pods privilegiados.

.Aplicación de la política pod-privileged
[%collapsible]
======

[subs="+attributes",shell]
----
$ kubectl apply -f - <<EOF
apiVersion: policies.kubewarden.io/v1
kind: ClusterAdmissionPolicy
metadata:
  name: psp-privileged
spec:
  module: ghcr.io/kubewarden/policies/pod-privileged:v0.2.7
  rules:
    - apiGroups:
        - ""
      apiVersions:
        - v1
      resources:
        - pods
      operations:
        - CREATE
        - UPDATE
  mutating: false
  settings: null
EOF
----

======

Para probar la política, podemos intentar ejecutar un pod con la configuración privilegiada activada:

.Ejecución de pods con configuración privilegiada habilitada
[%collapsible]
======

[subs="+attributes",shell]
----
$ kubectl apply -f - <<EOF
apiVersion: v1
kind: Pod
metadata:
  name: nginx
spec:
  containers:
  - name: nginx
    image: nginx
    imagePullPolicy: IfNotPresent
    securityContext:
      privileged: true
  - name: sleeping-sidecar
    image: alpine
    command: ["sleep", "1h"]
EOF
Error from server: error when creating "STDIN": admission webhook "clusterwide-psp-privileged.kubewarden.admission" denied the request: Privileged container is not allowed
----

=== Configuración del espacio de nombres del host

Para finalizar el ejercicio de migración a PSP, debe desactivar el uso compartido del espacio de nombres del host. Para ello, utilizaremos la página https://github.com/kubewarden/host-namespaces-psp-policy[`+host-namespace-psp+`policy]. Permite al administrador del clúster bloquear individualmente los espacios de nombres IPC, PID y de red. También establece los puertos en los que los pods pueden estar abiertos, en la IP del host.

.Desactivación del uso compartido del espacio de nombres y configuración de puertos
[%collapsible]
======

[subs="+attributes",shell]
----
$ kubectl apply -f - <<EOF
apiVersion: policies.kubewarden.io/v1
kind: ClusterAdmissionPolicy
metadata:
  name: psp-hostnamespaces
spec:
  module: ghcr.io/kubewarden/policies/host-namespaces-psp:v0.1.6
  rules:
    - apiGroups:
        - ""
      apiVersions:
        - v1
      resources:
        - pods
      operations:
        - CREATE
        - UPDATE
  mutating: false
  settings:
    allow_host_ipc: false
    allow_host_pid: false
    allow_host_ports:
      - min: 443
        max: 443
    allow_host_network: false
EOF
----

======

== Script de migración a PSP

El equipo de {short-project-name} ha desarrollado un script para la migración de PSP. Utiliza la herramienta de migración de https://github.com/appvia/psp-migration[AppVia]. La herramienta AppVia lee una configuración PSP YAML. A continuación, genera las políticas correspondientes. Lo hace para {short-project-name} y otros motores de políticas.

[CAUTION]
====

La herramienta de migración de AppVia está fuera del control de los mantenedores de Kuberwarden.
Esto significa que es posible que genere políticas desactualizadas {short-project-name}. Utilizar con precaución. Necesitamos un pull request para AppVia para el que se está trabajando.
Póngase en contacto con nosotros si necesita más información.
====


El script está disponible en el repositorio de {short-project-name}https://github.com/kubewarden/utils/blob/main/scripts/psp-to-kubewarden[utilidades de]. Descarga la herramienta de migración AppVia en el directorio de trabajo a utilizar. Procesa los PSP definidos en el contexto por defecto `+kubectl+`. A continuación, imprime las definiciones de las políticas de Kuberwarden en la salida estándar. Los usuarios pueden redirigir el contenido a un archivo o a `+kubectl+` directamente.

[NOTE]
====

Este script sólo funciona en máquinas Linux x86_64.
====


Veamos un ejemplo. En una agrupación con el PSP:

* bloquear el acceso a los espacios de nombres de host
* bloqueo de contenedores privilegiados
* no permitir la escalada de privilegios
* capacidad de descarga de contenedores
* listado de los tipos de volumen permitidos
* definición de los usuarios y grupos autorizados
* controlar el grupo suplementario aplicado a los volúmenes
* obligar a los contenedores a ejecutarse en un sistema de archivos raíz de sólo lectura

Se podría utilizar el siguiente YAML.

.La configuración PSP
[%collapsible]
======

[subs="+attributes",yaml]
----
apiVersion: policy/v1beta1
kind: PodSecurityPolicy
metadata:
  name: restricted
spec:
  hostNetwork: false
  hostIPC: false
  hostPID: false
  hostPorts:
    - min: 80
      max: 8080
  privileged: false
  # Required to prevent escalations to root.
  allowPrivilegeEscalation: false
  requiredDropCapabilities:
    - ALL
  # Allow core volume types.
  volumes:
    - "configMap"
    - "emptyDir"
    - "projected"
    - "secret"
    - "downwardAPI"
    # Assume that ephemeral CSI drivers & persistentVolumes set up by the cluster admin are safe to use.
    - "csi"
    - "persistentVolumeClaim"
    - "ephemeral"
  runAsUser:
    # Require the container to run without root privileges.
    rule: "MustRunAsNonRoot"
  seLinux:
    # This policy assumes the nodes are using AppArmor rather than SELinux.
    rule: "RunAsAny"
  supplementalGroups:
    rule: "MustRunAs"
    ranges:
      # Forbid adding the root group.
      - min: 1
        max: 65535
  fsGroup:
    rule: "MustRunAs"
    ranges:
      # Forbid adding the root group.
      - min: 1
        max: 65535
  readOnlyRootFilesystem: true
----
