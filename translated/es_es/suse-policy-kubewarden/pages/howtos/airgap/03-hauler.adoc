= Instalación del entrehierro con Hauler
:revdate: 2026-03-18
:page-revdate: {revdate}
:description: Utilice Hauler para instalar {project-name} en entornos con entradas de aire.
:keywords: ["{project-name}", kubewarden, kubernetes, instalación del entrehierro, transportista]
:doc-persona: kubewarden-operator, kubewarden-integrator
:doc-type: cómo
:doc-topic: manual del operador, cámara de aire, instalación, transportista

Esta guía le muestra cómo instalar {project-name} en entornos con cámara de aire utilizandolink:https://docs.hauler.dev/docs/intro[Hauler]. Hauler es una herramienta que ayuda a los usuarios a ejecutar cargas de trabajo en entornos con separación por aire. Traslada los recursos necesarios para las aplicaciones a esos entornos protegidos por el aire.

{short-project-name} proporciona un archivo de manifiesto con todos los recursos necesarios para ejecutar {short-project-name} en un entorno privado. Esta documentación describe cómo puede utilizarlo. Consulte la documentación del transportista para obtener más información al respecto.

El flujo de trabajo básico utilizando el manifiesto Hauler es:

. Consígalo en la página de publicación {short-project-name} 
. Cargue todos los recursos necesarios para ejecutar {short-project-name} en un almacén local
. Exportarlos a un archivo
. Mover el archivo al entorno privado
. Cargar los recursos en Hauler ejecutándose dentro del entorno privado.
. Copie todo en un registro, que se utilizará en el proceso de instalación, en su entorno de air-gapped

== Descargue el manifiesto de Hauler desde la página de publicación de cartas de Helm

Descargue `+hauler_manifest.yml+` desde la {short-project-name}link:https://github.com/kubewarden/helm-charts/releases/[página de versiones].

Sincronice los recursos definidos en el manifiesto con su almacén Hauler:

[source,shell]
----
hauler store sync --filename hauler_manifest.yaml
----

El transportista descarga todos los recursos del manifiesto al almacén local. Este proceso dura unos minutos. Una vez completado, puede ver los recursos sincronizados con el comando `+hauler store info+`.

== Generar el archivo tar con todos los recursos de {short-project-name} 

Ejecute el comando Hauler para exportar a un archivo todos los recursos cargados previamente en el almacén:

[source,shell]
----
hauler store save --filename kubewarden-resources.tar.zst
----

[NOTE]
====

== Transfiera el tarball a su entorno aislado

Ahora que ya tienes todos los recursos de {short-project-name} en`+kubewarden-resources.tar.zst+`, cópialos en tu entorno air-gapped y cárgalos allí en el almacén Hauler:

[source,shell]
----
hauler store load --filename kubewarden-resources.tar.zst
# Check if the resources are loaded
hauler store info
----

Ahora todos los recursos necesarios para instalar {short-project-name} están en el almacén Hauler de su entorno aislado.

== Rellenar el registro privado

Para utilizar los recursos de su tienda Hauler, es necesario que estén disponibles en un registro interno. Puede utilizar los comandos Hauler para copiarlos en su registro privado.

[source,shell]
----
hauler store copy registry://localhost:5000
----

También puede ejecutar Hauler para iniciar un registro con todos los recursos de la tienda. Este registro es inseguro, y es necesario adaptar la configuración del clúster:

[source,shell]
----
# Find IP address of your host
# hostname -I

# Update registries.yaml for k3s based cluster to allow insecure access
# mirrors:
#   "<HOST_IP>:5000":
#     endpoint:
#       - "http://<HOST_IP>:5000"
# configs:
#   "<HOST_IP>:5000":
#     tls:
#       insecure_skip_verify: true

# Configure policy-server to allow pulling policies from insecure sources
# helm install .. kubewarden-defaults .. --set policyServer.insecureSources[0]=<HOST_IP:5000>

hauler store serve registry
----

Esto inicia un registro en la dirección `+localhost:5000+`. Desde este punto, puede utilizar otros comandos como Skopeo para copiar todas las imágenes de contenedor, módulos de políticas y gráficos Helm utilizados por {short-project-name} en su registro privado.

== Instale {short-project-name}

Ahora que su registro privado tiene todo lo necesario, puede instalar{short-project-name}. La diferencia con una instalación estándar de {short-project-name}es que es necesario cambiar el registro en las imágenes y políticas del contenedor para que sea el registro privado. Además, es necesario instalar los gráficos Helm a partir de artefactos OCI.

Instale la pila {short-project-name}:

[source,shell]
----
helm install --wait -n kubewarden kubewarden-crds --create-namespace \
    oci://<REGISTRY.YOURDOMAIN.COM:PORT>/hauler/kubewarden-crds
helm install --wait -n kubewarden kubewarden-controller \
    --set "global.cattle.systemDefaultRegistry=<REGISTRY.YOURDOMAIN.COM:PORT>" \
    oci://<REGISTRY.YOURDOMAIN.COM:PORT>/hauler/kubewarden-controller
----

[CAUTION]
====

Para utilizar el subgráfico PolicyReporter disponible en el gráfico `+kubewarden-controller+`es necesario definir otros valores específicos para el subgráfico en un entorno con air-gapped. Vea un ejemplo a continuación:

[source,shell]
----
helm install --wait -n kubewarden kubewarden-controller oci://<REGISTRY.YOURDOMAIN.COM:PORT>/hauler/kubewarden-controller \
    --set global.cattle.systemDefaultRegistry=<REGISTRY.YOURDOMAIN.COM:PORT> \
    --set auditScanner.policyReporter=true \
    --set policy-reporter.image.registry=<REGISTRY.YOURDOMAIN.COM:PORT> \
    --set policy-reporter.image.repository=kyverno/policy-reporter \
    --set policy-reporter.ui.image.registry=<REGISTRY.YOURDOMAIN.COM:PORT> \
    --set policy-reporter.ui.image.repository=kyverno/policy-reporter-ui
----

Es necesario definir `+auditScanner.policyReporter+` y otros cuatro valores para habilitar el subdiálogo y configurar el registro y el repositorio con la ubicación del almacén de imágenes de Policy Reporter. Para obtener más información sobre los valores de los subgráficos del informe de políticas, consulte el enlace:https://kyverno.github.io/policy-reporter-docs/getting-started/helm.html[Documentación del informe de políticas].

====

[source,shell]
----
helm install --wait -n kubewarden \
  kubewarden-defaults oci://<REGISTRY.YOURDOMAIN.COM:PORT>/hauler/kubewarden-defaults \
  --set global.cattle.systemDefaultRegistry=<REGISTRY.YOURDOMAIN.COM:PORT>
----

Por último, configure cada servidor de directivas para que obtenga las directivas de su registro privado. Consulte la sección xref:/howtos/policy-servers/02-private-registry.adoc[sobre el uso de registros privados] de la documentación.

Ahora cree políticas {short-project-name} en su cluster. Las políticas deben estar disponibles en su registro privado.

[source,yaml]
----
kubectl apply -f - <<EOF
apiVersion: policies.kubewarden.io/v1
kind: ClusterAdmissionPolicy
metadata:
  name: privileged-pods
spec:
  module: registry://<REGISTRY.YOURDOMAIN.COM:PORT>/kubewarden/policies/pod-privileged:v0.2.2
  rules:
  - apiGroups: [""]
    apiVersions: ["v1"]
    resources: ["pods"]
    operations:
    - CREATE
  mutating: false
EOF
----

[CAUTION]
====

`+PolicyServer+` recursos deben utilizar la imagen disponible en su registro privado.
Por ejemplo:

[source,yaml]
----
apiVersion: policies.kubewarden.io/v1
kind: PolicyServer
metadata:
  name: reserved-instance-for-tenant-a
spec:
  image: <REGISTRY.YOURDOMAIN.COM:PORT>/kubewarden/policy-server:v1.3.0
  replicas:
----
