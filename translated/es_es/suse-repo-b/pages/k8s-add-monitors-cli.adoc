= Añadir un monitor de umbral a los componentes mediante la CLI
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: Observabilidad de SUSE

== Visión general

SUSE Observability proporciona xref:/use/alerting/k8s-monitors.adoc[monitores listos para usar], que supervisan los problemas habituales que pueden producirse en un clúster de Kubernetes. También es posible configurar monitores personalizados para las métricas recopiladas por SUSE Observability o las métricas de aplicación ingestadas xref:/use/metrics/k8s-prometheus-remote-write.adoc[desde Prometheus].

== Escribir el esquema del monitor

Cree un nuevo archivo YAML llamado `+monitor.yaml+` y añádale esta plantilla YAML para crear su propio monitor. Ábrelo en tu editor de código favorito para modificarlo a lo largo de esta guía. Al final se utilizará la CLI de SUSE Observability para crear y actualizar el monitor en SUSE Observability.

----
nodes:
- _type: Monitor
  arguments:
    metric:
      query:
      unit:
      aliasTemplate:
    comparator:
    threshold:
    failureState:
    urnTemplate:
    titleTemplate:
  description:
  function: {{ get "urn:stackpack:common:monitor-function:threshold"  }}
  identifier: urn:custom:monitor:...
  intervalSeconds: 30
  name:
  remediationHint:
  status:
  tags: {}
----

Los campos de esta plantilla son:

* `+_type+`: SUSE Observability necesita saber que se trata de un monitor, por lo que el valor siempre debe ser `+Monitor+`
* `+query+`: Una consulta PromQL. Utilice el xref:/use/metrics/k8sTs-explore-metrics.adoc[explorador de métricas] de su instancia de SUSE Observability, http://your-instance/#/metrics, y utilícelo para construir la consulta de la métrica de interés.
* `+unit+`: La unidad de los valores de la serie temporal devuelta por la consulta o consultas, utilizada para representar el eje Y del gráfico. Consulte la referencia de xref:/develop/reference/k8sTs-chart-units.adoc[unidades compatibles] para ver todas las unidades.
* `+aliasTemplate+`: Un alias para las series temporales en el gráfico métrico. Se trata de una plantilla que puede sustituir las etiquetas de las series temporales utilizando el marcador de posición `+${my_label}+`.
* `+comparator+`: Elija una de las opciones LTE/LT/GTE/GT para comparar el umbral con la métrica. Las series temporales para las que `+<metric> <comparator> <threshold>+` sea cierta producirán el estado de fallo.
* `+threshold+`: Un umbral numérico con el que comparar.
* `+failureState+`: O "CRÍTICO" o "DESVIADO". "CRÍTICO" se mostrará como leído en SUSE Observability y "DESVIADO" como naranja, para denotar una gravedad diferente.
* `+urnTemplate+`: Una plantilla para construir la urna del componente <<_bind_the_results_of_the_monitor_to_the_correct_components,al que se vinculará>> el resultado del monitor.
* `+titleTemplate+`: Título para el resultado de un monitor. Dado que varios resultados de monitor pueden vincularse al mismo componente, es posible sustituir las etiquetas de las series temporales utilizando el marcador de posición `+${my_label}+`.
* `+description+`: Descripción del monitor.
* `+function+`: Una referencia a la función del monitor que ejecutará el monitor. Actualmente sólo se admite `+{{ get "urn:stackpack:kubernetes-v2:shared:monitor-function:threshold"  }}+`.
* `+identifier+`: Un identificador de la forma `+urn:custom:monitor:....+` que identifica unívocamente al monitor al actualizar su configuración.
* `+intervalSeconds+`: El intervalo en el que se ejecuta el monitor. Para una métrica regular en tiempo real se aconsejan 30 segundos. Para consultas métricas analíticas de larga duración se recomienda un intervalo mayor.
* `+name+`: El nombre del monitor
* `+remediationHint+`: Descripción de lo que puede hacer el usuario cuando falla el monitor. El formato es markdown, con el uso opcional de variables handlebars para personalizar la sugerencia basada en series temporales u otros datos<<_write_the_remediation_hint,(más explicaciones a continuación>>).
* `+status+`: O `+"DISABLED"+` o `+"ENABLED"+`. Determina si el monitor se ejecutará o no.
* `+tags+`: Añada etiquetas al monitor para ayudar a organizarlos en la vista general de monitores de su instancia de SUSE Observability, http://your-SUSE Observability-instance/#/monitors

Por ejemplo, podría ser el inicio de un monitor que supervise las réplicas disponibles de una implantación:

----
nodes:
- _type: Monitor
  arguments:
    metric:
      query: "kubernetes_state_deployment_replicas_available"
      unit: "short"
      aliasTemplate: "Deployment replicas"
    comparator: "LTE"
    threshold: 0.0
    failureState: "DEVIATING"
    urnTemplate:
  description: "Monitor whether a deployment has replicas.
  function: {{ get "urn:stackpack:kubernetes-v2:shared:monitor-function:threshold"  }}
  identifier: urn:custom:monitor:deployment-has-replicas
  intervalSeconds: 30
  name: Deployment has replicas
  remediationHint:
  status: "ENABLED"
  tags:
  - "deployments"
----

Las direcciones `+urnTemplate+` y `+remediationHint+` se rellenarán en los próximos pasos.

== Vincular los resultados del monitor a los componentes correctos

Los resultados de un monitor deben estar vinculados a componentes en SUSE Observability, para ser visibles y utilizables. El resultado de un monitor se vincula a un componente mediante el componente `+identifiers+`. Cada componente de SUSE Observability tiene uno o más identificadores que lo identifican de forma única. Para vincular un resultado de un monitor a un componente, es necesario proporcionar la dirección `+urnTemplate+`. `+urnTemplate+` sustituye las etiquetas de la serie temporal del resultado del monitor en la plantilla, produciendo un identificador que coincide con un componente. Esto se ilustra mejor con el ejemplo:

La métrica utilizada en este ejemplo es `+kubernetes_state_deployment_replicas_available+`. Ejecute la métrica en el explorador de métricas para observar qué etiquetas están disponibles en la serie temporal:

image::k8s/available-replicas-metric-inspector.png[Las réplicas disponibles en el explorador de métricas]

En la tabla anterior se muestra que la métrica tiene etiquetas como `+cluster_name+`, `+namespace+` y `+deployment+`.

Dado que la métrica se observa en los despliegues, lo más lógico es vincular los resultados del monitor a los componentes del despliegue. Para ello, es necesario entender cómo se construyen los identificadores de las implantaciones:

. En la interfaz de usuario, vaya a la vista `+deployments+` y seleccione una única implantación.
. Abra la vista `+Topology+` y haga clic en el componente de despliegue.
. Al desplegar `+Properties+` en el panel derecho de la pantalla, los identificadores se mostrarán al pasar el ratón por encima, como se muestra a continuación:

image::k8s/component-identifier.png[Encontrar un identificador de componente]

El identificador aparece como `+urn:kubernetes:/preprod-dev.preprod.stackstate.io:calico-system:deployment/calico-typha+`. Esto muestra que el identificador se construye basándose en el nombre del clúster, el espacio de nombres y el nombre del despliegue. Sabiendo esto, ahora es posible construir el `+urnTemplate+`:

----
  ...
  urnTemplate: "urn:kubernetes:/${cluster_name}:${namespace}:deployment/${deployment}"
  ...
----

A continuación se explica <<_verifying_the_results_of_a_monitor,cómo comprobar>> si la dirección `+urnTemplate+` es correcta.

== Escribir la pista de reparación

La sugerencia de solución está ahí para ayudar a los usuarios a encontrar la causa de un problema cuando se dispara un monitor. La sugerencia de corrección está escrita en https://en.wikipedia.org/wiki/Markdown[markdown]. También es posible utilizar las etiquetas que aparecen en la serie temporal del resultado del monitor utilizando una plantilla de manillar, como en el siguiente ejemplo:

----
  ...
  remediationHint: |-
    To remedy this issue with the deployment {{ labels.deployment }}, consider taking the following steps:

    1. Look at the logs of the pods created by the deployment
  ...
----

== Crear o actualizar el monitor en SUSE Observability

Una vez completado `+monitor.yaml+`, utilice xref:/setup/cli/cli-sts.adoc[la CLI de SUSE Observability] para crear o actualizar el monitor:

[,bash]
----
sts monitor apply -f monitor.yaml
----

Compruebe si el monitor produce los resultados esperados, siguiendo los pasos que <<_verifying_the_results_of_a_monitor,se indican a continuación>>.

[CAUTION]
====
El identificador se utiliza como clave única de un monitor. Al cambiar el identificador se creará un nuevo monitor en lugar de actualizar el existente.
====


El comando `+sts monitor+` tiene más opciones, por ejemplo puede listar todos los monitores:

[,bash]
----
sts monitor list
----

Para eliminar un monitor utilice

[,bash]
----
sts monitor delete --id <id>
----

Para editar un monitor, edite el original del monitor que se aplicó y vuelva a aplicarlo. O existe un comando `+sts monitor edit+` para editar directamente el monitor configurado en la instancia de SUSE Observability:

[,bash]
----
sts monitor edit --id <id>
----

El `+<id>+` en este comando no es el identificador sino el número en la columna `+Id+` de la salida `+sts monitor list+`.

== Activar o desactivar el monitor

Un monitor puede activarse o desactivarse. Activado significa que el monitor producirá resultados, desactivado significa que suprimirá toda salida. Utilice los siguientes comandos para activar/desactivar:

[,bash]
----
sts monitor enable/disable --id <id>
----

== Verificación de los resultados de un monitor

Es una buena práctica, después de hacer un monitor, validar si produce el resultado esperado. Se pueden tomar las siguientes medidas:

=== Verificar la ejecución del monitor

Vaya a la página de resumen de monitores (http://your-SUSE Observability-instance/#/monitors) y busque su monitor.

. Compruebe que la columna `+Status+` está en estado `+Enabled+`. Si el monitor está en estado `+Disabled+`, <<_enable_or_disable_the_monitor,actívelo>>. Si el estado es `+Error+`, siga los siguientes pasos <<_the_monitor_is_showing_an_error_in_the_monitor_status_overview,para depurar>>.
. Verifique que ve la cantidad esperada de estados en la columna `+Clear+`/`+Deviating+`/`+Critical+`. Si el número de estados es significativamente menor o mayor que la cantidad de componentes que se pretende monitorear, la consulta PromQL podría estar dando demasiados resultados.

=== Verificar la fijación del monitor

Observe si el monitor produce un resultado en uno de los componentes que debe controlar. Si el monitor no aparece, sigue <<_the_result_of_the_monitor_isnt_showing_on_a_component,estos pasos>> para remediarlo.

== Problemas comunes

=== El resultado del monitor no se muestra en un componente

En primer lugar, compruebe si el monitor <<_verify_the_execution_of_the_monitor,produce>> realmente <<_verify_the_execution_of_the_monitor,resultados>>. Si este es el caso, pero los resultados del monitor no aparecen en los componentes, puede que haya un problema con la unión. Primero utilice el siguiente comando para verificarlo:

[,bash]
----
sts monitor status --id <id>
----

Si la salida tiene `+Monitor health states with identifier which has no matching topology element (<nr>): ....+`, esto muestra que el `+urnTemplate+` puede no generar un identificador que coincida con la topología. Para remediarlo, <<_bind_the_results_of_the_monitor_to_the_correct_components,revise su urnTemplate>>.

=== El monitor muestra un error en el resumen de estado del monitor

Obtener el estado del monitor a través de la CLI

[,bash]
----
sts monitor status --id <id>
----

La sección `+Monitor Stream errors:+` mostrará los errores que se producen en el monitor y ofrecerá más ayuda.
