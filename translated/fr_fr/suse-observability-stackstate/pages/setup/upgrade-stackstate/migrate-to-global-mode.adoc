= Migrer vers le mode de configuration globale
:description: {stackstate-product-name} Auto-hébergé
:page-toclevels: 3

== Vue d'ensemble

Ce guide explique comment passer de l'ancienne installation en deux étapes utilisant la carte `+suse-observability-values+` à l'installation simplifiée à carte unique utilisant la configuration `+global.suseObservability+`.

[NOTE]
====
La méthode de configuration `+global.suseObservability+` est disponible à partir de la version `+2.8.0+` et constitue désormais l'approche d'installation recommandée. L'ancienne méthode du diagramme `+suse-observability-values+` est obsolète mais continuera à fonctionner pour les installations existantes.
====

=== Installation ancienne (obsolète)

L'ancienne méthode d'installation nécessitait deux étapes :

. Générer des fichiers de valeurs à l'aide de la carte `+suse-observability-values+` 
. Installer la carte `+suse-observability+` avec les valeurs générées

[,text]
----
# Step 1: Generate values
helm template suse-observability-values suse-observability/suse-observability-values \
  --set license="YOUR-LICENSE-KEY" \
  --set baseUrl="https://observability.example.com" \
  --set sizing.profile="150-ha" \
  --set adminPassword="your-password" \
  --set pullSecret.username="registry-user" \
  --set pullSecret.password="registry-pass" \
  > generated-values.yaml

# Step 2: Install with generated values
helm install suse-observability suse-observability/suse-observability \
  -f generated-values.yaml \
  -n suse-observability
----

=== Nouvelle installation (recommandé)

La nouvelle méthode utilise un seul fichier de valeurs avec la configuration `+global.suseObservability+`:

[,text]
----
helm install suse-observability suse-observability/suse-observability \
  -f values.yaml \
  -n suse-observability
----

== Convertir avant l'installation

Si vous avez préparé les paramètres de l'ancienne carte `+suse-observability-values+` mais que vous ne l'avez pas encore installée, vous pouvez passer directement à la nouvelle méthode sans créer de fichiers de valeurs intermédiaires.

=== Conversion directe des paramètres

Convertissez votre commande `+helm template+` en commande `+helm upgrade --install+` en faisant correspondre les paramètres :

[cols="1,1"]
|===
| Ancien paramètre | Nouveau paramètre

| `+--set license="..."+`
| `+--set global.suseObservability.license="..."+`

| `+--set baseUrl="..."+`
| `+--set global.suseObservability.baseUrl="..."+`

| `+--set sizing.profile="..."+`
| `+--set global.suseObservability.sizing.profile="..."+`

| `+--set receiverApiKey="..."+`
| `+--set global.suseObservability.receiverApiKey="..."+`

| `+--set adminPassword="..."+`
| `+--set global.suseObservability.adminPassword="..."+`

| `+--set imageRegistry="..."+`
| `+--set global.imageRegistry="..."+`

| `+--set pullSecret.username="..."+`
| `+--set global.suseObservability.pullSecret.username="..."+`

| `+--set pullSecret.password="..."+`
| `+--set global.suseObservability.pullSecret.password="..."+`
|===

=== Exemple : Avant et après

[NOTE]
====
Avant d'exécuter les commandes helm, assurez-vous que vous disposez de la dernière version de la carte :

[,bash]
----
helm repo update
----
====

.Commande héritée (ne pas utiliser)
[,bash]
----
export VALUES_DIR=.
helm template \
  --set license="YOUR-LICENSE-KEY" \
  --set baseUrl="https://observability.example.com" \
  --set sizing.profile="10-nonha" \
  --set receiverApiKey="YOUR-API-KEY" \
  suse-observability-values \
  suse-observability/suse-observability-values --output-dir $VALUES_DIR

# Then: helm install with generated files...
----

.Nouvelle commande (recommandé)
[,bash]
----
helm upgrade --install \
  --namespace suse-observability \
  --create-namespace \
  --set global.suseObservability.license="YOUR-LICENSE-KEY" \
  --set global.suseObservability.baseUrl="https://observability.example.com" \
  --set global.suseObservability.sizing.profile="10-nonha" \
  --set global.suseObservability.receiverApiKey="YOUR-API-KEY" \
  --set global.suseObservability.adminPassword='your-password' \
  suse-observability suse-observability/suse-observability
----

[NOTE]
====
* Le site `+adminPassword+` doit être un hachage bcrypt dans la nouvelle méthode. Voir <<bcrypt-hash>> pour savoir comment en générer un.
* Pour les environnements à air comprimé ou les registres personnalisés, voir xref:/k8s-suse-rancher-prime-air-gapped.adoc[Installer {stackstate-product-name} en mode à air comprimé] pour une configuration supplémentaire, y compris les paramètres `+imageRegistry+` et `+pullSecret+`.
====

=== Utilisation d'un fichier de valeurs

Si vous préférez utiliser un fichier de valeurs, créez `+values.yaml+` avec la configuration indiquée dans <<_step_3_create_new_values_file>>, assurez-vous que vous disposez du dernier graphique avec `+helm repo update+`, puis exécutez :

[,bash]
----
helm upgrade --install \
  --namespace suse-observability \
  --create-namespace \
  --values values.yaml \
  suse-observability suse-observability/suse-observability
----

== Avant de migrer

Lorsque vous effectuez une migration à partir de l'ancienne méthode d'installation, tenez compte des points suivants :

[CAUTION]
====
Ce guide de migration s'adresse aux utilisateurs qui utilisent actuellement la méthode graphique `+suse-observability-values+`. Si vous mettez à jour depuis StackState 6.0, suivez plutôt le xref:/setup/upgrade-stackstate/migrate-from-6.adoc[guide de migration depuis StackState].
====

[NOTE]
====
* La migration est effectuée à l'aide d'une commande standard de mise à niveau de Helm
* Les données et la configuration existantes sont conservées
* La migration peut être effectuée pendant une fenêtre de maintenance
* Il est possible de revenir en arrière en utilisant les procédures de retour en arrière standard de Helm.
====

== Les étapes de la migration

=== Étape 1 : Sauvegarde de la configuration actuelle

Avant de commencer la migration, sauvegardez vos valeurs Helm existantes et toutes les personnalisations :

[,text]
----
# Save current Helm values
helm get values suse-observability -n suse-observability > backup-values.yaml

# Save your original suse-observability-values input (if available)
# This is typically your values file used with helm template

# Backup important resources
kubectl get secrets -n suse-observability -o yaml > backup-secrets.yaml
----

=== Étape 2 : Identifier votre configuration actuelle

Examinez votre configuration existante pour identifier les paramètres suivants :

* *Profil de dimensionnement*: Vérifiez `+sizing.profile+` dans votre entrée suse-observability-values
* *Clé de licence*: Votre licence {stackstate-product-name} 
* *URL de base*: L'URL où {stackstate-product-name} est accessible
* *Mot de passe administrateur*: Le mot de passe administrateur (vous pouvez également utiliser le hachage bcrypt du mot de passe, la valeur pour cela est `+adminPasswordBcrypt+`)
* *Obtenir des informations d'identification secrètes*: Nom d'utilisateur et mot de passe du registre (si utilisé)
* *Paramètres d'affinité personnalisés*: Toute personnalisation de l'affinité des nœuds ou de l'anti-affinité des pods

Vous pouvez en extraire certains à partir de votre installation actuelle :

[,text]
----
# Check current values
helm get values suse-observability -n suse-observability

# The sizing profile is visible in resource configurations
# Look for patterns like replica counts, resource limits, etc.
----

=== Comprendre les valeurs générées automatiquement et les valeurs personnalisées

Lorsque vous exécutez `+helm get values+`, vous pouvez voir des centaines de lignes de configuration. Toutefois, la plupart d'entre elles sont *générées automatiquement par* le profil de dimensionnement et *ne* doivent *pas* être conservées manuellement dans votre fichier de valeurs de migration.

==== Ce que le profil de dimensionnement fournit automatiquement

Lorsque vous réglez `+global.suseObservability.sizing.profile+`, la carte se configure automatiquement :

* Demandes et limites de ressources pour tous les composants (Elasticsearch, Kafka, ClickHouse, serveur, récepteur, etc.)
* Nombre de répliques pour les services avec état
* Tailles de stockage pour les volumes persistants
* Mode de déploiement de HBase (Mono pour non-HA, Distribué pour HA)
* Configuration fractionnée du serveur et du récepteur
* Paramètres HA de Victoria Metrics
* Variables d'environnement (`+extraEnv+`) y compris les paramètres `+CONFIG_FORCE_*+` pour l'optimisation des performances

==== Ce qu'il faut préserver

Ne conservez que les valeurs *réellement personnalisées*, c'est-à-dire les configurations que vous avez ajoutées ou modifiées au-delà de ce que prévoit le profil de dimensionnement :

[cols="1,1"]
|===
| Préserver (sur mesure) | Ne pas conserver (généré automatiquement)

| Paramètres de base (licence, baseUrl, adminPassword, receiverApiKey)
| Limites et demandes de ressources

| Configuration de l'entrée (hôtes, TLS, annotations)
| Réplique compte

| Variables d'environnement personnalisées que vous avez ajoutées
| Tailles de stockage

| Paramètres d'authentification (LDAP, OIDC)
| `+CONFIG_FORCE_*+` variables d'environnement

| Sur mesure `+allowedOrigins+`
| Facteurs de réplication Kafka/Zookeeper

| OpenTelemetry collector custom ingress
| Mode de déploiement de HBase
|===

==== Exemple : Fichier des valeurs minimales de migration

Votre fichier de valeurs de migration devrait être *beaucoup plus petit* que la sortie de `+helm get values+`:

[,yaml]
----
global:
  suseObservability:
    license: "YOUR-LICENSE"
    baseUrl: "https://your-url.example.com"
    sizing:
      profile: "10-nonha"  # This handles all resource configuration!
    adminPassword: "your-password"
    receiverApiKey: "your-api-key"

# Only include truly custom configurations below
ingress:
  enabled: true
  ingressClassName: your-ingress-class
  annotations:
    # your custom annotations
  hosts:
  - host: your-url.example.com
  tls:
  - hosts:
    - your-url.example.com
    secretName: your-tls-secret
----

[TIP]
====
Si vous ne savez pas si une valeur est personnalisée ou générée automatiquement, comparez-la aux valeurs par défaut du profil de dimensionnement. Les valeurs telles que les variables d'environnement `+elasticsearch.resources+`, `+kafka.resources+`, `+stackstate.components.server.resources+`, et `+CONFIG_FORCE_*+` sont toutes générées par le profil et peuvent être omises sans risque.
====

=== Étape 3 : Créer un nouveau fichier de valeurs

Créez un nouveau site `+values.yaml+` avec la structure `+global.suseObservability+`:

[,yaml]
----
global:
  suseObservability:
    # Required: Your license key
    license: "YOUR-LICENSE-KEY"

    # Required: Base URL for {stackstate-product-name}
    baseUrl: "https://observability.example.com"

    # Required: Sizing profile
    # Available: trial, 10-nonha, 20-nonha, 50-nonha, 100-nonha,
    #            150-ha, 250-ha, 500-ha, 4000-ha
    sizing:
      profile: "150-ha"

    adminPassword: "your-password"
    # Instead of adminPassword you can provide a bcrypt hashed password
    # Generate with: htpasswd -bnBC 10 "" "your-password" | tr -d ':\n'
    # adminPasswordBcrypt: "$2a$10$..."

    # Optional: Receiver API key (auto-generated if not provided)
    receiverApiKey: "your-receiver-api-key"

    # Optional: Affinity configuration
    # These settings apply to all infrastructure components (Elasticsearch,
    # Kafka, ClickHouse, etc.) automatically - no need to configure per-component
    affinity:
      # Node affinity for all components
      nodeAffinity: null

      # Pod anti-affinity for HA profiles (infrastructure components)
      podAntiAffinity:
        requiredDuringSchedulingIgnoredDuringExecution: true
        topologyKey: "kubernetes.io/hostname"

  # For air-gapped or custom registry configurations, see:
  # xref:/k8s-suse-rancher-prime-air-gapped.adoc[Air-Gapped Installation Guide]
----

=== Étape 4 : Référence du mappage de la configuration

Utilisez ce tableau pour faire correspondre votre ancienne configuration au nouveau format :

[cols="1,1"]
|===
| Ancienne configuration (suse-observability-values) | Nouvelle configuration (global.suseObservability)

| `+license+`
| `+global.suseObservability.license+`

| `+baseUrl+`
| `+global.suseObservability.baseUrl+`

| `+sizing.profile+`
| `+global.suseObservability.sizing.profile+`

| `+adminPassword+`
| `+global.suseObservability.adminPassword+`

| `+receiverApiKey+`
| `+global.suseObservability.receiverApiKey+`

| `+affinity.nodeAffinity+`
| `+global.suseObservability.affinity.nodeAffinity+`

| `+affinity.podAntiAffinity.requiredDuringSchedulingIgnoredDuringExecution+`
| `+global.suseObservability.affinity.podAntiAffinity.requiredDuringSchedulingIgnoredDuringExecution+`

| `+affinity.podAntiAffinity.topologyKey+`
| `+global.suseObservability.affinity.podAntiAffinity.topologyKey+`

| `+rancherUrl+`

| `+imageRegistry+` (uniquement à air comprimé)
| `+global.imageRegistry+`

| `+pullSecret.username+` (uniquement à air comprimé)
| `+global.suseObservability.pullSecret.username+`

| `+pullSecret.password+` (uniquement à air comprimé)
| `+global.suseObservability.pullSecret.password+`
|===

[NOTE]
====
Pour les environnements à air comprimé ou les registres personnalisés nécessitant la configuration de `+imageRegistry+` et `+pullSecret+`, voir l'xref:/k8s-suse-rancher-prime-air-gapped.adoc[installation de {stackstate-product-name} en mode à air comprimé.]
====

=== Étape 5 : Gérer les surcharges personnalisées

Si votre fichier de valeurs généré contenait des dérogations personnalisées, vous pouvez toujours les utiliser. Le nouveau mode global propose des valeurs par défaut raisonnables qui peuvent être modifiées pour chaque composant :

[,yaml]
----
global:
  suseObservability:
    sizing:
      profile: "150-ha"
    license: "YOUR-LICENSE-KEY"
    baseUrl: "https://observability.example.com"
    adminPassword: "your-password"
    # Global affinity - applied to all components by default
    affinity:
      nodeAffinity:
        requiredDuringSchedulingIgnoredDuringExecution:
          nodeSelectorTerms:
          - matchExpressions:
            - key: topology.kubernetes.io/zone
              operator: In
              values:
              - us-west-2a

# Custom per-component overrides still work - they take precedence over global defaults
stackstate:
  components:
    api:
      resources:
        requests:
          memory: 16Gi  # Override default from sizing profile
        limits:
          memory: 20Gi

elasticsearch:
  volumeClaimTemplate:
    resources:
      requests:
        storage: 500Gi  # Override default storage size
  # Override affinity for Elasticsearch specifically
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: node-type
            operator: In
            values:
            - storage-optimized
----

=== Étape 6 : Effectuer la mise à niveau

Tout d'abord, assurez-vous que vous disposez de la dernière version de la carte :

[,text]
----
helm repo update
----

Effectuez ensuite la mise à niveau :

[,text]
----
# Dry-run first to see what will change
helm upgrade suse-observability suse-observability/suse-observability \
  -n suse-observability \
  -f values.yaml \
  --dry-run

# If everything looks good, perform the actual upgrade
helm upgrade suse-observability suse-observability/suse-observability \
  -n suse-observability \
  -f values.yaml
----

=== Étape 7 : Vérifier la mise à niveau

[,text]
----
# Check all pods are running
kubectl get pods -n suse-observability

# Check for any errors in events
kubectl get events -n suse-observability --sort-by='.lastTimestamp' | tail -20

# Verify the Helm release
helm status suse-observability -n suse-observability

# Check the applied values
helm get values suse-observability -n suse-observability
----

== Mot de passe administrateur

Pour l'ancienne méthode, le site `+adminPassword+` a été fourni en texte brut au tableau suse-observability-values, qui l'a haché pour obtenir un hachage bcrypt. Dans la nouvelle méthode, vous pouvez fournir le texte en clair `+adminPassword+` au tableau de suse-observabilité, ou fournir le hachage bcrypt directement au tableau de suse-observabilité en utilisant `+adminPasswordBcrypt+`.

Si vous devez générer un nouveau hachage bcrypt :

[,bash]
----
htpasswd -bnBC 10 "" "your-password" | tr -d ':\n'
----

== Dépannage

=== Pods bloqués en état d'attente

Cela indique généralement un problème d'ordonnancement avec les règles d'anti-affinité. Pour les grappes avec un nombre limité de nœuds, utilisez l'anti-affinité douce :

[,yaml]
----
global:
  suseObservability:
    affinity:
      podAntiAffinity:
        requiredDuringSchedulingIgnoredDuringExecution: false  # Use soft anti-affinity
----

=== Les ressources diffèrent de celles de l'installation précédente

Les profils de dimensionnement peuvent comporter des recommandations de ressources mises à jour. Pour préserver vos paramètres précédents, ajoutez des dérogations explicites :

[,yaml]
----
global:
  suseObservability:
    sizing:
      profile: "150-ha"

# Override with your previous resource values
stackstate:
  components:
    api:
      resources:
        requests:
          memory: 8Gi  # Your previous value
----

=== Conflits de style de configuration

Ne mélangez pas les anciens et les nouveaux styles de configuration. Utilisez l'un ou l'autre :

* `+global.suseObservability.*+` (nouveau style), OU
* `+stackstate.license.key+``+stackstate.baseUrl+`, etc. (ancien style)

[,text]
----
# Check for conflicting values
helm get values suse-observability -n suse-observability | grep -E "license|baseUrl"
----

=== Revenir en arrière si nécessaire

[,text]
----
# List release history
helm history suse-observability -n suse-observability

# Rollback to previous revision
helm rollback suse-observability <revision-number> -n suse-observability
----

== Principales différences

[cols="1,1,1"]
|===
| Aspect | Héritage (valeurs d'observabilité suse) | Nouveau (global.suseObservability)

| Étapes de l'installation
| 2 (modèle + installation)
| 1 (installation)

| Gestion des fichiers de valeurs
| Généré, doit être stocké
| Source unique de vérité

| Mises à jour du profil
| Régénération manuelle nécessaire
| Automatique avec les mises à jour des cartes

| Tirer le secret
| Sous-carte séparée
| Intégré, auto-configuré

| Victoria Metrics HA
| Manuel `+enabled: true/false+`
| Auto-configuration par profil

| Configuration de l'affinité
| Généré par composant (paramètres distincts pour chaque service)
| Paramètres globaux centralisés (pouvant encore être modifiés par composant si nécessaire)
|===

[NOTE]
====
La configuration globale de l'affinité fournit des paramètres par défaut pour tous les composants de l'infrastructure. Vous pouvez toujours remplacer l'affinité pour des composants spécifiques (comme Elasticsearch ou Kafka) si nécessaire en définissant des valeurs d'affinité au niveau du composant. Les paramètres au niveau du composant ont la priorité sur les valeurs par défaut globales.
====

== Nettoyage

Une fois la migration réussie, vous pouvez

. Supprimer les fichiers de valeurs générés (ils ne sont plus nécessaires)
. Mettez à jour vos scripts de déploiement pour utiliser la nouvelle installation en une seule étape.
. Supprimez toute référence au graphique `+suse-observability-values+` dans vos pipelines CI/CD
