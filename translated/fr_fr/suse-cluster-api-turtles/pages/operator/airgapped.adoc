= Environnement avec bouchon d'air
:revdate: 2026-01-06
:page-revdate: {revdate}

{product_name} prend en charge un environnement air-gapped dès le départ en exploitant les fonctionnalités de l'API Cluster Operator, la dépendance requise pour l'installation de {product_name}.

Pour approvisionner et configurer les fournisseurs d'API de cluster, Turtles utilise la ressource *CAPIProvider* qui permet de gérer les manifestes des opérateurs d'API de cluster de manière déclarative. Chaque champ fourni par la ressource CAPI Operator en amont pour le site `+spec.type+` souhaité est également disponible dans le site `+spec+` de la ressource *CAPIProvider*.

Une nouvelle installation de {product_name} ne comprend que le fournisseur principal de l'ICPA et ses CRD. Le mécanisme d'installation par défaut de ce fournisseur ne nécessite pas de récupérer le manifeste à partir d'une source distante, de sorte qu'il est pleinement fonctionnel dans un environnement à air comprimé, puisqu'il récupère la définition YAML à partir d'un site local `+ConfigMap+`, intégré dans le diagramme de l'application.

[TIP]
====
La version de l'ICPA de base livrée avec {product_name} est activement testée et validée, et la carte est préconfigurée pour sélectionner cette version par défaut à l'aide de l'Opérateur ICPA. Toutefois, si vous avez des exigences spécifiques en matière de versions ou de référentiel à utiliser, vous pouvez toujours personnaliser le comportement de l'opérateur CAPI avec votre propre https://github.com/rancher/turtles/blob/5b6b33d894edf265e5d2b09d2718538892e83cd0/charts/rancher-turtles/values.yaml#L80[] fetchConfig.
====

*Principaux utilisateurs*: Les utilisateurs de Rancher Prime bénéficient d'artefacts OCI de fournisseurs CAPI pré-mirroisés disponibles dans le Rancher Prime Registry (`+registry.rancher.com+`). Ces fournisseurs sont automatiquement validés, testés et maintenus pour votre version de Turtles. Pour connaître les fournisseurs et les versions disponibles pour votre version de Turtles, consultez le fichier de configuration https://github.com/rancher/turtles/blob/main/internal/controllers/clusterctl/config-prime.yaml[config-prime.yaml].
endif::[]
ifeval::["{build-type}" == "community"]
== Utilisateurs communautaires
*Utilisateurs communautaires*: Les utilisateurs communautaires ont accès au fournisseur principal de l'ICPA, mais ne disposent pas d'artefacts OCI pré-mirroisés du fournisseur. Ils sont libres de mettre en miroir manuellement n'importe quel fournisseur CAPI ou d'utiliser des manifestes récupérés en fonction de leurs besoins. La configuration https://github.com/rancher/turtles/blob/main/internal/controllers/clusterctl/config-community.yaml[config-community.yaml] montre la configuration minimale de la communauté sans l'épinglage de la version du fournisseur.
endif::[]

Cette section fournit des conseils sur la manière d'utiliser `+CAPIProvider+` et la fonctionnalité de l'opérateur CAPI dans différents scénarios d'encaissement par voie aérienne.

== Préfixation automatique du registre avec Rancher Default Registry

=== Conditions préalables

Assurez-vous que le registre privé configuré dans le paramètre `+system-default-registry+` de Rancher contient des images miroir pour tous les fournisseurs CAPI que vous avez l'intention d'utiliser. Les chemins d'accès aux images dans le registre miroir doivent préserver l'espace de noms et la structure des noms d'images d'origine (par exemple, `+cluster-api/cluster-api-controller+`).
Reportez-vous aux fichiers de configuration clusterctl intégrés (https://github.com/rancher/turtles/blob/main/internal/controllers/clusterctl/config-community.yaml[config-community.yaml] / https://github.com/rancher/turtles/blob/main/internal/controllers/clusterctl/config-prime.yaml[config-prime.yaml]) pour obtenir la liste complète des images des fournisseurs qui doivent être mises en miroir.

{product_name} comprend une porte alpha, `+use-rancher-default-registry+`, qui simplifie la gestion des images dans les environnements à air comprimé. Lorsque cette option est activée (par défaut), {product_name} lit les paramètres de gestion de `+system-default-registry+` de Rancher et préfixe automatiquement tous les référentiels d'images de fournisseurs CAPI avec le registre configuré.

Sans cette fonctionnalité, les images des fournisseurs qui font référence à des registres externes tels que `+registry.k8s.io+` ne sont pas accessibles dans un cluster air-gapped. Le pipeline de mise en miroir d'images de Rancher met en miroir les images en amont, mais seule la partie registre d'une référence d'image est généralement modifiable - pas le nom de l'image lui-même.
Cela peut entraîner des incohérences lorsque les images mises en miroir utilisent une convention de dénomination différente (par exemple, `+rancher/mirrored-cluster-api-controller+` au lieu de `+cluster-api/cluster-api-controller+`), ou lorsque certaines images (par exemple, `+registry.k8s.io/kubernetes/kubectl+`) ne sont pas mises en miroir du tout.

Lorsque `+use-rancher-default-registry+` est activé, {product_name} résout ce problème en lisant le paramètre de gestion de `+system-default-registry+` Rancher et en utilisant sa valeur pour réécrire tous les référentiels d'images des fournisseurs. Le fichier clusterctl-config intégré `+ConfigMap+` sert de source de vérité pour savoir quels fournisseurs et quelles images sont pris en charge.
Si le paramètre est vide, {product_name} revient aux références d'image par défaut.

=== Comment ça marche

. {product_name} lit le paramètre de gestion de `+system-default-registry+` Rancher.
. Si le paramètre contient une URL de registre non vide, {product_name} parcourt toutes les images de fournisseur définies dans la configuration clusterctl intégrée.
. Pour chaque image, le préfixe original du registre est supprimé et remplacé par la valeur de `+system-default-registry+`, en préservant l'espace de noms et le nom de l'image.
+
Par exemple, `+registry.k8s.io/cluster-api/cluster-api-controller+` devient `+my-registry.example.com/cluster-api/cluster-api-controller+`.
. Toute surcharge d'image spécifiée dans la ressource `+ClusterctlConfig+` est appliquée _après le_ préfixe du registre, ce qui permet des surcharges sélectives par image.

[NOTE]
====
Les remplacements d'images définis dans la ressource `+ClusterctlConfig+` sont toujours prioritaires. Si une image de fournisseur spécifique est configurée via `+ClusterctlConfig+`, elle remplace la valeur fixée dans le registre. Il s'agit d'un mécanisme permettant d'ignorer sélectivement des images spécifiques tout en s'appuyant sur le préfixe automatique du registre pour tout le reste.
====

=== Activation ou désactivation

Cette fonction est activée par défaut. Pour le désactiver et revenir aux références de l'image d'origine, définissez les valeurs suivantes dans votre Helm :

[source,yaml]
----
features:
  use-rancher-default-registry:
    enabled: false
----

== Installation du fournisseur CAPI avec l'artefact OCI

En tant qu'administrateur travaillant dans un environnement à air comprimé, vous devez récupérer les composants du fournisseur CAPI à l'intérieur de votre cluster car l'accès externe à l'internet est restreint. Cette section explique comment déployer des fournisseurs CAPI à l'aide d'artefacts OCI.

ifeval::["{build-type}" == "product"]
=== Principaux utilisateurs

En tant qu'utilisateur de Rancher Prime, vous pouvez directement mettre en miroir les artefacts OCI prévalidés des fournisseurs depuis le Rancher Prime Registry vers votre registre privé.

Définissez l'URL de votre registre privé :

[source,bash]
----
export REGISTRY=<YOUR_PRIVATE_REGISTRY>
----

Mettez en miroir le fournisseur de Rancher Prime Registry dans votre registre privé. Reportez-vous à https://github.com/rancher/turtles/blob/main/internal/controllers/clusterctl/config-prime.yaml[config-prime.yaml] pour connaître les versions exactes des fournisseurs disponibles pour votre version de Turtles.

Par exemple, pour utiliser le fournisseur Azure :

[source,bash]
----
# Set the version from config-prime.yaml
export PROVIDER_VERSION=<VERSION_FROM_CONFIG_PRIME>

# Pull from Rancher Prime Registry
oras pull registry.rancher.com/rancher/cluster-api-azure-controller-components:${PROVIDER_VERSION}

# Push to your private registry
oras push ${REGISTRY}/cluster-api-azure-controller-components:${PROVIDER_VERSION} infrastructure-components.yaml:application/vnd.test.file metadata.yaml:application/vnd.test.file
----

Créez et appliquez la ressource CAPIProvider qui pointe vers votre registre privé :

.capz-provider-oci.yaml
[source,yaml]
----
apiVersion: turtles-capi.cattle.io/v1alpha1
kind: CAPIProvider
metadata:
  name: azure
  namespace: capz-system
spec:
  type: infrastructure
  name: azure
  version: ${PROVIDER_VERSION}
  fetchConfig:
    oci: ${REGISTRY}/cluster-api-azure-controller-components:${PROVIDER_VERSION}
----
endif::[]

ifeval::["{build-type}" == "community"]
=== Utilisateurs communautaires

En tant qu'utilisateur de la communauté, vous devez construire manuellement et empaqueter les artefacts OCI du fournisseur dans votre registre privé. Les utilisateurs de la communauté n'ont pas accès aux artefacts pré-mirroisés de registry.rancher.com.

Définissez l'URL de votre registre privé et la version du fournisseur :

[source,bash]
----
export REGISTRY=<YOUR_PRIVATE_REGISTRY>
export RELEASE_TAG=<CAPI_PROVIDER_VERSION>
----

Vous devrez construire les manifestes des fournisseurs à partir des sources en suivant la documentation du fournisseur en amont. Voir la section "CAPI Provider fetch and push OCI artifact" ci-dessous pour des instructions détaillées sur la construction à partir des sources et l'empaquetage en tant qu'artefacts OCI.

Une fois l'artefact OCI construit et transféré dans votre registre privé, créez et appliquez la ressource CAPIProvider qui pointe vers votre registre privé :

.capz-provider-oci.yaml
[source,yaml]
----
apiVersion: turtles-capi.cattle.io/v1alpha1
kind: CAPIProvider
metadata:
  name: azure
  namespace: capz-system
spec:
  type: infrastructure
  name: azure
  version: ${RELEASE_TAG}
  fetchConfig:
    oci: ${REGISTRY}:${RELEASE_TAG}
----
endif::[]

== Comment refléter l'artefact OCI du fournisseur CAPI ?

ifeval::["{build-type}" == "product"]

== Installation du fournisseur CAPI avec le manifeste récupéré

Cette section présente une autre approche pour les installations à air comprimé utilisant des ConfigMaps au lieu d'artefacts OCI. Cette méthode fonctionne à la fois pour les utilisateurs Community et Prime et peut être utile lorsque l'accès au registre OCI est limité ou lorsque vous préférez gérer directement les manifestes des fournisseurs.

En tant qu'administrateur, vous devez récupérer les composants du fournisseur vSphere (CAPV) à l'intérieur du cluster, car vous travaillez dans un environnement à air comprimé.

Dans cet exemple, il existe un ConfigMap dans l'espace de noms `+capv-system+` qui définit les composants et les métadonnées du fournisseur. Il peut être créé manuellement ou en exécutant les commandes suivantes :

[source,bash]
----
# Get the file contents from the GitHub release
curl -L https://github.com/rancher-sandbox/cluster-api-provider-vsphere/releases/download/v1.12.0/infrastructure-components.yaml -o components.yaml
curl -L https://github.com/rancher-sandbox/cluster-api-provider-vsphere/releases/download/v1.12.0/metadata.yaml -o metadata.yaml

# Create the configmap from the files
kubectl create configmap v1.12.0 --namespace=capv-system --from-file=components=components.yaml --from-file=metadata=metadata.yaml --dry-run=client -o yaml > configmap.yaml
----

Cet exemple de commande doit être adapté au fournisseur et à la version que vous souhaitez utiliser. La carte de configuration résultante ressemblera à l'exemple ci-dessous :

[source,yaml]
----
apiVersion: v1
kind: ConfigMap
metadata:
  labels:
    provider-components: vsphere
  name: v1.12.0
  namespace: capv-system
data:
  components: |
    # Components for v1.12.0 YAML go here
  metadata: |
    # Metadata information goes here
----

Une ressource *CAPIProvider* devra être créée pour représenter le fournisseur d'infrastructure vSphere. Il devra être configuré avec un `+fetchConfig+`. Le sélecteur d'étiquettes permet à l'opérateur de déterminer les versions disponibles du fournisseur vSphere et des ressources Kubernetes qui doivent être déployées (c'est-à-dire contenues dans les ConfigMaps qui correspondent au sélecteur d'étiquettes).

La version du fournisseur étant marquée comme `+v1.12.0+`, l'opérateur utilise les informations sur les composants de ConfigMap avec l'étiquette correspondante pour installer le fournisseur vSphere.

[source,yaml]
----
apiVersion: turtles-capi.cattle.io/v1alpha1
kind: CAPIProvider
metadata:
  name: vsphere
  namespace: capv-system
spec:
  name: vsphere
  type: infrastructure
  version: v1.12.0
  configSecret:
    name: vsphere-variables
  fetchConfig:
    selector:
      matchLabels:
        provider-components: vsphere
  deployment:
    containers:
    - name: manager
      imageUrl: "registry.suse.com/rancher/cluster-api-vsphere-controller:v1.12.0"
  variables:
    CLUSTER_TOPOLOGY: "true"
    EXP_CLUSTER_RESOURCE_SET: "true"
    EXP_MACHINE_POOL: "true"
----

En outre, le *CAPIProvider* modifie l'image du conteneur à utiliser pour le fournisseur à l'aide du champ `+deployment.containers[].imageUrl+`. Cela permet à l'opérateur d'extraire l'image d'un registre dans l'environnement à air comprimé.

=== Limitation de la taille de la carte de configuration

La d'un ConfigMap sur https://kubernetes.io/docs/concepts/configuration/configmap/#motivation[taille maximale] est limitée à 1 Mo. Si les manifestes ne tiennent pas dans cette taille, Kubernetes génère une erreur et l'installation du fournisseur échoue. Pour éviter cela, vous pouvez archiver les manifestes et les placer dans le ConfigMap de cette manière.

Par exemple, vous avez deux fichiers : `+components.yaml+` et `+metadata.yaml+`. Pour créer une carte de configuration fonctionnelle, il faut

. Archivage de components.yaml à l'aide de l'outil cli de `+gzip+` 
+
[source,sh]
----
gzip -c components.yaml > components.gz
----
+
. Créer un manifeste ConfigMap à partir des données archivées
+
[source,sh]
----
kubectl create configmap v1.12.0 --namespace=capv-system --from-file=components=components.gz --from-file=metadata=metadata.yaml --dry-run=client -o yaml > configmap.yaml
----
+
. Modifier le fichier en ajoutant l'annotation "provider.cluster.x-k8s.io/compressed : true".
+
[source,sh]
----
yq eval -i '.metadata.annotations += {"provider.cluster.x-k8s.io/compressed": "true"}' configmap.yaml
----
+
NOTE: Sans cette annotation, l'opérateur ne pourra pas déterminer si les données sont compressées ou non.

. Ajouter les étiquettes qui seront utilisées pour faire correspondre le ConfigMap dans la section `+fetchConfig+` du fournisseur.
+
[source,sh]
----
yq eval -i '.metadata.labels += {"my-label": "label-value"}' configmap.yaml
----
+
. Créer un ConfigMap dans votre cluster Kubernetes à l'aide de kubectl
+
[source,sh]
----
kubectl create -f configmap.yaml
----
