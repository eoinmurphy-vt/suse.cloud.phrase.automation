= ClusterClass
:revdate: 2026-02-18
:page-revdate: {revdate}

Cette section traite de l'utilisation de https://cluster-api.sigs.k8s.io/tasks/experimental-features/cluster-class/[ClusterClass] avec {product_name}.

== Conditions préalables

* Cluster Rancher Manager avec {product_name} installé
* Le fournisseur https://github.com/rancher/cluster-api-addon-provider-fleet[CAAPF] doit être installé comme condition préalable à l'utilisation de toutes les ClusterClasses.
** Exemple d'installation d'un fournisseur CAAPF :
+
.Cliquer pour agrandir
[%collapsible]
======
[,yaml]
----
apiVersion: v1
kind: Namespace
metadata:
  name: fleet-addon-system
---
apiVersion: turtles-capi.cattle.io/v1alpha1
kind: CAPIProvider
metadata:
  name: fleet
  namespace: fleet-addon-system
spec:
  name: rancher-fleet
  type: addon
----
======

[WARNING]
====
CAAPF dépend du portail de fonctionnalités Kubernetes `+WatchList+`. Cette fonctionnalité doit être explicitement activée sur les versions de Kubernetes `+1.33+`. Voir la https://kubernetes.io/docs/reference/command-line-tools-reference/feature-gates/[documentation] Kubernetes pour plus d'informations.
====

* En fonction de votre moteur, l'installation de https://github.com/rancher/cluster-api-provider-rke2[CAPRKE2] ou de https://github.com/kubernetes-sigs/cluster-api[] Kubeadm Bootstrap/Control Plane est nécessaire avant de poursuivre l'installation de CAPIProvider :

== Mise en place

ifeval::["{build-type}" == "community"]
[NOTE]
.Épinglage de la version
====
Le CAPIProvider `+spec.version+` peut être défini à tout moment pour rattacher le fournisseur à une version spécifique. Pour que l'immutabilité persiste, vous pouvez également définir `+spec.enableAutomaticUpdate: false+` pour désactiver les mises à jour automatiques de la version de votre fournisseur CAPIP respectif.
====
endif::[]

[tabs]
======
Azure::
+
--
Pour préparer le cluster de gestion, nous allons installer le https://capz.sigs.k8s.io/[Cluster API Provider Azure], et créer une identité https://capz.sigs.k8s.io/topics/identities#service-principal[ServicePrincipal] pour provisionner un nouveau cluster sur Azure.

Avant de commencer, un ServicePrincipal doit être créé, avec au moins un accès Contributor à un abonnement Azure.
Pour plus d'informations, consultez la https://capz.sigs.k8s.io/topics/identities[documentation du CAPZ à l'adresse].

* Installation du fournisseur
+
[source,yaml]
----
apiVersion: v1
kind: Namespace
metadata:
  name: capz-system
---
apiVersion: turtles-capi.cattle.io/v1alpha1
kind: CAPIProvider
metadata:
  name: azure
  namespace: capz-system
spec:
  type: infrastructure
  name: azure
----

* Configuration de l'identité
+
[WARNING]
.Authentification pour le provisionnement des clusters AKS
====
Si vous déployez un cluster géré sur **AKS**, vous pouvez sauter l'étape de **configuration de l'identité** et passer à la xref:#_create_a_cluster_from_a_clusterclass[création d'un cluster à partir d'une ClusterClass].
Au lieu d'utiliser `+AzureClusterIdentity+`, vous allez créer un `+Secret+` avec les informations d'identification du Service Principal pour que l'Azure Service Operator puisse les utiliser lors de l'application du manifeste de cluster.
====
+
Dans cet exemple, nous allons utiliser un site `+AzureClusterIdentity+` pour provisionner des clusters Azure.
Il faut d'abord créer un Secret contenant les informations d'identification du Service Principal, qui sera référencé par la ressource `+AzureClusterIdentity+`.
Notez que le site `+AzureClusterIdentity+` est une ressource à espace de noms et qu'il doit être créé dans le même espace de noms que le cluster.
Pour plus d'informations sur les meilleures pratiques lors de l'utilisation des identités Azure, veuillez vous référer à la officielle https://capz.sigs.k8s.io/topics/identities-use-cases[documentation].
+
Notez que certaines variables sont laissées à l'appréciation de l'utilisateur.
+
[source,yaml]
----
apiVersion: v1
kind: Secret
metadata:
  name: <AZURE_CLUSTER_IDENTITY_SECRET_NAME>
  namespace: <AZURE_CLUSTER_IDENTITY_SECRET_NAMESPACE>
type: Opaque
stringData:
  clientSecret: <AZURE_CLIENT_SECRET>
---
apiVersion: infrastructure.cluster.x-k8s.io/v1beta1
kind: AzureClusterIdentity
metadata:
  labels:
    clusterctl.cluster.x-k8s.io/move-hierarchy: "true"
  name: cluster-identity
spec:
  allowedNamespaces: {}
  clientID: <AZURE_APP_ID>
  clientSecret:
    name: <AZURE_CLUSTER_IDENTITY_SECRET_NAME>
    namespace: <AZURE_CLUSTER_IDENTITY_SECRET_NAMESPACE>
  tenantID: <AZURE_TENANT_ID>
  type: ServicePrincipal
----
--

AWS::
+
--
Pour préparer le cluster de gestion, nous allons installer le https://cluster-api-aws.sigs.k8s.io/[Cluster API Provider AWS], et créer un secret avec les informations d'identification requises pour provisionner un nouveau cluster sur AWS.
Les identifiants globaux sont mis à blanc, car nous allons utiliser `+AWSClusterStaticIdentity+` à la place.

* Installation du fournisseur
+
[source,yaml]
----
apiVersion: turtles-capi.cattle.io/v1alpha1
kind: CAPIProvider
metadata:
  name: aws
  namespace: capa-system
spec:
  type: infrastructure
  variables:
    AWS_B64ENCODED_CREDENTIALS: ""
----

* Configuration de l'identité
+
Dans cet exemple, nous allons utiliser un site `+AWSClusterStaticIdentity+` pour provisionner des clusters AWS. +
Un Secret contenant les informations d'identification doit être créé dans l'espace de noms où le fournisseur AWS est installé. +
Pour plus d'informations sur la configuration des informations d'identification, reportez-vous à la link:https://cluster-api-aws.sigs.k8s.io/clusterawsadm/clusterawsadm[documentation de]. +
Le site `+AWSClusterStaticIdentity+` peut faire référence à ce secret pour permettre le provisionnement du cluster. Dans cet exemple, nous autorisons l'utilisation de l'identité dans tous les espaces de noms, afin qu'elle puisse être facilement réutilisée. +
Vous pouvez vous référer à la link:https://cluster-api-aws.sigs.k8s.io/topics/multitenancy[documentation officielle de] pour en savoir plus sur la gestion des identités.
+
Notez que certaines variables sont laissées à l'appréciation de l'utilisateur. +
+
[source,yaml]
----
apiVersion: v1
kind: Secret
metadata:
  name: <AWS_IDENTITY_SECRET_NAME>
  namespace: capa-system
type: Opaque
stringData:
  AccessKeyID: <AWS_ACCESS_KEY_ID>
  SecretAccessKey: <AWS_SECRET_ACCESS_KEY>
---
apiVersion: infrastructure.cluster.x-k8s.io/v1beta2
kind: AWSClusterStaticIdentity
metadata:
  name: cluster-identity
spec:
  secretRef: <AWS_IDENTITY_SECRET_NAME>
  allowedNamespaces: {}
----
--

GCP::
+
--
Pour préparer le cluster de gestion, nous allons installer le https://cluster-api-gcp.sigs.k8s.io/[Cluster API Provider GCP], et créer un secret avec les informations d'identification requises pour provisionner un nouveau cluster sur GCP.
Un compte de service est nécessaire pour créer et gérer des clusters dans GCP, ce qui nécessite des autorisations à l'adresse `+Editor+`. Vous pouvez suivre le guide officiel de https://cluster-api-gcp.sigs.k8s.io/quick-start#create-a-service-account[CAPG Book.]
La clé du compte de service codée en base64 doit être définie dans la variable `+GCP_B64ENCODED_CREDENTIALS+` du fournisseur.

* Installation du fournisseur
+
[source,yaml]
----
apiVersion: turtles-capi.cattle.io/v1alpha1
kind: CAPIProvider
metadata:
  name: gcp
  namespace: capg-system
spec:
  type: infrastructure
  variables:
    GCP_B64ENCODED_CREDENTIALS: xxx
----

* Configuration du réseau
+
Le provisionnement d'un cluster GCP autogéré nécessite qu'un réseau GCP soit configuré pour permettre aux nœuds Kubernetes de communiquer avec le plan de contrôle et de tirer des images du registre de conteneurs pour lequel les machines doivent disposer d'un accès NAT ou d'une IP publique.
Le comportement par défaut du fournisseur est de créer des machines virtuelles sans adresse IP publique, de sorte qu'un https://cloud.google.com/nat/docs/overview[NAT] est nécessaire pour permettre aux nœuds d'établir une connexion avec l'équilibreur de charge et le monde extérieur.
Veuillez vous référer au guide officiel https://cluster-api-gcp.sigs.k8s.io/prerequisites#configure-network-and-cloud-nat[CAPG Book] pour savoir comment préparer votre réseau GCP pour provisionner un cluster GCP autogéré.
+
[NOTE]
====
Les étapes suivantes sont nécessaires pour préparer le réseau GCP au provisionnement des clusters :

- Créer un routeur.
- Créer un NAT associé au routeur.
====

vSphere::
+
--
Pour préparer le cluster de gestion, nous allons installer le https://github.com/kubernetes-sigs/cluster-api-provider-vsphere/blob/main/docs/getting_started.md[Cluster API Provider vSphere].
Les identifiants globaux sont mis à blanc, car nous allons utiliser `+VSphereClusterIdentity+` à la place.

* Installation du fournisseur
+
[source,yaml]
----
apiVersion: v1
kind: Namespace
metadata:
  name: capv-system
---
apiVersion: turtles-capi.cattle.io/v1alpha1
kind: CAPIProvider
metadata:
  name: vsphere
  namespace: capv-system
spec:
  type: infrastructure
  variables:
    VSPHERE_USERNAME: "" 
    VSPHERE_PASSWORD: ""
----

* Configuration de l'identité
+
Dans cet exemple, nous allons utiliser un site `+VSphereClusterIdentity+` pour provisionner des clusters vSphere. +
Un Secret contenant les informations d'identification doit être créé dans l'espace de noms où le fournisseur vSphere est installé.
Le site `+VSphereClusterIdentity+` peut faire référence à ce secret pour permettre le provisionnement du cluster. Dans cet exemple, nous autorisons l'utilisation de l'identité dans tous les espaces de noms, afin qu'elle puisse être facilement réutilisée.
Vous pouvez vous référer à la https://github.com/kubernetes-sigs/cluster-api-provider-vsphere/blob/main/docs/identity_management.md[documentation officielle de] pour en savoir plus sur la gestion des identités.
+
[source,yaml]
----
apiVersion: v1
kind: Secret
metadata:
  name: cluster-identity
  namespace: capv-system
type: Opaque
stringData:
  username: xxx
  password: xxx
---
apiVersion: infrastructure.cluster.x-k8s.io/v1beta1
kind: VSphereClusterIdentity
metadata:
  name: cluster-identity
spec:
  secretName: cluster-identity
  allowedNamespaces:
    selector:
      matchLabels: {}
----

== Créer un cluster à partir d'une ClusterClass

[tabs]
======

Azure RKE2::
+
--
* Une classe Azure ClusterClass se trouve parmi les https://github.com/rancher/turtles/tree/main/examples/clusterclasses[exemples de].
+
Des applications telles que Azure Cloud Provider et Calico CNI seront installées sur des clusters en aval. Cette opération est effectuée automatiquement lors de la création du cluster en ciblant les clusters avec des étiquettes spécifiques, telles que `+cloud-provider: azure+` et `+cni: calico+`.
+
[tabs]
=======
CLI::
+
Une Azure RKE2 ClusterClass et les applications associées peuvent être appliquées à l'aide de l'outil d'exemples :
+
[source,bash, subs="attributes"]
----
go run github.com/rancher/turtles/examples@{turtles_version} -r azure-rke2 | kubectl apply -f -
----
kubectl::
+
* Vous pouvez également appliquer la Azure RKE2 ClusterClass directement à l'aide de kubectl :
+
[source,bash, subs="attributes"]
----
kubectl apply -f https://raw.githubusercontent.com/rancher/turtles/refs/tags/{turtles_version}/examples/clusterclasses/azure/rke2/clusterclass-rke2-example.yaml
----

* En outre, le https://capz.sigs.k8s.io/self-managed/cloud-provider-config[Azure Cloud Provider] devra être installé sur chaque cluster en aval, pour que les nœuds soient initialisés correctement.
Pour cet exemple, nous allons également installer https://docs.tigera.io/calico/latest/about/[Calico] comme CNI par défaut.
+
Nous pouvons le faire automatiquement lors de la création du cluster à l'aide de https://rancher.github.io/cluster-api-addon-provider-fleet/[Cluster API Add-on Provider Fleet].
Il faut d'abord créer deux sites `+HelmOps+`, qui seront appliqués à la nouvelle grappe par le biais de sélecteurs d'étiquettes.
+
[source,bash, subs="attributes"]
----
kubectl apply -f https://raw.githubusercontent.com/rancher/turtles/refs/tags/{turtles_version}/examples/applications/ccm/azure/helm-chart.yaml
kubectl apply -f https://raw.githubusercontent.com/rancher/turtles/refs/tags/{turtles_version}/examples/applications/cni/calico/helm-chart.yaml
----
=======

* Créer le cluster Azure à partir de l'exemple ClusterClass
+
Notez que certaines variables sont laissées à l'appréciation de l'utilisateur.
+
[source,yaml]
----
apiVersion: cluster.x-k8s.io/v1beta2
kind: Cluster
metadata:
  labels:
    cluster-api.cattle.io/rancher-auto-import: "true"
    cloud-provider: azure
    cni: calico
  name: azure-quickstart
spec:
  clusterNetwork:
    pods:
      cidrBlocks:
      - 192.168.0.0/16
  topology:
    classRef: 
      name: azure-rke2-example
    controlPlane:
      replicas: 1
    variables:
    - name: subscriptionID
      value: <AZURE_SUBSCRIPTION_ID>
    - name: location
      value: <AZURE_LOCATION>
    - name: resourceGroup
      value: <AZURE_RESOURCE_GROUP>
    - name: azureClusterIdentityName
      value: cluster-identity
    version: v1.34.3+rke2r1
    workers:
      machineDeployments:
      - class: rke2-default-worker
        name: md-0
        replicas: 1
----
--

* Créez le cluster Azure AKS à partir de l'exemple ClusterClass.
+
L'Azure AKS `+ClusterClass+` à partir duquel `+Cluster+` est créé utilise les ressources `+AzureASOManagedCluster+`, `+AzureASOManagedControlPlane+` et `+AzureASOManagedMachinePool+` qui sont gérées par l'Azure Service Operator.
Pour cette raison, vous n'utiliserez pas `+AzureClusterIdentity+` comme vous le feriez pour un cluster Azure autogéré (par exemple Azure avec RKE2). Au lieu de cela, un site `+Secret+` est créé avec les informations d'identification du Service Principal dans le même espace de noms que le site `+Cluster+`.
Notez que la définition de `+Secret+` est intégrée dans le manifeste `+Cluster+` pour des raisons de simplicité, mais elle peut également être créée au préalable.
+
Vous pouvez consulter la https://azure.github.io/azure-service-operator/guide/authentication/[documentation de] pour plus d'informations sur cette méthode d'authentification.
+
Notez que certaines variables sont laissées à l'appréciation de l'utilisateur.
+
[source,yaml]
----
apiVersion: v1
kind: Secret
metadata:
  name: aks-credentials
stringData:
  AZURE_SUBSCRIPTION_ID: <AZURE_SUBSCRIPTION_ID>
  AZURE_TENANT_ID: <AZURE_TENANT_ID>
  AZURE_CLIENT_ID: <AZURE_CLIENT_ID>
  AZURE_CLIENT_SECRET: <AZURE_CLIENT_SECRET>
type: Opaque
---
apiVersion: cluster.x-k8s.io/v1beta2
kind: Cluster
metadata:
  labels:
    cluster-api.cattle.io/rancher-auto-import: "true"
  name: azure-aks-quickstart
spec:
  clusterNetwork:
    pods:
      cidrBlocks:
      - 192.168.0.0/16
  topology:
    class: azure-aks-example
    variables:
    - name: location
      value: <AZURE_LOCATION>
    - name: resourceGroup
      value: <AZURE_RESOURCE_GROUP>
    - name: asoCredentialSecretName
      value: aks-credentials
    version: v1.34.2
    workers:
      machinePools:
      - class: default-system
        name: system-1
        replicas: 1
      - class: default-worker
        name: worker-1
        replicas: 1
----
--

Le vSphere Cloud Provider et le contrôleur vSphere CSI ont besoin d'une configuration supplémentaire pour être appliqués sur le cluster en aval.
De la même manière que pour les étapes précédentes, nous pouvons créer deux Fleet Bundles supplémentaires, qui seront appliqués au Cluster en aval.
Veuillez noter que ces Bundles sont configurés pour cibler le Cluster en aval par son nom : `+vsphere-kubeadm-quickstart+`.
Si vous utilisez un nom différent pour votre cluster, modifiez les cibles de l'ensemble en conséquence.
+
[source,yaml]
----
kind: Bundle
apiVersion: fleet.cattle.io/v1alpha1
metadata:
  name: vsphere-csi-config
spec:
  resources:
  - content: |-
      apiVersion: v1
      kind: Secret
      type: Opaque
      metadata:
        name: vsphere-config-secret
        namespace: vmware-system-csi
      stringData:
        csi-vsphere.conf: |+
          [Global]
          thumbprint = "<VSPHERE_THUMBPRINT>"
          [VirtualCenter "<VSPHERE_SERVER>"]
          user = "<VSPHERE_USER>"
          password = "<VSPHERE_PASSWORD>"
          datacenters = "<VSPHERE_DATACENTED>"
          [Network]
          public-network = "<VSPHERE_NETWORK>"
          [Labels]
          zone = ""
          region = ""
  targets:
  - clusterSelector:
      matchLabels:
        csi: vsphere
        cluster.x-k8s.io/cluster-name: 'vsphere-kubeadm-quickstart'
---
kind: Bundle
apiVersion: fleet.cattle.io/v1alpha1
metadata:
  name: vsphere-cloud-credentials
spec:
  resources:
  - content: |-
      apiVersion: v1
      kind: Secret
      type: Opaque
      metadata:
        name: vsphere-cloud-secret
        namespace: kube-system
      stringData:
        <VSPHERE_SERVER>.password: "<VSPHERE_PASSWORD>"
        <VSPHERE_SERVER>.username: "<VSPHERE_USER>"
  targets:
  - clusterSelector:
      matchLabels:
        cloud-provider: vsphere
        cluster.x-k8s.io/cluster-name: 'vsphere-kubeadm-quickstart'
----

* Créer le cluster vSphere à partir de l'exemple ClusterClass
+
Notez que pour cet exemple, nous utilisons https://kube-vip.io/[] kube-vip comme équilibreur de charge du plan de contrôle.
L'adresse `+KUBE_VIP_INTERFACE+` sera utilisée pour lier l'adresse `+CONTROL_PLANE_IP+` en mode ARP. En fonction de la configuration de votre système d'exploitation et de votre périphérique réseau, vous devez configurer cette valeur en conséquence - par exemple, sur `+eth0+`.
Le manifeste statique `+kube-vip+` est intégré dans la définition de ClusterClass. Pour plus d'informations sur la façon de générer un manifeste kube-vip statique pour vos propres ClusterClasses, veuillez consulter la officielle https://kube-vip.io/docs/installation/static/[documentation].
+
[source,yaml]
----
apiVersion: cluster.x-k8s.io/v1beta2
kind: Cluster
metadata:
  labels:
    cni: calico
    cloud-provider: vsphere
    csi: vsphere
    cluster-api.cattle.io/rancher-auto-import: "true"
  name: 'vsphere-kubeadm-quickstart'
spec:
  clusterNetwork:
    pods:
      cidrBlocks:
      - 192.168.0.0/16
  topology:
    classRef: 
      name: vsphere-kubeadm-example
    version: v1.35.0
    controlPlane:
      replicas: 3
    workers:
      machineDeployments:
      - class: vsphere-kubeadm-example-worker
        name: md-0
        replicas: 1
    variables:
    - name: vSphereClusterIdentityName
      value: cluster-identity
    - name: vSphereTLSThumbprint
      value: <VSPHERE_THUMBPRINT>
    - name: vSphereDataCenter
      value: <VSPHERE_DATACENTER>
    - name: vSphereDataStore
      value: <VSPHERE_DATASTORE>
    - name: vSphereFolder
      value: <VSPHERE_FOLDER>
    - name: vSphereNetwork
      value: <VSPHERE_NETWORK>
    - name: vSphereResourcePool
      value: <VSPHERE_RESOURCE_POOL>
    - name: vSphereServer
      value: <VSPHERE_SERVER>
    - name: vSphereTemplate
      value: <VSPHERE_TEMPLATE>
    - name: controlPlaneIpAddr
      value: <CONTROL_PLANE_IP>
    - name: controlPlanePort
      value: 6443
    - name: sshKey
      value: <SSH_KEY>
    - name: kubeVIPInterface
      value: <KUBE_VIP_INTERFACE>
----

vSphere RKE2::
+
--
* Une vSphere ClusterClass se trouve parmi les https://github.com/rancher/turtles/tree/main/examples/clusterclasses[exemples de].
+
Des applications telles que vSphere Cloud Provider, le pilote vSphere CSI et Calico CNI seront installées sur les clusters en aval. Cette opération est réalisée automatiquement lors de la création du cluster en ciblant les clusters avec des étiquettes spécifiques, telles que `+cloud-provider: vsphere+`, `+csi: vsphere+`, et `+cni: calico+`.
+
[tabs]
=======

CLI::
+
Une ClusterClass vSphere RKE2 et les applications associées peuvent être appliquées à l'aide de l'outil d'exemples :
+
[source,bash, subs="attributes"]
----
go run github.com/rancher/turtles/examples@{turtles_version} -r vsphere-rke2 | kubectl apply -f -
----

kubectl::
+
* Vous pouvez également appliquer la ClusterClass vSphere RKE2 directement à l'aide de kubectl :
+
[source,bash, subs="attributes"]
----
kubectl apply -f https://raw.githubusercontent.com/rancher/turtles/refs/tags/{turtles_version}/examples/clusterclasses/vsphere/rke2/clusterclass-rke2-example.yaml
----

* En outre, le https://github.com/kubernetes/cloud-provider-vsphere[Cloud Provider] vSphere devra être installé sur chaque cluster en aval, pour que les nœuds soient initialisés correctement.
Le https://github.com/kubernetes-sigs/vsphere-csi-driver[pilote] sera utilisé comme solution de stockage.
Enfin, pour cet exemple, nous allons installer https://docs.tigera.io/calico/latest/about/[Calico] comme CNI par défaut.
+
Nous pouvons installer toutes les applications automatiquement lors de la création du cluster à l'aide de https://rancher.github.io/cluster-api-addon-provider-fleet/[Cluster API Add-on Provider Fleet].
Il faut d'abord créer deux sites `+HelmOps+`, qui seront appliqués à la nouvelle grappe par le biais de sélecteurs d'étiquettes.
+
[source,bash, subs="attributes"]
----
kubectl apply -f https://raw.githubusercontent.com/rancher/turtles/refs/tags/{turtles_version}/examples/applications/ccm/vsphere/helm-chart.yaml
kubectl apply -f https://raw.githubusercontent.com/rancher/turtles/refs/tags/{turtles_version}/examples/applications/cni/calico/helm-chart.yaml
----
+
Puisque le pilote vSphere CSI n'est pas packagé dans Helm, nous allons inclure son manifeste complet dans un Fleet Bundle, qui sera appliqué au Cluster en aval.
+
[source,bash, subs="attributes"]
----
kubectl apply -f https://raw.githubusercontent.com/rancher/turtles/refs/tags/{turtles_version}/examples/applications/csi/vsphere/bundle.yaml
----
