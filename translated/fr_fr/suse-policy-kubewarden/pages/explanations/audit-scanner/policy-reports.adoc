Lorsque vous utilisez le scanner d'audit {project-name}, vous stockez les résultats des analyses de politiques à l'aide des ressources personnalisées d' https://openreports.io/[OpenReports].

[CAUTION]
====

Avec {project-name} 1.33, la fonction Audit Scanner enregistre par défaut les rapports dans des CRD OpenReports à partir de [openreports.io] (https://openreports.io).

Les CRDslink:https://htmlpreview.github.io/?https://github.com/kubernetes-sigs/wg-policy-prototypes/blob/045372e558b896695b2daae92e8c7a04d4d40282/policy-report/docs/index.html[PolicyReport]du groupe de travail K8s Policy ne sont pas installés par défaut, sont marqués comme dépréciés et seront supprimés dans une prochaine version.

Les utilisateurs qui souhaitent continuer à utiliser les PolicyReports dépréciés, c'est-à-dire le comportement antérieur à la version 1.33, doivent :- Changer la valeur du diagramme de Helm `+kubewarden-crds+` de `+.Values.install.installPolicyReportCRDs+` à `+true+`- Changer la valeur du diagramme de Helm `+kubewarden-controller+` de `+.Values.auditScanner.reportCRDsKind+` de `+openreports+` à `+policyreport+`.

====


Ces CRD OpenReport offrent un moyen structuré de stocker et de gérer les résultats de l'audit.

Vous stockez les résultats de l'analyse de la politique d'audit à l'aide de `+Report+` et `+ClusterReport+`.
L'analyseur d'audit crée un `+Report+` ou un `+ClusterReport+` pour chaque ressource analysée, en fonction de l'étendue de la ressource. Les objets `+Report+` sont disponibles dans l'espace de noms de la ressource, tandis que les objets `+ClusterReport+` sont disponibles dans l'étendue du cluster.

Les résultats de l'audit générés par le scanner sont les suivants :

* la politique évaluée
* la ressource scannée
* le résultat de l'évaluation (réussite, échec ou non)
* un horodatage indiquant le moment où l'évaluation a eu lieu.

Vous pouvez également définir des annotations de gravité et de catégorie pour vos politiques.

Les opérateurs peuvent interroger les rapports en utilisant `+kubectl+`. Ils peuvent également utiliser l'interface utilisateur optionnelle fournie par le projet open-sourcehttps://kyverno.github.io/policy-reporter[policy-reporter] pour le suivi et l'observabilité des CRD OpenReports.

== Interroger les rapports

En utilisant l'interface de commande kubectl, il est possible d'interroger les résultats de l'analyse :

Liste les rapports dans l'espace de noms par défaut :

[source,console,subs="+attributes"]
----
$ kubectl get report -o wide
NAME                                   KIND   NAME            PASS   FAIL   WARN   ERROR   SKIP   AGE
44da974b-6057-4745-a92c-9dd8a1b01cc4   Pod    nginx-secrets   6      1      0      0       0      33m
----

Liste des rapports sur l'ensemble du groupe :

[source,console,subs="+attributes"]
----
$ kubectl get clusterreport -o wide
NAME                                   KIND        NAME                 PASS   FAIL   WARN   ERROR   SKIP   AGE
6c66175f-e15f-47e8-8c33-2c63d5ecde3e   Namespace   namespace1           3      1     0       0       0      1h
d0ffbf4f-7348-4350-af7b-fdbfc079cc49   Namespace   namespace2           0      4     0       0       0      1h
e48d9bbc-a9c4-4c36-8519-bc7390d9ec64   Namespace   namespace3           4      0     0       0       0      15h
----

Obtenir les détails d'un rapport spécifique :

[source,console,subs="+attributes"]
----
kubectl get report 44da974b-6057-4745-a92c-9dd8a1b01cc4 -o yaml
----

Obtenir les détails d'un rapport de cluster spécifique :

[source,console,subs="+attributes"]
----
kubectl get clusterreport e48d9bbc-a9c4-4c36-8519-bc7390d9ec64 -o yaml
----

L'interface utilisateur du rapporteur de politique fournit un tableau de bord indiquant toutes les violations. Voir la capture d'écran ci-dessous :

image::policy-reporter_dashboard.png[Exemple de tableau de bord Policy Reporter]
