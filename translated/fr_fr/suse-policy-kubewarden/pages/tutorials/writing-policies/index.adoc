Cette section présente les politiques {project-name} en utilisant des analogies informatiques traditionnelles.

Une politique {short-project-name} est comme un programme qui fait un seul travail.
Il reçoit des données d'entrée, effectue un calcul avec ces données et renvoie un résultat.

Les données d'entrée sont les demandes d'admission de Kubernetes.
Le résultat du calcul est une réponse de validation, qui indique à Kubernetes s'il faut accepter, rejeter ou modifier les données d'entrée, c'est-à-dire la demande d'admission.

Le composant https://github.com/kubewarden/policy-server[serveur de politique] de {short-project-name} effectue ces opérations.

Le serveur de politiques ne comprend aucune capacité de traitement des données.
Vous ajoutez des capacités de traitement au moment de l'exécution à l'aide de modules complémentaires.
Ces compléments sont les polices {short-project-name}.

Ainsi, une politique {short-project-name} est comme unhttps://en.wikipedia.org/wiki/Plug-in_%28computing%29[plug-in traditionnel]du programme "Policy Server".

Pour résumer :

* {short-project-name} Les politiques sont des plug-ins qui exposent un ensemble de fonctionnalités bien définies (valider un objet de requête Kubernetes, valider les paramètres de politique fournis par l'utilisateur, et d'autres fonctions) à l'aide d'une API bien définie.
* Le serveur de politiques est le programme "principal" qui charge les plug-ins (alias politiques) et utilise leurs fonctionnalités exposées pour accepter, rejeter ou faire muter les demandes Kubernetes

L'écriture des politiques {short-project-name} consiste à écrire la logique de validation et à l'exposer par le biais d'une API bien définie.

== Exigences en matière de langage de programmation

Vous fournissez les politiques {short-project-name} sous forme de binaireshttps://webassembly.org/[WebAssembly].

Les auteurs de politiques peuvent rédiger des politiques à l'aide de n'importe quel langage de programmation prenant en charge WebAssembly en tant que cible de compilation.
La liste des langages supportés est en constante évolution, https://github.com/appcypher/awesome-wasm-langs[cette page] donne un bon aperçu du paysage de WebAssembly.

Actuellement, WebAssembly n'a pas de moyen officiel de partager des types de données complexes entre l'hôte et un invité WebAssembly.
Pour surmonter cette limitation, les politiques de {short-project-name} utilisent le projet https://github.com/wapc[waPC], qui fournit un canal de communication bidirectionnel.

Le langage de programmation que vous avez choisi doit donc disposer d'un kit de développement pour les invités waPC.
Si ce n'est pas le cas, n'hésitez pas à nous contacter.
L'équipe du projet peut vous aider à surmonter cette difficulté.
