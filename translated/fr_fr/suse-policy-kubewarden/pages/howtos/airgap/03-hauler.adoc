= Installation d'une lame d'air avec Hauler
:revdate: 2026-03-18
:page-revdate: {revdate}
:description: Utilisez Hauler pour installer {project-name} dans des environnements à air comprimé.
:keywords: ["{project-name}", kubewarden, kubernetes, air gap installation, hauler]
:doc-persona: kubewarden-operator, kubewarden-integrator
:doc-type: howto
:doc-topic: manuel d'utilisation, entrefer, installation, transporteur

Ce guide vous montre comment installer {project-name} dans des environnements à air comprimé à l'aide delink:https://docs.hauler.dev/docs/intro[Hauler]. Hauler est un outil qui aide les utilisateurs à exécuter des charges de travail dans des environnements à air comprimé. Il déplace les ressources requises pour les applications dans ces environnements à l'abri de l'air.

{short-project-name} fournit un fichier manifeste contenant toutes les ressources nécessaires à l'exécution de {short-project-name} dans un environnement privé. Cette documentation décrit comment vous pouvez l'utiliser. Pour en savoir plus, reportez-vous à la documentation sur le transporteur.

Le processus de base utilisant le manifeste Hauler est le suivant :

. Obtenez-le à partir d'une page de publication {short-project-name} 
. Charger toutes les ressources nécessaires à l'exécution de {short-project-name} dans un magasin local.
. Les exporter dans un fichier
. Déplacer le dossier dans l'environnement privé
. Chargement des ressources dans le transporteur fonctionnant dans l'environnement privé
. Copier le tout dans un registre, qui sera utilisé lors de la procédure d'installation, dans votre environnement climatisé.

== Télécharger le manifeste du transporteur à partir de la page de publication des cartes de Helm

Téléchargez `+hauler_manifest.yml+` à partir de la {short-project-name}link:https://github.com/kubewarden/helm-charts/releases/[page]

Synchroniser les ressources définies dans le manifeste avec votre magasin Hauler :

[source,shell]
----
hauler store sync --filename hauler_manifest.yaml
----

Le transporteur télécharge toutes les ressources du manifeste vers le magasin local. Ce processus prend quelques minutes. Une fois la synchronisation terminée, vous pouvez voir les ressources synchronisées à l'aide de la commande `+hauler store info+`.

== Générer le fichier tarball avec toutes les ressources {short-project-name} 

Exécutez la commande Hauler pour exporter dans un fichier toutes les ressources précédemment chargées dans le magasin :

[source,shell]
----
hauler store save --filename kubewarden-resources.tar.zst
----

[NOTE]
====

{short-project-name} Les images des conteneurs prennent en charge les architectures x86_64 et Arm. Par conséquent, lorsque vous enregistrez la ressource dans le fichier, vous pouvez voir des messages d'avertissement comme celui-ci :

[source,shell]
----

2025-08-12 18:55:54 WRN specify an export platform to prevent potential platform mismatch on import of index [ghcr.io/kyverno/policy-reporter-ui:2.4.1]
2025-08-12 18:55:54 WRN specify an export platform to prevent potential platform mismatch on import of index [ghcr.io/kubewarden/policy-server:v1.27.0]
2025-08-12 18:55:54 WRN specify an export platform to prevent potential platform mismatch on import of index [ghcr.io/kubewarden/audit-scanner:v1.27.0]

Pour éviter ce message d'avertissement, vous pouvez activer l'indicateur CLI `+--platform+` pour définir l'architecture de plate-forme que vous souhaitez enregistrer dans le fichier.

== Transférez l'archive dans votre environnement isolé

Maintenant que vous avez toutes les ressources {short-project-name} dans`+kubewarden-resources.tar.zst+`, copiez-les dans votre environnement à air comprimé et chargez-les dans le magasin Hauler :

[source,shell]
----
hauler store load --filename kubewarden-resources.tar.zst
# Check if the resources are loaded
hauler store info
----

Désormais, toutes les ressources nécessaires à l'installation de {short-project-name} se trouvent dans le magasin Hauler de votre environnement isolé.

== Alimentation du registre privé

Pour utiliser les ressources de votre magasin Hauler, il est nécessaire de les rendre disponibles dans un registre interne. Vous pouvez utiliser les commandes Hauler pour les copier dans votre registre privé.

[source,shell]
----
hauler store copy registry://localhost:5000
----

Vous pouvez également lancer Hauler pour démarrer un registre avec toutes les ressources du magasin. Ce registre n'est pas sécurisé et vous devez adapter la configuration du cluster :

[source,shell]
----
# Find IP address of your host
# hostname -I

# Update registries.yaml for k3s based cluster to allow insecure access
# mirrors:
#   "<HOST_IP>:5000":
#     endpoint:
#       - "http://<HOST_IP>:5000"
# configs:
#   "<HOST_IP>:5000":
#     tls:
#       insecure_skip_verify: true

# Configure policy-server to allow pulling policies from insecure sources
# helm install .. kubewarden-defaults .. --set policyServer.insecureSources[0]=<HOST_IP:5000>

hauler store serve registry
----

Cela permet de démarrer un registre à l'adresse `+localhost:5000+`. À partir de là, vous pouvez utiliser d'autres commandes comme Skopeo pour copier toutes les images de conteneurs, les modules de politique et les graphiques Helm utilisés par {short-project-name} dans votre registre privé.

== Installer {short-project-name}

Maintenant que votre registre privé a tout ce qu'il faut, vous pouvez installer{short-project-name}. La différence avec une installation standard de {short-project-name}est que vous devez changer le registre dans les images de conteneurs et les stratégies pour qu'il soit le registre privé. En outre, vous devez installer les cartes Helm à partir des artefacts OCI.

Installez la pile {short-project-name}:

[source,shell]
----
helm install --wait -n kubewarden kubewarden-crds --create-namespace \
    oci://<REGISTRY.YOURDOMAIN.COM:PORT>/hauler/kubewarden-crds
helm install --wait -n kubewarden kubewarden-controller \
    --set "global.cattle.systemDefaultRegistry=<REGISTRY.YOURDOMAIN.COM:PORT>" \
    oci://<REGISTRY.YOURDOMAIN.COM:PORT>/hauler/kubewarden-controller
----

[CAUTION]
====

Pour utiliser le sous-graphe PolicyReporter disponible dans le graphique `+kubewarden-controller+`, vous devez définir d'autres valeurs spécifiques pour le sous-graphe dans un environnement à air comprimé. Voir l'exemple ci-dessous :

[source,shell]
----
helm install --wait -n kubewarden kubewarden-controller oci://<REGISTRY.YOURDOMAIN.COM:PORT>/hauler/kubewarden-controller \
    --set global.cattle.systemDefaultRegistry=<REGISTRY.YOURDOMAIN.COM:PORT> \
    --set auditScanner.policyReporter=true \
    --set policy-reporter.image.registry=<REGISTRY.YOURDOMAIN.COM:PORT> \
    --set policy-reporter.image.repository=kyverno/policy-reporter \
    --set policy-reporter.ui.image.registry=<REGISTRY.YOURDOMAIN.COM:PORT> \
    --set policy-reporter.ui.image.repository=kyverno/policy-reporter-ui
----

Il est nécessaire de définir `+auditScanner.policyReporter+` et quatre autres valeurs pour activer le sous-graphe et configurer le registre et le référentiel avec l'emplacement du magasin d'images Policy Reporter. Pour plus d'informations sur les valeurs des sous-cartes des rapports de politique, consultez le lien suivant : \https://kyverno.github.io/policy-reporter-docs/getting-started/helm.html[Policy Reporter documentation].

====

[source,shell]
----
helm install --wait -n kubewarden \
  kubewarden-defaults oci://<REGISTRY.YOURDOMAIN.COM:PORT>/hauler/kubewarden-defaults \
  --set global.cattle.systemDefaultRegistry=<REGISTRY.YOURDOMAIN.COM:PORT>
----

Enfin, configurez chaque serveur de stratégies pour qu'il récupère les stratégies de votre registre privé. Voir la section de la documentation consacrée à xref:/howtos/policy-servers/02-private-registry.adoc[l'utilisation du registre privé].

Créez maintenant des politiques {short-project-name} dans votre cluster. Les politiques doivent être disponibles dans votre registre privé.

[source,yaml]
----
kubectl apply -f - <<EOF
apiVersion: policies.kubewarden.io/v1
kind: ClusterAdmissionPolicy
metadata:
  name: privileged-pods
spec:
  module: registry://<REGISTRY.YOURDOMAIN.COM:PORT>/kubewarden/policies/pod-privileged:v0.2.2
  rules:
  - apiGroups: [""]
    apiVersions: ["v1"]
    resources: ["pods"]
    operations:
    - CREATE
  mutating: false
EOF
----

[CAUTION]
====

`+PolicyServer+` doivent utiliser l'image disponible dans votre registre privé.
Par exemple :

[source,yaml]
----
apiVersion: policies.kubewarden.io/v1
kind: PolicyServer
metadata:
  name: reserved-instance-for-tenant-a
spec:
  image: <REGISTRY.YOURDOMAIN.COM:PORT>/kubewarden/policy-server:v1.3.0
  replicas:
----
