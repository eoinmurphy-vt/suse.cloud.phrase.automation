= Services de mise en réseau
:page-languages: [en, ja, ko, zh]
:revdate: 2026-01-05
:page-revdate: {revdate}

Cette page explique comment CoreDNS, le contrôleur Traefik Ingress, le contrôleur Network Policy et le contrôleur ServiceLB load balancer fonctionnent au sein de K3s.

Reportez-vous à la page xref:networking/basic-network-options.adoc[Options du réseau d'installation] pour plus de détails sur les options de configuration de Flannel et la sélection du backend, ou pour savoir comment configurer votre propre CNI.

Pour savoir quels ports doivent être ouverts pour les K3, reportez-vous à la section xref:installation/requirements.adoc#_networking[Exigences en matière de mise en réseau].

== CoreDNS

CoreDNS est déployé automatiquement au démarrage du serveur. Pour le désactiver, configurez tous les serveurs de la grappe avec l'option `+--disable=coredns+`.

Si vous n'installez pas CoreDNS, vous devrez installer vous-même un fournisseur de DNS en cluster.

== Contrôleur d'ingérence Traefik

https://traefik.io/[Traefik] est un proxy inverse HTTP moderne et un équilibreur de charge conçu pour déployer facilement des microservices. Il simplifie la mise en réseau lors de la conception, du déploiement et de l'exécution des applications.

Le contrôleur d'entrée Traefik déploie un service LoadBalancer qui utilise les ports 80 et 443, annonce les IP externes du service LoadBalancer dans le statut des ressources d'entrée qu'il gère.

Par défaut, ServiceLB utilisera tous les nœuds du cluster pour héberger le service Traefik LoadBalancer, ce qui signifie que les ports 80 et 443 ne seront pas utilisables par d'autres pods HostPort ou NodePort, et que l'état des ressources d'entrée affichera les IP des nœuds de tous les membres du cluster.

Pour restreindre les nœuds utilisés par Traefik, et par extension les IP de nœuds annoncées dans le statut d'entrée, vous pouvez suivre les instructions de la section <<_controlling_servicelb_node_selection,Contrôle de la sélection des nœuds Service>> LB ci-dessous pour limiter les nœuds sur lesquels ServiceLB s'exécute, ou en ajoutant certains nœuds à un pool LoadBalancer et en restreignant le service Traefik à ce pool en définissant des étiquettes correspondantes dans la configuration Traefik HelmChartConfig.

Traefik est déployé par défaut lors du démarrage du serveur. Les valeurs par défaut des graphiques se trouvent à l'adresse `+/var/lib/rancher/k3s/server/manifests/traefik.yaml+`, mais ce fichier ne doit pas être modifié manuellement, car K3s remplace le fichier par les valeurs par défaut au démarrage. Au lieu de cela, vous devez personnaliser Traefik en créant un manifeste `+HelmChartConfig+` supplémentaire dans `+/var/lib/rancher/k3s/server/manifests+`. Pour plus de détails et un exemple, voir xref:add-ons/helm.adoc#_customizing_packaged_components_with_helmchartconfig[Personnaliser les composants packagés avec HelmChartConfig]. Pour plus d'informations sur les valeurs de configuration possibles, reportez-vous à `+values.yaml+` du https://github.com/k3s-io/k3s-charts/tree/main/charts/traefik[tableau de bord] fourni avec votre version de K3s.

Pour supprimer Traefik de votre cluster, démarrez tous les serveurs avec l'option `+--disable=traefik+`. Pour plus d'informations, voir xref:installation/packaged-components.adoc[Gestion des composants packagés].

Pour plus de détails sur la version spécifique de Traefik incluse dans K3s, consultez les notes de mise à jour de votre version.

* Les versions de K3s à partir de la version **1.32** incluent Traefik v3. Les installations existantes de Traefik v2 sont automatiquement mises à jour vers la v3 lors de la mise à jour de K3s.
  Traefik v3 devrait être compatible avec la configuration de la v2 ; consultez les documents de v2 https://doc.traefik.io/traefik/migrate/v2-to-v3/[migration de la] pour plus d'informations.
* Les versions **1.21** à **1.31** de K3s incluent Traefik v2, sauf si une installation existante de Traefik v1 est trouvée, auquel cas Traefik n'a pas été mis à niveau vers v2.
* Les versions **1.20** et **antérieures** de K3s incluent Traefik v1.

== Contrôleur de politique réseau

K3s comprend un contrôleur de politique de réseau intégré. L'implémentation sous-jacente est https://github.com/cloudnativelabs/kube-router[ici.] kube-router's netpol controller library (aucune autre fonctionnalité de kube-router n'est présente) et peut être trouvée https://github.com/k3s-io/k3s/tree/main/pkg/agent/netpol[]

Pour le désactiver, démarrez chaque serveur avec l'option `+--disable-network-policy+`.

[NOTE]
====
Les règles iptables de la politique de réseau ne sont pas supprimées si la configuration de K3s est modifiée pour désactiver le contrôleur de politique de réseau. Pour nettoyer les règles de stratégie réseau kube-router configurées après la désactivation du contrôleur de stratégie réseau, utilisez le script `+k3s-killall.sh+` ou nettoyez-les à l'aide des scripts `+iptables-save+` et `+iptables-restore+`. Ces étapes doivent être exécutées manuellement sur tous les nœuds du cluster.

----
iptables-save | grep -v KUBE-ROUTER | iptables-restore
ip6tables-save | grep -v KUBE-ROUTER | ip6tables-restore
----
====

== Service Load Balancer

N'importe quel contrôleur LoadBalancer peut être déployé dans votre cluster K3s. Par défaut, K3s fournit un équilibreur de charge connu sous le nom de https://github.com/k3s-io/klipper-lb[ServiceLB] (anciennement Klipper LoadBalancer) qui utilise les ports hôtes disponibles.

Upstream Kubernetes permet de créer des services de type LoadBalancer, mais n'inclut pas d'implémentation de load balancer par défaut, donc ces services resteront `+pending+` jusqu'à ce qu'un soit installé. De nombreux services hébergés nécessitent qu'un fournisseur de services en nuage tel qu'Amazon EC2 ou Microsoft Azure propose une implémentation d'équilibreur de charge externe. En revanche, le ServiceLB de K3s permet d'utiliser les services LoadBalancer sans fournisseur de nuage ni configuration supplémentaire.

=== Fonctionnement de ServiceLB

Le contrôleur ServiceLB surveille les Kubernetes https://kubernetes.io/docs/concepts/services-networking/service/[services] dont le champ `+spec.type+` est défini sur `+LoadBalancer+`.

Pour chaque service LoadBalancer, un link:https://kubernetes.io/docs/concepts/workloads/controllers/daemonset/[DaemonSet] est créé dans l'espace de noms `+kube-system+`. Ce DaemonSet crée à son tour des Pods ServiceLB avec un préfixe `+svc-+`, sur chaque nœud. Ces pods s'appuient sur hostPort en utilisant le port de service, ils ne seront donc déployés que sur des nœuds qui ont ce port disponible. S'il n'y a pas de nœuds avec ce port disponible, le LB restera en attente. Notez qu'il est possible d'exposer plusieurs services sur le même nœud, à condition qu'ils utilisent des ports différents.

Lorsque le module ServiceLB s'exécute sur un nœud pour lequel une IP externe a été configurée, l'IP externe du nœud est ajoutée à la liste d'adresses `+status.loadBalancer.ingress+` du service à l'aide de `+ipMode: VIP+`. Dans le cas contraire, l'IP interne du nœud est utilisée.

Si le trafic vers l'IP externe est soumis à link:https://en.wikipedia.org/wiki/Network_address_translation[Network Address Translation (NAT)] - par exemple dans les nuages publics lors de l'utilisation de l'IP publique du nœud comme IP externe - le trafic est acheminé dans le pod ServiceLB via le hostPort. Le pod utilise ensuite iptables pour transférer le trafic vers l'adresse et le port ClusterIP du service. Si le trafic n'est pas soumis à la NAT et arrive avec une adresse de destination correspondant à l'adresse du LoadBalancer, le trafic est intercepté (normalement par les chaînes iptables ou ipvs de kube-proxy) et transféré vers l'adresse et le port ClusterIP du service.

=== Utilisation

Créer un https://kubernetes.io/docs/concepts/services-networking/service/#loadbalancer[service] dans K3s.

[WARNING]
.Problème connu
====
Si le trafic externe atteint le nœud en utilisant un NAT (par exemple dans les nuages publics) et que vous avez besoin de `+externalTrafficPolicy=local+` à des fins telles que la préservation de l'IP source du client, veuillez ne pas définir la configuration k3s `+node-external-ip+` pour l'un des nœuds, car cela ne fonctionnera pas correctement.
====

=== Contrôle de la sélection des nœuds ServiceLB

L'ajout du label `+svccontroller.k3s.cattle.io/enablelb=true+` à un ou plusieurs nœuds fait passer le contrôleur ServiceLB en mode liste d'autorisation, où seuls les nœuds portant le label peuvent héberger des pods LoadBalancer. Les nœuds non étiquetés seront exclus de l'utilisation par le ServiceLB.

[NOTE]
====
Par défaut, les nœuds ne sont pas étiquetés. Tant que tous les nœuds ne sont pas étiquetés, tous les nœuds ayant des ports disponibles seront utilisés par ServiceLB.
====


=== Création de pools de nœuds ServiceLB

Pour sélectionner un sous-ensemble particulier de nœuds afin d'héberger des pods pour un LoadBalancer, ajoutez le label `+enablelb+` aux nœuds souhaités et définissez des valeurs de label `+lbpool+` correspondantes sur les nœuds et les services. Par exemple :

. Etiqueter le nœud A et le nœud B avec `+svccontroller.k3s.cattle.io/lbpool=pool1+` et `+svccontroller.k3s.cattle.io/enablelb=true+`
. Etiqueter le nœud C et le nœud D avec `+svccontroller.k3s.cattle.io/lbpool=pool2+` et `+svccontroller.k3s.cattle.io/enablelb=true+`
. Créer un service LoadBalancer sur le port 443 avec le label `+svccontroller.k3s.cattle.io/lbpool=pool1+`. Le DaemonSet de ce service ne déploie que des Pods vers le Node A et le Node B.
. Créer un autre service LoadBalancer sur le port 443 avec le label `+svccontroller.k3s.cattle.io/lbpool=pool2+`. Le DaemonSet ne déploiera des pods que sur les nœuds C et D.

=== Désactivation de ServiceLB

Pour désactiver ServiceLB, configurez tous les serveurs de la grappe avec l'option `+--disable=servicelb+`.

Ceci est nécessaire si vous souhaitez utiliser un autre LB, tel que MetalLB.

== Déploiement d'un gestionnaire de contrôleur de nuage externe

K3s fournit un gestionnaire de contrôleur de nuage (CCM) intégré qui effectue les opérations suivantes :

* Héberge le contrôleur <<_service_load_balancer, Service Load Balancer>> LoadBalancer.
* Efface la souillure de `+node.cloudprovider.kubernetes.io/uninitialized+`.
* Définit les champs d'adresse du nœud en fonction des drapeaux `+--node-ip+`, `+--node-external-ip+`, `+--node-internal-dns+`, et `+--node-external-dns+`.

Avant de déployer un CCM externe, vous devez démarrer tous les serveurs K3s avec l'option `+--disable-cloud-controller+` pour désactiver le CCM intégré. Lors de l'utilisation d'un CCM externe, les adresses des nœuds seront fournies par les API de métadonnées des instances du fournisseur de cloud, au lieu des valeurs des drapeaux K3s.

[NOTE]
====
Si vous désactivez le CCM intégré et que vous ne déployez pas et ne configurez pas correctement un substitut externe, les nœuds resteront entachés et irrécupérables.
====
