== {elemental-product-name} aperçu de la gestion des noms d'hôtes

Lorsqu'un hôte démarre à partir de l'ISO {elemental-product-name}, le nom d'hôte est temporairement défini_(nom d'hôte transitoire_) sur celui fourni par le serveur DHCP.
Si le serveur DHCP ne fournit pas de nom d'hôte, l'ISO {elemental-product-name} fournit un nom d'hôte transitoire sous la forme : `+rancher-$\{RANDOM}+`.

Dès que le processus de démarrage est terminé, la phase d'enregistrement commence : l'hôte se connecte à l'opérateur {elemental-product-name}, qui crée un `+MachineInventory+` pour l'hôte.
Chaque hôte enregistré auprès de l'opérateur {elemental-product-name} est suivi par une ressource `+MachineInventory+`.

*Le site `+name+` de la ressource `+MachineInventory+` associée au nœud est le nom d'hôte permanent (statique**) qui sera attribué à l'hôte*.
Ce nom d'hôte permanent *est défini sur le nœud uniquement pendant la phase de provisionnement du cluster K8s*.
Avant la phase de provisionnement du K8s, le nom d'hôte du nœud est soit celui attribué par DHCP, soit `+rancher-$\{RANDOM}+`.

Dans la suite de cette section, nous ferons référence à `+hostname+` pour désigner le nom d'hôte _permanent_, c'est-à-dire le nom d'hôte défini après que l'hôte ait été provisionné comme faisant partie d'un cluster K8s.

== Nom d'hôte par défaut

Le nom par défaut attribué à chaque MachineInventory nouvellement créé est de la forme `+m-{$UUID}+`.
Lorsque l'hôte est provisionné en tant que partie d'un cluster, ce nom `+m-\{UUID}+` est défini comme le nom d'hôte de l'hôte correspondant, remplaçant le nom d'hôte attribué précédemment (`+rancher-{$RANDOM}+` ou celui attribué par DHCP).

== Définir un nom d'hôte personnalisé

Le nom d'hôte peut être spécifié en définissant le champ xref:references/machineregistration-reference.adoc#_machinename[`+machineName+`] dans la ressourcexref:references/machineregistration-reference.adoc["MachineRegistration"].

Le nom d'hôte défini dans le champ `+machineName+` est censé se présenter sous la forme d'un modèle, afin d'être généré de manière unique pour chaque nœud d'enregistrement, à l'aide des données SMBIOS et Hardware Labels.

[IMPORTANT]
====
Le champ `+machineName+` de la ressource `+MachineRegistration+` est utilisé comme modèle non seulement pour le nom d'hôte de l'hôte qui s'enregistre, mais aussi pour le nom de la ressource `+MachineInventory+` créée pour suivre l'hôte.

Cela signifie que si vous n'utilisez pas un modèle `+machineName+` tel que pour générer un nom unique pour chaque hôte qui démarrera en utilisant les mêmes données `+MachineRegistration+` (c'est-à-dire la même ISO), seul le premier hôte qui s'enregistre réussira, tandis que les autres échoueront : le nom `+MachineInventory+` doit être unique.
====

== Conserver le nom d'hôte attribué par DHCP

Afin de conserver le nom d'hôte attribué par le serveur DHCP avant que l'hôte ne s'enregistre auprès de l'opérateur, l'adresse `+MachineRegistration+` xref:references/machineregistration-reference.adoc#_machinename[`+machineName field+`] doit être réglé sur l'étiquette matérielle `+${System Data/Runtime/Hostname}+`.

De cette façon, {elemental-product-name} utilisera le nom d'hôte actuel comme nom `+MachineInventory+` pendant la phase d'enregistrement, qui sera ensuite défini comme nom d'hôte statique de l'hôte pendant la phase de provisionnement.
