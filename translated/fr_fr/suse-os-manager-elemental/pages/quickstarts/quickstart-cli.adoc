== Préparez vos ressources kubernetes

Le déploiement des nœuds commence par un `+MachineRegistration+`, qui identifie un ensemble de machines partageant la même configuration (disques, réseau, etc.).

Le site `+MachineRegistration+` est nécessaire pour effectuer le déploiement du système d'exploitation {elemental-product-name} sur les hôtes cibles. Au démarrage, chaque hôte s'enregistre auprès de l'opérateur {elemental-product-name} qui suit le nouvel hôte à l'aide d'une ressource `+MachineInventory+`.

Ensuite, il y a une ressource Cluster qui utilise `+MachineInventorySelectorTemplate+` pour savoir quelles machines font partie de ce cluster.

Ce sélecteur est une simple correspondance basée sur les étiquettes définies dans `+MachineInventory+`, donc si votre sélecteur correspond à l'étiquette `+cluster-id+` avec une valeur `+cluster-id-val+`et que votre `+MachineInventory+` a la même étiquette `+cluster-id+`:``+cluster-id-val+``, il correspondra et sera démarré comme faisant partie du cluster.

Dans ce quickstart, nous allons déployer les ressources pour provisionner un cluster nommé _volcano_ qui correspondra à ``+MachineInventory+s``avec le label _element___:fire.__

== Préparation de l'image d'installation (semence)

C'est la dernière étape : vous avez besoin d'une image de départ {elemental-product-name} qui inclut la configuration d'enregistrement initiale, de sorte qu'elle puisse être enregistrée automatiquement, installée et entièrement déployée dans le cadre de votre cluster.

[NOTE]
.note
====
Le fichier de configuration de l'enregistrement initial est généré lorsque vous créez un site `+Machine Registration+`.

Vous pouvez le télécharger avec :

[,shell]
----
wget --no-check-certificate `+kubectl get machineregistration -n fleet-default fire-nodes -o jsonpath="{.status.registrationURL}"+` -O initial-registration.yaml
----
====

Le contenu du fichier de configuration de l'enregistrement n'est rien d'autre que l'URL d'enregistrement dont le nœud a besoin pour s'enregistrer, le certificat de serveur approprié et quelques options pour le processus d'enregistrement.

Une fois générée, une image de départ peut être utilisée pour approvisionner n'importe quel nombre de machines.

[tabs]
========
Téléchargement de l'ISO de démarrage rapide::
+
======
L'image de base créée par la ressource `+SeedImage+` ci-dessus peut être téléchargée en tant qu'ISO via le script suivant : 

[,shell]
----
kubectl wait --for=condition=ready pod -n fleet-default fire-img 
wget --no-check-certificate `+kubectl get seedimage -n fleet-default fire-img -o jsonpath="{.status.downloadURL}"+` -O elemental.x86_64.iso 
----

La première commande attend que l'ISO soit construite et prête, la seconde la télécharge dans le répertoire courant sous le nom `+elemental-x86_64.iso+`. 
======

Préparation manuelle de l'image de base (aarch64)::
+
======
{elemental-product-name}La prise en charge du Raspberry Pi par la Commission européenne n'est pour l'instant qu'une simple démonstration. Par conséquent, le processus d'installation est modélisé de la même manière que pour x86-64. Vous démarrez à partir d'une image de départ (une clé USB dans ce cas) et installez sur un support de stockage (carte SD pour Raspberry Pi). 

*Récupération de l'image de base préconstruite*

[,shell]
----
wget -q https://download.opensuse.org/repositories/isv:/Rancher:/SUSE® Rancher Prime OS Manager:/Stable/containers/rpi.raw
----

_Vérification du téléchargement_ 

Afin de vérifier l'intégrité des artefacts téléchargés, vous devez effectuer une vérification de la somme de contrôle : 

[,shell]
----
wget -q https://download.opensuse.org/repositories/isv:/Rancher:/SUSE® Rancher Prime OS Manager:/Stable/containers/rpi.raw.sha256 sha256sum -c rpi.raw.sha256
----

Cette opération devrait permettre d'imprimer `+rpi.raw: OK+` en sortie. 

*Injection des informations d'enregistrement*

L'ajout du site `+initial-registration.yaml+` n'est pas encore programmé. Il s'agit toujours d'un processus manuel : 

La clé USB écrite aura deux partitions. `+RPI_BOOT+` contient les fichiers du chargeur de démarrage et `+COS_LIVE+` les fichiers de {elemental-product-name}. Montez la partition `+COS_LIVE+` et écrivez `+initial-registration.yaml+` comme `+livecd-cloud-config.yaml+` sur cette partition. 

Si vous avez monté la clé USB avec un gestionnaire de fichiers, cette commande devrait permettre de copier les informations d'enregistrement : 

[,shell]
----
sudo cp initial-registration.yaml /run/media/$USER/COS_LIVE/livecd-cloud-config.yaml
----

Si vous préférez utiliser des outils CLI : 

[,shell]
----
IMAGE=rpi.raw 
DEST=$(mktemp -d) 

SECTORSIZE=$(sfdisk -J $\{IMAGE} | jq '.partitiontable.sectorsize') 
DATAPARTITIONSTART=$(sfdisk -J $\{IMAGE} | jq '.partitiontable.partitions[1].start') 
sudo mount -o rw,loop,offset=$(($\{SECTORSIZE}*$\{DATAPARTITIONSTART})) $\{IMAGE} $\{DEST} 
sudo cp initial-registration.yaml $\{DEST}/livecd-cloud-config.yaml 
sudo umount $\{DEST} 
rmdir $\{DEST}
----

*Écriture de l'image de départ sur une clé USB*

L'image `+.raw+` doit être enregistrée sur une clé USB pour pouvoir démarrer. Cette opération peut être réalisée à l'aide de `+dd+` sur la ligne de commande Linux si vous êtes à l'aise avec cette commande. Le site [openSUSE](https://www.opensuse.org) contient de bonnes instructions sur la manière d'écrire une image sur un support de stockage pour [Linux](https://en.opensuse.org/SDB:Live_USB_stick), [Windows](https://en.opensuse.org/SDB:Create_a_Live_USB_stick_using_Windows), et [OS X](https://en.opensuse.org/SDB:Create_a_Live_USB_stick_using_macOS). 

*Démarrer le Raspberry Pi*

Démontez la clé USB et branchez-la sur votre Raspberry Pi. 

Branchez une carte micro SD de grande taille (32 Go ou plus) et **rapide** ( !!) dans l'emplacement prévu à cet effet. 

Connecter le système à ethernet. 

Un cycle d'alimentation redémarre le Pi. Tout le reste est identique à x86-64. 

[CAUTION]
====
Assurez-vous que la carte micro SD n'est pas partitionnée. Sinon, le chargeur de démarrage du Pi essaiera de démarrer à partir de ce fichier et échouera.
====

Vous pouvez maintenant démarrer vos nœuds avec cette image et ils le feront :

* S'enregistrer à l'aide de l'URL d'enregistrement indiquée et créer un fichier `+MachineInventory+`
* Installer SLE Micro sur l'appareil donné
* Reboot
======
========

=== Sélectionner les bonnes machines pour rejoindre un cluster

Le site `+MachineInventorySelectorTemplate+` sélectionne les machines nécessaires à l'approvisionnement du cluster à partir des ``+MachineInventory+s``ayant le label _element:fire_.
Nous avons ajouté le label _element___:fire__ dans la carte `+MachineRegistration+` `+machineInventoryLabels+`, donc tous les ``+MachineInventory+``qui en sont issus ont déjà le label.
De toute façon, il est possible d'ignorer l'étiquette sur le site `+MachineRegistration+` et de l'ajouter plus tard :

[,shell]
----
kubectl -n fleet-default label machineinventory $(kubectl get machineinventory -n fleet-default --no-headers -o custom-columns=":metadata.name") element=fire
----

Dès que des ``+MachineInventory+s``avec l'_élément___:fire__ sont présents, les machines correspondantes déploient automatiquement le cluster via le fournisseur choisi (k3s/rke).

Après quelques minutes, votre nouveau cluster sera entièrement provisionné !

== Comment puis-je choisir la version de kubernetes et le déployeur pour le cluster ?

Dans votre fichier cluster.yaml, il y a une clé dans `+Spec+` appelée `+kubernetesVersion+`. Cela définit la version et le déployeur qui seront utilisés pour le cluster, par exemple Kubernetes+v1``.24.8++`` pour rke2 serait `v1 +.24.8+rke2r1`+ et pour k3s +v1`.24.8+k3s1`.

Pour connaître toutes les versions compatibles, consultez le https://www.suse.com/suse-rancher/support-matrix/all-supported-versions/[site] Rancher PDF pour les versions rke/rke2/k3s et leurs composants.

Vous pouvez également consulter notre xref:references/kubernetesversions.adoc[documentation sur les] versions pour savoir comment obtenir ces versions.

Consultez notre page xref:references/cluster-reference.adoc[Cluster Spec] pour plus d'informations sur la ressource `+Cluster+`.

== Comment puis-je suivre ce qui se passe en coulisses ?

Vous devriez pouvoir suivre ce que fait la machine via :

* Pendant le démarrage ISO :
 ** se connecter en ssh à la machine (user/pass : root/ros) :
  *** L'exécution de `+journalctl -f -t elemental+` vous montre la progression de l'enregistrement_(elemental-register_) et de l'installation de {elemental-product-name} _(elemental install_).
* Une fois le système installé :
 ** Sur l'interface utilisateur de Rancher \-> `+Cluster Management+` vous permet de voir votre nouveau cluster et le `+Provisioning Log+` dans les détails du cluster.
 ** ssh sur la machine (user/pass : Ce que vous avez configuré dans le fichier registration.yaml sous `+Spec.config.cloud-config.users+`) :
  *** Le site `+journalctl -f -u elemental-system-agent+` montre la sortie de la configuration élémentaire initiale et l'installation de l'application `+rancher-system-agent+`
  *** L'exécution de `+journalctl -f -u rancher-system-agent+` montre la sortie du bootstrap des composants de la grappe comme k3s
  *** L'exécution de `+journalctl -f -u k3s+` montre les journaux du déploiement de k3s.

== En option : Installer l'extension Elemental UI via CLI

Créer et appliquer un nouveau ClusterRepo pour ajouter le dépôt officiel de Rancher Extensions.

[source,yaml]
----
apiVersion: catalog.cattle.io/v1
kind: ClusterRepo
metadata:
  name: rancher-ui-charts
spec:
  gitBranch: main
  gitRepo: https://github.com/rancher/ui-plugin-charts
----

Ajout d'un repo helm pour les cartes d'extensions de l'interface utilisateur Rancher.

[source,shell]
----
helm repo add rancher-ui-plugins https://raw.githubusercontent.com/rancher/ui-plugin-charts/main/
----

Installer l'extension Elemental UI.

[source,shell]
----
helm install elemental rancher-ui-plugins/elemental -n cattle-ui-plugin-system
----
