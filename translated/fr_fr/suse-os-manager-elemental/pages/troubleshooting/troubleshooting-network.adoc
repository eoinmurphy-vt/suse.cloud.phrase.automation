= Dépannage des réseaux déclaratifs
:revdate: 2025-09-08
:page-revdate: {revdate}

Compte tenu de l'exemple d'enregistrement suivant :

.exemple de MachineRegistration utilisant Declarative Networking
[,yaml]
----
include::example$network/machineregistration.yaml[]
----

On peut s'attendre à ce que chaque machine élémentaire soit configurée à l'aide du modèle `+nm-configurator+` `+_all.yaml+` défini.

Au tout premier démarrage, le site `+elemental-register+` essaiera de contacter l'API Rancher pour enregistrer un nouveau site `+MachineInventory+`.

À ce stade, le réseau de la machine n'est pas configuré et sera réglé par défaut sur DHCP. Il est nécessaire que la machine puisse contacter l'API Rancher dans cette configuration, sinon l'enregistrement ne peut pas avoir lieu.

Une fois que le site `+MachineInventory+` a été enregistré, le site `+elemental-operator+` va réclamer un site `+IPAddress+` pour chaque référence `+IPPool+` définie dans la configuration du réseau.

Sur le site `+MachineInventory+`, cela ressemblera à ce qui suit :

[,yaml]
----
apiVersion: elemental.cattle.io/v1beta1
kind: MachineInventory
metadata:
  finalizers:
  - machineinventory.elemental.cattle.io
  name: m-e5331e3b-1e1b-4ce7-b080-235ed9a6d07c
  namespace: fleet-default
spec:
  ipAddressClaims:
    inventory-ip:
      apiVersion: ipam.cluster.x-k8s.io/v1beta1
      kind: IPAddressClaim
      name: m-e5331e3b-1e1b-4ce7-b080-235ed9a6d07c-inventory-ip
      namespace: fleet-default
      uid: 78f2d07a-7b6d-4b58-b615-c4108b7964b9
  ipAddressPools:
    inventory-ip:
      apiGroup: ipam.cluster.x-k8s.io
      kind: InClusterIPPool
      name: elemental-inventory-pool
  network:
    config:
      dns-resolver:
        config:
          search: []
          server:
          - 192.168.122.1
      interfaces:
      - description: Main-NIC
        ipv4:
          address:
          - ip: "{inventory-ip}"
            prefix-length: 24
          dhcp: false
          enabled: true
        ipv6:
          enabled: false
        name: eth0
        state: up
        type: ethernet
      routes:
        config:
        - destination: 0.0.0.0/0
          metric: 150
          next-hop-address: 192.168.122.1
          next-hop-interface: eth0
          table-id: 254
    ipAddresses:
      inventory-ip: 192.168.122.150
status:
  conditions:
  - lastTransitionTime: "2024-07-30T11:50:47Z"
    message: NetworkConfig is ready
    reason: ReconcilingNetworkConfig
    status: "True"
    type: NetworkConfigReady
----

Vous remarquerez que le site `+MachineInventory+` contient le même site `+network.config+` que le site `+MachineRegistration+`, mais qu'au lieu de faire référence aux IPAddressPools, nous disposons maintenant d'une carte des adresses IPA réelles :

[,yaml]
----
    ipAddresses:
      inventory-ip: 192.168.122.150
----

Cette adresse `+inventory-ip+` sera ensuite remplacée dans la configuration de `+nm-configurator+` lorsque `+{inventory-ip}+` aura été défini.

Notez également que le site `+MachineInventory+` fait référence à chaque site `+IPAddressClaim+` qui lui est associé et en est propriétaire. Chaque demande suit la convention de dénomination prévisible `+$MachineInventoryName-$IPPoolRefKey+`: `+m-e5331e3b-1e1b-4ce7-b080-235ed9a6d07c-inventory-ip+`.

Ces créances suivront le cycle de vie de l'objet `+MachineInventory+` et seront supprimées en cascade, par exemple au cours du xref:node-operational-tasks/reset.adoc[processus de réinitialisation].

Si l'adresse `+IPAddresses+` ne peut pas être revendiquée, la condition `+NetworkConfigReady+` sera `+False+`, ce qui empêchera la machine de terminer l'installation. Cela peut être le cas si le site `+IPPool+` n'a plus de `+IPAddresses+` disponible.

== Du côté de la machine

Pendant la phase d'installation, le processus `+elemental-register+` fonctionnant sur la machine recevra le modèle de configuration `+nm-configurator+` `+_all.yaml+` et la liste des adresses IPA revendiquées avec leurs clés. Ces informations seront transformées en une configuration applicable sur le site `+nm-configurator+`:

[,yaml]
----
config:
  dns-resolver:
    config:
      search: []
      server:
      - 192.168.122.1
  interfaces:
  - description: Main-NIC
    ipv4:
      address:
      - ip: "192.168.122.150"
        prefix-length: 24
      dhcp: false
      enabled: true
    ipv6:
      enabled: false
    name: eth0
    state: up
    type: ethernet
  routes:
    config:
    - destination: 0.0.0.0/0
      metric: 150
      next-hop-address: 192.168.122.1
      next-hop-interface: eth0
      table-id: 254
----

Le site `+elemental-register+` invoquera ensuite `+nmc generate+` et `+nmc apply+` pour appliquer cette configuration au système en cours d'exécution.

À partir de ce moment et jusqu'à la réinitialisation, la machine utilisera toujours la configuration appliquée.

Notez également qu'en dehors de l'installation et de la réinitialisation, `+nm-configurator+` n'est plus utilisé, puisque `+elemental-register+` conservera les fichiers `+/etc/NetworkManager/system-connection/*.nmconnection+` générés par `+nmc+` plutôt que la configuration `+nmc+` elle-même.

Par exemple, sur tout système en cours d'exécution, vous trouverez un fichier de configuration https://github.com/rancher/yip[] yip (`+/oem/elemental-network.yaml+`) pour appliquer les paramètres souhaités `+nmconnections+`, par exemple :

[,yaml]
----
name: Apply network config
stages:
    initramfs:
        - files:
            - path: /etc/NetworkManager/system-connections/Wired connection 1.nmconnection
              permissions: 384
              owner: 0
              group: 0
              content: |
                [connection]
                id=Wired connection 1
                uuid=d26b4ae4-d525-3cbf-a557-33feb60343c0
                type=ethernet
                autoconnect-priority=-999
                interface-name=eth0
                timestamp=1722340245

                [ethernet]

                [ipv4]
                address1=192.168.122.150/24
                dhcp-timeout=2147483647
                dns=192.168.122.1;
                dns-options=
                dns-priority=40
                method=manual
                route1=0.0.0.0/0,192.168.122.1,150
                route1_options=table=254

                [ipv6]
                addr-gen-mode=eui64
                dhcp-timeout=2147483647
                method=disabled

                [proxy]

                [user]
                nm-configurator.interface.description=Main-NIC
              encoding: ""
              ownerstring: ""
----

=== Pendant la réinitialisation

Lorsque la xref:node-operational-tasks/reset.adoc[réinitialisation] est déclenchée, le site `+elemental-register+` fonctionnant sur la machine efface tout fichier `+/etc/NetworkManager/system-connection/*.nmconnection+` et redémarre la pile réseau. La machine devrait alors revenir à DHCP et confirmer ensuite à `+elemental-operator+`, du côté de l'administration, que la réinitialisation a réussi.

Notez que cela ne s'applique que lorsque la valeur de `+MachineInventory.spec.network.configurator+` est différente de celle de `+none+`. Dans le cas contraire, aucune action ne sera entreprise pour réinitialiser le réseau pendant la phase de réinitialisation de la machine.

Après la réinitialisation du réseau, la machine doit redémarrer en mode de récupération, effectuer la réinitialisation proprement dite et recevoir une nouvelle configuration réseau à appliquer. Il peut s'agir de la même adresse qu'auparavant (si le site `+MachineRegistration+` n'a pas été mis à jour), ou d'adresses IP différentes, les précédentes ayant été revendiquées par d'autres machines entre-temps.

Si le retour au DHCP a échoué ou si la machine est incapable de contacter l'API Rancher pour obtenir une confirmation, vous remarquerez que le site `+MachineInventory+` ne sera pas supprimé, bien qu'il ait un horodatage de suppression. Comme la machine a maintenant un problème de réseau, il ne sera pas possible de la réparer à distance.

Vous avez la possibilité d'accéder physiquement à la machine ou de corriger par n'importe quel moyen la configuration réseau pilotée par DHCP, ou encore de retirer le finalisateur `+machineinventory.elemental.cattle.io+` du site `+MachineInventory+`, afin de permettre sa suppression, si vous avez l'intention de mettre la machine hors service.
