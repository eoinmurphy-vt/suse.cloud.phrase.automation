== Servidor do SUSE Observability

=== Expondo o SUSE Observability Server com certificados autoassinados

Os componentes xref:k8s-suse-rancher-prime.adoc#_installation[do SUSE Observability Server] não podem encerrar o SSL por conta própria e devem usar um Ingress Controller ou LoadBalancer protegido por SSL para expor o serviço com HTTPS. Esta seção explica como configurar um Controlador Ingress com certificados autoassinados.

==== Uso do Ingress Controller com TLS

Para expor o SUSE Observability Server usando o https://kubernetes.github.io/ingress-nginx/[Ingress Controller] Nginx com certificados autoassinados, você precisa:

1. Crie segredos TLS do Kubernetes a partir do certificado e da chave de seu servidor
2. Configurar o recurso Ingress com as definições de TLS
3. Implantar o SUSE Observability Server com o Ingress ativado

===== Criação de segredos TLS

Crie segredos TLS do Kubernetes a partir de seus arquivos de certificado e chave privada:

[,bash]
----
# Create TLS secret for main SUSE Observability Server
kubectl create secret tls tls-secret \
  --cert=server.crt \
  --key=server.key \
  --namespace suse-observability

# Create TLS secret for OTLP GRPC endpoint
kubectl create secret tls otlp-tls-secret \
  --cert=otlp-server.crt \
  --key=otlp-server.key \
  --namespace suse-observability

# Create TLS secret for OTLP HTTP endpoint
kubectl create secret tls otlp-http-tls-secret \
  --cert=otlp-http-server.crt \
  --key=otlp-http-server.key \
  --namespace suse-observability
----

===== Configuração dos valores do leme

Crie um arquivo de valores (por exemplo, `ingress-with-tls-values.yaml`) com a configuração do Ingress:

[NOTE]
====
*Observação:*
Essa configuração é específica para o https://kubernetes.github.io/ingress-nginx/[Ingress Controller] Nginx. Se você estiver usando um controlador Ingress diferente, ajuste as anotações e a configuração de acordo.
====

[,yaml]
----
ingress:
  enabled: true
  ingressClassName: nginx
  annotations:
    nginx.ingress.kubernetes.io/proxy-body-size: "50m"
  hosts:
    - host: suse-observability.MY_DOMAIN
  tls:
    - hosts:
        - suse-observability.MY_DOMAIN
      secretName: tls-secret

opentelemetry-collector:
  ingress:
    enabled: true
    ingressClassName: nginx
    annotations:
      nginx.ingress.kubernetes.io/proxy-body-size: "50m"
      nginx.ingress.kubernetes.io/backend-protocol: GRPC
    hosts:
      - host: otlp-suse-observability.MY_DOMAIN
        paths:
          - path: /
            pathType: Prefix
            port: 4317
    tls:
      - hosts:
          - otlp-suse-observability.MY_DOMAIN
        secretName: otlp-tls-secret
    additionalIngresses:
      - name: otlp-http
        annotations:
          nginx.ingress.kubernetes.io/proxy-body-size: "50m"
        hosts:
          - host: otlp-http-suse-observability.MY_DOMAIN
            paths:
              - path: /
                pathType: Prefix
                port: 4318
        tls:
          - hosts:
              - otlp-http-suse-observability.MY_DOMAIN
            secretName: otlp-http-tls-secret
----

===== Instalação com Ingress

Implante o SUSE Observability Server usando a configuração Ingress:

[,bash]
----
helm upgrade --install \
  --namespace suse-observability \
  --create-namespace \
  --values ingress-with-tls-values.yaml \
  suse-observability \
  suse-observability/suse-observability
----

Para obter instruções completas de instalação, consulte xref:k8s-suse-rancher-prime.adoc#_installation[Instalação].

=== SUSE Observability Server conectando-se a sistemas externos com certificados autoassinados

[NOTE]
====
Esta seção serve para configurar o SUSE Observability Server para confiar em sistemas externos que usam certificados autoassinados. Isso se aplica quando o servidor precisa fazer conexões de saída com serviços externos (como webhooks) que são protegidos por certificados autoassinados.
====

O SUSE Observability Server tem vários pontos de interação com sistemas externos. Por exemplo, os manipuladores de eventos podem fazer chamadas para webhooks em outros sistemas. Com a configuração padrão, o SUSE Observability Server não conseguirá se comunicar com esses sistemas se eles estiverem protegidos com TLS usando um certificado autoassinado ou um certificado que não seja confiável por padrão pela JVM.

Para atenuar isso, o SUSE Observability Server permite a configuração de um armazenamento de confiança personalizado.

=== Criar um trust store personalizado

Você precisa ter o certificado TLS personalizado disponível. Se não tiver isso, você precisará xref:/use/security/self-signed-certificates.adoc#_retrieve_certificate_via_the_browser[recuperá-lo por meio do navegador].

Use a ferramenta keytool e o arquivo `cacerts` incluídos na instalação da JVM (Java Virtual Machine) para converter um arquivo de certificado TLS existente no formato necessário para o SUSE Observability Server. Você pode executá-lo em qualquer máquina, independentemente do tipo de sistema operacional.

Se você não tiver a JVM instalada em seu computador, também poderá xref:/use/security/self-signed-certificates.adoc#_using_a_docker_jvm[usar uma imagem do Docker da JVM].

==== Usando uma JVM instalada

Com a JVM instalada em seu computador e o certificado salvo como um arquivo `site.cert`, você pode criar um novo repositório de confiança usando o repositório de confiança da JVM e adicionando o certificado extra.

. Crie um diretório de trabalho `workdir` e copie o arquivo de certificado `site.cert` para esse diretório.
. Mude o diretório para `workdir` e faça uma cópia do arquivo `cacerts` de sua instalação do Java. `$JAVA_HOME` é uma variável de ambiente que contém o local de sua instalação do Java. Normalmente, isso é definido durante a instalação do Java.
+
[,bash]
----
cd workdir
cp $JAVA_HOME/lib/security/cacerts ./custom_cacerts
----

. Execute o seguinte comando keytool para adicionar o certificado. A senha necessária é `changeit`. O alias precisa ser um alias exclusivo para o certificado, por exemplo, o próprio nome de domínio sem nenhum ponto.
+
[,bash]
----
keytool -import -keystore custom_cacerts -alias <a-name-for-the-certificate>  -file site.cert
----

. O arquivo de armazenamento `custom_cacerts` agora incluirá o certificado `site.cert`. Você pode verificar isso procurando o alias na saída do
+
[,bash]
----
keytool -list -keystore custom_cacerts
----

==== Usando uma JVM do Docker

Se você não tiver a JVM instalada em seu computador, poderá usar uma imagem do Docker da JVM. O certificado deve ser recuperado e salvo em um arquivo `site.cert`.

. Crie um diretório de trabalho `workdir` e copie o arquivo de certificado `site.cert` para esse diretório.
. Inicie o contêiner do Java Docker com o `workdir` montado como um volume para que possa ser acessado:
+
[,bash]
----
docker run -it -v `+pwd+`/workdir:/workdir  adoptopenjdk:11 bash
----

. Mude o diretório para `workdir` e faça uma cópia do arquivo `cacerts`:
+
[,bash]
----
cd /workdir
cp $JAVA_HOME/lib/security/cacerts ./custom_cacerts
----

. Execute o seguinte comando keytool para adicionar o certificado. A senha necessária é `changeit`. O alias precisa ser um alias exclusivo para o certificado, por exemplo, o próprio nome de domínio sem nenhum ponto.
+
[,bash]
----
keytool -import -keystore custom_cacerts -alias <a-name-for-the-certificate>  -file site.cert
----

. O arquivo de armazenamento `custom_cacerts` agora incluirá o certificado `site.cert`. Você pode verificar isso procurando o alias na saída do
+
[,bash]
----
 keytool -list -keystore custom_cacerts
----

=== Usar um armazenamento de confiança personalizado

O armazenamento confiável e a senha podem ser especificados como valores. O armazenamento confiável só pode ser especificado na linha de comando do helm, pois é um arquivo. O valor da senha é especificado da mesma forma que no exemplo, mas também pode ser fornecido por meio de um arquivo `values.yaml`.

[,bash]
----
helm upgrade \
  --install \
  --namespace suse-observability \
  --values values.yaml \
  --set-file 'stackstate.java.trustStore'=custom_cacerts \
  --set 'stackstate.java.trustStorePassword'=changeit \
suse-observability \
suse-observability/suse-observability
----

[NOTE]
====
*Observação:*

* A primeira execução do comando helm upgrade resultará na reinicialização dos pods, o que pode causar uma breve interrupção da disponibilidade.
* Inclua esses argumentos em cada execução do site `helm upgrade`.
* A senha e o armazenamento de confiança são armazenados como um segredo do Kubernetes.
====


[discrete]
==== Armazenamentos confiáveis codificados em base64

Se necessário, o armazenamento de confiança Java também pode ser configurado passando strings codificadas em Base64 para os valores do Helm.

[tabs]

Para usar um armazenamento confiável codificado em base64, execute o seguinte comando `helm upgrade`:

[,bash]
----
helm upgrade \
  --install \
  --namespace suse-observability \
  --values values.yaml \
  --set 'stackstate.java.trustStoreBase64Encoded'=$(cat custom_cacerts | base64 -w0) \
  --set 'stackstate.java.trustStorePassword'=changeit \
suse-observability \
suse-observability/suse-observability
----

--

+

Para usar um armazenamento confiável codificado em base64, execute o seguinte comando `helm upgrade`:

[,bash]

suse-observability/suse-observability
----

=== Recuperar o certificado por meio do navegador

O certificado pode ser baixado diretamente do navegador Chrome. As etapas envolvidas podem variar um pouco, dependendo da versão que você estiver usando:

. Navegue até a URL da qual você precisa do certificado.
. Clique no ícone de cadeado na barra de localização.
. Clique em *Certificate (Certificado*).
. Selecione *os detalhes*.
. Selecione *Exportar*.
. Salve usando o tipo de arquivo de exportação padrão (codificado em Base64 ASCII).

====

Crie um Kubernetes ConfigMap com seu certificado e faça referência a ele na configuração do Helm:

[,yaml]
----
apiVersion: v1
kind: ConfigMap
metadata:
  name: tls-config
data:
  tls.crt: |
    -----BEGIN CERTIFICATE-----
    [Your certificate content here]
    -----END CERTIFICATE-----
----

Faça referência ao ConfigMap em sua configuração do Helm:

[,yaml]
----

  customCertificates:
    enabled: true
    configMapName: "tls-config"
----

=== Implementar com certificados personalizados

==== Uso de dados diretos do PEM

Para a abordagem direta de dados PEM, primeiro armazene seu certificado em uma variável de shell:

[,bash]
----
export CERT_DATA=$(cat <<'EOF'
-----BEGIN CERTIFICATE-----
[Your certificate content here]
-----END CERTIFICATE-----
EOF
)
----

Implante o agente com a configuração do certificado:

[,bash]
----
helm upgrade --install \
  --namespace suse-observability \
  --create-namespace \
  --set-string 'stackstate.apiKey'='YOUR_API_KEY' \
  --set-string 'stackstate.cluster.name'='YOUR_CLUSTER_NAME' \
  --set-string 'stackstate.url'='YOUR_SUSE_OBSERVABILITY_URL' \
  --set 'global.customCertificates.enabled'=true \
  --set 'global.customCertificates.pemData'="$CERT_DATA" \
  suse-observability-agent suse-observability/suse-observability-agent
----

==== Usando a referência do ConfigMap

Para a abordagem ConfigMap, crie o ConfigMap que contém seu certificado:

[,bash]
----
kubectl create configmap tls-config \
  --from-file=tls.crt=your-certificate.crt \
  --namespace suse-observability
----

Implante o Agent com a referência do ConfigMap:

[,bash]
----
helm upgrade --install \
  --namespace suse-observability \
  --create-namespace \
  --set-string 'stackstate.apiKey'='YOUR_API_KEY' \
  --set-string 'stackstate.cluster.name'='YOUR_CLUSTER_NAME' \
  --set-string 'stackstate.url'='YOUR_SUSE_OBSERVABILITY_URL' \
  --set 'global.customCertificates.enabled'=true \
  --set 'global.customCertificates.configMapName'='tls-config' \
  suse-observability-agent suse-observability/suse-observability-agent
----

== CLI do SUSE Observability

xref:setup/cli/cli-sts.adoc[A CLI do SUSE Observ] ability se conecta ao SUSE Observability Server via HTTPS. Quando o servidor usar certificados autoassinados ou certificados de uma autoridade de certificação (CA) privada, configure a CLI para confiar nesses certificados.

=== Configurar certificados de CA personalizados

Configure certificados de CA personalizados usando um destes métodos:

* **Configuração persistente**: Use `sts context save` para armazenar a configuração do certificado para comandos futuros
* **Uso único**: Adicionar sinalizadores de certificado a comandos individuais da CLI quando necessário

==== Método 1: Caminho do arquivo de certificado da CA

Especifique o caminho para o arquivo de certificado CA codificado em PEM:

[,bash]
----
sts context save \
  --name staging \
  --url https://staging.internal \
  --api-token YOUR_API_TOKEN \
  --ca-cert-path /path/to/ca.crt
----

==== Método 2: Dados de certificado CA codificados em base64

Forneça os dados do certificado da CA como uma cadeia de caracteres codificada em base64:

[,bash]
----
sts context save \
  --name staging \
  --url https://staging.internal \
  --api-token YOUR_API_TOKEN \
  --ca-cert-base64-data BASE64_ENCODED_CERTIFICATE_DATA
----

==== Uso de certificados CA com outros comandos

Use sinalizadores de certificado com qualquer comando da CLI para validação única do certificado:

[,bash]
----
# Using certificate file path
sts agent list \
  --url https://staging.internal \
  --api-token YOUR_API_TOKEN \
  --ca-cert-path /path/to/ca.crt

# Using base64-encoded certificate data
sts settings list \
  --url https://staging.internal \
  --api-token YOUR_API_TOKEN \
  --ca-cert-base64-data BASE64_ENCODED_CERTIFICATE_DATA
----

=== Precedência de configuração

Quando ambas as opções de certificado são fornecidas, o caminho do arquivo (`--ca-cert-path`) tem precedência sobre os dados de base64 (`--ca-cert-base64-data`).

=== Armazenamento

As configurações de certificado são armazenadas em: `~/.config/stackstate-cli/config.yaml`

[NOTE]
====
**Importante**: O sinalizador `--skip-ssl` desativa todas as verificações de SSL e ignora as configurações de certificado. Sempre use as opções de certificado CA para conexões seguras com certificados personalizados.
====

== Telemetria aberta do SUSE Observability

=== Criar um mapa de configuração para a autoridade de certificação personalizada

----
kubectl create configmap tls-config \
  --namespace open-telemetry \
  --from-file=tls.crt=<your-certificate>
----

. Monte o configmap adicionando o seguinte no nível superior de `otel-collector.yaml`:

----
# Mounting volume for CA certificate
extraVolumes: 
  - name: tls-config
    configMap:
      items:
        - key: tls.crt
          path: tls.crt
      name: tls-config
extraVolumeMounts:
  - mountPath: "/certs"
    name: tls-config
    readOnly: true
----

. Atualize a configuração do exportador em `otel-collector.yaml` para usar o *arquivo ca_file do* certificado, adicionando a chave `tls.ca_file`:

----
otlp/suse-observability:
  auth:
    authenticator: bearertokenauth
  # Put in your own otlp endpoint, for example suse-observability.my.company.com:443
  endpoint: <your-endpoint>
  compression: snappy
  tls:
    ca_file: /certs/tls.crt
----

. Instale (ou atualize) seu coletor Open Telemetry.

== Extensão da interface do usuário do Rancher para o SUSE Observability

Ao instalar a extensão da interface do usuário do Rancher para o SUSE Observability (consulte xref:/k8s-suse-rancher-prime.adoc#_installing_ui_extensions[Instalação de extensões da interface do usuário]), a extensão deve se comunicar com o SUSE Observability Server. Se o seu servidor usar certificados autoassinados, a instalação da extensão falhará.

**Solução**: Adicione seu certificado personalizado ao Rancher antes de instalar a extensão. Siga a documentação do Rancher: https://ranchermanager.docs.rancher.com/getting-started/installation-and-upgrade/resources/custom-ca-root-certificates[configurando certificados raiz de CA personalizados^].

Depois de configurar o certificado no Rancher, a extensão se conectará com êxito ao SUSE Observability Server.
