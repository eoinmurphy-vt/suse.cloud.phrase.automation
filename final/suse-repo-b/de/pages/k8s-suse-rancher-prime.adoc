== Einführung

{stackstate-product-name}, früher bekannt als StackState, kann für die Beobachtbarkeit Ihrer Kubernetes-Cluster und deren Workloads verwendet werden.

Die Installation von {stackstate-product-name}, der {stackstate-product-name} UI-Erweiterung und den {stackstate-product-name} Agenten dauert insgesamt etwa 30 Minuten.

== Hilfe holen

Wenn Sie Unterstützung benötigen, reichen Sie bitte einen Support-Fall im https://scc.suse.com/[Customer Center (SCC] SUSE) ein.

== Voraussetzungen

=== Lizenzschlüssel

Ein Lizenzschlüssel für {stackstate-product-name} Server kann über das SUSE Customer Center auf der Registerkarte Subscription bezogen werden und wird als "{stackstate-product-name}" Registrierungscode angezeigt. Diese Lizenz ist bis zum Ende Ihres Rancher Prime-Abonnements gültig.

=== Anforderungen

Um {stackstate-product-name} zu installieren, stellen Sie sicher, dass der Cluster über ausreichend CPU- und Speicherkapazität verfügt. Im Folgenden sind die spezifischen Anforderungen aufgeführt.

Für {stackstate-product-name} gibt es verschiedene Installationsmöglichkeiten. Es ist möglich, {stackstate-product-name} entweder in einer Hochverfügbarkeits- (HA) oder in einer Einzelinstanzkonfiguration (nicht-HA) zu installieren. Das Nicht-HA-Setup wird für Testzwecke oder kleine Umgebungen empfohlen. Für Produktionsumgebungen wird empfohlen, {stackstate-product-name} in einem HA-Setup zu installieren.

Die HA-Produktionseinrichtung kann zwischen 150 und 4000 beobachtete Knoten unterstützen. Ein beobachteter Knoten wird in dieser Dimensionierungstabelle mit 4 vCPUs und 16 GB Arbeitsspeicher angenommen, unsere `default node size`.
Wenn Knoten in Ihrem beobachteten Cluster größer sind, können sie für mehrere `default nodes` zählen, so dass ein Knoten mit 12vCPU und 48GB als 3 `default nodes` unter Beobachtung zählt, wenn Sie ein Profil auswählen.
Die Nicht-HA-Einrichtung kann bis zu 100 `default nodes` unter Beobachtung halten.

=== Lagerung

{stackstate-product-name} verwendet persistente Volumenansprüche für die Dienste, die Daten speichern müssen. Die Standardspeicherklasse für den Cluster wird für alle Dienste verwendet, es sei denn, sie wird durch Werte überschrieben, die in der Befehlszeile oder in einer Datei `values.yaml` angegeben werden. Alle Dienste werden mit einer vorkonfigurierten Volume-Größe ausgeliefert, die für den Anfang ausreichen sollte, die aber später mit Hilfe von Variablen nach Bedarf angepasst werden kann.

[NOTE]
====
{stackstate-product-name} erfordert, dass der zugrunde liegende Speicher auf Flash-Speicher (SSD) oder einer ähnlichen Leistung basiert.
====

[NOTE]
====
Für Produktionsumgebungen wird NFS für die Speicherbereitstellung in {stackstate-product-name} aufgrund des potenziellen Risikos der Datenbeschädigung nicht empfohlen und unterstützt.
====

=== Helm

{stackstate-product-name} wird über Helm installiert, das mit einer Mindestversion von 3.13.1 installiert sein muss.

=== Die verschiedenen Komponenten

==== {stackstate-product-name} Server

Dies ist der Teil der Installation, bei dem der Server vor Ort gehostet wird. Sie enthält eine Reihe von Diensten zur Speicherung von Beobachtbarkeitsdaten:

* Topologie (StackGraph)
* Metriken (VictoriaMetrics)
* Spuren (ClickHouse)
* Protokolle (ElasticSearch)

Darüber hinaus enthält sie eine Reihe von Diensten für alle Aufgaben im Zusammenhang mit der Beobachtbarkeit, z. B. Benachrichtigungen, Zustandsverwaltung, Überwachung usw.

==== {stackstate-product-name} Agent

Der leichtgewichtige {stackstate-product-name} -Agent wird auf Ihren nachgelagerten Worker-Nodes installiert. Es sammelt und meldet Metriken, Ereignisse, Traces und Protokolle und bietet Echtzeitbeobachtung und Einblicke, die eine proaktive Überwachung und Fehlerbehebung Ihrer IT-Umgebung ermöglichen.

Die Version {stackstate-product-name} des Agenten verwendet eBPF als leichtgewichtige Methode zur Überwachung all Ihrer Workloads und deren Kommunikation. Es dekodiert auch die RED-Signale (Rate, Errors and Duration) für die meisten der gängigen L7-Protokolle wie TCP, HTTP, TLS, Redis usw.

==== Rancher Prime - UI-Erweiterung für Beobachtungsfähigkeit

Dies ist eine UI-Erweiterung für Rancher Manager, die die von {stackstate-product-name} beobachteten Gesundheitssignale integriert. Sie bietet direkten Zugriff auf den Zustand jeder Ressource und einen Link zur Benutzeroberfläche {stackstate-product-name} für weitere Untersuchungen.

=== Wo wird {stackstate-product-name} Server installiert?

{stackstate-product-name} Server sollte in einem eigenen, für Observability vorgesehenen Downstream-Cluster installiert werden. Siehe das folgende Bild als Referenz.

Damit {stackstate-product-name} richtig funktionieren kann, muss es:

* https://ranchermanager.docs.rancher.com/how-to-guides/new-user-guides/manage-clusters/create-kubernetes-persistent-storage[Kubernetes Persistent Storage] muss im Observability-Cluster verfügbar sein, um Metriken, Ereignisse usw. zu speichern.
* den Observability-Cluster, um eine Möglichkeit zu unterstützen, {stackstate-product-name} auf einer HTTPS-URL für Rancher, {stackstate-product-name} -Benutzer und den {stackstate-product-name} -Agenten zugänglich zu machen. Dies kann über eine Ingress-Konfiguration unter Verwendung eines Ingress-Controllers erfolgen, alternativ könnte dies auch ein (Cloud-)Loadbalancer für die {stackstate-product-name} -Dienste übernehmen, weitere Informationen finden Sie in den https://ranchermanager.docs.rancher.com/how-to-guides/new-user-guides/kubernetes-resources-setup/load-balancer-and-ingress-controller[Rancher-Dokumenten].

=== Vor der Installation

Vor der Installation des {stackstate-product-name} Servers muss in dem Cluster, in dem der {stackstate-product-name} Server installiert werden soll, eine Standard-Speicherklasse eingerichtet werden:

* *Für k3s*: Die Speicherklasse local-path vom Typ rancher.io/local-path wird standardmäßig erstellt.
* *Für EKS, AKS, GKE* wird standardmäßig eine Speicherklasse festgelegt
* *Für RKE2-Knoten-Treiber*: Standardmäßig wird keine Speicherklasse erstellt. Diese müssen Sie vor der Installation von {stackstate-product-name} erstellen.

== Zugriff auf {stackstate-product-name}

Das {stackstate-product-name} Helm-Diagramm bietet Unterstützung für die Erstellung einer Ingress-Ressource, um {stackstate-product-name} außerhalb des Clusters zugänglich zu machen. Befolgen Sie xref:/setup/install-stackstate/kubernetes_openshift/ingress.adoc[diese Anweisungen], um dies einzurichten, wenn Sie einen Ingress-Controller im Cluster haben. Stellen Sie sicher, dass die resultierende URL TLS mit einem gültigen, nicht selbstsignierten Zertifikat verwendet.

Wenn Sie es vorziehen, einen Load Balancer anstelle des Ingress zu verwenden, stellen Sie den Dienst `suse-observability-router` zur Verfügung. Die URL für den Loadbalancer muss ein gültiges, nicht selbstsigniertes TLS-Zertifikat verwenden.

== Erforderliche Privilegien

Für die Bereitstellung des Agenten {stackstate-product-name} sind die folgenden Systemberechtigungen erforderlich:

. `hostPID: true`: Diese Berechtigung ist erforderlich, um Prozesskennungen (PIDs) mit den entsprechenden Kontrollgruppen (cgroups) zu verknüpfen. Diese Verknüpfung ist für die genaue Zuordnung von Prozessen zu ihren jeweiligen Containern unerlässlich.
. `hostNetwork: true` (fakultativ): Standardmäßig wird der Knotenagent mit `hostNetwork: true` ausgeführt, um das Abrufen von offenen Metrikdaten von allen konfigurierten Pods auf dem Knoten zu erleichtern, ohne dass zusätzliche Netzwerkrichtlinien erforderlich sind. Falls deaktiviert, müssen entsprechende Netzwerkrichtlinien definiert werden, um sicherzustellen, dass der Agent auf die erforderlichen Endpunkte zugreifen kann.
. `securityContext.privileged: true`: Diese erhöhte Berechtigung ist für mehrere kritische Funktionen erforderlich. In erster Linie erlaubt es dem Agenten, eBPF-Programme (erweiterter Berkeley Packet Filter) zu Überwachungszwecken in jeden Netzwerk-Namensraum zu injizieren. Sie ist auch notwendig, um die Tabellen der Verbindungsverfolgung (conntrack) in allen Netzwerk-Namensräumen zu lesen. Diese Liste ist zwar nicht erschöpfend, aber die künftige Entwicklung zielt darauf ab, dieses umfassende Privileg durch detailliertere Linux-Funktionen zu ersetzen, wo dies möglich ist.

Außerdem benötigt der Agent Container-Laufzeit-Sockets, die in seinem Pod eingehängt werden. Diese Konfiguration ist unabdingbar, da sie die direkte Kommunikation mit den Container-Laufzeitdämonen ermöglicht, was eine Voraussetzung für die Erfassung von Metriken und Metadaten aus allen Containern auf dem Host-System ist.

== Rancher-Restricted PSA Vorlage

Die Konfiguration `Rancher-restricted` (link:https://ranchermanager.docs.rancher.com/how-to-guides/new-user-guides/authentication-permissions-and-global-configuration/psa-config-templates[Pod Security Admission (PSA) Configuration Templates]) ist eine sehr restriktive Einrichtung, die den aktuellen Best Practices für die Sicherung von Pods entspricht.

Wenn Sie Rancher auf einem Kubernetes-Cluster ausführen, das standardmäßig eine restriktive Sicherheitsrichtlinie durchsetzt, gibt es zwei Möglichkeiten, das SUSE Observability Helm-Diagramm zu installieren:

* *Nehmen Sie den gesamten Diagramm-Namensraum aus*, zusammen mit anderen link:https://ranchermanager.docs.rancher.com/how-to-guides/new-user-guides/authentication-permissions-and-global-configuration/psa-config-templates#exempting-required-rancher-namespaces[erforderlichen Rancher-Namensräumen].
* *Deaktivieren Sie den privilegierten Elasticsearch-Init-Container*, indem Sie `elasticsearch.sysctlInitContainer.enabled` auf `false` setzen. Dazu müssen Sie die link:https://www.elastic.co/docs/deploy-manage/deploy/self-managed/vm-max-map-count[Einstellungen für den virtuellen Speicher] (`vm.max_map_count`) auf den Knoten manuell erhöhen. Siehe auch xref:/setup/install-stackstate/kubernetes_openshift/required_permissions.adoc#_elasticsearch[Elasticsearch Erforderliche Berechtigungen]. 

Da der SUSE Observability Agent im privilegierten Modus ausgeführt werden muss, empfiehlt es sich, ihn in einem Namespace zu installieren, den Sie von der restriktiven Richtlinie ausnehmen möchten.

[NOTE]
====
Alle SUSE Observability Helm-Diagrammcontainer werden ab der Version `v2.3.8` mit den folgenden `securityContext` -Einstellungen konfiguriert:

. `securityContext.capabilities.drop` ist `["ALL"]`
. `securityContext.seccompProfile.type` ist `"RuntimeDefault"`
. `securityContext.runAsNonRoot` ist `true`
. `securityContext.allowPrivilegeEscalation` ist `false`
====

== Häufig gestellte Fragen und Beobachtungen:

. Ist es zwingend erforderlich, einen {stackstate-product-name} -Agenten zu installieren, bevor man mit dem Hinzufügen der UI-Erweiterung fortfährt?
 ** Nein, dies ist nicht zwingend erforderlich, die UI-Erweiterung kann unabhängig davon installiert werden.
. Ist es zwingend erforderlich, {stackstate-product-name} Server zu installieren, bevor wir mit den UI-Erweiterungen fortfahren?
 ** Ja, dies ist obligatorisch, da Sie in der Konfiguration einen {stackstate-product-name} Endpunkt angeben müssen.
. Können wir {stackstate-product-name} auf einem lokalen Cluster oder auf einem nachgelagerten Cluster installieren?
 ** Beide Optionen sind möglich.
. Sollten wir zur Überwachung der nachgelagerten Cluster den {stackstate-product-name} Agent aus dem App Store installieren oder eine neue Instanz über die {stackstate-product-name} UI hinzufügen?
 ** Je nach Präferenz des Benutzers sind beide Optionen möglich.

== Offene Fragen

. Bei der Deinstallation und Neuinstallation der UI-Erweiterungen für Observability ist uns aufgefallen, dass das Service-Token nicht gelöscht wird und bei der Neuinstallation wieder verwendet wird. Wenn wir die Erweiterungen deinstallieren, sollte das Service-Token entfernt werden.
 ** Diese Informationen sollten gelöscht werden, wenn die UI-Erweiterungen deinstalliert werden.
. Nachdem die Erweiterungen installiert sind, wird die Benutzeroberfläche {stackstate-product-name} in derselben Registerkarte wie die Rancher-Benutzeroberfläche geöffnet.
 ** Mit einem Shift-Klick können Sie eine neue Registerkarte öffnen, dies wird dann zum Standardverhalten.
. Bitte beachten Sie, dass ein Upgrade oder Downgrade von HA auf NON-HA und umgekehrt noch nicht unterstützt wird.
