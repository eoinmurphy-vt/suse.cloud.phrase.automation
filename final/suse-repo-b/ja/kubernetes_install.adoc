=== 名前空間の作成

まず、{stackstate-product-name} をインストールするネームスペースを作成し、そのネームスペースにシークレットをデプロイします。このチュートリアルでは、名前空間`suse-observability` を使用します：

[,text]
----
kubectl create namespace suse-observability
----

[#create-values-yaml]
=== `values.yaml` 設定ファイルの作成

`values.yaml` ファイルには、ライセンスキー、ベース URL、管理者パスワード、サイジングプロファイルなど、{stackstate-product-name} の展開に必要なすべての構成が含まれています。

[NOTE]
====
*続ける前に*最新版のHelmチャート（`helm repo update` ）があることを確認してください。
====

[tabs]
====
推奨方法::
+
--
[NOTE]
=====
`global.suseObservability` 設定方法は、バージョン`2.8.0` 以降で利用可能です。それ以前のバージョンの場合は、レガシー・メソッドを使用する。
=====

`global.suseObservability` のコンフィギュレーション構造で`values.yaml` ファイルを作成する：

[source,yaml]
----
global:
  # Optional: Override image registry (defaults to registry.rancher.com)
  # Only needed for air-gapped environments or custom registries
  # imageRegistry: "your-private-registry.example.com"

  # Optional: Override storage class (uses cluster default if not set)
  # storageClass: "your-storage-class"

  suseObservability:
    # Required: Your {stackstate-product-name} license key
    license: "YOUR-LICENSE-KEY"

    # Required: Base URL for {stackstate-product-name}
    # Use http://localhost:8080 if ingress is not configured yet
    baseUrl: "https://observability.example.com"

    # Required: Sizing profile
    # Available: trial, 10-nonha, 20-nonha, 50-nonha, 100-nonha,
    #            150-ha, 250-ha, 500-ha, 4000-ha
    sizing:
      profile: "150-ha"

    # Required: Plain text admin password
    adminPassword: "your-password"
    # Instead of adminPassword you can provide a bcrypt hashed password with adminPasswordBcrypt
    # Generate with: htpasswd -bnBC 10 "" "your-password" | tr -d ':\n'
    # adminPasswordBcrypt: "$2a$10$..."

    # Optional: Receiver API key (auto-generated if not provided)
    # receiverApiKey: "your-receiver-api-key"

    # Optional: Pull secret for private registry
    # pullSecret:
    #   username: "registry-user"
    #   password: "registry-pass"
----

アフィニティ設定を含むその他の設定オプションについては、xref:/setup/install-stackstate/kubernetes_openshift/affinity.adoc[Kubernetesアフィニティの設定を]参照してください。

--
レガシーメソッド（非推奨）::
+
--
[WARNING]
=====
このメソッドは非推奨。新規設置の場合は、上記の推奨方法を使用してください。この方法を使用している既存のインストールについては、新しい構成フォーマットに移行するためのxref:/setup/upgrade-stackstate/migrate-to-global-mode.adoc[移行ガイドを]参照してください。

別のHelmチャートを実行することで、{stackstate-product-name} 設定ファイルを生成できます。これが`suse-observability/suse-observability-values` のチャートだ：

[,text]
----
export VALUES_DIR=.
helm template \
  --set license='<your license>' \
  --set baseUrl='<suse-observability-base-url>' \
  --set sizing.profile='<sizing.profile>' \
  suse-observability-values \
  suse-observability/suse-observability-values --output-dir $VALUES_DIR
----

このコマンドは、`$VALUES_DIR/suse-observability-values/templates/baseConfig_values.yaml` 、`$VALUES_DIR/suse-observability-values/templates/sizing_values.yaml` 、`$VALUES_DIR/suse-observability-values/templates/affinity_values.yaml` のファイルを生成する。

[NOTE]
=====
{stackstate-product-name} 管理者パスワードは上記のコマンドで自動生成され、生成された`basicConfig.yaml` ファイルにコメントとして出力される。実際の値には、これらのパスワードの`bcrypt` ハッシュが含まれており、クラスタ内のHelmリリースに安全に保存されます。
=====

このチャートに渡すことのできる値は以下の通りである：

[cols="1,1,2"]
|===
| 構成 | 価値 | 説明

| 基本設定の生成
| `basicConfig.generate`
| `basicConfig_values.yaml` ファイルの生成のオン・オフ。

| レシーバーAPIキー
| `receiverApiKey`
| {stackstate-product-name} がエージェントからデータを受信するために使用する API キー。これは秘密にしておくべき鍵である。これを省略すると、ランダムな鍵が生成される。

| ベースURL
| `baseUrl`
| `<STACKSTATE_BASE_URL>`.ユーザーやエージェントが接続に使用する{stackstate-product-name} の外部 URL。例えば`https://suse-observability.internal` 。まだIngressのコンフィギュレーションを決めていない場合は、`http://localhost:8080` 。これは後で生成されたファイルで更新することができる。

| ユーザー名とパスワード
| `-u` `-p`
| {stackstate-product-name} が画像を取得する際に使用するユーザ名とパスワード。エアギャップ環境の場合、これらはローカル・コン タイア・レジストリのユーザー名とパスワードである必要がある。

| ライセンスキー
| `license`
| {stackstate-product-name} ライセンスキー。

| デフォルトパスワード
| `adminPassword`
| {stackstate-product-name} の UI にアクセスするためのデフォルト・ユーザー (`admin`) のパスワード。あるいは、`adminPasswordBcrypt` を使用して、bcrypt ハッシュ化されたパスワードを指定することもできます。同時に使用できるのは、2つのオプションのうち1つだけである。

| 画像登録
| `imageRegistry`
| {stackstate-product-name} イメージがホストされているレジストリ。提供されない場合、デフォルト値は'registry.rancher.com'となる。エアギャップ環境またはカスタムレジストリにのみ必要。

| 秘密のユーザー名を引く
| `pullSecret.username`
| {stackstate-product-name} イメージがホストされている Docker レジストリからイメージを取得する際に使用するユーザー名。カスタム登録にのみ必要です。

| 秘密のパスワードを引き出す
| `pullSecret.password`
| {stackstate-product-name} イメージがホストされている Docker レジストリからイメージを取得する際に使用するパスワード。カスタム登録にのみ必要です。

| サイズ生成
| `sizing.generate`
| `sizing_values.yaml` ファイルの生成のオン・オフ。

| サイズプロファイル
| `sizing.profile`
| OneOfトライアル、10-nonha、20-nonha、50-nonha、100-nonha、150-ha、250-ha、500-ha。これらのプロファイルに基づいて、{stackstate-product-name} リソースのデフォルトサイズと、HA または非 HA モードでデプロイするための構成を含む`sizing_values.yaml` ファイルを生成できます。例えば、10-nonhaは、NonHa{stackstate-product-name} インスタンスをデプロイして、10ノード・クラスターをNon High availabilityモードで監視するための`sizing_values.yaml` 。現在、Non-HAからHA環境への移行は不可能である。もし、あなたの環境が150ノード程度を監視する必要があると予想されるのであれば、すぐにでもHAを導入した方がいいだろう。

| ポッド・アンチアフィニティ
| `affinity.podAntiAffinity.requiredDuringSchedulingIgnoredDuringExecution`
| 同じデータサービスのレプリカが異なるKubernetesノードにスケジュールされることを強制する。`HA` プロフィールにのみ適用されます。`true` もしくは`false` 。デフォルトは`true` - 異なるノードへのスケジューリングが強制される。
|===

[NOTE]
=====
生成された`basicConfig.yaml` と`sizing_values.yaml` ファイルを安全に保管する。これらのファイルをアップグレードの際に再利用することで、時間を節約し、{stackstate-product-name} 、同じAPIキーを使い続けることができます。これは、エイジェントやその他のデータプロバイダー（{stackstate-product-name} ）が更新の必要がないことを意味するので、望ましいことである。
`basicConfig.generate=false` 、`sizing.generate=false` のスイッチを使えば、`output-dir` に前回生成したバージョンのファイルを残したまま、いずれかのファイルを無効にすることができる。
=====

[NOTE]
=====
{stackstate-product-name} Values チャートは、ポッドのスケジューリング動作を制御するために、メインの{stackstate-product-name} チャートで使用できるアフィニティ構成を生成します。詳細については、「xref:/setup/install-stackstate/kubernetes_openshift/affinity.adoc[Configure Kubernetes Affinities]」のドキュメントを参照してください。
=====

=== ストレージクラスの設定（オプション）

{stackstate-product-name} が指定されていない場合は、クラスタのデフォルト・ストレージ・クラスを使用します。`values.yaml` ファイルで`global.storageClass` を設定することで、オプションでこれを上書きすることができる。

[tabs]
====
推奨方法::
+
--
特定のストレージクラスを使用する必要がある場合は、`values.yaml` ファイルの`global.storageClass` に追加してください：

[source,yaml]
----
global:
  storageClass: "your-storage-class"  # e.g., "local-path" for Rancher local-path provisioner
  suseObservability:
    # ... other configuration
----
--
レガシーメソッド（非推奨）::
+
--
ファイル`$VALUES_DIR/suse-observability-values/templates/baseConfig_values.yaml` を編集し、"storageClass "設定を追加する：

[,text]
----
# Source suse-observability-values/templates/baseConfig_values.yaml
global:
  storageClass: "your-storage-class" # Optional: override default storage class
----
--

=== {stackstate-product-name} UIにアクセス

{stackstate-product-name} がデプロイされた後、すべてのポッドが稼働しているかどうかを確認できます：

[,bash]

イングレス・コンフィギュレーションでは、`ingress_values.yaml` ファイルを作成し、以下のコードを追加する。

[NOTE]
====
TLSコンフィギュレーションにシークレットを設定するのを忘れないこと。その方法の詳細については、kubectlのドキュメントを参照のこと。
====

[,yaml]
----
ingress:
  enabled: true
  class: ""
  annotations:
    nginx.ingress.kubernetes.io/proxy-body-size: "50m"
    nginx.ingress.kubernetes.io/ssl-redirect: "true"
    #if external dns (cloudflare and similars) configure:
    #external-dns.alpha.kubernetes.io/target: <INGRESS_FQDN>
    #external-dns.alpha.kubernetes.io/ttl: "60"
    #end if
  hosts:
  - host: <SUSE_OBSERVABILITY_HOST>
  tls:
  - hosts:
    - <SUSE_OBSERVABILITY_HOST>
    secretName: suse-private-ai-tls
  existingSecret: suse-private-ai-tls
  # Set up your secrets in whichever way feels fit. Check k8s docs for details: https://kubernetes.io/docs/reference/kubectl/generated/kubectl_create/kubectl_create_secret_tls/
----
