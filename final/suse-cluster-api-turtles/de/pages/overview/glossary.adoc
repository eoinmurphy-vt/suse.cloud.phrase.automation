== A

=== Add-ons

Dienste, die über die grundlegenden Komponenten hinausgehen, die für die Bereitstellung eines Kubernetes-konformen Clusters erforderlich sind, werden in zwei Typen unterteilt:

* *Kern-Add-ons*: Addons, die für die Bereitstellung eines Kubernetes-konformen Clusters erforderlich sind: DNS, kube-proxy, CNI.
* *Zusätzliche Add-ons*: Addons, die für einen Kubernetes-konformen Cluster nicht erforderlich sind (z. B. Metriken/Heapster, Dashboard).

=== Klimatisierte Umgebung

Einrichtung und Betrieb von Kubernetes-Clustern ohne direkten Internetzugang.

== B

=== Bootstrap

Der Prozess der Umwandlung eines Servers in einen Kubernetes-Knoten. Dies kann die Zusammenstellung von Daten beinhalten, die bei der Erstellung des Servers, der die Maschine unterstützt, bereitgestellt werden müssen, sowie die Laufzeitkonfiguration der auf diesem Server laufenden Software.

=== Bootstrap-Cluster

Ein temporärer Cluster, der für die Bereitstellung eines Target Management-Clusters verwendet wird.

=== Bootstrap-Anbieter

Bezieht sich auf einen <<_provider,Anbieter>>, der eine Lösung für den <<_bootstrap,Bootstrap-Prozess>> implementiert.

=== Kinder-Cluster

Begriff, der häufig austauschbar mit dem Begriff <<_workload_cluster,Workload-Cluster>> verwendet wird.

=== Cluster

Eine vollständige Kubernetes-Bereitstellung. Siehe Management Cluster und Workload Cluster.

=== ClusterClass

Eine Sammlung von Vorlagen, die eine Topologie (Steuerebene und Arbeiter) definieren, die für die kontinuierliche Abstimmung eines oder mehrerer Cluster verwendet wird.
Siehe xref:../user/clusterclass.adoc[ClusterClass]

=== Cluster-API

Oder *Cluster-API-Projekt*

Die Cluster-API ist ein Teilprojekt der SIG-cluster-lifecycle. Er wird auch verwendet, um sich auf die Softwarekomponenten, APIs und die Gemeinschaft zu beziehen, die sie produziert.

Siehe <<_core_provider,Hauptanbieter>>

=== Cluster-API-Betreiber

Oder *Cluster-API-Operator-Projekt*

Der Cluster API Operator ist ein Teilprojekt der SIG-cluster-lifecycle. Es wurde entwickelt, um Cluster-Administratoren die Möglichkeit zu geben, den Lebenszyklus von Cluster-API-Anbietern innerhalb eines Management-Clusters mithilfe eines deklarativen Ansatzes zu verwalten.

=== Cluster-API-Anbieter RKE2

<<_caprke2,Cluster-API-Provider RKE2>> ist eine Kombination aus zwei Providertypen: einem Cluster-API-Control-Plane-Provider für die Bereitstellung von Kubernetes-Kontrolplane-Knoten und einem Cluster-API-Bootstrap-Provider für das Bootstrapping von Kubernetes auf einem Rechner, auf dem <<_rke2,RKE2>> als Kubernetes-Distro verwendet wird.

=== Steuerungsebene

Der Satz von Kubernetes-Diensten, der die Grundlage eines Clusters bildet. Siehe auch https://kubernetes.io/docs/concepts/#_kubernetes-control-plane Es gibt zwei Varianten:

* *Selbstversorger*: Eine Kubernetes-Kontrollebene, die aus Pods oder Maschinen besteht, die vollständig von einer einzigen Cluster-API-Bereitstellung verwaltet werden.
* *Extern* oder *verwaltet*: Eine Steuerungsebene, die von einem anderen System als der Cluster-API angeboten und gesteuert wird (z. B. GKE, AKS, EKS, IKS).

=== Anbieter der Steuerungsebene

Bezieht sich auf einen <<_provider,Anbieter>>, der eine Lösung für die Verwaltung einer <<_control_plane,Kubernetes-Kontrollebene>> implementiert.

Siehe <<_caprke2,CAPRRKE2>>, <<_kcp,KCP>>.

=== Hauptanbieter

Bezieht sich auf einen <<_provider,Anbieter>>, der Cluster-API-Kern-Controller implementiert; wenn Sie bedenken, dass das erste Projekt, das in einem Verwaltungscluster bereitgestellt werden muss, die Cluster-API selbst ist, sollte klar sein, warum das Cluster-API-Projekt auch als Kernanbieter bezeichnet wird.

Siehe <<_cluster_api,CAPI>>.

== I

=== Anbieter von Infrastruktur

Bezieht sich auf einen <<_provider,Anbieter>>, der die Bereitstellung der vom Cluster oder von Maschinen benötigten Infrastruktur-/Rechenressourcen (z. B. VMs, Netzwerke usw.) durchführt.
Zu den Cloud-Infrastrukturanbietern gehören AWS, Azure oder Google, während VMware, MAAS oder metal3.io als Bare-Metal-Anbieter definiert werden können.

=== IPAM-Anbieter

Bezieht sich auf einen <<_provider,Anbieter>>, der eine Cluster-API für die Interaktion mit IPAM-Lösungen ermöglicht.
Die Interaktion des IPAM-Anbieters mit der Cluster-API basiert auf den API-Typen `IPAddressClaim` und `IPAddress`.

== K

=== Kubernetes-conformant

Oder *Kubernetes-kompatibel*

Ein Cluster, der die Kubernetes-Konformitätstests besteht.

=== Kubernetes Operator

Ein Kubernetes-Operator ist eine Methode zum Verpacken, Bereitstellen und Verwalten einer Kubernetes-Anwendung. Siehe auch https://kubernetes.io/docs/concepts/extend-kubernetes/operator/ für weitere Informationen.

=== k/k

Verweist auf das https://github.com/kubernetes/kubernetes[Haupt-Kubernetes-Git-Repository] oder das Haupt-Kubernetes-Projekt.

=== KCP

Kubeadm Control plane Provider

== M

=== Maschine

Oder *Maschine Ressource*

Die benutzerdefinierte Ressource für Kubernetes, die eine Infrastrukturkomponente darstellt, die einen Kubernetes-Knoten hostet.

=== Verwalten eines Clusters

Erstellen, Skalieren, Aktualisieren oder Zerstören des Clusters.

=== Verwaltete Kubernetes

Verwaltetes Kubernetes bezieht sich auf eine beliebige Abstraktion für die Bereitstellung und Wartung von Kubernetes-Clustern, die in der Regel als API zur Verfügung steht und in einem Cloud-Anbieter nativ verfügbar ist. Zum Beispiel: https://aws.amazon.com/eks/[EKS], https://www.oracle.com/cloud/cloud-native/container-engine-kubernetes/[IBM Cloud Kubernetes Service] OKE, https://azure.microsoft.com/en-us/products/kubernetes-service[DOKS] AKS, https://cloud.google.com/kubernetes-engine[] GKE, https://www.ibm.com/cloud/kubernetes-service[], https://www.digitalocean.com/products/kubernetes[], und viele weitere im gesamten Kubernetes Cloud Native Ökosystem.

=== Verwaltete Topologie

Siehe <<_topology,Topologie>>

=== Management-Cluster

Der Cluster, in dem ein oder mehrere Infrastrukturprovider laufen und in dem Ressourcen (z. B. Maschinen) gespeichert sind. Wird in der Regel verwendet, wenn Sie mehrere Workload-Cluster bereitstellen.

== N

=== Knotenpunkt-Pools

Ein Knotenpool ist eine Gruppe von Knoten innerhalb eines Clusters, die alle dieselbe Konfiguration haben.

== O

=== Das Betriebssystem

Oder *OS*

Eine allgemein verstandene Kombination aus einem Kernel und einer Benutzeroberfläche auf Systemebene, wie z. B. Linux oder Windows, im Gegensatz zu einer bestimmten Distribution.

== P

=== Pivot

Pivot ist ein Prozess zum Verschieben der Provider-Komponenten und der deklarierten Cluster-Api-Ressourcen von einem Source Management-Cluster zu einem Target Management-Cluster.

Der Pivot-Prozess wird auch zum Löschen eines Management-Clusters verwendet und könnte auch bei einem Upgrade des Management-Clusters eingesetzt werden.

=== Anbieter

Oder *Cluster-API-Anbieter*

Dieser Begriff wurde ursprünglich als Abkürzung für <<_infrastructure_provider,Infrastructure provider (Infrastrukturanbieter>>) verwendet, aber derzeit wird er für jedes Projekt verwendet, das bereitgestellt werden kann und dem Cluster API management Cluster Funktionalität bietet.

Siehe <<_bootstrap_provider,Bootstrap-Anbieter>>, <<_control_plane_provider,Anbieter der Steuerungsebene>>, <<_core_provider,Kernanbieter>>,<<_infrastructure_provider,Infrastrukturanbieter>>, <<_ipam_provider,IPAM-Anbieter>>, <<_runtime_extension_provider,Anbieter der Laufzeitverlängerung>>.

=== Anbieter-Komponenten

Bezieht sich auf das YAML-Artefakt, das im Rahmen des Freigabeprozesses für <<_provider,Provider>> veröffentlicht wird; es enthält in der Regel Custom Resource Definitions (CRDs), Deployments (zur Ausführung des Controller Managers), RBAC usw.

In einigen Fällen wird derselbe Ausdruck verwendet, um sich auf die Instanzen der oben genannten Komponenten zu beziehen, die in einem Management-Cluster eingesetzt werden.

Siehe <<_provider_repository,Provider-Repository>>

=== Anbieter-Repository

Bezieht sich auf den Ort, an dem die YAML für <<_provider_components,Provider-Komponenten>> gehostet wird; in der Regel werden in einem Provider-Repository viele Versionen von Provider-Komponenten gehostet, eine für jede freigegebene Version.

== R

=== Rancher

Eine https://www.rancher.com/[Open-Source-Plattform], die die Bereitstellung und Verwaltung von Kubernetes-Clustern vereinfacht.

=== Rancher Cluster Agent

Eine Komponente, die von Rancher in jedem von ihm verwalteten Kubernetes-Cluster bereitgestellt wird. Seine Hauptaufgabe besteht darin, einen sicheren Kommunikationskanal zwischen dem Rancher-Server und dem Kubernetes-Cluster aufzubauen, damit Rancher den Cluster verwalten und mit ihm interagieren kann.

=== Rancher Manager

Der Rancher Manager (oder Rancher Server) ist der Ort, an dem die Rancher-Benutzeroberfläche und die API gehostet werden, und er kommuniziert mit verwalteten Clustern über Komponenten wie den <<_rancher_cluster_agent,Rancher Cluster Agent>>. Es ermöglicht Benutzern, ihre Kubernetes-Cluster, Anwendungen und Rancher-spezifischen Ressourcen wie Kataloge, Benutzer, globale Rollen und mehr zu verwalten.

=== Laufzeitverlängerung

Eine externe Komponente, die Teil eines Systems ist, das auf der Cluster-API aufbaut und Anfragen für einen bestimmten Runtime Hook bearbeiten kann.

=== Anbieter von Laufzeitverlängerungen

Bezieht sich auf einen <<_provider,Anbieter>>, der eine oder mehrere <<_runtime_extension,Laufzeit-Erweiterungen>> implementiert.

== S

=== Skalierung

Wenn nicht anders angegeben, handelt es sich um eine horizontale Skalierung.

=== Gestapelte Kontrollebene

Ein Steuerebenen-Knoten, auf dem etcd zusammen mit dem Kubernetes-API-Server untergebracht ist und als statischer Pod ausgeführt wird.

=== Server

Die Infrastruktur, die hinter einer <<_machine,Maschinenressource>> steht, in der Regel entweder eine Cloud-Instanz, eine virtuelle Maschine oder ein physischer Host.

=== {product_name}

Ein <<_kubernetes_operator,Kubernetes-Operator>>, der die Integration zwischen dem Rancher Manager und der Cluster-API (CAPI) ermöglicht, mit dem Ziel, Rancher eine vollständige CAPI-Unterstützung zu bieten.

== T

=== Topologie

Ein Feld in der Cluster-Objektspezifikation, das die Definition und Verwaltung der Form der Clusterkontrollebene und der Arbeitsmaschinen von einem einzigen Kontrollpunkt aus ermöglicht. Die Topologie des Clusters basiert auf einer <<_clusterclass,ClusterClass>>.
Manchmal wird sie auch als verwaltete Topologie bezeichnet.

Siehe <<_clusterclass,ClusterClass>>

== W

=== Arbeitslast-Cluster

Ein von einem ClusterAPI-Controller erstellter Cluster, der _kein_ Bootstrap-Cluster ist und für die Verwendung durch Endbenutzer und nicht durch CAPI-Werkzeuge vorgesehen ist.

=== WorkerClass

Eine Sammlung von Vorlagen, die einen Satz von Arbeitsknoten im Cluster definieren. Eine ClusterClass enthält null oder mehr WorkerClass-Definitionen.

Siehe <<_clusterclass,ClusterClass>>
