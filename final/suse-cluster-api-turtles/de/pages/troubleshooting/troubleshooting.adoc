= Rancher Schildkröten Fehlersuche
:revdate: 2025-12-17
:page-revdate: {revdate}

Dieser Leitfaden enthält Schritte zur Fehlerbehebung für Rancher Turtles und Cluster-API (CAPI)-Cluster. Es behandelt häufige Probleme und deren Lösungen, um Ihnen zu helfen, eine stabile und funktionale Umgebung zu erhalten.

Bevor Sie diesen Leitfaden lesen, können Sie zunächst einen Blick in die offizielle https://cluster-api.sigs.k8s.io/user/troubleshooting[`troubleshooting`] Dokumentation des CAPI-Projekts ansehen.

Das Verständnis der Feinheiten sowohl von Rancher Turtles als auch von CAPI ist entscheidend für eine effektive Fehlerbehebung. Dieser Leitfaden soll Sie mit dem Wissen ausstatten, um grundlegende Probleme in Rancher Turtles und CAPI zu diagnostizieren und zu beheben.

== Voraussetzungen

Für eine effektive Fehlersuche bei Rancher Turtles und CAPI-Clustern benötigen Sie:

* Zugriff auf ein Terminal mit kubectl, das für den Zugriff auf den Rancher-Management-Cluster konfiguriert ist.
* Der Krew-Plugin-Manager für kubectl ist installiert.
* Das Helm CLI-Tool ist installiert.

Sie können Krew installieren, indem Sie den Anweisungen in der https://krew.sigs.k8s.io/docs/user-guide/setup/install/[Krew-Installationsanleitung] folgen.

== So überprüfen Sie die Protokolle

Sie können die folgenden Pods anzeigen und ihre Protokolle in den folgenden Namespaces überprüfen:


|===
| Namespace | Einsatz | Beschreibung

| vieh-schildkröten-system
| rancher-turtles-controller-manager
| Handhabt die Rancher-Integration und verwaltet die benutzerdefinierten Ressourcen für Turtles

| fleet-addon-system
| caapf-controller-manager
| Steuert das Cluster API Provider Framework für benutzerdefinierte Infrastrukturanbieter

| vieh-kapi-system
| capi-controller-manager
| Core Cluster API Controller, der die Hauptressourcen des Clusters abgleicht

| rke2-bootstrap-system
| rke2-bootstrap-controller-manager
| Verwaltet die Bootstrap-Konfiguration für RKE2-Knoten

| rke2-control-plane-system
| rke2-control-plane-controller-manager
| Übernimmt die Initialisierung und Verwaltung der Komponenten der RKE2-Kontrollebene

| *-system
| *-controller-manager
| Erledigt die Initialisierung und Verwaltung der Komponenten des Infrastrukturanbieters

| $-bootstrap-system
| $-bootstrap-controller-manager
| Verwaltet die Bootstrap-Konfiguration für benutzerdefinierte Anbieter

| $-Steuerungsplan-System
| $-control-plane-controller-manager
| Erledigt die Initialisierung und Verwaltung von Komponenten der Steuerebene benutzerdefinierter Anbieter
|===

'*' - je nach verwendetem Infrastrukturanbieter '$' - je nach verwendetem Kontrollebene/Bootstrap-Anbieter, z. B. kubeadm oder rke2

Eine Möglichkeit zur Überwachung von Protokollen aus den Cluster-API-Steuerebenen-Pods ist die Verwendung von `kubectl logs` mit dem Label-Selektor.

[source,bash]
----
kubectl logs -l control-plane=controller-manager -n cattle-turtles-system -f
kubectl logs -l control-plane=controller-manager -n cattle-capi-system -f
kubectl logs -l control-plane=controller-manager -n rke2-bootstrap-system -f
kubectl logs -l control-plane=controller-manager -n rke2-control-plane-system -f
----

[CAUTION]
====
Die Verwendung von `kubectl krew` zur Installation des stern-Plugins beinhaltet Software von Drittanbietern, die in Produktionsumgebungen Sicherheitsrisiken darstellen kann. Es ist zwar ein nützliches Tool zur Anzeige von Protokollen, sollte aber hauptsächlich in Entwicklungs- oder Testumgebungen eingesetzt werden. Prüfen Sie Plugins von Drittanbietern immer, bevor Sie sie in sensiblen oder produktiven Clustern installieren.
====

Die beste Methode zur Überwachung von Protokollen aus verschiedenen Pods ist die folgende.

[source,bash]
----
kubectl krew install stern
----

Dann können Sie Protokolle von allen Pods in Echtzeit verfolgen.

[source,bash]
----
kubectl stern . -n cattle-turtles-system -n rke2-bootstrap-system -n rke2-control-plane-system -n cattle-capi-system
----

oder

[source,bash]
----
kubectl stern -A -l control-plane=controller-manager
----

== Wie verwalte ich Rancher Schildkröten und CAPI-Ressourcen?

*Rancher- und Cluster-API-Ressourcen auflisten*

Benutzer fragen oft, wie sie mit langen Listen verschiedener CRDs umgehen können, die von CAPI und Rancher erstellt wurden. Hier sind einige Ideen, wie Sie damit umgehen können.

. Listen Sie alle CAPI- und Rancher-bezogenen Berechtigungsnachweise auf.
+
[source,bash]
----
kubectl api-resources | grep 'cattle.io\|cluster.x-k8s.io'
----
+
. Erfahren Sie mehr über die Ressource. Hier erhalten Sie eine ausführliche Erklärung, was das Programm macht und welche Konfigurationsoptionen es gibt.
+
[source,bash]
----
kubectl api-resources | grep clusterclass
kubectl explain clusterclass.cluster.x-k8s.io
----
+
. Dann erhalten Sie eine Beschreibung der einzelnen Felder und Konfigurationsabschnitte.
+
[source,bash]
----
kubectl explain clusterclass.spec
kubectl explain clusterclass.spec.controlPlane
kubectl explain clusterclasses.spec.controlPlane.machineInfrastructure
----

*Vorhandene Ressourcen für Rancher- und Cluster-API-Objekte auflisten*

Der häufigste Fehler besteht darin, den Namespace `default` in Kubernetes für Ihre Arbeit zu verwenden. Das macht die Konfiguration unübersichtlich und erschwert die Verwaltung und Fehlerbehebung. Wir empfehlen dringend, separate Namespaces für die Bereitstellung und Erstellung von CAPI-Downstream-Clustern zu verwenden und die Verwendung von Kubernetes-System-Namespaces wie `default` und `kube-*` zu vermeiden.

Wenn Sie Vorlagen mit `kubectl` anwenden, geben Sie immer `-n <NAMESPACE>` an, um Ressourcen an einem bekannten Ort bereitzustellen. Mit diesem Kommando können Sie in *KUBECONFIG* zwischen den Namensräumen wechseln:

[source,bash]
----
kubectl config view
kubectl config set-context --current --namespace <NAMESPACE>
----

Mit getrennten Namensräumen können Sie Ressourcen mit dieser Methode leicht verwalten.

== So aktivieren Sie den Debug-Modus für {product_name} und CAPI-Anbieter

* {product_name} - Um den Debug-Modus für {product_name} zu aktivieren, müssen Sie `cattle-system/rancher-config` configmap wie folgt ändern:
+
[source,bash]
----
kubectl patch configmap -n cattle-system rancher-config --type merge --patch '{"data":{"rancher-turtles": "managerArguments:\n  - --v=5\n"}}'
----
(5 ist gleichbedeutend mit DEBUG)
+
Dies führt dazu, dass der Turtles-Pod neu gestartet werden muss, um die neue Konfiguration zu verwenden.

* Cluster-API-Provider - Bearbeiten Sie die CAPIProvider-Ressourcen für Provider, bei denen eine Erhöhung der Protokollierungsstufe erforderlich ist. Wechseln Sie zu der von Ihnen gewünschten Stufe:
+
[source,bash]
----
CAPIProvider.Spec.Manager.Verbosity=5
----
(5 ist gleichbedeutend mit DEBUG)

== Wie man Informationen von Rancher Schildkröten und CAPI sammelt

https://github.com/crust-gather/crust-gather[crust-gather] ist ein Projekt, das von Cluster-API-Entwicklern ins Leben gerufen wurde und speziell für das Sammeln von Protokollen und Ressourcenzuständen aus CAPI-Umgebungen konzipiert ist. Es ist ein sicheres und offizielles Tool, das in jeder Art von Umgebung (Produktion, Test oder Entwicklung) verwendet werden kann, um umfassende Diagnoseinformationen für die Fehlersuche zu sammeln.

Sie können es mit Hilfe der folgenden Anweisungen installieren:

[source,bash]
----
kubectl krew install crust-gather

kubectl crust-gather --help
----

Alternativ kann es auch über das Skript *install.sh* eigenständig installiert werden:

[source,bash]
----
curl -sSfL https://github.com/crust-gather/crust-gather/raw/main/install.sh | sh

crust-gather --help
----

Sie können eine Liste von Filtern angeben, um Daten zu sammeln. Standardmäßig wird ein vollständiger Snapshot des Clusters erstellt. Die Daten werden standardmäßig im Verzeichnis crust-gather gespeichert.
Crust-gather akzeptiert eine vordefinierte Filterkonfiguration für Rancher oder den Child-Cluster. Siehe unten für Beispielkonfigurationen.

Beispiel einer YAML-Filterkonfiguration für die Datei `child-crust-gather.yaml`:

[source,yaml]
----
filters:
  - include_kind:
    - Node
    - Namespace
    - CustomResourceDefinition
  - include_group:
    - management.cattle.io/.*
    - provisioning.cattle.io/.*
  - include_namespace:
    - cattle.*
    - fleet.*
    - kube-system
settings:
  insecure_skip_tls_verify: true
  secrets_file: secrets.txt
----

Beispiel einer YAML-Filterkonfiguration für die Datei `rancher-crust-gather.yaml`:

[source,yaml]
----
filters:
  - include_group:
    - .*cluster.x-k8s.io/.*
    - turtles-capi.cattle.io/.*
  - include_kind:
    - CustomResourceDefinition
    - Node
    - Namespace
  - include_group:
    - management.cattle.io/.*
    - provisioning.cattle.io/.*
  - include_namespace:
    - cattle-turtles-system
    - capi-.*
    - cattle.*
    - fleet.*
    - c-.*
    - rke2.*
    - cert-manager
    - kube-system
settings:
  insecure_skip_tls_verify: true
  secrets_file: secrets.txt
----

Stellen Sie sicher, dass kubeconfig auf den richtigen Cluster verweist, und führen Sie dann den folgenden Befehl aus:

[source,bash]
----
kubectl crust-gather collect-from-config -c config.yaml
----

Verwendung über reguläre Flaggen:

[source,bash]
----
kubectl crust-gather collect --include-namespace cattle-turtles-system --include-namespace capi-* --include-namespace cattle* --include-namespace c-* --include-namespace=<any-capi-cluster-namespace> --kubeconfig=<KUBECONFIG>
----

Sie können eine Datei mit Secrets oder Umgebungsvariablen mit Secrets-Strings angeben, um sie auszuschließen. 

Zum Beispiel:

[source,bash]
----
kubectl crust-gather collect -s ENV_WITH_SECRET --secrets-file=secrets.txt
----

Oder schließen Sie alle geheimen Ressourcen von der Sammlung aus: 

[source,bash]
----
kubectl crust-gather collect --exclude-kind Secret
----

== Wie man Rancher Turtles und CAPI-Ressourcen bereinigt

Manchmal kann die Bereinigung Ihrer Infrastruktur fehlschlagen, was zu ausstehenden Ressourcen führen kann. In diesem Fall müssen Sie die Ressourcen manuell entfernen.

Beachten Sie, dass das manuelle Entfernen von Finalizern **eine** manuelle Bereinigung der bereitgestellten Ressourcen durch den Infrastrukturanbieter **erfordert**.

[source,bash]
----
export NAMESPACE=capi-clusters

for RESOURCE in `kubectl get-all -n $NAMESPACE -o name | grep 'cattle.io\|cluster.x-k8s.io'`; 
do 
        echo "Patching $RESOURCE in namespace $NAMESPACE";
        kubectl patch $RESOURCE -n $NAMESPACE -p '{"metadata":{"finalizers":[]}}' --type=merge;
        kubectl delete $RESOURCE -n $NAMESPACE;
done
----

== So deinstallieren Sie Rancher Turtles und das CAPI-Projekt

Um Rancher Turtles und CAPI-Komponenten von Ihrem Management-Cluster zu deinstallieren, führen Sie die folgenden Schritte der Reihe nach aus:

. Löschen Sie zunächst alle nachgelagerten Cluster, die mit CAPI erstellt wurden. Für jedes Cluster:
+
[source,bash]
----
kubectl delete -n capi-clusters cluster.cluster.x-k8s.io cluster1
----
+
[NOTE]
====
Ersetzen Sie `capi-clusters` durch den Namespace, in dem Ihre Cluster bereitgestellt werden, und `cluster1` durch den Namen Ihres Clusters. Warten Sie, bis jeder Cluster vollständig gelöscht ist, bevor Sie mit dem nächsten Schritt fortfahren.
====

. Deinstallieren Sie die Rancher Turtles Helmkarte:
+
[source,bash]
----
helm uninstall -n cattle-turtles-system rancher-turtles
----

. Entfernen Sie alle Webhook-Konfigurationen, die möglicherweise von CAPI-Anbietern erstellt wurden:
+
[source,bash]
----
# List all webhook configurations
kubectl get validatingwebhookconfigurations.admissionregistration.k8s.io

# Delete provider-specific webhooks (examples)
kubectl delete validatingwebhookconfigurations.admissionregistration.k8s.io azureserviceoperator-validating-webhook-configuration
kubectl delete validatingwebhookconfigurations.admissionregistration.k8s.io capz-validating-webhook-configuration
kubectl delete validatingwebhookconfigurations.admissionregistration.k8s.io capi-validating-webhook-configuration
kubectl delete validatingwebhookconfigurations.admissionregistration.k8s.io rke2-webhook-configuration

# Also check and delete mutating webhooks if present
kubectl get mutatingwebhookconfigurations
kubectl delete mutatingwebhookconfigurations [webhook-name]
----

. Bereinigen Sie alle übrig gebliebenen Namespaces und Ressourcen. Die folgenden Namespaces können nach der Deinstallation bestehen bleiben:
   * vieh-schildkröten-system
   * rke2-bootstrap-system
   * rke2-control-plane-system
   * capi-system
   * capz-system (oder andere anbieterspezifische Namensräume wie capv-system, capa-system, usw.)
   * capi-clusters (oder andere Namensräume, in denen Sie Cluster eingesetzt haben)
+
Um diese Namensräume zu entfernen:
+
[source,bash]
----
# First remove any finalizers that might be blocking deletion
for NS in cattle-turtles-system rke2-bootstrap-system rke2-control-plane-system capi-system capz-system capi-clusters; do
  kubectl get namespace $NS -o json | jq '.spec.finalizers = []' | kubectl replace --raw "/api/v1/namespaces/$NS/finalize" -f -
done

# Then delete the namespaces
kubectl delete namespace cattle-turtles-system rke2-bootstrap-system rke2-control-plane-system capi-system capz-system capi-clusters
----

. Entfernen Sie schließlich die CRDs, die sich auf die Cluster-API und Rancher Turtles beziehen:
+
[source,bash]
----
# Delete all Cluster API and Rancher Turtles CRDs
kubectl get crds | grep 'cluster.x-k8s.io\|turtles-capi.cattle.io' | awk '{print $1}' | xargs kubectl delete crd

# Or manually delete them one by one
kubectl delete crd clusters.cluster.x-k8s.io
kubectl delete crd clusterclasses.cluster.x-k8s.io
kubectl delete crd machines.cluster.x-k8s.io
kubectl delete crd machinepools.cluster.x-k8s.io
kubectl delete crd providers.turtles-capi.cattle.io
kubectl delete crd clusterconfigs.turtles-capi.cattle.io
# ... and other related CRDs
----
+
[CAUTION]
====
Vergewissern Sie sich, dass alle Cluster vollständig gelöscht sind, bevor Sie diese CRDs entfernen, da sonst möglicherweise verwaiste Cloud-Ressourcen zurückbleiben.
====
