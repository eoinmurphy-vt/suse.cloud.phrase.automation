=== フリート向けアドオン・プロバイダー
この機能は Cluster API Add-on Provider for Fleet (CAAPF) をインストールし、https://github.com/rancher/fleet[Fleet]との統合を提供し、CAPI プロビジョニングされたクラスタへのアプリケーションの容易なデプロイを可能にします。CAAPFは自動的にFleetを管理クラスタにインストールし、新たにプロビジョニングされたCAPIクラスタをFleetに登録する。これにより、アプリケーションはGitOps、`Bundle` 、`HelmOp` リソース経由でこれらのクラスタにデプロイできる。CAAPFはまた、FleetのClusterGroupを通じて同じClusterClassを参照するCAPIクラスタをグループ化するサポートも含んでおり、実質的に大規模なクラスタ・グループへのアプリケーション配備を簡素化します。この機能はGA段階にあり、デフォルトで有効になっている。

=== エージェントTLSモード
この機能ゲートは、クラスタをインポートするためにクラスタ登録マニフェストを取得するときに、Rancherの`agent-tls-mode` 設定に従うかどうかをTurtlesに指示するために使用されます。有効にすると、タートルズはランチャーの`agent-tls-mode` の設定値を検索し、`system-store` と`strict` のどちらに設定されているかを判断する。`system-store` に設定すると、Let's Encryptのような認証機関によって署名された証明書を含め、オペレーティングシステムのトラストストアに含まれるパブリック認証機関によって生成された証明書を信頼します。`strict` に設定された場合、タートルズはRancherの`cacerts` 設定に含まれる認証局によって生成された証明書のみを信頼する。この設定を無効にすると、タートルズは`system-store` モードで動作するようにフォールバックする。この機能はベータ版で、デフォルトで有効になっている。

=== サーティ・マネージャーなし
この機能ゲートは、CAPI Providersマニフェストを変換し、cert-managerの使用をhttps://github.com/rancher/wrangler/[wranglerに]置き換えます。

特に、バリデーションとミューテート入場のウェブフックは、cert-managerの代わりにwranglerによって生成された証明書を使用するように更新され、cert-managerを使用しない環境を可能にする。

Rancher`CATTLE_SYNC_ONLY_CHANGED_OBJECTS` https://documentation.suse.com/cloudnative/rancher-manager/latest/en/installation-and-upgrade/best-practices/tuning-rancher-at-scale.html#_reducing_event_handler_executions[機能を]使用すると、証明書がデフォルトの有効期限である 1 年後に更新されない可能性があることに注意してください。

この機能を無効にすると、すべてのプロバイダ・マニフェストを元の cert-manager の使用法に戻すことができます。

=== Rancherのデフォルトレジストリを使用する

この機能ゲートは、{product_name} 、CAPIプロバイダーコンポーネントのイメージ参照を構築する際に、Rancherの`system-default-registry` 管理設定を活用するように指示する。有効にすると、{product_name} はこの設定を読み取り、レジストリが構成されている場合は、すべてのプロバイダー・イメージのリポジトリを書き換えて、そのレジストリを使用するようにします。
設定が空の場合、{product_name} 、デフォルトの画像参照に戻る。`ClusterctlConfig` リソースで定義されたイメージオーバーライドが常に優先されます。エアギャップ環境での詳しい使用法については、xref:operator/airgapped.adoc[エアギャップ環境を]参照のこと。この機能はアルファ版で、デフォルトで有効になっている。

== フィーチャー・ステージ

{product_name} Kuberneteshttps://kubernetes.io/docs/reference/command-line-tools-reference/feature-gates/#feature-stages[feature stages]processに従う。フィーチャーは以下のいずれかの段階にある：アルファ、ベータ、またはGA（General Availability）。

=== アルファステージ
- デフォルトでは無効。
- バグが多く、長期的なサポートがない可能性がある。
- APIは予告なく変更される場合がある。
- 短期間のテストクラスタにのみ推奨される。

=== ベータステージ
- デフォルトで常に有効。
- 十分にテストされ、安全だと考えられている。
- サポートは安定しているが、スキーマは将来のリリースで変更される可能性がある。
- 互換性のない変更の可能性があるため、ビジネスクリティカルでない用途に推奨される。

[NOTE]
====
アルファ版およびベータ版の機能に関するフィードバックをお寄せください！
====

=== GAステージ
- 常に有効で、無効にすることはできない。
- フィーチャーゲートは必要ない。
- 今後のリリースで長期的にサポートされる安定したもの。
