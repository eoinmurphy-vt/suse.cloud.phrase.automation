= Solução de problemas das tartarugas Rancher
:revdate: 2025-12-17
:page-revdate: {revdate}

Este guia fornece etapas de solução de problemas para clusters do Rancher Turtles e da API de cluster (CAPI). Ele aborda problemas comuns e suas resoluções para ajudá-lo a manter um ambiente estável e funcional.

Antes de ler este guia, você pode consultar a documentação oficial https://cluster-api.sigs.k8s.io/user/troubleshooting[`troubleshooting`] oficial do projeto CAPI.

Compreender as complexidades do Rancher Turtles e do CAPI é fundamental para a solução eficaz de problemas. Este guia tem como objetivo equipá-lo com o conhecimento necessário para diagnosticar e resolver problemas básicos no Rancher Turtles e no CAPI.

== Pré-requisitos

Para solucionar efetivamente os problemas dos clusters do Rancher Turtles e CAPI, você precisa:

* Acesso a um terminal com o kubectl configurado para acessar o cluster de gerenciamento do Rancher.
* O gerenciador de plug-ins Krew para kubectl instalado.
* A ferramenta Helm CLI instalada.

Você pode instalar o Krew seguindo as instruções do https://krew.sigs.k8s.io/docs/user-guide/setup/install/[guia de instalação do].

== Como verificar os registros

Você pode visualizar os seguintes pods e verificar seus logs nos seguintes namespaces:


|===
| Espaço de nome | Implantação | Descrição

| cattle-turtles-system
| rancher-turtles-controller-manager
| Lida com a integração do Rancher e gerencia os recursos personalizados do Turtles

| fleet-addon-system
| caapf-controller-manager
| Controla a estrutura do provedor de API do cluster para provedores de infraestrutura personalizados

| sistema cattle-capi
| capi-controller-manager
| Controlador da API do Core Cluster que reconcilia os principais recursos do cluster

| rke2-bootstrap-system
| rke2-bootstrap-controller-manager
| Gerencia a configuração de bootstrap para nós RKE2

| rke2-control-plane-system
| rke2-control-plane-controller-manager
| Trata da inicialização e do gerenciamento dos componentes do plano de controle do RKE2

| *Sistema
| *-controller-manager
| Lida com a inicialização e o gerenciamento dos componentes do provedor de infraestrutura

| $-bootstrap-system
| $-bootstrap-controller-manager
| Gerencia a configuração do bootstrap para provedores personalizados

| $-control-plane-system
| $-control-plane-controller-manager
| Trata da inicialização e do gerenciamento dos componentes do plano de controle dos provedores personalizados
|===

'*' - dependendo do provedor de infraestrutura em uso '$' - dependendo do provedor de plano de controle/bootstrap em uso, por exemplo, kubeadm ou rke2

Uma maneira de monitorar os logs dos pods do plano de controle da API do cluster é usar `kubectl logs` com o seletor de rótulo.

[source,bash]
----
kubectl logs -l control-plane=controller-manager -n cattle-turtles-system -f
kubectl logs -l control-plane=controller-manager -n cattle-capi-system -f
kubectl logs -l control-plane=controller-manager -n rke2-bootstrap-system -f
kubectl logs -l control-plane=controller-manager -n rke2-control-plane-system -f
----

[CAUTION]
====
O uso do site `kubectl krew` para instalar o plug-in stern envolve software de terceiros que pode representar riscos de segurança em ambientes de produção. Embora seja uma ferramenta útil para a visualização de registros, ela deve ser usada principalmente em ambientes de desenvolvimento ou teste. Sempre analise os plug-ins de terceiros antes de instalá-los em clusters confidenciais ou de produção.
====

O melhor método para monitorar os registros de diferentes pods é usar o seguinte método.

[source,bash]
----
kubectl krew install stern
----

Em seguida, você pode acompanhar os logs de todos os pods em tempo real.

[source,bash]
----
kubectl stern . -n cattle-turtles-system -n rke2-bootstrap-system -n rke2-control-plane-system -n cattle-capi-system
----

ou

[source,bash]
----
kubectl stern -A -l control-plane=controller-manager
----

== Como gerenciar as tartarugas Rancher e os recursos CAPI

*Listar recursos da API do Rancher e do Cluster*

Os usuários costumam perguntar como lidar com longas listas de diferentes CRDs criadas pelo CAPI e pelo Rancher. Aqui estão algumas ideias de como lidar com isso.

. Liste todas as credenciais relacionadas ao CAPI e ao Rancher.
+
[source,bash]
----
kubectl api-resources | grep 'cattle.io\|cluster.x-k8s.io'
----
+
. Saiba mais sobre o recurso. Você pode obter uma explicação completa sobre o que ele está fazendo ou quais são as opções de configuração.
+
[source,bash]
----
kubectl api-resources | grep clusterclass
kubectl explain clusterclass.cluster.x-k8s.io
----
+
. Em seguida, você recebe uma descrição sobre cada campo e seção de configuração.
+
[source,bash]
----
kubectl explain clusterclass.spec
kubectl explain clusterclass.spec.controlPlane
kubectl explain clusterclasses.spec.controlPlane.machineInfrastructure
----

*Listar recursos de objetos da API do Rancher e do Cluster existentes*

O erro mais comum é usar o namespace `default` no Kubernetes para seu trabalho. Isso torna sua configuração realmente confusa e difícil de gerenciar e solucionar problemas. É altamente recomendável usar namespaces separados para implantar e criar clusters CAPI downstream e evitar o uso de namespaces do sistema Kubernetes, como `default` e `kube-*`.

Ao aplicar modelos com `kubectl`, sempre especifique `-n <NAMESPACE>` para provisionar recursos em um local conhecido. Você pode alternar entre namespaces no *KUBECONFIG* com esse comando:

[source,bash]
----
kubectl config view
kubectl config set-context --current --namespace <NAMESPACE>
----

Com namespaces separados, você pode gerenciar facilmente os recursos seguindo esse método.

[CAUTION]
====
O uso do site `kubectl krew` para instalar o plug-in de linhagem envolve software de terceiros que pode representar riscos de segurança em ambientes de produção. Embora seja uma ferramenta útil para a visualização de registros, ela deve ser usada principalmente em ambientes de desenvolvimento ou teste. Sempre analise os plug-ins de terceiros antes de instalá-los em clusters confidenciais ou de produção.
====

. Instale os plug-ins `get-all` e `lineage` usando `krew` para `kubectl`
+
[source,bash]
----
kubectl krew install lineage
kubectl krew install get-all
----
+
. Em seguida, liste todos os recursos existentes no namespace, por exemplo, `capi-clusters`, onde sua configuração de cluster downstream está implantada.
+
[source,bash]
----
kubectl get-all -n capi-clusters
----
+

== Como ativar o modo de depuração para os provedores {product_name} e CAPI

* {product_name} - Para ativar o modo de depuração para {product_name}, você precisa corrigir o configmap de `cattle-system/rancher-config` da seguinte forma:
+
[source,bash]
----
kubectl patch configmap -n cattle-system rancher-config --type merge --patch '{"data":{"rancher-turtles": "managerArguments:\n  - --v=5\n"}}'
----
(5 é equivalente a DEBUG)
+
Isso faz com que o pod Turtles seja reiniciado para usar a nova configuração.

* Provedores de API de cluster - Edite os recursos CAPIProvider para provedores em que é necessário aumentar o nível de registro. Mude para o nível desejado:
+
[source,bash]
----
CAPIProvider.Spec.Manager.Verbosity=5
----
(5 é equivalente a DEBUG)

== Como coletar informações do Rancher Turtles e do CAPI

https://github.com/crust-gather/crust-gather[O crust-gather] é um projeto criado por desenvolvedores da API do Cluster, projetado especificamente para coletar registros e estados de recursos de ambientes CAPI. É uma ferramenta segura e oficial que pode ser usada em qualquer tipo de ambiente (produção, teste ou desenvolvimento) para coletar informações abrangentes de diagnóstico para solução de problemas.

Você pode instalá-lo por meio das instruções a seguir:

[source,bash]
----
kubectl krew install crust-gather

kubectl crust-gather --help
----

Como alternativa, ele pode ser instalado de forma autônoma por meio do script *install.sh*:

[source,bash]
----
curl -sSfL https://github.com/crust-gather/crust-gather/raw/main/install.sh | sh

crust-gather --help
----

Você pode especificar uma lista de filtros para coletar dados. Por padrão, ele tira um instantâneo completo do cluster. Por padrão, os dados são armazenados no diretório crust-gather.
O Crust-gather aceita uma configuração de filtro predefinida para o Rancher ou o cluster filho. Veja abaixo exemplos de configurações.

Exemplo de configuração de filtro YAML para o arquivo `child-crust-gather.yaml`:

[source,yaml]
----
filters:
  - include_kind:
    - Node
    - Namespace
    - CustomResourceDefinition
  - include_group:
    - management.cattle.io/.*
    - provisioning.cattle.io/.*
  - include_namespace:
    - cattle.*
    - fleet.*
    - kube-system
settings:
  insecure_skip_tls_verify: true
  secrets_file: secrets.txt
----

Exemplo de configuração de filtro YAML para o arquivo `rancher-crust-gather.yaml`:

[source,yaml]
----
filters:
  - include_group:
    - .*cluster.x-k8s.io/.*
    - turtles-capi.cattle.io/.*
  - include_kind:
    - CustomResourceDefinition
    - Node
    - Namespace
  - include_group:
    - management.cattle.io/.*
    - provisioning.cattle.io/.*
  - include_namespace:
    - cattle-turtles-system
    - capi-.*
    - cattle.*
    - fleet.*
    - c-.*
    - rke2.*
    - cert-manager
    - kube-system
settings:
  insecure_skip_tls_verify: true
  secrets_file: secrets.txt
----

Verifique se o kubeconfig aponta para o cluster correto e, em seguida, execute o comando abaixo:

[source,bash]
----
kubectl crust-gather collect-from-config -c config.yaml
----

Uso por meio de sinalizadores regulares:

[source,bash]
----
kubectl crust-gather collect --include-namespace cattle-turtles-system --include-namespace capi-* --include-namespace cattle* --include-namespace c-* --include-namespace=<any-capi-cluster-namespace> --kubeconfig=<KUBECONFIG>
----

Você pode especificar um arquivo com segredos ou variáveis de ambiente com strings de segredos a serem excluídos. 

Por exemplo:

[source,bash]
----
kubectl crust-gather collect -s ENV_WITH_SECRET --secrets-file=secrets.txt
----

Ou exclua todos os recursos secretos da coleta: 

[source,bash]
----
kubectl crust-gather collect --exclude-kind Secret
----

== Como limpar os recursos do Rancher Turtles e do CAPI

Às vezes, a limpeza de sua infraestrutura pode falhar e isso pode levar a recursos pendentes. Nessa situação, você precisa remover os recursos manualmente.

Lembre-se de que a remoção manual dos finalizadores **exige** a limpeza manual dos recursos provisionados pelo provedor de infraestrutura.

[source,bash]
----
export NAMESPACE=capi-clusters

for RESOURCE in `kubectl get-all -n $NAMESPACE -o name | grep 'cattle.io\|cluster.x-k8s.io'`; 
do 
        echo "Patching $RESOURCE in namespace $NAMESPACE";
        kubectl patch $RESOURCE -n $NAMESPACE -p '{"metadata":{"finalizers":[]}}' --type=merge;
        kubectl delete $RESOURCE -n $NAMESPACE;
done
----

== Como desinstalar o Rancher Turtles e o projeto CAPI

Para desinstalar o Rancher Turtles e os componentes CAPI de seu cluster de gerenciamento, siga estas etapas em ordem:

. Primeiro, exclua todos os clusters downstream criados com o CAPI. Para cada cluster:
+
[source,bash]
----
kubectl delete -n capi-clusters cluster.cluster.x-k8s.io cluster1
----
+
[NOTE]
====
Substitua `capi-clusters` pelo namespace em que seus clusters estão implantados e `cluster1` pelo nome do cluster. Aguarde até que cada cluster seja totalmente excluído antes de prosseguir para a próxima etapa.
====

. Desinstale o gráfico do Rancher Turtles Helm:
+
[source,bash]
----
helm uninstall -n cattle-turtles-system rancher-turtles
----

. Remova todas as configurações de webhook que possam ter sido criadas por provedores CAPI:
+
[source,bash]
----
# List all webhook configurations
kubectl get validatingwebhookconfigurations.admissionregistration.k8s.io

# Delete provider-specific webhooks (examples)
kubectl delete validatingwebhookconfigurations.admissionregistration.k8s.io azureserviceoperator-validating-webhook-configuration
kubectl delete validatingwebhookconfigurations.admissionregistration.k8s.io capz-validating-webhook-configuration
kubectl delete validatingwebhookconfigurations.admissionregistration.k8s.io capi-validating-webhook-configuration
kubectl delete validatingwebhookconfigurations.admissionregistration.k8s.io rke2-webhook-configuration

# Also check and delete mutating webhooks if present
kubectl get mutatingwebhookconfigurations
kubectl delete mutatingwebhookconfigurations [webhook-name]
----

. Limpe todos os namespaces e recursos restantes. Os seguintes namespaces podem permanecer após a desinstalação:
   * cattle-turtles-system
   * rke2-bootstrap-system
   * rke2-control-plane-system
   * sistema capi
   * capz-system (ou outros namespaces específicos do provedor, como capv-system, capa-system, etc.)
   * capi-clusters (ou outros namespaces onde você implantou clusters)
+
Para remover esses namespaces:
+
[source,bash]
----
# First remove any finalizers that might be blocking deletion
for NS in cattle-turtles-system rke2-bootstrap-system rke2-control-plane-system capi-system capz-system capi-clusters; do
  kubectl get namespace $NS -o json | jq '.spec.finalizers = []' | kubectl replace --raw "/api/v1/namespaces/$NS/finalize" -f -
done

# Then delete the namespaces
kubectl delete namespace cattle-turtles-system rke2-bootstrap-system rke2-control-plane-system capi-system capz-system capi-clusters
----

. Por fim, remova os CRDs relacionados à API do cluster e ao Rancher Turtles:
+
[source,bash]
----
# Delete all Cluster API and Rancher Turtles CRDs
kubectl get crds | grep 'cluster.x-k8s.io\|turtles-capi.cattle.io' | awk '{print $1}' | xargs kubectl delete crd

# Or manually delete them one by one
kubectl delete crd clusters.cluster.x-k8s.io
kubectl delete crd clusterclasses.cluster.x-k8s.io
kubectl delete crd machines.cluster.x-k8s.io
kubectl delete crd machinepools.cluster.x-k8s.io
kubectl delete crd providers.turtles-capi.cattle.io
kubectl delete crd clusterconfigs.turtles-capi.cattle.io
# ... and other related CRDs
----
+
[CAUTION]
====
Certifique-se de que todos os clusters sejam completamente excluídos antes de remover esses CRDs, ou você poderá deixar recursos de nuvem órfãos.
====
