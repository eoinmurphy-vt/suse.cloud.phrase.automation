= Práticas recomendadas do GitOps para Turtles e CAPI
:revdate: 2025-10-02
:page-revdate: {revdate}

ifeval::["{build-type}" == "product"]
Esta seção fornece as práticas recomendadas para implementar o GitOps com o Turtles e a API de cluster (CAPI), concentrando-se no uso do https://documentation.suse.com/cloudnative/continuous-delivery/v0.12/en/index.html[Continuous Delivery] em combinação com o *Cluster API Addon Provider Fleet*(`CAAPF`) para automatizar o provisionamento de agentes e a integração ao ecossistema `CAPI`.
endif::[]
ifeval::["{build-type}" == "community"]
Esta seção fornece as práticas recomendadas para implementar o GitOps com o Turtles e a API de cluster (CAPI), concentrando-se no uso do https://fleet.rancher.io/[Fleet] em combinação com o *Cluster API Addon Provider Fleet*(`CAAPF`) para automatizar o provisionamento de agentes e a integração ao ecossistema `CAPI`.
endif::[]

Essas dicas são transferíveis e podem ser aplicadas mesmo quando se usa uma solução GitOps diferente.

== Dicas comuns

ifeval::["{build-type}" == "product"]
- **Configurações separadas de clusters CAPI e de classes de cluster:**
  Mantenha as configurações de cluster do CAPI (`clusters/`) distintas das definições do CAPI `ClusterClass` (`templates/`). Essa separação permite que vários clusters sejam provisionados a partir de modelos compartilhados, reduzindo o risco de alterações não intencionais na infraestrutura.
- **Isolar configurações de complementos:**
  Armazene os manifestos do aplicativo em `applications/`, garantindo que eles sejam independentes de `Clusters` e `ClusterClasses`. Isso permite o provisionamento seletivo ou agrupado com base na demanda do usuário ou em seletores de correspondência, evitando a interferência na infraestrutura do cluster. Esteja ciente de que as dependências do Helm chart precisam ser https://documentation.suse.com/cloudnative/continuous-delivery/v0.12/en/gitrepo-content.html#_fleet_yaml[atualizadas] antes do provisionamento do complemento.
- **Utilizar recursos aninhados do GitRepo:**
  Aproveite os recursos https://documentation.suse.com/cloudnative/continuous-delivery/v0.12/en/gitrepo-content.html#_nested_gitrepo_crs[] aninhados `GitRepo` no Fleet para estruturar o conteúdo do repositório de forma eficiente. Essa abordagem permite o provisionamento modular, combinando definições de `ClusterClass`, complementos/aplicativos necessários e configurações de cluster, mantendo a separação das preocupações.
- **Use as ferramentas de integração do GitOps quando disponíveis:**
  Ao configurar uma instalação do GitOps, leve em consideração as ferramentas disponíveis e os recursos que elas oferecem. Um exemplo disso é o https://rancher.github.io/cluster-api-addon-provider-fleet/00_intro.html[`CAAPF`]que automatiza a instalação do Fleet GitOps em clusters do `CAPI` e oferece automação conveniente.
- **Aplique as sobreposições do Kustomize para personalização específica do ambiente:**
  Use `overlays/` para armazenar sobreposições Kustomize para diferentes ambientes. Isso garante que as alterações de configuração, como ajustes de escala, modificações de rede ou upgrades de componentes, possam ser aplicadas dinamicamente sem alterar os manifestos de base.
- **Utilize Compare Patches:**
  O Fleet permite ignorar seletivamente as diferenças de recursos se o pacote contiver regras adequadas para a https://documentation.suse.com/cloudnative/continuous-delivery/v0.12/en/troubleshooting.html#_suse_rancher_prime_continuous_delivery_deployment_stuck_in_modified_state[detecção de] Isso evitaria que os pacotes ficassem presos em um estado modificado.
endif::[]

== Estratégia de separação de pastas

Um repositório bem estruturado simplifica o gerenciamento e melhora a clareza. Recomenda-se a seguinte estratégia de separação de pastas:

----
repo-root/
│── fleet/
│   ├── providers/         # CAPIProvider files to use with templates
│   │   ├── core/          # Core provider configuration
│   │   ├── infra-docker/  # Docker provider configuration
│   │   ├── infra-aws/     # AWS provider configuration
│   ├── clusters/          # Cluster-specific configurations
│   │   ├── prod/          # Production clusters
│   │   ├── staging/       # Staging clusters
│   │   ├── dev/           # Development clusters
│   ├── addons/            # Application-specific manifests focused on cluster bootstrapping, like CNI, CCM, CPI configurations.
│   │   ├── cni/           # Group applications based on purpose. Provide fleet.yaml per each sub-directory to maintain bundle separation.
│   │   ├── ccm/
│   ├── templates/         # CAPI ClusterClass templates for cluster provisioning
│   ├── fleet.yaml         # Fleet bundle configuration
│── overlays/              # Kustomize overlays for different environments
│   ├── prod/.             # Kustomize overlays for production environments
│   ├── staging/.          # Kustomize overlays for staging environments
│   ├── dev/.              # Kustomize overlays for development environments
----

== Configuração de provedores

Os provedores de CAPI definem os componentes de infraestrutura necessários para o provisionamento de clusters. Uma configuração simples pode ser semelhante:

----
repo-root/
│── fleet/
│   ├── providers/
│   │   ├── core/
│   │   |   ├── core.yaml  # Core `CAPIProvider` definition
│   │   |   ├── fleet.yaml # Fleet.yaml definition for the provider setup
----

onde o provedor principal contém:

[source,yaml]
----
apiVersion: turtles-capi.cattle.io/v1alpha1
kind: CAPIProvider
metadata:
  name: cluster-api
spec:
  type: core
----

== Configuração de modelos

Especifique `dependsOn` em `fleet.yaml` para pacotes específicos de `CAPIProvider` ou componentes externos de `GitRepo`. Isso garante que os modelos sejam implementados na ordem correta e evita problemas em que os controladores de componentes estejam ausentes ou não estejam prontos para aceitar as definições do `ClusterClass`.

[source,yaml]
----
namespace: capa-clusters
dependsOn:
  - name: core      # Ensure core bundle is installed first
  - name: infra-aws # Wait for AWS provider installation
----

== Configuração de clusters

Use rótulos para combinar dinamicamente os clusters com os complementos necessários, permitindo a implementação apenas das cargas de trabalho necessárias.

Exemplo de uso do rótulo:

[source,yaml]
----
apiVersion: cluster.x-k8s.io/v1beta1
kind: Cluster
metadata:
  name: aws-cluster
  labels:
    cni: calico
    ccm: aws
    env: dev
----

[WARNING]
====
Recomenda-se separar as definições de cluster dos modelos, pois essa remoção do pacote de cluster permitiria que o controlador CAPI processasse a exclusão na ordem correta. A remoção simultânea dos sites `Clusters` e `ClusterClasses` pode gerar resultados inesperados.
====

== Configuração de complemento

A configuração de complementos é essencial para simplificar o provisionamento de clusters, garantindo que eles atinjam o estado `Ready` sem intervenção manual ou ferramentas de terceiros.

Os complementos de cluster devem definir regras de seleção precisas para fazer a correspondência com clusters adequadamente rotulados.

O exemplo abaixo mostra um cluster que requer a instalação do `Calico` CNI e do `AWS` Cloud Controller Manager (CCM):

[source,yaml]
----
apiVersion: cluster.x-k8s.io/v1beta1
kind: Cluster
metadata:
  name: aws-cluster
  labels:
    cni: calico
    ccm: aws
----

Essa configuração requer dois recursos `GitRepo`, cada um responsável pelo provisionamento de addons com base em rótulos específicos - um para `cni: calico` e outro para `ccm: aws`.

Aqui está um exemplo de um recurso `GitRepo` para implementar a CNI `Calico`:

[source,yaml]
----
apiVersion: fleet.cattle.io/v1alpha1
kind: GitRepo
metadata:
  name: calico
spec:
  branch: main
  paths:
  - /fleet/applications/calico
  repo: https://github.com/rancher-sandbox/cluster-api-addon-provider-fleet.git
  targets:
  - clusterSelector:
      matchLabels:
        cni: calico
        env: dev
----

Isso garante que o `Calico` CNI seja implantado somente em clusters rotulados com `cni: calico` e `env: dev`, permitindo o provisionamento seletivo do ambiente.

=== Modelo de valores do gráfico do leme

A configuração do complemento precisa ajustar dinamicamente a carga de trabalho do `Calico` com base na definição do cluster correspondente e no estado específico do cluster. Isso pode ser feito usando uma configuração `fleet.yaml`, semelhante à estrutura a seguir:

----
repo-root/
│── fleet/
│   ├── applications/
│   │   ├── calico/
│   │   │   ├── fleet.yaml # Fleet configuration for the Calico setup
----

O arquivo `fleet.yaml` define regras de modelagem e `comparePatches` para garantir uma implementação tranquila, independentemente da configuração do cluster. Para saber mais sobre o modelo `CAAPF`, que é aproveitado aqui, consulte a oficial https://rancher.github.io/cluster-api-addon-provider-fleet/04_reference/02_templating-strategy.html[documentação].

[source,yaml]
----
helm:
  releaseName: projectcalico
  repo: https://docs.tigera.io/calico/charts
  chart: tigera-operator
  templateValues:
    installation: |-
      cni:
        type: Calico
        ipam:
          type: HostLocal
      calicoNetwork:
        bgp: Disabled
        mtu: 1350
        ipPools:
          ${- range $cidr := .ClusterValues.Cluster.spec.clusterNetwork.pods.cidrBlocks }
          - cidr: "${ $cidr }"
            encapsulation: None
            natOutgoing: Enabled
            nodeSelector: all()${- end}

diff:
  comparePatches:
  - apiVersion: operator.tigera.io/v1
    kind: Installation
    name: default
    operations:
    - {"op":"remove", "path":"/spec/kubernetesProvider"}
----
