= Mise à jour vers CAPI v1beta2

La version `v0.26.0` de Rancher Turtles adopte le contrat CAPI `v1beta2`, qui est fourni avec CAPI core v1.12.2. Si vous utilisez le provisionnement basé sur CAPI, il y a quelques points à vérifier avant de procéder à la mise à niveau.

Les changements se répartissent en trois domaines :

* xref:#_clusterclass[Comment les ressources du site `ClusterClass` sont-elles structurées]?
* xref:#_cluster[Comment la topologie de `Cluster` fait référence à sa classe.]
* xref:#_rke2_provider_caprke2[Ce qui a changé dans le fournisseur RKE2].

Lisez les sections qui s'appliquent à votre installation avant d'effectuer des modifications.

== Une note sur la compatibilité ascendante

CAPI v1.12 continue à servir les ressources `v1beta1` aux côtés de `v1beta2`. Les ressources existantes de votre cluster continueront à fonctionner sans migration immédiate, car le serveur API passe d'une version à l'autre de manière transparente. Cela dit, tous les nouveaux manifestes que vous écrivez doivent cibler `v1beta2`, et vous devez prévoir de migrer vos manifestes existants au fil du temps. L'API `v1beta1` devrait être supprimée dans une prochaine version de CAPI (août 2026), une fois que l'écosystème aura eu le temps de s'adapter à `v1beta2`.

*Recommandation*: évaluez d'abord si vous êtes prêt à adopter `v1beta2` et agissez en conséquence. Si vous décidez de ne pas migrer tout de suite, créez un plan pour passer à l'utilisation de la nouvelle API afin d'éviter les perturbations à l'avenir.

En outre, la communauté en amont fournit un guide de migration officiel :

* https://cluster-api.sigs.k8s.io/developer/providers/migrations/v1.10-to-v1.11[CAPI v1.10 à v1.11]: cette version couvre toutes les modifications apportées au cœur du CAPI, de `v1beta1` à `v1beta2`.
* https://cluster-api.sigs.k8s.io/developer/providers/migrations/v1.11-to-v1.12[CAPI v1.11 à v1.12]: la plupart des changements dans CAPI v1.12 peuvent être considérés comme mineurs.

== ClusterClass

=== Références du modèle

Dans `v1beta1`, le champ qui pointe vers les objets du modèle `ClusterClass` est appelé `ref`. Dans le site `v1beta2`, il est appelé `templateRef`. Ceci s'applique à `spec.infrastructure`, `spec.controlPlane`, et `spec.controlPlane.machineInfrastructure`.

Avant :

[,yaml]
----
spec:
  infrastructure:
    ref:
      apiVersion: infrastructure.cluster.x-k8s.io/v1beta1
      kind: DockerClusterTemplate
      name: my-cluster-template
----

Après :

[,yaml]
----
spec:
  infrastructure:
    templateRef:
      apiVersion: infrastructure.cluster.x-k8s.io/v1beta2
      kind: DockerClusterTemplate
      name: my-cluster-template
----

=== Section des travailleurs

Dans `v1beta1`, les références d'amorçage et d'infrastructure pour les déploiements de machines étaient imbriquées dans un champ `template`. Cette clé racine a disparu dans `v1beta2`. `bootstrap` et `infrastructure` sont maintenant des enfants directs de l'entrée de déploiement de la machine.

Avant :

[,yaml]
----
workers:
  machineDeployments:
    - class: default-worker
      template:
        bootstrap:
          ref:
            apiVersion: bootstrap.cluster.x-k8s.io/v1beta1
            kind: KubeadmConfigTemplate
            name: my-bootstrap-template
        infrastructure:
          ref:
            apiVersion: infrastructure.cluster.x-k8s.io/v1beta1
            kind: DockerMachineTemplate
            name: my-worker-template
----

Après :

[,yaml]
----
workers:
  machineDeployments:
    - class: default-worker
      bootstrap:
        templateRef:
          apiVersion: bootstrap.cluster.x-k8s.io/v1beta2
          kind: KubeadmConfigTemplate
          name: my-bootstrap-template
      infrastructure:
        templateRef:
          apiVersion: infrastructure.cluster.x-k8s.io/v1beta2
          kind: DockerMachineTemplate
          name: my-worker-template
----

== Groupement d'entreprises

La ressource `Cluster` fait désormais référence à sa ressource `ClusterClass` par le biais d'un objet structuré plutôt que d'une chaîne de caractères en ligne. Dans `v1beta1`, le nom de la classe et l'espace de noms ont été spécifiés comme étant `spec.topology.class` et `spec.topology.classNamespace`. Dans `v1beta2`, les deux champs sont remplacés par un seul objet `spec.topology.classRef`.

Avant :

[,yaml]
----
apiVersion: cluster.x-k8s.io/v1beta1
kind: Cluster
spec:
  topology:
    class: my-clusterclass
    classNamespace: capi-system
----

Après :

[,yaml]
----
apiVersion: cluster.x-k8s.io/v1beta2
kind: Cluster
spec:
  topology:
    classRef:
      name: my-clusterclass
      namespace: capi-system
----

Le reste de la spécification `Cluster` (paramètres du réseau, `controlPlane.replicas`, `workers.machineDeployments`, `variables`, `version`, etc.) reste inchangé.

== Fournisseur de RKE2 (CAPRKE2)

Cette version intègre CAPRKE2 v0.23.1, en remplacement de la v0.22.1. L'API `v1beta2` pour les ressources RKE2 bootstrap et control-plane a été introduite dans la version 0.22.0 en même temps que la prise en charge de CAPI v1.11, et la version 0.23.0 a complété ce travail en passant à CAPI v1.12.2. Les versions v0.22.1 et v0.23.1 sont des correctifs qui s'ajoutent à ces versions mineures respectives.

*Remarque*: lisez la documentation du fournisseur sur les modifications de l'API dans le link:https://caprke2.docs.rancher.com/05_reference/01_api_versions.html#v1beta1-to-v1beta2[livre RKE2 de].

=== Nouvelle API v1beta2

CAPRKE2 propose désormais sa propre API `v1beta2` pour `RKE2ControlPlane`, `RKE2ControlPlaneTemplate`, et `RKE2Config`/`RKE2ConfigTemplate`. L'API `v1beta1` reste disponible mais plusieurs champs qui avaient été marqués comme obsolètes dans `v1beta1` sont maintenant supprimés dans `v1beta2`.

=== Champs supprimés dans RKE2ControlPlane

Les champs suivants ont été supprimés de `RKE2ControlPlane.spec` dans `v1beta2`:

* `infrastructureRef` - Utilisez plutôt `spec.machineTemplate.spec.infrastructureRef`.
* `nodeDrainTimeout` - Utilisez plutôt `spec.machineTemplate.spec.deletion.nodeDrainTimeout`.

=== Renommer les champs de délai d'attente

Les champs de délai d'attente dans `RKE2ControlPlaneMachineTemplate` ont été déplacés dans un sous-objet `deletion` et renommés. Ils ont également changé de type, passant de `metav1.Duration` à `int32`, et attendent désormais des valeurs exprimées en secondes plutôt qu'une chaîne de durée.

|===
| v1beta1 | v1beta2

| `nodeDrainTimeout`
| `deletion.nodeDrainTimeoutSeconds`

| `nodeVolumeDetachTimeout`
| `deletion.nodeVolumeDetachTimeoutSeconds`

| `nodeDeletionTimeout`
| `deletion.nodeDeletionTimeoutSeconds`
|===

L'objet `RKE2ControlPlaneMachineTemplate` nécessite désormais un champ `spec`.

=== v1alpha1 deprecation

L'API `v1alpha1` a été supprimée dans CAPRKE2 v0.22.0. Si vous utilisez encore les ressources `v1alpha1`, le moment est venu de migrer vers au moins `v1beta1`.

== Mesures recommandées avant la mise à niveau

. Mettez à jour tous les manifestes `ClusterClass` que vous maintenez : renommez `ref` en `templateRef`, supprimez le wrapper `template` dans la section des travailleurs, et mettez à jour les chaînes du noyau CAPI `apiVersion` en `v1beta2`.
. Mettre à jour les manifestes `Cluster`: remplacer `spec.topology.class` et `spec.topology.classNamespace` par `spec.topology.classRef.name` et `spec.topology.classRef.namespace`.
. Si vous utilisez les ressources `v1beta2` de CAPRKE2, supprimez tous les champs directs `infrastructureRef` ou `nodeDrainTimeout` de `RKE2ControlPlane.spec` et déplacez-les vers `spec.machineTemplate.spec`.
. Consultez les exemples mis à jour sur `examples/clusterclasses/` pour obtenir des manifestes complets et fonctionnels pour chaque fournisseur CAPI certifié.
