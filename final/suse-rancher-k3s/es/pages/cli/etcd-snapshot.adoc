= k3s etcd-snapshot
:page-languages: [en, ja, ko, zh]
:revdate: 2025-09-17
:page-revdate: {revdate}

Esta página describe cómo utilizar la herramienta CLI `k3s etcd-snapshot` para gestionar instantáneas etcd y cómo restaurar desde una instantánea etcd.

Las instantáneas de K3s etcd se almacenan en el sistema de archivos del nodo y, opcionalmente, pueden cargarse en un almacén de objetos compatible con S3 para escenarios de recuperación ante desastres. Las instantáneas pueden automatizarse de forma periódica o tomarse manualmente a demanda. La herramienta CLI `k3s etcd-snapshot` ofrece un conjunto de subcomandos que pueden utilizarse para crear, eliminar y gestionar instantáneas.

|===
| Subcomando | Descripción

| `delete`
| Borrar instantánea(s) dada(s)

| `ls`, `list`, `l`
| Lista de instantáneas

| `prune`
| Eliminar instantáneas que superen el recuento de retención configurado.

| `save`
| Activar una instantánea de etcd a petición
|===

Para obtener información adicional sobre los subcomandos de instantáneas etcd, ejecute `k3s etcd-snapshot --help`.

== Creación de instantáneas

[tabs,sync-group-id=snapshots]
======
Programado::
+
--
Las instantáneas programadas están activadas por defecto, a las 00:00 y a las 12:00 hora del sistema, con 5 instantáneas retenidas. Las instantáneas programadas tienen un nombre que empieza por `etcd-snapshot`, seguido del nombre del nodo y la marca de tiempo.

Las siguientes opciones controlan el funcionamiento de las instantáneas programadas:

|===
| Bandera | Descripción

| `--etcd-disable-snapshots`
| Desactivar las instantáneas programadas.

| `--etcd-snapshot-name`
| Establece el nombre base de las instantáneas programadas de etcd. (Por defecto: `etcd-snapshot`)

| `--etcd-snapshot-compress`
| Comprime las instantáneas de etcd.

| `--etcd-snapshot-dir`
| Directorio para guardar las instantáneas de la base de datos. (Ubicación por defecto: `$<data-dir>/db/snapshots`)

| `--etcd-snapshot-retention`
| Número de instantáneas a conservar. (Por defecto: 5)

| `--etcd-snapshot-schedule-cron`
| Tiempo de intervalo de instantáneas en cron spec. Por ejemplo, cada 5 horas `0 */5 * * *` (por defecto: `0 */12 * * *`).
|===

El valor por defecto de la dirección de datos es `/var/lib/rancher/k3s` y puede cambiarse independientemente configurando el indicador `--data-dir`.

Las instantáneas programadas se guardan en la ruta establecida por el valor `--etcd-snapshot-dir` del servidor. Si desea que se repliquen en almacenes de objetos compatibles con S3, consulte las xref:#_s3_compatible_object_store_support[opciones de configuración de S3].
--

A la carta::
+
--
Las instantáneas pueden guardarse manualmente ejecutando el comando `k3s etcd-snapshot save`. No hay retención para estas instantáneas bajo demanda y el usuario necesita eliminarlas manualmente utilizando los comandos `k3s etcd-snapshot delete` o `k3s etcd-snapshot prune`. Las instantáneas bajo demanda tienen un nombre que empieza por `on-demand`, seguido del nombre del nodo y la marca de tiempo.

Las siguientes opciones controlan el funcionamiento de las instantáneas a petición:

|===
| Bandera | Descripción

| `--name`
| Establece el nombre base de las instantáneas bajo demanda de etcd. (Por defecto: `on-demand`)

| `--etcd-snapshot-compress`
| Comprime las instantáneas de etcd.

| `--etcd-snapshot-dir`
| Directorio para guardar las instantáneas de la base de datos. (Ubicación por defecto: `$<data-dir>/db/snapshots`)
|===

El valor por defecto de la dirección de datos es `/var/lib/rancher/k3s` y puede cambiarse independientemente configurando el indicador `--data-dir`.

El indicador `--name` sólo puede activarse cuando se ejecuta el comando `k3s etcd-snapshot save`. Los otros dos también pueden formar parte del xref:installation/configuration.adoc#_configuration_file[archivo de configuración] de `k3s server`.

Las instantáneas bajo demanda se guardan en la ruta establecida por el valor `--etcd-snapshot-dir` del servidor. Si desea que se repliquen en almacenes de objetos compatibles con S3, consulte las xref:#_s3_compatible_object_store_support[opciones de configuración de S3].
--
======

== Borrar instantáneas

Las instantáneas programadas se eliminan automáticamente cuando el número de instantáneas supera el recuento de retención configurado (5 por defecto). Las instantáneas más antiguas se eliminan primero. 

Para eliminar manualmente instantáneas programadas o instantáneas bajo demanda, puede utilizar el comando `k3s etcd-snapshot delete`:

[,bash]
----
k3s etcd-snapshot delete <SNAPSHOT-NAME-1> <SNAPSHOT-NAME-2> ...
----

El subcomando `prune` elimina las instantáneas que coinciden con el prefijo del nombre (`on-demand` por defecto) y superan el recuento de retención configurado. Incluye el indicador `--snapshot-retention` para establecer el recuento de retenciones. Para las instantáneas programadas, anula la política de retención predeterminada. Las instantáneas bajo demanda no tienen política de retención, por lo que se requiere este indicador.

Reduzca las instantáneas "a la carta" a una cantidad menor:

[,bash]
----
k3s etcd-snapshot prune --snapshot-retention  <NUM-OF-SNAPSHOTS-TO-RETAIN>
----

Reduzca las instantáneas "programadas" a una cantidad menor:

[,bash]
----
k3s etcd-snapshot prune --name etcd-snapshot --etcd-snapshot-retention <NUM-OF-SNAPSHOTS-TO-RETAIN>
----

=== Soporte de almacén de objetos compatible con S3

K3s admite la replicación de instantáneas de etcd y la restauración de instantáneas de etcd desde almacenes de objetos compatibles con S3. La compatibilidad con S3 está disponible tanto para instantáneas bajo demanda como programadas.

|===
| Bandera | Descripción

| `--etcd-s3`
| Activar copia de seguridad en S3

| `--etcd-s3-endpoint`
| URL del punto final S3

| `--etcd-s3-endpoint-ca`
| Certificado de CA personalizado de S3 para conectarse al punto final de S3

| `--etcd-s3-skip-ssl-verify`
| Desactiva la validación de certificados S3 SSL

| `--etcd-s3-access-key`
| Clave de acceso S3

| `--etcd-s3-secret-key`
| Clave secreta S3

| `--etcd-s3-bucket`
| Nombre del bucket S3

| `--etcd-s3-region`
| Región S3 / ubicación del cubo (opcional). Por defecto `us-east-1`.

| `--etcd-s3-folder`
| Carpeta S3

| `--etcd-s3-proxy`
| Servidor proxy que se utilizará al conectarse a S3, anulando cualquier variable de entorno liberada por el proxy.

| `--etcd-s3-insecure`
| Desactiva S3 sobre HTTPS

| `--etcd-s3-timeout`
| Tiempo de espera de S3 (por defecto: `5m0s`)

| `--etcd-s3-config-secret`
| Nombre del secreto en el espacio de nombres kube-system utilizado para configurar S3, si etcd-s3 está habilitado y no se establecen otras opciones de etcd-s3.
|===

Por ejemplo, así es como funcionaría la creación y eliminación de instantáneas etcd bajo demanda en S3:

[,shell-session]
----
$ k3s etcd-snapshot --s3 --s3-bucket=test-bucket --s3-access-key=test --s3-secret-key=secret save
INFO[0155] Snapshot on-demand-server-0-1753178523 saved. 
INFO[0155] Snapshot on-demand-server-0-1753178523 saved. 

$ k3s etcd-snapshot --s3 --s3-bucket=test-bucket --s3-access-key=test --s3-secret-key=secret ls
Name                              Location                                                                          Size    Created
on-demand-server-0-1753178523     s3://test-bucket/test-folder/on-demand-server-0-1753178523                        5062688 2025-07-22T10:02:03Z
on-demand-server-0-1753178523     file:///var/lib/rancher/k3s/server/db/snapshots/on-demand-server-0-1753178523     5062688 2025-07-22T10:02:03Z

$ k3s etcd-snapshot --s3 --s3-bucket=test-bucket --s3-access-key=test --s3-secret-key=secret delete on-demand-server-0-1753178523
INFO[0000] Snapshot on-demand-server-0-1753178523 deleted.

$ k3s etcd-snapshot --s3 --s3-bucket=test-bucket --s3-access-key=test --s3-secret-key=secret ls
Name                              Location                                                                          Size    Created
----

=== Soporte secreto de configuración S3

[NOTE]
.Versión Puerta
====
La compatibilidad con S3 Configuration Secret está disponible a partir de las versiones de agosto de 2024: v1.30.4+k3s1, v1.29.8+k3s1, v1.28.13+k3s1.
====

K3s admite la lectura de la configuración de instantáneas S3 de etcd desde un Kubernetes Secret.
Esto puede ser preferible a hardcoding credenciales en K3s CLI banderas o archivos de configuración por razones de seguridad, o si las credenciales deben ser rotados sin reiniciar K3s.
Para pasar la configuración de instantáneas S3 a través de un Secreto, inicie K3s con `--etcd-s3` y `--etcd-s3-config-secret=<SECRET-NAME>`.
No es necesario que el secreto exista cuando se inicie K3s, pero se comprobará cada vez que se realice una operación de guardar/lista/eliminar/preparar instantáneas.

El S3 config Secret no se puede utilizar al restaurar una instantánea, ya que el apiserver no está disponible para proporcionar el secreto durante una restauración.
La configuración de S3 debe pasarse a través de la CLI al restaurar una instantánea almacenada en S3.

[NOTE]
====
Pase sólo las banderas `--etcd-s3` y `--etcd-s3-config-secret` para activar el Secreto.
Si se establece cualquier otra bandera de configuración S3, el Secreto será ignorado.
====

Las claves del Secreto corresponden a los indicadores de la CLI `--etcd-s3-*` enumerados anteriormente.
La clave `etcd-s3-endpoint-ca` acepta un paquete de CA codificado con PEM, o la clave `etcd-s3-endpoint-ca-name` puede utilizarse para especificar el nombre de un ConfigMap en el espacio de nombres `kube-system` que contenga uno o más paquetes de CA codificados con PEM.

[,yaml]
----
apiVersion: v1
kind: Secret
metadata:
  name: k3s-etcd-snapshot-s3-config
  namespace: kube-system
type: etcd.k3s.cattle.io/s3-config-secret
stringData:
  etcd-s3-endpoint: ""
  etcd-s3-endpoint-ca: ""
  etcd-s3-endpoint-ca-name: ""
  etcd-s3-skip-ssl-verify: "false"
  etcd-s3-access-key: "AWS_ACCESS_KEY_ID"
  etcd-s3-secret-key: "AWS_SECRET_ACCESS_KEY"
  etcd-s3-bucket: "bucket"
  etcd-s3-folder: "folder"
  etcd-s3-region: "us-east-1"
  etcd-s3-insecure: "false"
  etcd-s3-timeout: "5m"
  etcd-s3-proxy: ""
----

== Restaurar instantáneas

El K3s ejecuta varios pasos al restaurar una instantánea:

. Si la instantánea se almacena en S3, el archivo se descarga en el directorio de instantáneas.
. Si la instantánea está comprimida, se descomprime.
. Si está presente, los archivos actuales de la base de datos etcd se mueven a `<data-dir>/server/db/etcd-old-$TIMESTAMP/`.
. El contenido de la instantánea se extrae al disco y se verifica la suma de comprobación.
. Etcd se inicia, y todos los miembros del clúster etcd excepto el nodo actual se eliminan del clúster.
. Los certificados CA y otros datos confidenciales se extraen del almacén de datos y se escriben en el disco, para su uso posterior.
. La restauración se ha completado, y el K3s puede reiniciarse y utilizarse normalmente en el servidor donde se realizó la restauración.
. (opcional) Los agentes y servidores del plano de control pueden iniciarse normalmente. 
. (opcional) Los servidores Etcd pueden reiniciarse para volver a unirse al clúster tras eliminar los archivos de base de datos antiguos.

Al restaurar una instantánea, no es necesario utilizar la misma versión de K3s que la creó; también es aceptable una versión menor superior.

[#_snapshot_restore_steps]
=== Pasos de restauración de instantáneas

Seleccione la pestaña que corresponda a la configuración de su clúster.

[tabs,sync-group-id=etcdsnap]
======
Servidor único::
+
--
. Detenga el servicio K3s:
+
[,bash]
----
systemctl stop k3s
----

. Ejecute `k3s server` con la bandera `--cluster-reset`, y `--cluster-reset-restore-path` indicando la ruta a la instantánea a restaurar.
   Si la instantánea se almacena en S3, proporcione los indicadores de configuración de S3 (`--etcd-s3`, `--etcd-s3-bucket`, etc.) y proporcione únicamente el nombre de archivo de la instantánea como ruta de restauración.
+
[NOTE]
====
El uso de la bandera `--cluster-reset` sin especificar una instantánea para restaurar simplemente restablece el clúster etcd a un solo miembro sin restaurar una instantánea.
====
+
[,bash]
----
k3s server \
  --cluster-reset \
  --cluster-reset-restore-path=<PATH-TO-SNAPSHOT>
----
+
**Resultado:** K3s restaura la instantánea y restablece la pertenencia al clúster, luego imprime un mensaje indicando que está listo para ser reiniciado:
`Managed etcd cluster membership has been reset, restart without --cluster-reset flag now.`

. Empieza de nuevo con los K3:
+
[,bash]
----
systemctl start k3s
----

Si se define una configuración de copia de seguridad etcd-s3 en el archivo de configuración de K3s, la restauración de K3s intenta extraer el archivo de instantáneas del bucket S3 configurado. En este caso, sólo debe introducirse el nombre del archivo de instantánea en el argumento `--cluster-reset-restore-path`. Para restaurar a partir de un archivo de instantánea local, donde hay una configuración de copia de seguridad etcd-s3, añada el argumento `--etcd-s3=false` y pase la ruta completa al archivo de instantánea local en el argumento `--cluster-reset-restore-path`.

Como mecanismo de seguridad, cuando K3s reinicia el clúster, crea un archivo vacío en `/var/lib/rancher/k3s/server/db/reset-flag` que impide que los usuarios ejecuten accidentalmente varios reinicios de clúster seguidos. Este archivo se borra cuando el K3s se inicia normalmente.
--

Varios servidores::
+
--
En este ejemplo hay 3 servidores, `S1`, `S2`, y `S3`. La instantánea se encuentra en `S1`.

. Detener los K3 en todos los servidores:
+
[,bash]
----
systemctl stop k3s
----

. En S1, ejecute `k3s server` con la opción `--cluster-reset`, y `--cluster-reset-restore-path` indicando la ruta a la instantánea a restaurar.
   Si la instantánea se almacena en S3, proporcione los indicadores de configuración de S3 (`--etcd-s3`, `--etcd-s3-bucket`, etc.) y proporcione únicamente el nombre de archivo de la instantánea como ruta de restauración.
+
[NOTE]
====
El uso de la bandera `--cluster-reset` sin especificar una instantánea para restaurar simplemente restablece el clúster etcd a un solo miembro sin restaurar una instantánea.
====
+
[,bash]
----
k3s server \
  --cluster-reset \
  --cluster-reset-restore-path=<PATH-TO-SNAPSHOT>
----
+
**Resultado:** K3s restaura la instantánea y restablece la pertenencia al clúster, luego imprime un mensaje indicando que está listo para ser reiniciado:
`Managed etcd cluster membership has been reset, restart without --cluster-reset flag now.`
`Backup and delete $\{datadir}/server/db on each peer etcd server and rejoin the nodes.`

. En S1, empieza K3s otra vez:
+
[,bash]
----
systemctl start k3s
----

. En S2 y S3, elimine el directorio de datos, `/var/lib/rancher/k3s/server/db/`:
+
[,bash]
----
rm -rf /var/lib/rancher/k3s/server/db/
----

. En S2 y S3, inicie de nuevo los K3 para que se unan al clúster restaurado:
+
[,bash]
----
systemctl start k3s
----

Si se define una configuración de copia de seguridad etcd-s3 en el archivo de configuración de K3s, la restauración de K3s intenta extraer el archivo de instantáneas del bucket S3 configurado. En este caso, sólo debe introducirse el nombre del archivo de instantánea en el argumento `--cluster-reset-restore-path`. Para restaurar a partir de un archivo de instantánea local, donde hay una configuración de copia de seguridad etcd-s3, añada el argumento `--etcd-s3=false` y pase la ruta completa al archivo de instantánea local en el argumento `--cluster-reset-restore-path`.

Como mecanismo de seguridad, cuando K3s reinicia el clúster, crea un archivo vacío en `/var/lib/rancher/k3s/server/db/reset-flag` que impide que los usuarios ejecuten accidentalmente varios reinicios de clúster seguidos. Este archivo se borra cuando el K3s se inicia normalmente.
--

==== Restauración en nuevos hosts

Es posible restaurar una instantánea etcd en un host diferente al que se tomó. Al hacerlo, debe pasar el xref:cli/token.adoc#_server[token de servidor] que se utilizó originalmente al tomar la instantánea, ya que se utiliza para descifrar los datos de arranque dentro de la instantánea. El proceso es el mismo que el anterior pero cambiando el paso 2 por:

. En el nodo que tomó la instantánea guarde el valor de: `/var/lib/rancher/k3s/server/token`. Esto es `<BACKED-UP-TOKEN-VALUE>` en el paso 3.
. Copie la instantánea en el nuevo nodo. La ruta en el nodo es `<PATH-TO-SNAPSHOT>` en el paso 3.
. Inicie la restauración desde la instantánea en el primer nodo servidor con los siguientes comandos:
+
[,bash]
----
k3s server \
  --cluster-reset \
  --cluster-reset-restore-path=<PATH-TO-SNAPSHOT>
  --token=<BACKED-UP-TOKEN-VALUE>
----
+
El valor del token también puede establecerse en el archivo de configuración del K3s.


[WARNING]
======
. Los recursos de nodo también se incluyen en la instantánea etcd. Si restaura en un nuevo conjunto de nodos, deberá eliminar manualmente los nodos antiguos que ya no estén presentes en el clúster.
. Si hay un token establecido en el archivo de configuración de K3s, asegúrese de que es el mismo que el de `<BACKED-UP-TOKEN-VALUE>`, de lo contrario K3s no se iniciará.
======

== ETCDSnapshotFile Recursos personalizados

[NOTE]
.Versión Puerta

Los ETCDSnapshotFiles están disponibles a partir de las versiones de noviembre de 2023: v1.28.4+k3s2, v1.27.8+k3s2, v1.26.11+k3s2, v1.25.16+k3s4.
====

Las instantáneas se pueden ver de forma remota utilizando cualquier cliente de Kubernetes mediante la enumeración o descripción de los recursos `ETCDSnapshotFile` incluidos en el clúster.
