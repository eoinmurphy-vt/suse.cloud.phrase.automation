= BD externa de alta disponibilidad
:page-languages: [en, ja, ko, zh]
:revdate: 2026-03-03
:page-revdate: {revdate}

Esta sección describe cómo instalar un clúster de alta disponibilidad {k3s-product-name} con una base de datos externa.

[NOTE]
====
Para implantar rápidamente grandes clústeres de HA, consulte xref:related-projects.adoc[Proyectos relacionados]
====


Los clústeres de un solo servidor pueden satisfacer una variedad de casos de uso, pero para entornos en los que el tiempo de actividad del plano de control de Kubernetes es crítico, puede ejecutar K3 en una configuración de HA. Un clúster HA K3s se compone de:

* Dos o más *nodos de servidor* que servirán a la API de Kubernetes y ejecutarán otros servicios del plano de control.
* Un *almacén* de *datos externo* (en lugar del almacén de datos SQLite integrado que se utiliza en las configuraciones de un solo servidor).
* Opcional: Cero o más *nodos agentes* designados para ejecutar sus aplicaciones y servicios.
* Opcional: Una *dirección de registro fija* para que los nodos agentes se registren en el clúster.

Para más detalles sobre el funcionamiento conjunto de estos componentes, consulte la xref:architecture.adoc#_high_availability_k3s[sección de arquitectura.]

== Esquema de instalación

La configuración de un clúster de HA requiere los siguientes pasos:

=== 1. Crear un almacén de datos externo

En primer lugar, deberá crear un almacén de datos externo para el clúster. Consulte la documentación xref:datastore/datastore.adoc[Opciones de almacén de datos de clúster] para obtener más detalles.

=== 2. Lanzar nodos de servidor

{k3s-product-name} requiere dos o más nodos de servidor para esta configuración de HA. Consulte la guía de xref:installation/requirements.adoc[requisitos] para conocer los requisitos mínimos de la máquina.

Al ejecutar el comando `k3s server` en estos nodos, debe establecer el parámetro `datastore-endpoint` para que K3s sepa cómo conectarse al almacén de datos externo. El parámetro `token` también se puede utilizar para establecer un token determinista al añadir nodos. Cuando esté vacío, este token se generará automáticamente para su uso posterior.

Por ejemplo, se podría utilizar un comando como el siguiente para instalar el servidor K3s con una base de datos MySQL como almacén de datos externo y xref:cli/server.adoc#_cluster_options[establecer un token]:

[,bash]
----
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s sh -s - server \
  --token=SECRET \
  --datastore-endpoint="mysql://username:password@tcp(hostname:3306)/database-name" \
  --tls-san=<FIXED_IP> # Optional, needed if using a fixed registration address
----

Donde `INSTALL_K3S_ARTIFACT_URL` es la https://scc.suse.com/rancher-docs/rancherprime/latest/en/reference-guide.html#prime-artifacts-url[URL de]

El formato del punto final del almacén de datos varía en función del tipo de base de datos. Para más detalles, consulte la sección sobre xref:datastore/datastore.adoc#_datastore_endpoint_format_and_functionality[formatos de punto final de almacén de datos.]

Para configurar certificados TLS al lanzar nodos de servidor, consulte la xref:datastore/datastore.adoc#_external_datastore_configuration_parameters[guía de configuración de almacenes de datos.]

[NOTE]
====
Las mismas opciones de instalación disponibles para las instalaciones de un solo servidor también están disponibles para las instalaciones de alta disponibilidad. Para más detalles, consulte la documentación xref:installation/configuration.adoc[Opciones de configuración].
====


Por defecto, los nodos de servidor serán programables y, por tanto, sus cargas de trabajo podrán lanzarse en ellos. Si desea tener un plano de control dedicado en el que no se ejecuten cargas de trabajo de usuario, puede utilizar taints. El parámetro `node-taint` le permitirá configurar nodos con manchas, por ejemplo `--node-taint CriticalAddonsOnly=true:NoExecute`.

Una vez que haya iniciado el proceso `k3s server` en todos los nodos del servidor, asegúrese de que el clúster se ha iniciado correctamente con `k3s kubectl get nodes`. Debería ver sus nodos de servidor en el estado Listo.

=== 3. Opcional: Unirse a nodos de servidor adicionales

El mismo comando de ejemplo del paso 2 se puede utilizar para unirse a nodos de servidor adicionales, en los que es necesario utilizar el token del primer nodo.

Si el primer nodo servidor se inició sin la bandera CLI `--token` o la variable `K3S_TOKEN`, el valor del token puede recuperarse de cualquier servidor ya unido al clúster:

[,bash]
----
cat /var/lib/rancher/k3s/server/token
----

A continuación, se pueden añadir nodos de servidor adicionales xref:cli/server.adoc#_cluster_options[utilizando el token]:

[,bash]
----
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s sh -s - server \
  --token=SECRET \
  --datastore-endpoint="mysql://username:password@tcp(hostname:3306)/database-name"
----

Hay algunas banderas de configuración que deben ser las mismas en todos los nodos del servidor:

* Banderas relacionadas con la red: `--cluster-dns`, `--cluster-domain`, `--cluster-cidr`, `--service-cidr`
* Banderas que controlan el despliegue de determinados componentes: `--disable-helm-controller`, `--disable-kube-proxy`, `--disable-network-policy` y cualquier componente pasado a `--disable`
* Banderas relacionadas con las características: `--secrets-encryption`

[NOTE]
====
Asegúrese de conservar una copia de este token, ya que es necesario cuando se restaura a partir de una copia de seguridad y se añaden nodos. Anteriormente, K3s no exigía el uso de un token cuando se utilizaban almacenes de datos SQL externos.
====


=== 4. Opcional: Configurar una dirección de registro fija

Los nodos de agente necesitan una URL para registrarse. Puede ser la IP o el nombre de host de cualquier nodo servidor, pero en muchos casos pueden cambiar con el tiempo. Por ejemplo, si ejecuta su clúster en una nube que admite grupos de escalado, los nodos pueden crearse y destruirse con el tiempo, cambiando a IP diferentes del conjunto inicial de nodos de servidor. Lo mejor sería tener un punto final estable frente a los nodos del servidor que no cambie con el tiempo. Este punto final se puede configurar utilizando cualquier número de enfoques, tales como:

* Un equilibrador de carga de capa 4 (TCP)
* Round-robin DNS
* Direcciones IP virtuales o elásticas

Consulte xref:datastore/cluster-loadbalancer.adoc[Cluster Loadbalancer] para ver ejemplos de configuración.

Este endpoint también puede utilizarse para acceder a la API de Kubernetes. Así que puedes, por ejemplo, modificar tu archivo kubeconfig https://kubernetes.io/docs/concepts/configuration/organize-cluster-access-kubeconfig/[de] para que apunte a él en lugar de a un nodo específico.

Para evitar errores de certificado en una configuración de este tipo, debe configurar el servidor con la opción `--tls-san=YOUR_IP_OR_HOSTNAME_HERE`. Esta opción añade un nombre de host o IP adicional como Subject Alternative Name en el certificado TLS, y se puede especificar varias veces si desea acceder a través de la IP y el nombre de host.

=== 5. Opcional: Unir nodos de agentes

Dado que los nodos de servidor K3s son programables por defecto, los nodos de agente no son necesarios para un clúster K3s de HA. Sin embargo, es posible que desee disponer de nodos agentes dedicados para ejecutar sus aplicaciones y servicios.

La unión de nodos de agente en un clúster de HA es igual que la unión de nodos de agente en un clúster de servidor único. Sólo tiene que especificar la URL en la que debe registrarse el agente (ya sea una de las IP del servidor o una dirección de registro fija) y el token que debe utilizar.

[,bash]
----
K3S_TOKEN=SECRET k3s agent --server https://server-or-fixed-registration-address:6443
----
