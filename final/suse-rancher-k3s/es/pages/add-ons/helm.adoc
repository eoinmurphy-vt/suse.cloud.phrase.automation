= Timón
:page-languages: [en, ja, ko, zh]
:revdate: 2026-01-05
:page-revdate: {revdate}
:page-aliases: helm.adoc

Helm es la herramienta de gestión de paquetes preferida para Kubernetes. Los gráficos de Helm proporcionan una sintaxis de plantillas para los documentos de manifiesto YAML de Kubernetes. Con Helm, los desarrolladores o administradores de clústeres pueden crear plantillas configurables conocidas como Charts, en lugar de limitarse a utilizar manifiestos estáticos. Para obtener más información sobre la creación de su propio catálogo Chart, consulte la documentación en https://helm.sh/docs/intro/quickstart/.

K3s no requiere ninguna configuración especial para soportar Helm. Sólo asegúrese de que ha configurado correctamente la ruta kubeconfig según la documentación de xref:cluster-access.adoc[acceso al clúster].

K3s incluye un https://github.com/k3s-io/helm-controller/[controlador Helm] que gestiona la instalación, actualización/reconfiguración y desinstalación de cartas Helm utilizando una definición de recursos personalizada (CRD) HelmChart. Junto con xref:installation/packaged-components.adoc[el despliegue automático de manifiestos AddOn], la instalación de un gráfico Helm en su clúster puede automatizarse mediante la creación de un único archivo en el disco.

== Uso del controlador Helm

El https://github.com/k3s-io/helm-controller#helm-controller[recurso personalizado] captura la mayoría de las opciones que normalmente pasaría a la herramienta de línea de comandos `helm`.

=== Definiciones de los campos de HelmChart

[NOTE]
====
El campo `name` debe seguir las convenciones de nomenclatura de la carta Helm, además de las reglas de Kubernetes para https://kubernetes.io/docs/concepts/overview/working-with-objects/names/[nombres de objetos e IDs de]. Para obtener más información, consulte la https://helm.sh/docs/chart_best_practices/conventions/#chart-names[documentación].
====

|===
| Campo | Por defecto | Descripción | Argumento del timón / Equivalente de la bandera

| metadata.name
|
| Nombre de la carta del timonel
| NOMBRE

| spec.chart
|
| Nombre del gráfico Helm en el repositorio, o URL HTTPS completa al archivo del gráfico (.tgz)
| GRÁFICO

| spec.chartContent
|
| Archivo de gráficos .tgz codificado en base64 - anula spec.chart
| GRÁFICO

| spec.targetNamespace
| por defecto
| Espacio de nombres de destino de Helm Chart
| `--namespace`

| spec.createNamespace
| falso
| Crear espacio de nombres de destino si no está presente
| `--create-namespace`

| versión.spec
|
| Versión de Helm Chart (cuando se instala desde el repositorio)
| `--version`

| spec.repo
|
| URL del repositorio de Helm Chart
| `--repo`

| spec.repoCA
|
| Verifique los certificados de los servidores habilitados para HTTPS que utilizan este paquete de CA. Debe ser una cadena que contenga uno o más certificados de CA codificados en PEM.
| `--ca-file`

| spec.repoCAConfigMap
|
| Referencia a un ConfigMap que contiene Certificados CA en los que Helm debe confiar. Puede utilizarse junto con o en lugar de `repoCA`
| `--ca-file`

| spec.plainHTTP
| falso
| Utilice conexiones HTTP inseguras para la descarga de gráficos.
| `--plain-http`

| spec.insecureSkipTLSVerify
| falso
| Omitir las comprobaciones de certificados TLS para la descarga de gráficos.
| `--insecure-skip-tls-verify`

| spec.helmVersion
| v3
| Versión de Helm a utilizar. Actualmente sólo se admite `v3`.
|

| spec.bootstrap
| Falso
| Establecer en True si este gráfico es necesario para arrancar el cluster (Cloud Controller Manager, etc)
|

| spec.jobImage
|
| Especifique la imagen que se utilizará al instalar la carta de navegación. E.g. rancher/klipper-helm:v0.3.0 .
|

| spec.podSecurityContext
|
| Personalizar https://kubernetes.io/docs/reference/generated/kubernetes-api/v1.32/#podsecuritycontext-v1-core[`v1.PodSecurityContext`] para el pod de trabajo Helm
|

| spec.securityContext
|
| Personalizar https://kubernetes.io/docs/reference/generated/kubernetes-api/v1.32/#securitycontext-v1-core[`v1.SecurityContext`] para los contenedores del pod de trabajo Helm
|

| spec.backOffLimit
| 1000
| Especifique el número de reintentos antes de considerar un trabajo como fallido.
|

| spec.timeout
| 300s
| Tiempo de espera para las operaciones de Helm, como https://pkg.go.dev/time#ParseDuration[cadena de duración] (`300s`, `10m`, `1h`, etc)
| `--timeout`

| spec.failurePolicy
| reinstall
| Configurado en `abort`, en cuyo caso se interrumpe la operación del timón, a la espera de la intervención manual del operador.
|

| spec.authSecret
|
| Referencia a Secret de tipo `kubernetes.io/basic-auth` que contiene las credenciales Basic auth para el repositorio Chart.
|

| spec.authPassCredentials
| falso
| Pasar las credenciales Basic auth a todos los dominios.
| `--pass-credentials`

| spec.dockerRegistrySecret
|
| Referencia a Secret de tipo `kubernetes.io/dockerconfigjson` que contiene las credenciales de autenticación de Docker para el registro basado en OCI que actúa como repositorio de Chart.
|

| spec.set
|
| Anula los valores simples por defecto del Gráfico. Estas tienen prioridad sobre las opciones establecidas a través de valuesContent.
| `--set` / `--set-string`


| spec.valuesContent
|
| Sustitución de valores complejos predeterminados del gráfico mediante el contenido del archivo YAML
| `--values`

| spec.valuesSecrets
|
| Anulación de valores de Gráficos complejos mediante referencias a Secretos externos
| `--values`
|===

Se puede acceder de forma anónima al contenido colocado en `/var/lib/rancher/k3s/server/static/` a través del APIServer de Kubernetes desde dentro del clúster. Esta URL puede planificarse utilizando la variable especial `%{KUBERNETES_API}%` en el campo `spec.chart`. Por ejemplo, el componente empaquetado Traefik carga su gráfico desde `https://%{KUBERNETES_API}%/static/charts/traefik-VERSION.tgz`.

Los valores de los gráficos se utilizan en el siguiente orden, de menor a mayor precedencia:

. Valores por defecto del gráfico
. HelmChart `spec.valuesContent`
. HelmChart `spec.valuesSecrets` por orden de nombre secreto y claves
. HelmChartConfig `spec.valuesContent`
. HelmChartConfig `spec.valuesSecrets` por orden de nombre secreto y claves
. HelmChart `spec.set`

He aquí un ejemplo de cómo podría desplegar Apache desde el repositorio de gráficos de Bitnami, anulando algunos de los valores predeterminados del gráfico. Tenga en cuenta que el propio recurso HelmChart está en el espacio de nombres `kube-system`, pero los recursos del gráfico se desplegarán en el espacio de nombres `web`, que se crea en el mismo manifiesto. Esto puede ser útil si desea mantener sus recursos HelmChart separados de los recursos que despliegan.

[,yaml]
----
apiVersion: v1
kind: Namespace
metadata:
  name: web
---
apiVersion: helm.cattle.io/v1
kind: HelmChart
metadata:
  name: apache
  namespace: kube-system
spec:
  repo: https://charts.bitnami.com/bitnami
  chart: apache
  targetNamespace: web
  valuesContent: |-
    service:
      type: ClusterIP
    ingress:
      enabled: true
      hostname: www.example.com
    metrics:
      enabled: true
----

Un ejemplo de despliegue de un gráfico helm desde un repositorio privado con autenticación:

[,yaml]
----
apiVersion: helm.cattle.io/v1
kind: HelmChart
metadata:
  namespace: kube-system
  name: example-app
spec:
  targetNamespace: example-namespace
  createNamespace: true
  version: v1.2.3
  chart: example-app
  repo: https://secure-repo.example.com
  authSecret:
    name: example-repo-auth
  repoCAConfigMap:
    name: example-repo-ca
  valuesContent: |-
    image:
      tag: v1.2.2
---
apiVersion: v1
kind: Secret
metadata:
  namespace: kube-system
  name: example-repo-auth
type: kubernetes.io/basic-auth
stringData:
  username: user
  password: pass
---
apiVersion: v1
kind: ConfigMap
metadata:
  namespace: kube-system
  name: example-repo-ca
data:
  ca.crt: |-
    -----BEGIN CERTIFICATE-----
    <YOUR CERTIFICATE>
    -----END CERTIFICATE-----
----

=== Valores gráficos de Secretos

Los valores del gráfico se pueden leer de Secretos gestionados externamente, en lugar de almacenar los valores en los campos `spec.set` o `spec.valuesContent`. Esto debe hacerse cuando se transmite información confidencial, como credenciales, a gráficos que no admiten referencias a secretos existentes a través del patrón `existingSecret`.

Al igual que con otros Secretos (`spec.authSecret` y `spec.dockerRegistrySecret`), los Secretos referenciados en `spec.valuesSecrets` deben estar en el mismo espacio de nombres que el HelmChart.

Cada entrada de la lista `valuesSecrets` tiene los siguientes campos:

|===
| Campo | Descripción

| nombre
| El nombre del Secreto. Necesario.

| llaves
| Lista de claves de las que leer valores, los valores se utilizan en el orden de la lista. Necesario.

| ignoreUpdates
| Marque este Secreto como opcional, y no actualice el gráfico si el Secreto cambia. Opcional, por defecto `false`.
|===

* Si `ignoreUpdates` está configurado como `false` o no especificado, el Secreto y todas las claves listadas deben existir. Cualquier cambio en los valores de referencia Secret hará que el gráfico se actualice con los nuevos valores.
* Si `ignoreUpdates` se establece en `true`, el Secreto se utiliza si existe cuando se crea el Gráfico, o se actualiza debido a cualquier otro cambio en los recursos relacionados. Los cambios en el Secreto no harán que se actualice el gráfico.

Un ejemplo de despliegue de un cuadro de mando utilizando un Secreto existente con dos claves:

[,yaml]
----
apiVersion: helm.cattle.io/v1
kind: HelmChart
metadata:
  namespace: kube-system
  name: example-app
spec:
  targetNamespace: example-namespace
  createNamespace: true
  version: v1.2.3
  chart: example-app
  repo: https://repo.example.com
  valuesContent: |-
    image:
      tag: v1.2.2
  valuesSecrets:
    - name: example-app-custom-values
      ignoreUpdates: false
      keys:
        - someValues
        - moreValues
---
apiVersion: v1
kind: Secret
metadata:
  namespace: kube-system
  name: example-app-custom-values
stringData:
  moreValues: |-
    database:
      address: db.example.com
      username: user
      password: pass
  someValues: |-
    adminUser:
      create: true
      username: admin
      password: secret
----

== Personalización de componentes empaquetados con HelmChartConfig

Para permitir la sobreescritura de valores para componentes empaquetados que se despliegan como HelmCharts (como Traefik), K3s soporta la personalización de despliegues a través de recursos HelmChartConfig. El recurso HelmChartConfig debe coincidir con el nombre y el espacio de nombres de su HelmChart correspondiente, y admite proporcionar `valuesContent` adicional, que se pasa al comando `helm` como un archivo de valores adicional.

### Definiciones de los campos de HelmChartConfig

|===
| Campo | Descripción

| metadata.name
| Nombre del Helm Chart - debe coincidir con el nombre del recurso HelmChart.

| spec.valuesContent
| Anule los valores predeterminados complejos de los gráficos mediante el contenido del archivo YAML.

| spec.valuesSecrets
| Anule los valores predeterminados de la Carta mediante Secretos externos.

| spec.failurePolicy
| Configurado en `abort`, en cuyo caso se interrumpe la operación del timón, a la espera de la intervención manual del operador.
|===

[NOTE]
====
Los valores de HelmChart `spec.set` anulan los ajustes de HelmChart y HelmChartConfig `spec.valuesContent` y `spec.valuesSecrets`, como se ha descrito anteriormente.
====

Por ejemplo, para personalizar la configuración de entrada de Traefik empaquetada, puede crear un archivo llamado `/var/lib/rancher/k3s/server/manifests/traefik-config.yaml` y rellenarlo con el siguiente contenido:

[,yaml]
----
apiVersion: helm.cattle.io/v1
kind: HelmChartConfig
metadata:
  name: traefik
  namespace: kube-system
spec:
  valuesContent: |-
    image:
      repository: docker.io/library/traefik
      tag: 3.3.5
    ports:
      web:
        forwardedHeaders:
          trustedIPs:
            - 10.0.0.0/8
----
