Esta página se concentra nas opções que são comumente usadas ao configurar o K3s pela primeira vez. Consulte a documentação sobre xref:advanced.adoc[Advanced Options and Configuration] e a documentação do comando xref:cli/server.adoc[do servidor] e xref:cli/agent.adoc[do agente] para obter informações mais detalhadas.

[#_configuration_with_install_script]
== Configuração com script de instalação

Conforme mencionado no xref:quick-start.adoc[Guia de início rápido], você pode usar o script de instalação disponível em https://get.k3s.io para instalar o K3s como um serviço em sistemas baseados em systemd e openrc.

Você pode usar uma combinação de `INSTALL_K3S_EXEC`, variáveis de ambiente `K3S_` e sinalizadores de comando para passar a configuração para a configuração do serviço.
As variáveis de ambiente prefixadas, o valor `INSTALL_K3S_EXEC` e os argumentos finais do shell são todos mantidos na configuração do serviço.
Após a instalação, a configuração pode ser alterada editando o arquivo de ambiente, editando a configuração do serviço ou simplesmente executando novamente o instalador com novas opções.
Para {k3s-product-name}, você deve usar `INSTALL_K3S_ARTIFACT_URL`, que aponta para o https://scc.suse.com/rancher-docs/rancherprime/latest/en/reference-guide.html#prime-artifacts-url[URL]

Para ilustrar isso, todos os comandos a seguir resultam no mesmo comportamento de registro de um servidor sem flanelógrafo e com um token:

[,bash]
----
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s INSTALL_K3S_EXEC="server" sh -s - --flannel-backend none --token 12345
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s INSTALL_K3S_EXEC="server --flannel-backend none" K3S_TOKEN=12345 sh -s -
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s K3S_TOKEN=12345 sh -s - server --flannel-backend none
# server is assumed below because there is no K3S_URL
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s INSTALL_K3S_EXEC="--flannel-backend none --token 12345" sh -s -
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s sh -s - --flannel-backend none --token 12345
----

Ao registrar um agente, todos os comandos a seguir resultam no mesmo comportamento:

[,bash]
----
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s INSTALL_K3S_EXEC="agent --server https://k3s.example.com --token mypassword" sh -s -
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s INSTALL_K3S_EXEC="agent" K3S_TOKEN="mypassword" sh -s - --server https://k3s.example.com
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s K3S_URL=https://k3s.example.com sh -s - agent --token mypassword
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s K3S_URL=https://k3s.example.com K3S_TOKEN=mypassword sh -s - # agent is assumed because of K3S_URL
----

Para obter detalhes sobre todas as variáveis de ambiente, consulte xref:reference/env-variables.adoc[Variáveis de ambiente.]

[IMPORTANT]
.Observação
====
Se você definir a configuração ao executar o script de instalação, mas não a definir novamente ao reexecutar o script de instalação, os valores originais serão perdidos.

O conteúdo do <<_configuration_file,arquivo de configuração>> não é gerenciado pelo script de instalação.
Se quiser que sua configuração seja independente do script de instalação, use um arquivo de configuração em vez de passar variáveis de ambiente ou argumentos para o script de instalação.
====


== Configuração com binário

O script de instalação se preocupa principalmente com a configuração do K3s para ser executado como um serviço do sistema. Se optar por não usar o script de instalação, você poderá executar o K3s simplesmente baixando o binário da do GitHub https://github.com/k3s-io/k3s/releases/latest[página de lançamento], colocando-o no caminho e executando-o. Isso não é particularmente útil para instalações permanentes, mas pode ser útil ao realizar testes rápidos que não merecem o gerenciamento do K3s como um serviço do sistema.

[,bash]
----
curl -Lo /usr/local/bin/k3s <PRIME-ARTIFACTS-URL>/k3s/v1.35.1%2Bk3s1/k3s; chmod a+x /usr/local/bin/k3s
----

Você pode passar a configuração definindo as variáveis de ambiente do `K3S_`:

[,bash]
----
K3S_KUBECONFIG_MODE="644" k3s server
----

Ou sinalizadores de comando:

[,bash]
----
k3s server --write-kubeconfig-mode=644
----

O agente k3s também pode ser configurado dessa forma:

[,bash]
----
k3s agent --server https://k3s.example.com --token mypassword
----

Para obter detalhes sobre a configuração do servidor K3s, consulte a xref:cli/server.adoc[documentação`k3s server` ]. +
Para obter detalhes sobre a configuração do agente K3s, consulte a xref:cli/agent.adoc[documentação do site`k3s agent` ]. +
Você também pode usar o sinalizador `--help` para ver uma lista de todas as opções disponíveis e suas variáveis de ambiente correspondentes.

[IMPORTANT]
.Bandeiras correspondentes
====
É importante combinar os sinalizadores críticos em seus nós de servidor. Por exemplo, se você usar o sinalizador`--disable servicelb` ou `--cluster-cidr=10.200.0.0/16` em seu nó mestre, mas não o definir em outros nós do servidor, os nós não conseguirão se unir. Eles imprimirão erros como:
`failed to validate server configuration: critical configuration value mismatch.`
Consulte a documentação de configuração do servidor (link acima) para obter mais informações sobre quais sinalizadores devem ser definidos de forma idêntica nos nós do servidor.
====

== Configuração com imagem de contêiner

A imagem do contêiner K3s (https://hub.docker.com/r/rancher/k3s[\`docker.io/rancher/k3s`]) oferece suporte aos mesmos métodos de configuração que o binário disponível na página de lançamento do GitHub.

[#_configuration_file]
== Arquivo de configuração

Além de configurar o K3s com variáveis de ambiente e argumentos da CLI, o K3s também pode usar um arquivo de configuração. O arquivo de configuração é carregado independentemente de como o K3s é instalado ou executado.

Por padrão, a configuração é carregada de `/etc/rancher/k3s/config.yaml` e os arquivos drop-in são carregados de `/etc/rancher/k3s/config.yaml.d/*.yaml` em ordem alfabética.
Esse caminho pode ser configurado por meio do sinalizador `--config` CLI ou da variável env `K3S_CONFIG_FILE`.
Ao substituir o nome do arquivo de configuração padrão, o caminho do diretório drop-in também é modificado.

Um exemplo de um arquivo de configuração básico do `server` está abaixo:

[,yaml]
./etc/rancher/k3s/config.yaml
----
write-kubeconfig-mode: "0644"
tls-san:
  - "foo.local"
node-label:
  - "foo=bar"
  - "something=amazing"
cluster-init: true
----

Isso é equivalente aos seguintes argumentos da CLI:

[,bash]
----
k3s server \
  --write-kubeconfig-mode "0644"    \
  --tls-san "foo.local"             \
  --node-label "foo=bar"            \
  --node-label "something=amazing"  \
  --cluster-init
----

Em geral, os argumentos da CLI são mapeados para suas respectivas chaves YAML, com argumentos repetitivos da CLI sendo representados como listas YAML. Os sinalizadores booleanos são representados como `true` ou `false` no arquivo YAML.

Também é possível usar um arquivo de configuração e argumentos da CLI. Nessas situações, os valores serão carregados de ambas as fontes, mas os argumentos da CLI terão precedência. Para argumentos que podem ser repetidos, como `--node-label`, os argumentos da CLI substituirão todos os valores da lista.

=== Comportamento de mesclagem de valores

Se estiver presente em vários arquivos, será usado o último valor encontrado para uma determinada chave. Um `+` pode ser anexado à chave para anexar o valor à cadeia de caracteres ou fatia existente, em vez de substituí-la. Todas as ocorrências dessa chave em arquivos subsequentes também exigirão um `+` para evitar a substituição do valor acumulado.

Um exemplo de valores mesclados de vários arquivos de configuração está abaixo:

[,yaml]
./etc/rancher/k3s/config.yaml
----
token: boop
node-label:
  - foo=bar
  - bar=baz
----

[,yaml]
./etc/rancher/k3s/config.yaml.d/test1.yaml
----
write-kubeconfig-mode: 600
node-taint:
  - alice=bob:NoExecute
----

[,yaml]
./etc/rancher/k3s/config.yaml.d/test2.yaml
----
write-kubeconfig-mode: 777
node-label:
  - other=what
  - foo=three
node-taint+:
  - charlie=delta:NoSchedule
----

Isso resulta em uma configuração final de:

[,yaml]
----
write-kubeconfig-mode: 777
token: boop
node-label:
  - other=what
  - foo=three
node-taint:
  - alice=bob:NoExecute
  - charlie=delta:NoSchedule
----

== Juntando tudo

Todas as opções acima podem ser combinadas em um único exemplo.

Um arquivo `config.yaml` é criado em `/etc/rancher/k3s/config.yaml`:

[,yaml]
----
token: "secret"
debug: true
----

Em seguida, o script de instalação é executado com uma combinação de variáveis de ambiente e sinalizadores:

[,bash]
----
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s K3S_KUBECONFIG_MODE="644" INSTALL_K3S_EXEC="server" sh -s - --flannel-backend none
----

Ou se você já tiver instalado o K3s Binary:

[,bash]
----
K3S_KUBECONFIG_MODE="644" k3s server --flannel-backend none
----

Isso resulta em um servidor com:

* Um arquivo kubeconfig com permissões `644`
* Backend do Flannel definido como `none`
* O token definido como `secret`
* Registro de depuração ativado

== Arquivos de configuração do Kubelet

O Kubernetes oferece suporte à configuração do kubelet por meio de sinalizadores da CLI e arquivos de configuração. A configuração do kubelet por meio de sinalizadores da CLI foi descontinuada há muito tempo, mas ainda é compatível e é a maneira mais fácil de definir opções básicas. Algumas configurações avançadas do kubelet só podem ser definidas por meio de um arquivo de configuração. Para obter mais informações, consulte a documentação do Kubernetes sobre o kubelet https://kubernetes.io/docs/reference/command-line-tools-reference/kubelet/[para definir os parâmetros do kubelet por meio de um arquivo de configuração] e https://kubernetes.io/docs/tasks/administer-cluster/kubelet-config-file/[].

O suporte para arquivos de configuração drop-in do kubelet ou para o arquivo de configuração (opções 1 e 2 abaixo) só está disponível na versão 1.32 e posteriores. Para versões mais antigas, você deve usar os args do kubelet diretamente (opção número 3 abaixo).

O K3s usa uma configuração padrão de kubelet que é armazenada em `/var/lib/rancher/k3s/agent/etc/kubelet.conf.d/00-k3s-defaults.conf`. Se você quiser alterar os parâmetros de configuração padrão, há três maneiras de fazer isso:

. Coloque um arquivo de configuração drop-in em `/var/lib/rancher/k3s/agent/etc/kubelet.conf.d/` *(recomendado)*.
. Usando o sinalizador `--kubelet-arg=config=$PATHTOFILE`, em que `$PATHTOFILE` é o caminho para um arquivo que inclui parâmetros de configuração do kubelet (por exemplo, `/etc/rancher/k3s/kubelet.conf`) ou o sinalizador `--kubelet-arg=config-dir=$PATHTODIR`, em que `$PATHTODIR` é o caminho para um diretório que pode incluir arquivos que contêm parâmetros de configuração do kubelet (por exemplo, `/etc/rancher/k3s/kubelet.conf.d`).
. Usando o sinalizador `--kubelet-arg=$FLAG`, em que `$FLAG` é um parâmetro de configuração do kubelet (por exemplo, `image-gc-high-threshold=100`). 

Ao misturar sinalizadores da CLI do kubelet e drop-ins do arquivo de configuração, preste atenção à https://kubernetes.io/docs/tasks/administer-cluster/kubelet-config-file/#kubelet-configuration-merging-order[ordem de precedência do].
