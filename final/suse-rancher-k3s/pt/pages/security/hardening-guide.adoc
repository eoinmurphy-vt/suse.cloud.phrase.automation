= Guia de fortalecimento do CIS
:page-languages: [en, ja, ko, zh]
:revdate: 2025-12-31
:page-revdate: {revdate}

Este documento fornece orientação prescritiva para fortalecer uma instalação de produção do K3s. Ele descreve as configurações e os controles necessários para atender aos controles de referência do Kubernetes do Center for Internet Security (CIS).

O K3s tem várias atenuações de segurança aplicadas e ativadas por padrão e será aprovado em vários controles de CIS do Kubernetes sem modificações. Há algumas exceções notáveis que exigem intervenção manual para cumprir totalmente a referência de CIS:

. Os K3s não modificarão o sistema operacional do host. Qualquer modificação no nível do host precisará ser feita manualmente.
. Determinados controles de política do CIS para `NetworkPolicies` e `PodSecurityStandards` (`PodSecurityPolicies` na versão 1.24 e anteriores) restringirão a funcionalidade do cluster. Você deve optar pela configuração do K3s adicionando as opções apropriadas (ativação de plug-ins de admissão) aos sinalizadores de linha de comando ou ao arquivo de configuração, além de aplicar manualmente as políticas apropriadas. Mais detalhes são apresentados nas seções abaixo.

A primeira seção (1.1) da referência do CIS trata principalmente das permissões e da propriedade do manifesto do pod. O K3s não os utiliza para os componentes principais, pois tudo é empacotado em um único binário.

== Requisitos em nível de host

Há duas áreas de requisitos em nível de host: parâmetros do kernel e configuração do processo/diretório do etcd. Elas estão descritas nesta seção.

=== Verifique se o endereço `protect-kernel-defaults` está definido

Esse é um sinalizador de kubelet que fará com que o kubelet seja encerrado se os parâmetros necessários do kernel não forem definidos ou forem definidos com valores diferentes dos padrões do kubelet.

NOTE: `protect-kernel-defaults` é exposto como um sinalizador de nível superior para K3s.

==== Definir parâmetros do kernel

Crie um arquivo chamado `/etc/sysctl.d/90-kubelet.conf` e adicione o snippet abaixo. Em seguida, execute `sysctl -p /etc/sysctl.d/90-kubelet.conf`.

[,bash]
----
vm.panic_on_oom=0
vm.overcommit_memory=1
kernel.panic=10
kernel.panic_on_oops=1
----

== Requisitos de tempo de execução do Kubernetes

Os requisitos de tempo de execução para estar em conformidade com o CIS Benchmark estão centrados na segurança do pod (via PSP ou PSA), nas políticas de rede e nos registros de auditoria do API Server. Elas estão descritas nesta seção.

Por padrão, o K3s não inclui nenhuma política de rede ou segurança de pod. No entanto, o K3s é fornecido com um controlador que aplicará políticas de rede, se alguma for criada. O K3s não ativa a auditoria por padrão, portanto, a configuração do registro de auditoria e a política de auditoria devem ser criadas manualmente. Por padrão, o K3s é executado com os controladores de admissão `PodSecurity` e `NodeRestriction` ativados, entre outros.

=== Segurança do pod

[tabs,sync-group-id=pod-sec]
======
v1.25 e mais recente::
+
--
O K3s v1.25 e mais recentes são compatíveis com https://kubernetes.io/docs/concepts/security/pod-security-admission/[Pod Security Admissions (PSAs)] para controlar a segurança do pod. Os PSAs são ativados passando o seguinte sinalizador para o servidor K3s:

----
--kube-apiserver-arg="admission-control-config-file=/var/lib/rancher/k3s/server/psa.yaml"
----

A política deve ser gravada em um arquivo chamado `psa.yaml` no diretório `/var/lib/rancher/k3s/server`.

Aqui está um exemplo de um PSA em conformidade:

[,yaml]
----
apiVersion: apiserver.config.k8s.io/v1
kind: AdmissionConfiguration
plugins:
- name: PodSecurity
  configuration:
    apiVersion: pod-security.admission.config.k8s.io/v1beta1
    kind: PodSecurityConfiguration
    defaults:
      enforce: "restricted"
      enforce-version: "latest"
      audit: "restricted"
      audit-version: "latest"
      warn: "restricted"
      warn-version: "latest"
    exemptions:
      usernames: []
      runtimeClasses: []
      namespaces: [kube-system, cis-operator-system]
----
--

v1.24 e versões anteriores::
+
--
O K3s v1.24 e versões anteriores suportam https://kubernetes.io/docs/concepts/security/pod-security-policy/[Pod Security Policies (PSPs)] para controlar a segurança do pod. Os PSPs são ativados passando o seguinte sinalizador para o servidor K3s:

----
--kube-apiserver-arg="enable-admission-plugins=NodeRestriction,PodSecurityPolicy"
----

Isso terá o efeito de manter o plug-in `NodeRestriction` e habilitar o `PodSecurityPolicy`.

Quando as PSPs são ativadas, uma política pode ser aplicada para satisfazer os controles necessários descritos na seção 5.2 do CIS Benchmark.

Aqui está um exemplo de um PSP em conformidade:

[,yaml]
----
apiVersion: policy/v1beta1
kind: PodSecurityPolicy
metadata:
  name: restricted-psp
spec:
  privileged: false                # CIS - 5.2.1
  allowPrivilegeEscalation: false  # CIS - 5.2.5
  requiredDropCapabilities:        # CIS - 5.2.7/8/9
    - ALL
  volumes:
    - 'configMap'
    - 'emptyDir'
    - 'projected'
    - 'secret'
    - 'downwardAPI'
    - 'csi'
    - 'persistentVolumeClaim'
    - 'ephemeral'
  hostNetwork: false               # CIS - 5.2.4
  hostIPC: false                   # CIS - 5.2.3
  hostPID: false                   # CIS - 5.2.2
  runAsUser:
    rule: 'MustRunAsNonRoot'       # CIS - 5.2.6
  seLinux:
    rule: 'RunAsAny'
  supplementalGroups:
    rule: 'MustRunAs'
    ranges:
      - min: 1
        max: 65535
  fsGroup:
    rule: 'MustRunAs'
    ranges:
      - min: 1
        max: 65535
  readOnlyRootFilesystem: false
----

Para que o PSP acima seja eficaz, precisamos criar um ClusterRole e um ClusterRoleBinding. Também precisamos incluir uma "política irrestrita do sistema", que é necessária para pods em nível de sistema que exigem privilégios adicionais, e uma política adicional que permita os sysctls necessários para que o servicelb funcione corretamente.

Combinando a configuração acima com a <<_networkpolicies,Política de rede>> descrita na próxima seção, um único arquivo pode ser colocado no diretório `/var/lib/rancher/k3s/server/manifests`. Aqui está um exemplo de um arquivo `policy.yaml`:

NOTE: As adições críticas do Kubernetes, como CNI, DNS e Ingress, são executadas como pods no namespace `kube-system`. Portanto, esse namespace terá uma política menos restritiva para que esses componentes possam ser executados corretamente.

=== Políticas de rede

O CIS exige que todos os namespaces tenham uma política de rede aplicada que limite razoavelmente o tráfego em namespaces e pods.

As políticas de rede devem ser colocadas no diretório `/var/lib/rancher/k3s/server/manifests`, onde serão implantadas automaticamente na inicialização.

Aqui está um exemplo de uma política de rede em conformidade.

[,yaml]
----
kind: NetworkPolicy
apiVersion: networking.k8s.io/v1
metadata:
  name: intra-namespace
  namespace: kube-system
spec:
  podSelector: {}
  ingress:
    - from:
      - namespaceSelector:
          matchLabels:
            kubernetes.io/metadata.name: kube-system
----

Com as restrições aplicadas, o DNS será bloqueado, a menos que seja permitido propositalmente. Abaixo está uma política de rede que permitirá a existência de tráfego para o DNS.

[,yaml]
----
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-network-dns-policy
  namespace: <NAMESPACE>
spec:
  ingress:
  - ports:
    - port: 53
      protocol: TCP
    - port: 53
      protocol: UDP
  podSelector:
    matchLabels:
      k8s-app: kube-dns
  policyTypes:
  - Ingress
----

O servidor de métricas e o controlador de entrada Traefik serão bloqueados por padrão se as políticas de rede não forem criadas para permitir o acesso. Certifique-se de usar o exemplo de yaml abaixo:

[,yaml]
----
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-all-metrics-server
  namespace: kube-system
spec:
  podSelector:
    matchLabels:
      k8s-app: metrics-server
  ingress:
  - {}
  policyTypes:
  - Ingress
---
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-all-svclbtraefik-ingress
  namespace: kube-system
spec:
  podSelector:
    matchLabels:
      svccontroller.k3s.cattle.io/svcname: traefik
  ingress:
  - {}
  policyTypes:
  - Ingress
---
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-all-traefik-v121-ingress
  namespace: kube-system
spec:
  podSelector:
    matchLabels:
      app.kubernetes.io/name: traefik
  ingress:
  - {}
  policyTypes:
  - Ingress
---
----

[IMPORTANT]
====
Os operadores devem gerenciar as políticas de rede normalmente para os namespaces adicionais que forem criados.
====


=== Configuração de auditoria do servidor API

Os requisitos 1.2.22 a 1.2.25 do CIS estão relacionados à configuração de registros de auditoria para o Servidor de API. O K3s não cria por padrão o diretório de logs e a política de auditoria, pois os requisitos de auditoria são específicos para as políticas e o ambiente de cada usuário.

O diretório de registro, de preferência, deve ser criado antes de iniciar o K3s. Recomenda-se uma permissão de acesso restritiva para evitar o vazamento de possíveis informações confidenciais.

[,bash]
----
sudo mkdir -p -m 700 /var/lib/rancher/k3s/server/logs
----

Uma política de auditoria inicial para registrar os metadados da solicitação é fornecida abaixo. A política deve ser gravada em um arquivo chamado `audit.yaml` no diretório `/var/lib/rancher/k3s/server`. Informações detalhadas sobre a configuração da política para o servidor de API podem ser encontradas na do Kubernetes https://kubernetes.io/docs/tasks/debug-application-cluster/audit/[documentação].

[,yaml]
----
apiVersion: audit.k8s.io/v1
kind: Policy
rules:
- level: Metadata
----

Ambas as configurações devem ser passadas como argumentos para o API Server como:

[tabs]
======
configuração::
+
[,yaml]
----
kube-apiserver-arg:
  - 'admission-control-config-file=/var/lib/rancher/k3s/server/psa.yaml'
  - 'audit-log-path=/var/lib/rancher/k3s/server/logs/audit.log'
  - 'audit-policy-file=/var/lib/rancher/k3s/server/audit.yaml'
  - 'audit-log-maxage=30'
  - 'audit-log-maxbackup=10'
  - 'audit-log-maxsize=100'
----

cmdline::
+
[,bash]
----
--kube-apiserver-arg='audit-log-path=/var/lib/rancher/k3s/server/logs/audit.log'
--kube-apiserver-arg='audit-policy-file=/var/lib/rancher/k3s/server/audit.yaml'
----
======

Os K3s devem ser reiniciados para carregar a nova configuração.

[,bash]
----
sudo systemctl daemon-reload
sudo systemctl restart k3s.service
----

== Operações manuais
A seguir estão os controles pelos quais o K3s não passa atualmente com a configuração acima aplicada. Esses controles exigem intervenção manual para cumprir totalmente a Referência de CIS.

=== Controle 1.1.20

Certifique-se de que as permissões do arquivo de certificado PKI do Kubernetes estejam definidas como 600 ou mais restritivas (Manual)

.Remediação
[%collapsible]
======
Os arquivos de certificado PKI do K3s são armazenados em `/var/lib/rancher/k3s/server/tls/` com permissão 644.
Para corrigir, execute o seguinte comando:

[,bash]
----
chmod -R 600 /var/lib/rancher/k3s/server/tls/*.crt
----
======

=== Controle 1.2.9

Certifique-se de que o plug-in de controle de admissão EventRateLimit esteja definido

.Remediação
[%collapsible]
======
Siga a https://kubernetes.io/docs/reference/access-authn-authz/admission-controllers/#eventratelimit[documentação do] e defina os limites desejados em um arquivo de configuração.
Para essa e outras configurações de psa, esta documentação usa /var/lib/rancher/k3s/server/psa.yaml.
Em seguida, edite o arquivo de configuração do K3s /etc/rancher/k3s/config.yaml e defina os parâmetros abaixo.

[,yaml]
----
kube-apiserver-arg:
  - "enable-admission-plugins=NodeRestriction,EventRateLimit"
  - "admission-control-config-file=/var/lib/rancher/k3s/server/psa.yaml"
----
======

=== Controle 1.2.11

Certifique-se de que o plug-in de controle de admissão AlwaysPullImages esteja definido

.Remediação
[%collapsible]
======
Permissivo, de acordo com as diretrizes do CIS: "Essa configuração pode afetar clusters off-line ou isolados, que têm imagens pré-carregadas e não têm acesso a um registro para extrair imagens em uso. Essa configuração não é apropriada para clusters que usam essa configuração."
Edite o arquivo de configuração do K3s /etc/rancher/k3s/config.yaml e defina o parâmetro abaixo.

[,yaml]
----
kube-apiserver-arg:
  - "enable-admission-plugins=...,AlwaysPullImages,..."
----
======

=== Controle 1.2.21

Certifique-se de que o argumento --request-timeout esteja definido conforme apropriado

.Remediação
[%collapsible]
======
Permissivo, de acordo com as diretrizes do CIS, "recomenda-se definir esse limite conforme apropriado e alterar o limite padrão de 60 segundos somente se necessário".
Edite o arquivo de configuração do K3s /etc/rancher/k3s/config.yaml e defina o parâmetro abaixo, se necessário. Por exemplo,

[,yaml]
----
kube-apiserver-arg:
  - "request-timeout=300s"
----
======

=== Control 4.2.13 

Certifique-se de que um limite esteja definido nos PIDs de pod

.Remediação
[%collapsible]
======
Se estiver usando um arquivo de configuração do K3s /etc/rancher/k3s/config.yaml, edite o arquivo para definir `podPidsLimit` como

[,yaml]
----
kubelet-arg:
  - "pod-max-pids=<value>"
----
======

=== Controle 5.X

Todos os controles 5.X estão relacionados à configuração da política do Kubernetes. Esses controles não são aplicados pelos K3s por padrão.

Consulte a xref:security/self-assessment-1.8.adoc#_5_1_rbac_and_service_accounts[Seção 5 do CIS 1.8] para obter mais informações sobre como criar e aplicar essas políticas.

==== Controle 5.1.5

[NOTE]
====
A correção para atingir a pontuação de aprovação só é necessária para o cis-1.9.
====

.Remediação
[%collapsible]
======
Crie contas de serviço explícitas sempre que uma carga de trabalho do Kubernetes exigir acesso específico ao servidor da API do Kubernetes.
No mínimo, faça o seguinte patch:

[,bash]
----
kubectl patch serviceaccount --namespace default default --patch '{"automountServiceAccountToken": false}'
kubectl patch serviceaccount --namespace kube-node-lease default --patch '{"automountServiceAccountToken": false}'
kubectl patch serviceaccount --namespace kube-public default --patch '{"automountServiceAccountToken": false}'
----
======

== Conclusão

Se você seguiu este guia, seu cluster K3s estará configurado para estar em conformidade com o CIS Kubernetes Benchmark. Você pode consultar o xref:security/self-assessment-1.11.adoc[Guia de autoavaliação do CIS 1.11] para entender as expectativas de cada uma das verificações do benchmark e como você pode fazer o mesmo em seu cluster.
