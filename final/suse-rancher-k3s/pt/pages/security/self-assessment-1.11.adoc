== Visão geral

Este documento é um complemento do xref:security/hardening-guide.adoc[guia de fortalecimento da segurança do K3s]. O guia de fortalecimento fornece orientação prescritiva para o fortalecimento de uma instalação de produção do K3s, e este guia de referência tem como objetivo ajudá-lo a avaliar o nível de segurança do cluster fortalecido em relação a cada controle no CIS Kubernetes Benchmark. Ele deve ser usado por operadores, equipes de segurança, auditores e tomadores de decisão do K3s.

Este guia é específico para a linha de versão **v1.29-v1.34** do K3s e a versão **v1.11** do CIS Kubernetes Benchmark.

Para obter mais informações sobre cada controle, incluindo descrições detalhadas e correções para testes com falha, consulte a seção correspondente do CIS Kubernetes Benchmark v1.9. Você pode fazer o download do benchmark, após criar uma conta gratuita, em https://www.cisecurity.org/benchmark/kubernetes[Center for Internet Security (CIS)].

=== Metodologia de teste de controles

Cada controle no CIS Kubernetes Benchmark foi avaliado em um cluster K3s que foi configurado de acordo com o guia de proteção que o acompanha.

Quando as auditorias de controle diferem do benchmark original do CIS, os comandos de auditoria específicos dos K3s são fornecidos para teste.

Estes são os resultados possíveis para cada controle:

- **Aprovado** - O cluster K3s em teste foi aprovado na auditoria descrita no benchmark.
- **Não aplicável** - O controle não é aplicável aos K3s devido à forma como foi projetado para operar. A seção de correção explicará por que isso acontece.
- **Warn** - O controle é manual no benchmark do CIS e depende do caso de uso do cluster ou de algum outro fator que deve ser determinado pelo operador do cluster. Esses controles foram avaliados para garantir que o K3s não impeça sua implementação, mas nenhuma configuração ou auditoria adicional do cluster em teste foi realizada.

Este guia parte do pressuposto de que o K3s está sendo executado como uma unidade Systemd. Sua instalação pode variar e exigirá que você ajuste os comandos de "auditoria" para se adequar ao seu cenário.

**Resultado:** PASSE

**Auditoria:**

[,bash]
----
find /var/lib/cni/networks -type f ! -name lock 2> /dev/null | xargs --no-run-if-empty stat -c permissions=%a
----

**Resultado esperado:** as permissões têm 600 permissões, espera-se 600 ou mais restritivas

[%collapsible]
.*Valor retornado:*
======

[,log]
----
permissions=600
permissions=600
permissions=600
permissions=600
permissions=600
permissions=600
----
======

[%collapsible]
.*Remediação:*
======

Por padrão, o K3s define as permissões do arquivo CNI como 600.
Observe que, para muitos CNIs, um arquivo de bloqueio é criado com as permissões 750. Isso é esperado e pode ser ignorado.
Se você modificar a configuração da CNI, verifique se as permissões estão definidas como 600.
Por exemplo, `chmod 600 /var/lib/cni/networks/<filename>`

=== 1.1.12 Certifique-se de que a propriedade do diretório de dados do etcd esteja definida como etcd:etcd (Automatizado)

**Resultado:** Não aplicável

**Justificativa:**

Para o K3s, o etcd é incorporado ao processo do k3s. Não há um processo etcd separado.
Portanto, a propriedade do diretório de dados do etcd é gerenciada pelo processo k3s e deve ser root:root.

**Resultado:** AVISO
