= Options de configuration
:page-languages: [en, ja, ko, zh]
:revdate: 2026-03-03
:page-revdate: {revdate}

Cette page se concentre sur les options couramment utilisées lors de la première installation d'un K3. Reportez-vous à la documentation sur les xref:advanced.adoc[options avancées et la configuration], ainsi qu'à la documentation sur les commandes du xref:cli/server.adoc[serveur] et de l'xref:cli/agent.adoc[agent] pour plus de détails.

[#_configuration_with_install_script]
== Configuration avec le script d'installation

Comme indiqué dans le xref:quick-start.adoc[Guide de démarrage rapide], vous pouvez utiliser le script d'installation disponible à l'adresse https://get.k3s.io pour installer K3s en tant que service sur les systèmes basés sur systemd et openrc.

Vous pouvez utiliser une combinaison de `INSTALL_K3S_EXEC`, de variables d'environnement `K3S_` et d'indicateurs de commande pour transmettre la configuration à la configuration du service.
Les variables d'environnement préfixées, la valeur `INSTALL_K3S_EXEC` et les arguments shell de fin sont tous conservés dans la configuration du service.
Après l'installation, la configuration peut être modifiée en éditant le fichier d'environnement, en modifiant la configuration du service ou simplement en réexécutant le programme d'installation avec de nouvelles options.
Pour {k3s-product-name}, vous devez utiliser `INSTALL_K3S_ARTIFACT_URL` qui pointe vers l' https://scc.suse.com/rancher-docs/rancherprime/latest/en/reference-guide.html#prime-artifacts-url[URL].

Pour illustrer ce point, les commandes suivantes aboutissent toutes au même comportement, à savoir l'enregistrement d'un serveur sans canal et avec un jeton :

[,bash]
----
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s INSTALL_K3S_EXEC="server" sh -s - --flannel-backend none --token 12345
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s INSTALL_K3S_EXEC="server --flannel-backend none" K3S_TOKEN=12345 sh -s -
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s K3S_TOKEN=12345 sh -s - server --flannel-backend none
# server is assumed below because there is no K3S_URL
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s INSTALL_K3S_EXEC="--flannel-backend none --token 12345" sh -s -
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s sh -s - --flannel-backend none --token 12345
----

Lors de l'enregistrement d'un agent, les commandes suivantes ont toutes le même effet :

[,bash]
----
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s INSTALL_K3S_EXEC="agent --server https://k3s.example.com --token mypassword" sh -s -
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s INSTALL_K3S_EXEC="agent" K3S_TOKEN="mypassword" sh -s - --server https://k3s.example.com
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s K3S_URL=https://k3s.example.com sh -s - agent --token mypassword
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s K3S_URL=https://k3s.example.com K3S_TOKEN=mypassword sh -s - # agent is assumed because of K3S_URL
----

Pour plus de détails sur les variables d'environnement, voir xref:reference/env-variables.adoc[Variables d'environnement].

[IMPORTANT]
.Note
====
Si vous définissez la configuration lors de l'exécution du script d'installation, mais que vous ne la définissez pas à nouveau lors de la ré-exécution du script d'installation, les valeurs d'origine seront perdues.

Le contenu du <<_configuration_file,fichier de configuration>> n'est pas géré par le script d'installation.
Si vous souhaitez que votre configuration soit indépendante du script d'installation, vous devez utiliser un fichier de configuration au lieu de transmettre des variables d'environnement ou des arguments au script d'installation.
====


== Configuration avec binaire

Le script d'installation s'occupe principalement de configurer K3s pour qu'il fonctionne en tant que service système. Si vous choisissez de ne pas utiliser le script d'installation, vous pouvez exécuter K3s simplement en téléchargeant le binaire depuis notre https://github.com/k3s-io/k3s/releases/latest[page de publication] GitHub, en le plaçant sur votre chemin d'accès et en l'exécutant. Cela n'est pas particulièrement utile pour les installations permanentes, mais peut l'être lors de tests rapides qui ne nécessitent pas de gérer K3s en tant que service système.

[,bash]
----
curl -Lo /usr/local/bin/k3s <PRIME-ARTIFACTS-URL>/k3s/v1.35.1%2Bk3s1/k3s; chmod a+x /usr/local/bin/k3s
----

Vous pouvez transmettre la configuration en définissant les variables d'environnement `K3S_`:

[,bash]
----
K3S_KUBECONFIG_MODE="644" k3s server
----

Ou des drapeaux de commande :

[,bash]
----
k3s server --write-kubeconfig-mode=644
----

L'agent k3s peut également être configuré de cette manière :

[,bash]
----
k3s agent --server https://k3s.example.com --token mypassword
----

Pour plus de détails sur la configuration du serveur K3s, voir la xref:cli/server.adoc[documentation`k3s server` ]. +
Pour plus de détails sur la configuration de l'agent K3s, voir la xref:cli/agent.adoc[documentation`k3s agent` ]. +
Vous pouvez également utiliser l'option `--help` pour obtenir une liste de toutes les options disponibles et des variables d'environnement correspondantes.

[IMPORTANT]
.Drapeaux assortis
====
Il est important de faire correspondre les drapeaux critiques sur vos nœuds de serveur. Par exemple, si vous utilisez l'indicateur`--disable servicelb` ou `--cluster-cidr=10.200.0.0/16` sur votre nœud maître, mais que vous ne le définissez pas sur les autres nœuds de serveur, ces derniers ne parviendront pas à se joindre. Ils imprimeront des erreurs telles que :
`failed to validate server configuration: critical configuration value mismatch.`
Voir la documentation sur la configuration du serveur (lien ci-dessus) pour plus d'informations sur les drapeaux qui doivent être définis de manière identique sur les nœuds du serveur.
====

== Configuration avec l'image du conteneur

L'image du conteneur K3s (https://hub.docker.com/r/rancher/k3s[\`docker.io/rancher/k3s`]) supporte les mêmes méthodes de configuration que le binaire disponible sur la page de publication de GitHub.

[#_configuration_file]
== Fichier de configuration

Outre la configuration de K3s à l'aide de variables d'environnement et d'arguments CLI, K3s peut également utiliser un fichier de configuration. Le fichier de configuration est chargé indépendamment de la manière dont K3s est installé ou exécuté.

Par défaut, la configuration est chargée à partir de `/etc/rancher/k3s/config.yaml`, et les fichiers de dépôt sont chargés à partir de `/etc/rancher/k3s/config.yaml.d/*.yaml` dans l'ordre alphabétique.
Ce chemin est configurable via l'indicateur CLI `--config` ou la variable env `K3S_CONFIG_FILE`.
Lorsque le nom du fichier de configuration par défaut est remplacé, le chemin du répertoire de dépôt est également modifié.

Voici un exemple de fichier de configuration de base pour `server`:

[,yaml]
./etc/rancher/k3s/config.yaml
----
write-kubeconfig-mode: "0644"
tls-san:
  - "foo.local"
node-label:
  - "foo=bar"
  - "something=amazing"
cluster-init: true
----

Cela équivaut aux arguments CLI suivants :

[,bash]
----
k3s server \
  --write-kubeconfig-mode "0644"    \
  --tls-san "foo.local"             \
  --node-label "foo=bar"            \
  --node-label "something=amazing"  \
  --cluster-init
----

En général, les arguments de l'interface de programmation correspondent à leur clé YAML respective, les arguments répétables de l'interface de programmation étant représentés par des listes YAML. Les drapeaux booléens sont représentés par `true` ou `false` dans le fichier YAML.

Il est également possible d'utiliser à la fois un fichier de configuration et des arguments CLI. Dans ce cas, les valeurs seront chargées à partir des deux sources, mais les arguments de l'interface de programmation seront prioritaires. Pour les arguments répétables tels que `--node-label`, les arguments de l'interface de programmation remplaceront toutes les valeurs de la liste.

=== Comportement de fusion des valeurs

Si elle est présente dans plusieurs fichiers, la dernière valeur trouvée pour une clé donnée sera utilisée. Un `+` peut être ajouté à la clé pour ajouter la valeur à la chaîne ou à la tranche existante, au lieu de la remplacer. Toutes les occurrences de cette clé dans les fichiers suivants nécessiteront également une adresse `+` pour éviter d'écraser la valeur accumulée.

Voici un exemple de valeurs fusionnées à partir de plusieurs fichiers de configuration :

[,yaml]
./etc/rancher/k3s/config.yaml
----
token: boop
node-label:
  - foo=bar
  - bar=baz
----

[,yaml]
./etc/rancher/k3s/config.yaml.d/test1.yaml
----
write-kubeconfig-mode: 600
node-taint:
  - alice=bob:NoExecute
----

[,yaml]
./etc/rancher/k3s/config.yaml.d/test2.yaml
----
write-kubeconfig-mode: 777
node-label:
  - other=what
  - foo=three
node-taint+:
  - charlie=delta:NoSchedule
----

Il en résulte une configuration finale de :

[,yaml]
----
write-kubeconfig-mode: 777
token: boop
node-label:
  - other=what
  - foo=three
node-taint:
  - alice=bob:NoExecute
  - charlie=delta:NoSchedule
----

== La mise en place de l'ensemble

Toutes les options ci-dessus peuvent être combinées en un seul exemple.

Un fichier `config.yaml` est créé à l'adresse `/etc/rancher/k3s/config.yaml`:

[,yaml]
----
token: "secret"
debug: true
----

Le script d'installation est ensuite exécuté à l'aide d'une combinaison de variables d'environnement et de drapeaux :

[,bash]
----
curl -sfL https://get.k3s.io | INSTALL_K3S_ARTIFACT_URL=<PRIME-ARTIFACTS-URL>/k3s K3S_KUBECONFIG_MODE="644" INSTALL_K3S_EXEC="server" sh -s - --flannel-backend none
----

Ou si vous avez déjà installé le K3s Binary :

[,bash]
----
K3S_KUBECONFIG_MODE="644" k3s server --flannel-backend none
----

Il en résulte un serveur avec :

* Un fichier kubeconfig avec des permissions `644`
* Le backend de Flannel est prêt à `none`
* Le jeton fixé à `secret`
* Activation de la journalisation de débogage

== Fichiers de configuration Kubelet

Kubernetes prend en charge la configuration du kubelet via des drapeaux CLI et des fichiers de configuration. La configuration du kubelet via les drapeaux du CLI est depuis longtemps obsolète, mais elle est toujours prise en charge et constitue le moyen le plus simple de définir les options de base. Certaines configurations avancées des kubelets ne peuvent être définies qu'à l'aide d'un fichier de configuration. Pour plus d'informations, consultez la documentation Kubernetes pour le kubelet https://kubernetes.io/docs/reference/command-line-tools-reference/kubelet/[setting kubelet parameters via un fichier de configuration] et https://kubernetes.io/docs/tasks/administer-cluster/kubelet-config-file/[].

La prise en charge des fichiers de configuration drop-in des kubelets ou du fichier de configuration (options 1 et 2 ci-dessous) n'est disponible qu'à partir de la version 1.32. Pour les versions plus anciennes, vous devriez utiliser directement les args du kubelet (option numéro 3 ci-dessous).

K3s utilise une configuration de kubelet par défaut qui est stockée sous `/var/lib/rancher/k3s/agent/etc/kubelet.conf.d/00-k3s-defaults.conf`. Si vous souhaitez modifier les paramètres de configuration par défaut, il y a trois façons de le faire :

. Placez un fichier de configuration dans `/var/lib/rancher/k3s/agent/etc/kubelet.conf.d/` *(recommandé).*
. En utilisant l'indicateur `--kubelet-arg=config=$PATHTOFILE`, où `$PATHTOFILE` est le chemin vers un fichier qui contient des paramètres de configuration de kubelet (par exemple `/etc/rancher/k3s/kubelet.conf`) ou l'indicateur `--kubelet-arg=config-dir=$PATHTODIR`, où `$PATHTODIR` est le chemin vers un répertoire qui peut inclure des fichiers qui contiennent des paramètres de configuration de kubelet (par exemple `/etc/rancher/k3s/kubelet.conf.d`).
. En utilisant l'indicateur `--kubelet-arg=$FLAG`, où `$FLAG` est un paramètre de configuration du kubelet (par exemple `image-gc-high-threshold=100`). 

Lorsque vous mélangez les drapeaux de la CLI des kubelets et les drop-ins des fichiers de configuration, faites attention à l' https://kubernetes.io/docs/tasks/administer-cluster/kubelet-config-file/#kubelet-configuration-merging-order[ordre de préséance].
