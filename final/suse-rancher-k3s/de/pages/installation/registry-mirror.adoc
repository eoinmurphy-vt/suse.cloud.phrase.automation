= Eingebetteter Registry-Spiegel
:page-languages: [en, ja, ko, zh]
:revdate: 2026-01-05
:page-revdate: {revdate}

[IMPORTANT]
.Version Tor
====
Der Embedded Registry Mirror ist als experimentelle Funktion ab Januar 2024 verfügbar: v1.26.13+k3s1, v1.27.10+k3s1, v1.28.6+k3s1, v1.29.1+k3s1 und GA ab Dezember 2024: v1.29.12+k3s1, v1.30.8+k3s1, v1.31.4+k3s1.
====

K3s bettet https://github.com/XenitAB/spegel[Spegel] ein, einen zustandslosen, verteilten OCI-Registry-Spiegel, der den Peer-to-Peer-Austausch von Container-Images zwischen Knoten in einem Kubernetes-Cluster ermöglicht. Der verteilte Registrierungsspiegel ist standardmäßig deaktiviert. Damit K3s diese Funktion nutzen kann, müssen Sie sowohl die xref:#_enabling_the_distributed_oci_registry_mirror[verteilte OCI-Registrierungsspiegelung] als auch die xref:#_enabling_registry_mirroring[Registrierungsspiegelung] aktivieren, wie in den folgenden Unterabschnitten erläutert.

[#_enabling_the_distributed_oci_registry_mirror]
== Aktivieren des verteilten OCI-Registerspiegels

Um den eingebetteten Registrierungsspiegel zu aktivieren, müssen die Serverknoten mit dem Flag `--embedded-registry` oder mit `embedded-registry: true` in der Konfigurationsdatei gestartet werden.
Mit dieser Option wird der eingebettete Mirror für die Verwendung auf allen Knoten im Cluster aktiviert.

Wenn diese Funktion auf Clusterebene aktiviert ist, hosten alle Knoten eine lokale OCI-Registrierung an Port 6443 und veröffentlichen eine Liste der verfügbaren Images über ein Peer-to-Peer-Netzwerk an Port 5001.
Jedes Image, das im Containerd-Image-Store auf einem beliebigen Knoten verfügbar ist, kann von anderen Clustermitgliedern ohne Zugriff auf eine externe Registry abgerufen werden.
Bilder, die über xref:installation/airgap.adoc#_tabs_1_manually_deploy_images_method[Air-Gap-Image-Tar-Dateien] importiert werden, werden in Containerd gepinnt, um sicherzustellen, dass sie verfügbar bleiben und nicht von der Kubelet-Garbage-Collection gelöscht werden.

Der Peer-to-Peer-Port kann durch Setzen der Umgebungsvariablen `K3S_P2P_PORT` für den K3s-Dienst von 5001 geändert werden. Der Port muss auf allen Knoten auf denselben Wert gesetzt werden.
Das Ändern des Anschlusses wird nicht unterstützt und nicht empfohlen.

=== Anforderungen

Wenn der eingebettete Registrierungsspiegel aktiviert ist, müssen alle Knoten in der Lage sein, sich gegenseitig über ihre internen IP-Adressen an den TCP-Ports 5001 und 6443 zu erreichen.
Wenn die Knoten sich nicht gegenseitig erreichen können, kann es länger dauern, bis die Images gezogen werden, da die verteilte Registry zuerst von containerd versucht wird, bevor auf andere Endpunkte zurückgegriffen wird.

[#_enabling_registry_mirroring]
== Aktivieren der Registrierungsspiegelung

Wenn Sie die Spiegelung für eine Registrierung aktivieren, kann ein Knoten sowohl Bilder aus dieser Registrierung von anderen Knoten abrufen als auch die Bilder der Registrierung für andere Knoten freigeben.
Wenn eine Registrierung auf einigen Knoten für die Spiegelung aktiviert ist, auf anderen jedoch nicht, tauschen nur die Knoten, auf denen die Registrierung aktiviert ist, Bilder aus dieser Registrierung aus.

Um die Spiegelung von Images aus einer vorgelagerten Container-Registry zu ermöglichen, müssen die Nodes einen Eintrag im Abschnitt `mirrors` von `registries.yaml` für diese Registry haben.
In der Registrierung müssen keine Endpunkte aufgeführt sein, sie muss nur vorhanden sein.
Um beispielsweise die verteilte Spiegelung von Images von `docker.io` und `registry.k8s.io` zu aktivieren, konfigurieren Sie `registries.yaml` mit dem folgenden Inhalt auf allen Clusterknoten:

[,yaml]
----
mirrors:
  docker.io:
  registry.k8s.io:
----

Auch Endpunkte für Registrierungsspiegel können wie üblich hinzugefügt werden.
In der folgenden Konfiguration wird zuerst der eingebettete Spiegel, dann `mirror.example.com` und schließlich `docker.io` ausprobiert, um Bilder zu ziehen:

[,yaml]
----
mirrors:
  docker.io:
    endpoint:
      - https://mirror.example.com
----

Wenn Sie ein privates Register direkt und nicht als Spiegel für ein vorgeschaltetes Register verwenden, können Sie die verteilte Spiegelung auf dieselbe Weise aktivieren, wie dies bei öffentlichen Registern der Fall ist - indem Sie es im Abschnitt "Spiegel" auflisten:

[,yaml]
----
mirrors:
  mirror.example.com:
----

[IMPORTANT]
.Version Tor
====
Wildcard-Unterstützung ist ab März 2024 verfügbar: v1.26.15+k3s1, v1.27.12+k3s1, v1.28.8+k3s1, v1.29.3+k3s1
====


Der `"*"` Wildcard-Spiegelungseintrag kann verwendet werden, um die verteilte Spiegelung aller Registrierungen zu aktivieren. Beachten Sie, dass das Sternchen in Anführungszeichen gesetzt werden MUSS:

[,yaml]
----
mirrors:
  "*":
----

Wenn auf einem Knoten keine Register für die Spiegelung aktiviert sind, nimmt dieser Knoten in keiner Weise an der verteilten Registrierung teil.

Weitere Informationen über die Struktur der Datei `registries.yaml` finden Sie unter xref:installation/private-registry.adoc[Private Registry Configuration].

=== Standard-Endpunkt-Fallback

Standardmäßig greift containerd auf den Standard-Endpunkt zurück, wenn er von Registries mit konfigurierten Spiegel-Endpunkten abruft. Wenn Sie diese Funktion deaktivieren und nur Bilder von den konfigurierten Mirrors und/oder dem eingebetteten Mirror abrufen möchten, lesen Sie den Abschnitt xref:installation/private-registry.adoc#_default_endpoint_fallback[Default Endpoint Fallback]in der Dokumentation Private Registry Configuration.

Beachten Sie, dass Sie, wenn Sie die Option `--disable-default-registry-endpoint` verwenden und den Abruf direkt aus einer bestimmten Registrierung zulassen möchten, während Sie den Rest nicht zulassen, explizit einen Endpunkt angeben können, damit der Abruf des Bildes auf die Registrierung selbst zurückgreifen kann:

[,yaml]
----
mirrors:
  docker.io:           # no default endpoint, pulls will fail if not available on a node
  registry.k8s.io:     # no default endpoint, pulls will fail if not available on a node
  mirror.example.com:  # explicit default endpoint, can pull from upstream if not available on a node
    endpoint:
      - https://mirror.example.com
----

=== Letzter Tag

Wenn kein Tag für ein Containerbild angegeben wird, ist das implizite Standard-Tag `latest`. Diese Markierung wird häufig aktualisiert, um auf die neueste Version des Bildes hinzuweisen. Da diese Markierung auf eine andere Revision eines Bildes verweist, je nachdem, wann sie abgerufen wird, *wird* die verteilte Registrierung die Markierung `latest` *nicht* von anderen Knoten abrufen. Dies zwingt containerd dazu, eine vorgelagerte Registry oder einen Registry-Mirror aufzusuchen, um eine konsistente Ansicht dessen zu gewährleisten, worauf sich das Tag `latest` bezieht.

Dies entspricht der https://kubernetes.io/docs/concepts/containers/images/#imagepullpolicy-defaulting[speziellen], die von Kubernetes bei Verwendung des `latest` Tags für ein Container-Image beobachtet wird.

Die Spiegelung des Tags `latest` kann durch Setzen der Umgebungsvariablen `K3S_P2P_ENABLE_LATEST=true` für den K3s-Dienst aktiviert werden.
Dies wird aus den oben genannten Gründen nicht unterstützt und nicht empfohlen.

== Sicherheit

=== Authentifizierung

Für den Zugriff auf die Registrierungs-API des eingebetteten Mirrors ist ein gültiges Client-Zertifikat erforderlich, das von der Client-Zertifizierungsstelle des Clusters unterzeichnet wurde.

Für den Zugriff auf das Peer-to-Peer-Netzwerk der verteilten Hash-Tabelle ist ein gemeinsamer Schlüssel erforderlich, der von den Serverknoten kontrolliert wird.
Die Knoten authentifizieren sich gegenseitig mit dem gemeinsam genutzten Schlüssel und einem von der Cluster-Zertifizierungsstelle signierten Zertifikat.

=== Mögliche Bedenken

[CAUTION]
====
Die verteilte Registrierung basiert auf Peer-to-Peer-Prinzipien und setzt gleiche Rechte und gleiches Vertrauen zwischen allen Mitgliedern des Clusters voraus.
Wenn dies nicht mit der Sicherheitslage Ihres Clusters übereinstimmt, sollten Sie die eingebettete verteilte Registrierung nicht aktivieren.
====


Die eingebettete Registrierung kann Bilder zur Verfügung stellen, auf die ein Knoten sonst keinen Zugriff hat.
Wenn beispielsweise einige Ihrer Images von einer Registry, einem Projekt oder einem Repository gezogen werden, das eine Authentifizierung über Kubernetes Image Pull Secrets oder Anmeldedaten in `registries.yaml` erfordert, ermöglicht die verteilte Registry anderen Knoten, diese Images freizugeben, ohne der Upstream-Registry Anmeldedaten zu übermitteln.

Benutzer, die Zugriff auf den Containerd-Image-Speicher auf einem Knoten haben, können dies nutzen, um das Image für andere Clusterknoten zu "vergiften", da andere Knoten dem vom Knoten bekannt gegebenen Tag vertrauen und es ohne Überprüfung der vorgelagerten Registry verwenden.
Wenn die Integrität des Bildes wichtig ist, sollten Sie Bild-Digests anstelle von Tags verwenden, da der Digest auf diese Weise nicht verfälscht werden kann.

== Gemeinsame Nutzung von Luftspalt- oder manuell geladenen Bildern

Die Freigabe von Bildern wird auf der Grundlage der Quellregistrierung gesteuert.
Images, die über xref:installation/airgap.adoc#_tabs_1_manually_deploy_images_method[Air-Gap-Tarballs] direkt in den Containerd geladen, xref:add-ons/import-images.adoc#_pre_import_images[vorimportiert] oder mit dem Kommandozeilen-Tool `ctr` direkt in den Containerd-Image-Speicher geladen werden, werden zwischen den Knoten ausgetauscht, wenn sie als von einer Registry stammend gekennzeichnet sind, die für die Spiegelung aktiviert ist.

Beachten Sie, dass die vorgelagerte Registrierung, aus der die Bilder zu stammen scheinen, nicht tatsächlich existieren oder erreichbar sein muss.
Sie könnten zum Beispiel Bilder als von einer fiktiven Upstream-Registry stammend kennzeichnen und diese Bilder in den Bildspeicher von containerd importieren.
Sie können diese Bilder dann von allen Clustermitgliedern abrufen, sofern die Registrierung in `registries.yaml`

== Bilder schieben

Die eingebettete Registrierung ist schreibgeschützt und kann nicht direkt mit `docker push` oder anderen gängigen Tools, die mit OCI-Registrierungen interagieren, aufgerufen werden.

Bilder können manuell über die eingebettete Registrierung verfügbar gemacht werden, indem `ctr -n k8s.io image pull` ausgeführt wird, um ein Bild zu ziehen, oder indem Bildarchive geladen werden, die von `docker save` über den Befehl `ctr -n k8s.io image import` oder die xref:add-ons/import-images.adoc#_pre_import_images[Vorimportfunktion] erstellt wurden.
Beachten Sie, dass der Namespace `k8s.io` bei der Verwaltung von Bildern über `ctr` angegeben werden muss, damit sie für das Kubelet sichtbar sind.
