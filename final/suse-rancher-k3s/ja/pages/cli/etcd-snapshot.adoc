= k3s etcd-snapshot
:page-languages: [en、ja、ko、zh]。
:revdate: 2025-09-17
:page-revdate: {revdate}

このページでは、`k3s etcd-snapshot` CLI ツールを使用して etcd スナップショットを管理する方法と、etcd スナップショットからリストアする方法について説明します。

K3s etcdスナップショットはノードのファイルシステムに保存され、オプションで災害復旧シナリオ用にS3互換オブジェクトストアにアップロードすることもできます。スナップショットは、定期的なスケジュールで自動化することも、オンデマンドで手動で取得することもできる。`k3s etcd-snapshot` CLIツールには、スナップショットの作成、削除、管理に使用できるサブコマンドのセットが用意されています。

|===
| サブコマンド | 説明

| `delete`
| 指定されたスナップショットを削除する

| `ls`, `list`, `l`
| スナップショット一覧

| `prune`
| 設定した保持数を超えたスナップショットを削除する

| `save`
| オンデマンドetcdスナップショットのトリガー
|===

etcd snapshotサブコマンドの詳細については、`k3s etcd-snapshot --help` を実行してください。

== スナップショットの作成

[tabs,sync-group-id=snapshots]
======
予定::
+
--
スケジュール・スナップショットはデフォルトで有効で、システム時間の00:00と12:00に、5つのスナップショットが保持される。スケジュールされたスナップショットの名前は、`etcd-snapshot` で始まり、ノード名とタイムスタンプが続きます。

以下のオプションは、スケジュールされたスナップショットの動作を制御します：

|===
| フラッグ | 説明

| `--etcd-disable-snapshots`
| スケジュールされたスナップショットを無効にする。

| `--etcd-snapshot-name`
| etcd スケジュールスナップショットのベース名を設定します。(デフォルト:`etcd-snapshot`)

| `--etcd-snapshot-compress`
| etcdスナップショットを圧縮する。

| `--etcd-snapshot-dir`
| dbスナップショットを保存するディレクトリ。(デフォルトの場所:`$<data-dir>/db/snapshots`)

| `--etcd-snapshot-retention`
| 保持するスナップショット数。(デフォルト：5)

| `--etcd-snapshot-schedule-cron`
| cron仕様のスナップショット間隔時間。例えば、5時間ごとに`0 */5 * * *` （デフォルト：`0 */12 * * *` ）。
|===

data-dir値はデフォルトで`/var/lib/rancher/k3s` 、`--data-dir` フラグを設定することで個別に変更できる。

スケジュールされたスナップショットは、サーバーの`--etcd-snapshot-dir` 値で設定されたパスに保存される。S3互換のオブジェクトストアにレプリケートしたい場合は、xref:#_s3_compatible_object_store_support[S3の設定]オプションを参照してください。
--

オンデマンド::
+
--
スナップショットは、`k3s etcd-snapshot save` コマンドを実行して手動で保存できます。これらのオンデマンドスナップショットには保持機能がないため、`k3s etcd-snapshot delete` または`k3s etcd-snapshot prune` コマンドを使用して手動で削除する必要があります。オンデマンドスナップショットの名前は、`on-demand` で始まり、ノード名とタイムスタンプが続きます。

以下のオプションは、オンデマンドスナップショットの動作を制御します：

|===
| フラッグ | 説明

| `--name`
| etcdオンデマンドスナップショットのベース名を設定します。(デフォルト:`on-demand`)

| `--etcd-snapshot-compress`
| etcdスナップショットを圧縮する。

| `--etcd-snapshot-dir`
| dbスナップショットを保存するディレクトリ。(デフォルトの場所:`$<data-dir>/db/snapshots`)
|===

data-dir値はデフォルトで`/var/lib/rancher/k3s` 、`--data-dir` フラグを設定することで個別に変更できる。

`--name` フラグは、`k3s etcd-snapshot save` コマンド実行時にのみ設定できる。他の2つは、`k3s server` xref:installation/configuration.adoc#_configuration_file[コンフィギュレーション・ファイルの]一部とすることもできる。

オンデマンドスナップショットは、サーバーの`--etcd-snapshot-dir` 値で設定されたパスに保存される。S3互換のオブジェクトストアにレプリケートしたい場合は、xref:#_s3_compatible_object_store_support[S3の設定]オプションを参照してください。
--
======

== スナップショットの削除

スケジュールされたスナップショットは、スナップショットの数が設定された保持数（デフォルトでは5）を超えると自動的に削除されます。最も古いスナップショットが最初に削除される。

スケジュールスナップショットまたはオンデマンドスナップショットを手動で削除するには、`k3s etcd-snapshot delete` コマンドを使用します：

[,bash]
----
k3s etcd-snapshot delete <SNAPSHOT-NAME-1> <SNAPSHOT-NAME-2> ...
----

`prune` サブコマンドは、名前のプレフィックス（デフォルトでは`on-demand` ）に一致し、設定された保持カウントを超えるスナップショットを削除します。保持回数を設定するフラグ`--snapshot-retention` 。スケジュールされたスナップショットについては、デフォルトの保持ポリシーを上書きします。オンデマンドスナップショットには保持ポリシーがないため、このフラグが必要です。

オンデマンド」スナップショットをより少ない量に刈り込む：

[,bash]
----
k3s etcd-snapshot prune --snapshot-retention  <NUM-OF-SNAPSHOTS-TO-RETAIN>
----

スケジュールされた "スナップショットをより少ない量にプルーンする：

[,bash]
----
k3s etcd-snapshot prune --name etcd-snapshot --etcd-snapshot-retention <NUM-OF-SNAPSHOTS-TO-RETAIN>
----

=== S3互換オブジェクトストアのサポート

K3sは、etcdスナップショットのS3互換オブジェクトストアへのレプリケーションと、S3互換オブジェクトストアからのetcdスナップショットのリストアをサポートしています。S3のサポートは、オンデマンドとスケジュールされたスナップショットの両方で利用可能です。

|===
| フラッグ | 説明

| `--etcd-s3`
| S3へのバックアップを有効にする

| `--etcd-s3-endpoint`
| S3エンドポイントのURL

| `--etcd-s3-endpoint-ca`
| S3 エンドポイントに接続するための S3 カスタム CA 証明書

| `--etcd-s3-skip-ssl-verify`
| S3 SSL証明書の検証を無効にする

| `--etcd-s3-access-key`
| S3アクセスキー

| `--etcd-s3-secret-key`
| S3の秘密鍵

| `--etcd-s3-bucket`
| S3バケット名

| `--etcd-s3-region`
| S3 リージョン / バケットの場所 (オプション)。デフォルトは`us-east-1` 。

| `--etcd-s3-folder`
| S3フォルダ

| `--etcd-s3-proxy`
| S3への接続時に使用するプロキシサーバーで、プロキシで解放された環境変数を上書きする。

| `--etcd-s3-insecure`
| S3 over HTTPSを無効にする

| `--etcd-s3-timeout`
| S3 タイムアウト (デフォルト:`5m0s`)

| `--etcd-s3-config-secret`
| etcd-s3が有効であり、他のetcd-s3オプションが設定されていない場合に、S3の構成に使用されるkube-system名前空間内のsecretの名前
|===

例えば、S3におけるオンデマンドetcdスナップショットの作成と削除はこのように動作する：

[,shell-session]
----
$ k3s etcd-snapshot --s3 --s3-bucket=test-bucket --s3-access-key=test --s3-secret-key=secret save
INFO[0155] Snapshot on-demand-server-0-1753178523 saved. 
INFO[0155] Snapshot on-demand-server-0-1753178523 saved. 

$ k3s etcd-snapshot --s3 --s3-bucket=test-bucket --s3-access-key=test --s3-secret-key=secret ls
Name                              Location                                                                          Size    Created
on-demand-server-0-1753178523     s3://test-bucket/test-folder/on-demand-server-0-1753178523                        5062688 2025-07-22T10:02:03Z
on-demand-server-0-1753178523     file:///var/lib/rancher/k3s/server/db/snapshots/on-demand-server-0-1753178523     5062688 2025-07-22T10:02:03Z

$ k3s etcd-snapshot --s3 --s3-bucket=test-bucket --s3-access-key=test --s3-secret-key=secret delete on-demand-server-0-1753178523
INFO[0000] Snapshot on-demand-server-0-1753178523 deleted.

$ k3s etcd-snapshot --s3 --s3-bucket=test-bucket --s3-access-key=test --s3-secret-key=secret ls
Name                              Location                                                                          Size    Created
----

=== S3 コンフィギュレーション・シークレットのサポート

[NOTE]
.バージョン・ゲート
====
S3 Configuration Secretのサポートは、2024年8月のリリースで利用可能です：v1.30.4+k3s1、v1.29.8+k3s1、v1.28.13+k3s1。
====

K3sはKubernetes Secretからetcd S3スナップショット設定の読み込みをサポートしている。
これは、セキュリティ上の理由や、K3sを再起動せずにクレデンシャルをローテーションする必要がある場合に、K3sのCLIフラグや設定ファイルにクレデンシャルをハードコードするよりも好ましいかもしれません。
シークレット経由でS3スナップショットの設定を渡すには、`--etcd-s3` と`--etcd-s3-config-secret=<SECRET-NAME>` でK3sを起動します。
シークレットはK3sの起動時に存在する必要はありませんが、スナップショットの保存/リスト/削除/プルーンの操作が実行されるたびにチェックされます。

apiserverはリストア中にシークレットを提供することができないため、スナップショットをリストアする際にS3 config Secretを使用することはできません。
S3に保存されたスナップショットをリストアする際は、CLI経由でS3の設定を渡す必要がある。

[NOTE]
====
シークレットを有効にするには`--etcd-s3` と`--etcd-s3-config-secret` フラグのみを渡す。
他のS3設定フラグが設定されている場合、Secretは無視される。
====

シークレットのキーは、上記の`--etcd-s3-*` CLIフラグに対応しています。
`etcd-s3-endpoint-ca` 、PEMエンコードされたCAバンドルを受け付ける。また、`etcd-s3-endpoint-ca-name` 、1つ以上のPEMエンコードされたCAバンドルを含む`kube-system` 名前空間内のConfigMapの名前を指定するために、このキーを使用することもできる。

[,yaml]
----
apiVersion: v1
kind: Secret
metadata:
  name: k3s-etcd-snapshot-s3-config
  namespace: kube-system
type: etcd.k3s.cattle.io/s3-config-secret
stringData:
  etcd-s3-endpoint: ""
  etcd-s3-endpoint-ca: ""
  etcd-s3-endpoint-ca-name: ""
  etcd-s3-skip-ssl-verify: "false"
  etcd-s3-access-key: "AWS_ACCESS_KEY_ID"
  etcd-s3-secret-key: "AWS_SECRET_ACCESS_KEY"
  etcd-s3-bucket: "bucket"
  etcd-s3-folder: "folder"
  etcd-s3-region: "us-east-1"
  etcd-s3-insecure: "false"
  etcd-s3-timeout: "5m"
  etcd-s3-proxy: ""
----

== スナップショットの復元

K3sはスナップショットをリストアする際、いくつかのステップを踏む：

. スナップショットがS3に保存されている場合、ファイルはスナップショットディレクトリにダウンロードされます。
. スナップショットが圧縮されている場合は解凍される。
. 存在する場合、現在のetcdデータベースファイルは`<data-dir>/server/db/etcd-old-$TIMESTAMP/` に移動される。
. スナップショットの内容がディスクに展開され、チェックサムが検証される。
. Etcdが開始され、現在のノード以外のすべてのetcdクラスタ・メンバがクラスタから削除される。
. CA証明書やその他の機密データは、データストアから抽出され、後で使用できるようにディスクに書き込まれる。
. リストアは完了し、K3sはリストアが実行されたサーバー上で再起動し、通常通り使用することができます。
. (オプション）エージェントとコントロールプレーン・サーバーは通常通り起動できる。
. (オプション) 古いデータベース・ファイルを削除した後、クラスタに再参加するためにEtcdサーバを再起動することができます。

スナップショットをリストアする場合、スナップショットを作成したK3sと同じバージョンを使用する必要はありません。

[#_snapshot_restore_steps]
=== スナップショットの復元手順

クラスタ構成に合わせて以下のタブを選択します。

[tabs,sync-group-id=etcdsnap]
======
シングルサーバー::
+
--
. K3sのサービスを停止する：
+
[,bash]
----
systemctl stop k3s
----

. `--cluster-reset` フラグを付けて`k3s server` を実行し、`--cluster-reset-restore-path` はリストアするスナップショットへのパスを示す。
   スナップショットがS3に保存されている場合は、S3の設定フラグ（`--etcd-s3` 、`--etcd-s3-bucket` など）を指定し、スナップショットのファイル名のみをリストアパスとして指定します。
+
[NOTE]
====
リストアするスナップショットを指定せずに`--cluster-reset` フラグを使用すると、スナップショットをリストアせずにetcdクラスタを単一メンバーにリセットするだけです。
====
+
[,bash]
----
k3s server \
  --cluster-reset \
  --cluster-reset-restore-path=<PATH-TO-SNAPSHOT>
----
+
**結果**K3sはスナップショットをリストアし、クラスタ・メンバーシップをリセットし、再起動の準備ができたことを示すメッセージを表示します：
`Managed etcd cluster membership has been reset, restart without --cluster-reset flag now.`

. K3を再開する：
+
[,bash]
----
systemctl start k3s
----

etcd-s3バックアップ設定がK3s設定ファイル内で定義されている場合、K3sリストアは設定されたS3バケットからスナップショットファイルを取得しようとします。この場合、スナップショットのファイル名のみを引数`--cluster-reset-restore-path` に渡す必要がある。etcd-s3バックアップ設定が存在するローカルのスナップショットファイルからリストアするには、引数`--etcd-s3=false` を追加し、引数`--cluster-reset-restore-path` にローカルのスナップショットファイルへのフルパスを渡します。

安全なメカニズムとして、K3sがクラスタをリセットする際、`/var/lib/rancher/k3s/server/db/reset-flag` に空のファイルを作成し、ユーザーが誤って複数のクラスタリセットを連続して実行することを防ぎます。このファイルはK3sが正常に起動すると削除されます。
--

複数サーバー::
+
--
この例では、`S1` 、`S2` 、`S3` の3つのサーバーがある。スナップショットは`S1` にある。

. すべてのサーバーでK3を停止する：
+
[,bash]
----
systemctl stop k3s
----

. S1上で、`--cluster-reset` オプションを付けて`k3s server` 、リストアするスナップショットへのパスを示す`--cluster-reset-restore-path` 。
   スナップショットがS3に保存されている場合は、S3の設定フラグ（`--etcd-s3` 、`--etcd-s3-bucket` など）を指定し、スナップショットのファイル名のみをリストアパスとして指定します。
+
[NOTE]
====
リストアするスナップショットを指定せずに`--cluster-reset` フラグを使用すると、スナップショットをリストアせずにetcdクラスタを単一メンバーにリセットするだけです。
====
+
[,bash]
----
k3s server \
  --cluster-reset \
  --cluster-reset-restore-path=<PATH-TO-SNAPSHOT>
----
+
**結果**K3sはスナップショットをリストアし、クラスタ・メンバーシップをリセットし、再起動の準備ができたことを示すメッセージを表示します：
`Managed etcd cluster membership has been reset, restart without --cluster-reset flag now.`
`Backup and delete $\{datadir}/server/db on each peer etcd server and rejoin the nodes.`

. S1で再びK3を始める：
+
[,bash]
----
systemctl start k3s
----

. S2とS3で、データ・ディレクトリを削除する。`/var/lib/rancher/k3s/server/db/` ：
+
[,bash]
----
rm -rf /var/lib/rancher/k3s/server/db/
----

. S2とS3で、K3を再度起動して、リストアされたクラスタに参加する：
+
[,bash]
----
systemctl start k3s
----

etcd-s3バックアップ設定がK3s設定ファイル内で定義されている場合、K3sリストアは設定されたS3バケットからスナップショットファイルを取得しようとします。この場合、スナップショットのファイル名のみを引数`--cluster-reset-restore-path` に渡す必要がある。etcd-s3バックアップ設定が存在するローカルのスナップショットファイルからリストアするには、引数`--etcd-s3=false` を追加し、引数`--cluster-reset-restore-path` にローカルのスナップショットファイルへのフルパスを渡します。

安全なメカニズムとして、K3sがクラスタをリセットする際、`/var/lib/rancher/k3s/server/db/reset-flag` に空のファイルを作成し、ユーザーが誤って複数のクラスタリセットを連続して実行することを防ぎます。このファイルはK3sが正常に起動すると削除されます。
--
======

==== 新しいホストへのリストア

etcdスナップショットを取得したホストとは異なるホストにリストアすることは可能です。その際、スナップショット内のブートストラップデータの復号化に使用されるため、スナップショット取得時に元々使用されていたxref:cli/token.adoc#_server[サーバートークンを]渡す必要があります。手順は上記と同じだが、ステップ2を次のように変更する：

. スナップショットを撮ったノードで、`/var/lib/rancher/k3s/server/token` の値を保存する。これはステップ3で`<BACKED-UP-TOKEN-VALUE>` 。
. スナップショットを新しいノードにコピーします。ノード内のパスはステップ3で`<PATH-TO-SNAPSHOT>` 。
. 以下のコマンドを使用して、最初のサーバーノードのスナップショットからリストアを開始します：
+
[,bash]
----
k3s server \
  --cluster-reset \
  --cluster-reset-restore-path=<PATH-TO-SNAPSHOT>
  --token=<BACKED-UP-TOKEN-VALUE>
----
+
トークンの値はK3sの設定ファイルで設定することもできる。


[WARNING]
======
. ノードリソースもetcdスナップショットに含まれる。新しいノード・セットにリストアする場合は、クラスタに存在しなくなった古いノードを手動で削除する必要があります。
. K3sの設定ファイルにトークンが設定されている場合は、`<BACKED-UP-TOKEN-VALUE>` 、そうでない場合はK3sの起動に失敗します。
======

== ETCDSnapshotFile カスタムリソース

[NOTE]
.バージョン・ゲート
====
ETCDSnapshotFilesは2023年11月のリリースで利用可能です: v1.28.4+k3s2、v1.27.8+k3s2、v1.26.11+k3s2、v1.25.16+k3s4。
====

スナップショットは、クラスタスコープの`ETCDSnapshotFile` リソースを一覧表示または記述することで、任意の Kubernetes クライアントを使用してリモートで表示できます。そのノードに表示されているスナップショットのみを表示する`k3s etcd-snapshot list` コマンドとは異なり、`ETCDSnapshotFile` リソースはクラスタ・メンバーに存在するすべてのスナップショットを追跡します。

[,shell-session]
----

s3-on-demand-k3s-server-1-1730308816-79b15c      on-demand-k3s-server-1-1730308816   s3             s3://etcd/k3s-test/on-demand-k3s-server-1-1730308816                                2891808   2024-10-30T17:20:16Z
