= k3sトークン
:page-languages: [en、ja、ko、zh]。
:revdate: 2025-05-26
:page-revdate: {revdate}

K3sはトークンを使用してノード参加プロセスを保護し、データストアに永続化される機密情報を暗号化する。トークンはクラスタと参加ノードを認証し、ノードとクラスタを認証します。

== トークン・フォーマット

K3sトークンは、セキュアまたはショートフォーマットで指定できます。認証情報を送信する前に、クライアントが参加するクラスタのIDを認証できるため、セキュアな形式が望ましい。

=== セキュア

セキュア・トークン形式（「フル」トークンと呼ばれることもある）には、以下の部分が含まれる：

`<prefix><cluster CA hash>::<credentials>`

* `prefix``K10` トークンの形式を識別する固定のプレフィックス。
* `cluster CA hash`:クラスターのサーバーCA証明書のハッシュで、参加ノードに対するサーバーの認証に使用されます。
 ** 自己署名CA証明書の場合、これはディスクに保存されているPEM形式の証明書のSHA256和である。
 ** カスタムCA証明書の場合、これはルート証明書のDERエンコーディングのSHA256和であり、一般に証明書フィンガープリントとして知られている。
* `credentials`:クラスタに参加するノードの認証に使用するユーザ名とパスワード、またはベアラ・トークン。

==== TLSブートストラッピング

セキュアトークンが指定されると、参加ノードは以下の手順を実行し、接続先サーバーのIDを検証してから認証情報を送信する：

. TLS検証を無効にして、参加するサーバーの`/cacerts` 、CAバンドルをダウンロードする。
. 上記のように、CA証明書のSHA256ハッシュを計算する。
. 計算されたSHA256ハッシュとトークンのハッシュを比較する。
. ハッシュが一致する場合、サーバーから提示された証明書がサーバーのCAバンドルによって検証可能であることを検証する。
. サーバ証明書が有効な場合は、トークンの種類に応じて、ベーシック認証またはベアラ・トークン認証を使用してクラスタに参加するための認証情報を提示します。

=== ショート

短いトークン形式には、クラスタに参加するノードを認証するために使用されるパスワードまたはベアラ・トークンだけが含まれます。

ショートトークンが使用される場合、参加ノードはサーバーから提示されたCAバンドルを暗黙的に信頼する。最初の接続は、https://en.wikipedia.org/wiki/Man-in-the-middle_attack[マン・イン・ザ・ミドル]攻撃に対して脆弱かもしれない。

== トークンの種類

K3sは3種類のトークンをサポートしている。デフォルトではサーバートークンのみが利用可能で、その他のトークンタイプは管理者が設定または作成する必要があります。

|===
| タイプ | CLIオプション | 環境変数

| サーバー
| `--token`
| `K3S_TOKEN`

| 代理店
| `--agent-token`
| `K3S_AGENT_TOKEN`

| ブートストラップ
| `n/a`
| `n/a`
|===

[#_server]
=== サーバー

クラスタ内の最初のサーバを起動するときにトークンが提供されない場合、ランダムなパスワードでトークンが作成されます。サーバートークンは常に安全な形式で`/var/lib/rancher/k3s/server/token` に書き込まれる。

サーバ・トークンは、サーバ・ノードとエージェント・ノードの両方をクラスタに参加させるために使用できます。サーバ・トークンにアクセスできる者は、実質的にクラスタに対する完全な管理者アクセス権を持つ。このトークンは慎重に守られなければならない。

サーバートークンは、https://en.wikipedia.org/wiki/PBKDF2[]PBKDF2 パスフレーズとしても使用され、ブートストラップデータとして知られるデータストアに永続化される機密情報を暗号化する。ブートストラップ・データは、新しいサーバー・ノードをセットアップしたり、スナップショットからリストアしたりするのに不可欠です。このため、トークンはクラスタ・データストアそのものと一緒にバックアップする必要があります。

[CAUTION]
====
カスタムCA証明書が使用されていない限り、クラスタ内の最初のサーバを起動するときに使用できるのは、短い（パスワードのみの）トークン形式のみです。なぜなら、クラスタCAハッシュは、サーバが自己署名クラスタCA証明書を生成した後でなければ知ることができないからである。
====


カスタムCA証明書の使用に関する詳細は、xref:cli/certificate.adoc[`k3s certificate` のドキュメントを]参照してください。 +
クラスタのバックアップの詳細については、xref:datastore/backup-restore.adoc[「Backup and Restore」の]ドキュメントを参照してください。

=== 代理店

デフォルトでは、エージェントトークンはサーバートークンと同じです。エージェント・トークンは、クラスタ内のすべてのサーバでCLIオプションまたは環境変数を変更することで、クラスタの起動前または起動後に設定できます。エージェント・トークンは、サーバ・トークンと同様に静的に設定され、有効期限はありません。

エージェント・トークンは、安全な形式で`/var/lib/rancher/k3s/server/agent-token` に書き込まれる。エージェント・トークンが指定されていない場合、このファイルはサーバー・トークンへのリンクとなります。

=== ブートストラップ

[IMPORTANT]
.バージョン・ゲート
====
`k3s token` コマンドのサポートと、ブートストラップトークンによるノードへの参加機能は、2023-02 リリース (v1.26.2+k3s1, v1.25.7+k3s1, v1.24.11+k3s1, v1.23.17+k3s1) から利用可能です。
====

K3sは、動的に生成され、自動的に失効するエージェントhttps://kubernetes.io/docs/reference/access-authn-authz/bootstrap-tokens/[ブートストラップトークンを]サポートしています。

== k3sトークン

k3sトークンCLIツールは処理します：

* ブートストラップ・トークンのライフサイクル。`kubeadm token` ブートストラップ・トークンと同じ生成および検証コードを使用する。どちらのCLIも似ていることに注意。
* サーバートークンのローテーション

----
NAME:
   k3s token - Manage tokens

USAGE:
   k3s token command [command options] [arguments...]

COMMANDS:
   create    Create bootstrap tokens on the server
   delete    Delete bootstrap tokens on the server
   generate  Generate and print a bootstrap token, but do not create it on the server
   list      List bootstrap tokens on the server
   rotate    Rotate original server token with a new token

OPTIONS:
   --help, -h  show help
----

[discrete]
=== `k3s token create [token]`

新しいトークンを作成する。`[token]` は、`k3s token generate` によって生成された、実際に書き込むトークンである。トークンが与えられない場合は、ランダムなトークンが生成される。

クラスタCAハッシュを含む安全な形式のトークンが標準出力に書き出されます。トークンの秘密部分は二度と表示できないので、このコマンドの出力は保存しておくこと。

|===
| フラッグ | 説明

| `--data-dir` 価値
| 状態を保持するフォルダ (デフォルト: /var/lib/rancher/k3s または ${HOME}/.rancher/k3s if not root)

| `--kubeconfig` 価値
| 接続先サーバー [$KUBECONFIG］

| `--description` 価値
| このトークンがどのように使用されるかを、人間にわかりやすく説明する。

| `--groups` 価値
| このトークンが認証に使用される際に認証される追加グループ。(デフォルトはデフォルト："system:bootstrappers:k3s:default-node-token")

| `--ttl` 価値
| トークンが自動的に削除されるまでの時間（例：1s、2m、3h）。0」に設定すると、トークンは期限切れにならない（デフォルト：24h0m0s)

| `--usages` 価値
| このトークンの使用方法を記述します。(デフォルト: "signing,authentication")
|===

[discrete]
=== `k3s token delete`

| `--kubeconfig` 価値
| 接続先サーバー [$KUBECONFIG］
|===

[discrete]
=== `k3s token generate`

ランダムに生成されたブートストラップトークンを生成する。

トークンを生成するためにこのコマンドを使う必要はない。`[a-z0-9]\{6}.[a-z0-9]\{16}` 。最初の部分がトークンIDで、2番目の部分がシークレットです。

|===
| フラッグ | 説明

| `--data-dir` 価値
| 状態を保持するフォルダ (デフォルト: /var/lib/rancher/k3s または ${HOME}/.rancher/k3s if not root)

| `--kubeconfig` 価値
| 接続先サーバー [$KUBECONFIG］
|===

[discrete]
=== `k3s token list`

ブートストラップ・トークンを一覧表示し、ID、説明、残りの有効期限を表示します。

|===

[discrete]
=== `k3s token rotate`

[IMPORTANT]
.バージョン・ゲート
====
2023年10月リリース（v1.28.2+k3s1、v1.27.7+k3s1、v1.26.10+k3s1、v1.25.15+k3s1）で利用可能。
====


元のサーバートークンと新しいサーバートークンを入れ替えます。このコマンドを実行した後、古いトークンで参加したすべてのサーバーとエージェントは、新しいトークンで再起動する必要があります。

新しいトークンを指定しない場合は、トークンが生成されます。

|===
| フラッグ | 説明

| `--data-dir` 価値
| 状態を保持するフォルダ (デフォルト: /var/lib/rancher/k3s または ${HOME}/.rancher/k3s if not root)

| `--kubeconfig` 価値
| 接続先サーバー [$KUBECONFIG］

| `--server` 価値
| 接続先サーバー (デフォルト: "https://127.0.0.1:6443") [$K3S_URL].

| `--token` 価値
| サーバーまたはエージェントをクラスタに参加させるために使用される既存のトークン [$K3S_TOKEN].

| `--new-token` 価値
| 既存のトークンを置き換える新しいトークン
|===

[WARNING]
====
ローテーション前に取得されたスナップショットは、クラスタをリストアするときに古いサーバ・トークンを必要とします。
====
