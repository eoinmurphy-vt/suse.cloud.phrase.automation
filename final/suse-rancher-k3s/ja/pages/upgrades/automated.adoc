= 自動アップグレード
:page-languages: [en、ja、ko、zh]。
:revdate: 2025-09-16
:page-revdate: {revdate}

== 概要

Rancherのsystem-upgrade-controllerを使ってK3sクラスタのアップグレードを管理できます。これはクラスタのアップグレードに対するKubernetesネイティブなアプローチだ。これはhttps://github.com/rancher/system-upgrade-controller/blob/master/doc/plan.md#planspec[`Plan`]カスタムリソースを利用して、どのノードをどのバージョンにアップグレードするかを宣言的に記述します。

この計画は、アップグレードの方針と要件を定義するものである。また、https://kubernetes.io/docs/concepts/overview/working-with-objects/labels/[ラベルセレクタによって]、どのノードをアップグレードすべきかを定義する。K3sクラスタのアップグレードに適したデフォルトのプランについては、以下を参照してください。より高度なプラン設定オプションについては、上記のリンク先のプランに関する文書を参照してください。

システム・アップグレード・コントローラーは、プランを監視し、アップグレードhttps://kubernetes.io/docs/concepts/workloads/controllers/jobs-run-to-completion/[ジョブを]実行するノードを選択することによって、アップグレードをスケジュールします。ジョブが正常に完了すると、コントローラはそのジョブが実行されたノードにラベルを付けます。

[WARNING]
====
K3sクラスタがRancherによって管理されている場合は、Rancher UIを使用してアップグレードを管理する必要があります。

* K3sクラスタがRancherにインポート（登録）されている場合、Rancherはデフォルトでsystem-upgrade-controllerのデプロイと計画を管理します。Rancherでバージョン管理を無効にしていない限り、このページのステップに従わないでください。
  詳細については、https://rancher.github.io/product-docs-playbook/rancher-manager/latest/en/cluster-deployment/register-existing-clusters.html#_configuring_version_management_for_suse_rancher_prime_rke2_and_suse_rancher_prime_k3s_clusters[Configuring Version Management for RKE2 and K3s Clustersを]参照してください。
* K3sクラスタがRancherによってプロビジョニングされている場合、Rancherはバージョンアップを管理するためにシステムエージェントを使用します。このページの手順には従わないでください。
* K3sクラスタがRancherによって管理されて*いない*場合は、以下の手順に従ってください。
====

== システムアップグレードコントローラーの使用

アップグレードを自動化するには、以下のようにする必要がある：

. system-upgrade-controllerをクラスタにインストールする。
. どのノードのグループをどのようにアップグレードするかを記述した計画を作成する。

system-upgrade-controllerの設計とアーキテクチャ、またはK3sとの統合の詳細については、以下のGitリポジトリを参照してください：

* https://github.com/rancher/system-upgrade-controller[system-upgrade-controller]
* https://github.com/k3s-io/k3s-upgrade[k3s-upgrade]

[TIP]
====
K3sの新しいバージョンにアップグレードしようとすると、https://kubernetes.io/docs/setup/release/version-skew-policy/[]Kubernetes適用されます。アップグレードの際、プランが中間のマイナーバージョンをスキップしないようにしてください。system-upgrade-controller自体は、Kubernetesのバージョンに対するサポートされていない変更から保護しない。
====


=== インストール

system-upgrade-controller マニフェストは、カスタムリソース定義、デプロイメント、サービスアカウント、クラスタロールバインディング、および configmap をインストールします。これらのコンポーネントをインストールするには、以下のコマンドを実行する：

[,bash]
----
kubectl apply -f https://github.com/rancher/system-upgrade-controller/releases/latest/download/crd.yaml -f https://github.com/rancher/system-upgrade-controller/releases/latest/download/system-upgrade-controller.yaml
----

コントローラは、前述のconfigmapで設定およびカスタマイズできますが、変更を適用するには、コントローラのポッドを削除する必要があります。

=== 構成

サーバーノードは常にエージェントノードより先にアップグレードする必要があります。このため、サーバー（制御プレーン）ノードのアップグレード用プランと、エージェントノードのアップグレード用プランの、少なくとも2つのプランを作成することをお勧めします。必要に応じて追加のプランを作成し、ノード間でのアップグレードのロールアウトを制御できます。プランが作成されると、コントローラがそれを受け取り、クラスタのアップグレードを開始します。

以下の2つのプラン例では、安定xref:upgrades/manual.adoc#_release_channels[リリースチャネルを]ターゲットにすることで、クラスタを継続的に現在の安定リリースにアップグレードし続けます：

[,yaml]
----
# Server plan
apiVersion: upgrade.cattle.io/v1
kind: Plan
metadata:
  name: server-plan
  namespace: system-upgrade
spec:
  concurrency: 1
  cordon: true
  nodeSelector:
    matchExpressions:
    - key: node-role.kubernetes.io/control-plane
      operator: In
      values:
      - "true"
  serviceAccountName: system-upgrade
  upgrade:
    image: rancher/k3s-upgrade
  channel: https://update.k3s.io/v1-release/channels/stable
---
# Agent plan
apiVersion: upgrade.cattle.io/v1
kind: Plan
metadata:
  name: agent-plan
  namespace: system-upgrade
spec:
  concurrency: 1
  cordon: true
  nodeSelector:
    matchExpressions:
    - key: node-role.kubernetes.io/control-plane
      operator: DoesNotExist
  prepare:
    args:
    - prepare
    - server-plan
    image: rancher/k3s-upgrade
  serviceAccountName: system-upgrade
  upgrade:
    image: rancher/k3s-upgrade
  channel: https://update.k3s.io/v1-release/channels/stable
----

これらのプランに関して、いくつか重要なことがある：

. プランは、コントローラをデプロイするのと同じ名前空間に作成する必要があります。
. `concurrency` フィールドは、同時にアップグレードできるノードの数を示す。
. サーバープランは、`node-role.kubernetes.io/control-plane` ラベルを持つノードを選択するラベルセレクタを指定することで、サーバーノードをターゲットにする。エージェント・プランは、そのラベルを持たないノードを選択するラベル・セレクタを指定することで、エージェント・ノードをターゲットにする。
. エージェントプランの`prepare` ステップは、そのプランのアップグレードジョブが実行される前に、サーバープランの完了を待つようにします。このロジックは、準備ステップに使用されるイメージに組み込まれており、system-upgrade-controller自体の一部ではない。
. どちらのプランも、`channel` フィールドには安定版リリース・チャンネルの URL が設定されている。これにより、コントローラはそのURLを監視し、新しいリリースに解決するたびにクラスタをアップグレードします。これはxref:upgrades/manual.adoc#_release_channels[リリースチャンネルと]相性がいい。したがって、以下のチャンネルでプランを設定することで、クラスタが常にK3sの最新の安定リリースに自動的にアップグレードされるようにすることができます。あるいは、`channel` フィールドを省略し、`version` フィールドにK3sの特定のリリースを設定することもできる：
+
[,yaml]
----
apiVersion: upgrade.cattle.io/v1
kind: Plan
# ...
spec:
  # ...
  version: v1.33.4+k3s1
----

バージョンフィールドから、またはチャネルサーバをポーリングして、コントローラがプランのターゲットバージョンが解決されたことを検出するとすぐに、アップグレードが開始されます。プランを変更すると、コントローラはプランを再評価し、再度アップグレードが必要かどうかを判断します。チャンネルが設定されている場合、そのURLも定期的にポーリングされ、新しいバージョンがないかチェックされる。

kubectl を使ってプランとジョブを表示することで、アップグレードの進捗を監視できます：

[,bash]
----
kubectl -n system-upgrade get plans -o wide
kubectl -n system-upgrade get jobs
----

=== アップグレードのスケジューリング

プランは、プラン仕様内の`window` フィールドを設定することにより、特定の時間枠内で発生するよう制限することができます。タイムウィンドウフィールドは、https://kured.dev/docs/configuration/#setting-a-schedule[] kured互換性があり、同じフォーマットをとります。例えば、こうだ：

[,yaml]
----
apiVersion: upgrade.cattle.io/v1
kind: Plan
# ...
spec:
  # ...
  window:
    days:
      - monday
      - tuesday
      - wednesday
      - thursday
      - friday
    startTime: 19:00
    endTime: 21:00
    timeZone: UTC
----

あるプランのアップグレードを実行するジョブは、タイムウィンドウの外では作成されない。ジョブが作成された後、ウィンドウが閉じた後もプランが実行され続けることがある。

== ダウングレード防止

[IMPORTANT]
.バージョン・ゲート
====
2023-07リリースから（https://github.com/k3s-io/k3s-upgrade/releases/tag/v1.27.4%2Bk3s1[v1.27.4+k3s1] https://github.com/k3s-io/k3s-upgrade/releases/tag/v1.26.7%2Bk3s1[、] v1 https://github.com/k3s-io/k3s-upgrade/releases/tag/v1.25.12%2Bk3s1[] v1. https://github.com/k3s-io/k3s-upgrade/releases/tag/v1.24.16%2Bk3s1[.26.7+k3s1] v1.）。
====


Kubernetesは制御プレーンコンポーネントのダウングレードをサポートしていない。アップグレードプランで使用されるk3s-upgradeイメージは、K3のダウングレードを拒否し、プランを失敗させます。プランに`cordon: true` が設定されているノードは、障害発生後もコードネーションを維持する。

以下はクラスタの例で、アップグレードに失敗したポッドとコード化されたノードを示しています：

[,console]
----
$ kubectl get pods -n system-upgrade

apply-k3s-server-on-ip-172-31-10-23-with-7af95590a5af8e8c-9xvwg   0/1     Error     0          14m
apply-k3s-server-on-ip-172-31-13-213-with-7af95590a5af8e8-8j72v   0/1     Error     0          18m
system-upgrade-controller-7c4b84d5d9-kkzr6                        1/1     Running   0          20m
$ kubectl get nodes
NAME               STATUS                     ROLES                       AGE   VERSION
ip-172-31-0-16     Ready,SchedulingDisabled   control-plane,etcd,master   19h   v1.27.4+k3s1
ip-172-31-10-23    Ready,SchedulingDisabled   control-plane,etcd,master   19h   v1.27.4+k3s1
ip-172-31-13-213   Ready,SchedulingDisabled   control-plane,etcd,master   19h   v1.27.4+k3s1
ip-172-31-2-13     Ready                      <none>                      19h   v1.27.4+k3s1
----

コード化されたノードは、以下のいずれかの方法でサービスに戻すことができます：

* 計画が成功するように、クラスタで現在実行されているものと同じか新しいリリースをターゲットにするように、計画のバージョンまたはチャネルを変更してください。
* プランを削除し、手動でノードのコードを解除する。
`kubectl get plan -n system-upgrade` でプラン名を検索し、`kubectl delete plan -n system-upgrade PLAN_NAME` で削除する。プランが削除されたら、`kubectl uncordon NODE_NAME` を使用して、各ノードのコードを解除する。

== セキュリティ

起動されるアップグレード・ジョブは、基礎となるノードに変更を加えるために高度な特権を持つ必要があります。デフォルトでは、以下のように設定されている：

* ホスト`IPC` 、`NET` 、`PID` 名前空間
* `CAP_SYS_BOOT` 。
* ホストのルートは、読み取りと書き込みのパーミッションで`/host` にマウントされている。
