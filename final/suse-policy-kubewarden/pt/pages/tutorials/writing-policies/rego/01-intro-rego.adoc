A linguagem Rego é uma linguagem específica de domínio que permite políticas como código.https://www.openpolicyagent.org/docs/latest/policy-language/[Rego]é uma linguagem inspirada no https://en.wikipedia.org/wiki/Datalog[] Datalog.

Há duas maneiras de escrever políticas Rego para implementar políticas como código no Kubernetes, Open Policy Agent e Gatekeeper.

As próximas seções mostram como as duas estruturas são diferentes, embora usem a mesma linguagem.
Se estiver mais interessado na implementação do {short-project-name}, você pode visitar diretamente a documentação específica da estrutura com o link abaixo.

== Um idioma, duas estruturas

=== Agente de política aberta (OPA)

O Open Policy Agent é um projeto que permite que você implemente políticas como código em qualquer projeto.
Você pode usar a OPA para qualquer política com base nas necessidades de seu aplicativo.

Nesse contexto, escrever políticas para o Kubernetes é simplesmente exercitar a OPA.
Ao usar os webhooks de admissão do Kubernetes, é possível avaliar as solicitações usando a OPA, que executa as políticas escritas no Rego.

A OPA tem integração opcional com o Kubernetes por meio de seu sidecar `kube-mgmt`.
Quando implantado no Kubernetes e com o servidor OPA avaliando as políticas do Rego, ele é capaz de replicar os recursos configurados do Kubernetes no Rego.
Portanto, esses recursos do Kubernetes são visíveis para todas as políticas.
Com a OPA, você pode definir políticas dentro dos objetos ConfigMap do Kubernetes.
Você pode ler mais sobre ele emhttps://github.com/open-policy-agent/kube-mgmt[, sua página de projeto].

=== Gatekeeper

O Gatekeeper se concentra apenas no uso no Kubernetes e aproveita isso o máximo possível, tornando os fluxos de trabalho do Kubernetes mais fáceis do que o OPA em determinados casos.

== Observando as diferenças

As políticas do OPA e do Gatekeeper usam o Rego para descrever as políticas como código.
Cada solução tem diferenças quando se trata de escrever políticas no Rego, conforme mostrado nas próximas seções.

== Ponto de entrada

O ponto de entrada é o nome de uma regra em um pacote e é a regra invocada pelo tempo de execução quando a política é executada.

== Limitações atuais

=== Políticas com reconhecimento de contexto

As políticas com reconhecimento de contexto são políticas que não avaliam a solicitação de entrada isoladamente.
Eles levam em conta outros fatores para tomar uma decisão.
Por exemplo, uma política que avalia recursos namespaced e usa uma anotação no namespace pai para configurar algo na política.
Outro exemplo seria uma política que avalia os recursos do `Ingress`, mas para tomar uma decisão também tem a lista dos recursos existentes do `Ingress`.

A ideia de políticas com reconhecimento de contexto também pode se estender a recursos personalizados, portanto, sua política também pode querer avaliar uma solicitação com base nos recursos personalizados usados no momento.

Tanto o OPA quanto o Gatekeeper oferecem suporte a políticas com reconhecimento de contexto, a partir da versão {short-project-name} 1.9.

Mais detalhes sobre políticas com reconhecimento de contexto estãoxref:reference/spec/05-context-aware-policies.adoc[aqui].

=== Políticas mutantes

O Gatekeeper oferece suporte a políticas de mutação, mas o site {short-project-name} ainda não implementou políticas de mutação compatíveis com o Gatekeeper.
Você pode usar políticas que utilizam o SDK do {short-project-name} para escrever políticas de mutação, mas atualmente não é possível executar políticas de mutação do Gatekeeper no {short-project-name}.
