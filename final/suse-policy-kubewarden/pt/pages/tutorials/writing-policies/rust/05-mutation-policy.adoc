= Criação de uma nova política de mutação
:revdate: 2026-03-18
:page-revdate: {revdate}
:description: Configure uma nova política de mutação {project-name} para mutar objetos de entrada com anotações personalizadas e, em seguida, teste-a.
:doc-persona: ["kubewarden-policy-developer"]

:sidebar_label: Criação de uma nova política de mutação
:current-version: {page-origin-branch}

As políticas de mutação são semelhantes às de validação, mas também têm a capacidade de alterar um objeto de entrada.

Eles podem:

* Rejeitar uma solicitação
* Aceitar uma solicitação sem alterar o objeto de entrada
* Mutar o objeto de entrada conforme necessário e aceitar a solicitação

Escrever uma política de mutação {project-name} é simples.
Você usará a política de validação criada nas seções anteriores e, com algumas alterações, a transformará em uma política de mutação.

Sua política usa a mesma lógica de validação definida anteriormente, mas também adiciona uma anotação a todos os pods que têm um nome válido.

Tentando criar um pod como esse:

[subs="+attributes",yaml]
----
apiVersion: v1
kind: Pod
metadata:
  name: nginx
spec:
  containers:
    - name: nginx
      image: nginx:latest
----

Leva à criação desse POD:

[subs="+attributes",yaml]
----
apiVersion: v1
kind: Pod
metadata:
  name: nginx
  annotations:
    kubewarden.policy.demo/inspected: true
spec:
  containers:
    - name: nginx
      image: nginx:latest
----

== Escreva o código de mutação

O código de mutação está na função `validate`.
Você deve alterar essa função para aprovar a solicitação usandohttps://docs.rs/kubewarden-policy-sdk/0.1.0/kubewarden_policy_sdk/fn.mutate_request.html[`mutate_request`]em vez dehttps://docs.rs/kubewarden-policy-sdk/0.1.0/kubewarden_policy_sdk/fn.accept_request.html[`accept_request`].

Esta é a aparência da função `validate` em `lib.rs`:

[subs="+attributes",rust]
----
fn validate(payload: &[u8]) -> CallResult {
    let validation_request: ValidationRequest<Settings> = ValidationRequest::new(payload)?;

    info!(LOG_DRAIN, "starting validation");
    if validation_request.request.kind.kind != apicore::Pod::KIND {
        warn!(LOG_DRAIN, "Policy validates Pods only. Accepting resource"; "kind" => &validation_request.request.kind.kind);
        return kubewarden::accept_request();
    }

    match serde_json::from_value::<apicore::Pod>(validation_request.request.object) {
        // NOTE 1
        Ok(mut pod) => {
            let pod_name = pod.metadata.name.clone().unwrap_or_default();
            if validation_request
                .settings
                .invalid_names
                .contains(&pod_name)
            {
                kubewarden::reject_request(
                    Some(format!("pod name {:?} is not accepted", pod_name)),
                    None,
                    None,
                    None,
                )
            } else {
                // NOTE 2
                let mut new_annotations = pod.metadata.annotations.clone().unwrap_or_default();
                new_annotations.insert(
                    String::from("kubewarden.policy.demo/inspected"),
                    String::from("true"),
                );
                pod.metadata.annotations = Some(new_annotations);

                // NOTE 3
                let mutated_object = serde_json::to_value(pod)?;
                kubewarden::mutate_request(mutated_object)
            }
        }
        Err(_) => {
            // We were forwarded a request we cannot unmarshal or
            // understand, just accept it
            kubewarden::accept_request()
        }
    }
}
----

Em comparação com o código anterior, você fez três alterações:

. Definimos o objeto `pod` como mutável, consulte a palavra-chave `mut`. Isso é necessário porque estenderemos seu atributo `metadata.annotations`.
// TODO: How best to describe the use of clone().
. Esse é o código que pega o objeto `annotations` existente, adiciona o novo e, por fim, coloca o objeto `annotations` atualizado de volta na instância `pod` original.
. Serialize o objeto `pod` em um `serde_json::Value` genérico e, em seguida, retorne uma resposta de mutação.

== Atualizar os testes de unidade

Você pode atualizar o endereço `accept_pod_with_valid_name` em `lib.rs` para que tenha a seguinte aparência:

[subs="+attributes",rust]
----
#[test]
fn accept_pod_with_valid_name() -> Result<(), ()> {
    let mut invalid_names = HashSet::new();
    invalid_names.insert(String::from("bad_name1"));
    let settings = Settings { invalid_names };

    let request_file = "test_data/pod_creation.json";
    let tc = Testcase {
        name: String::from("Pod creation with valid name"),
        fixture_file: String::from(request_file),
        expected_validation_result: true,
        settings,
    };

    let res = tc.eval(validate).unwrap();
    // NOTE 1
    assert!(
        res.mutated_object.is_some(),
        "Expected accepted object to be mutated",
    );

    // NOTE 2
    let final_pod =
        serde_json::from_value::<apicore::Pod>(res.mutated_object.unwrap()).unwrap();
    let final_annotations = final_pod.metadata.annotations.unwrap();
    assert_eq!(
        final_annotations.get_key_value("kubewarden.policy.demo/inspected"),
        Some((
            &String::from("kubewarden.policy.demo/inspected"),
            &String::from("true")
        )),
    );

    Ok(())
}
----

Em comparação com o primeiro teste, há duas alterações:

. Altere a instrução `assert!` para que a solicitação ainda seja aceita, mas também inclua um objeto mutado
. Criou uma instância de `Pod` a partir do objeto mutado que faz parte da resposta.
Afirme que o objeto Pod alterado tem o endereço correto `metadata.annotations`.
