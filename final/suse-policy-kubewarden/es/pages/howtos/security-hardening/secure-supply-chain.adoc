Una infraestructura de cadena de suministro segura puede verificar la validez de sus partes, o eslabones. Permite a usuarios y desarrolladores mostrar la cadena de custodia de sus componentes de software, o artefactos. Es un enfoque activo para mitigar los problemas de seguridad.

El proyecto https://sigstore.dev/[Sigstore] proporciona herramientas e infraestructura para ello. Sirve para validar la integridad de la cadena de suministro de artefactos.

{short-project-name} utiliza https://github.com/sigstore/cosign[`cosign`] junto con la infraestructura https://github.com/SigStore/fulcio[fulcio] yhttps://github.com/sigstore/rekor[rekor] que ofrece el proyecto Sigstore.

Los operadores de clúster pueden configurar {project-name} para que sólo ejecute políticas firmadas por entidades de confianza. Los desarrolladores de políticas pueden firmar sus políticas y publicarlas en un registro.

== Requisitos previos

En las secciones siguientes, deberá instalar algunas herramientas. Sirven para que los usuarios puedan firmar y verificar las firmas de los artefactos OCI. Los ejemplos muestran el uso dehttps://docs.sigstore.dev/quickstart/quickstart-cosign/[`cosign`] yhttps://github.com/kubewarden/kwctl[`kwctl`] para firmar e inspeccionar políticas.

Para utilizar GitHub para firmar políticas es necesario instalarhttps://docs.sigstore.dev/cosign/system_config/installation/#github-actions[GitHub actions].

La firma sin llave utiliza las instancias predeterminadas https://github.com/SigStore/fulcio[fulcio] yhttps://github.com/sigstore/rekor[rekor] proporcionadas por el proyecto Sigstore. Consulte la documentación de Sigstore para más detalles sobre cómo utilizar su propia infraestructura para esto, si es necesario.

== Políticas de firma

{short-project-name} recomienda utilizar la utilidadhttps://github.com/sigstore/cosign[cosign] de Sigstore para firmar pólizas. Esta sección muestra un método basado en claves para firmar políticas. Para ello, debe generar un par de claves públicas y privadas. Las claves generadas ayudan a verificar si los artefactos firmados proceden del usuario esperado. Para generar este par de claves utilice este comando `cosign generate-key-pair`:

[subs="+attributes",bash]
----
cosign generate-key-pair
----

Se le pedirá que introduzca y verifique una contraseña:

[subs="+attributes",console]
----
Enter password for private key: ●●●●●●●●
Enter password for private key again: ●●●●●●●●
Private key written to cosign.key
Public key written to cosign.pub
----

Ahora puede utilizar esta clave para firmar pólizas.

No comparta el archivo de clave privada, `cosign.key`. Se trata de un archivo secreto para uso exclusivo del propietario de la clave para firmar políticas.

Para firmar una política puede utilizar `cosign sign` pasando el argumento de línea de comandos `--key` con su archivo de clave privada:

[subs="+attributes",bash]
----
cosign sign --key cosign.key ghcr.io/kubewarden/policies/user-group-psp:latest
----

Se solicita la contraseña de la clave privada especificada:

[subs="+attributes",console]
----
an error occurred: no provider found for that key reference, will try to load key from disk...
Enter password for private key: ●●●●●●●●
Pushing signature to: ghcr.io/kubewarden/policies/user-group-psp
----

Este comando firma la política creando un nuevo objeto de firma. A continuación, el objeto de firma se carga en el registro, junto con la política. Ahora la política está lista para ser utilizada en una instalación de {short-project-name} utilizando la verificación de firmas.

La misma política puede ser firmada varias veces, por el mismo usuario o por usuarios diferentes. Estas firmas se añaden al objeto de firma junto con la firma original.

Para más información sobre cómo funciona el proceso de firma, consulte lahttps://docs.sigstore.dev/[documentación del proyecto].

=== Firma sin llave

A menudo, las políticas se crean automáticamente mediante canales CI/CD. Esto complica el proceso de generación de claves. Este flujo de trabajo sin llave de Sigstore es para estas situaciones. En lugar de utilizar claves de canto de larga duración, el flujo de trabajo sin claves utiliza autoridades de certificación (CA) y cadenas de certificados.

La clave de certificado generada, de corta duración, se vincula a una cadena de confianza. Se hace mediante una impugnación de identidad para confirmar la identidad del firmante. El tiempo de vida de la clave del certificado es suficiente para que se produzca la firma. El desafío de identidad tiene lugar mediante la autenticación con un proveedor de OpenID Connect (OIDC). La infraestructura pública Fulcio de Sigstore proporciona la cadena de confianza.

La firma utiliza la utilidad cosign de Sigstore.

[subs="+attributes",bash]
----
$ cosign sign ghcr.io/kubewarden/policies/user-group-psp:latest
----

=== Cómo firmar artefactos en los flujos de trabajo de GitHub

Cuando se utiliza la firma sin clave, en una acción de GitHub, `cosign` no necesita que el usuario inicie sesión en un proveedor de OIDC. Un token de GitHub está disponible durante la ejecución del flujo de trabajo de GitHub. Se utiliza para autenticar al usuario y generar las claves efímeras. El proceso de firma es el mismo que se utiliza en el modo sin llave. Este es un ejemplo de cómo el proyecto {short-project-name} firma sus políticas:

.YAML que describe la firma de directivas {short-project-name} 
[%collapsible]
======

[,yaml]
----
# ... beginning of the workflow file ...
jobs:
  build:
    name: Build container image
    runs-on: ubuntu-latest
    steps:
      # ... other steps building the container image ...
      -
      name: Login to GitHub Container Registry
      uses: docker/login-action@v1
      with:
        registry: ghcr.io
        username: ${{ github.repository_owner }}
        password: ${{ inputs.GITHUB_TOKEN }}
      -
      name: Publish Wasm policy artifact to OCI registry with the 'latest' tag
      shell: bash
      if: ${{ startsWith(github.ref, 'refs/heads/') }}
      env:
        COSIGN_EXPERIMENTAL: 1
      run: |
        set -ex
        echo Pushing policy to OCI container registry
        IMMUTABLE_REF=$(kwctl push -o json ${{ PATH_TO_BUILT_WASM_FILE }} ghcr.io/myorg/policies/my-great-policy:latest | jq -r .immutable_ref)
        echo Keyless signing of policy using cosign
        cosign sign ${IMMUTABLE_REF}
      # ... other build steps ...

# ... remainder of the workflow file ...
----

== Verificación de las políticas

Puede comprobar si una póliza está correctamente firmada con `cosign` o `kwctl`. Tienen opciones de línea de comandos similares para comprobar las firmas de las políticas. Para comprobar el binario de firma con una clave, utilice `kwctl` de la siguiente manera:

[subs="+attributes",console]
----
$ kwctl verify -k cosign.pub ghcr.io/kubewarden/policies/user-group-psp:latest
2022-03-29T14:49:31.878180Z  INFO kwctl::verify: Policy successfully verified
----

O `cosign`:

[subs="+attributes",console]
----
$ cosign verify --key cosign.pub ghcr.io/kubewarden/policies/user-group-psp:latest

Verification for ghcr.io/kubewarden/policies/user-group-psp:latest --
The following checks were performed on each of these signatures:
  - The cosign claims were validated
  - The signatures were verified against the specified public key
  - Any certificates were verified against the Fulcio roots.

== Configuración del servidor de directivas para comprobar las firmas de las directivas

Puede configurar {short-project-name} con `ConfigMap` para que sólo ejecute políticas de confianza. La estructura `ConfigMap` se describe enxref:reference/verification-config.adoc#_signature_configuration_reference[Referencia de configuración de firmas]. Se utiliza para verificar una política mediante `kwctl`. `ConfigMap` debe definir las configuraciones permitidas en el campo`verification-config`.

Por ejemplo, desea ejecutar políticas firmadas por la organización {short-project-name} GitHub. Entonces un ejemplo de `ConfigMap` para este escenario sería:

.$ cat kubewarden_signatures.yaml

anyOf: ~

== Referencia de configuración de la firma

Puede validar los requisitos de firma contenidos en un fichero. He aquí un ejemplo:

.Un fichero de requisitos de firma
[%collapsible]
======
[subs="+attributes",yaml]
----
apiVersion: v1

allOf: <1>
  - kind: githubAction
    owner: kubewarden   # mandatory
    annotations:
      env: prod

anyOf: # at least `anyOf.minimumMatches` are required to match <2>
  minimumMatches: 2 # default is 1
  signatures:
  - kind: pubKey
    owner: flavio # optional
    key: .... # mandatory
    annotations:  # optional
      env: prod
      foo: bar
  - kind: pubKey
    owner: victor # optional
    key: .... # mandatory
  - kind: genericIssuer
    issuer: https://github.com/login/oauth
    subject:
      equal: alice@example.com
  - kind: genericIssuer
    issuer: https://token.actions.githubusercontent.com
    subject:
      equal: https://github.com/flavio/policy-secure-pod-images/.github/workflows/release.yml@refs/heads/main
  - kind: genericIssuer
    issuer: https://token.actions.githubusercontent.com
    subject:
      urlPrefix: https://github.com/flavio/
  - kind: genericIssuer
    issuer: https://token.actions.githubusercontent.com
    subject:
      urlPrefix: https://github.com/kubewarden # <- it will be post-fixed with `/` for security reasons
  - kind: githubAction
    owner: flavio   # mandatory
    repo: policy1 # optional
  - kind: pubKey
    owner: alice # optional
    key: .... # mandatory
----
<1> `allOf`
<2> `anyOf`
======

=== Validación de firmas

La configuración anterior contiene las dos secciones resaltadas, `allOf` y`anyOf`:

* `allOf`la política es de confianza sólo si todos los requisitos de firma aquí son válidos.
* `anyOf`la política es de confianza si se cumple el criterio `minimumMatches`.

Arriba, el campo `minimumMatches` es 2. Por tanto, deben cumplirse al menos dos de los requisitos de la firma. El valor por defecto del campo `minimumMatches` es `1`.

Deben cumplirse todos los requisitos de firmas de `allOf` *y* el número mínimo de`anyOf`.

=== Validación de clave pública

Para comprobar que una póliza está firmada con la clave pública correcta, hay que especificar los datos de la clave y el propietario de la misma. En este ejemplo, se establece `kind`, a `pubKey` y el `key` tiene la clave pública. El campo propietario es opcional, pero puede ser útil para aclarar a quién pertenece la clave.

[subs="+attributes",yaml]
----
  - kind: pubKey
    owner: bob # optional
    key: |
      -----BEGIN PUBLIC KEY-----
      MBFKHFDGHKIJH0CAQYIKoZIzj0DAQcDQgAEX0HFTtCfTtPmkx5p1RbDE6HJSGAVD
      BVDF6SKFSF87AASUspkQsN3FO4iyWodCy5j3o0CdIJD/KJHDJFHDFIu6sA==
      -----END PUBLIC KEY-----
----

=== Validación de firmas sin claves

Una política firmada en modo sin clave no tiene una clave pública que verificar. Aún puede verificar la política con los datos OIDC utilizados durante el proceso de firma. Para ello, es necesario definir la validación de la firma como `genericIssuer`.

Es posible verificar la información de la firma:

* `issuer`(obligatorio): Esto coincide con el atributo `Issuer` del certificado generado por Fulcio. Muestra el OIDC utilizado para firmar la política.
* `subject`: Campo utilizado para coincidir con el atributo `Subject` del certificado de Fulcio. El campo `Subject` (Fulcio) contiene el usuario utilizado para autenticarse contra el proveedor OIDC. El campo de verificación, `subject`, puede tener uno de dos subcampos:
 ** `equal`: El `Subject` (Fulcio) del certificado debe ser igual al valor en la validación de la firma;
 ** `urlPrefix`: El valor del campo `Subject` (Fulcio) del certificado debe ir precedido del valor definido en la validación de la firma.

[NOTE]
====

Tanto `cosign verify` como `kwctl inspect` pueden mostrar información sobre firmas sin llave.

====


Por ejemplo, esta configuración significa que la política debe tener una firma sin clave de Alice utilizando el OIDC de GitHub:

[subs="+attributes",yaml]
----
- kind: genericIssuer
  issuer: https://github.com/login/oauth
  subject:
    equal: alice@example.com
----

Esta configuración necesita que la política se firme en acciones de GitHub, desde un repositorio propiedad del usuario de GitHub `flavio`:

[subs="+attributes",yaml]
----
- kind: genericIssuer
  issuer: https://token.actions.githubusercontent.com
  subject:
    urlPrefix: https://github.com/flavio
----

=== Verificación de la firma de las acciones de GitHub

El "tipo", `githubAction` es para validar políticas firmadas en GitHub Actions.
También puede hacerlo con el tipo `genericIssuer`. Para simplificar el proceso de solicitud de firma, utilice dos campos adicionales para `githubAction`:

* `owner` (obligatorio): ID de GitHub del usuario u organización de confianza
* `repo`: el nombre del repositorio en el que confiar

Por ejemplo, el último fragmento, utilizando `genericIssuer`, podría reescribirse como:

[subs="+attributes",yaml]
----
- kind: githubAction
  owner: flavio
----

=== Validación de anotaciones de firma

Todos los tipos de firma pueden tener otros campos de validación opcionales, `annotations`.
Estos campos son datos clave/valor añadidos durante el proceso de firma.

Con {short-project-name}, puede comprobar que la firma de la política es de usuarios de confianza *y* que tienen anotaciones específicas.

La siguiente validación comprueba 2 condiciones para la política:

* que está firmado con una clave específica
* tiene una anotación de entorno de producción

[subs="+attributes",yaml]
----
- kind: pubKey
  key: |
    -----BEGIN PUBLIC KEY-----
    MBFKHFDGHKIJH0CAQYIKoZIzj0DAQcDQgAEX0HFTtCfTtPmkx5p1RbDE6HJSGAVD
    BVDF6SKFSF87AASUspkQsN3FO4iyWodCy5j3o0CdIJD/KJHDJFHDFIu6sA==
    -----END PUBLIC KEY-----
  annotations:
    environment: production
----
