Usted ha escrito, construido y ejecutado su política de Rego.
Ahora es el momento de distribuir la póliza.

Hay que anotar las políticas para que puedan ejecutarse en `policy-server`.
El `policy-server` es la parte que ejecuta las políticas, cuando se ejecuta en un clúster Kubernetes.

== Anotar la política

Anotar una política es un proceso que enriquece los metadatos de la política con información relevante.
Información como autoría, licencia, ubicación del código fuente, reglas, que describen qué tipo de recursos entiende y evalúa esta política.

Para anotar su política necesita escribir un archivo `metadata.yaml`:

[subs="+attributes",yaml]
----
rules:
- apiGroups: [""]
  apiVersions: ["*"]
  resources: ["*"]
  operations: ["CREATE"]
mutating: false
contextAware: false
executionMode: opa
annotations:
  io.kubewarden.policy.title: no-default-namespace
  io.kubewarden.policy.version: 0.1.0 # should match the OCI tag
  io.kubewarden.policy.description: This policy will reject any resource created inside the default namespace
  io.kubewarden.policy.author: The {project-name} Authors
  io.kubewarden.policy.url: https://github.com/kubewarden/some-policy
  io.kubewarden.policy.source: https://github.com/kubewarden/some-policy
  io.kubewarden.policy.license: Apache-2.0
  io.kubewarden.policy.usage: |
      This policy is just an example.

      You can write interesting descriptions about the policy here.
----

Puedes ver varios detalles:

* Reglas:
A qué recursos se dirige esta política.
* Mutando:
Si esta política está mutando.
En este caso, sólo se trata de validar.
* Conciencia de contexto:
Si esta política requiere contexto del clúster para evaluar la solicitud.
* Modo de ejecución:
Dado que se trata de una política de Rego es obligatorio especificar qué modo de ejecución espera,`opa` o `gatekeeper`.
Esta política está escrita al estilo `opa`, devolviendo un objeto `AdmissionReview` completo.
* Anotaciones: Metadatos almacenados en la propia política.

Anote su política:

[subs="+attributes",console]
----
$ kwctl annotate policy.wasm --metadata-path metadata.yaml --output-path annotated-policy.wasm
----

Ahora puede inspeccionar la política ejecutando `kwctl inspect annotated-policy.wasm`.

== Impulsar la política

Ahora que la política está anotada, puede enviarla a un registro OCI.

[subs="+attributes",console]
----
$ kwctl push annotated-policy.wasm registry.my-company.com/kubewarden/no-default-namespace:v0.0.1
Policy successfully pushed
----

Su política Rego, dirigida al marco OPA, tiene todo lo que necesita, para ser desplegado en producción, mediante la creación de un `ClusterAdmissionPolicy`.
También puedes prepararlo.
En primer lugar, tiene que llevar la póliza a la tienda local `kwctl`:

[subs="+attributes",console]
----
$ kwctl pull registry://registry.my-company.com/kubewarden/no-default-namespace:v0.0.1
pulling policy...
----

Crea un `ClusterAdmissionPolicy` a partir de él.
Esta operación tiene en cuenta los metadatos que posee sobre la política:

[subs="+attributes",console]
----
$ kwctl manifest registry://registry.my-company.com/kubewarden/no-default-namespace:v0.0.1 --type ClusterAdmissionPolicy
---
apiVersion: policies.kubewarden.io/v1alpha2
kind: ClusterAdmissionPolicy
metadata:
  name: generated-policy
spec:
  module: "registry://registry.my-company.com/kubewarden/no-default-namespace:v0.0.1"
  settings: {}
  rules:
    - apiGroups:
        - ""
      apiVersions:
        - "*"
      resources:
        - "*"
      operations:
        - CREATE
  mutating: false
----

Ahora puede utilizar este `ClusterAdmissionPolicy` como base para orientar los recursos que desee, o desplegar en Kubernetes tal cual.
