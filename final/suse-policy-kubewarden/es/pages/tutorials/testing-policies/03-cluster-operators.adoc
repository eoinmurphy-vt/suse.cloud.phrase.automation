= Pruebas para operadores de clúster
:revdate: 2026-03-18
:page-revdate: {revdate}
:description: Pruebe las políticas del operador de clúster con kwctl utilizando diferentes configuraciones y objetos de solicitud de validación para validar la configuración de las políticas.
:doc-persona: ["kubewarden-policy-developer"]
:doc-topic: ["testing-policies", "cluster-operators"]
:doc-type: ["tutorial"]
:keywords: ["kubewarden", "kubernetes", "cluster operators", "testing"]
:current-version: {page-origin-branch}

Como operador de clústeres Kubernetes, querrá realizar pruebas para las políticas {project-name} que desee utilizar.

Tendrás preguntas como:

* ¿Cuál es la configuración correcta de la política para obtener el resultado de validación/mutación necesario?
* ¿Cómo puedo estar seguro de que todo sigue funcionando como se esperaba cuando yo:
 ** ¿actualizar la política a una versión más reciente?
 ** ¿añadir/modificar recursos Kubernetes?
 ** cambiar los parámetros de configuración de la política?
 ** etc.

{short-project-name} tiene una utilidad, https://github.com/kubewarden/kwctl[`kwctl`]que permite probar las políticas fuera de Kubernetes.

Para utilizar `kwctl` hay que invocarlo con las siguientes entradas:

. Un URI de archivo binario WebAssembly de la política que se va a ejecutar.
La política {short-project-name} puede cargar desde el:
 ** sistema de archivos local `file://`
 ** un servidor HTTP(s) `https://`
 ** un registro OCI `registry://`.
. El objeto de solicitud de admisión que se va a comprobar.
Lo das con el argumento `--request-path`.
Utilice `stdin` ajustando `--request-path` a `-`.
. La configuración de la política para el tiempo de ejecución como un JSON en línea a través de `--settings-json` bandera.
O un archivo JSON, o YAML, cargado desde el sistema de archivos a través de `--settings-path`.

Después de la prueba `kwctl`, imprime el objeto `ValidationResponse` en la salida estándar.

Puede descargar los binarios precompilados de `kwctl`https://github.com/kubewarden/kwctl/releases[ aquí].

== Un ejemplo de prueba

Esta sección describe cómo probar la políticahttps://github.com/kubewarden/psp-apparmor[psp-apparmor]con diferentes configuraciones y objetos de solicitud de validación.

=== Crear solicitudes `AdmissionReview` 

Es necesario crear archivos que contengan los objetos `AdmissionReview` para probar la política.

Puede crear un archivo llamado `pod-req-no-specific-apparmor-profile.json` con el siguiente contenido:

.pod-req-no-specific-apparmor-profile.json
[%collapsible]
======

[subs="+attributes",json]
----
{
  "uid": "1299d386-525b-4032-98ae-1949f69f9cfc",
  "kind": {
    "kind": "Pod",
    "version": "v1"
  },
  "object": {
    "metadata": {
      "name": "no-apparmor"
    },
    "spec": {
      "containers": [
        {
          "image": "nginx",
          "name": "nginx"
        }
      ]
    }
  },
  "operation": "CREATE",
  "requestKind": {"version": "v1", "kind": "Pod"},
  "userInfo": {
    "username": "alice",
    "uid": "alice-uid",
    "groups": ["system:authenticated"]
  }
}
----

======

Esta petición intenta crear un Pod que no especifica ningún perfil AppArmor a utilizar.
Esto se debe a que no tiene un `annotation` con la tecla`container.apparmor.security.beta.kubernetes.io/<container-name>`.

Puede crear un archivo llamado `pod-req-apparmor-unconfined.json` con el siguiente contenido:

.pod-req-apparmor-unconfined.json
[%collapsible]
======

[subs="+attributes",json]
----
{
  "uid": "1299d386-525b-4032-98ae-1949f69f9cfc",
  "kind": {
    "kind": "Pod",
    "version": "v1"
  },
  "object": {
    "metadata": {
      "name": "privileged-pod",
      "annotations": {
        "container.apparmor.security.beta.kubernetes.io/nginx": "unconfined"
      }
    },
    "spec": {
      "containers": [
        {
          "image": "nginx",
          "name": "nginx"
        }
      ]
    }
  },
  "operation": "CREATE",
  "requestKind": {"version": "v1", "kind": "Pod"},
  "userInfo": {
    "username": "alice",
    "uid": "alice-uid",
    "groups": ["system:authenticated"]
  }
}
----

======

Esta petición intenta crear un Pod con un contenedor llamado `nginx` ejecutándose con el perfil `unconfined` AppArmor.
Esto es sólo para fines tutoriales.
Ejecutar en modo `unconfined` es una mala práctica de seguridad.

Ahora puede crear un archivo llamado`pod-req-apparmor-custom.json` con el siguiente contenido:

.pod-req-apparmor-custom.json
[%collapsible]
======

[subs="+attributes",json]
----
{
  "uid": "1299d386-525b-4032-98ae-1949f69f9cfc",
  "kind": {
    "kind": "Pod",
    "version": "v1"
  },
  "object": {
    "metadata": {
      "name": "privileged-pod",
      "annotations": {
        "container.apparmor.security.beta.kubernetes.io/nginx": "localhost/nginx-custom"
      }
    },
    "spec": {
      "containers": [
        {
          "image": "nginx",
          "name": "nginx"
        }
      ]
    }
  },
  "operation": "CREATE",
  "requestKind": {"version": "v1", "kind": "Pod"},
  "userInfo": {
    "username": "alice",
    "uid": "alice-uid",
    "groups": ["system:authenticated"]
  }
}
----

======

[NOTE]
====
Todos ellos son objetos simplificados de `AdmissionReview`.
Sólo se utilizan los campos relevantes para nuestra comprobación de la política.
====


=== Probar la política

Ahora puede utilizar `kwctl` para probar la creación de un Pod sin especificar un perfil AppArmor:

[subs="+attributes",console]
----
$ kwctl run \
    --request-path pod-req-no-specific-apparmor-profile.json \
    registry://ghcr.io/kubewarden/policies/psp-apparmor:v0.1.4 \
    | jq
----

La política acepta la solicitud y produce una salida como:

[subs="+attributes",console]
----
{
  "uid": "1299d386-525b-4032-98ae-1949f69f9cfc",
  "allowed": true
}
----

La directiva rechaza la creación de un Pod con un perfil `unconfined` AppArmor:

[subs="+attributes",console]
----
$ kwctl run \
    --request-path pod-req-apparmor-unconfined.json \
    registry://ghcr.io/kubewarden/policies/psp-apparmor:v0.1.4 \
    | jq
{
  "uid": "1299d386-525b-4032-98ae-1949f69f9cfc",
  "allowed": false,
  "status": {
    "message": "These AppArmor profiles are not allowed: [\"unconfined\"]"
  }
}
----

En ambas ocasiones ejecutó la política *sin* proporcionar ningún tipo de configuración.
Como se indica enhttps://github.com/kubewarden/psp-apparmor#configuration[la documentación de la política], esto impide el uso de perfiles no predeterminados.

El Pod que utiliza un perfil personalizado de `nginx` también es rechazado por la política:

[subs="+attributes",console]
----
$ kwctl run \
    --request-path pod-req-apparmor-custom.json \
    registry://ghcr.io/kubewarden/policies/psp-apparmor:v0.1.4 \
    | jq
{
  "uid": "1299d386-525b-4032-98ae-1949f69f9cfc",
  "allowed": false,
  "status": {
    "message": "These AppArmor profiles are not allowed: [\"localhost/nginx-custom\"]"
  }
}
----

Puede cambiar el comportamiento predeterminado, permitiendo que se utilicen los perfiles AppArmor elegidos:

[subs="+attributes",console]
----
$ kwctl run \
    --request-path pod-req-apparmor-custom.json \
    --settings-json '{"allowed_profiles": ["runtime/default", "localhost/nginx-custom"]}' \
    registry://ghcr.io/kubewarden/policies/psp-apparmor:v0.1.4 \
    | jq
----

Ahora la solicitud tiene éxito:

[subs="+attributes",console]
----
{
  "uid": "1299d386-525b-4032-98ae-1949f69f9cfc",
  "allowed": true
}
----

== Automatización

Puede automatizar todos estos pasos utilizandohttps://github.com/bats-core/bats-core[] bats.

Puede escribir una serie de pruebas e integrar su ejecución en sus procesos existentes de CI y CD.

Los comandos pueden "envolverse" en una prueba `bats`:

.Un batestest
[%collapsible]
======

[,bash]
----
@test "all is good" {
  run kwctl run \
    --request-path pod-req-no-specific-apparmor-profile.json \
    registry://ghcr.io/kubewarden/policies/psp-apparmor:v0.1.4

  # this prints the output when one the checks below fails
  echo "output = ${output}"

  # request accepted
  [ $(expr "$output" : '.*"allowed":true.*') -ne 0 ]
}

@test "reject" {
  run kwctl run \
    --request-path pod-req-apparmor-custom.json \
    registry://ghcr.io/kubewarden/policies/psp-apparmor:v0.1.4

  # this prints the output when one the checks below fails
  echo "output = ${output}"

  # request rejected
  [ $(expr "$output" : '.*"allowed":false.*') -ne 0 ]
}
----

======

Si el código `bats` se encuentra en el archivo `e2e.bats`, puede ejecutar la prueba como:

[subs="+attributes",console]
----
$ bats e2e.bats
 ✓ all is good
 ✓ reject

2 tests, 0 failures
----
