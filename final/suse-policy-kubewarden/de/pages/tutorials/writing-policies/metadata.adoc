:doc-persona: ["kubewarden-policy-developer"]
:doc-topic: ["kubewarden", "writing-policies", "policy-metadata"]

Die Metadaten-Datei {project-name}, `metadata.yaml`, ist eine Konfigurationsdatei, die wichtige Informationen und Einstellungen in Bezug auf die in {short-project-name} verwendeten Richtlinien enthält.
Diese Dokumentation gibt einen Überblick über den Zweck und die Verwendung der Metadaten-Datei.

Die Datei `metadata.yaml` enthält Standardwerte für die Richtlinie sowie Metadaten wie Autor und Beschreibung, die vom Autor der Richtlinie festgelegt wurden.
Der Befehl `kwctl annotate` verwendet diese Datei, um die Datei `.wasm`, die die Richtlinie enthält, mit Anmerkungen zu versehen.
Somit sind alle relevanten Informationen, die für die Durchführung der Police erforderlich sind, verfügbar.
Weitere Informationen darüber, wie Sie die Richtlinie mit Anmerkungen versehen können, finden Sie in der Anleitung zumxref:../../explanations/distributing-policies.adoc[Verteilen von Richtlinien (Distributing Policies] ).

Wenn Richtlinienbenutzer eine Richtlinie verwenden wollen, generieren sie mit `kwctl scaffold` ein YAML-Manifest.
Dieser Befehl liest die in das ausgelieferte Wasm-Modul eingebetteten Richtlinien-Metadaten, führt Prüfungen durch und gibt ein YAML-Manifest zurück, das der Autor unverändert verwenden oder ändern kann.

Als Autor einer Police können Sie die Datei `metadata.yaml` anpassen, die während der Erstellung Ihrer Police bereitgestellt wird.

Siehe das folgende Beispiel für eine `metadata.yaml`:

[subs="+attributes",yaml]
----
rules:
  - apiGroups: [""]
    apiVersions: ["v1"]
    resources: ["pods"]
    operations: ["CREATE"]
mutating: false
contextAwareResources: []
executionMode: kubewarden-wapc
policyType: kubernetes
backgroundAudit: true
annotations:
  # kubewarden specific:
  io.kubewarden.policy.title: My policy
  io.kubewarden.policy.version: 0.1.0 # should match the OCI tag
  io.kubewarden.policy.description: Short description
  io.kubewarden.policy.author: myself
  io.kubewarden.policy.url: https://github.com/yourorg/my-policy
  io.kubewarden.policy.source: https://github.com/yourorg/my-policy
  io.kubewarden.policy.license: Apache-2.0
  # The next two annotations are used in the policy report generated by the
  # Audit scanner. Severity indicates policy check result criticality and
  # Category indicates policy category. See more here at docs.kubewarden.io
  io.kubewarden.policy.severity: medium
  io.kubewarden.policy.category: Resource validation
  # artifacthub specific: (optional, to release in Artifact Hub)
  io.kubewarden.policy.ociUrl: ghcr.io/myorg/my-policy
  io.artifacthub.displayName: Policy Name
  io.artifacthub.resources: Pod
  io.artifacthub.keywords: pod, cool policy, kubewarden
----

== Ermöglichung von Hintergrundüberprüfungen

Die Metadatendatei enthält ein Kennzeichen, `backgroundAudit`, das die Hintergrundprüfungen für eine bestimmte Richtlinie aktiviert.
Standardmäßig ist dieses Flag auf `true` gesetzt.

Es gibt Richtlinien, die aufgrund ihrer Funktionsweise oder der Art der Ereignisse, für die sie gelten, dieses Feld auf `false` setzen sollten.
Weitere Informationen finden Sie in derxref:explanations/audit-scanner/limitations.adoc[Dokumentation des Audit-Scanners] unter dem Abschnitt Einschränkungen.

== Definieren von Kubernetes-Ressourcen, auf die Richtlinien zugreifen können

In der Metadatendatei können Benutzer über das Feld `contextAwareResources` festlegen, auf welche Kubernetes-Ressourcen die Richtlinie zugreifen kann.
Zum Beispiel, wenn die Richtlinie Zugriff auf die Ressource `Namespace` benötigt.
Der Verfasser der Richtlinie kann die `contextAwareResources` wie folgt definieren:

[subs="+attributes",yaml]
----
[...]
contextAwareResources:
  - apiVersion: v1 kind: Namespace
[...]
----

== Festlegung von Richtlinien als mutierend oder nicht mutierend

Die Metadatendatei enthält ein Flag, `mutating`, mit dem Benutzer eine Richtlinie entweder als mutierend oder nicht mutierend konfigurieren können.
Eine mutierende Richtlinie ändert die eingehenden Anfragen oder die verwalteten Ressourcen.
Bei einer nicht verändernden Richtlinie werden Beschränkungen eingehalten und durchgesetzt, ohne dass Änderungen vorgenommen werden.
Diese Unterscheidung ist entscheidend dafür, wie Richtlinien mit den Kubernetes-Ressourcen interagieren und welche Auswirkungen sie auf den Cluster haben.

== Festlegen des Richtlinientyps als Kubernetes oder Raw

Die Metadatendatei enthält ein Kennzeichen, `policyType`, mit dem die Benutzer eine Richtlinie entweder als `kubernetes` oder `raw` kennzeichnen können.
Eine Kubernetes-Richtlinie ist eine Richtlinie, die Kubernetes-Ressourcen validiert.
Eine Raw-Richtlinie ist eine Richtlinie, die beliebige JSON-Dokumente validiert.
Standardmäßig wird dieses Feld, wenn es nicht vom Benutzer angegeben wird, auf `kubernetes` gesetzt, wenn eine Richtlinie mit Anmerkungen versehen wird.
Weitere Informationen finden Sie im Abschnitt xref:../../howtos/raw-policies.adoc[über die Richtlinien für Rohdaten].

== Definition von Ressourcentyp-Zielen

Die Metadaten-Datei bietet den Benutzern die Möglichkeit, die Regeln im Feld `rules` zu definieren, das die Ressourcentypen bestimmt, für die die Richtlinie gilt.
Mit dieser Funktion können die Benutzer die Durchsetzung von Richtlinien genau steuern und sicherstellen, dass die Richtlinien nur auf die vorgesehenen Ressourcentypen angewendet werden.
Durch diese fein abgestufte Kontrolle können die Benutzer sicherstellen, dass die Richtlinien genau auf ihre spezifischen Anforderungen abgestimmt sind und eine unbeabsichtigte Anwendung von Richtlinien auf nicht verwandte Ressourcentypen vermieden wird.
