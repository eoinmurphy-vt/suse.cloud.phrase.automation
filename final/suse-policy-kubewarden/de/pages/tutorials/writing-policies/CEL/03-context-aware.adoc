{project-name}Die Website `cel-policy` unterstützt die Funktion " xref:reference/spec/05-context-aware-policies.adoc[Context Awareness] ". Die Richtlinie ist in der Lage, Clusterinformationen zu lesen und Entscheidungen zu treffen, die auf anderen vorhandenen Ressourcen basieren als der Ressource, die die Richtlinienbewertung über die Zulassungsanfrage ausgelöst hat.

Um dies zu erreichen, können wir die in der Richtlinie enthaltenen https://github.com/kubewarden/cel-policy?tab=readme-ov-file#host-capabilities[{short-project-name}'s CEL extension libraries for host capabilities] verwenden.

== Beispiel: Einzigartiges Eindringen

Schreiben wir eine Richtlinie, die bei der Erstellung oder Aktualisierung von Ingresses überprüft, ob der Ingress eindeutig ist, so dass Hosts höchstens eine Ingress-Regel haben.

Zu diesem Zweck erklären wir, dass die Richtlinie kontextabhängig ist. Wir deklarieren auch die fein abgestuften Berechtigungen, die wir zum Lesen anderer Ingress-Ressourcen benötigen. Dies wird mit`spec.contextAwareResources` (1) erreicht. Einen Startpunkt können wir wie üblich mit `kwctl` erhalten:

[subs="+attributes",console]
----
$ kwctl scaffold manifest -t ClusterAdmissionPolicy \
  registry://ghcr.io/kubewarden/policies/cel-policy:v1.0.0` \
  --allow-context-aware
----

Diese können wir dann so bearbeiten, dass sie für unsere Ingress-Ressourcen relevant sind:

[subs="+attributes", yaml, title="./cel-policy-ingress.yaml"]
----
apiVersion: policies.kubewarden.io/v1
kind: ClusterAdmissionPolicy
metadata:
  name: "unique-ingress"
  annotations:
    io.kubewarden.policy.category: Best practices
    io.kubewarden.policy.severity: low
spec:
  module: ghcr.io/kubewarden/policies/cel-policy:v1.0.0
  failurePolicy: Fail
  rules:
    - apiGroups: ["networking.k8s.io"]
      apiVersions: ["v1"]
      resources: ["ingresses"]
      operations: ["CREATE", "UPDATE"]
  contextAwareResources: # (1)
    - apiVersion: networking.k8s.io/v1
      kind: Ingress

----

Nun müssen wir den CEL-Code schreiben, der die vorhandenen Ingresses im Cluster abruft. Dazu verwenden wir die [{short-project-name} CEL extension library](https://github.com/kubewarden/cel-policy?tab=readme-ov-file#host-capabilities).
Insbesondere die `kw.k8s` Host-Funktionen, die es uns ermöglichen, den Cluster nach GroupVersionKinds abzufragen. Sie können die verfügbaren Dokumente für die CEL-Funktionen [hier] einsehen (https://pkg.go.dev/github.com/kubewarden/cel-policy/internal/cel/library).

Die Bibliothek verwendet ein Builder-Muster, genau wie die Upstream-Kubernetes-CEL-Erweiterungen; der Aufruf einer CEL-Funktionsmethode gibt ein CEL-Objekt zurück, das seinerseits über spezifische Funktionsmethoden verfügt. Dies vereinfacht es, sich des Umfangs und der Ergebnisse unseres CEL-Codes sicher zu sein.

In diesem Fall verwenden wir `kw.k8s.apiVersion("v1").kind("Ingress")`; hier rufen wir die Funktion `apiVersion()` der Bibliothek `kw.k8s` auf, die uns ein Objekt`<ClientBuilder>` zurückgibt. Dieses Objekt verfügt über die Methode `<ClientBuilder>.kind()`, die eine Liste aller Ressourcen in einem Array namens `items` zurückgibt.

Damit speichern wir die Liste der Ingresses im Cluster in einer Variablen:

```yaml
variables:
  - name: knownIngresses
    expression: kw.k8s.apiVersion("networking.k8s.io/v1").kind("Ingress").list().items
```

Dann erstellen wir eine Liste von Hosts aus diesen Ingresses. Beachten Sie, dass es mehrere Hosts pro Ingress geben kann, so dass dieser Ausdruck ein Array von Arrays enthält (was eine aktuelle Einschränkung der CEL-Sprache ist):

[subs="+attributes",yaml]
----
variables:
  - name: knownHosts
    expression: |
      variables.knownIngresses.map(i, i.spec.rules.map(r, r.host))
----

Dabei werden jedoch UPDATE-Operationen nicht korrekt behandelt; dazu müssen wir das aktuelle Objekt entfernen und die Hosts aus den verbleibenden Ingresses extrahieren.
Wir können das mit einer `filter()` auf dem aktuellen Objekt unter `object` tun.
Damit werden UPDATE-Operationen korrekt überprüft. Das bedeutet auch, dass die Richtlinie die Ergebnisse korrekt an den Audit Scanner meldet. Sie wird folgendermaßen aussehen:

[subs="+attributes",yaml]
----
variables:
  - name: knownHosts
    expression: |
      variables.knownIngresses
      .filter(i, (i.metadata.name != object.metadata.name) && (i.metadata.namespace != object.metadata.namespace))
      .map(i, i.spec.rules.map(r, r.host))
----

Außerdem benötigen wir eine Liste der Hosts in der aktuellen Ingress-Anfrage, mit der wir vergleichen können:

[subs="+attributes",yaml]
----
variables:
  - name: desiredHosts
    expression: |
      object.spec.rules.map(r, r.host)
----

Mit diesen beiden Variablen können wir eine Schnittmenge zwischen den bekannten Hosts und den gewünschten Hosts bilden, und wenn es eine gibt, lehnen wir sie ab:

[subs="+attributes",yaml]
----
validations:
  - expression: |
      !variables.knownHost.exists_one(hosts, sets.intersects(hosts, variables.desiredHosts))
    message: "Cannot reuse a host across multiple ingresses"
----

Zusammengefasst sieht die Politik wie folgt aus:

```yaml title="./cel-policy-ingress.yaml"
apiVersion: policies.kubewarden.io/v1
kind: ClusterAdmissionPolicy metadata: name: "unique-ingress" annotations: io.kubewarden.policy.category: Beste Praktiken io.kubewarden.policy.severity: low spec: module: ghcr.io/kubewarden/policies/cel-policy:v1.0.0 failurePolicy: Fail-Regeln:- apiGroups: ["networking.k8s.io"] apiVersions: ["v1"] resources: ["ingresses"] operations: ["CREATE", "UPDATE"] contextAwareResources:- apiVersion: networking.k8s.io/v1 kind: Ingress-Einstellungen: Variablen: - Name: knownIngresses Ausdruck: | kw.k8s.apiVersion("networking.k8s.io/v1").kind("Ingress").list().items - name: knownHosts Ausdruck: | variables.knownIngresses .filter(i, (i.metadata.name != object.metadata.name) && (i.metadata.namespace != object.metadata.namespace)) .map(i, i.spec.rules.map(r, r.host)) - name: desiredHosts Ausdruck: | object.spec.rules.map(r, r.host) Validierungen:- expression: | !variables.knownHosts.exists_one(hosts, sets.intersects(hosts, variables.desiredHosts)) message: "Ein Host kann nicht über mehrere Ingresses wiederverwendet werden"

```

## Deploying the policy

As normal, we can deploy our policy by instantiating its manifest:

```console
$ kubectl apply -f ./cel-policy-example.yaml
```

Jetzt können wir es testen, indem wir Ingresses instanziieren. Die erste wird erfolgreich sein, da es keine anderen Ziele für diesen Host gibt:

[subs="+attributes",console]
----
$ kubectl apply -f - <<EOF
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: ingress-host-foobar-1
spec:
  rules:
  - host: "foo.bar.com"
    http:
      paths:
      - pathType: Prefix
        path: "/bar"
        backend:
          service:
            name: service1
            port:
              number: 80
EOF
----

Die zweite wird jedoch zu einer Ablehnung führen:

[subs="+attributes",console]
----
$ kubectl apply -f - <<EOF
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: ingress-host-foobar-2
spec:
  rules:
  - host: "foo.bar.com"
    http:
      paths:
      - pathType: Prefix
        path: "/foo"
        backend:
          service:
            name: service2
            port:
              number: 80
EOF
Error from server: error when creating "STDIN":
  admission webhook "clusterwide-unique-ingress.kubewarden.admission" denied the request:
  Cannot reuse a host across multiple ingresses
----
