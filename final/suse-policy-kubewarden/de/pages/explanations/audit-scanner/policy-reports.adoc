= Audit Scanner - Richtlinienberichte
:revdate: 2026-03-18
:page-revdate: {revdate}

Wenn Sie den {project-name} Audit Scanner verwenden, speichern Sie die Ergebnisse von Richtlinien-Scans unter Verwendung der https://openreports.io/[OpenReports] Custom Resources.

[CAUTION]
====

Mit {project-name} 1.33 speichert die Audit Scanner-Funktion die Berichte standardmäßig in OpenReports CRDs von [openreports.io] (https://openreports.io).

Dielink:https://htmlpreview.github.io/?https://github.com/kubernetes-sigs/wg-policy-prototypes/blob/045372e558b896695b2daae92e8c7a04d4d40282/policy-report/docs/index.html[PolicyReport]CRDs der K8s Policy Working Group sind standardmäßig nicht installiert, als veraltet gekennzeichnet und werden in einer zukünftigen Version entfernt.

Benutzer, die die veralteten PolicyReports, d. h. das Verhalten vor 1.33, weiterhin verwenden möchten, müssen Folgendes tun:- Ändern Sie den `kubewarden-crds` Helm Chart-Wert von `.Values.install.installPolicyReportCRDs` in `true`. - Ändern Sie den `kubewarden-controller` Helm Chart-Wert von `.Values.auditScanner.reportCRDsKind` von `openreports` in `policyreport`.

====


Diese OpenReport CRDs bieten eine strukturierte Möglichkeit zur Speicherung und Verwaltung der Prüfungsergebnisse.

Sie speichern die Scanergebnisse der Audit-Scanner-Richtlinie unter `Report` und `ClusterReport`.
Der Audit-Scanner erstellt für jede gescannte Ressource ein `Report` oder ein `ClusterReport`, je nach Bereich der Ressource. `Report` Objekte sind im Namensraum der Ressource verfügbar, während `ClusterReport` Objekte im Clusterbereich verfügbar sind.

Die vom Scanner erzeugten Prüfergebnisse umfassen:

* die bewertete Politik
* die gesuchte Ressource
* das Ergebnis der Bewertung (bestanden, nicht bestanden oder übersprungen)
* einen Zeitstempel, der angibt, wann die Bewertung stattgefunden hat.

Sie können auch Schweregrad- und Kategoriekommentare für Ihre Richtlinien definieren.

Die Bediener können die Berichte mit `kubectl` abfragen. Sie können auch die optionale Benutzeroberfläche des Open-Source-Projektshttps://kyverno.github.io/policy-reporter[policy-reporter] zur Überwachung und Beobachtung der OpenReports CRDs verwenden.

== Abfrage der Berichte

Mit dem kubectl CLI können Sie die Ergebnisse des Scans abfragen:

Liste der Berichte im Standard-Namensraum:

[source,console,subs="+attributes"]
----
$ kubectl get report -o wide
NAME                                   KIND   NAME            PASS   FAIL   WARN   ERROR   SKIP   AGE
44da974b-6057-4745-a92c-9dd8a1b01cc4   Pod    nginx-secrets   6      1      0      0       0      33m
----

Liste der clusterweiten Berichte:

[source,console,subs="+attributes"]
----
$ kubectl get clusterreport -o wide
NAME                                   KIND        NAME                 PASS   FAIL   WARN   ERROR   SKIP   AGE
6c66175f-e15f-47e8-8c33-2c63d5ecde3e   Namespace   namespace1           3      1     0       0       0      1h
d0ffbf4f-7348-4350-af7b-fdbfc079cc49   Namespace   namespace2           0      4     0       0       0      1h
e48d9bbc-a9c4-4c36-8519-bc7390d9ec64   Namespace   namespace3           4      0     0       0       0      15h
----

Abrufen der Details eines bestimmten Berichts:

[source,console,subs="+attributes"]
----
kubectl get report 44da974b-6057-4745-a92c-9dd8a1b01cc4 -o yaml
----

Abrufen der Details eines bestimmten ClusterReports:

[source,console,subs="+attributes"]
----
kubectl get clusterreport e48d9bbc-a9c4-4c36-8519-bc7390d9ec64 -o yaml
----

== Beispiel für einen Bericht

Das folgende Beispiel zeigt eine `Report` für die Ressource `Deployment``deployment1` im Namensraum `default`. Der Bericht zeigt an, dass `Pod`die `safe-labels` AdmissionPolicy nicht bestanden hat.

[source,yaml,subs="+attributes"]
----
apiVersion: wgpolicyk8s.io/v1beta1
kind: PolicyReport
metadata:
  creationTimestamp: “2024-02-29T06:55:37Z”
  generation: 6
  labels:
    app.kubernetes.io/managed-by: kubewarden
    kubewarden.io/policyreport-version: v2
  name: 009805e4-6e16-4b70-80c9-cb33b6734c82
  namespace: default
  ownerReferences:
    - apiVersion: apps/v1
      kind: Deployment
      name: deployment1
      uid: 009805e4-6e16-4b70-80c9-cb33b6734c82
  resourceVersion: “2685996”
  uid: c5a88847-d678-4733-8120-1b83fd6330cb
results:
  - category: Resource validation
    message: “The following mandatory labels are missing: cost-center”
    policy: namespaced-default-safe-labels
    properties:
      policy-resource-version: “2684810”
      policy-uid: 826dd4ef-9db5-408e-9482-455f278bf9bf
      policy-name: “safe-labels”
      policy-namespace: “default”
      validating: “true”
    resourceSelector: {}
    result: fail
    scored: true
    severity: low
    source: kubewarden
    timestamp:

== ClusterReport Beispiel

Das folgende Beispiel zeigt eine `ClusterReport` für die Ressource `Namespace``default` . Der Bericht zeigt an, dass die Ressource die`safe-annotations` ClusterAdmissionPolicy-Validierung nicht bestanden hat.

[source,yaml,subs="+attributes"]
----
apiVersion: wgpolicyk8s.io/v1beta1
kind: ClusterPolicyReport
metadata:
  creationTimestamp: “2024-02-28T14:44:37Z”
  generation: 3
  labels:
    app.kubernetes.io/managed-by: kubewarden
    kubewarden.io/policyreport-version: v2
  name: 261c9492-deec-4a09-8aa9-cd464bb4b8d1
  ownerReferences:
    - apiVersion: v1
      kind: Namespace
      name: default
      uid: 261c9492-deec-4a09-8aa9-cd464bb4b8d1
  resourceVersion: “2403034”
  uid: 20a3d00e-e955-4f21-a887-317d40f3f052
results:
  - category: Resource validation
    message: “The following mandatory annotations are not allowed: owner”
    policy: clusterwide-safe-annotations
    properties:
      policy-resource-version: “2396437”
      policy-uid: 46780d6e-e51a-4d65-8572-a6af01380aa7
      policy-name: “safe-annotations”
      validating: “true”
    resourceSelector: {}
    result: fail
    scored: true
    severity: low
    source: kubewarden
    timestamp:
      nanos: 0
      seconds: 1709294251
scope:
  apiVersion: v1
  kind: Namespace
  name: default

  uid: 261c9492-deec-4a09-8aa9-cd464bb4b8d1

  error: 0
  fail: 1
  pass: 0

== Policy Reporter UI

Der Policy Reporter wird als Subchart von `kubewarden-controller` ausgeliefert. Weitere Informationen finden Sie auf der Seite xref:howtos/audit-scanner.adoc[Installation des Audit-Scanners].

Die Policy Reporter-Benutzeroberfläche bietet ein Dashboard, das alle Verstöße anzeigt. Siehe die folgende Abbildung:

image::policy-reporter_dashboard.png[Policy Reporter Dashboard Beispiel]

{short-project-name} liefert die Policy Reporter-Benutzeroberfläche mit einem vorkonfigurierten Filter, der nur {short-project-name} Richtlinien anzeigt, zusammen mit erweiterten Informationen:

image::policy-reporter_policyreports.png[Policy Reporter PolicyReports Beispiel]

Weitere Funktionen von Policy Reporter sind die Weiterleitung von Ergebnissen an verschiedene Clients (wie Grafana Loki, Elasticsearch, Chat-Anwendungen), Metrik-Endpunkte und mehr. Weitere Informationen finden Sie in denhttps://kyverno.github.io/policy-reporter[Gemeinschaftsdokumenten von].
