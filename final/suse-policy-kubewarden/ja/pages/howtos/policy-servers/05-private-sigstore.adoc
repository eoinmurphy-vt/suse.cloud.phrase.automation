= プライベートSigstoreインスタンスを使用するようにPolicyServersを設定する
:description: {project-name} 、ポリシーの署名検証にプライベートまたはセルフホスト型のSigstoreインスタンスを使用するようにPolicyServersを設定する。
:keywords: ["kubewarden"、"kubernetes"、"policyservers"、"sigstore"、"private sigstore"、"policy signing"]。
:doc-persona: ["kubewarden-operator"、"kubewarden-integrator"]]。
:doc-type: ["howto"]
:doc-topic: ["operator-manual", "policy-servers", "private-sigstore"]
:sidebar_label: シグストアのプライベートインスタンスを使用する
:sidebar_position: 5

ポリシー署名検証のために、デフォルトのパブリック Sigstore インフラストラクチャではなく、プライベートまたはセルフホスト Sigstore インスタンスを使用するように PolicyServer を構成できます。これは、エアギャップのある環境や、シグストアを独自に展開する場合に便利だ。

== 前提条件

* フルシオ、Rekor、TSA、およびCTログサービスにアクセス可能な、実行中のプライベートSigstoreインスタンス
* `kubectl` `kubewarden` 名前空間へのアクセス
* ローカルにインストールされたhttps://docs.sigstore.dev/cosign/system_config/installation/[`cosign`]CLIをローカルにインストールする。
* https://jqlang.org/[`jq`] ローカルにインストール

== ステップ1 -`ClientTrustConfig` JSONを生成する

PolicyServerは、プライベートSigstoreインスタンスのトラストアンカーと署名設定を記述する`ClientTrustConfig` JSONを期待する。

=== 証明書と公開鍵の取得

プライベートSigstoreインスタンスから以下を取得する：

* `fulcio.pem` - Fulcio CA 証明書チェーン（PEM）
* `rekor.pub` - レコールの透明性ログ公開鍵
* `tsa.pem` - タイムスタンプ機関証明書チェーン（PEM）
* `ctfe.pub` - CTログ公開鍵

[source,shell]
----
# Fulcio CA certificate chain
curl --fail -o fulcio.pem "${FULCIO_URL}/api/v1/rootCert"

# Rekor transparency log public key
curl --fail -o rekor.pub "${REKOR_URL}/api/v1/log/publicKey"

# Timestamp Authority certificate chain
curl --fail -o tsa.pem "${TSA_URL}/api/v1/timestamp/certchain"

# CT log public key (from the Kubernetes secret in tuf-system namespace)
kubectl get secret -o json -n tuf-system ctlog-public-key \
  | jq -r ".data.public" | base64 -d > ctfe.pub
----

[WARNING]
.セキュリティへの配慮
====

Sigstoreインスタンスから証明書と鍵を安全にダウンロードしてください。そうしないと、{project-name}、危殆化したデータで構成することになりかねない。

上記のコマンドは、テスト環境での実行方法の例である。
この情報を適切に入手する方法については、組織の適切なチームに相談すること。

====

=== 信頼されたルートと署名設定を生成する

プライベートSigstoreサービスのURLを指す環境変数を設定します：

[source,shell]
----
export FULCIO_URL=https://fulcio.example.com
export REKOR_URL=https://rekor.example.com
export TSA_URL=https://tsa.example.com
export CTLOG_URL=https://ctlog.example.com
export ISSUER_URL=https://oidc.example.com
----

`cosign` を実行して、信頼されたルートを生成する：

[source,shell]
----
cosign trusted-root create \
  --fulcio="url=$FULCIO_URL,certificate-chain=fulcio.pem" \
  --rekor="url=$REKOR_URL,public-key=rekor.pub,start-time=2024-01-01T00:00:00Z" \
  --tsa="url=$TSA_URL,certificate-chain=tsa.pem" \
  --ctfe="url=$CTLOG_URL,public-key=ctfe.pub,start-time=2024-01-01T00:00:00Z" \
  --out trusted_root.json
----

`cosign` を実行して署名設定を生成する：

[source,shell]
----
cosign signing-config create \
  --fulcio="url=$FULCIO_URL,api-version=1,start-time=2024-01-01T00:00:00Z,operator=sigstore.dev" \
  --rekor="url=$REKOR_URL,api-version=1,start-time=2024-01-01T00:00:00Z,operator=sigstore.dev" \
  --rekor-config="ANY" \
  --oidc-provider="url=$ISSUER_URL/auth,api-version=1,start-time=2024-01-01T00:00:00Z,operator=sigstore.dev" \
  --tsa="url=$TSA_URL/api/v1/timestamp,api-version=1,start-time=2024-01-01T00:00:00Z,operator=sigstore.dev" \
  --tsa-config="EXACT:1" \
  --out signing_config.json
----

[NOTE]
====
`--oidc-provider` はオプションである。プライベートのシグストア・インスタンスに専用のOIDCプロバイダーがある場合のみ、これを含める。ポリシーがKubernetesサービスアカウントトークンを使用して署名されている場合（`kubectl create token` 経由）、このフラグを省略します。

`start-time` の値は、証明書が発行される前の日付（または Sigstore インスタンスのデプロイ日）に設定する必要があります。
====

両コマンドで利用可能なすべてのフラグとJSONフィールドの定義の詳細については、https://docs.sigstore.dev/cosign/system_config/installation/[cosign CLI referenceと]https://github.com/sigstore/protobuf-specs[] Sigstore参照のこと。

=== に混ぜ合わせる。 `trust_config.json`

[source,shell]
----
cat << EOF > trust_config.json
{
  "mediaType": "application/vnd.dev.sigstore.clienttrustconfig.v0.1+json",
  "trustedRoot": $(cat trusted_root.json),
  "signingConfig": $(cat signing_config.json)
}
EOF
----

== ステップ2 -`kubewarden` ネームスペースにConfigMapを作成する

`ClientTrustConfig` JSONをConfigMapに`sigstore-trust-config` というキーで格納する。ConfigMapは、アドミッションコントローラの名前空間に**なければならない**。この例では、`kubewarden` ：

[source,shell]
----
kubectl --namespace kubewarden create configmap my-sigstore-trust-config \
  --from-file=sigstore-trust-config=trust_config.json
----

== ステップ 3 - 検証用コンフィグ ConfigMap の作成

署名済みポリシーの証明書 ID 制約を指定する`verification_config.yaml` ファイルを作成します。例えば、Kubernetesサービスアカウントトークンを使用する場合：

[source,yaml]
----
allOf:
  - kind: genericIssuer
    issuer: https://kubernetes.default.svc.cluster.local
    subject:
      equal: https://kubernetes.io/namespaces/<namespace>/serviceaccounts/<serviceaccount>
anyOf: null
----

`<namespace>` と`<serviceaccount>` は、ポリシーの署名時に使用した名前空間とサービスアカウントに置き換えてください。

ConfigMapを作成します：

[source,shell]
----
kubectl --namespace kubewarden create configmap my-verification-config \
  --from-file=verification-config=verification_config.yaml
----

[NOTE]
====
ConfigMapのキーは`verification-config` （ファイル名ではない）で**なければならない**。{short-project-name} 、このキーを正確に探す。
====

== ステップ 4 - PolicyServer を設定する

PolicyServerの`spec.sigstoreTrustConfig` と`spec.verificationConfig` を作成したConfigMapsの名前に設定します：

[source,yaml]
----
apiVersion: policies.kubewarden.io/v1
kind: PolicyServer
metadata:
  name: default
spec:
  image: ghcr.io/kubewarden/policy-server:latest
  replicas: 1
  sigstoreTrustConfig: my-sigstore-trust-config
  verificationConfig: my-verification-config
----

コントローラは、ConfigMapをpolicy-serverポッドにマウントし、署名検証のためにプライベートのSigstoreインスタンスを使用するように構成し、検証configで指定されたID制約を実施します。

[WARNING]
.セキュリティへの配慮
====

`sigstoreTrustConfig`によって参照される ConfigMap への書き込みアクセス権を持つユーザーは誰でも、ポリシーの署名検証に影響を与えることができる。これにより、別のシグストアのトラスト・ルートを代用し、署名チェックを迂回することが可能になる。

最小特権の原則に従い、Kubernetes RBACを使用してこのConfigMapへのアクセスを制限します。

====
