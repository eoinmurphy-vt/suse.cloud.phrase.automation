ポリシー・グループ機能は、より単純なポリシーを組み合わせて複雑なポリシーを作成することを可能にする。新たに2つのカスタムリソース定義（CRD）が導入された：

* `AdmissionPolicyGroup`特定のネームスペースに適用されるアドミッション・ポリシー。
* `ClusterAdmissionPolicyGroup`クラスタ全体に適用されるアドミッションポリシー。

これらのポリシーグループにより、ユーザーは既存のポリシーを使用できるため、カスタムポリシー作成の必要性が減り、再利用が促進される。ポリシーロジックの重複を避けることで、ユーザーは管理を簡素化し、DSLのような構成でカスタムポリシーを作成できる。

ポリシーグループは、論理演算子を使用して複数のポリシーを組み合わせて評価することができます。これにより、複雑なロジックを定義することができる。しかし、通常のポリシーは、入場時にリソースを変更する変異ロジックを含むことができるが、ポリシーグループは検証のみを行う。

ポリシーグループの設定は、通常のポリシーの設定と似ている。違いは、`expression` 、`message` 、`policies`フィールドの追加と、異なる場所でのコンテキスト認識ルールの宣言である。

これは`ClusterAdmissionPolicyGroup` の例で、次のセクションでさまざまなフィールドを説明するために使用できます：

.`ClusterAdmissionPolicyGroup` 、`latest` タグのついた画像を使用するPodは、信頼できる二人の当事者によって署名された画像でない限り拒否されます：アリスとボブ。
[%collapsible]
======
[,yaml]
----
apiVersion: policies.kubewarden.io/v1
kind: ClusterAdmissionPolicyGroup # or AdmissionPolicyGroup
metadata:
  name: demo
spec:
  rules:
    - apiGroups: [""]
      apiVersions: ["v1"]
      resources: ["pods"]
      operations:
        - CREATE
        - UPDATE
  policies:
    signed_by_alice:
      module: ghcr.io/kubewarden/policies/verify-image-signatures:v0.3.0
      settings:
        modifyImagesWithDigest: false
        signatures:
          - image: "*"
            pubKeys:
              - |
                -----BEGIN PUBLIC KEY-----
                MFkwEwYHKoZIzj0CAQYIKoZIzj0DAQcDQgAEyg65hiNHt8FXTamzCn34IE3qMGcV
                yQz3gPlhoKq3yqa1GIofcgLjUZtcKlUSVAU2/S5gXqyDnsW6466Jx/ZVlg==
                -----END PUBLIC KEY-----
    signed_by_bob:
      module: ghcr.io/kubewarden/policies/verify-image-signatures:v0.3.0
      settings:
        modifyImagesWithDigest: false
        signatures:
          - image: "*"
            pubKeys:
              - |
                -----BEGIN PUBLIC KEY-----
                MFkwEwYHKoZIzj0CAQYIKoZIzj0DAQcDQgAEswA3Ec4w1ErOpeLPfCdkrh8jvk3X
                urm8ZrXi4S3an70k8bf1OlGnI/aHCcGleewHbBk1iByySMwr8BabchXGSg==
                -----END PUBLIC KEY-----
    reject_latest:
      module: registry://ghcr.io/kubewarden/policies/trusted-repos:v0.2.0
      settings:
        tags:
          reject:
            - latest
  expression: "reject_latest() || (signed_by_alice() && signed_by_bob())"
  message: "the image is using the latest tag or is not signed by Alice and Bob"
----
======

== 主な設定フィールド

このセクションでは、ポリシーグループの主な設定フィールドについて説明します。

=== `policies` 属性

policyフィールドは、通常のポリシーのマップである。{project-name} 、ポリシーグループによってポリシーを呼び出し、評価対象のリソースを受け入れるか拒否するかを決定する。これらのポリシーの定義は、通常の{short-project-name} ポリシーの簡略版であり、`module` 、`settings` 、`contextAwareResources` 属性のみを含んでいる。これらの要素は、ポリシー・グループ内でポリシーが機能するために必要である。

一意の名前は、グループポリシーの各ポリシーを識別します。例えば、以下のスニペットは3つのポリシーを定義している：`signed_by_alice` `signed_by_bob`と`reject_latest_tag` 。

[,yaml]
----
policies:
  signed_by_alice:
    module: ghcr.io/kubewarden/policies/verify-image-signatures:v0.2.8
    settings: {} # settings for the policy
  signed_by_bob:
    module: ghcr.io/kubewarden/policies/verify-image-signatures:v0.2.8
    settings: {} # settings for the policy
  reject_latest_tag:
    module: ghcr.io/kubewarden/policies/trusted-repos-policy:v0.2.0
    settings: {} # settings for the policy
----

[TIP]
====

ポリシーグループには、同じポリシーを異なる設定で複数回含めることができます。

====


=== `expression` 属性

`expression` 属性は、ポリシーの識別子を論理演算子でつないだステートメントを含んでいます。

`expression` ステートメントの評価はブール値でなければならない。

ポリシーの表現は、`.spec.policies` マップで指定された識別子にちなんで名付けられた関数として行われる。ポリシーの評価によって生成された結果は、ユーザーによって提供された論理演算子を使用して評価される。

これらはサポートされている演算子である：

* `&&``AND` 。
* `||``OR` 。
* `!``NOT` 。

丸括弧`( )` を使って評価の優先順位を定義することができる。

例えば、次のような式があるとする：

[,console]
----
reject_latest() || (signed_by_alice() && signed_by_bob())
----

ポリシーは、アリスとボブの両方が画像に署名していない限り、`latest` タグを使用した画像を持つワークロードを拒否する。

=== `message` 属性とレスポンス・フォーマット

`message` フィールドは、`expression` の評価結果が拒否となった場合に返されるメッセージを指定する。レスポンスには、メッセージと個々のポリシーの評価結果が含まれる。

[IMPORTANT]
====

グループに属する政策の評価は、必要な場合にのみ行われる。

例えば、次のような式があるとする：

[,console]
----
reject_latest() || (signed_by_alice() && signed_by_bob())
----

`reject_latest` ポリシーが`true` を返すとき、`signed_by_bob` と`signed_by_alice` ポリシーは評価されない。

同じように、`signed_by_alice` と`reject_latest` ポリシーが`false` を返す場合、`signed_by_bob` ポリシーは評価されない。

これにより、グループ内のポリシーの不必要な評価を回避し、アドミッション要求への迅速な応答を実現する。

====

システムは、グループポリシーが拒否を実行したときに、AdmissionResponse`.status.details.causes` の一部としてグループポリシーのすべての評価詳細を送信します。

== コンテキスト・アウェア・ポリシー

ポリシーグループと通常のポリシーのもう一つの違いは、コンテキストを考慮したリソースルールの定義場所である。グループ内の各ポリシーは、評価中にポリシーがアクセスできるリソースを指定するために、オプションの`contextAwareResources` フィールドを受け入れます。通常のポリシーと同様に、`ClusterAdmissionPolicyGroup` を定義することによってのみ、コンテキスト対応機能を使用することができます。これはセキュリティ上の理由からで、`AdmissionPolicyGroup` リソースを配置できるのは権限のないユーザーだけだからです。詳細については、xref:explanations/context-aware-policies.adoc[コンテキスト対応ポリシーの]ドキュメントを参照してください。

.コンテキスト認識ポリシーを利用するポリシーグループの例。
[%collapsible]
======
[,yaml]
----
apiVersion: policies.kubewarden.io/v1
kind: ClusterAdmissionPolicyGroup # or AdmissionPolicyGroup
metadata:
  name: demo-ctx-aware
spec:
  rules:
    - apiGroups:
        - ""
      apiVersions:
        - v1
      resources:
        - services
      operations:
        - CREATE
        - UPDATE
  policies:
    unique_service_selector:
      module: registry://ghcr.io/kubewarden/policies/unique-service-selector-policy:v0.1.0
      contextAwareResources:
        - apiVersion: v1
          kind: Service
      settings:
        app.kubernetes.io/name: MyApp
    owned_by_foo_team:
      module: registry://ghcr.io/kubewarden/policies/safe-annotations:v0.2.9
      settings:
        mandatory_annotations:
          - owner
        constrained_annotations:
          owner: "foo-team"
  expression: "unique_service_selector() || (!unique_service_selector() && owned_by_foo_team())"
  message: "the service selector is not unique or the service is not owned by the foo team"
----
======

前の例では、`unique_service_selector` ポリシーは`Service` リソースにアクセスできる。しかし、`owned_by_foo_team` 、Kubernetesリソースにはアクセスできない。

== 設定の検証

ポリシーサーバーが起動すると、ポリシーグループと通常のポリシーの両方の設定を検証します。しかし、ポリシーグループは、式が有効であり、ブール値として評価されることをチェックするために、追加の検証ステップを受けます。

== 監査スキャナー

AdmissionPolicy および ClusterAdmissionPolicy CRD と同様に、`backgroundAudit` フィールドは、xref:../explanations/audit-scanner/audit-scanner.adoc[監査チェック]中にポリシー グループを含めるかどうかを示します。

== ポリシーサーバー

`policies.yml` 設定ファイルを拡張して、通常のポリシーに加えてポリシーグループを含めることができます。通常のポリシーと同様、モジュールのダウンロードは一度だけ行われる。同じポリシー モジュールをポリシー グループと通常のポリシーの両方で使用します。
