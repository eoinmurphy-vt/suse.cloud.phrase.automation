[[kwctl]]
=== `kwctl`

{project-name} ポリシーを管理するツール

*使い方：* `kwctl [OPTIONS] <COMMAND>`

==== サブコマンド

* `annotate` - WebAssembly モジュールに{short-project-name} メタデータを追加する。
* `bench` - ベンチマーク a{short-project-name} ポリシー
* `completions` - シェルコンプリートの生成
* `digest` - ポリシーのOCIマニフェストからダイジェストを取得する
* `docs` - kwctlコマンドのマークダウン・ドキュメントを生成する
* `info` - システム情報を表示する
* `inspect` -{short-project-name} ポリシーの点検
* `load` - tar.gzファイルからポリシーを読み込む
* `policies` - ダウンロードしたすべてのポリシーを一覧表示
* `pull` - 指定された URI から{short-project-name} ポリシーを引き出します。
* `push` -{short-project-name} ポリシーをOCIレジストリにプッシュする。
* `rm` - ストアから{short-project-name} ポリシーを削除します。
* `run` - 与えられたURIから{short-project-name} ポリシーを実行する。
* `save` - ポリシーをtar.gzファイルに保存する
* `scaffold` - Kubernetesリソースまたは設定ファイルをスキャフォールドする
* `verify` - Sigstoreを使用して、指定されたURIから{short-project-name} ポリシーを検証する。

==== オプション

* `-v``--verbose <VERBOSE>` - 冗長性を高める
* `--no-color <NO-COLOR>` - カラフルな出力を無効にする

[[kwctl-bench]]
=== `kwctl bench`

{short-project-name} ポリシーのベンチマーク。

ポリシーは以下の方法で指定できる：- URI: 例:`registry://ghcr.io/kubewarden/policies/psp-policy:latest` または`\https://example.com/kubewarden/policies/main/psp-policy/psp-policy.wasm`- SHA 接頭辞: 例:`c3b80a10f9c3` (ポリシーがすでにプルされている必要があります) - ローカル WASM ファイル: 例:`file://home/tux/new-policy/psp-policy.wasm`- ローカル YAML ファイル: 例:`file://home/tux/cluster-admission-policy.yaml` (`ClusterAdmissionPolicy`,`AdmissionPolicy` などの{short-project-name} カスタムリソースの宣言を含みます)

デフォルトの動作：
スキーマが省略された場合は、カレント・ディレクトリをルートとする`file://` 。

{short-project-name} カスタム・リソースに関する注意事項：- フラグ`--request-path`,`--settings-path`,`--settings-json` は無視されます; 設定はカスタムリソース定義から読み込まれます。-`--execution-mode` フラグは YAML ファイルのすべてのポリシーに適用されます。-{short-project-name} のカスタムリソースは`raw` ポリシーをサポートしないので、`--raw` フラグは使用できません。

カスタムリソース定義（CRD）の以下の属性のみが評価されます：- ポリシーモジュール - ポリシー設定 - ポリシーがアクセスできるコンテキスト対応リソース

`rules` 、`matchConditions` 、`objectSelector` 、`namespaceSelector` などの他のフィールドは無視される。

YAMLファイルは複数のカスタムリソース宣言を含むことができます。この場合、`kwctl` 、各評価の間に同じリクエストを使用してファイル内の各ポリシーを評価する。

*使い方：* `kwctl bench [OPTIONS] --request-path <PATH> <uri_or_sha_prefix_or_yaml_file>`

==== 議論

* `<URI_OR_SHA_PREFIX_OR_YAML_FILE>` -{short-project-name} ポリシーリソースを含むポリシー URI、SHA プレフィックス、または YAML ファイル。対応スキーム：registry://、https://, file://。schemaが省略された場合、カレント・ディレクトリをルートとするfile://が想定される。

==== オプション

* `--allow-context-aware <ALLOW-CONTEXT-AWARE>` - ポリシーの`contextAwareResources` セクションで定義された Kubernetes リソースへのアクセスを許可します。警告：悪用を避けるため、リソースのリストを注意深く確認してください。デフォルトでは無効
* `--cert-email <VALUE>` - フルシオ証明書に期待されるEメール
* `--cert-oidc-issuer <VALUE>` - フルチオ証明書で予想されるOIDC発行者
* `--disable-wasmtime-cache <DISABLE-WASMTIME-CACHE>` - wasmtimeキャッシュの使用をオフにする
* `--docker-config-json-path <PATH>` - Dockerの'config.json'ファイルがあるディレクトリへのパス。レジストリ認証の詳細を示すために使用できる
* `--dump-results-to-disk <DUMP_RESULTS_TO_DISK>` - ターゲットが見つかれば、target/tiny-bench/label/\...に結果を置く。
* `-e``--execution-mode <MODE>` - このポリシーを実行するために使用するランタイム。 +
  可能な値：`opa` `gatekeeper`,`kubewarden` 、 `wasi`
* `--github-owner <VALUE>` - CDパイプラインで生成される証明書に期待されるGitHubオーナー
* `--github-repo <VALUE>` - CDパイプラインで生成される証明書に期待されるGitHubリポジトリ
* `--measurement-time <SECONDS>` - num_samplesが優先されるため、ベンチされるコードがこの制限時間より遅い場合、num_samplesを収集できるまでベンチに時間がかかる。
* `--num-resamples <NUM>` - 何回の再サンプルを行うべきか
* `--num-samples <NUM>` - 何回再サンプルを行うべきか。少なくとも50を推奨し、100以上でも大きな違いはないようだ。
* `--raw <RAW>` - 生のリクエストを検証する +
  デフォルト値： `false`
* `--record-host-capabilities-interactions <FILE>` - 与えられたファイルにすべてのポリシーとホスト能力通信を記録する。後で「--replay-host-capabilities-interactions」フラグと組み合わせると便利。
* `--replay-host-capabilities-interactions <FILE>` - ポリシーとホストの能力交換の間、ホストは提供されたファイル内で見つかった答えをリプレイバックする。これは、OCIレジストリ、DNS、Kubernetesとの外部インタラクションが実行されないことから、再現可能な方法でポリシーをテストするのに便利である。
* `-r``--request-path <PATH>` - JSON形式のKubernetes入場許可リクエストオブジェクトを含むファイル。
* `--settings-json <VALUE>` - このポリシーの設定を含むJSON文字列
* `-s``--settings-path <PATH>` - このポリシーの設定を含むファイル。
* `--sigstore-trust-config <PATH>` - Sigstore protobuf仕様のClientTrustConfigメッセージに準拠したJSON形式のファイル。このファイルはSigstoreインスタンス全体の状態を構成します。これには、CAとアーティファクトの透過サービスにアクセスするために使用されるURIや、暗号化された信頼ルートそのものが含まれます。
* `--sources-path <PATH>` - ソース情報を保持する YAML ファイル（https、レジストリの安全でないホスト、カスタム CA の... ）。
* `-a``--verification-annotation <KEY=VALUE>` - key=value形式の注釈。何度も繰り返すことができる
* `--verification-config-path <PATH>` - 検証設定情報（署名、公開鍵... ）を保持するYAMLファイル。
* `-k``--verification-key <PATH>` - ポリシーの検証に使用する鍵へのパス。何度も繰り返すことができる
* `--warm-up-time <SECONDS>` - ベンチのウォームアップ時間

[[kwctl-push]]
=== `kwctl push`

{short-project-name} ポリシーをOCIレジストリにプッシュする。

*使い方：* `kwctl push [OPTIONS] <policy> <uri>`

ポリシーのメタデータの中にある注釈は、OCIマニフェストの一部になる。
複数行のアノテーションはOCI仕様と互換性がないため、スキップされる。
'io.kubewarden.policy.source'アノテーションは'org.opencontainers.image.source'として伝搬され、renovationbotのようなツールがポリシーの更新を検出できるようにする。

==== 議論

* `<POLICY>` - 後押しする政策ローカルファイルへのパス、ポリシーの URI、またはストア内のポリシーの SHA 接頭辞を指定できます。
* `<URI>` - ポリシーURI。サポートされるスキーム： registry://

==== オプション

* `--docker-config-json-path <PATH>` - Dockerの'config.json'ファイルがあるディレクトリへのパス。レジストリ認証の詳細を示すために使用できる
* `-f``--force <FORCE>` - 注釈のないポリシーも押してください。
* `-o``--output <PATH>` - 出力フォーマット +
  デフォルト値：`text` +
  取り得る値：`text`, `json`
* `--sources-path <PATH>` - ソース情報を保持する YAML ファイル（https、レジストリの安全でないホスト、カスタム CA の... ）。

=== `kwctl scaffold`

[[kwctl-scaffold-manifest]]
=== `kwctl scaffold manifest`

Kubernetesリソースマニフェストを出力する

*使い方：* `kwctl scaffold manifest [OPTIONS] --type <VALUE> <uri_or_sha_prefix>`

==== 議論

* `<URI_OR_SHA_PREFIX>` - ポリシーURIまたはSHA接頭辞。対応スキーム：registry://、https://, file://。schemaが省略された場合、カレント・ディレクトリをルートとするfile://が想定される。

==== オプション

* `--allow-context-aware <ALLOW-CONTEXT-AWARE>` - ポリシーのメタデータを使用して、ポリシーによってアクセスできるKubernetesリソースを定義します。警告：悪用を避けるため、リソースのリストを注意深く確認してください。デフォルトでは無効
* `--cert-email <VALUE>` - フルシオ証明書に期待されるEメール
* `--cert-oidc-issuer <VALUE>` - フルチオ証明書で予想されるOIDC発行者
* `--docker-config-json-path <DOCKER_CONFIG>` - Dockerの'config.json'ファイルがあるディレクトリへのパス。レジストリ認証の詳細を示すために使用できる
* `--fulcio-cert-path <PATH>` - フルシオ証明書への道何度も繰り返すことができる
* `--github-owner <VALUE>` - CDパイプラインで生成される証明書に期待されるGitHubオーナー
* `--github-repo <VALUE>` - CDパイプラインで生成される証明書に期待されるGitHubリポジトリ
* `--rekor-public-key-path <PATH>` - Rekor公開鍵へのパス。何度も繰り返すことができる
* `--settings-json <VALUE>` - このポリシーの設定を含むJSON文字列
* `-s``--settings-path <PATH>` - このポリシーの設定を含むファイル。
* `--sigstore-trust-config <PATH>` - Sigstore protobuf仕様のClientTrustConfigメッセージに準拠したJSON形式のファイル。このファイルはSigstoreインスタンス全体の状態を構成します。これには、CAとアーティファクトの透過サービスにアクセスするために使用されるURIや、暗号化された信頼ルートそのものが含まれます。
* `--sources-path <PATH>` - ソース情報を保持する YAML ファイル（https、レジストリの安全でないホスト、カスタム CA の... ）。
* `--title <VALUE>` - 政策タイトル
* `-t``--type <VALUE>` -{short-project-name} カスタム・リソース・タイプ +
  取り得る値：`ClusterAdmissionPolicy`, `AdmissionPolicy`
* `-a``--verification-annotation <KEY=VALUE>` - key=value形式の注釈。何度も繰り返すことができる
* `--verification-config-path <PATH>` - 検証設定情報（署名、公開鍵... ）を保持するYAMLファイル。
* `-k``--verification-key <PATH>` - ポリシーの検証に使用する鍵へのパス。何度も繰り返すことができる
