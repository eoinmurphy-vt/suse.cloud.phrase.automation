Open Policy AgentをターゲットとするRegoポリシーとまったく同じ方法でポリシーを構築し、実行することができます。
プロジェクトの構成はこうだ：

----
.
├── data
│   ├── default-ns.json
│   └── other-ns.json
└── policy.rego

1 directory, 3 files
----

== ビルド

`opa` コマンドを実行してポリシーを構築する：

[subs="+attributes",shell]
----
$ opa build -t wasm -e policy/violation policy.rego
----

これにより、レゴ・ポリシーが構築される：

* `target`: `wasm`.私たちは、`wasm` の目標に向けた政策を構築したい。
* `entrypoint`: `policy/violation`.エントリーポイントは、`policy` パッケージ内の`violation`ルールである。
* `policy.rego``policy.rego` 。

先のコマンドは、`bundle.tar.gz` ファイルを生成する。
そこからWasmモジュールを取り出すことができる：

[subs="+attributes",shell]
----
$ tar -xf bundle.tar.gz /policy.wasm
----

プロジェクト・ツリーは以下のようになる：

----
.
├── bundle.tar.gz
├── data
│   ├── default-ns.json
│   └── other-ns.json
├── policy.rego
└── policy.wasm

1 directory, 5 files
----

これでポリシーを実行できる。

== 走る

`kwctl` 、以下のようにポリシーを実行する：

 $ kwctl run -e gatekeeper --request-path data/other-ns.json policy.wasm | jq
 {
   "uid": "1299d386-525b-4032-98ae-1949f69f9cfc",
   "allowed": true
 }

これは、`other` というネームスペースで作成されたあなたのリソースです。

これで、ポリシーによって拒否されたリクエストを実行することができます：

[subs="+attributes",console]
----
$ kwctl run -e gatekeeper --request-path data/default-ns.json policy.wasm | jq
{
  "uid": "1299d386-525b-4032-98ae-1949f69f9cfc",
  "allowed": false,
  "status": {
    "message": "it is forbidden to use the default namespace"
  }
}
----

ゲートキーパーのポリシーは、このリソースを拒否することができます。
