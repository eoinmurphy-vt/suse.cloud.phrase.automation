使用{project-name} Audit Scanner 时，您可以使用https://openreports.io/[OpenReports]Custom Resources 来存储策略扫描的结果。

[CAUTION]
====

在{project-name} 1.33 中，审计扫描仪功能默认将报告从 [openreports.io](https://openreports.io) 保存到 OpenReports CRD 中。

来自 K8s Policy 工作组的link:https://htmlpreview.github.io/?https://github.com/kubernetes-sigs/wg-policy-prototypes/blob/045372e558b896695b2daae92e8c7a04d4d40282/policy-report/docs/index.html[PolicyReport]CRD 默认未安装，已被标记为过时，并将在未来的版本中移除。

如果用户希望继续使用已废弃的 PolicyReports，即 1.33 之前的行为，则需要- 将`kubewarden-crds` Helm 图表中`.Values.install.installPolicyReportCRDs` 的值更改为`true`. - 将`kubewarden-controller` Helm 图表中`.Values.auditScanner.reportCRDsKind` 的值从`openreports` 更改为`policyreport`.

====


这些 OpenReport CRD 提供了一种结构化的方式来存储和管理审计结果。

您可以使用`Report` 和`ClusterReport` 来存储审计扫描仪策略扫描结果。
审计扫描程序会根据资源的作用域，为扫描的每个资源创建`Report` 或`ClusterReport` 。`Report` 对象可在资源的命名空间中使用，而`ClusterReport` 对象可在群集作用域中使用。

扫描仪生成的审计结果包括

* 评估的政策
* 扫描的资源
* 评估结果（通过、未通过或跳过）
* 表示评估时间的时间戳。

您还可以为策略定义严重性和类别注释。

操作员可使用`kubectl` 查询报告。他们还可以使用https://kyverno.github.io/policy-reporter[policy-reporter]开放源代码项目提供的可选用户界面来监控和观察 OpenReports CRD。

== 查询报告

使用 kubectl CLI 可以查询扫描结果：

列出默认命名空间中的报告：

[source,console,subs="+attributes"]
----
$ kubectl get report -o wide
NAME                                   KIND   NAME            PASS   FAIL   WARN   ERROR   SKIP   AGE
44da974b-6057-4745-a92c-9dd8a1b01cc4   Pod    nginx-secrets   6      1      0      0       0      33m
----

列出整个群组的报告：

[source,console,subs="+attributes"]
----
$ kubectl get clusterreport -o wide
NAME                                   KIND        NAME                 PASS   FAIL   WARN   ERROR   SKIP   AGE
6c66175f-e15f-47e8-8c33-2c63d5ecde3e   Namespace   namespace1           3      1     0       0       0      1h
d0ffbf4f-7348-4350-af7b-fdbfc079cc49   Namespace   namespace2           0      4     0       0       0      1h
e48d9bbc-a9c4-4c36-8519-bc7390d9ec64   Namespace   namespace3           4      0     0       0       0      15h
----

获取特定报告的详细信息：

[source,console,subs="+attributes"]
----
kubectl get report 44da974b-6057-4745-a92c-9dd8a1b01cc4 -o yaml
----

获取特定 ClusterReport 的详细信息：

[source,console,subs="+attributes"]
----
kubectl get clusterreport e48d9bbc-a9c4-4c36-8519-bc7390d9ec64 -o yaml
----

== 报告范例

下面的示例显示了`default` 命名空间中`Deployment` 资源`deployment1` 的`Report` 。报告显示`Pod`未通过`safe-labels` AdmissionPolicy。

----

kind: PolicyReport
metadata:
  creationTimestamp: “2024-02-29T06:55:37Z”
  generation: 6
  labels:
    app.kubernetes.io/managed-by: kubewarden
    kubewarden.io/policyreport-version: v2
  name: 009805e4-6e16-4b70-80c9-cb33b6734c82
  namespace: default
  ownerReferences:
    - apiVersion: apps/v1
      kind: Deployment
      name: deployment1
      uid: 009805e4-6e16-4b70-80c9-cb33b6734c82
  resourceVersion: “2685996”
  uid: c5a88847-d678-4733-8120-1b83fd6330cb
results:
  - category: Resource validation
    message: “The following mandatory labels are missing: cost-center”
    policy: namespaced-default-safe-labels
    properties:
      policy-resource-version: “2684810”
      policy-uid: 826dd4ef-9db5-408e-9482-455f278bf9bf

== 群集报告示例

----

  name: 261c9492-deec-4a09-8aa9-cd464bb4b8d1
  ownerReferences:
    - apiVersion: v1
      kind: Namespace
      name: default

      policy-resource-version: “2396437”

== 政策报告用户界面

政策报告器作为`kubewarden-controller` 的子图表提供。有关详细信息，请参阅xref:howtos/audit-scanner.adoc[审核扫描仪安装]页面。

政策报告程序用户界面提供了一个显示所有违规行为的仪表板。请看下面的截图：

image::policy-reporter_dashboard.png[政策报告仪表板示例]

{short-project-name} 在政策报告程序用户界面上使用预先配置的过滤器，该过滤器只显示{short-project-name} 政策以及扩展信息：

image::policy-reporter_policyreports.png[政策报告政策报告示例]

策略报告器的其他功能包括将结果转发到不同的客户端（如 Grafana Loki、Elasticsearch、聊天应用程序）、度量端点等。更多信息，请参阅https://kyverno.github.io/policy-reporter[policy-reporter 社区文档]。
