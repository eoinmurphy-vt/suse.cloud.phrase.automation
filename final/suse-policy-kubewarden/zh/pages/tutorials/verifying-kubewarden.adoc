{project-name} 堆栈提供不同的证明和保证：

- 出处证明：告知构建过程、构建依赖关系，并帮助复制构建。执行https://slsa.dev/spec/v1.0/[SLSA]标准（在我们的案例中为第 3 级）。
- SBOMs 证明：包含软件依赖关系。帮助下游用户确定{short-project-name} 及其依赖系统的漏洞。
- 签名文物：显示人工制品的真伪，提供供应链安全。这不仅包括可交付成果，还包括出处和 SBOM 证明。

{short-project-name} 使用https://docs.sigstore.dev[] Sigstore 对工件、出处证明和 SBOM 进行签名，采用无密钥工作流程。这意味着签名证书包含以下信息，其中`*` 与以下任何字符匹配：

* 发行人： `https://token.actions.githubusercontent.com`
* 主题 `https://github.com/kubewarden/*`
* GHA 的 x509 证书扩展，"github_workflow_repository"： `kubewarden/*`

[IMPORTANT]
====
本教程中`--certificate-identity-regexp` cosign CLI 标志中使用的主题利用`https://github.com/kubewarden/*` 值来简化解释。不过，这样一来，具有相同前缀的资源库中的工件就可以绕过验证。例如：`github.com/kubewarden/policy-server1` 。

如果您想进行更安全的检查，则需要使用完整的 URL：

----
https://github.com/kubewarden/policy-server/.github/workflows/container-image.yml@refs/tags/v1.30.0
----

请注意，URL 包括完整的版本库路径、工作流文件路径和版本标记。如果您遵循这一最佳实践，就可以使用带有完整 URL 的 cosign CLI 标志`--certificate-identity` 。
====

{short-project-name} 团队还在努力改进安全供应链，使整个堆栈符合 SLSA 3 级标准。因此，主要人工制品还包括 SBOM 和出处文件。在接下来的章节中，我们将介绍如何验证{short-project-name} 团队制作的不同工件，以及如何使用 SBOM 和出处文件确保工件供应链的安全。

== 集装箱图像

要验证{short-project-name} 团队制作的无密钥签名容器映像，可以使用`cosign` CLI 工具。例如，要验证`kubewarden/policy-server` 映像，可以执行以下命令：

[source, console]
----
cosign verify ghcr.io/kubewarden/policy-server:v1.30.0 \
  --certificate-identity-regexp 'https://github.com/kubewarden/*' \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com

Verification for ghcr.io/kubewarden/policy-server:v1.30.0 --
The following checks were performed on each of these signatures:
  - The cosign claims were validated
  - Existence of the claims in the transparency log was verified offline
  - The code-signing certificate was verified using trusted certificate authority certificates

<snipped json>
----

然后，您就可以验证返回的 JSON 中的证书是否包含正确的签发人、主题和`github_workflow_repository` 扩展名。

您还可以使用link:https://github.com/rancherlabs/slsactl[`slsactl`].
例如，版本 1.30.0：

[source, console]
----
slsactl verify ghcr.io/kubewarden/policy-server:v1.30.0
----

这同样适用于{short-project-name} 团队制作的所有其他图像，如`kubewarden/kubewarden-controller` 和`kubewarden/audit-scanner` 。

所有容器镜像都有 SBOM 和出处文件，可用于确保镜像供应链的安全。您可以在组件的发布页面或 OCI 注册表的容器映像中找到证明文件。

我们使用 Docker 来构建镜像及其证明。不过，cosign 命令https://github.com/sigstore/cosign/issues/2688[尚不支持]验证 Docker 从 OCI 注册表生成的证明。
因此，您需要从发布页面或注册表下载文件，并在本地进行验证。如果想从 OCI 注册表中下载证明文件，可以按照https://docs.docker.com/build/metadata/attestations/attestation-storage/[Docker 文档]，使用`crane` 或`docker` 等工具从注册表中下载文件。

从{short-project-name} 组件的发布页面下载验证文件时，请检查这些文件：

[,console]
----
$ cosign verify-blob --bundle audit-scanner-attestation-amd64-checksum-cosign.bundle \
    --certificate-oidc-issuer=https://token.actions.githubusercontent.com \
    --certificate-identity="https://github.com/kubewarden/audit-scanner/.github/workflows/attestation.yml@refs/tags/v1.30.0" \
    audit-scanner-attestation-amd64-checksum.txt
Verified OK
----

现在，文件的完整性已得到验证，您可以检查 SBOM 和 Provenance 文件了。
您可以使用link:https://github.com/rancherlabs/slsactl[] `slsactl 从容器映像中获取这些信息。
例如，版本 1.30.0：

[,console]
----
slsactl download provenance ghcr.io/kubewarden/policy-server:v1.30.0
slsactl download sbom ghcr.io/kubewarden/policy-server:v1.30.0
----

=== 手动验证

自 Helm 3.8.0 起，Helm 已支持 OCI 注册表，但由于其中的限制，无法通过`helm`.Helm 搜索这些注册表。您可以找到https://github.com/orgs/kubewarden/packages?tab=packages&q=charts[在 GitHub Container Registry 中]。

要验证 Helm 图表，需要安装`cosign` 。然后执行以下命令，例如

----
cosign verify ghcr.io/kubewarden/charts/kubewarden-defaults:1.5.4 \
  --certificate-identity-regexp 'https://github.com/kubewarden/*' \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com

Verification for ghcr.io/kubewarden/charts/kubewarden-defaults:1.5.4 --
The following checks were performed on each of these signatures:
  - The cosign claims were validated
  - Existence of the claims in the transparency log was verified offline
  - The code-signing certificate was verified using trusted certificate authority certificates

<snipped json>
----

然后就可以验证返回的 json 中的证书是否包含正确的签发人、主题和`github_workflow_repository` 扩展名。

图表证明作为人工制品层被推送到 OCI 注册中心。有关如何验证xref:#container-images[容器图像]，请参阅xref:#container-images[容器图像]部分。

{short-project-name} 海图内的船舶`imagelist.txt` 和 (`policylist.txt` 相关时)。因此，如果您已经验证了图表，就可以使用这些列表来验证已消耗的容器映像和策略。

{short-project-name} 在舵手图方面也遵循惯例。因此，也可以使用https://github.com/cvila84/helm-image[helm-image] 等插件或以下脚本找到 Helm 图表中的所有图像：

[subs="+attributes",bash]
----
#!/usr/bin/env bash
helm pull --untar kubewarden/kubewarden-controller && \
helm pull --untar kubewarden/kubewarden-defaults && \
{ helm template ./kubewarden-controller; helm template ./kubewarden-defaults } \
    | yq '..|.image? | select(.)' \
    | sort -u | sed 's/"//g'
----

这让我们

----
ghcr.io/kubewarden/kubewarden-controller:v0.5.5
ghcr.io/kubewarden/policy-server:v0.3.1
ghcr.io/kubewarden/kubectl:v1.21.4
----

现在，对于列表中的每个图像，您都可以按照xref:#container-images[上一节]的说明验证它们的 Sigstore 签名。

== kwctl

kwctl 二进制文件使用https://docs.sigstore.dev/cosign/signing/signing_with_blobs/[的 blob 签名] Sigstore。

当您下载https://github.com/kubewarden/kwctl/releases/[`kwctl`版本] 时，每个压缩文件都包含一个 Sigstore 捆绑文件，您可以使用该文件进行验证：`kwctl-*bundle.sigstore` 。

要验证 kwctl，您需要安装 cosign，然后执行以下命令：

----
cosign verify-blob \
  --bundle kwctl-linux-x86_64.bundle.sigstore \
  --certificate-identity-regexp 'https://github.com/kubewarden/*' \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com \
  kwctl-linux-x86_64

Verified OK
----

然后就可以验证返回的 json 中的证书是否包含正确的签发人、主题和`github_workflow_repository` 扩展名。

SBOM 已签署并发布在该项目的https://github.com/kubewarden/kwctl/releases[Releases] GitHub中。

`kwctl` 的出处证明是通过使用https://cli.github.com/manual/gh_attestation_verify[`gh attestation verify`].例如，`gh attestation verify kwctl-linux-x86_64 --repo kubewarden/kwctl` 。
