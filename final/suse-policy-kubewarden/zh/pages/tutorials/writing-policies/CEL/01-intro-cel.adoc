https://cel.dev[通用表达式语言（CEL）]是一种通用表达式语言，设计用于快速、可移植和安全执行。CEL 作为一种语言，具有内存安全性、无副作用、终止性（程序不能永远循环）、强动态类型等特点。您可以在https://cel.dev[cel.dev] 上了解有关 CEL 的更多信息，并在https://playcel.undistro.io[CEL playground] 上进行练习。

== Kubernetes 中的 CEL

之所以选择 CEL 作为 Kubernetes 验证规则的语言，是因为 CEL 表达式可以轻松内嵌到 CRD 模式中，并在 "超前"（创建和更新 CRD 时）进行编译和类型检查。基于这些原因及其一般特性，它是扩展 Kubernetes API 的完美候选者。

已在 Kubernetes 1.30 中标记为稳定版，可在https://kubernetes.io/docs/reference/access-authn-authz/validating-admission-policy[ValidatingAdmissionPolicies]和其他验证规则中使用 CEL。更多信息，请参见 Kubernetes 文档https://kubernetes.io/docs/reference/using-api/cel[。]

=== Kubernetes 功能库

Kubernetes CEL 验证规则可访问多个函数库：

* https://github.com/google/cel-spec/blob/v0.7.0/doc/langdef.md#list-of-standard-definitions[CEL 标准功能]
* https://github.com/google/cel-spec/blob/v0.7.0/doc/langdef.md#macros[CEL 标准宏]
* https://pkg.go.dev/github.com/google/cel-go/ext#Strings[CEL 扩展字符串函数库]
* https://kubernetes.io/docs/reference/using-api/cel/#kubernetes-cel-libraries[Kubernetes CEL 扩展库]，为列表、regex、URL、授权者、数量、可选类型、数字比较等提供补充功能。

== {project-name} 中的 CEL ： `cel-policy`

{short-project-name} https://github.com/kubewarden/cel-policy[cel-policy].
这是一个策略，用于构建和捆绑上游https://pkg.go.dev/github.com/google/cel-go[cel-go]解释器，以及上文列出的、可用于 Kubernetes 中 CEL 的不同库（`apiextensions-apiserver` ）。

此外，`cel-policy` 捆绑了一个*{short-project-name} CEL 扩展库*，可将{short-project-name} 的https://github.com/kubewarden/cel-policy?tab=readme-ov-file#host-capabilities[主机功能作为本地 CEL] 公开：

* Sigstore 验证
* OCI 互动
* 加密功能
* 网络运行
* 访问 Kubernetes 资源

这意味着`cel-policy` 是 Kubernetes CEL 的超集，并且向后兼容。我们可以重复使用为 vanilla Kubernetes 编写的 CEL，和/或利用{short-project-name} 添加的功能。

`cel-policy` 已编译并作为 CEL 解释器运行。策略用户在（群集）准入策略的`spec.settings` 中传递所需的 CEL 表达式，由于 CEL 的特性，在创建或更新（群集）准入策略时，表达式会被编译并经过类型检查，以确保其正确性。这意味着无需定制构建`cel-policy`.NET 版本。

=== {short-project-name} 的`cel-policy` 与 ValidatingAdmissionPolicies 相比的优势

{short-project-name} `cel-policy` ：

* 它的 CEL 代码与 ValidatingAdmissionPolicies 向后兼容。
* 与 ValidatingAdmissionPolicies 相反，它不需要 ValidatingAdmissionPolicyBinding 这样的绑定，因为{short-project-name} 的 (Cluster)Admissionpolicies 定义中包含了这一点。
* 可部署到不支持 ValidatingAdmissionPolicies 的集群中。
* 可感知上下文，并利用{short-project-name} 的细粒度权限来xref:../../../reference/spec/05-context-aware-policies.adoc[感知上下文]。
* 作为（群集）AdmissionPolicy 部署。
* 受益于{short-project-name} 的政策跟踪和遥测功能。
* 审计扫描仪会将其考虑在内。
* 借助 kwctl，可在集群外进行开发和测试。
