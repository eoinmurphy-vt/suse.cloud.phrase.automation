本节将介绍如何编写端到端测试，以运行由 Javy 编译工具链生成的实际 WebAssembly 二进制文件。

== 先决条件

请记住，您需要在开发机器上安装这些工具：

* `bats`:用于编写测试并自动执行。
* `kwctl` >= v1.30:{project-name} 提供的 CLI 工具，用于在 Kubernetes 外部运行其策略，以及其他操作。文档中的xref:/tutorials/testing-policies/index.adoc[测试政策部分]对此有所涉及。

== 写作测试

您将使用`bats` 编写测试并使其自动化。每项测试都有以下步骤：

. 直接使用`kwctl run` 和 JSON 资源文件运行策略。
. 针对`kwctl` 生成的输出执行断言。

所有端到端测试都放在一个名为`e2e.bats` 的文件中。项目脚手架项目包括一个示例，`e2e.bats` 。您需要扩展其内容，为您的保单行为提供全面的测试保障。

== 测试数据文件

端到端测试用例测试特定 Kubernetes 资源的创建。

例如，在测试 Pod 资源的创建时：

[source,json]
----
{
  "apiVersion": "v1",
  "kind": "Pod",
  "metadata": {
    "name": "test-pod"
  },
  "spec": {
    "containers": [
      {
        "name": "nginx",
        "image": "nginx:latest"
      }
    ]
  }
}
----

端到端测试用例使用包含 Kubernetes AdmissionReview 的 JSON 文件。这些 JSON 对象 AdmissionReviews JSON 是针对 Kubernetes API 发送的，因此会得到xref:/reference/spec/03-validating-policies.adoc#an-example[验证]。

您可以使用`kwctl scaffold admission-request` 获取 AdmissionReviews JSON 对象。
请访问`test_data/pod.json` 查看带有 Pod 的 AdmissionRequest JSON。

== 基本测试用例

模板已经提供了几个基本测试。下面是`e2e.bats` 完整文件的样子：

[source,bash]
----
#!/usr/bin/env bats
----

以下是现有的测试及说明：

=== 测试 1：拒绝被拒绝的主机名

[source,bash]
----
@test "reject because hostname is on deny list" {
  run kwctl run annotated-policy.wasm -r test_data/pod_with_hostname.json --settings-json '{"denied_hostnames": ["forbidden-host", "test-hostname"]}'

  # this prints the output when one of the checks below fails
  echo "output = ${output}"

  # request rejected
  [ "$status" -eq 0 ]
  [ $(expr "$output" : '.*allowed.*false') -ne 0 ]
  [ $(expr "$output" : ".*Pod hostname 'test-hostname' is not allowed.*") -ne 0 ]
}
----

该测试可确保当 Pod 的主机名在拒绝列表中时，策略会拒绝接受 Pod。

=== 测试 2：接受允许的主机名

[source,bash]
----
@test "accept because hostname is not on the deny list" {
  run kwctl run annotated-policy.wasm -r test_data/pod_with_hostname.json --settings-json '{"denied_hostnames": ["forbidden-host"]}'

  # this prints the output when one of the checks below fails
  echo "output = ${output}"

  # request accepted
  [ "$status" -eq 0 ]
  [ $(expr "$output" : '.*allowed.*true') -ne 0 ]
}
----

当 Pod 的主机名不在拒绝列表中时，该测试将验证策略是否接受 Pod。

=== 测试 3：未提供设置时接受

[source,bash]
----
@test "accept because the deny list is empty" {
  run kwctl run annotated-policy.wasm -r test_data/pod_with_hostname.json

  # this prints the output when one of the checks below fails
  echo "output = ${output}"

  # request accepted
  [ "$status" -eq 0 ]
  [ $(expr "$output" : '.*allowed.*true') -ne 0 ]
}
----

该测试可确保策略在不提供任何设置的情况下接受请求。

=== 测试 4：接受不带主机名的 pod

[source,bash]
----
@test "accept because pod has no hostname set" {
  run kwctl run annotated-policy.wasm -r test_data/pod.json --settings-json '{"denied_hostnames": ["forbidden-host"]}'

  # this prints the output when one of the checks below fails
  echo "output = ${output}"

  # request accepted (no hostname to check)
  [ "$status" -eq 0 ]
  [ $(expr "$output" : '.*allowed.*true') -ne 0 ]
}
----

该测试验证策略是否接受不带主机名的 Pod，与拒绝列表无关。

=== 测试 5：接受非 iPod 资源

[source,bash]
----
@test "accept non-pod resources" {
  run kwctl run annotated-policy.wasm -r test_data/service.json --settings-json '{"denied_hostnames": ["forbidden-host"]}'

  # this prints the output when one of the checks below fails
  echo "output = ${output}"

  # request accepted (not a pod)
  [ "$status" -eq 0 ]
  [ $(expr "$output" : '.*allowed.*true') -ne 0 ]
}
----

该测试可确保策略接受不是 Pod 的资源，因为策略只验证 Pod 主机名。

== 了解测试输出

每个测试都使用`kwctl` 运行策略和检查：

* 退出状态：如果策略执行成功，命令`kwctl` 的退出状态应为 0。
* 允许的字段：JSON 输出包含一个`allowed` 字段，表示请求是否被接受。
* 信息内容：对于被拒绝的请求，输出会包含一条描述性错误信息。

每个测试中的`echo "output = $\{output}"` 语句都会在测试失败时显示实际策略输出，有助于调试。
