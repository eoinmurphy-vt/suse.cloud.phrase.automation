= 验证政策
:revdate: 2026-03-18
:page-revdate: {revdate}
:description: 使用特定策略设置评估您的验证请求，以确定是否接受或拒绝 Kubernetes AdmissionReview 对象。
:doc-persona: ["kubewarden-policy-developer"]
:doc-topic: ["writing-policies", "specification", "validating-policies"]
:doc-type: ["reference"]
:keywords: ["kubewarden"、"kubernetes"、"策略规范"、"策略验证"]
:sidebar_label: 验证政策
:current-version: {page-origin-branch}

{project-name} 策略服务器接收：

* Kuberneteshttps://godoc.org/k8s.io/api/admission/v1#AdmissionReview[`AdmissionReview`]对象。
然后将`request` 属性的值转发给策略进行评估。https://godoc.org/k8s.io/api/admission/v1#AdmissionRequest[`AdmissionRequest`]的属性值转发给策略进行评估。

或

* JSON`request` 属性，如果是原始策略，则包含自由格式的请求文档。详情请查看 "xref:../../howtos/raw-policies.adoc[原始政策]"部分。

政策会对`request` 进行评估，并说明是否应该接受。当请求被拒绝时，策略可能会提供解释信息和错误代码，以显示给最终用户。

按照惯例，在`policy-server` 项目中，客户机必须通过 waPC 客户机 SDK 公开一个名为`validate` 的函数，以便`policy-server` （waPC 主机）可以调用它。

`validate` 函数接收`ValidationRequest` JSON 对象，并返回`ValidationResponse` JSON 对象。

== `ValidationRequest` 对象

`ValidationRequest` 是`validate` 函数接收到的 JSON 对象。
看起来像

[subs="+attributes",yaml]
----
{
  "request": <AdmissionReview.request data> | <RawReviewRequest.request data>,
  "settings": {
    # your policy configuration
  }
}
----

`settings` 键指向一个自由格式的 JSON 文档，其中包含策略特定设置。上一章重点介绍了政策和设置。

== `ValidationResponse` 对象

`validate` 函数使用`ValidationResponse` 对象返回验证结果。

`ValidationResponse` 具有这种结构：

[subs="+attributes",yaml]
----
{
  # mandatory
  "accepted": <boolean>,

  # optional, ignored if accepted - recommended for rejections
  "message": <string>,

  # optional, ignored if accepted
  "code": <integer>,

  # optional, used by mutation policies
  "mutated_object": <string>
}
----

当请求未被接受时，可以指定这些`message` 和`code` 属性。`message` 是自由格式的文本错误，`code` 代表 HTTP 错误代码。

Kubernetes API 服务器在接受请求时会忽略`message` 和`code` 值。

请求被拒绝后，如果`message` 或`code` 存在，Kubernetes API 服务器会返回该信息。该信息是错误正文的一部分，服务器会将其返回给发出被拒请求的 Kubernetes API 客户端。

`mutated_object` 是一个可选字段，仅在变异策略中使用。
这就是下一章的主题。
