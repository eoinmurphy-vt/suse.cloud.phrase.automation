= 政策评估超时保护
:revdate: 2026-03-18
:page-revdate: {revdate}

:doc-persona: ["kubewarden-operator", "kubewarden-integrator"]
:doc-topic: ["operator-manual", "policy-evaluation-timeout"]
:doc-type: ["reference"]
:keywords: ["kubewarden"、"kubernetes"、"策略超时保护"]。
:current-version: {page-origin-branch}

[IMPORTANT]
====

该功能从{project-name} v1.5.0 版开始适用于每个策略服务器超时，从 v1.29.0 版开始适用于每个策略超时。

====

策略评估超时保护是策略和策略服务器的一项安全功能。其目的是限制申请评估所需的时间。

== 目的

您可以使用传统编程语言（如xref:../tutorials/writing-policies/go/01-intro-go.adoc[Go]、xref:../tutorials/writing-policies/rust/01-intro-rust.adoc[Rust]xref:../tutorials/writing-policies/index.adoc[等]）或特殊查询语言xref:../tutorials/writing-policies/rego/01-intro-rego.adoc[Rego] 编写{short-project-name} 策略。这两种方法各有优点，因此{short-project-name} 的目标是让政策制定者选择最适合其需要的工具。

在使用传统的https://en.wikipedia.org/wiki/Turing_completeness[] 图灵完备语言时，可能会出现以下问题：

* https://en.wikipedia.org/wiki/Infinite_loop[无限循环]
* https://en.wikipedia.org/wiki/Deadlock[僵局]
* 运行缓慢的代码缺乏优化
* 计算密集型操作

策略评估超时保护功能可在预定时间后终止对请求的评估。这可确保策略服务器始终有可用的计算资源来处理传入的请求。

== 局限性

目前，策略评估超时保护能够中断大多数长时间运行的评估。有些边缘情况尚未处理。
这包括从策略中调用`sleep` 指令和死锁。

未来发布的 Policy Server 将解决这些问题。

最后，策略评估超时会影响策略服务器实例托管的所有策略。目前，还无法按策略调整策略评估超时。

== 配置

每个策略评估超时、每个策略服务器评估超时和网络钩子超时的所有值都要经过验证，使它们都在可接受的范围内。例如，无法设置比 Kubernetes 的 webhook 超时更高的策略评估超时值。

### 根据政策

从{short-project-name} v1.29.0 开始，每个{short-project-name} 策略都可以通过`spec.timeoutEvalSeconds` 属性设置自己的超时值。这不能与用于 Webhook 超时的`spec.timeoutSeconds` 相混淆（请参阅 [下文](#与-kubernetes-dynamic-admission-controller-timeout) 一节）。

`spec.timeoutEvalSeconds` 细致入微，允许按政策调整其评估超时。

此设置优先于每个策略服务器的全局超时评估配置。

例如，为特定策略设置更长的评估超时：

[source,yaml]
----
apiVersion: policies.kubewarden.io/v1
kind: ClusterAdmissionPolicy
metadata:
  annotations:
    io.kubewarden.policy.category: Secrets
    io.kubewarden.policy.severity: medium
  name: env-variable-secrets-scanner
spec:
  module: registry://ghcr.io/kubewarden/policies/env-variable-secrets-scanner:v1.0.6
  settings: {}
  timeoutEvalSeconds: 10 // Set evaluation timeout
  mutating: false
  rules:
    - apiGroups: [""]
      apiVersions: ["v1"]
      resources: ["pods"]
      operations: ["CREATE"]
    - apiGroups: [""]
      apiVersions: ["v1"]
      resources: ["replicationcontrollers"]
      operations: ["CREATE", "UPDATE"]
    - apiGroups: ["apps"]
      apiVersions: ["v1"]
      resources: ["deployments", "replicasets", "statefulsets", "daemonsets"]
      operations: ["CREATE", "UPDATE"]
    - apiGroups: ["batch"]
      apiVersions: ["v1"]
      resources: ["jobs", "cronjobs"]
      operations: ["CREATE", "UPDATE"]
----

=== 每个策略服务器

从{short-project-name} v1.5.0 开始，策略服务器带有可配置的评估超时，默认已启用。请求评估 2 秒后中断。此配置会影响该策略服务器中计划的所有策略。按策略配置的`spec.timeoutEvalSeconds`超时优先于按策略服务器设置的超时。

您可以使用这些环境变量来调整这种行为：

* `KUBEWARDEN_DISABLE_TIMEOUT_PROTECTION`:这就完全禁用了政策评估。任何指定值都会关闭该功能。
* `KUBEWARDEN_POLICY_TIMEOUT`:这会设置不同的超时值。该值以秒为单位，默认值为`2` 。

当使用https://doc.crds.dev/github.com/kubewarden/kubewarden-controller/policies.kubewarden.io/PolicyServer/v1@v1.4.2[`PolicyServer`]Kubernetes 自定义资源定义时，可以如下设置这些环境变量：

[source,yaml]
----
# A Policy Server that has policy evaluation timeout disabled
apiVersion: policies.kubewarden.io/v1
kind: PolicyServer
metadata:
  name: no-policy-timeout
spec:
  env:
    - name: KUBEWARDEN_DISABLE_TIMEOUT_PROTECTION
      value: "true"
---

# A Policy Server that has policy evaluation timeout enabled,
# with a 3 seconds timeout value
apiVersion: policies.kubewarden.io/v1
kind: PolicyServer
metadata:
  name: custom-policy-timeout
spec:
  env:
    - name: KUBEWARDEN_POLICY_TIMEOUT
      value: "3"
----

== 与 Kubernetes 动态接纳控制器超时的比较

{short-project-name} 是https://kubernetes.io/docs/reference/access-authn-authz/extensible-admission-controllers/[Kubernetes 动态接纳控制器]的https://en.wikipedia.org/wiki/Webhook[webhook]实现。

在内部，Kubernetes API 服务器会向{short-project-name} 的策略服务器发出 HTTP 请求，描述即将发生的事件。发出 HTTP 请求后，Kubernetes API 服务器会等待应答。不过，Kubernetes API 服务器不会永远等待。经过一定时间后，它会认为请求超时。

引用https://kubernetes.io/docs/reference/access-authn-authz/extensible-admission-controllers/#timeouts[官方 Kubernetes 文档]：

____
由于网络钩子会增加 API 请求的延迟，因此应尽快对其进行评估。`timeoutSeconds` 允许配置 API 服务器在将调用视为失败之前应等待网络钩子响应多长时间。

如果超时时间在 Webhook 响应之前已过，则 Webhook 调用将被忽略，或根据https://kubernetes.io/docs/reference/access-authn-authz/extensible-admission-controllers/#failure-policy[失败策略] 拒绝 API 调用。

超时值必须在 1 至 30 秒之间。

接纳网络钩子的超时默认为 10 秒。
____

这意味着，无论策略评估超时功能如何，每个 Kubernetes 准入请求都会超时。

每个{short-project-name} 策略都可以通过`ClusterAdmissionPolicy` 和`AdmissionPolicy` 自定义资源的`timeoutSeconds`属性设置自己的超时值。
默认超时值为 10 秒。

[IMPORTANT]
====

向策略服务器发出的所有 Kubernetes 准入请求都有两种不同的超时：

* Kubernetes API 服务器超时值。默认设置为 10 秒，可通过策略自定义资源上的专用`spec.timeoutSeconds`属性按策略进行调整。
* 策略评估超时。通过环境变量在策略服务器中设置，或通过策略自定义资源上的`spec.timeoutEvalSeconds` 属性按策略设置。

====

现在，您可以检查以下场景，以便更好地理解 Kubernetes 的 Webhook 超时与{short-project-name} 的策略评估超时之间的差异。


=== {short-project-name} 策略评估超时已禁用

假设你有一个关闭了策略评估超时功能的策略服务器，而且它上计划的任何策略都没有设置`spec.timeoutEvalSeconds` 字段。此策略服务器托管的策略受一个错误的影响，该错误导致策略在评估过程中进入无限循环。

Kubernetes API 服务器会发送一个接纳请求，以便根据这个有漏洞的策略进行评估。结果，政策评估进入了一个无限循环。
与此同时，Kubernetes API 服务器正在等待响应。

10 秒后，Kubernetes 的网络钩子超时，请求将根据网络钩子的失败策略进行处理。

现在，策略服务器的计算资源被困在这个无限循环中。
随着时间的推移，越来越多的准入请求触发了存在漏洞的策略，策略服务器的计算资源也随之耗尽。它无法响应 Kubernetes API 服务器。这相当于对策略服务器的拒绝服务 (DOS) 攻击。

=== {short-project-name} 策略评估超时已启用

假设同一策略服务器现在启用了策略评估超时功能，可以在策略服务器中全局启用，也可以通过策略`spec.timeoutEvalSeconds` 在策略中启用，策略评估超时时间为 2 秒。

Kubernetes API 服务器会发送一个接纳请求，以便根据这个有漏洞的策略进行评估。结果，政策评估进入了一个无限循环。
与此同时，Kubernetes API 服务器正在等待响应。

两秒后，{short-project-name} 的策略评估超时功能会中断策略评估并产生拒绝响应。回复中包含一条信息，说明拒绝发生的原因是策略评估没有及时完成。

[NOTE]
====

将{short-project-name} 的策略评估超时设置为与 Kubernetes 的 webhook 超时一样高的值并不是一个好的选择。

虽然策略评估仍会中断，从而降低了 DOS 攻击的几率，但策略服务器不会做出最终的拒绝响应。拒绝来自 Kubernetes API 服务器的 webhook 超时。

因此，用户和 Kubernetes 操作员更难检测到这些缓慢/有漏洞的策略。策略评估中断的唯一证据就是策略服务器日志和跟踪事件。

====
