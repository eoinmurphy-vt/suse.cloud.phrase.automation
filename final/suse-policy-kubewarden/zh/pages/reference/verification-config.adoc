= 验证配置格式
:revdate: 2026-03-18
:page-revdate: {revdate}

== 导言

`verification-config` 格式：

* `policy-server` 验证策略模块的出处
* `verify-image-signatures` 验证群集图像来源的策略

更多信息，请参阅xref:/howtos/security-hardening/secure-supply-chain.adoc[安全供应链]。

== 格式

配置有 2 个根密钥：

* `allOf`:您应满足此处列出的所有验证信息，以验证容器图像已签名。
* `anyOf`:您必须至少满足`anyOf.minimumMatches` 所有列出的验证信息，才能接受已签名的容器映像。

这两个根键接受`kind` 类型的键数组。根据不同的使用情况，可接受密钥的完整列表如下：

* `pubKey`:对于使用传统公钥/私钥加密法进行的签名。
* `githubAction`:对于在 GitHub Actions 内使用 Sigstore 的无密钥工作流执行的签名，{short-project-name} 会根据 GitHub 的 OpenID Connect 创建的 x509 证书扩展`workflow_repository` 检查该信息，而不仅是`issuer` 和`subject` 。如果要处理 GitHub 操作，请使用`kind`。
* `genericIssuer`:对于使用 Sigstore 无密钥工作流程执行的签名，用户需要自行验证证书`issuer` 和`subject` 。它接受`subject` ，可以是：

** `equal`:此处传递的值必须与签名证书中的`subject` 完全一致。
** `urlPrefix`:此处传递的值用`/` 后缀，以防止错别字，并且必须是签名证书中`subject` 的前缀。

`kind` key 接受一个可选的`annotations` key，该 key 包含一个键值列表，必须出现在签名中。

== 示例

这是一个使用 Sigstore 工作流程验证签名的配置示例：

[subs="+attributes",yaml]
----
---
apiVersion: v1

allOf:
  - kind: githubAction
    owner: kubewarden   # mandatory
    repo: policy-server # optional
    annotations:  # optional
      env: prod

anyOf: # at least `anyOf.minimumMatches` are required to match
  minimumMatches: 2 # default is 1
  signatures:
  - kind: pubKey
    owner: alice # optional
    key: | # mandatory
         -----BEGIN PUBLIC KEY-----
         MFkwEwYHKoZIzj0CAQYIKoZIzj0DAQcDQgAEQiTy5S+2JFvVlhUwWPLziM7iTM2j
         byLgh2IjpNQN0Uio/9pZOTP/CsJmXoUNshfpTUHd3OxgHgz/6adtf2nBwQ==
         -----END PUBLIC KEY-----
    annotations:  # optional
      env: prod
  - kind: genericIssuer
    issuer: https://github.com/login/oauth
    subject:
      equal: alice@example.com
  - kind: genericIssuer
    issuer: https://token.actions.githubusercontent.com
    subject:
      equal: https://github.com/bob/app-example/.github/workflows/release.yml@refs/heads/main
    annotations:  # optional
      env: prod
  - kind: genericIssuer
    issuer: https://token.actions.githubusercontent.com
    subject:
      urlPrefix: https://github.com/bob # <- it will be post-fixed with `/` for security reasons
----

=== 签名验证

之前的配置包含`allOf` 和`anyOf` 两个部分：

* `allOf`:只有在所有签名要求都有效的情况下，您才能信任该策略。
* `anyOf`:如果`minimumMatches` 标准有效，您就可以信任该政策。

在示例中，`minimumMatches` 字段为 2。因此，您需要满足至少两项签名要求。`minimumMatches`字段的默认值是`1` 。

要进行签名验证，您需要满足`allOf` 中的所有签名要求*和* `anyOf` 中的最小签名数要求。

=== 公钥验证

要检查策略是否具有正确的公钥签名，需要指定密钥数据和密钥所有者。在此示例中，您将`kind` 设置为`pubKey` ，而`key` 拥有公钥。所有者字段是可选的，但对于明确密钥的所有者很有用。

[subs="+attributes",yaml]
----
  - kind: pubKey
    owner: bob # optional
    key: |
      -----BEGIN PUBLIC KEY-----
      MBFKHFDGHKIJH0CAQYIKoZIzj0DAQcDQgAEX0HFTtCfTtPmkx5p1RbDE6HJSGAVD
      BVDF6SKFSF87AASUspkQsN3FO4iyWodCy5j3o0CdIJD/KJHDJFHDFIu6sA==
      -----END PUBLIC KEY-----
----

=== 无钥匙签名验证

以无密钥模式签署的策略没有可以验证的公开密钥。您仍然可以使用签名过程中使用的 OpenID Connect (OIDC) 数据来验证策略。为此，有必要将签名验证定义为`genericIssuer` 。

可以通过签名验证信息：

* `issuer`(必填）：与 Fulcio 生成的证书中的`Issuer` 属性相匹配。这显示了用于签署政策的 OIDC。
* `subject`：用于匹配 Fulcio 证书中`Subject` 属性的字段。`Subject` (Fulcio) 字段包含用于验证 OIDC 提供商的用户。验证字段`subject` 可以有两个子字段：

** `equal`：证书中的`Subject` (Fulcio) 必须等于签名验证中的值；
** `urlPrefix`：证书的`Subject` （Fulcio）字段值必须以签名验证中定义的值为前缀。

[NOTE]
====

`cosign verify` 和`kwctl inspect` 都可以显示有关无钥匙签名的信息。

====


例如，该配置意味着策略必须由 Alice 使用 GitHub OIDC 进行无密钥签名：

[subs="+attributes",yaml]
----
- kind: genericIssuer
  issuer: https://github.com/login/oauth
  subject:
    equal: alice@example.com
----

该配置需要在 GitHub 操作中签署策略，策略来自 GitHub 用户`flavio` 所拥有的资源库：

[subs="+attributes",yaml]
----
- kind: genericIssuer
  issuer: https://token.actions.githubusercontent.com
  subject:
    urlPrefix: https://github.com/flavio
----

=== GitHub 行动签名验证

种"，`githubAction` 用于验证在 GitHub Actions 中签署的策略。
`genericIssuer` 也可以这样做。为简化签名要求程序，`githubAction` ：

* `owner` (强制性）：要信任的用户或组织的 GitHub ID。
* `repo`:要信任的存储库名称。

例如，使用`genericIssuer` 的最后一个片段可改写为

[subs="+attributes",yaml]
----
- kind: githubAction
  owner: flavio
----

=== 签名注释验证

所有签名类型都可以有其他可选的验证字段，`annotations` 。
这些字段是在签名过程中添加的键/值数据。

通过{short-project-name} ，您可以验证策略签名是否来自受信任的用户*并*具有特定注释。

下一步验证将检查政策的两个条件：

* 是用特定的密钥签署的
* 生产环境注释。

[subs="+attributes",yaml]
----
- kind: pubKey
  key: |
    -----BEGIN PUBLIC KEY-----
    MBFKHFDGHKIJH0CAQYIKoZIzj0DAQcDQgAEX0HFTtCfTtPmkx5p1RbDE6HJSGAVD
    BVDF6SKFSF87AASUspkQsN3FO4iyWodCy5j3o0CdIJD/KJHDJFHDFIu6sA==
    -----END PUBLIC KEY-----
  annotations:
    environment: production
----

=== 使用签名验证配置文件检查策略 OCI 工件

您可以使用验证配置文件测试策略是否通过验证。使用`kwctl 校验`命令的`--verification-config-path` 标志。

[subs="+attributes",console]
----
$ cat signatures_requirements.yaml
apiVersion: v1
allOf:
  - kind: pubKey
    key: |
      -----BEGIN PUBLIC KEY-----
      MFkwEwYHKoZIzj0CAQYIKoZIzj0DAQcDQgAE5Q+cN1Jj2S7N05J4AXnqwP2DyzSg
      Mc+raYce2Wthrd30MSgFtoh5ADAkCd/nML2Nx8UD9KBuASRb0gG5jXqgMQ==
      -----END PUBLIC KEY-----

$ kwctl verify --verification-config-path signatures_requirements.yaml ghcr.io/kubewarden/policies/user-group-psp:latest
2022-03-29T17:34:37.847169Z  INFO kwctl::verify: Policy successfully verified
----

最后一个示例测试给定的策略是否来自{short-project-name}组织：

[subs="+attributes",console]
----
$ cat kubewarden_signatures.yaml
apiVersion: v1
allOf:
  - kind: githubAction
    owner: kubewarden

$ kwctl verify --verification-config-path kubewarden_signatures.yaml ghcr.io/kubewarden/policies/user-group-psp:latest
2022-03-29T18:07:39.062292Z  INFO kwctl::verify: Policy successfully verified
----
