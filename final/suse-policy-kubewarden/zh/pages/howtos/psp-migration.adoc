对于 Kubernetes ≥ v1.25。
https://kubernetes.io/docs/concepts/security/pod-security-policy/[PodSecurityPolicy](PSP) 已移除。现在，您可以使用{project-name} 对 Kubernetes 集群进行准入控制。

{short-project-name} 与单片式 PSP 配置一样，该系统也有单独的策略来实现相同的目标。每个{short-project-name} 策略定义都是 PSP 规范中不同的配置部分。PSP 配置字段与各自{short-project-name} 策略的映射关系如下<<_mapping_kubewarden_policies_to_psp_fields,表>>所示。

通过{short-project-name} ，操作员可以对集群中的策略配置进行细粒度控制。

通过{short-project-name} 实例，可以部署策略来替换`PodSecurityPolicy` 对象。我们在本例中考虑这些规则：

* 禁止特权升级的 PSP
* 特权容器
* 阻止以根用户身份运行 pod
* 强制某个用户组
* 阻止主机命名空间
* 允许 pod 仅使用端口 443

该 PSP 的 YAML 定义为

.PSP YAML 定义
[%collapsible]
======

[subs="+attributes",yaml]
----
apiVersion: policy/v1beta1
kind: PodSecurityPolicy
metadata:
  name: restricted
spec:
  allowPrivilegeEscalation: false
  runAsUser:
    rule: MustRunAsNonRoot
  supplementalGroups:
    rule: MustRunAs
    ranges:
      - min: 1000
        max: 65535
  privileged: false
  hostNetwork: false
  hostIPC: false
  hostPID: false
  hostPorts:
    - min: 443
      max: 443
----

======

== {short-project-name} PSP 的替代品

现在，我们将创建{short-project-name} 政策，以实现同样的目标。
您可以通过单独的{short-project-name} 策略来执行每条规则。
因此，在这个示例中，您需要为执行每项规定制定单独的政策：

* 特权升级
* 用户和组配置
* 主机名称空间
* 特权容器配置。

=== 阻止容器权限升级

您可以如下图所示部署策略：

.用于策略部署的 kubectl 命令
[%collapsible]
======

[subs="+attributes",shell]
----
$ kubectl apply -f - <<EOF
apiVersion: policies.kubewarden.io/v1
kind: ClusterAdmissionPolicy
metadata:
  name: psp-allow-privilege-escalation
spec:
  module: ghcr.io/kubewarden/policies/allow-privilege-escalation-psp:v0.2.6
  rules:
    - apiGroups:
        - ""
      apiVersions:
        - v1
      resources:
        - pods
      operations:
        - CREATE
        - UPDATE
  mutating: false
  settings:
    default_allow_privilege_escalation: false
EOF
----

======

在该命令中，我们指定`default_allow_privilege_escalation` 为`false` 。此策略可限制那些试图以高于父容器的权限运行的 pod。

.试图提高权限的 kubectl 输出结果
[%collapsible]
======

[subs="+attributes",shell]
----
$ kubectl apply -f - <<EOF
apiVersion: v1
kind: Pod
metadata:
  name: nginx
spec:
  containers:
  - name: nginx
    image: nginx
    securityContext:
      allowPrivilegeEscalation: true
  - name: sidecar
    image: sidecar
EOF
Error from server: error when creating "STDIN": admission webhook "clusterwide-psp-allow-privilege-escalation.kubewarden.admission" denied the request: one of the containers has privilege escalation enabled
----

=== 用户和用户组配置

现在，要执行用户和组配置，可以使用link:https://github.com/kubewarden/user-group-psp-policy[user-group-psp 策略]。

.`kubectl` 命令来使用用户组-spp-策略
[%collapsible]
======
[subs="+attributes",shell]
----
$ kubectl apply -f - <<EOF
apiVersion: policies.kubewarden.io/v1
kind: ClusterAdmissionPolicy
metadata:
  name: psp-user-group
spec:
  module: ghcr.io/kubewarden/policies/user-group-psp:v0.4.9
  rules:
    - apiGroups:
        - ""
      apiVersions:
        - v1
      resources:
        - pods
      operations:
        - CREATE
        - UPDATE
  mutating: true
  settings:
    run_as_user:
      rule: MustRunAsNonRoot
    supplemental_groups:
      rule: MustRunAs
      ranges:
        - min: 1000
          max: 65535
EOF
----
======

您应该通过`mutation: true` 配置策略。这是必需的，因为当用户没有定义时，策略会添加https://kubernetes.io/docs/concepts/security/pod-security-policy/#users-and-groups[supplementalGroups]。

因此，现在用户无法部署以根用户身份运行的 pod：

[subs="+attributes",shell]

metadata:
  name: nginx
spec:
  containers:
  - name: nginx
    image: nginx
    securityContext:
      runAsNonRoot: false
      runAsUser: 0
EOF

=== 特权容器配置

您需要替换阻止特权容器的旧版 PSP 配置。有必要部署https://github.com/kubewarden/pod-privileged-policy[pod-privileged policy]。这项政策不需要任何设置。运行后，它将阻止有权限的 pod。

.应用 pod-privileged-policy
[%collapsible]
======

[subs="+attributes",shell]
----
$ kubectl apply -f - <<EOF
apiVersion: policies.kubewarden.io/v1
kind: ClusterAdmissionPolicy
metadata:
  name: psp-privileged
spec:
  module: ghcr.io/kubewarden/policies/pod-privileged:v0.2.7
  rules:
    - apiGroups:
        - ""
      apiVersions:
        - v1
      resources:
        - pods
      operations:
        - CREATE
        - UPDATE
  mutating: false
  settings: null
EOF
----

======

要测试该策略，我们可以尝试运行一个启用了特权配置的 pod：

.启用特权配置运行 Pod
[%collapsible]
======

[subs="+attributes",shell]
----
$ kubectl apply -f - <<EOF
apiVersion: v1
kind: Pod
metadata:
  name: nginx
spec:
  containers:
  - name: nginx
    image: nginx

      privileged: true
  - name: sleeping-sidecar
    image: alpine

EOF

=== 主机命名空间配置

要完成 PSP 迁移，需要禁用主机名空间共享。为此，我们将使用https://github.com/kubewarden/host-namespaces-psp-policy[`host-namespace-psp`policy]。它允许集群管理员单独阻止 IPC、PID 和网络命名空间。它还会设置 pod 在主机 IP 上可以打开的端口。

.禁用命名空间共享和设置端口
[%collapsible]
======

[subs="+attributes",shell]
----
$ kubectl apply -f - <<EOF
apiVersion: policies.kubewarden.io/v1
kind: ClusterAdmissionPolicy
metadata:
  name: psp-hostnamespaces
spec:
  module: ghcr.io/kubewarden/policies/host-namespaces-psp:v0.1.6
  rules:
    - apiGroups:
        - ""
      apiVersions:
        - v1
      resources:
        - pods
      operations:
        - CREATE
        - UPDATE
  mutating: false
  settings:
    allow_host_ipc: false
    allow_host_pid: false
    allow_host_ports:
      - min: 443
        max: 443
    allow_host_network: false
EOF
----

======

== PSP 移植脚本

{short-project-name} 团队开发了一个用于 PSP 迁移的脚本。它使用https://github.com/appvia/psp-migration[AppVia] 的迁移工具。AppVia 工具会读取 PSP YAML 配置。然后生成相应的政策。{short-project-name} 和其他策略引擎。

[CAUTION]
====

AppVia 迁移工具不受 Kuberwarden 维护者的控制。
这意味着它有可能生成过时的{short-project-name} 策略。谨慎使用。我们需要为 AppVia 提交拉取请求，相关工作正在进行中。
如有需要，请联系我们获取更多信息。
====


该脚本可在{short-project-name}https://github.com/kubewarden/utils/blob/main/scripts/psp-to-kubewarden[ utils]软件仓库中找到。它会将 AppVia 迁移工具下载到工作目录中使用。它处理`kubectl` 默认上下文中定义的 PSP。然后在标准输出上打印库伯沃登策略定义。用户可以将内容重定向到文件或直接重定向到`kubectl` 。

[NOTE]
====

此脚本仅适用于 Linux x86_64 机器。
====


让我们来看一个例子。与 PSP 组成一个群组：

* 阻止访问主机名空间
* 阻止特权容器
* 不允许特权升级
* 丢弃集装箱的能力
* 列出允许的卷类型
* 定义允许使用的用户和用户组
* 控制应用于卷的补充组
* 强制容器在只读根文件系统中运行

可以使用以下 YAML。

.PSP 配置
[%collapsible]
======

[subs="+attributes",yaml]
----
apiVersion: policy/v1beta1
kind: PodSecurityPolicy
metadata:
  name: restricted
spec:
  hostNetwork: false
  hostIPC: false
  hostPID: false
  hostPorts:
    - min: 80
      max: 8080
  privileged: false
  # Required to prevent escalations to root.
  allowPrivilegeEscalation: false
  requiredDropCapabilities:
    - ALL
  # Allow core volume types.
  volumes:
    - "configMap"
    - "emptyDir"
    - "projected"
    - "secret"
    - "downwardAPI"
    # Assume that ephemeral CSI drivers & persistentVolumes set up by the cluster admin are safe to use.
    - "csi"
    - "persistentVolumeClaim"
    - "ephemeral"
  runAsUser:
    # Require the container to run without root privileges.
    rule: "MustRunAsNonRoot"
  seLinux:
    # This policy assumes the nodes are using AppArmor rather than SELinux.
    rule: "RunAsAny"
  supplementalGroups:
    rule: "MustRunAs"
    ranges:
      # Forbid adding the root group.
      - min: 1
        max: 65535
  fsGroup:
    rule: "MustRunAs"
    ranges:
      # Forbid adding the root group.
      - min: 1
        max: 65535
  readOnlyRootFilesystem: true
----

======
