== 配置畸变和亲和/反亲和

通过使用`tolerations` 和`affinity` 字段，运营商可以对{short-project-name}堆栈的调度和可靠性进行微调，以满足其特定的部署需求和限制。有关具体字段及其配置的详细信息，请参阅https://kubernetes.io/docs/concepts/scheduling-eviction/taint-and-toleration/[Kubernetes 关于污点和容忍度的文档] 和https://kubernetes.io/docs/concepts/scheduling-eviction/assign-pod-node/#affinity-and-anti-affinity[亲和性和反亲和性]。

从{short-project-name} 堆栈的 1.15 版开始，{short-project-name} 舵手图表将提供两个新值：

* `.global.tolerations`
* `.global.affinity`

这些 Helm 图表值允许用户为{short-project-name} 堆栈定义 Kubernetes 容限和亲和/反亲和设置，包括控制器部署、审计扫描仪 cronjob 和默认的`PolicyServer` 自定义资源。

=== 允差

`tolerations` 值是一个数组，用户可在其中为{short-project-name} 组件指定 Kubernetes 容差。阈值允许在具有匹配污点的节点上调度 pod。这对于管理 Pod 的调度位置非常有用，尤其是在涉及节点维护、专用工作负载或特定硬件要求的情况下：

[source,yaml]
----
global:
  tolerations:
    - key: "key1"
      operator: "Equal"
      value: "value1"
      effect: "NoSchedule"
    - key: "key2"
      operator: "Equal"
      value: "value2"
      effect: "NoExecute"
----

在此示例中，定义的容差适用于控制器部署、审计扫描仪 cronjob 和默认`PolicyServer` 自定义资源。

=== 亲和力/反亲和力

`affinity` 值允许用户为{short-project-name} 组件定义 Kubernetes 亲和性和反亲和性规则。亲和性规则可将 pod 限制在特定节点上，而反亲和性规则则可防止 pod 被调度到某些节点上或靠近其他 pod。这些设置有助于确保集群的高可用性、容错和优化资源使用。

[source,yaml]
----
global:
  affinity:
    podAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        - labelSelector:
            matchExpressions:
              - key: security
                operator: In
                values:
                  - S1
          topologyKey: topology.kubernetes.io/zone
    podAntiAffinity:
      preferredDuringSchedulingIgnoredDuringExecution:
        - weight: 100
          podAffinityTerm:
            labelSelector:
              matchExpressions:
                - key: security
                  operator: In
                  values:
                    - S2
            topologyKey: topology.kubernetes.io/zone
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
          - matchExpressions:
              - key: kubernetes.io/os
                operator: In
                values:
                  - linux
      preferredDuringSchedulingIgnoredDuringExecution:
        - weight: 1
          preference:
            matchExpressions:
              - key: label-1
                operator: In
                values:
                  - key-1
        - weight: 50
          preference:
            matchExpressions:
              - key: label-2
                operator: In
                values:
                  - key-2
----

在此示例中，亲和规则将应用于控制器部署、审计扫描仪 cronjob 和默认`PolicyServer` 自定义资源。

`kubewarden-default`舵手图中以前的亲和配置只用于定义`PolicyServer` 的亲和配置，现在已被移除，改用全局`affinity`值。这一更改简化了配置过程，只需采用一种方法就能为所有{short-project-name} 组件定义亲和与反亲和规则。

[WARNING]
====
`kubewarden-default` 舵手图中的旧`affinity` 配置已被删除。现在，用户应使用`.global.affinity` 字段来配置整个{short-project-name} 堆栈的亲和与反亲和设置。
====

== 配置优先级类别

通过使用 priorityClasses，操作员可以为{short-project-name} 堆栈中的工作负载 pod 执行调度优先级。这可确保{short-project-name} 工作负载优先于其他工作负载，从而防止驱逐并确保服务可靠性。有关详细信息，请参阅https://kubernetes.io/docs/concepts/scheduling-eviction/pod-priority-preemption/[Kubernetes 关于 Priorityclasses 的文档]。

从{short-project-name} 协议栈的 1.25 版开始，{short-project-name} 舵手图表会有一个新的值：

* `.global.priorityClassName`

此值中由名称定义的 priorityClass 将应用于控制器部署 pod 和默认`PolicyServer` 自定义资源的 pod。

`.global.priorityClassName` 值希望是现有 PriorityClass 的名称。例如，我们可以使用

[source,yaml]
----
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: kubewarden-high-priority
value: 1000000
globalDefault: false
description: "This priority class should be used for XYZ service pods only."
----

Kubernetes 已经提供了两个适合的 PriorityClasses：`system-cluster-critical` 和`system-node-critical` 。这些都是常用类，用于https://kubernetes.io/docs/tasks/administer-cluster/guaranteed-scheduling-critical-addon-pods/[确保关键组件总是排在前面]。

[WARNING]
====
如果删除 PriorityClass，使用已删除 PriorityClass 名称的现有 Pod 将保持不变，但 Kubernetes 将不会创建使用已删除 PriorityClass 名称的后续 Pod。
====

== `PolicyServer` 生产配置

`PolicyServers` 对集群至关重要。它们的可靠性非常重要，因为它们会通过验证和突变 Webhooks 处理指向 Kubernetes API 的 "入网请求"。

与其他动态接纳控制器一样，这一过程在请求到达 Kubernetes API 服务器之前就已发生。动态接纳控制器的延迟或服务延迟可能会导致群集不一致、拒绝服务或死锁。

{short-project-name} 提供了几种提高`PolicyServers` 可靠性的方法。生产部署可能会有很大差异，这取决于操作员如何根据自己的需要配置部署。

=== PodDisruptionBudgets

{short-project-name} 控制器可为`PolicyServer` Pod 创建https://kubernetes.io/docs/tasks/run-application/configure-pdb/[PodDisruptionBudget](PDB)。这可以控制与`PolicyServer` 相关的`PolicyServer`Pod 复制的范围，确保在节点维护、扩展操作或群集升级时的高可用性和受控驱逐。

这可通过设置`spec.minAvailable` 或`PolicyServer` 资源的`spec.maxUnavailable`来实现：

* `minAvailable`规定了`PolicyServer` Pod 在任何时候都必须可用的最低数量。可以是整数或百分比。

  Useful for maintaining the operational integrity of the `PolicyServer`,
  ensuring that policies are continuously enforced without interruption.

* `maxUnavailable`：指定在任何给定时间内不可用的`PolicyServer` Pod 的最大数量。可以是整数或百分比。

  Useful for performing rolling updates or partial maintenance without
  fully halting the policy enforcement mechanism.

[NOTE]
====
只能指定`maxUnavailable` 和`minAvailable` 中的一个。
====

==== 配置 minAvailable 或 maxUnavailable

.例如设置最小可用
[source,yaml]
----
apiVersion: policies.kubewarden.io/v1
kind: PolicyServer
metadata:
  name: your-policy-server
spec:
  # Other configuration fields
  minAvailable: 2 # ensure at least two policy-server Pods are available at all times
----

.例如设置最大不可用
[source,yaml]
----
apiVersion: policies.kubewarden.io/v1
kind: PolicyServer
metadata:
  name: your-policy-server
spec:
  # Other configuration fields
  maxUnavailable: "30%" # ensure no more than 30% of policy-server Pods are unavailable at all times
----

=== 亲和力/反亲和力

{short-project-name} 控制器可以设置`PolicyServer` Pod 的亲和力。这允许将 Pod 限制在特定节点上，或将 Pod 限制在其他 Pod 上。有关 Affinity 的更多信息，请参阅https://kubernetes.io/docs/concepts/scheduling-eviction/assign-pod-node/#affinity-and-anti-affinity[Kubernetes docs]。

Kubernetes 亲缘配置允许将 Pod 限制在节点上（通过`spec.affinity.nodeAffinity` ）或将 Pod 限制在其他 Pod 上（通过`spec.affinity.podAffinity` ）。亲和力可以设置为软约束（使用`preferredDuringSchedulingIgnoredDuringExecution` ）或硬约束（使用`requiredDuringSchedulingIgnoredDuringExecution` ）。

针对特定标签的亲和/反亲和匹配，无论是节点标签（例如：`topology.kubernetes.io/zone` 设置为`antarctica-east1` ）还是 Pods 标签。根据`PolicyServer` 定义创建的 Pod 的标签`kubewarden/policy-server` 设置为`PolicyServer` 的名称（例如：`kubewarden/policy-server: default` ）。

[NOTE]
====
节点间亲和性/反亲和性需要大量处理，不建议在超过几百个节点的集群中使用。
====

要为`PolicyServer` 配置亲和性，请设置其`spec.affinity`字段。该字段接受与 Pod 的`spec.affinity` 内容相匹配的 YAML 对象。

==== 配置亲和力/反亲和力

.例如将`PolicyServer` Pod 分布在不同区域和主机名上
[source,yaml]
----
apiVersion: policies.kubewarden.io/v1
kind: PolicyServer
metadata:
  name: your-policy-server
spec:
  # Other configuration fields
  affinity:
    podAntiAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        - labelSelector:
            matchExpressions:
              - key: kubewarden/policy-server
                operator: In
                values:
                  - your-policy-server
          topologyKey: topology.kubernetes.io/zone
        - labelSelector:
            matchExpressions:
              - key: kubewarden/policy-server
                operator: In
                values:
                  - your-policy-server
          topologyKey: kubernetes.io/hostname
----

.例如仅在控制平面节点中调度`PolicyServer` pod
[source,yaml]
----
apiVersion: policies.kubewarden.io/v1
kind: PolicyServer
metadata:
  name: your-policy-server
spec:
  # Other configuration fields
  affinity:
    podAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        - labelSelector:
            matchExpressions:
              - key: kubewarden/policy-server
                operator: In
                values:
                  - your-policy-server
          topologyKey: node-role.kubernetes.io/control-plane
----

=== 限制和要求

{short-project-name} 控制器可以设置`PolicyServer` Pod 的资源限制和请求。这规定了与`PolicyServer` Pod 相关联的每个容器需要多少资源。对于`PolicyServers` ，只有`cpu` 和`memory` 资源是相关的。更多信息，请参阅https://kubernetes.io/docs/concepts/configuration/manage-resources-containers/#resource-units-in-kubernetes[Kubernetes docs] 资源单元。

这可以通过设置以下`PolicyServer` 资源字段来实现：

* `spec.limits`:由容器运行时强制执行的资源限制。不同的运行时可以有不同的限制执行方式。
* `spec.requests`:为每个容器预留的资源量。容器有可能使用比其`request` 更多的资源，这也是允许的。

  If omitted, it defaults to `spec.limits` if that is set (unless
  `spec.requests` of containers is set to some defaults via an admission
  mechanism).

[NOTE]
====
`PolicyServers` 的资源分配不足可能会导致群集出现可靠性问题。
====

==== 配置限制和请求

.例如为每个`PolicyServer` 容器设置硬限制
[source,yaml]
----
apiVersion: policies.kubewarden.io/v1
kind: PolicyServer
metadata:
  name: your-policy-server
spec:
  # Other configuration fields
  limits:
    cpu: 500m
    memory: 1Gi
----

=== PriorityClasses

{short-project-name} 控制器可以设置用于`PolicyServers` pod 的优先级类别。这意味着`PolicyServer` 工作负载会得到优先调度，防止被驱逐并确保服务可靠性。更多信息，请参阅https://kubernetes.io/docs/concepts/scheduling-eviction/pod-priority-preemption/[Kubernetes文档]。

[WARNING]
====
如果删除 PriorityClass，使用已删除 PriorityClass 名称的现有 Pod 将保持不变，但 Kubernetes 将不会创建使用已删除 PriorityClass 名称的后续 Pod。
====

==== 配置优先级类别

.例如使用默认的`system-cluster-critical` priorityClass：
[source,yaml]
----
apiVersion: policies.kubewarden.io/v1
kind: PolicyServer
metadata:
  name: your-policy-server
spec:
  # Other configuration fields
  priorityClassName: system-cluster-critical
----

=== 隔离政策工作量

为确保大规模运行时的稳定性和高性能，用户可以运行单独的*`PolicyServer`*部署，以隔离不同的工作负载。

* *将一个`PolicyServer` 专用于 "情境感知政策"：*这些策略需要更多资源，因为它们会查询 Kubernetes API 服务器或其他外部服务，如 Sigstore、OCI 注册表等。将它们隔离，可以防止速度慢的策略对其他速度快的策略造成瓶颈。
* *所有其他政策请使用另一个`PolicyServer` ：*在单独的服务器上运行常规的、独立的策略，以确保最常见的接纳请求的低延迟。

您还可以考虑进一步分担工作量。例如，如果有些策略执行速度较慢，需要较大的执行超时，可以考虑将它们移至专用的`PolicyServer` 。这样就能确保策略不会阻止工作人员评估其他请求。

=== 资源分配和扩展

为处理高流量并确保可用性，请提供充足的资源并扩展副本。

* *分配足够的资源：*在高流量环境中，为每个副本分配充足的资源。不要让`PolicyServers` 陷入饥饿状态，因为 CPU 或内存不足是导致请求超时的主要原因。请记住，`PolicyServers` 将接收来自控制平面和{short-project-name} 审计扫描仪的请求。
* *扩展高可用性：*对于每秒处理数百个请求的部署，可运行大量副本。这样可以有效地分配负载，确保少数 pod 出现故障不会影响集群的运行。

从 3-5 个副本的基线开始，监控 CPU 和内存的使用情况。根据需要调整副本数量。

=== 有效的大规模审计

要在大型集群上高效运行审计，需要对审计扫描仪的性能和并行性进行微调。

* *定期安排审计：*频繁运行扫描可以在捕捉配置漂移和尽量减少 API 服务器负载之间取得良好的平衡。
* *积极调整并行性：*快速审计的关键在于并行化。利用高并行性设置，可以将大规模集群的审计时间缩短到一个多小时。

[WARNING]
====
重要的是要记住，审计扫描仪会向`PolicyServers` 发送请求。因此，其并行性会影响`PolicyServer`的性能。如果想在大群集中采用积极的并行方式来缩短扫描时间，可能需要增加可用的策略服务器资源，以避免影响接纳控制器的性能。
====

如果您从日志中消耗审计结果，并且不需要策略`Reports` 或`ClusterReports` 群集中的自定义资源，则设置`disableStore: true` 以减少负载。
