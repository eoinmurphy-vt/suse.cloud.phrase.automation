{project-name} sont des programmes réguliers compilés en WebAssembly (Wasm).
Comme pour tout type de programme, il est important de disposer d'une bonne couverture de test.

Les auteurs de politiques peuvent utiliser leurs environnements de développement préférés.
Vous pouvez utiliser des outils familiers et des cadres de test pour vérifier le développement.

Ces politiques {short-project-name} sont des exemples écrits enxref:tutorials/writing-policies/rust/01-intro-rust.adoc[Rust] etxref:tutorials/writing-policies/go/01-intro-go.adoc[Go]:

* https://github.com/kubewarden/psp-apparmor[psp-apparmor]
* https://github.com/kubewarden/ingress-policy[politique d'entrée]

Ils disposent de suites de tests utilisant des tests standard pour leurs environnements de développement.

Les politiques utilisent les actions GitHub pour leurs pipelines de CI.

== Tests de bout en bout

Vous pouvez également écrire des tests qui s'exécutent contre le binaire Wasm contenant votre politique.
Pour ce faire sans avoir à déployer un cluster Kubernetes, vous pouvez utiliser ces outils :

* https://github.com/bats-core/bats-core[bats]: il s'agit d'écrire des tests et d'automatiser leur exécution.
* https://github.com/kubewarden/kwctl[`kwctl`]: outil CLI par défaut de {short-project-name} qui vous aide à effectuer les opérations liées aux politiques : tirer, inspecter, annoter, pousser et exécuter.

Pour utiliser `kwctl run`, vous devez disposer des éléments suivants :

. La référence du fichier binaire Wasm de la politique à exécuter.
La politique {short-project-name} peut être chargée à partir de :
 ** le système de fichiers local (`file://`)
 ** un serveur HTTP(s) (`https://`
 ** un registre OCI (`registry://`).
. L'objet de demande d'admission à tester.
Vous le donnez via l'argument `--request-path`, ou sur `stdin` en réglant `--request-path` sur `-`.
. Les paramètres de la politique d'exécution sous forme de JSON en ligne via l'indicateur `--settings-json`.
Ou un fichier JSON ou YAML, chargé depuis le système de fichiers via `--settings-path`.

Après le test `kwctl`, l'objet `ValidationResponse` est imprimé sur la sortie standard.

C'est ainsi que l'on utilise `kwctl` pour tester le binaire Wasm de l'application https://github.com/kubewarden/ingress-policy[`ingress-policy`] mentionné précédemment :

[subs="+attributes",console]
----
$ curl https://raw.githubusercontent.com/kubewarden/ingress-policy/v0.1.8/test_data/ingress-wildcard.json 2> /dev/null | \
    kwctl run \
        --settings-json '{"allowPorts": [80], "denyPorts": [3000]}' \
        --request-path - \
        registry://ghcr.io/kubewarden/policies/ingress:v0.1.8 | jq
----

Vous pouvez télécharger les binaires préconstruits de `kwctl`https://github.com/kubewarden/kwctl/releases[ ici.]

En utilisant `bats`, vous pouvez écrire un test qui exécute cette commande et recherche les résultats attendus :

.Un test sur les chauves-souris
[%collapsible]
======

[,bash]
----
@test "all is good" {
  run kwctl run \
    --request-path test_data/ingress-wildcard.json \
    --settings-json '{"allowPorts": [80], "denyPorts": [3000]}' \
    ingress-policy.wasm

  # this prints the output when one the checks below fails
  echo "output = ${output}"

  # settings validation passed
  [[ "$output" == *"valid: true"* ]]

  # request accepted
  [[ "$output" == *"allowed: true"* ]]
}
----

======

Vous pouvez placer le code dans un fichier, `e2e.bats`, par exemple, et invoquer `bats` par :

[subs="+attributes",console]
----
$ bats e2e.bats
 ✓ all is good

1 tests, 0 failures
----
