= Création d'une nouvelle politique de validation
:revdate: 2026-03-18
:page-revdate: {revdate}
:description: Explorer la création d'une nouvelle politique de validation qui rejette les Pods avec des étiquettes de liste de refus, tout en validant les étiquettes avec des expressions régulières fournies par l'utilisateur pour assurer la sécurité.
:doc-persona: ["kubewarden-policy-developer"]
:doc-topic: ["kubewarden", "writing-policies", "go", "creating a new validation policy"]
:doc-type: ["tutorial"]
:keywords: ["kubewarden", "kubernetes", "writing policies in Go", "new validation policy"].
:current-version: {page-origin-branch}

Ce tutoriel traite de la création d'une politique qui valide les étiquettes des objets Pod.

La politique consiste à rejeter tous les Pods qui utilisent une ou plusieurs étiquettes de la liste de refus.
La politique valide également certaines étiquettes à l'aide d'une expression régulière fournie par l'utilisateur.

En résumé, les paramètres de la politique devraient ressembler à ceci :

[subs="+attributes",yaml]
----
# List of labels that cannot be used
denied_labels:
- foo
- bar

# Labels that are validated with user-defined regular expressions
constrained_labels:
  priority: "[123]"
  cost-center: "^cc-\d+"
----

La politique rejette la création de ce pod :

[subs="+attributes",yaml]
----
apiVersion: v1
kind: Pod
metadata:
  name: nginx
  labels:
    foo: hello world
spec:
  containers:
    - name: nginx
      image: nginx:latest
----

Elle rejette également la création de ce Pod :

[subs="+attributes",yaml]
----
apiVersion: v1
kind: Pod
metadata:
  name: nginx
  labels:
    cost-center: cc-marketing
spec:
  containers:
    - name: nginx
      image: nginx:latest
----

Vous pouvez utiliser les paramètres de la politique pour imposer l'utilisation d'une spécification d'étiquette, quel qu'en soit le contenu :

[subs="+attributes",yaml]
----
constrained_labels:
  mandatory-label: ".*" # <- this label must be present, we don't care about its value
----

== Échafaudage pour un nouveau projet politique

Vous pouvez créer un nouveau projet de politique à l'aide duhttps://github.com/kubewarden/go-policy-template[référentiel de modèles].
Sélectionnez le bouton vert "Utiliser ce modèle" en haut de la page et suivez l'assistant de GitHub.

Clonez le dépôt localement et définissez la directive `module` dans le fichier `go.mod` comme suit :

[,go]
----
module <path to your repository>
----

[NOTE]
====
Une véritable politique utiliserait un chemin d'accès au référentiel, comme `github.com/kubewarden/go-policy-template`.
====


== Essais

Si les outils nécessaires sont en place, la commande `make test` utilise Docker pour extraire une image du compilateur TinyGo et l'utiliser pour construire et tester le modèle de politique.

La commande par défaut `make` construit la cible `policy.wasm`. Ensuite, `make test` exécute les tests Go définis.
La commande `make e2e-tests` exécute des tests à l'aide des battes https://github.com/bats-core/bats-core[] au sein d'un cluster {short-project-name}.
Après avoir cloné le site `go-policy-template`, l'exécution de ces commandes permet de vérifier que vous disposez des outils nécessaires à la réalisation de ce tutoriel.
