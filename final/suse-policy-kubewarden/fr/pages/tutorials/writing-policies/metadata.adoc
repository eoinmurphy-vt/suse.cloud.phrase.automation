= Métadonnées de la politique
:revdate: 2026-03-18
:page-revdate: {revdate}
:description: Configurez les métadonnées de la politique dans le fichier metadata.yaml pour définir les paramètres de la politique, les informations sur l'auteur et les autorisations sur les ressources.
:doc-persona: ["kubewarden-policy-developer"]
:doc-topic: ["kubewarden", "writing-policies", "policy-metadata"]]
:doc-type: ["tutorial"]
:keywords: ["kubewarden", "kubernetes policy development", "policy metadata"].
:current-version: {page-origin-branch}

Le fichier de métadonnées {project-name}, `metadata.yaml`, est un fichier de configuration contenant des informations et des paramètres importants liés aux politiques utilisées dans {short-project-name}.
Cette documentation donne un aperçu de l'objectif et de l'utilisation du fichier de métadonnées.

Le fichier `metadata.yaml` contient les valeurs par défaut de la politique, ainsi que des métadonnées telles que l'auteur et la description, définies par l'auteur de la politique.
La commande `kwctl annotate` utilise le fichier pour annoter le fichier `.wasm` contenant la politique.
Par conséquent, toutes les informations pertinentes nécessaires à la gestion de la politique sont disponibles.
Vous trouverez plus d'informations sur la façon d'annoter la politique dans le guidexref:../../explanations/distributing-policies.adoc[Distribuer des politiques.] 

Lorsque les utilisateurs d'une politique veulent l'utiliser, ils génèrent un manifeste YAML à l'aide de `kwctl scaffold`.
Cette commande lit les métadonnées de politique intégrées dans le module Wasm livré, effectue des vérifications et renvoie un manifeste YAML que l'auteur peut utiliser tel quel ou modifier.

En tant qu'auteur de la police, vous pouvez adapter le fichier `metadata.yaml` fourni lors de l'échafaudage de votre police.

Voir l'exemple suivant de `metadata.yaml`:

[subs="+attributes",yaml]
----
rules:
  - apiGroups: [""]
    apiVersions: ["v1"]
    resources: ["pods"]
    operations: ["CREATE"]
mutating: false
contextAwareResources: []
executionMode: kubewarden-wapc
policyType: kubernetes
backgroundAudit: true
annotations:
  # kubewarden specific:
  io.kubewarden.policy.title: My policy
  io.kubewarden.policy.version: 0.1.0 # should match the OCI tag
  io.kubewarden.policy.description: Short description
  io.kubewarden.policy.author: myself
  io.kubewarden.policy.url: https://github.com/yourorg/my-policy
  io.kubewarden.policy.source: https://github.com/yourorg/my-policy
  io.kubewarden.policy.license: Apache-2.0
  # The next two annotations are used in the policy report generated by the
  # Audit scanner. Severity indicates policy check result criticality and
  # Category indicates policy category. See more here at docs.kubewarden.io
  io.kubewarden.policy.severity: medium
  io.kubewarden.policy.category: Resource validation
  # artifacthub specific: (optional, to release in Artifact Hub)
  io.kubewarden.policy.ociUrl: ghcr.io/myorg/my-policy
  io.artifacthub.displayName: Policy Name
  io.artifacthub.resources: Pod
  io.artifacthub.keywords: pod, cool policy, kubewarden
----

== Activation des vérifications d'arrière-plan

Le fichier de métadonnées comprend un indicateur, `backgroundAudit`, qui active les contrôles d'audit en arrière-plan pour une politique spécifique.
Par défaut, cet indicateur est fixé à `true`.

Pour certaines politiques, en raison de leur mode de fonctionnement ou du type d'événements qu'elles concernent, ce champ doit être fixé à `false`.
Vous trouverez de plus amples informations dans laxref:explanations/audit-scanner/limitations.adoc[documentation du scanner d'audit], sous la rubrique "limitations".

== Définir les ressources Kubernetes auxquelles les politiques peuvent accéder.

Dans le fichier de métadonnées, à l'aide du champ `contextAwareResources`, les utilisateurs peuvent définir les ressources Kubernetes auxquelles la politique peut accéder.
Par exemple, si la politique a besoin d'accéder à la ressource `Namespace`.
L'auteur de la politique peut définir le site `contextAwareResources` comme suit :

[subs="+attributes",yaml]
----
[...]
contextAwareResources:
  - apiVersion: v1 kind: Namespace
[...]
----

== Spécifier les politiques comme mutantes ou non mutantes

Le fichier de métadonnées comporte un indicateur, `mutating`, qui permet aux utilisateurs de configurer une politique comme étant mutante ou non mutante.
Une politique de mutation modifie les demandes entrantes ou les ressources gérées.
Une politique de non-mutations observe et applique les restrictions sans apporter de modifications.
Cette distinction est cruciale pour déterminer comment les politiques interagissent avec les ressources Kubernetes et leur impact sur le cluster.

== Spécifier le type de politique comme Kubernetes ou Raw

Le fichier de métadonnées comporte un indicateur, `policyType`, qui permet aux utilisateurs de marquer une politique comme étant soit `kubernetes`, soit `raw`.
Une politique Kubernetes est une politique qui valide les ressources Kubernetes.
Une politique Raw est une politique qui valide des documents JSON arbitraires.
Par défaut, s'il n'est pas spécifié par l'utilisateur, ce champ est fixé à `kubernetes` lors de l'annotation d'une politique.
Pour plus d'informations, reportez-vous à la section xref:../../howtos/raw-policies.adoc[Politiques brutes].

== Définition des cibles des types de ressources

Le fichier de métadonnées permet aux utilisateurs de définir les règles dans le champ `rules`, qui détermine les types de ressources auxquels la politique s'applique.
Cette fonctionnalité permet aux utilisateurs d'exercer un contrôle précis sur l'application des politiques, en garantissant qu'elles ne sont appliquées qu'aux types de ressources prévus.
Grâce à ce contrôle fin, les utilisateurs peuvent garantir que les politiques sont ciblées avec précision, qu'elles correspondent à leurs besoins spécifiques et qu'elles évitent toute application involontaire de politiques à des types de ressources non liés.
