Le https://wasi.dev/[site] est une norme de WebAssembly qui fournit un ensemble d'interfaces permettant l'exécution de WebAssembly en dehors du navigateur.

[CAUTION]
====

Les auteurs qui rédigent des politiques ordinaires ne doivent jamais utiliser les interfaces système WASI ordinaires pour rédiger des politiques.

Cette page s'adresse aux mainteneurs de {project-name} ou aux auteurs de politiques de bas niveau qui souhaitent expérimenter les plates-formes WASM les plus récentes.
====


En utilisant WASI, vous pouvez avoir un module WebAssembly qui interagit avec les primitives du système comme STDOUT, STDERR, STDIN, les variables d'environnement et plus encore.

De nombreux compilateurs utilisés pour compiler les politiques {short-project-name} produisent des modules WebAssembly qui ciblent les interfaces WASI.
Cependant, les politiques {short-project-name} utilisent le projet https://github.com/wapc[waPC] pour mettre en œuvre une communication bidirectionnelle entre la politique et le moteur d'exécution de la politique (`kwctl` ou `policy-server`). L'utilisation du protocole de communication par{short-project-name} est décrite xref:reference/spec/01-intro-spec.adoc[ici].

Il existe des cas particuliers où le projet waPC ne peut pas encore être utilisé.
Dans ce cas, vous pouvez rédiger une politique en utilisant les interfaces fournies par WASI.

[IMPORTANT]
====

{short-project-name} prend en charge les politiques WASI à partir de la version 1.7.0 de {short-project-name}.
====


== Limites

Vous ne devriez pas utiliser les stratégies WASI dans des circonstances normales, car elles sont moins performantes au moment de l'évaluation que les stratégies waPC/Rego.

[NOTE]
====

La communication bidirectionnelle entre la politique et l'hôte est possible, mais nécessite des changements dans le SDK du langage. Cela est nécessaire pour utiliser les xref:/reference/spec/host-capabilities/01-intro-host-capabilities.adoc[capacités de l'hôte] et pour rédiger des politiquesxref:/explanations/context-aware-policies.adoc[tenant compte du contexte].

Actuellement, seuls les SDK {short-project-name} Go et JavaScript/TypeScript les exposent aux politiques WASI.

Si cela vous intéresse, n'hésitez pas à nous contacter. Nous pouvons alors établir des priorités.

====


== Cas d'utilisation

La seule raison d'écrire une politique "plain WASI" est de ne pas pouvoir utiliser le mécanisme de communication waPC.

Actuellement (à partir de juin 2023), la seule bonne raison de le faire est d'utiliser le compilateur Go officiel pour produire un module WebAssembly.

À partir de la version 1.21, le compilateur Go officiel est capable de produire des modules WebAssembly ciblant l'interface WASI.
Cependant, ces modules ne peuvent pas encore exporter de fonctions vers le moteur d'exécution WebAssembly.
Cette limitation, signalée par https://github.com/golang/go/issues/42372[], empêche l'adoption du protocole waPC.

L'équipe du projet {short-project-name} conseille d'écrire des politiques Go {short-project-name} en utilisant le compilateur TinyGo, comme décrit xref:tutorials/writing-policies/go/01-intro-go.adoc[ici.]

Cependant, certaines bases de code Go complexes ne peuvent pas être compilées à l'aide du compilateur TinyGo.
Cela inclut, par exemple, des bases de code telles que https://github.com/google/cel-go[Kyverno] CEL-go ou https://github.com/kyverno/kyverno/[].
Dans ces circonstances, l'utilisation du compilateur officiel Go peut s'avérer utile.

== Protocole de communication

Cette section décrit comment rédiger une politique WASI simple.

Vous devez écrire le code comme un programme CLI normal.
Le programme doit comporter les sous-commandes suivantes :

* `validate`Cette commande est invoquée par le moteur de politique pour évaluer une demande d'admission.
* `validate-settings`Cette commande est invoquée par le moteur de politique pour valider les paramètres de la politique.

Dans les deux cas, les données à valider sont fournies via STDIN.
La politique doit fournir la réponse via STDOUT.
Vous pouvez utiliser STDERR pour les messages de débogage ou d'erreur.

=== Validation

La validation d'une demande se fait lors de l'invocation du programme CLI de politique à l'aide de la sous-commande `validate`.

STDIN doit contenir un document JSON décrivant un objet `ValidationRequest`.
La politique doit écrire à STDOUT un document JSON contenant un objet `ValidationResponse`.

Les objets `ValidationRequest` et `ValidationResponse` sont décrits xref:reference/spec/03-validating-policies.adoc[ici.]

=== Mutation

Les politiques de mutation fonctionnent de la même manière que les politiques de validation.
Le programme CLI de politique est invoqué à l'aide de la sous-commande `validate`.

STDIN doit contenir un document JSON décrivant un objet `ValidationRequest`.
La politique doit écrire sur STDOUT un document JSON contenant un objet `ValidationResponse`.

Les objets `ValidationRequest` et `ValidationResponse` sont décrits xref:reference/spec/03-validating-policies.adoc[ici.]

Lorsqu'une mutation est nécessaire, l'objet `ValidationResponse` doit avoir une clé, `mutated_object`, contenant l'objet à créer.
Ce processus est décrit xref:reference/spec/04-mutating-policies.adoc[ici.]

== Context-aware

Pour l'instant, il n'est pris en charge que par le SDK Go. Le SDK Go expose les capacités contextuelles comme d'habitude, pour plus d'informations, voir xref:explanations/context-aware-policies.adoc[ici.]

Pour un exemple de politique WASI Go context-aware, voirhttps://github.com/kubewarden/go-wasi-context-aware-test-policy[go-wasi-context-aware-test-policy].

=== Validation des paramètres

La politique doit fournir une sous-commande nommée `validate-settings`.
Cette commande permet de valider les paramètres fournis par l'utilisateur.

Le programme doit recevoir sur STDIN un objet JSON contenant les paramètres fournis par l'utilisateur.
Il les valide ensuite et écrit un objet `SettingsValidationResponse` sur STDOUT.

Le format du site `SettingsValidationResponse` et le processus de validation des paramètres sont décrits xref:reference/spec/02-settings.adoc[ici.]

== Métadonnées de la politique

Chaque politique {short-project-name} doit être annotée par la commande `kwctl annotate`.
Les métadonnées d'une politique WASI ordinaire doivent avoir cette valeur :

[subs="+attributes",yaml]
----
executionMode: wasi
----
