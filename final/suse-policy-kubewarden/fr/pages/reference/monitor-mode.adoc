= Mode moniteur
:revdate: 2026-03-18
:page-revdate: {revdate}
:description: Configurez ClusterAdmissionPolicies ou AdmissionPolicies en mode moniteur pour suivre et analyser les actions de la politique sans modifier les demandes entrantes.
:doc-persona: ["kubewarden-user", "kubewarden-operator", "kubewarden-integrator"]
:doc-topic: ["operator-manual", "monitor-mode"]
:doc-type: ["reference"]
:keywords: ["kubernetes", "kubewarden", "monitor mode"].
:sidebar_label: Mode moniteur
:sidebar_position: 50
:current-version: {page-origin-branch}

Lors de la définition d'une politique, vous pouvez choisir entre deux modes, spécifiés dans sa section xref:reference/CRDs.adoc#admissionpolicyspec[`spec.mode`].
Par défaut, vous déployez une politique à l'adresse `mode: protect`.
La politique accepte, rejette ou modifie les demandes.

On peut choisir de déployer une politique en mode moniteur. En mode moniteur :

* La police accepte toutes les demandes, comme si elle n'était pas installée.
* Le site `policy-server` retrace la politique normalement. Si une demande est rejetée ou si une politique propose une mutation, la trace contient les détails.
* Les métriques de `policy-server` sont mises à jour normalement, le mode étant inclus dans le bagage métrique. Il est donc facile de filtrer les politiques par mode et de se concentrer sur celles qui sont déployées en mode `monitor`.

Le site `mode` est un attribut inclus dans les ressources `ClusterAdmissionPolicy` et`AdmissionPolicy`. L'attribut `mode` peut prendre deux valeurs : `monitor` et `protect`. L'adresse `mode` devient par défaut `protect` si elle est omise.

Pour créer une politique dans `monitor mode`, vous devez inclure l'énoncé `mode : monitor` dans la spécification de la ressource. Par exemple, dans la section`spec` (marquée ➀), de ce `ClusterAdmissionPolicy`:

[subs="+attributes",yaml]
----
apiVersion: policies.kubewarden.io/v1alpha2
kind: ClusterAdmissionPolicy
metadata:
  name: psp-capabilities
spec:
  mode: monitor <1>
  policyServer: reserved-instance-for-tenant-a
  module: registry://ghcr.io/kubewarden/policies/psp-capabilities:v0.1.3
  rules:
  - apiGroups: [""]
    apiVersions: ["v1"]
    resources: ["pods"]
    operations:
    - CREATE
    - UPDATE
  mutating: true
  settings:
    allowed_capabilities:
    - CHOWN
    required_drop_capabilities:
    - NET_ADMIN
----
<1> L'attribut `mode: monitor` se trouve dans la section `spec`.

== Changement de mode d'action

Pour des raisons de sécurité, un utilisateur disposant des autorisations `UPDATE` sur les ressources de la politique peut rendre la politique plus restrictive. Cela signifie que vous pouvez changer le `mode` d'un `ClusterAdmissionPolicy` ou d'un `AdmissionPolicy` existant de `monitor` à`protect`.

Cependant, vous ne pouvez pas changer le `mode` d'un `ClusterAdmissionPolicy` ou d'un`AdmissionPolicy` existant de `protect` à `monitor`.

Ainsi, pour modifier le site `mode` d'une politique de `protect` à `monitor`, vous devez supprimer la politique et la recréer en mode `monitor`. Faire passer une politique de`protect` à `monitor` revient à la supprimer. Cette approche suppose donc que l'utilisateur dispose d'autorisations de suppression de politique.

== Une note sur les politiques de mutation

La mutation des politiques en mode `monitor` n'entraîne pas de mutation sur la ressource.
En mode `monitor`, les politiques consignent ce qu'elles auraient fait. Ils enregistrent également le patch de mutation qu'ils auraient produit en mode `protect`.

[WARNING]
====
Lorsqu'une politique de mutation est en mode `monitor`, les politiques ultérieures évaluent une ressource inchangée, et donc différente, que lorsque la politique de mutation est en mode`protect`.
====
