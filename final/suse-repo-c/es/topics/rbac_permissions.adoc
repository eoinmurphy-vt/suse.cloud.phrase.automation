== Visión general

Los permisos en SUSE Observability permiten a los administradores gestionar las acciones que cada usuario o grupo de usuarios puede realizar dentro de SUSE Observability y la información que se mostrará en su interfaz de usuario de SUSE Observability. Sólo se presentará el conjunto de características relevantes para el rol activo de cada usuario. Las acciones, la información y las páginas a las que un usuario no tiene acceso simplemente no se muestran en su interfaz de usuario de SUSE Observability.

[NOTE]
====
Los permisos se almacenan en StackGraph. Esto significa que:

* Si realiza una actualización con "borrar todos los datos", también se eliminará la configuración de permisos.
* Para eliminar completamente a un usuario, también hay que eliminarlo manualmente de StackGraph.
====


== Permisos de SUSE Observability

Existen dos tipos de permisos en SUSE Observability. *Los permisos del sistema* abarcan las capacidades del usuario, como el acceso a la configuración, la ejecución de consultas y la creación de secuencias de comandos. *Los permisos de ámbito* determinan el acceso a los datos de observabilidad y a algunos tipos de recursos como SUSE Observability Views.  Para las operaciones CRUD en SUSE Observability Views, pueden concederse para una vista específica o para todas las vistas. Para obtener información detallada sobre los permisos asociados a cada función predefinida en SUSE Observability, consulte xref:/setup/security/rbac/rbac_roles.adoc#_predefined_roles[funciones predefinidas]

=== Permisos de alcance

Puede conceder determinados permisos a todo el sistema o limitarlos a un subconjunto de recursos.  Puede darles un *Nombre de recurso*, que indica el ámbito.  La interpretación de este nombre depende del tipo de recurso.  Tenga en cuenta que el alcance puede aplicarse sólo a algunos de los verbos.

* `views` - El nombre de la vista
* `topology` - Cualquier etiqueta de componente. En un entorno Kubernetes, esto puede ser, por ejemplo, `cluster-name:<cluster-name>` o `k8s-scope:<cluster-name>/<namespace>`
* `metrics` - En un entorno Kubernetes, `k8s:<cluster-name>:\\__any__` o `k8s:<cluster-name>:<namespace>`
* `traces` - En un entorno Kubernetes, `k8s.cluster.name:<cluster-name>` o `k8s.scope:<cluster-name>/<namespace>`
* `scoped-permissions` - El nombre del clúster para el que se gestionan los permisos

== Gestionar permisos

Puede gestionar los permisos de SUSE Observability mediante la CLI `sts`.
Para los permisos de ámbito, se puede especificar un nombre de recurso.  Si no se da ninguno, el permiso se concederá (o revocará) en todo el sistema.

[NOTE]
====
*Nota importante:* Todos los permisos en SUSE Observability distinguen entre mayúsculas y minúsculas.
====


=== Lista de todos los permisos disponibles

Lista todos los permisos disponibles:

[,text]
----
sts rbac list-permissions
----

=== Enumerar todas las asignaturas

Enumera todas las asignaturas:

[,text]
----
sts rbac describe-subjects

SUBJECT                                    | SOURCE
stackstate-admin                           | Static
suse-observability-instance-admin          | Kubernetes
suse-observability-agent-clusterA          | Observability
----
[NOTE]
====
`+La asignatura `+SOURCE` muestra a quién pertenece y quién gestiona la asignatura. Puede ser una de nuestras fuentes RBAC (Configuración|Estática, Observabilidad, Kubernetes)

* Configuración|Estática: Sujetos obtenidos mediante la configuración de Suse Observability a través de helm
* Observabilidad: Sujetos gestionados a través de la sts cli
* Kubernetes: Sujetos obtenidos a través del agente RBAC de Kubernetes RoleBindings y ClusterRoleBindings
====

=== Lista de temas de un usuario

Lista todos los sujetos que están asignados al contexto, la primera entrada que se muestra es el nombre de usuario, y los posibles roles que tiene el usuario:

[,text]
----
sts user-session roles

user-dkzwh
keycloakoidc_group://suse-observability-instance-troubleshooter
keycloakoidc_group://suse-observability-instance-observer
keycloakoidc_group://suse-observability-cluster-observer
keycloakoidc_group://suse-observability-instance-admin
keycloakoidc_group://suse-observability-observer
----

Lista todas las asignaturas que están asignadas a un usuario, proporcionando su `api-token`(útil ya que el administrador puede ayudar a depurar a los usuarios con menos privilegios):

[,text]
----
sts user-session roles --api-token [api-token]

u-3p775e7w7l
keycloakoidc_group://suse-observability-cluster-observer
----

=== Mostrar permisos concedidos

Mostrar los permisos concedidos a un usuario o rol específico.

[,text]
----
sts rbac describe-permissions --subject [role-name]

PERMISSION   | RESOURCE
get-topology | cluster-name:rbac-t2
get-traces   | k8s.cluster.name:rbac-t2
get-metrics  | k8s:rbac-t2:__any__
----

[NOTE]
====
`--subject` puede ser cualquiera de los temas disponibles en las diferentes fuentes RBAC (Configuración, Observabilidad, Kubernetes)
====

=== Conceder permisos
[NOTE]
====
`--subject` puede ser un tema de Observabilidad, los temas de Configuración y Kubernetes son de sólo lectura
====

==== Permitir que un usuario abra una vista

Dar permiso a un sujeto para abrir una vista:

[,text]
----
sts rbac grant --subject [role-name] --permission get-views --resource [view-name]
----

==== Permitir a un usuario crear vistas

Dar permiso a un sujeto con el sistema para crear vistas:

[,text]
----
sts rbac grant --subject [role-name] --permission create-views
----

==== Permitir que un usuario compruebe la configuración de SUSE Observability

Conceda a un sujeto del sistema permiso para comprobar la configuración de SUSE Observability:

[,text]
----
sts rbac grant --subject [role-name] --permission get-settings
----

=== Revocar permisos
[NOTE]
====
`--subject` puede ser un tema de Observabilidad, los temas de Configuración y Kubernetes son de sólo lectura
====

Revocar los permisos de un sujeto para abrir una vista:

[,text]
----
sts rbac revoke --subject [role-name] --permission get-views --resource [view-name]
----
