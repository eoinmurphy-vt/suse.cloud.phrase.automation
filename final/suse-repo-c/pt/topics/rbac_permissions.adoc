== Visão geral

As permissões no SUSE Observability permitem que os administradores gerenciem as ações que cada usuário ou grupo de usuários pode realizar no SUSE Observability e as informações que serão exibidas na interface de usuário do SUSE Observability. Somente o conjunto de recursos relevante para a função ativa de cada usuário será apresentado. As ações, informações e páginas às quais um usuário não tem acesso simplesmente não são exibidas na interface do usuário do SUSE Observability.

[NOTE]
====
As permissões são armazenadas no StackGraph. Isso significa que:

* Se você realizar uma atualização com "limpar todos os dados", a configuração de permissões também será removida.
* Para remover completamente um usuário, ele também deve ser removido manualmente do StackGraph.
====


== Permissões do SUSE Observability

Há dois tipos de permissão no SUSE Observability. *As permissões do sistema* abrangem os recursos do usuário, como acesso a configurações, execução de consultas e criação de scripts. *As permissões com escopo* determinam o acesso aos dados de observabilidade e a alguns tipos de recursos, como as visualizações do SUSE Observability.  Para operações CRUD nas visualizações do SUSE Observability, elas podem ser concedidas para uma visualização específica ou para todas as visualizações. Para obter detalhes sobre as permissões associadas a cada função predefinida no SUSE Observability, consulte xref:/setup/security/rbac/rbac_roles.adoc#_predefined_roles[funções predefinidas]

=== Permissões com escopo

Você pode conceder determinadas permissões em todo o sistema ou pode limitá-las a um subconjunto de recursos.  Você pode dar a eles um *Nome do recurso*, que indica o escopo.  A interpretação desse nome depende do tipo de recurso.  Observe que o escopo pode se aplicar apenas a alguns dos verbos.

* `views` - O nome da exibição
* `topology` - Qualquer rótulo de componente. Em um ambiente Kubernetes, isso pode ser, por exemplo, `cluster-name:<cluster-name>` ou `k8s-scope:<cluster-name>/<namespace>`
* `metrics` - Em um ambiente Kubernetes, `k8s:<cluster-name>:\\__any__` ou `k8s:<cluster-name>:<namespace>`
* `traces` - Em um ambiente Kubernetes, `k8s.cluster.name:<cluster-name>` ou `k8s.scope:<cluster-name>/<namespace>`
* `scoped-permissions` - O nome do cluster para o qual as permissões são gerenciadas

== Gerenciar permissões

Você pode gerenciar as permissões do SUSE Observability usando a CLI `sts`.
Para permissões com escopo, um nome de recurso pode ser especificado.  Se nenhuma for fornecida, a permissão será concedida (ou revogada) em todo o sistema.

[NOTE]
====
*Observação importante:* Todas as permissões no SUSE Observability diferenciam maiúsculas de minúsculas.
====


=== Listar todas as permissões disponíveis

Listar todas as permissões disponíveis:

[,text]
----
sts rbac list-permissions
----

=== Listar todos os assuntos

Listar todos os assuntos:

[,text]
----
sts rbac describe-subjects

SUBJECT                                    | SOURCE
stackstate-admin                           | Static
suse-observability-instance-admin          | Kubernetes
suse-observability-agent-clusterA          | Observability
----
[NOTE]
====
O assunto `+SOURCE` mostra quem é o proprietário e gerencia o assunto. Pode ser uma de nossas fontes RBAC (Configuração|Estática, Observabilidade, Kubernetes)

* Configuração|Static: Sujeitos obtidos por meio da configuração do Suse Observability através do helm
* Observabilidade: Assuntos gerenciados por meio do cli sts
* Kubernetes: Assuntos obtidos por meio do agente RBAC de Kubernetes RoleBindings e ClusterRoleBindings
====

=== Listar assuntos para um usuário

Lista todos os assuntos atribuídos ao contexto; a primeira entrada mostrada é o nome de usuário e as possíveis funções que o usuário tem:

[,text]
----
sts user-session roles

user-dkzwh
keycloakoidc_group://suse-observability-instance-troubleshooter
keycloakoidc_group://suse-observability-instance-observer
keycloakoidc_group://suse-observability-cluster-observer
keycloakoidc_group://suse-observability-instance-admin
keycloakoidc_group://suse-observability-observer
----

Liste todos os assuntos atribuídos a um usuário, fornecendo o endereço `api-token`(útil porque o administrador pode ajudar a depurar usuários com menos privilégios):

[,text]
----
sts user-session roles --api-token [api-token]

u-3p775e7w7l
keycloakoidc_group://suse-observability-cluster-observer
----

=== Mostrar permissões concedidas

Mostrar as permissões concedidas a um usuário ou função específica.

[,text]
----
sts rbac describe-permissions --subject [role-name]

PERMISSION   | RESOURCE
get-topology | cluster-name:rbac-t2
get-traces   | k8s.cluster.name:rbac-t2
get-metrics  | k8s:rbac-t2:__any__
----

[NOTE]
====
`--subject` pode ser qualquer um dos assuntos disponíveis nas diferentes fontes RBAC (Configuração, Observabilidade, Kubernetes)
====

=== Conceder permissões
[NOTE]
====
`--subject` pode ser um assunto de Observability; os assuntos de Configuration e Kubernetes são somente leitura
====

==== Permitir que um usuário abra uma visualização

Dê permissão a um sujeito para abrir uma visualização:

[,text]
----
sts rbac grant --subject [role-name] --permission get-views --resource [view-name]
----

==== Permitir que um usuário crie exibições

Dê a um sujeito com permissão do sistema para criar exibições:

[,text]
----
sts rbac grant --subject [role-name] --permission create-views
----

==== Permitir que um usuário verifique as configurações do SUSE Observability

Dê a um sujeito com permissão do sistema para verificar as configurações do SUSE Observability:

[,text]
----
sts rbac grant --subject [role-name] --permission get-settings
----

=== Revogação de permissões
[NOTE]
====
`--subject` pode ser um assunto de Observability; os assuntos de Configuration e Kubernetes são somente leitura
====

Revogar permissões para que um sujeito abra uma visualização:

[,text]
----
sts rbac revoke --subject [role-name] --permission get-views --resource [view-name]
----
