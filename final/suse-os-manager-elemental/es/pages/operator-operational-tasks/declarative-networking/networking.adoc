== Requisitos previos

* Sigue siendo necesario un servidor DHCP para el registro del primer arranque y el reinicio de las máquinas. Por esta razón, el tiempo de arrendamiento puede mantenerse al mínimo, ya que durante todo el ciclo de vida de la máquina se utilizarán las direcciones IP controladas por IPAM.
* Se instala un Proveedor IPAM de su elección en el clúster de gestión de Rancher.
Por ejemplo, https://github.com/kubernetes-sigs/cluster-api-ipam-provider-in-cluster[InCluster IPAM Provider].
* https://networkmanager.dev[NetworkManager] debe instalarse en las imágenes del sistema operativo y puede configurarse directamente mediante el <<_configurators,configurador de red>> `nmconnections`.
Ya incluido en las imágenes proporcionadas por Elemental.
* (opcionalmente) https://github.com/suse-edge/nm-configurator/releases[] nmc puede utilizarse con `nmc` <<_configurators,configurador de>> red.
* (opcionalmente) https://github.com/nmstate/nmstate/releases[] nmstatectl puede utilizarse con `nmstate` <<_configurators,configurador de>> red.

=== Instalación de nmc o nmstatectl en imágenes de SO

Cuando se utilizan los configuradores `nmc` o `nmstate`, es necesario instalar en la máquina las herramientas https://github.com/suse-edge/nm-configurator/releases[] nmc o https://github.com/nmstate/nmstate/releases[] nmstatectl.

Actualmente esto se puede lograr mediante la xref:operator-operational-tasks/custom-images.adoc#_remastering_an_os_image_with_a_custom_dockerfile[personalización de una imagen Elemental OS] con un comando personalizado:

=== Cómo instalar el proveedor CAPI IPAM

La forma recomendada de instalar cualquier Proveedor CAPI en Rancher es utilizar https://documentation.suse.com/cloudnative/cluster-api/v0.22/en/index.html[Rancher Turtles].

Rancher Turtles permitirá al usuario instalar y gestionar el ciclo de vida de cualquier Proveedor CAPI. Para instalarlo en su sistema, siga la https://documentation.suse.com/cloudnative/cluster-api/v0.22/en/tutorials/quickstart.html[documentación de instalación de].

Una vez instalado Rancher Turtles, la instalación de un IPAM CAPI Provider, por ejemplo el https://github.com/kubernetes-sigs/cluster-api-ipam-provider-in-cluster[InCluster IPAM Provider], se puede realizar aplicando el siguiente recurso:

[,yaml]
----
kind: CAPIProvider
metadata:
  name: in-cluster
  namespace: default
spec:
  name: in-cluster
  type: ipam
  fetchConfig:
    url: "https://github.com/kubernetes-sigs/cluster-api-ipam-provider-in-cluster/releases"
  version: v0.1.0
----

==== Sin tortugas rancheras

Una opción alternativa para instalar un Proveedor CAPI IPAM es aplicar directamente el manifiesto en el cluster Rancher.

Tenga en cuenta que esta solución puede dar lugar a conflictos con los CRD y los recursos aplicados, ya que deben aplicarse y mantenerse manualmente.

. Los CRDs `ipaddresses.ipam.cluster.x-k8s.io` y `ipaddressclaims.ipam.cluster.x-k8s.io` deben estar instalados en el cluster de gestión de Rancher:
+
[,bash]
----
 kubectl apply -f https://raw.githubusercontent.com/kubernetes-sigs/cluster-api/main/config/crd/bases/ipam.cluster.x-k8s.io_ipaddressclaims.yaml
 kubectl apply -f https://raw.githubusercontent.com/kubernetes-sigs/cluster-api/main/config/crd/bases/ipam.cluster.x-k8s.io_ipaddresses.yaml
----

[IMPORTANT]
.info
====
Se espera que, con el tiempo, estos CRD formen parte de Rancher y no requieran instalación manual. Ver: https://github.com/rancher/rancher/issues/46385
====


. Instale https://github.com/kubernetes-sigs/cluster-api-ipam-provider-in-cluster[InCluster IPAM Provider] a partir del manifiesto publicado:
+
[,bash]
----
 kubectl apply -f https://github.com/kubernetes-sigs/cluster-api-ipam-provider-in-cluster/releases/download/v0.1.0/ipam-components.yaml
----

. Una plantilla de configuración de red (en este caso se utiliza el configurador `nmc` ).

[,yaml]
----

----

Aquí podemos observar que se ha definido un `InClusterIPPool`, ya que estamos utilizando el https://github.com/kubernetes-sigs/cluster-api-ipam-provider-in-cluster[InCluster IPAM Provider] para este ejemplo.

A continuación vamos a hacer referencia a este IPPool en la página `MachineRegistration`. La clave para esta referencia es `inventory-ip`, y sólo vamos a necesitar una IP por Máquina registrada. Si su máquina tiene más de una NIC, puede definir más referencias y utilizar también diferentes IPPools, por ejemplo:

[,yaml]
----
ipAddresses:
  main-nic-ip:
    apiGroup: ipam.cluster.x-k8s.io
    kind: InClusterIPPool
    name: elemental-inventory-pool
  secondary-nic-ip:
    apiGroup: ipam.cluster.x-k8s.io
    kind: InClusterIPPool
    name: elemental-inventory-pool
  private-nic-ip:
    apiGroup: ipam.cluster.x-k8s.io
    kind: InClusterIPPool
    name: elemental-private-pool
----

Cada clave de referencia IPPool definida puede utilizarse para la plantilla de configuración de red:

[,yaml]
----
config:
  dns-resolver:
    config:
      server:
      - 192.168.122.1
      search: []
  routes:
    config:
    - destination: 0.0.0.0/0
      next-hop-interface: eth0
      next-hop-address: 192.168.122.1
      metric: 150
      table-id: 254
  interfaces:
    - name: eth0
      type: ethernet
      description: Main-NIC
      state: up
      ipv4:
        enabled: true
        dhcp: false
        address:
        - ip: "{inventory-ip}"
          prefix-length: 24
      ipv6:
        enabled: false
----

El fragmento anterior es casi 1:1 https://github.com/suse-edge/nm-configurator?tab=readme-ov-file#unified-configurations[syntax] nm-configurator, con la única excepción del marcador de posición `{inventory-ip}`. Durante las fases de instalación o reinicio de las máquinas Elementales, el `elemental-operator` reclamará una Dirección IP del Grupo IP referenciado, y sustituirá el marcador de posición `{inventory-ip}` por una Dirección IP real.

=== Direcciones IPA reclamadas

`IPAddressClaim` seguirá todo el ciclo de vida de `MachineInventory`, garantizando que a cada máquina registrada se le asigne una IP única. Cada reclamo lleva el nombre de la página `MachineInventory` que lo utiliza, como `$MachineInventoryName-$IPPoolRefKey`, por ejemplo:

[,yaml]
----
apiVersion: ipam.cluster.x-k8s.io/v1beta1
kind: IPAddressClaim
metadata:
  finalizers:
    - ipam.cluster.x-k8s.io/ReleaseAddress
  name: m-e5331e3b-1e1b-4ce7-b080-235ed9a6d07c-inventory-ip
  namespace: fleet-default
  ownerReferences:
    - apiVersion: elemental.cattle.io/v1beta1
      kind: MachineInventory
      name: m-e5331e3b-1e1b-4ce7-b080-235ed9a6d07c
spec:
  poolRef:
    apiGroup: ipam.cluster.x-k8s.io
    kind: InClusterIPPool
    name: elemental-inventory-pool
status:
  addressRef:
    name: m-e5331e3b-1e1b-4ce7-b080-235ed9a6d07c-inventory-ip
----

Siempre que se borre un `MachineInventory`, se restaurará la configuración de red por defecto (DHCP), borrando cualquier perfil de red en la máquina y reiniciando la pila de red. Finalmente se liberarán las IPs asignadas.

Para obtener más información y detalles sobre cómo solucionar problemas, consulte la xref:troubleshooting/troubleshooting-network.adoc[documentación Solución de problemas de redes declarativas].

=== Configuradores

En la máquina Elemental, `elemental-register` puede configurar el `NetworkManager` de diferentes maneras. El configurador en uso se define en el xref:references/machineregistration-reference.adoc#_config_network[archivo MachineRegistration.spec.network]:

* https://github.com/suse-edge/nm-configurator[nmc]
* https://nmstate.io/[nmstate]
* https://networkmanager.pages.freedesktop.org/NetworkManager/NetworkManager/nm-settings-keyfile.html[nmconexiones]
