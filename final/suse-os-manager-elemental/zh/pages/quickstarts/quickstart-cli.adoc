== 准备好 kubernetes 资源

节点部署从`MachineRegistration` 开始，确定一组共享相同配置（磁盘驱动器、网络等）的机器。

在目标主机上部署{elemental-product-name} 操作系统需要`MachineRegistration` 。启动时，每台主机都要向{elemental-product-name} Operator 注册，后者通过`MachineInventory` 资源跟踪新主机。

然后继续使用集群资源，该资源使用`MachineInventorySelectorTemplate` 来了解哪些机器属于该集群。

该选择器是一个简单的匹配器，基于`MachineInventory` 中设置的标签，因此，如果您的选择器匹配的是带有`cluster-id-val`值的标签`cluster-id` ，而您的`MachineInventory` 具有相同的`cluster-id`:``cluster-id-val`` 标签，那么它将匹配并被引导为群集的一部分。

在本快速入门中，我们将部署资源，以配置一个名为_volcano_的群集，该群集将匹配``MachineInventorys``上的标签_element___:fire__。

== 准备安装（种子）映像

这是最后一步：您需要一个包含初始注册配置的{elemental-product-name} 种子映像，这样它就可以作为集群的一部分进行自动注册、安装和全面部署。

[NOTE]
.备注
====
初始注册配置文件会在创建`Machine Registration`.NET 文件时生成。

您可以通过以下方式下载

[,shell]
----
wget --no-check-certificate `kubectl get machineregistration -n fleet-default fire-nodes -o jsonpath="{.status.registrationURL}"` -O initial-registration.yaml
----
====

注册配置文件的内容无非是节点需要注册的注册 URL、适当的服务器证书以及注册过程中的一些选项。

种子映像生成后，可用于配置任意数量的机器。

[tabs]
========

可通过以下脚本将上述`SeedImage` 资源创建的种子映像下载为 ISO：

kubectl wait --for=condition=ready pod -n fleet-default fire-img

第一条命令等待 ISO 生成并准备就绪，第二条命令将其下载到当前目录，名称为`elemental-x86_64.iso` 。

手动准备种子映像 (aarch64)::
+
======
{elemental-product-name}对 Raspberry Pi 的支持目前主要用于演示目的。因此，安装过程与 x86-64 类似。从种子映像（本例中为 U 盘）启动，然后安装到存储介质（Raspberry Pi 为 SD 卡）。

*检索预制种子图像*

[,shell]
----
wget -q https://download.opensuse.org/repositories/isv:/Rancher:/SUSE® Rancher Prime OS Manager:/Stable/containers/rpi.raw
----

_验证下载_

为了验证下载工件的完整性，应该进行校验和验证：

[,shell]
----
wget -q https://download.opensuse.org/repositories/isv:/Rancher:/SUSE® Rancher Prime OS Manager:/Stable/containers/rpi.raw.sha256 sha256sum -c rpi.raw.sha256
----

输出结果应为`rpi.raw: OK` 。

*注入注册信息*

添加`initial-registration.yaml` 还没有脚本。这仍然是一个手动过程：

写入的 U 盘将有两个分区。`RPI_BOOT` 包含引导加载程序文件，`COS_LIVE` 包含{elemental-product-name} 文件。挂载`COS_LIVE` 分区，并将`initial-registration.yaml` 作为`livecd-cloud-config.yaml` 写入该分区。

如果你已经用文件管理器挂载了 U 盘，这条命令应该可以复制注册信息：

[,shell]
----
sudo cp initial-registration.yaml /run/media/$USER/COS_LIVE/livecd-cloud-config.yaml
----

如果您喜欢使用 CLI 工具：

[,shell]
----
IMAGE=rpi.raw 
DEST=$(mktemp -d) 

SECTORSIZE=$(sfdisk -J $\{IMAGE} | jq '.partitiontable.sectorsize') 
DATAPARTITIONSTART=$(sfdisk -J $\{IMAGE} | jq '.partitiontable.partitions[1].start') 
sudo mount -o rw,loop,offset=$(($\{SECTORSIZE}*$\{DATAPARTITIONSTART})) $\{IMAGE} $\{DEST} 
sudo cp initial-registration.yaml $\{DEST}/livecd-cloud-config.yaml 
sudo umount $\{DEST} 
rmdir $\{DEST}
----

*将种子映像写入 U 盘*

`.raw` 映像需要写入 U 盘，以便启动。如果您能熟练使用`dd` 命令，可以在 Linux 命令行上执行此操作。[openSUSE](https://www.opensuse.org)有关于如何将映像写入[Linux](https://en.opensuse.org/SDB:Live_USB_stick)、[Windows](https://en.opensuse.org/SDB:Create_a_Live_USB_stick_using_Windows)和[OS X](https://en.opensuse.org/SDB:Create_a_Live_USB_stick_using_macOS)存储介质的详细说明。

*启动 Raspberry Pi*

现在卸载 U 盘并将其插入 Raspberry Pi。

将大容量（32 GB 或以上）、**快速**（！！）的微型 SD 卡插入相应的插槽。

将系统连接到以太网。

电源循环将重启 Pi。其他一切都与 x86-64 相同。

[CAUTION]
====
确保微型 SD 卡未分区。否则，Pi 引导加载程序将尝试从它启动，但会失败。
====

现在，您可以使用该镜像启动您的节点，它们就会启动：

* 使用给定的注册 URL 注册，并为每台机器创建一个 `MachineInventory`
* 将 SLE Micro 安装到指定设备
* 重新启动
======

=== 选择合适的机器加入群集

`MachineInventorySelectorTemplate` 从带有_element:fire_标签的``MachineInventorys``中选择配置群集所需的机器。
我们已在`MachineRegistration` `machineInventoryLabels` 地图中添加了_element___:fire__标签，因此源自该地图的所有``MachineInventory``都已拥有该标签。
总之，我们可以跳过`MachineRegistration` 中的标签，稍后再添加：

[,shell]
----
kubectl -n fleet-default label machineinventory $(kubectl get machineinventory -n fleet-default --no-headers -o custom-columns=":metadata.name") element=fire
----

一旦出现带有__fire__ _元素_的``MachineInventorys 机器``，相应的机器就会通过所选提供商（k3s/rke）自动部署集群。

几分钟后，您的新群集将完全配置完毕

== 如何为群集选择 kubernetes 版本和部署程序？

在`Spec` 中的 cluster.yaml 文件中有一个名为`kubernetesVersion` 的键。这将设置群集将使用的版本和部署器，例如，用于 rke2 的 Kubernetes+v1``.24.8++`` 将是 `v1.24.8``+`` +rke2r1`+，用于 k3s 的是 +v1`.24.8+k3s1`。

要查看所有兼容版本，请查看https://www.suse.com/suse-rancher/support-matrix/all-supported-versions/[支持矩阵] RancherPDF，了解 rke/rke2/k3s 版本及其组件。

您还可以查看我们的版本xref:references/kubernetesversions.adoc[文档]，了解如何获取这些版本。

有关`Cluster` 资源的更多信息，请查看我们的xref:references/cluster-reference.adoc[集群规格]页面。

== 如何了解幕后的情况？

您应该可以通过它了解机器在做什么：

* 在 ISO 启动期间：
 ** ssh 进入机器（用户/密码：root/ros）：
  *** 运行`journalctl -f -t elemental` 会显示注册_(elemental-register_) 和安装{elemental-product-name} _(elemental install_) 的进度。
* 系统安装完毕后：
 ** 在 Rancher UI \->`Cluster Management` 上可以查看新群集和群集详细信息中的`Provisioning Log` 
 ** ssh 进入机器（用户/密码：：`Spec.config.cloud-config.users` 下的 register.yaml 中配置的内容：）
  *** 运行`journalctl -f -u elemental-system-agent` 显示了初始元素配置和安装 `rancher-system-agent`
  *** 运行`journalctl -f -u rancher-system-agent` 显示了 k3s 等集群组件的自举输出结果
  *** 运行`journalctl -f -u k3s` 显示 k3s 部署日志

== 可选：通过 CLI 安装元素用户界面扩展

创建并应用新的 ClusterRepo 以添加官方 Rancher 扩展库。

[source,yaml]
----
apiVersion: catalog.cattle.io/v1
kind: ClusterRepo
metadata:
  name: rancher-ui-charts
spec:
  gitBranch: main
  gitRepo: https://github.com/rancher/ui-plugin-charts
----

为 Rancher UI 扩展图表添加 helm repo。

[source,shell]
----
helm repo add rancher-ui-plugins https://raw.githubusercontent.com/rancher/ui-plugin-charts/main/
----

安装 Elemental UI 扩展。

[source,shell]
----
helm install elemental rancher-ui-plugins/elemental -n cattle-ui-plugin-system
----
