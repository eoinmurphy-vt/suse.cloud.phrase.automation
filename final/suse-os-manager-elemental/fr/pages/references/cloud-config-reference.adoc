= Référence Cloud-config
:revdate: 2025-10-21
:page-revdate: {revdate}

Les images Node OS construites à l'aide de https://github.com/elemental-toolkit[{elemental-product-name} Toolkit] sont censées être initialisées et configurées à l'aide de https://github.com/rancher/yip[] yip. Yip est un petit utilitaire permettant d'appliquer un ensemble d'actions et de configurations au système décrit par des fichiers yaml. Yip est intégré et consommé comme une bibliothèque dans le binaire du client élémentaire (voir `elemental run-stage --help`). Yip regroupe les configurations et les actions à appliquer dans des _étapes_ arbitraires, par exemple l'appel de commande `elemental run-stage network` n'appliquerait que les actions et les configurations définies pour l'étape nommée `network`. Du point de vue de Yip, les étapes peuvent être exécutées à tout moment puisqu'elles regroupent simplement un ensemble d'actions sous un nom arbitraire.

{elemental-product-name} Toolkit intègre cinq étapes prédéfinies dans le processus de démarrage du système d'exploitation.

. *`pre-rootfs`*: Cette étape s'exécute au début du démarrage sur le disque init ram, juste avant de monter le périphérique racine (généralement à l'adresse `/sysroot`). Cette étape peut être utilisée pour définir les étapes du premier démarrage qui sont nécessaires avant de monter le périphérique racine. Un bon exemple est l'extension de la partition du périphérique racine. Exécuté dans le cadre du site `initrd-root-device.target`.
. *`rootfs`*: Cette étape s'exécute au début du démarrage sur le disque init ram, juste après le montage du périphérique racine (généralement à l'adresse `/sysroot`). Cette étape peut être utilisée pour définir les étapes du premier démarrage, comme la création de nouvelles partitions. Les chemins éphémères et persistants sont généralement définis à ce stade. Exécuté dans le cadre du site `initrd-root-fs.target`.
. *`initramfs`*: Cette étape se déroule également à l'intérieur du disque ram init, mais à un stade ultérieur, juste avant le passage à la racine. Cette étape s'exécute dans un environnement chrooté vers le dispositif racine réel. Cette étape est pratique pour définir certains paramètres du système qui pourraient être utiles à systemd après le passage à l'utilisateur root. Par exemple, des fichiers unitaires systemd supplémentaires peuvent être ajoutés ici avant que le systemd de la racine ne soit exécuté. Exécuté dans le cadre du site `initrd.target`.
. *`fs`*: Cette étape est la première à être exécutée après le passage à l'utilisateur root et elle est exécutée dans le cadre de `sysinit.target` qui s'exécute une fois que tous les systèmes de fichiers locaux et les points de montage définis dans fstab sont montés et prêts à l'emploi.
. *`network`*: Cette étape est exécutée dans le cadre du site `multi-user.target` et après que le site `network-online.target` a été atteint. Cette étape peut être utilisée pour exécuter des actions et des configurations qui nécessitent une connectivité réseau. Par exemple, cette étape est utilisée pour exécuter le tout premier enregistrement de nœud et l'installation à partir d'un ISO vivant.
. *`boot`*: Cette étape est exécutée dans le cadre de la cible multi-utilisateur et avant la cible getty. Il s'agit de l'étape par défaut pour exécuter les données de configuration du nuage fournies à l'aide de la syntaxe "cloud-init" prise en charge. Voir la section sur xref:references/cloud-config-reference.adoc#_compatibility_with_cloud_init_format[la compatibilité avec cloud-init].

Par défaut, `elemental` lit les fichiers de configuration yaml à partir des chemins suivants dans l'ordre : `/system/oem`, `/oem` et `/usr/local/cloud-config`.

Dans {elemental-product-name} Operator, toutes les ressources kubernetes comprenant un champ `cloud-config` peuvent être exprimées dans une syntaxe <<_compatibility_with_cloud_init_format,compatible avec>> <<_configuration_syntax,yip>> ou <<_compatibility_with_cloud_init_format,cloud-init>>. Il s'agit de ressources telles que `MachineRegistration`, `SeedImage`, et `ManagedOSImage`.

[NOTE]
====
Contrairement à des projets similaires tels que _Cloud Init_, Yip ne conserve pas d'enregistrements ou de caches des étapes exécutées, toutes les étapes et leur configuration associée sont exécutées à chaque démarrage.
====

== Crochets de configuration du nuage du client élémentaire

En plus des étapes de configuration du nuage au démarrage décrites dans la section précédente, le client Elemental honore également certaines étapes spécifiques, appelées "hooks", pour personnaliser le comportement de ces sous-commandes : `install`, `upgrade`, `reset` et `build-disk`. Chacune de ces sous-commandes possède son propre ensemble de quatre étapes différentes de configuration du nuage exécutées à des phases analogues de l'exécution de la sous-commande spécifique.

Les crochets sont essentiellement un moyen d'apporter des modifications permanentes au système qui ne peuvent pas être facilement exprimées dans le cadre d'un conteneur OCI ou qui ne sont pas facilement réalisables avec les options de configuration du client `elemental`. Un bon exemple pourrait être la gestion du micrologiciel dans la partition EFI pour les appareils Raspberry Pi.

[WARNING]
====
Notez que la plupart des hooks sont exécutés dans l'environnement hôte avec des privilèges, et qu'il s'agit donc d'opérations potentiellement destructrices. Dans la plupart des cas, des opérations régulières de configuration du nuage au moment du démarrage suffisent à configurer le système. De même, pour inclure des logiciels supplémentaires dans une image, il est préférable de construire une xref:operator-operational-tasks/custom-images.adoc[image dérivée] et de ne pas abuser des crochets pour installer des logiciels supplémentaires.
====

=== Stades d'accrochage

Étapes préalables : avant l'installation, avant la mise à niveau, avant la réinitialisation, avant le disque Ces étapes sont exécutées une fois que les répertoires et l'environnement de travail sont préparés, mais avant le début de l'action proprement dite. Dans les étapes d'installation, de mise à niveau et de réinitialisation, cela se produit une fois que toutes les partitions associées sont créées et montées, mais avant d'indiquer le déploiement d'une image.

Ces étapes sont exécutées après le déploiement du système cible dans la zone de travail dans un environnement chroot ancré dans l'image déployée. Comme cela se produit dans un chroot env, le client élémentaire analyse les crochets présents dans l'image déployée, et non dans l'hôte. Seul /oem est partagé avec l'hôte s'il est disponible.

Après les étapes : après l'installation, après la mise à niveau, après la réinitialisation, après le disque Ces étapes sont exécutées après le déploiement du système cible dans la zone de travail depuis l'environnement hôte. À ce stade, toutes les partitions sont encore montées et disponibles en mode RW. Cet ensemble particulier d'étapes analyse les crochets présents dans l'hôte et dans l'ensemble équivalent de chemins chrootés vers l'image déployée. Le crochet n'est cependant pas exécuté dans un environnement chroot. Ceci est utile pour fournir des crochets `after-*` dans l'image déployée.

Post-étapes : post-install, post-upgrade, post-reset, post-disk Ces étapes sont exécutées à la fin avant de quitter la commande et d'exécuter un processus de nettoyage. À ce stade, l'image est déjà déployée et verrouillée dans un sous-volume ou un système de fichiers en lecture seule. Les partitions sont encore montées à ce stade.

[NOTE]
====
Remarque : les crochets d'installation ne sont pas appliqués dans le cadre de la procédure d'appel d'offres. xref:references/machineregistration-reference.adoc#_config_cloud_config[`MachineRegistration.config.cloud-config`]. Afin de fournir des crochets d'installation, ils peuvent être inclus dans le fichier d'installation. xref:[`SeedImage.cloud-config`]car ils doivent être présents dans le média d'installation.
====

== Syntaxe de configuration

Yip a sa propre syntaxe, il faut essentiellement définir des _étapes_ et une liste d'étapes pour chaque étape. Les étapes sont exécutées dans l'ordre et chaque étape peut être une combinaison de différents types d'actions (par exemple, exécuter des commandes, créer des fichiers, définir un nom d'hôte, etc.)

Prenons l'exemple suivant :

[,yaml]
----
stages:
  initramfs:
  - name: "Setup users"
    ensure_entities:
    - path: /etc/shadow
      entity: |
          kind: "shadow"
          username: "root"
          password: "root"
  boot:
  - files:
    - path: /tmp/script.sh
      content: |
        #!/bin/sh
        echo "test"
      permissions: 0777
      owner: 1000
      group: 100
  - commands:
    - /tmp/script.sh
----

Dans l'exemple ci-dessus, il y a deux étapes : `initramfs` et `boot`. L'étape `initramfs` permet d'initialiser un exemple d'utilisateur. L'étape `boot` comprend deux étapes, l'une pour créer un fichier de script exécutable et l'autre pour exécuter le script.

Yip prend également en charge les modificateurs de suffixe `*.before` et `*.after` pour toute étape donnée. Par exemple, l'exécution de l'étape `network` entraîne l'exécution des étapes `network.before` trouvées dans les fichiers de configuration, puis `network` et enfin `network.after`.

Voir la référence complète des clés applicables dans les étapes documentées dans lehttps://github.com/rancher/yip?tab=readme-ov-file#configuration-reference[projet] yip lui-même.

== Synchronization

Pendant l'exécution d'une étape, tous les fichiers sont chargés et un graphique est calculé puis exécuté. Cela signifie que les étapes d'une même phase peuvent être exécutées dans n'importe quel ordre et quel que soit le nom du fichier.

Pour synchroniser l'exécution, l'utilisateur peut utiliser les étapes post/pre ou se référer au mot-clé `after` dans la https://github.com/rancher/yip/blob/ee4051d9ec2782989344f813bc6a0975bbd8f3fe/pkg/executor/default_test.go#L222[syntaxe] yip.

== Compatibilité avec le format Cloud Init

Un sous-ensemble de la officielle http://cloudinit.readthedocs.org/en/latest/topics/format.html#cloud-config-data[spécification] est implémenté par yip. Plus précisément, les clés "cloud-init" prises en charge sont les suivantes : `users` `ssh_authorized_keys` , `runcmd`, `hostname` et `write_files`.

Si un fichier yaml commence par `#cloud-config`, il est analysé comme un cloud-init standard, associé à l'étape yip `boot`.

Par exemple :

[,yaml]
----
#cloud-config
# Note groups are delivered as list, not as comma separated values
users:
- name: "bar"
  passwd: "foo"
  groups:
  - "users"
  homedir: "/home/foo"
  shell: "/bin/bash"
  ssh_authorized_keys:
  - faaapploo
# Assigns these keys to the first user in users or root if there
# is none
ssh_authorized_keys:
- asddadfafefa
# Run these commands once the system has fully booted
# Each command is expressed as a sinlge string, no nested lists
runcmd:
- echo hello world
# Hostname
hostname: myserver
# Write arbitrary files
write_files:
- encoding: b64
  content: CiMgVGhpcyBmaWxlIGNvbnRyb2xzIHRoZSBzdGF0ZSBvZiBTRUxpbnV4
  path: /foo/bar
  permissions: "0644"
  owner: "bar"
----
