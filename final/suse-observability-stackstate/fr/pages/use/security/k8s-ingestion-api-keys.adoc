= Clés API (obsolète)
:revdate: 2025-07-22
:page-revdate: {revdate}
:description: SUSE Observability

Une clé API a été utilisée pour s'authentifier auprès de {stackstate-product-name} afin d'envoyer des données télémétriques à {stackstate-product-name}. Dans le cas d'une installation de l'agent Helm, cela se fait en définissant la variable `stackstate.apiKey` helm.

Les clés API et les clés API d'ingestion sont obsolètes et seront supprimées au quatrième trimestre 2026. Les jetons de service sont désormais le mécanisme d'authentification privilégié, y compris pour l'envoi de données télémétriques. 

Le flux d'ajout de nouveaux clusters à {stackstate-product-name} dans le xref:/k8s-suse-rancher-prime.adoc#_installing_the_suse_observability_agent[flux d'installation de StackPack] inclut la création d'un jeton spécifique au cluster qui est compatible avec l'agent RBAC pour Rancher RBAC.

== Migration d'une clé API vers un jeton de service

Il existe deux options pour passer de l'utilisation d'une clé API unique à l'utilisation de jetons de service pour les données de télémétrie. 

1. Migrer vers un jeton de service par cluster. C'est un peu plus compliqué à mettre en place, mais cela prend en charge le RBAC de Rancher.
2. Migrez vers un jeton de service unique que vous utilisez à partir de tous les clusters Kubernetes et collecteurs OTel. Cela ne prend pas en charge le système RBAC de Rancher.

Après avoir migré tous les agents et les collecteurs Open Telemetry vers des jetons de service, vous pouvez maintenant supprimer `global.receiverApiKey` ou `stackstate.apiKey` des valeurs Helm pour installer {stackstate-product-name} et exécuter `helm upgrade` pour appliquer les changements.

=== Migrer vers un jeton de service par cluster

1. Ouvrir le Stackpack Kubernetes dans l'interface utilisateur
2. Développez le cluster que vous migrez
3. Cliquez sur le bouton "Create new service token" pour obtenir le jeton de service spécifique au cluster et copiez-le.
4. Suivez les instructions d'installation/mise à niveau de l'agent dans le stackpack pour redéployer l'agent sur le cluster avec le jeton de service comme clé API.

Au lieu d'utiliser l'interface utilisateur, vous pouvez également générer le jeton à l'aide de l'interface CLI, comme xref:setup/security/authentication/service_tokens#_telemetry_data_ingestion[indiqué ici.] Veillez également à affecter le site `update-scoped-permissions` au cluster que vous migrez. Les autres étapes sont les mêmes que pour l'interface utilisateur.

=== Migrer vers un jeton de service unique

Générer un jeton de service avec le CLI comme xref:setup/security/authentication/service_tokens#_telemetry_data_ingestion[documenté ici]\... :

[,sh]
----
> sts rbac create-subject --subject my-agent
✅ Created subject 'my-agent'
> sts rbac grant --subject my-agent --permission update-metrics
✅ Granted permission 'update-metrics' on 'system' to subject 'my-agent'
PERMISSION   | RESOURCE
update-metrics | system
----

N'ajoutez pas d'autorisation `update-scoped-permissions` car elle ne fonctionnera pas pour un jeton utilisé à partir de plusieurs clusters.

Redéployez vos agents et vos collecteurs Open Telemetry en suivant les instructions disponibles dans l'interface utilisateur du StackPack Kubernetes et dans la xref:setup/otel/getting-started/getting-started-k8s.adoc[documentation du collecteur Open Telemetry]. Utilisez le jeton de service généré comme clé API en procédant comme suit.

== Migrer des clés d'API d'ingestion

Les clés API d'ingestion ont été utilisées par des outils externes pour ingérer des données (telles que des mesures, des événements et des traces) dans le cluster SUSE Observability.
Ces outils comprennent l'agent STS et/ou le collecteur OTel. Les jetons de service ont les mêmes capacités, voire plus. Suivez les mêmes étapes que pour la clé API (dans la section précédente) pour migrer vers <<_migrate_to_a_service_token_per_cluster, un jeton de service par cluster>> ou <<_migrate_to_a_single_service_token, un jeton de service unique>>.
