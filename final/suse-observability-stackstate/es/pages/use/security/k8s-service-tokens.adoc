= Fichas de servicio
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: Observabilidad de SUSE

== Visión general

Utilizando tokens de servicio es posible autenticarse en SUSE Observability sin tener una cuenta de usuario asociada. Esto es útil para situaciones en las que se desea utilizar SUSE Observability desde servicios headless como un servidor CI. En este caso, normalmente no se desea aprovisionar una cuenta de usuario en el proveedor de identidades.

== Gestión de fichas de servicio

Los tokens de servicio pueden gestionarse a través de la xref:/setup/cli/cli-sts.adoc[CLI`sts` ]. Están disponibles los siguientes comandos:

[,sh]
----
> sts service-token --help
Manage service tokens.

Usage:
  sts service-token [command]

Available Commands:
  create      Create a service token
  delete      Delete a service token
  list        List service tokens

Use "sts service-token [command] --help" for more information about a command.
----

También es posible <<_set_up_a_bootstrap_service_token,configurar un token de servicio de arranque>> al instalar {stackstate-product-name}.

=== Crear fichas de servicio

[NOTE]
====
Los paquetes Open Telemetry y Kubernetes StackPacks le permiten crear un token de sujeto y servicio con todos los permisos necesarios configurados.
====

Para crear un token de servicio para {stackstate-product-name}, puede utilizar la CLI `sts`.

[,sh]
----
sts service-token create
----

[NOTE]
====
Tenga en cuenta que el token de servicio sólo se mostrará una vez. No es posible volver a ver la ficha.
====


Este comando recibe los siguientes argumentos en la línea de comandos:

|===
| Bandera | Descripción

| `--name`
| Nombre del token de servicio

| `--expiration`
| La fecha de caducidad del token de servicio, el formato es aaaa-MM-dd. La expiración es opcional.

| `--roles`
| Una lista de roles separados por comas para asignar al token de servicio
|===

Por ejemplo, el siguiente comando creará un token de servicio con el nombre `my-service-token` y el rol `stackstate-k8s-troubleshooter`:

[,sh]
----
> sts service-token create --name my-service-token --roles stackstate-k8s-troubleshooter
✅ Service token created: svctok-aaaaa-bbbb-ccccc-ddddd
----

=== Configurar un token de servicio de arranque

Al instalar {stackstate-product-name}, es posible arrancarlo con un token de servicio (temporal). Esto permite utilizar la CLI sin tener que interactuar primero con {stackstate-product-name} y obtener un token de API desde la interfaz de usuario. Para configurarlo, añada el siguiente fragmento al archivo de configuración {stackstate-product-name}:

Para configurar {stackstate-product-name} para crear un token de servicio bootstrap en Kubernetes, es necesario añadir los siguientes valores al archivo `authentication.yaml`. Por ejemplo:

[,yaml]
----
stackstate:
  authentication:
    serviceToken:
      bootstrap:
        token: <token>
        roles:
          - stackstate-admin
        ttl: 24h
----

Siga los siguientes pasos para configurar {stackstate-product-name} para crear un token de servicio de arranque:

. En `authentication.yaml` - añadir el token bootstrap:
 ** *token* - El token que se crea al inicio (inicial) de {stackstate-product-name}.
 ** *roles* - Una matriz de roles que se asignan al token bootstrap.
 ** *ttl* - Opcional. El tiempo de vida del token de servicio, expresado como una cadena de duración.
. Guarde el archivo `authentication.yaml` junto con el `values.yaml` de las instrucciones de instalación {stackstate-product-name}.
. Ejecute una actualización de Helm para aplicar los cambios.
+
[,text]
----
 helm upgrade \
   --install \
   --namespace suse-observability \
   --values values.yaml \
   --values authentication.yaml \
 suse-observability \
 suse-observability/suse-observability
----

[NOTE]
====
* La primera ejecución del comando helm upgrade provocará el reinicio de los pods, lo que puede causar una breve interrupción de la disponibilidad.
* Incluya `authentication.yaml` en cada ejecución de `helm upgrade`.
* La configuración de autenticación se almacena como un secreto de Kubernetes.
====


==== Configurar el token de servicio de arranque desde un secreto externo

En lugar de especificar el token de arranque a través de los valores de Helm, también puede leer el token de arranque desde un secreto que se crea y gestiona fuera del gráfico de Helm (un secreto externo). Para utilizar un token boostrap de un externo, siga xref:/setup/security/external-secrets.adoc#_getting_authentication_data_from_an_external_secret[estos pasos] y añada los siguientes datos:

[,yaml]
----
kind: Secret
metadata:
   name: "<custom-secret-name>"
type: Opaque
data:
  bootstrap_token: <base64 of token>
----

Este token puede añadirse al secreto junto a los datos que ya están allí.

=== Lista de fichas de servicio

El ID, el nombre, la fecha de caducidad y los roles de todos los tokens de servicio creados pueden verse utilizando la CLI `sts`. Por ejemplo:

[,sh]
----
> sts service-token list
ID              | NAME             | EXPIRATION | ROLES
107484341630693 | my-service-token |            | [stackstate-admin]
----

=== Eliminar fichas de servicio

Un token de servicio puede eliminarse mediante la CLI `sts`. Pasa el ID del token de servicio como argumento. Por ejemplo:

[,sh]
----
> sts service-token delete 107484341630693
✅ Service token deleted: 107484341630693
----

== Utilizar fichas de servicio

Los tokens de servicio se utilizan para autenticarse al utilizar las API de SUSE Observability:* Al utilizar la CLI * Al utilizar directamente las API HTTP * Al introducir datos de telemetría

=== SUSE Observability `sts` CLI

Se utiliza un token de servicio para la autenticación con la CLI `sts`. Para más información, consulte xref:/setup/cli/cli-sts.adoc#_authentication[la documentación de la CLI].

=== API de SUSE Observability

Para utilizar un token de servicio para hablar directamente con la API de SUSE Observability, añádalo a la cabecera de la solicitud de una de las siguientes formas:

* En la cabecera `Authorization`:
+
[,sh]
----
  > curl -X GET -H "Authorization: ApiKey <TOKEN>" http://localhost:8080/api/server/status
----

* En la cabecera `X-API-Key`:
+
[,sh]
----
  > curl -X GET -H "X-API-Key: <TOKEN>" http://localhost:8080/api/server/status
----

=== Introducción de datos telemétricos

Para crear un token de servicio para la ingesta de datos de telemetría, primero debe crear un rol:
[,sh]
----
> sts rbac create-subject --subject my-agent
✅ Created subject 'my-agent'
> sts rbac grant --subject my-agent --permission update-metrics
✅ Granted permission 'update-metrics' on 'system' to subject 'my-agent'
PERMISSION   | RESOURCE
update-metrics | system
----

Si desea utilizar el token de servicio para ingerir datos RBAC de Kubernetes, también debe conceder `update-scoped-permissions`:
[,sh]
----
> sts rbac grant --subject my-agent --permission update-scoped-permissions --resource my-cluster
✅ Granted permission 'update-scoped-permissions' on 'my-cluster' to subject 'my-agent'
PERMISSION                | RESOURCE
update-metrics            | system
update-scoped-permissions | my-cluster
----

Esto creará un nuevo rol en SUSE Observability llamado `my-agent` y le otorgará el permiso `update-metrics`. A continuación, puede crear un ServiceToken para esta función:

[,sh]
----
> sts service-token create --name my-agent --roles my-agent
✅ Service token created: svctok-XXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXX
----

El token de servicio recién creado se puede utilizar como la clave api para la autenticación de los datos de telemetría de:

* {stackstate-product-name} xref:k8s-suse-rancher-prime.adoc#_installing_the_suse_observability_agent[Agente (incluido el agente RBAC)]
* xref:setup/otel/getting-started/getting-started-k8s.adoc[Colector OTel]
