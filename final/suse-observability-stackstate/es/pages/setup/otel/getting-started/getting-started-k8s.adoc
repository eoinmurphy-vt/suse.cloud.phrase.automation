= Primeros pasos con Open Telemetry en Kubernetes
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: {stackstate-product-name}

Esta guía ofrece instrucciones para supervisar una aplicación.

* La aplicación/carga de trabajo supervisada que se ejecuta en el clúster A.
* El colector de telemetría abierta que se ejecuta cerca de la(s) aplicación(es) observada(s), es decir, en el clúster A, y envía los datos a {stackstate-product-name}.
* {stackstate-product-name} ejecutándose en el clúster B, o SUSE Cloud Observability.

image::otel/open-telemetry-collector-kubernetes.png[Instrumentación de contenedores con Open Telemetry a través de un colector que se ejecuta como despliegue de Kubernetes]

== El colector de telemetría abierto

[NOTE]
====
Para una configuración de producción, se recomienda encarecidamente instalar el recopilador, ya que permite que su servicio descargue datos rápidamente y el recopilador puede encargarse de tareas adicionales como reintentos, procesamiento por lotes, cifrado o incluso filtrado de datos confidenciales.
====

Instala el colector OTel (Open Telemetry) en el cluster A y configúralo para:

* Recibir datos de, potencialmente muchas, aplicaciones instrumentadas.
* Enriquezca los datos recopilados con atributos de Kubernetes.
* Generar métricas para las trazas.
* Envía los datos a {stackstate-product-name}, incluida la autenticación mediante la clave API.

NOTE: {stackstate-product-name} también reintenta el envío de datos cuando hay problemas de conexión.

=== Crear un token de servicio

Hay dos formas de crear un token de servicio:

* **{stackstate-product-name} UI** - abra el menú principal haciendo clic en la parte superior izquierda de la pantalla y vaya a `StackPacks` > `Open Telemetry`.  Si no lo ha hecho antes, pulse el botón `INSTALL`. Haga clic en el botón `CREATE NEW SERVICE TOKEN` y copie el valor en el portapapeles.
* **{stackstate-product-name} CLI** - véase xref:/use/security/k8s-service-tokens.adoc#_manage_service_tokens[Gestión de tokens de servicio]

El valor del testigo de servicio debe utilizarse cuando en las instrucciones siguientes se mencione `<SERVICE_TOKEN>`.

=== Crear el espacio de nombres y un secreto para el token de servicio

Instalar en el espacio de nombres `open-telemetry` y utilizar el token de servicio:

[,bash]
----
kubectl create namespace open-telemetry
kubectl create secret generic open-telemetry-collector \
    --namespace open-telemetry \
    --from-literal=API_KEY='<SERVICE_TOKEN>'
----

=== Configurar e instalar el colector

Instalamos el colector con una carta Helm proporcionada por el proyecto Open Telemetry. Asegúrese de que tiene configurado el repositorio de cartas de timón de Open Telemetry:

[,bash]
----
helm repo add open-telemetry https://open-telemetry.github.io/opentelemetry-helm-charts
----

Cree un archivo de valores `otel-collector.yaml` para el gráfico Helm. Este es un buen punto de partida para su uso con {stackstate-product-name}, sustituya `<otlp-suse-observability-endpoint:port>` por su punto final xref:/setup/otel/otlp-apis.adoc[OTLP] (consulte la xref:/setup/otel/otlp-apis.adoc[API OTLP] para su punto final) e inserte el nombre de su clúster Kubernetes en lugar de `<your-cluster-name>`:

.otel-collector.yaml
[,yaml]
----
# Set the API key from the secret as an env var:
extraEnvsFrom:
  - secretRef:
      name: open-telemetry-collector
mode: deployment
image:
  # Use the collector container image that has all components important for k8s. In case of missing components the ghcr.io/open-telemetry/opentelemetry-collector-releases/opentelemetry-collector-contrib image can be used which
  # has all components in the contrib repository: https://github.com/open-telemetry/opentelemetry-collector-contrib
  repository: "ghcr.io/open-telemetry/opentelemetry-collector-releases/opentelemetry-collector-k8s"
ports:
  metrics:
    enabled: true
presets:
  kubernetesAttributes:
    enabled: true
    extractAllPodLabels: true
# This is the config file for the collector:
config:
  receivers:
    otlp:
      protocols:
        grpc:
          endpoint: 0.0.0.0:4317
        http:
          endpoint: 0.0.0.0:4318
    # Scrape the collectors own metrics
    prometheus:
      config:
        scrape_configs:
        - job_name: opentelemetry-collector
          scrape_interval: 10s
          static_configs:
          - targets:
            - ${env:MY_POD_IP}:8888
  extensions:
    # Use the API key from the env for authentication
    bearertokenauth:
      scheme: SUSEObservability
      token: "${env:API_KEY}"
  exporters:
    nop: {}
    otlp/suse-observability:
      auth:
        authenticator: bearertokenauth
      # Put in your own otlp endpoint, for example otlp-suse-observability.my.company.com:443
      endpoint: <otlp-suse-observability-endpoint:port>
      compression: snappy
  processors:
    memory_limiter:
      check_interval: 5s
      limit_percentage: 80
      spike_limit_percentage: 25
    batch: {}
    resource:
      attributes:
      - key: k8s.cluster.name
        action: upsert
        # Insert your own cluster name
        value: <your-cluster-name>
      - key: service.instance.id
        from_attribute: k8s.pod.uid
        action: insert
        # Use the k8s namespace also as the open telemetry namespace
      - key: service.namespace
        from_attribute: k8s.namespace.name
        action: insert
  connectors:
    # Generate metrics for spans
    spanmetrics:
      metrics_expiration: 5m
      namespace: otel_span
  service:
    extensions: [ health_check,  bearertokenauth ]
    pipelines:
      traces:
        receivers: [otlp]
        processors: [memory_limiter, resource, batch]
        exporters: [debug, spanmetrics, otlp/suse-observability]
      metrics:
        receivers: [otlp, spanmetrics, prometheus]
        processors: [memory_limiter, resource, batch]
        exporters: [debug, otlp/suse-observability]
      logs:
        receivers: [otlp]
        processors: []
        exporters: [nop]
    telemetry:
      metrics:
        address: ${env:MY_POD_IP}:8888
----


[CAUTION]
====
*Utilice el mismo nombre de clúster que utilizó para instalar el agente {stackstate-product-name} * si también utiliza el agente {stackstate-product-name} con el stackpack Kubernetes. El uso de un nombre de clúster diferente dará lugar a una perspectiva de trazas vacía para los componentes de Kubernetes y, en general, hará que la correlación de la información sea mucho más difícil para {stackstate-product-name} y sus usuarios.
====

Instale el colector utilizando el archivo de configuración:

[,bash]
----
helm upgrade --install opentelemetry-collector open-telemetry/opentelemetry-collector \
  --values otel-collector.yaml \
  --namespace open-telemetry
----

El xref:/setup/otel/collector.adoc[recopilador] ofrece muchos más receptores de configuración, procesadores y exportadores, para más detalles consulte nuestra xref:/setup/otel/collector.adoc[página del recopilador]. Para el uso en producción, a menudo se generan grandes cantidades de tramos. Por lo tanto, establecer un xref:/setup/otel/sampling.adoc[muestreo].

== Recopile datos telemétricos de su aplicación

La forma habitual de recopilar datos de telemetría es instrumentar su aplicación utilizando los SDK de telemetría abiertos.

* xref:/setup/otel/instrumentation/java.adoc[Java]
* xref:/setup/otel/instrumentation/dot-net.adoc[.NET]
* xref:/setup/otel/instrumentation/node.js.adoc[Node.js]

Para otros idiomas sigue la documentación en https://opentelemetry.io/docs/languages/[opentelemetry.io] y asegúrate de configurar el exportador SDK para enviar datos al recolector que acabas de instalar siguiendo xref:/setup/otel/instrumentation/sdk-exporter-config.adoc[estas instrucciones].

== Ver los resultados

Vaya a {stackstate-product-name} y asegúrese de que la mochila de telemetría abierta está instalada (a través del menú principal \-> Mochilas).

Si sus pods están recibiendo tráfico, debería ser capaz de encontrarlos bajo su nombre de servicio en Open Telemetry \-> services and service instances overviews. Las trazas aparecen en el xref:/use/traces/k8sTs-explore-traces.adoc[explorador de trazas] y en la xref:/use/views/k8s-traces-perspective.adoc[perspectiva de trazas] para los componentes de servicio e instancia de servicio. Las métricas de alcance y las métricas específicas del idioma (si están disponibles) pasan a estar disponibles en la xref:/use/views/k8s-metrics-perspective.adoc[perspectiva de métricas] de los componentes.

Si también tiene instalado el stackpack de Kubernetes, los pods instrumentados también tendrán las trazas disponibles en la xref:/use/views/k8s-traces-perspective.adoc[perspectiva de trazas].

== Rancher RBAC

Para que xref:/setup/security/rbac/rbac_rancher.adoc[Rancher RBAC] funcione, los datos de telemetría necesitan tener presentes los siguientes atributos de recursos:

* `k8s.cluster.name` - el nombre *del clúster* utilizado por el stackpack de Kubernetes
* `k8s.namespace.name` - un *espacio de nombres* gestionado por un *proyecto* Rancher

Esto puede lograrse mediante una configuración como la anterior.  Allí, el preajuste `kubernetesAttributes` del gráfico `opentelemetry-collector` Helm inyecta un procesador `k8sattributes` en cada canalización.
