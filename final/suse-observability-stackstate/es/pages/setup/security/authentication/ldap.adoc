== Visión general

SUSE Observability puede utilizar un servidor LDAP (incluyendo AD) para autenticar y obtener roles/grupos. Requiere un servidor LDAP en ejecución que sea accesible para SUSE Observability.

Se comprobarán el directorio principal LDAP y todos los subdirectorios en busca de archivos de usuario. Las credenciales de enlace de la configuración de SUSE Observability se utilizan para autenticar SUSE Observability en el servidor LDAP. Tras la autenticación, SUSE Observability pasa el nombre del directorio LDAP superior del usuario que desea iniciar sesión en SUSE Observability.

== Configurar SUSE Observability para LDAP

=== Kubernetes

Para configurar SUSE Observability para que se autentique mediante un servidor de autenticación LDAP en Kubernetes, es necesario añadir los detalles de LDAP y la asignación de funciones de usuario al archivo `authentication.yaml`. Por ejemplo:

[tabs]
====
authentication.yaml::
+
--

[,yaml]
----
stackstate:
  authentication:
    ldap:
      host: sts-ldap
      port: 10389 # For most LDAP servers 389 for plain, 636 for ssl connections
      #ssl:
      #  sslType: ssl
      #  trustStore: <see below>
      #  trustCertificates <see below>
      bind:
        dn: "cn=admin,ou=employees,dc=acme,dc=com"
        password: "password"
      userQuery:
        parameters:
          - ou: employees
          - dc: acme
          - dc: com
        usernameKey: cn
        emailKey: mail
      groupQuery:
        parameters:
          - ou: groups
          - dc: acme
          - dc: com
        rolesKey: cn
        groupMemberKey: member
        # to return all nested groups, use:
        # groupMemberKey: "member:1.2.840.113556.1.4.1941:"

    # map the groups from LDAP to the
    # standard subjects in SUSE Observability (guest, powerUser and admin)
    roles:
      guest: ["ldap-guest-role-for-stackstate"]
      powerUser: ["ldap-power-user-role-for-stackstate"]
      admin: ["ldap-admin-role-for-stackstate"]
----

--
====

Siga los pasos que se indican a continuación para configurar SUSE Observability para autenticarse mediante LDAP:

. En `authentication.yaml` - añada los detalles LDAP (véase el ejemplo anterior):
 ** *host* - El nombre de host del servidor LDAP.
 ** *port* - El puerto en el que escucha el servidor LDAP.
 ** *sslType* - Opcional. El tipo de conexión segura LDAP `ssl` o `startTls`. Omitir si se utiliza una conexión LDAP simple.
 ** *trustCertificates* - Opcional, archivo de certificado para SSL. Se admiten los formatos PEM, DER y PKCS7.
 ** *trustStore* - Opcional, archivo Java trust store para SSL. Si se especifican tanto `trustCertificates` como `trustStore`, `trustCertificatesPath` tiene prioridad.
 ** *bind* - Opcional, se utiliza para autenticar SUSE Observability en el servidor LDAP si el servidor LDAP no admite búsquedas LDAP anónimas.
 ** *userQuery parameters y groupQuery parameters* - El conjunto de parámetros en su interior corresponden al dn base de su LDAP donde se pueden encontrar usuarios y grupos. El primero se utiliza para autenticar usuarios en SUSE Observability, mientras que el segundo se utiliza para recuperar el grupo de ese usuario y determinar si se trata de un Administrador, un Usuario avanzado o un Invitado.
 ** *usernameKey* - El nombre del atributo que almacena el nombre de usuario, el valor se compara con el nombre de usuario proporcionado en la pantalla de inicio de sesión.
 ** *emailKey* - El nombre del atributo que se utiliza como dirección de correo electrónico en SUSE Observability.
 ** *rolesKey* - Nombre del atributo que almacena el nombre del grupo.
 ** *groupMemberKey* - Nombre del atributo que indica si un usuario es miembro de un grupo. El filtro LDAP construido sigue este patrón: `<groupMemberKey>=<user.dn>,ou=groups,dc=acme,dc=com`. Para devolver todos los grupos anidados, utilice `groupMemberKey: "member:1.2.840.113556.1.4.1941:"`.
. En `authentication.yaml` - asigne funciones de usuario de LDAP a los temas correctos de SUSE Observability (consulte el ejemplo anterior):
 ** *roles* - para más detalles, consulte los xref:/setup/security/rbac/rbac_permissions.adoc#_predefined_roles[roles por defecto de SUSE Observability]. También se pueden crear más roles de SUSE Observability, consulte la xref:/setup/security/rbac/README.adoc[documentación de RBAC].
. Guarde el archivo `authentication.yaml` junto con el `values.yaml` de las instrucciones de instalación de SUSE Observability.
. Ejecute una actualización de Helm para aplicar los cambios. Si está utilizando SSL *con certificados personalizados*, los archivos binarios de certificado que deben utilizarse al conectarse a LDAP deben establecerse desde la línea de comandos, utilice el comando en *SSL con certificados personalizados*:

[,text]
----
helm upgrade \
  --install \
  --namespace suse-observability \
  --values values.yaml \
  --values authentication.yaml \
suse-observability \
suse-observability/suse-observability
----

*trustCertificates*

[,bash]
----
helm upgrade \
  --install \
  --namespace suse-observability \
  --values values.yaml \
  --values authentication.yaml \
  --set-file stackstate.authentication.ldap.ssl.trustCertificates=./ldap-certificate.pem \
suse-observability \
suse-observability/suse-observability
----

*trustStore*

[,bash]
----
helm upgrade \
  --install \
  --namespace suse-observability \
  --values values.yaml \
  --values authentication.yaml \
  --set-file stackstate.authentication.ldap.ssl.trustStore=./ldap-cacerts \
suse-observability \
suse-observability/suse-observability
----

[NOTE]
====
*Nota:*

* La primera ejecución del comando helm upgrade provocará el reinicio de los pods, lo que puede causar una breve interrupción de la disponibilidad.
* Incluya `authentication.yaml` en cada ejecución de `helm upgrade`.
* La configuración de autenticación se almacena como un secreto de Kubernetes.
====


=== Utilizar un secreto externo

Cuando la contraseña ldap deba proceder de un secreto externo, siga xref:/setup/security/external-secrets.adoc#_getting_authentication_data_from_an_external_secret[estos pasos] pero rellene los siguientes datos:

[,yaml]
----
kind: Secret
metadata:
   name: "<custom-secret-name>"
type: Opaque
data:
  ldap_password: <base64 of ldap password>
----
