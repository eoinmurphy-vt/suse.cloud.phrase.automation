= Ámbitos topológicos
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: SUSE Observability autoalojado

WARNING: El alcance de la topología por consulta es una funcionalidad obsoleta. Es sustituido por el permiso `get-topology` que puede conceder acceso a cualquier componente con una etiqueta específica.

== ¿Cómo funcionan los ámbitos topológicos?

El ámbito de la topología es una xref:/develop/reference/k8sTs-stql_reference.adoc[consulta STQL] que se combina con los permisos de `get-topology` como prefijo de cada consulta de topología ejecutada en SUSE Observability. Cuando un usuario desea seleccionar una vista o pasar una consulta en SUSE Observability, este prefijo de consulta se ejecuta como parte de la consulta del usuario. Esto limita los resultados para que coincidan con la función del usuario.

NOTE: No se admiten llamadas a funciones como `withCauseOf` y `withNeighborsOf`, ya que no serían eficaces en este contexto.

Si un usuario pertenece a varios grupos, puede tener varios ámbitos de topología, lo que se traduce en varios prefijos. Aquí, el prefijo se ejecuta como un OR de todos los ámbitos de topología que tenga el usuario.

Los usuarios deben cerrar sesión y autenticarse de nuevo en SUSE Observability cada vez que cambien sus funciones o permisos.

== ¿Por qué ámbitos topológicos?

Los ámbitos de topología son una función de seguridad para temas dentro de SUSE Observability. Los roles predefinidos de SUSE Observability Administrador, Usuario avanzado e Invitado no tienen definido el ámbito de la topología.

Es posible especificar un ámbito de topología como comodín de consulta, sin embargo, esto da como resultado el acceso a todo y no se recomienda. Si se necesita un acceso sin ámbito de topología, se recomienda utilizar en su lugar uno de los xref:/setup/security/rbac/rbac_permissions.adoc#_predefined_roles[roles predefinidos].

== Ejemplos

El siguiente ejemplo muestra la misma vista de topología denominada "Toda la infraestructura" para cuatro usuarios con distintos niveles de permiso.

=== Este usuario forma parte del grupo SUSE Observability Admin, por lo que no existe ámbito de topología:

image::v51_allperm.png[Ver todos los permisos]

La consulta para esta vista es la misma que para las demás, pero sin prefijo:

[,text]
----
'layer = "Infrastructure" AND domain IN ("Customer1", "Customer2")'
----

=== Abajo el usuario está en un grupo con el sujeto X configurado con el siguiente ámbito de topología:

[,text]
----
'domain = "Customer1"'
----

image::v51_esx1perm.png[Vista limitada]

La consulta con el prefijo para esta vista es:

[,text]
----
'(domain = "Customer1") AND (layer = "Infrastructure" AND domain IN ("Customer1", "Customer2"))'
----

=== Otro usuario que forma parte de un grupo con un tema Y configurado que tiene el siguiente ámbito de topología:

[,text]
----
'domain = "Customer2"'
----

obtiene esta topología:

image::v51_esx2perm.png[Vista limitada]

La consulta con el prefijo para esta vista es:

[,text]
----
'(domain = "Customer2") AND (layer = "Infrastructure" AND domain IN ("Customer1", "Customer2"))'
----

=== Usuario con varios prefijos

Es posible asignar un sujeto a más de un grupo. En este ejemplo, puede ver un Administrador de Infraestructura que puede ver toda la vista presentada anteriormente. Este usuario tiene que estar en los dos grupos que tienen configurados los temas X e Y. En este caso, el prefijo para la consulta del usuario tendrá el siguiente aspecto:

[,text]
----
'(domain = "Customer1" OR domain = "Customer2")'
----

La consulta con prefijo para este usuario es entonces:

[,text]
----
'(domain = "Customer1" OR domain = "Customer2") AND (layer = "Infrastructure" AND domain IN ("Customer1", "Customer2"))'
----

Lo que da como resultado la siguiente vista:

image::v51_allperm.png[Ver todos los permisos]
