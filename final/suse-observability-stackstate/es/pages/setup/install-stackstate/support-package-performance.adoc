== Visión general

Esta página proporciona un método para perfilar un clúster en ejecución para verificar que funciona según lo esperado. Esto se puede utilizar cuando se experimenta un rendimiento degradado para investigar activamente cómo están funcionando varias partes. Se trata de un complemento del recopilador xref:/setup/install-stackstate/support-package-logs.adoc[Paquete de soporte (Logs] ) para recopilar logs en general.

== Notas

El script debe ejecutarse desde un host que tenga kubectl configurado con acceso al cluster {stackstate-product-name} o desde cualquier otro host con los permisos necesarios y conectividad al cluster.
Esto se puede hacer directamente en el host utilizando el usuario root o a través de sudo. Si kubeconfig no está configurado, utilice el comando `export KUBECONFIG=$PATH-TO-YOUR/kubeconfig`.

== Utilización

El script debe descargarse y ejecutarse directamente en el host, utilizando el usuario `root` o `sudo`.

=== Descargar y ejecutar el script

Descargue el script xref::attachment$suse-observability_performance_collector.sh[suse-observability_performance_collector.sh].

* Guarda el script como: `suse-observability_performance_collector.sh`
* Ejecute el script utilizando los siguientes comandos:
+
[,bash]
----
bash suse-observability_performance_collector.sh
----
+
Esto generará un archivo en el directorio actual llamado `suse-observability_performance_<date>.tar.gz`. Sube este archivo al caso.

Se pueden especificar argumentos adicionales. Por ejemplo, para elegir un espacio de nombres diferente:
[,bash]
----
> bash suse-observability_performance_collector.sh -h
SUSE Observability performance measurement tool.
Runs some rudimentary performance tests on a deployed instance to validate performance.

Usage: $0 [options] [<namespace>]

options:
  -h  Print this help

<namespace>:
  The namespace that is running SUSE Observability, or
  "suse-observability" when not specified
----

Aclaraciones:

- La E/S con búfer está relacionada con el rendimiento bruto del disco, ya que permite que los datos se almacenen en búfer en el núcleo.
- El disco directo mide el rendimiento del disco utilizando O_DIRECT, sin permitir el almacenamiento en búfer, lo que se aproxima más a nuestras bases de datos utilizando fsync() y da una idea de la latencia del almacenamiento subyacente.
- Kafka Producer local pone los datos en kafka a través de localhost, remoto a través de la red. El remoto puede ser más rápido que el local cuando el cuello de botella es la CPU y no la red (como ocurría en este caso, por lo que el remoto era más eficiente).
