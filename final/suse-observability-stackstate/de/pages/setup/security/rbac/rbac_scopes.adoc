WARNING: Topologie-Scoping durch Abfrage ist eine veraltete Funktionalität. Sie wird durch die Berechtigung `get-topology` ersetzt, die den Zugriff auf jede Komponente mit einem bestimmten Label erlaubt.

== Wie funktionieren die Topologiebereiche?

Der Topologiebereich ist eine xref:/develop/reference/k8sTs-stql_reference.adoc[STQL-Abfrage], die mit den `get-topology` -Berechtigungen als Präfix für jede in SUSE Observability ausgeführte Topologieabfrage kombiniert wird. Wenn ein Benutzer eine Ansicht auswählen oder eine Abfrage in SUSE Observability durchführen möchte, wird diese Präfixabfrage als Teil der Benutzerabfrage ausgeführt. Dadurch werden die Ergebnisse auf die Rolle des Benutzers eingeschränkt.

NOTE: Funktionsaufrufe wie `withCauseOf` und `withNeighborsOf` werden nicht unterstützt, da sie in diesem Kontext nicht performant wären.

Wenn ein Benutzer zu mehreren Gruppen gehört, kann er mehrere Topologiebereiche haben, was zu mehreren Präfixen führt. Hier wird das Präfix als ODER aller Topologiebereiche, die der Benutzer hat, ausgeführt.

Benutzer müssen sich abmelden und erneut bei SUSE Observability authentifizieren, wenn sich ihre Rollen oder Berechtigungen ändern.

== Warum Topologieumfänge?

Topologiebereiche sind eine Sicherheitsfunktion für Subjekte innerhalb von SUSE Observability. Für die vordefinierten SUSE Observability-Rollen Administrator, Hauptbenutzer und Gast ist kein Topologiebereich definiert.

Es ist möglich, einen Topologiebereich als Platzhalter für eine Abfrage anzugeben, dies führt jedoch zu einem Zugriff auf alles und wird nicht empfohlen. Wenn ein Zugriff ohne Topologiebereich erforderlich ist, empfiehlt es sich, stattdessen eine der xref:/setup/security/rbac/rbac_permissions.adoc#_predefined_roles[vordefinierten Rollen] zu verwenden.

== Beispiele

Das folgende Beispiel zeigt dieselbe Topologieansicht mit dem Namen "All Infrastructure" für vier Benutzer mit unterschiedlichen Berechtigungsstufen.

=== Dieser Benutzer ist Teil der SUSE Observability-Administrationsgruppe, daher gibt es keinen Topologiebereich:

image::v51_allperm.png[Berechtigungen für die Vollansicht]

Die Abfrage für diese Ansicht ist dieselbe wie für die anderen, jedoch ohne Präfix:

[,text]
----
'layer = "Infrastructure" AND domain IN ("Customer1", "Customer2")'
----

=== Untenstehender Benutzer ist in einer Gruppe mit dem konfigurierten Subjekt X mit dem folgenden Topologiebereich:

[,text]
----
'domain = "Customer1"'
----

image::v51_esx1perm.png[Eingeschränkte Ansicht]

Die Abfrage mit dem Präfix für diese Ansicht lautet:

[,text]
----
'(domain = "Customer1") AND (layer = "Infrastructure" AND domain IN ("Customer1", "Customer2"))'
----

=== Ein anderer Benutzer, der Teil einer Gruppe mit einem konfigurierten Thema Y ist, das den folgenden Topologiebereich hat:

[,text]
----
'domain = "Customer2"'
----

erhält diese Topologie:

image::v51_esx2perm.png[Eingeschränkte Ansicht]

Die Abfrage mit dem Präfix für diese Ansicht lautet:

[,text]
----
'(domain = "Customer2") AND (layer = "Infrastructure" AND domain IN ("Customer1", "Customer2"))'
----

=== Benutzer mit mehreren Präfixen

Es ist möglich, ein Thema mehr als nur einer Gruppe zuzuordnen. In diesem Beispiel sehen Sie einen Infrastrukturmanager, der die gesamte oben dargestellte Ansicht sehen kann. Dieser Benutzer muss in beiden Gruppen sein, die als X und Y konfiguriert sind. In diesem Fall sieht das Präfix für die Benutzerabfrage wie folgt aus:

[,text]
----
'(domain = "Customer1" OR domain = "Customer2")'
----

Die Abfrage mit Präfix für diesen Benutzer lautet dann:

[,text]
----
'(domain = "Customer1" OR domain = "Customer2") AND (layer = "Infrastructure" AND domain IN ("Customer1", "Customer2"))'
----

Daraus ergibt sich die folgende Ansicht:

image::v51_allperm.png[Berechtigungen für die Vollansicht]
