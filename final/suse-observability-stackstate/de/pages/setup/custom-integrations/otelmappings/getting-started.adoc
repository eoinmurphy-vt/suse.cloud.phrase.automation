== Übersicht

Dieses Dokument bietet eine Anleitung für den Einstieg in die Topologieerstellung aus OpenTelemetry (OTel) Trace-Daten unter Verwendung von Komponenten- und Relation-Mappings, die als Teil eines StackPack verpackt sind.

Dieser Leitfaden konzentriert sich auf die Topologie, die sofort im Produkt anhand der bereits vorhandenen Telemetriedaten visualisiert werden kann.
Die Beispieltopologie, die generiert wird, modelliert, wie eine Dienstinstanz einen Prozess ausführt, der von OpenTelemetry-Ressourcenattributen abgeleitet ist.

== Voraussetzungen

* Sie sammeln bereits OpenTelemetry-Traces.
* Sie erstellen oder erweitern ein StackPack.
* Sie sind vertraut mit YAML und grundlegenden OTel-Konzepten (Ressourcen, Spans).

Der Leitfaden konzentriert sich auf folgende Themen:

* Spurbasierte Topologie (nur`TRACES` Signal)
* Service-Instanzen (keine logischen Services)
* Sichtbarkeit von Prozessen zur Laufzeit

[NOTE]
====
In diesem Handbuch werden die Zuordnungen von Komponenten und Relationen als YAML-Konfiguration ausgedrückt, die als Teil eines StackPacks verpackt, getestet und bereitgestellt wird.
====

== Topologie konfigurieren

Ziel ist es, die Topologie der Laufzeitausführung zu visualisieren, die ohne zusätzliche Benutzerkonfiguration sofort verfügbar ist.

Konkret wollen wir ein Modell erstellen:

[source]
----
service instance (component) -> executes (relation) -> process (component)
----

Wo:

* Eine `service instance` stellt eine laufende Instanz eines instrumentierten Dienstes dar.
* Eine `process` steht für den Betriebssystemprozess, der diesen Dienst ausführt.
* Die Beziehung `executes` zeigt an, dass die Dienstinstanz von einem bestimmten Prozess unterstützt wird und innerhalb dieses Prozesses läuft.

Die gesamte Topologie wird automatisch aus OpenTelemetry-Trace-Daten abgeleitet.

== Trace-Daten

Neben der Service-Identität enthält die Trace-Ressource auch Attribute auf Prozessebene, wie z. B.:

* `process.pid`
* `process.executable.name`
* `process.executable.path`
* `process.command_args`
* `process.runtime.name`
* `process.runtime.version`

Diese Attribute ermöglichen es uns, die Topologie auf Prozessebene zu modellieren, ohne Korrelationen oder Heuristiken zu berücksichtigen.

== Spuren zur Topologie: das mentale Modell

Bevor Sie eine Konfiguration schreiben, ist es wichtig zu verstehen, wie Topologie-Zuordnungen konzeptionell funktionieren.

=== Komponenten

Eine Komponentenzuordnung beschreibt, wie ein Topologieknoten aus Telemetriedaten erstellt wird.

Jede Komponente wird abgebildet:

* Wählt die Telemetrie anhand von Bedingungen aus.
* Extrahiert Werte mit Hilfe von Ausdrücken.
* Erzeugt eine einzelne logische Komponente, die durch einen stabilen Bezeichner identifiziert wird.

In diesem Leitfaden:

* Dienstinstanzen werden durch das OpenTelemetry StackPack bereitgestellt.
* Prozesse werden von OpenTelemetry-Ressourcenattributen abgeleitet.

=== Beziehungen

Ein Relationsmapping beschreibt, wie eine Verbindung zwischen zwei Komponenten hergestellt wird.

Jede Beziehung abbilden:

* Löst eine `sourceId` und eine `targetId` auf.
* Weist einen Beziehungstyp zu.
* Erzeugt eine gerichtete Kante.

Beziehungen werden erstellt, sobald sowohl die Quell- als auch die Zielkomponente existieren.

== Erstellen von Dienstkomponenten aus Traces

=== Definieren, was eine "Dienstleistung" ist

Bevor wir das Mapping schreiben, müssen wir eine Design-Entscheidung treffen.

In diesem Leitfaden wird ein Dienst definiert als:

* Identifiziert durch `service.namespace` + `service.name`.
* Stabil bei verschiedenen Einsätzen.
* Unabhängig von Service-Instanzen.

Dadurch bleibt die Topologie lesbar und die Kardinalität gering.

== Komponenten der Dienstinstanz

Die Komponenten der Dienstinstanz sind bereits definiert und werden vom OpenTelemetry StackPack bereitgestellt.

Jede Dienstinstanz:

* Ist abgeleitet von `service.name`, `service.namespace` und `service.instance.id`.
* Stellt eine konkrete laufende Instanz eines Dienstes dar.
* Ist über Signale hinweg stabil (Spuren und Metriken).

Da dieses Mapping bereits existiert, wird es in diesem Leitfaden nicht neu definiert.
Stattdessen bauen wir darauf auf.

== Erstellen von Prozesskomponenten aus Traces

=== Definieren, was ein "Prozess" bedeutet

In diesem Leitfaden wird ein Prozess definiert als:

* Identifiziert durch `host.name`, `process.pid` und ausführbare Metadaten.
* Gültig für eine einzige Laufzeitumgebung.
* Ausschließlich von OpenTelemetry-Ressourcenattributen abgeleitet.

Dadurch wird die Topologie auf eine niedrige Kardinalität reduziert, während gleichzeitig nützliche Details zur Laufzeit aufgedeckt werden.

=== Abbildung von Prozesskomponenten

Die folgende Komponentenzuordnung erzeugt eine Topologiekomponente pro beobachtetem Prozess.

[,yaml]
----
_type: "OtelComponentMapping"
name: "OTel Process"
description: "Represents an operating system process derived from OpenTelemetry"
identifier: "urn:stackpack:<stackpack-name>:shared:otel-component-mapping:process"
input:
  signal:
    - "TRACES"
  resource:
    condition: |
      'host.name' in resource.attributes &&
      'process.pid' in resource.attributes
    action: "CREATE"
vars:
  - name: "pid"
    value: "${string(int(resource.attributes['process.pid']))}"
  - name: "hostname"
    value: "${resource.attributes['host.name']}"
  - name: "executableName"
    value: >-
      ${
        'process.executable.name' in resource.attributes ?
         resource.attributes['process.executable.name'] :
          'process.command' in resource.attributes ?
           resource.attributes['process.command'] :
            'process.command_args' in resource.attributes ?
             resource.attributes['process.command_args'] :
                resource.attributes['process.executable.path']
      }
output:
  identifier: "urn:opentelemetry:process/${vars.hostname}:${vars.pid}"
  name: "${vars.hostname}/${vars.executableName}:${vars.pid}"
  typeName: "process"
  typeIdentifier: "urn:stackpack:open-telemetry:shared:component-type:process"
  layerName: "Applications"
  domainName: "Open Telemetry"
  required:
    tags:
      - source: "process-component"
        target: "custom"
      - source: "${resource.attributes}"
        pattern: "process.(.*)"
        target: "process.${1}"
expireAfter: 900000
----

==== Wie diese Zuordnung funktioniert

* Wir verarbeiten nur Trace-Daten (TRACES-Signal).
* Eine Prozesskomponente wird immer dann erstellt, wenn `host.name` und `process.pid` vorhanden sind.
* Der Identifikator ist während der gesamten Lebensdauer des Prozesses stabil.
* Ein benutzerdefinierter Tag wird hinzugefügt, um die Filterung (für die Überprüfungsphase) zu erleichtern.

Ersetzen Sie `<stackpack-name>` durch den Namen Ihres StackPacks (wenn Sie noch keinen haben, können Sie einen beliebigen Namen verwenden, z. B. `mystackpack`).

== Erstellen von ausführenden Beziehungen

Da nun Dienstinstanzen und Prozesse als Komponenten existieren, können wir sie miteinander verbinden.

=== Dienstinstanz führt Prozess aus

Die folgende Beziehungszuordnung erstellt eine `executes` Beziehung zwischen einer Dienstinstanz und einem Prozess.

Diese Zuordnung ist an die bestehende Beziehung "Host führt Dienstinstanz aus" angepasst, die vom OpenTelemetry StackPack bereitgestellt wird.

[,yaml]
----
_type: "OtelRelationMapping"
name: "Executes Relation (Service Instance -> Process)"
description: "Service instance executes a process"
identifier: "urn:stackpack:<stackpack-name>:shared:otel-relation-mapping:executes-service-instance"
input:
  signal:
    - "TRACES"
  resource:
    condition: |
      'service.name' in resource.attributes &&
      'host.name' in resource.attributes &&
      'process.pid' in resource.attributes
    action: "CREATE"
vars:
  - name: "namespace"
    value: "${'service.namespace' in resource.attributes && resource.attributes['service.namespace'] != '' ? resource.attributes['service.namespace'] : 'default'}"
  - name: "service"
    value: "${resource.attributes['service.name']}"
  - name: "instanceId"
    value: >-
      ${
        'service.instance.id' in resource.attributes && resource.attributes['service.instance.id'] != '' ?
        resource.attributes['service.instance.id'] :
        resource.attributes['service.name']
      }
  - name: "hostname"
    value: "${resource.attributes['host.name']}"
  - name: "pid"
    value: "${string(int(resource.attributes['process.pid']))}"
output:
  sourceId: "urn:opentelemetry:namespace/${vars.namespace}:service/${vars.service}:serviceInstance/${vars.instanceId}"
  targetId: "urn:opentelemetry:process/${vars.hostname}:${vars.pid}"
  typeName: "executes"
expireAfter: 900000
----

==== Wie diese Zuordnung funktioniert

* Wir verarbeiten nur Trace-Daten (`TRACES` signal).
* Die Beziehung wird immer dann erstellt, wenn sowohl Dienstinstanz als auch Prozessdaten vorhanden sind.
* Es ist keine Spannenkorrelation erforderlich.
* Der Ausdruck `targetId` muss mit dem Ausdruck `output.identifier` der Prozesskomponentenzuordnung übereinstimmen.

Der Beziehungsbezeichner wird (automatisch) aus `sourceId` und `targetId` gebildet und hat die Form: `sourceId-relationId`

Ersetzen Sie `<stackpack-name>` durch den Namen Ihres StackPacks (wenn Sie noch keinen haben, können Sie einen beliebigen Namen verwenden, z. B. `mystackpack`).

== Validierung der OTel-Zuordnungen

Es gibt zwei Möglichkeiten, die Korrektheit der Zuordnungen zu überprüfen, bevor sie in der Produktion eingesetzt werden.

1. Verwenden Sie den Befehl `sts stackpack test-deploy`, um ein StackPack mit den Mappings zu paketieren, hochzuladen und auf einer laufenden {stackstate-product-name} -Instanz zu installieren/aktualisieren.
2. Mit den Befehlen `sts otel-component-mapping apply` und `sts otel-relation-mapping apply` können Sie die Zuordnungen zu einer laufenden Instanz von {stackstate-product-name} individuell erstellen/aktualisieren.

=== Testen der Zuordnungen zusammen in einem StackPack

Angenommen, beide Zuordnungen befinden sich in Ihrem StackPack, dann können sie gemeinsam getestet werden. Weitere Informationen finden Sie in der xref:/setup/cli/cli-sts.adoc[StackPack CLI-Dokumentation].

Mit dem Befehl `sts stackpack test-deploy --yes` können Sie das tun:

* Paketieren, Hochladen und Installieren/Aktualisieren des StackPack
* Validierung der deklarativen Komponenten- und Beziehungszuordnungen im StackPack (z. B. Korrektheit des Ausdrucks, korrekte Referenz der Eingangssignaldaten auf der Grundlage der bereitgestellten Filterung)

[NOTE]
====
Der Befehl `sts stackpack test` speist keine Beispiel-Trace-Daten in die Mappings ein.
Um zu überprüfen, ob die Mappings die richtige Topologie ergeben, stellen Sie sicher, dass Open Telemetry-Trace-Daten an SUSE Observability gesendet werden.
Weitere Informationen zur Verwendung von `sts stackpack test` finden Sie im Abschnitt xref:/setup/custom-integrations/develop.adoc[Entwicklung einer benutzerdefinierten Integration (StackPack)].
====

=== Individuelle Prüfung der Zuordnungen

Unter der Voraussetzung, dass die oben genannten Komponenten- und Beziehungszuordnungen in YAML-Dateien definiert sind, können sie einzeln angewendet werden.

Ersetzen Sie `<stackpack-name>` durch den Namen Ihres StackPacks (wenn Sie noch keinen haben, können Sie einen beliebigen Namen verwenden, z. B. `mystackpack`).

[,sh]
----
$ sts otel-component-mapping apply -f process-component-mapping.yaml
✅ OTel component mapping upserted successfully! Identifier: urn:stackpack:mystackpack:shared:otel-component-mapping:process, Name: OTel Process

$ sts otel-relation-mapping apply -f process-relation-mapping.yaml
✅ OTel Relation Mapping upserted successfully! Identifier: urn:stackpack:mystackpack:shared:otel-relation-mapping:executes-service-instance, Name: Executes Relation (Service Instance -> Process)
----

== Daraus resultierende Topologie

Wenn diese Zuordnungen angewandt werden, bildet die sich ergebende Topologie einen Topologiediagramm des Dienstes und des Prozesses, das vollständig aus Trace-Daten abgeleitet ist.

Wenn Sie zum Beispiel die `checkoutservice` aus der OTel-Demo-App verwenden, sollte die Topologie wie folgt aussehen:

----
checkoutservice (instance) ─> executes ─> checkoutservice (process)
----

Diese Topologie wird laufend aktualisiert, wenn neue Traces eintreffen, und läuft automatisch ab, wenn der Verkehr aufhört.

=== Sehen Sie sich die resultierende Topologie in {stackstate-product-name}

Verwenden Sie die Benutzeroberfläche {stackstate-product-name}, um eine visuelle Bestätigung zu erhalten, dass die Mappings die erwartete Topologie ergeben.

1. Öffnen Sie die Benutzeroberfläche {stackstate-product-name} unter dem konfigurierten Wert `baseUrl` Helm
2. Klicken Sie in der linken Seitenleiste auf `Open Telemetry > Services Instances`
3. Suchen Sie die `checkoutservice` in der Liste der Dienstinstanzen und klicken Sie auf den Namen der Dienstinstanz, um die Seite Komponentenübersicht/Highlights zu öffnen
4. Wählen Sie in der unteren Navigationsleiste am oberen Rand `Topology`
5. Auf der Ebene `Outgoing` sollte ein Knoten `od-checkoutservice-<hostId>/checkoutservice:1 (process)` sichtbar sein.
6. Wählen Sie den Prozessknoten aus, und klicken Sie auf `Explore component`
7. Eine kleinere Topologie, die `checkoutservice -> executes -> checkoutservice (process)` visualisiert, sollte sichtbar sein.

Im Leitfaden zur xref:/setup/custom-integrations/otelmappings/troubleshooting.adoc[Fehlerbehebung] finden Sie Tipps zur Fehlerbehebung, wenn die Topologie nicht wie erwartet funktioniert.

== Aufräumen

Um die in dieser Anleitung erstellten Zuordnungen zu entfernen, verwenden Sie die folgenden Befehle:

Ersetzen Sie `<stackpack-name>` durch den Namen Ihres StackPacks.

[,sh]
----
$ sts otel-component-mapping delete --identifier urn:stackpack:<stackpack-name>:shared:otel-component-mapping:process
✅ OTel Component Mapping deleted: urn:stackpack:mystackpack:shared:otel-component-mapping:process

$ sts otel-relation-mapping delete --identifier urn:stackpack:<stackpack-name>:shared:otel-relation-mapping:executes-service-instance
✅ OTel Relation Mapping deleted: urn:stackpack:mystackpack:shared:otel-relation-mapping:executes-service-instance
----
