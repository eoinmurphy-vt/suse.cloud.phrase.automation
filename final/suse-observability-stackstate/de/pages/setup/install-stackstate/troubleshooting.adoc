== Kurzanleitung zur Fehlerbehebung

Hier finden Sie eine Kurzanleitung zur Fehlerbehebung beim Start von SUSE Observability:

. Überprüfen Sie, ob die Installation erfolgreich abgeschlossen wurde und die Version aufgelistet ist:
+
[,text]
----
helm list --namespace suse-observability
----

. Überprüfen Sie, ob alle Pods im SUSE Observability-Namensraum ausgeführt werden:
+
[,text]
----
kubectl get pods
----
+
Bei einem ersten Einsatz kann es vorkommen, dass Container in mehreren Pods ein paar Mal neu starten, weil sie darauf warten, dass andere Pods starten und sich im Zustand `ready` befinden. Dies kann sich aufgrund von Verzögerungen bei der Planung und beim Ziehen von Docker-Images verzögern.
+
Hülsen, die sich im Zustand `pending` befinden, sind in der Regel ein Hinweis auf ein Problem:

 ** Der Pod ist aufgrund mangelnder Ressourcen im Cluster nicht planbar. Wenn ein automatischer Cluster-Scaler aktiv ist, kann er dieses Problem oft automatisch lösen, andernfalls ist ein manueller Eingriff erforderlich, um dem Cluster weitere Knoten hinzuzufügen.
 ** Der Pod ist nicht planbar, es gibt zwar Knoten, auf die er passen würde, aber diese Knoten haben `taints`, die der Pod nicht toleriert. SUSE Observability kann aber auch so xref:/setup/install-stackstate/kubernetes_openshift/customize_config.adoc#_override_default_configuration[konfiguriert werden], dass es bestimmte Verunreinigungen toleriert und auf den verunreinigten Knoten ausgeführt wird.
 ** Der Pod wartet darauf, dass die Persistent Volumes (PVs) eingehängt werden. Eine Ursache kann sein, dass das SUSE Observability Helm-Diagramm keine `storageClassName` angibt, sondern sich darauf verlässt, dass der Cluster eine Standardspeicherklasse hat. Wenn es keinen Standard für den Cluster gibt, müssen Sie über die Helm-Werte von SUSE Observability xref:/setup/install-stackstate/kubernetes_openshift/storage.adoc[eine Storage-Klasse angeben].

+
Prüfen Sie bei Pods mit dem Status `ImagePullBackOff` auch die genaue Fehlermeldung, häufige Ursachen sind:

 ** Ein falscher Benutzername/Passwort wurde zum Abrufen der Bilder verwendet
 ** Die Verbindung zur Docker-Registry ist fehlgeschlagen. Dies kann auf Authentifizierungs- oder Verbindungsprobleme zurückzuführen sein (Firewalls, Installationen mit Luftschleusen)
 ** Ein Tippfehler in der überschriebenen Docker-Image-Registry-URL

+
Um eine genauere Ursache für die Zustände `Pending`, `ImagePullBackOff` oder `CrashLoopBackOff` herauszufinden, verwenden Sie diesen Befehl:
+
[,text]
----
kubectl describe pod <pod-name>
----
+
Die Ausgabe enthält am Ende einen Abschnitt `event`, der normalerweise das Problem enthält. Außerdem gibt es einen Abschnitt `State` für jeden Container, der weitere Einzelheiten zur Beendigung des Containers enthält.

. Wenn Sie ein Hauptkunde sind, wenden Sie sich an den SUSE Observability-Support unter https://scc.suse.com/, um Hilfe bei der Einrichtung von SUSE Observability in Ihrem lokalen Cluster zu erhalten. Verwenden Sie xref:/setup/install-stackstate/support-package-logs.adoc[Support Package (Logs)], um Informationen über Ihre Instanz für das Support-Team zu sammeln.
. Wenn das Problem mit der Leistung zusammenhängt, führen Sie xref:/setup/install-stackstate/support-package-performance.adoc[Support Package (Performance)] aus, um die Leistung aktiv zu untersuchen.
. Für den Fall, dass die oben genannten Schritte das Problem nicht beheben konnten, gibt es eine xref:/setup/install-stackstate/advanced-troubleshooting.adoc[erweiterte Anleitung zur Fehlerbehebung].
