=== Namespace erstellen

Erstellen Sie zunächst den Namespace, in dem Sie {stackstate-product-name} installieren möchten, und stellen Sie das Geheimnis in diesem Namespace bereit. In unserem Beispiel werden wir den Namensraum `suse-observability` verwenden:

[,text]
----
kubectl create namespace suse-observability
----

[#create-values-yaml]
=== `values.yaml` Konfigurationsdatei erstellen

Die Datei `values.yaml` enthält die gesamte Konfiguration, die für die Bereitstellung von {stackstate-product-name} erforderlich ist, einschließlich Ihres Lizenzschlüssels, der Basis-URL, des Administratorkennworts und des Größenprofils.

[NOTE]
====
*Bevor Sie fortfahren:* Stellen Sie sicher, dass Sie die neueste Version der Helm-Karten mit `helm repo update` haben.
====

[tabs]
====
Empfohlene Methode::
+
--
[NOTE]
=====
Die Konfigurationsmethode `global.suseObservability` ist ab der Version `2.8.0` verfügbar. Für frühere Versionen verwenden Sie die Legacy-Methode.
=====

Erstellen Sie eine Datei `values.yaml` mit der Konfigurationsstruktur `global.suseObservability`:

[source,yaml]
----
global:
  # Optional: Override image registry (defaults to registry.rancher.com)
  # Only needed for air-gapped environments or custom registries
  # imageRegistry: "your-private-registry.example.com"

  # Optional: Override storage class (uses cluster default if not set)
  # storageClass: "your-storage-class"

  suseObservability:
    # Required: Your {stackstate-product-name} license key
    license: "YOUR-LICENSE-KEY"

    # Required: Base URL for {stackstate-product-name}
    # Use http://localhost:8080 if ingress is not configured yet
    baseUrl: "https://observability.example.com"

    # Required: Sizing profile
    # Available: trial, 10-nonha, 20-nonha, 50-nonha, 100-nonha,
    #            150-ha, 250-ha, 500-ha, 4000-ha
    sizing:
      profile: "150-ha"

    # Required: Plain text admin password
    adminPassword: "your-password"
    # Instead of adminPassword you can provide a bcrypt hashed password with adminPasswordBcrypt
    # Generate with: htpasswd -bnBC 10 "" "your-password" | tr -d ':\n'
    # adminPasswordBcrypt: "$2a$10$..."

    # Optional: Receiver API key (auto-generated if not provided)
    # receiverApiKey: "your-receiver-api-key"

    # Optional: Pull secret for private registry
    # pullSecret:
    #   username: "registry-user"
    #   password: "registry-pass"
----

Weitere Konfigurationsoptionen, einschließlich Affinitätseinstellungen, finden Sie unter xref:/setup/install-stackstate/kubernetes_openshift/affinity.adoc[Konfigurieren von Kubernetes-Affinitäten].

+

Diese Methode ist veraltet. Bei Neuinstallationen ist die oben empfohlene Methode anzuwenden. Für bestehende Installationen, die diese Methode verwenden, lesen Sie bitte den xref:/setup/upgrade-stackstate/migrate-to-global-mode.adoc[Migrationsleitfaden] für die Umstellung auf das neue Konfigurationsformat.

Sie können die {stackstate-product-name} Konfigurationsdateien generieren, indem Sie ein separates Helm Chart ausführen. Dies ist die Karte `suse-observability/suse-observability-values`:

[,text]
----
export VALUES_DIR=.
helm template \
  --set license='<your license>' \
  --set baseUrl='<suse-observability-base-url>' \
  --set sizing.profile='<sizing.profile>' \
  suse-observability-values \
  suse-observability/suse-observability-values --output-dir $VALUES_DIR
----

Dieser Befehl erzeugt die Dateien `$VALUES_DIR/suse-observability-values/templates/baseConfig_values.yaml`, `$VALUES_DIR/suse-observability-values/templates/sizing_values.yaml` und `$VALUES_DIR/suse-observability-values/templates/affinity_values.yaml`.

[NOTE]
=====
Die {stackstate-product-name} Administrator-Passwörter werden durch den obigen Befehl automatisch generiert und als Kommentare in der generierten `basicConfig.yaml` Datei ausgegeben. Die eigentlichen Werte enthalten die `bcrypt` Hashes dieser Passwörter, so dass sie sicher in der Helm-Veröffentlichung im Cluster gespeichert sind.
=====

Die Werte, die an dieses Diagramm übergeben werden können, sind:

[cols="1,1,2"]
|===
| Konfiguration | Wert | Beschreibung

| Grundkonfiguration erzeugen
| `basicConfig.generate`
| Schalten Sie die Erstellung der Datei `basicConfig_values.yaml` ein oder aus.

| API-Schlüssel des Empfängers
| `receiverApiKey`
| Der API-Schlüssel, der von {stackstate-product-name} verwendet wird, um Daten von Agenten zu erhalten. Dies ist ein geheimer Schlüssel, der geheim gehalten werden sollte. Wenn Sie dies weglassen, wird ein Zufallsschlüssel für Sie generiert.

| Basis-URL
| `baseUrl`
| Die `<STACKSTATE_BASE_URL>`. Die externe URL für {stackstate-product-name}, die Benutzer und Agenten für die Verbindung verwenden. Zum Beispiel `https://suse-observability.internal`. Wenn Sie sich noch nicht für eine Ingress-Konfiguration entschieden haben, verwenden Sie `http://localhost:8080`. Dies kann später in der generierten Datei aktualisiert werden.

| Benutzername und Passwort**
| `-u` `-p`
| Der Benutzername und das Kennwort, die von {stackstate-product-name} verwendet werden, um Bilder abzurufen. In Umgebungen mit Luftschnittstelle müssen dies der Benutzername und das Kennwort für die lokale Benutzerregistrierung sein.

| Lizenzschlüssel
| `license`
| Der {stackstate-product-name} Lizenzschlüssel.

| Standard-Passwort
| `adminPassword`
| Das Passwort für den Standardbenutzer (`admin`) für den Zugriff auf die Benutzeroberfläche von {stackstate-product-name}. Alternativ können Sie ein mit bcrypt verschlüsseltes Passwort unter `adminPasswordBcrypt` angeben. Es kann nur eine der beiden Optionen gleichzeitig verwendet werden.

| Bild-Register
| `imageRegistry`
| Das Register, in dem die {stackstate-product-name} Bilder gehostet werden. Falls nicht angegeben, ist der Standardwert "registry.rancher.com". Nur für Umgebungen mit Luftschleusen oder benutzerdefinierten Registern erforderlich.

| Geheimen Benutzernamen ziehen
| `pullSecret.username`
| Der Benutzername, der zum Abrufen von Images aus der Docker-Registry verwendet wird, in der die {stackstate-product-name} -Images gehostet werden. Nur für benutzerdefinierte Registrierungen erforderlich.

| Geheimes Passwort ziehen
| `pullSecret.password`
| Das Passwort, das verwendet wird, um Images aus der Docker-Registry zu ziehen, in der die {stackstate-product-name} Images gehostet werden. Nur für benutzerdefinierte Registrierungen erforderlich.

| Dimensionierung erzeugen
| `sizing.generate`
| Schalten Sie die Erstellung der Datei `sizing_values.yaml` ein oder aus.

| Profil für die Größenbestimmung
| `sizing.profile`
| Ein Versuch, 10-Nonha, 20-Nonha, 50-Nonha, 100-Nonha, 150-ha, 250-ha, 500-ha. Basierend auf diesen Profilen können Sie die Datei `sizing_values.yaml` generieren, die Standardgrößen für die {stackstate-product-name} Ressourcen und die Konfiguration für die Bereitstellung im HA- oder Nicht-HA-Modus enthält. Zum Beispiel erzeugt 10-nonha eine `sizing_values.yaml`, um eine NonHa-Instanz {stackstate-product-name} zur Beobachtung eines 10-Knoten-Clusters in einem Nicht-Hochverfügbarkeitsmodus einzusetzen. Derzeit ist der Wechsel von einer Nicht-HA- zu einer HA-Umgebung nicht möglich. Wenn Sie davon ausgehen, dass in Ihrer Umgebung etwa 150 Knoten beobachtet werden müssen, ist es besser, sich sofort für HA zu entscheiden.

| Pod AntiAffinität
| `affinity.podAntiAffinity.requiredDuringSchedulingIgnoredDuringExecution`
| Erzwingen Sie, dass die Replikate der gleichen Datendienste auf verschiedenen Kubernetes-Knoten geplant werden. Gilt nur für `HA` Profile. Entweder `true` oder `false`. Die Standardeinstellung ist `true` - die Verteilung auf verschiedene Knoten wird erzwungen.
|===

[NOTE]
=====
Speichern Sie die erzeugten Dateien `basicConfig.yaml` und `sizing_values.yaml` sicher. Sie können diese Dateien für Upgrades wiederverwenden, was Zeit spart und sicherstellt, dass {stackstate-product-name} weiterhin denselben API-Schlüssel verwendet. Dies ist wünschenswert, da dies bedeutet, dass Agenten und andere Datenlieferanten für {stackstate-product-name} nicht mehr aktualisiert werden müssen.
Sie können die Dateien unabhängig voneinander neu generieren, indem Sie die Schalter `basicConfig.generate=false` und `sizing.generate=false` verwenden, um eine der Dateien zu deaktivieren, während die zuvor generierte Version der Datei in `output-dir` erhalten bleibt.
=====

[NOTE]
=====
Das Diagramm {stackstate-product-name} Werte erzeugt Affinitätskonfigurationen, die Sie zusammen mit dem Hauptdiagramm {stackstate-product-name} verwenden können, um das Pod-Planungsverhalten zu steuern. Weitere Informationen finden Sie in der Dokumentation xref:/setup/install-stackstate/kubernetes_openshift/affinity.adoc[Configure Kubernetes Affinities (Kubernetes-Affinitäten konfigurieren] ).
=====

=== Speicherklasse konfigurieren (optional)

{stackstate-product-name} verwendet die Standardspeicherklasse des Clusters, falls nicht angegeben. Sie können dies optional außer Kraft setzen, indem Sie `global.storageClass` in Ihrer `values.yaml` Datei einstellen.

[tabs]
====
Empfohlene Methode::
+
--
Wenn Sie eine bestimmte Speicherklasse verwenden müssen, fügen Sie diese in Ihre `values.yaml` Datei unter `global.storageClass` ein:

[source,yaml]
----
global:
  storageClass: "your-storage-class"  # e.g., "local-path" for Rancher local-path provisioner
  suseObservability:
    # ... other configuration
----
--
Veraltete Methode (veraltet)::
+
--
Bearbeiten Sie die Datei `$VALUES_DIR/suse-observability-values/templates/baseConfig_values.yaml`, um die Konfiguration "storageClass" hinzuzufügen:

[,text]
----
# Source suse-observability-values/templates/baseConfig_values.yaml
global:
  storageClass: "your-storage-class" # Optional: override default storage class
----
--

=== Bereitstellung von {stackstate-product-name} mit Helm

Der empfohlene Einsatz von {stackstate-product-name} ist ein produktionsbereites, hochverfügbares Setup mit vielen redundant ausgeführten Diensten. Bei Bedarf ist es auch möglich, {stackstate-product-name} in einer nicht-redundanten Konfiguration zu betreiben, bei der jeder Dienst nur eine einzige Replik hat. Diese Einstellung wird nur für eine Testumgebung empfohlen.

Für Umgebungen mit Luftzufuhr befolgen Sie die Anweisungen für Installationen mit Luftzufuhr.

[tabs]
====
Empfohlene Methode::
+
--

So stellen Sie {stackstate-product-name} auf Kubernetes bereit:

. Bevor Sie loslegen:
 ** xref:/setup/install-stackstate/kubernetes_openshift/kubernetes_install.adoc#_create_namespace[Erstellen Sie den Namensraum, in dem {stackstate-product-name} installiert werden soll.]
 ** <<create-values-yaml,`values.yaml` Konfigurationsdatei erstellen>>
. Stellen Sie die neueste Version von {stackstate-product-name} mit dem folgenden Befehl im Namensraum `suse-observability` bereit:

[,text]
----
helm upgrade \
  --install \
  --namespace suse-observability \
  --values values.yaml \
suse-observability \
suse-observability/suse-observability
----

Alternativ können Sie auch direkt die `--set` -Flags verwenden, ohne eine Wertedatei zu erstellen:

[,bash]
----
helm upgrade \
  --install \
  --namespace suse-observability \
  --set global.suseObservability.license="YOUR-LICENSE-KEY" \
  --set global.suseObservability.baseUrl="https://observability.example.com" \
  --set global.suseObservability.sizing.profile="150-ha" \
  --set global.suseObservability.adminPassword='$2a$10$...' \
suse-observability \
suse-observability/suse-observability
----

=== Zugriff auf die Benutzeroberfläche {stackstate-product-name} 

Nachdem {stackstate-product-name} bereitgestellt wurde, können Sie überprüfen, ob alle Pods funktionieren:

[,bash]
----
kubectl get pods --namespace suse-observability
----

Wenn alle Pods in Betrieb sind, können Sie eine Port-Weiterleitung aktivieren:

[,bash]
----
kubectl port-forward service/<helm-release-name>-suse-observability-router 8080:8080 --namespace suse-observability
----

Erstellen Sie für die Ingress-Konfiguration eine Datei `ingress_values.yaml` und fügen Sie den folgenden Code hinzu.

[NOTE]
====
Denken Sie daran, ein Geheimnis für die TLS-Konfiguration festzulegen. Einzelheiten dazu finden Sie in der kubectl-Dokumentation.
====

[,yaml]
----
ingress:
  enabled: true
  class: ""
  annotations:
    nginx.ingress.kubernetes.io/proxy-body-size: "50m"
    nginx.ingress.kubernetes.io/ssl-redirect: "true"
    #if external dns (cloudflare and similars) configure:
    #external-dns.alpha.kubernetes.io/target: <INGRESS_FQDN>
    #external-dns.alpha.kubernetes.io/ttl: "60"
    #end if
  hosts:
  - host: <SUSE_OBSERVABILITY_HOST>
  tls:
  - hosts:
    - <SUSE_OBSERVABILITY_HOST>
    secretName: suse-private-ai-tls
  existingSecret: suse-private-ai-tls
  # Set up your secrets in whichever way feels fit. Check k8s docs for details: https://kubernetes.io/docs/reference/kubectl/generated/kubectl_create/kubectl_create_secret_tls/
----

Führen Sie ein Upgrade durch, um die Ingress-Konfiguration anzuwenden:

[tabs]
====
Upgrade::
+
--

[,bash]
----
helm upgrade --namespace suse-observability --reuse-values --values ingress_values.yaml suse-observability suse-observability/suse-observability
----

--
Upgrade mit Luftabdeckung::
+
--

[,bash]
----
helm upgrade --namespace suse-observability --reuse-values --values suse-observability-values/templates/ingress_values.yaml suse-observability suse-observability-A.B.C.tgz
----

--

[NOTE]

Wenn Sie keinen externen DNS verwenden, fügen Sie auf Ihrem lokalen Rechner /etc/hosts hinzu: <CLUSTER_IP> <SUSE_OBSERVABILITY_HOST>.
