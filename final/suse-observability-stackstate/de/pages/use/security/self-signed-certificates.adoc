== SUSE Observability-Server

=== Freigabe von SUSE Observability Server mit selbstsignierten Zertifikaten

xref:k8s-suse-rancher-prime.adoc#_installation[SUSE Observability Server-Komponenten] können SSL nicht selbst terminieren und müssen einen SSL-geschützten Ingress Controller oder LoadBalancer verwenden, um den Service mit HTTPS bereitzustellen. In diesem Abschnitt wird erläutert, wie Sie einen Ingress Controller mit selbstsignierten Zertifikaten konfigurieren.

==== Verwendung von Ingress Controller mit TLS

Um SUSE Observability Server mit Traefik Ingress Controller mit selbstsignierten Zertifikaten bereitzustellen, müssen Sie Folgendes tun:

1. Erstellen von Kubernetes-TLS-Geheimnissen aus Ihrem Serverzertifikat und -schlüssel
2. Konfigurieren Sie die Ingress-Ressource mit TLS-Einstellungen
3. Bereitstellen von SUSE Observability Server mit aktiviertem Ingress

===== Erstellen von TLS-Geheimnissen

Erstellen Sie Kubernetes-TLS-Geheimnisse aus Ihren Zertifikats- und privaten Schlüsseldateien:

[,bash]
----
# Create TLS secret for main SUSE Observability Server
kubectl create secret tls tls-secret \
  --cert=server.crt \
  --key=server.key \
  --namespace suse-observability

# Create TLS secret for OTLP GRPC endpoint
kubectl create secret tls otlp-tls-secret \
  --cert=otlp-server.crt \
  --key=otlp-server.key \
  --namespace suse-observability

# Create TLS secret for OTLP HTTP endpoint
kubectl create secret tls otlp-http-tls-secret \
  --cert=otlp-http-server.crt \
  --key=otlp-http-server.key \
  --namespace suse-observability
----

===== Installieren mit Ingress

Stellen Sie SUSE Observability Server mit der Ingress-Konfiguration bereit:

[,bash]
----
helm upgrade --install \
  --namespace suse-observability \
  --create-namespace \
  --values ingress-with-tls-values.yaml \
  suse-observability \
  suse-observability/suse-observability
----

Eine vollständige Installationsanleitung finden Sie unter xref:k8s-suse-rancher-prime.adoc#_installation[Installation].

=== SUSE Observability Server verbindet sich mit externen Systemen mit selbstsignierten Zertifikaten

[NOTE]
====
In diesem Abschnitt wird SUSE Observability Server so konfiguriert, dass er externen Systemen vertraut, die selbstsignierte Zertifikate verwenden. Dies gilt, wenn der Server ausgehende Verbindungen zu externen Diensten (wie Webhooks) herstellen muss, die mit selbstsignierten Zertifikaten gesichert sind.
====

SUSE Observability Server verfügt über mehrere Interaktionspunkte mit externen Systemen. Zum Beispiel können Event-Handler Webhooks in anderen Systemen aufrufen. In der Standardkonfiguration kann SUSE Observability Server nicht mit diesen Systemen kommunizieren, wenn sie mit TLS und einem selbstsignierten Zertifikat oder einem Zertifikat gesichert sind, dem die JVM nicht standardmäßig vertraut.

Um dies zu verhindern, ermöglicht SUSE Observability Server die Konfiguration eines benutzerdefinierten Trust Stores.

=== Einen benutzerdefinierten Truststore erstellen

Sie müssen das benutzerdefinierte TLS-Zertifikat zur Verfügung haben. Wenn Sie diese nicht haben, müssen Sie xref:/use/security/self-signed-certificates.adoc#_retrieve_certificate_via_the_browser[sie über den Browser abrufen].

Verwenden Sie das Tool keytool und die in der JVM-Installation (Java Virtual Machine) enthaltene Datei `cacerts`, um eine vorhandene TLS-Zertifikatsdatei in das von SUSE Observability Server benötigte Format zu konvertieren. Sie können es auf jedem Rechner ausführen, unabhängig von der Art des Betriebssystems.

Wenn Sie die JVM nicht auf Ihrem Computer installiert haben, können Sie stattdessen auch xref:/use/security/self-signed-certificates.adoc#_using_a_docker_jvm[ein JVM-Docker-Image verwenden].

==== Verwendung einer installierten JVM

Wenn die JVM auf Ihrem Computer installiert und das Zertifikat als Datei `site.cert` gespeichert ist, können Sie einen neuen Truststore erstellen, indem Sie den Truststore der JVM nehmen und das zusätzliche Zertifikat hinzufügen.

. Erstellen Sie ein Arbeitsverzeichnis `workdir` und kopieren Sie die Zertifikatsdatei `site.cert` in dieses Verzeichnis.
. Wechseln Sie in das Verzeichnis `workdir` und erstellen Sie eine Kopie der Datei `cacerts` aus Ihrer Java-Installation. `$JAVA_HOME` ist eine Umgebungsvariable, die den Speicherort Ihrer Java-Installation enthält. Diese Einstellung wird normalerweise bei der Installation von Java vorgenommen.
+
[,bash]
----
cd workdir
cp $JAVA_HOME/lib/security/cacerts ./custom_cacerts
----

. Führen Sie den folgenden keytool-Befehl aus, um das Zertifikat hinzuzufügen. Das erforderliche Passwort lautet `changeit`. Der Alias muss ein eindeutiger Alias für das Zertifikat sein, z. B. der Domänenname selbst ohne Punkte.
+
[,bash]
----
keytool -import -keystore custom_cacerts -alias <a-name-for-the-certificate>  -file site.cert
----

. Die Speicherdatei `custom_cacerts` enthält jetzt das Zertifikat `site.cert`. Sie können dies überprüfen, indem Sie nach dem Alias in der Ausgabe von
+
[,bash]
----
keytool -list -keystore custom_cacerts
----

==== Verwendung einer Docker JVM

Wenn Sie keine JVM auf Ihrem Computer installiert haben, können Sie ein JVM-Docker-Image verwenden. Das Zertifikat sollte abgerufen und als Datei `site.cert` gespeichert werden.

. Erstellen Sie ein Arbeitsverzeichnis `workdir` und kopieren Sie die Zertifikatsdatei `site.cert` in dieses Verzeichnis.
. Starten Sie den Java-Docker-Container mit dem als Volume gemounteten `workdir`, damit darauf zugegriffen werden kann:
+
[,bash]
----
docker run -it -v `pwd`/workdir:/workdir  adoptopenjdk:11 bash
----

. Wechseln Sie in das Verzeichnis `workdir` und erstellen Sie eine Kopie der Datei `cacerts`:
+
[,bash]
----
cd /workdir
cp $JAVA_HOME/lib/security/cacerts ./custom_cacerts
----

. Führen Sie den folgenden keytool-Befehl aus, um das Zertifikat hinzuzufügen. Das erforderliche Passwort lautet `changeit`. Der Alias muss ein eindeutiger Alias für das Zertifikat sein, z. B. der Domänenname selbst ohne Punkte.
+
[,bash]
----
keytool -import -keystore custom_cacerts -alias <a-name-for-the-certificate>  -file site.cert
----

. Die Speicherdatei `custom_cacerts` enthält nun das Zertifikat `site.cert`. Sie können dies überprüfen, indem Sie nach dem Alias in der Ausgabe von
+
[,bash]
----
 keytool -list -keystore custom_cacerts
----

=== Einen benutzerdefinierten Truststore verwenden

Der Truststore und das Passwort können als Werte angegeben werden. Der Vertrauensspeicher kann nur über die helm-Befehlszeile angegeben werden, da er eine Datei ist. Der Passwortwert wird auf die gleiche Weise wie im Beispiel angegeben, kann aber auch über eine Datei `values.yaml` bereitgestellt werden.

[,bash]
----
helm upgrade \
  --install \
  --namespace suse-observability \
  --values values.yaml \
  --set-file 'stackstate.java.trustStore'=custom_cacerts \
  --set 'stackstate.java.trustStorePassword'=changeit \
suse-observability \
suse-observability/suse-observability
----

[NOTE]
====
*Anmerkung:*

* Die erste Ausführung des Befehls helm upgrade führt zu einem Neustart der Pods, was zu einer kurzen Unterbrechung der Verfügbarkeit führen kann.
* Geben Sie diese Argumente bei jedem `helm upgrade` -Lauf an.
* Das Passwort und der Trust Store werden als Kubernetes-Geheimnis gespeichert.
====


[discrete]
==== Base64-kodierte Vertrauensspeicher

Bei Bedarf kann der Java-Vertrauensspeicher auch konfiguriert werden, indem Base64-kodierte Strings an Helm-Werte übergeben werden.

[tabs]
====
Linux::
+
--

Um einen base64-kodierten Trust Store zu verwenden, führen Sie den folgenden Befehl `helm upgrade` aus:

[,bash]

  --install \
  --namespace suse-observability \
  --values values.yaml \
  --set 'stackstate.java.trustStoreBase64Encoded'=$(cat custom_cacerts | base64 -w0) \
  --set 'stackstate.java.trustStorePassword'=changeit \
suse-observability \
suse-observability/suse-observability
----

--

Erstellen Sie eine Kubernetes-ConfigMap mit Ihrem Zertifikat und referenzieren Sie sie in der Helm-Konfiguration:

[,yaml]
----
apiVersion: v1
kind: ConfigMap
metadata:
  name: tls-config
data:
  tls.crt: |

    [Your certificate content here]
    -----END CERTIFICATE-----

Verweisen Sie auf die ConfigMap in Ihrer Helm-Konfiguration:

    enabled: true

=== Bereitstellen mit benutzerdefinierten Zertifikaten

==== Verwendung direkter PEM-Daten

Für den direkten PEM-Datenansatz speichern Sie Ihr Zertifikat zunächst in einer Shell-Variablen:

[,bash]
----
export CERT_DATA=$(cat <<'EOF'
-----BEGIN CERTIFICATE-----
[Your certificate content here]
-----END CERTIFICATE-----
EOF
)
----

Stellen Sie den Agenten mit der Zertifikatskonfiguration bereit:

[,bash]
----
helm upgrade --install \
  --namespace suse-observability \
  --create-namespace \
  --set-string 'stackstate.apiKey'='YOUR_API_KEY' \
  --set-string 'stackstate.cluster.name'='YOUR_CLUSTER_NAME' \
  --set-string 'stackstate.url'='YOUR_SUSE_OBSERVABILITY_URL' \
  --set 'global.customCertificates.enabled'=true \
  --set 'global.customCertificates.pemData'="$CERT_DATA" \
  suse-observability-agent suse-observability/suse-observability-agent
----

==== Verwendung der ConfigMap-Referenz

Bei der ConfigMap-Methode erstellen Sie die ConfigMap, die Ihr Zertifikat enthält:

[,bash]
----
kubectl create configmap tls-config \
  --from-file=tls.crt=your-certificate.crt \
  --namespace suse-observability
----

Stellen Sie den Agenten mit der ConfigMap-Referenz bereit:

[,bash]
----
helm upgrade --install \
  --namespace suse-observability \
  --create-namespace \
  --set-string 'stackstate.apiKey'='YOUR_API_KEY' \
  --set-string 'stackstate.cluster.name'='YOUR_CLUSTER_NAME' \
  --set-string 'stackstate.url'='YOUR_SUSE_OBSERVABILITY_URL' \
  --set 'global.customCertificates.enabled'=true \
  --set 'global.customCertificates.configMapName'='tls-config' \
  suse-observability-agent suse-observability/suse-observability-agent
----

== SUSE Observability CLI

Die xref:setup/cli/cli-sts.adoc[SUSE Observability-Befehlszeilenschnittstelle] verbindet sich über HTTPS mit dem SUSE Observability-Server. Wenn Ihr Server selbstsignierte Zertifikate oder Zertifikate von einer privaten Zertifizierungsstelle (CA) verwendet, konfigurieren Sie die CLI so, dass sie diesen Zertifikaten vertraut.

=== Benutzerdefinierte CA-Zertifikate konfigurieren

Konfigurieren Sie benutzerdefinierte CA-Zertifikate mit einer der folgenden Methoden:

* **Dauerhafte Konfiguration**: Verwenden Sie `sts context save`, um die Zertifikatskonfiguration für zukünftige Befehle zu speichern.
* **Einmalige Verwendung**: Hinzufügen von Zertifikatsflags zu einzelnen CLI-Befehlen bei Bedarf

==== Methode 1: Pfad der CA-Zertifikatsdatei

Geben Sie den Pfad zu Ihrer PEM-kodierten CA-Zertifikatsdatei an:

[,bash]
----
sts context save \
  --name staging \
  --url https://staging.internal \
  --api-token YOUR_API_TOKEN \
  --ca-cert-path /path/to/ca.crt
----

==== Methode 2: Base64-kodierte CA-Zertifikatsdaten

Geben Sie die Daten des CA-Zertifikats als base64-kodierte Zeichenfolge an:

[,bash]
----
sts context save \
  --name staging \
  --url https://staging.internal \
  --api-token YOUR_API_TOKEN \
  --ca-cert-base64-data BASE64_ENCODED_CERTIFICATE_DATA
----

==== Verwendung von CA-Zertifikaten mit anderen Befehlen

Verwenden Sie Zertifikatsflags mit jedem CLI-Befehl zur einmaligen Zertifikatsvalidierung:

[,bash]
----
# Using certificate file path
sts agent list \
  --url https://staging.internal \
  --api-token YOUR_API_TOKEN \
  --ca-cert-path /path/to/ca.crt

# Using base64-encoded certificate data
sts settings list \
  --url https://staging.internal \
  --api-token YOUR_API_TOKEN \
  --ca-cert-base64-data BASE64_ENCODED_CERTIFICATE_DATA
----

=== Vorrang der Konfiguration

Wenn beide Zertifikatsoptionen angegeben werden, hat der Dateipfad (`--ca-cert-path`) Vorrang vor den base64-Daten (`--ca-cert-base64-data`).

=== Lagerung

Zertifikatskonfigurationen werden in gespeichert: `~/.config/stackstate-cli/config.yaml`

[NOTE]
====
**Das ist wichtig**: Das Flag `--skip-ssl` deaktiviert die gesamte SSL-Überprüfung und ignoriert die Zertifikatskonfigurationen. Verwenden Sie für sichere Verbindungen mit benutzerdefinierten Zertifikaten immer die Optionen für CA-Zertifikate.
====

== SUSE Observability Offene Telemetrie

=== Erstellen Sie eine configmap für die benutzerdefinierte Zertifizierungsstelle

----
kubectl create configmap tls-config \
  --namespace open-telemetry \
  --from-file=tls.crt=<your-certificate>
----

. Binden Sie die configmap ein, indem Sie auf der obersten Ebene von `otel-collector.yaml` Folgendes hinzufügen:

----
# Mounting volume for CA certificate
extraVolumes: 
  - name: tls-config
    configMap:
      items:
        - key: tls.crt
          path: tls.crt
      name: tls-config
extraVolumeMounts:
  - mountPath: "/certs"
    name: tls-config
    readOnly: true
----

. Aktualisieren Sie die Konfiguration des Exporteurs in `otel-collector.yaml`, um das Zertifikat *ca_file* zu verwenden, indem Sie den Schlüssel `tls.ca_file` hinzufügen:

----
otlp/suse-observability:
  auth:
    authenticator: bearertokenauth
  # Put in your own otlp endpoint, for example suse-observability.my.company.com:443
  endpoint: <your-endpoint>
  compression: snappy
  tls:
    ca_file: /certs/tls.crt
----

. Installieren (oder aktualisieren) Sie Ihren Open Telemetry-Kollektor.

== Rancher UI-Erweiterung für SUSE Observability

Wenn Sie die Rancher UI-Erweiterung für SUSE Observability installieren (siehe xref:/k8s-suse-rancher-prime.adoc#_installing_ui_extensions[Installation von UI-Erweiterungen]), muss die Erweiterung mit Ihrem SUSE Observability-Server kommunizieren. Wenn Ihr Server selbstsignierte Zertifikate verwendet, wird die Installation der Erweiterung fehlschlagen.

**Lösung**: Fügen Sie Ihr eigenes Zertifikat zu Rancher hinzu, bevor Sie die Erweiterung installieren. Folgen Sie der Rancher-Dokumentation: https://ranchermanager.docs.rancher.com/getting-started/installation-and-upgrade/resources/custom-ca-root-certificates[configuring custom CA root certificates^].

Nach der Konfiguration des Zertifikats in Rancher stellt die Erweiterung erfolgreich eine Verbindung zu Ihrem SUSE Observability Server her.
