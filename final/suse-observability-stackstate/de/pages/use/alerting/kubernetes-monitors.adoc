== Sofort einsatzbereite Kubernetes-Monitore

=== Verfügbare Dienstendpunkte

Es ist wichtig sicherzustellen, dass Ihre Dienste für die Nutzer verfügbar und zugänglich sind. Um dies zu überwachen, hat SUSE Observability einen Check eingerichtet, der prüft, ob für einen Service mindestens ein Endpunkt verfügbar ist. Endpunkte sind Netzwerkadressen, die die Kommunikation zwischen verschiedenen Komponenten in einem verteilten System ermöglichen, und sie müssen verfügbar sein, damit der Dienst ordnungsgemäß funktioniert.
Wenn innerhalb der letzten 10 Minuten keine Endpunkte mehr verfügbar sind, bleibt der Monitor abweichend und zeigt an, dass möglicherweise ein Problem mit dem Dienst vorliegt, das behoben werden muss.
Erlaubt xref:/use/alerting/k8s-override-monitor-arguments.adoc[Override Monitor Argumente]

=== Cpu-Grenzen Ressourcenkontingent

Benutzer erstellen Ressourcen (Pods, Dienste usw.) im Namensraum, und das Quotensystem verfolgt die Nutzung, um sicherzustellen, dass die in einem ResourceQuota definierten harten Ressourcengrenzen für Cpu nicht überschritten werden. Der Monitor gibt eine Warnung aus, wenn die Gesamtzahl der Cpu-Limits im Namensraum 90 % oder mehr des durch die Quote festgelegten Wertes erreicht. Jedes `resourcequota` im Namespace erzeugt einen Monitor-Zustand.

=== Daemonset gewünschte Repliken

Es ist wichtig, dass die gewünschte Anzahl von Replikaten für ein Daemonset eingehalten wird. Daemonsets werden verwendet, um eine Reihe von Pods zu verwalten, die auf allen oder einer Untergruppe von Knoten in einem Cluster ausgeführt werden müssen, wobei sichergestellt wird, dass auf jedem Knoten, der die angegebenen Kriterien erfüllt, eine Kopie des Pods ausgeführt wird. Dies ist nützlich für Aufgaben wie Protokollierung, Überwachung und andere Aufgaben auf Clusterebene, die auf jedem Knoten des Clusters ausgeführt werden müssen. 

Um dies zu überwachen, hat SUSE Observability einen Check eingerichtet, der prüft, ob die verfügbaren Replikate mit der gewünschten Anzahl von Replikaten übereinstimmen. Diese Prüfung wird nur auf DaemonSets angewendet, die eine gewünschte Anzahl von Replikaten größer als Null haben. 

* Wenn die Anzahl der verfügbaren Replikate geringer ist als die gewünschte Anzahl, meldet der Monitor einen DEVIATING-Zustand, was auf ein Problem mit dem StatefulSet hinweist.
* Ist die Anzahl der verfügbaren Replikate gleich Null, meldet der Monitor einen CRITICAL-Zustand, was bedeutet, dass das StatefulSet überhaupt nicht funktioniert. 

Um die vollständige Definition des Monitors zu verstehen, lesen Sie die Details.

=== HTTP - 5xx Fehlerquote

HTTP-Antworten mit einem Statuscode im Bereich 5xx weisen auf serverseitige Fehler wie Fehlkonfiguration, Überlastung oder interne Serverfehler hin.
Um ein gutes Nutzererlebnis zu gewährleisten, sollte der Prozentsatz der 5xx-Antworten weniger als 5 % der gesamten HTTP-Antworten für einen Kubernetes-Dienst betragen.

Da der genaue Schwellenwert und der Schweregrad anwendungsabhängig sein können, können die Schwellenwerte xref:/use/alerting/k8s-override-monitor-arguments.adoc[über eine Kubernetes-Anmerkung] zum Dienst xref:/use/alerting/k8s-override-monitor-arguments.adoc[überschrieben] werden. Um zum Beispiel die vorkonfigurierte Abweichungsschwelle außer Kraft zu setzen und stattdessen nur eine kritische Schwelle von 6 % zu haben, fügen Sie diese Anmerkung zu Ihrem Dienst hinzu:
```
monitor.kubernetes-v2.stackstate.io/http-error-ratio-for-service: | 
  { 
    "criticalThreshold": 0.06,
    "deviatingThreshold": null
  }
```

Das Weglassen des abweichenden Schwellenwerts in diesem json-Snippet hätte den konfigurierten Wert von 5 % beibehalten, wobei der kritische Schwellenwert bei 6 % liegt, was bedeutet, dass die Überwachung nur bei einem Fehlerverhältnis zwischen 5 % und 6 % zu einem abweichenden Zustand führen würde.

=== HTTP - Antwortzeit - Q95 liegt über 3 Sekunden

Die Verfolgung der 95. Perzentile (Q95) der HTTP-Antwortzeit für Ihre Kubernetes-Dienste hilft Ihnen, langsame Anfragen und Leistungsausreißer zu erkennen, die die Benutzer beeinträchtigen könnten. Wenn die Reaktionszeit des Q95 innerhalb eines bestimmten Zeitfensters mehr als 3 Sekunden beträgt, gibt das Überwachungsgerät einen abweichenden Status aus.

Sie können diesen Monitor an die Bedürfnisse Ihrer Anwendung anpassen, indem Sie xref:/use/alerting/k8s-override-monitor-arguments.adoc[die Standardeinstellungen], wie z. B. das Zeitfenster, den Schwellenwert oder das Quantil, mithilfe einer Kubernetes-Anmerkung zu Ihrem Dienst xref:/use/alerting/k8s-override-monitor-arguments.adoc[überschreiben]. Diese Flexibilität ist nützlich, wenn Ihr Dienst besondere Leistungsanforderungen hat. Zum Beispiel lange Abrufdienste.

Um den Monitor anzupassen, fügen Sie einen Vermerk wie den folgenden zu Ihrem Dienst hinzu und passen Sie die Werte nach Bedarf an:

```
monitor.kubernetes-v2.stackstate.io/http-response-time/overrides: |
  {
    "deviatingTimeWindow": "10m",            // Time window for a deviating state (e.g., "10m", "1h")
    "deviatingThreshold": 1.2,               // Threshold in seconds for a deviating state
    "criticalTimeWindow": "30m",             // Time window for a critical state
    "criticalThreshold": 2.0,                // Threshold in seconds for a critical state
    "quantile": 0.99,                        // Quantile to monitor (e.g., 0.95, 0.99)
    "nameTemplate": "API latency (p{{ quantile }}) > {{ threshold }}", // Custom monitor name
    "enabled": true                          // Set to false to disable this monitor for the service
  }
```

- Sie können jedes Feld auslassen, um seinen Standardwert zu verwenden.
- Der Monitor wertet das gewählte Quantil der Antwortzeit über das angegebene Fenster aus. Liegt der Wert über dem Schwellenwert, wird der Monitor ausgelöst.
- Im obigen Beispiel signalisiert der Monitor einen abweichenden Zustand, wenn die 99. Perzentil-Antwortzeit (Q99) über 1,2 Sekunden liegt (über die letzten 10 Minuten), und einen kritischen Zustand, wenn sie über 2,0 Sekunden liegt (über die letzten 30 Minuten).

=== Druck der Knotenscheibe

Der Druck auf Knotenfestplatten bezieht sich auf eine Situation, in der die an einen Knoten angeschlossenen Festplatten einer übermäßigen Belastung ausgesetzt sind. Auch wenn es aufgrund der in Kubernetes eingebauten Vorbeugungsmaßnahmen unwahrscheinlich ist, dass die Festplatte eines Knotens unter Druck gerät, kann dies dennoch sporadisch auftreten. Es gibt zwei Hauptgründe, warum der Druck in den Bandscheiben entstehen kann. Der erste Grund ist, dass Kubernetes unbenutzte Images nicht bereinigt. Unter normalen Umständen prüft Kubernetes regelmäßig, ob Images vorhanden sind, die nicht verwendet werden, und löscht diese. Es handelt sich also um eine seltene Ursache für den Druck auf die Bandscheibe, die aber nicht außer Acht gelassen werden sollte. Wahrscheinlicher ist die Ansammlung von Protokollen. In Kubernetes werden Protokolle in der Regel in zwei Szenarien gespeichert: wenn Container ausgeführt werden und wenn die Protokolle des zuletzt beendeten Containers zu Fehlerbehebungszwecken aufbewahrt werden. Dieser Ansatz zielt darauf ab, ein Gleichgewicht zwischen der Bewahrung wichtiger Protokolle und der Beseitigung unnötiger Protokolle im Laufe der Zeit herzustellen. Wenn jedoch ein lang laufender Container eine große Menge an Protokollen erzeugt, können sich diese bis zu dem Punkt ansammeln, an dem sie die Kapazität der Knotenplatte überlasten. Um die vollständige Definition des Monitors zu verstehen, lesen Sie die Details.

=== Knotenspeicherdruck

Kubernetes-Speicherdruck bezieht sich auf eine Situation, in der die Speicherressourcen eines Kubernetes-Knotens übermäßig beansprucht werden. Aufgrund der in Kubernetes eingebauten Ressourcenverwaltungsmechanismen ist ein Druck auf den Knotenspeicher zwar ungewöhnlich, kann aber unter bestimmten Umständen dennoch auftreten. Es gibt zwei Hauptgründe für die Entstehung von Knotenspeicherdruck. Der erste Grund hängt mit falsch konfigurierten oder unzureichenden Ressourcenanforderungen und -beschränkungen für die auf dem Knoten laufenden Container zusammen. Kubernetes stützt sich auf Ressourcenanfragen und -grenzen, um Ressourcen effektiv zuzuweisen und zu verwalten. Wenn Container nicht genau mit ihren Speicheranforderungen konfiguriert sind, können sie mehr Speicher als erwartet verbrauchen, was zu Speicherdruck im Knoten führt. Der zweite Grund ist das Vorhandensein von speicherintensiven Anwendungen oder Prozessen. Bestimmte Arbeitslasten oder Anwendungen können höhere Speicheranforderungen haben, was zu einer erhöhten Speichernutzung auf dem Knoten führt. Wenn mehrere Pods oder Container mit hohem Speicherbedarf auf demselben Knoten geplant werden, ohne dass eine angemessene Ressourcenzuweisung erfolgt, kann dies zu Speicherdruck führen. Um den Druck auf den Knotenspeicher zu mindern, ist es wichtig, die Ressourcenanforderungen und -grenzen für Container zu überprüfen und anzupassen, um sicherzustellen, dass sie mit dem tatsächlichen Speicherbedarf der Anwendungen übereinstimmen. Die Überwachung und Optimierung der Speichernutzung in den Anwendungen selbst kann ebenfalls zur Verringerung des Speicherverbrauchs beitragen. Ziehen Sie außerdem eine horizontale Pod-Autoskalierung in Betracht, um die Anzahl der Pods dynamisch auf der Grundlage der Speichernutzung zu skalieren. Regelmäßige Überwachung, Analyse von speicherbezogenen Metriken und proaktive Zuweisung von Speicherressourcen können dazu beitragen, einen gesunden Speicherstatus auf Kubernetes-Knoten aufrechtzuerhalten. Es ist wichtig, die spezifischen Anforderungen Ihrer Workloads zu verstehen und die Ressourcenzuweisung entsprechend anzupassen, um Speicherdruck zu vermeiden und optimale Leistung zu gewährleisten.
Erlaubt xref:/use/alerting/k8s-override-monitor-arguments.adoc[Override Monitor Argumente]

=== Knotenpunkt PID Druck

Knoten-PID-Druck tritt auf, wenn die verfügbaren Ressourcen zur Prozessidentifizierung (PID) auf einem Kubernetes-Knoten übermäßig belastet werden. Der erste Grund hängt mit falsch konfigurierten oder unzureichenden Ressourcenanforderungen und -beschränkungen für die auf dem Knoten laufenden Container zusammen. Kubernetes ist auf genaue Ressourcenanforderungen und -begrenzungen angewiesen, um Ressourcen effektiv zuzuweisen und zu verwalten. Wenn Container nicht korrekt mit ihren PID-Anforderungen konfiguriert sind, können sie mehr PIDs als erwartet verbrauchen, was zu einem PID-Druck auf den Knoten führt. Der zweite Grund ist das Vorhandensein von PID-intensiven Anwendungen oder Prozessen. Einige Arbeitslasten oder Anwendungen stellen höhere Anforderungen an die Prozessidentifizierung, was zu einer erhöhten PID-Auslastung auf dem Knoten führt. Wenn mehrere Pods oder Container mit erheblichen PID-Anforderungen auf demselben Knoten ohne angemessene Ressourcenzuweisung eingeplant werden, kann dies zu PID-Druck führen. Um dem PID-Druck der Knoten entgegenzuwirken, ist es wichtig, die Ressourcenanforderungen und -grenzen für Container zu überprüfen und anzupassen, um sicherzustellen, dass sie mit dem tatsächlichen PID-Bedarf der Anwendungen übereinstimmen. Auch die Überwachung und Optimierung der PID-Nutzung in den Anwendungen selbst kann dazu beitragen, den PID-Verbrauch zu senken. Darüber hinaus kann die Anzahl der Pods unter Berücksichtigung der horizontalen Pod-Autoskalierung dynamisch anhand der PID-Auslastung skaliert werden. Regelmäßige Überwachung, Analyse von PID-bezogenen Metriken und proaktive Zuweisung von PID-Ressourcen sind entscheidend für die Aufrechterhaltung eines gesunden Zustands der PID-Nutzung auf Kubernetes-Knoten. Es ist wichtig, die spezifischen Anforderungen Ihrer Arbeitslasten zu verstehen und die Ressourcenzuweisung entsprechend anzupassen, um PID-Druck zu vermeiden und optimale Leistung zu gewährleisten.

=== Neustarts für Container

Es ist wichtig, die Neustarts für jeden Container in einem Kubernetes-Cluster zu überwachen. Container können aus einer Vielzahl von Gründen neu gestartet werden, z. B. bei Problemen mit der Anwendung oder der Infrastruktur.

Wenn innerhalb dieses Zeitraums mehr als 3 Neustarts erfolgen, wird der Gesundheitszustand des Containers auf DEVIATING gesetzt, was darauf hinweist, dass möglicherweise ein Problem vorliegt, das untersucht werden muss.

=== Statefulset gewünschte Replikate

Es ist wichtig, dass die gewünschte Anzahl von Replikaten für ein StatefulSet eingehalten wird. StatefulSets werden zur Verwaltung zustandsabhängiger Anwendungen verwendet und erfordern eine bestimmte Anzahl von Replikaten, um ordnungsgemäß zu funktionieren.

Um dies zu überwachen, hat SUSE Observability einen Check eingerichtet, der prüft, ob die verfügbaren Replikate mit der gewünschten Anzahl von Replikaten übereinstimmen. Diese Prüfung wird nur auf StatefulSets angewendet, die eine gewünschte Anzahl von Replikaten größer als Null haben.

* Wenn die Anzahl der verfügbaren Replikate geringer ist als die gewünschte Anzahl, meldet der Monitor einen DEVIATING-Zustand, was auf ein Problem mit dem StatefulSet hinweist.
* Ist die Anzahl der verfügbaren Replikate gleich Null, meldet der Monitor einen CRITICAL-Zustand, was bedeutet, dass das StatefulSet überhaupt nicht funktioniert.

=== Unschedulierbarer Knoten

Wenn Sie in Kubernetes auf ein "NodeNotSchedulable"-Ereignis stoßen, bedeutet dies, dass der Kubernetes-Scheduler einen Pod aufgrund von Einschränkungen oder Problemen mit dem Knoten nicht auf einem bestimmten Knoten platzieren konnte. Dieses Ereignis tritt ein, wenn der Scheduler keinen geeigneten Knoten für die Ausführung des Pods entsprechend den Ressourcenanforderungen und anderen Einschränkungen findet.

=== Aggregierter Gesundheitszustand eines Clusters

Der Cluster hat selbst keine Gesundheit. Ein Cluster besteht jedoch aus wenigen Komponenten, von denen einige für den normalen Betrieb entscheidend sind. Der Monitor aggregiert die Zustände dieser Komponenten:

* alle Pods im Namensraum 'kube-system'
* alle Knoten und nimmt dann den kritischsten Gesundheitszustand.

=== Abgeleiteter Workload-Zustandsstatus (Bereitstellung, DaemonSet, ReplicaSet, StatefulSet)

Der Monitor aggregiert die Zustände aller obersten Abhängigkeiten und gibt dann den kritischsten Gesundheitszustand auf der Grundlage direkter Beobachtungen (z. B. aus Metriken) zurück.
Dieser Ansatz stellt sicher, dass Gesundheitssignale von technischen Komponenten auf niedriger Ebene (z. B. Pods) an logische Komponenten auf höherer Ebene weitergegeben werden, aber nur, wenn die Komponente selbst keinen beobachteten Gesundheitszustand aufweist.
Um diesen Monitor effektiv zu nutzen, stellen Sie sicher, dass einige oder alle der folgenden Gesundheitsprüfungen deaktiviert sind:

* Gewünschter Einsatz von Replikaten
* DaemonSet gewünschte Replikate
* ReplicaSet gewünschte Replikate
* StatefulSet gewünschte Repliken

Wenn Sie einen Anwendungsfall haben, in dem logische Komponenten keine direkten Monitore haben, können Sie die Funktion xref:/use/alerting/k8s-derived-state-monitors.adoc[Abgeleiteter Zustandsmonitor] verwenden, um ihren Zustand auf der Grundlage der technischen Komponenten abzuleiten, von denen sie abhängen.
