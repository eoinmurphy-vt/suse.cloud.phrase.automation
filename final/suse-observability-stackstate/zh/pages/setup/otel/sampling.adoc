= 取样
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: SUSE 可观察性

采样用于减少输出到 SUSE Observability 的数据量，同时尽可能不影响遥测数据的质量。应用采样的主要原因是降低成本（网络、存储等）。

如果您的应用程序产生的数据很少，就没有必要进行采样，而且由于缺乏遥测数据，采样甚至会妨碍可观测性。但是，如果您的应用流量很大，例如每秒超过 1000 个跨度，那么应用采样就很有意义了。

主要有两种取样方式：头部取样和尾部取样。

== 头部取样

头部取样会尽早做出取样决定（是否输出数据）。因此，决定不能基于整个跟踪，而只能基于非常有限的现有信息。otel 收集器拥有https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/processor/probabilisticsamplerprocessor[概率采样处理器]，它实现了一致性概率采样。采样器可进行配置，并根据跟踪 ID（适用于跟踪）或属性哈希值（适用于日志）做出采样决定。这可确保跟踪的所有跨度始终采样或不采样，从而在 SUSE Observability 中获得完整的跟踪。

头部取样的优势在于

* 简单易懂
* 高效
* 配置简单

但其缺点是无法对整个轨迹进行采样决策，例如，无法对所有失败轨迹进行采样，只能对一小部分成功轨迹进行采样。

要启用头部采样，请配置处理器并将其纳入流水线。此示例根据轨迹 ID 从 4 个轨迹中采样 1 个：

[,yaml]
----
processors:
  probabilistic_sampler:
    sampling_percentage: 25
    mode: "proportional"
----

== 尾部取样

尾部取样将取样决定推迟到（几乎）完成跟踪之后。这样，尾部采样器就能根据整个轨迹做出采样决定，例如始终对故障轨迹和/或慢速轨迹进行采样。还有更多的可能性，例如根据特定属性或服务参与度进行抽样。

OpenTelemetry Collector 提供https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/processor/tailsamplingprocessor[尾部采样处理器]，可用于应用尾部采样策略。

尾部取样的主要优势在于它提供了额外的灵活性，可以根据完整的痕量数据而不是创建单个跨度时的部分信息来做出取样决定。


[WARNING]
====
尾部取样虽然有利，但也有以下缺点：

* 更难配置和理解。
* 必须是有状态的，以便在做出采样决定之前存储轨迹跨度。
* 因此使用的资源也更多（更多
* 采样器可能跟不上，因此需要额外的监控和缩放。
====

要启用尾部采样，请配置处理器并将其纳入流水线。

[,yaml]
----
processors:
  tail_sampling:
    decision_wait: 10s
    policies:
    - name: rate-limited-composite
      type: composite
      composite:
        max_total_spans_per_second: 500
        policy_order: [errors, slow-traces, rest]
        composite_sub_policy:
        - name: errors
          type: status_code
          status_code:
            status_codes: [ ERROR ]
        - name: slow-traces
          type: latency
          latency:
            threshold_ms: 1000
        - name: rest
          type: always_sample
        rate_allocation:
        - policy: errors
          percent: 33
        - policy: slow-traces
          percent: 33
        - policy: rest
          percent: 34
----

示例样本：

* 每秒最多 500 个跨度
* 误差不超过最大误差 33% 的轨迹中的所有跨度
* 慢于 1 秒的轨迹中的所有跨度，最高可达最大跨度的 33
* 其他跨度达到允许的最大跨度

有关配置选项和不同策略的更多详情，请参阅https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/processor/tailsamplingprocessor[tail sampling readme]。

不过，它也不是完全设置好了就不管了。如果资源使用量开始增加，就需要扩展使用多个收集器来处理尾部采样，这就需要https://github.com/open-telemetry/opentelemetry-collector-contrib/blob/main/connector/routingconnector/README.md[路由]，根据跟踪 ID 来路由流量。

== 结合跨度指标的采样轨迹

在入门部分，采集器配置不包括采样。在添加采样时，我们要小心谨慎，尽可能保持根据轨迹计算出的指标的准确性。特别是尾部取样会导致指标非常不稳定，因为通常情况下，相对误差量要大得多。为避免出现这种情况，我们将跟踪管道分成多个部分，并用前向连接器将它们连接起来。修改配置，加入额外的连接器和采样处理器。然后修改管道，如图所示：

[,yaml]
----
connectors:
  # enable the forwarder
  forward:
processors:
  # Configure the probabilistic sampler to sample 25% of the traffic
  probabilistic_sampler:
    sampling_percentage: 25
    mode: "proportional"
service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [memory_limiter, resource]
      exporters: [forward]
    traces/spanmetrics:
      receivers: [forward]
      processors: []
      exporters: [spanmetrics]
    traces/sampling:
      receivers: [forward]
      processors: [probabilistic_sampler, batch]
      exporters: [debug, otlp/stackstate]
    metrics:
      receivers: [otlp, spanmetrics, prometheus]
      processors: [memory_limiter, resource, batch]
      exporters: [debug, otlp/stackstate]
----

示例中使用的概率采样器配置为对 25% 的流量进行采样。您可能需要根据自己的情况调整百分比，或者改用<<_tail_sampling,尾部采样器>>。尾部采样器的管道设置相同，只需将`probabilistic_sampler` 的引用替换为`tail_sampling` 。
