== 快速故障排除指南

以下是 SUSE Observability 启动故障排除快速指南：

. 检查安装是否成功完成，版本是否已列出：
+
[,text]
----
helm list --namespace suse-observability
----

. 检查 SUSE Observability 命名空间中的所有 pod 是否都在运行：
+
[,text]
----
kubectl get pods
----
+
在首次部署中，几个 pod 中的容器可能会重启几次，因为它们在等待其他 pod 启动并进入`ready` 状态。由于调度和 docker 映像拉取延迟，这可能会造成延迟。
+
处于`pending` 状态的豆荚通常表明出现了问题：

 ** 由于集群资源不足，该 pod 无法调度。如果群集自动分频器处于活动状态，通常可以自动解决这个问题，否则就需要人工干预，向群集添加更多节点
 ** pod 是不可调度的，有一些节点可以容纳它，但这些节点有`taints` ，pod 无法容忍。为了解决这个问题，可以添加更多没有污点的节点，但也可以将 SUSE Observabilityxref:/setup/install-stackstate/kubernetes_openshift/customize_config.adoc#_override_default_configuration[配置]为容忍某些污点，并在有污点的节点上运行。
 ** pod 正在等待挂载持久卷 (PV)。原因可能是 SUSE Observability Helm 图表没有指定`storageClassName` ，而是依赖于群集的默认存储类别。如果群集没有默认值，则需要通过 SUSE Observability 的 Helm 值xref:/setup/install-stackstate/kubernetes_openshift/storage.adoc[指定存储类]。

+
对于状态为`ImagePullBackOff` 的 pod，也要检查确切的错误信息，常见的原因有：

 ** 提取图像时使用的用户名/密码不正确
 ** 连接到 docker 注册表失败，这可能是由于身份验证问题或连接问题（防火墙、隔空安装）造成的
 ** 覆盖的 docker 镜像注册表 URL 中的一个错字

+
要查找`Pending` 、`ImagePullBackOff` 或`CrashLoopBackOff` 状态的更详细原因，请使用此命令：
+
[,text]
----
kubectl describe pod <pod-name>
----
+
输出结果的末尾包含一个`event` 部分，通常包含问题。它还为每个容器提供了`State` 部分，其中包含终止容器的更多详细信息。

. 如果您是主要客户，请通过https://scc.suse.com/ 联系 SUSE Observability 支持部门，获取在本地集群中设置 SUSE Observability 的帮助。使用xref:/setup/install-stackstate/support-package-logs.adoc[支持软件包（日志）]为支持团队收集有关实例的信息。
. 如果问题与性能有关，请运行xref:/setup/install-stackstate/support-package-performance.adoc[支持软件包（性能）]以积极调查性能。
. 如果上述步骤仍不能解决问题，可参考《xref:/setup/install-stackstate/advanced-troubleshooting.adoc[高级故障排除指南》]。
