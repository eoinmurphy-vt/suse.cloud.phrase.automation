== 概述

{stackstate-product-name} 它有一个内置的备份和还原机制，可以配置为将备份存储到本地集群、AWS S3 或 Azure Blob Storage。

=== 备份范围

可以自动备份以下数据：

* 存储在 StackGraph 中的*配置和拓扑数据*
* 存储在 SUSE Observability 的 Victoria Metrics 实例中的*度量指标*
* *遥测数据*存储在 SUSE Observability 的 Elasticsearch 实例中
* 存储在 SUSE Observability 的 ClickHouse 实例中的*OpenTelemetry 数据*

将*不*备份以下数据：

* 存储在 Kafka 中的传输中拓扑和遥测更新 - 这些更新只有临时价值，在恢复备份时没有用处
* 存储在 ZooKeeper 中的主节点谈判状态\--该运行时状态在恢复时可能不正确，将在运行时自动确定
* Kubernetes 配置状态和原始持久卷状态\--可通过重新安装 SUSE Observability 和恢复备份来重建该状态。
* Kubernetes 日志 - 这些都是短暂的。

=== 存储选项

备份会被发送到https://min.io/[MinIO (min.io)] 实例，当启用自动备份时，`suse-observability` Helm 图表会自动启动该实例。MinIO 是一种对象存储系统，其 API 与 AWS S3 相同。它可以在本地存储数据，也可以充当https://docs.min.io/docs/minio-gateway-for-s3.html[AWS S3 (min.io)]、https://docs.min.io/docs/minio-gateway-for-azure.html[Azure BLob Storage (min.] 和其他系统的网关。

内置 MinIO 实例可配置为将备份存储在三个位置：

* xref:/setup/data-management/backup_restore/kubernetes_backup.adoc[AWS S3]
* xref:/setup/data-management/backup_restore/kubernetes_backup.adoc[Azure Blob 存储]
* xref:/setup/data-management/backup_restore/kubernetes_backup.adoc[Kubernetes 存储]

== 启用备份

=== AWS S3

[CAUTION]
====

*加密*

在加密存储备份的 S3 存储桶时，应使用 Amazon S3 管理的密钥 (SSE-S3)。

⚠️ 不支持使用存储在 AWS Key Management Service (SSE-KMS) 中的 AWS KMS 密钥进行加密。这将导致 Elasticsearch 日志中出现类似这样的错误：

`Caused by: org.elasticsearch.common.io.stream.NotSerializableExceptionWrapper: sdk_client_exception: Unable to verify integrity of data upload. Client calculated content hash (contentMD5: ZX4D/ZDUzZWRhNDUyZTI1MTc= in base 64) didn't match hash (etag: c75faa31280154027542f6530c9e543e in hex) calculated by Amazon S3. You may need to delete the data stored in Amazon S3. (metadata.contentMD5: null, md5DigestStream: com.amazonaws.services.s3.internal.MD5DigestCalculatingInputStream@5481a656, bucketName: suse-observability-elasticsearch-backup, key: tests-UG34QIV9s32tTzQWdPsZL/master.dat)",`
====


要启用到 AWS S3 存储桶的计划备份，请在用于安装 SUSE Observability 的 Helm`values.yaml` 文件中添加以下 YAML 片段：

[,yaml]
----
global:
  backup:
    enabled: true
backup:
  stackGraph:
    bucketName: AWS_STACKGRAPH_BUCKET
  elasticsearch:
    bucketName: AWS_ELASTICSEARCH_BUCKET
  configuration:
    bucketName: AWS_CONFIGURATION_BUCKET
victoria-metrics-0:
  backup:
    bucketName: AWS_VICTORIA_METRICS_BUCKET
victoria-metrics-1:
  backup:
    bucketName: AWS_VICTORIA_METRICS_BUCKET
clickhouse:
  backup:
    bucketName: AWS_CLICKHOUSE_BUCKET
minio:
  accessKey: YOUR_ACCESS_KEY
  secretKey: YOUR_SECRET_KEY
  s3gateway:
    enabled: true
    accessKey: AWS_ACCESS_KEY
    secretKey: AWS_SECRET_KEY
----

替换以下值：

* `YOUR_ACCESS_KEY` 和`YOUR_SECRET_KEY` 是用于确保 MinIO 系统安全的凭证。这些凭证在 MinIO 系统上设置，并被自动备份任务和还原任务使用。如果要手动访问 MinIO 系统，也需要它们。
 ** YOUR_ACCESS_KEY 应包含 5 至 20 个字母数字字符。
 ** YOUR_SECRET_KEY 应包含 8 至 40 个字母数字字符。
* `AWS_ACCESS_KEY` 和`AWS_SECRET_KEY` 是 IAM 用户的 AWS 凭据，该用户有权访问存储备份的 S3 存储桶。请参阅下文，了解需要附加给该用户的权限策略。
* `AWS_STACKGRAPH_BUCKET`,`AWS_CONFIGURATION_BUCKET`,`AWS_ELASTICSEARCH_BUCKET`,`AWS_VICTORIA_METRICS_BUCKET` 和`AWS_CLICKHOUSE_BUCKET` 是应存储备份的 S3 存储桶的名称。请注意：AWS S3 存储桶的名称是整个 AWS 的全局名称，因此，默认名称为 (`sts-elasticsearch-backup`,`sts-configuration-backup`,`sts-stackgraph-backup`,`sts-victoria-metrics-backup` 和`sts-clickhouse-backup` ) 的 S3 存储桶可能无法使用。

`AWS_ACCESS_KEY` 和`AWS_SECRET_KEY` 标识的 IAM 用户必须配置以下权限策略才能访问 S3 存储桶：

[,javascript]
----
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "AllowListMinioBackupBuckets",
            "Effect": "Allow",
            "Action": [
                "s3:ListBucket",
                "s3:GetBucketLocation"
            ],
            "Resource": [
                "arn:aws:s3:::AWS_STACKGRAPH_BUCKET",
                "arn:aws:s3:::AWS_ELASTICSEARCH_BUCKET",
                "arn:aws:s3:::AWS_VICTORIA_METRICS_BUCKET",
                "arn:aws:s3:::AWS_CLICKHOUSE_BUCKET",
                "arn:aws:s3:::AWS_CONFIGURATION_BUCKET"
            ]
        },
        {
            "Sid": "AllowWriteMinioBackupBuckets",
            "Effect": "Allow",
            "Action": [
                "s3:PutObject",
                "s3:GetObject",
                "s3:DeleteObject"
            ],
            "Resource": [
                "arn:aws:s3:::AWS_STACKGRAPH_BUCKET/*",
                "arn:aws:s3:::AWS_ELASTICSEARCH_BUCKET/*",
                "arn:aws:s3:::AWS_VICTORIA_METRICS_BUCKET/*",
                "arn:aws:s3:::AWS_CLICKHOUSE_BUCKET/*",
                "arn:aws:s3:::AWS_CONFIGURATION_BUCKET"
            ]
        }
    ]
}
----

=== Azure Blob 存储

要启用备份到 Azure Blob Storage 账户，请在用于安装 SUSE Observability 的 Helm`values.yaml` 文件中添加以下 YAML 片段：

[,yaml]
----
global:
  backup:
    enabled: true
minio:
  accessKey: AZURE_STORAGE_ACCOUNT_NAME
  secretKey: AZURE_STORAGE_ACCOUNT_KEY
  azuregateway:
    enabled: true
----

替换以下值：

* `AZURE_STORAGE_ACCOUNT_NAME` -https://learn.microsoft.com/en-us/azure/storage/common/storage-account-create?tabs=azure-portal[Azure 存储账户名称（learn.microsoft.com）]
* `AZURE_STORAGE_ACCOUNT_KEY` -https://learn.microsoft.com/en-us/azure/storage/common/storage-account-keys-manage?tabs=azure-portal[Azure 存储账户密钥（learn.microsoft.com）]，备份应存储在该账户中。

StackGraph、Elasticsearch 和 Victoria Metrics 备份分别存储在名为`sts-stackgraph-backup`,`sts-configuration-backup`,`sts-elasticsearch-backup`,`sts-victoria-metrics-backup`,`sts-clickhouse-backup` 的 BLOB 容器中。这些名称可分别通过设置 Helm 值`backup.stackGraph.bucketName`,`backup.elasticsearch.bucketName`,`victoria-metrics-0.backup.bucketName`,`victoria-metrics-1.backup.bucketName` 和`clickhouse.backup.bucketName` 进行更改。

=== Kubernetes 存储

[CAUTION]
====
如果 MinIO 被配置为将数据存储在 Kubernetes 存储中，则会使用 PersistentVolumeClaim (PVC) 向 Kubernetes 集群请求存储。分配的存储类型取决于群集的配置。

建议在 Amazon AWS 上运行的群集使用 AWS S3，在 Azure 上运行的群集使用 Azure Blob Storage，原因如下：

. 在云提供商中运行的 Kubernetes 集群通常会将 PVC 映射到块存储，如 AWS 的 Elastic Block Storage 或 Azure Block Storage。块存储的成本很高，尤其是对于大数据量而言。
. 当创建持久卷的群集被销毁时，持久卷也会被销毁。这意味着（意外）删除群集也会破坏存储在持久卷中的所有备份。
. 无法从另一个群集访问持久卷。这意味着无法从另一个群集上的备份恢复 SUSE Observability。
====


要启用备份到群集本地存储，请在用于安装 SUSE Observability 的 Helm`values.yaml` 文件中添加以下 YAML 片段，以启用 MinIO：

[,yaml]
----
global:
  backup:
    enabled: true
minio:
  accessKey: YOUR_ACCESS_KEY
  secretKey: YOUR_SECRET_KEY
  persistence:
    enabled: true
----

替换以下值：

* `YOUR_ACCESS_KEY` 和`YOUR_SECRET_KEY` - 用于确保 MinIO 系统安全的凭证。自动备份工作和还原工作将使用它们。`YOUR_ACCESS_KEY` 应包含 5 至 20 个字母数字字符，`YOUR_SECRET_KEY` 应包含 8 至 40 个字母数字字符。

== 配置和拓扑数据（StackGraph）

配置和拓扑数据（StackGraph）备份为完整备份，存储在扩展名为`.graph` 的单个文件中。每个文件都包含完整备份，可根据需要移动、复制或删除。

=== 备份时间表

默认情况下，StackGraph 备份在服务器时间每天凌晨 03:00 创建。

备份计划可使用https://kubernetes.io/docs/concepts/workloads/controllers/cron-jobs/#_cron_schedule_syntax[Kubernetes cron 计划语法（kubernetes.io）]中指定的 Helm 值`backup.stackGraph.scheduled.schedule` 进行配置。

=== 备份保留

默认情况下，StackGraph 备份保存 30 天。由于 StackGraph 备份是完全备份，因此需要大量存储空间。

备份保留 delta 可以使用 Helm 值`backup.stackGraph.scheduled.backupRetentionTimeDelta` 进行配置，该值以 GNU 日期`--date` 参数格式指定。例如，默认值为`30 days ago` 。更多示例请参阅https://www.gnu.org/software/coreutils/manual/html_node/Relative-items-in-date-strings.html[日期字符串中的相对项]。

=== 禁用计划备份

要禁用 StackGraph 计划备份，请使用 Helm 值`backup.stackGraph.scheduled.schedule` 将备份计划设置为远在过去的日期：

[,yaml]
----
backup:
  stackGraph:
    scheduled:
      schedule: '0 0 1 1 1970'  # January 1, 1970 (epoch start)
----

== 度量标准（维多利亚度量标准）

[WARNING]
====
Victoria Metrics 使用增量备份，不对数据桶进行版本控制，这意味着新的备份会*完全取代之前的*备份。

如果您遇到以下情况之一：

* 将空卷挂载到`/storage` Victoria Metrics 实例目录下
* 从 Victoria Metrics 实例中删除`/storage` 目录或其中的文件

创建的下一个（空）备份将标记为新版本，而清空卷之前的上一个备份将被保留。
从那时起，两个备份都将被<<_list_victoria_metrics_backups,列为可还原>>备份。
====


度量（Victoria Metrics）使用即时快照在增量备份中存储数据。Victoria Metrics 的多个实例可以将备份存储到同一个备份桶中，每个备份桶都将存储在不同的目录中。目录中的所有文件都应视为一个整体，只能作为一个整体进行移动、复制或删除。

[NOTE]
====
高可用部署应部署两个 Victoria Metrics 实例。每个备份都是独立启用/配置的。

以下代码片段/命令用于维多利亚度量衡`victoria-metrics-0` 的第一个实例。要备份/配置第二个实例，应使用 `victoria-metrics-1`
====


=== 备份时间表

默认情况下，每 1 小时创建一次 Victoria Metrics 备份：

* `victoria-metrics-0` - 整点 25 分
* `victoria-metrics-1` - 整点 35 分

可使用 Helm 值`victoria-metrics-0.backup.scheduled.schedule` 根据https://github.com/aptible/supercronic/tree/master/cronexpr[cronexpr 格式]配置备份计划

=== 禁用计划备份

要禁用计划中的 Victoria Metrics 备份，请将两个实例的备份计划都设置为远在过去的日期：

[,yaml]
----
victoria-metrics-0:
  backup:
    scheduled:
      schedule: '0 0 1 1 1970'  # January 1, 1970 (epoch start)
victoria-metrics-1:
  backup:
    scheduled:
      schedule: '0 0 1 1 1970'  # January 1, 1970 (epoch start)
----

== OpenTelemetry (ClickHouse)

ClickHouse 同时使用增量备份和完全备份。默认情况下，每天凌晨 00:45 执行完全备份，每小时执行增量备份。每次备份都会创建一个新目录，旧的备份（目录）会被自动删除。位于备份目录中的所有文件都被视为一个单独的组，只能作为一个组进行移动、复制或删除。建议使用`suse-observability-clickhouse-shard0-0` Pod 上的`clickhouse-backup` 工具管理备份。

=== 备份时间表

默认情况下，会创建 ClickHouse 备份：

* 全面备份 - 每天 00:45
* 增量备份 - 45 分钟后（从凌晨 3 点到上午 12 点）

[CAUTION]
====
备份在并行执行方面举步维艰。如果在第一个备份完成前启动第二个备份，则会中断第一个备份。因此，避免并行执行至关重要。例如，第一次增量备份应在完整备份三小时后执行。
====


可使用 Helm 值`clickhouse.backup.scheduled.full_schedule` 和`clickhouse.backup.scheduled.incremental_schedule` 根据https://github.com/aptible/supercronic/tree/master/cronexpr[cronexpr 格式]配置备份计划

=== 备份保留

默认情况下，工具会保留最后 308 次备份（完整和增量备份），相当于 14 天。

备份保留可通过 Helm 值`clickhouse.backup.config.keep_remote` 进行配置。

=== 禁用计划备份

要禁用计划中的 ClickHouse 备份，请将完整备份和增量备份计划设置为较远的过去日期：

[,yaml]
----
clickhouse:
  backup:
    scheduled:
      full_schedule: '0 0 1 1 1970'         # January 1, 1970 (epoch start)
      incremental_schedule: '0 0 1 1 1970'  # January 1, 1970 (epoch start)
----

== 遥测数据（Elasticsearch）

遥测数据（Elasticsearch）快照是递增的，存储在扩展名为`.dat` 的文件中。Elasticsearch 备份存储位置中的文件应视为一个整体，只能作为一个整体进行移动、复制或删除。

xref:/setup/data-management/backup_restore/kubernetes_backup.adoc#_enable_backups[启用备份]部分提供的配置片段将启用每日 Elasticsearch 快照。

=== 禁用计划快照

要禁用计划的 Elasticsearch 快照，请使用 Helm 值`backup.elasticsearch.scheduled.schedule` 将快照计划设置为远在过去的日期：

[,yaml]
----
backup:
  elasticsearch:
    scheduled:
      schedule: '0 0 1 1 1970'  # January 1, 1970 (epoch start)
----

=== 快照时间表

默认情况下，Elasticsearch 快照在服务器时间每天凌晨 03:00 创建。

备份计划可使用https://www.elastic.co/guide/en/elasticsearch/reference/7.6/cron-expressions.html[Elasticsearch cron 计划语法 (elastic.co)] 中指定的 Helm 值`backup.elasticsearch.scheduled.schedule` 进行配置。

=== 保留快照

默认情况下，Elasticsearch 快照保存 30 天，最少保存 5 个快照，最多保存 30 个快照。

可以使用以下 Helm 值配置快照的保留时间和数量：

* `backup.elasticsearch.scheduled.snapshotRetentionExpireAfter`，以https://www.elastic.co/guide/en/elasticsearch/reference/7.6/common-options.html#_time_units[Elasticsearch 时间单位 (elastic.co)] 指定。
* `backup.elasticsearch.scheduled.snapshotRetentionMinCount`
* `backup.elasticsearch.scheduled.snapshotRetentionMaxCount`

[NOTE]
====
默认情况下，保留任务本身https://www.elastic.co/guide/en/elasticsearch/reference/7.6/slm-settings.html#_slm_retention_schedule[在世界协调时每天凌晨 1:30 运行（elastic.co）]。如果设置快照的过期时间快于一天，例如出于测试目的，则需要更改保留任务的计划。
====


=== 快照指数

默认情况下，会为名称以`sts` 开头的 Elasticsearch 索引创建快照。

创建快照的索引可使用 Helm 值`backup.elasticsearch.scheduled.indices` 进行配置，该值以https://www.w3schools.com/js/js_json_arrays.asp[JSON 数组格式]指定。

=== 恢复维多利亚度量标准备份

[CAUTION]
====
*还原功能始终优先于数据。您必须小心谨慎，避免意外丢失现有数据。*
====


[CAUTION]
====
*恢复功能需要在运行过程中停止 Victoria Metric 实例。*

所有新指标将在还原过程中由`vmagent` 缓存，请确保`vmagent` 有足够的内存缓存指标。
====


要恢复 Victoria Metrics 备份，请选择实例名称和备份版本，并将其作为参数传递给以下命令：

[,bash]
----
./restore/restore-victoria-metrics-backup.sh victoria-metrics-0 victoria-metrics-0-20231128160000
----

输出结果应该是这样的

kubectl delete job victoria-metrics-restore-backup-20231017t092607

=== 删除 Elasticsearch 索引

[NOTE]
====
您可以在还原脚本中使用`--delete-all-indices` 标志，在还原前自动删除所有索引。有关详细信息，请参阅xref:/setup/data-management/backup_restore/kubernetes_backup.adoc#_restore_an_elasticsearch_snapshot[还原 Elasticsearch 快照]。
====

要手动删除现有的 Elasticsearch 索引并还原快照，请按照以下步骤操作：

. 停止索引 - 将所有使用 Elasticsearch 的部署缩减为 0：
+
[,bash]
----
kubectl scale --replicas=0 deployment -l observability.suse.com/scalable-during-es-restore="true"
----

. 打开指向 Elasticsearch 主站的端口转发：
+
[,bash]
----
kubectl port-forward service/suse-observability-elasticsearch-master 9200:9200
----

. 获取所有指数的列表：
+
[,bash]
----
curl "http://localhost:9200/_cat/indices?v=true"
----
+
输出结果应该是这样的
+
[,bash]
----
health status index                              uuid                   pri rep docs.count docs.deleted store.size pri.store.size dataset.size
green  open   .ds-sts_k8s_logs-2025.09.28-004619 9p7RZwNCR-aQwInTMr5Bow   3   1   24511032            0        6gb            3gb          3gb
green  open   sts_topology_events-2025.10.01     86I2JZIeRzqWkK1dolHzhg   1   1    1576132            0    111.6mb         55.8mb       55.8mb
green  open   sts_topology_events-2025.10.02     T-bcrok_S1uVPLusQuCMxw   1   1     999748            0     75.2mb         37.6mb       37.6mb
green  open   .ds-sts_k8s_logs-2025.09.30-004653 rwlcAr0sTPe9NaImtJLIiw   3   1   24387607            0        6gb            3gb          3gb
green  open   sts_topology_events-2025.09.10     T0x-qvyUR2-dg4fyvdZIaQ   1   1    1746143            0    131.6mb         65.8mb       65.8mb
----

. 使用以下命令删除索引
+
[,bash]
----
curl -X DELETE "http://localhost:9200/INDEX_NAME?pretty"
----
+
将`INDEX_NAME` 替换为要删除的索引名称，例如
+
[,bash]
----
curl -X DELETE "http://localhost:9200/sts_internal_events-2021.02.19?pretty"
----

. 输出结果应为
+
[,javascript]
----
{
"acknowledged" : true
}
----

=== 还原 Elasticsearch 快照

[WARNING]
====
*恢复快照时，现有索引不会被覆盖。*

你可以使用`--delete-all-indices` 标志通过还原脚本自动删除所有索引，或者按照xref:/setup/data-management/backup_restore/kubernetes_backup.adoc#_delete_elasticsearch_indices[删除 Elasticsearch 索引]中的说明手动删除它们。

====

[WARNING]
====
*如果重新创建了 Elasticsearch`PersistentVolumes` *（例如，意外删除和重新启动 pod 之后），则必须使用以下两种方法之一重新创建 Elasticsearch 备份配置：

* 使用与初始安装相同的配置重新安装 SUSE Observability Helm 图表 (OR)
* 手动触发备份初始化 CronJob：
+
[,bash]
----
kubectl create job --from=cronjob.batch/suse-observability-backup-init "suse-observability-backup-init-$(date +%s)"
----
====
要还原 Elasticsearch 快照，请选择快照名称并将其作为第一个参数传递。您可以选择指定

* 以逗号分隔的要恢复的索引列表。如果未指定，将恢复与 Helm 值`backup.elasticsearch.scheduled.indices` 匹配的所有索引。默认值为`"sts*"` ）。
* `2.6.1`--delete-all-indices "标志，在恢复前自动删除所有现有索引。

==== 基本还原

[,bash]
----
./restore/restore-elasticsearch-snapshot.sh \
  sts-backup-20210223-0300-ppss8nx40ykppss8nx40yk
----

==== 恢复特定索引

[,bash]
----
./restore/restore-elasticsearch-snapshot.sh \
  sts-backup-20210223-0300-ppss8nx40ykppss8nx40yk \
  "<INDEX_TO_RESTORE>,<INDEX_TO_RESTORE>"
----

==== 恢复时自动删除索引

[,bash]
----
./restore/restore-elasticsearch-snapshot.sh \
  sts-backup-20210223-0300-ppss8nx40ykppss8nx40yk \
  --delete-all-indices
----
