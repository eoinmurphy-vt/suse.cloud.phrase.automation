= 启用备份
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: SUSE Observability 自助托管

[NOTE]
====
本页适用于{stackstate-product-name} v2.7.0 或更新版本。
====

== 概述

{stackstate-product-name} 有一个内置备份机制，可配置为将备份存储到 AWS S3、Azure Blob Storage 或 Kubernetes 持久卷。

大多数备份都是通过一个 Helm 值`global.backup.enabled` 启用的。设置备份始终处于启用状态，但根据以下值会有不同的表现：

=== 备份范围

可以自动备份以下数据：

* *设置*（堆栈包、监控器、视图、令牌） - 始终启用：
** `global.backup.enabled` 是`true` 时：备份通过 MinIO 存储到配置的存储后端
** 当`global.backup.enabled` 是`false` 时：绕过 MinIO 将备份存储到专用的 Kubernetes 持久卷中
* 存储在 StackGraph 中的*拓扑数据和设置*- 在`global.backup.enabled` 时启用。 `true`
* 存储在{stackstate-product-name} 的 Victoria Metrics 实例中的*度量指标*- 当`global.backup.enabled` 为 "Victoria "时启用。 `true`
* *遥测数据*存储在{stackstate-product-name} 的 Elasticsearch 实例中 - 在`global.backup.enabled` 时启用。 `true`
* 存储在{stackstate-product-name} 的 ClickHouse 实例中的*OpenTelemetry 数据*- 当`global.backup.enabled` 是 `true`

=== 存储选项

备份使用https://min.io/[MinIO (min.io)]作为连接存储后端的 S3 兼容网关。启用备份时，Helm 图表会自动部署 MinIO。

内置 MinIO 实例可配置为将备份存储在三个位置：

* xref:/setup/data-management/backup_restore/backup_enable.adoc[AWS S3]
* xref:/setup/data-management/backup_restore/backup_enable.adoc[Azure Blob 存储]
* xref:/setup/data-management/backup_restore/backup_enable.adoc[Kubernetes 存储]

== 启用备份

You may need to delete the data stored in Amazon S3. ==== 使用单独的 S3 存储桶

要启用计划备份到单独的 AWS S3 存储桶（每个数据存储一个），请将以下 YAML 片段添加到用于安装{stackstate-product-name} 的 Helm`values.yaml` 文件中：

[,yaml]
----
global:
  backup:
    enabled: true
backup:
  stackGraph:
    bucketName: AWS_STACKGRAPH_BUCKET
  elasticsearch:
    bucketName: AWS_ELASTICSEARCH_BUCKET
  configuration:
    bucketName: AWS_CONFIGURATION_BUCKET
victoria-metrics-0:
  backup:
    bucketName: AWS_VICTORIA_METRICS_BUCKET
victoria-metrics-1:
  backup:
    bucketName: AWS_VICTORIA_METRICS_BUCKET
clickhouse:
  backup:
    bucketName: AWS_CLICKHOUSE_BUCKET
minio:
  accessKey: YOUR_ACCESS_KEY
  secretKey: YOUR_SECRET_KEY
  s3gateway:
    enabled: true
    accessKey: AWS_ACCESS_KEY
    secretKey: AWS_SECRET_KEY
----

替换以下值：

* `YOUR_ACCESS_KEY` 和`YOUR_SECRET_KEY` 是用于确保 MinIO 系统安全的凭证。这些凭证在 MinIO 系统上设置，并被自动备份任务和还原任务使用。如果要手动访问 MinIO 系统，也需要它们。
 ** YOUR_ACCESS_KEY 应包含 5 至 20 个字母数字字符。
 ** YOUR_SECRET_KEY 应包含 8 至 40 个字母数字字符。
* `AWS_ACCESS_KEY` 和`AWS_SECRET_KEY` 是 IAM 用户的 AWS 凭据，该用户有权访问存储备份的 S3 存储桶。请参阅下文，了解需要附加给该用户的权限策略。
* `AWS_*_BUCKET` 是存储备份的 S3 存储桶的名称。
+
[Note]
====
AWS S3 存储桶的名称在整个 AWS 中是全局性的。因此，默认名称为`sts-elasticsearch-backup`,`sts-configuration-backup`,`sts-stackgraph-backup`,`sts-victoria-metrics-backup` 和`sts-clickhouse-backup` 的 S3 存储桶可能无法使用。
====

==== 使用带前缀的单一 S3 存储桶

您可以使用具有不同前缀的单个 S3 存储桶，而不是为每个数据存储使用单独的存储桶：

[,yaml]
----
global:
  backup:
    enabled: true
backup:
  elasticsearch:
    bucketName: BUCKET
    s3Prefix: elasticsearch
  stackGraph:
    bucketName: BUCKET
    s3Prefix: stackgraph
  configuration:
    bucketName: BUCKET
    s3Prefix: configuration
victoria-metrics-0:
  backup:
    bucketName: BUCKET
    s3Prefix: victoria-metrics-0
victoria-metrics-1:
  backup:
    bucketName: BUCKET
    s3Prefix: victoria-metrics-1
clickhouse:
  backup:
    bucketName: BUCKET
    s3Prefix: clickhouse
minio:
  accessKey: YOUR_ACCESS_KEY
  secretKey: YOUR_SECRET_KEY
  s3gateway:
    enabled: true
    accessKey: AWS_ACCESS_KEY
    secretKey: AWS_SECRET_KEY
----

将`BUCKET` 替换为您的 S3 存储桶名称。不同数据存储的备份使用配置的`s3Prefix` 值进行组织。上一节中的`YOUR_ACCESS_KEY` 、`YOUR_SECRET_KEY` 、`AWS_ACCESS_KEY` 和`AWS_SECRET_KEY` 值同样适用于此处。

=== AWS S3 权限

`AWS_ACCESS_KEY` 和`AWS_SECRET_KEY` 标识的 IAM 用户必须配置以下权限策略才能访问 S3 存储桶：

[,javascript]
----
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "AllowListMinioBackupBuckets",
            "Effect": "Allow",
            "Action": [
                "s3:ListBucket",
                "s3:GetBucketLocation"
            ],
            "Resource": [
                "arn:aws:s3:::AWS_STACKGRAPH_BUCKET",
                "arn:aws:s3:::AWS_ELASTICSEARCH_BUCKET",
                "arn:aws:s3:::AWS_VICTORIA_METRICS_BUCKET",
                "arn:aws:s3:::AWS_CLICKHOUSE_BUCKET",
                "arn:aws:s3:::AWS_CONFIGURATION_BUCKET"
            ]
        },
        {
            "Sid": "AllowWriteMinioBackupBuckets",
            "Effect": "Allow",
            "Action": [
                "s3:PutObject",
                "s3:GetObject",
                "s3:DeleteObject"
            ],
            "Resource": [
                "arn:aws:s3:::AWS_STACKGRAPH_BUCKET/*",
                "arn:aws:s3:::AWS_ELASTICSEARCH_BUCKET/*",
                "arn:aws:s3:::AWS_VICTORIA_METRICS_BUCKET/*",
                "arn:aws:s3:::AWS_CLICKHOUSE_BUCKET/*",
                "arn:aws:s3:::AWS_CONFIGURATION_BUCKET"
            ]
        }
    ]
}
----

=== Azure Blob 存储

==== 使用单独的容器

要启用备份到单独的 Azure Blob Storage 容器（每个数据存储一个），请将以下 YAML 片段添加到用于安装{stackstate-product-name} 的 Helm`values.yaml` 文件中：

[,yaml]
----
global:
  backup:
    enabled: true
minio:
  accessKey: AZURE_STORAGE_ACCOUNT_NAME
  secretKey: AZURE_STORAGE_ACCOUNT_KEY
  azuregateway:
    enabled: true
----

替换以下值：

* `AZURE_STORAGE_ACCOUNT_NAME` -https://learn.microsoft.com/en-us/azure/storage/common/storage-account-create?tabs=azure-portal[Azure 存储账户名称（learn.microsoft.com）]
* `AZURE_STORAGE_ACCOUNT_KEY` -https://learn.microsoft.com/en-us/azure/storage/common/storage-account-keys-manage?tabs=azure-portal[Azure 存储账户密钥（learn.microsoft.com）]，备份应存储在该账户中。

StackGraph、Elasticsearch、Victoria Metrics 和 ClickHouse 备份分别存储在名为`sts-stackgraph-backup`,`sts-configuration-backup`,`sts-elasticsearch-backup`,`sts-victoria-metrics-backup`,`sts-clickhouse-backup` 的 BLOB 容器中。这些名称可分别通过设置 Helm 值`backup.stackGraph.bucketName`,`backup.elasticsearch.bucketName`,`victoria-metrics-0.backup.bucketName`,`victoria-metrics-1.backup.bucketName` 和`clickhouse.backup.bucketName` 进行更改。

==== 使用带前缀的单一容器

您可以使用具有不同前缀的单个 Azure Blob Storage 容器，而不是为每个数据存储使用单独的容器：

[,yaml]
----
global:
  backup:
    enabled: true
backup:
  elasticsearch:
    bucketName: CONTAINER
    s3Prefix: elasticsearch
  stackGraph:
    bucketName: CONTAINER
    s3Prefix: stackgraph
  configuration:
    bucketName: CONTAINER
    s3Prefix: configuration
victoria-metrics-0:
  backup:
    bucketName: CONTAINER
    s3Prefix: victoria-metrics-0
victoria-metrics-1:
  backup:
    bucketName: CONTAINER
    s3Prefix: victoria-metrics-1
clickhouse:
  backup:
    bucketName: CONTAINER
    s3Prefix: clickhouse
minio:
  accessKey: AZURE_STORAGE_ACCOUNT_NAME
  secretKey: AZURE_STORAGE_ACCOUNT_KEY
  azuregateway:
    enabled: true
----

将`CONTAINER` 替换为您的 Azure Blob Storage 容器名称。不同数据存储的备份将使用配置的`s3Prefix` 值进行组织。上一节中的`AZURE_STORAGE_ACCOUNT_NAME` 和`AZURE_STORAGE_ACCOUNT_KEY` 值同样适用于此处。

=== Kubernetes 持久卷

[CAUTION]
====
使用 Kubernetes 持久卷进行备份有很大的局限性：

* 昂贵 - 云提供商通常使用块存储（EBS/Azure Block），这对于大型备份来说成本高昂
* 无灾难恢复 - 如果删除群集，PV 将被销毁
* 不可移动 - 无法将备份还原到不同的群集

**建议**在生产环境中，请使用 AWS S3 或 Azure Blob Storage。
====

==== 基本配置

要启用备份到群集本地存储，请在用于安装{stackstate-product-name} 的 Helm`values.yaml` 文件中添加以下 YAML 片段，以启用 MinIO：

[,yaml]
----
global:
  backup:
    enabled: true
minio:
  accessKey: YOUR_ACCESS_KEY
  secretKey: YOUR_SECRET_KEY
  persistence:
    enabled: true
----

替换以下值：

* `YOUR_ACCESS_KEY` 和`YOUR_SECRET_KEY` - 用于确保 MinIO 系统安全的凭证。自动备份工作和还原工作将使用它们。`YOUR_ACCESS_KEY` 应包含 5 至 20 个字母数字字符，`YOUR_SECRET_KEY` 应包含 8 至 40 个字母数字字符。

== 配置和拓扑数据（StackGraph）

配置和拓扑数据（StackGraph）备份为完整备份，存储在扩展名为`.graph` 的单个文件中。每个文件都包含完整备份，可根据需要移动、复制或删除。

=== 备份时间表

默认情况下，StackGraph 备份在服务器时间每天凌晨 03:00 创建。

备份计划可使用https://kubernetes.io/docs/concepts/workloads/controllers/cron-jobs/#_cron_schedule_syntax[Kubernetes cron 计划语法（kubernetes.io）]中指定的 Helm 值`backup.stackGraph.scheduled.schedule` 进行配置。

=== 备份保留

默认情况下，StackGraph 备份保存 30 天。由于 StackGraph 备份是完全备份，因此需要大量存储空间。

备份保留 delta 可以使用 Helm 值`backup.stackGraph.scheduled.backupRetentionTimeDelta` 进行配置，该值以 GNU 日期`--date` 参数格式指定。默认值为`30 days ago` 。更多示例请参阅https://www.gnu.org/software/coreutils/manual/html_node/Relative-items-in-date-strings.html[日期字符串中的相对项]。

=== 禁用计划备份

要禁用 StackGraph 计划备份，请使用 Helm 值`backup.stackGraph.scheduled.schedule` 将备份计划设置为远在过去的日期：

[,yaml]
----
backup:
  stackGraph:
    scheduled:
      schedule: '0 0 1 1 1970'  # January 1, 1970 (epoch start)
----

== 设置

设置（之前的配置）包括已安装的 StackPacks 实例及其配置和用户创建的其他自定义配置，如监控器、自定义视图和服务令牌。

设置备份重量轻（通常只有几兆字节），恢复速度快，停机时间最短。恢复设置备份后，将像以前一样处理新数据，重新创建拓扑、健康状态和警报。不过，拓扑历史（包括健康状况）不会在设置备份中保留\--为此，请使用上述 StackGraph 备份。

[NOTE]
====
无论`global.backup.enabled` 的值如何，*设置备份始终处于启用状态*：

* `global.backup.enabled` 时，`true` ：设置备份通过 MinIO 存储到您配置的存储后端（AWS S3、Azure Blob Storage 或 Kubernetes Persistent Volume）。
* `global.backup.enabled` 时，`false` ：设置备份存储到专用的 Kubernetes 持久卷，绕过 MinIO
====

=== 备份时间表

默认情况下，设置备份在服务器时间每天凌晨 04:00 创建。

备份计划可使用https://kubernetes.io/docs/concepts/workloads/controllers/cron-jobs/#_cron_schedule_syntax[Kubernetes cron 计划语法（kubernetes.io）]中指定的 Helm 值`backup.configuration.scheduled.schedule` 进行配置。

=== 备份保留

备份保留取决于`global.backup.enabled` 设置：

** `global.backup.enabled` 时，`true`**（通过 MinIO 存储备份）：

* 默认情况下，设置备份保存 365 天
* 使用`backup.configuration.scheduled.backupRetentionTimeDelta` 配置保留 - 以 GNU 日期`--date` 参数格式指定。默认值为 `365 days ago`

** `global.backup.enabled` 时，`false`**（备份存储到专用 PV）：

* 默认情况下，最后 10 个备份文件保存在持久卷上
* 使用`backup.configuration.maxLocalFiles` 配置文件的最大数量（默认值：`10`)
* 使用`backup.configuration.scheduled.pvc.size` 配置 PV 大小（默认值：`1Gi`)

专用光伏储能系统配置示例：

[,yaml]
----
backup:
  configuration:
    maxLocalFiles: 10
    scheduled:
      pvc:
        size: '1Gi'
----

=== 禁用计划备份

要禁用计划设置备份，请使用 Helm 值`backup.configuration.scheduled.schedule` 将备份计划设置为远在过去的日期：

[,yaml]
----
backup:
  configuration:
    scheduled:
      schedule: '0 0 1 1 1970'  # January 1, 1970 (epoch start)
----

== 度量标准（维多利亚度量标准）

维多利亚度量标准创建备份时，会对当前度量标准数据进行快照，并只存储上次备份后的更改（增量方法）。

[WARNING]
====
*备份版本限制*

每次运行新的备份时，Victoria Metrics 备份都会替换之前的备份。这意味着你只能访问*最近的备份*，而不能访问多个备份版本的历史记录。

**重要**：如果备份失败或数据损坏，之前的备份数据可能会丢失。只有最新的成功备份才能用于还原。

**例外情况**：如果 Victoria Metrics 存储卷被清空或重置（如`/storage` 目录被挂载为空或被删除），系统将检测到并自动创建新的备份系列。在这种情况下，旧备份（重置前）和新备份都将被保留并可用于还原。
====

[NOTE]
====
*高可用性部署*

高可用部署使用两个 Victoria Metrics 实例 (`victoria-metrics-0` 和`victoria-metrics-1`)。每个实例的备份都是独立配置的。
====

=== 备份时间表

维多利亚指标备份每 1 小时创建一次：

* `victoria-metrics-0` - 整点 25 分
* `victoria-metrics-1` - 整点 35 分

=== 禁用计划备份

要禁用计划中的 Victoria Metrics 备份，请将两个实例的备份计划都设置为远在过去的日期：

[,yaml]
----
victoria-metrics-0:
  backup:
    scheduled:
      schedule: '0 0 1 1 1970'  # January 1, 1970 (epoch start)
victoria-metrics-1:
  backup:
    scheduled:
      schedule: '0 0 1 1 1970'  # January 1, 1970 (epoch start)
----

== OpenTelemetry (ClickHouse)

ClickHouse 同时使用增量备份和完全备份。根据配置的保留策略删除旧备份。

=== 备份时间表

创建 ClickHouse 备份：

* 全面备份 - 每天 00:45
* 增量备份 - 45 分钟后（从凌晨 3 点到上午 12 点）

=== 备份保留

默认情况下，工具会保留最后 308 次备份（完整和增量备份），相当于 14 天。

备份保留可通过 Helm 值`clickhouse.backup.config.keep_remote` 进行配置。

=== 禁用计划备份

要禁用计划中的 ClickHouse 备份，请将完整备份和增量备份计划设置为较远的过去日期：

[,yaml]
----
clickhouse:
  backup:
    scheduled:
      full_schedule: '0 0 1 1 1970'         # January 1, 1970 (epoch start)
      incremental_schedule: '0 0 1 1 1970'  # January 1, 1970 (epoch start)
----

== 遥测数据（Elasticsearch）

Elasticsearch 快照是增量的。

=== 快照时间表

Elasticsearch 快照于服务器时间每天凌晨 03:00 创建。

=== 保留快照

默认情况下，Elasticsearch 快照保存 30 天，最少保存 5 个快照，最多保存 30 个快照。

可以使用以下 Helm 值配置快照的保留时间和数量：

* `backup.elasticsearch.scheduled.snapshotRetentionExpireAfter`，以https://www.elastic.co/guide/en/elasticsearch/reference/7.6/common-options.html#_time_units[Elasticsearch 时间单位 (elastic.co)] 指定。
* `backup.elasticsearch.scheduled.snapshotRetentionMinCount`
* `backup.elasticsearch.scheduled.snapshotRetentionMaxCount`

=== 禁用计划快照

要禁用计划的 Elasticsearch 快照，请使用 Helm 值`backup.elasticsearch.scheduled.schedule` 将快照计划设置为远在过去的日期：

[,yaml]
----
backup:
  elasticsearch:
    scheduled:
      schedule: '0 0 1 1 1970'  # January 1, 1970 (epoch start)
----
