= グローバル・コンフィギュレーション・モードへの移行
:description: {stackstate-product-name} セルフホスト
:page-toclevels: 3

== 概要

このガイドでは、`suse-observability-values` チャートを使用した従来の 2 ステップ・インストールから、`global.suseObservability` 構成を使用した簡略化されたシングル・チャート・インストールへの移行方法について説明します。

[NOTE]
====
`global.suseObservability` の設定方法は、バージョン`2.8.0` から利用できるようになり、現在推奨されているインストール方法です。レガシーの`suse-observability-values` チャートメソッドは非推奨ですが、既存のインストールでは引き続き使用できます。
====

=== レガシーインストール（非推奨）

従来のインストール方法では、2つのステップが必要だった：

. `suse-observability-values` チャートを使った数値ファイルの作成
. 生成された値で`suse-observability` チャートをインストールする。

[,text]
----
# Step 1: Generate values
helm template suse-observability-values suse-observability/suse-observability-values \
  --set license="YOUR-LICENSE-KEY" \
  --set baseUrl="https://observability.example.com" \
  --set sizing.profile="150-ha" \
  --set adminPassword="your-password" \
  --set pullSecret.username="registry-user" \
  --set pullSecret.password="registry-pass" \
  > generated-values.yaml

# Step 2: Install with generated values
helm install suse-observability suse-observability/suse-observability \
  -f generated-values.yaml \
  -n suse-observability
----

=== 新規インストール（推奨）

新しい方法では、`global.suseObservability` のコンフィギュレーションを持つ単一の値ファイルを使用する：

[,text]
----
helm install suse-observability suse-observability/suse-observability \
  -f values.yaml \
  -n suse-observability
----

== インストール前に変換する

レガシー・チャート（`suse-observability-values` ）用のパラメータを準備しているが、まだインストールしていない場合は、中間値ファイルを作成することなく、新方式に直接変換することができる。

=== パラメータの直接変換

パラメータをマッピングして、`helm template` コマンドを`helm upgrade --install` コマンドに変換する：

[cols="1,1"]
|===
| 旧パラメーター | 新しいパラメータ

| `--set license="..."`
| `--set global.suseObservability.license="..."`

| `--set baseUrl="..."`
| `--set global.suseObservability.baseUrl="..."`

| `--set sizing.profile="..."`
| `--set global.suseObservability.sizing.profile="..."`

| `--set receiverApiKey="..."`
| `--set global.suseObservability.receiverApiKey="..."`

| `--set adminPassword="..."`
| `--set global.suseObservability.adminPassword="..."`

| `--set imageRegistry="..."`
| `--set global.imageRegistry="..."`

| `--set pullSecret.username="..."`
| `--set global.suseObservability.pullSecret.username="..."`

| `--set pullSecret.password="..."`
| `--set global.suseObservability.pullSecret.password="..."`
|===

=== 例ビフォー・アフター

[NOTE]
====
helmコマンドを実行する前に、チャートのバージョンが最新であることを確認してください：

[,bash]
----
helm repo update
----
====

.レガシーコマンド（使用しない）
[,bash]
----
export VALUES_DIR=.
helm template \
  --set license="YOUR-LICENSE-KEY" \
  --set baseUrl="https://observability.example.com" \
  --set sizing.profile="10-nonha" \
  --set receiverApiKey="YOUR-API-KEY" \
  suse-observability-values \
  suse-observability/suse-observability-values --output-dir $VALUES_DIR

# Then: helm install with generated files...
----

.新しいコマンド（推奨）
[,bash]
----
helm upgrade --install \
  --namespace suse-observability \
  --create-namespace \
  --set global.suseObservability.license="YOUR-LICENSE-KEY" \
  --set global.suseObservability.baseUrl="https://observability.example.com" \
  --set global.suseObservability.sizing.profile="10-nonha" \
  --set global.suseObservability.receiverApiKey="YOUR-API-KEY" \
  --set global.suseObservability.adminPassword='your-password' \
  suse-observability suse-observability/suse-observability
----

[NOTE]
====
* `adminPassword` 、新しいメソッドのbcryptハッシュでなければならない。生成方法は<<bcrypt-hash>> を参照。
* エアギャップ環境またはカスタム登録の場合、`imageRegistry` および`pullSecret` 設定を含む追加設定については、xref:/k8s-suse-rancher-prime-air-gapped.adoc[エアギャップモードでの{stackstate-product-name} のインストールを]参照してください。
====

=== 代わりにバリューファイルを使う

数値ファイルを使用したい場合は、<<_step_3_create_new_values_file>> に示す構成で`values.yaml` を作成し、`helm repo update` で最新のチャートを持っていることを確認してから実行してください：

[,bash]
----
helm upgrade --install \
  --namespace suse-observability \
  --create-namespace \
  --values values.yaml \
  suse-observability suse-observability/suse-observability
----

== 移行する前に

レガシーインストール方法からの移行を実行する場合は、以下の点に注意してください：

[CAUTION]
====
この移行ガイドは、現在`suse-observability-values` チャート方式を使用しているユーザー向けのものです。StackState 6.0からアップグレードする場合は、代わりにxref:/setup/upgrade-stackstate/migrate-from-6.adoc[StackStateからの移行ガイドに従って]ください。
====

[NOTE]
====
* 移行は標準のHelm upgradeコマンドを使って実行します。
* 既存のデータと設定は保持されます。
* 移行はメンテナンス・ウィンドウの間に実行できる
* ロールバックは、標準的な Helm のロールバック手順を使用して行うことができます。
====

== マイグレーション手順

=== ステップ1：現在の設定をバックアップする

移行を開始する前に、既存のHelmの値とカスタマイズを保存してください：

[,text]
----
# Save current Helm values
helm get values suse-observability -n suse-observability > backup-values.yaml

# Save your original suse-observability-values input (if available)
# This is typically your values file used with helm template

# Backup important resources
kubectl get secrets -n suse-observability -o yaml > backup-secrets.yaml
----

=== ステップ2：現在の構成を確認する

既存の設定を見直し、以下の設定を確認する：

* *サイズプロファイル*：suse-observability-valuesの入力で`sizing.profile` 。
* *ライセンスキー*：{stackstate-product-name} ライセンス
* *ベースURL*：{stackstate-product-name} にアクセスできるURL
* *管理者パスワード*`adminPasswordBcrypt`管理者パスワード（パスワードのbcryptハッシュを使用することもできます。）
* *秘密のクレデンシャルを引き出す*：レジストリのユーザー名とパスワード（使用している場合）
* *カスタムアフィニティ設定*：ノードのアフィニティやポッドのアンチアフィニティのカスタマイズ

現在のインストールから、これらのいくつかを抽出することができる：

[,text]
----
# Check current values
helm get values suse-observability -n suse-observability

# The sizing profile is visible in resource configurations
# Look for patterns like replica counts, resource limits, etc.
----

=== 自動生成値とカスタム値を理解する

`helm get values` を実行すると、何百行ものコンフィギュレーションが表示されるかもしれない。しかし、これらのほとんどはサイジング・プロファイルによって*自動生成される*ため、移行値ファイルに手動で保存する必要は*ありません*。

==== サイジング・プロファイルが自動的に提供するもの

`global.suseObservability.sizing.profile` を設定すると、チャートは自動的に設定される：

* すべてのコンポーネント（Elasticsearch、Kafka、ClickHouse、サーバー、レシーバーなど）のリソース要求と制限。
* ステートフルサービスのレプリカカウント
* 永続ボリュームのストレージサイズ
* HBaseのデプロイモード（非HAではMono、HAではDistributed）
* サーバーとレシーバーの分割構成
* ビクトリア・メトリクスのHA設定
* パフォーマンス・チューニングのための`CONFIG_FORCE_*` 設定を含む環境変数 (`extraEnv`)

==== 保存に必要なもの

*本当にカスタムな*値、つまりサイズプロファイルが提供する値以上に追加または変更した構成のみを保存します：

[cols="1,1"]
|===
| プリザーブ（カスタム） | 保存しない（自動生成）

| コア設定（ライセンス、baseUrl、adminPassword、receiverApiKey）
| リソース制限とリクエスト

| イングレスの設定（ホスト、TLS、アノテーション）
| レプリカカウント

| 追加したカスタム環境変数
| 収納サイズ

| 認証設定（LDAP、OIDC）
| `CONFIG_FORCE_*` 環境変数

| カスタム `allowedOrigins`
| Kafka/Zookeeperのレプリケーション要因

| OpenTelemetryコレクタ・カスタム・イングレス
| HBaseデプロイモード
|===

==== 例最小移行値ファイル

マイグレーション値のファイルは、`helm get values` の出力よりも*ずっと小さく*なるはずです：

[,yaml]
----
global:
  suseObservability:
    license: "YOUR-LICENSE"
    baseUrl: "https://your-url.example.com"
    sizing:
      profile: "10-nonha"  # This handles all resource configuration!
    adminPassword: "your-password"
    receiverApiKey: "your-api-key"

# Only include truly custom configurations below
ingress:
  enabled: true
  ingressClassName: your-ingress-class
  annotations:
    # your custom annotations
  hosts:
  - host: your-url.example.com
  tls:
  - hosts:
    - your-url.example.com
    secretName: your-tls-secret
----

[TIP]
====
値がカスタムか自動生成かわからない場合は、サイズプロファイルのデフォルト値と比較してください。`elasticsearch.resources` 、`kafka.resources` 、`stackstate.components.server.resources` 、`CONFIG_FORCE_*` のような環境変数はすべてプロファイルによって生成されるため、省略しても問題ない。
====

=== ステップ3：新しい数値ファイルを作成する

`global.suseObservability` 構造で新しい`values.yaml` を作成する：

[,yaml]
----
global:
  suseObservability:
    # Required: Your license key
    license: "YOUR-LICENSE-KEY"

    # Required: Base URL for {stackstate-product-name}
    baseUrl: "https://observability.example.com"

    # Required: Sizing profile
    # Available: trial, 10-nonha, 20-nonha, 50-nonha, 100-nonha,
    #            150-ha, 250-ha, 500-ha, 4000-ha
    sizing:
      profile: "150-ha"

    adminPassword: "your-password"
    # Instead of adminPassword you can provide a bcrypt hashed password
    # Generate with: htpasswd -bnBC 10 "" "your-password" | tr -d ':\n'
    # adminPasswordBcrypt: "$2a$10$..."

    # Optional: Receiver API key (auto-generated if not provided)
    receiverApiKey: "your-receiver-api-key"

    # Optional: Affinity configuration
    # These settings apply to all infrastructure components (Elasticsearch,
    # Kafka, ClickHouse, etc.) automatically - no need to configure per-component
    affinity:
      # Node affinity for all components
      nodeAffinity: null

      # Pod anti-affinity for HA profiles (infrastructure components)
      podAntiAffinity:
        requiredDuringSchedulingIgnoredDuringExecution: true
        topologyKey: "kubernetes.io/hostname"

  # For air-gapped or custom registry configurations, see:
  # xref:/k8s-suse-rancher-prime-air-gapped.adoc[Air-Gapped Installation Guide]
----

=== ステップ5：カスタム・オーバーライドの処理

生成されたバリュー・ファイルにカスタム・オーバーライドがある場合は、それを使用することができます。新しいグローバル・モードは、コンポーネントごとにオーバーライドできる賢明なデフォルトを提供する：

[,yaml]
----
global:
  suseObservability:
    sizing:
      profile: "150-ha"
    license: "YOUR-LICENSE-KEY"
    baseUrl: "https://observability.example.com"
    adminPassword: "your-password"
    # Global affinity - applied to all components by default
    affinity:
      nodeAffinity:
        requiredDuringSchedulingIgnoredDuringExecution:
          nodeSelectorTerms:
          - matchExpressions:
            - key: topology.kubernetes.io/zone
              operator: In
              values:
              - us-west-2a

# Custom per-component overrides still work - they take precedence over global defaults
stackstate:
  components:
    api:
      resources:
        requests:
          memory: 16Gi  # Override default from sizing profile
        limits:
          memory: 20Gi

elasticsearch:
  volumeClaimTemplate:
    resources:
      requests:
        storage: 500Gi  # Override default storage size
  # Override affinity for Elasticsearch specifically
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: node-type
            operator: In
            values:
            - storage-optimized
----

== 管理者パスワード

レガシー方式では、`adminPassword` はプレーンテキストとしてsuse-observability-valuesチャートに提供され、suse-observability-valuesチャートはそれをbcryptハッシュにハッシュした。新しい方法では、プレーンテキストの`adminPassword` をsuse-observability chartに提供するか、`adminPasswordBcrypt` を使ってbcryptハッシュを直接suse-observability chartに提供することができる。

新しいbcryptハッシュを生成する必要がある場合：

[,bash]
----
htpasswd -bnBC 10 "" "your-password" | tr -d ':\n'
----

== トラブルシューティング

=== PodがPending（保留）状態から抜け出せない

これは通常、アンチ・アフィニティ・ルールのスケジューリングに問題があることを示している。ノード数が限られているクラスタでは、ソフト・アンチ・アフィニティーを使用する：

[,yaml]
----
global:
  suseObservability:
    affinity:
      podAntiAffinity:
        requiredDuringSchedulingIgnoredDuringExecution: false  # Use soft anti-affinity
----

=== リソースが以前のインストールと異なる

サイジング・プロファイルの推奨リソースが更新される可能性があります。以前の設定を保持するには、明示的なオーバーライドを追加します：

[,yaml]
----
global:
  suseObservability:
    sizing:
      profile: "150-ha"

# Override with your previous resource values
stackstate:
  components:
    api:
      resources:
        requests:
          memory: 8Gi  # Your previous value
----

=== 構成スタイルの衝突

新旧の構成スタイルを混ぜないこと。どちらかを使う：

* `global.suseObservability.*` (新スタイル）、または
* `stackstate.license.key``stackstate.baseUrl` など（旧式）

[,text]
----
# Check for conflicting values
helm get values suse-observability -n suse-observability | grep -E "license|baseUrl"
----

=== 必要に応じてロールバック

[,text]
----
# List release history
helm history suse-observability -n suse-observability

# Rollback to previous revision
helm rollback suse-observability <revision-number> -n suse-observability
----

== 主な相違点

[cols="1,1,1"]
|===
| アスペクト | レガシー（観測値） | 新規 (global.suseObservability)

| インストール手順
| 2 (テンプレート + インストール)
| 1（インストール）

| バリュー・ファイル管理
| 生成され、保存されなければならない
| 真実の単一ソース

| プロフィール更新
| 手動再生が必要
| チャートのアップグレードで自動化

| 秘密の引き出し
| 個別のサブチャート
| 内蔵、自動設定

| Victoria Metrics HA
| マニュアル `enabled: true/false`
| プロファイルごとに自動設定

| アフィニティ・コンフィギュレーション
| コンポーネントごとに生成（サービスごとに個別の設定）
| 一元化されたグローバル設定（必要に応じてコンポーネントごとにオーバーライド可能）
|===

[NOTE]
====
グローバル・アフィニティ構成は、すべてのインフラストラクチャ・コンポーネントのデフォルト設定を提供する。コンポーネントレベルのアフィニティ値を設定することで、必要に応じて特定のコンポーネント（ElasticsearchやKafkaなど）のアフィニティをオーバーライドすることができます。コンポーネント・レベルの設定は、グローバル・デフォルトよりも優先される。
====

== クリーンアップ

移行に成功したら、次のことができる：

. 生成された値ファイルを削除する（もう必要ない）
. 新しいシングルステップ・インストールを使用するように、デプロイメント・スクリプトを更新します。
. CI/CDパイプライン内の`suse-observability-values` チャートへの参照をすべて削除する。
