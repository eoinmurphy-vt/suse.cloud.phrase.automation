= Kubernetes上でOpen Telemetryオペレータを始める
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: SUSE Observability

監視が必要なアプリケーションのセットアップを紹介しよう：

* クラスタAで稼働している監視対象のアプリケーション/ワークロード。
* クラスタAのオープン・テレメトリー・オペレーター
* オペレーターが作成したコレクター
* クラスタBで動作するSUSE Observability、またはSUSE Cloud Observability

== オペレーターの取り付け

Open Telemetryオペレーターは、通常のKubernetesセットアップよりもいくつかの追加機能を提供する：

* サポートされている言語（Java、.NET、Python、Golang、Node.js）に対応したアプリケーション・ポッドに、アプリケーションやドッカー・イメージを一切変更することなく、自動計測を行うことができます。
* Prometheusオペレーターの代わりとしてドロップインすることができ、サービスとポッドモニターに基づいてPrometheusエクスポーターエンドポイントのスクレイピングを開始します。

=== サービス・トークンの作成

サービス・トークンを作成するには2つの方法がある：

* **SUSE Observability UI**- 画面左上をクリックしてメインメニューを開き、`StackPacks` >`Open Telemetry`. まだの方は、`INSTALL` ボタンをクリックしてください。 `CREATE NEW SERVICE TOKEN` ボタンをクリックし、値をクリップボードにコピーする。
* **SUSE Observability CLI**-xref:/use/security/k8s-service-tokens.adoc#_manage_service_tokens[サービス・トークンの管理を]参照してください。

サービス・トークンの値は、以下の説明で`<SERVICE_TOKEN>` と記述されている場所で使用する必要があります。

=== サービス・トークンのネームスペースとシークレットを作成する。

`open-telemetry` ネームスペースにインストールし、サービス・トークンを使用する：

[,bash]
----
kubectl create namespace open-telemetry
kubectl create secret generic open-telemetry-collector \
    --namespace open-telemetry \
    --from-literal=API_KEY='<SERVICE_TOKEN>'
----

=== オペレーターの設定とインストール

オペレータはHelmチャートと一緒にインストールされているので、最初にチャートリポジトリを設定します。

[,bash]
----
helm repo add open-telemetry https://open-telemetry.github.io/opentelemetry-helm-charts
----

`otel-operator.yaml` 、オペレーターを設定するファイルを作成しよう：

.otel-operator.yaml
[,yaml]
----
# Add image pull secret for private registries
imagePullSecrets: []
manager:
  image:
    # Uses chart.appVersion for the tag
    repository: ghcr.io/open-telemetry/opentelemetry-operator/opentelemetry-operator
  collectorImage:
    # find the latest collector releases at https://github.com/open-telemetry/opentelemetry-collector-releases/releases
    repository: otel/opentelemetry-collector-k8s
    tag: 0.123.0
  targetAllocatorImage:
    repository: ""
    tag: ""
  # Only needed when overriding the image repository, make sure to always specify both the image and tag:
  autoInstrumentationImage:
    java:
      repository: ""
      tag: ""
    nodejs:
      repository: ""
      tag: ""
    python:
      repository: ""
      tag: ""
    dotnet:
      repository: ""
      tag: ""
    # The Go instrumentation support in the operator is disabled by default.
    # To enable it, use the operator.autoinstrumentation.go feature gate.
    go:
      repository: ""
      tag: ""

admissionWebhooks:
  # A production setup should use certManager to generate the certificate, without certmanager the certificate will be generated during the Helm install
  certManager:
    enabled: false
  # The operator has validation and mutation hooks that need a certificate, with this we generate that automatically
  autoGenerateCert:
    enabled: true
----


設定ファイルを使用して、コレクタをインストールします：

[,bash]
----
helm upgrade --install opentelemetry-operator open-telemetry/opentelemetry-operator \
  --namespace open-telemetry \
  --values otel-operator.yaml
----

これはオペレーターをインストールするだけである。コレクタのインストールを続行して、自動測定を有効にします。

== オープン・テレメトリー・コレクター

オペレータは、`OpenTelemetryCollector` という種類の Kubernetes カスタムリソースを介して、1 つ以上のコレクターのデプロイメントを管理する。xref:/setup/otel/getting-started/getting-started-k8s.adoc[Kubernetesのスタートガイドで]使用されているのと同じ設定を使って作成しよう。

このガイドでは、先に作成したシークレットを使用する。`<otlp-suse-observability-endpoint:port>` を OTLP エンドポイントに置き換え（エンドポイントについてはxref:/setup/otel/otlp-apis.adoc[OTLP API を]参照）、`<your-cluster-name>` の代わりに Kubernetes クラスタの名前を挿入してください：

.collector.yaml
[,yaml]
----
apiVersion: opentelemetry.io/v1beta1
kind: OpenTelemetryCollector
metadata:
  name: otel-collector
spec:
  mode: deployment
  envFrom:
  - secretRef:
      name: open-telemetry-collector
  # optional service-account for pulling the collector image from a private registries
  # serviceAccount: otel-collector
  config:
    receivers:
      otlp:
        protocols:
          grpc:
            endpoint: 0.0.0.0:4317
          http:
            endpoint: 0.0.0.0:4318
      # Scrape the collectors own metrics
      prometheus:
        config:
          scrape_configs:
          - job_name: opentelemetry-collector
            scrape_interval: 10s
            static_configs:
            - targets:
              - ${env:MY_POD_IP}:8888
    extensions:
      health_check:
        endpoint: ${env:MY_POD_IP}:13133
      # Use the API key from the env for authentication
      bearertokenauth:
        scheme: SUSEObservability
        token: "${env:API_KEY}"
    exporters:
      debug: {}
      nop: {}
      otlp/suse-observability:
        auth:
          authenticator: bearertokenauth
        # Put in your own otlp endpoint, for example otlp-suse-observability.my.company.com:443
        endpoint: <otlp-suse-observability-endpoint:port>
        compression: snappy
    processors:
      memory_limiter:
        check_interval: 5s
        limit_percentage: 80
        spike_limit_percentage: 25
      batch: {}
      resource:
        attributes:
        - key: k8s.cluster.name
          action: upsert
          # Insert your own cluster name
          value: <your-cluster-name>
        - key: service.instance.id
          from_attribute: k8s.pod.uid
          action: insert
          # Use the k8s namespace also as the open telemetry namespace
        - key: service.namespace
          from_attribute: k8s.namespace.name
          action: insert
    connectors:
      # Generate metrics for spans
      spanmetrics:
        metrics_expiration: 5m
        namespace: otel_span
    service:
      extensions: [ health_check,  bearertokenauth ]
      pipelines:
        traces:
          receivers: [otlp]
          processors: [memory_limiter, resource, batch]
          exporters: [debug, spanmetrics, otlp/suse-observability]
        metrics:
          receivers: [otlp, spanmetrics, prometheus]
          processors: [memory_limiter, resource, batch]
          exporters: [debug, otlp/suse-observability]
        logs:
          receivers: [otlp]
          processors: []
          exporters: [nop]
      telemetry:
        metrics:
          address: ${env:MY_POD_IP}:8888
----


[CAUTION]
====
KubernetesスタックパックでSUSE Observabilityエージェントも使用する場合は、*SUSE Observabilityエージェントのインストールに使用したものと同じクラスタ名を使用*します。異なるクラスタ名を使用すると、Kubernetesコンポーネントのトレースパースペクティブが空になり、全体的にSUSE Observabilityとユーザーにとって情報の関連付けが非常に難しくなります。
====


`open-telemetry` ネームスペースにこの`collector.yaml` を適用して、コレクターをデプロイする：

[,bash]
----
kubectl apply --namespace open-telemetry -f collector.yaml
----

コレクターは、レシーバー、プロセッサー、エクスポーターなど、より多くの設定を提供します。詳細はxref:/setup/otel/collector.adoc[コレクターのページを]ご覧ください。本番使用では大量のスパンが生成されることが多く、xref:/setup/otel/sampling.adoc[サンプリングの]設定を始めたくなるだろう。

== 自動計測

=== 自動計測の設定

ここで、別のカスタムリソース（`Instrumentation` ）を使用して、異なる言語の自動インストゥルメンテーションを設定する方法をオペレーターに伝える必要があります。これは、主に、測定されたアプリケーションのテレメトリエンドポイン トとしてデプロイされたコレクタを設定するために使用される。

一箇所で定義してクラスタ内のすべてのポッドで使用することもできますが、名前空間ごとに異なる`Instrumentation` 。ここでは前者を行う。別の名前空間または別の名前を otel コレクタに使用する場合は、このファイル のエンドポイントを適宜更新する必要があることに注意。

`instrumentation.yaml` を作成する：

.instrumentation.yaml
[,yaml]
----
apiVersion: opentelemetry.io/v1alpha1
kind: Instrumentation
metadata:
  name: otel-instrumentation
spec:
  exporter:
    # default endpoint for the instrumentation
    endpoint: http://otel-collector-collector.open-telemetry.svc.cluster.local:4317
  propagators:
    - tracecontext
    - baggage
  defaults:
    # To use the standard app.kubernetes.io/ labels for the service name, version and namespace:
    useLabelsForResourceAttributes: true
  python:
    env:
      # Python autoinstrumentation uses http/proto by default, so data must be sent to 4318 instead of 4317.
      - name: OTEL_EXPORTER_OTLP_ENDPOINT
        value: http://otel-collector-collector.open-telemetry.svc.cluster.local:4318
  dotnet:
    env:
      # Dotnet autoinstrumentation uses http/proto by default, so data must be sent to 4318 instead of 4317.
      - name: OTEL_EXPORTER_OTLP_ENDPOINT
        value: http://otel-collector-collector.open-telemetry.svc.cluster.local:4318
  go:
    env:
      # Go autoinstrumentation uses http/proto by default, so data must be sent to 4318 instead of 4317.
      - name: OTEL_EXPORTER_OTLP_ENDPOINT
        value: http://otel-collector-collector.open-telemetry.svc.cluster.local:4318
----


次に、`instrumentation.yaml` を`open-telemetry` 名前空間にも適用する：

[,bash]
----
kubectl apply --namespace open-telemetry -f instrumentation.yaml
----

=== ポッドの自動計測を有効にする

アプリケ-ションポッドに自動計測を指示するには、ポッドにアノテーションを追加する必要があります：

* Javaだ： `instrumentation.opentelemetry.io/inject-java: open-telemetry/otel-instrumentation`
* NodeJS： `instrumentation.opentelemetry.io/inject-nodejs: open-telemetry/otel-instrumentation`
* パイソン `instrumentation.opentelemetry.io/inject-python: open-telemetry/otel-instrumentation`
* 行くんだ： `instrumentation.opentelemetry.io/inject-go: open-telemetry/otel-instrumentation`

アノテーションの値は、作成した`Instrumentation` リソースの名前空間と名前を参照していることに注意してください。その他のオプションは以下の通り：

* "true" - 名前空間からカスタム・リソースを注入し、`Instrumentation` 。
* "my-instrumentation" - 現在のネームスペースにある`Instrumentation` カスタムリソースの名前。
* "my-other-namespace/my-instrumentation" - 別の名前空間にある`Instrumentation` カスタムリソースの名前空間と名前。
* 「false" - 注入しない

注釈の1つを持つポッドが作成されると、オペレータは変異フックを介してポッドを変更します：

* 自動計測ライブラリを提供するinitコンテナが追加される。
* 起動時にインスツルメンテーションをロードするようにポッドの最初のコンテナを変更し、インスツルメンテーションを設定するための環境変数を追加します。

どのコンテナをインスツルメンテーションするかカスタマイズする必要がある場合は、https://github.com/open-telemetry/opentelemetry-operator?tab=readme-ov-file#_multi_container_pods_with_multiple_instrumentations[オペレータのドキュメントを]使用してください。

[CAUTION]
====
自動計測を行うには、昇格した権限が必要です。これらの権限はオペレーターが自動的に設定する：

[,yaml]
----
securityContext:
  privileged: true
  runAsUser: 0
----

====


== 結果を見る

SUSE Observabilityにアクセスし、Open Telemetry Stackpackがインストールされていることを確認します（メインメニュー \-> Stackpacks）。

しばらくして、ポッドがトラフィックを得ていれば、Open Telemetry \-> services and service instances overviews で、サービス名の下にポッドを見つけることができるはずです。トレースは、サービスとサービス インスタンス コンポーネントのxref:/use/traces/k8sTs-explore-traces.adoc[トレース エクスプローラ]およびxref:/use/views/k8s-traces-perspective.adoc[トレース パースペクティブに]表示されます。スパン・メトリクスと言語固有のメトリクス（利用可能な場合）は、コンポーネントのxref:/use/views/k8s-metrics-perspective.adoc[メトリクス・パースペクティブで]利用可能になります。

Kubernetesのstackpackもインストールされている場合、インスツルメンテッドポッドもトレースxref:/use/views/k8s-traces-perspective.adoc[パースペクティブで]トレースが利用できるようになります。

== Rancher RBAC

xref:/setup/security/rbac/rbac_rancher.adoc[Rancher RBACが]機能するためには、遠隔測定データには以下のリソース属性が必要です：

* `k8s.cluster.name` - Kubernetesスタックパックで使用される*クラスタ*名
* `k8s.namespace.name` - Rancher*プロジェクトによって*管理される*名前空間*

Kubernetes Operatorはデフォルトでこれらの属性を送信されるテレメトリデータに注入します。
