= SDKエクスポータの設定
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: SUSE Observability

SUSE Observabilityにデータを送信するには、アプリケーションのインスツルメンテーションに使用するSDKに組み込みのエクスポーターを使用します。本番環境でのセットアップでは、インスツルメンテッド・アプリケーションの近くにある<<_with_a_collector_production_setup,コレクターを>>使用してSUSE Observabilityにデータを送信しますが、インスツルメンテッド・アプリケーションが<<_without_a_collector,直接>>SUSE Observabilityにテレメトリ・データを送信することも可能です。

== コレクター付き（生産セットアップ）

=== Kubernetes用のSDK Exporter設定

言語に関係なく、すべての SDK は、Open Telemetryhttps://opentelemetry.io/docs/concepts/glossary/#_service[サービス名と]エクスポータ・エンドポイント（すなわち、テレメトリの送信先）を定義するために、同じ基本設定を使用します。

これらは、計測アプリケーションの環境変数を設定することで構成できる。

Kubernetesでは、ワークロードのマニフェストでこれらの環境変数を設定します（`<the-service-name>` をアプリケーションサービスの名前に置き換えてください）：

[,yaml]
----
...
spec:
  containers:
  - env:
    - name: OTEL_EXPORTER_OTLP_ENDPOINT
      value: http://opentelemetry-collector.open-telemetry.svc.cluster.local:4317
    - name: OTEL_SERVICE_NAME
      value: <the-service-name>
    - name: OTEL_EXPORTER_OTLP_PROTOCOL
      value: grpc
...
----

例で指定されているエンドポイントは、コレクタがxref:/setup/otel/collector.adoc[インストールガイドの] デフォルトを使用してインストールされていることを前提としています。ポート`4317` 、OTLPプロトコルの`gRPC` 。インストルメンテーションによってはHTTPしかサポートしていないものもあるので、その場合はポート`4318` を使ってください。

サービス名は、すでに存在する可能性のあるKubernetesラベルから派生させることもできる。例えばこんな感じだ：

[,yaml]
----
spec:
  containers:
  - env:
    - name: OTEL_SERVICE_NAME
      valueFrom:
        fieldRef:
          apiVersion: v1
          fieldPath: metadata.labels['app.kubernetes.io/component']
----

=== 他のインストレーションのSDK Exporter設定

SDKを設定するには、アプリケーション用に以下の環境変数を設定します：

[,bash]
----
export OTEL_EXPORTER_OTLP_ENDPOINT="http://<the-host-for-the-collector>:4317"
export OTEL_EXPORTER_OTLP_PROTOCOL="grpc"
export OTEL_SERVICE_NAME="<the-service-name>"
export OTEL_RESOURCE_ATTRIBUTES='service.namespace=<the-namespace>'
----

この例では、OTLPプロトコルの`gRPC` バージョンを使用するポート`4317` を使用している。インスツルメンテーションの中には、プロトコルを`http` に設定し、ポート`4318` を使用する HTTP のみをサポートするものもある。SDKがどのプロトコルをサポートしているかは、各言語のSDKドキュメントを参照してください。`OTEL_EXPORTER_OLTP_ENDPOINT` 」と「`OTEL_EXPORTER_OTLP_PROTOCOL` 」は省略可能で、ローカルホスト上の優先エンドポイントにデータを送信するデフォルト値が設定されている。

`OTEL_RESOURCE_ATTRIBUTES` はオプションで、サービス名前空間を定義する以外に、カンマ区切りのリストでリソース属性を設定することができます。

=== gRPC vs HTTP

OTLP（Open Telemetry Protocol）は、HTTP上でgRPCとprotobufをサポートする。前節では、エクスポーターのプロトコルは`gRPC` に設定されていた。SDKがgRPCをサポートしていないことの他に、HTTPを好む理由がある：

* 一部のファイアウォールはgRPCを処理するようにセットアップされていません。
* (リバース）プロキシやロードバランサーは、追加設定なしでgRPCをサポートしない場合があります。
* gRPCの長寿命接続は、負荷分散の際に問題を引き起こすかもしれない。

gRPCの代わりにHTTPに切り替えるには、プロトコルを`http` に変更_し、_ポート`4318` を使用します。

まとめると、gRPCがうまくいかない場合はHTTPを使ってみるといい：

* `grpc` プロトコルは、コレクタのポート`4317` を使用する。
* `http` プロトコルは、コレクタのポート`4318` を使用する。

== コレクターなし

小規模なテストセットアップでは、インスツルメンテッド・アプリケーションからSUSE Observabilityに直接データを送信すると便利です。上記のコレクターのセットアップとの唯一の違いは、`OTEL_EXPORTER_OTLP_ENDPOINT` に異なる値を使用することである：

* gRPCについては、SUSE ObservabilityのOTLPエンドxref:/setup/otel/otlp-apis.adoc[ポイントを]使用してください。
* HTTPでは、SUSE ObservabilityのOTLP over HTTPエンドxref:/setup/otel/otlp-apis.adoc[ポイントを]使用します。

[NOTE]
====
コレクタの*URLと*ポートの両方をSUSE Observabilityのエンドポイントに置き換えます。お使いのSUSE Observabilityのインストールによって、ポートは異なります。
====
