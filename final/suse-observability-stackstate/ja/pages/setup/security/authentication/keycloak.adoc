== 概要

SUSE Observabilityは、認証プロバイダとしてKeyCloakを使用して認証を行うため、SUSE ObservabilityとKeyCloakの両方が互いに通信できるように設定する必要があります。以下のセクションでは、それぞれのセットアップについて説明する。

== 認証フロー

認証プロバイダとしてKeycloakを使用する場合、SUSE ObservabilityはOIDC (OpenID Connect)を使用してユーザーを認証します。以下の図に、認証の流れを示す。

image::k8s/keycloak-flow.png[キークローク認証フロー]

== Configure KeyCloak

KeyCloakを使用して認証するようにSUSE Observabilityを設定する前に、KeyCloak認証サーバに新しいクライアント設定を追加する必要があります。クライアントに必要な設定は以下の通り：

* *クライアントID*- 接続するクライアントのID。 `stackstate`
* *クライアント・プロトコル* `openid-connect`
* *Access Type*-`confidential` に設定し、KeyCloakとSUSE Observability間の接続を確立するためにシークレットを使用する。
* *標準フロー有効*- に設定 `Enabled`
* *暗黙フロー有効*- に設定 `Disabled`
* *Root URL*- SUSE Observabilityのルートの場所(SUSE Observability設定のベースURLで設定された値と同じ)
* *有効なリダイレクト URI*- これは次のようになります。 `/loginCallback/*`
* *ベースURL*- SUSE Observabilityのルートロケーションを指す必要があります。

== SUSE Observabilityの設定

=== Kubernetes

KeyCloakを使用して認証するようにSUSE Observabilityを設定するには、KeyCloakの詳細とユーザロールマッピングをファイル`authentication.yaml` に追加する必要があります。例えば、こうだ：

[,yaml]
----
stackstate:
  authentication:
    keycloak:
      url: "https://keycloak.acme.com/auth"
      realm: acme
      authenticationMethod: client_secret_basic
      clientId: stackstate
      secret: "8051a2e4-e367-4631-a0f5-98fc9cdc564d"
      jwsAlgorithm: RS256
      # scope is optional. By default `openid`, `profile` and `email` are requested
      #_ scope: ["openid", "profile", "email"]
      # jwtClaims:
      #   usernameField: preferred_username
      #   groupsField: roles

    # map the roles from Keycloak to the
    # 3 standard subjects in SUSE Observability (guest, powerUser and admin)
    roles:
      guest: ["keycloak-guest-role-for-stackstate"]
      powerUser: ["keycloak-power-user-role-for-stackstate"]
      admin: ["keycloak-admin-role-for-stackstate"]
----

[NOTE]
====
*注：*
デフォルトでは、ユーザー認証の際、KeyCloakへのリクエストは、コンフィギュレーションでカスタムスコープが指定されていない場合、`openid profile email` のデフォルトスコープを指定する。KeyCloakインスタンスの`Client scopes` 、デフォルトのスコープが正しいか、カスタムのスコープが必要かを確認する。
====


以下の手順に従って、KeyCloakを使用して認証するようにSUSE Observabilityを設定します：

. `authentication.yaml` - KeyCloak認証プロバイダの詳細を追加する（上記の例を参照）。KeyCloak固有の値は、KeyCloakのクライアント設定から取得できる：
 ** *url*- KeyCloak インスタンスのベース URI。
 ** *realm*- 接続先の KeyCloak レルム。
 ** *authenticationMethod*-`client_secret_basic` に設定。現在、この値しかサポートされていない。
 ** *clientId*- KeyCloakで設定されているKeyCloakクライアントのID。
 ** *secret*- KeyCloakクライアントに付けられた秘密。このクライアントをKeyCloakに認証するために使われる。
 ** *redirectUri*- オプション：SUSE Observabilityのログインコールバックエンドポイントに到達可能なURI。デフォルトでは、`stackstate.baseUrl` を使用して入力されますが、オーバーライドすることもできます（`/loginCallback` のパスを指す完全修飾 URL である必要があります）。
 ** *jwsAlgorithm*- これを`RS256` に設定する。現在、この値しかサポートされていない。
 ** *jwtClaims*- オプション：ロールまたはユーザー名は、Keycloakのデフォルトの動作とは異なる属性から取得できます。
  *** *usernameField*- オプション：ユーザー名として使用されるOIDCユーザープロファイルのフィールド。デフォルトでは、`preferred_username` 。
  *** *groupsField*- オプション：SUSE Observabilityは常に、デフォルトではKeycloakが提供する`roles` 。しかし、ここで指定したフィールドからロールを追加することもできる。これは主にKeycloakがサードパーティシステムからロール/グループをマッピングする場合に便利です。
. `authentication.yaml` -`roles.guest`,`roles.powerUser` または`roles.admin` 設定を使用して、KeyCloakから正しいSUSE Observabilityサブジェクトにユーザロールをマッピングします（上記の例を参照してください）。詳細については、xref:/setup/security/rbac/rbac_permissions.adoc#_predefined_roles[デフォルトのSUSE Observabilityロールを]参照してください。その他のSUSE Observabilityロールも作成できます。xref:/setup/security/rbac/README.adoc[RBACドキュメントを]参照してください。
. SUSE Observabilityのインストール手順にある`values.yaml` ファイルと一緒に`authentication.yaml` 。
. Helmアップグレードを実行して変更を適用します：
+
[,text]
----
 helm upgrade \
   --install \
   --namespace suse-observability \
   --values values.yaml \
   --values authentication.yaml \
 suse-observability \
 suse-observability/suse-observability
----

[NOTE]
====
*注：*

* helm upgrade コマンドを最初に実行するとポッドが再起動し、可用性が短時間中断する可能性があります。
* `helm upgrade` 実行するたびに`authentication.yaml` を含める。
* 認証設定はKubernetesシークレットとして保存される。
====


=== 外部秘密の使用

キークロックシークレットを外部シークレットから取得する場合は、xref:/setup/security/external-secrets.adoc#_getting_authentication_data_from_an_external_secret[以下の手順に]従いますが、以下のデータを記入してください：

[,yaml]
----
kind: Secret
metadata:
   name: "<custom-secret-name>"
type: Opaque
data:
  keycloak_client_id: <base64 of client id>
  keycloak_secret: <base64 of secret>
----
