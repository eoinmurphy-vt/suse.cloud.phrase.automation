= 識別子
:revdate: 2025-07-10

== 概要

SUSE Observabilityでは、トポロジ要素(コンポーネントやリレーション)や関数などのオブジェクトを識別するために識別子が使用されます。このページでは、トポロジー要素に使用されるさまざまなタイプの識別子と、SUSE Observabilityでのグローバルスコープ識別子の構造について説明します。

== トポロジー識別子

SUSE Observabilityでは、トポロジー要素に2種類の識別子を使用します：

* *統合スコープ識別子*- 統合内のコンポーネントと関係を識別するために使用される。各コンポーネントまたはリレーションは、1つの統合スコープ識別子しか持たない。識別子は任意であるが、統合自体の範囲内で一貫していなければならない。
* *グローバルスコープ識別子*- ServiceNowとSUSE Observability Agentなどの統合間でコンポーネントをマージするために使用します。各コンポーネントは複数のグローバルスコープ識別子を持つことができるが、リレーションはグローバル識別子を持たない。これらはSUSE Observabilityによって割り当てられ、xref:/configure/topology/identifiers.adoc#_global_scope_identifiers[SUSE Observabilityグローバル識別子規約に従って]フォーマットされます。

以下のコード・サンプルは、両方のタイプの識別子を持つコンポーネントを示しています。

* 統合スコープ識別子 `this-host-unique-identifier`
* グローバルスコープ識別子 `urn:host:/this-host-fqdn`


[,text]
----
self.component("this-host-unique-identifier", "Host", {
    "name": "this-host",
    "domain": "Webshop",
    "layer": "Machines",
    "identifiers": ["urn:host:/this-host-fqdn"],
    "labels": ["host:this-host", "region:eu-west-1"],
    "environment": "Production"
})
----


== SUSE Observabilityエージェントの識別子

SUSE Observability Agentが同期トポロジ要素を識別するために使用するグローバルスコープ識別子を、以下の表に示します。

|===
| リソースタイプ | URN識別子の形式

| ホスト
| `urn:host:/[hostName]`

| プロセス
| `urn:process:/[hostName]:[pid]:[createTime]`

| コンテナ
| `urn:container:/[hostName]:[containerId]`

| 痕跡から発見されたサービス
| `urn:service:/[serviceName]`

| トレースで発見されたサービスインスタンス
| `urn:service-instance:/[serviceName]:/[hostName]`
|===

xref:/configure/topology/identifiers.adoc#_stackstate_agent[SUSE Observability AgentのURNグローバルスコープ識別子の例を]参照してください。

== グローバルスコープ識別子

SUSE Observabilityが異なる外部ソースからグローバルスコープ識別子が一致するコンポーネントを受信すると、コンポーネントとそのプロパティ(ラベル、ストリーム、チェック)が1つのコンポーネントにマージされます。これにより、さまざまなソースからのデータを組み合わせて、ITランドスケープの全体像を把握することが可能になる。

SUSE Observabilityのグローバルスコープ識別子は、次の規約に一致するグローバルに一意のURNです：

[,text]
----
urn:<prefix>:<type-name>:<free-form>
----

URN識別子にはすべての文字が許されるわけではないことに注意。以下のURN正規表現で識別子をチェックできる：

[,text]
----
^urn:[a-z0-9][a-z0-9-]{0,31}:[a-z0-9()+,\-.:=@;$_!*'%/?#]+$
----

`<prefix>` 、`<type-name>:<free-form>` セグメントのフォーマットは以下の通り。

=== プリフィックス

`prefix` セグメントは、グローバル識別子の必須部分である。これは、識別子が属するスコープに名前を付けるもので、純粋に組織的な目的のために使用される。

[NOTE]
====
プレフィックスにStackPack名が含まれている場合、そのオブジェクトはそのStackPackの管理下に置かれる。これは、StackPackがアンインストールされるときに、オブジェクトがアンインストールされることを意味する。
====


認識されるURN接頭辞は以下の通り：

* `stackpack:<name>` - オブジェクトはStackPacksに属する。
* `stackpack:<name>:shared` - オブジェクトは、StackPack のインスタンス間で共有される。
* `stackpack:<name>:instance:{{instanceId}}`-`{{instanceId}}` はハンドルバーで、StackPack の特定のインスタンスごとに、StackPack のインストールプロセスで提供された ID を返します。
* `system:auto` - 特定のStackPacksに属さない、システムによって作成されたオブジェクト。

=== タイプ名と自由形式

識別子は、`<type-name>:<free-form>` セグメントによって一意に識別される。

* `<type-name>` は、識別子が割り当てられているオブジェクトのドメイン・オブジェクト・タイプと一致します（大文字と小文字は区別されません）。
* `<free-form>` は任意であるが、型に対して一意でなければならない。自由形式のセグメントのフォーマットはユーザーが決める。オブジェクトの名前と一致する必要はなく（存在する場合）、それ自体が複数のセグメントで構成されていても構わない。

== 識別子の例

=== 共通スタックパック

一般的なStackPackのURNグローバルスコープ識別子の例。オブジェクトは、共通のStackPackがアンインストールされるときにアンインストールされます。

* コンポーネントタイプのサーバー：
 ** `urn:stackpack:common:component-type:server`

=== SUSE Observabilityエージェント

SUSE Observability AgentのURNグローバルスコープ識別子の例。

* ホストだ：
 ** `urn:host:/example.org`
* プロセス
 ** `urn:process:/db.infra.company.org:161841:1602158335000`
* コンテナ：
 ** `urn:container:/compnode5.k8s.example.org:8b18c68a820904c55b4909d7f5a9a52756d45e866c07c92bf478bcf6cd240901`
* トレースで発見されたサービス：
 ** `urn:service:/prod-db`
* トレースで発見されたサービスインスタンス：
 ** `urn:service-instance:/prod-db:/main.example.org`

=== その他のStackPacks

様々なStackPackからのURNグローバルスコープ識別子の例。オブジェクトは、指定された StackPack がアンインストールされるとアンインストールされます。

* AWS StackPackインスタンス間で共有されるAWS Event Run Stateの機能を確認します：
 ** `urn:stackpack:aws:shared:check-function:aws-event-run-state`
* ServiceNow StackPack の cmdb_ci_netgear というコンポーネントタイプ：
 ** `urn:stackpack:servicenow:componenttype:cmdb_ci_netgear`
