= Linguagem de consulta de observabilidade do SUSE (STQL)
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: Observabilidade do SUSE

== Visão geral

Esta página descreve como usar a SUSE Observability Query Language (STQL) integrada para escrever filtros avançados de componentes de topologia. As consultas STQL são usadas no SUSE Observability para escrever xref:/use/views/k8s-filters.adoc#_advanced_topology_filters[filtros de topologia avançados].

Uma consulta STQL consiste em <<_component_filters,filtros>> e <<_functions,funções>> <<_component_filters,de componentes>>. A saída da consulta é um componente ou um conjunto de componentes filtrados da topologia completa.

== Filtros de componentes

Os filtros de componentes são usados de duas maneiras no STQL:

* Defina o conjunto de componentes a serem incluídos na saída da consulta.
* Especifique o conjunto de componentes a serem tratados por uma função STQL incorporada.

=== Filtros

Os filtros descritos abaixo podem ser combinados usando os <<_operators,operadores>> disponíveis para obter seleções complexas de componentes.

|===
| Filtro | Padrão | Descrição

| `healthstate`
| "todos"
| Componentes com o estado de saúde nomeado.

| `label`
| "todos"
| Componentes com os rótulos nomeados.

| `name`
| "todos"
| Componentes com o nome especificado.

| `type`
| "todos"
| Componentes do tipo especificado.

| `identifier`
| "todos"
| Componentes com o identificador URN especificado. O filtro de identificador só é compatível com a filtragem básica quando é especificado usando `identifier IN (...)` e combinado com outros filtros usando um operador `OR`. Quando o filtro definido for compatível com a filtragem básica, o número de identificadores de componentes consultados será informado na xref:/use/views/k8s-filters.adoc#_other_filters[caixa*Outros filtros*].

| `layer`
| "todos"
| Componentes na camada nomeada.

| `domain`
| "todos"
| Componentes no(s) domínio(s) especificado(s).

| `environment`
| "todos"
| Componentes no ambiente nomeado.
|===

=== Operadores

Os operadores descritos abaixo estão disponíveis para uso em consultas STQL. Observe que os operadores booleanos serão executados na ordem padrão: NOT, OR, AND.

|===
| Operador | Descrição | Exemplo

| =
| Correspondência de igualdade
| `name = "cert-manager"`

| !=
| Correspondência de desigualdade
| `name != "coredns"`

| IN
| O valor está no subconjunto
| `name in ("cert-manager", "cluster_autoscaler")`

| NÃO
| Negação
| `name NOT in ("cert-manager", "cluster_autoscaler")`

| E e OU
| Filtrar com base em mais de uma condição ou subconsulta
| `name = "cert-manager" OR type = "deployment"`

| ()
| Use parênteses para agrupar resultados
| `(name = … AND type = …) OR (…)`
|===

Por exemplo:

[,yaml]
----
# Return all components named cert-manager or coredns regardless of type:
  name = "cert-manager" OR name = "coredns"

# Return only deployments named coredns and configmaps named kube-root-ca.crt:
  (name = "coredns" AND type = "deployment") OR (name = "kube-root-ca.crt" AND type = "configmap")
----

=== Curinga

Você pode usar `*` como um curinga completo em um filtro de componente. Não é possível filtrar correspondências parciais usando um caractere curinga.

=== Exemplos

[,text]
----
# Select all components
name = "*"

# Select all components with name "etcd-manager"
name = "etcd-manager"

# Select all components in the "Containers" layer:
layer = "Containers"

# Select all components named either "etcd-manager" or "coredns" that don't have a label "cluster-name:prod.stackstate.io"
name IN ("etcd-manager","coredns") NOT label = "cluster-name:prod.stackstate.io"

# Select all components named "coredns" that don't have a label "bck" or "test"
name = "cert-manager" NOT label in ("image_name:cert-manager/cert-manager-controller:testA", "image_name:cert-manager/cert-manager-controller:testB")
----

== Funções

=== withNeighboursOf

A função withNeighborsOf amplia a saída da consulta STQL, adicionando componentes conectados na(s) direção(ões) especificada(s). O número de níveis de topologia incluídos pode ser ajustado até um máximo de 15.

`withNeighborsOf(components=(), levels=, direction=)`

Para ser compatível com a filtragem básica, a função só pode ser combinada com outros filtros usando um operador `OR`. Quando um filtro avançado inclui uma função `withNeighborsOf` que é compatível com a filtragem básica, o número de componentes cujos vizinhos são consultados é mostrado na xref:/use/views/k8s-filters.adoc#_other_filters[caixa*Outros filtros*].

==== Parâmetros / campos

|===
| Parâmetro | Padrão | Valores permitidos | Descrição

| `components`
| "todos"
| Um filtro de componentes
| O(s) componente(s) para o(s) qual(is) os vizinhos serão retornados; consulte <<_component_filters,filtros de componentes>>.

| `levels`
| 1
| "all", [1:14]
| O número de níveis a serem incluídos na saída. Use "all" para exibir todos os níveis disponíveis (máximo de 15)

| `direction`
| "ambos"
| "para cima", "para baixo", "ambos"
| *up*: somente os componentes que dependem do(s) componente(s) nomeado(s) serão adicionados *down*: somente as dependências do(s) componente(s) nomeado(s) serão adicionadas *both*: componentes que dependem e dependências do(s) componente(s) nomeado(s) serão adicionados.
|===

==== Exemplo

O exemplo abaixo retornará todos os componentes da camada de aplicativos que têm um estado de integridade de `DEVIATING` ou `CRITICAL`. Os componentes com nomes "appA" ou "appB" e seus vizinhos também serão incluídos.

[,text]
----
layer = "Containers"
  AND (healthstate = "CRITICAL" OR healthstate = "DEVIATING")
  OR withNeighborsOf(components = (name in ("cert-manager","coredns")))
----

== Compatibilidade de filtros básicos e avançados

=== Filtragem básica a avançada

Você pode alternar da filtragem básica para a avançada selecionando *Advanced* em *Filter Topology (Topologia de filtro* ) no painel *View Filters (Exibir filtros* ).

É sempre possível alternar da filtragem básica para a avançada. Os filtros básicos selecionados serão convertidos diretamente em uma consulta STQL.

=== Filtragem avançada a básica

Você pode alternar da filtragem avançada para a básica selecionando *Basic* em *Filter Topology (Topologia de filtro* ) no painel *View Filters (Exibir filtros* ).

Nem sempre é possível alternar da filtragem avançada para a filtragem básica. As consultas simples do Mpst podem ser convertidas em filtros básicos, mas algumas consultas avançadas não são compatíveis com os filtros básicos.

* Os filtros básicos não podem conter uma desigualdade.
* Os filtros básicos não usam `=`, eles são sempre formatados usando o operador `IN`. Por exemplo, `name IN ("cert-manager”)` e não `name = "cert-manager”`.
* Os filtros básicos usam AND/OR de uma maneira específica:
 ** Todos os itens em cada caixa de filtro básico são unidos por um *OR*: `layer IN ("Containers", "Services", "Storage")`
 ** As diferentes caixas de filtro básicas são encadeadas com um *AND*: `layer IN ("Containers") AND domain IN ("cluster.test.stackstate.io”)`
 ** A caixa de filtro básico *Incluir componentes* (`name`) é a exceção - ela é encadeada às outras caixas de filtro com um OU: `layer IN ("Containers") AND domain IN ("cluster.test.stackstate.io") OR name IN ("cert-manager”)`
 ** Para ser compatível com a filtragem básica, a função *withNeighborsOf* e o filtro *de identificador* devem ser unidos a outros filtros com um *OR*: `layer in ("Containers") OR identifier IN ("urn:kubernetes:/cluster.test.stackstate.io:kube-system:pod/cert-manager-7749f44bb4-vspjj:container/cert-manager")`

Se você tentar mudar de um filtro avançado para um filtro básico e a consulta não for compatível, o SUSE Observability solicitará confirmação antes de remover os filtros incompatíveis. Para manter os filtros, você pode optar por permanecer na filtragem avançada.
