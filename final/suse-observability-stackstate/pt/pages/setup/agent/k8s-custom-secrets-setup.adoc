= k8s Gerenciamento de segredos personalizados
:revdate: 2025-10-27
:page-revdate: {revdate}

== Visão geral

O SUSE Observability Agent suporta a especificação do nome de um segredo personalizado que contém a chave da API e o token de autorização do cluster. Esse recurso é útil para usuários que desejam gerenciar seus próprios segredos e evitar a criação automática de segredos pelo Helm chart.

NOTE: *Havia um método anterior de especificar isso que agora está obsoleto, veja o xref:/setup/agent/k8s-custom-secrets-setup-deprecated.adoc[método obsoleto].*

== O gráfico do leme

=== Opções de configuração

* `global.apiKey.fromSecret`: Especifique um nome secreto pré-existente que resida no mesmo namespace e que contenha um campo `STS_API_KEY` contendo a chave da API.
* `global.clusterAgentAuthToken.fromSecret`: Especifique um nome secreto pré-existente que resida no mesmo namespace e que contenha um campo `STS_CLUSTER_AGENT_AUTH_TOKEN` com um token para proteger as conexões entre o cluster e os agentes de nó.

=== Descrição do comportamento

* *Criação automática de segredos*: Por padrão, o gráfico requer a especificação de um `stackstate.apiKey` e criará um segredo por si só. O site `STS_CLUSTER_AGENT_AUTH_TOKEN` é gerado automaticamente.
* *Gerenciamento de segredos personalizado*: Ao substituir os campos `fromSecret`, a chave da API e o token de autenticação do cluster serão extraídos desses segredos.
* *Omissão implícita*: Ao especificar que você gostaria de gerenciar seus próprios segredos, o gráfico ignorará os valores de `stackstate.apiKey` e `stackstate.cluster.authToken`.

== Como usar em values.yaml

. *Usando a criação automática de segredos (padrão)*:
+
[,yaml]
----
 stackstate:
   apiKey: "<your api key>"
----

. *Gerenciando seus próprios segredos*:
+
[,yaml]
----
 global:
   apiKey:
     fromSecret: "name-of-my-api-key-secret"
   clusterAgentAuthToken:
     fromSecret: "name-of-my-cluster-agent-auth-token-secret"
----
