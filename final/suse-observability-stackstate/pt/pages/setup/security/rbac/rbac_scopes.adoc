= Escopos de topologia
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: SUSE Observability Self-hosted

WARNING: O escopo de topologia por consulta é uma funcionalidade obsoleta. Ela é substituída pela permissão `get-topology`, que pode conceder acesso a qualquer componente com um rótulo específico.

== Como funcionam os escopos de topologia?

O escopo da topologia é uma xref:/develop/reference/k8sTs-stql_reference.adoc[consulta STQL] que é combinada com as permissões `get-topology` como um prefixo para cada consulta de topologia executada no SUSE Observability. Quando um usuário deseja selecionar uma visualização ou passar uma consulta no SUSE Observability, essa consulta de prefixo é executada como parte da consulta do usuário. Isso limita os resultados para que correspondam à função do usuário.

NOTE: Chamadas de função como `withCauseOf` e `withNeighborsOf` não são suportadas, pois não seriam eficazes nesse contexto.

Se um usuário pertencer a vários grupos, ele poderá ter vários escopos de topologia, o que se traduz em vários prefixos. Aqui, o prefixo é executado como um OU de todos os escopos de topologia que o usuário possui.

Os usuários precisam fazer logout e autenticar-se novamente no SUSE Observability sempre que suas funções ou permissões forem alteradas.

== Por que escopos de topologia?

Os escopos de topologia são um recurso de segurança para assuntos dentro do SUSE Observability. As funções predefinidas do SUSE Observability Administrator, Power User e Guest não têm escopo de topologia definido.

É possível especificar um escopo de topologia como um curinga de consulta; no entanto, isso resulta em acesso a tudo e não é recomendado. Se houver necessidade de acesso sem um escopo de topologia, é recomendável usar uma das xref:/setup/security/rbac/rbac_permissions.adoc#_predefined_roles[funções predefinidas].

== Exemplos

O exemplo abaixo mostra a mesma visualização de topologia chamada "All Infrastructure" para quatro usuários com diferentes níveis de permissão.

=== Esse usuário faz parte do grupo SUSE Observability Admin, portanto, não há escopo de topologia:

image::v51_allperm.png[Permissões de visualização completa]

A consulta para essa exibição é a mesma que para as outras, mas sem nenhum prefixo:

[,text]
----
'layer = "Infrastructure" AND domain IN ("Customer1", "Customer2")'
----

=== O usuário abaixo está em um grupo com o assunto X configurado com o seguinte escopo de topologia:

[,text]
----
'domain = "Customer1"'
----

image::v51_esx1perm.png[Visão limitada]

A consulta com o prefixo para essa exibição é:

[,text]
----
'(domain = "Customer1") AND (layer = "Infrastructure" AND domain IN ("Customer1", "Customer2"))'
----

=== Outro usuário que faz parte de um grupo com um assunto Y configurado que tem o seguinte escopo de topologia:

[,text]
----
'domain = "Customer2"'
----

obtém essa topologia:

image::v51_esx2perm.png[Visão limitada]

A consulta com o prefixo para essa exibição é:

[,text]
----
'(domain = "Customer2") AND (layer = "Infrastructure" AND domain IN ("Customer1", "Customer2"))'
----

=== Usuário com vários prefixos

É possível atribuir um assunto a mais de um grupo. Neste exemplo, você pode ver um gerente de infraestrutura que pode ver toda a visualização apresentada acima. Esse usuário precisa estar em ambos os grupos que configuraram os assuntos como X e Y. Nesse caso, o prefixo da consulta do usuário será semelhante ao seguinte:

[,text]
----
'(domain = "Customer1" OR domain = "Customer2")'
----

A consulta com o prefixo para esse usuário é então:

[,text]
----
'(domain = "Customer1" OR domain = "Customer2") AND (layer = "Infrastructure" AND domain IN ("Customer1", "Customer2"))'
----

O que resulta na seguinte exibição:
