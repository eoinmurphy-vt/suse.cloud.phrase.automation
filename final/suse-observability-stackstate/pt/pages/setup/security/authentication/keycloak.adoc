== Visão geral

O SUSE Observability pode se autenticar usando o KeyCloak como provedor de autenticação; você precisará configurar o SUSE Observability e o KeyCloak para que possam se comunicar entre si. As seções a seguir descrevem as respectivas configurações.

== Fluxo de autenticação

Ao usar o Keycloak como provedor de autenticação, o SUSE Observability usará o OIDC (OpenID Connect) para autenticar os usuários. O diagrama a seguir descreve o fluxo de autenticação.

image::k8s/keycloak-flow.png[Fluxo de autenticação do Keycloak]

== Configurar o KeyCloak

Antes de configurar o SUSE Observability para autenticar usando o KeyCloak, você precisa adicionar uma nova configuração de cliente ao KeyCloak Authentication Server. As configurações necessárias para o cliente são:

* *Client ID (* ID do cliente) - A ID do cliente que está se conectando; recomendamos nomeá-la `stackstate`
* *Protocolo do cliente* - Defina como `openid-connect`
* *Access Type (Tipo de acesso* ) - Defina como `confidential`, para que um segredo seja usado para estabelecer a conexão entre o KeyCloak e o SUSE Observability
* *Fluxo padrão ativado* - Defina como `Enabled`
* *Implicit Flow Enabled (Fluxo implícito ativado* ) - Defina como `Disabled`
* *URL da raiz* - O local da raiz do SUSE Observability (o mesmo valor configurado no URL de base da configuração do SUSE Observability)
* *URIs de redirecionamento válidos* - Devem ser `/loginCallback/*`
* *URL base* - Deve apontar para o local da raiz do SUSE Observability

== Configurar o SUSE Observability

=== Kubernetes

Para configurar o SUSE Observability para autenticar usando o KeyCloak, os detalhes do KeyCloak e o mapeamento da função do usuário precisam ser adicionados ao arquivo `authentication.yaml`. Por exemplo:

[,yaml]
----
stackstate:
  authentication:
    keycloak:
      url: "https://keycloak.acme.com/auth"
      realm: acme
      authenticationMethod: client_secret_basic
      clientId: stackstate
      secret: "8051a2e4-e367-4631-a0f5-98fc9cdc564d"
      jwsAlgorithm: RS256
      # scope is optional. By default `openid`, `profile` and `email` are requested
      #_ scope: ["openid", "profile", "email"]
      # jwtClaims:
      #   usernameField: preferred_username
      #   groupsField: roles

    # map the roles from Keycloak to the
    # 3 standard subjects in SUSE Observability (guest, powerUser and admin)
    roles:
      guest: ["keycloak-guest-role-for-stackstate"]
      powerUser: ["keycloak-power-user-role-for-stackstate"]
      admin: ["keycloak-admin-role-for-stackstate"]
----

[NOTE]
====
*Observação:*
Por padrão, ao autenticar um usuário, a solicitação ao KeyCloak especifica um escopo padrão de `openid profile email` se um escopo personalizado não tiver sido especificado na configuração. Verifique o endereço `Client scopes` em sua instância do KeyCloak para ter certeza de que o escopo padrão está correto ou de que você precisa de um escopo personalizado.
====


Siga as etapas abaixo para configurar o SUSE Observability para autenticar usando o KeyCloak:

. Em `authentication.yaml` - adicione detalhes do provedor de autenticação KeyCloak (veja o exemplo acima). Os valores específicos do KeyCloak podem ser obtidos na configuração do cliente no KeyCloak:
 ** *url* - O URI de base para a instância do KeyCloak
 ** *realm* - O realm do KeyCloak ao qual se conectar
 ** *authenticationMethod* - Defina como `client_secret_basic`, esse é atualmente o único valor compatível.
 ** *clientId* - O ID do cliente do KeyCloak, conforme configurado no KeyCloak.
 ** *secret* - O segredo anexado ao cliente do KeyCloak, que é usado para autenticar esse cliente no KeyCloak
 ** *redirectUri* - opcional: O URI em que o ponto de extremidade de retorno de chamada de login do SUSE Observability pode ser acessado. Preenchido por padrão usando o endereço `stackstate.baseUrl`, mas pode ser substituído (deve ser um URL totalmente qualificado que aponte para o caminho `/loginCallback` )
 ** *jwsAlgorithm* - Defina esse valor como `RS256`, atualmente esse é o único valor compatível.
 ** *jwtClaims* - Opcional: As funções ou o nome de usuário podem ser recuperados de um atributo diferente do comportamento padrão do Keycloak
  *** *usernameField* - opcional: O campo no perfil de usuário do OIDC que deve ser usado como nome de usuário. Por padrão, esse será o `preferred_username`.
  *** *groupsField* - Opcional: O SUSE Observability sempre usará, e por padrão apenas, o `roles` Keycloak fornecido. Mas ele também pode adicionar funções do campo especificado aqui. Isso é útil principalmente quando o Keycloak está mapeando funções/grupos de um sistema de terceiros.
. Em `authentication.yaml` - mapeie as funções de usuário do KeyCloak para os assuntos corretos do SUSE Observability usando as configurações `roles.guest`, `roles.powerUser` ou `roles.admin` (veja o exemplo acima). Para obter detalhes, consulte as xref:/setup/security/rbac/rbac_permissions.adoc#_predefined_roles[funções padrão do SUSE Observability]. Mais funções do SUSE Observability também podem ser criadas, consulte a xref:/setup/security/rbac/README.adoc[documentação do RBAC].
. Armazene o arquivo `authentication.yaml` junto com o arquivo `values.yaml` das instruções de instalação do SUSE Observability.
. Execute um upgrade do Helm para aplicar as alterações:
+
[,text]
----
 helm upgrade \
   --install \
   --namespace suse-observability \
   --values values.yaml \
   --values authentication.yaml \
 suse-observability \
 suse-observability/suse-observability
----

[NOTE]
====
*Observação:*

* A primeira execução do comando helm upgrade resultará na reinicialização dos pods, o que pode causar uma breve interrupção da disponibilidade.
* Inclua `authentication.yaml` em cada execução de `helm upgrade`.
* A configuração de autenticação é armazenada como um segredo do Kubernetes.
====


=== Usando um segredo externo

Quando os segredos do keycloak forem provenientes de um segredo externo, siga xref:/setup/security/external-secrets.adoc#_getting_authentication_data_from_an_external_secret[estas etapas], mas preencha os dados a seguir:

[,yaml]
----
kind: Secret
metadata:
   name: "<custom-secret-name>"
type: Opaque
data:
  keycloak_client_id: <base64 of client id>
  keycloak_secret: <base64 of secret>
----
