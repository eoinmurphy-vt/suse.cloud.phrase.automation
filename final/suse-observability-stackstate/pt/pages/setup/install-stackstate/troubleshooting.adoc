== Guia rápido de solução de problemas

Aqui está um guia rápido para solucionar problemas de inicialização do SUSE Observability:

. Verifique se a instalação foi concluída com êxito e se a versão está listada:
+
[,text]
----
helm list --namespace suse-observability
----

. Verifique se todos os pods no namespace do SUSE Observability estão em execução:
+
[,text]
----
kubectl get pods
----
+
Em uma primeira implantação, pode ser que os contêineres em vários pods sejam reiniciados algumas vezes, porque estão esperando que outros pods sejam iniciados e estejam no estado `ready`. Isso pode ser atrasado devido a atrasos no agendamento e na extração da imagem do docker.
+
Os pods que estão no estado `pending` geralmente são uma indicação de um problema:

 ** O pod não pode ser programado devido à falta de recursos no cluster. Se um escalonador automático de cluster estiver ativo, ele poderá resolver isso automaticamente; caso contrário, será necessária uma intervenção manual para adicionar mais nós ao cluster
 ** O pod não é programável, há nós nos quais ele caberia, mas esses nós têm `taints` que o pod não tolera. Para resolver esse problema, é possível adicionar mais nós que não tenham as impurezas, mas o SUSE Observability também pode ser xref:/setup/install-stackstate/kubernetes_openshift/customize_config.adoc#_override_default_configuration[configurado] para tolerar determinadas impurezas e ser executado nos nós contaminados.
 ** O pod está aguardando que os Volumes Persistentes (PVs) sejam montados. Uma causa pode ser o fato de o gráfico do SUSE Observability Helm não especificar um `storageClassName`, mas confiar no fato de o cluster ter uma classe de armazenamento padrão. Quando não há um padrão para o cluster, é necessário xref:/setup/install-stackstate/kubernetes_openshift/storage.adoc[especificar uma classe de armazenamento] por meio dos valores do Helm do SUSE Observability.

+
Para pods com estado `ImagePullBackOff`, verifique também a mensagem de erro exata. As causas comuns são:

 ** Um nome de usuário/senha incorretos usados para extrair as imagens
 ** A conexão com o registro da plataforma falhou; isso pode ocorrer devido a problemas de autenticação ou de conectividade (firewalls, instalações com air-gap)
 ** Um erro de digitação no URL do registro de imagem do docker substituído

+
Para descobrir uma causa mais detalhada para os estados `Pending`, `ImagePullBackOff` ou `CrashLoopBackOff`, use este comando:
+
[,text]
----
kubectl describe pod <pod-name>
----
+
O resultado contém uma seção `event` no final, que geralmente contém o problema. Ele também tem uma seção `State` para cada contêiner, com mais detalhes sobre o encerramento do contêiner.

. Quando você for um cliente principal, entre em contato com o suporte do SUSE Observability em https://scc.suse.com/ para obter ajuda na configuração do SUSE Observability em seu cluster local. Use xref:/setup/install-stackstate/support-package-logs.adoc[o Support Package (Logs)] para coletar informações sobre a sua instância para a equipe de suporte.
. Se o problema estiver relacionado ao desempenho, execute o xref:/setup/install-stackstate/support-package-performance.adoc[Support Package (Performance)] para investigar ativamente o desempenho.
. Caso as etapas acima não tenham resolvido o problema, há um xref:/setup/install-stackstate/advanced-troubleshooting.adoc[Guia avançado de solução de problemas] disponível.
