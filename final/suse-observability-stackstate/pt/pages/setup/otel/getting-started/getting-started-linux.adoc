= Primeiros passos com a telemetria aberta
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: Observabilidade do SUSE

Esta é a configuração que criaremos para um aplicativo que precisa ser monitorado:

* O aplicativo/carga de trabalho monitorado em execução em um host Linux
* O coletor Open Telemetry em execução no mesmo host Linux
* SUSE Observability ou SUSE Cloud Observability

image::otel/open-telemetry-collector-linux.png[Instrumentação de aplicativos em um host Linux com o coletor Open Telemetry em execução no host]

== Instalar o coletor Open Telemetry

[NOTE]
====
Para uma configuração de produção, é altamente recomendável instalar o coletor, pois ele permite que seu serviço descarregue dados rapidamente e o coletor pode cuidar do tratamento adicional, como novas tentativas, lotes, criptografia ou até mesmo filtragem de dados confidenciais.
====


Primeiro, instalaremos o coletor. Nós o configuramos para:

* Receber dados de, potencialmente, muitos aplicativos instrumentados
* Enriquecer os dados coletados com atributos do host
* Gerar métricas para traços
* Encaminhar os dados para o SUSE Observability, incluindo a autenticação usando a chave da API

Além disso, ele também tentará enviar dados novamente quando houver problemas de conexão.

=== Criar um token de serviço

Há duas maneiras de criar um token de serviço:

* **SUSE Observability UI** - abra o menu principal clicando no canto superior esquerdo da tela e vá para `StackPacks` > `Open Telemetry`.  Se você não tiver feito isso antes, clique no botão `INSTALL`.  Clique no botão `CREATE NEW SERVICE TOKEN` e copie o valor para sua área de transferência.
* **CLI do SUSE Observability** - consulte xref:/use/security/k8s-service-tokens.adoc#_manage_service_tokens[Gerenciar tokens de serviço]

O valor do token de serviço deve ser usado quando as instruções abaixo mencionarem `<SERVICE_TOKEN>`.

=== Instalar e configurar o coletor

O coletor fornece pacotes (apk, deb e rpm) para a maioria das versões e arquiteturas do Linux e usa o site `systemd` para a configuração automática do serviço. Para instalá-lo, encontre a https://github.com/open-telemetry/opentelemetry-collector-releases/releases[versão mais recente do] e atualize o URL no exemplo para usar a versão mais recente:

[tabs]
====

sudo dpkg -i otelcol-contrib_0.123.1_linux_amd64.deb

====

Para outras opções de instalação, use as https://opentelemetry.io/docs/collector/installation/#_linux[instruções do site].

Após a instalação, modifique a configuração do coletor editando `/etc/otelcol-contrib/config.yaml`. Altere o arquivo de modo que ele se pareça com o exemplo `config.yaml` aqui, substitua `<otlp-suse-observability-endpoint:port>` pelo seu endpoint OTLP (consulte a xref:/setup/otel/otlp-apis.adoc[API OTLP] para obter seu endpoint) e insira seu token de serviço para `<SERVICE_TOKEN>`:

.config.yaml
[,yaml]
----
receivers:
  otlp:
    protocols:
      # Only bind to localhost to keep the collector secure, see https://github.com/open-telemetry/opentelemetry-collector/blob/main/docs/security-best-practices.md#safeguards-against-denial-of-service-attacks
      grpc:
        endpoint: 127.0.0.1:4317
      http:
        endpoint: 127.0.0.1:4318
  # Collect own metrics
  prometheus:
    config:
      scrape_configs:
      - job_name: 'otel-collector'
        scrape_interval: 10s
        static_configs:
        - targets: ['0.0.0.0:8888']
extensions:
  health_check: {}
  pprof:
    endpoint: 0.0.0.0:1777
  zpages:
    endpoint: 0.0.0.0:55679
  # Use the API key from the env for authentication
  bearertokenauth:
    scheme: SUSEObservability
    token: "<SERVICE_TOKEN>"
exporters:
  nop: {}
  debug: {}
  otlp/suse-observability:
    compression: snappy
    auth:
      authenticator: bearertokenauth
    # Put in your own otlp endpoint, for example otlp-suse-observability.my.company.com:443
    endpoint: <otlp-suse-observability-endpoint:port>
processors:
  memory_limiter:
    check_interval: 5s
    limit_percentage: 80
    spike_limit_percentage: 25
  batch: {}
  # Optionally include resource information from the system running the collector
  resourcedetection/system:
    detectors: [env, system] # Replace system with gcp, ec2, azure when running in cloud environments
    system:
      hostname_sources: ["os"]
connectors:
  # Generate metrics for spans
  spanmetrics:
    metrics_expiration: 5m
    namespace: otel_span
service:
  extensions: [ bearertokenauth, health_check, pprof, zpages ]
  pipelines:
    traces:
      receivers: [otlp]
      processors: [memory_limiter, resourcedetection/system, batch]
      exporters: [debug, spanmetrics, otlp/suse-observability]
    metrics:
      receivers: [otlp, spanmetrics, prometheus]
      processors: [memory_limiter, batch, resourcedetection/system]
      exporters: [debug, otlp/suse-observability]
    logs:
      receivers: [otlp]
      processors: []
      exporters: [nop]
----


Por fim, reinicie o coletor:

[,bash]
----
sudo systemctl restart otelcol-contrib
----

Para ver os registros do coletor, use:

[,bash]
----
sudo journalctl -u otelcol-contrib
----

== Colete dados de telemetria do seu aplicativo

A maneira comum de coletar dados de telemetria é instrumentar seu aplicativo usando o Open Telemetry SDK. Documentamos alguns guias de início rápido para alguns idiomas, mas há muitos outros:

* xref:/setup/otel/instrumentation/java.adoc[Java]
* xref:/setup/otel/instrumentation/dot-net.adoc[.NET]
* xref:/setup/otel/instrumentation/node.js.adoc[Node.js]

Não é necessária nenhuma configuração adicional para os SDKs, pois eles exportam para o localhost via OTLP ou OTLP sobre HTTP (dependendo dos protocolos compatíveis) por padrão.

Para outros idiomas, siga a documentação em https://opentelemetry.io/docs/languages/[opentelemetry.io].

== Veja os resultados

Acesse o SUSE Observability e verifique se o Open Telemetry Stackpack está instalado (no menu principal \-> Stackpacks).

Depois de algum tempo e se o seu aplicativo estiver processando algum tráfego, você poderá encontrá-lo com o nome do serviço nas visões gerais de Open Telemetry \-> serviços e instâncias de serviço. Os rastreamentos aparecerão no xref:/use/traces/k8sTs-explore-traces.adoc[explorador] de rastreamento e na xref:/use/views/k8s-traces-perspective.adoc[perspectiva de rastreamento] dos componentes de serviço e instância de serviço. As métricas de abrangência e as métricas específicas do idioma (se disponíveis) ficarão disponíveis na xref:/use/views/k8s-metrics-perspective.adoc[perspectiva de métricas] dos componentes.
