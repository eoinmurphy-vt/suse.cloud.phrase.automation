= Conceitos de telemetria aberta
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: Observabilidade do SUSE

Este é um resumo dos conceitos mais importantes da Telemetria Aberta e deve ser suficiente para começar. Para obter uma introdução mais detalhada, use a https://opentelemetry.io/docs/concepts/[documentação do]

== Sinais

O Open Telemetry reconhece 3 sinais de telemetria:

* Traços
* Métricas
* Registros

No momento, o SUSE Observability é compatível com rastreamentos e métricas; os registros serão compatíveis em uma versão futura. Para os registros do Kubernetes, é possível usar o xref:/k8s-quick-start-guide.adoc[agente SUSE Observability].

=== Traços

Os rastreamentos nos permitem visualizar o caminho de uma solicitação em seu aplicativo. Um traço consiste em um ou mais vãos que, juntos, formam uma árvore. Um único traço pode estar inteiramente dentro de um único serviço, mas também pode atravessar vários serviços. Cada intervalo representa uma operação no processamento da solicitação e tem:

* um nome
* hora de início e de término, a partir da qual uma duração pode ser calculada
* status
* atributos
* atributos de recursos (consulte <<_resources,recursos>>)
* eventos

Os atributos do intervalo são usados para fornecer metadados para o intervalo, por exemplo, um intervalo para uma operação que coloca um pedido pode ter o `orderId` como atributo, ou um intervalo para uma operação HTTP pode ter o método HTTP e o URL como atributos.

Os eventos do vão podem ser usados para representar um ponto no tempo em que algo importante aconteceu na operação do vão. Por exemplo, se o intervalo falhar, pode haver um evento `exception` ou `error` que capture a mensagem de erro, um stacktrace e o momento exato em que o erro ocorreu.

=== Métricas

As métricas são medições capturadas no tempo de execução e resultam em um evento de métrica. As métricas são indicadores importantes para o desempenho e a disponibilidade do aplicativo e são frequentemente usadas para alertar sobre uma interrupção ou um problema de desempenho. As métricas têm:

* um nome
* um registro de data e hora
* um tipo (contador, medidor, histograma, etc.)
* atributos
* atributos de recursos (consulte <<_resources,recursos>>)

Os atributos fornecem os metadados de uma métrica.

== Recursos

Um recurso é a entidade que produz os dados de telemetria. Os atributos do recurso fornecem os metadados do recurso. Por exemplo, um processo em execução em um contêiner, em um pod, em um namespace em um cluster do Kubernetes pode ter atributos de recursos para todas essas entidades.

Os atributos de recursos geralmente são atribuídos automaticamente pelos SDKs. No entanto, é recomendável sempre definir explicitamente os atributos `service.name` e `service.namespace`. O primeiro é o nome lógico do serviço; se não for definido, o SDK definirá um valor `unknown_service`, o que dificultará muito o uso dos dados posteriormente no SUSE Observability. O namespace é uma maneira conveniente de organizar seus serviços, especialmente útil se você tiver os mesmos serviços em execução em vários locais.

== Convenções semânticas

A Open Telemetry define nomes comuns para operações e dados, o que eles chamam de convenções semânticas. As convenções semânticas seguem um esquema de nomenclatura que permite a padronização do processamento de dados entre linguagens, bibliotecas e bases de código. Há convenções semânticas para todos os sinais e para os atributos de recursos. Eles são definidos para muitas plataformas e operações diferentes no https://opentelemetry.io/docs/specs/semconv/attributes-registry/[site]. Os SDKs usam as convenções semânticas para atribuir esses atributos, e o SUSE Observability também respeita as convenções e se baseia nelas, por exemplo, para reconhecer os recursos do Kubernetes.
