= Adicionar gráficos personalizados aos componentes

== Visão geral

{stackstate-product-name} já fornece muitos gráficos de métricas por padrão na maioria dos tipos de componentes que representam recursos do Kubernetes. Gráficos métricos extras podem ser adicionados a qualquer conjunto de componentes sempre que necessário. Ao adicionar métricas aos componentes, há duas opções:

. As métricas já são coletadas pelo site {stackstate-product-name}, mas não são visualizadas em um componente, por padrão
. As métricas ainda não foram coletadas pelo site {stackstate-product-name} e, portanto, ainda não estão disponíveis

Para a opção 1, as etapas abaixo o instruirão sobre como criar uma vinculação de métrica que configurará o site {stackstate-product-name} para adicionar uma métrica específica a um conjunto específico de componentes.

Para a opção 2, garanta que as métricas estejam disponíveis em {stackstate-product-name} enviando-as para {stackstate-product-name} usando o xref:/use/metrics/k8s-prometheus-remote-write.adoc[protocolo de gravação remota do Prometheus]. Continue adicionando gráficos para as métricas aos componentes SOMENTE depois de garantir que as métricas estejam disponíveis.

Por exemplo, as etapas adicionarão uma associação de métrica para `Replica counts` das implantações do Kubernetes. Essa vinculação de métrica já existe em {stackstate-product-name}, por padrão.

=== Criar um esboço da ligação métrica

Abra o arquivo `metricbindings.yaml` YAML em seu editor de código favorito para alterá-lo ao longo deste guia.  Você pode usar a CLI para xref:/setup/custom-integrations/develop.adoc#_test_your_stackpack[testar seu StackPack].

----
- _type: MetricBinding
  chartType: line
  enabled: true
  identifier: urn:stackpack:my-stackpack:shared:metric-binding:node-memory-bytes-available-scheduling
  layout:
    metricPerspective:
      section: Resources
      tab: Kubernetes Node
  name: Memory available for scheduling (Custom)
  priority: medium
  queries:
  - alias: ${cluster_name} - ${node}
    expression: max_over_time(kubernetes_state_node_memory_allocatable{cluster_name="${tags.cluster-name}", node="${name}"}[${__interval}])
  scope: (label = "stackpack:kubernetes" and type = "node")
  unit: bytes(IEC)
----

A seção de consultas e escopo será preenchida nas próximas etapas. Observe que a unidade usada é `short`, que simplesmente renderizará um valor numérico. Caso ainda não tenha certeza sobre a unidade da métrica, você pode deixá-la em aberto e decidir a unidade correta ao escrever a consulta PromQL.

=== Selecione os componentes aos quais se vincular

Salve uma visualização da xref:/use/views/k8s-topology-perspective.adoc[perspectiva da Topologia] e use os filtros (Filtros -> Topologia -> Alternar para STQL) para consultar os componentes que precisam mostrar a nova métrica. Os campos mais comuns para selecionar a topologia para ligações métricas são `type` para o tipo de componente e `label` para selecionar todos os rótulos. Por exemplo, para as implantações:

----
type = "deployment" and label = "stackpack:kubernetes"
----

O filtro de tipo seleciona todas as implantações, enquanto o filtro de rótulo seleciona apenas os componentes criados pelo stackpack do Kubernetes (o nome do rótulo é `stackpack` e o valor do rótulo é `kubernetes`). O último também pode ser omitido para obter o mesmo resultado.  Todos os xref:/develop/reference/k8sTs-stql_reference.adoc#_component_filters[filtros de componentes de consulta STQL] podem ser usados para filtragem.

Alterne para o modo avançado para copiar a consulta de topologia resultante e colocá-la no campo `scope` da ligação métrica.

[NOTE]
====
Os vínculos de métricas são compatíveis apenas com os filtros de consulta. Funções de consulta como `withNeighborsOf` não são compatíveis e não podem ser usadas.
====


=== Escreva a consulta PromQL

Acesse o xref:/use/metrics/k8sTs-explore-metrics.adoc[explorador de métricas] de sua instância {stackstate-product-name}, http://your-instance/#/metrics, e use-o para consultar a métrica de interesse. O explorador tem preenchimento automático para métricas, rótulos, valores de rótulos, mas também funções PromQL e operadores para ajudá-lo. Comece com um intervalo de tempo curto, por exemplo, de uma hora, para obter os melhores resultados.

Para obter o número total de réplicas, use a métrica `kubernetes_state_deployment_replicas`. Para mostrar os gráficos de métricas dos dados de série temporal, estenda a consulta para fazer uma agregação usando o parâmetro `${__interval}`:

----
max_over_time(kubernetes_state_deployment_replicas[${__interval}])
----

Nesse caso específico, use `max_over_time` para garantir que o gráfico sempre mostre o maior número de réplicas em um determinado momento. Para intervalos de tempo mais longos, uma pequena queda nas réplicas não é mostrada. Para enfatizar o menor número de réplicas, use `min_over_time`.

Copie a consulta na propriedade `expression` da primeira entrada no campo `queries` da ligação métrica. Use `Total replicas` como um alias. para que ele seja exibido na legenda do gráfico.

[NOTE]
====
Em {stackstate-product-name}, o tamanho do gráfico de métricas determina automaticamente a granularidade da métrica mostrada no gráfico. As consultas PromQL podem ser ajustadas para otimizar o uso desse comportamento e obter um gráfico representativo da métrica. xref:/setup/custom-integrations/metric-bindings/writing-promql.adoc[Escrever PromQL para gráficos] explica isso em detalhes.
====


=== Associe a série temporal correta a cada componente

A ligação métrica com todos os campos preenchidos:

----
_type: MetricBinding
chartType: line
enabled: true
tags: {}
unit: short
name: Replica counts
priority: MEDIUM
identifier: urn:stackpack:my-stackpack:metric-binding:my-deployment-replica-counts
queries:
  - expression: max_over_time(kubernetes_state_deployment_replicas[${__interval}])
    alias: Total replicas
scope: type = "deployment" and label = "stackpack:kubernetes"
----

Criá-lo em {stackstate-product-name} e visualizar o gráfico "Replica count" (Contagem de réplicas) em um componente de implantação apresenta um resultado inesperado. O gráfico mostra as contagens de réplicas para todas as implantações. Logicamente, seria de se esperar apenas uma série temporal: a contagem de réplicas para essa implantação específica.

image::k8s/k8s-replica-counts-without-binding.png[O gráfico incorreto para uma única implantação, it shows the replica count for all deployments]

Para corrigir isso, faça a consulta PromQL específica para um componente usando informações do componente. Filtre com base em rótulos de métricas suficientes para selecionar apenas a série temporal específica do componente. Essa é a "vinculação" da série temporal correta ao componente. Para quem tem experiência na criação de painéis do Grafana, isso é semelhante a um painel com parâmetros que são usados em consultas no painel. Vamos alterar a consulta na vinculação de métricas para isso:

----
max_over_time(kubernetes_state_deployment_replicas{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", deployment="${name}"}[${__interval}])
----

image::k8s/k8s-replica-counts-with-binding.png[Depois de adicionar os filtros parametrizados, o gráfico resultante tem a aparência esperada, only one time series for this component]

A consulta PromQL agora filtra em três rótulos, `cluster_name`, `namespace` e `deployment`. Em vez de especificar um valor real para esses rótulos, é usada uma referência variável aos campos do componente. Nesse caso, são usados os rótulos `cluster-name` e `namespace`, referenciados por `${tags.cluster-name}` e `${tags.namespace}`. Além disso, o nome do componente é referenciado com `${name}`.

As referências de variáveis compatíveis são:

* Qualquer rótulo de componente, usando `${tags.<label-name>}`
* O nome do componente, usando `${name}`

image::k8s/k8s-carts-highlights.png[Página Destaques do componente que mostra os rótulos e o nome do componente (ambos destacados em vermelho)]

[NOTE]
====
O nome do cluster, o namespace e uma combinação do tipo e do nome do componente geralmente são suficientes para selecionar as métricas de um componente específico do Kubernetes. Essas etiquetas, ou etiquetas semelhantes, geralmente estão disponíveis na maioria das métricas e componentes.
====


== Avançado

=== Mais de uma série temporal em um gráfico

[NOTE]
====
Há apenas uma unidade para uma ligação métrica (ela é plotada no eixo y do gráfico). Como resultado, você só deve combinar consultas que produzam séries temporais com a mesma unidade em uma ligação métrica. Às vezes, pode ser possível converter a unidade. Por exemplo, o uso da CPU pode ser relatado em mili-cores ou núcleos; os mili-cores podem ser convertidos em núcleos multiplicando-se por 1.000, como em `(<original-query>) * 1000`.
====


Há duas maneiras de obter mais de uma série temporal em uma única ligação de métrica e, portanto, em um único gráfico:

. Escreva uma consulta PromQL que retorne várias séries temporais para um único componente
. Adicionar mais consultas PromQL à vinculação de métricas

Para a primeira opção, um exemplo é dado na xref:/setup/custom-integrations/metric-bindings/index.adoc#_using_metric_labels_in_aliases[próxima seção]. A segunda opção pode ser útil para comparar métricas relacionadas. Alguns casos de uso típicos:

* Comparação do total de réplicas com o desejado e o disponível
* Uso de recursos: limites, solicitações e uso em um único gráfico

Para adicionar mais consultas a uma vinculação de métrica, basta repetir xref:/setup/custom-integrations/metric-bindings/index.adoc#_steps[as etapas] 3 e 4 e adicionar a consulta como uma entrada extra na lista de consultas. Para as contagens de réplicas de implementação, há várias métricas relacionadas que podem ser incluídas no mesmo gráfico:

----
- _type: MetricBinding
  chartType: line
  enabled: true
  tags: {}
  unit: short
  name: Replica counts
  priority: MEDIUM
  identifier: urn:stackpack:my-stackpack:metric-binding:my-deployment-replica-counts
  queries:
    - expression: max_over_time(kubernetes_state_deployment_replicas{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", deployment="${name}"}[${__interval}])
      alias: Total replicas
    - expression: max_over_time(kubernetes_state_deployment_replicas_available{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}",  deployment="${name}"}[${__interval}])
      alias: Available - ${cluster_name} - ${namespace} - ${deployment}
    - expression: max_over_time(kubernetes_state_deployment_replicas_unavailable{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}",  deployment="${name}"}[${__interval}])
      alias: Unavailable - ${cluster_name} - ${namespace} - ${deployment}
    - expression: min_over_time(kubernetes_state_deployment_replicas_desired{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}",  deployment="${name}"}[${__interval}])
      alias: Desired - ${cluster_name} - ${namespace} - ${deployment}
  scope: type = "deployment" and label = "stackpack:kubernetes"
----

image::k8s/k8s-replica-counts-multiple-timeseries.png[Vinculação de métricas com várias métricas]

=== Uso de rótulos métricos em aliases

Quando uma única consulta retorna várias séries temporais por componente, isso será mostrado como várias linhas no gráfico. Mas, na legenda, todos eles usarão o mesmo alias. Para poder ver a diferença entre as diferentes séries temporais, o alias pode incluir referências aos rótulos de métricas usando a sintaxe `${label}`. Por exemplo, aqui está uma associação de métrica para a métrica "Reinicialização de contêineres" em um pod; observe que um pod pode ter vários contêineres:

----
type: MetricBinding
chartType: line
enabled: true
id: -1
identifier: urn:stackpack:my-stackpack:metric-binding:my-pod-restart-count
name: Container restarts
priority: MEDIUM
queries:
- alias: Restarts - ${container}
  expression: max by (cluster_name, namespace, pod_name, container) (kubernetes_state_container_restarts{cluster_name="${tags.cluster-name}", namespace="${tags.namespace}", pod_name="${name}"})
scope: (label = "stackpack:kubernetes" and type = "pod")
unit: short
----

Observe que o endereço `alias` faz referência ao rótulo `container` da métrica. Certifique-se de que o rótulo esteja presente no resultado da consulta. Quando o rótulo estiver ausente, o site `${container}` será renderizado como texto literal para ajudar na solução de problemas.

=== Layouts

Cada componente pode ser associado a várias tecnologias ou protocolos, como k8s, redes, ambientes de tempo de execução (por exemplo, JVM), protocolos (HTTP, AMQP) etc.
Consequentemente, uma infinidade de métricas diferentes pode ser exibida para cada componente. Para facilitar a leitura, o site {stackstate-product-name} pode organizar esses gráficos em guias e seções.
Para exibir um gráfico (`MetricBinding`) em uma guia ou seção específica, é necessário configurar a propriedade de layout.
