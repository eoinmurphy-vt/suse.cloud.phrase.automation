== Visão geral

Esta página descreve os esquemas para definir um `OtelComponentMapping` ou `OtelRelationMapping`, juntamente com explicações detalhadas sobre construções, sintaxe de expressão e semântica.

== Esquemas para mapeamentos de componentes e relações da OTel

=== OtelComponentMapping

Mapeamento de cada componente:

* Seleciona a telemetria usando as condições.
* Extrai valores usando expressões.
* Produz um único componente lógico identificado por um identificador estável.

Vários registros de telemetria podem ser resolvidos com o mesmo identificador de componente; nesse caso, o componente é mesclado e atualizado.

[,yaml]
----
_type: "OtelComponentMapping"
name: string
input:
  signal: ["TRACES" | "METRICS"]
  resource:
    condition: <cel-boolean>        # default: true
    action: CONTINUE                # default
    scope:
      condition: <cel-boolean>      # default: true
      action: CONTINUE              # default
      span:                         # TRACES only
        condition: <cel-boolean>    # default: true
        action: CONTINUE            # default
      metric:                       # METRICS only
        condition: <cel-boolean>    # default: true
        action: CONTINUE            # default
        datapoint:                  # METRICS only
          condition: <cel-boolean>  # default: true
          action: CONTINUE          # default
vars:                               # Optional
  - name: string
    value: <cel-expression>
output:
  identifier: <cel-string>
  name: <cel-string>
  typeName: <cel-string>
  typeIdentifier: <cel-string>      # Optional
  domainName: <cel-string>
  domainIdentifier: <cel-string>    # Optional
  layerName: <cel-string>
  layerIdentifier: <cel-string>     # Optional
  required:                         # Optional (required means that if any expression fails, the mapping fails)
    version: <cel-string>           # Optional
    additionalIdentifiers:          # Optional
      - <cel-string>
    tags:                           # Optional
      - source: <cel-string>
        target: string
      - source: <cel-string>
        pattern: regex
        target: string
  optional:                         # Optional (besides being optional on the schema, optional means that if any expression under `optional` fails, the mapping will continue, but without the failed expression/field)
    version: <cel-string>           # Optional
    additionalIdentifiers:          # Optional
      - <cel-string>
    tags:                           # Optional
      - source: <cel-string>
        target: string
      - source: <cel-map>
        pattern: regex
        target: string
expireAfter: duration-ms
----

=== OtelRelationMapping

Cada mapeamento de relação:

* Resolve um `sourceId` e `targetId`.
* Atribui um tipo de relação.
* Produz uma borda direcionada entre componentes existentes ou futuros.

As relações são materializadas quando existem componentes de origem e destino.

[,yaml]
----
_type: "OtelRelationMapping"
name: string
input:
  signal: ["TRACES" | "METRICS"]
  resource:
    condition: <cel-boolean>        # default: true
    action: CONTINUE                # default
    scope:
      condition: <cel-boolean>      # default: true
      action: CONTINUE              # default
      span:                         # TRACES only
        condition: <cel-boolean>    # default: true
        action: CONTINUE            # default
      metric:                       # METRICS only
        condition: <cel-boolean>    # default: true
        action: CONTINUE            # default
        datapoint:                  # METRICS only
          condition: <cel-boolean>  # default: true
          action: CONTINUE          # default
vars:                               # Optional
  - name: string
    value: <cel-expression>
output:
  sourceId: <cel-string>
  targetId: <cel-string>
  typeName: <cel-string> 
  typeIdentifier: <cel-string>      # Optional
expireAfter: duration-ms
----

== Identidade, fusão e ciclo de vida de componentes e relações

=== Identidade do componente

O campo `output.identifier` define a identidade principal de um componente e deve ser "globalmente" exclusivo em toda a topologia. Em `output.required.additionalIdentifiers`, você pode especificar mais identificadores para o componente.  Os componentes que têm pelo menos um identificador sobreposto são considerados a mesma entidade lógica e são mesclados pela plataforma. Isso faz com que a construção do identificador seja uma opção crítica de projeto.

Os identificadores devem:

* Siga o formato do identificador {stackstate-product-name} (ou seja, `urn:...`). Consulte a documentação xref:/configure/topology/identifiers.adoc[dos identificadores] para obter mais informações.
* Ser estável ao longo do tempo.
* Refletir a cardinalidade pretendida (serviço, instância, banco de dados e similares).
* Evite dimensões não limitadas.

[NOTE]
====
Usando a lista `output.required.additionalIdentifiers`, você pode especificar identificadores adicionais para associar o componente gerado. Isso pode ser útil para correlacionar um componente com um identificador externo em outro sistema.
====

=== Identidade da relação

A relação de identidade é uma composição da forma `output.sourceId-output.targetId`, em que `sourceId` e `targetId` são identificadores de componentes (por exemplo, de `output.identifier`).

=== Atualização e expiração

Cada mapeamento define uma duração de `expireAfter`. Esse valor controla o tempo que um componente ou relação permanece na topologia sem ser atualizado por uma nova telemetria correspondente. Se não houver atualização, o componente ou a relação será removido da topologia.

Internamente, o coletor OTel atualiza continuamente os componentes conforme a correspondência da telemetria - essa janela de atualização é baseada no campo `expireAfter` por mapeamento.

== Modelo de passagem de entrada

Os mapeamentos de topologia percorrem os dados da OpenTelemetry hierarquicamente. `resource → scope → metric/span → datapoint`

Cada nível pode definir:

* Uma condição: uma expressão booleana CEL.
* Uma ação: o que fazer se a condição corresponder.

=== Campos disponíveis por sinal

O sinal selecionado determina quais estruturas de dados e atributos estão disponíveis para a avaliação da expressão.

**TRAÇOS**

Fornece acesso a:

* resource.attributes
* scope.name, scope.version, scope.attributes
* span.name, span.kind, span.statusMessage, span.statusCode, span.attributes

Exemplos:

[source]
----
resource.attributes['service.name'] == 'checkout'
# OR
span.kind == 'SPAN_KIND_SERVER' && span.attributes['http.method'] == 'POST'
----

**MÉTRICAS**

Fornece acesso a:

* resource.attributes
* scope.name, scope.version, scope.attributes
* metric.name, metric.description, metric.unit
* datapoint.attributes

Exemplos:

[source]
----
metric.name == 'traces_service_graph_request_total'
# OR
datapoint.attributes['connection_type'] == 'database'
----


=== Condições e ações

**Condições**

As condições determinam se o processamento continua em um determinado nível. Se uma condição for avaliada como falsa, o mapeamento será ignorado para esse elemento de telemetria.

Padrões:

* O padrão das condições é "true". Isso permite que a omissão de condições explícitas em cada nível atue como um fall-through.

Restrições:

* Quando há vários sinais de entrada declarados (por exemplo, `TRACES` e `METRICS`), os dados de dentro de uma expressão só podem ser acessados no nível do ancestral comum, que em todos os casos é o nível do escopo.
* Não é permitido o acesso a dados/campo de um nível inferior/criança. Por exemplo, os campos no nível do escopo não podem ser acessados no nível do recurso.
* Não é permitido o acesso a dados/campos de outro sinal de entrada. Por exemplo, os campos de nível métrico não podem ser acessados no nível de abrangência e vice-versa.

As condições podem acessar dados de um nível superior. Por exemplo, no nível de abrangência, os dados de escopo e de recursos podem ser acessados.

[,yaml]
----
input:
  signal:
    - "TRACES"
  resource:
    condition: "${'service.name' in resource.attributes}"
    # action omitted since the default is `CONTINUE`
    scope:
      action: "CREATE" # input block describes that it expects to filter until this level only
      condition: "${scope.name.contains('http') && resource.attributes['service.name'] == 'cart-svc'}" # it's allowed to access resource-level fields at scope-level
----

**Ações**

Ações suportadas:

* CONTINUE - continua a avaliação para o próximo nível.
* CREATE - cria um componente ou uma relação nesse nível.

Padrões:

* O padrão das ações é "CONTINUAR".

Isso significa que uma ação `CREATE` explícita precisa ser especificada em um nível de entrada para que o mapeamento seja aplicado.

Veja a seguir mais alguns exemplos de declarações de entrada válidas:

**TRAÇOS**

[,yaml]
----
input:
  signal:
    - "TRACES"
  resource:
    condition: "${'service.name' in resource.attributes}"
    # action omitted since the default is `CONTINUE`
    scope:
      # action and condition have been omitted since the defaults are `CONTINUE` and `true` respectively
      span:
        action: "CREATE"
        condition: "${span.attributes['http.method'] == 'POST'}"
----

**MÉTRICAS**

[,yaml]
----
input:
  signal:
    - "METRICS"
  resource:
    # action and condition have been omitted since the defaults are `CONTINUE` and `true` respectively
    scope:
      condition: "scope.name == 'traces_service_graph'"
      metric:
        # action omitted since the default is `CONTINUE`
        condition: "metric.name == 'traces_service_graph_request_total'"
        datapoint:
          action: "CREATE"
          condition: |
            'client' in datapoint.attributes &&
            'server' in datapoint.attributes
----

== Variáveis (vars)

Os mapeamentos podem definir variáveis para evitar repetições e melhorar a legibilidade.

Variáveis:

* São avaliados usando expressões CEL.
* Pode fazer referência a quaisquer campos que estejam disponíveis no nível de entrada mais profundo correspondente aos filtros de entrada. Por exemplo, se a seleção de entrada tiver a ação `CREATE` no nível do intervalo, os campos de recurso, escopo e intervalo poderão ser usados.
* Pode ser reutilizado em campos de saída.

Exemplo:

[,yaml]
----
vars:
- name: "service"
  value: "${resource.attributes['service.name']}"
----

As variáveis são resolvidas antes da avaliação das expressões de saída.

== Saída

A seção `output` define como os sinais do OpenTelemetry (traços/métricas) devem ser mapeados para componentes ou relações.

Campos de saída:

* Use expressões CEL para gerar valores dinamicamente.
* Pode fazer referência a quaisquer campos que estejam disponíveis no nível de entrada mais profundo correspondente aos filtros de entrada. Por exemplo, se a seleção de entrada tiver a ação `CREATE` no nível do intervalo, os campos de recurso, escopo e intervalo poderão ser usados.

Se o valor de um campo não precisar ser gerado dinamicamente, um literal de cadeia de caracteres poderá ser definido onde o esquema especifica uma expressão `<cel-string>` a ser declarada.

=== Necessário vs. Subseções opcionais

O esquema de mapeamento de componentes faz distinção entre dois comportamentos de tratamento de erros:

==== obrigatório (subseção opcional)

Se qualquer expressão sob required falhar, o mapeamento inteiro falhará.

==== opcional (subseção opcional)

Se alguma expressão em opcional falhar, o mapeamento continuará, mas sem o campo que falhou.

==== Mapeamentos de tags

Os mapeamentos de componentes podem definir mapeamentos de tags para enriquecer os componentes com metadados.

Há suporte para dois formulários:

**Mapeamento direto**

[,yaml]
----
- source: "value"
  target: "key"
----
Resultado: `key:value`

[,yaml]
----
- source: "${resource.attributes['service.name']}"
  target: "service.name"
----
Dado: `resource.attributes { 'service.name': 'cart-svc' }`

Resultado: `service.name:cart-svc`

A fonte para um mapeamento direto precisa ser a:

* literal de string
* expressão de string

**Extração baseada em Regex**

[,yaml]
----
- source: "${resource.attributes}"
  pattern: "telemetry.sdk\.(.*)"
  target: "telemetry.sdk.${1}"
----
Dado: `resource.attributes: { 'telemetry.sdk.language': 'go', 'telemetry.sdk.version': '1.23.1' }`

Resultado: `telemetry.sdk.language:go;telemetry.sdk.version:1.23.1`

Os mapeamentos baseados em Regex são compatíveis com vários grupos de captura, que podem ser referenciados posicionalmente na expressão de destino.

A fonte para um mapeamento baseado em regex precisa ser:

* expressão do mapa

Exemplo com vários grupos de captura:

[,yaml]
----
- source: "${resource.attributes}"
  pattern: "^(os|host|cloud|azure|gcp)\.(.*)"
  target: "${1}.${2}"
----

Em mapeamentos baseados em regex, os grupos de captura são substituídos no destino.

== Explicação da expressão CEL (`<cel-*>`):

* `<cel-string>` - precisa retornar uma string; pode ser um dos seguintes:
** literal de string (por exemplo, "hello")
** expressão de cadeia de caracteres, envolvida em `${...}` (por exemplo, "${resource.attributes['service.name']}")
** interpolação de cadeia de caracteres (por exemplo, "urn:opentelemetry:namespace/$\{resource.attributes['namespace']}:service/$\{resource.attributes['service.name']}") - observação: para a interpolação de cadeia de caracteres, a expressão inteira não é envolvida em `$\{...}`
* `cel-boolean` - precisa retornar um booleano; pode ser um dos seguintes:
** literal booleano (por exemplo, "true")
** expressão booleana (por exemplo, "'namespace' in resource.attributes") - observação: as expressões booleanas não são agrupadas em `${...}`
* `cel-map` - precisa retornar um mapa; pode ser um dos seguintes:
** literal do mapa (por exemplo, "${{'a': 1, 'b': 'two'}}")
** expressão de mapa (por exemplo, "${resource.attributes}")
* `cel-expression` - retorna "qualquer" tipo, que pode ser um dos seguintes:
** expressão de string
** expressão booleana
** expressão de mapa (por exemplo, "${resource.attributes}" - retorna o mapa de atributos)
** expressão de lista (por exemplo, "${resource.attributes['process.command_args']}" - retorna uma lista)

=== Interpolação

A interpolação de strings permite incorporar expressões em literais de strings. Embora não sejam diretamente compatíveis com a especificação CEL, as configurações de mapeamento OTel fornecem a sintaxe de interpolação do formulário `prefix-$\{expression1}/suffix-$\{expression2}`, que é reescrito internamente como uma concatenação de cadeia de caracteres usando o operador `+` (por exemplo, `"prefix-" + expression1 + "/suffix-" + expression2`).

Entender esse comportamento de reescrita é importante por dois motivos:

1. *Não há suporte para interpolação aninhada*: Como as expressões de cadeia de caracteres são demarcadas com `${...}`, você não pode aninhar expressões de interpolação. No entanto, você pode escrever a concatenação diretamente usando o operador `+`. Por exemplo:
+
[source]
----
${
  'service.instance.id' in resource.attributes ?
    resource.attributes['service.name'] + " - " + resource.attributes['service.instance.id'] :
    resource.attributes['service.name'] + " - instance"
}
----

2. *A conversão de tipos é necessária para valores que não sejam strings*: O CEL é fortemente tipado e não converte automaticamente os tipos durante a concatenação. Ao interpolar valores de tipos diferentes, você deve converter explicitamente os tipos que não são strings em strings.
+
Por exemplo, se `process.pid` for um número inteiro (por exemplo, `1`), essa expressão falhará:
+
[source]
----
${resource.attributes['service.name']}/${resource.attributes['process.pid']}
----
+
A forma correta requer primeiro a conversão do valor dinâmico em um tipo concreto e, em seguida, a conversão em uma cadeia de caracteres:
+
[source]
----
${resource.attributes['service.name']}/${string(int(resource.attributes['process.pid']))}
----
+
A fundição dupla (`string(int(...))`) é necessária porque:
+
* `resource.attributes['process.pid']` retorna um tipo dinâmico (`dyn`)
* `int()` converte o valor dinâmico em um tipo de número inteiro concreto
* `string()` converte o número inteiro em uma string para concatenação
+
Para outros tipos, use padrões semelhantes: `string(double(...))` para floats, `string(bool(...))` para booleanos.

=== Referência de linguagem CEL

O CEL é uma linguagem de expressão segura e sem efeitos colaterais, projetada para casos de uso de configuração e política. No contexto dos mapeamentos de topologia do OpenTelemetry, o CEL é usado para:

* Definir condições que decidem se um mapeamento se aplica
* Calcular variáveis a partir de atributos de telemetria
* Criar identificadores, nomes e tags dinamicamente

As expressões são avaliadas em relação a um contexto tipificado e bem definido, derivado do sinal OpenTelemetry que está sendo processado (por exemplo, recurso, escopo, intervalo, métrica ou ponto de dados).

Visite https://github.com/google/cel-spec/blob/master/doc/langdef.md[o site] para obter uma referência completa.

Um playground on-line de CEL, como o https://playcel.undistro.io/, é uma ferramenta útil para fazer verificações rápidas de validade de expressões.

== Padrões comuns e práticas recomendadas

* Proteja os mapeamentos com condições para evitar cardinalidade não intencional
* Sempre lide com atributos ausentes de forma defensiva
* Manter os identificadores estáveis e previsíveis
* Prefira variáveis para expressões complexas e para promover a legibilidade
* Use os valores expireAfter adequados à frequência do sinal

Consulte a página xref:/setup/custom-integrations/otelmappings/troubleshooting.adoc[de solução de problemas] para obter orientação sobre como solucionar problemas de mapeamentos OTel.
