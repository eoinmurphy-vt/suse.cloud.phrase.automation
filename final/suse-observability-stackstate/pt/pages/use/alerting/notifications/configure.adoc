== Criar uma nova notificação

Abra a página de notificações por meio do link na metade inferior do menu de hambúrguer na interface do usuário do SUSE Observability. Isso abre uma visão geral de todas as notificações que já estão configuradas, incluindo seu status.

Você pode verificar se a notificação desejada já existe. Caso contrário, crie uma nova com o botão "Adicionar nova notificação".

== Configurar quando notificar

image::k8s/notifications-add-new-notification.png[Adição de uma nova notificação]

Configure a notificação:

. Nome - Escolha um nome curto, mas que ainda descreva a intenção dessa notificação. É para sua própria referência na visão geral das notificações.
. Status - A notificação pode ser desativada temporariamente, caso ainda não seja necessária, seja muito barulhenta etc.
. Notificar quando - um estado de saúde crítico sempre aciona uma notificação, mas opcionalmente também podem ser incluídos estados divergentes.
. Escopo - no exemplo, serão enviados os estados de integridade de todos os monitores em pods no namespace padrão do Kubernetes. Use os filtros disponíveis na seção de <<_scope,escopo>> para alterar essa seleção.

=== Escopo

Há quatro filtros de escopo possíveis. Por padrão, uma notificação será enviada para cada estado de saúde crítico (e, opcionalmente, divergente). Os filtros são usados para limitar esse escopo. Um estado de saúde só resultará em uma notificação quando corresponder a todos os filtros.

* Monitores: Selecione um ou mais monitores específicos. As notificações serão enviadas apenas para os estados de saúde dos monitores selecionados.
* Tags do monitor: Selecione uma ou mais tags de monitor. As notificações só serão enviadas para estados de saúde de monitores que tenham pelo menos uma das tags selecionadas.
* Tipos de componentes: Selecione um ou mais tipos de componentes. As notificações serão enviadas apenas para os estados de integridade dos componentes dos tipos selecionados.
* Tags de componentes: Selecione uma ou mais tags de componentes. As notificações só serão enviadas para estados de integridade de componentes que tenham pelo menos um valor correspondente para cada chave distinta nas tags selecionadas.

Embora esses componentes não sejam compatíveis:

* um componente com tags [ `k8s-scope:prod-emea/checkout` ]
* um componente com tags [ `k8s-scope:prod-us/checkout`, `team:green` ]
* um componente com tags [ `k8s-scope:prod-emea/carts`, `team:blue` ]

== Para onde enviar as notificações?

O SUSE Observability pode enviar notificações para diferentes sistemas externos por meio de canais. Os canais compatíveis são:

* xref:/use/alerting/notifications/channels/slack.adoc[Slack] - Enviar notificações para o Slack
* xref:/use/alerting/notifications/channels/teams.adoc[Teams] - Enviar notificações para o Microsoft Teams
* xref:/use/alerting/notifications/channels/webhook.adoc[Webhook] - Envie notificações para um webhook, o ponto de extremidade do webhook pode traduzir a carga útil do SUSE Observability em qualquer API personalizada de terceiros necessária
* xref:/use/alerting/notifications/channels/opsgenie.adoc[Opsgenie] - Enviar notificações para o OpsGenie

Em geral, o SUSE Observability envia dois tipos de mensagens para notificações:

. Uma mensagem do site `open` quando um estado de saúde passa para Crítico ou Desviante. Essa mensagem pode ser repetida quando houver alterações no estado de saúde
. Uma mensagem `close` quando o estado de saúde não é mais Crítico ou Desviante ou quando, por outros motivos (o componente desapareceu, o monitor foi removido etc.), a notificação não está mais ativa.
