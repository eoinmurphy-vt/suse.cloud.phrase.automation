= Konfiguration der Affinitätswerte
:revdate: 2025-07-10
:page-revdate: {revdate}
:description: {stackstate-product-name} Affinität Konfiguration

Affinitätskonfigurationen steuern das Pod-Scheduling-Verhalten in {stackstate-product-name}. Die Affinitätswerte tragen dazu bei, die Ressourcennutzung zu optimieren und eine hohe Verfügbarkeit zu gewährleisten, indem sie steuern, wo die Pods eingeplant werden.

== Verfügbare Konfigurationsoptionen

=== Knotenpunkt-Affinität

Sie können die Knotenaffinität verwenden, um Pods für bestimmte Knoten oder Instanzgruppen zu planen, z. B. für EC2-Knoten, die in derselben Verfügbarkeitszone bereitgestellt werden.

[,yaml]
----
affinity:
  # Node Affinity settings - applied to all components when configured
  nodeAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
      nodeSelectorTerms:
      - matchExpressions:
        - key: topology.kubernetes.io/zone
          operator: In
          values:
          - us-west-2a
----

=== Pod Anti-Affinität

Mit Pod-Anti-Affinität können Sie Replikate von Datendiensten auf verschiedenen Knoten planen, um eine hohe Verfügbarkeit zu gewährleisten. Standardmäßig ist das Scheduling erforderlich (harte Anti-Affinität) und `topologyKey` ist `kubernetes.io/hostname`.

[,yaml]
----
affinity:
  podAntiAffinity:
    # Enable required pod anti-affinity (true = hard, false = soft)
    requiredDuringSchedulingIgnoredDuringExecution: true
    # Topology key for pod anti-affinity
    topologyKey: "kubernetes.io/hostname"
----

== Verhalten

=== Knotenpunkt-Affinität

* **Wenn konfiguriert**: Angewandt auf alle Komponenten

=== Pod Anti-Affinität

* **Wenn konfiguriert**: Wird nur angewendet, wenn `sizing.profile` mit `-ha` endet (Hochverfügbarkeitsprofile)
* **HA-Profile**: `150-ha`, `250-ha`, `500-ha`, `4000-ha`
* **Betroffene Komponenten**: Alle zustandsbezogenen Datendienste, einschließlich Clickhouse, Kafka, Zookeeper, VictoriaMetrics, StackGraph, Elasticsearch

== Beispielkonfigurationen

=== Grundlegende Knotenaffinität (gleiche Availability Zone) + harte PodAntiAffinität für HA-Bereitstellung

Angenommen, Sie haben `sizing.profile` auf eines der HA-Profile eingestellt. Die nachstehenden Werte konfigurieren nodeAffinity so, dass die Komponenten von {stackstate-product-name} in der Zone `us-west-2a` eingeplant werden und die Replikate der gleichen Datendienste auf verschiedenen Hosts (standardmäßig`affinity.podAntiAffinity.requiredDuringSchedulingIgnoredDuringExecution`=`true` ) eingeplant werden müssen.
[,yaml]
----
affinity:
  # Schedule all pods to nodes in the same AZ
  nodeAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
      nodeSelectorTerms:
      - matchExpressions:
        - key: topology.kubernetes.io/zone
          operator: In
          values:
          - us-west-2a
----

=== Soft PodAntiAffinity für HA-Bereitstellung
Wenn es nicht erwünscht ist, die Planung von Datendiensten auf verschiedenen Knoten zu erzwingen, können Sie podAntiAffinity im "weichen" Modus konfigurieren:
[,yaml]
----
affinity:
  podAntiAffinity:
    requiredDuringSchedulingIgnoredDuringExecution: false
----

== Verwendung

[tabs]
====
Empfohlene Methode::
+
--
[NOTE]
=====
Die Konfigurationsmethode `global.suseObservability` ist ab der Version `2.8.0` verfügbar. Für frühere Versionen verwenden Sie die Legacy-Methode.
=====

Fügen Sie die Affinitätskonfiguration direkt in Ihre `values.yaml` Datei unter `global.suseObservability.affinity` ein:

[source,yaml]
----
global:
  suseObservability:
    # Your existing configuration
    license: "YOUR-LICENSE-KEY"
    baseUrl: "https://observability.example.com"
    sizing:
      profile: "150-ha"
    adminPassword: "your-password"

    # Affinity configuration
    affinity:
      # Node affinity - schedule all pods to nodes in the same AZ
      nodeAffinity:
        requiredDuringSchedulingIgnoredDuringExecution:
          nodeSelectorTerms:
          - matchExpressions:
            - key: topology.kubernetes.io/zone
              operator: In
              values:
              - us-west-2a

      # Pod anti-affinity for HA profiles (infrastructure components)
      podAntiAffinity:
        # true = hard anti-affinity, false = soft anti-affinity
        requiredDuringSchedulingIgnoredDuringExecution: true
        topologyKey: "kubernetes.io/hostname"
----

Dann installieren oder aktualisieren Sie mit einem einzigen Befehl:

[,bash]
----
helm upgrade --install \
  --namespace suse-observability \
  --create-namespace \
  --values values.yaml \
  suse-observability \
  suse-observability/suse-observability
----

TIP: Bei HA-Profilen wird die Pod-Anti-Affinität automatisch auf alle zustandsabhängigen Datendienste angewendet. Sie müssen es nur konfigurieren, wenn Sie das Standardverhalten ändern möchten.

Weitere Einzelheiten zur Migration von der alten Methode finden Sie in der xref:/setup/upgrade-stackstate/migrate-to-global-mode.adoc[Migrationsanleitung].
--

[WARNING]

==== Schritt 1: Erstellen Sie Ihre Affinitätswerte-Datei

Erstellen Sie eine separate Wertedatei mit Ihrer gewünschten Affinitätskonfiguration. Speichern Sie zum Beispiel folgendes als `suse-observability-values-values.yaml`:

[,yaml]
----
affinity:
  nodeAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
      nodeSelectorTerms:
      - matchExpressions:
        - key: topology.kubernetes.io/zone
          operator: In
          values:
          - us-west-2a
  podAntiAffinity:
    requiredDuringSchedulingIgnoredDuringExecution: true
----

==== Schritt 2: Erzeugen von Affinitätsvorlagenwerten

Führen Sie den folgenden Befehl aus, um die Vorlage für die Affinitätswerte zu erstellen:

[,text]
----
export VALUES_DIR=.
helm template \
  --set license='<your license>' \
  --set baseUrl='<suse-observability-base-url>' \
  --set sizing.profile='<sizing.profile>' \
  --values suse-observability-values-values.yaml \
  suse-observability-values \
  suse-observability/suse-observability-values --output-dir $VALUES_DIR
----

==== Schritt 3: Generierte Werte in der Helm-Installation verwenden

Nehmen Sie die generierten Affinitätswerte in Ihre Helm-Installation auf:

[,bash]
----
helm upgrade \
  --install \
  --namespace suse-observability \
  --values $VALUES_DIR/suse-observability-values/templates/baseConfig_values.yaml \
  --values $VALUES_DIR/suse-observability-values/templates/sizing_values.yaml \
  --values $VALUES_DIR/suse-observability-values/templates/affinity_values.yaml \
  suse-observability \
  suse-observability/suse-observability
----
