== 概要

このドキュメントでは、StackPack の一部としてパッケージ化されたコンポーネントと関係のマッピングを使用して、OpenTelemetry (OTel) のトレース・データからトポロジーを構築するための入門ガイドを提供します。

このガイドでは、すでに存在するテレメトリデータを使用して、製品内ですぐに視覚化できるトポロジーに焦点を当てる。
生成されるトポロジーの例は、OpenTelemetry のリソース属性に由来する、サービスインスタンスがどのようにプロセスを実行するかをモデル化したものです。

== 前提条件

* あなたはすでにOpenTelemetryのトレースを収集している。
* StackPackを作成または拡張しています。
* YAMLと基本的なOTelの概念（リソース、スパン）に慣れている。

このガイドが注目するのは

* トレースベースのトポロジー（`TRACES` 信号のみ）
* サービスインスタンス（論理サービスではない）
* ランタイム・プロセスの可視化

[NOTE]
====
このガイドでは、コンポーネントとリレーションのマッピングは、StackPackの一部としてパッケージ化され、テストされ、デプロイされるYAML設定として表現されます。
====

== トポロジーの設定

その目的は、ユーザーによる追加設定なしに即座に利用可能なランタイム実行トポロジーを可視化することである。

具体的には、モデル化したい：

[source]
----
service instance (component) -> executes (relation) -> process (component)
----

どこでだ：

* `service instance` は、インスツルメンテッドサービスの実行中のインスタンスを表す。
* `process` は、そのサービスを実行するオペレーティング・システム・プロセスを表す。
* `executes` リレーションは、サービスインスタンスが特定のプロセスによってバックアップされ、その中で実行されていることを示す。

このトポロジーはすべて、OpenTelemetryのトレースデータから自動的に導き出される。

== トレースデータ

サービス ID に加えて、トレースリソースには以下のようなプロセスレベルの属性が含まれる：

* `process.pid`
* `process.executable.name`
* `process.executable.path`
* `process.command_args`
* `process.runtime.name`
* `process.runtime.version`

これらの属性は、スパンの相関関係やヒューリスティックなしに、プロセスレベルのトポロジーをモデル化することを可能にする。

== トポロジーへの痕跡：メンタル・モデル

コンフィギュレーションを書く前に、トポロジーマッピングが概念的にどのように機能するかを理解することが重要である。

=== コンポーネント

コンポーネントマッピングは、トポロジーノードがテレメトリデータからどのように作成されるかを記述する。

各コンポーネントのマッピング：

* 条件を使用してテレメトリーを選択します。
* 式を使用して値を抽出します。
* 安定識別子で識別される単一の論理コンポーネントを生成する。

このガイドでは

* サービスインスタンスはOpenTelemetry StackPackによって提供される。
* プロセスはOpenTelemetryリソース属性から派生する。

=== 関係

関係マッピングは、2つのコンポーネント間の接続がどのように作成されるかを記述する。

それぞれの関係マッピング：

* `sourceId` と`targetId` を解決する。
* 関係タイプを割り当てる。
* 有向エッジを生成する。

リレーションは、ソースとターゲットの両方のコンポーネントが存在した時点で作成される。

== トレースからサービスコンポーネントを作成する

=== サービス」とは何かを定義する

マッピングを書く前に、デザインを決める必要がある。

このガイドでは、サービスを次のように定義する：

* `service.namespace` +`service.name` 。
* デプロイメントを超えた安定性。
* サービスインスタンスに依存しない。

これにより、トポロジーは読みやすく、カーディナリティは低く保たれる。

== サービス・インスタンス・コンポーネント

サービス・インスタンス・コンポーネントはすでに定義されており、OpenTelemetry StackPackによって提供されています。

各サービスインスタンス：

* `service.name` 、`service.namespace` 、`service.instance.id` に由来する。
* サービスの具体的な実行インスタンスを表します。
* 信号（トレースとメトリクス）間で安定している。

このマッピングはすでに存在するため、本ガイドでは再定義しない。
その代わりに、私たちはその上に構築する。

== トレースからプロセスコンポーネントを作成する

=== プロセス」とは何かを定義する

このガイドでは、プロセスを次のように定義する：

* `host.name` 、`process.pid` 、実行可能なメタデータによって識別される。
* 単一のランタイム環境にスコープされる。
* OpenTelemetry リソース属性からのみ派生。

これにより、トポロジーを低基数性に保ちつつ、実行時に有用な詳細情報を公開することができる。

=== プロセス・コンポーネント・マッピング

以下のコンポーネントマッピングは、1つの観測プロセスにつき1つのトポロジーコンポーネントを作成する。

[,yaml]
----
_type: "OtelComponentMapping"
name: "OTel Process"
description: "Represents an operating system process derived from OpenTelemetry"
identifier: "urn:stackpack:<stackpack-name>:shared:otel-component-mapping:process"
input:
  signal:
    - "TRACES"
  resource:
    condition: |
      'host.name' in resource.attributes &&
      'process.pid' in resource.attributes
    action: "CREATE"
vars:
  - name: "pid"
    value: "${string(int(resource.attributes['process.pid']))}"
  - name: "hostname"
    value: "${resource.attributes['host.name']}"
  - name: "executableName"
    value: >-
      ${
        'process.executable.name' in resource.attributes ?
         resource.attributes['process.executable.name'] :
          'process.command' in resource.attributes ?
           resource.attributes['process.command'] :
            'process.command_args' in resource.attributes ?
             resource.attributes['process.command_args'] :
                resource.attributes['process.executable.path']
      }
output:
  identifier: "urn:opentelemetry:process/${vars.hostname}:${vars.pid}"
  name: "${vars.hostname}/${vars.executableName}:${vars.pid}"
  typeName: "process"
  typeIdentifier: "urn:stackpack:open-telemetry:shared:component-type:process"
  layerName: "Applications"
  domainName: "Open Telemetry"
  required:
    tags:
      - source: "process-component"
        target: "custom"
      - source: "${resource.attributes}"
        pattern: "process.(.*)"
        target: "process.${1}"
expireAfter: 900000
----

==== このマッピングの仕組み

* トレースデータ（TRACES信号）のみを処理する。
* プロセス・コンポーネントは、`host.name` 、`process.pid` 。
* この識別子は、プロセスの有効期間中、安定している。
* カスタムタグは、フィルタリングを助けるために追加されます（検証段階用）。

`<stackpack-name>` をあなたのStackPackの名前に置き換えてください（まだStackPackを持っていない場合は、`mystackpack` のようにお好きな名前を使ってください）。

== リレーションシップの作成

サービス・インスタンスとプロセスがコンポーネントとして存在するようになったので、それらを接続することができる。

=== サービス・インスタンスがプロセスを実行する

次のリレーションマッピングは、サービスインスタンスからプロセスへの`executes` リレーションを作成します。

このマッピングは、OpenTelemetry StackPackが提供する既存の "ホストがサービスインスタンスを実行する "関係から適応されている。

[,yaml]
----
_type: "OtelRelationMapping"
name: "Executes Relation (Service Instance -> Process)"
description: "Service instance executes a process"
identifier: "urn:stackpack:<stackpack-name>:shared:otel-relation-mapping:executes-service-instance"
input:
  signal:
    - "TRACES"
  resource:
    condition: |
      'service.name' in resource.attributes &&
      'host.name' in resource.attributes &&
      'process.pid' in resource.attributes
    action: "CREATE"
vars:
  - name: "namespace"
    value: "${'service.namespace' in resource.attributes && resource.attributes['service.namespace'] != '' ? resource.attributes['service.namespace'] : 'default'}"
  - name: "service"
    value: "${resource.attributes['service.name']}"
  - name: "instanceId"
    value: >-
      ${
        'service.instance.id' in resource.attributes && resource.attributes['service.instance.id'] != '' ?
        resource.attributes['service.instance.id'] :
        resource.attributes['service.name']
      }
  - name: "hostname"
    value: "${resource.attributes['host.name']}"
  - name: "pid"
    value: "${string(int(resource.attributes['process.pid']))}"
output:
  sourceId: "urn:opentelemetry:namespace/${vars.namespace}:service/${vars.service}:serviceInstance/${vars.instanceId}"
  targetId: "urn:opentelemetry:process/${vars.hostname}:${vars.pid}"
  typeName: "executes"
expireAfter: 900000
----

==== このマッピングの仕組み

* トレースデータ（`TRACES` 信号）のみを処理する。
* リレーションは、サービスインスタンスとプロセスデータの両方が存在する場合に作成されます。
* スパン相関は必要ない。
* `targetId` 式は、プロセス・コンポーネント・マッピングの`output.identifier` 式と同じである必要がある。

関係識別子は`sourceId` と`targetId` から(自動的に)構築され、以下のような形式となる： `sourceId-relationId`

`<stackpack-name>` をあなたのStackPackの名前に置き換えてください（まだStackPackを持っていない場合は、`mystackpack` のようにお好きな名前を使ってください）。

== OTelマッピングの検証

本番環境にデプロイする前にマッピングの正しさを検証するには、2つのオプションがある。

1. `sts stackpack test-deploy` コマンドを使用して、実行中の{stackstate-product-name} インスタンスにマッピングを含む StackPack をパッケージング、アップロード、インストール/アップグレードします。
2. `sts otel-component-mapping apply` 、`sts otel-relation-mapping apply` コマンドを使用して、実行中の{stackstate-product-name} インスタンスに対して個別にマッピングを作成/更新する。

=== StackPackでマッピングをテストする

両方のマッピングがStackPack内にあると仮定すれば、一緒にテストすることができる。詳細については、xref:/setup/cli/cli-sts.adoc[StackPack CLI]ドキュメントを参照してください。

`sts stackpack test-deploy --yes` ：

* StackPackのパッケージ化、アップロード、インストール/アップグレード
* StackPackに存在する宣言的コンポーネントと関係のマッピングを検証する（例えば、式の正しさ、提供されたフィルタリングに基づく入力信号データの正しい参照）。

[NOTE]
====
`sts stackpack test` コマンドは、マッピングを通してサンプルのトレースデータをフィードしない。
マッピングが正しいトポロジーを生成していることを確認するには、Open TelemetryのトレースデータがSUSE Observabilityに送信されていることを確認します。
`sts stackpack test` の使用方法の詳細については、xref:/setup/custom-integrations/develop.adoc[カスタム統合の開発 (StackPack)]を参照してください。
====

=== マッピングを個別にテストする

上記のコンポーネントとリレーションのマッピングがYAMLファイルで定義されているとして、個別に適用することができます。

`<stackpack-name>` をあなたのStackPackの名前に置き換えてください（まだStackPackを持っていない場合は、`mystackpack` のようにお好きな名前を使ってください）。

[,sh]
----
$ sts otel-component-mapping apply -f process-component-mapping.yaml
✅ OTel component mapping upserted successfully! Identifier: urn:stackpack:mystackpack:shared:otel-component-mapping:process, Name: OTel Process

$ sts otel-relation-mapping apply -f process-relation-mapping.yaml
✅ OTel Relation Mapping upserted successfully! Identifier: urn:stackpack:mystackpack:shared:otel-relation-mapping:executes-service-instance, Name: Executes Relation (Service Instance -> Process)
----

== トポロジーの結果

これらのマッピングが適用されると、結果として得られるトポロジーは、完全にトレースデータから得られるサービス-プロセスのトポロジーグラフを形成する。

例えば、OTelデモ・アプリの`checkoutservice` 、視覚的にトポロジーは次のように表示される：

----
checkoutservice (instance) ─> executes ─> checkoutservice (process)
----

このトポロジーは、新しいトレースが到着すると継続的に更新され、トラフィックが停止すると自動的に失効する。

=== 出来上がったトポロジーを {stackstate-product-name}

{stackstate-product-name} UIを使用して、マッピングが期待されるトポロジーに具体化されていることを視覚的に確認する。

1. `baseUrl` Helmの設定値で{stackstate-product-name} UIを開く。
2. 左側のサイドバーで `Open Telemetry > Services Instances`
3. サービス・インスタンスのリストで`checkoutservice` を見つけ、サービス・インスタンス名をクリックすると、コンポーネントの概要/ハイライト・ページが開きます。
4. 上部のサブナビゲーションバーで `Topology`
5. `Outgoing` レイヤーには、`od-checkoutservice-<hostId>/checkoutservice:1 (process)` ノードが表示される。
6. プロセス・ノードを選択し `Explore component`
7. `checkoutservice -> executes -> checkoutservice (process)` 、より小さなトポロジーが表示される。

トポロジーが期待通りに実現しない場合のデバッグのヒントについては、xref:/setup/custom-integrations/otelmappings/troubleshooting.adoc[トラブルシューティング・]ガイドを参照のこと。

== 後片付け

このガイドで作成したマッピングを削除するには、以下のコマンドを使用する：

`<stackpack-name>` 、StackPackの名前に置き換えてください。

[,sh]
----
$ sts otel-component-mapping delete --identifier urn:stackpack:<stackpack-name>:shared:otel-component-mapping:process
✅ OTel Component Mapping deleted: urn:stackpack:mystackpack:shared:otel-component-mapping:process

$ sts otel-relation-mapping delete --identifier urn:stackpack:<stackpack-name>:shared:otel-relation-mapping:executes-service-instance
✅ OTel Relation Mapping deleted: urn:stackpack:mystackpack:shared:otel-relation-mapping:executes-service-instance
----


== 概要

このガイドでは

* 既存のサービス・インスタンス・コンポーネントをベースに構築。
* OpenTelemetryリソース属性から派生したプロセスコンポーネント。

== 次のステップ

ここからできること

* ホストやコンテナにプロセスをアタッチする。
* データベースやメッセージング関係を導入する。
* ランタイム固有のレイヤーを導入（JVM、Go、Node.js）。
* メトリックスベースのサービスグラフをご覧ください。
* トポロジーレイヤーを追加してStackPackを拡張する。
* 詳細なコンポーネントとリレーションマッピングのxref:/setup/custom-integrations/otelmappings/schemas-ref.adoc[リファレンスに慣れて]ください。
