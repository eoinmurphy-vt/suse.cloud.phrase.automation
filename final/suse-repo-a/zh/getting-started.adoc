== 概述

本文档提供了一份入门指南，指导如何使用作为 StackPack 一部分打包的组件和关系映射，从 OpenTelemetry (OTel) 跟踪数据构建拓扑结构。

本指南的重点是利用已有的遥测数据，在产品中实现拓扑结构的可视化。
将生成的拓扑示例模拟了服务实例如何执行由 OpenTelemetry 资源属性导出的流程。

== 先决条件

* 您已经收集了 OpenTelemetry 跟踪信息。
* 您正在创建或扩展一个 StackPack。
* 熟悉 YAML 和 OTel 基本概念（资源、跨度）。

指南的重点是

* 基于轨迹的拓扑结构（仅限`TRACES` 信号）
* 服务实例（非逻辑服务）
* 运行时进程可见性

[NOTE]
====
在本指南中，组件和关系映射以 YAML 配置的形式表示，并作为 StackPack 的一部分进行打包、测试和部署。
====

== 配置拓扑结构

目标是可视化运行时执行拓扑结构，无需用户额外配置即可立即使用。

具体来说，我们希望模拟

[source]
----
service instance (component) -> executes (relation) -> process (component)
----

在哪里？

* `service instance` 代表仪器服务的一个运行实例。
* `process` 代表执行该服务的操作系统进程。
* `executes` 关系表示服务实例由特定进程支持并在其中运行。

所有这些拓扑结构都是从 OpenTelemetry 跟踪数据中自动得出的。

== 跟踪数据

除服务标识外，跟踪资源还包括流程级属性，如

* `process.pid`
* `process.executable.name`
* `process.executable.path`
* `process.command_args`
* `process.runtime.name`
* `process.runtime.version`

这些属性使我们能够在不跨越相关性或启发式的情况下建立流程级拓扑模型。

== 拓扑痕迹：心智模式

在编写任何配置之前，必须从概念上理解拓扑映射是如何工作的。

=== 组件

组件映射描述了如何根据遥测数据创建拓扑节点。

每个组件映射：

* 选择遥测使用条件。
* 使用表达式提取数值。
* 生成由稳定标识符标识的单个逻辑组件。

在本指南中

* 服务实例由 OpenTelemetry StackPack 提供。
* 进程源自 OpenTelemetry 资源属性。

=== 关系

关系映射描述了如何创建两个组件之间的连接。

每个关系映射：

* 解决`sourceId` 和`targetId` 。
* 指定关系类型。
* 产生一条有向边缘。

一旦源组件和目标组件都存在，就会创建关系。

== 从轨迹创建服务组件

=== 定义 "服务 "的含义

在编写映射之前，我们需要做出一个设计决定。

在本指南中，服务被定义为：

* 通过`service.namespace` +`service.name` 确定。
* 跨部署稳定。
* 独立于服务实例。

这样可以保持拓扑结构的可读性和低卡性。

== 服务实例组件

OpenTelemetry StackPack 已经定义并提供了服务实例组件。

每个服务实例：

* 源于`service.name`,`service.namespace`, 和`service.instance.id` 。
* 代表服务的具体运行实例。
* 跨信号（轨迹和度量）稳定。

由于该映射已经存在，本指南不再重新定义。
相反，我们在此基础上进行建设。

== 从轨迹创建流程组件

=== 定义 "流程 "的含义

在本指南中，流程被定义为：

* 通过`host.name` 、`process.pid` 和可执行元数据识别。
* 适用于单一运行环境。
* 完全来自 OpenTelemetry 资源属性。

这样既能保持拓扑结构的低卡方性，又能暴露有用的运行时细节。

=== 流程组件映射

以下组件映射为每个观察进程创建一个拓扑组件。

[,yaml]
----
_type: "OtelComponentMapping"
name: "OTel Process"
description: "Represents an operating system process derived from OpenTelemetry"
identifier: "urn:stackpack:<stackpack-name>:shared:otel-component-mapping:process"
input:
  signal:
    - "TRACES"
  resource:
    condition: |
      'host.name' in resource.attributes &&
      'process.pid' in resource.attributes
    action: "CREATE"
vars:
  - name: "pid"
    value: "${string(int(resource.attributes['process.pid']))}"
  - name: "hostname"
    value: "${resource.attributes['host.name']}"
  - name: "executableName"
    value: >-
      ${
        'process.executable.name' in resource.attributes ?
         resource.attributes['process.executable.name'] :
          'process.command' in resource.attributes ?
           resource.attributes['process.command'] :
            'process.command_args' in resource.attributes ?
             resource.attributes['process.command_args'] :
                resource.attributes['process.executable.path']
      }
output:
  identifier: "urn:opentelemetry:process/${vars.hostname}:${vars.pid}"
  name: "${vars.hostname}/${vars.executableName}:${vars.pid}"
  typeName: "process"
  typeIdentifier: "urn:stackpack:open-telemetry:shared:component-type:process"
  layerName: "Applications"
  domainName: "Open Telemetry"
  required:
    tags:
      - source: "process-component"
        target: "custom"
      - source: "${resource.attributes}"
        pattern: "process.(.*)"
        target: "process.${1}"
expireAfter: 900000
----

==== 如何进行映射

* 我们只处理跟踪数据（TRACES 信号）。
* 只要`host.name` 和`process.pid` 出现，就会创建一个流程组件。
* 该标识符在整个流程中都是稳定的。
* 添加自定义标签以帮助过滤（用于验证阶段）。

用你的 StackPack 名称代替`<stackpack-name>` （如果你还没有 StackPack，可以使用任何你喜欢的名称，如`mystackpack` ）。

== 创建执行关系

既然服务实例和流程都作为组件存在，我们就可以将它们连接起来。

=== 服务实例执行进程

下面的关系映射创建了一个从服务实例到流程的`executes` 关系。

这种映射关系改编自 OpenTelemetry StackPack 提供的现有 "主机执行服务实例 "关系。

[,yaml]
----
_type: "OtelRelationMapping"
name: "Executes Relation (Service Instance -> Process)"
description: "Service instance executes a process"
identifier: "urn:stackpack:<stackpack-name>:shared:otel-relation-mapping:executes-service-instance"
input:
  signal:
    - "TRACES"
  resource:
    condition: |
      'service.name' in resource.attributes &&
      'host.name' in resource.attributes &&
      'process.pid' in resource.attributes
    action: "CREATE"
vars:
  - name: "namespace"
    value: "${'service.namespace' in resource.attributes && resource.attributes['service.namespace'] != '' ? resource.attributes['service.namespace'] : 'default'}"
  - name: "service"
    value: "${resource.attributes['service.name']}"
  - name: "instanceId"
    value: >-
      ${
        'service.instance.id' in resource.attributes && resource.attributes['service.instance.id'] != '' ?
        resource.attributes['service.instance.id'] :
        resource.attributes['service.name']
      }
  - name: "hostname"
    value: "${resource.attributes['host.name']}"
  - name: "pid"
    value: "${string(int(resource.attributes['process.pid']))}"
output:
  sourceId: "urn:opentelemetry:namespace/${vars.namespace}:service/${vars.service}:serviceInstance/${vars.instanceId}"
  targetId: "urn:opentelemetry:process/${vars.hostname}:${vars.pid}"
  typeName: "executes"
expireAfter: 900000
----

==== 如何进行映射

* 我们只处理跟踪数据（`TRACES` 信号）。
* 只要服务实例和流程数据都存在，就会创建关系。
* 无需跨度相关性。
* `targetId` 表达式必须与流程组件映射的`output.identifier` 表达式相同。

关系标识符由`sourceId` 和`targetId` 自动生成，其形式为 `sourceId-relationId`

用你的 StackPack 名称代替`<stackpack-name>` （如果你还没有 StackPack，可以使用任何你喜欢的名称，如`mystackpack` ）。

== 验证 OTel 映射

在将映射部署到生产环境之前，有两种方法可以验证映射的正确性。

1. 使用`sts stackpack test-deploy` 命令将包含映射的 StackPack 打包、上传和安装/升级到运行中的{stackstate-product-name} 实例。
2. 使用`sts otel-component-mapping apply` 和`sts otel-relation-mapping apply` 命令分别创建/更新运行中的{stackstate-product-name} 实例的映射。

=== 在堆栈包中一起测试映射

假设这两个映射都在 StackPack 中，那么它们就可以一起测试。有关详细信息，请参阅xref:/setup/cli/cli-sts.adoc[StackPack CLI]文档。

使用`sts stackpack test-deploy --yes` 命令，您可以

* 打包、上传和安装/升级 StackPack
* 验证堆栈包中的声明式组件和关系映射（例如，表达式的正确性、基于所提供过滤的输入信号数据的正确引用）

[NOTE]
====
`sts stackpack test` 命令不会通过映射提供示例跟踪数据。
要验证映射是否产生了正确的拓扑，请确保将 Open Telemetry 跟踪数据发送到 SUSE Observability。
有关如何使用`sts stackpack test` 的详细信息，请参阅xref:/setup/custom-integrations/develop.adoc[开发自定义集成（StackPack）]。
====

=== 单独测试映射

假设上述组件和关系映射是在 YAML 文件中定义的，那么它们可以单独应用。

用你的 StackPack 名称代替`<stackpack-name>` （如果你还没有 StackPack，可以使用任何你喜欢的名称，如`mystackpack` ）。

[,sh]
----
$ sts otel-component-mapping apply -f process-component-mapping.yaml
✅ OTel component mapping upserted successfully! Identifier: urn:stackpack:mystackpack:shared:otel-component-mapping:process, Name: OTel Process

$ sts otel-relation-mapping apply -f process-relation-mapping.yaml
✅ OTel Relation Mapping upserted successfully! Identifier: urn:stackpack:mystackpack:shared:otel-relation-mapping:executes-service-instance, Name: Executes Relation (Service Instance -> Process)
----

== 拓扑结果

应用这些映射后，拓扑结构会形成一个完全由跟踪数据生成的服务流程拓扑图。

例如，使用 OTel 演示应用程序中的`checkoutservice` ，拓扑结构应如下所示：

----
checkoutservice (instance) ─> executes ─> checkoutservice (process)
----

该拓扑会随着新轨迹的到来不断更新，并在流量停止时自动失效。

=== 在 {stackstate-product-name}

使用{stackstate-product-name} UI 直观确认映射是否实现了预期的拓扑结构。

1. 在配置的`baseUrl` Helm 值上打开{stackstate-product-name} UI
2. 在左侧边栏中，点击 `Open Telemetry > Services Instances`
3. 在服务实例列表中找到`checkoutservice` ，点击服务实例名称打开组件概述/亮点页面
4. 在顶部的子导航栏中，选择 `Topology`
5. 在`Outgoing` 层，应能看到`od-checkoutservice-<hostId>/checkoutservice:1 (process)` 节点
6. 选择进程节点，然后单击 `Explore component`
7. 应能看到更小的拓扑图，可视化`checkoutservice -> executes -> checkoutservice (process)` 

如果拓扑未按预期实现，请参阅xref:/setup/custom-integrations/otelmappings/troubleshooting.adoc[故障排除]指南，了解调试技巧。

== 清理

要删除本指南中创建的映射，请使用以下命令：

用 StackPack 的名称代替`<stackpack-name>` 。

[,sh]
----
$ sts otel-component-mapping delete --identifier urn:stackpack:<stackpack-name>:shared:otel-component-mapping:process
✅ OTel Component Mapping deleted: urn:stackpack:mystackpack:shared:otel-component-mapping:process

$ sts otel-relation-mapping delete --identifier urn:stackpack:<stackpack-name>:shared:otel-relation-mapping:executes-service-instance
✅ OTel Relation Mapping deleted: urn:stackpack:mystackpack:shared:otel-relation-mapping:executes-service-instance
----
