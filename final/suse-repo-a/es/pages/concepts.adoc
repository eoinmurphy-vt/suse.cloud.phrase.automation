== Asignación de componentes

En `OtelComponentMapping` se describe cómo se crea (o actualiza) un componente de topología a partir de los datos de OpenTelemetry.

Ejemplos típicos de componentes creados a partir de OpenTelemetry incluyen (estos son una parte de la OpenTelemetry StackPack por defecto):

* Servicios (derivados de `service.name` / `service.namespace`)
* Instancias de servicio (derivadas de `service.instance.id`)
* Bases de datos
* Hosts y recursos Kubernetes (nodos, espacios de nombres, pods)
* Funciones y tareas en entornos sin servidor o por lotes

== Asignaciones de relaciones

Un `OtelRelationMapping` describe cómo se crea una relación entre dos componentes a partir de los datos de OpenTelemetry.

Ejemplos comunes de relaciones incluyen (estos son una parte de la OpenTelemetry StackPack por defecto):

* Llamadas de servicio síncronas (cliente -> servidor)
* Flujos de mensajería asíncrona (productor -> consumidor, consumidor -> par, productor -> par)
* Conexiones servicio-base de datos
* Instancia de servicio proporcionada por el servicio
* Relaciones de ejecución (el servicio o la función ejecuta la tarea, el host o la función)

== Expresiones y lenguaje común de expresiones (CEL)

Los mapas topológicos se basan en el https://cel.dev/[lenguaje de expresión común (CEL) de] para seleccionar telemetría, extraer valores y construir campos de componentes y relaciones.

No es necesario tener conocimientos previos de CEL para empezar: la mayoría de las expresiones son simples comprobaciones booleanas, búsquedas de atributos o construcciones de cadenas. Las expresiones más avanzadas pueden introducirse gradualmente según sea necesario.

Para obtener más información sobre cómo aprovechar CEL en las asignaciones de topología, consulte la página de xref:/setup/custom-integrations/otelmappings/schemas-ref.adoc[esquemas].
