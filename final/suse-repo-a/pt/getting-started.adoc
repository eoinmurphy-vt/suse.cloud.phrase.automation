= Primeiros passos
:revdate: 2025-12-23
:page-revdate: {revdate}
:description: Primeiros passos com a derivação de topologia da OpenTelemetry

== Visão geral

Este documento fornece um guia de introdução para a criação de topologia a partir de dados de rastreamento do OpenTelemetry (OTel) usando mapeamentos de componentes e relações empacotados como parte de um StackPack.

Este guia se concentra na topologia que pode ser visualizada imediatamente no produto usando dados de telemetria que já estão presentes.
A topologia de exemplo que será gerada modela como uma instância de serviço executa um processo, derivado dos atributos de recursos da OpenTelemetry.

== Pré-requisitos

* Você já coleta os rastros do OpenTelemetry.
* Você está criando ou ampliando um StackPack.
* Você está familiarizado com YAML e com os conceitos básicos da OTel (recursos, spans).

O guia se concentrará em:

* Topologia baseada em traços (somente sinal`TRACES` )
* Instâncias de serviço (não serviços lógicos)
* Visibilidade do processo em tempo de execução

[NOTE]
====
Neste guia, os mapeamentos de componentes e relações são expressos como uma configuração YAML que é empacotada, testada e implantada como parte de um StackPack.
====

== Configuração da topologia

O objetivo é visualizar a topologia de execução em tempo de execução que está imediatamente disponível sem configuração adicional do usuário.

Especificamente, queremos modelar:

[source]
----
service instance (component) -> executes (relation) -> process (component)
----

Onde:

* O site `service instance` representa uma instância em execução de um serviço instrumentado.
* Um `process` representa o processo do sistema operacional que executa esse serviço.
* A relação `executes` indica que a instância do serviço é apoiada e executada em um processo específico.

Toda essa topologia é derivada automaticamente dos dados de rastreamento do OpenTelemetry.

== Dados de rastreamento

Além da identidade do serviço, o recurso de rastreamento inclui atributos em nível de processo, como

* `process.pid`
* `process.executable.name`
* `process.executable.path`
* `process.command_args`
* `process.runtime.name`
* `process.runtime.version`

Esses atributos nos permitem modelar a topologia em nível de processo sem correlação de extensão ou heurística.

== Traços para topologia: o modelo mental

Antes de escrever qualquer configuração, é importante entender como os mapeamentos de topologia funcionam conceitualmente.

=== Componentes

Um mapeamento de componentes descreve como um nó de topologia é criado a partir de dados de telemetria.

Mapeamento de cada componente:

* Seleciona a telemetria usando as condições.
* Extrai valores usando expressões.
* Produz um único componente lógico identificado por um identificador estável.

Neste guia:

* As instâncias de serviço são fornecidas pelo OpenTelemetry StackPack.
* Os processos são derivados dos atributos de recursos do OpenTelemetry.

=== Relações

Um mapeamento de relação descreve como é criada uma conexão entre dois componentes.

Cada mapeamento de relação:

* Resolve um `sourceId` e um `targetId`.
* Atribui um tipo de relação.
* Produz uma borda direcionada.

As relações são criadas quando existem os componentes de origem e de destino.

== Criação de componentes de serviço a partir de rastreamentos

=== Definição do que significa um "serviço"

Antes de escrever o mapeamento, precisamos tomar uma decisão de design.

Para este guia, um serviço é definido como:

* Identificado por `service.namespace` + `service.name`.
* Estável em todas as implementações.
* Independente das instâncias de serviço.

Isso mantém a topologia legível e com baixa cardinalidade.

== Componentes de instância de serviço

Os componentes da instância de serviço já estão definidos e são fornecidos pelo OpenTelemetry StackPack.

Cada instância de serviço:

* É derivado de `service.name`, `service.namespace` e `service.instance.id`.
* Representa uma instância concreta em execução de um serviço.
* É estável em todos os sinais (traços e métricas).

Como esse mapeamento já existe, ele não é redefinido neste guia.
Em vez disso, construímos sobre ele.

== Criação de componentes de processo a partir de rastros

=== Definição do que significa um "processo"

Para este guia, um processo é definido como:

* Identificado por `host.name`, `process.pid` e metadados executáveis.
* Com escopo para um único ambiente de tempo de execução.
* Derivado exclusivamente de atributos de recursos do OpenTelemetry.

Isso mantém a topologia com baixa cardinalidade e, ao mesmo tempo, expõe detalhes úteis do tempo de execução.

=== Mapeamento de componentes do processo

O mapeamento de componentes a seguir cria um componente de topologia por processo observado.

[,yaml]
----
_type: "OtelComponentMapping"
name: "OTel Process"
description: "Represents an operating system process derived from OpenTelemetry"
identifier: "urn:stackpack:<stackpack-name>:shared:otel-component-mapping:process"
input:
  signal:
    - "TRACES"
  resource:
    condition: |
      'host.name' in resource.attributes &&
      'process.pid' in resource.attributes
    action: "CREATE"
vars:
  - name: "pid"
    value: "${string(int(resource.attributes['process.pid']))}"
  - name: "hostname"
    value: "${resource.attributes['host.name']}"
  - name: "executableName"
    value: >-
      ${
        'process.executable.name' in resource.attributes ?
         resource.attributes['process.executable.name'] :
          'process.command' in resource.attributes ?
           resource.attributes['process.command'] :
            'process.command_args' in resource.attributes ?
             resource.attributes['process.command_args'] :
                resource.attributes['process.executable.path']
      }
output:
  identifier: "urn:opentelemetry:process/${vars.hostname}:${vars.pid}"
  name: "${vars.hostname}/${vars.executableName}:${vars.pid}"
  typeName: "process"
  typeIdentifier: "urn:stackpack:open-telemetry:shared:component-type:process"
  layerName: "Applications"
  domainName: "Open Telemetry"
  required:
    tags:
      - source: "process-component"
        target: "custom"
      - source: "${resource.attributes}"
        pattern: "process.(.*)"
        target: "process.${1}"
expireAfter: 900000
----

==== Como esse mapeamento funciona

* Processamos somente dados de rastreamento (sinal TRACES).
* Um componente de processo é criado sempre que `host.name` e `process.pid` estão presentes.
* O identificador é estável durante o tempo de vida do processo.
* Uma tag personalizada é adicionada para ajudar na filtragem (para o estágio de verificação).

Substitua `<stackpack-name>` pelo nome do seu StackPack (se você ainda não tiver um, use qualquer nome que desejar, como `mystackpack`).

== Criação de relações de execução

Agora que as instâncias de serviço e os processos existem como componentes, podemos conectá-los.

=== A instância de serviço executa o processo

O mapeamento de relação a seguir cria uma relação `executes` de uma instância de serviço para um processo.

Esse mapeamento é adaptado da relação existente "host executa instância de serviço" fornecida pelo OpenTelemetry StackPack.

[,yaml]
----
_type: "OtelRelationMapping"
name: "Executes Relation (Service Instance -> Process)"
description: "Service instance executes a process"
identifier: "urn:stackpack:<stackpack-name>:shared:otel-relation-mapping:executes-service-instance"
input:
  signal:
    - "TRACES"
  resource:
    condition: |
      'service.name' in resource.attributes &&
      'host.name' in resource.attributes &&
      'process.pid' in resource.attributes
    action: "CREATE"
vars:
  - name: "namespace"
    value: "${'service.namespace' in resource.attributes && resource.attributes['service.namespace'] != '' ? resource.attributes['service.namespace'] : 'default'}"
  - name: "service"
    value: "${resource.attributes['service.name']}"
  - name: "instanceId"
    value: >-
      ${
        'service.instance.id' in resource.attributes && resource.attributes['service.instance.id'] != '' ?
        resource.attributes['service.instance.id'] :
        resource.attributes['service.name']
      }
  - name: "hostname"
    value: "${resource.attributes['host.name']}"
  - name: "pid"
    value: "${string(int(resource.attributes['process.pid']))}"
output:
  sourceId: "urn:opentelemetry:namespace/${vars.namespace}:service/${vars.service}:serviceInstance/${vars.instanceId}"
  targetId: "urn:opentelemetry:process/${vars.hostname}:${vars.pid}"
  typeName: "executes"
expireAfter: 900000
----

==== Como esse mapeamento funciona

* Processamos somente dados de rastreamento (sinal`TRACES` ).
* A relação é criada sempre que a instância de serviço e os dados do processo estiverem presentes.
* Não é necessária nenhuma correlação de span.
* A expressão `targetId` precisa ser a mesma que a expressão `output.identifier` do mapeamento do componente do processo.

O identificador de relação é construído (automaticamente) a partir de `sourceId` e `targetId`, e tem o formato: `sourceId-relationId`

Substitua `<stackpack-name>` pelo nome do seu StackPack (se você ainda não tiver um, use qualquer nome que desejar, como `mystackpack`).

== Validação dos mapeamentos OTel

Há duas opções para validar a exatidão dos mapeamentos antes de implantá-los na produção.

1. Usar o comando `sts stackpack test-deploy` para empacotar, fazer upload e instalar/atualizar um StackPack contendo os mapeamentos para uma instância em execução do {stackstate-product-name}.
2. Usar os comandos `sts otel-component-mapping apply` e `sts otel-relation-mapping apply` para criar/atualizar os mapeamentos individualmente em uma instância {stackstate-product-name} em execução.

=== Testar os mapeamentos juntos em um StackPack

Supondo que ambos os mapeamentos estejam dentro de seu StackPack, eles podem ser testados juntos. Consulte a documentação xref:/setup/cli/cli-sts.adoc[da CLI do StackPack] para obter mais informações.

Usando o comando `sts stackpack test-deploy --yes`, você pode:

* Empacotar, fazer upload e instalar/atualizar o StackPack
* Validar o componente declarativo e os mapeamentos de relação que residem no StackPack (por exemplo, correção da expressão, referência correta dos dados do sinal de entrada com base na filtragem fornecida)

[NOTE]
====
O comando `sts stackpack test` não alimenta os dados de rastreamento de exemplo por meio dos mapeamentos.
Para verificar se os mapeamentos produzem a topologia correta, certifique-se de que os dados de rastreamento do Open Telemetry sejam enviados ao SUSE Observability.
Consulte a seção xref:/setup/custom-integrations/develop.adoc[Desenvolver uma integração personalizada (StackPack)] para obter mais detalhes sobre como usar o site `sts stackpack test`.
====

=== Testar os mapeamentos individualmente

Supondo que os mapeamentos de componentes e relações acima estejam definidos em arquivos YAML, eles podem ser aplicados individualmente.

Substitua `<stackpack-name>` pelo nome do seu StackPack (se você ainda não tiver um, use qualquer nome que desejar, como `mystackpack`).

[,sh]
----
$ sts otel-component-mapping apply -f process-component-mapping.yaml
✅ OTel component mapping upserted successfully! Identifier: urn:stackpack:mystackpack:shared:otel-component-mapping:process, Name: OTel Process

$ sts otel-relation-mapping apply -f process-relation-mapping.yaml
✅ OTel Relation Mapping upserted successfully! Identifier: urn:stackpack:mystackpack:shared:otel-relation-mapping:executes-service-instance, Name: Executes Relation (Service Instance -> Process)
----

== Topologia resultante

Quando esses mapeamentos são aplicados, a topologia resultante forma um gráfico de topologia de processo de serviço derivado inteiramente de dados de rastreamento.

Por exemplo, usando o site `checkoutservice` do aplicativo de demonstração da OTel, visualmente, a topologia deve ter a seguinte aparência:

----
checkoutservice (instance) ─> executes ─> checkoutservice (process)
----

Essa topologia é atualizada continuamente à medida que novos rastros chegam e expira automaticamente quando o tráfego é interrompido.

=== Visualize a topologia resultante em {stackstate-product-name}

Use a interface de usuário {stackstate-product-name} para obter confirmação visual de que os mapeamentos se materializam na topologia esperada.

1. Abra a interface do usuário {stackstate-product-name} no valor configurado do `baseUrl` Helm
2. Na barra lateral esquerda, clique em `Open Telemetry > Services Instances`
3. Localize o site `checkoutservice` na lista de instâncias de serviço e clique no nome da instância de serviço para abrir a página Visão geral dos componentes/Highlights
4. Na barra de navegação secundária na parte superior, selecione `Topology`
5. Na camada `Outgoing`, um nó `od-checkoutservice-<hostId>/checkoutservice:1 (process)` deve estar visível
6. Selecione o nó do processo e clique em `Explore component`
7. Uma topologia menor visualizando `checkoutservice -> executes -> checkoutservice (process)` deve estar visível

Consulte o guia xref:/setup/custom-integrations/otelmappings/troubleshooting.adoc[Solução de problemas] para obter dicas de depuração se a topologia não estiver se materializando conforme o esperado.

== Limpeza

Para remover os mapeamentos criados neste guia, use os seguintes comandos:

Substitua `<stackpack-name>` pelo nome do seu StackPack.

[,sh]
----
$ sts otel-component-mapping delete --identifier urn:stackpack:<stackpack-name>:shared:otel-component-mapping:process
✅ OTel Component Mapping deleted: urn:stackpack:mystackpack:shared:otel-component-mapping:process

$ sts otel-relation-mapping delete --identifier urn:stackpack:<stackpack-name>:shared:otel-relation-mapping:executes-service-instance
✅ OTel Relation Mapping deleted: urn:stackpack:mystackpack:shared:otel-relation-mapping:executes-service-instance
----


== Resumo

Neste guia, você:

* Criado com base em componentes de instância de serviço existentes.
* Componentes de processo derivados de atributos de recursos do OpenTelemetry.

== Próximas etapas

A partir daqui, você pode:

* Anexe processos a hosts ou contêineres.
* Introduzir relações de banco de dados ou de mensagens.
* Introduzir camadas específicas de tempo de execução (JVM, Go, Node.js).
* Explore gráficos de serviços baseados em métricas.
* Amplie o StackPack com camadas adicionais de topologia.
* Familiarize-se com uma xref:/setup/custom-integrations/otelmappings/schemas-ref.adoc[referência] detalhada de mapeamento de componentes e relações.
