== Visão geral

{stackstate-product-name} pode ser exposto com um recurso do Kubernetes Ingress. Os exemplos nesta página mostram como configurar um controlador Traefik ou Nginx Ingress usando xref:/setup/install-stackstate/kubernetes_openshift/ingress.adoc#_configure_ingress_via_the_suse_observability_helm_chart[o Helm para SUSE Observability em execução no Kubernetes]. Esta página também documenta qual combinação de serviço/porta deve ser exposta ao usar um método diferente de configuração do tráfego de entrada.

Ao observar o cluster que também hospeda o site {stackstate-product-name}, o tráfego do agente pode ser mantido inteiramente dentro do próprio cluster, xref:/setup/install-stackstate/kubernetes_openshift/ingress.adoc#_agents_in_the_same_cluster[alterando a configuração do agente] durante a instalação do agente.

== Configure o ingresso por meio do gráfico {stackstate-product-name} Helm

O gráfico do {stackstate-product-name} Helm expõe uma seção `ingress` em seus valores. Isso é desativado por padrão. Os exemplos abaixo mostram como usar o gráfico Helm para configurar um controlador de entrada com a criptografia TLS ativada. Observe que a configuração do próprio controlador e dos certificados está além do escopo deste documento.

Para configurar o ingresso para {stackstate-product-name}, crie um arquivo `ingress_values.yaml` com o conteúdo abaixo. Substitua `MY_DOMAIN` pelo seu próprio domínio (que está vinculado ao seu controlador de entrada) e defina o nome correto para `tls-secret`. Consulte a documentação do seu controlador de entrada para obter as anotações corretas a serem definidas. Todos os campos abaixo são opcionais; por exemplo, se não for usado TLS, omita essa seção, mas saiba que {stackstate-product-name} também não criptografa o tráfego.

[CAUTION]
====
Observe que a configuração do TLS é necessária para o uso da extensão da interface do usuário do rancher.
====

Traefik::
+
--
.Problemas conhecidos
[WARNING]
=====
Certifique-se de usar o Traefik < 3.6.3 or >= 3.6.7 (com a versão do gráfico Helm >= 39.0.0), pois as versões 3.6.3 a 3.6.6 apresentam um problema que causa erros em alguns pontos de extremidade da API. Em particular, as páginas de destaque do componente não são carregadas devido ao bloqueio das chamadas de API pelo Traefik com um erro `400 Bad Request`.

[,text]
----
ingress:
  enabled: true
  ingressClassName: traefik
  annotations:
    traefik.ingress.kubernetes.io/router.entrypoints: websecure
  hosts:
    - host: suse-observability.MY_DOMAIN
  tls:
    - hosts:
        - suse-observability.MY_DOMAIN
      secretName: tls-secret
----

        - suse-observability.MY_DOMAIN

O que se destaca nesse arquivo é a anotação do Nginx para aumentar a permissão de `proxy-body-size` para `50m` (maior do que qualquer solicitação esperada). Por padrão, o Nginx permite tamanhos de corpo de no máximo `1m`. {stackstate-product-name} Os agentes e outros provedores de dados podem, às vezes, enviar solicitações muito maiores. Por esse motivo, você deve se certificar de que o tamanho do corpo permitido seja grande o suficiente, independentemente de estar usando o Nginx ou outro controlador de entrada.

== Configurar regra de entrada para telemetria aberta

O gráfico do {stackstate-product-name} Helm expõe um serviço `opentelemetry-collector` em seus valores, no qual um `ingress` dedicado pode ser criado. Isso é desativado por padrão. O ingresso necessário para fins de `opentelemetry-collector` precisa ser compatível com o protocolo GRPC. Observe que a configuração do próprio controlador e dos certificados está além do escopo deste documento.

Para configurar a entrada `opentelemetry-collector` para {stackstate-product-name}, crie um arquivo `ingress_otel_values.yaml` com o conteúdo abaixo. Substitua `MY_DOMAIN` pelo seu próprio domínio (que está vinculado ao seu controlador de entrada) e defina o nome correto para `otlp-tls-secret`. Consulte a documentação do seu controlador de entrada para obter as anotações corretas a serem definidas. Todos os campos abaixo são opcionais; por exemplo, se não for usado TLS, omita essa seção, mas saiba que {stackstate-product-name} também não criptografa o tráfego.

[NOTE]
=====
A exposição do ponto de extremidade GRPC do OpenTelemetry Collector via Traefik é compatível a partir da versão v2.8.0 do gráfico do {stackstate-product-name} Helm.

Observe que a entrada do host GRPC requer um `serviceName: suse-observability-otel-collector-grpc` explícito para rotear o tráfego para o serviço de backend correto.
=====

[,text]
----
opentelemetry-collector:
  ingress:
    enabled: true
    ingressClassName: traefik
    annotations:
      traefik.ingress.kubernetes.io/router.entrypoints: websecure
    hosts:
      - host: otlp-suse-observability.MY_DOMAIN
        paths:
          - path: /
            pathType: Prefix
            serviceName: suse-observability-otel-collector-grpc
            port: 4317
    tls:
      - hosts:
          - otlp-suse-observability.MY_DOMAIN
        secretName: otlp-tls-secret
    additionalIngresses:
      - name: otlp-http
        ingressClassName: traefik
        annotations:
          traefik.ingress.kubernetes.io/router.entrypoints: websecure
        hosts:
          - host: otlp-http-suse-observability.MY_DOMAIN
            paths:
              - path: /
                pathType: Prefix
                port: 4318
        tls:
          - hosts:
              - otlp-http-suse-observability.MY_DOMAIN
            secretName: otlp-http-tls-secret
----

== Configurar por meio de ferramentas externas

Para tornar o {stackstate-product-name} acessível fora do cluster do Kubernetes em que está instalado, basta rotear o tráfego para a porta `8080` do serviço `<namespace>-stackstate-k8s-router`. A interface do usuário de {stackstate-product-name} pode ser acessada diretamente no caminho raiz desse serviço (ou seja, `http://<namespace>-stackstate-k8s-router:8080`), enquanto os agentes usarão o caminho `/receiver` (`http://<namespace>-stackstate-k8s-router:8080/receiver`).

Certifique-se de atualizar o endereço `baseUrl` no arquivo de valores gerado durante a instalação inicial, pois ele será usado por {stackstate-product-name} para gerar instruções de instalação convenientes para o agente.

[NOTE]
====
Ao configurar manualmente um servidor HTTP Nginx ou similar como proxy reverso, certifique-se de que ele também possa fazer proxy de websockets. Para o Nginx, isso pode ser configurado com a inclusão das seguintes diretivas na diretiva `location`:

[,text]
----
proxy_set_header Upgrade                 $http_upgrade;
proxy_set_header Connection              "Upgrade";
----

====


[CAUTION]
====
{stackstate-product-name} não usa tráfego criptografado por TLS, espera-se que a criptografia TLS seja tratada pelo controlador de entrada ou por balanceadores de carga externos.
====


== Agentes no mesmo cluster

Os agentes implantados no mesmo cluster que {stackstate-product-name} podem, obviamente, usar a URL externa na qual {stackstate-product-name} está exposto, mas também é possível configurar o agente para se conectar diretamente à instância {stackstate-product-name} somente por meio da rede interna do Kubernetes. Para fazer isso, substitua o valor de `'stackstate.url'` no comando `helm install` da xref:/k8s-quick-start-guide.adoc[instalação do Agente Kubernetes] pela URL do cluster interno do serviço de roteador (veja também acima): `http://<namespace>-suse-observability-router.<namespace>.svc.cluster.local:8080/receiver/stsAgent` (as seções `<namespace>` precisam ser substituídas pelo namespace de {stackstate-product-name}).


== Encaminhamento da porta de serviço do roteador {stackstate-product-name} 

Por padrão, o gráfico do {stackstate-product-name} Helm implementa um pod e um serviço de roteador. Esse serviço expõe a porta `8080`, que é o único ponto de entrada que precisa ser exposto via Ingress.
Também é possível acessar {stackstate-product-name} por meio de encaminhamento de porta; ao fazer isso, você deve permitir localhost como origem da solicitação.

[WARNING]
====
* A permissão de `localhost` em `stackstate.allowedOrigins` destina-se apenas ao desenvolvimento local ou à depuração ao usar o encaminhamento de porta.
* Esta *não* é uma configuração de produção. Remova essa origem permitida ao expor {stackstate-product-name} via Ingress.
====

Para acessar a interface do usuário sem configurar o Ingress, encaminhe a porta de serviço do roteador:

[,text]
----
kubectl port-forward service/<helm-release-name>-suse-observability-router 8080:8080 --namespace suse-observability
----

Ao acessar {stackstate-product-name} por meio de encaminhamento de porta, o navegador usa `http://localhost:8080` como a origem da solicitação.
Para permitir solicitações dessa origem, adicione-a à lista `stackstate.allowedOrigins` nos valores do Helm ou passe-a diretamente para o comando Helm::

[,bash]
----
helm upgrade \
  --install \
  --namespace suse-observability \
  --values $VALUES_DIR/suse-observability-values/templates/baseConfig_values.yaml \
  --values $VALUES_DIR/suse-observability-values/templates/sizing_values.yaml \
  --values $VALUES_DIR/suse-observability-values/templates/affinity_values.yaml \
  --set stackstate.allowedOrigins={"http://localhost:8080"} \
suse-observability \
suse-observability/suse-observability
----
